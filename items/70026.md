## Summary

Keep open. Current `main` still routes supervisor cancellation to `SIGKILL`, the child adapter still hard-kills the supervised process immediately on that path, and the session-lock cleanup code still only handles catchable termination signals. The latest release `v2026.4.23` shipped the same supervisor behavior, so this is not implemented on `main` and not obsolete.

## Evidence

- **Supervisor still cancels runs with SIGKILL on main:** `createProcessSupervisor()` sets `cancelAdapter` to call `adapter.kill("SIGKILL")` for cancellation paths, including timeout-driven cancellation.
  - file: [src/process/supervisor/supervisor.ts:172](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/process/supervisor/supervisor.ts#L172)

- **Child adapter still hard-kills the supervised process immediately:** When the supervisor passes `SIGKILL`, the child adapter enters the `SIGKILL` branch, calls `killProcessTree(pid)`, and then immediately calls `child.kill("SIGKILL")`, so the supervised process itself still receives an uncaught hard kill.
  - file: [src/process/supervisor/adapters/child.ts:288](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/process/supervisor/adapters/child.ts#L288)
