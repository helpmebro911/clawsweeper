## Summary

Keep #44523 open. Current main still has the reported gap: explicit model/provider overrides are copied when an existing entry is reused and when an explicit `/new`/`/reset` runs, but not when a stale daily freshness rollover creates a new session with `resetTriggered=false`. The PR is stale against newer `modelOverrideSource` semantics, but the user-visible bug it targets is not implemented on main.

## Best Possible Solution

Keep the PR open and ask for a rebase/update against current main. The best fix is to preserve user-driven or legacy model/provider selections across implicit daily/scheduled stale rollovers, likely by reusing the current reset-preserved-selection policy outside the `resetTriggered`-only branch, while keeping behavior overrides scoped to explicit `/new` and `/reset`. Add a regression test for a stale 3 AM entry observed at 5 AM with `modelOverrideSource: "user"`, and include an auto-sourced override case if the helper is reused.
