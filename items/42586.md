## Summary

Keep open. Current `main` still has the health-snapshot/runtime-state gap the PR is trying to fix: `health` builds channel snapshots without gateway runtime state, while `channels.status` explicitly reads and merges runtime snapshots. I did not find evidence in the checked-out `main` code that this PR is already implemented or obsolete.

## Evidence

- **`getHealthSnapshot()` still lacks runtime input and builds minimal snapshots:** On current `main`, `getHealthSnapshot()` only accepts `{ timeoutMs, probe }`, and the per-account snapshot it passes to `buildChannelSummary` is built from `accountId`, `enabled`, `configured`, plus optional `probe`/`lastProbeAt`; there is no runtime merge for `running`/`connected`.
  - file: [src/commands/health.ts:215](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/commands/health.ts#L215)
  - command: `nl -ba src/commands/health.ts | sed -n '215,340p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Gateway health refresh path does not thread channel runtime state:** `refreshGatewayHealthSnapshot()` still calls `getHealthSnapshot({ probe: opts?.probe })` with no runtime snapshot, so the gateway `health` endpoint cannot pick up live channel runtime fields from this path.
  - file: [src/gateway/server/health-state.ts:72](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/gateway/server/health-state.ts#L72)

- **Maintenance/startup wiring still omits `getRuntimeSnapshot` for health refresh:** `startGatewayMaintenanceTimers()` does not accept or set any runtime-snapshot callback, and the startup callsite passes `refreshGatewayHealthSnapshot` without a `getRuntimeSnapshot` parameter.
