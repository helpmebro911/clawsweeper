## Summary

Current `main` still lacks the runtime-state merge described in PR #42586, so the health endpoint/channel-status mismatch appears unresolved and this PR should remain open.

## Evidence

- **`getHealthSnapshot` still has no runtime merge path:** On `main`, `getHealthSnapshot` only accepts `timeoutMs` and `probe`. It builds each `ChannelAccountSnapshot` from config/probe data and never merges runtime fields like `running` or `connected` before calling `buildChannelSummary`.
  - file: [src/commands/health.ts:215](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/health.ts#L215)
  - command: `nl -ba src/commands/health.ts | sed -n '215,336p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Gateway health refresh is not wired to runtime snapshots:** `refreshGatewayHealthSnapshot` still calls `getHealthSnapshot({ probe: opts?.probe })` directly, and there is no `setRuntimeSnapshotFn` hook in this file on `main`, so the cached health payload cannot see channel manager runtime state.
  - file: [src/gateway/server/health-state.ts:72](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/gateway/server/health-state.ts#L72)
  - command: `nl -ba src/gateway/server/health-state.ts | sed -n '1,140p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **`channels.status` already uses runtime state:** The gateway `channels.status` handler pulls `context.getRuntimeSnapshot()`, resolves per-account runtime entries, and passes them into `buildChannelAccountSnapshot`. That means the discrepancy described in the PR remains plausible on current `main`: `channels.status` sees runtime state while `health` does not.
  - file: [src/gateway/server-methods/channels.ts:161](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/gateway/server-methods/channels.ts#L161)
