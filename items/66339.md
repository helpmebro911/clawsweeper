## Summary

Keep this issue open. Current main still has mitigations for QMD SQLite contention, but not a source-backed fix for the reported gateway-vs-CLI runtime race: gateway background QMD managers and CLI memory search managers can be separate processes, while QMD update serialization is process-local and search only waits on the same manager's pending update.

## Best Possible Solution

Keep this issue open and fix it in the memory-core QMD owner path. The likely maintainer direction is a per-agent cross-process coordination mechanism around QMD update/embed maintenance and foreground search, such as a shared update lock plus lock-aware wait/skip/read behavior, with a regression test that models gateway background QMD maintenance and a separate CLI `memory search` contending for the same agent index. Preserve the existing same-process queue, read-only busy_timeout, embed lock, and builtin fallback as mitigations, but do not treat them as the complete fix.

- A future fix should avoid a broad blocking lock that can deadlock or make searches hang; it needs bounded wait/skip behavior and stale-lock handling.
