## Summary

Keep this issue open. Current main has useful QMD contention mitigations, but the reported normal-runtime gateway-vs-CLI race is still not fully coordinated: gateway startup and CLI memory search can create separate QMD managers in separate processes, while update serialization is process-local and search only waits briefly for the same manager instance's pending update.

## Best Possible Solution

Keep this open and fix it in the memory-core QMD owner path. The likely maintainer direction is per-agent cross-process coordination around QMD update/embed maintenance and foreground search, with bounded wait/skip/read behavior and a regression test that models gateway background QMD work contending with a separate CLI `memory search` for the same agent index.

## Risks / Open Questions

- No live macOS/QMD reproduction was run in this read-only review, but the current source still shows the missing cross-process coordination needed for the reported race.
- A future fix should avoid unbounded cross-process locks; it needs bounded wait, stale-lock handling, and skip/read-only behavior so foreground searches do not hang.
