## Summary

Keep #66339 open. Current main has mitigations around QMD busy failures, but it still does not fully coordinate foreground `openclaw memory search` with gateway/background QMD update work. CLI search builds a full QMD manager, full QMD initialization starts boot update by default, search waits only a bounded 500ms for pending in-process updates, and QMD update serialization is in-process only while the shared file lock is only applied to embeds. That leaves the reported cross-process SQLite lock contention path plausible and unclosed.

## Best Possible Solution

Keep the issue open and fix QMD runtime coordination directly. The likely product path is to add cross-process coordination for `qmd update` similar to the existing embed file lock, or make foreground CLI search/index detect an active gateway/background update and wait, skip refresh, or use a read-only query path. Add a regression that simulates separate managers/processes contending on boot/onSearch/watch updates, and update QMD docs only if user-visible behavior or config changes.
