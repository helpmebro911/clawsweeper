## Summary

Keep open. Current main still renders Telegram model buttons from the provider-local model id when no catalog display name overrides it, so an OpenRouter ref like `openrouter/openai/gpt-5.4-mini` still falls back to visible text `openai/gpt-5.4-mini`. The related #67797 was about OpenRouter reasoning defaults, not this Telegram label path.

## Best Possible Solution

Keep this issue open and fix it in the bundled Telegram plugin. Update `extensions/telegram/src/model-buttons.ts` so fallback button labels use the canonical `provider/model` text for nested provider-local ids, while preserving callback data and selection resolution. Add a regression test in `extensions/telegram/src/model-buttons.test.ts` for provider `openrouter` and model `openai/gpt-5.4-mini` expecting visible text to include `openrouter/openai/gpt-5.4-mini`.
