## Summary

Keep this PR open. Current main has the later local-timezone session-memory fix, but it still derives the memory filename date, fallback slug, and Markdown header from the `/new` or `/reset` hook event timestamp rather than the persisted session creation timestamp. The PR is still aimed at a real remaining behavior gap, although it needs rebase and review fixes before it is mergeable.

## Best Possible Solution

Keep the PR open, but require a current-main rebase and a narrower implementation: add a transcript helper that returns both recent content and the selected transcript header timestamp, preserve the current local-timezone formatting behavior, use reset-header fallback only when the reset transcript is the content source, and add regression tests for normal headers, rotated reset transcripts, invalid/missing headers, and local-timezone day boundaries.

## Risks / Open Questions

- The PR is based on older session-memory code and must be rebased over the current local-timezone implementation so session-created timestamps are formatted consistently with host-local filename/header behavior.
- The review discussion identifies a correctness gap: reset-file timestamp fallback should match the transcript actually selected by the content fallback and should not choose an older reset sibling for a non-empty transcript with an invalid or legacy header.
- The patch adds another full transcript read for the first-line timestamp; a refreshed implementation should avoid doubling I/O on large transcripts, for example by extracting the header during the same transcript read or by reading only the first line.
- The fetched PR patch references `makeSessionMemoryConfig(tempDir)` in a new test, but that helper is not present in current main’s `handler.test.ts`; the rebase should make the test compile against current helpers.
