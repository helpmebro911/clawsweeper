## Summary

Keep open. This is still a recent, concrete Matrix regression report against Docker/local Synapse where only OpenClaw changed. Current main has relevant Matrix E2EE delivery and verify-status fixes, but I did not find enough evidence that the broader no-response/no-inbound-log symptom is fixed for the reporter's deployment.

## Best Possible Solution

Keep this issue open until a Matrix owner reproduces or verifies against current main or the next release in a Docker deployment with local Synapse. Capture Matrix gateway startup logs, sync readiness states, room encryption status, `openclaw matrix verify status --verbose`, and whether inbound events reach `room.message`, `room.decrypted_event`, or `room.failed_decryption`. If current main fixes it, close as implemented with the specific Matrix decrypted-message/verify-status commits and release evidence; if it still reproduces, fix likely belongs in `extensions/matrix` around shared-client startup, sync readiness, or encrypted/decrypted event delivery.
