## Summary

Keep this PR open. Current main still hardcodes the 5 MB media store, media serving, and sandbox staging limits, and the config schema/type/help surface still has no `media.maxBytes`. The PR is a plausible fix for the linked bug, but it is stale/incomplete against current main and has unresolved review-blocking runtime concerns around missing imports, startup wiring, and global-vs-channel cap semantics.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40880, but require an update against current main. The best fix is to add `media.maxBytes` consistently to schema, types, labels/help, generated config metadata, and user docs; initialize the effective cap unconditionally during gateway startup with a default reset; expose/import the getter through the media runtime boundary; apply it to store, media serving, and sandbox staging; define interaction with channel-specific limits; and add focused regression tests. Security review found no supply-chain-sensitive file changes in the provided PR file list, but the resource cap behavior needs explicit bounded enforcement before merge.
