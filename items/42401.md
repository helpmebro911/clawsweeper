## Summary

Current `main` does not implement this request on the shipped `models status --probe` surface. `auth.probes` still returns only status/error/latency-style results, while the newer usage work is a separate human-readable status summary with normalized quota windows, not per-probe rate-limit headroom fields.

## Evidence

- **Probe result schema has no headroom fields:** `AuthProbeResult` only includes `provider`, `model`, `profileId`, `label`, `source`, `mode`, `status`, `reasonCode`, `error`, and `latencyMs`. There is no `rateLimit` or equivalent structure for remaining/limit/reset data.
  - file: [src/commands/models/list.probe.ts:61](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/models/list.probe.ts#L61)
  - command: `nl -ba src/commands/models/list.probe.ts | sed -n '50,90p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Probe execution only builds status/error/latency results:** `probeTarget()` constructs each probe result via `buildResult()` and only populates `status`, optional `error`, and `latencyMs` after `runEmbeddedPiAgent()`. No response headers or provider headroom data are captured or surfaced.
  - file: [src/commands/models/list.probe.ts:449](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/models/list.probe.ts#L449)
