## Summary

Keep open. Current main still does not set `Cache-Control`, `Pragma`, or `Expires` on temporary `/media/:id` responses, while the PR is a focused security hardening change for that exact endpoint. It is not superseded by the related noVNC hardening because that lives under the browser bridge route, not `src/media`. The PR needs a rebase onto the current `createMediaRequestHandler` shape and should address the minor missing test assertions noted by Greptile, but the underlying change remains useful.

## Best Possible Solution

Keep the PR open, rebase it onto current `src/media/server.ts`, set anti-cache headers immediately after a matched `/media/:id` route and before all validation/read branches, and expand regression assertions for success, missing-media, HEAD, and representative error responses. After rebase, maintain the narrow scope: no behavior changes to path validation, TTL expiry, cleanup timing, or file serving.
