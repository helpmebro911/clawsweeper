## Summary

Keep PR #70757 open. The checked-out main still has the BlueBubbles webhook auth boundary coupled to `channels.bluebubbles.password`, and the docs still tell operators to publish that same password on the webhook URL. The PR is a plausible focused fix for linked issue #70755, with later commits addressing the review comments called out in the provided discussion.

## Best Possible Solution

Keep PR #70757 open for maintainer security review. The desired direction is to land or refine the dedicated BlueBubbles `webhookSecret` approach: runtime webhook auth should compare against that separate secret, setup and schemas should require it to differ from the API password, docs should stop publishing the API password in webhook URLs, and regression tests should cover password-only and duplicate-secret rejection.
