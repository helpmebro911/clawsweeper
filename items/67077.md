## Summary

Keep this PR open. Current main still allows auth-profile bookkeeping persistence failures to escape through the post-success mark paths, and the linked Windows EPERM issue remains a concrete, reproducible bug. The PR is a focused implementation candidate with regression coverage and no obvious supply-chain or infrastructure drift.

## Best Possible Solution

Keep this PR open, rebase it onto current main, review the exact catch/logging boundaries, and land this or an equivalent targeted fix with regression coverage for non-fatal post-completion auth-profile bookkeeping save failures. OAuth/token-refresh persistence failures should continue to propagate. After merge, #62099 can be closed for the gateway failure cascade, with separate follow-ups only if maintainers want ReadOnly attribute repair or error-loop context hardening.
