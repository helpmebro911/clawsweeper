## Summary

Keep this issue open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends and passes that same value as the outbound RPC `text`, so the reported visible placeholder behavior remains present. The related fix PRs #40341 and #46343 are closed unmerged in the provided GitHub context, and current tests still encode placeholder-based media echo/cache behavior.

## Best Possible Solution

Keep this issue open. The best fix belongs in the iMessage plugin send path: stop generating visible `<media:...>` fallback text for media-only sends, send an empty string plus `file` to the iMessage RPC when media exists, preserve text+media captions and reply IDs, and update the media reply/self-chat dedupe tests so placeholder text is no longer required for correctness.
