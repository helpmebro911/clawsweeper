## Summary

Keep open. Current main still turns blank iMessage media sends into visible `<media:image>` / `<media:${kind}>` text, then sends that value as the outbound RPC `text` field with the media file. The two related fix PRs in the provided context, #40341 and #46343, are closed unmerged, and the latest release tag still contains the same implementation.

## Best Possible Solution

Keep this issue open. The best fix belongs in the iMessage plugin send path: stop generating visible `<media:...>` fallback text for media-only outbound sends, send an empty string plus `file` to the iMessage RPC when media exists, preserve captions and `reply_to` metadata for text+media sends, and update the deliver/echo/self-chat tests so dedupe no longer depends on user-visible placeholder text.
