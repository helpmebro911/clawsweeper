## Summary

Keep open. Current `main` still synthesizes `<media:image>`/`<media:...>` for media-only iMessage sends and passes that value as the outbound `text`, and the iMessage delivery tests still assert that placeholder as the actual sent text. I found no evidence of a fix on `main` or in the latest release `v2026.4.22`.

## Evidence

- **Current iMessage send path still injects placeholder text:** `resolveDeliveredIMessageText()` returns `<media:image>` or `<media:${kind}>` when `text` is blank, and `sendMessageIMessage()` applies it for media sends before building RPC params.
  - file: [extensions/imessage/src/send.ts:86](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/extensions/imessage/src/send.ts#L86)

  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Placeholder is still sent as RPC text on media sends:** After resolving media, the send path sets `params.text = message`; with media-only sends that `message` is the synthesized placeholder, so the outbound RPC still carries visible text alongside the attachment.
  - file: [extensions/imessage/src/send.ts:126](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/extensions/imessage/src/send.ts#L126)
