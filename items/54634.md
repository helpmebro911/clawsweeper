## Summary

Keep open. Current main still does not implement the reported cross-HOME update safeguard. It has useful partial hardening for same-path config clobbers, status mismatch diagnostics, and service env filtering, but the package-update/service-refresh path remains rooted in the current effective HOME unless it can already find and carry the prior service environment. That does not reliably detect, preserve, migrate, warn, or block when an update is run with HOME=/root after a service/config lived under /home/openclaw.

## Best Possible Solution

Keep this issue open and fix the update service refresh/install path directly. Before regenerating an installed gateway service, compare the previous effective roots (HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, OPENCLAW_CONFIG_PATH) with the update invocation roots, discover an existing service even when HOME differs, and then preserve the prior root when safe or emit an explicit migration/warning/block. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root.
