## Summary

Keep open. Current main has partial config-clobber recovery and doctor diagnostics, but the specific HOME-change update path is still not fixed: update refresh still force-runs gateway install from the current invocation environment, forced reinstall still drops existing service HOME and OPENCLAW_* roots, and config recovery remains scoped to the active config path.

## Best Possible Solution

Keep this issue open and fix the service refresh/install path directly. Before regenerating an installed gateway service, preserve or validate the previous effective config root (HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, OPENCLAW_CONFIG_PATH) and warn or block when a forced reinstall would silently switch roots. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root; expected behavior should be explicit preservation, audited migration, or a hard warning rather than creating a fresh root config.

## Evidence

- **Issue discussion supports the root cause:** The report and first comment identify the same failure mode: updating with a different root/HOME causes the service to ignore the existing /home/openclaw/.openclaw config and start from /root/.openclaw instead. The later GHCR version-tag comment is useful rollback feedback, but it does not supersede the service/config-root bug.
