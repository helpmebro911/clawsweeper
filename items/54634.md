## Summary

Keep open. Current main still lacks the reported cross-HOME update safeguard: package update, doctor, service discovery, and service refresh still follow the updater process environment unless explicit OpenClaw path overrides are already present. Existing config-clobber and systemd env-file hardening helps adjacent problems, but it does not detect, preserve, migrate, warn, or block when a gateway installed under one HOME is refreshed from another HOME.

## Best Possible Solution

Keep this issue open and fix the Linux update/service refresh path directly. Before regenerating an installed gateway service, discover the previous unit/config root even when the updater runs under a different HOME, compare HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, then either preserve the prior root when safe or emit an explicit migration/warning/block. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root. Treat the GHCR version-tag request from the discussion as separate rollback/pinning work, not as the core fix for this bug.
