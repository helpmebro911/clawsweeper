## Summary

Keep this issue open. Current main has useful partial completion-delivery work, task records, state-change notification plumbing, and subagent plugin hooks, but it still does not implement the central request: framework-owned immediate user-facing spawn ACKs, per-run subagent progress lifecycle notifications, or a durable idempotent announce queue that survives gateway restart/channel outage independently of the parent session lane.

## Best Possible Solution

Keep this open as the umbrella runtime architecture request. The best implementation path is to extend the existing subagent completion delivery and task ledger work with runtime-owned lifecycle events for spawned/progress/completed/failed/cancelled, a framework-generated immediate user-facing spawn acknowledgement, and durable idempotent announce persistence/retry. Coordinate with #22358 for completion hook work and #54488 for session-lane starvation rather than closing this until those remaining pieces ship.

## Risks / Open Questions

- Current completion delivery, task notifications, and subagent hooks are partial foundations; replacing them wholesale could regress direct completion delivery, idempotency, or thread-bound routing.
- Any channel-direct lifecycle notice should remain a narrow framework/status-event path, not a bypass for arbitrary model output or normal channel safety/routing rules.
