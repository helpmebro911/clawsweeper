## Summary

Keep #68977 open. Current main still exposes `enqueueSystemEvent` as an in-memory, session-scoped queue with a boolean return and only consecutive duplicate-text suppression; it does not support a scoped `idempotencyKey`, an `EnqueueResult`, or durable dedupe tied atomically to enqueue. The plugin SDK has a separate disk-backed dedupe helper, but that is not the requested core enqueue primitive and does not remove the enqueue-vs-marker race by itself.

## Best Possible Solution

Keep the issue open for maintainer/API design. The likely implementation path is an additive core change around `src/infra/system-events.ts` and the plugin runtime/docs: accept a scoped `idempotencyKey`, return a structured enqueue result, persist keys at least until consumption, and add tests for same-session duplicates, same key across different sessions, payload mismatch behavior, restart durability, and cleanup semantics. If maintainers decide plugin-level dedupe is sufficient instead, document that as an explicit workaround rather than closing as already implemented.
