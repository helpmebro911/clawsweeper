## Summary

Keep this issue open. Current main has advisory/plugin-owned memory recall paths, including Active Memory and memory-lancedb auto-recall, but it still does not provide the requested enforced recall-before-response mechanism with block-on-failure semantics across the reported agent/subagent orchestration surface. The linked implementation PR #43825 is closed and unmerged.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in runtime policy or plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally block or return a controlled failure when recall cannot run, with explicit coverage for embedded runs, CLI/harness execution, subagents/workflow orchestration, and existing Active Memory/memory-lancedb behavior.

## Risks / Open Questions

- A blanket forced recall step could add latency, cost, and hidden personalization to automation or internal-worker paths, so the eventual design should be opt-in and scoped.
- Changing `before_prompt_build` globally into a hard gate could break existing plugins that rely on best-effort prompt mutation semantics.
- Resurrecting #43825 directly would still need maintainer design review, rebase, and validation across embedded runs, CLI/harness paths, subagents, Active Memory, and memory-lancedb interactions.
