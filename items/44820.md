## Summary

Keep open. Current main has partial session-maintenance coverage for stale session entries and archived `.deleted`/`.reset` artifacts, but it does not fully implement the cleanup requested in #44820: current tests explicitly keep old `*.bak.*` session artifacts during normal maintenance, and there is no separate scavenger for orphan `*.tmp` files in the sessions directory.

## Evidence

- **Normal maintenance only ages out deleted/reset archives:** `saveSessionStoreUnlocked` runs `cleanupArchivedSessionTranscripts` for `reason: "deleted"` and, when configured, `reason: "reset"`; there is no matching age-based cleanup call for `bak` or `tmp` artifacts.
  - file: [src/config/sessions/store.ts:317](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/store.ts#L317)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '290,335p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Current test suite proves old bak artifacts are retained:** `store.pruning.integration.test.ts` asserts that an old `.deleted` archive is removed, a recent `.deleted` archive is kept, and `bak-session.jsonl.bak.*` is still present after `saveSessionStore(...)`. That is direct evidence that normal maintenance does not clean old `*.bak.*` files.
  - file: [src/config/sessions/store.pruning.integration.test.ts:151](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/store.pruning.integration.test.ts#L151)
  - command: `nl -ba src/config/sessions/store.pruning.integration.test.ts | sed -n '150,220p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Bak cleanup is only tied to disk-budget pressure:** Disk-budget enforcement removes files from a queue filtered by `isSessionArchiveArtifactName(...)` or unreferenced primary transcripts. So `*.bak.*` files are only eligible when `maxDiskBytes`/`highWaterBytes` cleanup is triggered, not via ordinary age-based maintenance.
  - file: [src/config/sessions/disk-budget.ts:258](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/disk-budget.ts#L258)
  - command: `nl -ba src/config/sessions/disk-budget.ts | sed -n '258,284p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Tmp files have no independent scavenger:** Atomic writes create `...<uuid>.tmp` files and remove them in the same write path's `finally` block. I did not find any startup/timer/session-maintenance path that later scans the sessions directory for stale `*.tmp` leftovers from interrupted writes or older versions.
  - file: [src/infra/json-files.ts:65](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/infra/json-files.ts#L65)
  - command: `nl -ba src/infra/json-files.ts | sed -n '51,89p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Artifact classification excludes tmp files from maintenance paths:** Session artifact helpers recognize timestamped `.deleted.`, `.reset.`, and `.bak.` names, plus legacy `sessions.json.bak.<n>`. They do not classify `*.tmp` files as session artifacts.
  - file: [src/config/sessions/artifacts.ts:16](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/artifacts.ts#L16)
  - command: `nl -ba src/config/sessions/artifacts.ts | sed -n '1,52p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)

## Risks / Open Questions

- Closing now would overstate the current implementation: the reported file classes are only partially covered on main.
- Users can still accumulate stale `*.bak.*` and `*.tmp` files unless disk-budget cleanup happens or they remove them manually.
