## Summary

Keep open. The issue is concrete, recent, and still matches current main: light dreaming ingests daily/session signals, updates short-term recall state, then sorts light candidates by `lastRecalledAt` before `recallCount`; deep promotion still gates `minRecallCount` using a combined signal count that includes daily/grounded support; and the rehydration search span is still `Math.max(preferredSpan + 3, 8)`. The provided related items do not supersede this report; the only timeline cross-reference is an unrelated DeepSeek issue, and the local memory-status/perf reports cover different surfaces.

## Best Possible Solution

Keep this issue open for memory-core maintainer triage. The likely fix belongs in `extensions/memory-core/src/dreaming-phases.ts` and `extensions/memory-core/src/short-term-promotion.ts`: prioritize meaningful recall strength in light-phase staging, split true recall count from daily/grounded support in deep promotion gates or rename/document the contract if combined signals are intended, recalibrate scoring so valid recalled entries can pass default thresholds, widen or otherwise improve rehydration range matching, and add focused regression tests for daily-heavy stores plus shifted broad snippets.

## Risks / Open Questions

- Changing the `minRecallCount` gate from combined signals to true recalls would invalidate current tests and may alter intentionally supported daily/grounded promotion behavior, so maintainers should decide the product contract explicitly.
- Widening `relocateCandidateRange` should preserve bounded runtime for large daily files; a regression test with shifted multi-line snippets would make the behavior change safer.
