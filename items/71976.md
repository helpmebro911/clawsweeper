## Summary

Keep open. Current main still matches the reported memory-core behavior: light dreaming ingests daily/session signals before selection, sorts light candidates by `lastRecalledAt` before `recallCount`, deep promotion gates on a combined signal count that includes `dailyCount` and `groundedCount`, and rehydration still uses the narrow `preferredSpan + 3` / `8` span. This is a concrete current bug report, not stale cleanup or already implemented work.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-core`. Add regression coverage where fresh daily-only entries compete with older real recalls in light dreaming, decide whether `minRecallCount` means true `recallCount` or an explicitly renamed aggregate signal gate, retune scoring so real multi-query recalls can pass documented thresholds, widen or replace the fragile rehydration span search, and update docs/status wording if aggregate signals remain intentional.
