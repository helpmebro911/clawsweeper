## Summary

Current `main` only partially mitigates this webchat race. It now defers `session.message`-triggered history reloads while a run is active and clears stream state on terminal events, but it still reloads `chat.history` after every non-tool `chat final`, and the current tests explicitly lock that eager final-reload behavior in. That is not strong evidence for `implemented_on_main` or `cannot_reproduce`, so this issue should stay open.
