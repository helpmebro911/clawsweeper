## Summary

Keep this issue open. Current main still lacks a pinned key-value facts editor and still lacks the broader manifest/gateway/SPA seam for persistent Control UI plugin panels raised in the discussion. Existing behavior is raw workspace markdown editing plus chat-local tool cards/sidebar content, not a persistent Control UI panel with guaranteed structured context injection semantics.

## Best Possible Solution

Keep this open unless maintainers split a narrower canonical issue. The strongest path is an additive, manifest-first Control UI panel contract in the plugin manifest/schema, loader, gateway protocol, and SPA mounting layer, followed by a pinned facts editor backed by a documented structured store or `USER.md` section with tests/docs proving those facts are included in agent context on every normal turn.

## Risks / Open Questions

- A persistent Control UI plugin panel seam needs product and security design around refresh behavior, iframe/canvas/tool rendering, permissions, and persistence before third-party plugins can safely mount long-lived UI surfaces.
- This overlaps with broader sidebar/Control UI requests in #46630 and #49828; maintainers may want to split a canonical manifest-first panel API issue and keep this issue focused on pinned facts UX.
- A pinned facts editor still needs explicit storage and injection semantics; raw `USER.md` or `MEMORY.md` editing alone does not satisfy the requested structured add/edit/delete workflow.
