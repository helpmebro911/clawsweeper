## Summary

Keep open. Current main still lacks the PR's reconnect-grace behavior: post-startup `connected === false` is immediately unhealthy, the monitor does not pass `lastDisconnect` timing into health evaluation, and current tests still assert restart of a long-running disconnected channel. Related stale-socket/transport-liveness fixes reduce adjacent false restarts, but they do not supersede this PR's mid-lifecycle disconnect race.

## Best Possible Solution

Keep the PR open and either land a rebased version of the reconnect-grace behavior or replace it with an equivalent mainline change: add a typed reconnect-grace policy, consume `ChannelAccountSnapshot.lastDisconnect.at` (with a fallback if desired), cover within-grace and past-grace paths in health policy/monitor tests, and decide whether readiness should share the same grace semantics.
