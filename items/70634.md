## Summary

Keep #70634 open. Current main still has a single FIFO queue per command lane, and the enqueue API has no origin/priority field. `sessions_send` already marks agent-originated work as `inter_session`, but `runEmbeddedPiAgent` still serializes all turns for the same target session through the same outer `session:<key>` lane before the nested/global lane, so human/external turns do not jump ahead of queued agent-to-agent work. The issue is recent, non-maintainer-authored, unprotected, and corroborated by the existing GitHub comment describing the same production starvation pattern.

## Best Possible Solution

Keep the issue open and implement a core command-lane priority contract: carry normalized input provenance or an explicit priority from gateway/channel ingress into `enqueueCommandInLane`, route `external_user` work ahead of `inter_session` work within the same session lane, preserve FIFO within each tier, and add regression tests that enqueue inter-session work before an external user turn for the same session. Consider bounded fairness/back-pressure for sustained A2A loops as follow-up scope.

## Risks / Open Questions

- A naive priority queue could starve background agent-to-agent work unless it includes fairness or back-pressure rules.
- Classifying human messages inside the queue by payload inspection would be brittle; the existing `inputProvenance` boundary is safer.
