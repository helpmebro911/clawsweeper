## Summary

Keep open. Current main has related safety valves for sub-agent spawning, including max depth, active-child caps, run timeouts, generic tool loop detection, and provider/auth cooldowns, but it does not implement the requested native per-subagent consecutive-failure circuit breaker with cooldown, override, notification, config schema, CLI, or UI visibility. The related same-feature #65817 is already closed and does not provide an active canonical tracker; the other related items cover provider fallback, heartbeat, task cancellation, or event-drain behavior rather than this sub-agent breaker.

## Best Possible Solution

Keep this issue open and implement the breaker in the core subagent spawn/registry path. The likely path is to persist breaker state keyed by requester/target agent plus a normalized task signature, update it from subagent completion outcomes, check it before the gateway `agent` call, reset it on success, add strict config/schema/docs plus CLI and Control UI visibility, and cover it with focused spawn/registry tests. Provider-level cooldown/cost-cap work should remain separate follow-up scope unless maintainers explicitly combine them.

## Risks / Open Questions

- A native breaker needs careful keying; agentId-only may block unrelated tasks, while task-signature-only may miss repeated bad-agent/config failures.
- Provider billing/rate-limit cooldowns and sub-agent failure circuit breaking should remain separate but coordinated, so one does not hide the other in operator status surfaces.
- Too-aggressive defaults could block legitimate recovery retries; implementation should include clear cooldown, reset-on-success, manual override, and visible trip reasons.
