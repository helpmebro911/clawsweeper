## Summary

Current main still lacks the requested native subagent cascade circuit breaker. The subagent spawn path enforces depth, per-requester active-child limits, explicit agent IDs, allowlists, sandbox compatibility, timeouts, and per-run registration, but there is no aggregate consecutive-failure state keyed by subagent or task, no cooldown/tripped-state precheck, no force override/manual reset, and no Control UI or CLI exposure for breaker state. The issue discussion also adds concrete user impact and separates this from adjacent provider-level cooldown/cost-cap work, so closing would lose a still-missing core reliability feature.

## Best Possible Solution

Keep this issue open. Implement the native breaker in the core subagent spawn and registry path: persist breaker state keyed by requester plus target agent and/or normalized task signature, update it on subagent completion outcomes, reset on success, check it before gateway spawn, support explicit force override and manual reset, add strict config/schema/docs, and expose breaker state through subagent/task status, agents/tasks CLI, and Control UI. Treat provider-level billing or rate-limit breakers as coordinated follow-up scope unless maintainers intentionally combine them.

## Risks / Open Questions

- Provider/profile cooldowns, tool-loop blocking, announce delivery retry backoff, active-child limits, and run timeouts are adjacent safeguards but do not prevent repeated future subagent spawns after repeated terminal failures.
- The related closed #65817 describes the same missing capability but is not an open canonical tracker; this issue currently carries the stronger discussion and should remain the active tracking item unless maintainers choose a different canonical issue.
- Breaker keying and defaults still need product design: target-agent-only may over-block unrelated work, while task-signature-only may miss broken agent configuration. Provider billing/rate-limit breakers should likely remain related but separate scope.
