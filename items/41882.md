## Summary

Current main has moved Discord handling into the bundled Discord plugin, but the same product bug is still present: the Discord inbound worker unconditionally serializes same-session jobs before `processDiscordMessage` can reach the inner `steer` / `steer-backlog` queue logic. This PR is still useful work, but it needs to be rebased/adapted to the current `extensions/discord` worker and should address the session-override concern from review before landing.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase/adaptation to current `extensions/discord` code. The best fix is a queue-mode-aware Discord inbound worker path that bypasses the outer per-session `KeyedAsyncQueue` for `steer` and `steer-backlog`, preserves serialization for non-steer modes, honors session-level `/queue` overrides where possible, and adds focused regression tests for both bypass and non-bypass behavior without violating the plugin SDK boundary.

## Risks / Open Questions

- Landing the PR mechanically against current main is unlikely to be correct because the touched path moved from `src/discord` to `extensions/discord` and the queue is now in `extensions/discord/src/monitor/inbound-worker.ts`.
- A config-only bypass can violate per-session queue overrides: if global Discord queue mode is `steer` but a session was switched to `collect` or `followup`, bypassing the outer worker would allow concurrent processing for a session that requested serialization.
- The rebased fix should avoid extension-to-core internal imports; if a queue-mode resolver is needed by Discord, promote or reuse an SDK-safe helper rather than importing `src/auto-reply` internals from the plugin.
