## Summary

Keep #71487 open. Current main has TTS backend, CLI commands, gateway RPCs, docs, and a per-message browser-native Read aloud button, but it still lacks a first-class Control UI Settings panel that reflects and persists TTS enabled state, default provider, or default voice selection. The existing maintainer/Codex comment already reached the same conclusion, and current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f does not supersede it.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech panel in Control UI Settings. It should call existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert where possible, show configured versus unavailable providers, filter voices by selected provider, and add or reuse a safe persistence path for default voice selection. Cover the UI/RPC behavior with focused tests and update the public Control UI/TTS docs when the surface ships.

## Risks / Open Questions

- The generic config editor can expose some messages.tts fields, so a dedicated TTS panel should avoid conflicting sources of truth between openclaw.json and local ~/.openclaw/settings/tts.json preferences.
- Persistent default voice selection likely needs either a narrow new prefs/API surface or a deliberate provider-specific config write path; current RPCs support voice overrides for conversion but not a default voice preference.
