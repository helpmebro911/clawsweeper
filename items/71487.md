## Summary

Keep #71487 open. Current main still has TTS backend, CLI/docs, gateway RPCs, and a per-message browser-native Read aloud button, but no first-class Control UI Settings panel that reflects or persists TTS enabled state, provider selection, or default voice selection. The prior Codex review comment on the issue already reached the same keep-open conclusion, and the latest checked-out main change is only a Thai locale refresh, not a TTS settings implementation.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech panel in Control UI Settings. It should call the existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert gateway methods where possible; show configured and unavailable providers clearly; filter voices by selected provider; add a deliberate persistence path for default voice; and cover the UI/RPC behavior with focused tests.
