## Summary

Current main still lacks the requested first-class Control UI Settings surface. TTS exists in backend RPCs, CLI/docs, provider voice metadata/listVoices hooks, and a per-message browser-native Read aloud button, but the Settings UI has no TTS enable/disable control, provider selector, provider-filtered default voice picker, or provider-backed sample playback.

## Best Possible Solution

Keep this issue open. Implement a first-class Text-to-Speech panel in Control UI Settings that reads tts.status/providers, calls tts.enable/disable and tts.setProvider, surfaces configured versus unavailable providers, exposes provider-filtered voice choices via existing provider voice metadata/listVoices or an added safe RPC if needed, persists a default voice through the agreed TTS preference/config contract, and uses tts.convert for a short provider-backed sample playback. Add focused UI and gateway tests for the settings flow.
