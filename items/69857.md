## Summary

Keep #69857 open. Current main still mints Ollama synthetic auth for meaningful explicit Ollama config without classifying the base URL, and current tests assert that configured remote Ollama providers without an API key receive `ollama-local`. The PR is not superseded by #59954, which the supplied timeline shows was closed and superseded by this PR, and #43945 remains the broader subagent/fallback issue rather than a replacement for this focused change.

## Best Possible Solution

Keep the PR open or rework it on current main: add an Ollama-plugin-owned locality gate before minting `ollama-local`, update the current tests that expect remote synthetic auth, cover Cloud/public remote, LAN, mDNS, bare hostnames, and loopback aliases, and avoid deep-importing core `src/**` helpers unless the needed IP classifier is promoted through a documented plugin SDK surface.
