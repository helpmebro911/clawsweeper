## Summary

Current main still allows the reported Telegram `/new` leak path. Bare `/new`/`/reset` prepends runtime startup memory into the first reset prompt; startup context still loads raw dated memory files; the bundled session-memory hook still captures assistant transcript text from reset sessions into those files; and Telegram delivery uses the shared payload path without a guard for `[note self-improvement (before reset): ...]`-style output. The related WhatsApp report #58704 is a different self-chat echo bug, not a canonical fix for this Telegram/startup-memory path.

## Best Possible Solution

Keep this issue open. Add a focused regression test for Telegram/native bare `/new` or `/reset` where startup memory or session-memory contains `[note self-improvement (before reset): ...]`, then prevent internal hook/self-improvement artifacts from entering the startup prelude or suppress that artifact before user-visible reset delivery. Likely touch points are `src/auto-reply/reply/startup-context.ts`, `src/hooks/bundled/session-memory/transcript.ts`, and possibly shared reply sanitization if maintainers choose output-time suppression.
