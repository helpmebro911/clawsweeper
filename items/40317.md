## Summary

Keep this issue open. Current main still has no strict/tolerant config validation mode surface, and the read/startup path still treats unknown schema keys as invalid config, with last-known-good recovery rather than tolerant stripping. The related write-path crash class is mitigated by strict validation before config.set/config.apply/config.patch writes, but the explicit configurable read/write policy requested here remains unimplemented.

## Best Possible Solution

Keep this issue open for maintainer product judgment. A complete solution should add a documented validation policy to the config schema/types/help/docs, keep strict validation for config.set/config.apply/config.patch writes, define tolerant read/startup/hot-reload semantics for root and nested unknown keys, log stripped unknown keys at the configured level, and add regression tests for root/nested unknown keys, plugin-local invalidity, write rejection, and last-known-good recovery interactions. If maintainers decide not to expose a validation-mode surface, consolidate that decision into #40268 before closing this as duplicate or superseded.
