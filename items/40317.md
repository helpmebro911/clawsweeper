## Summary

Keep this issue open. Current main still has strict config schema validation and last-known-good recovery, not an operator-configurable strict/tolerant validation mode. The write-path crash class is mitigated by validating config.set/config.apply/config.patch before writing, but startup/read behavior still rejects invalid snapshots rather than stripping unknown keys and continuing.

## Best Possible Solution

Keep this open for maintainer product judgment. Either add and document a config validation policy in the schema/types/help/docs with strict write validation and well-defined tolerant read/startup/hot-reload behavior, or explicitly decide that tolerant startup belongs only in #40268 and close this later as superseded after that narrower path is resolved.

## Risks / Open Questions

- Closing now would drop an unresolved product/API decision: whether OpenClaw should expose an operator-configurable validation policy or keep strict validation plus recovery only.
- A tolerant read mode still needs precise semantics for root and nested unknown keys, known-key type/value errors, plugin-local schema failures, logging level, hot reload, startup, and last-known-good recovery interactions.
- Any future implementation must preserve the current strict write rejection for config.set, config.apply, and config.patch so invalid keys are not persisted through OpenClaw-owned writes.
