## Summary

Keep open. Current main has the intended Feishu inbound debounce path and prior changelog evidence for same-chat batching, but the reporter reproduced the Feishu DM burst failure on 2026.4.14 after enabling messages.inbound.debounceMs, and I did not find direct current-main regression coverage for the exact p2p/DM three-message idle case. The separate active-run collect serialization problem is still tracked by open #54409, so closing this as implemented would be premature.

## Best Possible Solution

Keep this issue open. Add a focused regression test around createFeishuMessageReceiveHandler using three same-sender Feishu p2p text events inside messages.inbound.debounceMs while the agent is idle, asserting one delayed merged dispatch and suppressed earlier message ids. If that passes, add diagnostics/docs clarifying messages.queue versus messages.inbound and ask the reporter for runtime logs; if it fails, fix the Feishu debounce key/text/effective-window path and coordinate the active-run queue.collect behavior with #54409.
