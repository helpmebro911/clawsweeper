## Summary

Keep this issue open. Current `main` still routes the OpenAI Codex OAuth browser login through a TLS preflight that hard-fails on `UNABLE_TO_GET_ISSUER_CERT_LOCALLY` before the upstream OAuth flow starts. There are adjacent mitigations, including device-code auth, proxy dispatching, and startup CA environment handling, but they do not prove the reported macOS custom-CA precheck/runtime regression is fixed.

## Best Possible Solution

Keep this issue open and align implementation with the intended behavior: either make the OpenAI Codex OAuth TLS preflight advisory for browser login, or add targeted diagnostics/repair for effective `NODE_EXTRA_CA_CERTS`, `NODE_USE_SYSTEM_CA`, cert-bundle readability, CLI/gateway env inheritance, and Codex runtime fetch dispatcher behavior. Add regression coverage that direct custom-CA startup does not get blocked by the speculative preflight, plus a runtime transport test or documented macOS verification for `openai-codex/gpt-5.4`/`gpt-5.5`.
