## Summary

Keep open. Current main still creates another Feishu streaming card for a distinct late final after idle close, so the PR is not obsolete. The PR is focused and touches only the Feishu dispatcher/test, but it still needs maintainer review/revision because the latest approach can intentionally drop distinct final text and relies on async closeout state in a lifecycle where onIdle/onError are fire-and-forget.

## Best Possible Solution

Keep this PR open. The best path is to revise the Feishu dispatcher fix so late-final suppression is scoped to the successfully closed streaming reply generation, or limited to true duplicate/correlated finals, while preserving media delivery. Add regression coverage for distinct late finals, close-error recovery, back-to-back reply lifecycle overlap, and the intended auto-mode behavior before maintainer review/merge.

## Risks / Open Questions

- The PR intentionally suppresses distinct late `final` text after streaming close; if Feishu chat is used as the record of the answer, corrected/tool-derived/disclaimer text could be hidden.
- A shared `streamingClosedForReply` boolean written after awaited closeout can be set by stale async close work because the dispatcher does not await `onIdle`/`onError` callbacks.
- The author notes the auto-mode short-late-final case remains out of scope, so the patch is narrower than a general late-final duplication fix.
