## Summary

Keep PR #45465 open. Current main does not implement cron lifecycle hooks, and the existing plugin/internal hook surfaces do not provide an equivalent before-run/after-run cron execution boundary with abort semantics. The PR is meaningful core/plugin-API work, not an implemented, duplicate, or ClawHub-only item. It still needs maintainer/security review because the PR adds gateway-side user script execution and the later review comments call out unresolved per-job hook path/type validation gaps.

## Best Possible Solution

Keep the PR open for maintainer judgment. The best direction is either to rework it into a documented cron lifecycle plugin hook/API seam with strict schema/path validation and coverage for timer, manual, and startup catch-up runs, or to explicitly reject the product direction in a maintainer comment. Current main does not solve the central request, and existing plugin hooks are not enough to build the requested abortable cron lifecycle externally.

## Risks / Open Questions

- The PR adds user-defined script execution inside the gateway cron lifecycle, including per-job configuration, so path validation, schema validation, trust boundaries, timeout behavior, and module-cache behavior need maintainer/security review before merge.
- A ClawHub/plugin implementation cannot fully satisfy the requested `beforeRun` abort and scheduler-level `afterRun` semantics with current plugin hooks; closing as ClawHub would likely discard a real missing extension seam.
