## Summary

Keep open. Current `main` does not implement bare `group:<numeric_chat_id>` normalization in the Telegram target parser or chat-id normalization helpers, so the requested behavior is still missing. The submitted PR also appears incomplete relative to its stated goal because `parseTelegramTarget` remains the entry point for sends and approval-target chat-type resolution.

## Evidence

- **Bare `group:` targets are still not normalized on current main:** `extensions/telegram/src/targets.ts:32` and `extensions/telegram/src/targets.ts:47` still feed `normalizeTelegramChatId` and `normalizeTelegramLookupTarget` through `stripTelegramInternalPrefixes(...)` only. That helper intentionally leaves bare `group:<id>` unchanged unless a `telegram:`/`tg:` prefix was stripped first, so `group:-100123` and `group:123456789` are still rejected as numeric chat IDs on current `main`. The matching test at `extensions/telegram/src/targets.test.ts:30` still asserts that bare `group:` is preserved.
  - file: [extensions/telegram/src/targets.ts:32](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/extensions/telegram/src/targets.ts#L32)
  - command: `nl -ba extensions/telegram/src/targets.ts | sed -n '1,140p'; nl -ba extensions/telegram/src/targets.test.ts | sed -n '21,33p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **The parser still misinterprets `group:<digits>` as `chatId:threadId`:** `extensions/telegram/src/targets.ts:91` still parses targets by first calling `stripTelegramInternalPrefixes(to)` and then matching `^(.+):(\d+)$`. For a bare delivery target like `group:123456789`, that regex yields `chatId = "group"` and `messageThreadId = 123456789` instead of a numeric chat ID. No current parser branch removes a bare `group:` prefix before this split.
