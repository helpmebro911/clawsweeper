## Summary

Keep #70472 open. Current main still has the reported failure shape: missing compaction checkpoint session entries only warn/return null, and compaction/overflow failure handling can still rotate the active session mapping to a newly generated session id. The provided triage comment and open PR #70479 identify the same remaining bug, but that fix is not present on current main.

## Best Possible Solution

Keep the issue open and land a focused fix that preserves the active WebChat session mapping on overflow/compaction failure. The implementation should stop using resetReplyRunSession for compaction failures, return explicit retry or /compact guidance without changing sessionId, make missing agent:main:main checkpoint entries an auditable invariant/recovery path, and add regression coverage proving the next WebChat message continues on the same active session id after compaction failure.
