## Summary

Keep this PR open. Current main still lacks a default dangerous-tool deny policy on the plugin-tools MCP stdio path, while the HTTP tool invoke path has a default deny boundary. The PR remains a plausible security hardening candidate, but it needs a current-main rebase and should stay focused on MCP policy rather than dependency release-age cleanup.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase onto current main, implement a universal dangerous-tool deny policy in the active MCP plugin-tools handler/server path, preserve existing ownerOnly filtering and before_tool_call enforcement, wire gateway.tools.allow and gateway.tools.deny with clear precedence for both listing and invocation, add targeted tests, and split or drop unrelated dependency release-age changes.

## Risks / Open Questions

- The PR patch shown in the provided context targets an older wrapper shape in src/mcp/plugin-tools-serve.ts; current main centralizes active MCP list/call behavior in createPluginToolsMcpHandlers, so a rebase should implement and test the policy in the shared handler/server flow without bypassing ownerOnly filtering or before_tool_call hooks.
- Changing default MCP exposure can break operators or plugins that intentionally expose high-risk tool names through the opt-in bridge; the fix needs explicit gateway.tools.allow and gateway.tools.deny precedence tests for both listTools and callTool.
- The provided PR diff mixes MCP security hardening with an unrelated supply-chain-sensitive pnpm minimumReleaseAge exclusion for axios. Current main already contains both axios and axios@1.15.0 exclusions in pnpm-workspace.yaml, so dependency policy should be reviewed separately from this MCP change.
