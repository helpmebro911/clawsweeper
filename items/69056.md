## Summary

Keep #69056 open. Current main still has the reported Windows-sensitive `SIGUSR1` fallback in `emitGatewayRestart()`, the scheduler still reports no-listener restarts as `signal`, and the PR discussion contains real unresolved review concerns around Windows fallback token handling. This is core gateway restart behavior, not cleanup-safe to close.

## Best Possible Solution

Keep the item open and resolve it as a narrow core gateway fix: preserve the listener-driven path when a `SIGUSR1` handler exists, add a Windows no-listener fallback that uses the scheduled-task handoff without leaving restart tokens stuck, return/roll back correctly on handoff failure, update `mode` reporting, and add focused tests for Windows listener/no-listener success and failure paths. If #68853 is the preferred implementation, mark #69056 superseded only after maintainers choose that PR as canonical.
