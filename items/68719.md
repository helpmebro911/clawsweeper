## Summary

Keep open. Current main has generic `tools.media.audio` transcription for inbound audio, but the BlueBubbles plugin does not implement the requested BlueBubbles-native `/api/v1/message/audio-transcript/:guid` enricher, config flag, or `message_transcribed` hook. The bundled BlueBubbles path still builds the inbound body from webhook text or a media placeholder, downloads attachments into `MediaPaths`, and leaves any transcription to the shared media-understanding pipeline rather than Apple/BlueBubbles' no-ASR transcript endpoint.

## Best Possible Solution

Keep #68719 open and implement it in `extensions/bluebubbles`: add an audio attachment detector using MIME/UTI metadata, a BlueBubbles client method for `GET /api/v1/message/audio-transcript/:guid` with existing auth/SSRF policy reuse, version/capability gating, a documented `channels.bluebubbles.inboundAudioEnricher` config surface, tests proving transcript replacement before agent dispatch and mention/command handling, and a privacy-safe scoped hook or an explicit decision to reuse an existing message hook instead of adding `message_transcribed`.
