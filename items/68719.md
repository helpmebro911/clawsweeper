## Summary

Keep open. Current main has a generic inbound audio/media-understanding path that can transcribe saved BlueBubbles audio attachments when audio understanding is configured or auto-detected, but it does not implement the requested BlueBubbles-owned pre-dispatch enricher that calls `/api/v1/message/audio-transcript/:guid`, defaults/flags `channels.bluebubbles.inboundAudioEnricher`, detects audio UTIs, or emits a scoped `message_transcribed` hook. The BlueBubbles plugin still turns empty attachment-only messages into `<media:audio>` placeholders and passes downloaded media paths into the shared reply pipeline; the transcript rewrite, when it happens, is generic provider/CLI media understanding later in `getReplyFromConfig`, not the Apple on-device BlueBubbles transcript endpoint requested here. The provided discussion also cites a concrete openclaw-infra overlay/workaround and tracking item, so this is a real remaining channel feature request rather than stale noise.

## Best Possible Solution

Keep this open for a BlueBubbles plugin-owned implementation or an explicit maintainer product decision. The likely fix belongs under `extensions/bluebubbles`: add a typed client method for the BlueBubbles audio-transcript endpoint, detect audio attachments by MIME and UTI before building the final inbound body, honor a documented `channels.bluebubbles.inboundAudioEnricher` config/default, avoid transcript logging, and add focused BlueBubbles tests for empty/placeholder voice notes, opt-out behavior, and scoped hook payload. If maintainers decide the shared `tools.media.audio` path is the supported answer instead, document that explicitly in the BlueBubbles channel docs rather than auto-closing this request.

## Risks / Open Questions

- Closing as implemented would conflate the generic provider/CLI audio transcription path with the requested no-ASR-cost BlueBubbles server transcript endpoint and opt-out defaults.
- The implementation must preserve the existing BlueBubbles client auth and SSRF policy when adding the new endpoint, because the client currently centralizes guarded API requests.
- Transcripts are PII; any fix should keep transcript content out of verbose logs and hook/debug output, and should avoid passing full OpenClaw config through a new scoped hook.
