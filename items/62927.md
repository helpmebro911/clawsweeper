## Summary

Keep this PR open. Current main still has a Slack attachment-loss path around paired `message` and `app_mention` events, and the media resolver still swallows per-file download failures. The submitted patch is relevant but should be revised before merge: the current Slack handler no longer debounces media-bearing messages, so a debounce-entry union alone may miss the in-flight app_mention-wins race; the patch also drops id-less `SlackFile` objects and logs raw error messages instead of using the repo's redacted formatter.

## Best Possible Solution

Keep this PR open, but request a narrow revision in the Slack plugin: preserve attachment metadata across the same-ts message/app_mention race even when the file-bearing message is not debounced, keep id-less Slack files, add regression coverage for attachment preservation and media failure logging, and route diagnostics through the repo's redacted error formatter. Do not move this into core; the behavior is Slack-plugin-owned.

## Risks / Open Questions

- The submitted debounce union drops `SlackFile` objects without `id`; preserve id-less files and dedupe only when an ID exists.
- Because media-bearing messages are not debounced on current main, a debounce-entry union alone may not fix the app_mention-wins in-flight race; the revision should cover same-ts message/app_mention suppression explicitly.
- The proposed raw `err.message`/`err.cause.message` verbose logging should be replaced with `formatErrorMessage` or an equivalent redacted formatter.
- The PR needs focused regression tests for preserving Slack attachments across the app_mention race and for verbose media-resolution diagnostics.
