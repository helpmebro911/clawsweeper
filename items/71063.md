## Summary

Current main does not contain the PR's stdin writability guard or an equivalent regression test. The reported path is still plausible because CLI runner passes an `input` value into `createChildAdapter`, and the adapter writes/ends stdin before any protected `ManagedRunStdin` wrapper is used. This is a focused, recent bug-fix PR and should stay open for maintainer review rather than be closed by cleanup.

## Best Possible Solution

Keep #71063 open. The best code path is a small fix in `src/process/supervisor/adapters/child.ts` that makes the initial stdin write/end safe for failed or already-closed child stdin streams, plus a regression test in `src/process/supervisor/adapters/child.test.ts` that fails against current main and verifies caller-facing `adapter.stdin` behavior.

## Risks / Open Questions

- The PR's current regression test should be checked carefully: overriding `PassThrough.writable` may not fully reproduce a real closed or EPIPE-ing stdin stream unless the test fails on current main or uses a stream/stub whose write/end actually fails.
- A `writable` pre-check may not cover races where stdin is writable at check time and closes before `write` or `end`; maintainer review should decide whether a try/catch/error-listener around the initial write/end path is needed.
