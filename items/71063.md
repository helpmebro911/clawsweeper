## Summary

Keep this PR open. Current main still has the unguarded initial stdin write path in the child process adapter, and the CLI backend route described by the PR is real. The proposed patch is focused, but a read-only Node probe shows a missing binary behind the Linux shell wrapper can still have `stdin.writable === true` immediately before emitting async `EPIPE`, so maintainer review should verify whether the one-line writable guard is sufficient or broaden the fix.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is a scoped supervisor fix in `src/process/supervisor/adapters/child.ts` that prevents initial stdin write/end from crashing failed CLI spawns, with regression coverage in `src/process/supervisor/adapters/child.test.ts` that models a closed or EPIPEing stdin stream, not only a pre-set `writable: false` property.
