## Summary

Keep open. Current main still has an unprotected initial stdin write in the supervisor child adapter, so the PR targets a real remaining crash path. However, the current PR patch only checks `stdin.writable`, and a live read-only probe of the Linux shell-wrapped missing-binary path showed `stdin.writable === true` before an async `EPIPE`, so the proposed fix likely needs maintainer amendment rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or contributor amendment. The best fix belongs in `src/process/supervisor/adapters/child.ts`: protect the initial `params.input` write/end path from both already-non-writable stdin and asynchronous stdin `error`/`EPIPE` after early child exit, then add focused regression coverage in `src/process/supervisor/adapters/child.test.ts` for both shapes while preserving normal writable stdin behavior. The current PR can be amended, or superseded by an equivalent targeted supervisor fix.

## Risks / Open Questions

- The PR's current one-line `stdin.writable` guard may leave the reported gateway crash path open because the shell-wrapped missing-binary case can be writable initially and then emit async `EPIPE`.
