## Summary

Current main still has the web_fetch network-error problem, so this PR is not obsolete. It should stay open for maintainer rework or replacement, but it should not land as-is because the branch bundles unrelated behavior changes and has concrete install/supply-chain and Feishu route-construction risks.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement. The best path is a narrow web_fetch patch that wraps direct network-level failures only after provider fallback misses, extracts `error.cause.message` or code, preserves the original error as `cause`, includes a safe/redacted requested URL, and adds a regression test for the no-fallback network-error case. The unrelated Feishu, Telegram, config schema, model/table, tool-result, and package override changes should be split into separate PRs or dropped, with the Feishu separator and lockfile blockers resolved before any landing decision.

## Risks / Open Questions

- Landing this PR as-is would mix a focused web_fetch error-message fix with unrelated Feishu, Telegram, config, model-scan, list rendering, tool-result, and package-manager changes.
- The package override deletion is install and supply-chain sensitive because it changes package resolution metadata without the matching lockfile update while CI uses frozen installs.
- The Feishu validation refactor would allow `/` or `\\` inside keys later used as Feishu SDK path params, creating a route-construction regression from current main's stricter opaque-key guard.
- The headline web-fetch error message uses the raw requested URL from the PR diff; a final implementation should consider redacting sensitive query/path data in chat-facing errors while still preserving useful debug context.
