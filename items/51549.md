## Summary

Keep open. Current `main` appears to have added chat history reload from the gateway, but the reported refresh-loss bug is only partially addressed: queued follow-up messages and the unsent draft are still stored only in UI memory and are reinitialized empty on page load, so a browser refresh still drops that state.

## Evidence

- **Queued messages are still only in memory:** `enqueueChatMessage()` and `enqueuePendingRunMessage()` append directly to `host.chatQueue`; there is no storage or reload path here.
  - file: [ui/src/ui/app-chat.ts:112](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/ui/src/ui/app-chat.ts#L112)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '110,170p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Draft is cleared into in-memory state before queueing/sending:** `handleSendChat()` reads `host.chatMessage`, clears it, and queues/sends from memory only. There is no persistence hook for refresh recovery.
  - file: [ui/src/ui/app-chat.ts:320](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/ui/src/ui/app-chat.ts#L320)
