## Summary

Keep this issue open. Current main still lacks compaction-owned same-provider auth-profile failover: the normal run path has profile candidate rotation, but manual/default compaction still resolves one effective auth profile and runs one compaction attempt. Recent compaction commits on main address transcript rotation/checkpoint handling, not auth-profile retry within compaction.

## Best Possible Solution

Keep this issue open. The best fix is to factor the existing auth-profile order/cooldown/failure classification into a reusable compaction-safe retry helper, then make compaction snapshot the transcript once, try eligible same-provider profiles against that snapshot, mark transient failures/cooldowns, and commit only one successful summary/checkpoint/transcript rotation. Add tests for manual compaction and overflow/timeout-triggered compaction where profile A rate-limits and profile B succeeds, plus a partial-failure/idempotency regression test.

## Risks / Open Questions

- A naive outer-run retry after a compaction failure can replay an already-completed prompt/tool turn or double-compact; the fix needs a compaction-owned snapshot and single-commit boundary.
- Manual `/compact`, overflow recovery, timeout recovery, and context-engine delegated compaction should be covered separately because they enter the compaction path with different runtime context and session ownership.
