## Summary

Keep open. Current `main` still contains a plausible Feishu-specific regression path: Feishu media keys are rejected when they contain `..`, so inbound image/file keys can be discarded before download, leaving no media attached for analysis.

## Evidence

- **Feishu external-key validator still rejects `..`:** `normalizeFeishuExternalKey` returns `undefined` when a Feishu key contains `/`, `\\`, or `..`. That means a legitimate Feishu `image_key`/`file_key` containing `..` is dropped immediately.
  - file: [extensions/feishu/src/external-keys.ts:15](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/extensions/feishu/src/external-keys.ts#L15)
  - command: `nl -ba extensions/feishu/src/external-keys.ts | sed -n '1,220p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Inbound media parsing depends on that validator:** `parseMediaKeys` normalizes `parsed.image_key` and `parsed.file_key` through `normalizeFeishuExternalKey`; if both normalize to nothing, `resolveFeishuMediaList` returns an empty list and no inbound media is downloaded.
  - file: [extensions/feishu/src/bot-content.ts:287](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/extensions/feishu/src/bot-content.ts#L287)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '120,330p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **No media list means no attachment reaches the agent context:** `handleFeishuMessage` builds `mediaPayload` from `resolveFeishuMediaList(...)` and spreads it into the inbound context. If the media list is empty, the agent context gets no `MediaPath`/`MediaUrl` fields, which matches an "I don't see any image attached" outcome.
  - file: [extensions/feishu/src/bot.ts:712](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/extensions/feishu/src/bot.ts#L712)
