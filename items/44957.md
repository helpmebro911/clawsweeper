## Summary

Keep open. Current main does not implement the PR's central `outputHistory` / `recentOutputs` behavior, and this is a core cron/gateway scheduler contract change rather than a ClawHub-only plugin idea. The PR remains potentially useful, but it still needs maintainer review and likely fixes for the latest unresolved review comments.

## Best Possible Solution

Keep the PR open for maintainer judgment. If accepted, implement this as a core cron feature: add additive `payload.outputHistory` and bounded `state.recentOutputs` to cron types/protocol/UI/docs; inject bounded prior-output context before delivery instructions and inside the external-content guard; record only substantive successful outputs with silent/control tokens filtered; normalize malformed boolean inputs; and cover scheduled, manual, startup catch-up, no-delivery, and external-hook paths with focused tests.

## Risks / Open Questions

- Landing the PR as-is may record control tokens such as `HEARTBEAT_OK` / `NO_REPLY` or non-substantive interim output as future prompt context unless the latest review concern is addressed.
- Because this is a gateway protocol/state change, cron types, validators, UI types, docs, and tests need to stay aligned.
- A persistent custom session can preserve history today, but it is not equivalent to bounded per-job output history for fresh isolated cron runs.
