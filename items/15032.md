## Summary

Current main still does not implement per-spawn tool restrictions for `sessions_spawn`. The issue remains a concrete core API gap: the schema, spawn params, child session metadata, and subagent policy resolver have no per-call allow/deny path, while existing docs only describe global `tools.subagents.tools` policy.

## Best Possible Solution

Keep #15032 open as the canonical core API gap. The best implementation is an additive `sessions_spawn` parameter such as `tools: { allow?: string[], deny?: string[] }`, threaded through the schema, `SpawnSubagentParams`, child session persistence, and `resolveSubagentToolPolicyForSession`, with deny-wins semantics, focused unit/e2e coverage, and updates to the Sub-agents docs.
