## Summary

Keep #15032 open. Current main still does not implement per-spawn tool restrictions for `sessions_spawn`: the public tool schema has no `tools`/allow/deny field, the execution path does not parse or forward any per-call tool policy, child session metadata has no persistent policy carrier, and subagent tool policy resolution remains global/config-derived. The issue discussion also shows a real Claw Drive use case and a closed-unmerged implementation PR (#23506); the narrower #23906 remains open rather than a shipped replacement.

## Best Possible Solution

Keep #15032 open as the canonical core API/security gap. Implement an additive `sessions_spawn` tool-policy parameter such as `tools: { allow?: string[], deny?: string[] }`, optionally including `tools: "none"` if maintainers want to cover #23906. Thread it through the tool schema, `SpawnSubagentParams`, child-session persistence, and `resolveSubagentToolPolicyForSession`; enforce deny-wins and narrow-only semantics; add focused unit/e2e coverage and update the public Sub-agents docs.

## Risks / Open Questions

- No live `sessions_spawn` run was executed because this review was constrained to read-only inspection; the missing schema, type carrier, session metadata, resolver input, and docs path are sufficient to keep the feature open.
- The eventual implementation needs explicit maintainer security semantics: per-spawn policy should narrow only, deny should win, and it must not re-grant tools removed by profile, provider, group, sandbox, or hard subagent policy.
