## Summary

Current `main` no longer uses the reported ACPX startup path: the ACP runtime is embedded in the bundled `acpx` plugin, `acpx` is staged as a bundled runtime dependency, and the Windows install path resolves a safe npm executable without relying on `.cmd` wrapper execution.

## Evidence

- **Embedded ACPX runtime replaced the old external hop:** The service now constructs `AcpxRuntime` directly from the bundled plugin and only probes that embedded runtime; it no longer shells out through the old ACPX CLI startup path described in the report.
  - file: [extensions/acpx/src/service.ts:43](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/acpx/src/service.ts#L43)
  - command: `nl -ba extensions/acpx/src/service.ts | sed -n '1,166p'`
- **Current bundled ACPX version is pinned in plugin metadata:** The bundled `acpx` plugin now declares `acpx` as a runtime dependency at `0.5.3` and marks runtime deps for staging, so the active version comes from plugin metadata rather than a separate hardcoded `0.1.16` constant.
  - file: [extensions/acpx/package.json:6](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/acpx/package.json#L6)
  - command: `nl -ba extensions/acpx/package.json | sed -n '1,20p'`
- **Windows runtime-deps install avoids `.cmd` wrapper shell execution:** The bundled runtime dependency installer resolves `npm-cli.js` via `node` when available, falls back to `npm.exe` on Windows, and explicitly errors instead of using an unsafe shell-wrapper fallback.
  - file: [src/plugins/bundled-runtime-deps.ts:483](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/plugins/bundled-runtime-deps.ts#L483)

  - file: [src/plugins/bundled-runtime-deps.test.ts:39](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/plugins/bundled-runtime-deps.test.ts#L39)
  - command: `nl -ba src/plugins/bundled-runtime-deps.test.ts | sed -n '39,125p'`

## Close Comment

Closing from a Codex review because current `main` no longer uses the failing ACPX path from this report. The bundled `acpx` plugin now embeds the ACP runtime directly (`extensions/acpx/src/service.ts:43`, changelog entry at `CHANGELOG.md:1209`), ships `acpx` as a bundled runtime dependency pinned in plugin metadata (`extensions/acpx/package.json:6`), and the Windows dependency-install path resolves `npm-cli.js`/`npm.exe` without relying on `.cmd` wrapper shell execution (`src/plugins/bundled-runtime-deps.ts:483`, `src/plugins/bundled-runtime-deps.test.ts:39`). The old `expectedVersion` / `command` config path from the report is not part of the current ACPX schema anymore (`extensions/acpx/src/config-schema.ts:26`), and `strictWindowsCmdWrapper=false` is only kept as a legacy ignored field (`extensions/acpx/src/service.ts:67`). On current `main`, this issue’s specific version-mismatch plus Windows-wrapper failure mode is implemented away.
