## Summary

Keep #20756 open. Current main still selects only an explicit `accountId`, a tool/default account id, or the first agent binding before dispatch; it does not inspect the selected channel's enabled accounts to auto-select the sole enabled account when `default` is disabled. WhatsApp's omitted-account path still falls through to the configured/default account helper, which prefers `default` if present and ignores enabled state. The related fix PR #20801 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #20756 open. Implement shared outbound account resolution in `runMessageAction` after channel selection: preserve explicit params/tool defaults/agent bindings, use a configured default only when enabled and configured, auto-select exactly one enabled account when no usable default exists, and emit clear zero-or-multiple-enabled-account errors. Add focused regression tests for the message action path and the WhatsApp disabled-`default` plus single-enabled-account case, then update the multi-account/channel-routing docs.
