## Summary

Keep #20756 open. Current main still only preserves explicit/defaultAccountId/agent-binding account selection, then lets WhatsApp fall back to its configured default account path; it does not resolve the sole enabled account when `accountId` is omitted and `default` is disabled. The related fix PR #20801 was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep #20756 open. Implement shared outbound account resolution after channel selection in the message action path: preserve explicit/tool/agent-binding account IDs, use a configured/default account only when it is usable, auto-select exactly one enabled account when no usable default exists, and return clear zero/multiple-enabled-account errors. Add focused regression tests for `runMessageAction` and the WhatsApp disabled-default/single-enabled-account case, then update the public multi-account/channel routing docs.
