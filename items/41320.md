## Summary

Keep this PR open. Current main does not implement `tirith` or `tools.exec.commandSecurity`, and the PR is a focused security-hardening proposal for the built-in exec path rather than an obsolete or already-shipped change. It may still be redirected to a ClawHub/native plugin using `before_tool_call`, but that is a maintainer product/security decision, not a conservative auto-close.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is either to merge a narrowed core implementation after validating the external-binary trust model, warning/approval semantics, docs, and changed-gate tests, or ask the contributor to move Tirith scanning to a ClawHub/native plugin using `before_tool_call` if maintainers decide external scanner integration should not live in core.

## Risks / Open Questions

- The PR introduces a security-sensitive external binary execution path (`tirithPath`) inside built-in exec handling. Even with `execFile` and PATH hardening, maintainers should explicitly review trust, installation, spoofing, and fail-open semantics before merge.
- The current plugin hook surface may be sufficient for a ClawHub/native plugin implementation, but core integration would provide tighter exec approval behavior and node/sandbox host handling. That scope decision needs maintainer judgment.
- Because this is an open PR, not an issue, stale age alone is not a supported close reason; there is still concrete code and tests that may be useful after review.
