## Summary

Current main still does not implement #7679. Telegram continues to default to `dmPolicy: "pairing"` in config schema, runtime fallback, tests, and public docs. QuickStart can force an allowlist prompt and write `dmPolicy: "allowlist"`, but that path asks the user to manually enter a numeric Telegram user ID; it does not auto-capture the owner ID during onboarding, nor does it implement the token-verified capture flow raised in the issue discussion.

## Best Possible Solution

Keep #7679 open. The best product path is to implement a Telegram onboarding flow that verifies the intended owner, captures that sender's numeric Telegram ID, writes explicit `dmPolicy: "allowlist"` plus `allowFrom`, and then updates the schema/runtime/docs/tests so new Telegram setups no longer present pairing as the safer default. Existing configured installs should keep a clear migration/backcompat story.

## Risks / Open Questions
