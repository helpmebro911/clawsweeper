## Summary

Current main still does not restore the abortable chat-run identity after a Control UI refresh during streaming. The UI can show streamed text after a later delta, but the Stop/New session state is still keyed to `chatRunId`, and current code does not rehydrate that field from the active run after page load.

## Best Possible Solution

Keep this issue open. The best fix is to add an authoritative active webchat run hydration path for Control UI refresh/reconnect, for example by returning active run id and buffered stream state from a gateway method or initial event, then setting `chatRunId`, `chatStream`, and `chatStreamStartedAt` before rendering controls. Add regression coverage that refreshes during an active stream and verifies New session/Send become Stop/Queue immediately, plus terminal-event cleanup still clears the run.
