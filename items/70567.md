## Summary

Keep #70567 open. Current main has transient embedding retry and atomic full-reindex safeguards, but `memory status` still reports `Dirty` from process-local state plus index metadata existence, not from a source-file-to-index coverage audit. A fresh status process can still show `Indexed: 2/10` while printing `Dirty: no`, which is the core operator-facing bug in this issue. The related #26772 is adjacent checkpoint/resume work, not a canonical fix for this status correctness gap.

## Best Possible Solution

Keep the issue open and fix the builtin memory status/index contract: persist incomplete indexing state or compare configured source files against `files`/`chunks` coverage so missing paths make status dirty or are explicitly listed. Add a regression test that simulates a transient embedding 500 during indexing, then creates a fresh status/index manager and verifies the missing files are surfaced or reprocessed without requiring `--force`.

## Risks / Open Questions

- A full source coverage audit in plain `memory status` may be expensive for large `extraPaths`; a cached completeness marker or `--deep` audit may be the safer product shape.
- Builtin SQLite and QMD memory backends have different status semantics, so the fix should be scoped or expressed through a clear shared status contract.
