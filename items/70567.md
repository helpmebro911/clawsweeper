## Summary

Keep #70567 open. Current main has helpful 5xx embedding retry behavior, but it still does not persist failed indexing state or reconcile `Dirty` against source coverage for normal `openclaw memory status`. A partial index with existing metadata can still report `Dirty: no` while only showing an indexed-count mismatch, which is the core operator-visible problem in the issue. Related #26772 is adjacent crash/checkpoint work, not a canonical duplicate for this transient embedding/status bug.

## Best Possible Solution

Keep the issue open and fix memory-core status/indexing reconciliation. A robust path would persist failed/incomplete indexing state or compute dirty/missing coverage from the current source scan versus indexed file rows, make incremental `memory index` repair missing files reliably, surface missing-file counts clearly in `memory status`, and add regression coverage for embedding failure after some files have already been indexed. Keep #26772 separate for durable checkpoint/resume of full reindex operations.
