## Summary

Keep issue #59409 open. Current main has inbound Feishu media filename plumbing, but the visible code still passes Feishu-provided `file_name` through directly when download metadata does not supply a better filename, and the tests I found cover outbound Unicode upload/header extraction rather than the reported inbound Latin-1 mojibake case. The timeline also links an open PR, #59431, that appears to be the active fix path for this issue.

## Evidence

- **Issue is a concrete current bug report with linked active fix PR:** The issue provides version 2026.4.1, Linux/npm install, Feishu inbound webhook path, expected/actual filenames, and the timeline cross-references open PR #59431 titled "fix(feishu): correct Chinese filename encoding in attachments" plus a closed alternative PR #59467. This is not stale or unactionable.
- **Inbound event filename is used directly from Feishu message content:** `parseMediaKeys` parses message JSON and returns `parsed.file_name` for image/file/audio/sticker/video/media without any visible mojibake or Latin-1 repair step.
