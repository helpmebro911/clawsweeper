## Summary

The issue remains actionable. Current main still has the task registry loader requiring `./task-registry-control.runtime.js` or `.ts`, `openclaw tasks cancel` still routes into that cancellation path, and the latest published `openclaw@2026.4.24` tarball still contains the hashed task-registry chunk and type declarations but not `dist/task-registry-control.runtime.js`. The build/release checks do not currently require that sidecar as a stable packaged artifact.

## Best Possible Solution

Keep the issue open. The likely fix is to either emit `src/tasks/task-registry-control.runtime.ts` as a stable `dist/task-registry-control.runtime.js` entry or change the loader to a packaged target that is actually emitted, then add release-check/prepack coverage and a packaged CLI smoke that exercises `openclaw tasks cancel` or `openclaw tasks flow cancel` through an ACP/subagent-backed task path before shipping the next npm release.
