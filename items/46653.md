## Summary

Keep open. Current main does not implement the proposed persistent `openclaw tunnel up/down/status` CLI, and the existing SSH support is only transient gateway probing plus manual docs/app flows. The PR is still a plausible core remote-gateway/operator UX change, but it needs rebase and the unresolved review issues fixed before merge. The related title-search hits are narrower SSH tunnel docs PRs, not superseding implementations.

## Best Possible Solution

Keep the PR open for maintainer/contributor follow-up. The best path is to rebase it onto current main, decide whether the command should be top-level `openclaw tunnel` or nested under `openclaw gateway`, reuse the existing `src/infra/ssh-tunnel.ts` helper where possible, fix the review-identified PID/state/port-race bugs, add targeted Vitest coverage, and update `docs/cli/index.md`, `docs/docs.json`, and remote-access docs consistently.

## Risks / Open Questions

- The proposed detached PID-file design needs process identity verification before `tunnel down` signals a stored PID, otherwise PID reuse could terminate an unrelated process.
- The startup readiness logic should distinguish the new SSH child from a pre-existing listener on the same port and should surface unexpected bind errors clearly.
- The PR needs rebase/adaptation to the current subcommand descriptor and registration structure, plus focused tests for state directory creation, stale state cleanup, port collision behavior, and stop/status safety.
