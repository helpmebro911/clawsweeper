## Summary

Keep open. Current main still rejects symlink targets outside the configured boundary root, and this PR is paired with the open linked issue #64472 by the same author. The submitted patch is a plausible implementation candidate, but it relaxes shared boundary enforcement in a security-sensitive way and needs a narrower design plus tests before merge.

## Best Possible Solution

Keep this PR and #64472 open, but require a revised implementation before merge. The safest path is a narrow, tested exception for the intended workspace-bootstrap symlink traversal only: use the effective and canonical OpenClaw state root, avoid allowing direct absolute reads of sibling state directories, preserve rejection for credentials/session paths and non-workspace boundaries, and add positive symlink plus negative direct-path/security regression tests.

## Risks / Open Questions

- The submitted patch could allow direct absolute access to other OpenClaw state paths under `~/.openclaw/`, including credentials or sessions, rather than only allowing intended bootstrap-file symlink traversal.
- The fallback is added to shared boundary enforcement used by sandbox, plugin, config, and other file-open paths, so it may weaken root confinement outside the workspace-bootstrap use case.
- The proposed fallback uses `os.homedir()` instead of the effective/canonical OpenClaw state root, so it can be inconsistent with `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, or symlinked state roots.
- The PR has no automated positive or negative regression tests for the relaxed security path.
