## Summary

Keep PR #70944 open. Current main does not implement the requested Discord identity-link principal flow: Discord sender resolution has no `identityLinks` input or trusted-principal output, message/native command contexts do not pass `TrustedSenderPrincipal`, inbound metadata does not render `sender_principal`, and the reusable identity-link resolver remains private to session-key routing. The PR is focused, recent, and its review discussion shows the reported PluralKit and import-placement concerns were addressed in follow-up commits rather than leaving the work obsolete.

## Best Possible Solution

Keep the PR open for normal maintainer review and CI. If accepted, the best implementation is the shape already proposed: expose a stable identity-link resolver through the plugin SDK, resolve Discord trusted principals from configured `session.identityLinks` using the real PluralKit sender ID when available, thread `TrustedSenderPrincipal` through Discord message/native-command/component contexts, normalize it in inbound context, render `sender_principal` in trusted inbound metadata, and keep the focused regression tests covering these paths.
