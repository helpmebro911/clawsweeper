## Summary

Keep #70944 open. Current main does not implement Discord trusted-principal resolution into inbound metadata: identityLinks currently collapse DM session keys, but Discord sender identity has no trustedPrincipal field, Discord message/native-command contexts do not pass TrustedSenderPrincipal, inbound meta does not render sender_principal, and the generic resolver is not exported through the plugin SDK. The PR remains useful, and the supplied review context shows the early bot comments were addressed on the PR branch rather than superseding the work on main.

## Best Possible Solution

Keep the PR open. The best path is to review/rebase and land or adapt the PR’s focused implementation: expose a generic identity-link resolver through the plugin SDK if maintainers accept that seam, resolve Discord trusted principals from identityLinks including PluralKit sender handling, thread TrustedSenderPrincipal through preflight/process/native command contexts, normalize it in inbound context, render sender_principal in inbound meta, and keep the regression tests around those paths.
