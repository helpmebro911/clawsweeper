## Summary

Keep open. Current main already de-duplicates exact SecretRef keys and batches exec refs per provider, but it does not implement the requested provider-scoped memoization for shared backend lookups such as 1Password vault-name, item-name, or item-payload fetches during a startup resolution pass. The issue is recent, concrete, unprotected, and not superseded by the provided related items.

## Best Possible Solution

Keep the issue open. A good fix would add focused startup-pass memoization where core has enough information, especially making `resolveSecretRefValues()` consult and populate `resolvedByRefKey` for batch calls, then add regression coverage with a fake exec provider proving repeated same refs across runtime phases do not re-invoke the provider. If maintainers want to reduce 1Password Connect vault/item sublookups for distinct ids, implement or document provider-local memoization in the 1Password exec resolver path, or track a separate SecretRef provider API design if core should own those backend semantics.

## Risks / Open Questions

- No live 1Password Connect reproduction was run in this read-only review, so the exact 84-call observation was not independently replayed.
- The fix needs a careful boundary decision: core can cache exact SecretRef results across the startup pass, while vault/item sublookup memoization may need to live in a 1Password/exec resolver unless OpenClaw adds a richer provider contract.
