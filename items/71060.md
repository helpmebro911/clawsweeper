## Summary

Keep #71060 open. Current main has hardened the normal `gateway restart` path, but the reported slow-drain `gateway stop` failure mode is still present: `stopLaunchAgent()` waits only 10 x 100ms, then falls back to `bootout` and emits warnings that the service was left unloaded. Existing tests explicitly assert that fallback behavior when the service remains running or stop confirmation fails, so this is not already fixed by the earlier restart-only LaunchAgent work.

## Best Possible Solution

Keep the issue open and fix the macOS `gateway stop` slow-drain path directly. The likely product/code direction is to stop treating a >1s graceful shutdown as grounds for destructive `bootout`: increase the stop confirmation budget to match the gateway's real graceful drain, and if launchd still cannot confirm the stop, either fail with actionable diagnostics or re-register the LaunchAgent in a deliberate disabled/stopped state so `start`/`restart` does not depend on manual bootstrap. Add focused tests for a slow-but-eventually-stopped LaunchAgent and for the chosen recovery behavior after an unavoidable bootout fallback.
