## Summary

Keep open. Current main has adjacent fixes and clarified docs, but it does not prove the umbrella problem is gone. The explicit disabled-plugin path avoids normal registration, and the fallback loop can recover from a stale requested provider by adding the configured primary later. However, current message/session model selection can still honor stored overrides, legacy `codex/*` refs remain accepted, and the related merged harness work explicitly did not claim to close #68615's broader stale route/auth/model cleanup.

## Best Possible Solution

Keep #68615 open as the umbrella for stale Codex route cleanup. The best maintainer path is a focused follow-up that defines the intended disabled-Codex contract, then adds regression coverage for message-channel runs with `plugins.entries.codex.enabled = false`, stale session/model/auth overrides, and a configured proxy primary such as `codex-proxy/gpt-5.4`; either normalize before runtime execution or document and test that fallback-only recovery is the intended behavior.
