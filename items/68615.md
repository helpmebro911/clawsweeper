## Summary

Keep open. Current main has related improvements and clearer docs, but it does not fully close the reported stale Codex route/state problem: the loader still executes plugin register during non-activating discovery loads before restoration, fallback-selected models can still persist as auto session overrides, and there is no targeted coverage for the reported proxy-backed codex-proxy route.

## Best Possible Solution

Keep this open as the umbrella unless maintainers split it. The best path is to add a focused regression for a configured replacement/proxy provider not trying stale codex or openai-codex first, decide cleanup semantics for auto fallback overrides and stale auth-profile ordering, and either rework non-activating loader discovery or document/test which register-time effects are safely captured and restored.
