## Summary

Keep #70819 open. Current main still rejects any node_modules symlink whose real target resolves outside the staged install root, and there is no trusted OpenClaw peer-link exception in the scanner. The installer separately creates a host `node_modules/openclaw` peer symlink after scanning, so the PR’s reported package-manager workspace conflict remains plausible and not implemented on main. The provided review thread shows the earlier broad allowlist, repeated root resolution, equality branch, and non-exact path matching concerns were addressed in later PR commits, but those commits are not present in the current checkout. The related #69971/#45744 context covers plugin discovery/profile install symlinks, not this install-security scanner exception.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a narrow scanner change that allows only exact trusted `node_modules/openclaw` and `node_modules/.bin/openclaw` peer-link shapes when their real targets resolve inside the trusted host OpenClaw package root, with the negative regression tests described in the PR. Review whether the CI stabilization edits should remain in the same PR or be split into a separate maintainer-owned change.
