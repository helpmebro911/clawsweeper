## Summary

Keep #70819 open. Current main still fails any `node_modules` symlink whose real target escapes the install root, so it does not implement the PR's narrower trusted `node_modules/openclaw` / `node_modules/.bin/openclaw` peer-link exception. The existing mainline peer-dependency helper only creates the host `openclaw` symlink after source and installed dependency scans, so it does not solve package-manager workspace source trees that already contain those symlinks. Related symlink PRs in the provided context cover plugin discovery/profile install behavior, not this install security scanner case.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land or revise a narrowly scoped scanner change that permits only exact `node_modules/openclaw` and `node_modules/.bin/openclaw` peer symlinks whose real targets are inside the trusted host OpenClaw package root, while retaining fail-closed behavior and negative tests for spoofed, nested, arbitrary, or non-exact links. Consider splitting unrelated CI timeout/worker changes if maintainers want the security fix isolated.
