## Summary

Keep this issue open. Current main does not expose or forward Telegram Bot API `entities` for the `message` tool, so custom emoji via `custom_emoji` entities is still a real missing Telegram plugin capability. The existing Telegram-specific send surface covers poll extras, inline buttons, quote text, pinning, silent sends, and document mode, but not message/caption entities.

## Best Possible Solution

Keep the issue open and implement it inside the bundled Telegram plugin, not shared core. A focused fix should add a Telegram-owned schema contribution for send entities, validate/narrow the entity payload, thread it through `extensions/telegram/src/action-runtime.ts` into `TelegramSendOpts`, and have `extensions/telegram/src/send.ts` send Bot API `entities` or caption entities only on compatible unsplit text/caption payloads. Add regression coverage for the message tool schema and the Telegram API params sent for a `custom_emoji` entity.

## Risks / Open Questions

- Implementing this needs Telegram-specific validation and tests for Bot API `MessageEntity` shape, especially `custom_emoji_id`, offset/length semantics, chunking, and interaction with the existing HTML parse-mode fallback.
- Entity support may need separate handling for text messages, media captions, and follow-up chunks so custom emoji is not silently lost or sent with incompatible `parse_mode` params.
