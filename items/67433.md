## Summary

Keep open. Current main still implements `POST /hooks/agent` as an authenticated fire-and-forget hook that returns only a run id; it does not parse `blocking`, await the isolated agent run, return response text, or expose blocking-run errors. This PR remains a live implementation candidate for the result-return capability tracked by #49019, not conservative cleanup.

## Best Possible Solution

Keep this PR open as the concrete implementation path. Maintainers should rebase/review it against current main’s split hook handler, validate blocking success/error/idempotency/trust semantics with targeted Gateway tests and docs, then either land it if the API shape is accepted or keep #49019 as the canonical product/API tracker if the contract needs redesign.
