## Summary

Keep open. Current main still lacks the PR's specific null-to-empty-string normalization for OpenAI-compatible Chat Completions assistant messages that have tool calls. The existing `requiresStringContent` path only flattens structured text-part arrays and does not handle `content: null`; the locked upstream `@mariozechner/pi-ai` conversion still has the null-content/tool-call shape described by the PR.

## Best Possible Solution

Keep the PR open and review it as a real bug-fix candidate. The best implementation is a current-main patch that normalizes only assistant messages with `tool_calls` and nullish `content` before OpenAI-compatible Chat Completions requests are sent, with regression coverage for both the wrapper/onPayload path and `buildOpenAICompletionsParams` or whichever transport path maintainers want to own this compatibility behavior. It should preserve null content on assistant messages without tool calls and leave Responses API payloads unchanged.
