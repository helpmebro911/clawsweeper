## Summary

Keep open. Current main still enqueues `openclaw system event --mode now` through the heartbeat path and intentionally skips the run when the target session lane is busy, which is the core failure reported here. There is no CLI/API priority or bypass-queue option on current main, and the KeepMyClaw comments describe a complementary snapshot/recovery workaround rather than an OpenClaw core fix for immediate in-session alert delivery.

## Evidence

- **CLI has no priority/bypass option:** `openclaw system event` accepts `--text`, `--mode`, and `--json`, then sends a `wake` RPC with only `{ mode, text }`. No `--priority` or `--bypass-queue` flag is exposed.
  - file: [src/cli/system-cli.ts:52](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/cli/system-cli.ts#L52)
  - command: `nl -ba src/cli/system-cli.ts`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Wake path only queues the event and requests heartbeat:** The cron wake implementation trims text, calls `enqueueSystemEvent(text)`, and for `mode: now` calls `requestHeartbeatNow({ reason: "wake" })`; it does not provide a direct session-message injection path.
  - file: [src/cron/service/timer.ts:1409](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/cron/service/timer.ts#L1409)
