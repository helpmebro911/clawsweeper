## Summary

Keep open. Current main still matches the reported root cause: sandbox file-tool writes create a `0o600` temp file and atomically replace the target without preserving the existing file mode. The issue discussion correctly identified this as the fix point, and the referenced fix PR #44114 was closed unmerged rather than shipped.

## Best Possible Solution

Fix `src/agents/sandbox/fs-bridge-mutation-helper.ts` in the `write_atomic()` path: capture the existing regular file's mode before replacement, apply that mode to the temp file with `os.fchmod()` before `os.replace()`, and preserve current symlink/hardlink safety checks. Add a regression test that writes over an existing `0644` file through the pinned mutation helper or sandbox bridge and asserts the mode remains `0644`; also cover new-file behavior deliberately.
