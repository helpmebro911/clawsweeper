## Summary

Keep this PR open. Current main still does not implement ACP control-plane audit logging, and the related gateway-tool audit work is a different surface. The PR remains potentially useful, but it should not merge as-is because it is behind current ACP lifecycle behavior, defaults to a null audit sink in normal construction, and has unresolved audit correctness and security hardening issues.

## Best Possible Solution

Keep the PR open but require a rebased, narrower ACP audit implementation before merge. The maintainable path is to preserve current `initializeSession` and `closeSession` semantics and tests, wire an explicit real audit sink under the normal OpenClaw state/log directory with restrictive permissions, make audit emission best-effort so sink failures cannot change ACP lifecycle outcomes, cover the claimed session/run/cancel/runtime-control events with focused regression tests, harden buffering/query/retention behavior, and split unrelated config/schema changes out of the PR. Treat #63557 as separate gateway tool-call audit work, not a replacement for ACP control-plane audit logging.

## Risks / Open Questions

- Merging this branch as-is would likely regress current ACP session lifecycle behavior and runtime option propagation because the branch is behind current `main` manager logic and tests.
- The advertised audit trail is not actually enabled for normal ACP manager construction; the PR adds a null default logger unless a real sink is wired through config/startup.
- Audit sink failures are security-relevant here: logger rejections can affect lifecycle calls, repeated flush failures can grow memory, query/flush races can miss or duplicate audit records, retention is not scheduled correctly, and file writes lack the restrictive state-dir/file-mode shape expected for sensitive audit data.
- The branch mixes unrelated config/schema churn into a security-sensitive ACP audit feature, increasing review and merge risk.
