## Summary

Keep this issue open. Current main still has Google Chat paths that can thread replies despite `channels.googlechat.replyToMode: "off"`: the inbound monitor sends the default typing placeholder into `message.thread?.name`, that placeholder can be edited into the first visible reply, and the routed outbound adapter still converts `threadId ?? replyToId` into a Google Chat thread send. The earlier attempted fixes #42705, #43076, and #43112 were closed unmerged; the later PR review context for #67055 also says the bug remains, though that PR needs revision before merge.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands. The best path is to suppress implicit inbound-thread propagation for `replyToMode: "off"` at the monitor/typing boundary and in routed outbound delivery, preserve explicit reply tags or explicit thread targets, and add focused Google Chat regression tests covering typing placeholders, text/media replies, and outbound `threadId` suppression.

## Risks / Open Questions

- A future fix needs to distinguish automatic Google Chat thread propagation from explicit reply directives; the shared reply policy intentionally preserves explicit `[[reply_to_*]]` targets for known channels even when `replyToMode` is `off`.
- Fixing only final payload delivery would miss the typing-placeholder path, while fixing only the monitor would still leave routed outbound sends able to pass `threadId` through the Google Chat adapter.
- The active/follow-up PR path should be revised against the current split `monitor-reply-delivery.ts` implementation and should include regression tests for typing placeholders, text/media reply delivery, and routed outbound text/media sends.
