## Summary

Keep #70470 open. Current main does not contain a source-, test-, or docs-backed fix for the reported WhatsApp sender-read-receipt behavior. The existing triage comment correctly points at the inbound read-receipt path, but the specific `access.shouldMarkRead` one-line diagnosis is not enough to close: blocked messages return before mark-read, while accepted messages still await `sock.readMessages` before downstream processing.

## Best Possible Solution

Keep the issue open and ask for maintainer or reporter validation on current main with sender read receipts disabled. If confirmed, the likely product fix belongs in the WhatsApp plugin inbound path: make mark-as-read non-blocking or bounded so `sock.readMessages` cannot prevent message enrichment/dispatch, then add a focused regression test around failed or delayed `readMessages` preserving accepted-message processing. If live testing disproves the WhatsApp path, retarget the issue to the Calendar/Zapier integration with logs.
