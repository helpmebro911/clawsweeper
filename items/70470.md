## Summary

Keep #70470 open. The current WhatsApp plugin still sends and awaits Baileys read receipts for accepted inbound messages before normal message processing, with `sendReadReceipts` defaulting to true. That means the reported read-receipt dependency is not clearly fixed on main or in v2026.4.24. The triage comment usefully points at the read-receipt path, but the exact suggested `access.shouldMarkRead` condition looks incomplete because blocked messages already return before `maybeMarkInboundAsRead`, while accepted messages set `shouldMarkRead: true`.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the bundled WhatsApp plugin inbound monitor: make read-receipt sending non-blocking or time-bounded and ensure message enrichment/enqueue proceeds regardless of `sock.readMessages` behavior. Add a focused regression test around accepted inbound messages where `readMessages` rejects or remains pending, plus retain existing tests for self-chat and `sendReadReceipts: false`.

- The existing triage comment’s proposed `access.shouldMarkRead` change may not address accepted-message failures; implementation should reproduce the sender-read-receipts-disabled case or at least add a regression test where `readMessages` stalls or fails without blocking `onMessage`.
