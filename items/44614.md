## Summary

Keep #44614 open. Current main still runs global npm updates in the gateway process and only schedules a SIGUSR1 restart after a successful update, so it does not implement the detached Windows update flow this PR targets. The PR is not safe to merge as-is because review comments identify real correctness gaps, but it remains a focused core gateway/update bugfix rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open or replace it with a focused maintainer-owned Windows update fix. The best implementation should make `update.run` safe for Windows global npm installs by delegating replacement work outside the locked gateway process, preserving the prepared install environment and npm fallback behavior, gating forced exits on a real restart-capable supervisor or fallback launcher, and reconciling detached install results on the next gateway start with tests for the default-delay and failure paths.
