## Summary

Keep open. Current main still accepts arbitrary non-empty CLI session-id strings from JSON, JSONL, and Claude live-session output, then can persist and reuse them with Claude CLI `--resume`. The related fix PRs cited in the discussion were closed unmerged, and their backend-aware validation is not present in current source.

## Best Possible Solution

Keep this issue open and implement backend-aware session-id validation. The likely fix belongs in the CLI backend contract and parser: declare UUID-shaped session ids for `claude-cli`, reject non-UUID canonical Claude `session_id`/`sessionId` values before persistence in JSON, JSONL, streaming, and live-session paths, and preserve opaque IDs for backends such as Codex. Add regression coverage for `session_id: "rate-limited"`, a valid Claude UUID, Claude live-session fallback, and an opaque non-Claude backend ID.
