## Summary

Keep open. Current main still forwards the Feishu `react` emoji value directly into the reaction add/remove paths, while the linked Feishu bug reports raw Unicode emoji such as `👍` failing with Feishu error 231001. This PR is a focused implementation candidate, but it should be revised before landing to avoid pulling Feishu runtime dependencies into the top-level channel module and to verify mappings against Feishu's current emoji_type vocabulary.

## Best Possible Solution

Keep this PR open for maintainer review or land an equivalent Feishu-owned fix for #66406. The safest implementation is a pure Feishu emoji normalizer backed by the current Feishu/go-lark emoji_type vocabulary, applied before both create and list/remove reaction calls without importing runtime SDK modules into `channel.ts`, with focused tests for Unicode thumbs-up, pass-through values, mixed-case Feishu types, trimming, unknown input behavior, and remove/list filtering.

## Risks / Open Questions

- Functional correctness: several proposed mappings may still be rejected by Feishu if the accepted emoji_type is mixed-case rather than the local uppercase convenience value.
- Runtime packaging: the PR's direct `channel.ts` import from `reactions.ts` would pull `@larksuiteoapi/node-sdk` through `client.ts` at top-level, undermining the current lazy runtime dependency boundary for the Feishu plugin.
- Security/supply-chain review: the PR does not touch workflows, dependency manifests, lockfiles, install scripts, release metadata, permissions, or downloaded artifacts; the observable risk is runtime dependency loading, not a new third-party execution path.
