## Summary

Keep open. Current main still trims Feishu reaction emoji input and forwards it unchanged to the add and remove/list reaction API paths, while the linked bug remains open. This PR is a focused implementation candidate, but it should be revised before merge to preserve the Feishu lazy runtime boundary and verify the emoji mapping against the current Feishu/Lark emoji_type vocabulary.

## Best Possible Solution

Keep this PR open for maintainer review, or land an equivalent Feishu-owned fix. The best implementation is a lightweight Feishu emoji normalizer that does not import the runtime client, applied before both addReactionFeishu and listReactionsFeishu in the react action, with tests for Unicode thumbs-up, Feishu type passthrough, mixed-case input, trimming, unknown input behavior, and remove/list filtering.

## Risks / Open Questions

- Merging this PR as-is could pull the Feishu runtime client and @larksuiteoapi/node-sdk into top-level channel import paths, weakening the staged/lazy runtime dependency model.
- The mapping table in the PR should be checked against Feishu/Lark's current emoji_type vocabulary before landing; otherwise OpenClaw could normalize to unsupported values.
- The linked bug remains open and current main still lacks normalization, so closing this PR as obsolete would leave the reported Feishu reaction failure unresolved.
