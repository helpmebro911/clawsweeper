## Summary

Current `main` still does not implement the PR’s explicit-empty-tools behavior. The system prompt builder falls back to a generic tool-enabled prompt when `toolNames` is empty, the skills block/reporting are still keyed off raw `skillsPrompt`, embedded prompts still do not advertise hosted/client-only tools correctly, and provider prompt transforms still do not receive the final tool-name context. This PR remains actionable rather than closeable.

## Evidence

- **Empty tool lists still render a tool-enabled fallback prompt:** `buildAgentSystemPrompt` has no explicit-empty-tools branch. When the normalized tool list is empty, it falls back to a hardcoded catalog (`exec`, `process`, `browser`, `cron`, `sessions_*`, `subagents`, etc.) and keeps tool-only guidance such as poll-loop and sub-agent instructions, which is the opposite of a true no-tools prompt.
  - file: [src/agents/system-prompt.ts:679](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/system-prompt.ts#L679)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Skills are still injected whenever `skillsPrompt` is present:** `buildSkillsSection` returns `## Skills (mandatory)` for any non-empty `skillsPrompt`; there is no guard for an explicitly empty effective tool set.
  - file: [src/agents/system-prompt.ts:156](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/system-prompt.ts#L156)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Prompt report still uses raw skills input instead of rendered prompt:** `buildSystemPromptReport` still parses `params.skillsPrompt` directly and reports `promptChars: params.skillsPrompt.length`, so report metadata can stay non-zero even if a future prompt transform omits or rewrites the skills section.
  - file: [src/agents/system-prompt-report.ts:96](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/system-prompt-report.ts#L96)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Embedded prompt builder still ignores hosted/client-only tools:** `buildEmbeddedSystemPrompt` only forwards `params.tools.map((tool) => tool.name)` into `buildAgentSystemPrompt`; there is no client-tools input here, so client-tool-only runs still look tool-less from the prompt builder’s perspective.
  - file: [src/agents/pi-embedded-runner/system-prompt.ts:80](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/system-prompt.ts#L80)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Runtime allowlist already includes client tools, so prompt/runtime can disagree:** The embedded runner’s actual allowlist includes both native tools and `clientTools`, which means current `main` can expose callable hosted tools at runtime while the prompt builder still advertises only native-tool names.
  - file: [src/agents/pi-embedded-runner/tool-name-allowlist.ts:20](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/tool-name-allowlist.ts#L20)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Provider transforms still lack final tool-name context:** `ProviderTransformSystemPromptContext` still exposes only the assembled `systemPrompt`; it does not provide the final advertised tool-name list, so provider-specific prompt rewrites still cannot reliably align no-tools behavior with the real runtime catalog.
  - file: [src/plugins/types.ts:1111](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/plugins/types.ts#L1111)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Existing discussion still points to an unfixed mainline bug:** In the PR thread, the author repeatedly reported a live repro that still failed on `main`/`v2026.4.11` and only passed on the branch after follow-up fixes. Current `main` source still matches the pre-fix shape described there, and the thread does not contain contrary maintainer evidence.
