## Summary

Current `main` still bails out when `reserveTokensFloor + softThresholdTokens >= contextWindowTokens`, so the underlying bug this PR targets is still plausible on the memory-flush path. The PR itself is stale against the April 24 refactor and carries an unrelated `pnpm-lock.yaml` diff, but there is not strong evidence to close it as already implemented or obsolete.

## Evidence

- **Current gate still disables flush on non-positive threshold:** `resolveMemoryFlushGateState` computes `threshold = Math.max(0, contextWindow - reserveTokens - softThreshold)` and returns `null` when `threshold <= 0`, which means `shouldRunMemoryFlush` and `shouldRunPreflightCompaction` both skip the flush/compaction path for oversized reserves.
