## Summary

Current main still does not implement the requested opt-in custom safe-header extension for cross-origin fetch-guard redirects. The redirect filter is a fixed safe-header allowlist, fetchWithSsrFGuard applies it unconditionally on cross-origin redirects, tests explicitly keep x-api-key in the stripped-header set, and docs/config searches did not find a fetchGuard.additionalSafeHeaders-style surface. This is a concrete core security/product design request, not a ClawHub-only item or duplicate of the related bearer-token fallback issue.

## Best Possible Solution

Keep this issue open for maintainer security/product judgment. If accepted, implement a narrow opt-in in the core fetch-guard/redirect-header path, validate header names, preserve the existing default safe allowlist, deny known credential headers regardless of config, document the approved config or SDK surface, and add regression tests for allowed custom headers, denied credential headers, and cross-origin redirect behavior. Coordinate with the related #57588 security discussion but do not treat it as a duplicate.

## Risks / Open Questions

- A global additional-safe-header setting could let auth-like custom headers follow attacker-controlled redirects; any implementation needs fail-closed defaults, header-name validation, and explicit denial of credential headers such as authorization, cookie, and proxy-authorization.
- Because fetchWithSsrFGuard is used by core, media, provider transports, and plugins, a broad API/config change could alter security behavior across many call sites unless scoped carefully.
