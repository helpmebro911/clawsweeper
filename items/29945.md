## Summary

Keep #29945 open. Current main still hardcodes branded pairing replies for newly created unpaired DM pairing requests, has no `unpairedResponse` config contract, and the direct maintainer PR #39065 remains the relevant unmerged fix path in the provided GitHub context.

## Best Possible Solution

Keep #29945 open until #39065 or an equivalent maintainer-approved change lands on main. The fix should add an explicit unpaired-DM response policy, wire it through shared pairing helpers and affected channel/plugin call sites, update config schema/generated metadata and public docs, add regression coverage for silent, code-only, and branded behavior, and audit related pairing logs/error responses for raw sender identifiers.
