## Summary

Keep this issue open. Current main and v2026.4.24 address several likely subproblems from the 2026.4.11 report, but I did not find enough end-to-end evidence that the native Windows Google/Gemini Control UI hang is fixed for the reporter's setup.

## Best Possible Solution

Keep this issue open. Ask the reporter or a maintainer to retry on the latest release/current main and collect redacted Gateway logs, `openclaw doctor --fix` output, `openclaw models status --probe --json`, active provider/model config, Node version, install method/package version, and whether the native Windows Gateway process sees `GEMINI_API_KEY` or `GOOGLE_API_KEY`. If still reproducible, fix the Windows native Google/Gemini `chat.send` path or the generic Control UI chat lifecycle so provider/auth failures always produce a visible terminal error instead of leaving the run active.
