## Summary

Keep this issue open. Current main still appears to miss the reporter's exact Slack binding shape: inbound Slack routing passes the raw channel ID into generic route matching, while configured route-binding peer IDs are only trimmed and compared literally. A binding with `peer.id: "channel:C..."` therefore still does not match inbound `message.channel: "C..."`, causing fallback to the default agent and `agent:main:...` session keys. The related PR #42162 was closed unmerged, and I did not find an equivalent shipped fix on current main.

## Best Possible Solution

Keep this issue open and fix the Slack route-binding normalization path. The likely fix is to normalize Slack route-binding peer IDs so target-style forms such as `channel:C123` and `user:U123` match the raw inbound Slack IDs before generic route comparison, without changing the canonical session key shape. Add focused regression tests for Slack channel and user bindings using both raw and target-prefixed IDs, and update config help/docs or doctor warnings if maintainers want raw IDs to remain the preferred form.
