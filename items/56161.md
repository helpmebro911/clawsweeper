## Summary

Keep open. Current main still resolves the skills CLI workspace through the default agent rather than inferring the agent from the current working directory, and an open, unmerged implementation candidate (#72726) is linked with a closing reference to this issue. The current behavior still conflicts with the active-workspace behavior described in docs.

## Best Possible Solution

Keep this issue open and review, land, or supersede #72726. The best fix is a shared skills workspace resolver that prefers an explicit agent override if accepted, then `resolveAgentIdByWorkspacePath(config, process.cwd())`, then the default-agent fallback, with regression tests for install/update and likely list/info/check for consistency.
