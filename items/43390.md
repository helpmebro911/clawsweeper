## Summary

Keep this PR open. Current main still uses a module-level cron `storeLocks` map keyed by `storePath` with `get()` and `set()` but no `delete()` or `finally` cleanup, so the reported unbounded retained-entry behavior remains present. The PR is a focused implementation candidate with targeted regression coverage, and the related #43386 refactor is closed unmerged rather than a superseding shipped fix.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused cleanup in `src/cron/service/locked.ts` that deletes the per-store lock entry only when the completing promise is still the current map entry, with regression coverage for post-completion pruning and queued-operation retention. Maintainers can adjust the test seam if they do not want to export the mutable map.
