## Summary

Keep this PR open. Current main still retains one module-level `storeLocks` entry per cron `storePath` with no cleanup after queued work resolves, while this PR is a narrow cron-service fix with targeted coverage. The related refactor #43386 is closed and unmerged, so it does not supersede the cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused `finally` cleanup that deletes the per-store lock only when the completing promise is still the current map entry, with regression coverage for both post-completion pruning and queued-operation retention.
