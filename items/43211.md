## Summary

Current main still does not implement hyphen-stripped allowlist variants, but this PR should stay open rather than be closed or merged as-is. The proposed unconditional variant expansion would broaden the model authorization allowlist for provider ids that current code and docs treat as distinct unless they are explicit aliases.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. The best path is to provide a concrete reproduction for the runtime path that rewrites a hyphenated provider id, then fix that specific normalization path or express the equivalence through explicit provider/plugin alias metadata. Regression coverage should prove known aliases resolve while distinct providers such as `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Security/authorization risk: merging the PR as written can allow `myprovider/some-model` when only `my-provider/some-model` was configured, even though current provider normalization permits those provider ids to be distinct.
- The PR body asserts that some runtime path strips provider hyphens, but the provided discussion and current main inspection do not identify a concrete reproduction path for that rewrite.
- The PR adds tests for the broadened behavior but not for collision cases where hyphenated and non-hyphenated provider ids are intentionally separate.
- The PR touches only source and tests, so I did not find a supply-chain, workflow, dependency, lockfile, or secret-permission regression beyond the model authorization broadening itself.
