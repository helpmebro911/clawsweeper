## Summary

Current main does not implement the PR’s hyphen-stripped allowlist behavior, but the PR should stay open rather than be closed as obsolete. The proposed fix changes the model allowlist authorization path and can allow a distinct provider id (`myprovider`) when only `my-provider` was configured. The safer path is explicit provider alias/canonicalization metadata with collision tests, not unconditional hyphen stripping.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. If the mismatch is real, implement it through an explicit provider alias/canonicalization contract owned by the provider/plugin or config metadata, then add regression tests showing known aliases resolve while distinct providers such as `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Merging the PR as written can grant access to a distinct provider id when both hyphenated and hyphenless providers exist, because authorization is keyed by exact `provider/model` strings today.
- The PR’s tests prove the new acceptance behavior but, based on the provided patch, do not cover the collision case where `my-provider` and `myprovider` are separate configured providers.
- The reported mismatch may still represent a real bug, but it needs a concrete reproduction path showing which runtime path rewrites a hyphenated custom provider id to a hyphenless id before the fix should be accepted.
