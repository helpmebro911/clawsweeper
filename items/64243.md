## Summary

Not safe to auto-close. Current main still treats `tools.exec.ask`/`security` as a requested policy that can be tightened by the host exec-approvals file, so `tools.exec.ask="off"` alone does not prove a bug. The repo also only proves no-approval execution when the effective host policy is `security="full"` + `ask="off"`. Mattermost now has approval-command auth wiring, but this review did not find strong end-to-end evidence that the specific deploy-probe timeout path is fixed on main.

## Evidence

- **Host policy can still override tool config:** Gateway/node exec computes effective policy by taking the stricter host-side values: `hostSecurity = minSecurity(...)`, `hostAsk = maxAsk(...)`, and `askFallback = minSecurity(...)`. That means `tools.exec.ask="off"` does not guarantee no prompt if `~/.openclaw/exec-approvals.json` is stricter.

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Approval is skipped only for effective full/off policy:** `requiresExecApproval(...)` returns `true` for `ask="always"`, or for `ask="on-miss"` in allowlist misses; otherwise it does not require approval. That matches the expected no-prompt behavior only when the effective policy really is `full/off`.
