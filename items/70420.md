## Summary

Keep open. Current main still has the reported contract gap: Discord health-monitor reconnect updates runtime channel status, while `sessions_list` is backed by persisted session-store entries that are created from inbound/session activity, not by provider reconnect. The existing docs point users to live health/channel probes but do not explicitly state that `sessions_list` is not a channel-liveness signal after reconnect.

## Best Possible Solution

Keep the issue open for a channel-health owner decision. The conservative path is a docs/status guidance fix: state that `sessions_list` is persisted session inventory, not channel socket liveness, and point heartbeat/connectivity checks at runtime channel status such as `openclaw channels status --probe`, `openclaw status --deep`, or `openclaw health --json`. If maintainers want `sessions_list` to be a stable liveness signal instead, add an explicit core/plugin lifecycle contract to materialize or refresh a channel heartbeat/session row on successful start/reconnect, with tests for quiet post-reconnect workspaces.
