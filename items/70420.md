## Summary

Keep #70420 open. Current main still has the exact contract gap described in the issue: `sessions_list` is backed by stored conversation/session rows, while the channel health monitor restarts the Discord runtime through stop/start and the Discord lifecycle only publishes channel status on reconnect. Discord session rows are still materialized by inbound interaction/message processing, so a quiet workspace after a successful reconnect can show Discord healthy while `sessions_list` has no Discord entries. The related items in the provided context are adjacent but not canonical duplicates for this sessions_list-vs-channel-liveness contract/docs issue.

## Best Possible Solution

Keep the issue open for channel-health ownership. The likely best fix is to document and surface that channel status/readiness (`openclaw status --deep`, `openclaw health`, `/readyz`, or `channels status --probe`) is authoritative for Discord socket/channel liveness, while `sessions_list` is stored conversation inventory. Only implement synthetic channel session materialization if maintainers intentionally want `sessions_list` to become a liveness signal across all channel plugins.
