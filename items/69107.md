## Summary

Keep open. The report is a recent, concrete packaging request, and current main still publishes Docker images from the same Node 24 Debian bookworm/bookworm-slim baseline that the issue says blocks derived images needing newer glibc. This is not a ClawHub/plugin close and not external-only administration: the requested change lives in OpenClaw's Dockerfile, Docker docs, and Docker release workflow.

## Best Possible Solution

Keep the issue open for maintainer packaging judgment. A concrete resolution should either rebase the main Docker image or add a supported newer-base image variant, then update `Dockerfile`, Docker docs, and the Docker release workflow together; if maintainers decide derived-image libc compatibility is out of scope, document that boundary explicitly in the Docker docs.

## Evidence

- **Issue reports concrete derived-image failure:** The issue body says a custom image derived from the OpenClaw Docker image installs `@googleworkspace/cli`, but `gws --help` fails with `GLIBC_2.39 not found` while the container reports Debian GLIBC 2.36. The issue explicitly narrows the request to considering a newer Docker base, not bundling or supporting `gws`.

## Risks / Open Questions

- Rebasing the Docker base can affect image size, native dependency compatibility, apt package availability, Docker CLI repository setup, security update cadence, and multi-arch release behavior.
- OpenClaw cannot reasonably guarantee every third-party CLI added by derived images will run; maintainers may choose a documented support boundary, a newer base, or a separate image variant.
