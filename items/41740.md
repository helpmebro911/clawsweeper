## Summary

Current main does not appear to implement the requested fix. Discord native exec approval resolution still reaches `GatewayClient` through the shared operator-approvals client, and that client only forwards URL/auth/bootstrap data, not the local self-signed TLS fingerprint. The linked fix attempts in the provided context were closed unmerged, so this remains an actionable core/gateway bug rather than a stale cleanup close.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the shared operator approvals gateway client/bootstrap path, not only Discord-specific code: resolve the same local gateway TLS runtime fingerprint used by `callGateway` when the configured local gateway is `wss://` with `gateway.tls.enabled === true`, avoid applying it to explicit CLI URL overrides, pass it into `GatewayClient`, and add focused regression coverage for `createOperatorApprovalsGatewayClient` or `resolveGatewayClientBootstrap`. The previous closed PRs #41741 and #46232 are useful prior art but should not close the issue unless a tested equivalent lands on main.
