## Summary

Keep open. Current main still has the same provider/model-specific gap described in this issue: Qwen/legacy ModelStudio exposes the affected Alibaba Coding Plan models through the generic OpenAI-compatible transport, and that transport only turns structured `choice.delta.tool_calls` into tool calls while preserving `reasoning_content` as thinking. The Kimi marker parser is real but scoped to the separate Kimi Coding wrapper and only rewrites standalone text blocks, not tagged tool calls inside thinking blocks. The MiniMax thinking mitigation is similarly scoped to native MiniMax Anthropic-message routes, not `qwen/MiniMax-M2.5`.

## Best Possible Solution

Keep this issue open and fix it in provider-owned stream normalization. The likely path is a Qwen/ModelStudio OpenAI-compatible wrapper or shared OpenAI-compatible normalization seam that detects valid tagged Kimi-style tool-call sections in reasoning/thinking content, converts them to structured tool-call blocks, and updates `stopReason` only when extraction succeeds. Add focused regression coverage for `qwen/kimi-k2.5`, `qwen/MiniMax-M2.5`, and the existing Kimi Coding wrapper so text-block and thinking-block behavior are explicit.
