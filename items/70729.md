## Summary

Current main still appears to drop the trusted-proxy user before webchat messages reach the agent. The auth layer extracts and authorizes the header value, but the WebSocket client state does not retain it, and `chat.send` fills `SenderId`/`SenderName` from the static Control UI client metadata instead of the authenticated proxy identity.

## Best Possible Solution

Keep the issue open and implement a core gateway fix: persist the authenticated trusted-proxy subject on `GatewayWsClient` at handshake time, thread it into webchat `chat.send` message context as either `SenderId`/`SenderName` or a new explicit authenticated-user field, and add a regression test for trusted-proxy webchat `chat.send` propagation. The optional sessions metadata request can be handled in the same patch or a narrower follow-up.

## Risks / Open Questions

- The fix should use the server-authenticated trusted-proxy result from the WS handshake, not any client-supplied RPC field or arbitrary header value on `chat.send`.
- Mapping trusted-proxy identity into existing `SenderId`/`SenderName` may affect command, hook, and tool authorization semantics; a distinct trusted identity field may be safer if maintainers want to preserve channel-native sender semantics.
