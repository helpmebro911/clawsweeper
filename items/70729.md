## Summary

Keep #70729 open. Current main authenticates trusted-proxy requests and extracts the forwarded user, but the WebSocket client state and `chat.send` agent message context do not preserve that authenticated user. Webchat currently supplies `SenderId`/`SenderName` from the Control UI client identity, not from `X-Forwarded-User`, and `sessions.create` also does not record the proxy-authenticated owner.

## Best Possible Solution

Keep the issue open and implement this in the gateway path: preserve the trusted-proxy authenticated user on the accepted WS client/auth context, thread it into webchat-originated `chat.send`, populate `MsgContext` sender identity or a distinct authenticated-user field by maintainer decision, optionally persist the owner on `sessions.create`, and add a regression test for trusted-proxy Control UI connect plus webchat `chat.send` context.

## Risks / Open Questions

- The implementation needs a clear security boundary: the proxy-authenticated identity should come from verified auth state, not from browser-provided personal identity or arbitrary `chat.send` params.
- A maintainer should choose whether to map the trusted-proxy user directly to `SenderId`/`SenderName` or add a distinct authenticated-user field to avoid conflating proxy identity with channel-native sender identity.
