## Summary

Keep open. Current `main` at `11cffb230013c2d8bd1e0e18da04b0b985b8abd5` still uses ASCII-only slug regexes, so non-Latin group names collapse to an empty token and `buildGroupDisplayName` returns only the provider key, matching the report.

## Evidence

- **ASCII-only normalization still on main:** `normalizeHyphenSlug` and `normalizeAtHashSlug` still strip everything outside `[a-z0-9...]` via `/[^a-z0-9#@._+-]+/g` and `/[^a-z0-9-]+/g` in `src/shared/string-normalization.ts`.
  - file: [src/shared/string-normalization.ts:54](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/shared/string-normalization.ts#L54)
  - command: `nl -ba src/shared/string-normalization.ts | sed -n '45,90p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Display-name path still depends on that normalization:** `buildGroupDisplayName` normalizes the subject/space/channel label, and if normalization returns empty it falls back to provider-only output; `deriveGroupSessionPatch` still assigns `displayName` from that helper.
  - file: [src/config/sessions/group.ts:45](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/config/sessions/group.ts#L45)
