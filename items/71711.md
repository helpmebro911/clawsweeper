## Summary

Keep #71711 open. Current main does not implement a per-job `activeUntil`/`expiresAt` or `onExpire` contract for recurring cron jobs: the cron data model, protocol schema, CLI add path, UI form mapping, docs, and scheduler lifecycle only support schedule kind, enabled state, `deleteAfterRun`, and runtime next-run state. The linked #71712 RFC depends on this idea for agent-created recurring schedules, but it is broader and open; it does not supersede this narrower core cron feature request.

## Best Possible Solution

Keep the issue open and implement it in core cron as an additive job field: update cron types, normalization, Gateway protocol validators, CLI add/edit/show/list, Control UI form/list mapping, scheduler pre-run expiration handling (`remove` or `disable`), audit/run-log behavior, docs, and focused regression tests for `every` and `cron` schedules. Treat #71712 as related product/security design input, not as a replacement for this issue.

## Risks / Open Questions

- This is a public cron job and Gateway protocol surface change, so it should be additive and keep existing job files compatible.
- The narrower core feature should be coordinated with the open #71712 agent-facing scheduling RFC if maintainers want recurring agent-created jobs to require bounded lifetimes and provenance/audit semantics.
