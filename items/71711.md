## Summary

Keep #71711 open. Current main does not have a recurring cron expiration field or scheduler behavior; the request is a concrete core cron lifecycle feature, not stale, not duplicated by the related #71712 RFC, and not appropriate to close as ClawHub because it requires changes in OpenClaw's persistent cron schema, gateway protocol, CLI, scheduler, and docs.

## Best Possible Solution

Keep #71711 open for maintainer/product review and implementation in core cron. The best path is an additive optional recurring-job expiration contract, probably activeUntilMs or activeUntil ISO plus onExpire, validated through the gateway protocol and CLI, rejected for one-shot at jobs, checked before execution and maintenance recompute, and surfaced in list/show/docs/tests. This should not be redirected to ClawHub because the missing behavior lives in OpenClaw's built-in scheduler and durable cron job lifecycle.

## Risks / Open Questions

- Implementing this is an additive persisted cron contract change, so it should update the core CronJob types, gateway protocol validators, CLI add/edit/list/show surfaces, agent cron tool schema, UI/app cron models where applicable, tests, and public docs together.
- The scheduler behavior needs careful ordering: expire before starting a due recurring run, persist remove/disable atomically, and emit/audit a distinct expiration event without breaking existing one-shot deleteAfterRun semantics.
