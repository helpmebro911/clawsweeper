## Summary

Keep open. Current main still clears auto-sourced model fallback overrides before the next run and retries the configured primary, so it does not provide the requested cache-aware sticky fallback window across future turns. The related OpenRouter routing work helps #62992 inside OpenRouter requests, but it does not solve cross-provider prompt-cache bouncing after automatic failover.

## Best Possible Solution

Implement a core/session failover policy that records a successful automatic fallback with an expiry, honors it across future direct turns before clearing the auto override, probes the configured primary once after expiry, and extends the sticky window when that probe fails. Put config near auth cooldown/model failover settings, support fixed and cacheRetention-derived durations, preserve explicit user overrides, and add regression tests for primary timeout -> fallback success -> next turn starts on fallback -> expiry probes primary. Keep the closed OpenRouter provider-routing work as a separate mitigation for #62992, not a replacement for this issue.
