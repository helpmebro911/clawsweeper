## Summary

Current main partially mitigates the reported failure by using a 48-hour default embedded-agent runtime cap and adding an LLM stream idle watchdog, but it does not implement the requested activity-aware agent run timeout. The embedded runner still arms a single wall-clock abort timer for the run and does not reset it on tool completion or assistant activity, and there is no agent-level idle/max timeout config surface.

## Best Possible Solution

Keep this issue open and narrow the desired implementation: add an agent-run idle timeout plus an absolute max timeout, or an equivalent watchdog that resets on assistant stream activity and tool completion while retaining a hard cap. The change should update config schema/help/docs and include tests proving active multi-tool runs are not killed solely because the original wall-clock window elapsed.

## Risks / Open Questions

- Closing now would mark a real remaining feature as done: users who configure or invoke a shorter run timeout can still have productive multi-tool runs aborted by wall-clock time.
- The current 48-hour default and LLM idle watchdog are useful mitigations, so maintainers should decide whether the product wants the full idle/max run-timeout contract or a smaller documentation/config cleanup.
