## Summary

Keep #71609 open. Current main has partial Control UI auth hardening, including explicit shared-token precedence, a stable app-level instanceId, and a bounded cached-device-token retry, but it still does not fully solve the reported stale device-token recovery loop. Browser-side cached-token-as-primary failures can still reconnect on `AUTH_TOKEN_MISMATCH` without clearing stale device auth, `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as terminal, and gateway/user guidance still uses the rotate/reissue wording the issue calls out.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to clear stale cached device auth for stale device-token failures, avoid automatic reconnect loops once no safe bounded retry is available, preserve the current stable app-level instanceId behavior, and replace the gateway/client-facing rotate/reissue wording with stale identity cleanup or reconnect guidance. Add focused browser tests for stored-token-as-primary mismatch, explicit retry failure, and no rate-limiter-burning reconnect loop.
