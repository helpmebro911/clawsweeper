## Summary

Keep #71609 open. Current main has partial Control UI auth hardening, including explicit shared-token precedence and a one-shot cached-device-token retry, but the exact stale cached device-token mismatch path can still reconnect in a loop and burn the gateway auth limiter. The prior ClawSweeper comment correctly distinguishes this report from older closed token/signature mismatch issues: those fixes cover shared-token precedence, not full cleanup and non-looping recovery after `AUTH_DEVICE_TOKEN_MISMATCH`.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to clear stale cached device auth on `AUTH_DEVICE_TOKEN_MISMATCH` when the browser used stored device-token auth, stop automatic reconnect loops when no safe retry is possible, avoid consuming the auth rate limiter, preserve the stable app-level `clientInstanceId`, and replace the `rotate/reissue` message with stale-identity cleanup or reconnect guidance. Add focused browser tests for stale cached device-token recovery after scope upgrade/approval and for the non-looping mismatch path.
