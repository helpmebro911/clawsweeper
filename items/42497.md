## Summary

Keep this PR open. Current main still sends agent gateway-tool calls through the WebSocket-backed `callGateway` path, and the cron agent tool still depends on that helper for cron status/list/add/update/remove/run/runs. The PR remains a plausible implementation candidate for the WS self-contention reports, but it needs a current-main rebase/adaptation because the gateway-tool contract now includes explicit scopes, URL/token/timeout handling, `expectFinal`, and WebSocket serialization behavior that the PR's shown local-dispatch path does not fully preserve.

## Best Possible Solution

Keep this PR open and ask for a current-main rebase or adapted patch. The maintainer-ready fix should add an embedded in-process path only for gateway-native calls while preserving explicit `gatewayUrl`, explicit `gatewayToken`, `timeoutMs`, `expectFinal`, explicit `extra.scopes`, least-privilege default scopes, and WebSocket-equivalent JSON request normalization. Add focused regression tests for cron status/list/add/update/remove/run/runs plus URL, token, timeout, expectFinal, explicit-scope, default-scope, and optional/undefined parameter cases before merge.

## Risks / Open Questions

- The PR is based on an older gateway-tool contract and, as shown, does not carry current `extra.scopes` through the local path.
- The local dispatch path changes a security-sensitive boundary from WebSocket-authenticated gateway RPC to in-process dispatch; credential, explicit token, URL override, and scope parity need deliberate handling before merge.
- The local path can change timeout and `expectFinal` semantics unless it applies equivalent deadline and final-response behavior or explicitly avoids local dispatch for those cases.
- Forwarding params directly to in-process handlers can differ from the WebSocket JSON round trip, especially for optional properties set to `undefined`.
- Closing this PR now would discard a focused candidate for a bug that current main still appears not to solve.
