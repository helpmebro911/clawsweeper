## Summary

Keep this PR open. Current main still has `callGatewayTool` resolve options/scopes and call the WebSocket-backed `callGateway` path, while the cron tool still routes status/list/add/run/runs through that helper. The PR is still the canonical implementation candidate for the linked gateway self-contention bug, but it should be rebased and reworked before merge because the proposed in-process dispatch path touches an auth-sensitive boundary and can bypass current token, timeout, scope, and serialization semantics.

## Best Possible Solution

Keep this PR open and ask for a current-main rebase or adapted patch. The maintainer-ready fix should add an in-process dispatch path for embedded gateway-native tool calls while preserving gateway URL/token resolution, explicit remote targeting, timeout behavior, `expectFinal`, explicit and least-privilege scopes, and JSON-like parameter normalization. Add focused regression tests for cron tool calls plus gatewayUrl, gatewayToken, timeoutMs, expectFinal, scope, and undefined-parameter cases, then do maintainer security review before merge.

## Risks / Open Questions

- Landing the PR as-is against current main could bypass newer `extra.scopes` and least-privilege gateway authorization behavior for embedded tool calls.
- The proposed local path is security-sensitive because it avoids normal WebSocket authentication and transport timeout handling; explicit token and timeout semantics need equivalent handling or a deliberate documented exclusion.
- The local path must preserve WebSocket serialization behavior for `undefined` fields, or validators can reject embedded calls that currently succeed over the WS path.
