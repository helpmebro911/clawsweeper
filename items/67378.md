## Summary

Keep open. Current main still routes proactive WhatsApp phone-number sends through PN JIDs (`digits@s.whatsapp.net`) and only uses LID mapping for inbound/reverse normalization, not the outbound phone-to-LID path described by the reporter.

## Best Possible Solution

Keep this issue open and fix it in the WhatsApp plugin outbound path. The likely fix is an account-aware recipient resolver wired from the active listener/authDir and Baileys `signalRepository.lidMapping` (`getLIDsForPNs` and/or persisted `lid-mapping-{digits}.json`), then use that resolved JID for proactive text, media, polls, presence, reactions, quoted-message lookup, and focused regression tests.
