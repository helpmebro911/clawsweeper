## Summary

Keep this issue open. Current main still sends proactive WhatsApp phone-number targets through plain PN JID normalization in the outbound web send path, while the existing LID support is inbound/reverse-only. The discussion says there was no PR and only a local compiled-bundle workaround, and current source/tests still preserve the outbound PN-to-LID gap.

## Best Possible Solution

Keep this issue open and fix it in the WhatsApp plugin outbound path. Add an account-aware recipient resolver with access to the active listener/auth directory and Baileys LID mapping, resolve plain phone-number targets to mapped @lid JIDs when available, keep unmapped numbers falling back to @s.whatsapp.net, and route text/media, polls, reactions, composing presence, quoted-message context, logs, and returned toJid through the resolved JID. Add focused regression tests for mapped and unmapped proactive sends.
