## Summary

Keep #70777 open. Current main has some workspace-cwd protections for configured ACP bindings and normal embedded runs, but the ACP manager still trusts persisted ACP cwd/runtimeOptions when rehydrating a session. That is close to the reporter's suspected failure mode, and there is no strong current-main evidence that an intermittent bad main-session cwd is repaired or impossible.

## Best Possible Solution

Keep the issue open and fix it in the ACP session lifecycle. The likely maintainer path is to derive a desired cwd for main/default ACP sessions from agents.list[].runtime.acp.cwd or the resolved agent workspace, reconcile persisted ACP meta/runtimeOptions against that desired cwd on startup/run, and add a regression test where a stored main ACP session has cwd=/home/user while config has agents.defaults.workspace=/home/user/.openclaw/workspace. If changing existing persisted sessions is risky, add a doctor/fix repair path plus targeted runtime reinitialization for mismatched configured workspaces.
