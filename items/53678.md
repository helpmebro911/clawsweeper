## Summary

Keep this issue open. Current main still does not provide a cross-channel Gateway-received timestamp in shared inbound context, agent-visible inbound metadata, hook payloads, or the plugin SDK helpers used by channel plugins. The existing Telegram and QQBot `receivedAt` values are plugin-local timing/debug data, not a shared `receivedAt` or `gatewayReceivedAt` metadata field exposed to agents.

## Best Possible Solution

Add an additive typed gateway receipt timestamp, such as `GatewayReceivedAtMs` or `ReceivedAtMs`, to the shared inbound context. Set it once at channel ingress or in a shared dispatch/finalization helper, propagate it through plugin SDK direct-DM/envelope helpers and hook payloads, render it to agents as a stable ISO `receivedAt` or `gatewayReceivedAt` user-context metadata field, and add tests for the metadata builder plus at least one buffered or long-poll-style channel path.

## Risks / Open Questions

- The fix would touch public plugin SDK and agent-visible prompt metadata, so it should be additive and preserve `Timestamp` as the source-channel timestamp.
- Adding a per-message receipt timestamp to prompt/user metadata can affect prompt-cache stability; it should stay out of stable system-prefix metadata and be covered by focused tests.
