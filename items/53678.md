## Summary

Keep open. Current main still exposes the source-channel timestamp through shared inbound context and agent-visible metadata, but it does not expose a generic Gateway-received timestamp. Local receipt-time variables exist in a few bundled plugins for logging or command-local latency, yet they are not part of the shared MsgContext, plugin SDK inbound helpers, hook events, or prompt metadata path that agents see.

## Best Possible Solution

Add an additive typed gateway receipt timestamp to the shared inbound contract, likely `GatewayReceivedAt` or `ReceivedAtMs` alongside `Timestamp` so source-channel time remains intact. Set it once at channel ingress or in shared inbound dispatch helpers, propagate it through plugin SDK helpers such as `direct-dm`, `inbound-envelope`, and `inbound-reply-dispatch`, expose it to hook events if hooks should observe latency, render it in agent-visible inbound metadata with a stable ISO/timezone format, and add focused tests for the metadata builder plus at least one webhook/long-poll-style channel path. External long-poll plugins such as openclaw-weixin should then be able to pass the value through the public channel/plugin contract without core learning channel-specific behavior.
