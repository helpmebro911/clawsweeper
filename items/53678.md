## Summary

Keep this issue open. Current main still does not expose a shared gateway-received timestamp in inbound context, agent-visible inbound metadata, plugin SDK dispatch/envelope helpers, or message hook payloads. The existing `receivedAt`/`receivedAtMs` usages are plugin-local timing/debug fields, not a cross-channel `receivedAt` or `gatewayReceivedAt` field available to agents.

## Best Possible Solution

Keep this issue open. Add an additive typed receipt timestamp such as `GatewayReceivedAtMs` or `ReceivedAtMs` to shared inbound context, set it once at channel ingress or finalization while allowing buffered/long-poll plugins to preserve the original receipt time, propagate it through plugin SDK direct-DM/envelope helpers and hook payloads, render a stable ISO `receivedAt` or `gatewayReceivedAt` field in agent-visible inbound metadata, and add focused tests for the metadata builder plus at least one buffered or long-poll-style path.

## Risks / Open Questions

- The fix would touch public plugin SDK and hook contracts, so it should be additive and preserve existing `Timestamp` as the source-channel timestamp.
- A per-message receipt timestamp is volatile; adding it to stable system-prefix metadata could hurt prompt-cache stability. It should be rendered in user-role inbound context or another intentionally dynamic surface.
