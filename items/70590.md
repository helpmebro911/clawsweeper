## Summary

Keep open. Current main already has a Control UI i18n system, persisted locale selection, lazy-loaded locale bundles, and changelog evidence of earlier multilingual work, but the reported problem is still observable: multiple current Control UI surfaces render user-facing English literals outside the translation map. This is a real core UI/i18n cleanup request, not a ClawHub/plugin-bound item and not a duplicate of the unrelated provided related reports.

## Best Possible Solution

Keep this issue open as a core Control UI i18n cleanup. The practical path is to inventory user-facing raw English in `ui/src/ui/**`, add source strings to `ui/src/i18n/locales/en.ts`, replace visible literals with `t()` calls, regenerate non-English locale bundles with the existing Control UI i18n pipeline, and add or extend a lightweight raw-string guard so new hard-coded UI copy is caught before release.
