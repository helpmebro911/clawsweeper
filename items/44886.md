## Summary

Keep this issue open. Current main still accepts Telegram `direct.<peer>.tools` and uses the same direct config for DM prompts/access, but the mounted tool-policy enforcement path remains group/channel-oriented and does not resolve Telegram `direct` policy as a per-DM tool policy.

## Best Possible Solution

Keep this issue open. The best maintainer path is to adapt Telegram's legacy `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]` surface onto the shared direct-session tool-policy resolver, enforce both `tools` and `toolsBySender` for normal Telegram DM runs and native-command target sessions, preserve explicit-over-wildcard and account-scoped precedence, and add focused Telegram regression tests for mounted tool inventory, sender-scoped overrides, and DM topic/thread fallback.
