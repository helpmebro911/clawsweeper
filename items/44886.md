## Summary

Current main still accepts Telegram direct-message tool policy in schema/types and uses Telegram direct config for DM routing, access, topic, and prompt behavior, but the mounted tool-policy path remains group/channel-oriented. Direct Telegram session keys are still not parsed into tool-policy candidates, and Telegram's plugin resolver delegates to group policy under `channels.telegram.groups`, not `channels.telegram.accounts.<account>.direct`. The related #53530 is useful context but is still an open, unmerged PR in the provided context and was explicitly described as not adapting Telegram's legacy `direct.*` surface by itself.

## Best Possible Solution

Keep this issue open. The best maintainer path is to adapt Telegram's legacy `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]` surface onto the shared direct-session tool-policy resolver, preserve explicit-over-wildcard and account-scoped precedence, apply `tools` and `toolsBySender` through the mounted agent tool path for normal DM and native-command runs, and add focused Telegram regression tests. #53530 can remain related shared resolver work, but it is not sufficient closure until current main enforces Telegram `direct.*` tool policy.

## Risks / Open Questions

- Authorization-sensitive fix: Telegram DM tool policy should be derived from trusted Telegram account/session/sender context, not caller-supplied group IDs or untrusted payload fields.
- Regression coverage should include explicit peer versus `direct["*"]` precedence, account-scoped lookup, `toolsBySender`, native command target sessions, mounted tool inventory, and thread/parent fallback behavior.
- The related #53530 may provide shared resolver pieces, but closing this issue before Telegram `direct.*` enforcement is wired and tested on main would leave the reported user-visible policy gap open.
