## Summary

Keep #44886 open. Current main still accepts Telegram `direct.<peer>.tools` in schema/types, and Telegram direct config is used for DM prompt/access behavior, but the runtime tool-policy enforcement path still resolves only group/channel policy. The linked #53530 is a useful related PR for the shared direct-session resolver, but the provided issue discussion explicitly says it does not adapt Telegram's legacy `direct.*` surface, so it is not enough to close this Telegram-specific report.

## Best Possible Solution

Keep #44886 open and implement the Telegram adaptation on top of the shared direct-session policy work: parse trusted Telegram direct-session/account context, resolve `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]`, apply `tools` and `toolsBySender` with explicit-over-wildcard precedence, and add regression coverage through the agent tool-policy path for Telegram DM runs. #53530 can be reused or linked as related shared resolver work, but it should not be treated as the canonical closure until Telegram `direct.*` enforcement is covered.
