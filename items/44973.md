## Summary

Keep #44973 open. Current main still has the same main-session cron path the PR targets: the cron service enqueues the raw systemEvent with a plain `cron:<jobId>` context key, heartbeat prompt resolution still wraps cron events into the scheduled-reminder prompt, and reply setup still drains the same queued event into `System:` lines. The PR is a focused bug fix with a linked open bug (#44922) and useful discussion tracing the peek/drain duplication path, so it is not obsolete or safely closeable.

## Best Possible Solution

Keep the PR open or land an equivalent fix on main: mark main-session injected cron systemEvents, or otherwise teach `resolveHeartbeatRunPrompt` not to wrap cron events that will already be drained into the active main-session system-event block. Add a regression test for `sessionTarget: "main"` + `payload.kind: "systemEvent"` + cron wake with `delivery.mode: "none"`, and update existing timer expectations if the context key changes.
