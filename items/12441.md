## Summary

Keep #12441 open. Current main still supports bearer credentials for authenticated Control UI HTTP reads, but the browser WebSocket handshake is authenticated from explicit `connect.params.auth` state, Tailscale identity headers, or trusted-proxy identity mode. A reverse-proxy-injected `Authorization` or `X-OpenClaw-Token` header is not mapped into shared-secret Control UI WebSocket auth, and the bootstrap config endpoint does not issue a browser session credential for the WebSocket.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best path is either to validate `Authorization: Bearer` and/or `X-OpenClaw-Token` on the Control UI WebSocket upgrade and feed it into the existing shared-secret auth path after the normal origin/proxy checks, or to validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. Add focused gateway WS/auth tests, Control UI bootstrap tests, and docs updates under the public Control UI/dashboard pages.

## Risks / Open Questions

- Closing this would leave shared-secret reverse-proxy deployments dependent on URL fragment/sessionStorage/manual entry, which is the user problem the issue reports.
- A fix needs explicit security design for proxy trust, browser origin checks, failed-auth rate limiting, credential lifetime, and whether the token is accepted on WS upgrade or exchanged through a same-origin bootstrap/session endpoint.
