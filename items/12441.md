## Summary

Keep #12441 open. Current main still authenticates Control UI WebSockets from `connect.params.auth`, Tailscale Serve identity, or trusted-proxy identity, but it does not convert a reverse-proxy-injected `Authorization: Bearer ...` or `X-OpenClaw-Token` header into shared-secret WebSocket auth. Related token persistence/dashboard URL work is implemented, but that does not solve the reported header-bootstrap flow.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best path is either to validate `Authorization: Bearer` and/or `X-OpenClaw-Token` on the Control UI WebSocket upgrade and feed it into the existing shared-secret auth path after normal origin/proxy checks, or to validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. Add focused gateway WS/auth tests, Control UI bootstrap tests, and public dashboard/Control UI docs.

## Risks / Open Questions

- Closing this would leave shared-secret reverse-proxy deployments dependent on URL fragments, sessionStorage/manual entry, or a different trusted-proxy identity mode rather than the requested injected bearer-token flow.
- Accepting bearer headers on the WebSocket upgrade needs a careful security design around browser origin checks, trusted proxy boundaries, failed-auth rate limiting, credential lifetime, and device-pairing interaction.
- Reusing `X-OpenClaw-Token` outside hooks would need explicit Control UI/Gateway documentation so a hook-specific header does not become an undocumented cross-surface auth contract.
