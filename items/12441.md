## Summary

Keep open. Current main has partial bearer support for Control UI HTTP/bootstrap/read requests, but it still does not satisfy the requested reverse-proxy flow because the browser WebSocket shared-secret auth is still derived from UI-stored/manual credentials in `connect.params.auth`, not from an injected `Authorization` or `X-OpenClaw-Token` header, and the bootstrap payload does not issue a session credential.

## Best Possible Solution

Keep tracking this in core as a Gateway/Control UI auth design issue. The best fix is to either validate `Authorization: Bearer ...` / `X-OpenClaw-Token` on the WebSocket upgrade and map it into the existing shared-secret auth path, or validate the header on an initial same-origin HTTP request and issue a short-lived UI session/bootstrap credential for the browser WebSocket. Add focused server/UI tests and update the public Control UI/dashboard docs.
