## Summary

Keep open. Current main still lacks a remote node-host guard for `openclaw models auth login`, the linked bug #42291 remains open, and this PR is an unmerged implementation candidate rather than obsolete cleanup. The PR is focused and security-improving in intent, but it needs a rebase and maintainer review of guard scope before landing.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review/land a focused guard in `src/commands/models/auth.ts` with regression coverage in `src/commands/models/auth.test.ts`. The implementation should prevent node-local OAuth credential writes when `~/.openclaw/node.json` points at a non-loopback gateway, preferably using the existing loopback-host semantics and with an explicit maintainer decision on whether API-key/custom login methods should also be blocked. Once merged, #42291 can close as implemented.

## Risks / Open Questions

- The PR is currently `mergeable: false` and its diff uses older import paths than current main, so it likely needs a rebase or conflict resolution before review can land it.
- The proposed guard runs before auth method selection, while provider auth kinds include `oauth`, `api_key`, `token`, `device_code`, and `custom`; maintainers should confirm whether blocking all `models auth login` methods on remote node hosts is intentional or should be scoped to OAuth/credential-writing methods.
- The proposed local loopback helper recognizes only empty host, `localhost`, `127.0.0.1`, and `::1`; current `src/gateway/net.ts:isLoopbackHost` handles broader local forms such as `127.x`, bracketed IPv6, and IPv4-mapped IPv6, so the patch should probably reuse the existing helper or match its semantics.
