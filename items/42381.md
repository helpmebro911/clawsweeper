## Summary

Keep open. Current `main` does not implement the remote-node-host OAuth guard from PR #42381, and the underlying bug remains plausible: `models auth login` still writes auth profiles to the local agent state even though docs and node-host status make clear that remote node hosts target a separate gateway host.

## Evidence

- **Login path still lacks node-host guard:** In current `main`, `modelsAuthLoginCommand` only checks for interactive TTY and then proceeds into auth-context resolution; there is no `loadNodeHostConfig()` check or any remote-node-host rejection before provider auth runs.
  - file: [src/commands/models/auth.ts:584](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/models/auth.ts#L584)

- **OAuth results still persist to local agent storage:** `persistProviderAuthResult()` writes returned credentials with `upsertAuthProfile({ ..., agentDir })`, so a login on a node host still targets that machine's local auth store.
  - file: [src/commands/models/auth.ts:239](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/models/auth.ts#L239)
