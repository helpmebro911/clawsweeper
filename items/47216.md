## Summary

Keep open. Current main still lacks the PR's direct-webchat sender guard: direct webchat conversation metadata is suppressed, but `buildInboundUserContextPrefix` still emits `Sender (untrusted metadata)` whenever sender identity resolves to a label. The TUI/gateway path still provides transport client identity fields, and the PR's regression test is not present on main.

## Best Possible Solution

Keep the PR open or land an equivalent scoped fix on main: add a sender-info guard for direct `webchat` contexts in `buildInboundUserContextPrefix`, preserve sender metadata for external-channel direct and group chats, and add the regression test for `openclaw-tui` / `gateway-client` transport identities.
