## Summary

Keep open. The provided repro is a concrete gateway/session stability bug, and current main still has no clear session-level queue/reject path for parallel same-session agent/exec activity. Main has adjacent mitigations such as transcript/session file locks and background exec update suppression, but those do not prove that two concurrent Codex exec subprocesses from one Telegram session now fail gracefully or cannot destabilize the session.

## Evidence

- **Issue discussion identifies actionable same-session concurrency failure:** The issue body reports a reproducible Telegram DM flow where two parallel `codex exec --full-auto` runs make the assistant unresponsive until gateway restart; the only comment points to missing session-level state isolation/locking and suggests queueing or isolated sessions as the desired mitigation.
- **Gateway accepts and tracks active runs without rejecting same-session concurrency:** `agent` handling registers a new abort controller and returns an accepted run response. Earlier in the same handler it explicitly iterates other active runs in the same session to add tool-event recipients, which indicates concurrent same-session runs can exist rather than being rejected or queued at this boundary.
