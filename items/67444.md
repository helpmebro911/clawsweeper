## Summary

Keep this PR open. Current main still suppresses `ANNOUNCE_SKIP` before the completion-delivery call, so the PR targets a real remaining behavior difference. The submitted branch is not ready as-is because the proposed fallthrough can keep the literal `ANNOUNCE_SKIP` token as completion content, and the semantic change should be covered by stronger tests and docs/product review.

## Best Possible Solution

Keep the PR open for maintainer review and require a follow-up patch before merge: allow `ANNOUNCE_SKIP` to skip only the soft announce step when completion delivery is required, clear the control token before building completion content, preserve `NO_REPLY` suppression, strengthen tests around the actual delivery call, and update the FAQ/docs if maintainers accept the semantic change.
