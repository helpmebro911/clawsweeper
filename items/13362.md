## Summary

Keep #13362 open. Current main still lacks a doctor-specific `--dry-run` flag or equivalent no-write validation mode, and the current doctor flow still has confirmed/repair config writes plus other side-effecting repair paths. The related global agent dry-run issue #41418 is complementary and does not replace this doctor/config validation request.

## Best Possible Solution

Keep #13362 open until OpenClaw implements or explicitly rejects a doctor-specific dry-run mode. The likely path is to add `openclaw doctor --dry-run`, thread `dryRun` through `DoctorOptions` and every doctor contribution, make all repair/write paths emit planned changes without mutating config/state/services/runtime deps, define exit-code behavior for findings, update `docs/cli/doctor.md` and public docs, and add focused tests proving dry-run does not mutate config, cron store, service state, or bundled plugin dependency state.

## Risks / Open Questions

- A narrow CLI-only flag would be incomplete unless all doctor side effects respect the same no-write contract: config writes, cron/state migrations, service repair, shell/profile changes, plugin registry/runtime dependency repairs, and channel/plugin-owned doctor repairs.
- The requested non-zero exit behavior for dry-run findings still needs explicit maintainer/product semantics so scripts can distinguish validation findings from command failures.
