## Summary

Keep open. Current main still has the reported macOS false `No GUI detected` path: `resolveBrowserOpenCommand()` short-circuits on `SSH_*` plus no display for every non-Windows platform before macOS can use `/usr/bin/open`. The issue also has an open, unmerged implementation candidate, PR #67115, using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review PR #67115, or replace it with an equivalent narrow fix in `src/infra/browser-open.ts` that narrows the SSH/no-display early return to Linux or otherwise excludes `darwin`. Add regression tests for macOS resolving to `open` with lingering `SSH_*` env vars and no display, plus Linux preserving `ssh-no-display`.

## Evidence

- **Current main still short-circuits macOS:** At current main, `resolveBrowserOpenCommand()` treats any `SSH_CLIENT`, `SSH_TTY`, or `SSH_CONNECTION` plus no `DISPLAY`/`WAYLAND_DISPLAY` as `ssh-no-display` whenever `platform !== "win32"`. On macOS, normal shells commonly have no display env, so this returns before the later `platform === "darwin"` branch can probe and use `open`.
