## Summary

Keep open. Current main still does not implement opt-in native raw audio/video forwarding: media config has no `nativeForwarding`, media understanding still produces text outputs, and the embedded runner only forwards `imageResult.images` to `activeSession.prompt`. The PR is a plausible core media/prompt feature, but it is not merge-ready as written because the review discussion identified a session-key cleanup leak and the pinned `@mariozechner/pi-ai` media content contract is still image-only.

## Best Possible Solution

Keep the PR open for maintainer review, but require a revision before merge: land or verify the pi-ai audio/video content contract, replace the force-cast with a typed or guarded media block path, thread the resolved `agentSessionKey` into native media capture/cleanup, and add focused tests plus config/docs for opt-in default-off behavior, Telegram voice/video forwarding, and native-command redirect cleanup.

## Risks / Open Questions

- Merging the PR as-is could leak up-to-25 MiB native media blocks in native-command redirect scenarios unless the resolved session key is threaded consistently through capture, consume, and discard.
- The current pinned pi-ai type/runtime contract is image-only, so native audio/video content blocks need upstream support or an explicit runtime guard before OpenClaw can rely on them safely.
- The feature touches core prompt assembly, media understanding, config, and provider compatibility; it needs focused tests and docs rather than closure as stale or external plugin work.
