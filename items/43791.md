## Summary

Keep this PR open. Current main still does not preserve full embedded tool-result ReplyPayload objects in final run payloads; it only has a narrower orphaned-media fallback. The PR remains a plausible implementation candidate, with remaining maintainer review needed around timeout semantics, dedupe with the existing media fallback, and the returned payload contract.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase onto current main, preserve full ReplyPayloads only for non-streaming callers, reconcile the existing orphaned-media fallback, settle timeout behavior with regression coverage, document or deliberately type the embedded run result payload contract, and fix changelog attribution before merge.

## Risks / Open Questions

- Buffered tool-result payloads can alter timeout semantics: a timed-out run with no assistant reply may look like a partial success if a tool payload is returned before the timeout guard.
- The PR needs reconciliation with current main's orphaned tool-media fallback so media/audio payloads are not duplicated, reordered, or attached twice.
- Returning full ReplyPayload objects widens the embedded run result contract; maintainers should explicitly confirm how metadata, interactive payloads, sensitiveMedia, and channelData are expected to flow for non-streaming callers.
- The PR changelog entry in the live diff lacks the required Thanks @author attribution before merge.
