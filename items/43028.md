## Summary

Current main partially mitigates the reported cron pain with timeouts, recurring error backoff, failure alerts, session retention pruning, and JSONL run history, but the requested recurring-job auto-pause circuit breaker, immediate remove-time cleanup flag, and `.learnings/changelog.md` recording are still not implemented.

## Best Possible Solution

Keep open and narrow the work into cron-owned follow-ups: add a backward-compatible recurring execution circuit breaker in `src/cron/service/timer.ts` with config/types/docs, reuse existing failure-alert delivery for notifications, decide whether `cron.remove` should gain an immediate cleanup option across gateway/CLI/UI, and explicitly document whether `cron.runs` is the supported audit surface instead of `.learnings/changelog.md`.

## Evidence

- **Issue context:** The issue asks for three concrete OpenClaw cron changes: auto-pause/circuit breaker after repeated timeouts, `openclaw cron rm <id> --cleanup`, and automatic `.learnings/changelog.md` entries. A commenter also tied the 2026.3.8 report to real cron execution bugs, so this remains actionable cron work rather than an incoherent or stale report.
- **Recurring errors back off but do not auto-pause:** `applyJobResult` increments `consecutiveErrors` and can emit failure alerts, but for non-`at` recurring jobs it only applies exponential backoff and computes `nextRunAtMs`; it does not set `job.enabled = false` after repeated execution failures/timeouts.
  - file: [src/cron/service/timer.ts:433](https://github.com/openclaw/openclaw/blob/aac83e00cfe72e5395b4cdfe77ead1c534fe7ef8/src/cron/service/timer.ts#L433)

  - file: [src/config/types.cron.ts:3](https://github.com/openclaw/openclaw/blob/aac83e00cfe72e5395b4cdfe77ead1c534fe7ef8/src/config/types.cron.ts#L3)
