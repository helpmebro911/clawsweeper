## Summary

Current main still does not implement the reported per-session or per-turn memory-lancedb autoCapture behavior. Auto-capture remains an agent_end-only path that processes only successful runs, user-role messages, hard-coded trigger matches, length limits, prompt-injection filters, duplicate checks, and at most three captured texts. The adjacent doctor complaint also remains structurally valid for a slot-only memory-lancedb config because doctor asks for an active memory runtime/capability, while memory-lancedb registers tools/hooks/CLI/service but no memory capability/runtime. Later memory-lancedb history only fixed float embedding encoding, not capture firing semantics.

## Best Possible Solution

Keep this open. Maintainers should define the intended memory-lancedb autoCapture contract, then either implement broader session/turn capture with tests and poisoning safeguards or document the current heuristic-only behavior clearly. The doctor path should also treat a valid plugins.slots.memory = "memory-lancedb" setup correctly, or memory-lancedb should register the memory capability/runtime if it is meant to satisfy the memory-search runtime contract.
