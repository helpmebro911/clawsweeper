## Summary

Keep open. The report is fresh and underspecified, but it maps to core-owned session lifecycle behavior: current main intentionally starts a new session after the default 4:00 AM local daily reset boundary, which matches the reported 11 PM to 5 AM gap. That may be expected behavior, but the issue still needs maintainer triage because the reporter provided no config, session timestamps, logs, WeChat plugin version, or session key evidence to distinguish intended reset behavior from a WeChat routing/session persistence regression.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, the affected session key, redacted sessions.json timestamps (sessionStartedAt, lastInteractionAt, updatedAt), and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, improve the first-turn reset notice or docs/config guidance; if not, add a focused regression test around WeChat direct-session reuse or session-key stability.
