## Summary

Keep open. Current main intentionally starts a new session after the default 4:00 AM local daily reset boundary, which matches the reported 11 PM to 5 AM WeChat gap, but the report is fresh and still maps to core-owned session lifecycle behavior. There is not enough evidence to close it as fixed, unreproducible, stale, external, or duplicate.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, affected session key, redacted sessions.json timestamps, and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, improve first-turn reset notice or docs/config guidance; if not, add a focused regression test for WeChat/direct session-key stability or channel session reuse.
