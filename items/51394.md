## Summary

Keep open. Current `main` has partial browser-network support in the bundled browser plugin (`/requests` listing and `/response/body` lookup), but the agent-facing browser tool still does not expose any `network` act kind or `requests`/`responsebody` action. The issue's requested browser-tool surface is not implemented on `main`.

## Evidence

- **Browser tool schema lacks any network action/kind:** `extensions/browser/src/browser-tool.schema.ts` only allows act kinds `click|type|press|hover|drag|select|fill|resize|wait|evaluate|close` and top-level actions `status|start|stop|profiles|tabs|open|focus|close|snapshot|screenshot|navigate|console|pdf|upload|dialog|act`. There is no `network` act kind and no `requests` or `responsebody` tool action.
  - file: [extensions/browser/src/browser-tool.schema.ts:4](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/browser/src/browser-tool.schema.ts#L4)
  - command: `nl -ba extensions/browser/src/browser-tool.schema.ts | sed -n '1,140p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Browser tool dispatcher has no network/debug request handler:** `extensions/browser/src/browser-tool.ts` dispatches `console`, `pdf`, `upload`, `dialog`, and `act`, but there is no case for `requests`, `responsebody`, or a network-interception command path.
  - file: [extensions/browser/src/browser-tool.ts:653](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/browser/src/browser-tool.ts#L653)
  - command: `nl -ba extensions/browser/src/browser-tool.ts | sed -n '620,730p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Bundled browser runtime already captures network activity internally:** `extensions/browser/src/browser/pw-session.ts` records Playwright `page.on("request")`, `page.on("response")`, and `page.on("requestfailed")` events into per-page request state, so the underlying browser plugin has request-observation support.
  - file: [extensions/browser/src/browser/pw-session.ts:350](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/browser/src/browser/pw-session.ts#L350)
  - command: `nl -ba extensions/browser/src/browser/pw-session.ts | sed -n '340,390p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Network inspection exists in control API/CLI, not in agent tool:** `extensions/browser/src/browser/routes/agent.debug.ts` exposes `GET /requests`, `extensions/browser/src/browser/routes/agent.act.ts` exposes `POST /response/body`, and `docs/tools/browser-control.md` documents `GET /requests` plus `openclaw browser requests` / `openclaw browser responsebody`. That shows partial implementation, but on the control API/CLI surface rather than the browser agent tool requested in the issue.

## Risks / Open Questions

- The issue asks for CDP Network-domain exposure and interception; current `main` only provides passive request capture plus response-body retrieval, not an agent-tool API for interception/blocking/modifying requests.
- If maintainers view the existing CLI/control API as sufficient, the issue could be reframed as an agent-tool parity request rather than a new browser-runtime capability request.
