## Summary

Keep #17840 open. Current main still treats channel reactions as passive reaction notification system events, not as an opt-in wake/agent-turn source. The issue has concrete Slack, Telegram, and Discord use cases, a newer overlapping #56708 that is less canonical than this older richer issue, and a recent maintainer-facing Codex review comment explicitly keeping this item open.

## Best Possible Solution

Keep #17840 open as the canonical product request. Implement an opt-in per-channel/plugin wake mode such as reactionTriggersTurn: "off" | "own" | "all" | "allowlist"; after existing reaction authorization and notification filtering, enqueue the reaction system event and request a scoped heartbeat/agent turn for that session. Update channel schemas, docs, and focused reaction tests while preserving the passive default. Treat #56708 as an overlapping duplicate candidate, not a superseding canonical tracker.

## Risks / Open Questions

- Default reaction behavior should remain passive/off to avoid waking agents on ordinary group-chat reaction noise.
- Any implementation must run after existing reactionNotifications filtering, allowlists, and channel access checks so reactions do not bypass authorization.
- Session/thread routing needs channel-specific care, especially Telegram reactions where the API does not provide message_thread_id.
