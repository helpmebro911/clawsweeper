## Summary

Keep #17840 open. Current main still lacks an opt-in reaction-triggered agent-turn mode for the affected channel paths; reactions are generally queued as passive system events and drained on a later reply/heartbeat, while Feishu has only a channel-specific synthetic-turn implementation that does not solve the broader Slack/Telegram/Discord request. The richer older issue remains the canonical tracker over overlapping #56708, and the latest issue discussion already includes a maintainer-facing keep-open review.

## Best Possible Solution

Keep #17840 open as the canonical product request. Implement a documented opt-in channel/plugin wake mode such as reactionTriggersTurn: "off" | "own" | "all" | "allowlist"; after existing reaction authorization and notification filtering, enqueue the reaction event and request a scoped heartbeat/agent turn for that session. Update channel schemas, public docs, and focused Slack/Telegram/Discord reaction tests while preserving the passive default. Treat #56708 as an overlapping duplicate candidate rather than the canonical tracker.

## Risks / Open Questions

- Default behavior should remain passive/off to avoid waking agents for ordinary group-chat reaction noise.
- Any implementation must run only after existing reactionNotifications filtering, allowlists, channel access checks, and sender authorization so reactions cannot bypass current channel security.
- Session/thread routing needs channel-specific handling, especially Telegram where reaction updates do not include message_thread_id.
