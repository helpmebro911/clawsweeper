## Summary

Keep this PR open. Current main still drops `accountId` before `/hooks/agent` dispatch, while the cron delivery layer already supports `delivery.accountId`. The PR is paired with open #43866 from the same author and remains a plausible implementation candidate, but it needs rebase/adaptation and maintainer review before landing.

## Best Possible Solution

Keep this PR open. Best path is to rebase it onto current main, implement the focused hook-side fix (`accountId` normalization, propagation into `CronDelivery`, replay-key scoping, observable normalization failure, and regression tests), preserve the current lazy cron-runner import boundary, adapt or remove the stale cron tool-policy hunk against current `deliveryMode`/`forceMessageTool` behavior, update public `/hooks/agent` docs, and fix changelog attribution before maintainer review.
