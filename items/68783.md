## Summary

Keep this PR open. The linked cron delivery regression is still plausible on current main because timer/startup executions still pass the stored job object through execution without a delivery snapshot or equivalent immutable merge. However, the PR as described is not ready to merge: restoring the whole pre-run delivery object conflicts with existing behavior that intentionally preserves persisted delivery writebacks, and current tests assert that Telegram target normalization/writeback survives manual runs.

## Best Possible Solution

Keep the PR open and request changes. The fix should prevent in-flight execution from rewriting user-selected delivery.channel/mode to last without replacing the entire post-reload delivery object. A field-wise reconcile, execution-only clone, or narrower immutable-field snapshot should preserve existing target/account/thread writebacks. Regression coverage should include both the reported channel-preservation case and the existing Telegram writeback behavior for manual, timer-driven, and startup catch-up paths.
