## Summary

Keep open. Current main still has the same Control UI add-attachment shape described in the report: image/file inputs are read with FileReader.readAsDataURL(), converted into ChatAttachment objects carrying dataUrl, and assigned into the Lit @state chatAttachments array. The related gateway/chat.send fixes and tests prove later send-side attachment normalization, but they do not prove the immediate browser-side add-image path no longer throws. The current test coverage still exercises a small PDF picker path, not a large image paste/drop/select regression through OpenClawApp state.

## Best Possible Solution

Keep this as a Control UI attachment-add bug. Add a focused browser/UI regression for selecting, pasting, and dropping a large image through OpenClawApp, then either close with runtime evidence if current main no longer reproduces or move image payloads out of chatAttachments reactive state, for example by storing Blob/object URLs or id-keyed raw payloads and converting to base64 only when chat.send is invoked.
