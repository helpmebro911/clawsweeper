## Summary

Keep open. Current main still reads pasted, selected, and dropped Control UI chat attachments with FileReader.readAsDataURL(), stores the resulting dataUrl on ChatAttachment, and assigns that array into Lit state before any chat.send serialization runs. The recent shipped attachment work added non-video file support and downstream send/gateway handling, but it does not move pending image payloads out of the browser-side add/preview state path described here.

## Best Possible Solution

Keep this open as a Control UI Webchat attachment-add bug. The likely fix belongs in ui/src/ui: move large pending attachment payloads out of Lit @state, store Blob/object URLs or id-keyed raw payloads in a non-reactive store, convert to base64 only when chat.send is built, and add focused regression coverage for selecting, pasting, and dropping a large image through OpenClawApp.
