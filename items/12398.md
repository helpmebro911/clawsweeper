## Summary

Keep #12398 open. Current main still lacks a cron `postconditions` contract, deterministic post-run checker execution, persisted checker state/output, checker-failure retry prompt injection, or a plugin/API seam that can fail an otherwise successful isolated cron run and drive cron-owned retry/escalation. The existing implementation covers execution status, delivery status, transient-error backoff, and failure alerts only.

## Best Possible Solution

Keep #12398 open for maintainer/product implementation. The likely core path is an additive cron job contract plus gateway schema fields for postconditions, deterministic checker execution after isolated agent completion and before applyJobResult, persisted checker status/output in cron state and run logs, retry prompt injection with checker output, and integration with existing retry/failure-alert delivery. If maintainers prefer a plugin route, the prerequisite is a first-class cron post-run hook/API that can fail a cron run and schedule cron-owned retry/escalation.
