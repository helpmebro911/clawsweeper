## Summary

Keep #12398 open. Current main still lacks the requested cron postconditions contract: there is no `postconditions` field in cron types or Gateway schemas, no persisted postcondition state or run-log fields, no deterministic checker execution between isolated agent completion and cron state update, and no plugin hook that can fail a cron run and drive cron-owned retry/escalation. The existing retry path only handles execution errors such as transient provider failures, not an `ok` agent turn followed by a deterministic verifier failure.

## Best Possible Solution

Keep #12398 open for maintainer/product implementation. The best path is an additive cron contract and Gateway schema field for postconditions, deterministic checker execution after isolated agent completion and before `applyJobResult`, persisted postcondition state/run-log evidence, retry prompt injection using checker output, and integration with existing cron retry/failure-alert delivery. If maintainers prefer a plugin route, the missing prerequisite is a first-class cron post-run hook/API that can fail a cron run and schedule cron-owned retry/escalation.
