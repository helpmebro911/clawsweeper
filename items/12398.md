## Summary

Keep #12398 open. Current main still lacks the requested first-class cron postcondition contract: no `postconditions` field in cron job types or gateway schemas, no durable postcondition result/retry state in cron state or run logs, and no deterministic verifier step between isolated agent completion and `applyJobResult()`. Existing retry and failure-alert behavior remains error-status based, so a clean but incomplete `agentTurn` run cannot be converted into cron-owned retry or escalation by a verifier.

## Best Possible Solution

Keep #12398 open for maintainer/product design and implementation. The likely path is an additive cron job contract plus gateway/CLI/schema/docs/UI updates, deterministic verifier execution after isolated agent completion and before `applyJobResult()`, durable verifier state and run-log fields, bounded retry prompt injection with checker output, and integration with cron-owned retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook/API that can fail a run and request cron-owned retry/escalation.
