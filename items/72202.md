## Summary

Keep #72202 open. The report is recent, specific, and not covered by a protected/maintainer auto-keep rule, but I did not find enough current-main evidence to close it. Current main has intended OpenAI image routing and config-env plumbing, yet the exact reported Windows case, `OPENAI_API_KEY` supplied only through `openclaw.json` env with no `openai-codex` profile, is not directly covered by the inspected tests or history. The linked #70703 shipped the inverse capability, Codex OAuth image generation when no API key is available, and does not supersede this API-key route-selection bug.

## Best Possible Solution

Keep the issue open and add a targeted regression around `image_generate` with `agents.defaults.imageGenerationModel.primary = "openai/gpt-image-2"`, `env.OPENAI_API_KEY` or `env.vars.OPENAI_API_KEY`, no `openai-codex` profile, and a Windows-compatible gateway/tool path. The expected assertion should be direct OpenAI Images API routing (`/v1/images/generations`) and no Codex Responses call. If that fails, fix config env propagation or OpenAI image auth precedence; if it passes, ask the reporter for sanitized auth-store/config shape and gateway logs to identify an unexpected Codex credential source. Track provider-config persistence separately under the related provider-config overwrite issue rather than closing this one as #70703.
