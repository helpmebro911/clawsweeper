## Summary

Keep open. Current main has separate Control UI embed and browser screenshot/snapshot paths, and the canvas tool now tries a default canvas-capable node, but `canvas(action="present")` and `canvas(action="navigate")` are still node-backed commands that return JSON success rather than a chat-visible webpage preview. The reported `node required` class is still reachable when no default canvas node can be selected, so this issue still needs maintainer triage of the intended product contract.

## Best Possible Solution

Keep this issue open and decide the intended product contract. If `canvas present`/`canvas navigate` should visibly confirm state in chat, add a preview-producing result, automatic optional snapshot, or explicit Control UI status artifact with regression coverage. If they are only node controls, tighten tool descriptions, docs, and error guidance so no-node cases direct users to pair/select a canvas node, use `canvas snapshot` for visual confirmation, use the `browser` screenshot/snapshot flow for webpages, or use `[embed ...]` for Control UI inline rendering.
