## Summary

Current `main` already has native MCP server support in the shipped codebase: an `openclaw mcp` CLI, config-backed `mcp.servers`, transport handling for stdio/SSE/streamable HTTP, and runtime discovery/invocation of MCP tools for agent sessions. The implementation landed on `main` after the latest tagged release, so it is implemented on `main` but not in `v2026.4.22`.

## Risks / Open Questions

- The strongest evidence is for MCP server registration, transport support, and MCP tool discovery/invocation; I did not find equivalent explicit support for every MCP primitive mentioned in the body (for example prompt/resource APIs).
- `VISION.md` still states a preference for the `mcporter` bridge model over a broader first-class core MCP runtime, so future MCP expansion requests may still need narrower follow-up issues.

## Close Comment

Closing as implemented on current `main` based on a Codex review. OpenClaw now has an `openclaw mcp` surface for both serving OpenClaw over MCP and managing outbound MCP server definitions (`docs/cli/mcp.md:10`, `src/cli/mcp-cli.ts:26`). Current main also supports configured `mcp.servers` with stdio plus remote HTTP/SSE transport settings (`docs/cli/mcp.md:360`, `src/agents/mcp-transport-config.ts:95`), and the agent runtime connects to configured MCP servers, catalogs their tools, and invokes them through the MCP client path (`src/agents/pi-bundle-mcp-runtime.ts:163`). The implementation was introduced by commit `bff212822cf73c7a37fa7a77d9a725c2d8895b1e`, which is on `main` but not in the latest tagged release `v2026.4.22`, so there is no shipped release tag for it yet. If you meant broader MCP resource/prompt coverage beyond tool-server support, please open a narrower follow-up issue for that gap.
