## Summary

Keep open. Current main only partially addresses the report: Slack status-reaction dispatch suppresses `already_reacted` and `no_reaction`, but the shared Slack reaction helpers and the agent-facing Slack `react` action still call the raw Slack reactions API without idempotent guards. The requested no-op behavior for duplicate reaction adds/removes is therefore not implemented broadly enough to close #50733.

## Evidence

- **Shared Slack reaction helpers still propagate Slack API errors:** `reactSlackMessage` directly awaits `client.reactions.add`, and `removeSlackReaction` directly awaits `client.reactions.remove`; neither helper catches `already_reacted` or `no_reaction`.
  - file: [extensions/slack/src/actions.ts:95](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/slack/src/actions.ts#L95)
  - command: `nl -ba extensions/slack/src/actions.ts`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Agent Slack reaction action still uses the unguarded helpers:** The `react` tool path awaits `slackActionRuntime.reactSlackMessage` and `slackActionRuntime.removeSlackReaction` without catching Slack idempotency errors, so duplicate reaction failures can still escape through the agent action surface.
  - file: [extensions/slack/src/action-runtime.ts:195](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/slack/src/action-runtime.ts#L195)
