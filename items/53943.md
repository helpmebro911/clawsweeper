## Summary

Current main still does not implement Slack `message` read by `messageId`. The Slack read path forwards `threadId` but never accepts, forwards, or filters by exact Slack message timestamp, so the reported thread-reply lookup gap remains open.

## Best Possible Solution

Keep this issue open. The focused fix should stay in the Slack plugin read path: accept `messageId` for Slack `read`, pass it through the dispatcher and runtime handler, extend `readSlackMessages` to return/filter the exact Slack timestamp for both `messageId + threadId` and messageId-only reads, and add regression tests for both cases. If Slack URL parsing feeds this tool path, cover `pMESSAGE_TS?thread_ts=THREAD_TS` normalization in the same change. No broad core API change is clearly required from the current evidence.
