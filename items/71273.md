## Summary

Keep open. Current main does not contain high-confidence evidence that the reported Kimi Code repeated tool-call loop is fixed. The issue's proposed root cause is partly contradicted by source and tests because the Kimi wrapper only flips `stopReason` after it has parsed tagged tool-call markup, and normal assistant text is already left unchanged. But the bundled Kimi provider still uses that wrapper, PR #71274 is closed unmerged in the provided GitHub context, and the generic repeated-tool loop guard is disabled by default, so this remains a valid issue for a focused repro/regression rather than a conservative close.

## Best Possible Solution

Keep the issue open and narrow it with a sanitized transcript or a regression test that models the post-tool-result Kimi turn. If Kimi returns ordinary final text, the existing wrapper should preserve `stop`; if Kimi re-emits identical tagged tool calls after identical tool results, the fix likely belongs in Kimi-specific replay/transcript handling or an always-on safety guard for repeated identical tool calls, not in the redundant `hasToolCalls` check from PR #71274 alone.
