## Summary

Keep this PR open. Current main still does not support per-binding workspace overrides for normal route bindings, and the PR remains a plausible core routing/config feature. It is not ready to close or merge as-is because the PR-head implementation re-derives the workspace from only part of the binding match instead of consuming the already resolved route binding.

## Best Possible Solution

Keep this PR open as the implementation candidate. The best path is to revise it so `AgentRouteBinding` and the strict schema expose optional `workspace`, `resolveAgentRoute` carries the matched binding workspace through `ResolvedAgentRoute`, and reply execution consumes that resolved value instead of re-scanning bindings. Add docs/config-help/generated-schema updates plus tests for peer, account, team, guild/role scoped bindings, uppercase peer IDs, case-insensitive agent IDs, and fallback to the agent-level workspace.

## Risks / Open Questions

- PR-head workspace lookup can select the wrong workspace in multi-account or multi-team/channel cases because it does not tie the override to the already matched binding scope.
- PR-head workspace lookup can silently drop an override when route resolution accepts a differently cased agent ID but `resolveBindingWorkspace` compares `binding.agentId` exactly.
- The config/help/docs/generated schema surface and targeted regression tests are still missing for a new route-binding config key.
