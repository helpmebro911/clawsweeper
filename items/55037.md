## Summary

Keep this PR open. Current main still only supports agent-level workspaces for normal route bindings, and this PR is a plausible core routing/config feature. It should not be closed as implemented or obsolete, but it needs revision before merge: PR head re-derives the binding workspace from a peer-id suffix instead of consuming the already resolved route, and current main has routing/session changes the PR must preserve.

## Best Possible Solution

Keep this PR open as the implementation candidate. Revise it so AgentRouteBinding and the strict schema expose optional workspace, resolveAgentRoute carries the matched binding workspace together with existing session overrides, and reply execution consumes that resolved route value instead of re-scanning bindings. Add config help/generated schema and docs updates plus regression tests for peer, account, team, guild/role scoped bindings, uppercase peer ids, case-insensitive agent ids, fallback to the agent-level workspace, and preservation of binding session.dmScope.
