## Summary

Keep this issue open. Current main still only warns at gateway startup when model resolution fully misses, while the reported OpenRouter stale-slug path can still resolve as a dynamic text-only model and therefore avoids the startup unknown-model warning and doctor-style configured-model validation.

## Best Possible Solution

Keep this issue open and add non-fatal configured-model diagnostics for `agents.defaults.model.primary` and configured fallback refs. Gateway startup and `openclaw doctor` should warn when a provider catalog was successfully fetched and proves a configured ref is absent, while preserving OpenRouter dynamic pass-through when the catalog is unavailable or intentionally incomplete. Known retired or renamed slugs should be documented in changelog/docs when OpenClaw changes bundled defaults or compatibility behavior.

## Risks / Open Questions

- Hard-failing every uncatalogued OpenRouter ID would regress OpenRouter's intentional dynamic pass-through for newly published upstream models.
- A useful fix needs to distinguish catalog unavailable from catalog fetched and model absent, otherwise transient network failures could produce false invalid-model warnings.
