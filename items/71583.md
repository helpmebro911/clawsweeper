## Summary

Keep #71583 open. It is a recent, concrete QQ Bot bundled-plugin failure report with reproduction commands and follow-up diagnostics offered by the reporter. Current main still performs the QQ access-token POST directly against `https://bots.qq.com/app/getAppAccessToken` without a plugin-level timeout/signal, and `channels status --probe` has no QQ-specific token endpoint probe because the QQ Bot status adapter does not implement `probeAccount`. This leaves actionable OpenClaw work for bounded token-fetch behavior and clearer diagnostics, even if Tencent-side routing or regional policy is ultimately the root cause.

## Best Possible Solution

Keep the issue open for a focused QQ Bot plugin fix: add an explicit token-fetch timeout/AbortSignal and structured error, expose a lightweight QQ token/gateway probe through `status.probeAccount`, and document diagnostics for `bots.qq.com:443` reachability and routing/egress failures. If follow-up proves this is purely Tencent-side policy, close later with documented diagnostics and a user-facing workaround.
