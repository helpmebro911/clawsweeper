## Summary

Keep this issue open. The report is recent, specific, and still matches current main: QQ Bot token acquisition remains a plugin-owned raw fetch to `bots.qq.com` without an explicit timeout/proxy-aware helper, while the bundled channel depends on that fetch before the WebSocket can connect.

## Best Possible Solution

Keep this open as a QQ Bot plugin reliability and troubleshooting issue. Add a bounded timeout and structured error metadata around `TokenManager.doFetchToken()`, route token acquisition through the existing proxy-aware or dispatcher-aware plugin SDK fetch helper if appropriate, add focused regression tests for timeout/abort behavior, and update https://docs.openclaw.ai/channels/qqbot plus channel troubleshooting with `bots.qq.com` token endpoint diagnostics and expected timeout signatures. Keep core generic unless the plugin fix proves a missing SDK seam is needed.
