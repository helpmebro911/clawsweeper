## Summary

Keep #71583 open. The report is fresh, non-maintainer-authored, and concrete. The user's own curl evidence strongly suggests an upstream/egress reachability problem for `bots.qq.com`, but current `main` still has QQ Bot startup directly gated on the token fetch and lacks token-endpoint-specific timeout/diagnostic guidance, so there is still useful repo work to do in the QQ Bot plugin/docs rather than closing this as external-only.

## Best Possible Solution

Keep this open for a targeted QQ Bot plugin/docs improvement: add a bounded timeout and clearer error classification around `TokenManager.doFetchToken()`, include `bots.qq.com` reachability/proxy/egress diagnostics in the QQ Bot troubleshooting docs, and mention the existing `HTTPS_PROXY`/`HTTP_PROXY` path as an operator workaround when direct QQ egress is blocked. Do not treat this as a core gateway failure unless a separate repro shows core RPC/dashboard degradation.
