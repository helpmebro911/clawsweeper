## Summary

Keep this backup-design issue open. Current main still has useful backup primitives, including tar.gz create/verify, manifest checks, config-only mode, workspace opt-out, and top-level canonical path dedupe. The central requested work is still missing: configurable exclusion rules, recursive archive pruning such as plugin dependency filtering, service-aware or quiesced backup behavior for live state, resilient live-file handling, and broader restore/full-mode semantics. Related open items and PRs cover pieces, but no single canonical item fully supersedes the proposal.

## Best Possible Solution

Keep this issue open, but narrow it into a maintainer-readable backup design. The best path is to keep the shipped create/verify/config-only behavior, add a core exclusion policy surface or CLI/config option, land or supersede the live-file race fixes tracked by #67417 and #72251, decide whether service control should be an explicit quiesce mode, and coordinate broader restore/import disaster-recovery semantics with #13616.

## Risks / Open Questions

- The issue is broad; maintainers may want to split it into narrower implementation tasks for exclusion policy, live-file resilience, quiesce/service lifecycle, and restore/import semantics.
- Open PR #72251 may resolve one live-backup failure mode after merge, but it would not by itself implement configurable exclusions or service-aware backup behavior.
- Gateway stop/start or quiesce behavior is product-sensitive and should likely be explicit opt-in behavior rather than an implicit side effect of backup create.
