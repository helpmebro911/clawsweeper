## Summary

Keep open. Current `main` has two adjacent mitigations for duplicate webhook work: stable hook/cron session reuse and optional idempotency-key replay dedupe. But the requested feature itself is not implemented: webhook deliveries for an already-active `sessionKey` are still accepted, dispatched, and answered with HTTP 200 plus a `runId`, with no session-busy check and no 409 Conflict path.

## Evidence

- **`/hooks/agent` still accepts and dispatches same-session requests:** The `/hooks/agent` handler validates payloads, optionally reuses a cached `runId` only when an idempotency key matches, then calls `dispatchAgentHook(...)` and returns `200 { ok: true, runId }`. There is no active-session lookup, no `replyRunRegistry`/queue-state check, and no 409 response path.

## Risks / Open Questions

- Because ingress still returns 200 for a new same-session webhook without a matching idempotency key, long-running sessions can accumulate queued duplicate turns instead of being skipped.
- Per-session lane serialization lowers concurrency but does not satisfy the issue's requested external contract (`409 Conflict` when the `sessionKey` is already active).
- Closing this as implemented would conflate two different behaviors: retry dedupe by idempotency key versus skip-if-running by session identity.
