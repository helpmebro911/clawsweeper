## Summary

Keep #69735 open. Current main still passes native Ollama inbound `tool_calls[].function.arguments` through unchanged in `buildAssistantMessage`, so JSON-string arguments from Ollama can still reach downstream tool execution as strings. Existing normalization and tests cover replay/outbound conversion paths, not the reported inbound response assembly path; the latest release tag inspected has the same raw pass-through. The related fix PRs in the provided GitHub context were closed unmerged, so they do not supersede the issue.

## Best Possible Solution

Land a focused Ollama plugin fix in `extensions/ollama/src/stream.ts` that applies `ensureArgsObject(toolCall.function.arguments)` when building inbound assistant tool-call content, adjust the local Ollama tool-call type if needed to reflect string/unknown provider input, and add a regression test for `buildAssistantMessage` with JSON-string arguments, including malformed JSON or unsafe-integer behavior if maintainers want parity with the replay helper.
