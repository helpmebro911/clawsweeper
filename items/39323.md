## Summary

Keep PR #39323 open. Current main still lacks owner notifications for fresh DM pairing requests, has no `pairing:request` internal or plugin hook, and has no top-level `pairing.notify` config surface. The existing Telegram notification support is for node/device pairing, not DM sender pairing, and the closed related issue #38998 does not supersede this PR's remaining implementation work.

## Best Possible Solution

Keep the PR open as the active implementation candidate, or replace it with a maintainer-owned narrower design. The best path is to add a documented, versioned `pairing:request` event or plugin hook emitted only for freshly created DM pairing requests, then implement owner notification through a plugin or minimal core config with fixed-format sanitized text, PII-safe logs, target normalization, and regression tests. Do not close as ClawHub-only while the core/plugin surface still lacks an immediate DM pairing request event.
