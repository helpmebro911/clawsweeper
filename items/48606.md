## Summary

Keep open. Current main still has the reported macOS Edge default-browser fallback gap: Edge LaunchServices IDs are recognized, but if dynamic app-path resolution fails, the resolver returns null and falls into the generic macOS browser search where Chrome is tried before Edge. The PR is still directionally useful, but it should not be merged without the regression tests requested in the maintainer review.

## Best Possible Solution

Keep the PR open and ask for a small, tested version of the current approach: after macOS default-browser bundle detection succeeds but dynamic app-path resolution fails, consult an own-key known bundle-id mapping that checks both system and user Applications locations before falling through to generic browser discovery. Add regression coverage for `com.microsoft.edgemac` with osascript/defaults failure plus both Edge and Chrome present, and a missing-known-Edge-path case that still falls back to Chrome.
