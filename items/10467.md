## Summary

Keep #10467 open. Current main still does not expose a `lane` parameter on `sessions_spawn`, native and ACP spawned child runs are still dispatched on the hardcoded `subagent` lane, and subagent concurrency config remains a single scalar `agents.defaults.subagents.maxConcurrent` rather than per-lane limits. This is a concrete core/gateway/tool-schema feature request, not a cleanup close.

## Best Possible Solution

Keep #10467 open until core implements an optional validated `sessions_spawn.lane` field, forwards it through native and ACP spawn paths to the Gateway `agent` call, adds backward-compatible per-lane subagent concurrency configuration with default fallback to `subagent`, and updates docs plus regression tests for default lane behavior, custom lanes, invalid lane names, and config reload.

## Risks / Open Questions

- The lower-level Gateway `agent` protocol and queue can accept arbitrary lane strings, but `sessions_spawn` does not expose that control and unconfigured lanes default to concurrency 1, so the feature still needs explicit tool/config semantics.
- Implementation should define lane-name validation, fallback to `subagent`, per-lane config shape, reload behavior, and regression tests for native and ACP spawn paths.
