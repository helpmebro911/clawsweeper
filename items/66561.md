## Summary

Keep this issue open. Current main has adjacent OpenAI Codex transport normalization and model-call diagnostics, but the embedded Pi runner still treats the whole attempt timeout as a generic timedOut/aborted state, and assistant failover still normalizes that state to timeout/408 without carrying whether an upstream SSE response had already started.

## Best Possible Solution

Keep this open and fix it in the embedded Pi runner/failover path. Track provider-call lifecycle separately from the whole-attempt timer, carry request-started/response-started or first-stream-event state into the attempt result and failover observation, and distinguish no-response timeout from post-start local abort, parser/stream failure, and upstream explicit errors. Add a focused regression test that simulates a streamed Codex response being aborted by the enclosing timeout and asserts the surfaced reason/status/log fields are not the generic no-response timeout shape.
