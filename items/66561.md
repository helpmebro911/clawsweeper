## Summary

Keep #66561 open. Current main has implemented part of the related Codex transport/base-url workaround, but the central report remains actionable: OpenAI/Codex SSE can still surface a local post-start abort through the same `Request was aborted`/`AbortError` path that current failover code classifies as `timeout`/408, and the embedded attempt watchdog still marks the enclosing run as timed out without carrying a distinct “upstream stream already started” signal.

## Best Possible Solution

Keep the issue open and implement a transport-progress-aware classification path. The likely maintainer path is to propagate whether the OpenAI/Codex Responses stream reached headers/start/events, classify post-start local aborts separately from pre-response timeouts, and add regression coverage in the OpenAI transport, failover classifier, and embedded-run failover logging paths. Leave the existing Codex base-url/SSE normalization in place as the narrower #41282-era fix.
