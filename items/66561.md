## Summary

Keep this issue open. Current main still has adjacent OpenAI/Codex transport support and model-call diagnostics, but the embedded run/failover boundary still collapses an enclosing attempt timeout into `timeout`/408 without carrying request-started or response-started state. The discussion also points this way: #66599 was a closed, unmerged narrow PR, and later comments explicitly said the baseUrl/SSE workaround from #41282 did not solve the root classification problem for the reporter.

## Best Possible Solution

Keep this issue open and fix the embedded PI runner/failover boundary. Track provider-call lifecycle separately from the whole-attempt timer, carry request-started/response-started or first parsed stream event state into `EmbeddedRunAttemptResult` and failover observation, and surface distinct reasons for no-response timeout, post-start local abort, parser/stream failure, and upstream explicit errors. Add a focused regression test that simulates a Codex SSE stream receiving at least one chunk before the enclosing timeout aborts it and asserts the surfaced failure is not the generic no-response timeout/408 shape.
