## Summary

Keep this issue open. Current main still lacks the requested local transcript-token estimate for providers that return missing or all-zero usage: status and usage surfaces recover provider-reported transcript usage, but they do not estimate tokens from stored JSONL message content or show an estimated marker. Preflight compaction has a partial transcript estimate path, so part of the compaction concern is improved, but the central MiniMax-style visibility gap remains.

## Best Possible Solution

Keep this open and implement a shared estimated context snapshot path for sessions whose provider usage is missing or all zero. The likely maintainer path is to reuse readSessionMessages plus estimateMessagesTokens or a shared chars/token helper, expose the result as explicitly estimated/stale-aware context usage, render an estimated marker in /status and /usage, and let compaction gates consume the same estimate without overwriting fresh provider-reported snapshots.

## Risks / Open Questions

- Some provider-specific paths may now supply promptTokens or lastCallUsage for MiniMax in cases not covered by the issue, but the provider-independent missing/all-zero usage fallback requested here is still absent in the shared status and usage paths.
