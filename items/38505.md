## Summary

Keep #38505 open. Current main still wires Discord inbound conversation handling through `MessageCreateListener` only; there is no `MessageUpdateListener` registration or edit-aware dispatch path. The existing replay guard is keyed by account, channel, and message id, so simply sending an edited message through the current handler would be dropped as a duplicate after the original delivery. Related #53654 overlaps but is broader, open, and does not supersede this focused edit-reprocess request.

## Best Possible Solution

Keep the issue open and implement the missing behavior in the bundled Discord plugin. Add a Carbon `MessageUpdateListener`, ignore updates that do not change user-visible content, run qualifying edited messages through the same preflight/authorization/session path with the edited content as authoritative, make replay/dedupe edit-aware, and add focused tests for mention-added edits, DMs or active threads, non-content update suppression, and duplicate update handling. Coordinate with open #53654 if delete-to-cancel is implemented in the same area.
