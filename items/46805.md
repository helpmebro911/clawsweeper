## Summary

Keep open. Current main and the latest release still resolve Feishu sender names through a single sender id, effectively preferring `open_id` when present, so the PR's central bug is not implemented or superseded. The PR is a plausible bundled Feishu channel fix, but it needs to be rebased onto the current split-out sender-name helper and should retain the cache/error fallback fixes raised in review.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase that ports the same behavior into `extensions/feishu/src/bot-sender-name.ts`: pass both `sender_id.user_id` and `sender_id.open_id`, prefer `user_id`, fall back to `open_id` and the historical id, and add focused Feishu tests for priority, fallback, cache ordering, and cache-poisoning cases.
