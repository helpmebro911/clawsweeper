## Summary

Keep this issue open. Current main has separate TUI session-key canonicalization and Ollama native tool-call support, but I did not find a direct fix or regression test for the combined `openclaw tui --deliver` + local Ollama + session-aware tool path that produced the `openclaw-tui` session-id failure.

## Best Possible Solution

Keep the issue open and add a focused regression for `openclaw tui --deliver` with the UI client identity `openclaw-tui`, canonical session key `agent:main:main` or `current`, and a native Ollama tool-capable response that exercises a session-aware tool. If the model is being shown or encouraged to pass `openclaw-tui` as a session reference, fix the prompt/session metadata or resolver guidance so tools use `current`, `main`, or the canonical agent session key instead.
