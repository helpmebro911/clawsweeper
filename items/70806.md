## Summary

Keep #70806 open. The report is recent, has a concrete symptom plus follow-up memory-file evidence, and current main still has a plausible core/provider transport path that can produce repeated `[object Object]` in assistant text when an OpenAI-compatible provider emits non-string structured `delta.content` during tool-use streaming. I did not find source, test, or history evidence that current main has fixed this specific chat contamination path.

## Best Possible Solution

Keep the issue open. The likely maintainer path is to add a regression test for an OpenAI-compatible/Mistral-style stream where `delta.content` is structured or otherwise non-string during tool use, then harden `processOpenAICompletionsStream` so only real text is appended as assistant text, structured tool/reasoning content is routed through the proper handlers, and malformed objects cannot contaminate chat, session history, or memory.
