## Summary

Keep this PR open. Current main still does not send `encoding_format` on the relevant OpenAI-compatible memory embedding request bodies, so the reported compatibility problem is not implemented. However, after the memory embedding refactor, the PR's one-line change now lands in the shared remote helper used by Mistral/LM Studio, while the OpenAI plugin constructs its own direct and batch embedding bodies and still omits `encoding_format`; the PR likely needs rework or consolidation with the broader open #43910 rather than cleanup closure. Security review found no supply-chain or secrets-sensitive change in the provided diff.

## Best Possible Solution

Keep this PR open, but ask for a current-main update that targets the actual OpenAI-compatible embedding request paths: add `encoding_format: "float"` where OpenAI-compatible providers construct direct and batch bodies, preserve existing `input_type` behavior, and add regression tests for the strict-proxy case. Maintainers should also decide whether #43910's configurable `float`/`base64` surface should absorb this narrower compatibility fix or remain a separate follow-up.
