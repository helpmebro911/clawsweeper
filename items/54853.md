## Summary

Current main still lacks automatic cross-channel account binding for Feishu tool calls. The existing behavior supports inbound account-to-agent routing, Feishu defaults, and same-id contextual routing via agentAccountId, but not an agent-level or generic target-channel account binding for a DingTalk-originated agent to select a different Feishu account.

## Best Possible Solution

Keep this issue open. The best fix is a generic, documented target-channel account resolution contract, likely agents.list[].channelBindings or an equivalent lookup keyed by agentId plus target channel. Feishu tool resolution should prefer explicit tool accountId, then the current agent's bound Feishu account, then valid same-id contextual Feishu account routing, then channels.feishu.defaultAccount/runtime default. Add config type/schema/help/docs and regression coverage for a DingTalk-originated agent whose DingTalk account id differs from the intended Feishu account id.

## Risks / Open Questions

- Deployments where the origin channel account id intentionally matches the Feishu account id may work through the same-id contextual fallback, but that is narrower than the requested explicit cross-channel binding.
- Because this is a multi-account permission isolation request, closing it without a new target-channel account resolution contract could leave Feishu tools using channels.feishu.defaultAccount in mixed-user deployments.
