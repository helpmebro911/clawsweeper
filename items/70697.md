## Summary

Current main has partial mitigation and Docker-native helper/docs, but the reported Docker lifecycle gap is not fully resolved. `openclaw gateway restart` is still primarily a managed-service command with an unmanaged SIGUSR1 fallback, while the bundled Docker Compose CLI container shares only the gateway network namespace and the gateway respawn path has no Docker/container supervisor detection.

## Best Possible Solution

Keep the issue open and make the Docker lifecycle explicit. Either make containerized gateways use a deterministic restart mode, such as setting a Docker/container marker or `OPENCLAW_NO_RESPAWN` so SIGUSR1 stays in-process or hands off to Docker cleanly, or make `openclaw gateway restart` detect Docker and fail with a Docker-native command such as `docker compose restart openclaw-gateway`/`clawdock-restart`. Add a focused regression test for the container/PID-namespace path and update Docker docs to distinguish managed-service restart from container restart.

  - file: [src/cli/daemon-cli/register-service-commands.ts:115](https://github.com/openclaw/openclaw/blob/b979f2964c9609a4d2ce9e96d643ed020e3fc00e/src/cli/daemon-cli/register-service-commands.ts#L115)

- **Unmanaged fallback sends SIGUSR1 and waits for health:** When the service is not loaded, restart falls back to locating a single gateway listener PID, signaling it with SIGUSR1, and then polling gateway health. That can work only when the CLI can see and signal the gateway process; it is not a Docker-native restart path.
  - file: [src/cli/daemon-cli/lifecycle.ts:124](https://github.com/openclaw/openclaw/blob/b979f2964c9609a4d2ce9e96d643ed020e3fc00e/src/cli/daemon-cli/lifecycle.ts#L124)

- **Docker Compose uses separate gateway and CLI services:** The bundled Compose file runs `openclaw-gateway` as a foreground `node dist/index.js gateway ...` process, while `openclaw-cli` is a separate service sharing the gateway network namespace via `network_mode: service:openclaw-gateway`; it does not declare a shared PID namespace for process signaling.
  - file: [docker-compose.yml:32](https://github.com/openclaw/openclaw/blob/b979f2964c9609a4d2ce9e96d643ed020e3fc00e/docker-compose.yml#L32)

There is no Docker/container supervisor marker here, which leaves the container-entrypoint restart behavior ambiguous.
