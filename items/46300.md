## Summary

Current main does not implement #46300: `workspaceAccess: "none"` is still treated as read-only by Docker/browser mounts, the local and remote sandbox FS bridges, memory-flush gating, and regression tests. The PR addresses a real unresolved behavior gap and the author responded to the in-thread hash and memory review comments, but the diff shown in the provided context still appears incomplete for current main because OpenShell/remote FS and current public docs continue to model `"none"` as non-writable.

## Best Possible Solution

Keep the PR open, but require a complete maintainer-reviewed direction. If `workspaceAccess: "none"` should be writable, finish the change across Docker/browser mounts, local FS bridge, remote/OpenShell FS bridge, memory-flush gating, config-hash recreation, tool/write capability checks, and public docs/tests. If maintainers instead want `none` to stay write-restricted, close or rewrite this PR in favor of suppressing write-capable tools under `none`.
