## Summary

Keep this PR open. Current main still preserves the status scan source config at the overview layer but does not thread it into the status memory probe/runtime bootstrap path. The linked bug issue was closed after broader loader/source-snapshot fixes, but this PR still targets a narrower remaining gap in the status memory path and should be reviewed as a scoped rebase rather than closed as obsolete.

## Best Possible Solution

Keep this PR open and review a scoped rebase. The best implementation is to thread `overview.sourceConfig` as an internal `activationSourceConfig` through `executeStatusScanFromOverview`, `resolveStatusMemoryStatusSnapshot`, `resolveSharedMemoryStatusSnapshot`, the status runtime dependency wrapper, and `getActiveMemorySearchManager`/`ensureMemoryRuntime`, while passing only the declared `{ cfg, agentId, purpose }` fields to `MemoryPluginRuntime` and preserving omission of optional `purpose`. Split unrelated extension/type cleanup, then run the focused status and memory-runtime regression tests plus `pnpm check:changed`.
