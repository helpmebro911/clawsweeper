## Summary

Keep #55986 open. Current main has an OpenRouter media-understanding provider, but it is image-only; audio transcription still requires a provider with transcribeAudio, and the OpenRouter docs/manifest currently declare only image capability. The PR remains relevant, though it needs rebasing against current main and API verification before it could land.

## Best Possible Solution

Keep the PR open, but require an update against current main: extend the existing OpenRouter media-understanding provider to add audio without regressing image support, update manifest metadata and user docs to include audio, keep focused registration/request-shape tests, and verify or clearly document OpenRouter model compatibility for input_audio.
