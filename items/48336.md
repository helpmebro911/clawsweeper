## Summary

Keep #48336 open. Current main still does not provide a general all-HTTP-request per-IP gateway limiter; it only has adjacent protections such as failed-auth throttling, hook auth throttling, preauth WebSocket connection budgets, control-plane write limits, and baseline security headers. The PR is also not safe to merge as-is because review comments correctly flagged that it keys the new limiter from raw socket remoteAddress before trusted-proxy client IP resolution.

## Best Possible Solution

Keep the PR open for maintainer action. The best path is to rework the PR so the limiter is configured through the gateway config contract, is wired once in the HTTP request pipeline after loadConfig and resolveRequestClientIp, uses trusted-proxy semantics consistently, has tests for direct, Tailscale/reverse-proxy, loopback exemption, and 429 Retry-After behavior, and leaves the broader security-labeled #20373 as the canonical product/security tracker until the full gateway rate-limiting scope is shipped.
