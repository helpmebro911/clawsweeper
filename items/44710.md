## Summary

Keep open. Current main has static config-driven tool allow/deny policy and per-tool-call plugin hooks, but it does not have the PR's requested list-level `before_tools_resolve` hook that can filter the final tool list before LLM exposure. This is a real core/plugin API seam rather than ClawHub-only work, because external plugins cannot add a new hook invocation point inside the agent tool assembly path themselves. The provided PR discussion also shows focused iteration on security-sensitive merge and final-tool-set coverage issues, not a stale or duplicate cleanup item.

## Best Possible Solution

Keep the PR open for maintainer API/security review. The best implementation path is a documented public hook in the plugin contract and runner, invoked exactly once over the final LLM-visible tool set for both attempt and compaction paths, including core, client, bundled MCP, and bundled LSP tools; merge semantics should be deny-union and normalized allow-intersection with focused tests for multi-hook, empty-allow, identity context, and final-tool-set edge cases. If maintainers decide dynamic list-level tool policy is not desired, that should be an explicit product/API decision rather than automated cleanup.
