## Summary

Keep PR #19328 open. Current main at e672b614 still has the stale cached-entry writeback path that the PR targets: the agent handler reads a potentially cached session entry, copies store-owned fields including modelOverride/providerOverride into nextEntryPatch, then merges that patch over the fresh row loaded by updateSessionStore. The related #70575 only tracks narrower residual delivery/channel fields, not the central modelOverride race.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix in src/gateway/server-methods/agent.ts. The best implementation is to stop copying cache-derived store-owned fields into nextEntryPatch before merging over the fresh store row, or build those fields from the fresh row inside updateSessionStore, then add regression coverage that simulates stale loadSessionEntry data racing with fresh updateSessionStore data. Leave the narrower lastChannel/channel residual to #70575.
