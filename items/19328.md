## Summary

Keep PR #19328 open. Current main at cf834e2a21eac7fd218989fb5b85e7e8dd4c3246 still has the stale cached-entry writeback pattern the PR targets: the agent handler builds a patch from `loadSessionEntry()` data, including `modelOverride` and `providerOverride`, then merges that patch over a fresh `updateSessionStore()` read. The recent maintainer/Codex comment correctly identified the same remaining race, and the PR remains a focused useful fix rather than obsolete cleanup.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop putting cache-derived session fields into `nextEntryPatch` before merging over the fresh store, preserve fresh store-only fields through `mergeSessionEntry`, and add regression tests that simulate stale `loadSessionEntry()` data against a fresh `updateSessionStore()` store for `modelOverride`, `providerOverride`, and the broader stale-writeback field class.

## Evidence

If `entry` is stale, these patch values can overwrite fresher store state.
