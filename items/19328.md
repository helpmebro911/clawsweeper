## Summary

Keep PR #19328 open. Current main still builds the agent session patch from a potentially cached `loadSessionEntry()` result, including `modelOverride`, `providerOverride`, send policy, skill, thinking, spawn, and CLI binding fields, then merges that patch over the fresh row loaded by `updateSessionStore`. That means a stale cached entry can still clobber a fresh `sessions.patch` model override. The latest maintainer/Codex discussion also explicitly kept this PR open for the same code-backed reason, and the PR remains a focused fix rather than obsolete cleanup.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop copying cache-derived store-only fields into `nextEntryPatch` before merging over the fresh store row, or build those fields from the fresh row inside `updateSessionStore`, and add regression coverage that simulates stale `loadSessionEntry` data against fresh `updateSessionStore` data. Keep the narrower lastChannel/channel residual tracked in #70575.
