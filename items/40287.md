## Summary

Keep this PR open. Current main still does not register or document Slack `file_shared` events, so the large-file Socket Mode fallback remains missing. The PR is stale because it targets the removed `src/slack` layout, but the central behavior is still useful and should be ported into the current `extensions/slack` plugin.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but request a rebase/port into `extensions/slack`. The best fix is a Slack plugin `file_shared` event handler under `extensions/slack/src/monitor/events`, with `file_shared` added to setup manifest and public Slack docs, hydration through Slack APIs anchored by file/channel/event identity, race-safe dedupe against normal `message` delivery, preservation of existing account/team and allowlist/mention policies, and focused tests for the fallback and setup surface.

## Risks / Open Questions

- The PR likely cannot merge as-is because it modifies old `src/slack` files rather than the current `extensions/slack` plugin layout.
- A correct fallback must avoid duplicate processing when Slack delivers both normal `message` and `file_shared` events, and it should anchor history lookup to event/file identity rather than a fragile recent-message window.
- The port needs focused tests for large-file fallback, normal-message/file_shared dedupe, busy-channel lookup behavior, setup manifest coverage, and docs coverage.
- Security-sensitive behavior is in the event handler path: the implementation must preserve Slack app/team mismatch checks, channel/DM authorization, mention gating, and scoped Slack API reads before passing hydrated events into the message pipeline.
