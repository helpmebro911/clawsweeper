## Summary

Current main still matches the reported memory-wiki lint bug. The wikilink extractor runs a raw `[[...]]` regex over page content after removing only the managed related block, so fenced code, inline code, frontmatter, comments, and transcript text can still be interpreted as real wikilinks. The broken-link resolver still indexes only extensionless relative paths and basenames, while `toWikiPageSummary` already reads frontmatter titles that the resolver does not use. The related #64696 remains a narrower open issue and does not supersede this broader extractor/resolver report.

## Best Possible Solution

Keep #70986 open. The best fix belongs in `extensions/memory-wiki`: make `extractWikiLinks` markdown-aware by masking or parsing frontmatter, fenced code, inline code, and HTML comments before wikilink extraction; expand broken-link resolution to include normalized page titles, slugified titles, aliases if supported, and relative Markdown targets; then add regression tests covering fenced/inline false positives and `[[<frontmatter title>]]` resolution. #64696 can remain a narrower related durability issue unless maintainers choose to consolidate it under this broader lint resolver work.
