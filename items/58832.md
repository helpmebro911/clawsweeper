## Summary

Keep open. Current `main` still routes top-level Slack DMs with `replyToMode: "all"` into thread-scoped session keys, so the reported behavior remains implemented rather than fixed.

## Evidence

- **DM routing still auto-threads top-level messages in `replyToMode="all"`:** `resolveSlackRoutingContext` keeps DM `autoThreadId = messageTs` when the inbound Slack message is not a real thread reply and `replyToMode === "all"`, then uses that as `canonicalThreadId` for non-room chats. That still produces per-message `:thread:<ts>` routing for direct messages.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:63](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L63)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '63,83p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Slack thread context still assigns a thread id to top-level messages in `all` mode:** `resolveSlackThreadContext` sets `messageThreadId` to the message timestamp for non-thread replies whenever `replyToMode === "all"`.
  - file: [extensions/slack/src/threading.ts:16](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/threading.ts#L16)
  - command: `nl -ba extensions/slack/src/threading.ts | sed -n '16,28p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Inbound session creation consumes that thread id:** The Slack inbound context writes `MessageThreadId: threadContext.messageThreadId`, and session initialization uses `ctx.MessageThreadId` to classify the session as a thread. That is the path that turns the DM into a `:thread:` session.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:683](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/monitor/message-handler/prepare.ts#L683)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare.ts | sed -n '678,690p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Current tests lock in the reported behavior:** `extensions/slack/src/threading.test.ts` explicitly expects top-level Slack messages in `replyToMode: "all"` to get `messageThreadId === ts`, so current `main` is intentionally asserting the same condition the issue reports.
  - file: [extensions/slack/src/threading.test.ts:76](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/threading.test.ts#L76)
  - command: `nl -ba extensions/slack/src/threading.test.ts | sed -n '64,90p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)

## Risks / Open Questions

- I did not run a live Slack reproduction because this review had to remain read-only, so the conclusion is from current code paths and tests rather than a runtime trace.
- The issue body claims the bug affects any `replyToMode != off`, while current code evidence is strongest for `replyToMode: "all"`; `first` and `batched` already avoid the top-level auto-thread path in `extensions/slack/src/threading.test.ts`.
