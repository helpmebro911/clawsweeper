## Summary

Keep #70901 open. The report is a recent, concrete bug with another user confirming it and an open PR (#71035) proposing the same fix, but current `main` still passes HTTP MCP `url` and `headers` through as literal strings. The documented configuration example even shows `Authorization: "Bearer ${MCP_REMOTE_TOKEN}"`, so the current implementation/docs contract is still inconsistent until the fix lands.

## Best Possible Solution

Keep the issue open and land a focused fix, likely through #71035 or an equivalent patch, that expands `${ENV_VAR}` in global HTTP MCP `url` and `headers` before transport construction. Add regression coverage around `resolveHttpMcpServerLaunchConfig()` / `resolveMcpTransportConfig()` for expanded headers, missing env behavior, and docs-aligned examples; separately decide whether `SecretRef` support should be added to the global `mcp.servers.*.headers.*` schema or documented as unsupported.
