## Summary

Keep this PR open. Current main still sends WhatsApp inbound text to the agent through raw `BodyForAgent`, so the requested hardening is not implemented. The submitted patch is also not safe to merge unchanged because it wraps legacy envelope text used for echo keys while the active prompt path prefers `BodyForAgent`, and it would place trusted `messagePrefix` text inside the untrusted block.

## Best Possible Solution

Keep this PR open as an incomplete security-hardening candidate. The maintainer path is to revise or replace it so WhatsApp wraps or structurally isolates the actual model-facing `BodyForAgent`, reply-context body, and model-visible group-history text, while keeping echo keys deterministic and keeping trusted operator `messagePrefix` outside untrusted markers. Add focused WhatsApp tests for direct messages, audio transcript bodies, group history, reply context, prefix handling, and duplicate-message echo suppression.

## Risks / Open Questions

- Current main still exposes WhatsApp inbound message text, audio transcripts, and group-history text to the model outside the external-content wrapping path.
- Merging the submitted patch unchanged would likely break combined-message duplicate suppression because wrapped content includes a fresh random marker id per call.
- Merging the submitted patch unchanged would put trusted operator `messagePrefix` text inside the untrusted-content block because the patch wraps the whole `baseLine`.
- The PR’s cited Discord precedent is only a partial pattern: Discord appends wrapped message text as `UntrustedContext` for guild context while still keeping `BodyForAgent` separate, so WhatsApp needs a targeted design rather than a direct legacy-envelope wrap.
