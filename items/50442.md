## Summary

Keep #50442 open. Current main has improved atomic publication and cleanup for ordinary backup errors, but the reported timeout-kill path is still not clearly fixed: `openclaw backup create` still writes a large `*.tmp` archive next to the requested output and only removes it from normal async control flow. The issue discussion's workaround to periodically delete stale `.tmp` files is still relevant, and the core backup tool remains the right owner because this temporary file is created by core backup code, not by a plugin.

## Evidence

- **Temporary archive is still written as outputPath.UUID.tmp:** Current main builds the temporary archive path by appending a random UUID and `.tmp` to the final output path, so default backups can still create large `.tmp` files in the destination/current directory while tar is running.
  - file: [src/infra/backup-create.ts:134](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/infra/backup-create.ts#L134)

  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Cleanup only runs in normal async finally path:** The implementation removes `tempArchivePath` in `finally`, which covers thrown tar/copy errors, but it does not make the cleanup survive an external timeout killing the process before JS control returns.
  - file: [src/infra/backup-create.ts:337](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/infra/backup-create.ts#L337)
