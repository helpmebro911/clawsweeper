## Summary

Keep #71738 open. Current main still has the Matrix thread-history and reply-placement gaps this PR targets: Matrix room history and watermarks are scoped only by room, Matrix readMessages has no threadId/relations path, thread sends still emit fallback m.in_reply_to/is_falling_back metadata, and threaded edits still put m.in_reply_to on the outer replace relation. The PR is useful, but the provided review comments identify real edge cases that should be revised rather than closed as obsolete.

## Best Possible Solution

Keep #71738 open for revision on current main. The best path is to preserve the core Matrix behavior change, but fix the review-commented edge cases: add thread-scoped room-history queues and watermarks, pass threadRootId through handler history calls, add deliberate threadId reads via Matrix relations, filter main-room reads without lossy pagination cursors, and remove fallback reply metadata from thread sends/edits unless explicitly requested.
