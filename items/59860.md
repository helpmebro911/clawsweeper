## Summary

Current `main` still appears vulnerable to the reported iMessage self-chat echo case. In the self-chat `is_from_me` path, `skipIdShortCircuit` is only enabled when the inbound GUID is missing, but the echo helper still falls back using the first inbound ID (typically the SQLite row ID) and `SentMessageCache.has()` still short-circuits before text matching when that ID misses. The existing tests on `main` explicitly preserve that short-circuit behavior and do not cover the reported `new row id + new/different guid + same text` self-chat reflection path.

## Evidence

- **Self-chat path only skips ID short-circuit when inbound GUID is absent:** For `is_from_me` self-chat messages, the inbound decision calls `hasIMessageEchoMatch(..., skipIdShortCircuit: !hasInboundGuid)`. If the inbound message carries any GUID, the code keeps ID short-circuiting enabled.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:250](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/imessage/src/monitor/inbound-processing.ts#L250)
  - command: `nl -ba extensions/imessage/src/monitor/inbound-processing.ts | sed -n '220,280p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Echo helper retries bare IDs first, then reuses the first inbound ID for text fallback:** `hasIMessageEchoMatch()` checks each inbound ID alone, then performs the text fallback with `messageId: params.messageIds[0]`. Because `resolveInboundEchoMessageIds()` orders `message.id` before `guid`, the fallback normally reuses the SQLite row ID rather than the GUID.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:70](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/imessage/src/monitor/inbound-processing.ts#L70)
  - command: `nl -ba extensions/imessage/src/monitor/inbound-processing.ts | sed -n '1,140p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **SentMessageCache still short-circuits before text match on ID-backed entries:** When a `messageIdKey` is present but misses the ID cache, `has()` returns `false` unless `skipIdShortCircuit` is true or the text cache entry is text-only. For outbound entries remembered with both text and GUID, the text fallback is intentionally blocked.
  - file: [extensions/imessage/src/monitor/echo-cache.ts:69](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/imessage/src/monitor/echo-cache.ts#L69)

## Risks / Open Questions

- iMessage self-chat users can still get duplicate agent turns or echo loops when reflected messages carry non-matching row IDs/GUIDs but matching text.
- The current test suite leaves a gap around GUID-backed outbound sends reflected back as `is_from_me` self-chat rows with different identifiers.
