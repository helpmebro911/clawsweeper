## Summary

Keep this PR open, but do not merge it as written. Current main already has safer Google Chat auth plumbing and channels.status runtime snapshots, but health snapshots still do not receive gateway runtime state. The PR’s remaining useful idea should be replaced with a narrow health/runtime snapshot change that preserves probe fields and current read-only health behavior; the manual Google auth rewrite is security-sensitive and stale against current main.

## Best Possible Solution

Do not merge this PR as-is. Replace it with a focused gateway/health change that passes the current ChannelRuntimeSnapshot into getHealthSnapshot while preserving the read-only plugin path and existing probe, lastProbeAt, configured, enabled, and plugin summary behavior for every channel. Leave Google Chat auth on the current google-auth.runtime path unless a separately evidenced auth bug is fixed with the same SSRF and credential validation guarantees.

## Risks / Open Questions

- The PR’s manual Google service-account token exchange would bypass current main’s SSRF-guarded google-auth transport and strict Google endpoint validation, widening a credential-bearing network path.
- The PR’s health refactor can drop probe data unless it reattaches probe/audit baseline fields or fixes buildSnapshotFromAccount’s default path.
- The PR is stale against current health architecture, which now uses read-only channel plugin metadata in health.ts; applying the old diff directly would risk reintroducing heavier plugin loading on a cold health path.
