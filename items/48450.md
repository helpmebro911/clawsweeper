## Summary

Keep open. Current main still has the reported side-effect/return divergence for Discord `thread-create`: `createThreadDiscord` creates the thread, then posts the initial content as a second REST call, and only returns the thread after that second call succeeds. If the second call fails, the `threadCreate` action rejects before returning the created thread ID, so the agent still cannot distinguish a failed create from a created thread with failed initial content. The existing issue comments correctly identify the needed partial-success/thread-id behavior, and current tests cover the two-step success path but not recovery from an initial-message failure.

## Evidence

- **Issue discussion identifies the remaining contract gap:** Ryce and Hollychou924 both describe the same failure mode: Discord thread creation succeeds, a later initial-message step fails, and OpenClaw collapses that into an error without a created thread ID, causing retries and duplicate threads.
- **Thread creation still returns only after the follow-up message succeeds:** `createThreadDiscord` posts to `Routes.threads(...)`, stores the returned `thread`, then for non-forum content posts to `Routes.channelMessages(thread.id)` before returning `thread`. There is no try/catch or partial-success return around the second post.
