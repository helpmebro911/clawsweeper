## Summary

Keep #17424 open. Current main has a generic `outputDimensionality` config field and provider API plumbing, and Gemini/Bedrock plus the separate memory-lancedb plugin use dimension controls, but the requested default memory-search OpenAI/OpenAI-compatible embedding path still drops the dimension value before calling `/embeddings`. The issue specifically asks for `agents.defaults.memorySearch` and per-agent overrides to pass OpenAI-compatible `dimensions` for models like `text-embedding-3-large`, and that meaningful capability remains missing.

## Evidence

- **Issue request and follow-up are still about OpenAI-compatible memory embeddings:** The issue requests a `dimensions` setting under `agents.defaults.memorySearch` and per-agent overrides, passed as `dimensions` in OpenAI-compatible embedding requests; the only comment reinforces this for qwen3 embedding models up to 4096 dimensions. The provided related PR #18734 is a closed, unmerged gateway pairing PR whose `Fixes #17424` reference is unrelated, and #19028 is a closed, unmerged MongoDB backend proposal rather than a resolution of this config surface.
- **Config accepts only `outputDimensionality`, not the requested `dimensions` alias:** `MemorySearchConfig` includes `outputDimensionality?: number`, and the runtime schema accepts `outputDimensionality`, but there is no `dimensions` key in the memorySearch config schema. This means the example from the issue would not be accepted as written.
