## Summary

Keep open. Current main has adjacent memory safeguards and optional memory plugins, but still lacks the requested built-in bounded memory CRUD tool with per-store hard character budgets, rejected over-limit writes, substring replace/remove, duplicate handling for Markdown stores, and secret-screened structured writes.

## Best Possible Solution

Keep open for a focused memory-core or active-memory design. The likely implementation is a structured `memory` or `memory_write` tool owned by the active memory slot, with config-backed per-store character budgets, atomic add/replace/remove operations, duplicate and credential-pattern checks, clear over-limit rejection responses that include current usage, and docs/tests. If enforcement must apply beyond that tool, add a generic write-guard seam rather than hardcoding one memory file path into unrelated filesystem tools.
