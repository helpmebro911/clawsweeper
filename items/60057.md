## Summary

Current `main` still contains the unsafe `job.name` dereference in `printCronList`, so this PR is addressing a live bug rather than something already implemented or obsolete. The review comment also correctly identifies that the fix should ideally land with a regression test.

## Evidence

- **Unsafe dereference still on main:** `printCronList` still builds `nameLabel` as `pad(truncate(job.name, CRON_NAME_PAD), CRON_NAME_PAD)`. In the same file, `truncate` immediately reads `value.length` and `pad` calls `value.padEnd(width)`, so `job.name === undefined` is still a crash path on current `main`.
  - file: [src/cli/cron-cli/shared.ts:279](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cli/cron-cli/shared.ts#L279)
  - command: `nl -ba src/cli/cron-cli/shared.ts | sed -n '156,290p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Adjacent fields already use fallback guards:** The same formatter already applies `?? "-"` guards for `sessionTarget`, `agentId`, and the model label. That makes the proposed `job.name ?? "-"` change consistent with current defensive handling in this codepath.
  - file: [src/cli/cron-cli/shared.ts:291](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cli/cron-cli/shared.ts#L291)
  - command: `nl -ba src/cli/cron-cli/shared.ts | sed -n '288,305p'`

- **Type contract hides the runtime gap:** `CronJobBase` still declares `name: string`, so typechecking alone will not catch older persisted jobs that surface without `name` at runtime.
  - file: [src/cron/types-shared.ts:6](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/types-shared.ts#L6)

- Closing this PR would leave a plausible `openclaw cron list` crash on current `main` for legacy cron records missing `name`.
- If the fix lands without the analogous regression test, this legacy-compatibility path could regress again without automated coverage.
