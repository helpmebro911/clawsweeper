## Summary

Keep #30067 open. Current main still does not add the requested proactive setup heads-up about temporary provider API rate-limit errors before the TUI/Web/chat handoff; existing docs and runtime error formatting are reactive, not an initial setup notification.

## Best Possible Solution

Keep #30067 open. Add a concise first-run onboarding notice in the classic finalizeSetupWizard path before the TUI/Web/Later hatch choice, and mirror equivalent guidance in Crestodian/setup handoff if that path can launch chat. The copy should explain that first-run provider initialization may temporarily hit provider rate limits, suggest waiting/retrying or configuring a fallback model, and add focused tests asserting the note appears in the relevant first-run paths.
