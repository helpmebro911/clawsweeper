## Summary

Keep this PR open. Current main still requires `MessageSid` to build an inbound dedupe key, so idless inbound retries are treated as invalid and continue through dispatch. The PR is a focused active implementation candidate, and the author addressed the bot-reviewed provider-id key regression by preserving the legacy raw message-id suffix while keeping the new `content:<hash>` fallback for idless messages.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is to land or refine the PR's content/timestamp fallback in `src/auto-reply/reply/inbound-dedupe.ts`, preserving the current raw `MessageSid` key format for provider-id messages and keeping regression coverage for idless retries plus same-body/different-timestamp non-collapse.
