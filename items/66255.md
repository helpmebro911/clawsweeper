## Summary

Keep this open. Current main has a useful partial mitigation: gateway restart-deferral now reads embedded-run activity from a lightweight state module instead of importing the embedded runner during boot. But the remaining crash surface is still real: OpenClaw still pins `@mariozechner/pi-coding-agent@0.70.2`, still resolves the Linux x64 GNU native clipboard package, and still imports the top-level Pi package for embedded agent/runtime paths where upstream still loads `@mariozechner/clipboard` whenever Linux has `DISPLAY` or `WAYLAND_DISPLAY` and `TERMUX_VERSION` is not set. The issue discussion also has concrete coredump evidence plus a confirmed `TERMUX_VERSION=1` workaround, so this is actionable and not stale.

## Best Possible Solution

Keep this issue open. The best fix is an OpenClaw-owned or upstream-supported guard that prevents native clipboard loading in gateway/background embedded Pi paths before `@mariozechner/pi-coding-agent` is imported, such as a documented `OPENCLAW_DISABLE_CLIPBOARD=1`/Pi equivalent, a Pi upgrade that makes clipboard explicit and lazy, or process isolation for native clipboard access. Add focused regression coverage proving Linux gateway boot and the first non-fast embedded agent/cron path do not require `@mariozechner/clipboard` when the guard is enabled, then document the supported workaround.
