## Summary

Keep open. Current `main` does not implement per-skill model overrides: the skill config/schema/update surfaces only support enablement, API keys, env, and generic config, while the docs describe skills as `SKILL.md` instructions read on demand under the active session model.

## Evidence

- **Skill config type has no model field:** `SkillConfig` only exposes `enabled`, `apiKey`, `env`, and `config`; there is no per-skill `model` override in the persisted config shape.
  - file: [src/config/types.skills.ts:3](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/config/types.skills.ts#L3)
  - command: `nl -ba src/config/types.skills.ts | sed -n '1,80p'`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Runtime schema rejects extra skill-entry fields:** `SkillEntrySchema` is strict and only allows `enabled`, `apiKey`, `env`, and `config`, so `skills.entries.<key>.model` is not part of current main's validated config contract.
  - file: [src/config/zod-schema.ts:149](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/config/zod-schema.ts#L149)
  - command: `nl -ba src/config/zod-schema.ts | sed -n '120,170p'`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Gateway skills.update API cannot write model overrides:** The gateway protocol and handler for `skills.update` only accept and patch `enabled`, `apiKey`, and `env`; there is no request field or write path for a skill-level model setting.
  - file: [src/gateway/protocol/schema/agents-models-skills.ts:228](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/gateway/protocol/schema/agents-models-skills.ts#L228)

  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Gateway handler only mutates enabled/apiKey/env:** The concrete `skills.update` implementation copies the existing entry and mutates only `enabled`, `apiKey`, and `env` before writing config back out.
  - file: [src/gateway/server-methods/skills.ts:300](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/gateway/server-methods/skills.ts#L300)

- **Skills are documented as on-demand prompt reads, not separate model runs:** Current docs say skill instructions are not injected by default and the model reads `SKILL.md` only when needed, which means current main does not already have a documented per-skill execution boundary where a temporary model swap would occur.
  - file: [docs/concepts/context.md:117](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/docs/concepts/context.md#L117)

  - file: [docs/tools/skills.md:159](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/docs/tools/skills.md#L159)

## Risks / Open Questions

- A future implementation needs a clear contract for when a skill-level override applies, because current docs describe skills as on-demand `SKILL.md` reads inside the active model turn rather than as isolated sub-runs.
- There may be adjacent plugin/subagent model-override mechanisms elsewhere in core, but they do not provide the requested `skills.entries.<skillKey>.model` behavior on current `main`.
