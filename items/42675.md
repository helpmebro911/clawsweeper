## Summary

Keep this PR open. Current main still allows Discord startup to continue with a missing bot identity when fetchUser("@me") throws or returns no usable id, and the PR is a focused implementation candidate for the linked open security bug. The submitted patch is stale against the current extension path and incomplete, so the best maintainer path is revision, not cleanup closure.

## Best Possible Solution

Keep the PR open, request a rebase onto current main, and revise it to fail Discord startup from extensions/discord/src/monitor/provider.startup.ts when fetchUser("@me") rejects or returns no normalized id. Add focused tests for both rejection and invalid-identity cases, then review whether the preflight botId && mention-gate guard should remain as channel fallback defense or be tightened.
