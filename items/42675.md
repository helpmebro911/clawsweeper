## Summary

Keep open. Current `main` still allows the Discord provider to continue with `botUserId` unset after `fetchUser("@me")` fails, so the underlying security/routing problem is not implemented on `main` and cannot be closed. The submitted PR is also stale/incomplete: its one-line throw targeted the old `provider.ts` location, while current startup logic lives in `provider.startup.ts` and still returns `undefined` on failure without aborting startup.

## Evidence

- **Startup still swallows bot identity fetch failure:** `fetchDiscordBotIdentity()` calls `client.fetchUser("@me")`, but its `catch` block only logs and returns `{ botUserId: undefined, botUserName: undefined }`. That means current main still proceeds without a validated bot identity after a fetch failure.
