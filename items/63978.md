## Summary

Keep this PR open. Current main still reads a selected transcript and extracts usage without validating that a parseable session header belongs to the requested session id, and the PR remains a focused candidate for that gap.

## Best Possible Solution

Keep this PR open and review or land it after minor fixture cleanup and any needed rebase. The best implementation is to make `readLatestSessionUsageFromTranscript` return `null` when a discovered transcript has a parseable `{ type: "session", id }` header for a different session, while preserving legacy/custom transcripts without a parseable session header, plus a regression test for the post-reset stale transcript scenario.
