## Summary

Keep open. Current `main` still allows REM output to cite stale `memory/.dreams/session-corpus/*.txt` paths directly from the short-term recall store, and the built-in repair path does not prune recall entries whose backing corpus files were deleted. Some follow-up symptoms are partially addressed on `main` (role-word blacklist in REM tags, cron transcript exclusion), but the stale-corpus/re-pin claim is not convincingly fixed. The report's `recallCount: 0` diagnosis is also partly overstated on current `main`, because deep ranking gates use total signal count, not `recallCount` alone.

## Evidence

- **REM preview still trusts stored evidence paths verbatim:** `selectRemCandidateTruths` and `buildRemReflections` both emit evidence from `entry.path:start-end` with no file-existence check or rehydration step. If the recall store still contains entries for an older corpus file, REM can keep citing that deleted path.
  - file: [extensions/memory-core/src/dreaming-phases.ts:1395](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/dreaming-phases.ts#L1395)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **REM sweep feeds raw recall entries straight into preview and reinforcement:** `runRemDreaming` reads short-term recall entries, filters only by lookback, then passes them directly to `previewRemDreaming` and records REM hits from the selected keys. There is no source-file validation between store read and REM output.
  - file: [extensions/memory-core/src/dreaming-phases.ts:1614](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/dreaming-phases.ts#L1614)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Repair path normalizes store metadata but does not drop missing-source entries:** `repairShortTermPromotionArtifacts` removes stale locks and rewrites normalized counters/tags, but it does not check whether each entry's `path` still exists on disk. Clearing `memory/.dreams/session-corpus/` therefore would not, by itself, remove stale REM evidence from the recall store.
  - file: [extensions/memory-core/src/short-term-promotion.ts:1821](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/short-term-promotion.ts#L1821)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Deep promotion is not gated by recallCount alone:** `totalSignalCountForEntry` sums `recallCount + dailyCount + groundedCount`. On current `main`, a candidate with `recallCount: 0` can still satisfy the deep `minRecallCount` gate if it accumulated daily or grounded signals, so that part of the report is not sufficient reason to close or dismiss the item.
  - file: [extensions/memory-core/src/short-term-promotion.ts:390](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/short-term-promotion.ts#L390)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Main has only partial symptom hardening, not a stale-path fix:** Current `main` blacklists scaffold tags like `assistant/user/system/the` from REM reflections and excludes cron-generated transcripts from session-corpus ingestion. That helps with the follow-up 'assistant' theme artifact, but it does not address stale recall entries or phantom corpus paths.
