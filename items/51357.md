## Summary

Keep open. Current `main` still treats a successful WebSocket connect followed by a read-probe timeout/error as not reachable, which matches the reported false-negative class rather than fixing it.

## Evidence

- **Probe flips to failure after handshake:** After `hello-ok` sets `connectLatencyMs`, the probe arms a second timer and settles `{ ok: false, error: "timeout" }` if follow-up detail RPCs do not finish in time.
  - file: [src/gateway/probe.ts:262](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/probe.ts#L262)
  - command: `nl -ba src/gateway/probe.ts | sed -n '240,320p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Reachability excludes post-connect read failures:** `isProbeReachable` only returns true for `probe.ok` or the narrow `missing scope: operator.read` case. A connected probe with another error/timeout is still treated as unreachable.
  - file: [src/commands/gateway-status/helpers.ts:272](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/gateway-status/helpers.ts#L272)
  - command: `nl -ba src/commands/gateway-status/helpers.ts | sed -n '260,310p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **CLI exit/result depends on that narrow reachability check:** `writeGatewayStatusJson` derives top-level `ok` from `isProbeReachable(...)` and exits non-zero when no targets satisfy it, so post-connect read-probe timeouts still become overall probe failures.
  - file: [src/commands/gateway-status/output.ts:100](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/gateway-status/output.ts#L100)
  - command: `nl -ba src/commands/gateway-status/output.ts | sed -n '85,150p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Docs promise broader reachability semantics:** The CLI docs say `Reachable: yes` means a target accepted a WebSocket connect, and exit is non-zero only when no target is reachable. That does not match the current implementation for post-connect read timeouts/errors.
  - file: [docs/cli/gateway.md:224](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/docs/cli/gateway.md#L224)

- **Tests currently encode the false-negative behavior:** `gateway-status` output tests explicitly treat a probe with `connectLatencyMs: 40` plus `error: "unknown method: status"` as an unreachable probe for summary purposes.
  - file: [src/commands/gateway-status/output.test.ts:79](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/gateway-status/output.test.ts#L79)
