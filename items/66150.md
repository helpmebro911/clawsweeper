## Summary

Keep this PR open. Current main still has no equivalent ingress filter for operational alert text before delivery-mirror transcript persistence; it trims and writes any non-empty mirrored text. Main does have downstream replay/display hygiene for `delivery-mirror` entries, but that is not the same as preventing stale backup/git alert text from being stored. The PR remains a focused implementation candidate, and the author already addressed the initial review concern by narrowing the broad Spanish `Error en` match.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or revise a narrow, regression-tested filter for non-user-facing operational alerts, or move the suppression closer to the backup/git alert producer if maintainers do not want owner-specific strings in the generic transcript mirror layer. Broader replay and duplicate delivery work can remain tracked separately by #51628.

## Risks / Open Questions

- The proposed filter is content-destructive: any legitimate mirrored assistant text matching the hard-coded patterns would be silently dropped as `empty text`, so maintainers should review the final false-positive surface before merge.
- The alert strings look backup/git-owner-specific; maintainers should decide whether suppression belongs in the generic transcript mirror resolver or closer to the alert-producing owner boundary.
- No supply-chain or CI security regression was visible in the provided diff; the main security/privacy consideration is intentional suppression of transcript content.
