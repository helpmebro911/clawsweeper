## Summary

Keep this PR open. Current main still persists delivery-mirror text through `resolveMirroredTranscriptText` without an operational-alert filter, and the existing replay hygiene only drops delivery-mirror entries from in-memory provider replay after they are already persisted. The PR remains a focused implementation candidate; the earlier broad Spanish `Error en` false-positive was addressed in the PR diff by requiring the warning prefix and adding a regression assertion.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or revise a narrow, regression-tested suppression for non-user-facing operational alert mirrors, or move the filtering closer to the backup/git alert producer if maintainers do not want owner-specific strings in the generic transcript mirror layer. Broader delivery replay and duplicate work should remain tracked separately by #51628.

## Risks / Open Questions

- The proposed filter is intentionally content-destructive: false positives would silently drop legitimate mirrored assistant text as `empty text`, so maintainer review should focus on matching scope and observability.
- The alert strings are backup/git-owner-specific. Maintainers should decide whether suppression belongs in the generic transcript mirror resolver or closer to the alert-producing owner boundary.
- Broader delivery replay and duplicate behavior remains separate from this narrow PR and is still tracked by open issue #51628.
