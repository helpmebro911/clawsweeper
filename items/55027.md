## Summary

Current main still does not implement true progressive Feishu block delivery. The Feishu dispatcher still suppresses non-card block payloads, routes accepted card-mode blocks into the Feishu streaming-card session, and hard-disables core block streaming, so the user problem remains open despite related duplicate-card fixes.

## Best Possible Solution

Keep this open for a Feishu plugin-owned fix. The best path is to make Feishu preview-card streaming and block-message streaming separate modes, honor an explicit Feishu block-streaming config by allowing core block delivery when enabled, deliver `info.kind === "block"` through normal Feishu message/card sends immediately instead of Card Kit playback, and update Feishu schema/manifest docs plus regression tests for both disabled and enabled block delivery.

## Risks / Open Questions

- No live Feishu API delivery test was run during this read-only review, but the source and tests show the requested path is disabled or redirected before platform API behavior would matter.
- A fix needs to preserve the recent duplicate streaming-card/final suppression behavior while separating true block replies from streaming-card preview delivery.
- Feishu docs currently describe `blockStreaming`, while the Feishu plugin schema and dispatcher do not expose or honor it as true progressive block delivery; implementation and docs/config metadata should be aligned together.
