## Summary

Keep #10960 open. Current main still only attempts messaging-channel steer injection when the active run reports streaming; non-streaming active runs reject queued injection and active steer-mode input falls back to follow-up queue behavior. The docs still describe next-boundary/fallback semantics rather than a clean mid-stream pause/abort/resume soft steer, and the related implementation PR #64758 is still presented in the provided GitHub context as open and unmerged.

## Best Possible Solution

Keep #10960 open as the canonical product/runtime tracker. The complete fix should live in the auto-reply queue and embedded/reply-run control path: when steer-mode input arrives during active non-streaming work, tool waits, or generation, OpenClaw should cleanly pause or abort the active work, preserve usable partial context, append the new steering message, and resume in the same session. Regression coverage should cover streaming steer success, non-streaming/tool-wait fallback, steer-backlog/followup behavior, and the distinction from explicit gateway `sessions.steer`.
