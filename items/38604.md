## Summary

Keep open. Current main still does not set a default Docker sandbox `pidsLimit`; the resolved config only preserves explicit agent/global values, Docker args emit `--pids-limit` only when a positive value is already present, and docs/tests still cover configuration examples rather than a runtime default. The related open PR #44098 remains a possible implementation path, but it is not merged or shipped.

## Best Possible Solution

Keep #38604 open until a focused core fix lands. The likely path is to add a maintainer-approved `DEFAULT_SANDBOX_PIDS_LIMIT`, apply it in `resolveSandboxDockerConfig` while preserving explicit agent/global overrides, add regression coverage for default and override behavior, ensure `buildSandboxCreateArgs` emits the resolved default, and update the public sandbox/config docs. PR #44098 can remain the canonical implementation path if maintainers want to continue it; otherwise land an equivalent focused patch.

## Risks / Open Questions

- Closing now would leave default Docker sandbox containers without a core PID-limit fallback unless users explicitly configure `pidsLimit`.
- The docs example may continue to imply a PID limit exists by default even though runtime behavior only applies it when configured.
- Maintainers still need to choose and approve a compatible default value, then verify override behavior and documentation.
