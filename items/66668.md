## Summary

Keep open. Current `main` does not support the issue's requested outcome end-to-end: linked installs are intentionally not deleted on uninstall, manually copied plugins in `~/.openclaw/extensions` are still not uninstallable unless they have managed config/install records, and the docs still read broadly enough to make the report plausible.

## Evidence

- **Linked installs reuse the source path instead of a managed copied install:** `plugins install --link` persists a path install record with `sourcePath` and `installPath` both set to the resolved local path, so current main does not treat `--link` as a copied state-dir install.
  - file: [src/cli/plugins-install-command.ts:412](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/cli/plugins-install-command.ts#L412)
  - command: `nl -ba src/cli/plugins-install-command.ts | sed -n '399,418p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Uninstall refuses unmanaged plugins:** The CLI exits when a discovered plugin is not present in `plugins.entries` or `plugins.installs`, which means a plugin manually copied into the auto-discovered extensions root still cannot be uninstalled through `openclaw plugins uninstall`.
  - file: [src/cli/plugins-cli.ts:555](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/cli/plugins-cli.ts#L555)
  - command: `nl -ba src/cli/plugins-cli.ts | sed -n '547,564p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Core uninstall intentionally skips deleting path-based installs:** `resolveUninstallDirectoryTarget` returns `null` for `source === "path"`, and the uninstall implementation documents that linked plugins never have their source directory deleted.
  - file: [src/plugins/uninstall.ts:40](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/plugins/uninstall.ts#L40)
  - command: `nl -ba src/plugins/uninstall.ts | sed -n '24,310p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Tests lock in preserving linked plugin directories:** The uninstall tests explicitly expect linked plugins to keep their directory and only remove the load-path/config state.
  - file: [src/plugins/uninstall.test.ts:687](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/plugins/uninstall.test.ts#L687)
  - command: `nl -ba src/plugins/uninstall.test.ts | sed -n '650,735p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Docs still make the report plausible:** The CLI docs still say uninstall removes the plugin install directory under the active state-dir plugin root, while the configuration reference says plugins are loaded from `~/.openclaw/extensions`; together that leaves the manual-copy case looking supported from a user perspective.
  - file: [docs/cli/plugins.md:240](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/docs/cli/plugins.md#L240)
  - command: `nl -ba docs/cli/plugins.md | sed -n '214,252p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Prior discussion claimed a fix, but current main does not satisfy the filed expectation:** The issue timeline links PR #66737 and has a comment saying "Fixed, closing.", but current main still deliberately preserves linked directories and blocks uninstall for unmanaged manual copies, so there is not strong enough evidence to close conservatively.
