## Summary

Current main still matches the reported Discord native voice output-contract gap. Discord VC sends transcribed speech into the agent as normal Discord ingress, then requires non-empty returned text before synthesizing playback; the tool policy still only hides `tts` for the `voice` provider, so a tool-only `tts` turn plus silent final payload remains plausible and unaddressed.

## Best Possible Solution

Keep this open and implement a native Discord VC spoken-output contract. The fix likely belongs in `extensions/discord/src/voice/manager.ts` and/or the generic embedded-agent ingress context: preserve Discord routing metadata, but run VC turns under a voice-specific output/tool policy so `tts` is hidden and the agent is prompted to return spoken text for DiscordVoiceManager to synthesize and play. Add focused regression coverage for the VC segment policy and for tool-only `tts` plus `NO_REPLY` or empty final payload not silently satisfying the turn.
