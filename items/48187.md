## Summary

Keep open. Current main still does not expose Telegram topic routing/binding state in generic `/status`; it only reports the generic session line plus existing verbose-only plugin debug status. The underlying Telegram configured/runtime topic binding state exists and is covered by tests, so the PR addresses a real visibility gap, but it is stale against the current status architecture and should be rebased or rewritten rather than closed.

## Best Possible Solution

Keep the PR open, but ask for a focused rebase/rewrite: implement Telegram topic routing visibility on the current status path, showing the topic conversation id, delivery target, configured ACP binding, live runtime binding, and drift warning for Telegram forum-topic `/status` requests. Drop unrelated gaxios/CI changes and add focused tests in the current `src/status/*` and Telegram native-command status surfaces.
