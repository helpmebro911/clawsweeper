## Summary

Current main still lacks the requested public session lifecycle management. There are useful lower-level primitives such as `sessions.abort`, `sessions.delete`, `sessions.reset`, `sessions.compact`, token display, and session-store cleanup, but no `openclaw sessions stop/kill/pause/resume` CLI commands, no Sessions-page Stop/Pause/Resume controls, and no active resource quota enforcement for running sessions. The related open PR #35735 only covers deletion cleanup (`rm`/`clear`) and does not supersede this broader lifecycle request.

## Best Possible Solution

Keep this issue open as the broad session lifecycle tracker. Implement a narrow `openclaw sessions stop <session-key>` first, likely backed by existing `sessions.abort`, with CLI docs and tests. Coordinate delete/rm/clear cleanup with #35735. Defer pause/resume and resource quotas until the Gateway protocol, runtime state, harness/provider cancellation behavior, and cost/accounting model can enforce them. Add Sessions-page controls only when each button maps to real Gateway behavior.

## Risks / Open Questions

- Closing now would hide a real public CLI and Sessions UI lifecycle gap that lower-level Gateway primitives only partially address.
- A pause/resume implementation that only flips metadata would be misleading unless Gateway runtime, harnesses, and providers can actually enforce paused and resumed execution states.
- The existing `sessions.abort` primitive looks like the right backend for a narrow `sessions stop` command, but it still needs command wiring, docs, tests, and UI decisions.
