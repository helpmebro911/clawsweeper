## Summary

Keep this issue open. Current main has prompt guidance and partial NO_REPLY hardening in shared helpers, auto-reply, and cron delivery, but the gateway live-chat path still does not strip a standalone trailing NO_REPLY suffix from otherwise substantive text before delta/final chat delivery.

## Best Possible Solution

Keep the issue open and implement a conservative shared cleanup contract for gateway chat/display output. The likely fix belongs in the gateway live-chat projection/buffering path, reusing existing token helpers where appropriate: exact NO_REPLY suppresses, malformed leading control-token output follows the existing lead-fragment rules, and a standalone trailing NO_REPLY suffix on otherwise substantive text is stripped before final and delta delivery. Add regression tests for exact, leading, glued-leading, trailing suffix, newline suffix, mixed case, and non-trailing explanatory mentions; then decide whether outbound payload normalization should adopt the same behavior repo-wide.

## Risks / Open Questions
