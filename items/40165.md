## Summary

Keep this issue open. Current main still does not strip a standalone trailing `NO_REPLY` in the gateway live chat/TUI projection path. Exact-token suppression, lead-fragment suppression, and glued leading-token stripping exist, and a shared terminal-token helper exists elsewhere, but the gateway projector still emits `Task complete. NO_REPLY` unchanged.

## Best Possible Solution

Implement the missing gateway display cleanup in `projectLiveAssistantBufferedText` or a shared display normalizer used by both gateway live chat and TUI embedded chat. Reuse the existing standalone terminal-token behavior from `stripSilentToken`, then add regression tests for exact suppression, separated leading malformed output, glued-leading stripping, trailing suffix (`Task complete. NO_REPLY`), newline suffix (`Git is clean.\nNO_REPLY`), mixed case, and embedded/non-terminal mentions that must remain visible.

## Risks / Open Questions

- A broad suffix-strip rule could remove legitimate explanatory mentions of `NO_REPLY`; use a standalone terminal-token rule and keep tests for embedded/non-terminal mentions.
- Gateway live chat, TUI embedded chat, chat history, auto-reply delivery, and cron/announce delivery currently have different silent-token contracts, so the fix should deliberately scope whether this is display normalization or a shared delivery contract.
