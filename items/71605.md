## Summary

Keep this issue open. Current main still has the same CLI/Gateway shape described in the report: `openclaw agent` waits for the final Gateway `agent` response, falls back to same-state embedded execution on Gateway failure, and embedded runs still acquire the same per-session transcript lock with the 10s default wait. The related #62981 tracks the lock-contention subset, but it does not cover the broader fresh-session Gateway dispatch timeout reported here.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The likely fix is to split the CLI path into a fast `agent` accept followed by explicit `agent.wait` final waiting, then prevent same-state embedded fallback while a running Gateway owns the session directory or add explicit alternate state-dir support. Update the public `openclaw agent` and agent-send docs, and add regression coverage for Gateway timeout handling plus embedded fallback/session-lock contention. Coordinate with #62981 for the lock-contention subset rather than closing this broader report.
