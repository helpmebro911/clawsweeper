## Summary

Keep #71605 open. The issue is recent, concrete, and current `main` still matches the core reported shape: `openclaw agent` waits on the Gateway `agent` RPC final response with a timeout derived from `--timeout`, falls back to embedded execution on Gateway failure, and the embedded runner still uses the session write lock with the 10s default acquisition timeout. The docs mention fallback but do not document a daemon/embedded same-state concurrency limitation. Related Discord issues are adjacent motivation, not canonical duplicates of this Gateway/embedded-lock report.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The best fix is to make `openclaw agent` avoid the brittle final-response wait by using the accepted `runId` plus `agent.wait` or equivalent lifecycle tracking, and to stop falling back to same-state embedded execution when a daemon owns the same state directory. The embedded path should either support an explicit isolated state dir, cooperate safely with daemon-held session locks, or fail fast with a clear daemon/state-dir message. Update `docs/cli/agent.md` and add regression coverage for the Gateway timeout/fallback and session-lock contention behavior.
