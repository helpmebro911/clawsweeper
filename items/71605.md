## Summary

Keep #71605 open. The report is recent, concrete, and still matches current main: the CLI Gateway path waits for the final `agent` response with a `(timeout + 30s)` wrapper timeout, then falls back to same-state embedded execution on Gateway failure; the embedded PI attempt still acquires the session transcript write lock with the default 10s acquisition timeout, and the docs still do not document a daemon-versus-embedded concurrency limitation.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. A maintainer fix should make the CLI/Gateway contract explicit, likely by returning the accepted `runId` promptly and using `agent.wait` or another tracked wait path for completion, or by otherwise preventing healthy Gateway runs from falling into a wrapper timeout. The embedded fallback should not silently run against the same state directory while the daemon may own session locks; add a daemon/state-dir guard, separate state-dir support, or a clear refusal/error, plus docs in `docs/cli/agent.md` and regression coverage for the Gateway timeout/fallback and 10s session-lock behavior.
