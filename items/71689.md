## Summary

Keep open. Current main still has the reported failure shape: task registry restore loads `tasks/runs.sqlite` through the default SQLite store and only catches/logs restore errors, with no corruption detection, bad-DB snapshot, `.recover`/rebuild path, or task-registry audit finding. The related ClawSweeper matches are tangential and do not supersede this concrete malformed-SQLite recovery request.

## Best Possible Solution

Keep the issue open and implement a focused task-registry recovery path in core: detect SQLite corruption on restore/open/read, close the cached handle, snapshot the bad `runs.sqlite` and sidecars, attempt a conservative recover/rebuild or quarantine-and-recreate flow, and surface an explicit operator/audit finding. Add regression coverage for the restore-failure path and, if feasible without brittle fixtures, a malformed SQLite image case.
