## Summary

Keep #71689 open. Current main still opens and restores `tasks/runs.sqlite` directly; malformed SQLite failures are caught only as a warning, with no corruption classifier, quarantine/snapshot, recovery/rebuild retry, or task-registry audit/operator signal. The related reports are adjacent registry/storage issues, not a canonical duplicate for corrupted `tasks/runs.sqlite` recovery.

## Best Possible Solution

Keep #71689 open as core task-registry storage resilience work. Add a narrow SQLite corruption recovery path around open/restore: classify malformed-image SQLite errors, close handles, preserve/quarantine `runs.sqlite` plus WAL/SHM sidecars, attempt bounded recovery or clean rebuild, reopen and restore recoverable rows, surface an explicit operator/audit signal, and add regression coverage with a corrupt DB fixture or injected `DatabaseSync` failure. Coordinate the operator signal with existing Task Flow `restore_failed` audit behavior, but do not close this as #66747 because that issue covers chmod `EPERM`, not SQLite image corruption.
