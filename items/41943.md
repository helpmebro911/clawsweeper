## Summary

Keep open. Current main still does not apply ambient HTTP(S) proxy handling to Feishu/Lark Axios HTTP requests; it only injects request timeouts for the Lark SDK HTTP instance and applies ambient proxy handling to the Feishu WebSocket client. This PR is still a plausible fix for the linked proxy bug, but it needs revision before merge for current-main drift, NO_PROXY behavior, tests, and plugin dependency metadata.

## Best Possible Solution

Revise this PR rather than closing it: add Feishu HTTP proxy support in extensions/feishu/src/client.ts using the shared proxy-env/proxy-agent helpers or an equivalent target-aware adapter, preserve NO_PROXY behavior for Feishu and Lark domains, account for proxy env changes in cache behavior if supported, add focused Feishu client tests for HTTPS_PROXY and NO_PROXY, and update Feishu plugin dependency metadata if a new direct runtime import remains.

## Risks / Open Questions

- Merging the PR as-is could route Feishu/Lark requests through a proxy even when NO_PROXY excludes the target host, because the proposed Axios options disable built-in proxy handling while unconditionally supplying an agent when a proxy env var exists.
- The PR branch was authored against an older Feishu client shape; current main now uses resolveAmbientNodeProxyAgent for WebSocket proxy handling, so the patch should be rebased and aligned with the shared proxy helper surface.
- If the PR imports https-proxy-agent directly without updating extensions/feishu/package.json, packaged/bundled plugin runtime dependency staging may miss the dependency even though the root package has it.
