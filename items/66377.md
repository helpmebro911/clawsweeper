## Summary

Keep open. Current main has useful deleted-agent safeguards, including session-store purging and deleted-agent rejection, but it does not implement the central multi-location cleanup/reset requested here. Agent deletion still only prunes core agent config, route bindings, agent-to-agent allow entries, session-store entries, and local workspace/agent/session directories; it does not remove or disable cron jobs for that agent, and it cannot directly clean a third-party Lossless Claw/LCM database without a plugin lifecycle contract. The related reports in the provided context are narrower or still open and do not supersede this issue.

## Best Possible Solution

Keep the issue open. Implement a scoped cleanup/repair path that extends `agents delete` and gateway `agents.delete`, or adds `agents reset --clean-state`, to archive core-owned sessions/transcripts and purge session-store entries, remove/disable or explicitly report cron jobs whose `agentId` matches the deleted agent, emit a plugin/context-engine lifecycle hook so Lossless Claw can clean its own `lcm.db`, and add doctor/session cleanup repair plus regression tests. Do not hard-code third-party LCM SQL in OpenClaw core.
