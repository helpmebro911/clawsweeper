## Summary

Keep open. Current main has partial deleted-agent hardening, but it still does not satisfy the issue's central request for full multi-location agent state cleanup or a first-class agent reset/repair path. Agent deletion prunes config, removes matching core session-store entries, and trashes agent/workspace/transcript directories, but it does not clean cron jobs targeting the deleted agent or invoke the richer session deletion lifecycle for plugin/context-engine/runtime-owned state.

## Best Possible Solution

Keep this issue open. The best path is a scoped cleanup/repair design: extend `agents delete` and Gateway `agents.delete`, or add `agents reset --clean-state`, so agent cleanup enumerates affected sessions through the shared session mutation lifecycle, removes/disables/reports cron jobs targeting the agent, and gives plugin/context-engine owners a versioned cleanup hook for session-owned persistence. Add regression coverage for CLI deletion, Gateway deletion, cron jobs targeting deleted agents, plugin/context-engine cleanup, and the reported corruption/error-message path.

## Risks / Open Questions

- The exact reported `TypeError: Cannot read properties of undefined (reading 'length')` was not reproduced in this read-only review; a fix should capture or synthesize a focused corruption fixture before closure.
- Cron cleanup needs product judgment: deleting an agent could remove matching jobs, disable them, or fail/report actionable cleanup.
- Plugin/context-engine-owned state should be handled through a generic lifecycle or repair seam, not by hard-coding a specific plugin database such as `lcm.db` in core.
