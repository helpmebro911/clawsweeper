## Summary

Keep open. Current main still only partially addresses deleted-agent state: `agents delete` and `agents.delete` prune core config/session-store entries and trash agent/session directories, but they do not remove cron jobs targeting the deleted agent, do not run per-session/plugin cleanup for purged sessions, and there is still no first-class agent reset/clean-state command. The issue’s multi-location cleanup/reset request remains actionable.

## Best Possible Solution

Keep this issue open. The best fix is a scoped core cleanup/repair path: extend `agents delete`/`agents.delete` or add `agents reset --clean-state` to purge core session-store/transcript state, remove/disable/report cron jobs whose `agentId` targets the deleted agent, run appropriate session/plugin lifecycle cleanup for purged sessions, and add a versioned context-engine/plugin cleanup hook so Lossless Claw and other engines can clean agent/session-owned state themselves. Add regression coverage for CLI deletion, Gateway deletion, cron jobs targeting the deleted agent, and plugin/context-engine cleanup behavior.

## Risks / Open Questions

- The exact reported `TypeError: Cannot read properties of undefined (reading 'length')` was not reproduced in this read-only review; a fix should add a logs-backed fixture or focused reproduction before closing.
- Cron cleanup needs maintainer product judgment: deleting an agent could remove matching jobs, disable them, or fail/report actionable cleanup.
- Plugin-owned LCM cleanup should be solved through a generic lifecycle/API seam or plugin-owned repair path, not hard-coded `lcm.db` SQL in OpenClaw core.
