## Summary

Keep open. On current `main` (`d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132`), the macOS speech paths still feed `SFSpeechRecognizer` the input device’s native tap format directly, with no mono/stereo downmix, channel-layout conversion, or `channelCount` guard beyond `> 0`. I found no evidence on `main` that issue #42533’s multi-channel-input failure has been implemented away, and the existing tests do not cover this case.

## Evidence

- **macOS push-to-talk still appends raw input-node format to Speech:** `VoicePushToTalk.startRecognition()` reads `input.outputFormat(forBus: 0)`, installs the tap with that format, and appends each buffer straight to `SFSpeechAudioBufferRecognitionRequest`. There is no `AVAudioMixerNode`, converter, or channel normalization in this path.
  - file: [apps/macos/Sources/OpenClaw/VoicePushToTalk.swift:256](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/apps/macos/Sources/OpenClaw/VoicePushToTalk.swift#L256)

No multi-channel handling is present here either.
