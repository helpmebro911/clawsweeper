## Summary

Current main still matches the reported macOS speech-input failure mode. The affected macOS PTT, Talk Mode, Voice Wake runtime, and Voice Wake tester paths continue to tap the hardware input format and append those raw buffers directly to SFSpeechAudioBufferRecognitionRequest, with no shared mono/downmix or Speech-compatible conversion layer wired into the app.

## Best Possible Solution

Keep this issue open and fix the macOS app speech input pipeline. Add a shared macOS voice/audio helper that converts hardware capture buffers from multi-channel input devices into a Speech-compatible mono format before appending to SFSpeechAudioBufferRecognitionRequest, wire it through VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester, and add focused conversion/downmix coverage plus manual validation on a >2-channel macOS input device.
