## Summary

Keep open. The report is concrete, recent, and still matches current main: macOS Push-to-Talk, Talk Mode, Voice Wake, and the wake tester all install taps with the input node's hardware format and append raw buffers directly to SFSpeechAudioBufferRecognitionRequest. I found a converter-backed SpeechPipeline in Swabble, but it is not wired into the macOS app Talk/PTT paths, and there is no focused regression coverage for multi-channel-to-mono speech input normalization.

## Best Possible Solution

Keep this issue open and fix it in the macOS app audio capture owner paths. A good fix would add a shared speech-input normalization helper used by VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester, converting multi-channel hardware input to a Speech-compatible mono format before appending buffers. Add focused tests for multi-channel format conversion/downmix behavior and manually validate on a multi-channel macOS audio interface before closing.
