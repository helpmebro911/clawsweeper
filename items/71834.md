## Summary

Keep open. The 2026.4.22 generated thinking-bundle syntax failure appears obsolete in current main, but the remaining Telegram webhook startup problem is still present in the current source: webhook mode binds the local listener, calls Telegram `setWebhook`, and closes the listener if that call fails. The generic channel supervisor then waits for restart backoff, matching the reported interval where the reverse proxy has no local listener and returns 502.

## Best Possible Solution

Keep the issue open and fix the Telegram webhook startup lifecycle in the Telegram plugin. The best direction is to decouple the local listener lifetime from Telegram webhook registration: once the local listener binds, keep it alive on recoverable `setWebhook` failures, retry registration with bounded backoff, and add a regression test that fails `setWebhook` once while proving the local health/webhook route remains reachable or comes up before retry backoff. The old 2026.4.22 bundle syntax failure can be treated as already superseded by later builds unless a separate published-tarball validation gap remains.
