## Summary

Current main consistently treats `OPENCLAW_HOME` as the home directory and `OPENCLAW_STATE_DIR` as the state-root override. The nested `.openclaw/.openclaw/workspace` path only appears when `OPENCLAW_HOME` is pointed at the `.openclaw` state dir itself, which does not match the documented/current contract. The PR also only special-cases workspace resolution, which would make workspace behavior diverge from agent/state path resolution on main.

## Close Comment

Closing based on a Codex review. Current `main` consistently treats `OPENCLAW_HOME` as the home-directory override, and `OPENCLAW_STATE_DIR` as the supported state-directory override: see `docs/help/environment.md:108-134`, `src/config/paths.ts:60-69`, and the expectation in `src/config/config.nix-integration-u3-u5-u9.test.ts:53-58`. `src/agents/workspace.ts:17-26` follows that same contract for the default workspace path. Under that supported configuration, the reported nested workspace path does not reproduce; it only appears when `OPENCLAW_HOME` is pointed at the `.openclaw` state dir itself. This PR also only special-cases workspace resolution, while agent/state paths still derive from `OPENCLAW_HOME` via `resolveStateDir()`, so landing it would make path resolution inconsistent. If there is still a current repro with `OPENCLAW_HOME` set to the actual home directory, or with `OPENCLAW_STATE_DIR` set, please open a fresh issue with the exact env values and resulting paths.
