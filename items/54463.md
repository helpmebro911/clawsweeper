## Summary

Keep this issue open. The current main branch still has the QMD traversal robustness gap described in the report: documentation now warns about workspace-visible temp repos and QMD scanner behavior, but runtime still registers plain QMD collections and delegates indexing to `qmd update` without OpenClaw-side symlink pruning, realpath cycle detection, or an exclusion/no-follow config surface. The issue discussion also contains maintainer direction to keep the runtime problem open after the docs correction plus a current-behavior controlled repro from the reporter.

## Best Possible Solution

Keep this issue open as a focused QMD traversal robustness item. Add the controlled symlink-loop repro as regression coverage, then make traversal cycle-safe/no-follow upstream or add OpenClaw-side pre-update pruning/exclusion before `qmd update` can recurse through workspace-visible temp monorepos. Keep the current docs workaround until a runtime fix ships.

## Risks / Open Questions

- Users who enable QMD with broad indexed workspace roots can still break indexing when visible temporary monorepos include recursive package-manager symlink structures.
- A general directory-exclusion feature would reduce exposure only if pruning happens before the scanner follows symlinks; otherwise cycle-safe traversal is still unproven.
- The precise fix point remains a maintainer decision between upstream QMD traversal behavior, OpenClaw-side pre-update pruning/exclusion, or both.
