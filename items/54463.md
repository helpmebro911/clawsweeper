## Summary

Keep this issue open. Current main still treats the QMD symlink-loop problem as a live runtime limitation: docs warn users to move workspace-visible temp repos under hidden directories or outside indexed QMD roots, while runtime still registers plain QMD collections and delegates indexing to `qmd update`. I found no source, schema, docs, or regression-test evidence of symlink pruning, realpath cycle detection, or a QMD no-follow/exclusion contract that would satisfy the reported `ENAMETOOLONG` robustness gap.

## Best Possible Solution

Keep this issue open as a focused QMD traversal robustness item. Add the controlled symlink-loop repro as regression coverage, then make traversal cycle-safe/no-follow upstream or add an OpenClaw-side pre-update pruning/exclusion layer before `qmd update` can recurse through workspace-visible temp monorepos. Keep the current docs workaround until a runtime fix ships.

## Risks / Open Questions

- Users who enable QMD with broad indexed workspace roots can still break indexing when workspace-visible temporary monorepos include recursive package-manager symlink structures.
- A directory-exclusion feature would only satisfy this issue if pruning happens before QMD/Bun traversal follows symlinks; post-yield filtering would not prevent path expansion.
- The precise fix point remains a maintainer decision between upstream QMD traversal behavior, OpenClaw-side pre-update pruning/exclusion, or both.
