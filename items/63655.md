## Summary

Current main does not resolve the reported ZhiPu/unknown-parameter case. The bundled memory-lancedb plugin still always sends `encoding_format: "float"`; that addresses the earlier Ollama/base64 mismatch but conflicts with the reporter's follow-up that some providers reject the `encoding_format` key entirely.

## Best Possible Solution

Keep this issue open. Implement a memory-lancedb embedding compatibility mode in the plugin-owned config and manifest, with a provider-safe omit/raw-request path that does not let the OpenAI SDK inject `encoding_format`, normalizes float-array and base64/string responses explicitly, and adds regression tests for both forced-float providers and providers that reject the parameter.

## Risks / Open Questions

- A fix must preserve the already-landed float-encoding behavior for Ollama/base64 dimension mismatches while adding a provider-compatible path for APIs such as ZhiPu that reject unknown `encoding_format` parameters.
- Simply deleting `encoding_format: "float"` from the current SDK call is likely insufficient because the pinned OpenAI SDK injects `encoding_format: "base64"` when the field is omitted.
