## Summary

Keep this issue open. Current main still does not strip, parse, or execute the reported WhatsApp-visible `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo tool call. Existing coverage handles XML `<tool_call>` blocks, line-oriented bracketed JSON tool blocks, and `[Tool Call: ...]` downgraded markers, but not the exact uppercase bracket form reported here; WhatsApp delivery still sends normalized reply text through `msg.reply`.

## Best Possible Solution

Keep this issue open. Add a shared assistant-visible runtime/delivery fix for `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]`: safely convert it into a structured tool call if OpenClaw can do that deterministically, or suppress it from user-visible replies when it is only leaked scaffolding. Add focused regression coverage in `src/shared/text/assistant-visible-text.test.ts`, likely `src/shared/text/tool-call-shaped-text.test.ts`, and a WhatsApp auto-reply/block delivery test.

## Evidence

- **Reporter context is concrete:** The provided issue body and follow-up comments describe a reproducible WhatsApp auto-reply failure on OpenClaw 2026.4.5 with `dmPolicy=pairing`: model output appears as `[TOOL_CALL]{tool => "web_search", args => {...}}[/TOOL_CALL]` text across providers, while structured tool calls work in the main agent path.
- **Shared sanitizer does not match the reported uppercase bracket form:** The assistant-visible sanitizer quick-checks XML-style `<tool_call>`, `<tool_result>`, `<function_call(s)>`, `<function>`, and `<tool_calls>` tags, and its downgraded marker path checks `[Tool Call]`, `[Tool Result]`, and `[Historical context]`; it does not include `[TOOL_CALL]...[/TOOL_CALL]`.
