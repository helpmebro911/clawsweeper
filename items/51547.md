## Summary

Keep open. Current `main` does not implement the Windows GBK/code-page fix in `runExec`, and this PR’s proposed approach is unsafe because it relies on unsupported `"gbk"` as a Node `BufferEncoding`. The underlying bug still appears unresolved on the `runExec` path, but this PR is not a strong close candidate because it is neither implemented on `main` nor obsolete.

## Evidence

- **Current main still hardcodes UTF-8 in runExec:** `runExec` in `src/process/exec.ts:131`-`151` passes `encoding: "utf8"` to `execFileAsync` for both numeric and object options, so the PR’s requested behavior is not already on `main`.
  - file: [src/process/exec.ts:131](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/process/exec.ts#L131)
  - command: `nl -ba src/process/exec.ts | sed -n '130,160p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Proposed GBK encoding is not supported by Node BufferEncoding:** Local Node check returned `Buffer.isEncoding('gbk') === false` and `Buffer.from([0xb2,0xe2]).toString('gbk')` throws `ERR_UNKNOWN_ENCODING Unknown encoding: gbk`. That matches the review concern that passing `encoding: "gbk"` to `execFile` is not a safe fix.
  - command: `node -e "console.log(process.version); console.log(Buffer.isEncoding('gbk')); try { Buffer.from([0xb2,0xe2]).toString('gbk'); } catch (err) { console.log(err && err.code, err && err.message); }"`
- **runExec callers expect strings immediately:** `src/link-understanding/runner.ts:68`-`72` destructures `stdout` from `runExec()` and immediately calls `stdout.trim()`. Any PR change that causes non-string output or throws during decoding would break downstream callers.
  - file: [src/link-understanding/runner.ts:68](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/link-understanding/runner.ts#L68)
  - command: `nl -ba src/link-understanding/runner.ts | sed -n '60,80p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Another exec path still decodes with default toString():** `runCommandWithTimeout` in `src/process/exec.ts:340`-`345` appends `d.toString()` for stdout/stderr, so even a `runExec`-only patch would not fully solve Windows non-UTF-8 output across process execution paths.
  - file: [src/process/exec.ts:340](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/process/exec.ts#L340)
  - command: `nl -ba src/process/exec.ts | sed -n '260,360p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Repo already has a better decoding pattern elsewhere, and docs still treat this as an open workaround:** `src/node-host/invoke.ts:146`-`165` decodes Windows buffers via `TextDecoder(encoding)` after code-page detection, which is the viable pattern for GBK-like encodings. Separately, `docs/help/faq-first-run.md:395`-`420` still documents garbled Chinese exec output as a current workaround/report issue, indicating the bug is not closed out on main.
  - file: [src/node-host/invoke.ts:146](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/node-host/invoke.ts#L146)
