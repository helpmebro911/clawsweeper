## Summary

Keep open. Current main still routes Telegram `/new` and other built-in slash commands through native `bot.command` handlers that do not reuse the regular inbound feedback setup. Regular Telegram messages build ack/status reactions and typing callbacks in the message pipeline; the native slash-command path creates its reply pipeline without typing callbacks and contains no reaction or chat-action calls. The provided cross-reference to security-for-ai-agent/openclaw#2 is useful context, but it is not merged into openclaw/openclaw main and does not supersede this issue.

## Best Possible Solution

Keep the issue open and implement Telegram native-command feedback in the Telegram plugin. Factor or reuse the regular message feedback helper so built-in `/new` and `/reset` native handlers, and ideally plugin native commands, can set the queued ack/status reaction and start typing after auth/context resolution but before slow command execution. Pass typing callbacks into the native command reply pipeline, finalize or clear reactions on success/error/no-delivery, and add focused tests for `/new` in a DM with `reactionLevel: "ack"` plus `ackReactionScope: "direct"`, including a negative unauthorized-command case.

## Risks / Open Questions

- A fix should decide whether ack reactions are emitted only after command authorization; reacting before authorization could acknowledge commands from blocked senders.
- Native Telegram commands have built-in, plugin, custom/menu, and callback-replay paths. The implementation should avoid duplicate feedback for callback/menu paths that already re-enter `processMessage`.
- Telegram status reactions should reuse existing allowed-reaction variant logic, `ackReactionScope`, `removeAckAfterReply`, and `messages.statusReactions` lifecycle behavior rather than adding a one-off reaction call.
