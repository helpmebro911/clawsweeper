## Summary

Keep this issue open. Current main still has no crontab-specific source, docs, tests, or history-backed fix for macOS `/usr/bin/crontab` failing through exec, and the `openclaw cron` CLI path still uses paired CLI Gateway authorization for admin-scoped cron mutations rather than same-host automatic cron authorization. The recent devices CLI local fallback is a useful terminal workaround for approving pending devices, but it does not fix raw system `crontab` exec failures or make cron mutations bypass pairing.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l` and `echo ... | crontab -` through OpenClaw exec on a current macOS LaunchAgent install, then either fix the exec environment/TTY/timeout behavior or document a supported alternative. Separately make an explicit product/security decision for same-host CLI cron administration: keep manual admin scope upgrades with documented `openclaw devices list/approve`, surface that terminal approval path directly in cron errors, or add a narrowly scoped same-host token/device reuse path that preserves the Gateway trust boundary.
