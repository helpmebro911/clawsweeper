## Summary

Keep this issue open. Current main still models Telegram group sessions as agent-scoped session keys and then applies generic session-tool visibility plus agent-to-agent policy; I did not find a shipped same-Telegram-group cross-agent symmetry rule or a regression test covering two agents in one Telegram group listing and sending to each other both directions. The related #59933 remains a separate open same-agent group-listing report, not a canonical fix for this cross-agent failure.

## Best Possible Solution

Keep this issue open. Reproduce with two configured agents in one Telegram group under explicit `tools.sessions.visibility: "all"` and enabled, allowlisted `tools.agentToAgent`. If the asymmetry reproduces, add regression coverage for both agents' `agent:<agent>:telegram:group:<id>` keys appearing in `sessions_list` and resolving through `sessions_send` both directions, then fix the gateway/session-store/visibility path. If same-group cross-agent sends are intentionally opt-in only, improve the tool error/docs so users get config guidance instead of a one-way `No session found`. Track #59933 separately because it is same-agent group listing, not this cross-agent mismatch.
