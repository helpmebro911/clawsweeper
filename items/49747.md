## Summary

Keep open. Current main has partial coverage for deriving announce targets from agent-prefixed Slack thread session keys, but the reported `resolveGroupSessionKey()` boundary bug remains: it still only treats the first `From` segment as the provider and otherwise falls back to `ctx.Provider`, so `agent:<id>:slack:channel:...` can still resolve under `webchat`. The provided related context also shows PR #49750 tracks this same Slack A2A routing fix and was previously kept open rather than superseded.

## Evidence

- **Parser still assumes provider is segment 0:** `resolveGroupSessionKey()` splits `ctx.From`, checks only `parts[0]` against deliverable surfaces, and chooses `providerHint` when the head is not a surface. For `agent:builder:slack:channel:...` with `ctx.Provider`/surface as webchat, this still selects webchat instead of the embedded slack provider.
