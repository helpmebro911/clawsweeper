## Summary

Keep open. Current `main` does not show this PR's ACPX-side fix, and the only auth-related ACPX code now present is a no-op bridge. I found provider-auth env stripping on the `openclaw acp client` path, but not an equivalent Codex-specific env scrub on the embedded ACPX runtime path described by the PR.

## Evidence

- **ACPX Codex auth bridge is a no-op:** `prepareAcpxCodexAuthConfig` ignores `stateDir`/`logger` and returns the plugin config unchanged; it does not inspect Codex auth mode, `CODEX_HOME`, or strip env keys.
  - file: [extensions/acpx/src/codex-auth-bridge.ts:3](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/extensions/acpx/src/codex-auth-bridge.ts#L3)
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Current ACPX tests cover non-copy behavior, not env scrubbing:** The tests assert that no Codex ACP auth home or copied `auth.json` is created and that explicit commands are preserved. There is no regression coverage for stripping `OPENAI_API_KEY`/`CODEX_API_KEY` based on Codex `auth_mode`.
  - file: [extensions/acpx/src/codex-auth-bridge.test.ts:39](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/extensions/acpx/src/codex-auth-bridge.test.ts#L39)
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Env stripping exists on ACP client path only:** `createAcpClient` builds `stripKeys` and passes them to `resolveAcpClientSpawnEnv`, which removes provider auth env vars and sets `OPENCLAW_SHELL=acp-client`. This is the CLI ACP client path, not the embedded ACPX runtime path from the PR.
  - file: [src/acp/client.ts:128](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/acp/client.ts#L128)
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **No local ACPX runtime seam for spawn env customization:** The embedded ACPX wrapper just instantiates external `acpx/runtime` delegates. The local compat type for `AcpRuntimeOptions` includes cwd/sessionStore/agentRegistry/probe settings, but no env override hook that would implement this PR's `resolveAcpxSpawnEnv` behavior in repo code.
  - file: [extensions/acpx/src/runtime.ts:218](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/extensions/acpx/src/runtime.ts#L218)
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Search shows ACPX production code lacks the proposed auth/env handling:** Repo search for `OPENAI_API_KEY`, `CODEX_API_KEY`, `auth.json`, `chatgpt`, and related symbols under `extensions/acpx` found only the no-op bridge plus tests; the only provider-env stripping implementation is under `src/acp/client*`.
  - command: `rg -n "OPENAI_API_KEY|CODEX_API_KEY|ACPX_AUTH_OPENAI_API_KEY|ACPX_AUTH_CODEX_API_KEY|CODEX_HOME|auth.json|chatgpt" extensions/acpx src/acp`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)

## Risks / Open Questions

- `acpx/runtime` is an external dependency (`extensions/acpx/package.json` depends on `acpx` 0.5.3), and its installed source was not vendored in this checkout. If that package now performs internal env scrubbing, this local inspection would not see it.
- GitHub CLI inspection was unavailable here because `GH_TOKEN` was not set, so I relied on the provided PR context plus the checked-out `main` tree.
