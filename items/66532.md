## Summary

Keep open. Current main still has the embedded-run abort/wait/active/queue helpers internally, and it has adjacent Gateway `chat.abort`/`sessions.abort` wiring for agent RPC runs, but the requested in-process plugin runtime contract is still missing from `api.runtime.agent`. The CoClaw plugin context in the issue remains a valid example of a plugin needing a stable runtime surface instead of a global-state side door.

## Best Possible Solution

Keep this issue open and implement the additive plugin runtime contract: re-export the embedded-run abort/wait/isActive/queue helpers from `src/plugins/runtime/runtime-embedded-pi.runtime.ts`, lazy-bind them in `createRuntimeAgent()`, declare them on `PluginRuntimeCore["agent"]`, and update runtime shape tests plus public plugin runtime docs/API inventory. Treat the shipped Gateway abort path and harness-only aliases as related surfaces, not substitutes for the requested in-process plugin API.

## Risks / Open Questions

- Exposing abort and queue controls is an additive trusted-runtime capability change; maintainers should review naming and ownership semantics, likely deciding whether to expose the neutral EmbeddedAgent aliases, the Pi-named helpers, or both.
- Leaving this unresolved keeps plugin authors dependent on unstable global state or a Gateway round trip for cancellation flows, which is exactly the fragility described by the CoClaw plugin report.
