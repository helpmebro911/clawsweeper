## Summary

Keep open. Current main no longer appears to bypass the Feishu plugin for `message(action=send)`, but it still does not detect native Feishu interactive-card JSON passed as the plain `message` string. The plugin send handler builds interactive cards only from the semantic `presentation` parameter; a raw JSON-looking `message` falls through to the normal post/text transport, which matches the reported remaining regression.

## Best Possible Solution

Keep this issue open for Feishu-owned maintainer handling. The narrow implementation path is to add a conservative native Feishu card JSON detector in `extensions/feishu/src/channel.ts` for `send` and `thread-reply` text/message params, route valid v1/v2/wrapped interactive card payloads to `sendCardFeishu`, and add regression coverage. The alternative is an explicit maintainer decision and docs update that raw provider-native Feishu card JSON in `message` is unsupported and users should send semantic `presentation` payloads instead.

## Risks / Open Questions

- A compatibility detector must be conservative so ordinary JSON text is not silently converted into Feishu interactive cards.
- Maintainers may intentionally choose semantic `presentation` as the only supported proactive-card API, but that needs an explicit product/docs resolution because the report describes prior raw-card behavior as a regression.
