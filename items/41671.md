## Summary

Current main does not implement the requested fix. Telegram network config is honored for the initial dispatcher, but the retry path still arms an IPv4-only sticky fallback for direct/env-bypass transports and the tests currently assert that behavior even when network config is explicit. The provided GitHub context also shows an open PR, #42059, proposing the missing "skip runtime IPv4 fallback when channels.telegram.network is explicitly configured" change, so this issue should stay open until that fix lands or is replaced.

## Best Possible Solution

Keep this issue open and either land a narrow fix or split the relevant part from #42059: in `extensions/telegram/src/fetch.ts`, do not arm the sticky IPv4 fallback when `channels.telegram.network.autoSelectFamily` or `dnsResultOrder` was explicitly configured for the account, or replace it with a symmetric policy that preserves explicit IPv6-capable config. Add regression coverage for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"` on both API and media download dispatcher paths.
