## Summary

Keep open. Current main still has binary/allowlist-based exec approvals, but no path-keyed RWX filesystem policy for read/write/edit/exec. The two linked implementation PRs in the provided GitHub context (#42653 and #42780) were closed unmerged, and the broader open #48532 does not supersede this exact feature request.

## Evidence

- **Exec approvals schema lacks filesystem RWX policy:** `ExecApprovalsAgent` only extends exec defaults with `allowlist`, and `ExecApprovalsFile` only defines `version`, `socket`, `defaults`, and `agents`; there is no `permissions.filesystem`, `rules`, `deny`, `default`, or sidecar access-policy schema.
  - file: [src/infra/exec-approvals.ts:129](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/infra/exec-approvals.ts#L129)

  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Allowlist entries remain executable-pattern based:** `ExecAllowlistEntry` stores a command pattern and optional `argPattern`/usage metadata, not path-scoped RWX permissions for target filesystem resources.
  - file: [src/infra/exec-approvals.types.ts:1](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/infra/exec-approvals.types.ts#L1)
