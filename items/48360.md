## Summary

Keep open. Current main has related probe/status improvements, but it still does not fully satisfy this issue's core request: a target that accepted a WebSocket connection can still be classified as unreachable when the follow-up read/detail RPC fails for a non-scope reason such as timeout. The issue discussion also framed this as a broader gateway health visibility gap, and the related daemon-status PR explicitly did not address the broader loopback probe/call inconsistency.
