## Summary

Current main does not automatically recover the reported failure mode. The browser plugin now has stronger CDP readiness checks and launch cooldowns, but an unresponsive process that still holds the managed CDP port still falls through to launch, where `ensurePortAvailable` rejects with `PortInUseError`; there is no owner-scoped stale Chrome process discovery/kill before launch. The linked attempted fix in #41758 was closed unmerged, and the comments only provide external/manual workarounds.

## Best Possible Solution

Keep this issue open and implement the recovery in the bundled browser plugin: before launching a local managed profile, probe the configured loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, identify only OpenClaw-owned Chrome/Chromium processes by `--remote-debugging-port` plus the managed `--user-data-dir`, terminate them with platform-specific fallbacks, wait for the port to clear, then launch fresh. Add focused unit tests for unhealthy port holder, healthy pre-existing managed browser, attachOnly/remote profiles, and refusal to kill unrelated listeners.

## Risks / Open Questions

- A future fix would add process-killing behavior, so it must be tightly scoped to local managed OpenClaw browser profiles, the expected user-data-dir, and the expected CDP port to avoid killing a user's normal Chrome or an intentionally external CDP service.
- The issue describes an unresponsive process, not necessarily a Unix zombie in the strict process-state sense; recovery should be based on short CDP health probes plus ownership checks rather than RSS or process-state heuristics alone.
