## Summary

Keep this PR open. Current main still treats Slack interactive reply-button clicks as queued Slack interaction system events plus UI confirmation; it does not proactively dispatch or wake an agent turn. The PR remains a plausible implementation candidate for the unresolved Slack interaction wake gap, but it overlaps with #61374 and the broader #61502 design discussion and needs maintainer choice on the canonical Slack interaction wake architecture.

## Best Possible Solution

Keep this PR open for maintainer review. Reconcile it with #61374 and #61502, choose the safest Slack interaction wake architecture, verify reply-mode/thread/session routing against the normal Slack inbound path, confirm the Slack authorization and dedupe behavior, then land the chosen implementation with focused Slack interaction tests and the changed gate or deliberately close superseded PRs after a canonical path is selected.
