## Summary

Keep this PR open. Current main still does not route BlueBubbles inbound text through an untrusted-content boundary, so the security hardening goal remains relevant. The submitted patch is not safe to merge as-is because it wraps only the legacy envelope `Body` value while the model-facing path still prefers raw `BodyForAgent`; the existing empty-body guard review is also valid. Security review found no supply-chain or infrastructure changes in the PR diff, but the prompt-boundary change is security-sensitive and needs targeted tests before landing.

## Best Possible Solution

Keep this PR open and request a targeted revision or replacement. The fix should protect the actual BlueBubbles model-facing text path, likely via `BodyForAgent` or a structured `UntrustedContext` approach, while preserving raw `RawBody`, `CommandBody`, and `BodyForCommands` for command parsing, reset handling, dedupe, and metadata. It should also add the Discord-style non-empty guard and focused BlueBubbles tests that prove inbound text reaches the agent with the intended untrusted-content boundary.

## Risks / Open Questions

- Merging this PR as-is would create a false sense of hardening because BlueBubbles text can still reach the model through raw `BodyForAgent`.
- The unconditional wrap can turn attachment-only or empty-body messages into marker-only prompt text, diverging from the Discord guard pattern already called out in review.
- The PR touches a security-sensitive prompt boundary without regression tests proving the actual model-facing BlueBubbles prompt is protected.
