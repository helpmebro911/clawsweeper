## Summary

Keep this PR open. Current main still computes and passes a CLI no-output watchdog timeout into normal child spawns even when the prompt is sent through stdin and stdin is closed, so the proposed behavior is not implemented or obsolete. The PR is a plausible focused fix, but it still needs functional revision and regression coverage before landing.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to revise the condition to key off raw stdin presence or explicit stdin mode, avoid misleading timeout text when the watchdog is disabled, add a CLI-runner regression test proving `noOutputTimeoutMs` is omitted for stdin-fed closed-stdin child spawns while `timeoutMs` remains set, and have maintainers decide whether the separate Claude live-session timer should keep or change its current behavior.

## Risks / Open Questions

- The PR diff shown in the provided review context keys off truthiness of `stdinPayload`; an explicit stdin-mode empty prompt would still arm the watchdog even though stdin is closed.
- The PR diff's `?? 0` timeout-message fallback can produce misleading `0s` no-output messages if an unexpected no-output branch is reached after the watchdog is disabled.
- The default Claude Code path now routes through `claude-stdio` live-session handling with its own no-output timer, so maintainers should decide whether the same behavior is intended there or only for normal child spawns.
- Disabling the no-output watchdog for stdin-fed child spawns makes silent hung backends rely on the overall timeout, so the change needs a focused regression test and ideally manual/live validation.
