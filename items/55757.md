## Summary

Keep PR #55757 open. Current main still resolves and passes a no-output timeout into the standard CLI runner spawn even when the prompt is carried over stdin, while the child adapter closes stdin after writing provided input. The PR addresses a real remaining behavior, though its exact patch should incorporate the review feedback about checking raw `stdin` rather than truthy `stdinPayload` and avoiding misleading `0s` timeout messages.

## Best Possible Solution

Keep the PR open and ask for a focused revision: disable `noOutputTimeoutMs` based on raw `stdin != null` for the standard one-shot CLI spawn path, keep `timeoutMs` as the runaway guard, avoid misleading `0s` no-output messages, and add a regression test proving stdin-mode or long-prompt stdin runs do not pass a no-output watchdog while normal interactive/live-session watchdog behavior remains covered.

## Risks / Open Questions

- The PR's current patch is slightly under-specified: `stdinPayload` truthiness misses empty stdin-mode prompts, as noted in the review comments.
- Disabling the no-output watchdog for stdin-delivered prompts leaves only the overall timeout for truly hung silent CLI backends, so the final implementation should preserve and test the overall timeout behavior.
- Related watchdog-cap items such as #40982/#41022 do not supersede this PR; they track timeout sizing, while this PR targets stdin-closed false positives.
