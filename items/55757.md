## Summary

Keep this PR open. Current main still passes a CLI no-output watchdog into normal child spawns even when the prompt is supplied through stdin and stdin is closed, so the PR is not obsolete or already implemented. The PR is a plausible focused fix, but it needs revision for the empty-stdin edge case, clearer timeout messaging, and regression coverage before it should land.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to revise the condition to use raw stdin presence or explicit stdin mode, avoid misleading timeout text when the watchdog is disabled, add a CLI-runner regression test proving `noOutputTimeoutMs` is omitted for stdin-fed closed-stdin child spawns while `timeoutMs` remains set, and have maintainers decide whether Claude live-session no-output behavior should remain separate.

## Risks / Open Questions

- The PR diff keys off truthiness of `stdinPayload`; an explicit stdin-mode empty prompt would still arm the watchdog even though stdin is closed.
- The PR diff's `?? 0` fallback can produce misleading `0s` no-output timeout text if an unexpected no-output branch is reached after the watchdog is disabled.
- The default `claude-cli` route can use `claude-stdio` live-session handling with its own no-output timer, so maintainers should decide whether this PR should affect only normal child spawns or also live sessions.
- Disabling the no-output watchdog for stdin-fed child spawns makes silent hung backends rely on the overall timeout, so the change needs a focused regression test and ideally manual or live validation.
