## Summary

Current `main` already mitigates the reported failure mode: session stores are now bounded by age/count/disk-budget maintenance, startup recovery is limited to orphaned subagent sessions instead of all `sessions.json` bindings, and those recovery resumes use the `subagent` lane with a 10s gateway call timeout rather than contending on `lane=main` indefinitely.

## Evidence

- **Session store maintenance exists for `sessions.json`:** `session.maintenance` now resolves retention/count/rotation/disk-budget controls, including `pruneAfter` and `maxEntries`, so stale bindings no longer accumulate indefinitely when enforcement is enabled.
  - file: [src/config/sessions/store-maintenance.ts:135](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/store-maintenance.ts#L135)

- **Oversized stores are pruned on load:** Loading a session store applies maintenance when the store exceeds the configured max entries, pruning stale entries and capping the store before use.
  - file: [src/config/sessions/store-load.ts:131](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/config/sessions/store-load.ts#L131)

## Close Comment

Closing after a Codex review because current `main` already addresses the reported startup-resume failure mode. `src/config/sessions/store-maintenance.ts:135`, `src/config/sessions/store-load.ts:131`, and `src/config/sessions/store.ts:287` now bound `sessions.json` by age/count/disk budget instead of letting stale entries grow without limit. Startup also no longer resumes every stored binding: `src/gateway/server-startup-post-attach.ts:352` only schedules orphaned subagent recovery, and `src/agents/subagent-orphan-recovery.ts:146` resumes only sessions that are still active in the subagent registry and marked `abortedLastRun`; those recovery calls use `lane: "subagent"` with a 10s timeout at `src/agents/subagent-orphan-recovery.ts:104`. Embedded reply runs are session-scoped via `src/agents/pi-embedded-runner/lanes.ts:3` and `src/auto-reply/reply/get-reply-run.ts:593`, so the originally reported behavior of resuming all sessions onto `lane=main` for 600000ms no longer matches current OpenClaw. This implementation is also present in release `v2026.4.22`. If you still hit a similar stall on `v2026.4.22` or newer, please open a fresh issue with current restart logs, the affected agent/session counts, and the exact lane/timeout lines you see now.
