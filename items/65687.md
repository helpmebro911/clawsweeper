## Summary

Current main still appears to leave the reported Discord voice Deepgram follow-up failure open. The media provider resolver can still return a partial active media-understanding registry before compat loading, Discord voice still uses that shared transcription path, and the existing tests still miss the partial-active-registry plus configured-Deepgram case described in the issue and prior ClawSweeper comment.

## Best Possible Solution

Keep this issue open. Fix src/plugins/capability-provider-runtime.ts so configured media-understanding providers from compat loading are merged into a partial active registry for requested tools.media.audio models, without pulling unrelated bundled providers. Add a regression test where active mediaUnderstandingProviders contains OpenAI/MiniMax, cfg.tools.media.audio.models requests Deepgram, and the resolved provider set includes Deepgram.
