## Summary

Keep #70900 open. Current `main` still has the exact `surface_error` throw guard that the PR targets, and the surrounding code confirms the reported mismatch: `failoverReason` can come from a raw string classifier while `failoverFailure` is the stopReason-gated provider-failure signal. The PR is a focused, recent bug fix with a relevant regression test and is not implemented or clearly superseded on `main`.

## Best Possible Solution

Keep #70900 open for maintainer review and normal CI. The best code direction is to land this PR or an equivalent `surface_error` guard that only throws when a genuine stopReason-gated provider failure occurred, while preserving the #70848/#70124 behavior for real billing/auth/rate-limit failures and keeping the new regression test for `failoverFailure=false` with a classified stale `failoverReason`.
