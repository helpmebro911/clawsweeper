## Summary

Keep this issue open. Current main has useful mitigations for macOS TCC exposure, including app-owned TCC surfaces for some node actions, a local-prefix installer path, stable Homebrew Node resolution, and an optional gateway wrapper. It still does not make a bundled or dedicated OpenClaw-owned Node/Gateway runtime the supported default for npm/global macOS Gateway installs, so the requested TCC boundary remains unresolved.

## Best Possible Solution

Keep this as core macOS security and packaging architecture work. The best fix is to make the supported macOS Gateway/FDA runtime use a stable OpenClaw-owned identity by default, either a signed app-owned Gateway/runtime or a dedicated installed Node binary, with migration, doctor/onboarding, docs, and LaunchAgent reinstall behavior coordinated with #7227, #69561, and #22179. Treat the local-prefix installer and `--wrapper` as mitigations until maintainers explicitly make one the supported TCC boundary.

## Risks / Open Questions

- Closing this now would conflate optional local-prefix or wrapper mitigations with the requested default OpenClaw-owned TCC identity.
- Users running the supported npm/global CLI path can still end up granting macOS TCC permissions to a shared Node binary.
- The docs now describe app-owned TCC for some macOS surfaces while the Gateway runtime docs/code still use an external CLI runtime, so the product boundary needs maintainer alignment.
