## Summary

Keep this issue open. Current main still injects channel metadata into agent-visible prompts, including a trusted inbound system block and untrusted JSON user-context blocks for direct external-channel chats, and I did not find a config option that disables or minimizes that behavior for trusted Telegram or Discord DMs. The related #41299 work is about clearer separation, not a metadata-injection toggle.

## Best Possible Solution

Keep the issue open for a maintainer product/security decision. A good implementation would add a documented, narrowly scoped config option in the shared channel/auto-reply prompt path, with tests for Telegram and Discord DMs, while preserving group/reply/forwarded/location safety metadata by default and documenting the trust tradeoff.

## Risks / Open Questions

- A broad off switch could remove sender/chat provenance that prompt safety, routing, replay cleanup, and copied-metadata stripping currently assume. Any implementation should be narrow and explicit, likely limited to trusted direct chats and preserving enough system-level provenance for safety and routing.
