## Summary

Keep this issue open. Current main still does not implement WhatsApp disappearing-message expiration for outbound replies: there is no config/schema/type field, resolved account state has nowhere to carry an effective expiration, active send options cannot transport it, and text/media/voice-note/poll send paths only pass existing quote/media options. The prior ClawSweeper comment is still accurate against current main, and the related reports cover quote threading, mentions, or delivery rather than disappearing-message expiration.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin. Add channel-level and account-level disappearingMessagesSeconds to the config types, zod schema, generated metadata/help, and docs; resolve a positive effective value with account override over channel default and 0/null/omitted meaning no explicit override; add an expiration field to ActiveWebSendOptions and a poll options path; merge it into Baileys MiscMessageGenerationOptions alongside quoted options; apply it across normal text, quoted replies, media sends, voice-note companion text, payload/tool sends, fallback/chunked sends, and polls; add targeted tests for configured, account override, unset, quoted, media/voice-note, and poll cases.

## Risks / Open Questions

- A partial implementation could leave privacy gaps if it only covers the main text path and misses quoted replies, media sends, voice-note companion text, fallback/chunked text, polls, or tool/payload sends.
- The optional behavior to honor a Baileys credential default should be verified against the pinned @whiskeysockets/baileys 7.0.0-rc.9 source/types before shipping.
- Because the WhatsApp config schema and generated metadata are strict, config types, zod schema, generated metadata/help, docs, and tests need to land together.
