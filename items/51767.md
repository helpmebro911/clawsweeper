## Summary

Keep this issue open. Current main still treats `cron.sessionRetention` as a cron-run-only reaper, while default kept subagent/ACP spawns are not covered by an equivalent short, restart-safe retention path. The related implementation PR #51861 was closed unmerged, and #52327 was closed as a duplicate rather than becoming the canonical tracker.

## Best Possible Solution

Keep this issue open and implement a restart-safe retention path for completed subagent and ACP session-store entries. The likely maintainer direction is a generic ephemeral-session reaper or documented subagent/ACP retention setting that preserves active and intentionally persistent sessions, archives/deletes transcripts consistently, runs on startup/store-write paths, and adds focused tests for expired cron runs, completed subagent/ACP runs, running subagents, keep/session-mode behavior, and transcript cleanup. PR #51861 remains useful prior art but needs review against current main and the restart-safety details from #52327.

## Risks / Open Questions

- A naive cron-reaper expansion could delete active, resumable, keep-mode, or intentionally persistent session-mode rows unless it uses durable terminal-state and cleanup-policy markers.
- The registry sweep can remove registry metadata for keep/session-mode runs without deleting the session row, so a future retention implementation should not depend only on registry records still existing.
- General `session.maintenance` can prune by age/count, but it is broad all-session maintenance and does not satisfy the reported short `cron.sessionRetention` behavior for completed subagent/ACP sessions.
