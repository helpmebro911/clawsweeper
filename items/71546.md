## Summary

Keep #71546 open. The issue is a fresh, concrete Discord ingest-lag report against 2026.4.23, and current main does not yet show a source-backed fix for the reported buffered-message/RESUME path. Main includes the earlier #68159 startup readiness fix and the later #38596 health-monitor restart escalation, but those address adjacent lifecycle/restart failures rather than proving that messages delivered during Carbon reconnect/RESUME are replayed, caught up, or promptly handed to the Discord inbound worker.

## Best Possible Solution

Keep #71546 open for Discord plugin owner investigation. The best path is to add targeted instrumentation/tests around Carbon close/RESUME/READY and MessageCreate delivery, verify whether listeners survive internal reconnects, and either prove Discord gateway replay reaches the existing inbound worker promptly or add a bounded, deduped catch-up read using the existing Discord message read surface after successful reconnect/RESUMED. Also answer whether the idle close-1000 cadence is expected baseline behavior.
