## Summary

Keep this issue open. Current main has adjacent Discord gateway fixes for startup readiness (#68159/#52372) and health-monitor restart-loop escalation (#38596), but I did not find an implementation that addresses the reported user-visible problem: delayed Discord MESSAGE_CREATE ingestion during Carbon reconnect/RESUME windows. Message ingress is still Carbon-event driven, with no reconnect hook or bounded REST catch-up/backfill path wired into the Discord monitor lifecycle.

## Best Possible Solution

Keep this issue open for the Discord plugin owner. The best path is to instrument Carbon close/RESUME/READY and MESSAGE_CREATE delivery timing, add regression coverage that listeners survive internal reconnects and replayed events reach the inbound worker promptly, and add a bounded deduped catch-up read using readMessagesDiscord(... after ...) if Carbon replay is not reliable. Maintainers should also explicitly answer whether the reported idle close-1000 cadence is expected baseline behavior.
