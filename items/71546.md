## Summary

Keep #71546 open. It is a fresh, concrete current-version Discord ingest-latency report, and current main still shows reconnect/READY hardening without a source-backed RESUME catch-up or proof that the reported 100-400s ingest delay is fixed. Related #38596 is broader and open, but it does not safely supersede this narrower post-login Carbon reconnect/buffering report.

## Best Possible Solution

Keep this as an active Discord plugin reliability issue. The best path is a targeted repro or chaos test around Carbon close/resume/READY timing, then either preserve/verify listener delivery across RESUME or add a deduped, opt-in catch-up pass using last seen Discord message IDs after successful reconnect. If maintainers want one umbrella, explicitly link #71546 from #38596 before closing it as a duplicate.
