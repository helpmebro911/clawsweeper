## Summary

Keep #8673 open. Current main still lacks bounded retry for ordinary transient OAuth refresh failures: OpenClaw calls the shared OAuth refresh path once, the manager adds locks/timeouts/adoption fallbacks but not generic retry, and the only retry-like branch is refresh_token_reused-specific. The related retry PRs in the provided context are closed unmerged, and the latest issue discussion already points the same direction.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the OAuth refresh operation. Prefer reusing `src/infra/retry.ts` with a narrow transient-error classifier, avoid retrying permanent auth failures, preserve the current lock/adoption/reused-token recovery behavior, and add focused auth-profile tests for success after a transient failure, retry exhaustion preserving the final error, and non-retryable OAuth failures.

## Risks / Open Questions

- A fix must distinguish transient refresh failures such as network timeouts, 429, and 5xx from permanent OAuth failures like invalid_grant or revoked refresh tokens.
- The retry policy must preserve existing file-lock serialization, hard refresh timeout, external/main-store credential adoption, provider-plugin refresh hooks, and refresh_token_reused recovery.
- Provider plugins may already own custom refresh behavior, so shared retry placement should avoid surprising double-retry semantics.
