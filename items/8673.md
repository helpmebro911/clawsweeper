## Summary

Keep #8673 open. Current main still lacks bounded retry for ordinary transient OAuth refresh failures: the shared OAuth path calls the upstream refresh API once, the manager adds locking/timeouts/adoption paths but not generic retry, and the only second-attempt path is the refresh_token_reused recovery. The related retry PRs #8677 and #45975 are closed unmerged, and the latest issue discussion also concluded the item remains relevant.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the existing OAuth refresh operation. Prefer reusing src/infra/retry.ts with a narrow transient-error classifier, avoid retrying permanent auth failures, preserve current lock/adoption/reused-token recovery behavior, and add focused auth-profile tests for success after a transient failure, retry exhaustion preserving the final error, and non-retryable OAuth failures.

## Risks / Open Questions

- A fix must retry only transient refresh failures such as network timeouts, 429, and 5xx responses, not permanent OAuth errors like invalid_grant or expired/revoked refresh tokens.
- The retry policy must preserve the existing file-lock serialization, hard refresh timeout, external/main-store credential adoption, provider plugin hook behavior, and refresh_token_reused recovery path.
- Provider plugins may implement provider-specific refresh behavior; shared retry should avoid surprising double-retry semantics where a plugin deliberately handles its own policy.
