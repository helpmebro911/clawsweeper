## Summary

Current main still does not implement the requested built-in L1-L4 command-text risk classifier for exec approvals. The shipped model is policy, allowlist, durable approval, strict inline-eval, and narrow safeBins based; plugin hooks can block or add plugin approval, but they do not provide a documented exec-policy auto-allow classifier that would safely replace core exec approval decisions.

## Best Possible Solution

Keep this issue open for maintainer product/security design. If pursued in core, implement an explicit opt-in exec policy layer with documented threat model, UI/config affordances, and regression coverage for safe reads, pipelines, shell wrappers, interpreters, Windows/POSIX parsing, and dangerous-command denial. If maintainers prefer plugin-first classification, first add or document an exec-specific plugin API that can classify/block/require-approval/auto-allow while preserving the existing host exec approval policy boundaries.

## Risks / Open Questions

- A broad built-in classifier would materially change host exec security posture and needs explicit maintainer/security design for shell parsing, wrappers, interpreters, pipelines, POSIX/Windows differences, aliases, PATH resolution, and destructive-pattern semantics.
- Closing as ClawHub would be premature unless maintainers first accept a plugin-first direction and expose or document an exec-specific policy hook that can safely block, require approval, and optionally auto-allow without weakening host approval boundaries.
