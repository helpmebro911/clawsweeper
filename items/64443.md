## Summary

Keep this open. The issue is recent, concrete, hardware-specific, and current main still uses the native macOS WebChat path with transparent non-opaque windows, active visual effect material, material/gradient chat backgrounds, shadows, and visible update animations. I found smoke/unit coverage for basic construction and chat model behavior, but no source, docs, tests, or history evidence that the reported WindowServer/compositor CPU problem has been reproduced, fixed, or measured away on Intel Retina hardware.

## Best Possible Solution

Keep this open as a macOS chat rendering performance investigation. Profile current native WebChat on affected Intel Retina hardware, compare opaque/no-material/no-shadow/no-animation variants, verify Reduce Motion behavior, and add a measured low-compositor fallback for older Intel or high-resolution displays if the regression is confirmed.

- The native SwiftUI chat implementation may have changed since the reporter's 2026.4.8 build, but current main still contains plausible compositor stressors and no measurement-backed resolution evidence.
