## Summary

Keep this PR open. Current main still does not implement offline APNs chat-reply alerts, and this PR is an active but incomplete implementation candidate rather than obsolete cleanup. It is also paired with the same author's open iOS-side work in #63697, while the prior gateway attempt #63800 is closed and unmerged.

## Best Possible Solution

Keep this PR open as an implementation candidate or supersede it with a complete gateway+iOS change. The maintainer path is to add a dedicated chat-reply APNs payload builder/kind, wire it into both server-chat agent-event completion and direct chat.send completion with request device/connection tracking, use current main-session canonicalization, cover direct and relay delivery plus skip/cleanup cases, and coordinate iOS notification routing with the open companion #63697.

## Risks / Open Questions

- Merging as-is would add an unused helper because the PR body explicitly defers lifecycle wiring to a follow-up.
- The helper currently reuses sendApnsAlert, which emits openclaw.kind="push.test"; chat replies need a dedicated APNs payload kind and matching iOS route.
- The proposed main-session guard appears incompatible with current canonical main-session keys such as global and agent:<id>:<mainKey>.
- The PR tests do not cover relay sending, hasConnectedClientForDevice early exit, or stale-registration cleanup.
- Assistant reply text in APNs alert bodies is a privacy-sensitive product decision that should be explicitly confirmed before landing.
