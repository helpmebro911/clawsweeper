## Summary

Close as `clawhub`. Current `main` already provides the core safety primitives for updates and config recovery, while `VISION.md` explicitly keeps convenience-wrapper orchestration and optional capability out of OpenClaw core and routes new skills/plugins to ClawHub first.

## Risks / Open Questions

- Some users may still want a bundled one-command ops workflow, but that is a product-scope choice rather than missing core infrastructure.
- The issue also asks for channel notifications; that is optional orchestration and even more naturally fits a ClawHub skill/plugin than core CLI.

## Close Comment

Closing as `clawhub` after Codex review. The requested flow is useful, but current scope points away from adding this as a built-in core wrapper. `VISION.md:54-70` says optional capability should usually ship as plugins and new skills should go to ClawHub first, and `VISION.md:87-91` explicitly says OpenClaw does not want convenience wrappers that hide critical security decisions from users. Current `main` already has the core primitives this workflow would orchestrate: `openclaw update` (`docs/cli/update.md:9-117`) runs the update path with doctor/restart integration, the updater runs `openclaw doctor --non-interactive --fix` (`src/infra/update-runner.ts:1184-1191`), restart health is verified after update (`src/cli/update-cli/update-command.ts:781-818`), and config writes/recovery already preserve rejected payloads and restore last-known-good backups (`docs/cli/config.md:357-386`, `docs/gateway/troubleshooting.md:265-314`, `src/config/backup-rotation.ts:115-124`, `src/config/io.observe-recovery.ts:651-679`). Given that, the missing piece here is higher-level orchestration and channel notification, which is a better fit for a ClawHub skill/plugin than OpenClaw core. If you still want to pursue it, please package the existing `upgrade.sh` workflow as a ClawHub skill/plugin and link it from a new proposal focused on that surface.
