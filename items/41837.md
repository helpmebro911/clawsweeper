## Summary

Keep open. Current main still does not detect the reported Mattermost deaf WebSocket condition: the Mattermost socket path has no ping/pong or pong-timeout surface, only the bot update_at REST watchdog, and the generic stale-socket monitor only applies when a channel publishes provider-proven transport activity. The linked fix PR #57621 is open and unmerged, so this remains actionable.

## Best Possible Solution

Keep this issue open until PR #57621 or an equivalent Mattermost-owned fix lands. The fix should add WebSocket ping/pong or another provider-proven transport liveness signal in extensions/mattermost/src/mattermost/monitor-websocket.ts, terminate missed-pong sockets so runWithReconnect restarts them, publish lastTransportActivityAt only for real transport heartbeat evidence, and add focused monitor-websocket regression tests.
