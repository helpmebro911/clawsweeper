## Summary

Keep open. Current main still does not implement the PR’s central behavior: a proactive gateway startup/hourly sweep for stale session archive files, orphaned atomic-write tmp files, and excess transcript/session backups. Main has reactive/session-store maintenance and an on-demand cleanup command, but those do not fully supersede this PR, and the PR discussion still contains an unresolved custom `session.store` coverage concern that needs maintainer judgment before landing or closing.

## Best Possible Solution

Keep the PR open as useful but incomplete implementation work. The best path is to rebase it onto current main, preserve the startup/hourly proactive sweep, add coverage for orphaned atomic tmp files and transcript/session `.bak.*` trimming, and route discovery through the existing all-agent session-store target machinery so custom `session.store` roots are covered. If maintainers prefer issue-first tracking, keep #44820 as the canonical bug and ask the PR author to address the custom-store review comment rather than auto-closing this PR.
