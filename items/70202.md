## Summary

Keep #70202 open. Current main has the underlying model primitives the request needs: primary model, fallback list, model allowlist, post-setup model switching, and a multi-select allowlist/fallback path in `openclaw configure --section model`. The initial `openclaw onboard` flow, however, still wires only `promptDefaultModel` after auth and custom provider onboarding still collects one endpoint/model at a time, so the requested onboarding-time “select multiple models and one default” workflow remains real work. The provided related context does not supersede it: #29874 covered optional automatic routing, while #10480 is a provider-specific onboarding gap.

## Best Possible Solution

Keep the issue open and implement the missing onboarding UX by reusing the existing `promptModelAllowlist` and fallback/allowlist application helpers in `runSetupWizard` after provider auth/default selection. The best product shape is: authenticate one or more providers/endpoints, choose one primary default, multi-select additional models for `/model`, and optionally seed `agents.defaults.model.fallbacks` from that selection without clobbering existing allowlist entries.
