## Summary

Current main partially supports the broader workflow through `openclaw configure --section model`, model allowlists, fallbacks, and UI/session model switching, but the specific first-run onboarding request is not fully implemented. The interactive onboarding wizard still prompts for a single default model via `promptDefaultModel` and applies it with `applyPrimaryModel`; the multi-select allowlist/fallback flow is only wired through configure, not `openclaw onboard`.

## Best Possible Solution

Keep the issue open and implement the narrow missing onboarding integration: after onboarding picks or preserves the default model, call the existing `promptModelAllowlist` flow with the same preferred-provider scoping, then apply `applyModelFallbacksFromSelection` and `applyModelAllowlist`. Add onboarding wizard tests proving a user can select one primary plus additional enabled models/fallbacks during first-run setup. Leave automatic task-complexity routing as a separate provider/plugin or ClawHub discussion rather than blocking this onboarding UX fix.
