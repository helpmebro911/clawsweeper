## Summary

Do not auto-close PR #70586. Current main partly supersedes the stated stale-socket concern because channel health now uses provider-proven transport activity instead of app-event freshness, and normal Slack messages/app_mentions already call `trackEvent` through `createSlackMessageHandler`. But main still does not thread `trackEvent` into `registerSlackMessageEvents`, so early-return message branches like subtype system events and DM-skipped app_mentions remain a plausible status-activity gap. The PR needs revision rather than cleanup closure because its current approach would double-update liveness for normal handled messages.

## Best Possible Solution

Keep the PR open, but revise it to track only message-event branches that do not reach `createSlackMessageHandler` or otherwise centralize message-event tracking so each accepted inbound Slack event updates status at most once. Update the PR tests to prove subtype system events and DM-skipped app_mentions tick activity while normal message/app_mention flows still call `trackEvent` exactly once and mismatched events do not track.
