## Summary

Keep open. Current main still has a real gap for Slack message-event branches that return before `handleSlackMessage`, but PR #70586 should not be closed as already implemented and should not be merged as-is because accepted messages already call `trackEvent` in `createSlackMessageHandler`, so the proposed registrar-level callback would double-update liveness for normal message/app_mention flows.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: thread `trackEvent` into `registerSlackMessageEvents`, but call it only for valid branches that return before `handleSlackMessage` can run, or otherwise centralize tracking so regular message/app_mention paths update exactly once. Add regression tests for mismatch drops at zero, normal handled messages at one, subtype system events at one, and DM-skipped app_mention at one.
