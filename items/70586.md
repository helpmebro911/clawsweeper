## Summary

Keep this PR open for revision. Current main still has a real Slack message-event app-activity tracking gap for accepted registrar branches that return before the message handler, but the submitted patch tracks too early and would double-update normal handled messages/app mentions. The PR's stale-socket rationale is also obsolete because current channel health uses transport activity, not `lastEventAt`, for stale socket detection.

## Best Possible Solution

Request a narrower revision of this PR: thread `trackEvent` into `registerSlackMessageEvents`, but invoke it only for accepted message-event branches that return before `handleSlackMessage` and genuinely represent inbound app activity, such as authorized subtype system events and DM/MPIM `app_mention` skips. Keep `createSlackMessageHandler` as the single tracking point for normal handled messages/app mentions, and add tests proving handled events track once, accepted early-return branches track once, and mismatched, duplicate, invalid, unauthorized, or self-generated events do not track. Reframe the change around app-activity timestamp completeness rather than stale socket liveness.
