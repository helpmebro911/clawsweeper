## Summary

Keep this PR open. Current main still omits `trackEvent` from `registerSlackMessageEvents`, so accepted Slack message-event branches that return before `createSlackMessageHandler` can miss app-activity updates. The submitted patch is not safe as-is because normal handled `message` and non-DM `app_mention` paths already track inside `createSlackMessageHandler`, and the PR rationale about stale socket liveness is outdated by the transport-activity health policy shipped from #69833.

## Best Possible Solution

Keep this PR open and request a narrower revision: thread `trackEvent` into `registerSlackMessageEvents`, but invoke it only for accepted branches that return before `handleSlackMessage` and genuinely represent inbound app activity. Preserve `createSlackMessageHandler` as the single tracking point for normal handled messages/app mentions, and add tests proving handled events track once, accepted early-return branches track once, and mismatched, duplicate, invalid, or unauthorized/self-generated events do not track.
