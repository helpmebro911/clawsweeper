## Summary

Keep #13364 open. Current main still does not expose `before_tool_call` or `after_tool_call` through managed/internal `HOOK.md` hooks; tool-call interception remains a plugin hook surface, and a fresh maintainer comment explicitly kept this issue as the canonical tracker.

## Best Possible Solution

Keep #13364 open for maintainer product/API design. A complete implementation should add typed managed/internal before/after tool-call events, define blocking and parameter-rewrite return semantics, bridge both tool execution paths without double-firing plugin hooks, validate or clearly warn on unsupported `HOOK.md` event names, and update docs plus focused internal-hook tests. If maintainers decide this must remain plugin-only, close later with an explicit product decision and point users to plugin hooks and ClawHub packaging.
