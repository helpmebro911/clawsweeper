## Summary

Keep open. Current main has meaningful Web UI chat responsiveness pieces, but the central request is broader than those pieces: gateway/config UI responsiveness and end-to-end latency attribution are still not implemented. The provided discussion and related-item context also point to overlapping, still-open work rather than a canonical merged fix.

## Best Possible Solution

Keep this open as a UX/performance tracker, or split it into scoped maintainer-owned follow-ups: add client-to-gateway-to-provider timing attribution for chat/config actions, preserve the existing optimistic/non-blocking chat path, isolate slow config/status panels, add per-control pending states and stale-response guards, and cross-link narrower work such as #64004 and #71630.
