## Summary

Keep open. Current main has partial chat responsiveness work, including optimistic user-message display, live stream/reading indicators, and guards against mid-run transcript reloads, but the broader request is not fully implemented: gateway config still uses coarse global loading/saving/applying state, end-to-end UI-to-model latency instrumentation is not present, and the related open items do not provide a single canonical superseding tracker.

## Best Possible Solution

Keep #71631 open and triage it as a Web UI/gateway responsiveness enhancement. The best path is to preserve the already-shipped optimistic chat/streaming behavior, coordinate with #64004 and #71630, and add missing end-to-end timing diagnostics plus config-specific UI pending states, cancellation/stale-response guards, and slow-RPC isolation where current code still uses coarse global state.
