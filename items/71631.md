## Summary

Keep #71631 open. Current main already has important pieces of responsive Web UI chat, including optimistic local user messages, a non-blocking `chat.send` ack, streaming chat events, and some config loading/saving flags. The issue is broader than those shipped pieces: current code does not provide end-to-end UI-to-gateway-to-provider timing instrumentation, per-control config operation states, request cancellation/stale guards across slow panels, or proof that slow config/status/background RPCs cannot block unrelated chat rendering. The related #64004 remains an open narrower bug for Control UI slowness, and #71630 remains an open complementary runtime status indicator request, so this issue still tracks real remaining product work rather than cleanup closure.

## Best Possible Solution

Keep this open as a product/UI performance umbrella, but narrow the next implementation into concrete tasks: preserve the shipped non-blocking chat/streaming path, add end-to-end client and gateway timing marks from click/send through first visible chat feedback and final completion, add per-panel/per-control config loading/saving/testing states with stale-response or cancellation guards, and coordinate with #64004 for Control UI slow-call isolation and #71630 for composer runtime status.
