## Summary

Keep PR #46380 open. Current main still sends MiniMax VLM image-understanding requests with raw `fetch()` to a host derived from `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl`, so the SSRF/key exfiltration concern is not implemented on main or safely superseded. The exact patch needs redesign because review discussion and current tests show strict guarded mode can break env-proxy-only deployments.

## Best Possible Solution

Keep the PR open but request a revised implementation that preserves proxy compatibility: route MiniMax VLM through the shared guarded provider HTTP helper or an equivalent `fetchWithSsrFGuard` call with `auditContext: "minimax-vlm"`, proxy-aware mode selection, cleanup coverage, and regression tests for private/cloud-metadata `apiHost` and `modelBaseUrl` blocking plus HTTP(S)_PROXY compatibility.

## Risks / Open Questions

- The submitted patch’s `withStrictGuardedFetchMode` approach may regress proxy-required MiniMax VLM environments; current main explicitly initializes the env proxy dispatcher before the raw fetch.
- The security gap remains unresolved on main: private/internal `apiHost` or `modelBaseUrl` destinations are not blocked at request time on the MiniMax VLM path.
