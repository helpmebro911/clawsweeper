## Summary

Keep open. Current main still lacks the proposed config `$include` hardening: include paths are resolved and normalized before any explicit NUL-byte or path-length validation, tests only assert that a NUL path throws somehow, and the user-facing docs do not specify a NUL or path-length contract. The PR is a focused security-hardening candidate, but it needs the existing review findings addressed before merge, especially enforcing any PATH_MAX-style limit by byte length rather than UTF-16 string length.

## Best Possible Solution

Keep this PR open as the implementation candidate. Revise it to reject NUL before resolution, enforce the raw and normalized/resolved include-path cap with `Buffer.byteLength(..., "utf-8")`, make the exclusive 4096-byte boundary unambiguous in errors/docs, add regression tests for NUL, raw limit, resolved limit, boundary acceptance, and multibyte byte-limit cases, then run the targeted config include test and changed gate before maintainer merge.
