## Summary

Keep open. Current main documents the strict-agentic/update_plan interaction and reduces update_plan result noise, but it still does not prove a runtime cap or retry path for the reported shape: many consecutive tool-only update_plan calls with no real tool progress before the next non-plan tool.

## Best Possible Solution

Keep this issue open as the focused tracker for excessive consecutive update_plan-only bursts. The best fix belongs in the embedded Pi runtime/tool-loop layer: treat pure update_plan-only activity as non-progress even without visible assistant prose, cap consecutive plan-tool-only calls or turns before retry/blocking, and add regression coverage for many update_plan-only calls before a real non-plan tool. Keep the current docs and tools.experimental.planTool=false mitigation because they help operators but do not solve the runtime loop.
