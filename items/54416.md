## Summary

Keep this issue open. Current main still has no hard exec-boundary guard that blocks or requires explicit approval for model-issued OpenClaw self-management shell commands such as `openclaw config set ...`, `openclaw models set ...`, or `openclaw gateway restart` when host exec is in the documented full/off mode. Prompt guidance and the first-class gateway config guard are useful partial hardening, but they do not close the raw `exec` path reported here.

## Best Possible Solution

Keep this open for a maintainer/security fix at the exec boundary. Add a hard runtime guard that blocks or requires explicit user approval for model-initiated OpenClaw self-management shell commands such as `openclaw config set`, `openclaw models set`, `openclaw gateway restart`, update, and uninstall equivalents unless they come from an explicit user-approved config-change action. Add regression coverage for ambiguous model-switch prompts that may show example commands but must not execute them automatically, and coordinate the broader destructive-command policy with #65307.

## Risks / Open Questions

- I did not attempt a live reproduction because doing so would intentionally mutate OpenClaw config and restart the gateway; source and docs review are enough to show the enforcement gap.
- Deployments with manually tightened `tools.exec` or host approvals policy may not reproduce, but current main still permits the reported class under the documented host full/off behavior.
- The eventual fix should coordinate with #65307 so config mutation, model switching, gateway lifecycle, update, uninstall, and similar self-management commands share one conservative exec-boundary policy.
