## Summary

Keep this issue open. Current main has prompt guidance and a guarded first-class gateway config tool, but model-issued `exec` commands still default to no approval on gateway/node hosts and there is no hard runtime block or confirmation boundary for `openclaw config set ...` or `openclaw gateway restart` executed through shell exec. The related #65307 is still open and reinforces that the broader no-confirmation exec class remains unresolved.

## Best Possible Solution

Keep the issue open for a maintainer/security fix. The strongest product path is a hard runtime guard at the agent exec boundary that requires explicit approval or blocks model-initiated OpenClaw config/lifecycle mutations, including `openclaw config set`, config apply/patch equivalents, `openclaw gateway restart`, and adjacent destructive self-management commands unless a user-approved config-change action is present. Add regression coverage for ambiguous model-switch prompts that may show example commands but must not execute them automatically, and coordinate the broader destructive-command policy with #65307 rather than closing either report prematurely.
