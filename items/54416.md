## Summary

Keep this issue open. Current main still allows model-issued host exec to run OpenClaw CLI config/self-management commands without a command-specific confirmation guard when the effective exec policy is the documented gateway/node default of `security="full"` and `ask="off"`. Prompt guidance and the owner-only `gateway` tool guard are useful hardening, but they do not enforce the exec boundary described in this issue.

## Best Possible Solution

Keep this open for a maintainer/security fix at the exec boundary. Add a hard runtime guard that blocks or requires explicit user approval for model-initiated OpenClaw self-management commands such as `openclaw config set`, `openclaw models set`, `openclaw gateway restart`, update/uninstall equivalents, and related lifecycle commands unless they are tied to a user-approved config-change action. Add regression coverage for ambiguous model-switch prompts that may show example commands but must not execute them automatically, and coordinate the broader destructive-command policy with #65307.

## Risks / Open Questions

- No live reproduction was attempted because the reported path would intentionally mutate OpenClaw config and restart the gateway; the source and docs still show the relevant enforcement gap.
- Deployments that already set stricter exec approval policy may not reproduce, but the documented default gateway/node behavior still permits this class of model-issued shell mutation.
- The fix should coordinate with #65307 so self-management, uninstall/update, config mutation, and restart commands share one conservative policy instead of separate narrow blocks.
