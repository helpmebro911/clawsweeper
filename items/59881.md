## Summary

Keep open. Current `main` still wires agent-hook `ctx.channelId` to the channel/plugin id (`messageChannel`, or fallback `messageProvider`) instead of the underlying conversation/channel identifier, while the actual conversation target is carried separately elsewhere.

## Evidence

- **Embedded runner `before_prompt_build` still passes provider/plugin id as `ctx.channelId`:** `src/agents/pi-embedded-runner/run/attempt.ts` builds the hook context for prompt hooks with `messageProvider: params.messageProvider` and `channelId: params.messageChannel ?? params.messageProvider`, so Discord/Telegram-style conversation ids are not exposed here.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2055](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/attempt.ts#L2055)

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Embedded runner `agent_end` uses the same mapping:** The `agent_end` hook path repeats the same assignment: `channelId: params.messageChannel ?? params.messageProvider`.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2703](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/attempt.ts#L2703)

  - file: [src/agents/cli-runner/prepare.test.ts:272](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/cli-runner/prepare.test.ts#L272)

## Risks / Open Questions

- A fix is behavior-sensitive: current code and tests rely on `ctx.channelId` meaning the routed channel/plugin id, so changing it may break existing plugins or bundled extensions that reuse `ctx.channelId` as `messageChannel`.
- There is no evidence on current `main` of a backward-compatible replacement field on `PluginHookAgentContext` for the real conversation/channel identifier, so the original report remains unresolved rather than obsolete.
