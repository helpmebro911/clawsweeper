## Summary

Keep PR #71612 open. Current main still records tool-call loop history against the stable cron agent session key and `recordToolCall` has no idle-gap or run-boundary reset, so repeated isolated cron runs can still accumulate identical tool calls. The PR is a focused recent bug fix with targeted tests; the related session reset items do not supersede this specific loop-detection false positive.

## Best Possible Solution

Keep the PR open for maintainer review. The best fix is to scope tool-loop history to the real isolated cron run boundary for jobs sharing `agent:<id>:cron:<jobId>`, with regression coverage in `src/agents/tool-loop-detection.test.ts`; the PR's idle-gap reset is a plausible minimal implementation, but maintainers may prefer passing the existing per-run key into diagnostics.
