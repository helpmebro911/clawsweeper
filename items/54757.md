## Summary

Keep this issue open. Current main still suppresses only WhatsApp text-only tool lifecycle chatter; final tool-error warning payloads such as mutating write/edit/message failures can still be built as `⚠️ ... failed` final replies and delivered through WhatsApp. The existing `messages.suppressToolErrors` setting does not suppress mutating tool warnings, and WhatsApp does not pass the stronger per-run `suppressToolErrorWarnings` option for inbound replies.

## Best Possible Solution

Keep the issue open and implement an external-channel-safe tool-error policy. The likely fix is to pass a WhatsApp/SMS-scoped `suppressToolErrorWarnings` or equivalent into inbound auto-reply runs, or to add a channel-aware policy that suppresses final mutating tool-error warnings for external recipients while preserving internal logs and operator-visible diagnostics. Add regression tests proving final `⚠️ ... failed` payloads from read/edit/write/message failures are not delivered through WhatsApp, while intentional media/tool artifacts still deliver.
