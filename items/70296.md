## Summary

Keep #70296 open. Current main still has an actionable restart-sentinel channel-noise gap: config/apply writes a restart sentinel with a doctor hint, startup formats that message, wakes the session with it, and delivers it to the resolved channel route, while no `sentinel.suppressChannelNotify` or equivalent suppression config exists. The Control UI model read-back path appears improved on main, and the BOOT token concern needs narrower product clarification, but the combined issue should not be closed while Issue 1 remains real.

## Best Possible Solution

Keep the issue open, preferably split or narrow it. The concrete core fix should add a documented config-backed restart-sentinel channel-notice suppression option, wire it into the startup sentinel path before `enqueueRestartSentinelWake`/`deliverRestartSentinelNotice`, and add regression tests proving channel notices can be suppressed without dropping durable continuations. Separately, add/keep a Control UI model read-back regression test and decide whether BOOT should support `NO_REPLY_NEEDED` as an alias or document that `NO_REPLY` is the only protocol token.
