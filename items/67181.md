## Summary

Keep open. Current main still builds and sends the exact internal resume-fallback banner on direct follow-up fallback after a session resume failure, including the successful `code 0` Discord path described by the issue. The Telegram confirmation in the discussion matches the same shared channel-generic exec approval follow-up path, so this is still an active core behavior rather than a Discord-only stale report.

## Best Possible Solution

Keep this issue open and fix the shared exec approval follow-up fallback in `src/agents/bash-tools.exec-approval-followup.ts`: for successful command results, direct fallback should send only the sanitized result or neutral user copy, while the session-resume failure reason stays in logs. Update `src/agents/bash-tools.exec-approval-followup.test.ts` so successful `code 0` fallback no longer expects the internal prefix, and coordinate any retry/backoff work with the related open follow-up tracking if maintainers choose to use it.
