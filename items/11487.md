## Summary

Keep #11487 open. Current main still does not accept optional human-readable `name`, `label`, or `description` metadata inside group config entries. Account-level `name` fields exist in several channel schemas, but the group-entry Zod schemas, TypeScript config types, generated config metadata, docs/examples, and at least one explicit strict-schema regression test still treat group entries as behavior-only objects and reject unknown fields.

## Best Possible Solution

Keep the issue open. The best fix is to choose one canonical optional metadata field, likely `name`, add it to group-entry config types and Zod schemas for core and bundled channel surfaces, regenerate config metadata, update public group/channel docs, and add focused schema tests proving the field is accepted and ignored functionally.

## Risks / Open Questions

- The implementation needs a product choice for the canonical field name (`name`, `label`, or `description`) so core and bundled plugin schemas do not drift.
- Because these schemas are strict and generated metadata is consumed by config tooling, adding the field in only TypeScript types or only docs would leave `config.patch`/validation behavior inconsistent.
