## Summary

Keep this PR open. Current main still lacks the gateway provider-qualified catalog lookup and the Control UI pending-attachment guard, so the PR is not obsolete. The gateway half is useful, but the Control UI patch needs revision because it still relies on stale render props for pending counters and attachment appends, can block unrelated command sends, and is behind the current run-controls refactor. Security review found no supply-chain or secret-handling regression in the changed-file surface.

## Best Possible Solution

Keep this PR open and request a focused revision: preserve the gateway provider-qualified catalog lookup and regression test, then rework Control UI attachment handling so pending-read deltas and attachment appends apply against live app state, limit pending-read blocking to user sends that depend on in-flight attachments, update the current run-controls component, and add tests for concurrent paste/select/drop races plus command-bypass behavior.

## Risks / Open Questions

- Merging the current Control UI patch can still re-enable send too early or leave the counter wrong because pending-count deltas are computed from stale render props.
- Merging the current Control UI patch can still drop attachments because FileReader callbacks append to captured attachment arrays instead of the latest app state.
- The current handleSendChat guard in the PR blocks all sends while reads are pending, including messageOverride/local command flows that do not depend on attachments, such as new-session and compact actions.
- The PR patch modifies an older inline send-button surface; current main routes run controls through ui/src/ui/chat/run-controls.ts, so the Control UI part needs a rebase/rework before merge.
