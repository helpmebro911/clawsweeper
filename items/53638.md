## Summary

Keep #53638 open. Current main partially satisfies the per-channel/per-group part through the shipped shared `channels.modelByChannel` config, so the Discord-channel request in the comment is largely covered. But the issue also asks for persistent per-DM model pinning and per-target config fields; the current resolver does not receive or match direct peer/native user IDs, and the WhatsApp/Slack target schemas/docs still expose prompt/access controls rather than `model` fields. I did not find a canonical related item that covers the remaining per-DM work; #53447 is narrower and tracks Discord thread inheritance only.

## Best Possible Solution

Keep the issue open and build on the existing shared `channels.modelByChannel` surface rather than adding isolated provider-only behavior by default. The best implementation path is to add direct-message candidate resolution from stable direct peer identifiers (`nativeDirectUserId`, provider `from`/`to`, and explicit direct session-key raw IDs where available), document the supported keys for WhatsApp, Slack, Discord, and other channels, and add tests for per-DM, per-group/channel, wildcard/channel default, thread-parent behavior, and explicit session override precedence. If maintainers prefer typed per-provider fields like `channels.whatsapp.direct.<id>.model`, route them through the same shared resolver so behavior stays consistent.
