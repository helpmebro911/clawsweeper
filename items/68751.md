## Summary

Keep open. Current main still implements the reported session-memory write behavior: `/reset` triggers the hook, `sessionContent` is written directly under `## Conversation Summary`, and tests explicitly expect raw prior `user:`/`assistant:` lines in the generated memory file. There is a useful current mitigation on the load path because startup context quotes recent memory as untrusted notes, but that does not fully satisfy the requested hook-side fix or make the report obsolete.

## Best Possible Solution

Keep this issue open and fix the bundled session-memory hook in core. The likely maintainer path is to stop persisting on `/reset` or make that configurable, and to wrap or sanitize persisted `sessionContent` before writing `memory/YYYY-MM-DD-*.md`; update `handler.test.ts`, `transcript.ts` coverage for inbound metadata/control-token stripping, and docs under `docs/automation/hooks.md` / `src/hooks/bundled/session-memory/HOOK.md`. Coordinate with the related raw session-memory poisoning reports (#69943, #71031) so the fix handles replay, chat-template/token contamination, and reset user expectations together.
