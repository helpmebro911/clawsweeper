## Summary

Current main still does not implement Discord trusted webhook ingress. Discord preflight resolves webhook IDs, but bot-authored webhook messages are still dropped when `allowBots` is off unless they are PluralKit; config types, strict zod schemas, generated metadata, and docs expose only the existing global `allowBots` behavior. Generic HTTP hooks and the bundled webhooks plugin are related automation surfaces, but they do not turn a Discord webhook post into the same-channel Discord conversation event requested here.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it inside `extensions/discord`: add typed and zod-validated `trustedWebhooks` config, resolve trusted webhook IDs before the bot-message drop, validate optional `mapToUser` against configured guild users or another explicit policy before changing sender identity, preserve own-message and bound-thread echo guards, update Discord docs/config metadata, and add preflight regression tests for allowed, denied, mention-only, map-to-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would change effective sender identity, so implementation needs explicit security review around configured guild users, channel allowlists, command authorization, tool policy, and auditability.
- Trusted webhook handling must preserve current self-loop protections, especially own-message filtering and bound-thread webhook echo suppression.
- Closing this as implemented because generic HTTP hooks exist would lose the reporter's Discord same-channel workflow.
