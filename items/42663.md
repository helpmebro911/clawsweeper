## Summary

Current main still does not implement Discord trusted webhook ingress. Discord preflight resolves webhook IDs, but only uses them for PluralKit exclusion and bound-thread echo suppression; bot-authored webhook messages are still dropped unless global `allowBots` permits them. The existing `/hooks/agent` HTTP ingress is a useful workaround for some automations, but it is not the same Discord same-channel webhook flow requested here.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it inside `extensions/discord`: add typed and zod-validated `trustedWebhooks` config, resolve trusted webhook IDs before the bot drop, optionally map to a configured user only after validation, preserve existing self-loop and bound-thread echo guards, update Discord docs/config metadata, and add preflight regression tests for allowed, denied, mention-only, map-to-user, and bound-thread cases.

## Risks / Open Questions

- A `mapToUser` feature would intentionally alter sender identity, so it needs explicit security design around guild/channel allowlists, role checks, command authorization, tool policy, and audit expectations.
- Trusted Discord webhook support must preserve existing loop protections, including own-message filtering and bound-thread webhook echo suppression.
- Closing this as implemented because `/hooks/agent` exists would lose the Discord-only, same-channel workflow requested by the reporter.
