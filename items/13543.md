## Summary

Keep #13543 open. Current main still implements sandboxing as a session/agent-level runtime decision plus sandbox-only tool allow/deny filtering, with separate browser target and exec host escape behavior. It does not implement the requested per-tool placement policy that can force web_fetch/browser into a sandbox while keeping exec/message/cron/gateway on host in the same session. The related Docker sandbox PR #29974 remains useful backend setup context, but it only partially unblocks this feature.

## Best Possible Solution

Keep this as a core sandbox/tool-routing enhancement. A good fix would add a documented, schema-backed per-tool execution placement policy; teach runtime/tool construction to route selected tools such as web_fetch and browser through a sandbox backend while keeping selected host tools such as exec, message, cron, and gateway on the host in the same session; and add tests/docs that clearly distinguish placement from tools.sandbox.tools availability filtering, tools.exec.host, elevated exec, and browser target selection.
