## Summary

Keep this issue open. Current main still routes QQBot tool/progress deliveries through `info.kind === "tool"`, stores their text in `toolTexts`, and returns without sending it; a later block/final reply clears the tool-only timeout, so the buffered progress text is not shown. QQBot `streaming.mode` still only gates block streaming, and QQBot has no documented preview/tool-progress path comparable to Telegram/Discord/Slack/Mattermost.

## Best Possible Solution

Keep this issue open and implement the fix in the QQBot plugin boundary. `extensions/qqbot/src/engine/gateway/outbound-dispatch.ts` should deliver useful `kind: "tool"` progress text immediately or through a QQBot preview/edit path when `streaming.mode` is partial, clear/dedupe it correctly when a final reply arrives, and preserve current media fallback semantics. Add a focused regression test in `extensions/qqbot/src/engine/gateway/outbound-dispatch.test.ts` for tool/progress text followed by a block/final reply, then update QQBot schema/docs if a public `streaming.preview.toolProgress` or equivalent option is added.
