## Summary

Keep open. Current main has Telegram preview tool-progress support, but the QQBot gateway source that matches the issue still buffers `kind: "tool"` text into `toolTexts`, waits on the 60s tool-only fallback, and clears that fallback when a block/final response arrives. I did not find a QQBot test or config path that makes `streaming.mode: "partial"` deliver tool-kind text immediately.

## Best Possible Solution

Keep #66509 open for a QQBot-specific fix. The likely path is to add visible progress delivery for QQBot when `streaming.mode: "partial"` is active, using an edited preview or an immediate progress send while preserving existing media fallback and text-only tool-summary suppression rules. Add a focused regression test in `extensions/qqbot/src/engine/gateway/outbound-dispatch.test.ts` that emits `deliver({ text: "Let me check" }, { kind: "tool" })` followed by a final/block reply and proves the progress text is visible instead of being cleared with the timeout.
