## Summary

Keep open. Current `main` does not implement `AbortSignal` support for `runTasksWithConcurrency`, and the proposed PR shape appears unsafe because it would allow aborted runs to return `hasError === false` with partial `results`, which conflicts with how current callers use the helper.

## Evidence

- **Current helper has no abort parameter:** On current `main` (`11cffb230013c2d8bd1e0e18da04b0b985b8abd5`), `runTasksWithConcurrency` still accepts only `tasks`, `limit`, `errorMode`, and `onTaskError`; there is no `signal?: AbortSignal` support in the shipped implementation.
  - file: [src/utils/run-with-concurrency.ts:3](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/utils/run-with-concurrency.ts#L3)
  - command: `nl -ba src/utils/run-with-concurrency.ts | sed -n '1,220p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Success path currently implies complete results:** The helper advertises `results: T[]` and returns that array on non-error completion, with no aborted/incomplete state in the return contract.
  - file: [src/utils/run-with-concurrency.ts:8](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/utils/run-with-concurrency.ts#L8)
  - command: `nl -ba src/utils/run-with-concurrency.ts | sed -n '1,220p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Callers rely on `hasError` as the only failure signal:** `runWithConcurrency` in the memory host SDK throws only when `hasError` is true and otherwise returns `results` directly as `T[]`. A cancellation path that leaves `hasError === false` but skips tasks would violate that assumption.
  - file: [src/memory-host-sdk/host/internal.ts:513](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/memory-host-sdk/host/internal.ts#L513)
  - command: `nl -ba src/memory-host-sdk/host/internal.ts | sed -n '500,535p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Some callers dereference every result after checking `hasError`:** `resolveSecretRefs` checks `taskResults.hasError` once, then iterates `taskResults.results` and immediately reads `result.group.refs`. Partial holes from an abort-success path would break this usage.
  - file: [src/secrets/resolve.ts:888](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/secrets/resolve.ts#L888)
  - command: `nl -ba src/secrets/resolve.ts | sed -n '876,915p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Abort support is a plausible request, but this patch needs redesign:** Existing utility code already treats abort as a first-class control path via `AbortSignal` composition in `fetch-timeout.ts`, so the feature request itself is coherent. The blocker is the proposed success-shaped partial-result behavior, matching the PR review's P1 concern.
  - file: [src/utils/fetch-timeout.ts:14](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/utils/fetch-timeout.ts#L14)
  - command: `nl -ba src/utils/fetch-timeout.ts | sed -n '1,220p'`
