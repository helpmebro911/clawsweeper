## Summary

Keep open. Current `main` still has the startup catch-up path that compares the cron slot before `now` against `job.state.lastRunAtMs`, and startup runs still record only the actual catch-up start time. That means a late catch-up can still be reclassified as a missed slot on the next restart, matching the report.

## Evidence

- **Startup missed-run path still uses `previousRunAtMs > lastRunAtMs`:** `isRunnableJob()` first checks `nextRunAtMs`, but if `nextRunAtMs` is in the future and there is no active error backoff, it falls through to the startup-only cron path and returns `previousRunAtMs > lastRunAtMs`. That is the exact comparison the issue says causes the duplicate replay.
  - file: [src/cron/service/timer.ts:894](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/timer.ts#L894)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '850,925p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Startup catch-up still enables that fallback on every restart:** `planStartupCatchup()` calls `collectRunnableJobs(..., { allowCronMissedRunByLastRun: true })`, so the `previousRunAtMs > lastRunAtMs` branch remains active during restart recovery.
  - file: [src/cron/service/timer.ts:989](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/timer.ts#L989)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '975,1025p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Successful runs still persist only the actual run start time:** `applyJobResult()` writes `job.state.lastRunAtMs = result.startedAt`. There is no separate persisted field for the cron slot that was fulfilled by a late catch-up run.
  - file: [src/cron/service/timer.ts:416](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/timer.ts#L416)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '391,455p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Cron job state has no fulfilled-slot marker:** `CronJobState` includes `nextRunAtMs`, `runningAtMs`, and `lastRunAtMs`, but no `lastFulfilledSlotMs` or equivalent field that would let startup recovery distinguish a late catch-up from an on-time execution of the prior slot.
  - file: [src/cron/types.ts:142](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/types.ts#L142)
  - command: `nl -ba src/cron/types.ts | sed -n '142,170p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **The reported ordering still evaluates to a duplicate under the current comparison:** Using the issue's shape of timestamps, a late catch-up at `2026-04-04T01:56:00Z` and a later computed cron slot at `2026-04-04T18:00:00Z` satisfy `previousRunAtMs > lastRunAtMs`, which is exactly what current `main` checks.

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
