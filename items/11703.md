## Summary

Keep #11703 open. Current main still batches WhatsApp inbound messages with the shared receive-time debouncer keyed by account, conversation, and sender. Baileys `messageTimestamp` is preserved as message metadata, but it is not used as a timestamp bucket, secondary coalescing window, or delayed same-timestamp grouping signal. Config, docs, and tests still expose normal debounce behavior only. The linked #19716 does not supersede this because it was a closed, unmerged PR about surfacing timestamps to agent context, not batching WhatsApp deliveries.

## Best Possible Solution

Keep the issue open. Implement this in the WhatsApp-owned inbound path, or add an additive shared inbound debouncer capability if maintainers want it reusable: group by account/conversation/sender plus same or near `messageTimestamp`, tolerate Baileys delivery delays beyond the receive-time debounce window, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any new config option if exposed.
