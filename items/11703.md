## Summary

Keep #11703 open. Current main still batches WhatsApp inbound messages with the shared receive-time debounce path; `messageTimestamp` is parsed and preserved, but it is not used to coalesce late Baileys deliveries that share the same WhatsApp server timestamp. The existing maintainer/Codex discussion also already identified #19716 as unrelated to batching and closed unmerged.

## Best Possible Solution

Keep #11703 open. Implement a WhatsApp-owned timestamp-aware coalescing path, or an additive shared inbound debouncer capability if maintainers want it reusable. The fix should group same account/chat/sender messages with the same or near `messageTimestamp` even when delivery occurs after the normal receive-time debounce window, and should include regression tests plus config/docs if a new option is exposed.
