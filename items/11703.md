## Summary

Keep #11703 open. Current main preserves WhatsApp `messageTimestamp` and now surfaces inbound timestamps to model context, but batching still uses the shared receive-time debouncer keyed by account, conversation, and sender. There is no timestamp bucket, delayed same-timestamp coalescing window, or `coalesceByTimestamp`/`coalesceWindowMs` config/docs/test coverage. The related #19716 was closed unmerged and addressed timestamp visibility, not batching.

## Best Possible Solution

Keep the issue open. Implement this in the WhatsApp-owned inbound path, or add an additive shared inbound debouncer capability if maintainers want reuse: group by account/conversation/sender plus same or near `messageTimestamp`, tolerate Baileys delivery delays beyond the receive-time debounce window, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any new config option if exposed.
