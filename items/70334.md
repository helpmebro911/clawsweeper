## Summary

Keep this issue open. The report is recent, concrete, and corroborated, and current main still has the explicit overflow compaction success -> retry path that matches the logs. Current main has adjacent idle/aggregate timeout safeguards, but I did not find source-and-test proof that the reported post-compaction retry lock is cleared and follow-up channel messages proceed.

## Best Possible Solution

Keep this issue open as the canonical tracker for post-overflow-compaction retry session locks. The best fix is a focused integration regression that drives overflow compaction success followed by retry success, retry error, and a lost/hung retry terminalization path, then asserts the active embedded run clears, diagnostics leave `processing` or enter a bounded blocked state, `queueDepth` drains, and a follow-up channel message can run.
