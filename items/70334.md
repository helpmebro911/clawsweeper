## Summary

Keep #70334 open. The report is recent, concrete, and corroborated, and current main has only partial lifecycle safeguards around compaction retry waits. I did not find source, test, changelog, or related-item evidence that the exact failure mode, a session remaining in diagnostic `processing` with `queueDepth=0` after successful overflow compaction and prompt retry, is fixed on main.

## Best Possible Solution

Keep the issue open and add a focused regression around the embedded overflow-compaction retry lifecycle: simulate successful explicit overflow compaction followed by retry success, retry error, and lost compaction-retry resolution, then assert the diagnostic session state returns to idle or the run is boundedly aborted and the session lane drains. If the root cause is a stale active command lane rather than embedded-run state, add a targeted watchdog/recovery path instead of relying on the warn-only stuck-session diagnostic.
