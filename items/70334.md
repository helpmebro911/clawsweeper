## Summary

Keep this issue open. The report is recent, has a second user confirmation, and current main still lacks a source/test-backed recovery path for the reported post-compaction retry lock. Main has adjacent mitigations, including bounded compaction retry waiting and compacted transcript rotation, but the active session still returns to `idle` only when the embedded attempt reaches terminal/finally cleanup; stuck-session diagnostics remain warning/telemetry only.

## Best Possible Solution

Keep this issue open as the canonical tracker for the post-overflow-compaction retry/session-lock freeze. The best fix is a focused regression that exercises overflow compaction followed by retry success, retry error, and a hung/lost retry terminalization path; it should assert that the embedded active-run registry clears, diagnostics leave `processing` or enter a bounded abort state, queue depth drains, and follow-up messages can proceed. If compacted transcript rotation or the aggregate compaction retry timeout is intended to fix this, add an integration-level proof connecting that path to the reported `state=processing queueDepth=0` diagnostics before closing.
