## Summary

Keep #70334 open. The report is recent, concrete, and corroborated, and current main still does not provide direct source/test evidence that a successful overflow-compaction retry cannot leave the session diagnostic state stuck in `processing` with `queueDepth=0`. Related #70591 was closed as a duplicate of this tracker, while #71455 is a provider-specific DeepSeek replay issue and does not supersede this general compaction/session-lock failure.

## Best Possible Solution

Keep #70334 open as the canonical tracker for the compaction retry/session-lock freeze. Add focused regression coverage for explicit overflow compaction followed by retry success, retry error, and lost retry/lifecycle resolution; assert diagnostic state returns to idle or the run is boundedly aborted, active-run state clears, and queued follow-up messages can drain. If the root cause is stale active-run state rather than compaction wait resolution, wire recovery through existing abort/cleanup primitives instead of leaving stuck-session diagnostics warning-only.
