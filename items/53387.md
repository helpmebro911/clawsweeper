## Summary

Keep open. Current main still treats state-dir `.env` entries as durable service environment, merges them into the gateway install plan, and passes that environment to the LaunchAgent plist renderer. The plist renderer writes every non-empty planned environment entry into `EnvironmentVariables`, while runtime dotenv loading still refuses to override an already-present process env var. The linked fix attempts (#53452 and #53462) identified the same root cause, but both are closed and unmerged.

## Best Possible Solution

Keep this issue open and fix the macOS LaunchAgent install path. The likely patch is to stop copying state-dir `.env` secrets into LaunchAgent `EnvironmentVariables`, keep only internal service metadata and explicitly supported service-level env, update `daemon-install-helpers` and LaunchAgent coverage to assert user-managed dotenv keys are excluded, and add repair guidance for already-installed plists.
