## Summary

Current `main` already contains a targeted Telegram fix for the compaction-replay case described in #40343, plus regression tests that cover the reported sequence. The fix is present on `main` in commit `51186d2725439646f7fea0e59bb466f957404c46`, but the provided latest release (`v2026.4.22`) predates that commit, so no shipped release can be confirmed from the local checkout.

## Evidence

- **Compaction marks the active streamed Telegram preview as a replay boundary instead of rotating it:** `pendingCompactionReplayBoundary` is set when compaction starts while an answer preview is already visible, specifically guarding the transient streamed preview during auto-compaction.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:1012](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/extensions/telegram/src/bot-message-dispatch.ts#L1012)
  - command: `nl -ba extensions/telegram/src/bot-message-dispatch.ts | sed -n '1008,1024p'`
  - sha: [51186d272543](https://github.com/openclaw/openclaw/commit/51186d2725439646f7fea0e59bb466f957404c46)
- **Next assistant-message start skips preview rotation after compaction replay:** When the replayed assistant message begins after compaction, `onAssistantMessageStart` clears `pendingCompactionReplayBoundary` and returns early instead of calling `rotateAnswerLaneForNewAssistantMessage()`, which is the path that would materialize/archive/delete the already-visible preview.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:944](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/extensions/telegram/src/bot-message-dispatch.ts#L944)

- **Cleanup logic retains finalized/archived previews instead of deleting delivered messages:** The answer lane archives materialized previews with `deleteIfUnused: false`, and final cleanup skips `clear()` / `deleteMessage()` for active or archived previews marked to retain, preventing already-delivered Telegram messages from being clawed back during teardown.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:455](https://github.com/openclaw/openclaw/blob/51186d2725439646f7fea0e59bb466f957404c46/extensions/telegram/src/bot-message-dispatch.ts#L455)

## Close Comment

Closing as `implemented_on_main` based on this Codex review. Current `main` now carries a targeted Telegram compaction-replay fix in `extensions/telegram/src/bot-message-dispatch.ts` that marks an in-flight streamed preview as a compaction replay boundary (`:1012`), skips preview rotation on the replayed assistant-message start (`:944`), and retains finalized/archived previews during cleanup instead of deleting them (`:455`, `:1038`). There is also direct regression coverage in `extensions/telegram/src/bot-message-dispatch.test.ts:1203` for the reported sequence: streamed partial reply, compaction, replay of the same assistant message, and final delivery without deleting the already-visible Telegram message.
