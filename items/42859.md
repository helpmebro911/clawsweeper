## Summary

Keep open. Current `main` still injects gateway client identity into `chat.send` sender metadata, so this PR is not implemented or obsolete. More importantly, the same `SenderId` currently flows into command/binding ownership checks; removing it without a replacement identity path would weaken existing `boundBy` protections for session lifecycle, focus, unfocus, and ACP rebind flows.

## Evidence

- **Current main still has the targeted behavior:** `chat.send` on current `main` still writes `SenderId`, `SenderName`, and `SenderUsername` from `clientInfo` into the dispatch context, matching the code this PR deletes.
  - file: [src/gateway/server-methods/chat.ts:2320](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/gateway/server-methods/chat.ts#L2320)

  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Sender metadata is still consumed as command identity:** The reply pipeline maps `sessionCtx.SenderId` into `run.senderId`, so stripping the gateway-populated sender fields changes downstream command identity, not just prompt metadata.
