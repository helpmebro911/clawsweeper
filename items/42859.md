## Summary

Keep this PR open. Current main still injects Control UI gateway client identity into `MsgContext` sender fields, which still renders as prompt-visible `Sender (untrusted metadata)` when inbound metadata is built. The PR targets a real regression, but its unconditional removal of `SenderId`/`SenderName`/`SenderUsername` from all `chat.send` requests is too broad because `SenderId` also feeds command authorization and sender-bound lifecycle ownership checks.

## Best Possible Solution

Keep the PR open, but request a narrower implementation: preserve trusted gateway sender identity in `MsgContext` for command authorization, run attribution, abort keys, and lifecycle ownership checks, while suppressing Control UI/internal gateway sender fields only from prompt-visible untrusted metadata. Add paired regression coverage for both sides: Control UI prompts should not include synthetic sender metadata, and a different gateway sender still must not update a binding owned by another sender.
