## Summary

Keep this PR open. Current main still renders Android chat text parts through Markdown only and has no Android Adaptive Card marker parser or equivalent rich-card rendering path. The PR is a concrete implementation candidate, but it is not merge-ready: the live PR head still has unresolved runtime/security risks around untrusted card JSON, and the same author has the paired #42350 mobile Adaptive Cards PR open, so this needs maintainer product direction rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review, not as ready-to-merge. The best path is to decide the product boundary first: if native Android rich chat is desired, rebase and rework this into a generic mobile MessagePresentation or documented client-rendering seam, add parser/render tests, and fix the unresolved crash/action/template-data issues. If maintainers decide Adaptive Cards should remain third-party plugin work, close the coordinated Android/iOS/Web Adaptive Cards client PRs together with an explicit ClawHub/plugin-boundary explanation rather than closing only this Android half.

## Risks / Open Questions

- The PR parses and renders untrusted model/plugin-supplied card JSON in Android chat; current head can still crash on malformed payloads through recursive rendering, invalid Compose weights, or negative rating bounds.
- Action.Submit and Action.Execute render tappable controls but only log locally, leaving action routing unresolved and presenting a misleading interactive UI.
- The implementation is tied to a third-party HTML-comment marker protocol instead of the existing MessagePresentation contract, which could create a parallel rich-message surface across clients.
- No Android parser/render regression tests are added for malformed JSON, nested cards, invalid widths, template data, multiple card blocks, or action behavior.
