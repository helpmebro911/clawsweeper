## Summary

Keep this PR open. Current main still lacks Android Adaptive Card parsing/rendering, so this remains a concrete implementation candidate for the Android chat surface. It is not merge-ready: the live PR head has functional and security issues for model/plugin-supplied card JSON, a member review requested changes, and the same author still has the paired iOS/macOS Adaptive Cards PR #42350 open.

## Best Possible Solution

Keep this PR open for maintainer product review. If native Android rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with defensive parsing, depth and numeric guards, URL scheme allowlisting, real action routing or visibly disabled actions, template-data handling, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close the Android and paired mobile/client Adaptive Cards PRs together with an explicit ClawHub/plugin-boundary explanation that points to the existing openclaw-adaptive-cards plugin.

## Risks / Open Questions

- The PR head is not safe to merge as-is because untrusted card JSON can crash rendering or trigger unsafe URI handling.
- Action.Submit and Action.Execute appear interactive but do not route back to the agent or gateway.
- The implementation uses a third-party HTML-comment marker protocol instead of the documented MessagePresentation contract, which could create a parallel rich-message surface across clients.
- No Android parser/render regression tests are included for malformed JSON, invalid widths, negative rating bounds, multiple card blocks, template data, URL schemes, or action behavior.
- Closing only this PR would split it from the same-author open iOS/macOS pair #42350 without a clear maintainer product decision.
