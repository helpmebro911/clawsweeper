## Summary

Keep this PR open. Current main still routes memory-lancedb baseUrl embeddings through the OpenAI SDK path, while the PR proposes a distinct direct-HTTP baseUrl path. The work is plausible but needs maintainer review or refresh because current main has adjacent float-encoding behavior and #69707 proposes a conflicting dimension-forwarding policy.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best path is a refreshed memory-lancedb embedding transport fix that handles `embedding.baseUrl` providers deliberately, preserves configured LanceDB vector dimensions, keeps native OpenAI on the SDK path, includes `encoding_format: "float"` or an equivalent float-vector guarantee, uses existing guarded/proxy-aware HTTP semantics, validates malformed and empty embedding responses, adds focused baseUrl regression tests, and coordinates the final dimension-forwarding policy with #69707.

## Risks / Open Questions

- Landing this PR as-is could omit `encoding_format: "float"` on the new direct baseUrl path and regress current protection for OpenAI-compatible servers that otherwise return base64 embeddings.
- The direct fetch path sends the configured bearer token to the intended baseUrl, but it bypasses the current SDK proxy dispatcher and does not reuse guarded remote HTTP helpers; final code should preserve proxy, redirect, and SSRF expectations.
- #69707 remains open with a competing dimension policy, so merging either PR without coordination could leave provider compatibility inconsistent.
