## Summary

Keep #20935 open. Current main has a narrow memory host event journal and a provenance-oriented memory-wiki bridge, but it still does not provide an append-only audit log for every MEMORY.md or memory/*.md mutation with what/when/why/session/user context or tamper evidence. The issue discussion also supports keeping this as remaining core memory work rather than closing it as implemented or ClawHub-only.

## Best Possible Solution

Keep #20935 open. The best implementation is a first-class memory-change audit layer in memory-core plus shared host/tool write hooks that emit append-only records for every MEMORY.md and memory/*.md mutation. Each record should include timestamp, operation, relative path, changed-content summary or diff/hash, trigger/why, session/run/user context when available, and optional integrity signing or hash chaining. Cover promotion, memory flush, dreaming, ordinary write/edit, apply_patch, and sandbox write paths with regression tests, then document the audit surface in the public memory docs.

## Risks / Open Questions

- Closing this now would leave normal write/edit/apply_patch mutations, memory flush appends, and some memory-core direct writes outside any canonical memory-change audit trail.
- A good implementation must decide how much changed content to log versus hashing or summarizing it, because memory files may contain sensitive personal data.
- The audit hook needs to cover host and sandboxed file operations plus plugin-owned memory writes without breaking existing append-only flush behavior or memory-core promotion/dreaming flows.
