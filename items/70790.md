## Summary

Keep PR #70790 open. Current main still lacks the PR's general embedded-runner payload guard: stream resolution returns session/boundary/custom stream functions without stripping `tools: []` or orphaned `tool_choice`, while OpenAI-compatible payload construction can still emit `tools: []`. The related `toolsAllow` forwarding fix has landed, but that does not solve the remaining strict-provider payload shape that this PR targets.

## Best Possible Solution

Keep the PR open for maintainer review or a successor patch. The best path is a narrow, tested embedded-runner/OpenAI-compatible payload sanitizer that strips `tools: []` and orphaned `tool_choice` at the provider egress boundary, preserves non-empty tools, composes with existing payload hooks, and does not duplicate already-landed `toolsAllow` forwarding work.
