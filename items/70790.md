## Summary

Keep PR #70790 open. Current main has related `toolsAllow` forwarding and an explicit-allowlist fail-closed guard, but it does not implement the PR's embedded-runner payload guard. The OpenAI-compatible transport builders can still serialize `tools: []` when `context.tools` is an empty array, and the only current empty-tools deletion I found is the narrow `/btw` path, not embedded sub-agent stream resolution.

## Best Possible Solution

Keep the PR open, but do not merge it blindly as-is. The best product/code path is a current-main patch that preserves the fail-closed behavior for explicit allowlists with no matching tools, adds a shared OpenAI-compatible payload guard for legitimate tool-less embedded calls so `tools: []` and orphaned `tool_choice` are omitted, and adds focused regression tests around the resolver and transport payload builders. The prior PR discussion and the Greptile fallback-path fix should be carried forward because they identify real coverage requirements.

## Risks / Open Questions

- The best fix needs a maintainer decision on semantics: explicit `toolsAllow` resolving to no tools may be a configuration error that should fail closed, while generic tool-less embedded calls should avoid sending `tools: []`.
- The PR should be rebased or reshaped around current main because `toolsAllow` forwarding and the empty-allowlist precheck already exist.
- Regression coverage should include openai-completions, openai-responses, websocket/session HTTP fallback, boundary-aware streams, custom streams, and `tool_choice` without non-empty tools.
