## Summary

Keep PR #70790 open. Current main has the adjacent `toolsAllow` forwarding/filtering fix and a `/btw`-specific empty-tools patch, but it still does not implement the embedded-runner payload guard this PR proposes. OpenAI-compatible payload construction can still serialize `tools: []` or an orphaned `tool_choice` in embedded stream paths, so the PR is not implemented or safely superseded.

## Best Possible Solution

Keep this PR or a successor open, but rework it against current main as a focused OpenAI-compatible provider-boundary guard. Strip `tools: []` and orphaned `tool_choice` where embedded runner payloads are serialized, preserve existing custom/provider stream semantics, and add regression coverage for boundary-aware, custom, session-http-fallback, completions, Responses, and WebSocket paths. Coordinate with #47947 so one canonical fix covers the remaining empty-tools/tool_choice behavior.
