## Summary

Keep open. Current main already has per-job cron timeouts and some cleanup/notification primitives, but it does not implement the requested job-level lifecycle API for hooks and auto-cleanup policies.

## Evidence

- **Per-job timeout is already supported:** Cron agent-turn payloads include `timeoutSeconds`, and the cron runner aborts execution when that timeout elapses.
  - file: [src/gateway/protocol/schema/cron.ts:12](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/gateway/protocol/schema/cron.ts#L12)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '1,220p'; nl -ba src/cron/service/timer.ts | sed -n '72,120p'`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Timeout behavior is covered by current tests:** Current regression coverage explicitly verifies that manual isolated `cron.run` executions honor `timeoutSeconds` and abort the run.
  - file: [src/cron/service/ops.regression.test.ts:200](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/cron/service/ops.regression.test.ts#L200)
  - command: `nl -ba src/cron/service/ops.regression.test.ts | sed -n '190,245p'`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Cleanup controls are global/coarse, not the requested per-job auto-cleanup API:** Current cron surfaces expose `deleteAfterRun`, global `cron.sessionRetention`, and global run-log pruning. There is no cron schema/config field for per-job `autoCleanup.afterRuns`, `afterCompleted`, or `ttlHours`.
  - file: [src/gateway/protocol/schema/cron.ts:258](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/gateway/protocol/schema/cron.ts#L258)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '250,320p'; nl -ba src/config/types.cron.ts | sed -n '1,140p'; nl -ba docs/cli/cron.md | sed -n '78,82p'`
  - sha: [9f4ef6162dbd](https://github.com/openclaw/openclaw/commit/9f4ef6162dbde384797abc002cd4241080eb65e3)
- **Session cleanup exists, but only as retention-based reaping:** The session reaper prunes completed isolated cron run sessions after a configurable retention window; it is not a job-level cleanup policy keyed by run count/status.
  - file: [src/cron/session-reaper.ts:24](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/cron/session-reaper.ts#L24)
  - command: `nl -ba src/cron/session-reaper.ts | sed -n '1,180p'`

- **Lifecycle notifications are partial and not the requested per-job hooks API:** Current main emits internal cron `started`/`finished` events and can deliver finished/failure webhooks, but the public cron schema exposes `failureAlert`/`failureDestination`, not per-job `hooks.onStart` / `onComplete` / `onFailure` fields.
  - file: [src/cron/service/state.ts:16](https://github.com/openclaw/openclaw/blob/9f4ef6162dbde384797abc002cd4241080eb65e3/src/cron/service/state.ts#L16)

## Risks / Open Questions

- The requested feature overlaps existing `failureAlert`, `failureDestination`, finished-webhook delivery, and `deleteAfterRun` behavior, so any future implementation needs a careful design to avoid duplicating cron surfaces.
- Per-job cleanup policies would likely touch persisted cron schema, session retention/reaping, run logs, and docs, so this is broader than a small cleanup-only change.
