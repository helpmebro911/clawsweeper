## Summary

Current main still lacks the requested cron job-level lifecycle contract. It has payload-level agent timeout, one-shot deleteAfterRun, retry/backoff, failure alerts, run-log/session retention, and plugin hooks with cron job context, but not a per-job lifecycle block covering timeoutMs, maxRetries, auto-cleanup by runs/time/status, or per-job start/complete/failure hook configuration.

## Best Possible Solution

Keep this issue open as a product/API design item. The best path is an additive cron lifecycle contract that defines job-level timeout semantics, retry and cleanup policies such as max runs, TTL, and status cleanup, plus start/complete/failure hook delivery through the existing hook/plugin architecture, with protocol/schema, scheduler, CLI/UI, docs, and regression tests updated together.

## Risks / Open Questions

- Running user-defined lifecycle hooks from cron is security-sensitive; an implementation should reuse existing plugin/internal hook trust, permissions, timeout, and isolation rules rather than adding an ad hoc script runner.
- The design crosses protocol/schema, cron store normalization, scheduler semantics, CLI/UI, docs, and tests, and should preserve current payload.timeoutSeconds, one-shot deleteAfterRun, failureAlert, run-log, and session-retention behavior.
- Open PR #45465 may be useful for the hook portion, but it should not be treated as resolving job-level timeout or automatic cleanup policy without additional implementation.
