## Summary

Keep this issue open. Current main still has only partial cron lifecycle coverage: payload-level timeout, one-shot delete-after-run, failure alerts, run-log/session pruning, and cron event broadcasts. It does not implement the requested job-level lifecycle contract with timeoutMs, retry policy, auto-cleanup by runs/time/status, or start/complete/failure hooks.

## Best Possible Solution

Keep this issue open as a product/API design item. The best path is an additive cron lifecycle design that formalizes job-level timeout semantics, recurring-job cleanup policies such as max runs/TTL/status cleanup, and cron start/complete/failure hook delivery through the existing internal hook or plugin-hook architecture. #37855 and #45465 can inform the hook portion, but they do not close the timeout and cleanup portions by themselves.

## Risks / Open Questions

- A broad lifecycle schema would touch public gateway protocol, config validation, CLI/UI/docs, and scheduler semantics; it should land additively and preserve existing payload.timeoutSeconds, deleteAfterRun, failure alert, run-log, and session-retention behavior.
- Any lifecycle hook implementation that executes user scripts needs the same path validation, trust, timeout, and isolation scrutiny as the existing hook systems.
