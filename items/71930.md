## Summary

Keep open. Current main still drops Mattermost `post_edited` WebSocket events before post parsing, mention detection, debounce, and wake routing. The issue is actionable against the bundled Mattermost plugin, not a ClawHub/off-repo request, and no duplicate or shipped fix was found in the provided related context or local source search.

## Best Possible Solution

Keep #71930 open and fix it in `extensions/mattermost/src/mattermost/monitor-websocket.ts` plus Mattermost monitor tests. Accept `post_edited` as a post-bearing event, feed the updated post through the existing Mattermost mention/access/session path, and adjust inbound replay dedupe so an original unmentioned `posted` event does not suppress a later edit-to-mention while duplicate edit frames still do not re-wake the agent. Add regression coverage for posted-without-mention followed by post_edited-with-mention, plus a non-mention edit that stays quiet.

## Risks / Open Questions

- A fix must avoid double-waking for ordinary edits while still allowing an edit that newly adds a mention after the original unmentioned post was already deduped.
- Existing replay keys probably need an edit-aware component such as event type plus `edit_at`/`update_at`, or the wake decision must happen before committing the original post id.
