## Summary

Keep #13948 open. Current main still implements core tool policy as whole-tool allow/deny, while `message` remains a multiplexed tool with an `action` argument. Channel-specific action gates and plugin hooks exist, but they do not provide the requested config-level `tools.deny: ["message:send"]` behavior or define `/tools/invoke`/Lobster privileged-send semantics.

## Best Possible Solution

Keep #13948 open as the canonical action-level tool-policy tracker. The best implementation path is to extend the core config schema/types/docs and policy evaluator with action-aware matching for multiplexed tools such as `message` and `gateway`, specify deny precedence and `/tools/invoke`/Lobster behavior, and add regression tests proving `message:send` can be denied while `message:read` remains available.
