## Summary

Keep #13948 open. Current main still enforces tool policy at whole-tool name granularity, while the relevant `message` and `gateway` actions are parsed inside each tool after policy filtering. The provided issue thread also already treats #13948 as the canonical tracker, with related gateway action-level reports #54260 and #46684 closed back toward it.

## Best Possible Solution

Keep #13948 open as the canonical tracker. The best implementation path is to add an action-aware policy contract to config types, schema, generated config help, and docs; extend shared policy enforcement so it can evaluate tool name plus action before multiplexed tool execution; define precedence across existing policy scopes; and add regression coverage proving cases like `message:send` denied while `message:read` remains available and `gateway:config.patch` denied while `gateway:config.schema.lookup` remains available.
