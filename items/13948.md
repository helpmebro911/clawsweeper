## Summary

Keep #13948 open. Current main still models tool policy as whole-tool allow/deny lists and filters tool objects by `tool.name` before execution, while `message` and `gateway` remain multiplexed tools whose `action` argument is parsed only inside `execute`. The provided issue context also shows this issue is the canonical tracker, with related action-level reports #46684 and #54260 closed back toward it.

## Best Possible Solution

Keep #13948 open as the canonical action-level tool policy tracker. The best path is to add a typed and documented action-aware policy contract, extend shared enforcement so it can evaluate both tool name and action before multiplexed execution, define precedence and bypass semantics, and add regression coverage for cases like `message:send` denied while `message:read` remains available and `gateway:config.patch` denied while `gateway:config.schema.lookup` remains available.
