## Summary

Keep this issue open. Current main still contains the same conservative session-lock gap described in the issue discussion and prior ClawSweeper comment: same-PID lock files without a verifiable starttime can be treated as orphaned when absent from the process-local held-lock map, and current macOS behavior cannot use Linux starttime identity checks. The recent session-lock-scope change reduces one pre-model startup blocking case, but it does not fix the dead-pid/no-starttime cleanup path or prove immediate Telegram delivery after tool execution. PR #71903 remains the open implementation candidate.

## Best Possible Solution

Keep this issue open and review, land, or supersede PR #71903 with a conservative session-write-lock fix. The fix should preserve live same-PID/no-starttime locks when ownership cannot be proven, retain safe cleanup for dead or verifiably recycled PIDs, and add regression coverage for the macOS/no-starttime tool-execution delivery path plus any `sessions_history` or restart-recovery behavior needed after stale lock cleanup.
