## Summary

Keep #38846 open. Current `main` still has only the existing narrow Windows `cmd.exe` metacharacter guard in `src/process/exec.ts`; it does not export the proposed validator or add null/control-character validation. The PR also is not safe to close as implemented or merge as-is: the supplied review comments remain material because the current cmd wrapper escapes the resolved `.cmd`/`.bat` path and each argument, so the PR's blacklist over `\`, `/`, and `=` would reject normal Windows paths, switches, package specs, and npm-style flags.

## Best Possible Solution

Keep the PR open for rework, or replace it with a narrower canonical Windows command-hardening issue before closing. A safe fix should harden the Windows `cmd.exe` wrapper boundary without rejecting normal paths, slash switches, scoped package names, or equals flags; keep PowerShell-specific checks scoped to PowerShell execution; redact argument values in errors; and add focused Windows regression tests for `.cmd`/`.bat`, package-manager fallback, scoped package specs, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely break Windows `.cmd`/`.bat` execution for normal absolute paths containing backslashes.
- The proposed `/` and `=` blacklist would reject valid Windows switches, scoped npm package specs, and npm/npm-style flags already used by OpenClaw flows.
- The proposed error messages echo full argument strings, which could expose secrets in thrown errors or logs.
- Closing now would drop an unimplemented security-hardening thread without a narrower canonical issue or replacement PR.
