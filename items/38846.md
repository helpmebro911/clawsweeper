## Summary

Keep open. Current main has Windows cmd.exe hardening, but it has not absorbed this PR's proposed validator/export/PowerShell-pattern work, and the PR discussion identifies concrete unresolved regressions in normal Windows command-wrapper behavior. This is core process-execution/security surface, not a ClawHub item or stale issue close.

## Best Possible Solution

Keep the PR open for maintainer or author follow-up. The safe path is to rework the hardening around the actual cmd.exe wrapper contract: preserve Windows path separators, scoped npm package names, and equals-style flags; avoid logging full argument values; add focused regression tests for .cmd/.bat wrappers, npm/pnpm fallback, scoped package installs, equals flags, and any null/control-byte blocking; then reassess whether a public validator export is a stable API OpenClaw wants to support.
