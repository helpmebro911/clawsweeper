## Summary

Keep open. Current `main` still does not fail Gateway PROBE mode immediately on malformed/non-JSON WebSocket frames, so the reported `gateway status` failure path remains plausible and is not implemented on `main` or in the latest tagged release available in this checkout.

## Evidence

- **Current client only debug-logs parse failures:** `GatewayClient.handleMessage()` catches `JSON.parse` failures and only calls `logDebug(...)`; it does not notify `onConnectError`, close with a protocol error, or mark probe mode terminal.
  - file: [src/gateway/client.ts:732](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/gateway/client.ts#L732)

- **Probe only fast-fails before hello:** `probeGateway()` settles on close only when `connectLatencyMs == null`. After a successful hello, an unexpected close caused by malformed WS traffic does not have a dedicated parse-error fast-fail path here.
  - file: [src/gateway/probe.ts:249](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/gateway/probe.ts#L249)
