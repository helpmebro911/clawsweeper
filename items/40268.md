## Summary

Current main still does not implement tolerant read-time stripping of unknown config keys. The root schema remains strict, the normal read/load paths still call strict validation and mark unknown-key configs invalid, and startup/hot reload recovery uses last-known-good restoration or narrow repair paths rather than accepting the edited config with unknown keys stripped. The discussion also points to #42201 as an open implementation candidate, so this issue should stay open until that work is reviewed and either lands or is closed.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and either land or close #42201; if it lands, close this as implemented with regression coverage for startup and hot reload accepting unknown keys while strict write/config-validate behavior and invalid known-field failures remain intact. If #42201 is abandoned, implement the same behavior directly in the config read/snapshot path, with focused tests for the root and nested misplaced-key examples from this issue. #40317 can remain the broader validation-mode discussion, but this issue has the concrete crash-loop reproduction and should stay as the focused bug until the fix ships.

## Risks / Open Questions

- Closing now would hide a real remaining behavior gap: last-known-good recovery can reduce outage impact, but it does not accept a valid config after stripping arbitrary unknown keys as requested.
- A future fix needs careful review so tolerant handling only strips genuinely unknown keys and does not silently accept invalid known fields, legacy config, SecretRef policy violations, plugin-local invalidity, or union/discriminator mistakes.
