## Summary

Current `main` already avoids this false positive. The cron payload-kind normalizer now returns `true` only when it actually rewrites `payload.kind`, and the doctor warning counter is only incremented when that mutation happens. The same logic is present in the shipped `v2026.4.22` release.

## Evidence

## Close Comment

Closing this as implemented on current `main`. In this Codex review, I checked `src/commands/doctor-cron-store-migration.ts:71` and the payload-kind normalizer now returns `true` only when it actually rewrites `payload.kind`; if the stored value is already canonical (`agentTurn` / `systemEvent`), it returns `false`. The doctor warning counter is only incremented from that return path at `src/commands/doctor-cron-store-migration.ts:334`, so already-normalized jobs should no longer be reported as needing payload-kind normalization. I also verified the same logic is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see this on `v2026.4.22` or newer, please open a fresh issue with the exact `jobs.json` shape and `openclaw doctor` output.
