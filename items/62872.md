## Summary

Current main still leaves the reported persistent-session `sessions_send` duplicate-message behavior unresolved. The merged #69817 change is a narrow parent-owned one-shot ACP child fix; normal peer sessions and persistent ACP/session targets still enter the A2A ping-pong plus announce path, and the tool still has no per-call ping-pong disable/max-turn option. #62814 was closed as a duplicate, so this remains the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely fix belongs in `src/agents/tools/sessions-send-tool.ts` and `src/agents/tools/sessions-send-tool.a2a.ts`: make `sessions_send` context-aware for persistent peer-session sends or add an explicit per-call ping-pong max/disable option, while preserving the shipped parent-owned one-shot ACP child skip and adding regression coverage for persistent peer-session sends.

## Risks / Open Questions

- The global `session.agentToAgent.maxPingPongTurns=0` workaround can reduce the symptom, but it disables intended A2A chaining everywhere rather than solving persistent peer sends per call or per context.
- Changing this path too broadly could break legitimate peer A2A follow-up/announce workflows, so the fix should distinguish persistent peer sessions from the parent-owned one-shot ACP child path already shipped.
- Closing this now would drop the remaining canonical tracker; #62814 is closed as a duplicate and #69817 only resolved the ACP-child slice.
