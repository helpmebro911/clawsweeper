## Summary

Keep open. Current main still makes `doctor.memory.status` synchronously await a live embedding probe before responding, and the built-in memory manager implements that probe by embedding `["ping"]` through the normal batch embedding path with remote/local batch timeouts of 120s/600s. I did not find a cache, `lastProbeAt` state, strict default probe budget, or separate lightweight memory-health method on this gateway path. Related memory probe issues (#44426 and #47235) overlap, but they do not fully supersede this narrower gateway/control-plane latency ask. One part of the report is weaker on current main: the channel health monitor itself appears to use channel runtime snapshots, not `doctor.memory.status`, so that coupling needs fresh confirmation before driving a fix.

## Best Possible Solution

Keep #71568 open and fix the gateway memory-status contract. The best path is to make `doctor.memory.status` return fast by default from cached per-agent/provider probe state with `lastProbeAt`/`lastProbeOk` metadata, put the long live embedding probe behind an explicit deep/refresh path or separate method, and add regression tests proving repeated gateway status calls do not invoke `probeEmbeddingAvailability()` inside the cache window and that timeout/unavailable probe results are reported as live-provider state rather than blocking core status/dreaming data.
