## Summary

Keep open. Current main still implements `doctor.memory.status` as a live status handler that awaits the memory manager embedding probe before responding, and the built-in memory manager still probes by embedding `["ping"]` through the normal batch embedding timeout path. I did not find a shipped cache, strict short-budget default, or separate lightweight memory-health path on main, so the issue remains actionable.

## Best Possible Solution

Keep the issue open and fix the memory status control-plane path in core/plugin-owned memory runtime: cache the embedding-readiness result per agent/provider/backend with a short TTL and `lastProbeAt`/`lastProbeOk`, or make the default `doctor.memory.status` return last-known readiness while moving the long live embedding probe behind an explicit deep path. Add targeted tests proving repeated status calls do not invoke `probeEmbeddingAvailability()` inside the cache window and that doctor/status do not wait on the full embedding batch timeout by default.
