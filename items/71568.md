## Summary

Keep open. Current main still routes `doctor.memory.status` through a live embedding availability probe that calls `embedBatchWithRetry(["ping"])` with the normal batch timeout path, so the reported control-plane stall remains plausible and actionable. The provided GitHub context also shows PR #72443 is open, unmerged, and uses closing syntax for this issue, so the issue should stay open until that implementation candidate is reviewed and merged or rejected.

## Best Possible Solution

Keep this issue open and review PR #72443 as the current implementation candidate. The likely product/code path is to make default doctor/status memory readiness fast and bounded by caching last-known embedding probe readiness, or moving long live embedding probes behind an explicit deep check, with focused regression tests in `extensions/memory-core` and the doctor/status path as needed.
