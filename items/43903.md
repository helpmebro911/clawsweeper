## Summary

Current main still lacks the requested multi-token gateway auth model. Token mode is a single shared secret with no per-token label/scope identity, while the existing identity-bearing alternatives are trusted-proxy and Tailscale Serve rather than the simpler token-mode workflow requested here.

## Best Possible Solution

Keep this open for a deliberate core gateway design: add a backwards-compatible multi-token auth contract, support SecretInput-backed token entries with non-secret label/scopes metadata, resolve matched token identity into HTTP/WebSocket auth results, propagate that identity into Control UI sender/session metadata, preserve current single-token and device-pairing behavior, and update config schema, docs, and gateway auth/session tests.

## Risks / Open Questions

- This is a security-sensitive gateway auth/session contract change. A future implementation needs careful secret redaction, SecretRef/env precedence, rate-limit behavior that does not leak token inventory, and constant-time matching across multiple configured tokens as far as practical.
- Trusted-proxy and Tailscale identity paths partly solve identity today, but they require external identity infrastructure and do not preserve the requested token-mode/local CLI/device-pairing workflow.
