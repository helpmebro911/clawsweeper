## Summary

Keep this issue open. Current main still has the reported DAVE-disabled Discord voice pre-Ready abort shape: `/vc join` passes `voice.daveEncryption` through to `@discordjs/voice`, waits for `VoiceConnectionStatus.Ready` with a fixed 15s timeout, destroys the connection on readiness failure, and only records a voice session after Ready. The receive/DAVE recovery path is post-Ready and does not cover a voice WebSocket close after Hello before UDP handshaking. Related open reports #39825 and #63098 are adjacent, but the supplied discussion and local ClawSweeper records do not show a maintainer-designated canonical replacement, and this issue adds useful Hello-before-Ready diagnostics.

## Best Possible Solution

Keep this issue open until maintainers decide the intended non-DAVE Discord voice behavior. The likely implementation work belongs in `extensions/discord/src/voice/manager.ts`: add explicit pre-Ready close/abort diagnostics and regression coverage for `entersState` failure before session activation, then either implement bounded retry/fallback to DAVE-enabled join or reject/deprecate `channels.discord.voice.daveEncryption=false` with clear operator guidance if Discord now requires DAVE. If maintainers consolidate with #39825 or #63098, preserve this issue's Hello-before-Ready trace in the canonical thread before closing it as duplicate.
