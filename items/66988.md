## Summary

Keep this issue open. Current main still returns `raw: null` from the redaction fallback and the Control UI still treats non-string raw snapshots as Raw mode unavailable, forcing Form mode and showing the reported disabled message. The older #59330/#59336 thread tracks one root cause for excessive fallback, but this issue’s requested display-vs-editability contract remains a separate product/security decision and is not implemented on main.

## Best Possible Solution

Keep this issue open unless maintainers decide that unsafe raw round-trips must hide Raw mode entirely. A durable fix should separate raw display availability from raw editability in the config snapshot/UI contract, preserve safely redacted raw text for read-only viewing when possible, expose an explicit unsafe-round-trip or editability reason, keep save/apply blocked while unsafe, and update gateway, Control UI, docs, and regression coverage. Continue using #59330/#59336 for the narrower safe-bin/materialized-config false-positive fallback work.

## Risks / Open Questions

- The current force-Form behavior is documented and covered by tests as a safety policy, so changing it should be an explicit maintainer product/security decision rather than a one-line reversal.
- Showing redacted raw text while blocking edits needs a clear snapshot/UI contract; otherwise the UI could imply that raw edits are safe when the gateway cannot safely round-trip that snapshot.
- The related #59330/#59336 path may reduce false-positive fallback causes, but it would not by itself implement read-only raw display for legitimate unsafe fallback cases.
