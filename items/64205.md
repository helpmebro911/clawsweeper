## Summary

Keep open. Current main still logs MCP startup failures and drops the failed server from the session catalog; it does not preserve agent-facing diagnostics, block required servers, or expose expected-vs-available tool awareness to the embedded runner or prompt.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostics path: preserve redacted failed-server records in the session MCP catalog, expose those diagnostics to embedded runner prompt/channel context, add typed `mcp.servers.<name>.required` semantics that can block agent start and notify the active channel, add expected-vs-available tool awareness where feasible, and cover stdio/SSE/streamable-HTTP startup failures with focused runtime, runner, config, and docs tests.

## Risks / Open Questions

- Startup diagnostics must redact sensitive command args, URLs, headers, and environment values before reaching the model or channel.
- Required-server semantics need a transport-agnostic contract so optional MCP failures can degrade while required server failures can stop agent start with a user-visible message.
- Because unknown MCP server fields are accepted today, users can configure `required` or `onStartFailure` and mistakenly believe they are enforced.
