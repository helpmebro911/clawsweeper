## Summary

Current main still appears to have the reported restart/resume gap. Exec approvals are kept in a process-local in-memory manager, stale approval IDs still resolve as `INVALID_REQUEST`, and the session replay/restart recovery paths I inspected do not invalidate or re-register persisted `approval-pending` exec tool results after a gateway restart. The newer approval replay runtime only replays approval notifications from the live gateway manager, so it cannot recover approvals lost with the prior process.

## Best Possible Solution

Keep this issue open. The fix should live in session replay/restart recovery or in a deliberate approval-manager persistence contract: when resumed history contains an exec result with `status: "approval-pending"` whose ID is absent from the live manager, rewrite it to an expired/error result so the agent re-issues exec, or securely persist and re-register pending approvals with original command/cwd/session/host/caller binding metadata. Add a regression test that simulates gateway restart with an approval-pending transcript tail and proves the resumed turn does not keep resolving the stale ID.
