## Summary

Keep this issue open. Current main still treats missing or unauthorized credentials for the selected remote provider as model-fallback signals, and the Cloudflare AI Gateway plugin/config path still creates auth for the `cloudflare-ai-gateway` provider only rather than sharing credentials across additional custom gateway-backed providers. Main has better fallback-decision logging, so the failure is less silent in logs, but the reported product problem remains.

## Best Possible Solution

Keep this issue open. Implement a generic policy for explicitly selected remote providers: if the selected provider has no credentials or returns a clear auth/missing-key error, fail with a configuration error unless the session/run explicitly opted into fallback. Add regression coverage for a Cloudflare AI Gateway-style second provider without `apiKey`, and update https://docs.openclaw.ai/providers/cloudflare-ai-gateway plus https://docs.openclaw.ai/concepts/model-failover to explain per-provider SecretRef reuse or a deliberately scoped shared credential mechanism.

## Risks / Open Questions

- Changing auth-error fallback is shared model infrastructure and must preserve intentional auth profile rotation plus explicitly configured fallback chains.
- A gateway-level credential sharing mechanism must be explicit; inferring shared credentials only from similar base URLs could leak a provider key to an unrelated provider.
- Docs currently document auth failures as fallback triggers, so any behavior change needs coordinated docs and regression tests.
