## Summary

Keep open. Current main still resolves missing credentials for remote custom providers as a concrete `No API key found` error, but that error is classified as auth failover and the model fallback chain still appends `agents.defaults.model.primary` for override/current models. The docs also still describe auth failures falling through model fallback and do not document a shared Cloudflare AI Gateway credential pattern for additional providers behind the same gateway.

## Best Possible Solution

Keep this issue open and implement a generic policy change: missing credentials for an explicitly selected remote provider should surface as a clear error unless that run/agent has an explicit fallback policy. Add a regression test for a Cloudflare AI Gateway-style second remote provider without `apiKey`, and update `docs/providers/cloudflare-ai-gateway.md` plus model failover docs to show repeated per-provider `apiKey` or a supported shared credential mechanism if one is introduced.

## Risks / Open Questions

- Changing auth-error fallback is shared model infrastructure; a broad change could regress intentional auth-profile rotation, rate-limit failover, or operator-configured model fallback unless scoped carefully.
- A Cloudflare-only patch would not fully address the generic remote custom-provider case where a selected provider id has no usable credential and currently falls through fallback.
- The public docs should be updated with the behavior change or with explicit repeated `apiKey`/SecretRef guidance so users do not infer that onboarding one Cloudflare AI Gateway provider automatically covers sibling provider ids.
