## Summary

Keep this PR open. Current main still does not expose a unified `auth.providerUsage` JSON schema or OpenRouter metered billing/account data in `openclaw models status --json`, and the PR is a real implementation candidate. It should not be closed as stale cleanup, but it needs maintainer review and rework around the current provider-owned usage hook boundary plus the open latency/error-handling concerns.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate. Ask for a rebase onto current main and reshape the implementation so OpenRouter-specific auth, HTTP calls, and response parsing live in `extensions/openrouter` through `resolveUsageAuth`/`fetchUsageSnapshot`, while core gets a documented additive discriminated JSON schema if metered usage needs to be represented alongside existing windowed snapshots. Gate OpenRouter billing fetches to output modes that render them, preserve existing rich OAuth usage visibility, surface non-2xx and partial failures accurately, and add focused command/fetcher tests plus docs for the new JSON fields.

## Risks / Open Questions

- The PR head appears to place OpenRouter-specific billing fetch/parsing in core `src/infra` instead of the bundled OpenRouter plugin, while current main already has provider-owned usage hooks for this class of behavior.
- Review comments indicate OpenRouter billing may still be fetched for default non-plain status even though the metered data is only emitted in JSON, adding avoidable network latency and account-query side effects.
- Review comments indicate non-2xx or partial OpenRouter billing/key failures can be collapsed into missing data or an overly healthy status, making JSON health inaccurate for revoked keys or mixed multi-key setups.
- The new JSON output exposes billing/account and per-key profile usage metadata; that is not a supply-chain regression, but it should be treated as sensitive operational output and documented before merge.
- The PR was based on an older main and current main now has provider-usage plugin boundary changes, so a rebase and design refresh are likely needed before review.
