## Summary

Keep open. Current `main` still lacks the Discord doc clarifications from PR #59904, while the runtime code supports the PR’s corrected wording and the PR discussion shows the author already addressed the initial review concerns.

## Evidence

- **Current docs still omit the requested clarification:** The current Discord troubleshooting docs only say bot-authored messages are ignored by default and recommend `allowBots="mentions"`; they do not explain that guild sender access is separately gated by guild/channel `users` and `roles`, or that `allowFrom` does not use a `bot:` prefix. See `docs/channels/discord.md:1114`-`1118`.
  - file: [docs/channels/discord.md:1114](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/channels/discord.md#L1114)
  - command: `nl -ba docs/channels/discord.md | sed -n '1114,1119p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Runtime allowlist parsing matches the PR claim about bare IDs vs prefixes:** Discord allowlists accept bare numeric IDs and recognized prefixes `discord:`, `user:`, and `pk:`. There is no `bot:` prefix handling, so the PR’s note that bot IDs should be listed as bare IDs is consistent with current code in `extensions/discord/src/monitor/allow-list.ts:57`-`87`.
  - file: [extensions/discord/src/monitor/allow-list.ts:57](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/discord/src/monitor/allow-list.ts#L57)

- **Runtime member access uses users OR roles:** Guild sender authorization is computed from `channelConfig.users ?? guildInfo.users` together with `channelConfig.roles ?? guildInfo.roles`, and a sender is allowed when either the user allowlist or role allowlist matches. See `extensions/discord/src/monitor/allow-list.ts:189`-`243`, including `return userOk || roleOk;`.
  - file: [extensions/discord/src/monitor/allow-list.ts:189](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/discord/src/monitor/allow-list.ts#L189)

That matches the PR author’s final wording adjustment that `allowBots` is orthogonal rather than the primary guild gate.
