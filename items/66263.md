## Summary

Current main still does not satisfy the native Codex compaction accounting request. The Codex app-server compaction path reports success without parsing or returning any post-compaction token snapshot, and both session-row and status/reporting fallbacks can still promote transcript-derived pre-compaction usage back into a fresh-looking current total after the store total is cleared. The merged #64669 fix is useful but explicitly scoped to cumulative token projection, not native compaction boundaries or transcript fallback.

## Best Possible Solution

Keep this issue open and implement the native Codex compaction accounting boundary end to end. The fix should parse and propagate a reliable post-compaction token snapshot when the app-server exposes one; otherwise persist an explicit unknown/stale post-compaction state, add or reuse a compaction boundary that transcript/status fallback actually honors, and add regression coverage for known tokensAfter, missing tokensAfter, and post-compaction session/status fallback. Keep #64669 as the already-merged narrower cumulative-usage guardrail, not the canonical fix for this issue.
