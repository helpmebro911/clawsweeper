## Summary

Keep this issue open. Current main still wires Discord inbound processing only from message-create events, reactions, thread updates, and optional presence updates; there is no messageUpdate/messageDelete listener path, no edit-aware replay semantics, and no active-run cancel index keyed by the source Discord message id. Related #38505 remains open and covers only the edit half, so it does not supersede this broader edit-and-delete request.

## Best Possible Solution

Implement this in the bundled Discord plugin. Add Carbon messageUpdate and messageDelete handling in the Discord monitor listener layer, register those listeners during Discord startup, route eligible edits through the existing mention/preflight/message processing path with edit-aware replay or supersede semantics, and add a precise Discord source-message-id to active job/run abort mapping so delete events cancel only the matching in-flight work. Coordinate the edit half with #38505, but keep this broader tracker open until both edit-to-reprocess and delete-to-cancel are shipped.

## Risks / Open Questions

- Carbon messageUpdate payloads may be partial, so a real implementation likely needs message hydration and content-change gating before reprocessing.
- Delete-to-cancel needs to cancel only the active run triggered by the deleted Discord message; cancelling by channel or session alone could abort newer unrelated work.
- Edit replay needs an explicit dedupe/supersede design so edited content is not dropped as an already-processed Discord message id.
