## Summary

Keep this issue open. Current main at 348728c28c1ea9ae7c5824350baba35ee08a275e still does not implement Discord edit-to-reprocess or delete-to-cancel: the bundled Discord plugin only wires message create, reaction, thread update, and optional presence listeners; the inbound queue is keyed by routed session rather than by source Discord message id; and the generic chat abort path is not connected to Discord message delete events. Related #38505 remains open and covers only the messageUpdate/edit half, so it does not supersede this broader request.

## Best Possible Solution

Keep this open and implement it in the bundled Discord plugin. Add MessageUpdateListener and MessageDeleteListener handling in extensions/discord/src/monitor/listeners.ts and register them from provider.startup; reuse the existing preflight/message processing path for edited content with content-change and mention gating; track the triggering Discord message id to the accepted session/run or pending job; on delete, abort only the matching active work through the existing chat abort/run abort surface. Add focused tests for listener registration, edit re-dispatch despite replay dedupe, mention filtering, active-vs-completed edit behavior, and delete cancellation. Coordinate with #38505 for the edit-only half, but keep this broader issue as the delete-to-cancel tracker too.
