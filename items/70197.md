## Summary

Keep open. Current main still ships a default-enabled Bonjour gateway discovery plugin that dynamically imports `@homebridge/ciao`, stages `@homebridge/ciao@1.3.6` without an OpenClaw patch, and starts local discovery by default unless `discovery.mdns.mode` is `off` or `OPENCLAW_DISABLE_BONJOUR` is set. The pinned upstream ciao source still runs the Windows `arp -a | findstr` command via `child_process.exec()` on a 15s polling loop without `windowsHide`. Existing OpenClaw Windows `windowsHide` fixes cover OpenClaw-owned process helpers, not ciao's internal `exec()` call. The prior related OpenClaw issue #25856 was stale-closed with instructions to reopen with fresh repro data, which #70197 provides; #58566 and #40340 are different spawn surfaces.

## Best Possible Solution

Keep #70197 open as the active OpenClaw-tracked Windows UX bug for the bundled Bonjour/ciao path. The likely maintainer fix is either to upgrade to an upstream ciao release that adds `windowsHide`, carry a scoped OpenClaw patch for `@homebridge/ciao@1.3.6` adding `{ windowsHide: true }` to the Windows ARP `exec()` call, or change Bonjour startup policy on Windows if maintainers prefer lazy/on-demand advertising. Verification should include a Windows gateway run with Bonjour enabled and a regression check proving no visible `cmd.exe` window is spawned by the ciao ARP poll.
