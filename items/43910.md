## Summary

Keep this PR open. Current main still does not expose or honor configurable `memorySearch.remote.encodingFormat`, and the PR remains a relevant implementation candidate, but it is stale against the current memory-host-sdk/provider-plugin architecture and needs more work before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase the configurable `encodingFormat` design onto the current memory-host-sdk and provider-plugin architecture, update schema/types/help/labels/docs plus package mirror paths, propagate the setting through resolved memory config, direct requests, OpenAI/Voyage batch requests, and parsers, add robust base64 decode validation and tests, and coordinate with #41670 on whether OpenClaw wants a narrow explicit-float fix or a broader configurable feature.

## Risks / Open Questions

- The PR should not merge as-is because it targets obsolete `src/memory/*` paths and does not update the active `src/memory-host-sdk/*`, `packages/memory-host-sdk/*`, and bundled provider-plugin implementations on current main.
- Base64 decoding is security-relevant untrusted input handling. A rebased implementation should validate that decoded bytes are well-formed float32 data, avoid typed-array alignment pitfalls, reject malformed payloads cleanly, and add regression tests for valid and invalid base64 responses.
- No supply-chain regression is visible from the provided PR file list: it does not touch workflows, package metadata, lockfiles, dependency sources, install/build/release scripts, or secret handling. The main security review concern is runtime parsing of external embedding-provider responses.
- The related #41670 is still open and narrower: it proposes always sending `encoding_format: "float"` for compatibility. Maintainers should decide whether to land a minimal float compatibility fix there or a broader configurable feature here.
