## Summary

Keep open. Current `main` still treats Anthropic 1M context as an explicit `params.context1m: true` opt-in, and the PR discussion surfaces a real planner-vs-request capability mismatch rather than evidence that this was already shipped or is obsolete.

## Evidence

- **Planner still requires `context1m: true`:** `resolveContextTokensForModel()` only returns `ANTHROPIC_CONTEXT_1M_TOKENS` for Anthropic 1M models when `modelParams?.context1m === true`; without that flag it falls through to configured/discovered context windows.

## Risks / Open Questions

- The PR review thread identified a concrete risk: changing session/planner context limits without matching `resolveAnthropicBetas()` behavior can overstate available context for OAuth or other non-1M-capable auth flows.
- Upstream Anthropic capability and model coverage need maintainer verification before merging any automatic 1M behavior, especially because the current OpenClaw contract and tests are explicitly opt-in.
