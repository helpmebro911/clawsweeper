## Summary

Keep #13870 open. Current main displays client-supplied paired-device displayName values in the CLI and Control UI, and storage can refresh that metadata on reconnect, but there is still no operator-owned paired-device rename/label workflow in the CLI, gateway protocol, server handlers, Control UI actions, or devices docs. Related #23916 and closed #57424 overlap on device identification, but they do not supersede this issue's specific label-management request.

## Best Possible Solution

Keep #13870 open until paired-device label management is implemented end to end: add an additive gateway method/schema/handler for paired-device rename, persist an operator-owned display-name override with a clear precedence rule over reconnect/client metadata, expose `openclaw devices rename --device <id> --name <displayName>`, add a Control UI rename action, cover storage/protocol/CLI/UI behavior with focused tests, and update the public devices docs. Keep `node.rename` separate, and let #23916 continue tracking broader device metadata display.
