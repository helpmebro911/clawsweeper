## Summary

Keep #13870 open. Current main displays client-supplied paired-device displayName metadata in the CLI and Control UI, and reconnects can refresh that metadata, but the requested operator-owned paired-device rename/label workflow is still missing from the CLI, gateway protocol/method list, server handlers, Control UI actions, docs, and persistence semantics. Related #23916 tracks broader device metadata display, and closed #57424 overlaps on device identification, but neither supersedes this specific rename request.

## Best Possible Solution

Keep #13870 open until paired-device label management is implemented end to end: add an additive gateway method/schema/handler for paired-device rename, persist an operator-owned display-name override with a documented precedence rule over reconnect/client metadata, expose `openclaw devices rename --device <id> --name <displayName>`, add a Control UI rename action, cover storage/protocol/CLI/UI behavior with focused tests, and update the public devices/protocol docs. Keep #23916 focused on broader metadata display and keep `node.rename` separate.

## Risks / Open Questions

- Closing this now would lose the specific operator-owned label-management request; current displayName support is only client-supplied metadata display/refresh.
- A naive implementation that reuses `displayName` directly would allow reconnect metadata to overwrite operator labels unless a clear override/precedence rule is added.
- `node.rename` is a separate node surface and does not cover browser/Control UI/CLI paired-device records.
