## Summary

Keep #72394 open. Current main still keeps system-event queues scoped to the exact session key: heartbeat preflight peeks the base session, isolated heartbeat creates/stores a separate `:heartbeat` session entry, and active-session event inspection then peeks the isolated key without cloning or moving base queued entries. Existing tests cover the current asymmetry rather than proving a transfer, and `memory-core` has a local workaround that checks the base key for isolated heartbeat cron events.

## Best Possible Solution

Keep this open for a narrow core contract fix. The best path is to add an explicit, tested transfer/read-through strategy for pending system-event entries at the isolated heartbeat boundary, likely in `src/infra/heartbeat-runner.ts` with a small helper in `src/infra/system-events.ts`, while preserving the existing guard against stale base-session delivery routing. Add regression coverage that future consumers of `peekSystemEventEntries(runSessionKey)` on `:heartbeat` see the intended pending entries, and document the contract if plugin-facing semantics change.
