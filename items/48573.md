## Summary

Keep open. Current main has useful partial mitigations, including subagent abort cascading and ACP task tracking, and the discussion correctly identified the ACP gap. But the central bug is still not fully solved on main: ACP child sessions are still persisted as session-store entries, task maintenance treats any ACP session entry as live backing state, and explicit subagent termination marks registry/session timing rather than synchronously deleting the child session state. The linked PRs #48583 and #68718 may become the canonical fix if merged, but they are open and not present in the checked-out main tree.

## Evidence

- **ACP spawns still create durable child session entries:** `spawnAcpDirect` creates `agent:<id>:acp:<uuid>` with `sessions.patch`, dispatches the child run, and only records a `createRunningTaskRun` after dispatch. This proves ACP child state is still backed by persisted session entries on current main.

- **ACP task maintenance does not detect parent-dead zombie sessions:** For `runtime === "acp"`, maintenance considers the task backed as long as `readAcpSessionEntry` returns an entry. It does not check parent liveness or reconcile `spawnedBy` against an active parent process/session before deciding the task is still backed.
