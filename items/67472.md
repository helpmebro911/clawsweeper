## Summary

Keep this PR open. Current main still uses the existing `tar.c` backup path with no archive-time ENOENT/ENOTDIR skip handling, and the linked backup race remains plausible. The PR is a focused implementation candidate, but it should be revised rather than closed because manifest consistency, verification behavior, and the tar writer contract still need proof.

## Best Possible Solution

Keep this PR open and request revision, or land an equivalent superseding fix. The best path is to handle archive-time ENOENT/ENOTDIR for volatile session files in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for skipped payloads, add a regression test that simulates a session transcript disappearing during packing, and coordinate explicitly with #67417 and #72251 before closing any implementation candidate.

## Risks / Open Questions

- Merging the submitted patch unchanged could produce an archive whose embedded manifest still declares payloads omitted during packing, which can make `backup verify` fail or misrepresent the artifact.
- The proposed tar mock, as described in review context, cannot faithfully test the swallowed file-level ENOENT success path because it also emits a stream error.
- Replacing `tar.c` with direct `tar.Pack` stream wiring needs explicit proof for gzip, portable mode, path remapping, stream close/error ordering, temp archive cleanup, and atomic publish semantics.
- Related broader live-file backup work in #72251 is still only a candidate from the provided context and does not currently ship this fix on main.
