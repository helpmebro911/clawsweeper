## Summary

Current main does not implement PR #47007's central request. Feishu has streaming-card reply support for inbound runs and subagent completion routing, but subagent/manual delivery still resolves to final payload delivery rather than a channel-native live streaming bridge. The PR also has unresolved review findings around duplicate main-run streaming/delivery, thread id loss, blockStreaming semantics, and throttling, so it is not a conservative close-as-implemented or merge-ready cleanup item.

## Best Possible Solution

Keep PR #47007 open unless a maintainer wants to close it manually as abandoned. The best technical path is a fresh, lane-scoped implementation against current src/agents/* that wires existing Feishu replyOptions only for subagent/outbound streaming, preserves replyTo/thread ids, avoids main-run duplicate delivery, respects blockStreaming defaults and card update throttling, and adds focused tests for Feishu reply-dispatcher, streaming-card, subagent routing, and agent delivery behavior.
