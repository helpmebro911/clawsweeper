## Summary

Current main has related stale-port cleanup, but it has not implemented the issue's requested two-phase macOS LaunchAgent restart. `restartLaunchAgent` still prefers `launchctl kickstart -k` on the loaded service, tests explicitly assert no `bootout`/`bootstrap` on the normal restart path, and the detached handoff path also uses `kickstart -k` after waiting for the caller PID.

## Best Possible Solution

Keep this issue open. A maintainer-quality fix should decide whether macOS restart should move from the current `kickstart -k` flow to an explicit loaded-service stop/wait/start sequence, probably in `src/daemon/launchd.ts` plus `src/daemon/launchd-restart-handoff.ts`, with tests that assert the new command ordering and preserve recovery for not-loaded services and in-process detached handoff restarts.

## Evidence

## Risks / Open Questions

- Changing this path could regress macOS launchd service-state recovery, especially the already-loaded/not-loaded bootstrap fallback and the detached self-restart handoff. The existing tests currently encode the opposite ordering, so implementation should update tests alongside behavior.
