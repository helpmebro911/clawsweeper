## Summary

Keep this issue open. Current main still lacks a native Control UI page contribution seam: navigation and page rendering are static, bootstrap config has no plugin page/import-map metadata, the manifest loader does not preserve `ui.pages`, and the plugin SDK only exposes routes/services rather than Control UI tabs. The discussion includes a real Openclaw-Quest workaround and unresolved API/security questions, so this remains valid product/API design work rather than conservative cleanup.

## Best Possible Solution

Keep this issue open as the canonical design and implementation track for a versioned Control UI plugin-page extension API. A good solution would add manifest or SDK page declarations, gateway bootstrap metadata, a browser Custom Element loader, route/navigation grouping, docs/tests, and explicit security decisions for non-Lit Custom Elements and plugin-owned backend traffic.

## Risks / Open Questions

- Loading third-party plugin JavaScript inside the Control UI shell is security-sensitive and needs explicit design for CSP, auth scope, gateway client exposure, dependency/runtime sharing, unload behavior, and version compatibility.
- The issue is recent and active, with discussion through 2026-04-27 and no merged implementation on current main; closing now would discard a real plugin API gap.
