## Summary

Keep open. Current `main` still treats Control UI navigation as a fixed built-in tab set, the gateway bootstrap contract does not expose plugin page metadata, and the plugin manifest/SDK only support separate HTTP routes rather than native in-shell plugin pages. The issue is also recent, concrete, and backed by active discussion plus a real plugin use case (`openclaw-quest`), so there is not strong evidence for any conservative auto-close reason.

## Evidence

- **Control UI navigation is still closed over built-in tabs:** `ui/src/ui/navigation.ts` defines a fixed `TAB_GROUPS` array, a closed `Tab` union, and static `TAB_PATHS`. There is no manifest-fed or registry-fed plugin tab/page extension point on current `main`.
  - file: [ui/src/ui/navigation.ts:5](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/ui/src/ui/navigation.ts#L5)

- **Gateway bootstrap config has no plugin-page payload:** `ControlUiBootstrapConfig` only carries base path, assistant identity, version, media preview roots, and embed settings. It does not include anything like `pluginPages` or a plugin import map for native UI injection.
  - file: [src/gateway/control-ui-contract.ts:1](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/gateway/control-ui-contract.ts#L1)
