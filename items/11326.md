## Summary

Keep #11326 open. Current main still decodes built-in `web_fetch` response bytes through default UTF-8 paths, exposes no encoding parameter/config, and passes the already-decoded string into HTML extraction. The prior Codex review comment is still accurate on current main; related GBK items found by search are Windows exec/process-output issues, not this web_fetch charset gap.

## Best Possible Solution

Keep the issue open until core `web_fetch` decodes response bytes with charset awareness before extraction. The likely fix belongs in `src/agents/tools/web-shared.ts` or a narrow helper used by `web_fetch`: preserve SSRF and `maxResponseBytes` behavior, detect charset from `Content-Type` and early HTML meta bytes, support labels such as `gbk`/`gb2312` when the runtime can decode them, consider a manual encoding parameter/config only if maintainers want that surface, and add focused web_fetch tests plus `docs/tools/web-fetch.md` updates.
