## Summary

Keep #11326 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still decodes built-in web_fetch response bodies through UTF-8/default text paths before extraction, with no Content-Type/meta charset detection, no GBK/GB2312 decoding path, no manual encoding parameter, and no focused regression coverage. The related #56538 work is about Windows node-host exec output decoding, not HTTP response decoding for web_fetch.

## Best Possible Solution

Keep the issue open until core web_fetch performs charset-aware byte decoding before extraction. A narrow fix should preserve SSRF guarding and maxResponseBytes behavior, decode from bounded bytes using Content-Type charset and early HTML meta detection, support GBK/GB2312 labels where the runtime can decode them or via an approved dependency, decide whether a manual encoding parameter/config is warranted, and add focused web_fetch regression tests plus docs/config help updates.
