## Summary

Keep #11326 open. Current main e672b61417af5c45b0431df6d9109a1f4b618ef5 still decodes built-in web_fetch HTTP bodies through UTF-8/default text paths before extraction, with no Content-Type/meta charset handling, no GBK/GB2312 decode path, no manual encoding parameter, and no focused regression coverage. The prior Codex review comment remains accurate on current main; the related GBK work is Windows node-host command-output decoding, not web_fetch response-body decoding.

## Best Possible Solution

Keep the issue open until core web_fetch performs charset-aware bounded byte decoding before extraction. A narrow fix should preserve SSRF guarding and maxResponseBytes behavior, choose encoding from Content-Type and early HTML meta charset where possible, support GBK/GB2312 labels through the runtime or an approved dependency, decide whether a manual encoding parameter/config is warranted, and add focused regression tests plus docs/config help updates.
