## Summary

Keep open. Current main still does not expose inbound media paths, URLs, or MIME types on `PluginHookAgentContext` for `before_agent_reply`, and a handled hook still returns before normal inbound media staging. The PR addresses a real plugin API gap, but it is stale against the current hook-types split and needs tests, docs, and maintainer API/security review before landing.

## Best Possible Solution

Keep this PR open. The best path is to rebase it onto current main, add `mediaPaths`, `mediaUrls`, and `mediaTypes` to `src/plugins/hook-types.ts`, pass normalized defensive copies from `getReplyFromConfig`, document the fields on the public plugin hooks page, and add focused tests for text-only, local-path, URL-only, media-type, and mutation-safety cases before maintainer API/security review.

## Risks / Open Questions

- The PR is stale against current main because public hook context types now live in `src/plugins/hook-types.ts`, while the PR edits the old inline type location in `src/plugins/types.ts`.
- The current PR has no focused tests for text-only turns, local media paths, URL-only attachments, media type cardinality/defaulting, or defensive-copy mutation safety in the `getReplyFromConfig` hook context.
- Exposing absolute inbound media paths and remote media URLs to `before_agent_reply` expands what third-party plugins can observe before normal staging/sandboxing, so the final API shape and docs should be reviewed deliberately.
