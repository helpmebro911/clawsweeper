## Summary

Keep #39248 open. Current main still treats native `sessions_spawn` child sessions as non-main sandbox candidates, resolves sandbox context before the embedded child run starts, and lacks a regression test proving a Docker-backed `agents.defaults.sandbox.mode: "non-main"` child actually initializes and writes message/transcript/token state. The only relevant diff since the prior ClawSweeper review touches unrelated webchat attachment files.

## Best Possible Solution

Keep the issue open and add a targeted regression test or live verification for native `sessions_spawn` with `agents.defaults.sandbox.mode: "non-main"`. The proof should assert the child run starts, writes transcript/messages, updates run/session state such as `systemSent` or token usage where applicable, and surfaces sandbox initialization or dispatch failures instead of silently timing out. If the repro still fails, patch the Docker-backed `resolveSandboxContext` or async child dispatch path with explicit error propagation/logging.
