## Summary

Keep open. Current main has assistant avatar identity support in gateway/Control UI, but the macOS Talk Mode overlay still renders the generic Circle-based TalkOrbView and its model carries only visibility, phase, paused state, and level. The reporter's follow-up is correct that the closed #22199 web/Control UI identity work does not cover this native Talk overlay surface.

## Best Possible Solution

Keep the issue open and implement it in the macOS app: fetch the effective assistant identity for the active Talk session through the existing gateway identity surface, store the resolved avatar state on the Talk overlay/controller path, render it as a circular clipped image/text avatar inside TalkOrbView, preserve the existing rings/arcs/paused visuals, and fall back to the current orb when no renderable avatar is available. Add focused macOS overlay/config tests plus docs if the user-facing Talk Mode behavior changes.

## Risks / Open Questions

- Native macOS implementation should reuse the existing identity/avatar resolution contract instead of directly trusting arbitrary config paths, so local avatar path, symlink, data URL, and authenticated avatar-route behavior stay consistent.
- Remote or authenticated avatar rendering in an always-on overlay may need async loading, caching, and fallback behavior so Talk Mode startup remains fast and still falls back to the orb when resolution fails.
