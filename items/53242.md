## Summary

Keep open. Current main still truncates chat.history text fields at the default 8,000-character projection cap before byte-budget enforcement, and the normal Control UI request path does not pass a larger maxChars value. The same-author PR #53243 remains the implementation candidate in the provided GitHub context and includes closing syntax for this issue, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #53243 or an equivalent gateway fix. The implementation should raise or derive the default chat-history text cap consistently with the byte budget, preserve long assistant text that fits the configured history budget, keep byte-heavy messages readable instead of replacing normal content with placeholders, add a default-path regression test for long assistant text, and align config help/generated docs with the shipped default.
