## Summary

Keep open. Current main still applies a small per-text-field chat history character cap before the byte-budget logic, and the default WebChat/Control UI path does not request a larger `maxChars`. The provided GitHub context also shows #53243 is an open, unmerged same-author PR with closing syntax for this issue, so this issue should remain open until that implementation is reviewed, landed, or explicitly closed.

## Best Possible Solution

Keep this issue open and treat #53243 as the active implementation candidate. The best path is to review and land a gateway fix that raises or derives the default cap consistently with the byte budget, keeps truncation byte-aware, adds regression coverage for a long assistant message that fits the history budget, and aligns config help/generated docs with the shipped default; otherwise close the PR and implement the same behavior directly on main.
