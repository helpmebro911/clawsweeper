## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR is a plausible implementation candidate, but it is not ready to close or merge: its signal recording is wired to `executeJob`, while the live scheduled, startup catch-up, and manual cron run paths use `executeJobCoreWithTimeout` plus shared outcome application instead. The prior review concerns about real-path coverage and recovery classifier loss still hold against current main and the PR diff.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require revision before merge. Move playbook signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and propagate `lastErrorKind`; add tests for the real scheduled, startup catch-up, and manual paths; settle concurrency, retention, and privacy semantics for the new local memory file; and consider splitting the autonomy roadmap doc from the runtime change.

## Risks / Open Questions

- Functional readiness: the PR records signals from `executeJob`, but scheduled ticks, startup catch-up, and manual runs do not use that path on current main.
- Functional correctness: recovery counts can miss entries when failure classification depends on `errorKind`, because persisted cron job state lacks `lastErrorKind`.
- Privacy and retention: the PR introduces durable local memory with job identifiers, job names, and raw error text; retention and operator-data expectations need explicit maintainer review.
- Concurrency: current cron supports `cron.maxConcurrentRuns`; the proposed recorder constructs a fresh file-backed layer and load/save cycle per signal, so parallel due jobs could lose updates without serialized writes or shared state.
- Scope hygiene: the PR bundles a large autonomy roadmap doc with a cron runtime change; maintainers may want that split before detailed review.
- Security pass found no obvious supply-chain or CI regression in the diff, but the new persistent-memory file is a security-sensitive data surface and should not be merged without the privacy/retention decision above.
