## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, the same-author predecessor #42984 was closed unmerged, and this PR is a real implementation candidate rather than obsolete cleanup. It still needs revision before merge because the proposed hook, per the provided PR discussion and current main code paths, misses the scheduler/startup/manual cron execution paths and has a recovery matching gap around unpersisted errorKind data.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a revision before merge. The best path is to move playbook signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual cron.run; propagate and persist errorKind/lastErrorKind where recovery matching depends on it; add tests that exercise onTimer, startup catch-up, and manual run paths; and review the new local memory file's concurrency, retention, and privacy semantics.

## Risks / Open Questions

- Functional readiness risk: as proposed, signal recording is attached to executeJob while scheduled, startup catch-up, and manual cron runs use executeJobCoreWithTimeout plus shared result-application helpers.
- Functional correctness risk: recovery counting can miss entries when the original failure classification depended on errorKind, because current job state and protocol schema do not persist lastErrorKind.
- Persistence/privacy review risk: the PR introduces a durable local memory file containing cron failure signatures, job ids, job names, and last error text; that should be explicitly reviewed for retention and operator-data expectations before merge.
- Concurrency risk: the proposed memory layer loads and saves a JSON file per signal, and current cron supports concurrent due-job workers, so the final design should avoid lost updates once instrumentation moves into live paths.
