## Summary

Keep open. Current main has adjacent WhatsApp recovery hardening, but the specific Windows listener flap/stall report is not proven fixed. The same visible failure signatures remain in current code and docs: `No active WhatsApp Web listener`, watchdog reconnects after 30 minutes, reconnect loops, and only partial Bad MAC handling.

## Best Possible Solution

Keep the issue open as the concrete Windows WhatsApp listener-flap tracker. A conservative fix should focus on the WhatsApp plugin lifecycle: reproduce or instrument the Windows Scheduled Task/Baileys disconnect path, verify whether libsignal Bad MAC/closed-session errors escape the current Baileys-only crypto matcher, avoid delivery loss while the listener is reconnecting, and add regression coverage around the exact 408/428/499, watchdog, and no-listener delivery sequence.
