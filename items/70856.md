## Summary

Keep open. Current main has several WhatsApp reconnect and outbound-listener hardenings, but the specific reported behavior still has live source-backed gaps: the WhatsApp monitor still forces a reconnect after 30 minutes without inbound app messages, logs the same 499 watchdog path shown in the issue, can overlap with the gateway health-monitor stop/start path that produced the 5000ms stop timeout, and the current Bad MAC unhandled-rejection classifier may not catch the libsignal-only stack from the reporter's follow-up comment.

## Best Possible Solution

Keep #70856 open as a real WhatsApp lifecycle reliability bug. The best fix is likely in the WhatsApp plugin lifecycle: distinguish transport liveness from inbound user-message silence, make the watchdog/health-monitor restart path serialize with active reconnect and queued outbound delivery, broaden Bad MAC/libsignal recovery if appropriate, and align socket teardown with the open Baileys cleanup work. Add focused regression coverage for 408/428 retry loops, 499 watchdog reconnects, libsignal Bad MAC handling, and cron/message sends during reconnect on a Windows scheduled-task style gateway lifecycle.
