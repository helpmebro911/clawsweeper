## Summary

Keep #69633 open. Current main does not already contain WebView2/`window.chrome.webview` SPA support, and the request is a plausible first-party Windows companion-app UI integration rather than an obvious ClawHub/plugin cleanup item. The PR is still draft and appears incomplete as submitted, but that is a review/iteration concern, not a strong auto-close reason.

## Best Possible Solution

Keep the PR open for maintainer/product review. If accepted, turn it into a wired first-party Control UI/native-host adapter: type and validate WebView2 messages, register/unregister the listener in the app lifecycle, connect `draft-text`, recording, and voice messages to the existing chat/realtime surfaces, add focused tests for no-op browser behavior and WebView2 message handling, and coordinate the exact contract with the linked Windows host PRs.
