## Summary

Keep this PR open. Current main still has no outbound Slack `replyBroadcast` / Slack API `reply_broadcast` support, and the PR remains a plausible implementation candidate rather than obsolete. It is not merge-ready as-is because the provided review context still identifies docs/schema/dispatch/media semantics that need cleanup before shipping.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, wire `replyBroadcast` through the current Slack-owned schema, message-tool dispatch, action adapter, and send path; emit Slack `reply_broadcast: true` only for explicit threaded `chat.postMessage` calls; explicitly support or reject upload/media semantics; fix the docs example to use the current message tool contract; and add focused Slack send/dispatch/discovery tests. Keep broader automatic inbound/outbound mirroring behavior tracked in #63904 unless a maintainer chooses to merge those scopes.
