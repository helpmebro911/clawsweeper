## Summary

Keep #11665 open. Current `main` still routes `/hooks/agent` and mapped hook agent actions through an isolated cron-agent turn, and that path forces a new session id even when the caller supplies a stable `sessionKey`. The request is concrete, still actionable in core, and the related fix PRs in the provided context were closed unmerged rather than shipped.

## Best Possible Solution

Keep the issue open until OpenClaw adds a backward-compatible persistent hook-session path. The likely fix is an explicit `sessionMode: "isolated" | "persistent"` or equivalent on hook mappings plus request/transform payloads, threaded through hook normalization, config schema/types, mapping merge, dispatch, docs, and tests, with `forceNew` disabled only for the persistent hook path while preserving current isolated behavior by default.
