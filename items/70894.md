## Summary

Keep #70894 open. Current main still routes `/hooks/agent` to an isolated cron agent run with the approved `sessionKey`, but `prepareCronRunContext` still passes `forceNew: true` for every isolated run, so `resolveCronSession` cannot reuse an existing fresh session. The linked PR #70895 is a relevant open/unmerged fix, not evidence that the issue is already resolved on main.

## Best Possible Solution

Keep #70894 open and review/land #70895 or an equivalent maintainer-approved patch. The patch should compute whether an isolated run is hook/webhook-sourced before `resolveCronSession`, pass `forceNew: false` for accepted stable hook session keys, keep normal isolated cron jobs forced fresh, and add focused regression coverage for both paths.
