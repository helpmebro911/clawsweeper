## Summary

Keep #70915 open. Current main still advertises and registers ZAI image understanding, but the ZAI media provider delegates directly to the generic image helper and current model resolution can preserve a configured Coding Plan baseUrl for `glm-4.6v`/other vision models. The linked #70886 bug remains plausible and unimplemented on main; the PR is a focused fix attempt, though the request-scoped config rewrite should be revised before merge because the review thread identified a models.json race risk.

## Best Possible Solution

Keep the PR open and ask for a provider-owned routing fix. The best path is to implement ZAI vision endpoint normalization in the ZAI plugin/model-resolution layer, likely via `normalizeResolvedModel` or an equivalent direct model baseUrl rewrite for known vision models on coding endpoints, plus focused tests that prove `zai/glm-4.6v`, `glm-4.5v`, and `glm-5v-turbo` route to `/api/paas/v4` while ordinary coding models keep `/api/coding/paas/v4`. Avoid request-scoped config mutation around the shared models cache.
