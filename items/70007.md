## Summary

Keep open. Current main has strong adjacent audio support for channel/media-understanding flows and realtime VoiceClaw/Talk surfaces, but it still does not define the requested canonical custom-chat voice ingress contract. `/v1/responses` is text/image/file only, `chat.send.attachments` remains untyped at the protocol boundary, the parser drops non-image attachments, and the Control UI/WebChat send path only emits image attachments.

## Best Possible Solution

Keep this open for maintainer/API design. The likely fix is an authenticated, documented custom-client voice ingress path: either `input_audio` for `/v1/responses`, a typed `chat.send` audio attachment schema, or an upload-then-send media reference flow. It should persist or validate audio safely, route it through `tools.media.audio`/media-understanding, inject the transcript into the normal chat session, and add protocol docs plus regression tests for accepted/rejected audio MIME types and size limits.
