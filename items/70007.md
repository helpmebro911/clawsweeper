## Summary

Keep open. Current main has several adjacent audio/STT paths, including provider media understanding, Control UI Talk, and a VoiceClaw realtime endpoint, but it still does not provide the canonical custom chat-client voice ingress requested in #70007. `/v1/responses` does not accept `input_audio`, `chat.send` attachments remain untyped and image-oriented, and the attachment parser explicitly drops non-image payloads instead of routing audio through server-side transcription/media understanding.

## Best Possible Solution

Keep #70007 open for maintainer design. The best path is to choose and document one core Gateway contract: either add `input_audio` to `/v1/responses`, add a typed `chat.send.attachments` audio shape, or define an authenticated upload/media-ref flow. The implementation should save/sniff/bound audio safely, route it through the existing `mediaUnderstanding.transcribeAudioFile` / `tools.media.audio` provider pipeline, inject/persist the resulting transcript and media metadata, and update `docs/gateway/openresponses-http-api.md`, `docs/gateway/protocol.md`, `src/gateway/protocol/schema/logs-chat.ts`, `src/gateway/chat-attachments.ts`, and focused Gateway tests.

## Risks / Open Questions

- Adding custom-client audio ingress changes a public Gateway contract, so it needs protocol schema, auth, size-limit, MIME-sniffing, persistence, and transcript semantics rather than a UI-only workaround.
- The existing VoiceClaw realtime endpoint may be tempting to document as the answer, but it is Gemini/VoiceClaw-specific and does not cover the requested voice-note upload/transcribe path for ordinary custom chat clients.
