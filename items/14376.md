## Summary

Keep #14376 open. Current main records `lastErrorReason`, has one-shot transient retry/permanent-disable behavior, failure alerts, and auth-profile cooldowns, but recurring cron scheduling still applies a generic consecutive-error backoff and does not implement the requested reason-aware recurring guard policy, Retry-After handling, job-level cooldown/disable fields, or billing/auth/format circuit breaker.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely path is `src/cron/service/timer.ts`, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify the terminal run error once, persist the reason, respect provider Retry-After where available for rate limits, auto-pause or circuit-break recurring jobs for billing/auth/format with safe configurable defaults and one alert, keep timeout/network/rate-limit jobs enabled with bounded backoff, and coordinate with existing auth-profile cooldowns plus any provider/global rate-limit breaker.
