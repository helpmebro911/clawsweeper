## Summary

Keep #71166 open. Current main does not yet implement a WhatsApp channel/account `readOnly` mode or a hard transport-level outbound gate. The linked PR #71382 is the correct active implementation path, but it is still open, unmerged, and has review follow-ups, so the issue should remain open until that or an equivalent fix lands.

## Best Possible Solution

Keep #71166 open and use PR #71382, or an equivalent maintainer patch, as the implementation path. The best fix is a documented `channels.whatsapp.readOnly` plus per-account override, schema/metadata/docs coverage, and a transport-level gate that blocks or drops every outbound WhatsApp path: pairing replies, normal text/media sends, message-tool sends, polls, reactions, typing/presence, read receipts, and default notices. Add regression tests proving no Baileys send/read/presence methods are called in read-only mode while inbound processing still runs.
