## Summary

Keep #71040 open. Current main still has the reported instrumentation gap: cron active-job marking exists for due-job and execute-job paths, but startup catch-up and manual run create/finish cron task records without marking the job active while execution is in flight. Gateway startup configures task maintenance as the authoritative cron runtime, and the maintenance check uses `isCronJobActive(jobId)` after the five-minute grace window, so the reported false `lost` classification remains a real current-main risk.

## Best Possible Solution

Keep the PR open for maintainer review. The best code path is still the PR's narrow fix: add `markCronJobActive`/`clearCronJobActive` symmetry around startup catch-up and manual-run execution using `try/finally`, plus a regression test that observes `isCronJobActive` mid-flight on the real cron entry points. Before landing, update the PR rationale for the current timeout policy and run the repo's changed cron/task lanes.
