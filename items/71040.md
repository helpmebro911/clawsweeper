## Summary

Keep #71040 open. Current main still has the active-job producer gap the PR describes: scheduled due runs and `executeJob()` mark/clear `activeJobIds`, but startup catch-up and manual `cron.run()` paths create cron task records and execute jobs without marking the job active. Task maintenance still treats cron backing as `isCronJobActive(jobId)` after the 5-minute grace, while durable recovery only helps after terminal run-log/job-state evidence exists, so it does not cover an in-flight long manual/startup run.

## Best Possible Solution

Keep the PR open or land an equivalent core fix: add `markCronJobActive`/`clearCronJobActive` with `try/finally` coverage to `runStartupCatchupCandidate` and the prepared manual-run execution path, then add the proposed regression coverage that drives the real `CronService.start()` and `cron.run()` entry points against the real active-job singleton. Durable cron terminal recovery should remain as a separate safety net because it does not protect in-flight runs before terminal run history exists.
