## Summary

Keep PR #71040 open. Current main still has the cron active-job tracking gap this PR describes: the task registry treats cron tasks as backed only by `isCronJobActive(jobId)`, but startup catch-up and prepared manual-run execution still do not mark the job active while running. The PR is a focused bug-fix candidate with a plausible regression test; it is not implemented on main and is not superseded by the recovery-hook work in #69313.

## Best Possible Solution

Keep the PR open and either merge it after current-base validation or apply the same scoped fix on main: mark startup catch-up and prepared manual runs active immediately before execution, clear them in `finally`, and keep the regression coverage that observes `isCronJobActive` mid-flight without stubbing the consumer.
