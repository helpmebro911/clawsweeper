## Summary

Keep PR #71820 open. Current main still lacks the proposed BlueBubbles inbound reply-context API fallback: inbound processing uses webhook reply fields plus the process-local reply cache only, and there is no shipped config/type/helper surface for `replyContextApiFallback`. The related #43764/#65447 fix already shipped, but it is outbound Private API server-info refresh for send-time reply threading, not inbound reply-context recovery. The PR is a recent focused draft with useful tests and unresolved review comments that should be addressed, not closed by cleanup.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback to reuse or extend `BlueBubblesClient` for a guarded `GET /api/v1/message/{guid}` reply-context lookup, preserve channel-level defaults when per-account config omits the new flag, keep the feature opt-in unless maintainers choose otherwise, and retain focused tests for cache misses, failures, dedupe, config inheritance, and SSRF policy.
