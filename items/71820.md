## Summary

Keep PR #71820 open. Current main still only resolves BlueBubbles inbound reply context from the webhook payload and the in-memory reply cache; it does not implement the PR's opt-in HTTP API fallback for cache misses. The related #43764/#65447 work already shipped, but that covers outbound Private API status refresh for reply threading, not inbound reply-context recovery. The PR has useful focused work and tests, plus unresolved review comments that should be addressed rather than using maintainer cleanup to close it.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback so it extends/reuses the existing BlueBubbles client request path for guarded `GET /api/v1/message/{guid}`, preserve channel-level defaults when account configs omit the new flag, keep the behavior opt-in unless maintainers decide otherwise, and retain focused tests for cache misses, failures, dedupe, config inheritance, and SSRF policy.
