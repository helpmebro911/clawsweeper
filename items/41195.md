## Summary

Keep this open. Current main has persistent per-account Feishu message-id dedupe and startup warmup, so the same-message-id restart case is covered, but this issue explicitly reports Feishu reconnect replay with new message IDs. I did not find a startup/reconnect replay window, old-create-time dispatch filter, or Feishu config option to ignore offline/replayed messages before agent dispatch.

## Best Possible Solution

Keep this issue open and fix it inside the Feishu plugin. The likely path is a per-account WebSocket startup/reconnect replay guard that records message ids but skips agent dispatch for replayed events whose Feishu create_time predates the current connection by a conservative cutoff, with a config/default policy if maintainers want user control. Add regression coverage for different message_id values with stale create_time being skipped, plus fresh post-connect messages still dispatching, while preserving the existing persistent same-id dedupe path.
