## Summary

Keep this issue open. Current main covers Feishu duplicate delivery when the same `message_id` is replayed and restores persistent dedupe on startup, but the reported failure is narrower: Feishu WebSocket replay with fresh message IDs after reconnect. Current receive and dispatch paths still treat a new `message_id` as fresh, and I found no startup/reconnect replay guard, inbound `create_time` age filter, or Feishu `ignoreOfflineMessages`-style config.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin. The likely direction is a per-account WebSocket startup/reconnect replay guard that records incoming IDs while skipping stale events whose Feishu `create_time` predates the active connection by a conservative cutoff, with focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, and existing same-ID persistent dedupe continuing to work.
