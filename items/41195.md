## Summary

Keep open. Current main has stronger Feishu dedupe than the report's original `feishu-dedup.json is not read back` claim: it uses a persistent per-account dedupe store and warms it from disk on monitor startup. However, the remaining requested behavior is not implemented: inbound suppression is still keyed to `message_id`, and I found no startup grace window, reconnect replay window, `ignoreOfflineMessages`, or timestamp-age filter that would handle the issue's claimed Feishu replay-with-new-IDs case.
