## Summary

Keep PR #56538 open. Current main still hardcodes UTF-8 decoding in the process exec paths that the PR is trying to fix, while only the node-host system.run path has a Windows code-page decoder. The PR is not merge-ready as-is because review comments and the current diff show remaining quality/scope issues, but the underlying Windows GBK/CP936 exec-output bug remains real and is still tracked by #50519.

## Best Possible Solution

Keep the PR open but require a focused rework: implement Windows code-page-aware decoding in `src/process/exec.ts` for both `runExec` and `runCommandWithTimeout`, use a streaming decoder where output is processed by chunks, avoid hardcoded GBK for non-Chinese Windows code pages, add regression coverage for GBK and UTF-8 Windows outputs, and drop unrelated ACPX/CLAUDE.md churn. The remaining product bug should continue to be coordinated with #50519.
