## Summary

Current main still matches the reported stop-timeout failure mode. The timeout path returns before clearing the task/abort maps, leaves runtime marked running, and the start path still no-ops when a stale task exists. The existing test still asserts the buggy behavior, while the linked fix PR #71456 is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and land PR #71456 or an equivalent core gateway fix. On the stop-timeout path, release the stale store.aborts and store.tasks entries before returning, mark the runtime stopped instead of running, and update the regression test to prove a subsequent start registers a fresh task. Treat any openclaw doctor liveness enhancement as a separate follow-up unless maintainers choose to bundle it.
