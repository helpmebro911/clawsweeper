## Summary

Current main still matches the reported failure mode. `stopChannel` marks the timed-out channel runtime as `running: true` and returns before deleting the task/abort entries, while `startChannelInternal` still no-ops when `store.tasks` already has that account. The existing regression test still asserts the buggy behavior, and the related fix PR #71456 is closed and unmerged in the provided context, so this issue should remain open.

## Best Possible Solution

Keep #71412 open and land PR #71456 or an equivalent focused patch. The fix should clear the stale `store.aborts` and `store.tasks` entries on stop timeout, mark the runtime stopped instead of running, and update the regression test so a subsequent start registers a fresh account task. Treat the `openclaw doctor` Telegram liveness check as a separate follow-up unless maintainers want it bundled.
