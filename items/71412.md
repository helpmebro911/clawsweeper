## Summary

Current main still matches the reported stop-timeout failure mode. The timeout branch leaves the task/abort entries registered, marks the runtime as running, and returns before cleanup; `startChannelInternal` still no-ops when the stale task exists; and the current unit test still asserts that a second start is suppressed after a stop timeout. Related PR #71456 is closed and unmerged in the provided context, so this issue should remain open until an equivalent fix lands.

## Best Possible Solution

Keep #71412 open and land/review PR #71456 or an equivalent patch that clears `store.aborts` and `store.tasks` on stop timeout, records the account as stopped rather than running, and updates the regression test to assert that a subsequent start registers a fresh task. Treat the `openclaw doctor` Telegram liveness enhancement as a separate follow-up unless maintainers want it bundled.
