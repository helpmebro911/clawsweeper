## Summary

Current main still contains the reported stop-timeout failure mode. `stopChannel` returns on timeout before clearing `store.aborts` and `store.tasks`, marks the runtime as `running: true`, and `startChannelInternal` still silently returns when a task entry remains. The health monitor still treats the stop/start sequence as completed when `stopChannel` returns normally, and the existing regression test still asserts the stuck behavior. The related implementation candidate, #71456, is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and land #71456 or an equivalent core gateway fix. On the stop-timeout path, clear the stale `store.aborts` and `store.tasks` entries before returning, mark the runtime stopped instead of running, and update the regression test to prove that a subsequent start registers a fresh task. Treat any `openclaw doctor` liveness improvement as a separate follow-up unless maintainers choose to bundle it.
