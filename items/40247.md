## Summary

Keep #40247 open. Current main still returns only the primary non-loopback bind host from `resolveGatewayListenHosts()`, while local gateway calls still target `127.0.0.1`; that leaves the reported tailnet/custom bind self-connection failure unresolved. The PR is focused, has targeted tests in its diff, and related #50751 overlaps the symptom but does not clearly supersede this server-side dual-bind approach.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is either to land a current-main version of this server-side dual-bind fix with the PR's tailnet/custom regression tests, or explicitly choose the client-side host-resolution path in #50751 and close #40247 only after that shipped path demonstrably fixes internal calls, CLI probes, and `sessions_spawn` for `gateway.bind=tailnet` and `gateway.bind=custom`.
