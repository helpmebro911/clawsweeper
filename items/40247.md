## Summary

Keep this PR open. Current main still does not implement the PR's server-side dual-bind behavior: `resolveGatewayListenHosts()` returns only the selected non-loopback bind host, while local gateway calls still target loopback. The PR remains a focused implementation candidate for the reported tailnet/custom bind regression, and the overlapping client-side alternative #50751 is still open in the provided context rather than a merged/shipped replacement.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a current-main version of this server-side dual-bind change with the PR's tailnet/custom regression tests and matching docs updates, or explicitly choose the #50751 client-side host-resolution approach and close this PR only after the shipped path demonstrably fixes internal calls, CLI probes, and `sessions_spawn` for `gateway.bind=tailnet` and `gateway.bind=custom`.

## Evidence

That means a resolved tailnet/custom host is not paired with `127.0.0.1`.
