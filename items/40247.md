## Summary

Keep this PR open. Current main still listens only on the resolved non-loopback host for tailnet/custom binds while local gateway callers and tests still target 127.0.0.1, so the PR is not obsolete and no shipped superseding fix is evident. #50751 is an overlapping open alternative, not a landed replacement.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase and land the server-side dual-bind fix with the PR's tailnet/custom regression tests plus docs/security expectation updates, or explicitly choose #50751 or another client-side host-resolution fix and close this PR only after a shipped implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding would intentionally change documented direct-tailnet behavior where loopback does not work, so docs and operator expectations should change if this lands.
- Adding a loopback listener for tailnet/custom mode changes same-host reachability and may interact with local pairing/auth assumptions; maintainers should explicitly accept that security model change.
- #50751 overlaps with a client-side host-resolution approach. Maintainers should choose one coherent listener/target strategy or coordinate both to avoid inconsistent gateway semantics.
