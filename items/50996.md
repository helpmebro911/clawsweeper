## Summary

Keep open. Current main has partial mitigations, but it does not implement the requested interruptible or parallel handling for long `process(action: "poll")` waits. The process tool still ignores the abort signal and waits with a timer loop until completion or timeout, while ordinary Telegram messages remain serialized behind an active same-chat/topic turn.

## Evidence

- **Process tool ignores abort signal:** `createProcessTool` receives the tool abort signal as `_signal`, so the process tool does not consume the run abort signal that would be needed to interrupt an in-flight poll.
  - file: [src/agents/bash-tools.process.ts:132](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/agents/bash-tools.process.ts#L132)
  - command: `nl -ba src/agents/bash-tools.process.ts | sed -n '126,136p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Poll still waits inside the tool call:** `process` action `poll` resolves a timeout, then loops with `await setTimeout(...)` until the process exits or the deadline passes. There is no abort-signal check in this wait path.
  - file: [src/agents/bash-tools.process.ts:306](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/agents/bash-tools.process.ts#L306)
  - command: `nl -ba src/agents/bash-tools.process.ts | sed -n '258,360p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Long-poll wait is intentional current behavior:** The existing test expects a `process poll` with `timeout: 2000` to remain unresolved after 200 ms, confirming current main intentionally allows the tool call to hold the turn while waiting.
  - file: [src/agents/bash-tools.process.poll-timeout.test.ts:86](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/agents/bash-tools.process.poll-timeout.test.ts#L86)
  - command: `nl -ba src/agents/bash-tools.process.poll-timeout.test.ts | sed -n '86,94p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Run abort signals are available to tools:** The Pi tool wrapper combines the active run abort signal and passes it into `tool.execute`; this makes the process tool's ignored `_signal` the relevant missing piece for interrupting a pending process poll.
  - file: [src/agents/pi-tools.abort.ts:61](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/agents/pi-tools.abort.ts#L61)
  - command: `nl -ba src/agents/pi-tools.abort.ts | sed -n '48,68p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Telegram ordinary messages remain serialized:** Telegram installs grammY `sequentialize(getTelegramSequentialKey)`, and the sequential key groups ordinary messages by chat/topic. The current test verifies a second same-topic message does not start until the first turn completes.
  - file: [extensions/telegram/src/bot-core.ts:386](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/telegram/src/bot-core.ts#L386)

## Risks / Open Questions

- Current mitigations such as `sessions.steer`, `chat.abort`, `sessions_yield`, automatic completion wake, and the 120s process-poll cap reduce some stall cases but do not satisfy the core request for inbound user messages to interrupt or run in parallel with a pending tool call.
- This is core session/tool runtime behavior, not a ClawHub plugin/skill fit, and the provided related-item context does not identify a canonical duplicate or superseding tracker.
