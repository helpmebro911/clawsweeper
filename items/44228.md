## Summary

Keep open. Current `main` still compares Windows `file:///C:/...` URLs, `C:/...`, and `C:\...` as different strings in reply-media dedupe, so the reported bug remains plausible. The PR is also not landable as-is because review discussion identified both unrelated config/protocol changes and a remaining lowercase drive-letter gap.

## Evidence

- **Current main lacks Windows path canonicalization in reply-media dedupe:** `filterMessagingToolMediaDuplicates` still returns raw non-`file://` inputs unchanged and returns `decodeURIComponent(parsed.pathname)` for `file://` URLs, with no drive-letter or slash normalization. That means `file:///C:/...` becomes `/C:/...` while local Windows paths stay `C:/...` or `C:\...`.
