## Summary

Keep open. The Langfuse portion should not merge into core as-is because current main already provides a generic diagnostics/OpenTelemetry and plugin-hook direction, but the PR also contains a real cron/subagent completion delivery-confirmation fix that does not appear implemented on main. Current main still treats a successful gateway agent call with visible payloads as delivered, without an explicit external-delivery confirmation contract.

## Best Possible Solution

Do not merge this PR as-is. Split it: extract a small core PR for cron/subagent completion delivery confirmation, with tests for missing delivery metadata despite visible payloads, best-effort delivery failure, successful confirmed delivery, and silent duplicate announce behavior. Move the Langfuse PoC to ClawHub or an external plugin built on diagnostics-otel/plugin hooks; if that exposes a real missing public API, propose a generic plugin SDK seam rather than adding Langfuse directly to core.

## Risks / Open Questions

- Merging the whole PR would add a Langfuse-specific runtime dependency and env-secret surface to core despite the current plugin/ClawHub direction for optional integrations.
- The proposed observability code has unresolved privacy/security review findings around redaction, secret retention, and trace attribution, so it should not land without redesign or a focused security pass.
- Closing the PR now would lose a still-relevant core bug fix: completion announces need an explicit external-delivery confirmation or a guaranteed fallback path before being marked delivered.
