## Summary

Keep open. Current main still implements `runTimeoutSeconds` as a hard sub-agent runtime/wait timeout with after-the-fact timeout announce handling, but I found no grace-period config, pre-timeout timer, or system-message injection path that would give the child agent a chance to checkpoint before the run is killed. The related #37250 partial-progress work helps preserve already-persisted transcript/tool history, but it does not implement this issue's requested pre-timeout warning behavior.

## Best Possible Solution

Keep #6625 open for maintainer product/design work. A good implementation would add a core sub-agent lifecycle grace period, likely under `agents.defaults.subagents`, schedule a warning before `runTimeoutSeconds`, deliver it through the existing child-session messaging/runtime path, and cover native plus ACP behavior with focused tests and docs. The existing #37250 partial-progress handling should remain as the fallback after timeout.
