## Summary

Keep #71669 open. Current main still enforces the remote plain-HTTP Control UI/device-identity rejection, and the docs explain the expected HTTPS/localhost/break-glass paths, but the requested startup/runtime warning with actionable server-side guidance is not implemented. Related #32473 remains the broader product/security thread, and #56830 is an open copy-clarification PR rather than a shipped startup-warning implementation.

## Best Possible Solution

Keep this issue open as the narrow warning/guidance follow-up. The best fix is a small, tested gateway warning path that emits actionable guidance when Control UI is enabled in a configuration likely to reject remote plain-HTTP browser sessions, plus alignment with #56830's error-copy work if that PR lands. The broader policy question about allowing homelab remote HTTP while preserving device auth should remain on #32473.

## Risks / Open Questions

- The server cannot know a browser's `isSecureContext` until request time, so the implementation should define a conservative startup trigger such as Control UI enabled, TLS not enabled for the served surface, non-loopback exposure possible, and device auth not disabled.
- Warning text should avoid normalizing `dangerouslyDisableDeviceAuth` for public/VPS deployments; it should present it as break-glass and point to HTTPS, localhost/SSH tunnel, Tailscale Serve, or trusted reverse-proxy documentation first.
