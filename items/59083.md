## Summary

Keep open. Current `main` already avoids the missing-filename crash, but it does not fully implement this PR's path-sanitization goal: the shared upload filename resolver still uses `path.basename()` only, which leaves Windows-style paths intact on Linux/macOS. The PR is also outdated against `main` because filename construction moved out of `openai-compatible-audio.ts` into `shared.ts`.

## Evidence

- **Shared resolver still leaks Windows-style paths:** `resolveAudioTranscriptionUploadFileName()` trims the input and then calls `path.basename(trimmed)`. On the Linux/macOS runtime used by OpenClaw, that strips POSIX separators but not backslash-separated Windows paths, so `C:\Users\alice\private\note.wav` would still be sent as the multipart filename.
  - file: [src/media-understanding/shared.ts:28](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/media-understanding/shared.ts#L28)
  - command: `nl -ba src/media-understanding/shared.ts | sed -n '24,60p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Missing-filename crash is already fixed on main:** The same shared resolver now returns `"audio"` when `fileName` is absent or blank, so the PR's crash fix is no longer needed in its original form.
  - file: [src/media-understanding/shared.ts:29](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/media-understanding/shared.ts#L29)

## Risks / Open Questions

- Windows-style local paths can still be exposed to third-party audio transcription APIs in multipart filenames.
- Merging this PR as-is would miss the current seam on `main`; the fix needs rebasing onto `src/media-understanding/shared.ts` and should preserve the newer AAC-to-M4A logic.
