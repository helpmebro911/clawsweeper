## Summary

Keep open. Current main still has a real main-session cron behavior gap: a `sessionTarget: "main"` `systemEvent` is enqueued and wakes heartbeat delivery to the last user route without consulting `delivery.mode`, while `lastDeliveryStatus: "not-requested"` only reflects the separate cron delivery plan. That matches the core complaint in #47865 rather than an implemented fix or close candidate.

## Evidence

- **Main cron path ignores delivery.mode:** `executeMainSessionCronJob` resolves the systemEvent text, calls `enqueueSystemEvent`, and then runs or queues heartbeat with `heartbeat: { target: "last" }`; there is no guard for `job.delivery?.mode === "none"` in this path.
  - file: [src/cron/service/timer.ts:1206](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/cron/service/timer.ts#L1206)

  - sha: [d228463120a5](https://github.com/openclaw/openclaw/commit/d228463120a5057f7ccf9a5c31af62fe4a922088)
- **not-requested is only delivery-plan state:** `resolveDeliveryState` returns `not-requested` when `resolveCronDeliveryPlan(job).requested` is false, so the reporter's `lastDeliveryStatus: "not-requested"` can coexist with main-session system-event injection and heartbeat routing.
