## Summary

Keep open. Current `main` still does not implement the requested runtime behavior: generic channel account listing omits an implicit `default` account whenever named accounts exist, while Slack/Discord/Zalo still let named accounts inherit top-level auth. The PR’s goal is still relevant, and the current branch also appears to need follow-up for cross-channel edge cases like Zalo `tokenFile` and explicit blank-token overrides.

## Evidence

- **Current main still omits implicit default at runtime:** `createAccountListHelpers().listAccountIds()` only returns configured account ids plus `default` when the list is empty. There is no base-token/default-injection logic on current `main`.
  - file: [src/channels/plugins/account-helpers.ts:53](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/channels/plugins/account-helpers.ts#L53)

  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Gateway startup uses exactly that account list:** Gateway startup takes `plugin.config.listAccountIds(cfg)` and starts those accounts. If `default` is omitted from `listAccountIds()`, the base-config provider is never started.
  - file: [src/gateway/server-channels.ts:299](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/gateway/server-channels.ts#L299)
