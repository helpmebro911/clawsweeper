## Summary

Keep open. Current `main` still has the unguarded `env.vars` path, and config loading applies `env.vars` before schema validation, so truthy non-string JSON values can still flow into `containsEnvVarReference` and crash. The PR remains relevant and is not implemented on `main`.

## Evidence

- **Missing type guard on current main:** `collectConfigEnvVarsByTarget` still accepts any truthy `env.vars` value and stores it into `entries` without checking `typeof value === "string"` first.
  - file: [src/config/config-env-vars.ts:22](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/config-env-vars.ts#L22)
  - command: `nl -ba src/config/config-env-vars.ts | sed -n '1,120p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Crash site still expects a string:** `containsEnvVarReference` immediately calls `.includes()` on its `value` parameter, so a numeric or boolean `env.vars` entry will throw at runtime if it reaches this function.
  - file: [src/config/env-substitution.ts:137](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/env-substitution.ts#L137)

  - file: [src/config/io.ts:1054](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/io.ts#L1054)

## Risks / Open Questions

- Closing this PR would leave a real pre-validation crash path in config loading for JSON configs that contain numeric or boolean values under `env.vars`.
- Because the crash happens before schema validation, users may see a raw TypeError instead of a normal config validation error, which is a poor failure mode for startup and config recovery paths.
