## Summary

Keep #71555 open. Current main still opens the embedded-run transcript with `SessionManager.open(params.sessionFile)`, while OpenClaw's cache only prewarms a 4 KiB read and the pinned `@mariozechner/pi-coding-agent@0.70.2` implementation still fully reads/parses the JSONL once in `open()` and again in `setSessionFile()`. The related stale-lock cleanup only improves restart recovery after a crash; it does not address the large-session duplicate-load failure class reported here.

## Best Possible Solution

Keep the issue open and fix the core embedded-session path. The best direction is to update or patch `@mariozechner/pi-coding-agent` so `SessionManager.open()` reads only the header or reuses preloaded entries instead of parsing the full JSONL twice, then add focused regression/performance coverage for large session JSONL context rebuild and a controlled error path that does not exit the gateway.
