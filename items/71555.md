## Summary

Keep this issue open. Current main still has the reported embedded Pi/session path: OpenClaw pins `@mariozechner/pi-coding-agent` 0.70.2, opens the existing session file before SDK session creation, and the pinned SDK still performs a whole-file JSONL parse in `SessionManager.open()` and another in `SessionManager` construction before `createAgentSession()` calls `buildSessionContext()`. Recent compaction successor transcript rotation is useful, but it is config-gated and runs after successful compaction, so it does not eliminate the initial large-session open/rebuild crash path reported here.

## Best Possible Solution

Keep this issue open and fix it at the embedded Pi/session boundary. The best path is to upgrade or patch `@mariozechner/pi-coding-agent` so `SessionManager.open()` reads the header without a second full parse or reuses parsed entries, or add a narrow OpenClaw adapter that avoids duplicate parsing while preserving Pi session semantics. Add focused regression or stress coverage with a large JSONL/long tool-result transcript proving embedded context rebuild/truncation returns a controlled error, compacts/rotates, or otherwise avoids gateway exit.
