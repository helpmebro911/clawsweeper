## Summary

Current main only partially addresses the report. `qmd embed` now uses `memory.qmd.update.embedTimeoutMs`, and config/schema help expose the update knobs, but OpenClaw still runs QMD embedding by default on boot and still intentionally runs embeddings in `searchMode: "search"`. The reported ARM/WSL timeout loop can therefore still occur when QMD embedding is what triggers the node-llama-cpp/Vulkan build path.

## Best Possible Solution

Keep this issue open for QMD memory-core work. The best path is to define the product contract for BM25-only QMD operation, then update `extensions/memory-core/src/memory/qmd-manager.ts`, `packages/memory-host-sdk/src/host/backend-config.ts`, QMD docs, and regression tests so constrained ARM/WSL hosts can run without triggering llama.cpp/Vulkan embedding work. If maintainers want vectors warmed by default, add an explicit embedding/vector maintenance switch and cross-link #59234 for the narrower boot probe problem.

## Risks / Open Questions

- Changing `searchMode: "search"` to suppress all embedding work could surprise users who expect vectors to stay warm for later hybrid/vector mode switches. Maintainers should choose an explicit QMD embedding contract rather than relying on ambiguous side effects.
- The current docs/config split is easy to misread: `limits.timeoutMs` is for search queries, while `update.embedTimeoutMs` is for `qmd embed`; the user-facing QMD docs should make that distinction clear.
