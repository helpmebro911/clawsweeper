## Summary

Keep open. Current main has useful session maintenance knobs, but it still stores `skillsSnapshot` inline in each `sessions.json` entry and still performs whole-file parse/serialize on session-store updates. The related #69451 remains open and covers a memory/RSS angle, but it does not supersede this latency/run-budget report.

## Best Possible Solution

Keep #71227 open and treat it as a core session-store performance bug. The best path is a backwards-compatible storage change, starting with content-addressed/out-of-line `skillsSnapshot` or another dedupe mechanism, plus follow-up sharding or lazy/async persistence so session updates do not parse and serialize unrelated sessions on the run's critical path. Link and coordinate with #69451, but do not collapse this latency-specific report into the memory/RSS issue.
