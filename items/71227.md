## Summary

Keep #71227 open. Current main has useful session maintenance mitigations, but the reported hot path still exists: per-agent `sessions.json` is read, parsed, merged, and stringified as one object, while `skillsSnapshot` and `systemPromptReport` remain inline on each session row. That means recent or fat Slack thread stores below the entry cap can still hit the latency/run-budget failure described here. Related #69451 remains open and adjacent, but it tracks the broader memory/RSS problem rather than fully superseding this skillsSnapshot/run-budget report.

## Best Possible Solution

Keep this open and fix the core session-store scalability path: store repeated `skillsSnapshot` data out of line or content-addressed, avoid full `sessions.json` parse/stringify on request paths via sharding/SQLite/incremental metadata, and add regression coverage for a Slack-like store with fewer than 500 recent but large thread sessions that must not consume the agent wait/run budget.
