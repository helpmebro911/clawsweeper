## Summary

Keep #71227 open. Current main has session maintenance that can prune or cap some stores, but the reported core failure mode remains: session metadata is still one `sessions.json` object, `skillsSnapshot` remains inline per entry, and hot update paths still re-read, parse, stringify, and atomically rewrite the whole store on session updates. The related #69451 remains a neighboring memory/RSS report, not a canonical replacement for this latency/run-budget issue.

## Best Possible Solution

Keep the issue open and fix the session metadata hot path in core. The strongest product/code direction is to stop storing repeated `skillsSnapshot` and possibly `systemPromptReport` inline per row, or shard/session-index metadata so an ordinary session update does not parse and stringify unrelated sessions. Add a regression or perf-oriented test fixture that exercises a multi-MB `sessions.json` with under-500 entries and verifies session updates avoid full-store latency or remain well below the 60s agent wait budget.
