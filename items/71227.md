## Summary

Keep open. Current main still performs size-proportional monolithic session-store updates: it re-reads and parses `sessions.json` inside the write lock, then stringifies and atomically rewrites the whole store. The large `skillsSnapshot` and `systemPromptReport` fields remain inline on each `SessionEntry`, and the current maintenance controls only prune/cap/rotate stores; they do not dedupe snapshots, shard per session, use SQLite, or move writes off the run-critical path.

## Best Possible Solution

Keep this issue open and fix the core session metadata storage path. A narrow first step would be content-addressing or otherwise deduplicating `skillsSnapshot`; the more complete solution is per-session metadata sharding or SQLite plus off-critical-path writes. Add regression coverage for a large active `sessions.json` under `maxEntries` with repeated snapshots so updating one session no longer scales with whole-store size or consumes the agent wait/run budget.
