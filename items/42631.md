## Summary

Current main still does not provide a first-class job-level model/provider override for cron jobs that run as `sessionTarget: "main"` with `payload.kind: "systemEvent"`. Model selection remains scoped to `agentTurn`/isolated cron jobs, while system-event payloads and legacy top-level override fields are stripped. The related implementation PR #42708 was closed unmerged, so there is no landed fix to close against.

## Best Possible Solution

Keep this issue open. The best path is an additive cron contract for job-level `model` and provider selection on `main + systemEvent` jobs, validated against the existing model/provider allowlist rules and forwarded transiently into the cron-triggered main-session heartbeat/run without persisting into unrelated interactive session state. Update protocol schema, runtime normalization, Gateway execution, CLI/tool/UI surfaces, docs, and regression tests together; treat #42708 as prior discussion, not a landed fix.

## Risks / Open Questions

- Closing this now would hide a still-present feature gap for users who want stable main-session cron system events on a cheaper model without editing internal session metadata.
- A future implementation needs to scope model/provider overrides to the cron-triggered heartbeat or run; persisting them into shared main-session metadata could leak the cron model into later interactive work.
- The fix is a protocol/product change across schema, normalization, execution, CLI/tool/UI/docs, and tests, not a stale cleanup item.
