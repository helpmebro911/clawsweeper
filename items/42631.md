## Summary

Keep open. Current main still supports cron model overrides only for isolated/current/session `agentTurn` jobs. `main + systemEvent` jobs still reject or strip model/provider override fields and execute by enqueueing a system event plus heartbeat with no scoped model override. The related implementation PR #42708 was closed unmerged, so it does not supersede the remaining feature request.

## Best Possible Solution

Keep this issue open and implement a first-class, validated model override for `main + systemEvent` cron jobs. The clean path likely needs an additive cron protocol field or documented `payload.model` support, normalization/type updates, gateway/runtime execution support that applies the override only for the cron-triggered heartbeat/session turn, and regression tests proving the override reaches the run without persisting into unrelated interactive use.

## Risks / Open Questions

- Closing this as stale would hide a real remaining feature gap: users still cannot choose a cheap model for stable `main + systemEvent` cron jobs without mutating session metadata manually.
