## Summary

Keep open. Current main does not implement automatic sensitive-value redaction for outbound channel messages: the shared send/poll service still forwards raw message params to plugin actions and raw text to core fallback delivery. The PR discussion identifies real defects in the proposed implementation, so it should not merge as-is, but the underlying security hardening remains actionable in this repo.

## Best Possible Solution

Keep the item open, but require a revised implementation. The best path is a core outbound redaction seam that sanitizes both shared core delivery and plugin action params before external sends, covers send and poll text/options, respects existing logging redaction contracts or introduces a clearly separate outbound setting, reuses the repo's sensitive-path/SecretRef knowledge instead of broad key substring scans, and adds regression tests for plugin-handled Telegram/Discord/Slack-style send and poll paths plus direct/gateway fallback delivery.
