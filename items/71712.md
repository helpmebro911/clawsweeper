## Summary

Keep open. Current main has an owner-only cron tool and admin-scoped `cron.add`, but it does not implement the RFC's narrower agent-facing scheduling API: no playbook allowlist, no gateway-injected non-forgeable creator provenance, no required expiration for recurring jobs, no per-agent active-job quota, and no creation audit surface. The related items are adjacent, not canonical duplicates.

## Best Possible Solution

Keep the RFC open as a core scheduler/security design item. The best path is an additive gateway-owned scheduling surface that injects creator identity/session and createdAt, accepts only registered playbook ids plus schema-validated params, requires an expiration for recurring agent-created jobs in coordination with #71711, enforces per-agent quotas, records creation/update attempts in an audit ledger, and adds protocol/source/tests/docs. If maintainers prefer a plugin implementation, first define the trusted requester/provenance API that plugin tools can consume.

## Risks / Open Questions

- This touches admin-scoped gateway methods, cron store schema, tool owner gating, task/audit records, and security docs, so it needs maintainer product/security design rather than automated closure.
- A plugin could prototype UX, but non-forgeable provenance likely requires core gateway-injected requester identity or a new trusted plugin API seam before it can be safely externalized.
