## Summary

Keep open. Current main has a generic owner-only cron tool and gateway `cron.add` method, but it does not implement the RFC's hardened agent-facing scheduling path: no `schedule_task` surface, playbook allowlist, gateway-injected non-forgeable `createdBy`, recurring `activeUntil`, per-agent quota, or schedule-creation audit contract. The request is recent, security-sensitive core/gateway work, and related items do not supersede it.

## Best Possible Solution

Keep the RFC open for maintainer scoping. The best path is an additive gateway-owned scheduling method/tool separate from generic `cron.add`: require registered operator playbooks with parameter schemas, gateway-inject requester agent/session identity and `createdAt`, require or depend on #71711-style `activeUntil` for recurring jobs, enforce per-agent active-job quotas, emit schedule-creation audit records, and update protocol schema, agent tool docs, cron docs, and focused tests. If maintainers want smaller work, split active-until and provenance into canonical tracked prerequisites and keep this issue as the API/security design anchor.
