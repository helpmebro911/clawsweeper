## Summary

Keep this issue open. Current main still has the core ingredients reported here: cron creation paths default jobs to enabled unless disabled explicitly, persisted or externally edited jobs missing `enabled` are treated as enabled, isolated agent cron jobs fall through to agent/default model selection, and I did not find an implemented pre-run cost/usage circuit breaker or confirmation/diff flow for externally registered recurring jobs. The related reports in the provided context cover different cron concerns and do not supersede this safety/cost issue.

## Best Possible Solution

Keep this open and implement a generic cron safety contract: distinguish user-created, plugin-managed, and externally repaired jobs; require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs; surface inherited model and estimated call cadence in doctor/update dry-runs; warn or block high-cost inherited defaults unless the job has an explicit model; and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs with tests and docs for the plugin `getCron` surface.

## Risks / Open Questions

- Changing the global cron default from enabled to disabled would be behavior-breaking for existing CLI flows and managed jobs, so the fix likely needs an origin-aware policy rather than a blanket default flip.
- A usage circuit breaker needs careful classification of user-approved versus plugin-managed recurring jobs, plus provider-specific quota semantics, to avoid pausing critical automation unexpectedly.
