## Summary

Keep #71882 open. Current main still short-circuits memory status before asking a runtime-capable memory plugin for status, so the underlying host-side problem is real and cannot be solved purely in the third-party plugin. The linked plugin-side work is useful because it registers a MemoryRuntime/MemoryCapability status/search manager, but this OpenClaw PR needs revision before merge: the proposed patch hardcodes `memory-lancedb-pro` into shared core status logic, and its console subsystem filter would not suppress raw `console.log`/`console.info` output from plugin code.

## Best Possible Solution

Keep the PR open but require a generic revision: derive runtime-status eligibility from the active registered memory capability/runtime or an explicit documented capability field, not from `memory-lancedb-pro`; update the memory status typing/contract if plugin-specific backend names are expected; and protect `status --json --all` from raw plugin console output, with focused regression tests for runtime-only memory status and strict JSON stdout.

## Risks / Open Questions

- Merging this PR as-is would add a hardcoded third-party plugin id to shared core status logic, contrary to the extension-agnostic core boundary.
- The JSON-cleanup part of the PR is incomplete because raw plugin `console.log`/`console.info` calls can still reach stdout.
- Closing as ClawHub would leave a real host-side status/runtime contract gap for third-party memory plugins that correctly register a memory runtime but do not use the built-in sqlite/qmd store path.
