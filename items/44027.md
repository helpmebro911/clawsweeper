## Summary

Keep open. Current main still suppresses ordinary group text-only tool summaries without consulting `/verbose on` or `/verbose full`, and the PR addresses a real channel-plugin delivery gap called out by the author and DingTalk-related discussion. The branch is stale against current main, though: its proposed native-source suppression conflicts with current dispatch tests that require native sessions to deliver tool summaries and media.

## Best Possible Solution

Keep this PR open as a plausible focused bug fix, but require a rebase and implementation update against current `src/auto-reply/reply/dispatch-from-config.ts`. The best implementation is to let effective session verbose `on` or `full` opt ordinary non-Slack group chats into text tool summaries while preserving current native-command delivery, Slack non-DM suppression, forum behavior, sendPolicy deny, preview suppression, media-only delivery, and exec approvals. Add focused regression tests for group verbose off/on/full and coordinate broader configurable group tool-result policy in #54663.

## Risks / Open Questions

- Merging the PR unchanged could regress current native command-source behavior because the PR still suppresses native tool summaries while current tests expect native text summaries and media to be delivered.
- Verbose tool summaries in group chats can expose tool args, URLs, plugin diagnostics, and data the model saw; the implementation should keep this explicit opt-in and preserve Slack non-DM, forum, sendPolicy, preview-suppression, media-only, and exec-approval behavior.
- The PR currently lacks focused regression tests for ordinary group chats with verbose off/on/full and for adjacent delivery gates affected by `shouldSendToolSummaries`.
