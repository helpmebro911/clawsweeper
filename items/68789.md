## Summary

Keep open. The report is recent, concrete, and not solved by current main. Main has improvements around Slack Socket Mode reconnects, proxy-agent wiring, stale-socket health, and fatal unhandled-rejection classification, but the central fresh-send failure can still propagate through the Slack WebClient write path: direct Slack sends use a cached write WebClient with `retryConfig: { retries: 0 }`, `maxRequestConcurrency: 1`, and no OpenClaw-level transient DNS retry or path-specific diagnostics around `conversations.open` / `chat.postMessage`. Related #24581 only covers crash-loop prevention, while this issue explicitly says the gateway stays up and fresh WebClient sends continue failing.

## Best Possible Solution

Keep this issue open for a Slack-plugin-owned fix. The likely direction is to add targeted diagnostics that distinguish `conversations.open`, `chat.postMessage`, Socket Mode `apps.connections.open`, and WebSocket ping/pong failures, then add a conservative transient DNS recovery path for safe/idempotent Slack WebClient operations. For actual message posts, use an idempotency/dedupe strategy or avoid blind retries to prevent duplicate sends. Add regression tests with Slack-like wrapped `EAI_AGAIN` request errors and, if feasible, a live Slack validation path on a configured gateway.

## Risks / Open Questions

- A naive retry around `chat.postMessage` can duplicate Slack messages; the current no-retry write client appears intentional and any recovery should preserve idempotency or only retry safe pre-send steps.
- There is no deterministic repro in the repository yet, so the best next step needs an injectable Slack WebClient/request-error regression test plus live Slack validation if maintainers can reproduce the affected environment.
