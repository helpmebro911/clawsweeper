## Summary

Current `main` does not implement the requested mapping for `sessions_spawn`. The ACP spawn path still passes the requested `agentId` straight through as the ACP runtime agent, while current docs describe that `agentId` as an ACP harness id, not an OpenClaw agent profile id. This looks like an unresolved behavior/design gap rather than something already fixed on `main`.

## Evidence

- **ACP session manager still forwards raw input agent:** `initializeSession()` normalizes `input.agent` directly and sends that value to `runtime.ensureSession(...)` without resolving `agents.list[].runtime.acp.agent`.
  - file: [src/acp/control-plane/manager.core.ts:309](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/acp/control-plane/manager.core.ts#L309)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '297,330p'`
  - sha: [b45329334965](https://github.com/openclaw/openclaw/commit/b453293349657dedd069d291fa5b078d910067b1)
- **ACP spawn path passes targetAgentId unchanged:** `sessions_spawn` initializes ACP with `agent: params.targetAgentId`, so the spawned ACP session uses the requested id verbatim.
  - file: [src/agents/acp-spawn.ts:847](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/agents/acp-spawn.ts#L847)

- **Docs currently define sessions_spawn agentId as harness id:** The ACP docs say `sessions_spawn(..., runtime: "acp")` uses `agentId` as the ACP target harness id and falls back to `acp.defaultAgent`; they do not describe resolving an OpenClaw agent profile alias here.
  - file: [docs/tools/acp-agents.md:311](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/docs/tools/acp-agents.md#L311)

- **Per-agent runtime.acp.agent is consumed for configured bindings:** Configured ACP bindings do resolve `agents.list[].runtime.acp.agent` into `acpAgentId`, which shows the mapping exists elsewhere but not on the ACP spawn path under review.
  - file: [src/channels/plugins/acp-configured-binding-consumer.ts:27](https://github.com/openclaw/openclaw/blob/b453293349657dedd069d291fa5b078d910067b1/src/channels/plugins/acp-configured-binding-consumer.ts#L27)

## Risks / Open Questions

- The issue body assumes `sessions_spawn.agentId` should accept an OpenClaw agent profile id, but current docs and command UX describe a harness id; maintainers may need to decide whether to change code, change docs, or reframe the issue as a feature request.
- There may be related policy implications because `acp.allowedAgents` is also enforced against the same raw `agentId` value today.
