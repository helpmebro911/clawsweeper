## Summary

Keep open. Current main still uses the generic missing-register diagnostics in the legacy plugin loader paths, so the PR is not implemented or obsolete. The PR is recent and plausibly useful, but the supplied review comments identify a real regression in the branch: it drops the documented debug module-shape detail for ordinary malformed plugins.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: preserve `formatMissingPluginRegisterError(mod, env)` for normal missing-register errors in both loader paths, add the bundled-channel-entry wrong-loader diagnostic without weakening the plugin kind contract, and add focused assertions for the full loader, CLI metadata loader, and bundled entry missing-field warning path. The earlier #67380 context appears to be a prior unmerged attempt, not a canonical replacement for this active PR.

## Risks / Open Questions

- Merging the branch as supplied would regress documented `OPENCLAW_PLUGIN_LOAD_DEBUG=1` diagnostics for ordinary malformed plugins.
- The requested bundled-channel-entry diagnostic should be added with a type-safe check; `PluginKind` currently only covers `memory` and `context-engine`, so a naive `hasKind(record.kind, "bundled-channel-entry")` call does not match current loader types.
