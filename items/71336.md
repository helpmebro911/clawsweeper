## Summary

Keep open. Current main still keeps the Control UI/WebChat pending chat queue in browser memory only; refresh/remount recreates it as an empty Lit state field, session switching clears it, and reconnect only flushes an existing in-memory queue. The PR was recently narrowed after maintainer feedback, and the related items do not supersede the remaining WebChat queue persistence work.

## Best Possible Solution

Keep #71336 open as the current focused WebChat queue-persistence candidate. The best path is to review it against current main, require tests for persist/restore/TTL/clear-on-drain/reconnect drain/session switch, and keep unrelated session-store crash-window work separate from this PR. If merged, leave any broader #51549 draft or history gaps tracked explicitly rather than closing them by implication.
