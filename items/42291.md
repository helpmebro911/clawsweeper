## Summary

Current `main` still appears to route `openclaw models auth login` through the local agent auth store, with no code path that forwards OAuth results to a remote gateway or blocks/warns on node-host remote-gateway setups. The issue remains actionable and should stay open.

## Evidence

- **Auth login resolves a local agent directory:** `resolveModelsAuthContext()` loads config, resolves the default agent id, and immediately derives `agentDir` with `resolveAgentDir(config, defaultAgentId)`. `modelsAuthLoginCommand()` then uses that resolved `agentDir` directly; there is no branch for `gateway.mode=remote`, node-host detection, or gateway sync.
  - file: [src/commands/models/auth.ts:104](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/commands/models/auth.ts#L104)
  - command: `nl -ba src/commands/models/auth.ts | sed -n '104,127p'; nl -ba src/commands/models/auth.ts | sed -n '584,637p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **OAuth results are persisted to the local auth store path:** `persistProviderAuthResult()` writes each returned profile with `upsertAuthProfile({ ..., agentDir: params.agentDir })`, and `runProviderAuthMethod()` passes the same local `agentDir` through after the provider auth flow completes. No gateway RPC or remote-store write is present in this path.
  - file: [src/commands/models/auth.ts:231](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/commands/models/auth.ts#L231)
