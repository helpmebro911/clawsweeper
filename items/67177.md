## Summary

Keep open. Current main still leaves the reported MS Teams `file.download.info` path unfixed: the `content.downloadUrl` branch returns the raw SharePoint/OneDrive URL, while the existing Graph shares rewrite is only applied to the separate `contentUrl` path. The tests cover generic `file.download.info` downloads and Graph-shares routing for `reference`/`contentUrl` attachments, but not the reported SharePoint `downloadUrl` case.

## Best Possible Solution

Patch `extensions/msteams/src/attachments/download.ts` so the `file.download.info` branch applies `tryBuildGraphSharesUrlForSharedLink(downloadUrl)`, uses the rewritten Graph shares URL when available, and handles `contentTypeHint` consistently with the existing `contentUrl` rewrite. Add a regression test using a SharePoint `content.downloadUrl` that verifies only `graph.microsoft.com/v1.0/shares/.../driveItem/content` is fetched and the Graph-scoped token retry path succeeds. Keep the current warning logger behavior for failed downloads.
