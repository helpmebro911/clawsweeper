## Summary

Keep open. Current `main` has partial overlap with the request through provider-level `parallel_tool_calls`, transcript compaction/context-engine plugins, and the experimental `update_plan` tool, but it does not implement the core asks in this issue: generic batch built-in tool calls, named context block save/load/unload, or hard plan-step enforcement.

## Evidence

- **Partial batch support only at provider payload level:** `applyDefaultOpenAIGptRuntimeParams()` auto-injects OpenAI `parallel_tool_calls = true` for eligible GPT models. This is provider request shaping, not generic built-in tool batching such as array-based `read_file`/`write_file`/`search` APIs or gateway-side multi-call coalescing.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:200](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/agents/pi-embedded-runner/extra-params.ts#L200)
  - command: `nl -ba src/agents/pi-embedded-runner/extra-params.ts | sed -n '200,216p'`
  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **`update_plan` is tracking, not enforcement:** The tool schema accepts an explanation plus a list of steps/statuses, validates that at most one step is `in_progress`, and then returns those details. There is no persisted plan state, no `plan_submit`/`plan_step_done`/`plan_status` API, and no step-gating logic here.
  - file: [src/agents/tools/update-plan-tool.ts:11](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/agents/tools/update-plan-tool.ts#L11)
