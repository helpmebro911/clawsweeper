## Summary

Current main still does not persist ACPX subprocess stderr after initialization. The OpenClaw ACPX plugin delegates child-process ownership to the pinned `acpx@0.6.1` runtime, and that published runtime still only records bounded startup stderr plus optional verbose mirroring. There is no OpenClaw config, manifest, docs, or test surface for an ACPX stderr log directory or retention policy.

## Best Possible Solution

Keep this issue open. The best path is to implement bounded, redacted per-process stderr teeing in upstream `openclaw/acpx` `AcpClient.start()`, expose a typed runtime option for log directory and retention, wire those options through `extensions/acpx` config/schema/manifest/docs, add tests for startup and post-initialize stderr capture, then bump the bundled `acpx` dependency in OpenClaw.

## Risks / Open Questions

- Persisting raw child stderr may capture API keys, OAuth diagnostics, environment details, filesystem paths, prompt content, or tool arguments; the implementation needs redaction, restrictive file permissions, discoverability, and retention before broad enablement.
- The scoped ACPX guide says reusable ACP runtime behavior belongs in `openclaw/acpx`; patching only the OpenClaw wrapper would create drift now that current main delegates subprocess ownership to the published runtime package.
- Open PR #53997 may contain useful artifact-design ideas, but it is open/unmerged and not sufficient to close this issue as implemented or superseded.
