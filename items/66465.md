## Summary

Keep open. Current main still does not persist ACPX subprocess stderr for post-mortem debugging. The bundled ACPX plugin delegates child-process lifecycle to the pinned `acpx@0.6.1` runtime, whose client still keeps only an 8 KB startup stderr buffer and mirrors stderr only when verbose mode is enabled. OpenClaw's ACPX plugin config, manifest, service wiring, and tests expose no stderr log path, retention, redaction, or artifact controls. Related PR #53997 is useful implementation context, but it is open/unmerged and not present on main.

## Best Possible Solution

Keep this issue open. Implement bounded, redacted per-process stderr teeing in upstream `openclaw/acpx` `AcpClient.start()`, expose typed runtime options for log directory and retention, then wire those through `extensions/acpx` config/schema/manifest/docs with focused startup and post-initialize stderr capture tests. Review #53997 for artifact-design ideas, but close this issue only after that work lands or a narrower canonical item supersedes it.

## Risks / Open Questions

- Persisting raw child stderr can capture secrets, auth diagnostics, local paths, prompt/tool content, or environment details; an implementation needs redaction, restrictive file permissions, retention, and clear operator discoverability.
- A wrapper-only implementation in `extensions/acpx` would drift from the owned child-process lifecycle in `openclaw/acpx`; the cleaner path is a typed upstream runtime option consumed by the bundled plugin.
- Open PR #53997 overlaps with durable ACP artifacts, but it is open/unmerged, so it should be reviewed as an implementation candidate rather than used as a reason to close this issue now.
