## Summary

Keep open. Current `main` still lacks the Slack restart-persistent thread-participation behavior described in PR #58731, and the PR discussion still shows unresolved follow-up around the bounded `conversations.replies` fallback and restart-survival coverage.

## Evidence

- **Current main still records Slack thread participation in memory only:** `dispatchPreparedSlackMessage` records participation only through `recordSlackThreadParticipation(...)`, and `sent-thread-cache.ts` implements that as a 24-hour in-memory dedupe cache. There is no persistent JSON/file-backed store on `main`.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:995](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/monitor/message-handler/dispatch.ts#L995)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/dispatch.ts | sed -n '988,1010p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Mention gating on main does not consult any persistent participation store:** `prepareSlackMessage` still treats `bot_thread_participant` as `hasSlackThreadParticipation(account.accountId, message.channel, message.thread_ts)` only. That means `requireMention` thread continuation still depends on transient process memory and will not survive a gateway restart.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:327](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/slack/src/monitor/message-handler/prepare.ts#L327)
