## Summary

Keep this issue open. Current main has a narrower Telegram/agents fix that hides failed-tool warnings only when the assistant explicitly acknowledges the failed mutating action, but the broader reported warning-policy behavior remains: recoverable mutating tool errors still produce user-facing `⚠️ ... failed` payloads, and tests explicitly lock that in.

## Best Possible Solution

Keep this issue open. The likely maintainer path is a focused change around `src/agents/pi-embedded-runner/run/payloads.ts`, with possible `lastToolError` clearing or proof-of-application handling in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, plus regression tests proving recoverable or proven-applied mutating failures do not emit false `failed` notifications while real mutating failures remain visible. Coordinate with #59868 and PR #60664, but do not treat either as shipped replacement work.

## Risks / Open Questions

- A fix should preserve warnings for genuine mutating failures such as permission errors, disk-full writes, failed sends, and failed deletes.
- The existing recoverable keyword classifier is broad; maintainers should decide whether mutating suppression should use that classifier or require a narrower proven-applied/post-write recovery signal.
- Related #59868 and PR #60664 address nearby edit-tool recovery behavior, but the provided context shows they are not a shipped replacement for this warning-policy issue.
