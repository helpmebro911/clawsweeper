## Summary

Keep open. Current `main` does add Microsoft Teams per-thread session isolation for clean channel routes, which lines up with the reporter's link to the earlier `#58615`/`#59294` discussion, but I do not have high-confidence evidence that the specific mixed-key failure (`...:thread:OLD:thread:NEW`) is actually fixed. The shared thread-session helper still blindly appends a `:thread:` suffix to whatever base session key it receives, existing MSTeams tests only cover clean base keys, and downstream reply routing still falls back to the thread encoded in the current session key when MSTeams inbound context omits `MessageThreadId`. That leaves the old-session-reselection hypothesis materially unruled out on current `main`.

## Evidence
