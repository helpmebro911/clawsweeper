## Summary

Keep this issue open. Current main still has a plausible and concrete mixed Teams thread-session key path: the MSTeams handler resolves a base channel route, mutates that route with a thread suffix, the shared helper blindly appends `:thread:<id>`, and the route cache can retain the mutated object from a cache miss for later messages in the same normalized Teams channel.

## Best Possible Solution

Keep this issue open. Fix MSTeams thread routing so cached base routes remain immutable: derive the thread-qualified session key from a clean base route, avoid mutating the object returned by `resolveAgentRoute()`, and add a regression test for two routed Teams channel-thread messages in the same channel asserting the second key is `base:thread:new`, never `base:thread:old:thread:new`.
