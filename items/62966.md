## Summary

Keep this PR open. Current main still has the underlying gap: reply metadata exists on finalized inbound contexts and is populated by channel code, but `deriveInboundMessageHookContext` and `toPluginInboundClaimEvent` do not carry `ReplyToId` or `ReplyToBody` into `inbound_claim` metadata. The PR remains a narrow additive implementation candidate, not obsolete or superseded.

## Best Possible Solution

Keep this PR open and review it as a small plugin hook API fix. The best implementation is to add optional `replyToId` and `replyToBody` to the canonical inbound hook context, map them from `ctx.ReplyToId` and `ctx.ReplyToBody`, expose them in `inbound_claim` metadata, and add a focused assertion in `src/hooks/message-hook-mappers.test.ts`. A short docs note may be useful if maintainers want plugin authors to rely on these metadata keys.
