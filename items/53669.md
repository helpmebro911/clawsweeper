## Summary

Keep this issue open. Current main still has correct runtime cron timezone evaluation, but the reported failure is model argument selection for CronCreate/cron.add. The model-facing cron tool schema and description still do not explicitly say that a cron expression paired with `tz` must use that timezone's local wall-clock time and must not be UTC-converted first.

## Best Possible Solution

Make a narrow model-facing fix in `src/agents/tools/cron-tool.ts`: explicitly state that for `schedule.kind="cron"`, `expr` is written in the supplied `tz` timezone's local wall-clock time and must not be converted to UTC. Include the reported Shanghai example, such as 6pm Shanghai -> `expr: "0 18 * * *"`, `tz: "Asia/Shanghai"`, and add focused schema/description regression coverage in `src/agents/tools/cron-tool.schema.test.ts` or the existing cron tool tests.
