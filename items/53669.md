## Summary

Keep open. Current main still has the intended cron runtime behavior and user docs for local wall-clock scheduling, but the reported bug is upstream of runtime: the model can choose the wrong cron expression before calling `cron.add`. The model-facing cron tool schema and description still do not explicitly say that when `tz` is set, `expr` must stay in that timezone's local wall clock and must not be UTC-converted.

## Best Possible Solution

Keep this issue open and make a narrow model-facing fix in `src/agents/tools/cron-tool.ts`: explicitly state that for `schedule.kind="cron"`, `expr` is written in the supplied `tz` timezone's local wall clock and must not be converted to UTC. Include the reported example, such as Shanghai 6pm -> `expr: "0 18 * * *"`, `tz: "Asia/Shanghai"`, then add focused schema/description regression coverage in `src/agents/tools/cron-tool.schema.test.ts` or the existing cron tool tests.
