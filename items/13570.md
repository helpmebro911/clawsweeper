## Summary

Keep #13570 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still does not expose `contextPruning.forcePruneRatio` or any equivalent high-usage TTL bypass. The prior issue comment’s conclusion remains accurate on the newer checkout: the closed PR #13572 was not merged, schemas/settings/docs omit the key, and the runtime hook still returns during a fresh TTL window before pruning can run.

## Best Possible Solution

Keep #13570 open until a maintainer implements or explicitly declines the feature. The likely fix is to add optional `agents.defaults.contextPruning.forcePruneRatio` to the config type, strict Zod schema, generated schema/help, and user-facing docs; carry it through `computeEffectiveSettings()` and `ContextPruningRuntimeValue`; estimate context usage before the fresh-TTL early return; bypass TTL only when the configured threshold is met; preserve the existing TTL gate below threshold; and add regression tests for both above-threshold bypass and below-threshold preservation.
