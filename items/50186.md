## Summary

Keep open. Current main has adopted `KillMode=control-group` for newly rendered systemd gateway units, but the central requested doctor/status observability is still missing: neither the systemd runtime path nor the service audit exposes `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so an existing `KillMode=process` unit can still pass without the focused hygiene warning requested in #50186.

## Evidence

- **New systemd units use control-group:** `buildSystemdUnit` renders `KillMode=control-group` with a comment tying it to child-process cleanup, so the safer-new-unit part is partially implemented on current main.
  - file: [src/daemon/systemd-unit.ts:83](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/daemon/systemd-unit.ts#L83)
  - command: `nl -ba src/daemon/systemd-unit.ts | sed -n '75,90p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Systemd runtime status omits requested hygiene properties:** `readSystemdServiceRuntime` calls `systemctl show` for only `ActiveState,SubState,MainPID,ExecMainStatus,ExecMainCode`; it does not request `KillMode`, `TasksCurrent`, or `MemoryCurrent`, so `gateway status --deep` cannot surface the requested cgroup hygiene data.
  - file: [src/daemon/systemd.ts:683](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/daemon/systemd.ts#L683)

- **Runtime type has no fields for KillMode/tasks/memory:** `GatewayServiceRuntime` only carries status/state/substate/pid/exit/detail-style fields, with no place for `KillMode`, `TasksCurrent`, or `MemoryCurrent`.
  - file: [src/daemon/service-runtime.ts:1](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/daemon/service-runtime.ts#L1)
