## Summary

Keep this issue open. Current main at 646a268d2710bc54e478349ef5690f3c8803df4f still uses strict-priority contextTokens resolution in the main resolver, auto-reply, run/session persistence, and gateway defaults. maxTokens also remains handled at provider/transport boundaries rather than through a shared minimum resolver.

## Best Possible Solution

Implement a shared positive-token-limit resolver that computes the minimum of valid per-agent overrides, agent defaults, provider model contextTokens/contextWindow, discovery-cache values, and safe defaults for contextTokens. Wire it through agent run setup, auto-reply, gateway session defaults/status/session persistence, and transcript/session fallback paths. Add a provider-aware maxTokens clamp at request construction or stream-option normalization so explicit agent params cannot exceed model/provider limits, with focused tests for configured caps higher than model/discovery limits.
