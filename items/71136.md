## Summary

Keep #71136 open. Current main partially caps the embedded Pi runtime context window when `agents.defaults.contextTokens` is smaller than the model/config window, but shared context-token resolution still short-circuits on agent overrides in reply/session/status paths, and positive `maxTokens` request/config values are still forwarded without a general selected-model clamp in major transport paths.

## Best Possible Solution

Keep the issue open and implement a shared token-limit resolver used by reply/session/status/runtime paths. It should compute the minimum of positive finite agent default/per-agent overrides, provider model `contextTokens`/`contextWindow`, discovery/cache values, and `DEFAULT_CONTEXT_TOKENS`, then add focused regression tests for oversized `agents.defaults.contextTokens` and `agents.list[].contextTokens`. For `maxTokens`, clamp positive configured/requested values against the selected model's `maxTokens` at the common extra-params/transport boundary while preserving explicitly tested provider semantics such as OpenAI WS `maxTokens: 0` unless maintainers decide to retire them.
