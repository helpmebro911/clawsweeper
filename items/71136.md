## Summary

Keep #71136 open. Current main has partial runtime cap handling for `agents.defaults.contextTokens`, but the broader requested minimum-of-agent-config, provider config, discovery, and default behavior is not implemented consistently, and the `maxTokens` half remains priority/override based.

## Best Possible Solution

Keep the issue open and implement a shared token-limit resolver that collects positive finite candidates from per-agent config, agent defaults, configured provider model metadata, discovered/runtime model metadata, and defaults, then uses the minimum where the value is a safety cap. Apply the same policy to configured/requested `maxTokens` before transport dispatch, with targeted tests for oversized agent defaults, per-agent overrides, configured model values, discovery/runtime metadata, and provider-specific max-token forwarding.

## Risks / Open Questions

- A fix needs to preserve the existing intentional case where a smaller `agents.defaults.contextTokens` acts as a runtime cap, while changing oversized agent/model config values to clamp to smaller model or discovery limits.
- `maxTokens` has provider-specific semantics and existing transport-level clamps in some providers, so the implementation should centralize shared limit resolution without double-clamping provider-specific behavior.
