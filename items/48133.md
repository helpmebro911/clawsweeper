## Summary

Keep #48133 open. Current main still forwards the direct `firecrawl_scrape` URL to Firecrawl without a local SSRF check, while the explicit Firecrawl scrape tool is registered and documented as a callable plugin tool. The cache-size portion appears already covered by the shared cache helper on main, but the central SSRF fix is not implemented or superseded.

## Best Possible Solution

Keep the PR open for maintainer/security review. The best path is a narrow Firecrawl-owned fix that validates `params.url` before cache lookup and before `postFirecrawlJson`, adds regression tests for blocked literal/private/metadata/non-HTTP/malformed URLs, keeps the already-shared cache eviction instead of duplicating a new cap, and explicitly documents or resolves the remote-DNS limitation before landing.
