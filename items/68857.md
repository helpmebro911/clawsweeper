## Summary

Keep this PR open. Current main still has the reported hardcoded gateway-side `node.invoke` default and does not expose `gateway.nodes.invokeTimeoutMs`, so the work is not implemented or obsolete. The PR is a plausible implementation candidate, but it needs revision before merge because it misses schema/docs/test coverage and does not propagate the resolved default timeout to the node-side payload.

## Best Possible Solution

Keep the PR open and ask the contributor to finish it as a complete config contract change: add `gateway.nodes.invokeTimeoutMs` to the config type, strict zod schema, labels/help/tags/generated metadata, and public config docs; validate it as a positive integer with any maintainer-approved bound; wire it into `NodeRegistry`; resolve the effective timeout once so both the gateway timer and `node.invoke.request.timeoutMs` receive the same value; preserve per-call `timeoutMs` precedence; and add focused tests for schema acceptance/rejection, default 30s behavior, configured default behavior, and per-call override behavior. The paired feature issue being closed does not make this PR obsolete because current main still lacks the capability.

## Risks / Open Questions

- As written, the PR’s advertised config key would still fail strict runtime config validation unless `src/config/zod-schema.ts` and generated/help metadata are updated.
- The PR resolves the gateway wait timeout but leaves the node payload timeout undefined when callers omit per-call `timeoutMs`, so Swift/iOS nodes can still enforce their 30s default and abort long operations early.
- The proposed config needs positive-integer validation, and preferably an explicit upper bound or documented policy, to avoid instant timeouts from zero/negative values or unexpectedly long pending gateway invocations.
