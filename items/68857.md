## Summary

Keep open. Current main still has the gateway-side node.invoke default hardcoded to 30 seconds, the public gateway node config type and strict runtime config schema do not accept gateway.nodes.invokeTimeoutMs, and the linked feature request #68090 remains actionable. The PR is a focused implementation attempt, but it is incomplete as submitted because it omits strict zod schema support and does not propagate the resolved default timeout into the node.invoke.request payload used by Swift nodes.

## Best Possible Solution

Keep the PR open and ask for a complete config-contract implementation: add gateway.nodes.invokeTimeoutMs to types, zod schema, labels/docs as appropriate, validate it as a finite positive integer, wire it into NodeRegistry, resolve the effective timeout once, send that resolved timeout in node.invoke.request, and add focused gateway/config tests covering default behavior, configured default, per-call override, and invalid config handling.

## Risks / Open Questions

- A config-only timeout would not work until the strict zod schema, generated/help labels or docs, and startup wiring are updated together.
- If the gateway waits longer but sends no resolved timeoutMs to node.invoke.request, Swift nodes can still enforce their local 30s default and abort long operations early.
- The PR currently has no visible regression test coverage for default preservation, configured default use, invalid values, and per-call override precedence.
