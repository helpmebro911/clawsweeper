## Summary

Keep PR #47277 open. Current main does not already implement user-specific memory-file isolation: the shipped memory host SDK and memory-core plugin still read and index agent-wide `MEMORY.md` plus the shared `memory/` tree, with no user/requester scope in the relevant public runtime contracts. The PR is not a ClawHub close because it touches the bundled memory plugin and privacy/security behavior, but the latest PR discussion still shows unresolved correctness and security review concerns, so the best maintainer path is continued review/redesign rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review, but require a security-focused redesign before merge: introduce a first-class, filesystem-safe memory requester scope through the memory plugin/runtime and memory-host SDK contracts; derive it only from trusted plugin context or well-defined session identity; apply it consistently to builtin indexing/search/read, QMD collections/fallbacks, config schema, docs, and tests; and add regression coverage for traversal, case-insensitive filesystems, DM/channel/account identity, missing requesterSenderId fallback, QMD parity, and cache/resource behavior.

## Risks / Open Questions

- A partial implementation could create a misleading privacy boundary unless builtin and QMD search, exact reads, cache keys/lifetimes, path normalization, case sensitivity, and session/requester identity derivation are handled together.
- The remaining design needs an explicit decision on whether root `memory/` files are shared by design or should be migrated/scoped, because current docs and indexing behavior treat that tree as agent-wide memory.
