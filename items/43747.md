## Summary

Keep open. Current main documents and implements a clearer memory backend model, but it still does not implement the issue's requested single strict memory mode covering builtin-only, qmd-only, mixed behavior, compaction flush writes, fallback policy, and workspace-scoped SQLite storage.

## Best Possible Solution

Keep this open as a focused maintainer design task. Define a single user-facing memory-mode contract, or an explicitly documented successor, across `src/config/types.memory.ts`, `src/config/zod-schema.ts`, config help/docs, `src/agents/memory-search.ts`, `extensions/memory-core/src/flush-plan.ts`, and `extensions/memory-core/src/memory/search-manager.ts`. The design should cover durable write target, search/index backend, fallback policy, index path scoping, and migration/doctor guidance for users who already have both Markdown and SQLite memory artifacts.

## Evidence

- **Issue discussion defines remaining product request:** The reporter moved from inconsistent observed behavior to a concrete request for one `memory.type` contract with `built-in-only`, `qmd-only`, and `mixed` modes, plus per-workspace SQLite behavior. Follow-up discussion correctly identified today's split between `agents.defaults.compaction.memoryFlush`, `memory.backend`, and `memory.qmd.*`.
- **Current config type exposes backend selection, not strict memory modes:** `MemoryConfig` only supports `backend?: "builtin" | "qmd"`, `citations`, and `qmd` settings. There is no `memory.type` or enum for `built-in-only`, `qmd-only`, or `mixed`.

## Risks / Open Questions

- The request is broad and product-design oriented; a fix should not simply add `memory.type` without defining how it controls durable writes, search backend, fallback, index storage, startup context, migration, and doctor guidance.
- Teams can still see different behavior when embedding credentials, QMD availability, `memorySearch.enabled`, `memory.backend`, plugin slot selection, or `memoryFlush.enabled` differ.
