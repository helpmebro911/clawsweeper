## Summary

Keep #71330 open. Current main still hardcodes short-term promotion output to `MEMORY.md`; there is no memory-core `promotionTarget` config field, no CLI/dreaming pass-through for an alternate target, and docs/tests still describe and assert promotion into `MEMORY.md`. The related memory-wiki/adaptive-memory items are adjacent but do not cover this concrete bundled memory-core behavior.

## Best Possible Solution

Keep the issue open and implement this in the bundled `memory-core` plugin: add a validated `promotionTarget` or `dreaming.deep.promotionTarget` config with default `MEMORY.md`, safe workspace-relative path normalization, optional date-token expansion if maintainers accept that scope, and pass it through both manual `openclaw memory promote --apply` and scheduled dreaming. Update docs/schema/tests so promoted files under `memory/**` remain indexed by default, and document when users need `memorySearch.extraPaths` for root-level custom files such as `MEMORY_PROMOTIONS.md`.
