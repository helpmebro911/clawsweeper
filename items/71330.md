## Summary

Keep #71330 open. Current main still hard-codes memory-core short-term promotion output to workspace `MEMORY.md`, the plugin config schema has no `promotionTarget` or equivalent, and the CLI/docs explicitly describe `memory promote --apply` and dreaming deep promotion as writing only to `MEMORY.md`. The related memory-wiki/adaptive-memory issues are broader adjacent work and do not supersede this exact configurable target-file request.

## Best Possible Solution

Keep the issue open and implement a focused `memory-core` promotion target configuration. A good path is to add a strict schema/runtime field such as `plugins.entries.memory-core.config.dreaming.promotionTarget` or `dreaming.phases.deep.promotionTarget`, default it to `MEMORY.md`, validate it as a safe workspace-relative Markdown path, support documented date tokens, create parent directories/frontmatter when needed, thread it through manual CLI apply and scheduled dreaming, and add tests/docs. If targets outside `memory/**/*.md` are allowed, the implementation should also make them searchable by updating memory indexing configuration or constraining/documenting the supported target locations.
