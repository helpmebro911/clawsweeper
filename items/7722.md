## Summary

Keep open. Current main still implements only the narrower `tools.fs.workspaceOnly` workspace guard for filesystem-style tools and prompt/media paths. It does not implement the requested `tools.fileAccess` shape, allow/deny path lists, or exec-command path enforcement, and the latest issue discussion explicitly preserves that remaining scope.

## Best Possible Solution

Keep tracking #7722 until OpenClaw either implements a real filesystem access policy surface, likely under `tools.fs` or an equivalent versioned config contract, with allow/deny path rules enforced across read/write/edit/apply_patch and a defensible exec strategy, or a maintainer explicitly narrows/rejects the product direction. The existing `tools.fs.workspaceOnly` implementation should remain documented as the partial workspace-only hardening path, not treated as completion of this issue.
