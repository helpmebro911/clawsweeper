## Summary

Keep open. Current `main` has coarse filesystem hardening via `tools.fs.workspaceOnly` and sandbox controls, but it does not implement the requested `tools.fileAccess` config, per-path `allowedPaths`/`denyPaths`, or exec-layer host filesystem restrictions.

## Evidence

- **Config types only expose workspace-only guards:** `ExecToolConfig` only adds `applyPatch.workspaceOnly`, and `FsToolsConfig` is a single boolean `workspaceOnly`; there is no `tools.fileAccess`, `allowedPaths`, or `denyPaths` surface in the typed config.
  - file: [src/config/types.tools.ts:279](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/config/types.tools.ts#L279)
  - command: `nl -ba src/config/types.tools.ts | sed -n '270,310p'`
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Runtime schema rejects unknown filesystem keys beyond workspaceOnly:** The runtime Zod schema defines only `tools.exec.applyPatch.workspaceOnly` and `tools.fs.workspaceOnly` for filesystem path guarding. No schema entries exist for `tools.fileAccess`, `allowedPaths`, or `denyPaths`.
  - file: [src/config/zod-schema.agent-runtime.ts:438](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/config/zod-schema.agent-runtime.ts#L438)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '438,490p'`
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Tool assembly uses a boolean workspaceOnly policy, not path allow/deny lists:** `pi-tools.ts` resolves only `fsConfig.workspaceOnly`, wraps `read`/`write`/`edit` with workspace guards when true, and separately configures `exec` with host/security/ask/node settings. There is no per-path allowlist/denylist enforcement for exec.
  - file: [src/agents/pi-tools.ts:425](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/agents/pi-tools.ts#L425)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '424,555p'`
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Host FS operations are all-or-nothing relative to the workspace:** When `workspaceOnly` is false, host write/edit operations resolve arbitrary host paths; when true, they enforce the workspace root. There is no evaluation against configurable allowed/denied host paths.
  - file: [src/agents/pi-tools.read.ts:760](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/agents/pi-tools.read.ts#L760)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '760,835p'`
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Current security guidance still points users to workspace-only mode, sandboxing, and separate OS trust boundaries:** The docs describe `tools.fs.workspaceOnly` as the filesystem hardening knob, and the security audit remediation for mutually untrusted users is still separate gateways/credentials and ideally separate OS users/hosts. That is materially weaker/different than the requested per-path host FS sandbox.
