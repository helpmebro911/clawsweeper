## Summary

Keep open. Current main has useful partial abort plumbing, including session-scoped `chat.abort` and `/stop`, but it still does not fully solve the reported loss-of-control path: the Control UI Stop button is tied to local `chatRunId`, reconnect clears that state, reload/reconnect can leave no Stop button for a still-active server run, and the Gateway still relies on application-level `tick` keepalives rather than a WebSocket protocol ping/pong keepalive.

## Best Possible Solution

Keep #70991 open as the canonical issue. The best fix is to expose or recover active abortable run state from the Gateway after reconnect/reload, make the Control UI Stop control depend on server-side abortable session/run state instead of only local `chatRunId`, preserve `/stop`/`sessions.abort` as session-scoped fallbacks, and address the transport stability side through #56668 or an equivalent protocol-level keepalive with focused regression coverage.
