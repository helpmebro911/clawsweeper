## Summary

Keep this issue open. Current main still has the Control UI recovery gap after disconnect/reload: Stop visibility is tied to browser-local chatRunId, reconnect clears that state, disconnected abort without a local run id is a no-op, and the gateway still only advertises/broadcasts an application-level tick rather than a WebSocket protocol ping. Session-scoped chat.abort and /stop handling are useful partial fixes, but they do not prove the long-task disconnect is fixed or preserve a Stop affordance after reload.

## Best Possible Solution

Keep this issue open as the canonical Control UI/Gateway long-task recovery regression. A maintainer-quality fix should expose recoverable active run or abortable session state after reconnect/reload, keep a session-level Stop affordance available while gateway work can be aborted, make /stop visibly work after a fresh reload or disconnected recovery path, add WS protocol ping/pong or equivalent proven keepalive, and add regression coverage for active-run disconnect/reconnect/reload.
