## Summary

Keep open. Current main has useful partial abort support, including session-scoped `chat.abort` and `/stop` routing, but the reported Control UI recovery bug is still present: Stop visibility depends on volatile browser `chatRunId`, reconnect clears that state, disconnected aborts without a local run id are no-ops, and the gateway still has only application-level tick keepalive rather than WS protocol ping evidence. Related #56668 remains open and only covers the transport keepalive part.

## Best Possible Solution

Keep this issue open as the canonical Control UI/Gateway long-task recovery regression. A maintainer-quality fix should expose or recover active-run state for the selected session after reconnect/reload, keep a session-level Stop affordance available while the gateway has abortable work, preserve visible `/stop` feedback after a fresh reload, add WS protocol ping/pong or equivalent proven keepalive for long idle tool-call windows, and add regression coverage for active run disconnect/reconnect/reload.
