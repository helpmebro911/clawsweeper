## Summary

Keep #70991 open. Current main has useful partial fixes for session-scoped aborts and queued aborts across reconnect, but the reported Stop-button disappearance after reconnect/reload is still plausible because the Control UI renders Stop only from local `chatRunId`, and reconnect explicitly clears that state. The connection-drop part is also not proven fixed, and related context says the protocol-level WebSocket keepalive PR #56668 is still open.

## Best Possible Solution

Keep #70991 open and treat it as a Control UI/gateway reliability bug. The best fix is to make active run state recoverable after reconnect or reload, keep a human-accessible session-scoped Stop affordance when the selected session is still running, ensure `/stop` works after a fresh reload, and separately verify whether a gateway WebSocket keepalive/ping change from #56668 is needed for the long-task disconnects.
