## Summary

Keep open. The current main code has abort plumbing, but it still leaves the reported product failure plausible and actionable: Control UI clears active run state after reconnect, so the Stop button can disappear, and there is no evidence in current main that the long-task gateway disconnect path is fixed.

## Best Possible Solution

Keep the issue open and fix it in core Control UI/Gateway behavior. The likely path is to make active chat runs recoverable after reconnect/page reload, keep a session-scoped Stop control available while the gateway reports an active run, return visible feedback for `/stop`, and investigate WebSocket keepalive/reconnect robustness with regression coverage for long-running `chat.send` across disconnect and reload.

## Evidence

  - file: [docs/web/control-ui.md:186](https://github.com/openclaw/openclaw/blob/8bbb143ab87e0a45f9d3f986768d75168750b672/docs/web/control-ui.md#L186)

That directly supports the report that the Stop button can disappear after disconnect/reload while the gateway-side task may still be running.
  - file: [ui/src/ui/app-gateway.ts:307](https://github.com/openclaw/openclaw/blob/8bbb143ab87e0a45f9d3f986768d75168750b672/ui/src/ui/app-gateway.ts#L307)
