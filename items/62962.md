## Summary

Current main still does not implement the requested schema-driven tool security classification contract. OpenClaw has owner-only metadata, allow/deny filtering, hook-driven approvals, ACP/name-action heuristics, and a provider/request-level parallel-tool-call setting, but not standardized `isReadOnly`, `isDestructive`, `isConcurrencySafe`, `interruptBehavior`, `isOpenWorld`, or `toAutoClassifierInput` fields on tool definitions or conformance checks.

## Best Possible Solution

Keep this issue open. The best implementation path is a backwards-compatible core/public SDK tool metadata contract: add the classification type to the core tool surface, preserve it through wrappers, plugin/channel tool APIs, adapters, and catalogs where needed, annotate core tools, extend `TOOL_POLICY_CONFORMANCE` and tests, document the contract, then incrementally consume the metadata for approval wording, interrupt handling, and scheduling while keeping current heuristics as conservative fallbacks.

## Risks / Open Questions

- This is a core/public SDK contract change, so defaults and compatibility need maintainer design before implementation.
- Existing ACP and mutation heuristics may need to become fallback behavior rather than being removed abruptly.
- Provider-level `parallel_tool_calls` and any future per-tool concurrency metadata need a careful mapping so unsafe tools are not scheduled concurrently by accident.
