## Summary

Keep open. Current main partially addresses the report by converting binary image clipboard items into chat attachments and centralizing draft/attachment restoration in the send pipeline, but the reported massive `data:image/...;base64,...` text-paste path is still not safely handled: text/plain data-URI clipboard content falls through to the textarea. The installed package also still publishes dist assets while excluding source maps, so the source-to-dist traceability concern remains.

## Evidence

- **Issue is eligible for review but asks for behavior still missing:** The issue author association is NONE and labels are only bug/bug:behavior, so there is no protected maintainer signal. The body reports that pasting can inject a very large `data:image/...;base64,...` string into the Control UI composer and asks for attachment/reject behavior plus regression tests and source-to-dist clarity.
- **Composer paste handler only intercepts binary image clipboard items:** `handlePaste` reads `e.clipboardData?.items`, collects entries whose `item.type` starts with `image/`, and returns when no such items exist.
