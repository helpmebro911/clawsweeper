## Summary

Current main does not yet satisfy the reported behavior. The `sessions_history` tool still exposes `limit` with only a minimum, forwards any finite caller-provided value directly to `chat.history`, and the gateway protocol still rejects `chat.history.limit > 1000` before the server-side clamp runs. The related open PR #66650 tracks the closest fix, but until a clamp/retry lands on main this remains a real bug rather than an auto-close candidate.

## Best Possible Solution

Keep the issue open until main clamps `sessions_history.limit` to the `chat.history` maximum, ideally via a shared constant or local `1000` cap plus runtime clamping before the gateway call, and add a regression test proving `limit: 2000` results in a compliant `chat.history` request or a successful retry. PR #66650 is the closest current implementation direction; #66615's server-max increase changes the protocol contract and should not be treated as the conservative fix without maintainer review.
