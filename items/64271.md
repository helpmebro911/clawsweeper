## Summary

Current main still lacks the requested per-profile circuit breaker/backoff for repeated managed Chrome CDP launch failures. The browser plugin deduplicates concurrent availability checks only, then clears that guard after settlement, and the existing test suite still expects a later call to retry launch after a prior launch failure.

## Best Possible Solution

Keep this issue open. Implement the fix inside the bundled browser plugin by adding per-profile managed-launch failure tracking in `extensions/browser/src/browser/server-context.availability.ts`, short-circuiting repeated `launchOpenClawChrome()` attempts after a small threshold with a cooldown or disabled-until-restart state, resetting the state on successful CDP readiness, adding clearer guidance to disable browser control when unused, and updating the existing ensure-browser-available tests so persistent launch/readiness failures do not keep spawning Chrome.
