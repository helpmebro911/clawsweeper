## Summary

Keep open. Current `main` still retries managed-browser startup on each `browser.request` without any per-profile consecutive-failure cap, cooldown, or backoff, so the reported CPU-burning loop remains plausible and the requested fix is not implemented.

## Evidence

- **Current availability path still relaunches on every request:** In the `!httpReachable` path, managed loopback profiles go straight to `launchOpenClawChrome(...)`, then either return or throw after `waitForCdpReadyAfterLaunch()`. There is no stored failure counter, cooldown window, or circuit-breaker state in this path.
  - file: [extensions/browser/src/browser/server-context.availability.ts:220](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/browser/src/browser/server-context.availability.ts#L220)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '203,307p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Per-attempt startup wait is still 15 seconds:** `CHROME_LAUNCH_READY_WINDOW_MS` remains `15_000`, which matches the issue report that each failed spawn burns CPU for about 15 seconds before failing.
  - file: [extensions/browser/src/browser/cdp-timeouts.ts:11](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/browser/src/browser/cdp-timeouts.ts#L11)
