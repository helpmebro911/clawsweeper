## Summary

Keep #70333 open. Current main still persists auto-enable-derived `enabled: true` flags during gateway startup, and `doctor` still treats auto-enable as a repair mutation that can be written back. That directly conflicts with the issue's requested runtime-only behavior, even though current reload code appears to have a hash-dedupe mitigation for one previous restart-churn path.

## Best Possible Solution

Keep the issue open and fix auto-enable as a runtime/diagnostic overlay, not a persisted config repair. Specifically, gateway startup should stop writing `autoEnable.config` back to `openclaw.json`, `doctor --fix` should report auto-enable reasoning without applying that mutation to the persisted candidate, and regression tests should cover startup and doctor preserving configs where `channels.whatsapp.enabled`, `channels.telegram.enabled`, and `plugins.entries.openai.enabled` are absent while runtime activation still works.
