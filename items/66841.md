## Summary

Keep open. Current main still implements `openclaw logs` as a `logs.tail` RPC over the configured gateway file log, with local loopback pairing/scope failures falling back to `readConfiguredLogTail`; the requested Linux/systemd journal preference, service provenance, PID/ExecStart reporting, service-version reporting, and mismatch warning are still absent.

## Best Possible Solution

Keep this issue open and implement the Linux/systemd local path in `src/cli/logs-cli.ts`: detect the active `openclaw-gateway.service`, prefer `journalctl --user _PID=<pid>` for local follow when available, print CLI version, service ExecStart, service PID, service version, and exact log source before tailing, warn on mismatches, retain RPC/file-log behavior for explicit remote `--url` and Control UI surfaces, add focused CLI tests for pairing/scope fallback to journal, and update the public logs/logging docs.

## Risks / Open Questions

- The unresolved behavior can still mislead operators after side-by-side cutover because an old CLI/install context can tail a configured rolling file log instead of the active gateway service journal.
- A fix needs to preserve explicit remote `--url` RPC behavior and Control UI `logs.tail` file-log behavior while adding a local Linux/systemd journal path only where appropriate.
- The JSON output contract should be designed deliberately before adding provenance so scripts consuming `openclaw logs --json` are not broken unexpectedly.
