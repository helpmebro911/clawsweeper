## Summary

Current main still has the reported behavior: `openclaw logs` calls `logs.tail`, falls back to `readConfiguredLogTail` on local loopback pairing/scope failures, and prints file-log metadata rather than active systemd service provenance. The reporter's validated local prototype is useful implementation evidence, but that behavior is not present on main and no open PR with closing syntax for this issue was found.

## Best Possible Solution

Keep this issue open. Implement the local Linux/systemd path in `src/cli/logs-cli.ts`: detect the active `openclaw-gateway.service` with the existing daemon service helpers, prefer `journalctl --user _PID=<pid>` for local `--follow` when available, print CLI version, service ExecStart, service PID, service version, and exact log source before tailing, warn on mismatches, keep explicit `--url` and Control UI `logs.tail` file-log behavior unchanged, add focused CLI tests for pairing/scope fallback to journal, and update the public logs/logging docs.

## Risks / Open Questions

- Operators can still be misled during side-by-side Linux/systemd cutover because an old CLI/install context may tail a configured rolling file instead of the active gateway service journal.
- A fix should preserve explicit remote `--url` RPC behavior and Control UI `logs.tail` file-log behavior while adding the Linux/systemd local CLI path only where appropriate.
- Any provenance added to `--json` output should be designed deliberately so existing scripts consuming `openclaw logs --json` are not broken unexpectedly.
