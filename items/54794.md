## Summary

Current main still does not implement Telegram inline-query UX. The issue discussion accurately identifies a real gap: OpenClaw receives allowed update metadata for inline_query/chosen_inline_result, and can display Telegram's BotFather inline-query capability flag, but the Telegram runtime has no inline_query/chosen_inline_result handler, no answerInlineQuery path, no InlineQueryResult construction, and no public config/docs surface for enabling or constraining inline mode. The closed unmerged PR #51963 remains useful reference material, not shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical Telegram inline-query tracker. The likely fix belongs in the bundled Telegram plugin: add explicit config/schema/docs for inline queries, handle inline_query and chosen_inline_result updates, authorize senders conservatively, route bounded requests through the agent flow, answer with Telegram InlineQueryResultArticle results, and add focused Telegram plugin tests. Treat closed PR #51963 as reference material unless a new reviewed implementation lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram contexts, so an implementation needs explicit sender authorization, privacy/session boundaries, debounce/rate limiting, and bounded execution before routing into the agent flow.
- The existing Telegram status flag can mislead operators because inlineQueries reports BotFather/account capability, not OpenClaw's ability to answer inline queries.
