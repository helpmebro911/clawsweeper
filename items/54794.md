## Summary

Keep this issue open. Current main still does not implement Telegram inline query UX: the Telegram plugin has allowed-update awareness and status reporting for BotFather inline capability, but no `inline_query`/`chosen_inline_result` runtime handler, no `answerInlineQuery` path, and no config/docs surface for enabling inline queries. The related PR #51963 is closed unmerged, so it is useful implementation context rather than shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely implementation belongs in the bundled Telegram plugin: add an explicit inline-query config/schema/docs surface, intentionally request `inline_query` and `chosen_inline_result` updates, register grammY handlers, authorize senders conservatively, route bounded inline requests through the agent flow, answer with Telegram `InlineQueryResultArticle` results, and add focused Telegram plugin tests. PR #51963 should be treated as useful reference material, not as a shipped fix.

## Risks / Open Questions

- Implementing inline queries needs explicit authorization, privacy, and session/conversation design because queries can be typed from arbitrary Telegram chats and do not map cleanly to the existing DM/group message flow.
- The existing `inlineQueries=` status flag may confuse operators because it reflects BotFather capability, not OpenClaw inline-query support.
