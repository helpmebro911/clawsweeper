## Summary

Keep #13440 open. Current main still treats session.store as a filesystem path and persists sessions through sessions.json load/save/update helpers plus a .lock file. The plugin SDK exposes those same helpers rather than a driver/adapter registration surface, and the related PostgreSQL/MongoDB work in the discussion is memory-backend prior art, not a shipped pluggable session-store backend.

## Best Possible Solution

Keep #13440 open until maintainers add a core SessionStoreAdapter contract or a stable plugin API that can own session persistence end to end. The likely implementation path is to keep the current file backend as the default, add driver-shaped session.store config/types/help/docs, route load/save/update/readUpdatedAt/session discovery through the abstraction, and add contract tests for the file adapter plus a mocked non-file adapter.
