## Summary

Keep #13440 open. Current main still models `session.store` as a string filesystem path and routes session load/save/update through path-based `sessions.json` helpers. There is no `SessionStoreAdapter`, driver-shaped config, backend registry, or plugin seam that would let Convex, SQLite, Redis, PostgreSQL, or custom modules own session persistence. The related memory-backend PRs and RFCs show demand for pluggable persistence, but they do not implement the session-store backend API requested here, and an existing maintainer/Codex issue comment already reached the same keep-open conclusion.

## Best Possible Solution

Keep #13440 open until maintainers either implement a core `SessionStoreAdapter` contract or intentionally add a plugin API that lets external backends own session persistence. A complete solution should update `session.store` schema/types/help/docs, preserve the current file backend as the default, route load/save/update/readUpdatedAt and discovery through the abstraction, and add contract tests for the file adapter plus at least one mocked non-file adapter path.
