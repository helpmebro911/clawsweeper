## Summary

Keep #13440 open. Current main still treats `session.store` as a filesystem path, resolves it to `sessions.json`, and routes load/save/update/readUpdatedAt through file-backed helpers and filesystem lock files. There is no driver-shaped config, `SessionStoreAdapter`, registry, or plugin hook that would let Redis, SQLite, Convex, PostgreSQL, or a custom module own session persistence. The existing plugin runtime only exposes the same file-backed helpers, and the related memory-backend items are adjacent prior art rather than a shipped session-store backend abstraction.

## Best Possible Solution

Keep #13440 open until maintainers either add a core `SessionStoreAdapter` contract or expose a stable plugin API that lets external backends own session persistence. The best implementation path is to keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/session discovery through the abstraction, and add contract tests for the file adapter plus a mocked non-file adapter.
