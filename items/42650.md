## Summary

Keep open. Current main has adjacent memory review surfaces, including short-term promotion preview/apply, Dream Diary review, grounded backfill/reset, Active Memory recall, and memory-wiki claim lint/apply, but it still lacks the requested first-class operator maintenance flows for review queues, edit, non-destructive forget/tombstone or superseded status, and conflict resolution over active memory records.

## Best Possible Solution

Keep this issue open and implement the missing Memory MVP maintenance layer in the memory owner modules. The likely path is a canonical record/status model plus operator-facing review/list, edit, non-destructive forget/tombstone or superseded transitions, and conflict-resolution flows, exposed through CLI/RPC or UI with tests and docs. Memory-wiki claim lint/apply can be reused where provenance-rich claims fit, but active memory still needs first-class maintenance semantics.
