## Summary

Keep this issue open. Current main has memory search, promotion, dreaming, and a companion memory-wiki layer, but it still lacks first-class operator flows for reviewing, editing, non-destructively forgetting, tombstoning/superseding, and resolving conflicts over canonical active-memory records. The provided discussion also shows same-author open paired Memory MVP issues #42646, #42648, and #42651, so this should remain open until that suite is implemented or explicitly superseded by maintainers.

## Best Possible Solution

Keep this issue open and implement it with the memory owner modules alongside #42646, #42648, and #42651. The likely product path is a canonical durable memory record/status model plus operator-facing review/list, edit, non-destructive forget/tombstone or superseded transitions, and conflict-resolution flows exposed through CLI/RPC or UI, with docs and tests. Memory-wiki should remain a provenance-rich companion layer unless maintainers explicitly decide it owns these active-memory maintenance semantics.

## Risks / Open Questions

- The eventual MVP should define a clean boundary between canonical active-memory records and the provenance-rich memory-wiki belief layer so wiki claims do not become the only source of truth for active memory maintenance.
- The existing `memory-lancedb` plugin has a `memory_forget` tool that physically deletes records, so a default-memory non-destructive forget flow should avoid implying identical semantics across memory plugins.
