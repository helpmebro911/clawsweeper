## Summary

Keep open. Current main has corrected several cron configuration boundaries: CLI-created main jobs must use `systemEvent`, service creation rejects `sessionTarget: "main"` with `agentTurn`, and persisted invalid main/agentTurn jobs are skipped with an explicit error. But the specific `ANNOUNCE_SKIP` confusion is not fully resolved for main-session cron prompts: exact assistant `ANNOUNCE_SKIP` replies are still globally suppressed from chat/history, and the cron docs do not warn users not to put that control token in main-session cron payloads.

## Best Possible Solution

Keep this issue open and narrow the fix to cron/control-token ergonomics. A conservative maintainer path would add targeted docs to the public cron guide and CLI reference warning that `ANNOUNCE_SKIP` is an internal exact-output control token, add a warning or validation diagnostic when a main-session `systemEvent` or cron agent prompt contains that token as an instruction, and cover it with a focused cron/service or CLI regression test. The existing `cron run <jobId>` manual trigger already covers much of the requested dry-run workflow, so the remaining product decision is whether to add a separate dry-run command or just document the manual-run/run-history workflow.
