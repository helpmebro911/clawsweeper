## Summary

Current main does not support closing this as implemented: cron main-session jobs enqueue the full systemEvent text and wake heartbeat, while ANNOUNCE_SKIP suppression is implemented as an exact assistant reply token elsewhere. The report’s proposed warning/docs/validation gap for cron payloads remains actionable, especially because cron docs explain main vs isolated jobs but do not call out ANNOUNCE_SKIP in cron troubleshooting.

## Best Possible Solution

Keep the issue open and address it narrowly in the cron surface: document that main-session cron payloads are system-event reminders interpreted by heartbeat, recommend isolated/current/session agentTurn jobs with announce delivery for report-style channel output, and add either validation or a warning when a main-session systemEvent payload contains an exact control-token instruction such as ANNOUNCE_SKIP. A focused regression test can stub heartbeat output to ANNOUNCE_SKIP to prove the warning/docs behavior without needing live Feishu.
