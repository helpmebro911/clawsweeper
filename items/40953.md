## Summary

Keep this PR open. Current main still has the gateway-facing CLI timeout validation gap the PR targets: several commands register raw string `--timeout` options and then pass `Number(opts.timeout ?? ...)` into gateway calls, so invalid values can still reach the gateway layer. The PR is stale against current main’s `gateway-rpc` wrapper/runtime split and overlaps with other open timeout-validation work, but it remains a plausible focused fix rather than obsolete. Security review found no unexplained supply-chain or secrets-sensitive changes in the provided diff.

## Best Possible Solution

Keep this PR open for maintainer review or contributor rebase. The best path is to adapt the strict shared timeout parser to current main, apply it to the gateway-facing call sites that still use `Number(...)`, add focused regression tests for invalid values and valid passthrough, and coordinate with #54646/#59397 so duplicate timeout fixes do not diverge.
