## Summary

Keep #70440 open. Current main has narrow suppression for already-committed messaging-tool output and group/chat silent-reply cases, but it still has no global or per-channel config option to silently swallow genuine empty-turn fallback errors for direct/TTS surfaces. The maintainer comment explicitly says this remains valid after the #70396/#70623 fallback fixes.

## Best Possible Solution

Keep the issue open and implement an explicit empty-turn delivery policy, likely under agent defaults with per-surface/channel override support. The shipped behavior should default to visible error text, allow opt-in silent delivery for TTS/voice-note channels, log the suppressed failure at WARN, and add focused tests covering direct Signal/TTS-style delivery, group silent-reply behavior, and the existing committed-messaging-tool suppression path.

## Risks / Open Questions

- A silent mode could hide real provider/config failures, so the default should remain the current visible error behavior and silent mode should keep WARN/server-log evidence.
- The implementation should not conflate intentional NO_REPLY/silent-reply policy with failure-delivery policy; direct/TTS empty-turn suppression needs its own explicit config and tests.
