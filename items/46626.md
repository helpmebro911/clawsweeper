## Summary

Current main does not implement the PR's all-path own-line NO_REPLY suppression. The cron delivery path has a partial later fix for trailing NO_REPLY, but normalizeReplyPayload, parseReplyDirectives, outbound payload planning, and Gateway live chat/history paths still use exact/envelope, leading-token, or strip-only behavior that can leave preceding text visible. The PR is still addressing a real remaining gap, though it needs rebase/rework because review comments identified merge blockers.

## Best Possible Solution

Keep the PR open or ask for a refreshed maintainer patch. The best implementation is a shared, tested silent-token classifier that distinguishes exact/envelope silence, own-line trailing silence, inline token stripping, media/channelData preservation, and streaming lead-fragment suppression, then uses it consistently in normalizeReplyPayload, parseReplyDirectives/outbound payload planning, cron follow-up/dispatch, Gateway live chat projection, and chat history sanitization.
