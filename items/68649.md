## Summary

Keep #68649 open. Current main still has the lower-level remote media idle-timeout machinery, but the PDF tool path does not pass a timeout into `loadWebMediaRaw`, and `loadWebMediaRaw`/`loadWebMediaInternal` do not expose or forward `readIdleTimeoutMs` to `fetchRemoteMedia`. The related PR #68747 proposed the narrow fix, but it was closed unmerged and the change is not present on current main.

## Best Possible Solution

Keep the issue open and land a narrow core fix: add `readIdleTimeoutMs` to the web-media options contract, forward it into `fetchRemoteMedia`, set a conservative default for remote PDF loads in `src/agents/tools/pdf-tool.ts`, and add a PDF-tool regression test that asserts remote PDF loads pass a positive idle timeout. Then split any broader `/new`/`/restart` queue bypass or session watchdog work into a separate focused issue if maintainers want that behavior.
