## Summary

Close as `clawhub`. The request is an optional observability/analytics pipeline, and current `main` explicitly keeps that kind of capability out of core and on plugin/external integration paths. OpenClaw already exposes the raw JSONL/session surfaces plus a plugin-based observability exporter, which makes a core JSONL→SQLite daemon a scope mismatch rather than an unimplemented core requirement.

## Close Comment

Closing this as a ClawSweeper/Codex scope review. The request is useful, but it fits the plugin/external-observability path rather than OpenClaw core. `VISION.md:54-59` is explicit that optional capability should usually ship as plugins and that the bar for adding this kind of feature to core is intentionally high. Current `main` already treats observability that way: `docs/logging.md:258-290` documents the `diagnostics-otel` plugin, and `extensions/diagnostics-otel/index.ts:1-10` implements that exporter plugin. Core also already exposes the raw data surfaces you want to ingest, including cache trace JSONL (`docs/gateway/configuration-reference.md:3592-3615`) and session JSONL (`docs/gateway/security/index.md:380-382`). On top of that, the issue discussion already points to a standalone implementation (`clawmetry`) built on those existing logs. Given the current architecture, this is better carried as a ClawHub/plugin or external tool, not as a new core JSONL→SQLite daemon.
