## Summary

Keep this PR open. Current main still has the slow status path: `doctor.memory.status` awaits `probeEmbeddingAvailability()`, and the built-in memory manager still probes by calling `embedBatchWithRetry(["ping"])` without a cache. The PR is a focused implementation candidate for #71568, with no obvious security-sensitive file changes, but it still deserves normal maintainer review around the test strategy and slow-provider behavior before landing.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #71568. The best path is to review the focused memory-core change, decide whether the regression test should exercise production code more directly, run the appropriate changed gate, and land it if the cache semantics are accepted.

## Risks / Open Questions

- The proposed test harness duplicates the cache algorithm instead of directly exercising `MemoryIndexManager`, so it may miss future drift unless the maintainer accepts that tradeoff or asks for a narrower extracted helper/direct class test.
- The PR intentionally caches failures as well as successes for 30 seconds; that is likely acceptable for status calls, but it can briefly hide provider recovery.
- The PR body says live slow-provider gateway verification was not performed, so final review should rely on targeted tests plus any feasible manual latency proof before merge.
