## Summary

Keep open. Current main fixed the specific unquoted-colon frontmatter case behind related issue #22134 via a tolerant frontmatter parser, but it still does not implement this PR's requested parse-error reporting contract: the skill/status types have no parseError field, CLI list/info/check have no parse-error state, and loader failures are not preserved as diagnostics. The open PR is also incomplete per the provided review discussion because it only covers part of the CLI surface.

## Best Possible Solution

Keep the PR open or supersede it with a focused maintainer issue/PR that preserves SKILL.md load diagnostics through the current local-loader path, adds a parseError-style field to the status contract, handles parse errors consistently in skills list, skills info, skills check, JSON output, and any UI/status consumers, and adds regression tests for malformed frontmatter plus the already-fixed unquoted-colon case.
