## Summary

Keep this PR open. Current main still has the gateway config lost-update race: `config.set`, `config.patch`, and `config.apply` each read a snapshot, validate the caller's `baseHash` against that snapshot, compute a new config, and call `writeConfigFile()` directly without serializing the full read/check/write window. The PR remains a relevant implementation candidate, but it needs rebase/review because current main now has SecretRef and shared-auth behavior that must be preserved, and the existing review comments identify real landing gaps around queue rejection responses and `config.set` race coverage.

## Best Possible Solution

Keep this PR open as a still-relevant implementation candidate. Rebase or reimplement the gateway config write serialization on current main, wrap the full read/base-hash-check/validate/write sequence for `config.set`, `config.patch`, and `config.apply`, preserve current SecretRef and shared-auth semantics, map command-lane rejection errors to explicit RPC failures, and add stale concurrent-write tests for all three methods before merge.
