## Summary

Keep #71482 open. The PR addresses a plausible data-loss path in current main: `loadSessionStore()` still returns `{}` for missing or malformed stores, and the dreaming scrubber still treats an empty reference set as authority before scanning and archiving stale dreaming transcripts. However, the submitted guard is over-broad, has no regression test, and both bot review comments correctly note that it would also suppress legitimate orphan cleanup for agents with a genuinely empty store. This needs a tighter fix, not cleanup closure.

## Best Possible Solution

Keep the PR open and ask for a narrower fix: give the scrubber a reliable way to tell load failure from a valid empty store, either by making the session-store load path expose/throw read-parse failures for this caller or by validating the store file separately before destructive cleanup. Add focused tests for malformed/unreadable `sessions.json` with old active-looking dreaming transcripts, and for a genuinely empty store where stale dreaming orphans should still archive.
