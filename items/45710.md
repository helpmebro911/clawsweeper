## Summary

Keep this PR open. Current main still lacks the VS Code debugging support proposed here, and the PR is a focused developer-experience change rather than something that fits a close reason. However, the PR should not be merged as-is without maintainer review because the clean-and-run debug path can remove build artifacts that `scripts/run-node.mjs` does not fully recreate, and the late VS Code launch-config concern still deserves direct validation.

## Best Possible Solution

Keep the PR open and ask for a maintainer-quality revision: preserve the useful `OUTPUT_SOURCE_MAPS`/docs idea, but change the launch flow so it does not wipe full-build artifacts before a partial `run-node` rebuild. A safer path is likely a source-map-enabled debug launch that either requires an existing full build, runs an explicit full build when needed, or forces only the relevant tsdown rebuild while preserving `dist` assets. The VS Code `pnpm` launch config should also be validated or adjusted to the documented `runtimeExecutable`/`runtimeArgs` pattern before merge.
