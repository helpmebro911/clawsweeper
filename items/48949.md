## Summary

Keep open. Current main still routes Feishu REST/token calls through a wrapper around the Lark SDK default axios HTTP instance without disabling ambient proxy use or adding Feishu-specific NO_PROXY handling, so the reported tenant_access_token proxy failure is still plausible on the reviewed SHA.

## Evidence

- **Feishu REST client path still delegates to SDK default HTTP instance:** createTimeoutHttpInstance wraps feishuClientSdk.defaultHttpInstance, and createFeishuClient passes that wrapper as Lark.Client httpInstance. There is timeout injection, but no proxy:false, NO_PROXY matcher, or Feishu/Lark domain bypass on this REST/token path.

Current main does not implement either on the REST client path.
