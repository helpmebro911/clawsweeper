## Summary

Keep this PR open. Current main still lacks per-conversation Active Memory filters, so the feature is not implemented or obsolete. The PR is a focused plugin-owned privacy/control improvement, but the submitted parser still needs revision for supported session-key shapes and threaded sessions before it is safe to merge.

## Best Possible Solution

Keep this PR open and request a focused revision. The implementation should resolve the base conversation ID through existing session-key utilities or a shared plugin-safe helper, strip generic thread suffixes before matching, support all current direct/group/channel key shapes, and add regression tests for per-peer, per-channel-peer, per-account-channel-peer, group/channel, and threaded cases. After that, the manifest/docs additions can land as a plugin-owned Active Memory feature.

## Risks / Open Questions

- Merging as-is could make `allowedChatIds` and `deniedChatIds` unreliable for supported direct-message scopes, including per-peer, per-channel-peer, and per-account-channel-peer sessions.
- Threaded channel sessions could miss allowlist entries or bypass denylist entries if the matcher treats `:thread:<id>` as part of the conversation ID.
- Closing this PR would discard a useful Active Memory privacy/control feature that is still absent from current main.
