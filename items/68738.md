## Summary

Current main still matches the reported failure mode: `readPageSummaries` fans out all page reads with an unbounded `Promise.all`, and `compileMemoryWikiVault` may call it up to three times in one compile. There is no shipped compile/read concurrency setting or regression coverage, so this remains actionable in `memory-wiki`.

## Best Possible Solution

Keep the issue open and fix it in `extensions/memory-wiki/src/compile.ts` by reading page summaries through a bounded async map or small limiter with a conservative default, then add a focused regression test that instruments concurrent reads during `compileMemoryWikiVault`. If maintainers want user control, add a documented `plugins.entries.memory-wiki.config.compile` concurrency setting across the TypeScript schema, plugin manifest, and memory-wiki docs.

## Risks / Open Questions

- The iCloud/FileProvider `EDEADLK` symptom is platform-specific, so CI should verify the concurrency ceiling directly rather than try to reproduce the filesystem deadlock.
- If the fix exposes a user setting, config schema, manifest metadata, docs, and defaults need to stay aligned; a fixed conservative internal default may be lower-risk.
