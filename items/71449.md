## Summary

Keep #71449 open. Current main still lacks the PR's central startup UX changes: `ready (...)` is not qualified as gateway-listening readiness, there is no post-sidecar `sidecars settled` channel-health breakdown, Telegram polling still retries `deleteWebhook` before polling on recoverable failures, and Feishu startup probing still only treats timeout/abort specially. Related #67921 covers bounded channel startup, while #56369/#70728 remain broader readiness follow-ups rather than shipped superseders for this PR.

## Best Possible Solution

Keep #71449 open for maintainer review. The strongest path is to preserve the legacy `ready (` marker while clarifying that it means the gateway is listening, add a deterministic `sidecars settled` summary using the existing channel runtime snapshot and health-policy machinery, and review the Telegram/Feishu changes as channel-owned startup hardening tied to #70728 rather than closing this as implemented or duplicate.
