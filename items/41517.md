## Summary

Keep this PR open. Current main has some model-ref normalization in session/config paths, but the embedded resolver still does not handle the PR’s direct case where it receives `provider="google"` and `modelId="google/gemini-2.5-pro"`; that can still miss a bare configured Google model and fall back with the prefixed ID. The PR is relevant, but it should be rebased/reviewed carefully with nested-ID regression coverage before merge.

## Best Possible Solution

Keep this PR open, but request a current-main revision. The best fix is a narrow resolver-level normalization that strips a duplicated same-provider prefix only when the provider contract/configured or discovered model metadata supports the bare id, with regression tests for Google same-provider inline resolution, OpenRouter cross-provider pass-through, and nested provider-native ids.
