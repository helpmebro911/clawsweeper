## Summary

Current main still has the reported ambiguity: after the target `agent` RPC accepts a run, `sessions_send` waits for a reply and returns a plain `status: "timeout"` if `agent.wait` times out, without delivery/acceptance metadata or a distinct reply-wait-timeout result.

## Best Possible Solution

Keep this issue open and implement a backward-compatible clarification in the `sessions_send` result contract. After the target `agent` RPC has accepted a run, a later wait timeout should include explicit acceptance/delivery metadata, such as `status: "timeout"` plus `deliveryStatus: "accepted" | "delivered"` and `replyStatus: "timeout"`, or another additive pending/accepted shape that preserves old callers. Add focused regression coverage for accepted `agent` followed by `agent.wait` timeout, and update the session-tool docs to distinguish send failure, accepted delivery, and inline reply timeout.

## Risks / Open Questions

- Changing the top-level `status` value could break existing callers that branch on `timeout`; an additive contract such as `deliveryStatus`, `replyStatus`, or `accepted: true` is safer.
- The fix should only mark post-acceptance wait timeouts as accepted/delivered. Initial `agent` RPC failures before a `runId` is accepted should remain send/start errors.
