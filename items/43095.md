## Summary

Current main still has the reported ambiguity. `sessions_send` can successfully accept/start the target agent run, then return only `status: "timeout"` when the later `agent.wait` window expires, with no delivery/acceptance or reply-wait metadata to distinguish that from a failed send. The related closed report #53527 points back to this remaining contract issue rather than superseding it.

## Best Possible Solution

Keep this issue open and implement a backward-compatible result-contract clarification in `sessions_send`: after the target `agent` RPC accepts a run, a later wait timeout should include explicit accepted/delivery and reply-wait metadata, while preserving pre-accept failures as errors. Add focused regression coverage for accepted run plus `agent.wait` timeout, and update the public session-tool docs to distinguish delivery failure, accepted delivery, reply wait timeout, and success.

## Risks / Open Questions

- Changing the top-level `status` away from `timeout` could break existing callers; an additive contract such as `delivery: { status: "accepted" }` plus `replyStatus: "timeout"` is safer.
- Any fix should only mark post-acceptance wait timeouts as accepted/delivered. Failures before the `agent` RPC accepts a run should remain send/start errors.
