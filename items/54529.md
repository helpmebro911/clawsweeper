## Summary

Keep this issue open. Current main has Telegram `apiRoot` passthrough, but it still does not scope persisted polling offsets by Bot API endpoint and still treats a Telegram 400 `Logged out` response from `deleteWebhook` as a fatal setup error rather than a local Bot API migration/cooldown condition.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin: store a normalized `apiRoot` or endpoint fingerprint with update-offset state, ignore/reset stored offsets when the endpoint changes, narrowly handle Telegram 400 `Logged out` during local Bot API migration/cooldown without crash-looping, and add focused regression tests plus Telegram docs for local Bot API migration and offset reset behavior.

## Risks / Open Questions

- Closing now would leave a concrete local Bot API migration failure where valid local-server updates can be ignored because the stored official-API update watermark is higher.
- The fix needs to preserve replay protection for normal restarts against the same endpoint while resetting or ignoring offsets only when the Bot API endpoint identity changes.
- `Logged out` handling should be narrow so unrelated Telegram 400 authentication or configuration errors are still surfaced clearly.
