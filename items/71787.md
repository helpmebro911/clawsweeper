## Summary

Keep #71787 open. Current main has the narrower Feishu Schema 2.0 `context.open_chat_id` fix, but it still does not normalize nested Schema 2.0 operator identity from `operator.user_id.{open_id,user_id,union_id}`. The PR remains useful and is not superseded by #71692 because the provided maintainer discussion explicitly called #71787 the better base for #71670.

## Best Possible Solution

Keep #71787 open and land the nested Feishu Schema 2.0 identity normalization on top of current main: extract `operator.user_id.{open_id,user_id,union_id}` and equivalent event-level nested identity fields, preserve existing `context.open_chat_id` handling and malformed-payload guards, keep the lifecycle regression test for the nested callback shape, and update the changelog attribution without duplicating the already-shipped #71670 wording.

## Evidence

It does not dereference object-valued `operator.user_id.open_id`, `operator.user_id.user_id`, or `operator.user_id.union_id`.
