## Summary

Keep open. Current main only implements the narrower Feishu Schema 2.0 `context.open_chat_id` fallback; it still does not extract nested Schema 2.0 operator identity from `operator.user_id.{open_id,user_id,union_id}`, so this PR remains useful. The provided PR discussion also contains maintainer signal that #71787 is the better base for #71670 compared with the narrower #71692.

## Best Possible Solution

Keep #71787 open and adapt it on top of current main: add nested Feishu Schema 2.0 identity extraction for `operator.user_id.{open_id,user_id,union_id}`, preserve the existing `context.open_chat_id` handling and malformed-payload guards, keep or add lifecycle coverage for the nested identity callback, and update the changelog without duplicating the existing #71670 entry.

## Evidence

- **Current parser misses nested operator identity:** `parseFeishuCardActionEventPayload` reads `openId` from `operator.open_id`, `value.open_id`, or `context.open_id`, and `userId` from `operator.user_id`, `value.user_id`, or `context.user_id`. It never dereferences `operator.user_id.open_id`, `operator.user_id.user_id`, or `operator.user_id.union_id`. With the PR's nested Schema 2.0 shape and no `context.open_id`, `openId` remains missing and the parser returns null.
