## Summary

Keep #70531 open. Current main still stores cron job definitions in one `jobs.json` array and does not discover or persist individual per-job definition files under a `cron/jobs/` directory. Recent/current code does split runtime state into `jobs-state.json` and per-job run logs, which helps reduce runtime churn, but it does not satisfy the requested per-job config-file layout, migration, or CLI/gateway save behavior.

## Best Possible Solution

Keep this open as a core cron storage enhancement. If maintainers accept the product direction, implement a versioned cron store that can load legacy `jobs.json`, discover deterministic per-job definition files under `~/.openclaw/cron/jobs/`, save add/edit/delete operations atomically per job, migrate or fall back safely, and update doctor/config docs plus cron store/service tests.

## Risks / Open Questions

- The current state split partially addresses runtime churn, so any implementation should preserve the `jobs-state.json` and per-job run-log behavior or define a deliberate migration path.
- Changing cron storage is user-data-sensitive: filename mapping, deterministic load order, atomic writes/deletes, fallback from `jobs.json`, doctor repair, docs, and gateway/CLI/UI tests would all need to move together.
