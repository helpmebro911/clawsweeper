## Summary

Current main still does not emit `llm_output` per LLM call, and there is no `llm_call` hook. The embedded runner records assistant usage per assistant message, accumulates it into attempt totals, and emits a single aggregate `llm_output` after the attempt. The related open PR #45990 remains an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and either land or replace #45990 with an additive, documented per-call observability contract: either a new `llm_call` hook or a backwards-compatible `llm_output` expansion that preserves aggregate semantics while exposing per-call usage, provider/model, run/session/agent ids, optional latency, and tool-call data. The implementation should update hook types, embedded subscription/attempt threading, plugin docs, and focused multi-round agentic-loop tests.
