## Summary

Keep #71195 open. Current main already has the backend/browser OpenAI Realtime Talk path and the voice-call realtime bridge/tooling, but native macOS Talk still uses the older SFSpeechRecognizer -> chat.send -> chat.history polling -> talk.speak/TTS playback loop. The useful remaining scope is the maintainer-narrowed macOS client integration with the existing `talk.realtime.session` flow, not a new parallel gateway voice stack.

## Best Possible Solution

Keep this issue open as the native macOS Realtime Talk tracker. The best implementation path is to add a Swift `GatewayConnection` wrapper for `talk.realtime.session`, a native WebRTC/AVAudioEngine realtime session modeled on `ui/src/ui/chat/realtime-talk.ts`, Talk overlay phase/cancel integration, and a clean fallback to the existing SFSpeechRecognizer -> chat.send -> talk.speak path. Split any remaining ElevenLabs repeated PCM 403 behavior into a separate focused Mac Talk TTS issue.
