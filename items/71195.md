## Summary

Keep #71195 open. Current main already has the backend/browser OpenAI Realtime Talk path, but the native macOS Talk client still uses the classic speech recognition -> chat -> `talk.speak` playback chain and has no `talk.realtime.session` client flow. The issue discussion explicitly narrows the remaining work to macOS client integration against the existing realtime Talk session surface.

## Best Possible Solution

Keep #71195 open and scope it to native macOS Talk consuming the existing `talk.realtime.session` realtime Talk flow: add a `GatewayConnection` wrapper, a Swift realtime audio/WebRTC session, overlay/runtime phase and cancellation wiring, and keep the current STT -> chat -> `talk.speak` path as fallback. Split the ElevenLabs repeated PCM 403 behavior into a separate focused Mac Talk TTS issue if it still reproduces after configuring `talk.providers.elevenlabs.outputFormat`.
