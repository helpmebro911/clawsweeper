## Summary

Keep this PR open. Current main still removes sandbox runtimes and registry entries without deleting local sandbox workspace directories, so the linked disk-leak bug remains unresolved. The PR is a relevant implementation attempt, but it is stale against current backend-aware prune code and still needs safer deletion semantics plus regression tests before merge.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The best fix is a backend-aware, container-prune-only cleanup that uses stored workspace identity and the original workspace root, skips shared or still-active shared-scope cases, deletes only after runtime removal is known safe, and adds regression tests for session, agent, shared, browser, backend, duplicate-entry, and failure cases. If a safer patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- The PR adds recursive workspace deletion, so an incomplete guard can delete a workspace still used by another active sandbox, browser runtime, session, or agent-scoped runtime.
- The PR is merge-dirty and was based before current backend-aware prune behavior; it needs a rebase/update for Docker, SSH, and OpenShell/plugin backend semantics.
- No current regression test covers workspace cleanup during prune, including shared scope, agent scope, duplicate session entries, browser pruning, backend failures, or stale workspace roots.
