## Summary

Keep this PR open. Current main still prunes sandbox runtimes and registry entries without removing Docker sandbox workspace directories, and the linked bug remains open. The PR is a relevant implementation attempt, but its current diff is stale against backend-aware prune code and still needs a safer deletion contract plus regression tests.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The safest path is a backend-aware container-prune-only cleanup that uses stored workspace identity, skips shared and still-active shared-scope cases, only deletes after a clear runtime-removal success contract, and adds regression tests for session, agent, shared, browser, backend, and failure cases. If a better patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- The current PR could delete the wrong workspace if it recomputes a stored scope key as though it were a raw session key.
- Workspace deletion must be guarded against any remaining container, browser, or backend runtime using the same workspace.
- Deleting after a swallowed Docker removal failure can remove workspace data while the runtime still exists.
- The PR is stale against current backend-aware prune code and likely needs a rebase or replacement patch.
