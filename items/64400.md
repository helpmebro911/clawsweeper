## Summary

Keep this issue open. Current main still uses the heavyweight gateway `health` RPC as doctor’s Phase A liveness gate with a 3000ms non-interactive budget, and a failed gate can still fall through to the running-service restart prompt. The related PR #64511 was cited in discussion as the intended fix, but the provided GitHub context shows it was closed unmerged, and the current source does not contain that change.

## Best Possible Solution

Keep this issue open. Implement the narrow core fix in the doctor/gateway health path: use a lightweight liveness signal such as the gateway `status` RPC before offering daemon restart repair, classify accepted local probes as reachable/degraded when only follow-up detail RPCs time out, and add regression tests around slow `health` and the restart path. Treat any unauthenticated loopback auth relaxation as a separate maintainer security review item.
