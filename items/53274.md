## Summary

Keep open. Current main still skips browser device identity whenever `crypto.subtle` is unavailable and still fingerprints the browser public key with WebCrypto, so the primary LAN HTTP/reverse-proxy request is not implemented. The secondary `dangerouslyDisableDeviceAuth` scope-bypass and operator scope-normalization pieces are implemented and covered, but they do not solve the Secure Context dependency.

## Best Possible Solution

Keep this issue open unless a maintainer explicitly consolidates it into the older open tracker #32473. The concrete implementation path is to decide whether OpenClaw should support browser device identity over non-secure HTTP, then remove the `crypto.subtle` guard, replace the WebCrypto hash dependency with an approved pure-JS hash path or direct dependency, and update gateway policy tests plus Control UI/security docs to match. A refreshed version of the closed, unmerged PR #53276 could be the starting point, while preserving the already-fixed bypass and scope-normalization behavior.
