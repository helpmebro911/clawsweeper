## Summary

Current main still matches the central report: the browser client only creates device identity when `crypto.subtle` exists, and browser fingerprinting still calls WebCrypto. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope requests are already implemented, but the Secure Context/LAN HTTP device-identity behavior remains unresolved.

## Best Possible Solution

Keep this issue open. Maintainers should decide the product/security policy for non-secure LAN browser device identity; if supported, update `ui/src/ui/gateway.ts` to attempt identity without the `crypto.subtle` gate, replace the WebCrypto-only fingerprint path in `ui/src/ui/device-identity.ts` with an approved browser-safe hash implementation, and refresh Control UI tests plus public docs. The already-shipped bypass and scope-normalization fixes should remain intact.
