## Summary

Keep open. Current main still does not resolve iMessage inbound reply/quote context from chat.db or GUID-only imsg payloads; it only forwards reply context when imsg already supplies resolved `reply_to_*` fields. The PR is stale because it targets the removed `src/imessage` tree, but it remains a focused implementation candidate for the open iMessage reply/thread-context gap tracked by #10872.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate. The maintainer path is to rebase/adapt the idea into `extensions/imessage/src/monitor`, resolve GUID-only iMessage replies into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` context fields before dispatch, preserve local-only remoteHost protection, keep chat.db reads opt-in and read-only, prefer or validate `thread_originator_guid`, and add regression coverage for GUID-only replies plus permission/schema failure cases.

## Risks / Open Questions

- This PR is not merge-ready as-is because it targets the removed `src/imessage` tree instead of the current `extensions/imessage` plugin layout.
- An adapted implementation reads local macOS Messages `chat.db`; it must stay opt-in through configured `dbPath`, read-only, local-runtime gated, and fail closed on permissions, locking, schema mismatch, or remote SSH-wrapper sessions.
- The related issue says `reply_to_guid` can be unreliable and that `thread_originator_guid` better matches the iMessage UI, so the final resolver should prefer or validate `thread_originator_guid` when available.
- The old helper imports core internals from the former core iMessage tree; any current implementation must respect the bundled plugin boundary and use plugin-safe SDK or local extension seams.
