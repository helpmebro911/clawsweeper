## Summary

Current main still does not resolve iMessage inbound reply/quote context from chat.db or GUID-only imsg payloads. This PR is stale against the current plugin layout but remains a useful implementation candidate rather than obsolete.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate. The best path is to rebase/adapt it into extensions/imessage/src/monitor, resolve reliable GUID-only inline replies into the existing ReplyToId/ReplyToBody/ReplyToSender context fields, preserve the local-only remoteHost guard, avoid plugin boundary violations, and add regression coverage for GUID-only replies plus permission/schema failure cases.

## Risks / Open Questions

- The PR is not merge-ready against current main because iMessage now lives under extensions/imessage, while the diff targets the removed src/imessage tree.
- Adapting the helper into the plugin must respect the extension boundary; direct imports from core internals such as src/memory or src/utils would need a plugin-safe seam or local implementation.
- The linked #10872 analysis says reply_to_guid can be unreliable; the final implementation should prefer or validate thread_originator_guid/current imsg fields before relying on reply_to_guid.
- Reading chat.db touches local Messages content, so the final fix should stay opt-in through configured dbPath, read-only, local-runtime gated, and tolerant of permission/schema failures.
