## Summary

Keep PR #39350 open. Current main still allows a run that ends after buffering only a partial NO_REPLY lead fragment to emit that fragment as a final chat message, but the PR should not be closed as obsolete or merged unchanged because its proposed final-path prefix guard would also suppress a legitimate uppercase "NO" assistant reply.

## Best Possible Solution

Keep the PR open and request a revised gateway final-path fix. The best direction is to carry provenance that the buffered text came from a suppressed control-token lead fragment during streaming, then suppress only that final payload while preserving intentional uppercase "NO" replies. Add regression coverage for lifecycle end after only "NO" or "NO_" is buffered, plus a case proving an intentional final uppercase "NO" remains visible.
