## Summary

Current main still does not implement OS-level TCP keepalive for the underlying WhatsApp WebSocket. The WhatsApp plugin constructs a Baileys socket with auth/version/logger/proxy options, and the existing watchdog only detects or recovers stale activity; it does not keep the TCP NAT entry alive. The issue is recent, has a same-problem comment, and the related #61788 report remains open rather than superseding it.

## Best Possible Solution

Keep this issue open and implement the mitigation in the WhatsApp plugin socket setup, likely near `extensions/whatsapp/src/session.ts` or a small WhatsApp-local helper. The fix should enable TCP keepalive on the underlying Node socket with a conservative initial delay such as 15000 ms, preserve proxy-agent support, and add a focused unit test with a mocked Baileys/ws socket proving the keepalive call is applied.
