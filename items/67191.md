## Summary

Current main still has the reported BOOT.md startup delivery gap. The boot-md hook runs BOOT.md through `runBootOnce`, and `runBootOnce` still starts an agent turn with `deliver: false` and no channel, target, `to`, `replyTo`, or runContext. The runtime prompt now tells BOOT.md to use `channel + target`, so a hard-coded destination may work, but OpenClaw still does not provide the default/current startup delivery route described by the issue and follow-up Telegram reproduction.

## Best Possible Solution

Keep this issue open. The best maintainer path is to define the boot-md delivery contract: either pass a well-defined startup delivery route into the boot run/message-tool context, or explicitly document and validate that BOOT.md sends must provide `channel` and `target`. Preserve `NO_REPLY` as final assistant reply suppression, and add a regression test proving the chosen gateway-startup BOOT.md message-tool behavior.
