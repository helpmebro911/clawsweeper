## Summary

Current main still does not resolve the reported BOOT.md startup delivery problem. The boot-md hook still runs BOOT.md through `runBootOnce`, `runBootOnce` still calls `agentCommand` with `deliver: false` and a no-op runtime log, and the existing unit test still asserts that suppressed final-delivery behavior. The docs/prompt direct BOOT.md to use the message tool followed by `NO_REPLY`, so the fix needs to make startup message-tool delivery reliable without simply leaking or duplicating final assistant replies.

## Best Possible Solution

Keep this issue open. The best fix is a focused change in the boot-md/gateway boot path that makes BOOT.md message-tool sends reliably reach an intended channel/target while preserving final `NO_REPLY` suppression, backed by a regression test covering gateway startup and the configured delivery route.

## Risks / Open Questions

- Changing `deliver` to true by itself could deliver final assistant text in addition to explicit message-tool sends, or surface `NO_REPLY` behavior incorrectly. The fix should separate final-reply suppression from explicit startup message-tool delivery.
- The desired default delivery target for boot sessions still needs a maintainer decision: persisted main-session delivery context, explicit BOOT.md `channel`/`target`, or a new boot-specific config/API.
