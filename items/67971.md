## Summary

Current main still matches the reported Discord startup bug. `fetchDiscordApplicationId()` falls back to the token-derived ID only when the lookup throws; a non-OK HTTP response such as 429 still returns `undefined`, and the Discord provider still converts that into `Failed to resolve Discord application id`. The earlier merged PR #29695 is related but narrower: it covered transport/timeout failures, not transient non-OK HTTP responses.

## Best Possible Solution

Keep this issue open and implement a narrow Discord plugin fix in `extensions/discord/src/probe.ts`: classify application-id lookup non-OK responses so 401/403 fail fast, while transient statuses such as 429 and likely 5xx use the existing `parseApplicationIdFromToken()` fallback. Add focused Vitest coverage for 429/5xx fallback and 401/403 hard-failure behavior; no core change appears necessary.
