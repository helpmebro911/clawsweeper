## Summary

Keep this issue open. Current main at 646a268d2710bc54e478349ef5690f3c8803df4f still exposes only the existing `messages.queue` modes and timer/debounce-based followup drain behavior; there is no operator-selectable readiness-aware `until_ready` mode or equivalent public queue option. The narrower subagent announce queue has a busy-parent defer hook and 15s safety cap, but that is internal announce delivery behavior, not the requested public chat queue mode.

## Best Possible Solution

Keep this open as core auto-reply queue work. Add an operator-selectable readiness-aware `messages.queue` mode or option, wire it through config types/schema/generated help, `/queue`, command metadata, docs, and focused tests, then implement a generic drain condition tied to real session/run readiness with a clear starvation cap. The internal subagent announce deferral should be considered prior art, not the complete fix.
