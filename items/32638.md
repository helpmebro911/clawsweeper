## Summary

Keep #32638 open. Current main still cannot reliably emit provider-native reasoning values for Groq or LM Studio/Qwen-style OpenAI-compatible providers. The later fallback-loop concern appears partly improved because unknown errors can now advance to remaining fallback candidates, but the root request-shaping bug remains: Groq `default` and LM Studio `on`/`off` are not preserved as valid outbound reasoning values.

## Best Possible Solution

Keep the issue open and fix reasoning compatibility at the provider/model contract level. The maintainer path should preserve provider-native allowed reasoning values, allow typed request-value mapping outside the OpenAI effort enum, apply Groq `none`/`default` handling to all affected Groq reasoning models, wire LM Studio/Qwen `on`/`off` discovery into request shaping, and add focused regression tests for Groq and LM Studio OpenAI-compatible completions.
