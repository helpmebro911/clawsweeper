## Summary

Keep this issue open. Current main still emits the reported TUI pairing guidance without the concrete `openclaw devices approve <requestId>` command, and the TUI regression test only asserts `openclaw devices list`. Other current surfaces already document and display the approval command, so this remains a narrow TUI copy and test cleanup rather than an implemented fix.

## Best Possible Solution

Update `src/tui/tui.ts` so `resolveGatewayDisconnectState` includes the concrete approval command after the list command. Prefer using `readConnectPairingRequiredMessage` from `src/gateway/protocol/connect-error-details.ts` to extract a sanitized request ID when the close reason includes one, emitting `openclaw devices approve <actual-request-id>` in that case and `openclaw devices approve <requestId>` as the fallback. Tighten `src/tui/tui.test.ts` to assert both the list command and approval command, including request-id parsing if added.
