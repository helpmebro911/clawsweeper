## Summary

Keep #71697 open. Current main still documents and routes cloud-only Ollama through the native `api: "ollama"` `/api/chat` path, but the native Ollama provider does not add a replay policy or converter normalization for strict user/assistant alternation. The prior ClawSweeper comment remains consistent with current SHA 4a3030df9efa72c44bab567a4390ac0c4876a73f.

## Best Possible Solution

Keep the issue open and fix the native Ollama provider path. The likely maintainer path is an Ollama-owned replay policy or converter-level normalization for `api: "ollama"` that reshapes non-alternating Gateway/agent replay into Ollama Cloud-compatible `/api/chat` messages while preserving native Ollama tool-call behavior, plus regression tests covering the non-alternating replay history reported here.
