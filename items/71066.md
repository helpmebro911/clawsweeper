## Summary

Keep this issue open. Current main has adjacent Telegram polling hardening, including the #71502/#69064 watchdog fix, but the reported sticky IPv4 fallback path still exists and the maintainer comments already identified this as a separate network/region-specific failure that needs current-main affected-host evidence before closing.

## Best Possible Solution

Keep this issue open and request a current-main retest on the affected host with `OPENCLAW_DIAG=telegram` through one full stall/rebuild cycle. If an in-gateway probe using `resolveTelegramFetch` fails, focus on `extensions/telegram/src/fetch.ts` fallback diagnostics, dispatcher lifecycle, and error-chain logging; if that probe succeeds while gateway polling still wedges, continue in `extensions/telegram/src/polling-session.ts` and runner ownership/restart behavior.
