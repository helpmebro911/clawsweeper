## Summary

Keep this issue open. Current main contains related Telegram polling watchdog and transport-rebuild hardening, including the #71502 pending-runner fix, but the exact sticky IPv4 fallback path and warning reported here still exist and there is no affected-host current-main retest showing the India/macOS fallback loop is gone.

## Best Possible Solution

Keep this issue open and ask for a current-main retest on the affected host with Telegram diagnostics around one full stall/rebuild cycle. If an in-gateway probe using `resolveTelegramFetch` fails, focus the fix in `extensions/telegram/src/fetch.ts` around fallback diagnostics, dispatcher lifecycle, and error-chain logging. If that probe succeeds while gateway polling still wedges, continue in `extensions/telegram/src/polling-session.ts` around runner ownership and restart behavior, using #71502 as the baseline.
