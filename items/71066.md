## Summary

Keep #71066 open. Current main has several Telegram polling hardening fixes, including the #71502 runner-watchdog fix, but the issue is recent, region/network-specific, and maintainer comments explicitly say the sticky IPv4 fallback loop remains a separate open question that needs current-main diagnostic logs from the affected host.

## Best Possible Solution

Keep the issue open and use current main after PR #71502 as the retest baseline. The next maintainer action should be to collect OPENCLAW_DIAG=telegram logs from the affected host around a full stall/rebuild cycle and compare a gateway-process resolveTelegramFetch getMe/getUpdates probe with actual polling behavior. If the probe succeeds while polling wedges, fix the polling-session/runner ownership path; if it fails with the same codes, improve the Telegram fetch transport diagnostics and fallback behavior.
