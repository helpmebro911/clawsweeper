## Summary

Do not close #71066. The current checkout still contains the exact Telegram sticky IPv4 fallback path and warning described in the report, while the related merged work only covers a generic wedged-runner watchdog hole. The issue is recent and the provided maintainer comments explicitly leave it open pending current-main diagnostics from the affected host.

## Best Possible Solution

Keep #71066 open. Retest the affected host against current main after #71502, collect focused Telegram diagnostic logs around a full fallback/stall/rebuild cycle, and compare gateway polling with a small probe that uses OpenClaw's `resolveTelegramFetch`. If it still reproduces, the likely fix belongs in `extensions/telegram/src/fetch.ts` and/or `extensions/telegram/src/polling-session.ts`, with clearer dispatcher construction/reuse logging and full error-chain diagnostics.
