## Summary

Keep this PR open. Current main still resolves sandbox Docker `pidsLimit` only from agent/global config with no default, while the Docker create path already applies `--pids-limit` whenever a positive value is present. The PR is a narrow implementation candidate for the still-open same-author security issue it closes, so conservative cleanup should not close either half of the pair before maintainer review or merge.

## Best Possible Solution

Keep the PR open as the implementation candidate for the linked security issue. The best maintainer path is to review the product/security choice of a default `1024` PID cap, add focused coverage for unset/global/agent override behavior, update the config docs or JSDoc so operators know how the default works, then merge or supersede it with an equivalent patch that closes #38604.
