## Summary

Keep open. Current main preserves explicit empty plugin scope semantics, but it does not implement this PR's O(1) fast-path: `loadOpenClawPlugins` still reaches plugin discovery and manifest registry loading before scoped filtering. The related merged scope-handling work covers correctness, not this performance optimization. The PR should be clarified because current CLI activation-planner no-match behavior intentionally omits `onlyPluginIds` instead of passing `[]`.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main, clarify the intended caller behavior, and add focused loader coverage proving that explicit empty scopes return `createEmptyPluginRegistry()` without discovery or manifest loading while preserving the existing reentry and activation side effects. If maintainers want no-match CLI planning to load zero plugins, update `src/plugins/cli-registry-loader.ts` and its no-match test explicitly rather than relying on the loader change alone.
