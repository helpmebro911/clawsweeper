## Summary

Keep open. Current main has memory embedding configuration and even a latent `dreaming.execution.*.model` resolver shape, but memory-core does not pass any provider/model override into Dream Diary subagent runs, the plugin manifest does not accept a Dreaming model/execution key, and user-facing docs still say the diary uses the default runtime model. magejosh's comment is directionally right that most consolidation is embedding/heuristic work, but the remaining LLM-backed diary subagent still inherits the agent model.

## Best Possible Solution

Keep the issue open. The best path is to make one documented memory-core Dreaming model override official, probably by wiring the existing `dreaming.execution.defaults.model` or a simpler `dreaming.model` alias into `generateAndAppendDreamNarrative`, parsing provider/model refs safely, using the existing trusted plugin subagent override gate, preserving default inheritance when unset, and updating the memory-core manifest schema, Dreaming docs, memory config reference, and focused tests. Also document that core promotion/ranking remains heuristic/embedding-backed and memory embedding provider/model selection stays under `agents.defaults.memorySearch`.

## Risks / Open Questions

- The issue's call-count/cost framing appears to conflate heuristic/embedding-backed Dreaming phases with the LLM-backed Dream Diary narrative; the implementation should scope model selection to the actual subagent calls unless maintainers intentionally add LLM-backed phase execution.
- There is a partially normalized but undocumented `dreaming.execution.*.model` shape; making that official without aligning manifest validation, docs, and runtime wiring would leave confusing dead config.
