## Summary

Keep #13601 open. Current main has partial shipped cron history support: finished runs are persisted to per-job JSONL files, `cron.runs` can page/filter by status/query/delivery in the Gateway protocol, and the Control UI exposes run-history filters. The issue’s core CLI workflow and several acceptance criteria are still missing: no `cron logs`/`cron history` command, no CLI status/time-range filters, no protocol time-window filter, no success-rate/list aggregates, and no manual prune/vacuum or retention-days policy. The existing issue comment from steipete already reached the same keep-open conclusion, and current source still supports it.

## Best Possible Solution

Keep #13601 open until core cron history is completed or maintainers deliberately narrow the scope. The likely path is to extend `cron.runs` and `readCronRunLogEntriesPage` with time-window filters, expose user-facing CLI flags or a `cron logs/history` command, add success-rate/list aggregation, and either implement manual history pruning/retention-days or document a deliberate config-only retention decision with tests and docs.
