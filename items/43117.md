## Summary

Keep this issue open. Current main implements the partial path already discussed: Gemini Embedding 2, outputDimensionality, and structured image/audio embedding for memory-file indexing. The requested conversation-history attachment workflow is still missing because session transcript indexing flattens message text and ignores persisted MediaPath/MediaPaths, and memory multimodal support still does not cover video or PDF.

## Best Possible Solution

Keep this issue open and narrow it to the missing session-memory attachment path: extend session transcript indexing to collect eligible persisted MediaPath/MediaPaths plus MediaType/MediaTypes, resolve them through existing media safety boundaries, and build structured Gemini embedding inputs for those attachments. Preserve the shipped gemini-embedding-2-preview, outputDimensionality, and memory-file indexing behavior; add regression coverage for chat or Telegram attachment recall; and update the memory docs to distinguish memory-file media indexing from conversation-history attachment indexing and supported modalities.

## Risks / Open Questions

- Closing this as implemented would lose the requested recall workflow for screenshots/photos received through channels such as Telegram unless users manually place those files into an indexed memory path.
- A future implementation must preserve media safety boundaries: managed media path resolution, sandbox/private path checks, MIME validation, file-size limits, provider capability checks, and fallback behavior need explicit design.
- Video and PDF remain separate gaps for memorySearch multimodal indexing because the current memory multimodal modality set is image/audio only.
