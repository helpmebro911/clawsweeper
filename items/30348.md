## Summary

Keep open. Current `main` partially addresses the motivation by hardcoding `Accept: text/markdown` in `web_fetch`, and the browser control service/CLI can set extra headers, but the requested configurable header support is not implemented as a `tools.web.fetch.headers` or `browser.headers` config surface, and the agent `browser` tool still does not expose a headers action.

## Evidence

- **web_fetch now prefers Markdown responses by default:** `runWebFetch` sends a fixed `Accept: text/markdown, text/html;q=0.9, */*;q=0.1` header, with a matching test proving that behavior.
  - file: [src/agents/tools/web-fetch.ts:423](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/agents/tools/web-fetch.ts#L423)

  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **web_fetch config still has no headers field:** The typed config and runtime schema for `tools.web.fetch` include `enabled`, `provider`, limits, `userAgent`, `readability`, and `ssrfPolicy`, but no `headers` property. The docs show the same config surface.
  - file: [src/config/types.tools.ts:540](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/config/types.tools.ts#L540)

## Risks / Open Questions

- The motivating `web_fetch` use case is partly solved already, so another reviewer could argue for closure based on that narrower interpretation rather than the requested configurable header surface.
- The browser subsystem already has header-setting support in the control API/CLI, which could be mistaken for full agent-tool/config support even though the schema and config do not expose it.
