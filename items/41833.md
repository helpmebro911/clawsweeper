## Summary

Keep open. Current main has meaningful adjacent work: cron jobs now have task-ledger completion tracking, failure alerts, run history, and a reminder-specific guard that warns when the assistant says it will remind/ping/follow up without a successful cron.add. That does not fully satisfy this issue’s broader request for a native commitment/deferred-task primitive with add/list/done semantics, generic commitment records, user-visible status, and overdue escalation for non-reminder commitments like reports or weekend fixes.

## Best Possible Solution

Keep this issue open and scope a core design that builds on existing cron and task-ledger primitives: a first-class commitment record or CLI/tool surface with due time, linked cron/task run, done/cancel/list/status operations, completion reconciliation, and overdue/failure escalation. The existing reminder guard can remain as a narrow safety net, but it is not a replacement for the broader requested primitive.

## Risks / Open Questions

- A native commitment primitive would need careful product design to avoid false positives in natural-language promise detection and unexpected autonomous scheduling.
- The implementation should reuse existing cron failure alerts, run history, task-ledger maintenance, and the reminder guard rather than adding a parallel scheduler.
