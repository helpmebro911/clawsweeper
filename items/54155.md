## Summary

Keep open. Current main has partial mitigations for Gateway memory growth, especially seqByRun cleanup and session maintenance, but the broad reported RSS/session-accumulation problem is still not proven fixed. The session-store structuredClone/cache path remains, several Gateway history surfaces still materialize full transcript files before applying limits, and I found no multi-day plateau evidence for the reported cron/channel/subagent workload.

## Best Possible Solution

Keep this issue open as the broad Gateway RSS/session-accumulation tracker. The best maintainer path is to finish or supersede #45438, remove or bound the structuredClone/session-cache retention path, avoid full transcript materialization before limits for Gateway history surfaces, preserve the shipped seqByRun and session-maintenance mitigations, and add long-running or targeted soak evidence showing RSS plateaus under cron jobs, channel traffic, and subagent sessions.
