## Summary

Keep open. Current main has partial memory-growth fixes and mitigations, including the seqByRun cleanup and session maintenance, but the broader Gateway RSS/session-accumulation report is not fully resolved: the session-store cache still clones and retains store state, active Gateway paths still materialize whole transcripts, and compaction transcript rotation remains opt-in rather than a default proof that long-running RSS now plateaus.

## Best Possible Solution

Keep this issue open as the canonical broad Gateway RSS/session-accumulation tracker. The best maintainer path is to land or supersede the #45438 structuredClone fix, reduce retained dual serialized/object session-store state, stream or cap transcript reads used by Gateway sequencing/history paths, decide whether compaction transcript rotation should become a safer default or have stronger automatic policy, and add targeted regression or soak evidence showing long-running Gateway RSS plateaus under cron, channel traffic, and subagent sessions.
