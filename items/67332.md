Keep this PR open. Current main still does not ship the requested owner-only `task` and `team` agent tools, and this PR remains the active implementation candidate rather than obsolete work. It needs maintainer review because it is stacked on #67331, mixes task/team wiring with filesystem hardening, and has unresolved security-sensitive review feedback around schema shape, direct tool invocation, owner authorization, and error mapping.

## Best Possible Solution

Keep this PR open for maintainer review. Resolve or split #67331 first, rebase the task/team changes onto current main, add task/team to the core catalog/docs/tool-policy surfaces only after an owner-only authorization decision, use the existing owner-scoped task access helpers where appropriate, fix the unresolved review comments, and validate non-owner denial through focused `/tools/invoke`, MCP/listing, and agent-tool tests.

## Risks / Open Questions

- The PR depends on still-open #67331 and includes filesystem permission/error hardening alongside task/team tool wiring, so merging it as-is could blend separate behavior changes and complicate review.
- New owner-only `task` and `team` tools need explicit coverage across normal agent use, direct `/tools/invoke`, and MCP/listing paths so non-owner callers cannot invoke them by bypassing schema or catalog assumptions.
- The PR review comments identify caller-visible issues that should be addressed before merge: schema `anyOf` shape, unsupported task action handling, team update error classification, and permission-denied reason classification.
- The proposed `team` registry appears to add new in-memory coordination state; maintainers should decide whether that is the intended persistence and lifecycle model before shipping it as a core tool.
