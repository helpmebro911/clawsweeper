## Summary

Keep open. Current main still implements the reported mixed-signal behavior: `minRecallCount` is enforced using aggregate `signalCount`, and `signalCount` still includes recall, daily, and grounded evidence. The current tests and docs also encode daily/dreaming signals as promotion evidence, so this is an active product/semantics decision rather than a cleanup-close candidate.

## Best Possible Solution

Keep this issue open for maintainer product judgment. If accepted, split the semantics in `extensions/memory-core/src/short-term-promotion.ts` so `minRecallCount` checks real `recallCount` only, keep daily/grounded/session evidence under a separately named aggregate signal metric, recalibrate score thresholds so recall-only candidates remain reachable, and update the daily-only promotion tests, CLI diagnostics, and docs to match the chosen behavior.
