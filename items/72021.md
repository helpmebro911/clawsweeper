## Summary

Keep #72021 open. Current main still implements the reported behavior: promotion ranking treats `recallCount + dailyCount + groundedCount` as the candidate signal count, uses that aggregate for the `minRecallCount` gate, and uses it in average-score/frequency calculation. Existing tests also explicitly expect repeated daily-only dreaming signals with `recallCount: 0` and `dailyCount: 3` to clear the default gates, so this is not already fixed or obsolete.

## Best Possible Solution

Keep the issue open and fix `memory-core` promotion semantics directly. The likely path is to split promotion gating from aggregate evidence: make the `minRecallCount` gate use real recall signals only, decide explicitly how daily/session and grounded signals should influence score/frequency, update `memory promote-explain` so its pass/fail output matches the actual gate, and replace the daily-only default-gate test with regression coverage for the chosen behavior.
