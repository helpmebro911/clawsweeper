## Summary

Keep open. Current main has related edit-tool hardening, but it does not clear a stale mutating edit failure after the file is fixed externally; the warning path still keys off `lastToolError` until a later matching edit tool success occurs.

## Evidence

- **Edit tool recovery is narrower than the reported case:** `wrapEditToolWithRecovery` only does two things: convert post-write throws into success when the file actually changed, or append current file contents to exact-match mismatch errors. It does not clear any later conversation/tool error state after an out-of-band manual repair.
  - file: [src/agents/pi-tools.host-edit.ts:145](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-tools.host-edit.ts#L145)
  - command: `nl -ba src/agents/pi-tools.host-edit.ts | sed -n '145,209p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Both host and sandbox edit tools use that wrapper:** Current main routes both edit tool variants through the same recovery wrapper, so there is no separate external-fix clearing path elsewhere in tool construction.
  - file: [src/agents/pi-tools.read.ts:644](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-tools.read.ts#L644)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '644,671p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Stale mutating tool failures are retained until the same action succeeds:** `handleToolExecutionEnd` stores mutating failures in `lastToolError` and only clears them when a later successful tool call matches `isSameToolMutationAction(...)`. Non-matching successes do not clear a mutating edit failure.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:808](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-subscribe.handlers.tools.ts#L808)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '808,833p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Matching requires a later mutating tool fingerprint:** `buildToolActionFingerprint` and `isSameToolMutationAction` compare later mutating tool calls by normalized path/action fingerprint and fail closed when fingerprints do not match. An external manual file fix or a readback does not produce that clearing signal.
  - file: [src/agents/tool-mutation.ts:155](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/tool-mutation.ts#L155)

- **The stale tool error is still surfaced to users:** When `lastToolError` is present, payload building emits a user-facing warning for mutating tool failures. That matches the report's confusing persistent error symptom if no matching successful edit tool call occurs afterward.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:339](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/payloads.ts#L339)

## Risks / Open Questions

- A too-aggressive fix could hide real unresolved mutating failures; current code intentionally fails closed for mutation-state clearing.
- There is already partial hardening for related edit failures, so any follow-up should distinguish post-write recovery, retry-success clearing, and truly out-of-band manual repair.
