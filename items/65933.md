## Summary

Keep this PR open. Current main still lets the exec tool's model-supplied `security` argument reduce the effective configured policy via `minSecurity`, so the reported `tools.exec.security="full"` plus model `security:"allowlist"` failure remains real. The PR is a plausible implementation candidate, but the supplied head still needs cleanup before it should land.

## Best Possible Solution

Keep this PR open and review it as a real exec policy bug fix. The best landing shape is a minimal change in `src/agents/bash-tools.exec.ts` that makes configured/session exec security authoritative for model tool calls, preserves the explicit operator-granted elevated-full override, removes unused imports and the unused `maxSecurity` helper/tests, adds focused regressions for configured `full` with model `allowlist`, configured `allowlist` with model `deny`, and configured `deny` with model `full`, and makes an explicit compatibility decision for the model-facing `security` schema.

## Risks / Open Questions

- The supplied PR head still adds an exported `maxSecurity` helper and tests even though the final implementation no longer uses it; `maxSecurity("deny", "full") === "full"` would be a security footgun if reused later, so it should be removed before landing.
- After the PR changes `security` to the configured value, the existing `minSecurity` import in `src/agents/bash-tools.exec.ts` appears to become unused and should be cleaned up before CI/lint review.
- If model-supplied `security` is ignored, maintainers should decide whether the exec schema keeps the field as a compatibility no-op or updates its description so models are not encouraged to set operator policy.
