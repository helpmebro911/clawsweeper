## Summary

Keep this issue open. Current main still has no `memorySearch.local.maxThreads` setting or equivalent thread-control wiring for the built-in local node-llama-cpp embedding provider: repo-wide search has no `maxThreads` hits, strict config only accepts `modelPath`, `modelCacheDir`, and `contextSize`, resolved memory-search config only preserves those fields, and `createLocalEmbeddingProvider` still calls `getLlama` with only `logLevel`. Upstream node-llama-cpp does expose `getLlama(options?: LlamaOptions)` and `LlamaOptions.maxThreads?: number`, so this is a feasible missing implementation rather than an obsolete request. citeturn1view0turn1view1

## Best Possible Solution

Keep this issue open. The best fix is to add a validated `agents.defaults.memorySearch.local.maxThreads` setting with the same per-agent override behavior as other `memorySearch.local` keys, wire it through public config types, strict Zod schema, generated schema labels/help, `ResolvedMemorySearchConfig`, memory embedding provider create options, the node-llama shim, and `createLocalEmbeddingProvider`, then pass it to `getLlama` only when configured. Add tests for default omission, configured forwarding, invalid-value rejection, default/per-agent merge behavior, and docs coverage.
