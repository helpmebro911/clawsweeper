## Summary

Keep this PR open. Current main still has the single generic self-hosted vLLM setup path, not the endpoint lifecycle management, multi-endpoint `vllm-2` style selection, or stale managed-vLLM default cleanup proposed here. The PR is still potentially useful, but it needs rework against the current provider-plugin architecture and a cleanup pass for unrelated/security-sensitive changes before maintainer review.

## Best Possible Solution

Keep the PR open for now. The best path is to rework it narrowly around the current bundled vLLM provider plugin: implement endpoint add/update/delete and multi-model discovery in the provider-owned setup flow, add focused tests for multi-endpoint keys and stale default/fallback pruning, wire model-picker/auth-choice paths through existing plugin setup contracts, and drop unrelated `.secrets.baseline`, memory-test, A2UI script, and zizmor changes or split them into separate reviewed PRs.
