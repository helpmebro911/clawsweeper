## Summary

Current main still does not surface a typed session-lane busy or wait-timeout error for `openclaw agent`. The session queue only logs long waits, the Gateway accepts the run before dispatch settles and maps dispatch failures to generic `UNAVAILABLE`, and the CLI has no lane-busy decode or queue-policy/force-queue option.

## Best Possible Solution

Keep this issue open and fix the Gateway/CLI contract. Add a typed session-lane busy or lane-wait-timeout result from the session queue/agent dispatch path, propagate `lane`, `queueAhead`, and `waitedMs` through the Gateway final/error response, make `openclaw agent` print an actionable message, and add focused regression coverage for blocked same-session CLI dispatch. If queueing remains default behavior, add and document an explicit queue policy or force-queue option.
