## Summary

Keep open. Current `main` at `38caa6832d4e1d344736d635b0c5fdad37e4430b` still exhibits the reported behavior: nested `sessions_send` A2A steps fall back to internal channel `webchat` when no explicit `channel` is passed, and the A2A flow does not recover the requester channel from session delivery context.

## Evidence

- **Nested agent steps still default to webchat:** `runAgentStep()` sends `channel: params.channel ?? INTERNAL_MESSAGE_CHANNEL`, and `INTERNAL_MESSAGE_CHANNEL` is still defined as `"webchat"` on current `main`. Any caller that omits `channel` still records/runs under webchat.
  - file: [src/agents/tools/agent-step.ts:39](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/agents/tools/agent-step.ts#L39)
  - command: `nl -ba src/agents/tools/agent-step.ts | sed -n '1,240p'; nl -ba src/utils/message-channel-constants.ts | sed -n '1,80p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **A2A sessions_send flow still omits explicit channel on reply and announce steps:** `runSessionsSendA2AFlow()` resolves only the target announce target, then calls `runAgentStep()` for ping-pong and announce without a `channel` field. It passes `sourceChannel`, but that does not affect `runAgentStep()`'s fallback channel. There is no requester-session delivery-context hydration here on current `main`.
  - file: [src/agents/tools/sessions-send-tool.a2a.ts:64](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/agents/tools/sessions-send-tool.a2a.ts#L64)

That is the behavior this PR is trying to change, and it remains the expectation on `main`.
