## Summary

Current main already knows how to pass through `reply_to_*` fields from iMessage notifications, but it still only does that when the upstream `imsg` payload already contains those fields. I found no OpenClaw-side fallback for `reply_to_guid` / `thread_originator_guid` / direct `chat.db` lookup, and the iMessage monitor code is unchanged from release `v2026.4.22`, so there is not strong evidence that #42266 is implemented or obsolete.

## Evidence

- **Reply context is only derived from existing reply_to_* fields:** `describeReplyContext()` reads only `message.reply_to_text`, `message.reply_to_id`, and `message.reply_to_sender`; there is no alternate resolution path from GUIDs or DB data.
