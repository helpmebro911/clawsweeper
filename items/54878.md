## Summary

Keep this issue open. Current main still applies the cross-agent rejection before honoring spawned-session lineage, so a parent session reading a spawned ACP child with a different `agent:` id remains blocked under default `tree` visibility or without the child agent in the agent-to-agent allowlist. The related fix PRs in the provided context are closed and unmerged, and current regression coverage still does not cover the reported cross-agent spawned-child shape.

## Best Possible Solution

Keep this issue open and implement the narrow guard fix in `src/plugin-sdk/session-visibility.ts`: in `tree` mode, allow the requester session and sessions present in the spawned set before applying generic cross-agent rejection, while preserving agent-to-agent enforcement for non-spawned cross-agent targets. Add RED/GREEN coverage for `agent:main:main` reading a spawned `agent:codex:acp:<id>` child with default/tree visibility and no broad agent-to-agent allowlist, ideally at both guard level and `sessions_history` tool level. The closed PRs #55983, #56019, and #56028 are useful references but are not merged.
