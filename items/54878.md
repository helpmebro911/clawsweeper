## Summary

Keep this issue open. Current main still rejects cross-agent session keys before honoring `tree` spawned-session lineage, so a parent can still be denied when reading its own ACP child such as `agent:codex:acp:<uuid>`. The related fix attempts (#55983, #56019, #56028) were closed unmerged, and the narrow lineage fix is not present on current main.

## Best Possible Solution

Keep this issue open and implement the narrow lineage fix in `src/plugin-sdk/session-visibility.ts`: for `tree` visibility, allow the requester session and targets present in the spawned set before applying generic cross-agent rejection. Add regression coverage for `agent:main:main` reading a spawned `agent:codex:acp:<id>` or `agent:opencode:acp:<id>` child with default/tree visibility and no broad `tools.agentToAgent` allowlist, preferably at both guard and `sessions_history` tool levels. Then audit other session tools, especially `session_status`, for pre-guard cross-agent checks that should honor proven spawned lineage.

## Risks / Open Questions

- Closing now would leave ACP orchestration unable to reliably retrieve parent-owned child history when the child session belongs to a different agent id.
- The fix needs to allow only proven spawned-tree lineage before generic cross-agent policy; non-spawned cross-agent access should remain governed by `tools.sessions.visibility` and `tools.agentToAgent`.
- `session_status` has an additional explicit-agent cross-agent gate before the shared guard, so maintainers should decide whether the same lineage rule applies there after the shared history/list/send guard is fixed.
