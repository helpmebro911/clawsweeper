## Summary

Keep open. Current main still does not implement this PR's central fix: incomplete-turn resolution trusts assistant-authored exact NO_REPLY and committed messaging-tool delivery, but it does not inspect the latest current-attempt tool result for exact NO_REPLY. The PR is also not safe to close as ready/superseded because the provided review discussion identifies a real remaining issue: the PR's silent-tool fast path can bypass provider-error handling unless gated to clean terminal states.

## Best Possible Solution

Keep the PR open and request a narrow update: pass messagesSnapshot and prePromptMessageCount into the incomplete-turn resolver, inspect only the latest non-error tool result from the current attempt, accept exact NO_REPLY only for clean terminal states such as stop, and keep provider-error handling ahead of or outside that fast path. Add regression tests for quiet-success toolResult NO_REPLY, stale prior-turn NO_REPLY, and stopReason="error" with a latest NO_REPLY tool result.
