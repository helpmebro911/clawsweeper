## Summary

Keep open. Current main still does not implement the PR’s specific silent tool-result path, and the PR head is not safely obsolete or merge-ready because the latest discussion identifies a real provider-error regression in the proposed fast path.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: carry prePromptMessageCount into EmbeddedRunAttemptResult, scan only current-attempt tool messages for the latest non-error exact NO_REPLY result, and gate that silent success so aborts, timeouts, tool errors, and provider stopReason="error" still use the existing retry/error paths. Add tests for the successful tool-result NO_REPLY case, stale prior-turn NO_REPLY being ignored, and provider-error-after-NO_REPLY not being suppressed.
