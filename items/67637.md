## Summary

Keep open. Current main still does not expose generalized tool-call traces on QaBusMessage; it only has mock-provider /debug/requests plannedToolName/plannedToolArgs signals that help some scenarios but do not satisfy the requested QA bus trace surface populated from actual agent/tool execution.

## Best Possible Solution

Keep this issue open. Implement a backwards-compatible QA trace surface, either optional structured tool-call traces on QaBusMessage or a dedicated scenario assertion API, populated from actual agent/tool execution logs rather than only mock-provider planned requests. Carry it through create, clone, snapshot, read, search, poll, and wait paths, and cover it with focused qa-channel/qa-lab tests plus at least one scenario that fails when the agent fabricates success without calling the expected tool.
