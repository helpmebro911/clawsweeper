## Summary

Current main recomputes `nextRunAtMs` for normal `cron.update` / `openclaw cron edit` schedule changes, but the issue's remaining direct-file-edit path is still unresolved: editing `jobs.json` can leave `jobs-state.json` with a still-future stale `nextRunAtMs`, and load/reload preserves that value without schedule drift detection.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for externally edited cron definitions. A conservative path is to persist a schedule identity with runtime state, recompute and persist `nextRunAtMs` when the loaded job definition's identity differs from the sidecar, and add regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`. Preserve the existing `cron.update` recomputation and restart catch-up safeguards; an explicit `openclaw cron reschedule <id>` command could be a useful operator escape hatch.
