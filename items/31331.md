## Summary

Keep #31331 open. Current main still builds Docker sandbox and sandbox-browser workspace mounts from gateway-visible `workspaceDir` / `agentWorkspaceDir` strings and passes them directly to `docker create`. The Docker install/setup path still maps host storage into `/home/node/.openclaw/...` inside the gateway container, so Docker-outside-of-Docker can still hit the reported host/container path mismatch. The related fix PRs are closed unmerged in the provided context, and there are recent same-problem confirmations.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed sandbox workspace and agent workspace bind sources through the gateway container mount table before validation, config hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none` plus browser containers. If maintainers prefer host-path parity as the supported contract, Docker setup/onboarding should generate or validate that parity by default and fail clearly when it is absent; documentation alone should not close this bug.
