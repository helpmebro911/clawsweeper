## Summary

Keep #31331 open. Current main still builds Docker sandbox and sandbox-browser workspace bind mounts from gateway-visible paths and passes those paths directly to Docker, while the Docker install mounts host storage into `/home/node/.openclaw/...` inside the gateway container. New docs describe a manual DooD host-path parity constraint, but current code/setup do not implement automatic host-path remapping or make the default Docker install satisfy that parity. The provided issue context also shows this is the canonical item with recent same-problem confirmations and related fix PRs closed unmerged.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed sandbox workspace and agent workspace bind sources through the gateway container mount table before validation, hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none` plus browser containers. If maintainers prefer host-path parity as the official contract, Docker setup/onboarding should generate or validate host-canonical workspace/sandbox paths and identical gateway volume maps by default; documentation alone is not enough to close this bug.
