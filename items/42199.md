## Summary

Keep open. Current `main` has newer trusted-env-proxy plumbing, but the memory embedding HTTP path this PR targets still calls `fetchWithSsrFGuard` in default strict mode, so `HTTPS_PROXY` is not automatically honored there. The PR discussion also already identified correctness gaps in the proposed patch (DNS-before-proxy ordering and proxy-env semantics), so there is not strong evidence to close this as already implemented or obsolete.

## Evidence

- **`fetchWithSsrFGuard` still defaults to strict pinned-DNS mode:** `resolveGuardedFetchMode` returns `STRICT` unless the caller explicitly sets `mode` or deprecated env-proxy flags. In the main request path, env-proxy dispatch only happens when `mode === TRUSTED_ENV_PROXY`; otherwise the code still resolves and pins DNS locally.
