## Summary

Keep open. Current main partially narrows the original abort-controller overwrite hazard because `registerChatAbortController()` refuses to replace an existing run id, but `chat.send` still does not satisfy the documented same-`idempotencyKey` attachment-race contract: after attachment parsing, it ignores `registered: false`, still sends `{status:"started"}`, adds another chat-run entry, and dispatches the duplicate path. The linked PR #70142 remains the active exact fix path and is not merged into main.

## Best Possible Solution

Keep #70139 open until #70142 or an equivalent main-branch fix lands. The best implementation is to make the post-parse loser return the same `{status:"in_flight"}` response as the early guard, either with an explicit post-parse re-check before `addChatRun`/`respond`/`dispatchInboundMessage`, or by honoring `registerChatAbortController().registered === false` before doing any duplicate gateway-side work. Add a focused regression test for two concurrent same-`idempotencyKey` attachment sends that proves exactly one `started` ack and one `in_flight` ack.
