## Summary

Current `main` still matches the reported `chat.send` attachment-branch race. The handler checks `chatAbortControllers` before two awaited attachment/model-capability steps, then calls `registerChatAbortController(...)` later; that helper declines duplicate `runId`s, but `chat.send` ignores the returned `registered` flag and still adds the run plus responds `{status:"started"}`. That means the documented same-`idempotencyKey` `in_flight` contract is not reliably enforced for attachment sends, so this should stay open.

## Evidence

- **Pre-parse in-flight guard happens before awaited attachment work:** `chat.send` returns `{status:"in_flight"}` only from the early `chatAbortControllers.get(clientRunId)` check. For attachment sends, that check runs before `await resolveGatewayModelSupportsImages(...)` and before `await parseMessageWithAttachments(...)`, leaving an async gap before the run is claimed.

  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Attachment parsing does real async work before registration:** `parseMessageWithAttachments(...)` awaits MIME sniffing and, for offloaded images, `saveMediaBuffer(...)`, so the attachment branch is not atomic between the early `get` and later registration.
  - file: [src/gateway/chat-attachments.ts:345](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/gateway/chat-attachments.ts#L345)

- **Abort registration refuses duplicate run IDs but does not overwrite existing owner:** `registerChatAbortController(...)` returns `{ registered: false }` when `chatAbortControllers.has(runId)` is already true. That prevents overwrite, but only if the caller handles the false return.
  - file: [src/gateway/chat-abort.ts:89](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/gateway/chat-abort.ts#L89)

So a retried attachment send can still receive `started` after another caller has already claimed the run ID.
