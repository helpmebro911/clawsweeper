## Summary

Current main still uses Baileys `useMultiFileAuthState` for the WhatsApp session auth store. OpenClaw has atomic `creds.json` persistence and backup/restore mitigations, but it has not replaced the Baileys multi-file signal-key store with a SQL or consolidated auth state. Related PR #66863 is useful prior art, but it is closed unmerged per the provided GitHub context and its referenced commit is not present in this checkout.

## Best Possible Solution

Keep #9544 open until the WhatsApp plugin replaces `useMultiFileAuthState` with a production auth store for credentials and signal keys, adds migration from the existing Baileys multi-file layout, preserves or supersedes the current atomic `creds.json` safety behavior, and covers the path with focused WhatsApp plugin tests. PR #66863 can be treated as prior art, but it is not a shipped or canonical fix.
