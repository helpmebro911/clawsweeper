## Summary

Keep open. Current main still does not implement the requested shared-secret fallback for `gateway.auth.mode="trusted-proxy"`: the dispatcher returns the trusted-proxy failure directly, startup validation rejects mixed trusted-proxy plus shared-token config, and tests/docs now codify fail-closed loopback behavior. The provided GitHub context shows related follow-up work (#64122, #67524), but no merged implementation or clearly canonical superseding item closes the remaining request.

## Evidence

- **Trusted-proxy dispatcher still returns before token/password auth:** `authorizeGatewayConnectCore` handles `auth.mode === "trusted-proxy"` by calling `authorizeTrustedProxy`; if no trusted-proxy user is returned, it immediately returns `{ ok: false, reason: result.reason }`. The token and password branches are below this block and are still unreachable for trusted-proxy failures.
