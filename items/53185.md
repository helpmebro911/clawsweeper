## Summary

Keep this issue open. Current main has meaningful built-in `web_fetch` cleanup for hidden HTML and invisible Unicode, and web search providers wrap returned content as untrusted, but the requested surface is still not implemented: no `content-sanitizer.py` integration, no config-driven per-tool script preprocessing hook, no shared `web_search` sanitizer/threat decision point, and no `threat_level` or HIGH/CRITICAL block semantics. Current tool-result middleware is a trusted bundled-plugin seam only, not an external/plugin or user-script preprocessing API.

## Best Possible Solution

Keep this issue open but narrow it. Treat the current `web_fetch` hidden-content stripping as substantially implemented, then make a maintainer product/security decision on a trusted pre-model tool-result preprocessing API for `web_fetch` and `web_search`: either a bundled-only sanitizer/classifier with documented threat-level semantics, or a new explicitly trusted plugin/script seam with fail-closed behavior, auditability, and tests. Keep broader provenance and execution guardrails tracked in #54508.

## Risks / Open Questions

- Closing as implemented would conflate shipped `web_fetch` hidden-content cleanup with the still-missing `web_search`, custom preprocessing, and structured threat-decision path.
- A user-configured script preprocessing hook would execute local code and rewrite tool output before model ingestion, so it needs an explicit trust model, failure semantics, auditability, and runtime coverage.
- Related #54508 overlaps on broader taint/risk-class enforcement, but it remains open and does not supersede this issue's concrete custom script/preprocessing API request.
