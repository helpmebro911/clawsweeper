## Summary

Keep this PR open. Current main still drops MIME types on Discord media uploads and still sends WebP images through the shared optimizer, so the PR addresses a real open bug. It should stay under maintainer review rather than be auto-closed because the WebP bypass changes shared media behavior with WhatsApp implications, unresolved validation/privacy concerns remain, the PR does not cover the native Discord interaction upload path, and overlapping open PRs #41265 and #41714 should be consolidated deliberately.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it with a focused canonical fix for #41256. The best path is to preserve verified `media.contentType` in every Discord upload Blob path, including standard sends, component sends, and native interaction replies; decide whether WebP and animated-image preservation belongs in shared media loading or in Discord-specific send policy; add regression tests for MIME preservation and animation/frame preservation; resolve WebP metadata and unknown-dimension validation before bypassing optimization; keep WhatsApp tests/docs aligned with the chosen behavior; and consolidate the overlapping work in #41265 and #41714.

## Risks / Open Questions

- PR security pass: the diff touches shared media processing and would return verified WebP buffers without re-encoding, which can preserve EXIF/XMP/ICC metadata that the existing optimizer likely strips.
- PR security pass: the proposed WebP passthrough relies on `assertImagePixelLimit`; current non-sips behavior returns when header metadata cannot be determined, so crafted or unsupported image headers need a fail-closed policy before shared passthrough is safe.
- The PR does not cover `extensions/discord/src/monitor/native-command.ts`, leaving one Discord reply upload path with typeless Blob objects.
- The PR changes WhatsApp-observed shared media behavior while current WhatsApp tests/docs still expect WebP/common images to be recompressed to JPEG.
- There are multiple open implementations for the same linked bug (#41265 and #41714), so maintainers should choose or combine one canonical fix rather than close or merge this PR in isolation.
