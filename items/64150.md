## Summary

Keep this PR open. Current main still has the reported Anthropic OAuth Claude Code tool-name gap, and the default OpenClaw coding tool set still does not register `find`, `grep`, or `ls`. The PR is a plausible fix direction, but it is not merge-ready: its dependency import shape is stale, it appends upstream filesystem discovery tools without OpenClaw sandbox/workspace policy wrappers, and it does not update the core tool catalog, group policy, docs, or regression tests.

## Best Possible Solution

Keep this PR open as a real but incomplete implementation candidate. The best path is a focused revision or successor that registers guarded OpenClaw wrappers around `createFindTool`, `createGrepTool`, and `createLsTool`, routes filesystem access through the same workspace/sandbox policy surfaces as `read`/`write`/`edit`, updates `tool-catalog` and `docs.openclaw.ai` tool/group docs, handles the `fd`/`rg` executable policy explicitly, and adds regression tests for Anthropic OAuth `Glob` -> `find`, `Grep` -> `grep`, `LS` -> `ls`, plus `group:fs` allow/deny behavior.

## Risks / Open Questions

- Merging the PR as submitted could expose `find`, `grep`, or `ls` outside the `tools.fs.workspaceOnly` and sandbox filesystem bridge paths used by existing OpenClaw filesystem tools.
- Merging the PR as submitted could make `group:fs`, profiles, allowlists, denylists, and public docs inaccurate because the new filesystem-reading tools are not added to the core tool catalog.
- The upstream discovery tools can invoke `fd` and `rg` through the dependency's tool manager path; OpenClaw needs an explicit policy decision for that executable acquisition/execution path before core registration.
- The PR appears stale against the pinned dependency because `findTool`, `grepTool`, and `lsTool` are not exported symbols in `@mariozechner/pi-coding-agent@0.70.2`.
