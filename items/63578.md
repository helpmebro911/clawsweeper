## Summary

Keep this PR open. Current main still does not hydrate plugin-handled `send` media/path/file hints into shared attachment fields, and the PR head is a plausible implementation candidate because it changes that behavior behind an opt-in `preloadSendMedia` adapter contract. It is not implemented on main and not cleanly superseded by the Telegram/buffer-only related work.

## Best Possible Solution

Keep this PR open for maintainer review or contributor refinement. The best path is a guarded public plugin action contract for send-media preloading, wired from `runMessageAction` into `hydrateAttachmentParamsForAction`, with tests proving both default URL/path forwarding and opt-in hydrated `buffer`/`contentType`/`filename`. Coordinate with #41779, #42674, #53641, and #57335, but do not close this PR as implemented or duplicate while current main lacks the plugin send media hydration path.

## Risks / Open Questions

- This is a public plugin channel contract change, so maintainers should review the exact `preloadSendMedia` semantics and require matching SDK docs/API checks before merge.
- Opt-in media preloading changes where remote URLs or local paths are fetched/read; the implementation must keep using the existing sandbox and outbound media-access policy.
- Existing plugin send handlers such as Google Chat, Matrix, and Discord already load or forward URL/path media themselves, so only handlers that consume hydrated `buffer`/`contentType`/`filename` should opt in.
