## Summary

Keep open. Current main still treats plugin-entry config schema failures as global invalid config and as strict plugin-load errors in key registry paths, so the requested per-plugin quarantine/graceful degradation is not implemented. The linked PR #70394 is relevant but open and unmerged in the provided context, not a shipped resolution.

## Best Possible Solution

Keep #70371 open and finish a core fix. The best path is to quarantine plugin-entry schema failures per plugin: preserve clear diagnostics, mark the offending plugin failed or disabled in registry/status, skip its registration and tools, continue booting unaffected gateway/channels/jobs, and extend doctor/status output or repair for `plugins.entries.<id>.config` schema failures. PR #70394 is a useful starting point but should be reviewed against the config-load, runtime-loader, metadata-loader, and doctor requirements.

## Risks / Open Questions

- Closing now would hide a real remaining outage path where one bad plugin config can block gateway or CLI startup before OpenClaw can disable only that plugin.
- A narrow fix that only flips `throwOnLoadError` may not satisfy the issue's doctor/status repair acceptance criteria unless config-load and plugin registry diagnostics are handled together.
