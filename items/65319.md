## Summary

Keep open. Current main has better manifest-backed command-alias diagnostics, but the exact lossless-claw path remains unresolved: if `lcm` or `lossless` is not resolved through manifest `commandAliases`, the CLI fallback still tells users to add the command token to `plugins.allow`, while validation still treats unknown allow entries as stale plugin ids. A maintainer explicitly left this open and called #64732 adjacent rather than canonical.

## Best Possible Solution

Keep this issue open until the lossless-claw path is verified end to end. The best fix is either for `@martian-engineering/lossless-claw` to declare `lcm` and `lossless` as manifest `commandAliases` with appropriate `runtime-slash` or CLI metadata, or for core to harden the unknown-command fallback so it does not recommend command tokens as `plugins.allow` entries when ownership is unresolved. Coordinate broader fallback cleanup with #64732, but do not treat it as canonical unless maintainers explicitly designate it.
