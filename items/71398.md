## Summary

Keep #71398 open. Current main still has no parser or migration for the reported `secretref:/...` string shorthand: SecretInput string values are treated as plaintext first, QQ Bot passes the resolved `clientSecret` into its token request, and the docs only define structured SecretRef objects. The existing maintainer/Codex comment also explicitly kept this issue open, and the provided related items do not supersede this QQ Bot regression.

## Best Possible Solution

Keep the issue open. Maintainers should decide whether `secretref:/...` is an old supported shorthand that needs compatibility/migration or invalid syntax that should fail validation with clear doctor/docs remediation. The likely fix is shared SecretInput migration/validation plus QQ Bot regression coverage for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, with docs and the SecretRef credential surface aligned to the supported behavior.
