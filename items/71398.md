## Summary

Current main still leaves the reported QQ Bot SecretRef failure open. A `secretref:/...` value is still treated as ordinary plaintext, QQ Bot still routes `clientSecret` through that shared string path, and the QQ Bot plugin still lacks the channel SecretRef contract/target-registry coverage needed to materialize runtime SecretRefs for `channels.qqbot.clientSecret` or account-level secrets.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the QQ Bot plugin plus the shared SecretRef policy surface: add a QQ Bot `secret-contract-api` and runtime assignment collector for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, update the canonical SecretRef credential docs, and add regression coverage for both structured SecretRef objects and the literal `secretref:/...` marker path. For `secretref:/...`, either implement a precise legacy migration/resolution path or reject it during config validation with actionable structured SecretRef guidance.
