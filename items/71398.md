## Summary

Keep this issue open. Current main still has the same QQ Bot SecretRef gap described in the discussion: structured SecretRef objects are recognized, but a literal `secretref:/...` string is treated as ordinary plaintext, accepted by the QQ Bot schema, and can flow into the QQ Bot `clientSecret` runtime path instead of being resolved or rejected with remediation.

## Best Possible Solution

Keep this issue open. The best path is a targeted maintainer fix in shared SecretInput validation/migration/runtime handling plus QQ Bot regression coverage for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`: either resolve or migrate `secretref:/...` precisely if it is supported legacy syntax, or reject it early with docs-backed remediation to the structured SecretRef object form. Also confirm whether QQ Bot needs an explicit secret contract/target-registry entry so runtime SecretRef resolution and docs stay aligned.
