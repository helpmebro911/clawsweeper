## Summary

Keep issue #71398 open. Current main still treats non-empty SecretInput strings as plaintext before SecretRef coercion, so `secretref:/QQBOT_...` would be passed through as the QQ Bot client secret rather than resolved. I found structured SecretRef support for QQ Bot, but no current-main support, validation, migration, release evidence, or superseding item for the reported `secretref:/...` string-form upgrade regression.

## Best Possible Solution

Keep the issue open. The best path is to make a maintainer decision on `secretref:/...`: either add shared SecretInput migration/runtime compatibility plus QQ Bot regression tests for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, or explicitly reject that syntax through validation/doctor with docs pointing users to the structured SecretRef object form. The QQ Bot docs, SecretRef target registry, and credential-surface docs should be aligned either way.

## Evidence
