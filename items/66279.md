## Summary

Current main still matches the reported TUI-specific TLS trust bug. `openclaw tui` resolves the gateway URL/auth and constructs `GatewayClient` without a `tlsFingerprint`, while the shared gateway call and status/probe paths already load or forward TLS fingerprints for secure gateway connections. The related local self-signed TLS reports cover other surfaces and do not supersede this TUI gap.

## Best Possible Solution

Keep this issue open. Implement the TUI trust handoff in `src/tui/gateway-chat.ts`: extend the resolved connection with an optional `tlsFingerprint`, load the local gateway TLS runtime for configured local `wss://` targets, avoid applying it to URL overrides or remote endpoints, pass it into `GatewayClient`, and add focused tests in `src/tui/gateway-chat.test.ts` for propagation, unavailable runtime fallback, URL override, and remote cases.
