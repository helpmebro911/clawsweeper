## Summary

Keep open. Current main still has the reported TUI-specific TLS trust gap: `openclaw tui` resolves the local WSS URL and auth, but the resolved connection has no `tlsFingerprint` field and `GatewayChatClient` never passes one to `GatewayClient`. The status/probe paths already load the local TLS runtime and pass the fingerprint for local loopback WSS probes, so this remains a current-code mismatch rather than an implemented cleanup candidate.

## Best Possible Solution

Keep this issue open. Implement the TUI trust handoff in `src/tui/gateway-chat.ts`: add optional `tlsFingerprint` to the resolved connection, load the local gateway TLS runtime only for configured local `wss://` targets with `gateway.tls.enabled === true` and no URL override or remote URL, pass it into `GatewayClient`, and add focused tests in `src/tui/gateway-chat.test.ts` for propagation, unavailable-runtime fallback, URL override, and remote cases.
