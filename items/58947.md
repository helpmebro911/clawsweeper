## Summary

Keep open. Current `main` still implements and documents `tools.fs.workspaceOnly` as opt-in with a default of `false`, so the proposal has not been implemented. The discussion’s note that tool profiles and FS policy are orthogonal matches the current code, but that is an argument about design tradeoffs, not evidence that the request is already resolved.

## Risks / Open Questions

- Users may reasonably assume the `coding` profile is workspace-contained, but current `main` still leaves host FS access unrestricted unless `tools.fs.workspaceOnly=true` is set explicitly.
- The existing docs frame this as optional hardening, so the security posture requested by the issue is not yet the out-of-box behavior.
