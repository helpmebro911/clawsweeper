## Summary

Keep this issue open. Current main still builds the OpenClaw tool list once for a session, passes that fixed list into the embedded agent session, and tells users to restart the Gateway after plugin install or runtime plugin changes. I found no registry-change broadcast, active-session toolset delta, or per-turn plugin tool re-resolution path that would make a tool-providing plugin usable inside an already-active agent session immediately after install.

## Best Possible Solution

Implement this in the core plugin-runtime and agent-session lifecycle: after successful plugin install or registry refresh, either notify active sessions or re-resolve plugin tools at the next turn boundary, then update customTools, session active-tool allowlists, and prompt/tool inventory before the next model call. Add regression coverage for install -> immediate use in an existing session and for registry reuse without duplicate plugin runtime bindings. The related open GatewayRestart broadcast work is adjacent but does not by itself prove this specific toolset-refresh behavior is solved.

## Risks / Open Questions

- A fix must preserve explicit tool policy, plugin-tool allowlists, optional tool gating, and owner/security restrictions when adding tools to active sessions.
- The implementation should reuse the active gateway-bindable plugin registry without duplicating plugin runtimes, HTTP routes, hooks, or channel bindings.
- The model prompt/tool inventory, session allowlist, and runtime execution surface need to refresh together at a safe turn boundary.
