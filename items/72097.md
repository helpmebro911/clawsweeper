## Summary

Keep #72097 open. Current main still builds plugin tools at embedded run/session construction and passes a fixed `customTools` set plus active-tool allowlist into the Pi session; I found no plugin-install registry-change broadcast, per-turn re-query, or active-session tool refresh path. The installer refreshes persisted registry metadata, but still tells users to restart the gateway to load plugins. Related PR #44401 is adjacent restart-sentinel work, but it does not implement live plugin-tool injection for active sessions.

## Best Possible Solution

Keep the issue open and implement a core runtime path that refreshes active agent session tool definitions after a successful plugin install or registry reload. The likely implementation belongs around the embedded runner session lifecycle: rebuild plugin tools with the same session context and policy filters, update `customTools` plus active tool names before the next model call if the session API supports it, or intentionally interrupt/restart the active run with a clear system event if live mutation is not supported. Add regression coverage for install -> registry refresh -> same active session or next model turn seeing the new plugin tool, and update plugin docs once the restart workaround is no longer required.
