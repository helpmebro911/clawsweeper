## Summary

Keep this PR open. Current main still forces every `sessionTarget: "isolated"` agent turn through `forceNew: true`, so webhook-sourced `/hooks/agent` runs with stable, policy-approved hook session keys cannot reuse a fresh existing session. The PR is also paired with open bug #70894 by the same author and links it with closing syntax, so the right maintainer path is review/land this implementation candidate or a narrow equivalent, not cleanup closure.

## Best Possible Solution

Review and land this PR or a narrow equivalent on main: compute `forceNew` from true scheduled isolated-cron semantics rather than every isolated agent turn, allow hook/webhook-sourced isolated turns with explicit hook provenance or stable hook session keys to reuse fresh existing sessions, and keep regression coverage proving ordinary isolated cron jobs still force fresh sessions.

## Risks / Open Questions

- The eventual fix must preserve `forceNew: true` for ordinary scheduled isolated cron jobs so documented fresh-per-run isolation does not regress.
- Because this changes webhook session continuity, maintainer review should confirm direct `/hooks/agent`, mapped hooks, generated/default hook session keys, and hook-derived `externalContentSource` remain bounded by existing token, agent, and prefix policies.
- The PR appears narrow and not supply-chain sensitive, but the provenance check should stay tied to real hook/webhook dispatch rather than letting unrelated scheduled cron jobs opt out of isolation accidentally.
