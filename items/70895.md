## Summary

Keep this PR open. Current main still forces every isolated cron-style agent turn through `resolveCronSession` with `forceNew: true`, while `/hooks/agent` dispatches webhook agent turns through that isolated runner with stable hook session keys and forwarded webhook provenance. The lower-level session resolver only reuses a session when `forceNew` is false, so the reported webhook session rollover remains present. This PR is also paired with open bug #70894 by the same author, so it should be reviewed or landed rather than closed as obsolete.

## Best Possible Solution

Review and land this PR or a narrow equivalent on main: compute `forceNew` from true scheduled isolated cron semantics rather than every isolated agent turn, allow hook/webhook-sourced isolated turns with explicit hook provenance or stable hook session keys to reuse fresh existing sessions, and add regression coverage for direct `/hooks/agent` plus mapped hook session keys while retaining `forceNew: true` for ordinary isolated cron runs.
