## Summary

Keep open. Current `main` already has some adjacent behavior for reply context, action bias, and push-based sub-agent completion, but the proposal is only partially covered and is not implemented as stated. In particular, the shipped memory prompt still tells agents to run `memory_search` before answering many prior-work/preference questions, which cuts against the requested retrieval-guardrail default.

## Evidence

- **Reply context is already separated and surfaced to the model:** Inbound prompt assembly sends `reply_to_id` in conversation metadata and injects a dedicated `Replied message (untrusted, for context)` block, so reply-target context exists today, but this is not the same as a proven reply-first decision policy.

## Risks / Open Questions

- The issue bundles several separate policy changes: reply prioritization, retrieval policy, progress-update policy, sub-agent orchestration, and anti-stall behavior. Any implementation would likely need to split these into smaller maintainable items.
- There is partial overlap with existing prompt/configuration surfaces, so maintainers still need a product decision about what should become a core default versus optional agent customization.
