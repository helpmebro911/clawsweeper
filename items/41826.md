## Summary

Current main does not implement custom TUI client display names. The TUI command exposes no `--client-name`, `--username`, or display-name option, `TuiOptions` has no corresponding field, and Gateway-mode TUI connections still hard-code `clientDisplayName: "openclaw-tui"`. The related display-name reports in the provided context cover Control UI/WebChat or agent names, not this TUI Gateway client identity request.

## Best Possible Solution

Keep this issue open. Add a focused TUI implementation: parse a `--client-name` or `--display-name` option in `src/cli/tui-cli.ts`, extend `TuiOptions` and `launchTuiCli` so relaunches preserve it, pass the sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, and add targeted tests plus TUI docs.

## Risks / Open Questions

- Implementation should keep the stable protocol client id as `openclaw-tui` and only customize `clientDisplayName`; changing `clientName` would risk breaking existing Gateway client-id behavior.
- Free-form display names should be trimmed and terminal/log safe before being forwarded to the Gateway or shown in presence/session UI.
