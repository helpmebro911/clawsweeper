## Summary

Keep open. Current main has some related infrastructure (a process-tree helper and SIGKILL wait fallback), but the observable request is not implemented: supervisor timeout cancellation still asks adapters to kill with SIGKILL, and the child adapter still calls child.kill("SIGKILL") immediately after invoking the graceful process-tree helper. The issue is recent, concrete, and reinforced by related open reports (#70026 and #71710), so it should not be auto-closed.

## Best Possible Solution

Keep the issue open and implement a real supervisor cancellation contract: on timeout/manual cancellation request graceful termination first, wait a bounded grace window, then force-kill and settle via a cross-platform drain timeout. The adapter API should distinguish graceful termination from force kill, remove the immediate child.kill("SIGKILL") from the graceful path, preserve killProcessTree's detached/process-group safeguards, and add regression tests for child and PTY modes covering SIGTERM-first, delayed SIGKILL escalation, POSIX drain fallback, Windows close fallback, and timeout outcome reporting.

## Risks / Open Questions

- There is already partial signal infrastructure in killProcessTree; a fix should preserve the #71662 detached:false safeguard so a child sharing the gateway process group cannot SIGTERM the gateway itself.
- Windows and PTY behavior need separate tests because Node signal semantics, taskkill behavior, close-event delivery, and PTY onExit behavior differ from POSIX child processes.
- Related open issues #70026 and #71710 overlap with this area; implementation should coordinate the exec supervisor, agent CLI, and session-lock cleanup behavior rather than fixing only one caller.
