## Summary

Current main still supports the reported failure mode: background exec failures can enqueue raw tail output as an untrusted system event, the exec-event heartbeat path includes those details in a user-relay prompt when Discord delivery is available, and Discord does not provide a narrow card-only or sanitized exec-failure delivery path. The prior ClawSweeper keep-open comment remains accurate; the related local matches cover different raw-text leaks or already-fixed Discord feedback loops, not this exec notification behavior.

## Best Possible Solution

Keep this issue open and fix it at the exec notification or heartbeat-to-channel delivery boundary. A conservative maintainer path is to keep exec card updates and intentional async completion alerts, but suppress user relay for exec notify events already represented by a card, sanitize failure output into a short user-facing summary, or add a scoped Discord exec-notification policy. Add regression coverage that proves Discord no longer receives raw `System (untrusted)`/tail-output follow-ups after exec-card failures while legitimate completion notices still work.
