## Summary

Do not close #66366. Current main still has a plausible path for the reported Discord-visible raw exec failure text: background exec failures enqueue untrusted raw-ish completion output, system-event formatting produces the exact `System (untrusted): [timestamp] ...` shape, and exec-event replies can be routed back to Discord as a normal outbound message. The existing global `tools.exec.notifyOnExit` knob can suppress the class of event, but it is not a Discord-specific delivery policy and the default behavior still allows the noisy follow-up path described by the issue.

## Best Possible Solution

Implement a targeted fix in the exec-event follow-up path: either keep Discord exec completion failures attached to native/tool update UI only, sanitize the heartbeat follow-up so `System (untrusted): ... Exec failed ... :: <raw output>` cannot be sent verbatim, or add a channel/session policy that suppresses these exec completion follow-ups for Discord without disabling all background exec notifications globally. Cover it with tests around `exec-event` routing and Discord outbound delivery.
