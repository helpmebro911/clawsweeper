## Summary

Keep #72184 open. Current main has related restart-continuation hardening, but the specific config.patch restart path still persists and delivers the caller-provided restart note as a normal outbound queued message, and generic config.patch requests explicitly do not create an agent continuation. Outbound delivery recovery still replays queued payloads verbatim, so main does not yet enforce the requested safety behavior of delivering only a reconstructed intended reply or dropping unsafe/unreconstructable content.

## Best Possible Solution

Keep the issue open and fix the config.patch/restart delivery contract. The likely product path is to stop treating arbitrary config.patch `note` text as a recoverable user-facing reply, add an explicit safe continuation/reply field for agent-initiated config writes if needed, and make recovery drop or quarantine restart-notice payloads that cannot be tied to a safe original reply. Add regression coverage for config.patch-triggered restarts and Telegram/outbound recovery so raw session/bootstrap/workspace context cannot be replayed to a channel.
