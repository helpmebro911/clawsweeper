## Summary

Keep #72121 open. Current main still validates Anthropic `paste-token` input against only `sk-ant-oat01-`, so `sk-ant-api03-...` keys deterministically hit the reported error. The separate Anthropic API-key login path exists, but it does not make the reported `paste-token` workflow accept API keys or clearly redirect from that command. The related fix PR #72124 is still open and unmerged.

## Best Possible Solution

Keep the issue open until #72124 or an equivalent fix lands. The product path should either accept `sk-ant-api03-` in the Anthropic paste-token validator with regression coverage, or make `paste-token` clearly reject API keys with an actionable redirect to `openclaw models auth login --provider anthropic --method api-key` while preserving the documented API-key setup path.
