## Summary

Keep open. Current `main` still does not implement this fix end to end: core outbound delivery preserves `audioAsVoice`, but Telegram outbound never maps that flag to Telegram's `asVoice` send option, so outbound media still falls back to regular audio/file sends instead of native voice notes.

## Evidence

- **Core delivery still carries the flag:** `deliverOutboundPayloadsCore` passes `audioAsVoice` into channel send overrides before choosing `sendPayload`/`sendMedia`, so the remaining gap is in the Telegram adapter layer, not upstream normalization.
  - file: [src/infra/outbound/deliver.ts:869](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/infra/outbound/deliver.ts#L869)

- **Telegram outbound send options omit `asVoice`:** `buildTelegramSendOptions` includes media/thread/reply/silent/forceDocument fields but has no `audioAsVoice` input and never sets Telegram's `asVoice` option.
  - file: [extensions/telegram/src/channel.ts:154](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/telegram/src/channel.ts#L154)
