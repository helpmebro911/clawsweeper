## Summary

Keep this issue open. Current main still has the reported precedence problem for `openclaw agent --agent ... --channel ... --to ...`: the CLI resolves the session key before normalizing/passing channel context, and the resolver treats `--agent` as an explicit main-session key whenever no `--session-id` is present. The gateway handler has the same fallback, so both gateway and embedded paths still route this combination to the agent main session rather than a per-recipient target session. The linked fix PR #41557 was closed unmerged, and current regression coverage only protects `--agent` plus `--session-id`, not `--agent` plus `--to`/`--channel`.

## Best Possible Solution

Keep the issue open. The best fix is to define the intended `--agent` + explicit target semantics, then update both CLI pre-resolution and gateway agent handling so explicit `--to` can derive a per-recipient session scoped to the requested agent, with channel-aware target canonicalization where available. Add regression tests for gateway and embedded paths covering `--agent chatwoot_agent --channel whatsapp --to cw_111`, distinct recipients, and `--session-id` compatibility, then update the CLI docs to match the chosen precedence.

## Risks / Open Questions

- Changing this behavior needs an explicit precedence decision for `--agent` plus `--to`, because current docs also describe `--agent` as targeting the agent main session and `--channel` as delivery-only.
- A robust fix likely needs plugin/channel-aware target normalization; the current generic `normalizeE164` fallback is phone-number-oriented and would not preserve custom ids like `cw_111` as the reporter expected.
