## Summary

Keep open. Current main still appears to ignore `--to` for `openclaw agent --agent ... --to ...` session routing: an explicit `--agent` with no `--session-id` resolves to the agent main session before the `to`-derived context key is considered. The related fix PR #41557 directly addressed this issue but was closed unmerged, so it does not supersede the open bug.

## Evidence

- **Agent main key still wins over --to:** `resolveSessionKeyForRequest` creates `explicitSessionKey` from `agentId` whenever there is no requested session id, then sets `sessionKey` to `explicitSessionKey` before falling back to the `opts.to` context. That means `--agent chatwoot_agent --to cw_111` resolves to the agent main key path instead of a recipient-specific key.
  - file: [src/agents/command/session.ts:147](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/agents/command/session.ts#L147)
  - command: `nl -ba src/agents/command/session.ts | sed -n '135,206p'`
  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Explicit agent session key is the agent main session:** `resolveExplicitAgentSessionKey` returns `resolveAgentMainSessionKey` whenever an agent id is present, so the explicit key used above is `agent:<agentId>:<mainKey>`.
  - file: [src/config/sessions/main-session.ts:36](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/config/sessions/main-session.ts#L36)
