## Summary

Keep open. Current main still contains the reported Slack inbound drop path: `resolveSlackMedia()` only uses `url_private_download` or `url_private` and returns `null` when both are missing, and the inbound message preparation path does not pass a Slack WebClient that could call `files.info`. The current tests also assert that files without private URLs are skipped. PR #50200 appears to be the exact proposed fix and PR #62792 is a broader related Slack file-ingestion fix, but both are still open and their changes are not present on main.

## Evidence
