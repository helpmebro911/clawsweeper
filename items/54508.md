## Summary

Keep this issue open. Current main still has useful external-content markers, web_fetch metadata, message-scoped input provenance, plugin approval hooks, and outbound/send hooks, but it does not implement the requested end-to-end model: persistent taint through derived content, default content-derived execution denial, exec risk classes, or provenance-aware secret and outbound enforcement. The reporter’s related-thread list is useful and matches the current architecture gap rather than making this umbrella obsolete.

## Best Possible Solution

Keep this issue open as the core security architecture umbrella unless maintainers explicitly split it into canonical implementation issues. The best path is structured provenance/taint metadata for external inputs and tool outputs, preservation through summaries and persistence, stable action/risk/provenance fields in `before_tool_call` and approval payloads, and core enforcement for content-derived exec, secret access, and outbound actions, coordinated with narrower follow-ups such as #54373, #48519, #48503, #41140, #47492, and #53185.

## Risks / Open Questions

- Closing now would drop a real core security architecture umbrella while the requested enforcement model is still absent from the tool-call, exec approval, persistence, secret, and outbound paths.
- Treating the narrower related issues as sufficient would leave cross-cutting gaps between taint propagation, risk-classed approvals, content-derived exec denial, and authorization for secret or outbound actions.
- A future fix likely needs careful public-contract design because `before_tool_call`, `message_sending`, approval payloads, transcript persistence, and plugin hook events are extension-facing surfaces.
