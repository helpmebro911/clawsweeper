## Summary

Keep this issue open. Current main still has useful marker-based external-content wrapping, web_fetch result metadata, narrow user-message provenance, plugin before_tool_call approval hooks, and transcript/tool-result hygiene, but it does not implement the requested end-to-end security model: persistent taint through derived content, default content-derived execution denial, exec risk classes, or provenance-aware secret and outbound-action enforcement.

## Best Possible Solution

Keep this issue open as the core security architecture umbrella unless maintainers split it into canonical implementation issues. The best path is structured provenance/taint metadata for external inputs and tool outputs, preservation through summaries and persistence, stable provenance/action/risk fields in before_tool_call and approval flows, and core enforcement for content-derived exec, secret access, and outbound actions, coordinated with narrower follow-ups such as #54373, #48519, #48503, #41140, #47492, and #53185.
