## Summary

Keep this issue open. Current main still does not provide a supported path to forward arbitrary inbound OpenAI-compatible HTTP headers, such as `custom-token`, into an agent run or expose them to a Python skill. Existing support covers fixed OpenClaw request-control headers and config-backed skill env/API-key injection, which does not satisfy dynamic per-request header passthrough.

## Best Possible Solution

Keep open for maintainer product/security design. If OpenClaw should support this, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, per-skill opt-in, sandbox behavior, docs, and gateway tests. If dynamic header passthrough is intentionally unsupported, document the supported alternative: fixed skill credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution is security-sensitive: it could leak operator credentials or untrusted caller data unless designed with explicit allowlists, redaction, per-skill opt-in, sandbox semantics, and tests.
- The report names `/v1/completions`, while current main exposes `/v1/chat/completions` and `/v1/responses`; maintainers should clarify the intended compatibility endpoint before implementing.
