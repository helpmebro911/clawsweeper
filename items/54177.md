## Summary

Keep open. Current main supports fixed skill env/API-key configuration and a small set of OpenAI-compatible control headers, but it still does not provide a supported way to pass arbitrary inbound `/v1/chat/completions` request headers into an agent run or expose them to a Python skill script. The related provider-header/API-key reports are outbound provider transport problems and do not supersede this inbound skill-context request.

## Best Possible Solution

Keep open for maintainer/product design. The safest implementation path is an explicit allowlist that maps selected inbound OpenAI-compatible HTTP headers into per-run context or per-skill environment with redaction, sandbox semantics, per-skill opt-in, docs, and gateway tests. If maintainers do not want dynamic secret passthrough, document that custom skill credentials must be configured statically through `skills.entries.<skill>.env` or `apiKey`.
