## Summary

Keep open. Current `main` has partial proxy support already, but not the first-class cross-surface/system-custom proxy support requested here. OpenClaw bootstraps env-proxy support for some HTTP paths and supports `request.proxy` on model providers plus a few channel-specific proxy knobs, but `web_fetch` still deliberately avoids env-proxy routing for ordinary untrusted URLs, browser strict mode blocks env-proxy navigation, and agent-level proxy env overrides are intentionally blocked by host env security.

## Evidence

- **CLI already bootstraps env-proxy support:** `main` enables Undici env-proxy bootstrap at CLI startup, so standard host proxy env vars are already partially supported today.
  - file: [src/cli/run-main.ts:187](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cli/run-main.ts#L187)
- **Model providers already have first-class proxy config:** The config schema already accepts `models.providers.*.request.proxy` in `env-proxy` and `explicit-proxy` modes, and provider request transport builds matching dispatcher policies from that config.
  - file: [src/config/zod-schema.core.ts:270](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/config/zod-schema.core.ts#L270)
- **web_fetch still avoids env-proxy for ordinary URLs:** The regression test on `main` explicitly asserts that `HTTP_PROXY` must not switch untrusted `web_fetch` traffic onto `EnvHttpProxyAgent`, which means the requested broad web/tool proxy behavior is not implemented end-to-end.
  - file: [src/agents/tools/web-tools.fetch.test.ts:256](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/tools/web-tools.fetch.test.ts#L256)
- **Browser strict mode still blocks proxy-env navigation:** Browser navigation fails closed when proxy env vars are set and private-network browser navigation is not explicitly allowed, so browser automation is not first-class proxy-aware in the way the issue requests.
  - file: [extensions/browser/src/browser/navigation-guard.ts:112](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/browser/src/browser/navigation-guard.ts#L112)
- **Per-agent proxy env overrides are intentionally blocked:** Host env security policy marks `HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`, and `NO_PROXY` as blocked override keys, which directly conflicts with the issue's proposed per-agent/per-tool env override model.
  - file: [src/infra/host-env-security-policy.json:254](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/host-env-security-policy.json#L254)
- **Proxy support is still surface-specific:** OpenClaw already documents explicit channel-level proxy knobs such as `channels.matrix.proxy`, which reinforces that proxy support exists today in scattered surfaces rather than as one global OpenClaw feature.
  - file: [docs/gateway/config-channels.md:651](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/gateway/config-channels.md#L651)

## Risks / Open Questions

- Closing as implemented would overstate the current state and hide remaining work on web tools, browser automation, and agent/tool-level override behavior.
- The feature cuts across network policy and SSRF boundaries, so any future closure should verify both usability and the security model rather than only checking for isolated proxy knobs.
