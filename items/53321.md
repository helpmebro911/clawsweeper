## Summary

Keep open. Current main has useful partial mitigations: preflight compaction, silent pre-compaction memory flush, and opt-in transcript rotation after compaction. It still does not implement the requested proactive auto session split policy with configurable context thresholds, default rollover behavior, active tool-call boundary semantics, or continuity payloads carrying task/pending-confirmation state. The duplicate #53330 was closed in favor of this issue, so this remains the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical product/design tracker. Build on the existing memory-flush, preflight compaction, queued-followup refresh, and `truncateAfterCompaction` machinery, then add a supported auto-split policy with threshold config, documented defaults/notifications, active tool-call boundaries, session-key routing tests, and continuity payload handling for summary plus task or pending-confirmation state.

## Risks / Open Questions

- `truncateAfterCompaction` may satisfy some deployments, so maintainers should decide whether the product answer is a distinct `session.autoSplit` feature or a smaller supported preset/docs path around compaction rotation.
- The requested active-task and pending-confirmation carry-over semantics need explicit product and API design; they should not be inferred from current compaction summaries alone.
