## Summary

Current main still omits Telegram video width/height in both outbound video paths, so this PR is not obsolete or implemented. The PR is a plausible bug fix with concrete reproduction, but it should stay open because the submitted patch crosses the plugin boundary, adds a bare ffprobe execution path, changes an unrelated global media limit, and lacks regression coverage for the requested behavior.

## Best Possible Solution

Keep this PR open and request a focused revision: add a Telegram-local helper that uses `openclaw/plugin-sdk/media-runtime` `runFfprobe`, or promote a narrow public media-runtime helper for probing video dimensions; pass width and height only for regular Telegram `sendVideo` calls in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts`; keep GIF animations and video notes excluded; add regression tests for both send paths; and split or remove the unrelated global `MAX_VIDEO_BYTES` change.

## Risks / Open Questions

- Merging as-is would add a new child-process execution path that resolves ffprobe from PATH instead of the repo's trusted system-binary resolver.
- Merging as-is would add relative core `src/**` imports from Telegram plugin production code, which violates the plugin boundary used for bundled and third-party plugins.
- The unrelated global `MAX_VIDEO_BYTES` increase could alter non-Telegram media behavior and should be split or dropped from this bug fix.
- Without regression tests for both Telegram send paths, the width/height behavior can regress during later media or Telegram refactors.
