## Summary

Keep this PR open. Current main still omits width/height on both regular Telegram sendVideo paths, so the reported bug is not implemented on main. The submitted patch is still useful directionally, but should not be merged as-is because it crosses the bundled-plugin boundary, adds bare ffprobe execution instead of the trusted media-runtime helper path, includes an unrelated global media limit change, and lacks regression coverage for the requested behavior.

## Best Possible Solution

Keep this PR open and request a focused revision: add or expose a video-dimension helper through openclaw/plugin-sdk/media-runtime using the trusted ffprobe resolver, pass width and height only for regular Telegram sendVideo calls in extensions/telegram/src/send.ts and extensions/telegram/src/bot/delivery.replies.ts, keep GIF animations and video notes excluded, add regression tests for both outbound paths, and remove or split the unrelated MAX_VIDEO_BYTES change.
