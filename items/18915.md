## Summary

Keep #18915 open. Current main still omits `width`/`height` when sending regular Telegram videos in both direct-send and reply-delivery paths, and no shipped video-dimension probing helper or regression test exists. The PR addresses a plausible reproduced Telegram bug, but its current diff is not merge-ready because it crosses the plugin boundary with relative imports into core `src/**` and bundles an unrelated global video-size change.

## Best Possible Solution

Keep the PR open and request a focused revision: add a boundary-compliant video-dimension probe through `openclaw/plugin-sdk/media-runtime` or a Telegram-local runtime seam, forward `width` and `height` in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts` for regular videos only, keep video notes and GIF animations excluded, add regression tests for both send paths, and split or drop the unrelated global `MAX_VIDEO_BYTES` change.
