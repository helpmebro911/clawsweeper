## Summary

Keep PR #39487 open. Current main still does not implement the Feishu-specific liveness/status plumbing proposed by the PR: Feishu only publishes the gateway port at startup, and no status callback is threaded through monitor startup, event handlers, or transport lifecycle paths. Current main does include a conservative generic health-policy mitigation that avoids stale-socket restarts when transport liveness is absent, but that does not make this PR implemented or clearly superseded. The overlapping #41940 remains open context, not a merged or maintainer-selected canonical replacement.

## Best Possible Solution

Keep #39487 open for Feishu plugin follow-up. The best path is to thread the gateway `setStatus` callback through `extensions/feishu/src/channel.ts`, `monitor.ts`, `monitor.account.ts`, and `monitor.transport.ts`; report `connected` only after WebSocket start or webhook listen succeeds; report disconnect/error state on abort, startup failure, runtime failure, and early-abort paths; update inbound event liveness for meaningful Feishu events; publish transport activity when Feishu transport activity proves socket liveness; and add focused lifecycle plus inbound regression tests. If #41940 is later merged or explicitly selected as canonical, #39487 can then close as duplicate/superseded.
