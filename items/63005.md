## Summary

Current main still has the raw-vs-agent-canonical session-key mismatch described here. Explicit inbound `SessionKey` values can remain as raw structured keys, while gateway/session tool paths canonicalize the same key into an `agent:<defaultAgent>:...` store key and prune non-canonical candidates. The related closed WebUI UUID issue is narrower and does not cover arbitrary structured external keys like `conversation:pair:...`.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make inbound explicit-key initialization and gateway `resolveSessionStoreKey` / `resolveGatewaySessionStoreTarget` share one store-key contract, or preserve raw external aliases that future inbound traffic still depends on. Add a regression test that creates a raw key like `conversation:pair:user_a::user_b:space:space_123`, exercises a gateway `sessions.resolve` or `sessions.send` path, and verifies later inbound traffic with the same raw key reuses the original `sessionId` and transcript instead of minting a new session.

## Risks / Open Questions

- External integrations that keep sending stable structured raw session keys can still lose continuity after a gateway resolve/send/patch/delete/compact path migrates the same logical session to an agent-prefixed key and prunes the raw alias.
- Treating the closed WebUI UUID issue as covering this would miss non-UUID structured keys, which current gateway tests still canonicalize to agent-scoped keys.
