## Summary

Keep #70575 open. Current main still builds the agent session patch from a potentially cached `entry` and includes stale fallback values for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`; `updateSessionStore` then re-reads the fresh store but merges that stale patch over it. The linked PR #19328 is still open/unmerged and its current diff leaves the exact residual delivery/channel patterns described here.

## Best Possible Solution

Keep the issue open and either fold the fix into #19328 or land a follow-up that makes the agent handler patch contain only request-derived values. Use a `Partial<SessionEntry>`, omit cached-entry fallback fields such as `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`, or compute those fallbacks from the fresh store entry inside the `updateSessionStore` mutator. Add a regression test modeled on the #5369 stale-cache/fresh-store pattern that proves fresh delivery/channel fields survive.
