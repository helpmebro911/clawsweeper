## Summary

Keep #70575 open. Current main still constructs `nextEntryPatch` from the cached `entry` before the locked store reread, and then merges that patch over the fresh store entry. The residual `lastChannel`/`lastTo`/`lastAccountId`/`lastThreadId` and `channel` writebacks described in the issue are still present, and the requested stale-cache/fresh-store regression coverage is not present on current main. The related #19328 is an open, unmerged PR for the earlier #5369 model override race and does not supersede this narrower residual bug.

## Best Possible Solution

Fix `src/gateway/server-methods/agent.ts` so fields that can be concurrently patched are derived from the fresh `store[primaryKey]` inside the `updateSessionStore` mutator, or only include request/effective delivery fields in `nextEntryPatch` when they are actually defined. Add focused regression tests that simulate a stale cached `entry` plus a fresh store entry for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`.
