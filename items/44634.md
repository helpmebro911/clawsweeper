## Summary

Issue #44634's reported stale-socket restart path is already addressed on current main and was shipped in v2026.4.22: gateway stale-socket health now keys off provider-proven transport activity instead of inbound app-event freshness, which prevents quiet Slack DMs from being restarted just because no user traffic arrived.

## Close Comment

Closing this as implemented on current `main` after Codex review. The stale-socket path described here has been changed so gateway health no longer treats stale inbound app events as proof of a dead socket: `src/gateway/channel-health-policy.ts:115` now only uses provider-proven `lastTransportActivityAt` for stale-socket detection, and the regression test at `src/gateway/channel-health-policy.test.ts:139` locks in that stale app events without transport activity stay healthy. Slack inbound handling still only bumps `lastEventAt`/`lastInboundAt` (`extensions/slack/src/monitor/provider.ts:292`), so the old ~35 minute Slack restart loop from quiet DM traffic should not reproduce on current main. This shipped in `v2026.4.22` (`CHANGELOG.md:266`, release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce on `2026.4.22` or newer, please open a new issue with current logs and channel status snapshots, because that would point to a different Slack turn-trigger problem.
