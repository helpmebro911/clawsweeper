## Summary

Keep this issue open. Current main still escapes ampersands in the Slack mrkdwn formatter and routes normal outbound Slack text through that formatter before `chat.postMessage`; the linked fix PRs were closed unmerged after discussion questioned removing ampersand escaping. Slack’s own formatting contract supports exactly-one escaping, so the remaining work is to reproduce where the visible `&#38;` is introduced rather than close this as fixed. ([docs.slack.dev](https://docs.slack.dev/messaging/formatting-message-text/?utm_source=openai))

## Best Possible Solution

Keep the issue open and add a focused Slack regression around raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` through normal send, draft preview final edit, native streaming `markdown_text`, and block fallback paths. Fix the boundary that introduces the second entity pass so Slack receives exactly one valid escaping pass, without disabling required Slack mrkdwn escaping globally.
