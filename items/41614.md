## Summary

Keep open. Current main still reuses the in-memory cron store for normal service operations, so a live Gateway can serve stale cron data until a force-reload path runs. The PR is still aimed at a real gap, although it needs refresh for the current split `jobs.json` / `jobs-state.json` store and should address the reviewed mtime ordering concern before merge.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for an updated patch against current main: invalidate the cached cron store for normal live operations when either the definition store or runtime state sidecar changes, handle non-file paths, place mtime capture so a concurrent write cannot permanently pin stale contents, and keep regression coverage for external deletion/list/run behavior plus sidecar runtime-state recovery.
