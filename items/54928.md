## Summary

Current main still does not implement the requested Exa `webSearch.baseUrl` support or provider-specific unsupported-field guidance. The Exa manifest remains strict and only accepts `apiKey`, runtime still posts to the fixed Exa endpoint, docs still document only the API key path, and open PR #54939 is an unmerged implementation candidate that explicitly targets this issue.

## Best Possible Solution

Keep this issue open and review PR #54939 as the implementation candidate. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to the manifest schema and UI hints, resolve and validate it in the Exa runtime, pass it through the trusted web-search endpoint path, include the resolved endpoint in cache-key behavior, update Exa/web-search docs, and add tests proving the field is accepted and used. If maintainers decide Exa should not support custom endpoints, document that provider capability difference and improve validation guidance instead.

## Risks / Open Questions

- Adding configurable Exa endpoints is security-sensitive outbound behavior: the implementation should preserve guarded fetch behavior and avoid accidentally sending Exa API keys to unintended URLs without deliberate validation.
- If `baseUrl` is added, the resolved endpoint should be included in the search cache key so responses from different proxies or gateways cannot share cached payloads.
