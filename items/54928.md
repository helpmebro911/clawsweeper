## Summary

Keep this issue open. Current main still does not support `plugins.entries.exa.config.webSearch.baseUrl`: the Exa manifest schema only allows `apiKey`, validation still reports strict schema failures, the runtime config only reads `apiKey`, and Exa requests are sent to the hard-coded default endpoint. The linked implementation PR #54939 is open and unmerged with closing syntax for this issue, so it should be reviewed/landed or closed before the issue is resolved.

## Best Possible Solution

Keep this issue open and review #54939 as the implementation candidate. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json` and UI hints, plumb it through `extensions/exa/src/exa-web-search-provider.runtime.ts`, include the endpoint in cache-key behavior, document it in the Exa and web-search docs, and add Exa plugin coverage proving config validation accepts `baseUrl` and runtime requests use it. If maintainers decide Exa should intentionally not support custom endpoints, document that explicitly and improve the provider-specific validation guidance instead of leaving the generic additional-properties error.
