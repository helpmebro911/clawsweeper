## Summary

Keep this issue open. Current main still does not support `plugins.entries.exa.config.webSearch.baseUrl`, and open PR #54939 is an explicit implementation candidate with closing syntax, so the issue should remain open until that PR is reviewed, landed, or closed.

## Best Possible Solution

Keep this issue open and review PR #54939 as the implementation candidate. The fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json` and UI hints, plumb it through `extensions/exa/src/exa-web-search-provider.runtime.ts`, preserve trusted-endpoint validation, include the resolved endpoint in cache-key behavior, document it on `https://docs.openclaw.ai/tools/exa-search` and `https://docs.openclaw.ai/tools/web`, and add Exa plugin tests proving config validation accepts `baseUrl` and runtime requests use it. If maintainers intentionally do not want custom Exa endpoints, document the provider capability difference and return provider-specific validation guidance instead.

## Risks / Open Questions

- Adding configurable Exa endpoints is security-sensitive outbound behavior; the implementation should keep the request on the existing trusted web-search endpoint path and reject unsafe URLs consistently with other providers.
- If Exa `baseUrl` is implemented, the resolved endpoint should be part of the cache key so results from different proxies or gateways cannot share cached payloads.
