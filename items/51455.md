## Summary

Keep open. Current main still routes PDF fallback rendering through the bundled document-extract plugin, and that active pdf.js getDocument call passes only data and disableWorker, with no standardFontDataUrl. The issue is real and currently has an open focused replacement PR (#70936), but it is not merged or shipped yet.

## Best Possible Solution

Keep #51455 open until the focused fix lands. The best code path is to review or adapt PR #70936: resolve the installed pdfjs-dist package root, pass a trailing-slash standard_fonts path as standardFontDataUrl in extensions/document-extract/document-extractor.ts, update the local PdfJsModule/test shim type, and add a regression test asserting the forwarded path exists and no standardFontDataUrl warning path is left unconfigured.
