## Summary

Keep open. Current main still queues follow-up messages while an active run continues, then schedules the queued follow-up drain while still returning and dispatching the original final reply. The generic dispatcher has a cancellable `beforeDeliver` hook, but current default and Telegram wiring do not add a session queue-depth/pending-message suppression check. The existing Telegram stale-reply suppression appears abort-fence specific, not the collect-mode pending-correction behavior requested in #48814. This is core queue/delivery behavior across channels, not a ClawHub/plugin cleanup close.
