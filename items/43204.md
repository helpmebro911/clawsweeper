## Summary

Keep this PR open. Current main still silently cleans up when the typing TTL expires and exposes no TTL-expiry feedback hook, so the core UX problem remains. The PR is useful as a reference, but it should not land as-is because the head is stale/mixed-scope and still has an uncontained synchronous-throw path in the custom TTL handler wrapper.

## Best Possible Solution

Keep this PR open as a useful reference implementation, but do not land it as-is. The maintainable path is a fresh narrow auto-reply change against current main that adds the chosen TTL-expiry feedback contract, contains both synchronous throws and async rejections from custom handlers, preserves routed-final ordering, adds focused tests for TTL expiry/suppressed typing/routed replies, and drops or separately rebases the unrelated macOS and Tlon changes. Coordinate with #40760; if that lands an equivalent user-visible solution, this PR can then be closed as superseded.

## Risks / Open Questions

- Landing this PR as-is could leave custom `onTypingTtlExpired` handlers able to crash or surface as uncaught timer errors when they throw synchronously.
- The PR head mixes the typing UX change with unrelated macOS remote-connection UI/probe work and stale Tlon dependency/lockfile changes, including package-resolution changes outside the stated fix.
- The product direction overlaps with #40760, so maintainers should choose whether the durable solution is TTL-expiry feedback, longer timeout-aware TTL budgeting, explicit timeout final replies, or a combination.
