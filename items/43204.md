## Summary

Keep this PR open for maintainer judgment/reference. The underlying user-visible TTL-expiry gap is still not implemented on current main: the typing TTL still logs and cleans up without a caller notification hook, and the public reply options still lack `onTypingTtlExpired`. The PR is plausible but not landable as-is because it mixes the intended auto-reply change with unrelated macOS remote-connection changes and a Tlon GitHub dependency bump, and its custom handler wrapper still lets synchronous throws escape.

## Best Possible Solution

Keep this PR open only as reference for the TTL-expiry feedback approach. The maintainable path is a fresh, narrow auto-reply PR against current main that either implements a well-tested TTL-expiry notification contract or reconciles with the adjacent keepalive/configurable-TTL work. It should drop the macOS and Tlon dependency changes, update the current `GetReplyOptions` surface, contain both synchronous throws and async rejections from custom handlers, and add focused tests for TTL expiry, suppressed typing, routed replies, and final-reply ordering.

## Risks / Open Questions

- Landing this PR as-is would mix a narrow auto-reply UX fix with unrelated macOS remote connection behavior and a GitHub-sourced Tlon dependency bump, both of which require their own review paths.
- The PR's custom TTL handler wrapper still does not contain synchronous throws, so a failing handler can escape the timeout callback even though cleanup runs in `typing.ts`.
- The product direction is still unsettled: a generic block message at TTL expiry may be noisier than keeping the typing indicator alive, making TTL configurable, or using existing channel-specific streaming/status affordances.
