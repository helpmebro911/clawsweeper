## Summary

Keep this issue open. Current main has substantial ACPX/runtime work since the report, including bundled acpx 0.6.1, ACP child dispatch on the subagent lane, runtime timeouts, and dead-handle recovery. I did not find a current source/test/release-history proof that the specific rapid concurrent ACP spawn race is fixed. The ACP manager still intentionally allows different ACP sessions to run in parallel, so a backend concurrent-init race remains actionable if it reproduces.

## Best Possible Solution

Keep this open until a maintainer or contributor verifies current main with the bundled acpx 0.6.1+ using two rapid `sessions_spawn(runtime:"acp", agentId:"claude")` calls. If it still reproduces, add a focused regression around concurrent ACP spawn initialization/run dispatch for distinct session keys and fix the ACPX runtime/control-plane launch path, possibly with a backend-level init guard or retry if the acpx adapter requires it. If current acpx no longer reproduces, close later with live proof and the exact fixing SHA/release.
