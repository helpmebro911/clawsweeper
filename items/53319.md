## Summary

Keep this issue open. The report is concrete and recent, and current main still has no source, test, doc, release-note, or live-smoke evidence proving that two rapid Claude Code ACP `sessions_spawn` calls both launch their underlying ACPX/Claude process. Current main has adjacent ACPX, completion, timeout, and retry improvements, but those do not close the specific first-spawn-never-starts failure mode.

## Best Possible Solution

Keep this open until current main is verified with the bundled ACPX runtime, preferably `acpx` 0.6.1 or newer, using two rapid `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` calls. If it still reproduces, fix the ACPX/OpenClaw launch path with a backend-level init guard, startup health check, or retry that covers silent launch hangs, then add focused regression coverage proving both distinct ACP sessions launch. If it no longer reproduces, close later with the exact live-smoke evidence and fixing commit or release.

## Evidence

- **Issue context describes a distinct launch failure:** The provided issue body reports two accepted ACP spawns 37 seconds apart on OpenClaw 2026.3.22/acpx 0.1.16. The first emitted lifecycle start and stall but no `assistant_delta` and no matching Claude process; the second eventually produced output. That is an ACPX/Claude launch/init failure, not just parent completion relay loss. The existing review comment in the thread also kept this open for lack of focused rapid two-spawn proof.

This is still the surface where accepted can precede observable external harness output.
