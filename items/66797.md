## Summary

Keep open. The report matches current `main`: natural-language Telegram group turns on a non-default account are still dropped when there is no explicit routing binding, while native slash commands take a separate path and can still succeed. That reproduces the issue’s core symptom pattern from source inspection, so this is not safe to auto-close.

## Evidence

- **Named-account group messages are intentionally dropped on the normal message path:** `buildTelegramMessageContext` treats a non-default Telegram account routed only by the default fallback as invalid for groups and returns `null` before dispatch, with reason `non-default account requires explicit binding`. The issue config uses `accounts.rasoi` with `bindings` unset, which matches this guard.
  - file: [extensions/telegram/src/bot-message-context.ts:236](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/telegram/src/bot-message-context.ts#L236)
  - command: `nl -ba extensions/telegram/src/bot-message-context.ts | sed -n '222,255p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current tests codify the drop behavior:** The Telegram test suite explicitly asserts that a named-account group message without an explicit binding returns `null` and is dropped.
  - file: [extensions/telegram/src/bot-message-context.named-account-dm.test-support.ts:123](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/telegram/src/bot-message-context.named-account-dm.test-support.ts#L123)
  - command: `nl -ba extensions/telegram/src/bot-message-context.named-account-dm.test-support.ts | sed -n '110,140p'`

- **Native slash commands use a different path and do not apply that guard:** `resolveCommandRuntimeContext` resolves the Telegram conversation route for native commands, but it does not reject named-account group fallback routes. The native-command flow then records slash-session metadata and proceeds, which fits the reporter’s observation that `/whoami` works while plain text does not.
