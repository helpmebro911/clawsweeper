## Summary

Keep #72037 open. Current main has moved the bundled ACPX plugin to `acpx@0.6.1` and adds an OpenClaw Codex wrapper, but the observable problem remains: the default Codex ACP path still shells through npm/npx to fetch `@zed-industries/codex-acp`, and the bundled plugin dependency graph does not include that package. The reporter’s related note about #60693 is relevant because that issue asks for the same packaging direction; #71549 was closed unmerged and would only have bumped ACPX to 0.6.0, not proven bundled Codex ACP binaries.

## Best Possible Solution

Keep the issue open and fix the bundled ACPX/Codex adapter packaging path. The likely product-quality fix is to make the OpenClaw ACPX plugin or upstream `acpx` resolve a plugin-local bundled `@zed-industries/codex-acp` binary before any npm/npx fallback, stage the required platform packages in bundled runtime dependencies, and add a regression test proving the default Codex ACP command does not invoke `npm exec`/`npx`. The related #60693 packaging recommendation should be considered supporting evidence; #71549 is not a sufficient superseding item because it closed unmerged and did not prove the no-npx behavior.
