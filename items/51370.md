## Summary

Current main already addresses this Discord startup-outage report well enough: transient gateway metadata failures no longer leave startup dead, the initial gateway READY phase is actively watched and restarted once if needed, the misleading pre-READY login log is fixed, and the shipped changelog for v2026.4.22 includes the Discord startup logging fix.

## Evidence

- **Transient gateway metadata errors are now recoverable:** `fetchDiscordGatewayInfo` marks transport/body-read/5xx-style gateway metadata failures as transient `fetch failed`, and `resolveGatewayInfoWithFallback` converts those into a default gateway URL instead of leaving startup stuck on the initial metadata request.

  - file: [extensions/discord/src/monitor/gateway-plugin.ts:274](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/discord/src/monitor/gateway-plugin.ts#L274)

## Close Comment

Closing from a Codex review. Current `main` already covers this well enough: transient Discord gateway metadata failures now fall back to the default gateway URL during startup instead of leaving the plugin in a dead state (`extensions/discord/src/monitor/gateway-plugin.ts:76`, `extensions/discord/src/monitor/gateway-plugin.ts:274`), and the initial gateway startup path now watches for READY and restarts the gateway once if it never becomes ready (`extensions/discord/src/monitor/provider.lifecycle.ts:276`, with regression coverage in `extensions/discord/src/monitor/provider.lifecycle.test.ts:218`). The misleading pre-READY `logged in to discord` message was also corrected so startup now reports `discord client initialized ...; awaiting gateway readiness` until the gateway is actually ready (`extensions/discord/src/monitor/startup-status.ts:1`). This logging fix is recorded in the shipped `v2026.4.22` release notes (`CHANGELOG.md:2486`, release tag `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
