## Summary

Keep open. Current main still runs Tailscale Serve/Funnel and immediately performs a single `tailscale status --json` hostname lookup with no bounded retry/backoff, and verbose `runExec` can still emit the noisy `Command failed: tailscale status --json` line before the caller catches the failure. The alternate `gateway.tailscale.binaryPath` config is also not implemented.

## Best Possible Solution

Add a bounded retry/backoff around the post-Serve/Funnel Tailscale status lookup, preferably in the shared Tailscale hostname/status helper or the startup hostname path, and add regression coverage where the first `tailscale status --json` fails and the retry succeeds without surfacing a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate schema/docs-backed enhancement if maintainers want that operator control.
