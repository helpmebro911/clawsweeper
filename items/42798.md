## Summary

Keep open. Current main still enables Tailscale Serve/Funnel and immediately performs a single `tailscale status --json` hostname lookup with no retry/backoff, while verbose exec logging still emits the confusing failure before the caller suppresses the thrown error. The suggested `gateway.tailscale.binaryPath` operator config also is not present.

## Best Possible Solution

Add bounded retry/backoff around the post-Serve/Funnel Tailscale hostname/status lookup, preferably in the shared Tailscale status helper or the gateway startup hostname path. Add regression coverage where the first `tailscale status --json` call fails and a later retry succeeds without emitting a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate schema, docs, and config enhancement if maintainers want explicit operator control.
