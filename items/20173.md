## Summary

Keep open. Current `main` does not implement Discord inbound edit re-processing: the Discord monitor only registers a `MessageCreateListener`, there is no `MESSAGE_UPDATE`/`messageUpdate` handling or `handleEdits` config surface, and the inbound replay guard dedupes by Discord message id, which would suppress a second delivery for the same message unless that logic changes too.

## Evidence

- **Discord startup registers no edit listener:** `registerDiscordMonitorListeners` adds `DiscordMessageListener`, reaction listeners, `DiscordThreadUpdateListener`, and optional presence listener. There is no message-update listener registration path.
  - file: [extensions/discord/src/monitor/provider.startup.ts:234](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/extensions/discord/src/monitor/provider.startup.ts#L234)

  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Inbound message listener is create-only:** `DiscordMessageListener` extends `MessageCreateListener`; the monitor listener layer imports `MessageCreateListener`, `MessageReaction*Listener`, `PresenceUpdateListener`, and `ThreadUpdateListener`, but no message-update listener type.
  - file: [extensions/discord/src/monitor/listeners.ts:1](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/extensions/discord/src/monitor/listeners.ts#L1)

## Risks / Open Questions

- If this feature is implemented later, replay dedupe will need a versioned key or edit-aware bypass; adding only a gateway listener would likely be insufficient.
- Edited-message handling also needs product decisions around canceling or superseding in-flight replies, because current inbound processing assumes one turn per message id.
