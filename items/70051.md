## Summary

Keep PR #70051 open. Current main still handles node `exec.started` / `exec.finished` / `exec.denied` by canonicalizing the session key, enqueueing a system event, and requesting an `exec-event` heartbeat wake without checking whether the session entry is a synthetic heartbeat-isolated session. The heartbeat runner does mark those synthetic sessions with `heartbeatIsolatedBaseSessionKey`, so the PR's proposed guard targets a real current path. Related #59606 and #67059 cover nearby heartbeat behavior but do not supersede this narrow gateway node-event suppression.

## Best Possible Solution

Keep the PR open for maintainer review or integrate the same narrow fix on main: load the full session entry in `src/gateway/server-node-events.ts`, return before enqueue/wake when `entry.heartbeatIsolatedBaseSessionKey` is present, and add focused regression coverage in `src/gateway/server-node-events.test.ts` proving synthetic heartbeat-isolated exec events do not enqueue or self-wake while normal exec notifications remain unchanged.
