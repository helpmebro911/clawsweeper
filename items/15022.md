## Summary

Keep #15022 open. Current main still does not provide the requested opt-in coalescing/final-only mode for tool-interleaved assistant text: final payload assembly preserves multiple assistant text blocks as multiple payloads, dispatch sends each payload separately, block streaming still flushes before tool execution, and the hold/resume API proposed in PR #16275 is absent from main.

## Best Possible Solution

Keep #15022 open as the focused core reply-pipeline issue. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around tool execution, likely using hold/defer semantics for sequential and parallel tool calls, with config schema/docs and regression tests for text -> tool -> text -> tool -> text turns. PR #16275 remains useful prior art but is closed unmerged and not implemented on current main.
