## Summary

Keep #15022 open. Current main still lacks the requested opt-in coalescing/final-only mode for assistant text blocks interleaved with tool calls. The closed PR #16275 remains relevant prior art, but its hold/resume approach is not present on main, and current tests still assert the opposite behavior: flushing block replies before tool execution.

## Best Possible Solution

Keep #15022 open as the focused core reply-pipeline issue. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, likely with hold/defer semantics, config schema/docs, and regression tests for text -> tool -> text -> tool -> text turns. PR #16275 should be treated as useful prior art, not as a shipped fix.
