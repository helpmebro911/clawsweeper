## Summary

Keep #71744 open. Current main has structured model-fallback diagnostic logs and per-attempt `FallbackSummaryError` data, but it still does not emit a first-class fallback-step runtime/trajectory event carrying the primary failure reason/detail when the fallback chain advances or exhausts. Related #62684 is closed and covers a different Ollama timeout symptom; #71710 remains a separate host/process leak root cause, not a duplicate of this observability gap.

## Best Possible Solution

Keep the issue open and implement the observability gap in core. Add a sanitized per-candidate fallback-step runtime/trajectory event from the `runWithModelFallback` path, carrying from/to provider-model refs, reason/status/code, bounded error detail, attempt/total, and final outcome. Add regression coverage for primary failure followed by fallback failure so both the primary root cause and terminal fallback error are present in the structured runtime event stream.
