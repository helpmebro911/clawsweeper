## Summary

Keep #71744 open. Current main already has useful fallback attempt data in `FallbackSummaryError` and structured `model_fallback_decision` log records, but it still does not emit a persisted `fallback_step`/runtime-event record for each fallback transition or exhausted chain. The related #62684 and #71710 reports are adjacent incidents, not canonical replacements for this missing fallback-chain observability.

## Best Possible Solution

Keep the issue open and implement core-owned, sanitized fallback-step observability in the existing model fallback path. Emit a structured event when moving from one candidate to the next and when the chain is exhausted, with from/to provider and model, classified reason, truncated detail, chain position, final outcome, and run/session correlation. Add focused tests for primary failure followed by fallback failure, and expose the event through the documented runtime-event/trajectory or agent-event surface without requiring verbose user-facing replies.

## Risks / Open Questions

- A fix should choose the canonical sink deliberately: OpenClaw currently has diagnostic logs, trajectory events, agent lifecycle events, and plugin hooks, but no first-class `runtime_events.jsonl` writer in this checkout.
- Fallback-step details need the same redaction/truncation treatment as existing trajectory and log records so provider errors do not leak secrets.
