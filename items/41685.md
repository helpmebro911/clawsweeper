## Summary

Current main only partially mitigates deleted-agent fallout. It purges deleted-agent store entries and blocks targeting deleted-agent sessions, but the Gateway/Control UI session listing path still aggregates disk-only agent stores, and the UI still preserves explicit session keys from URL/browser state. The closed, unmerged #41687 was the direct proposed fix for the reported listing behavior, so this issue should stay open unless maintainers intentionally want the documented disk-only/ACP behavior to supersede it.

## Best Possible Solution

Keep this issue open and reconcile the product boundary: make the Control UI/default sessions.list view ignore unknown deleted-agent namespaces, or add an explicit includeDiskOnly/includeAcp-style path for intentional ACP/disk-only stores. Also normalize or reject URL/browser-restored session keys whose agent id is not configured, falling back to sessionDefaults.mainSessionKey, while preserving the existing deleted-agent purge and send/resolve guards.
