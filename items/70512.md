## Summary

Current main still does not implement Feishu/Lark DM topic-based session isolation. The Feishu plugin parses `root_id` and `thread_id`, but only group routing uses those fields for session keys; direct messages still route by sender open ID, and the config/docs/tests only expose group topic session scope. The related #10242 context is distinct because it concerns DM thread creation/binding behavior, not inbound DM topic session isolation.

## Best Possible Solution

Keep this issue open. Implement the feature inside the bundled Feishu plugin by adding an explicit DM topic/session-scope config, deriving an opt-in direct-message route/session id from sender plus the canonical DM topic/root/thread id, propagating thread context consistently for DM topics, preserving current sender-scoped DMs by default, and adding focused Feishu p2p/private tests plus config schema/docs updates.

## Risks / Open Questions

- Feishu/Lark DM topic event semantics need verification before choosing canonical precedence between `root_id` and `thread_id`.

- The existing #10242-related DM thread work does not supersede this inbound routing request, so closing now would leave the reported workflow untracked.
