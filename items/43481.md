## Summary

Keep this issue open. Current main still lacks provider-level fallback trust configuration for image or attachment data, still strips explicit images with a coarse fallback-retry boolean rather than a provider-boundary policy, and still loads prompt-detected local images without a fallback trust guard. The provided context also shows open same-author PR #57959 with closing syntax for this issue, so the issue should remain open until that implementation candidate is reviewed, landed, or closed.

## Best Possible Solution

Keep this issue open while maintainers review, revise, land, or close PR #57959. The best implementation path is a focused fallback-boundary change that passes primary and target provider context through runWithModelFallback and runAgentAttempt, centralizes a tested image/attachment trust decision, suppresses prompt-detected image loading when the fallback target is not trusted, and documents any user-facing trust configuration if maintainers choose configurable behavior rather than conservative cross-provider stripping.
