## Summary

Keep this PR open. Current main already includes the Feishu schema part, but the PR is not obsolete: gateway health still builds snapshots from the pinned runtime config, while current main now has a fresh `readBestEffortConfig()` helper that makes the health hunk salvageable after rebase. The provider-normalization and skill-symlink hunks should not land as-is because they conflict with current provider/auth and path-containment contracts.

## Best Possible Solution

Keep this PR open, but ask for a rebase and split. Drop the already-shipped Feishu hunk; implement the gateway health fix against current `readBestEffortConfig()` with a regression test for stale runtime snapshots; leave plan-provider identity distinct and use manifest auth aliases for shared auth; and handle any skill symlink change as a separate security-sensitive path-canonicalization PR with targeted tests.

- The health fix needs a regression test proving gateway health reads fresh config after the runtime snapshot is pinned.
- The PR bundles four independent fixes; landing it as-is would mix an already-shipped Feishu change with unrelated health, provider, and path-security changes.
