## Summary

Keep #35835 open. Current main still does not implement native audio attachments for the read tool: the read wrapper is image-only, sandbox MIME detection only exposes image MIME types, and provider tool-result transports still forward text/images rather than audio. Existing audio transcription, webchat audio display, and memory multimodal support are adjacent infrastructure, not the requested read("*.audio") multimodal session attachment behavior. Related PR #46607 is a narrow open Google/Gemini headless @file path and does not supersede this general read-tool request.

## Best Possible Solution

Keep #35835 open. The best product/code path is to extend the read wrapper or upstream PI read contract to detect supported audio files with MIME sniffing and size/path guards, add a stable audio content block to the tool-result/session contract, update capable provider transports such as Google/Gemini and OpenAI to forward audio only when the selected model supports it, return a clear unsupported-model message otherwise, and add focused read plus transport regression tests. Coordinate with #46607 only for the narrower Google/Gemini headless @file payload path.
