## Summary

Keep open. Current main still clears the Overview Gateway Token when the WebSocket URL input changes, and the linked bug remains open. This PR is a focused implementation candidate with tests, but the exact behavior needs maintainer judgment because preserving a token across arbitrary URL edits can carry a selected-gateway credential to a different endpoint.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix should stop ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token while preserving the selected-gateway credential boundary, with regression coverage in the Overview/navigation UI tests and explicit handling for when a URL change should require confirmation or a new token.

## Risks / Open Questions

- Merging the exact patch as-is could preserve a shared gateway token across an arbitrary WebSocket endpoint edit and send that credential to the newly entered endpoint on Connect.
- Because applySettings saves settings immediately and persistSessionToken keys by gateway URL, preserving the token while typing may write the existing token under a new or partial gateway URL sessionStorage key.
- There are multiple open implementation candidates for the same linked bug; maintainers should choose or request one canonical, security-aware fix rather than closing this focused PR as stale cleanup.
