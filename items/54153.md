## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal plugin downloads the selected GitHub release asset through a private raw `node:https` downloader, recursively follows redirects, and streams directly to disk without the shared guarded fetch helper, explicit timeout wiring, SSRF validation on redirects, or byte-limit enforcement.

## Best Possible Solution

Fix this in the bundled Signal plugin. Replace the private raw `node:https` `downloadToFile` implementation with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, pass an explicit install-download timeout and redirect limit, choose whether trusted env-proxy mode is appropriate for this GitHub release flow, enforce both declared and streamed byte caps, sanitize the archive temp filename, clean partial downloads, and add focused Signal installer regression tests for timeout, redirect SSRF blocking, proxy mode, oversized downloads, streamed over-limit abort, and cleanup.

## Risks / Open Questions

- The user-triggered Signal install path can still hang on a stalled download because the raw `node:https` request has no explicit timeout or AbortSignal wiring.
- Redirect chains from a GitHub release asset are still followed without the shared SSRF guard validating each redirected target.
- The response body is still streamed to disk without declared-size or incremental byte limits, leaving a disk-exhaustion failure mode on this installer path.
- The eventual fix should decide conservative Signal archive size limits, safe temp-file naming, proxy mode semantics, and cleanup behavior for fetch, size, extraction, and binary-discovery failures.
