## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal plugin downloads the selected GitHub release asset through raw `node:https` redirect recursion and streams it directly to disk without the guarded fetch helper, an explicit timeout, or byte-limit enforcement. The prior ClawSweeper discussion matches the current source state.

## Best Possible Solution

Fix this in the bundled Signal plugin. Replace the private raw `node:https` `downloadToFile` implementation in `extensions/signal/src/install-signal-cli.ts` with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, use an explicit install-download timeout and redirect limit, choose trusted env-proxy mode only for the trusted GitHub release download flow if maintainers want proxy support, enforce both `content-length` and incremental streamed byte caps, clean up partial temp files, and add focused Signal installer tests for timeout wiring, private-network redirect blocking, proxy mode selection, oversized downloads, streamed over-limit abort, and cleanup.

## Risks / Open Questions

- Leaving this open is appropriate because the current user-triggered Signal install path can still hang, follow unguarded redirects, and write an unbounded response body to disk.
- The fix should pick a conservative archive size cap and guarantee temp-file cleanup on content-length rejection, streamed over-limit abort, fetch failure, and extraction failure.
