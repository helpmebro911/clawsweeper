## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal plugin downloads the selected GitHub release asset through a private raw `node:https` downloader, follows redirects itself, and streams directly to disk without the shared SSRF guard, explicit timeout, proxy-aware mode, or declared/streamed size limit.

## Best Possible Solution

Keep this issue open and fix it in the bundled Signal plugin. Replace the private `node:https` `downloadToFile` path with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, pass explicit install-download timeout and redirect limits, decide whether trusted env-proxy mode is appropriate for this GitHub release flow, enforce both declared and streamed archive byte caps, sanitize the temporary archive filename, clean partial downloads on failure, and add focused Signal installer regression tests for the hardened downloader behavior.

## Risks / Open Questions

- The user-triggered Signal release install can still hang if the asset download stalls because the raw `node:https` request has no explicit timeout or abort wiring.
- Redirect chains from a GitHub release asset are still followed without the shared SSRF guard validating each redirected target.
- The asset body is streamed to disk without declared-size or incremental byte limits, so an oversized response can consume local disk during setup.
- The eventual fix should also choose conservative Signal archive size limits, safe temp-file naming, trusted env-proxy semantics, partial-download cleanup, and regression coverage for timeout, redirect blocking, size caps, and cleanup.
