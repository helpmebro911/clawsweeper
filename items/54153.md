## Summary

Keep this issue open. Current main still has the reported Signal installer hardening gap: the bundled Signal setup path downloads a GitHub release asset through a private raw `node:https` helper that follows redirects and streams to disk without the shared SSRF guard, explicit timeout, proxy-aware guarded fetch mode, or archive byte cap.

## Best Possible Solution

Keep this issue open and fix it in the bundled Signal plugin. Replace the private `node:https` `downloadToFile` path with `fetchWithSsrFGuard` from `openclaw/plugin-sdk/ssrf-runtime`, pass explicit timeout and redirect limits, decide whether trusted env-proxy mode is appropriate for this GitHub release flow, enforce both declared and streamed archive byte caps, clean partial downloads on failure, sanitize the archive filename before joining it into the temp directory, and add Signal installer tests for timeout, redirect SSRF blocking, size caps, and cleanup.

## Risks / Open Questions

- The Signal release asset download can still hang if the server stalls because the raw `node:https` request has no explicit timeout or abort wiring.
- Redirect chains from the GitHub release asset are still followed without validating each redirected target through the shared SSRF guard.
- The response body is streamed directly to disk without declared-size or incremental byte limits, so an oversized response can consume local disk during setup.
- The eventual fix should also choose conservative Signal archive limits, sanitize temporary archive filenames, clean partial downloads on failure, and add focused downloader regression tests.
