## Summary

Current `main` only supports `POST` on core `/hooks/*` ingress, and hook transforms still do not receive raw request bytes. Request header access is already partially implemented, so this is a valid open enhancement rather than something already fixed or too stale to evaluate.

## Evidence

- **Core hooks still reject GET:** `createHooksRequestHandler` returns `405 Method Not Allowed` with `Allow: POST` for any non-`POST` request on `/hooks/*`.
  - file: [src/gateway/server-http.ts:595](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/gateway/server-http.ts#L595)

  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Tests lock POST-only behavior:** Gateway hook tests explicitly send `GET /hooks/wake` and assert `405`; another test asserts the `Allow` header is `POST`.
  - file: [src/gateway/server.hooks.test.ts:231](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/gateway/server.hooks.test.ts#L231)

## Risks / Open Questions

- There is a plugin-level workaround path: plugin HTTP routes can inspect the full `IncomingMessage`, and the public webhook helpers already support configurable allowed methods and raw-body reads (`src/plugins/http-registry.ts:7`, `src/plugin-sdk/webhook-request-guards.ts:153`, `src/plugin-sdk/webhook-request-guards.ts:245`). That does not satisfy the core `/hooks/*` request, but it means maintainers may choose a plugin-first design instead of extending core hooks.
- The checkout appears shallow and `gh` was unavailable without `GH_TOKEN`, so this review is based on current `main` source, docs, and in-repo tests rather than full GitHub history.
