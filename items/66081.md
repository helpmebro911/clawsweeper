## Summary

Keep this PR open. Current main still does not request or read Codex CLI `--output-last-message` artifacts, and the PR remains a plausible implementation candidate, but its current head has actionable correctness and cleanup review findings before it should merge.

## Best Possible Solution

Keep this PR open and ask for a scoped revision: inject the Codex `--output-last-message` option before positional prompt args, keep the fallback parser scoped to `codex-cli` or an explicit dialect, read a non-empty artifact as the final reply while preserving stdout-derived session/usage/raw text, mask or abbreviate the generated artifact path in verbose logs, create and clean the artifact inside the existing guarded cleanup flow, and retain focused tests for artifact precedence, empty-artifact fallback, and Codex JSONL message-content parsing.

## Risks / Open Questions

- If landed as-is, appending `--output-last-message` after `buildCliArgs()` may put the option after a positional prompt for arg-mode backends, making Codex CLI parsing fragile.
- The PR head’s JSONL fallback extracts top-level `message`, `content`, `result`, and `response` for every JSONL backend, which can fold diagnostics or tool metadata into replies outside the stated `codex-cli` scope.
- The PR touches a command execution path and introduces a temp artifact file. There is no new supply-chain surface, but temp path logging and cleanup ordering should be tightened before merge.
