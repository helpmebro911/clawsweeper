## Summary

Keep this PR open. Current main still finalizes `codex-cli` replies from stdout/JSONL only, and Codex CLI exposes the requested `--output-last-message` artifact contract. The PR is a plausible implementation candidate, but it should be revised before merge because the proposed diff has unresolved command-ordering, parser-scoping, temp-cleanup, and verbose-log concerns in a command execution path.

## Best Possible Solution

Keep this PR open and request a scoped revision: rebase onto current main, insert the Codex `--output-last-message` flag before positional prompt args, scope any Codex JSONL content fallback to `codex-cli` or an explicit dialect, prefer a non-empty artifact for final reply text while preserving stdout-derived session/usage/raw text, mask the artifact path in verbose logs, create and clean the artifact inside the existing guarded cleanup flow, and retain focused tests for artifact precedence, empty-artifact fallback, and Codex JSONL content parsing.

## Risks / Open Questions

- Merging the PR as-is could make generic JSONL backends incorrectly fold status, diagnostic, tool-result, or metadata fields into final reply text because the proposed fallback is not scoped to `codex-cli` or a named dialect.
- Appending `--output-last-message` after the positional prompt can make Codex argv parsing fragile for arg-mode runs; the flag should be inserted before prompt placement or through a backend-owned base-arg seam.
- The PR touches a command execution path and introduces a temp artifact file. The artifact path should be masked or abbreviated in verbose logs, and artifact creation/cleanup should sit inside the existing guarded cleanup flow.
