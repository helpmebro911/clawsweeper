## Summary

Keep open. Current main does not provide strong evidence that #39588 is fixed; the desktop launch-at-login setting is still derived from live launchd state and can write that state back to the LaunchAgent. The related #28423 covers gateway LaunchAgent entrypoint drift, and the visible changelog fix #40213 only removed KeepAlive from the desktop app LaunchAgent, which does not resolve the reported toggle persistence/update reconciliation problem.

## Evidence

- **Startup refresh can overwrite the toggle from launchd state:** AppState initializes `launchAtLogin` to false, then asynchronously reads `LaunchAgentManager.status()` and assigns it back to `launchAtLogin`; after initialization ends, `updateLaunchAtLogin` writes the setting back through `LaunchAgentManager.set`. That leaves a path where a false status result becomes a real disable operation rather than a reconcile/repair.
  - file: [apps/macos/Sources/OpenClaw/AppState.swift:387](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/apps/macos/Sources/OpenClaw/AppState.swift#L387)
  - command: `nl -ba apps/macos/Sources/OpenClaw/AppState.swift | sed -n '292,307p;387,408p;850,861p'`
  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Status conflates enabled state with loaded launchd state:** `LaunchAgentManager.status()` returns false unless the plist exists and `launchctl print gui/<uid>/ai.openclaw.mac` exits 0; `set(enabled: false)` removes the plist. The code does not distinguish an installed-but-unloaded/stale LaunchAgent from a user-disabled setting, which is central to the reported update-related drift.
