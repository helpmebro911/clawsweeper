## Summary

Keep #37943 open. Current main still has the node-host startup gap described in the issue and the prior ClawSweeper comment: the installed node service runs `openclaw node run`, the global config guard does not allow `node` through invalid effective config, and `runNodeHost()` still calls strict `loadConfig()` before connecting. Gateway startup has a separate best-effort/last-known-good recovery path, so the reporter's “gateway recovers, node service remains stuck” scenario remains plausible on current main.

## Best Possible Solution

Keep the issue open. A maintainer should decide the intended behavior for `openclaw node run` when the main config is JSON-valid but schema-invalid: either give node startup a safe subset of gateway best-effort/last-known-good recovery, or keep node strict but emit node-specific diagnostics and repair guidance. Likely surfaces are `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, `src/cli/node-cli/register.ts`, and `src/daemon/program-args.ts`, with regression coverage for node startup under invalid effective config.
