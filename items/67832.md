## Summary

Current main already has the direct HTTP `/tools/invoke` surface and the underlying `sessions_spawn` tool, but the reporter narrowed the request to a direct CLI path. Current main still has no `openclaw sessions spawn` command and no generic `openclaw tools invoke` command, so the requested CLI capability remains open work.

## Best Possible Solution

Keep this issue open. A conservative implementation would add a core CLI surface backed by the existing Gateway `/tools/invoke` contract: either `openclaw tools invoke --tool sessions_spawn --args ... --json`, or that generic command plus a discoverable `openclaw sessions spawn` wrapper mapping flags to `sessions_spawn`. It should reuse existing gateway URL/auth resolution, preserve all current HTTP and agent tool-policy gates, add CLI tests around denied and allowed `sessions_spawn`, and update the public CLI docs.

## Risks / Open Questions

- A direct spawn CLI would expose a high-risk session orchestration surface, so it should preserve Gateway auth, `gateway.tools.allow` default-deny behavior, agent tool policy, max child/depth limits, `allowAgents`, and sandbox inheritance checks.
- A generic `openclaw tools invoke` command would make many tools more scriptable; docs and errors should make dangerous-tool default denies and required opt-in explicit.
