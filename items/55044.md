## Summary

Keep open. Current main still recognizes explicit stop/no-action text, but the abort paths persist `abortedLastRun: true`, and restart recovery still uses that same marker to inject synthetic continuation messages through `callGateway`. The issue lacks full reproduction logs, but the current source, docs, tests, and prior ClawSweeper comment still match the central complaint closely enough that this should not be auto-closed.

## Best Possible Solution

Keep this open. The likely fix is to split restart-interruption recovery state from explicit user-abort state, persist a terminal user-abort or recovery-suppression marker for `/stop` and recognized no-action phrases, ensure main-session and subagent recovery skip those sessions and clear pending retry state, and add focused regression tests proving `/stop` followed by recovery startup does not call `callGateway` while ordinary gateway-reload recovery still works.
