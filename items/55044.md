## Summary

Keep this issue open. Current main still has no separate persisted marker for an explicit user stop versus restart-interruption recovery, and the existing recovery paths can still synthesize continuation turns from `abortedLastRun` state.

## Best Possible Solution

Split explicit user-abort state from restart-interruption recovery state. `/stop` and recognized no-action phrases should persist a terminal user-abort or recovery-suppression marker, recovery scans should skip those sessions, and regression tests should prove `/stop` followed by gateway startup does not call `callGateway` while legitimate restart recovery still resumes interrupted work.
