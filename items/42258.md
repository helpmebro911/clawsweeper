## Summary

Keep open. Current `main` still has no recovery path for a live Control UI chat run when the terminal `chat` event is missed: `chatRunId` only clears on terminal events, queued follow-ups stay blocked while that flag is set, and the existing reconnect cleanup only helps after an actual disconnect.

## Evidence

- **Active run state only clears on terminal chat events:** `handleChatEvent` clears `chatRunId` only in the `final`, `aborted`, and `error` branches. There is no fallback path here for a run that already finished server-side but never delivered its terminal UI event.
  - file: [ui/src/ui/controllers/chat.ts:592](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/ui/src/ui/controllers/chat.ts#L592)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '565,636p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Deferred history reload still depends on receiving a terminal event:** `handleSessionMessageGatewayEvent` defers `loadChatHistory` whenever `chatRunId` is set, and `handleChatGatewayEvent` only replays that deferred reload after a terminal state clears the active run. If the terminal event is the missing piece, the deferred reload never runs.
  - file: [ui/src/ui/app-gateway.ts:463](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/ui/src/ui/app-gateway.ts#L463)
  - command: `nl -ba ui/src/ui/app-gateway.ts | sed -n '445,507p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Current self-heal covers reconnects, not live stale runs:** On `hello`, the gateway client resets orphaned `chatRunId`/stream state and can resume queued chat after reconnect. That recovery only runs after the websocket reconnect path, not while the UI stays connected but misses one terminal event.
  - file: [ui/src/ui/app-gateway.ts:320](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/ui/src/ui/app-gateway.ts#L320)
  - command: `nl -ba ui/src/ui/app-gateway.ts | sed -n '320,334p'`

- **Queued follow-ups are expected to wait behind an active run:** The user docs explicitly say normal follow-ups queue while a run is active. Combined with the stale `chatRunId` path above, this matches the PR's reported failure mode that queued messages remain blocked until something clears the run state.

That makes the PR's client-side probe strategy technically coherent rather than out-of-scope or incoherent.
