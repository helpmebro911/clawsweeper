## Summary

Keep open. The current main branch still uses the file-backed delivery queue and startup recovery still replays pending entries without any TTL/expiry gate. Related reliability work in the issue context is useful background, but current code does not implement the requested max-age behavior or config surface.

## Evidence

- **Startup recovery has no age/TTL gate:** recoverPendingDeliveries loads every pending delivery file, sorts by enqueuedAt, then only checks max retries and retry backoff before delivering. There is no check against enqueuedAt age or configurable expiry action.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:354](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/infra/outbound/delivery-queue-recovery.ts#L354)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '342,436p'`
  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Fresh stale entries replay immediately:** isEntryEligibleForRecoveryRetry treats a retryCount=0 entry with no lastAttemptAt as immediately eligible. That means an old file left after a crash-before-ack can still be replayed on restart regardless of age.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:173](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/infra/outbound/delivery-queue-recovery.ts#L173)
