## Summary

Keep #16555 open. Current main still uses the file-backed outbound delivery queue with write-ahead enqueue, two-phase ack cleanup, max-retry handling, permanent-error classification, retry backoff, and reconnect drains, but it does not implement the requested configurable TTL/expiry policy for stale queued outbound messages.

## Best Possible Solution

Keep #16555 open until outbound delivery recovery has a bounded, documented expiry policy. The likely core fix is to add optional `messages.delivery` schema/types/help/docs, enforce an `enqueuedAt` age check before delivery in both `recoverPendingDeliveries` and `drainPendingDeliveries`, choose a conservative terminal action such as moving expired entries to `failed/`, and add regression tests for default backward compatibility, stale startup entries, reconnect drains, max-retry/backoff interaction, and delivered-marker cleanup.

## Risks / Open Questions

- Old `retryCount === 0` queue entries with no `lastAttemptAt` can still replay on startup no matter how old their `enqueuedAt` timestamp is.
- Reconnect drains can still replay stale matching entries if a channel reconnects before startup recovery handles them.
- The recent two-phase ack fix reduces duplicate replay for already-acked deliveries, but it does not provide the requested configurable age-based expiry for still-pending queue entries.
