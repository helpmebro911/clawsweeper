## Summary

Keep open. Current main still uses a transient IRC connection for proactive sends when no live client is passed, while the monitor's joined persistent client is only used for inbound replies. The PR addresses a real-looking gap but still needs maintainer review because the supplied review thread identifies unresolved process-boundary and join-readiness concerns.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to turn this into a source-backed IRC outbound fix with regression coverage: either route IRC proactive sends through the gateway/persistent monitor client, or make the direct send path safely join and confirm channel membership before PRIVMSG. Preserve the identity-checked deregistration idea from the PR, but address the process-boundary and JOIN-readiness review comments before landing.
