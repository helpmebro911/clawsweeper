## Summary

Keep open. Current main still maps the Dreaming Advanced daily-log summary/section and origin logic to `groundedCount`, while the memory-core store and tests keep `dailyCount` and `groundedCount` as distinct signal classes. The related PR #68942 is open and unmerged, so it does not make the issue obsolete.

## Best Possible Solution

Keep the issue open and make the Dreaming Advanced UI semantics consistent. Given current docs describe a distinct grounded historical replay lane, the likely product-safe fix is to rename the summary/section copy away from "From Daily Log" and ensure origin badges distinguish daily-only, grounded-only, and mixed entries correctly. If maintainers instead want "From Daily Log" to mean `dailyCount`, compute that count/section from `dailyCount > 0`. Add targeted `ui/src/ui/views/dreaming.test.ts` coverage for daily-only and grounded-only entries before closing.
