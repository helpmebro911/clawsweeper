## Summary

Keep this issue open. The report is recent, has concrete user evidence from v2026.4.15 and v2026.4.21 showing `action=new` plus `.jsonl.reset.*` transcript rotation, and current `main` still does not contain a targeted reconnect/page-refresh regression proving Control UI preserves the same gateway `sessionId`. Source inspection shows the nominal WebSocket reconnect path refreshes state via `chat.history`, but the gateway send/reset paths still have plausible session-rotation boundaries that need a focused test before closure.

## Best Possible Solution

Add a focused regression for Control UI WebSocket reconnect and browser refresh: seed an existing webchat session/transcript for `openclaw-control-ui`, connect, force close/reconnect or remount with the same settings, then call `chat.history` and `chat.send`. Assert the same `sessionId` is retained, no `sessions.reset` or command hook `action=new` fires, and no `.jsonl.reset.*` archive appears. If it fails, fix the frontend session-selection/reconnect path or the gateway reset/freshness trigger; if it passes, document the distinction from explicit New Session/reset and reconsider closure.
