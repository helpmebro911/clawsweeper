## Summary

Keep #38212 open. Current main has related progress: runtime config now migrates legacy `codex-cli/*` primary refs to canonical provider refs plus `agentRuntime.id`, and auth aliases can map deprecated `codex-cli` auth choices to `openai-codex`. But the exact helper-path parity requested here is not implemented or proven: `generateSlugViaLLM` still forwards the resolved provider/model directly into `runEmbeddedPiAgent`, the embedded model resolver still uses the incoming provider for `resolveModelAsync`, and `codex-cli` is registered as a CLI backend rather than an `openai-codex` provider alias for dynamic model resolution. The earlier exact-fix PRs #38215 and #47074 were closed unmerged; #70743 is useful related hardening, but it did not close this exact issue.

## Best Possible Solution

Keep #38212 open. A maintainer-quality fix should either add an explicit `codex-cli/gpt-5.4` slug/helper regression and normalize helper runs to a supported `openai-codex` model/auth route before `resolveModelAsync`, or document and enforce that legacy `codex-cli/*` primary refs must be migrated to canonical model refs plus `agentRuntime.id` and that helper slug generation will not use Codex OAuth/CLI auth. The implementation should make the desired product boundary explicit instead of relying on a partial runtime-config migration.
