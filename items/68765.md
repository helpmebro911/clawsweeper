## Summary

Keep PR #68765 open. Current main still routes `chat.history` through a single-file `readSessionMessages` implementation, so the reported compaction checkpoint-chain truncation is not implemented or superseded. The PR discussion includes useful production-derived evidence and a focused regression direction, but the branch also has unresolved correctness concerns around reset/restore epochs, checkpoint boundary duplication, duplicate store rows, and path containment, so it should be reworked rather than auto-closed.

## Best Possible Solution

Keep the item open and ask for a narrowed gateway fix. The likely implementation belongs around `src/gateway/session-utils.fs.ts` and the history consumers: resolve ordered transcript segments from the canonical current main-session row, use `compactionCheckpoints[].preCompaction.sessionFile` only when contained in the session store root, trim or de-duplicate snapshots at `firstKeptEntryId`/active-branch boundaries, define reset/restore epoch behavior, and add focused regression tests in `src/gateway/session-utils.fs.test.ts` plus a `chat.history` integration case.
