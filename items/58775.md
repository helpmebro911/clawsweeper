## Summary

Current main has partial `google-vertex` handling (alias lookup, model-id normalization, ADC marker detection, and docs), but it still does not implement the native Google Vertex Gemini transport described in the regression: the Google transport always builds `.../models/{model}:streamGenerateContent` URLs and sends Gemini API-key/OAuth-style headers, while the Google plugin manifest does not register `google-vertex` as its own provider. The linked PR #60860 appears to track the exact remaining work, and a commenter reported a `GOOGLE_CLOUD_LOCATION=global` workaround, but that PR is still open/unmerged in the provided context, so this bug should remain open until a native Vertex fix lands or maintainers intentionally scope it out.

## Best Possible Solution

Keep #58775 open as the user-visible regression tracker until PR #60860 or an equivalent maintainer-approved fix lands. The best code direction is a dedicated Google Vertex provider path that preserves Google model-id normalization while adding Vertex project/location URL construction, ADC/Bearer auth exchange, tests for `global` and regional endpoints, and docs that distinguish AI Studio, Gemini CLI OAuth, OpenAI-compatible Vertex endpoints, and native Vertex ADC.
