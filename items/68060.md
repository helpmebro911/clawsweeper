## Summary

Keep open. Current main still does not implement the proposed memory-lancedb smart extraction path: there is no smartExtraction config surface, no smart-extractor module, and the active auto-capture flow remains the existing heuristic path. This PR is a scoped bundled-plugin feature, not obsolete cleanup. The author did address the initial credential/dead-helper feedback in follow-up commits, but remaining review comments identify real integration and security issues that need rework before merge.

## Best Possible Solution

Keep this PR open for focused rework in the memory-lancedb plugin. The best path is to update the manifest and UI hints for smartExtraction, preserve shouldCapture and captureMaxChars before the LLM call, treat existing memories as untrusted prompt data, make extraction failures observable or safely fall back to heuristic capture, and add targeted config/parser/hook tests plus docs for the new opt-in behavior.

## Risks / Open Questions

- Closing now would discard an active, scoped memory-lancedb feature that current main has not shipped and that has no canonical replacement item in the provided related context.
- Merging as-is would likely reject user smartExtraction config because the plugin manifest schema was not updated with the new property.
- Merging as-is could send long, sensitive, or prompt-injection-like user content to the extraction model if the smart path bypasses shouldCapture and captureMaxChars.
- Merging as-is could introduce prompt-injection risk by inserting existing stored memories into the extractor prompt without the untrusted-memory escaping used by recall.
- Merging as-is could make extraction outages hard to diagnose if chat-completion failures are swallowed and auto-capture silently stops storing memories.
