## Summary

Keep #69066 open. Current main has tactical trusted-proxy and direct-loopback backend handling, but it has not implemented the RFC's central request: a first-class service-identity axis with separate service token/config/role and migrated internal callers. The item is a recent core gateway auth RFC, not stale, not duplicated by the provided related context, and it needs maintainer product/security judgment.

## Best Possible Solution

Keep the RFC open as the canonical discussion for separating internal service identity from user/operator auth. A complete path would add a documented `gateway.service` config/schema/type surface, persistent rotatable service token storage, `GatewayClient`/spawned-child token injection, a service-auth precheck before user auth, scoped service roles or an agreed equivalent, migration of internal callers, and regression coverage for trusted-proxy, loopback, subagent, cron, browser tool, and approval flows. Existing tactical trusted-proxy fixes can be evaluated separately and linked back to this RFC rather than using them to close it.
