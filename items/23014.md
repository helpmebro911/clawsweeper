## Summary

Current main does not implement the requested .m2a audio-extension support. The canonical audio extension set still omits .m2a, isAudioFileName depends on that set, and media-understanding selection/preflight paths depend on that classification. The related PR #23260 was draft/closed/unmerged in the provided GitHub context, so it does not supersede the issue.

## Evidence

- **Canonical audio extension list omits .m2a:** AUDIO_FILE_EXTENSIONS contains .aac, .caf, .flac, .m4a, .mp3, .oga, .ogg, .opus, and .wav, but not .m2a.
  - file: [src/media/mime.ts:56](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/media/mime.ts#L56)

  - sha: [bd796d1c85c7](https://github.com/openclaw/openclaw/commit/bd796d1c85c7dec0dcb89ed11502e4271ded1314)
- **Audio filename detection is extension-set based:** isAudioFileName returns AUDIO_FILE_EXTENSIONS.has(ext), so a .m2a filename with no audio MIME hint is not classified as audio.
  - file: [src/media/mime.ts:124](https://github.com/openclaw/openclaw/blob/bd796d1c85c7dec0dcb89ed11502e4271ded1314/src/media/mime.ts#L124)
