## Summary

Keep #23014 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still does not treat `.m2a` as audio in the shared MIME/extension map, audio filename classifier, voice-compatible audio set, Telegram document placeholder path, or audio preflight detection. The related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers implement or explicitly reject `.m2a` support. The likely fix belongs in shared media handling: add a confirmed `.m2a` MIME mapping and audio filename support, consider passing preserved original filenames into `detectMime` for inbound buffers, update Telegram document placeholder/preflight logic if document filenames should drive audio handling, and add focused tests for shared MIME detection, `isAudioFileName`, Telegram document media behavior, and adjacent WebChat/UI audio allowlists where support is intended.
