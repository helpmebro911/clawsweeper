## Summary

Keep #23014 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still has no `.m2a` support in the shared media MIME map, audio filename classifier, Telegram document placeholder path, or audio preflight detection. The related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers implement or explicitly reject `.m2a` support. The likely fix belongs in shared media handling: add `.m2a` to the shared extension/MIME and audio filename classifiers, pass preserved original filenames into MIME detection for inbound buffers, and add focused tests for `detectMime`, `isAudioFileName`, Telegram document placeholder/preflight behavior, and adjacent WebChat/UI audio allowlists if consistent support is intended.
