## Summary

Keep #70753 open. The report is recent, externally authored, has no protected labels, and the provided triage comment treats it as a valid P1 regression with concrete reproduction evidence. Current main still contains the same embedded bundle-MCP connection shape that can plausibly hit the reported failure: URL-only or `type: "http"` MCP configs resolve to OpenClaw's SSE transport unless `transport: "streamable-http"` is set, the SSE path still uses the SDK SSE transport with raw undici fetch, and the 30s connection guard still emits the exact timeout variant reported. I found no current-main source, tests, changelog, or release evidence that specifically fixes AgentMail/HTTP MCP startup timeouts.

## Best Possible Solution

Keep the issue open and fix the embedded bundle-MCP HTTP transport path. A concrete maintainer path is to decide whether imported/downstream `type: "http"` should be accepted as OpenClaw `transport: "streamable-http"` for `mcp.servers`, and to harden the SSE/streamable HTTP client against the undici HTTP/2 timeout class. Add focused tests for URL-only, `type: "http"`, `transport: "sse"`, and `transport: "streamable-http"` resolution plus a regression-style mocked transport test that proves a fast initialize response is not lost to the 15s/30s startup guard.
