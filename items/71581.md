## Summary

Keep #71581 open. Current main still routes forced heartbeat wakes for exec-event and cron-event through the literal same-agent session key, and the heartbeat/system-event queues are keyed by that resolved session. `session.dmScope=main` single-owner pinning exists for inbound DM handling, but the heartbeat/exec/cron paths do not consult it. The related PR #72160 from the provided context is closed, unmerged, and unrelated to heartbeat session routing, so it does not supersede this issue.

## Best Possible Solution

Keep the issue open and implement this in core heartbeat/session routing, not as a plugin. Add a helper that, for same-agent forced direct-message session keys, rewrites to the agent main session when `session.dmScope` is effectively `main` and the existing single-owner allowlist precondition holds. Decide whether to re-key system events at enqueue time or bridge/drain from the original queue into main, then add regression tests for background exec completion and cron-event wakes with `dmScope=main`, plus negative tests for multi-owner/wildcard allowlists and non-main dmScope values.
