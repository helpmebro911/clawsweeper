## Summary

Keep open. Current main still exposes `health` as a service/channel/agent/session summary and does not include the requested connection-level `connected`, `rttMs`, or `lastHeartbeatAt` fields. There is also no separate `gateway.ping`/`ping` RPC method on the advertised gateway method list. The existing WebSocket `tick` keepalive is internal client stall detection, not a response field a UI can poll from `health`.

## Best Possible Solution

Keep the issue open and implement an additive gateway-level connection health surface. The conservative path is either to add optional `connected`, `rttMs`, and `lastHeartbeatAt` fields to the `health` RPC response for the current caller, or add a lightweight `gateway.ping`/`ping` RPC plus client/UI typing and tests. The implementation should live around the gateway health handler/client request path, not the channel health monitor, because the requested signal is current gateway WebSocket reachability and latency.
