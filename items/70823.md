## Summary

Keep this issue open. Current main still models `plugins.slots.*` as bare string plugin IDs across the public config type, Zod/runtime schema, generated schema, normalization, activation, install/write paths, tests, and docs. The requested `{ owner, claimed_at, claimed_by_version }` read/write shape and takeover invariant remains a real core plugin/config contract RFC, not a stale report, duplicate, or ClawHub-only plugin idea.

## Best Possible Solution

Keep the RFC open for maintainer design review. A concrete path would add a backwards-compatible slot ownership union, normalize legacy string slots to an owner object on read, update install/register/write paths to persist canonical ownership metadata, add explicit conflict and force/allow-override semantics for slot takeover, and cover validation, generated schema/help, docs, activation, install/update/uninstall, default slots, and `"none"` behavior with focused tests.

## Risks / Open Questions

- This is a public config and plugin contract change; maintainers still need to choose the exact schema shape, migration order, provenance fields, and takeover semantics.
- Any implementation must preserve legacy bare-string configs, `"none"` disable behavior, inferred defaults such as `memory-core` and `legacy`, generated schema/help alignment, and install/update/uninstall compatibility.
- Existing `plugins install --force` is about replacing install targets, not authorizing slot takeover, so force/override semantics need a deliberate design.
