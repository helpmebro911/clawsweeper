## Summary

Keep open. Current main still models `plugins.slots.memory` and `plugins.slots.contextEngine` as bare string/null selectors. The RFC's structured owner record, provenance fields, read-side object normalization, canonical object writes, and overwrite invariant are not implemented, and this is a core config/plugin API design question rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep #70823 open for maintainer API review. A good implementation path would add an additive slot-owner type and read-side normalization in `src/config/types.plugins.ts`, `src/config/zod-schema.ts`, and `src/plugins/config-normalization-shared.ts`, while preserving bare-string reads. Follow-up work should canonicalize install/update writes, update validation/doctor/docs/schema help/tests, and define an explicit force/override invariant for replacing an existing slot owner.
