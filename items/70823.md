## Summary

Keep #70823 open. Current main still models `plugins.slots.memory` and `plugins.slots.contextEngine` as bare plugin-id strings, and the request is a legitimate core config/plugin-contract RFC rather than a ClawHub plugin idea or duplicate of the provided related reports.

## Best Possible Solution

Keep the RFC open for maintainer design review. A good implementation path would define a backward-compatible slot-claim type, normalize legacy string slots to an owner-bearing read shape, update validation and generated schema/docs, teach install/selection/update/uninstall/startup paths to read and write the new shape, and add focused tests for legacy strings, object-form ownership, provenance preservation, and explicit override semantics.
