## Summary

Keep open. Current main still models `plugins.slots.*` as bare plugin-id strings, validates only string slot values, normalizes slot reads to `string | null | undefined`, and install/selection flows overwrite a slot with a new string plus warnings. The RFC is coherent and in-scope as a core config/plugin-contract discussion, especially given the issue body's concrete `@plur-ai/claw` repair use case and plur-ai/plur#51 cross-reference.

## Best Possible Solution

Keep the RFC open for maintainer design review. A conservative path would add a shared slot-claim type, accept both string and object forms on read, expose a helper that returns the normalized owner ID plus optional provenance, update install/slot mutation paths deliberately, and document the migration before plugin authors adopt `slots.<slot>.owner`.
