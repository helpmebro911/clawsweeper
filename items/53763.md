## Summary

Keep this issue open. Current main already has a bundled browser plugin with headless mode, remote CDP, extra launch args, hosted-browser docs, optional Docker browser installs, and a sandbox browser image, but it still does not deliver the central request: a first-class bundled Chromium runtime that works without external browser setup across normal installs.

## Best Possible Solution

Keep this issue open, but narrow it to the remaining product decision: whether OpenClaw should ship a pinned Chromium runtime by default, publish an official browser-enabled Docker image, or add installer/doctor repair flows that install and persist Playwright Chromium and wire that path into executable discovery. Reuse the existing browser plugin, headless mode, remote CDP support, extraArgs, and sandbox browser image rather than creating a second browser-control stack.

## Risks / Open Questions

- Bundling Chromium by default would affect package size, Docker image size, platform support, sandbox defaults, and the browser security-update cadence.
- The existing browser plugin/control implementation should be reused; a separate browser stack would risk duplicate behavior and inconsistent security policy.
