## Summary

Keep this issue open. Current main exposes partial Dreams visibility, but it still does not expose the requested last run timestamp, run outcome or partial-failure state, per-run processed trace count, or diary updated/workspace status in the Dreams UI.

## Best Possible Solution

Keep this issue open. Implement a memory-core-owned Dreaming run summary, either on doctor.memory.status or a sibling Dreams status method, with last run time, outcome such as ok/partial/error/skipped, duration, processed trace/candidate/applied/failed counts, diary path/updatedAt/workspace status, and next run. Then render those fields in the Dreams tab and cover the gateway payload, UI controller normalization, and view rendering with focused tests.

## Risks / Open Questions

- A fix needs a memory-core-owned aggregation rule for multiple configured memory workspaces, otherwise a single last-run timestamp or diary status could be misleading.
- Cron state already tracks generic last-run metadata, but Dreams should not expose only raw Cron job state if phase processing, diary writing, or workspace fan-out can fail independently.
