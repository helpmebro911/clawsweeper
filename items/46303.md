## Summary

Keep open. Current main still does not implement the central restart-time drain behavior: inbound debounce buffers remain closure-local with no global flush surface, the SIGUSR1 run loop drains only active command tasks, embedded runs, and runtime dependency installs before closing the server, and followup queues only have per-key drain scheduling with no all-queue drain waiter. The PR is a plausible implementation candidate and the discussion includes useful fixes and review feedback, but it is broad and security-sensitive enough that it needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best maintainer path is a narrower revision that adds a lifecycle-safe, controlled drain surface for active inbound debouncers; preserves the existing public SDK contract; flushes debounced inbound work at the correct restart phase; waits for active turns, followup queues including dropped-summary state, and channel worker queues that can receive flushed work; and adds focused restart-ordering, teardown-race, and SDK compatibility tests.

## Risks / Open Questions

- The PR branch's mutable `Symbol.for` global debouncer registry is security-sensitive in a same-process plugin model; a safer design should expose controlled operations rather than a discoverable raw registry.
- The PR changes plugin-facing debounce contracts (`flushKey` return type and added lifecycle methods), which can break third-party plugins unless handled as backward-compatible additions.
- Restart ordering and lifecycle races remain subtle: late ingress after unregister, Discord-style inbound worker queues, active-turn followup drain sequencing, and `droppedCount` summary state all need explicit handling.
- The branch is large across many bundled channels and has recent merge/test-isolation fallout in the discussion, so keeping it open does not imply it should land unchanged.
