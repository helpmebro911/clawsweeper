## Summary

Keep open. Current `main` does not already contain this fix, and the PR still has unresolved correctness and validation problems from review: its suppression flag is not wired into the user-visible compaction side effects it claims to suppress, it changes the `runMemoryFlushIfNeeded` return contract without updating existing in-scope tests, and the new tests described in review do not exercise the production path.

## Evidence

- **Current main still has no memory-flush suppression path in `runReplyAgent`:** `runReplyAgent` still assigns `activeSessionEntry = await runMemoryFlushIfNeeded(...)`, then later unconditionally enqueues post-compaction system context and pushes the `🧹 Auto-compaction complete` notice whenever `autoCompactionCount > 0`. There is no memory-flush flag or guard on that side-effect block on current `main`.
