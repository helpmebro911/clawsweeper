## Summary

Keep #38907 open. Current main still contains the reported default ACP debug-client failure path: `openclaw acp client` creates a default bridge session with no `_meta` session key or ACP runtime metadata, the bridge falls back to an `acp:<uuid>` session key, Gateway canonicalizes that into an agent-scoped ACP-shaped key, and ACP reply dispatch treats ACP-shaped sessions without persisted `SessionAcpMeta` as `ACP_SESSION_INIT_FAILED` before normal assistant generation can run. The related closed ACP reports in the supplied context cover different thread-bound/runtime failure modes, not this default debug-client path.

## Best Possible Solution

Keep #38907 open until the ACP bridge/runtime boundary is fixed and covered. The likely product fix is to either stop default Gateway-backed ACP debug-client sessions from using ACP-shaped fallback keys that the ACP runtime claims, or persist valid `SessionAcpMeta` before the first prompt. Add focused regression coverage for default `createAcpClient` / `AcpGatewayAgent.newSession` first-prompt behavior proving `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
