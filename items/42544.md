## Summary

Keep this PR open. Current main still performs service-mode stale gateway cleanup unconditionally before startup, so the user-visible bug is not implemented or superseded. The PR targets the right code path, but its current patch is not merge-ready because of functional gaps and a security/portability concern around executing `curl` from PATH during gateway startup.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best implementation is a small gateway startup change that probes the already-running gateway with Node's built-in `fetch` and `AbortController`, uses the effective reachable host for the configured bind mode, exits successfully when the existing gateway is healthy, preserves stale cleanup only for nonresponding stale processes, and adds focused regression tests for service-marker healthy/unhealthy paths plus non-loopback bind coverage.

## Risks / Open Questions

- Closing this PR as stale would leave the current main behavior in place: service-mode cleanup can still kill a healthy gateway before startup binds.
- Merging the PR as-is could still fail the healthy-gateway path because it falls through toward `runGatewayLoop` instead of cleanly returning after detecting a healthy listener.
- The proposed `curl` subprocess is a portability and security-sensitive regression compared with Node's built-in `fetch`/`AbortController`.
- The proposed probe only checks `127.0.0.1`, so LAN, custom bind, tailnet, or container bind modes can still be misclassified as unhealthy.
- The PR does not include regression tests for healthy service-mode preservation, unhealthy stale cleanup, non-loopback bind behavior, or the external-probe failure mode.
