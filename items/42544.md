## Summary

Keep this PR open. Current main still runs service-mode stale gateway cleanup before any health probe, so the reported bug remains. The PR targets a real gateway startup problem, but the submitted patch is not merge-ready because it adds a PATH-resolved curl subprocess, probes only loopback, falls through toward startup after detecting a healthy gateway, and adds no regression tests.

## Best Possible Solution

Keep the PR open for maintainer review or contributor rework. The best fix belongs in gateway startup: after resolving the effective bind/probe target, probe the existing gateway with Node fetch plus AbortController against the existing /healthz or /health liveness endpoint, return successfully when the existing gateway is healthy, call cleanStaleGatewayProcessesSync only when the listener is nonresponding, avoid spawning curl, and add focused tests for healthy, unhealthy, and non-loopback service-marker paths.

## Risks / Open Questions

- Closing this PR as stale would leave current main able to kill a healthy service-mode gateway before startup binds.
- Merging the PR as-is could still fail the healthy-gateway scenario because the code logs the healthy gateway and then continues toward runGatewayLoop instead of returning successfully.
- The PR's curl subprocess is a portability and security-sensitive regression: correctness depends on an external executable resolved from PATH even though the repo requires Node >=22.14.0 and can use built-in fetch plus AbortController.
- The PR hard-codes 127.0.0.1, so custom or tailnet bind modes can be misclassified as unhealthy and still flow into stale cleanup.
- The PR does not add focused regression coverage for healthy service-marker preservation, unhealthy stale cleanup, non-loopback/custom bind behavior, or health probe failure handling.
