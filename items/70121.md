## Summary

Keep open. The report is recent, concrete, and still plausible against current `main`. Current OpenClaw docs/code confirm that native plugins run in-process with the Gateway, but the public plugin SDK still exposes only generic process exec helpers plus a signal-forwarding bridge, not a dedicated long-running subprocess lifecycle API for plugin-owned child services. I did not find a main-branch fix after the reported `2026.4.15` version.

## Evidence

- **Reporter provided concrete reproduction and controls:** The issue body includes 17 repeated SIGTERM samples, a stable direct `node server.js` control, and a failing Gateway `autoStart` path on OpenClaw `2026.4.15`; there are no comments or timeline items contradicting that evidence in the provided GitHub context.
  - command: `GitHub issue #70121 context provided in task`
- **Native plugins share the Gateway process:** The architecture docs say native plugins are loaded in-process via jiti and later state that native plugins run in-process with the Gateway and share the same process-level trust boundary as core code.
  - file: [docs/plugins/architecture.md:133](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/docs/plugins/architecture.md#L133)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Current public SDK does not expose a dedicated plugin subprocess manager:** `plugin-sdk/process-runtime` currently exports only the generic process exec helpers and OOM wrapper; the SDK subpaths docs describe it simply as `Process exec helpers`. That is weaker than the issue's requested plugin-owned long-running subprocess lifecycle surface.
  - file: [src/plugin-sdk/process-runtime.ts:1](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/plugin-sdk/process-runtime.ts#L1)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Current guidance is signal forwarding, not detached child ownership:** The background-process docs tell authors to attach the child-process bridge helper for long-running child processes, and the helper itself forwards SIGTERM/SIGINT/SIGHUP/SIGQUIT from the parent process to the child before detaching listeners on exit/error.
  - file: [src/process/child-process-bridge.ts:14](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/process/child-process-bridge.ts#L14)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **No relevant fix found on main after the reported version:** `git log --since='2026-04-15'` over the relevant process/runtime/docs files surfaced only commit `712f7b21` (`test: cover bundled MCP runtime cleanup gates`), which is not an implementation fix for plugin child-process lifecycle behavior.
  - command: `git log --oneline --since='2026-04-15' -- src/process/child-process-bridge.ts src/plugin-sdk/process-runtime.ts docs/gateway/background-process.md docs/plugins/sdk-subpaths.md docs/plugins/architecture.md`
  - sha: [712f7b218c09](https://github.com/openclaw/openclaw/commit/712f7b218c092c9d0fae1f583f7c06e617920115)
