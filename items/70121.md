## Summary

Keep open. The report is recent and has a concrete reproduction against a third-party native plugin. Current main has generic plugin service lifecycle support, but I did not find a shipped camofox-specific fix, a canonical duplicate, or a public plugin subprocess-management contract that clearly resolves the reported SIGTERM behavior. ## Best Possible Solution

Keep the issue open and reproduce it on current main with camofox-browser. If the plugin starts the helper outside `api.registrationMode === "full"`/`api.registerService`, document or patch the plugin pattern. If a valid full-runtime service child is still killed, add a narrow Plugin SDK subprocess/lifecycle contract or runtime fix with a regression test around long-running helper servers. Do not add camofox-specific behavior to core.
