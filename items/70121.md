## Summary

Keep open. The report is recent, has concrete reproduction data, and points at a plausible OpenClaw plugin lifecycle/API gap rather than only a third-party plugin request. Current main documents and exposes plugin services for long-lived runtime work, but I did not find a public plugin-facing subprocess supervisor or a source-backed fix that proves detached plugin child processes now survive the Gateway lifecycle.

## Best Possible Solution

Keep the issue open and investigate it as a plugin lifecycle/API gap. A good maintainer path is to reproduce with a tiny external test plugin that starts a child process from a full-runtime `registerService`, records parent/child PIDs and signal source, and then either document the supported pattern or add a narrow plugin SDK helper for managed long-running subprocesses with explicit start/stop semantics.

## Evidence
