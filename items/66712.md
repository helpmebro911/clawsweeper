## Summary

Keep PR #66712 open. Current main still only deduplicates direct cron delivery by execution ID, so the same scheduled slot can bypass the existing in-memory cache when replayed with a different runStartedAt. I also did not find the PR's startup catch-up deliveryAttempted propagation fix or its same-slot regression test on main.

## Best Possible Solution

Keep #66712 open for maintainer review. The likely best path is to land a focused process-local slot-level direct-delivery dedup guard keyed by job, scheduledAtMs, and normalized delivery target, while preserving the existing execution-level cache, adding the same-slot/different-runStartedAt regression coverage, and forwarding deliveryAttempted from startup catch-up. Persistent cross-process dedup should remain a separate design item if needed.
