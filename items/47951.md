## Summary

Keep open. Current main has better ACP task lifecycle tracking, but it still does not solve the central reported failure: an ACP/Codex child run can be accepted before the prompt is mirrored into the OpenClaw child transcript, and ACP transcript persistence happens only after the ACP turn completes. If the runtime stalls or fails before completion, the child session history can still be empty. The related #63702 remains open and is related, but it is not a clear canonical replacement for this specific empty-transcript accepted-run bug.

## Evidence

- **Issue evidence:** The provided issue body and comments describe a reproducible accepted ACP Codex spawn where the childSessionKey is returned but sessions_history stays messages: [] and the Codex transcript has no initial user message. Ryce’s comment explicitly frames the empty transcript as the core reliability/debuggability failure. Timeline also cross-references open related issue #63702, but that issue tracks a narrower Internal error after session/new, not this exact accepted-empty-history symptom.
