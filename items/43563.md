## Summary

Current main still does not implement the requested Feishu send-time cross-bot broadcast. Feishu now has an inbound broadcast/fan-out path for a single received platform event, but outbound replies still go directly through the Feishu send APIs and are not synthesized back into peer bot accounts in the same group. The issue body and related Feishu/Telegram reports describe a concrete multi-bot platform gap, so this should remain open.

## Best Possible Solution

Keep this issue open. Implement an opt-in Feishu-owned send-time synthetic inbound path after successful outbound group sends, keyed by configured peer bot accounts/groups, with dedupe, loop limits, sender/origin metadata, multi-account tests, outbound reply coverage, and Feishu docs. Add a generic plugin/channel seam only after proving multiple channels need the same capability.

## Risks / Open Questions

- Synthetic bot-to-bot delivery can create reply loops or token amplification unless it is opt-in, scoped per group/account, deduped, and guarded by consecutive-bot-turn limits.
- Existing Feishu inbound broadcast partially overlaps the multi-agent goal; a future implementation should reuse or deliberately evolve that path instead of adding a confusing parallel broadcast model.
- A cross-channel abstraction may be premature. Feishu should likely own the first concrete fix, with a generic SDK/core seam only if Telegram or another bundled/third-party channel needs the same mechanism.
