## Summary

Current main still does not implement the requested Feishu send-time cross-bot broadcast. Feishu has inbound broadcast fan-out for already-received platform events, but outbound replies and direct sends still go straight through Feishu send APIs without synthesizing peer-bot inbound events. The docs also still describe no built-in bot-to-bot bridge and note that broadcast agents do not see each other's responses.

## Best Possible Solution

Keep this issue open. Implement an opt-in Feishu-owned send-time synthetic inbound fan-out after successful outbound group sends, keyed by configured peer bot accounts/groups, with origin metadata, dedupe, loop limits, multi-account tests, outbound reply coverage, and Feishu docs. Add a generic channel/plugin seam only if Feishu and another channel need the same mechanism.

## Risks / Open Questions

- A send-time synthetic bot-to-bot path can create reply loops or token amplification unless it is opt-in, deduped, scoped per group/account, and guarded by consecutive-bot-turn limits that reset on human messages.
- Existing top-level broadcast and Feishu inbound broadcast behavior partially overlap the multi-agent goal, so an implementation should avoid making routing semantics ambiguous.
- A generic cross-channel abstraction may be premature until a Feishu-owned implementation proves which SDK or channel seam is actually needed.
