## Summary

Keep this issue open. Current main still rejects BOM-prefixed markdown frontmatter before skill parsing, and the affected skill then disappears because the loader treats missing `description` as an invalid skill. Closed #45432 remains separate: that was about write/edit tool BOM behavior, not BOM tolerance in skill discovery.

## Best Possible Solution

Fix the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before `---`, then add focused coverage in `src/markdown/frontmatter.test.ts` plus a skill loading or `skills list` path test proving a BOM-prefixed `SKILL.md` remains discoverable. Keep #45432 closed as the separate write/edit-tool BOM report.

## Evidence

## Risks / Open Questions

- `parseFrontmatterBlock` is shared by skills, hooks, and plugin bundle command parsing, so the fix should strip only one leading UTF-8 BOM before the opening delimiter check and avoid broadening parsing for arbitrary invisible characters.
- Because `pnpm` is not available in this checkout, I did not run tests; this review is based on read-only source, docs, history, and a small Node string-decoding check.
