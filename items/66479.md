## Summary

Current main still matches the reported bug: the shared frontmatter parser normalizes newlines but does not remove a leading UTF-8 BOM before `startsWith("---")`, and skill discovery drops a skill when parsed frontmatter lacks `description`. This is a concrete, recent, actionable OpenClaw code bug rather than a ClawHub/plugin boundary issue or duplicate of the broader Windows encoding reports in the provided context.

## Best Possible Solution

Keep #66479 open and fix it in the shared frontmatter parser, ideally by stripping a single leading `\uFEFF` before delimiter detection and adding regression coverage in `src/markdown/frontmatter.test.ts`; because hooks also use the shared parser, verify hook frontmatter behavior remains compatible.

## Evidence

- **Shared frontmatter parser still rejects BOM-prefixed frontmatter:** `extractFrontmatterBlock` only normalizes CRLF/CR, then immediately checks `normalized.startsWith("---")`; a leading `\uFEFF` therefore returns no frontmatter block and `parseFrontmatterBlock` returns `{}`.
  - file: [src/markdown/frontmatter.ts:183](https://github.com/openclaw/openclaw/blob/cf834e2a21eac7fd218989fb5b85e7e8dd4c3246/src/markdown/frontmatter.ts#L183)
