## Summary

Keep #20950 open. Current main still supports a single cron definition store file plus a derived runtime state sidecar; it does not support `cron.storeDir`, `jobs.d`, or one JSON file per cron job. The related state-splitting work is shipped, but that only moved runtime fields to `jobs-state.json` and does not solve the NixOS/GitOps per-file job definition request.

## Best Possible Solution

Keep the issue open until core cron storage gains an additive directory-backed definitions store or maintainers explicitly reject that storage shape. A strong implementation should add a documented `cron.storeDir` or backend contract, deterministic per-file loading, compatibility with existing `cron.store`, preservation of the current `jobs-state.json` runtime sidecar behavior, config schema/help/docs updates, doctor handling, and focused store/service tests. PR #41206 is the current implementation candidate to review, refresh, or replace.
