## Summary

Keep #20950 open. Current main still persists cron definitions through a single `cron.store` file and a derived `jobs-state.json` sidecar; it does not support `cron.storeDir`, `jobs.d`, or one JSON file per cron job. The related PR #41206 is an open implementation candidate in the provided GitHub context, but it is unmerged and the existing maintainer/Codex discussion explicitly kept this issue open because the feature is not shipped.

## Best Possible Solution

Keep the issue open until core cron storage gains an additive directory-backed definitions store or maintainers explicitly reject that storage shape. A likely implementation should add `cron.storeDir` or a store-backend contract, deterministic per-file loading, compatibility with existing `cron.store`, preservation of the current `jobs-state.json` runtime sidecar behavior, config schema/help/docs updates, doctor handling, and focused tests. PR #41206 is the current candidate to review, refresh, or replace.
