## Summary

Keep #20950 open. Current main still implements cron job definitions as a single aggregate `jobs.json` file plus a separate `jobs-state.json` runtime sidecar. It does not expose `cron.storeDir`, `jobsDir`, `jobs.d`, or one-file-per-job loading/saving. Related context also treats #20950 as the canonical open tracker: #70531 was closed into it, #53581 handled the complementary state-split problem, and #41206 remains an open, unmerged implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep #20950 open as the canonical product issue until maintainers land #41206 or a replacement implementation. The eventual fix should add an additive, documented directory-backed cron definitions store with deterministic per-file loading, compatibility with existing `cron.store`/`jobs.json`, preservation of the current `jobs-state.json` runtime sidecar behavior, config schema/help/docs/doctor updates, and focused load/save/gateway tests.
