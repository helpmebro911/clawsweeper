## Summary

Keep open. Current main still parses provider-level `models.providers.*.auth` values, including `token` and `oauth`, but `resolveApiKeyForProvider` and `hasAvailableAuthForProvider` still iterate the normal profile order without applying that provider-level token/oauth override. The submitted PR is directionally relevant, but the review discussion correctly identifies fallback and availability-check gaps, so the best maintainer path is a revised fix rather than closing this PR as implemented.

## Best Possible Solution

Keep the PR open or supersede it with a maintainer patch that centralizes provider-level token/oauth eligibility, applies the same rule in `resolveApiKeyForProvider` and `hasAvailableAuthForProvider`, avoids silent fallback to mismatched credential types, and adds focused regression tests for mixed-profile and no-matching-profile cases under `models.providers.<provider>.auth`.
