## Summary

Keep #66467 open. Current main still depends on an ACP SDK schema that requires `usage_update.used` and `usage_update.size` as integers, while the bundled ACPX/OpenClaw path does not sanitize `used: null` before SDK notification validation. Existing source and tests only prove numeric usage updates, not the reported null-tolerant behavior.

## Best Possible Solution

Keep the issue open and fix the ACP notification boundary. The robust path is to make the bundled ACPX/OpenClaw runtime tolerate non-critical `usage_update` fields before SDK validation can fail, or upgrade/pin an ACP SDK/runtime pair whose schema accepts null/omitted usage fields. Add a regression test that feeds `session/update` with `sessionUpdate: "usage_update", used: null, size: 200000` through the actual ACP runtime notification path and proves no `Invalid params` error destabilizes Discord-bound ACP turns.
