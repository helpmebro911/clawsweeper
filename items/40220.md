## Summary

Keep open. Current `main` still emits a top-level `time` field formatted with a local offset while preserving tslog’s `_meta.date` timestamp, which serializes as UTC. The inconsistency reported in #40220 is still present in the logging path and in current parsing/tests.

## Evidence

- **File logger still writes a separate local-offset `time` field:** The file-log transport formats `logObj.date` with `formatTimestamp(..., { style: "long" })` and then serializes `{ ...logObj, time }`, so `_meta.date` remains in the original object while `time` is added separately.
  - file: [src/logging/logger.ts:178](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/logging/logger.ts#L178)
  - command: `nl -ba src/logging/logger.ts | sed -n '150,240p'`
  - sha: [171322644dec](https://github.com/openclaw/openclaw/commit/171322644dece55402569c52b26370fd81769ca6)
- **Timestamp formatter produces offset-based local timestamps:** `formatTimestamp(..., { style: "long" })` returns `YYYY-MM-DDTHH:MM:SS.mmm±HH:MM`; tests explicitly cover `Asia/Shanghai` as `+08:00`, matching the issue report’s local-offset `time` field.
  - file: [src/logging/timestamps.ts:55](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/logging/timestamps.ts#L55)

  - file: [src/logging/parse-log-line.test.ts:26](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/logging/parse-log-line.test.ts#L26)
