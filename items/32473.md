## Summary

Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions that lack browser device identity. The issue discussion and docs provide HTTPS, localhost/SSH tunnel, trusted-proxy, Tailscale/Traefik, and break-glass workarounds, but the requested homelab/VPS path that keeps device authentication enabled over non-127.0.0.1 HTTP is not implemented or fully superseded.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection model that preserves browser device identity where feasible, or explicitly decide that remote plain HTTP must remain rejected and close only after the narrower warning/guidance work in #56830/#71669 lands or a maintainer records that policy decision.

- Closing as duplicate of #56830 or #71669 would narrow the issue to warning/copy work and leave the broader connection-model decision untracked.
