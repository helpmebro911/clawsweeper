## Summary

The request is a useful plugin-level orchestration idea, but current OpenClaw scope keeps optional multi-agent orchestration out of core and on the plugin/ClawHub path. Current main also closes the specific SDK gap mentioned in the issue by exposing `api.runtime.subagent` plus TaskFlow/webhook helpers to plugins.

## Close Comment

Thanks for the detailed write-up and prototype. This was reviewed by Codex against current `main`, and the request fits the plugin/ClawHub path rather than OpenClaw core.

The scope anchor is `VISION.md:54-61`, which says optional capability should usually ship as plugins, and `VISION.md:101-107`, which explicitly calls out agent-hierarchy frameworks and heavy orchestration layers as out of core scope for now. The community-plugin path is also the documented default in `docs/plugins/community.md:9-22`.

The specific blocker you called out in the issue body is also no longer a core gap on current `main`: plugins now have `api.runtime.subagent` (`docs/plugins/sdk-runtime.md:87-120`, `src/plugins/runtime/types.ts:77-87`, `src/plugins/runtime/index.ts:203-219`) plus TaskFlow/webhook support for gateway-owned flows (`docs/plugins/sdk-runtime.md:143-175`, `docs/plugins/webhooks.md:11-126`). That makes this kind of supervised agent-to-agent relay a better fit for a maintained community plugin than for merging a core orchestration feature.

Closing as ClawHub/plugin scope. If there is still a narrower missing SDK seam after using the current runtime surfaces, please open a focused plugin-SDK issue for that gap.
