## Summary

Keep open. The issue is recent, has a concrete reproduction, and a second user confirmed the same 406 Not Acceptable behavior on v2026.4.14. Current main still constructs StreamableHTTPClientTransport in src/agents/mcp-transport.ts by passing only resolved user headers through requestInit, with no OpenClaw-side regression test or explicit Accept-header merge visible in the touched surface. Upstream current SDK source appears to set the required POST Accept header internally, but I could not verify the checked-out lockfile/package version because local read-only shell inspection failed before execution in this environment, so this does not meet the high-confidence close bar.
