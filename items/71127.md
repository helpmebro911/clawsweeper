## Summary

Current main still treats stuck-session detection as diagnostics only. The heartbeat logs/emits `session.stuck`, the config surface only exposes `diagnostics.stuckSessionWarnMs`, and existing abort mechanisms are user/RPC timeout paths rather than a diagnostic-driven stuck-session recovery hook. The issue is recent, has concrete production logs, and describes a plausible core runtime recovery gap, so it should remain open.

## Best Possible Solution

Keep the issue open. The best product/code path is a core runtime recovery design that wires stuck-session diagnostics to a safe active-run abort/release mechanism, likely behind a conservative `diagnostics.stuckSessionAbortMs` or equivalent setting, with tests proving WARN-only behavior changes to controlled abort plus queue forward progress. The existing `chat.abort`/active-run controller plumbing should be reused where possible instead of adding a separate hidden cancellation path.
