## Summary

Keep this PR open. Current main still has the reported audit behavior, and the related duplicate PR was closed rather than merged. The current PR is a plausible implementation candidate, but its head remains security-sensitive because the shared loopback-equivalence classifier can suppress critical gateway audit findings for `auto` and fallback-prone `custom` binds.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is a conservative effective-bind audit model: downgrade only definitely local-only exposure, fail closed for ambiguous `auto` and fallback-prone `custom` binds, keep `trusted_proxies_missing`, `bind_no_auth`, `loopback_no_auth`, and Control UI origin checks semantically coherent, add regression coverage for loopback, lan, custom loopback, custom fallback/exposed, and auto/container behavior, then update the public audit-check docs if severity changes.

## Risks / Open Questions

- Leaving main unchanged keeps the default loopback Control UI path noisy and still does not warn for some non-loopback empty-`trustedProxies` configurations.

- Same-host reverse proxy deployments make a blanket loopback downgrade a maintainer security taxonomy decision, not a purely mechanical false-positive fix.
