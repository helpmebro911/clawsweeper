## Summary

Keep this PR open. Current main still has the Discord MIME-loss and WebP conversion behavior, and this PR is a plausible fix for the still-open linked bug. It should not be closed as implemented or duplicate yet because no canonical overlapping PR has landed, this PR covers the native Discord interaction path, and the current diff needs maintainer review for shared WebP behavior, security/privacy concerns, and a rebase onto the moved shared media loader.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it only after a canonical fix is chosen. The best path is to rebase, preserve `media.contentType` in all Discord Blob upload paths including standard sends, component sends, and native interaction replies, then handle WebP preservation deliberately: either make it Discord-specific or add a shared media option with verified WebP detection, pixel-limit enforcement, metadata policy, and regression tests. If the shared behavior intentionally changes, update WhatsApp tests and docs; otherwise keep WhatsApp recompression intact and consolidate the overlapping work in #41265 and #41419.

## Risks / Open Questions

- Merging as-is would change shared `loadWebMedia` semantics for every channel, not just Discord, so static WebP that currently compresses under WhatsApp caps could fail with an over-cap error.
- Raw WebP preservation needs an explicit privacy and validation policy: signature verification, fail-closed dimension/pixel limits, and metadata stripping or deliberate preservation tests.
- The PR is dirty/outdated against current main and edits the old WhatsApp media implementation path; it needs a rebase onto `src/media/web-media.ts`.
- Multiple open PRs target the same bug (#41265 and #41419), so maintainers should choose or combine a canonical fix before closing duplicates.
