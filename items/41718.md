## Summary

Keep open. Current main still has the Overview input behavior this PR targets, so the linked bug is not implemented on main. However, the patch as described is not safe or clean to land as-is: it preserves an existing gateway token while changing the WebSocket endpoint, which conflicts with current per-gateway token scoping in source, tests, and docs, and it also carries an unrelated/stale memory-lancedb change.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require a rework before landing: split or drop the memory-lancedb hunk, fix the URL-editing UX without copying credentials across endpoint scopes, and add focused browser/storage regression coverage. A good direction is a draft URL or confirmation flow that keeps typing ergonomic but clears or reloads only the selected gateway's scoped token before save/connect when the normalized endpoint actually changes.

## Risks / Open Questions

- Security review: preserving `settings.token` while changing `settings.gatewayUrl` can send a token intended for one gateway to a different WebSocket endpoint on the next Connect action.
- Storage regression risk: because session tokens are keyed by gateway URL, updating only `gatewayUrl` while keeping the old token can write that token under new or intermediate URL scopes during input.
- Scope risk: the PR bundles an unrelated memory-lancedb cache change with the UI fix, and that area has already changed on current main.
