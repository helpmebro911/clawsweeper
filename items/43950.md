## Summary

Current main still loads local skills for runtime/list/status paths, but `skills.search`, `openclaw skills search`, and the UI search flow remain ClawHub-only. The requested local skill registration or local search-index behavior is not implemented, and the provided related items do not identify a canonical duplicate or superseding issue.

## Best Possible Solution

Keep this issue open. The best path is a core-owned design that either extends `skills.search` and `openclaw skills search` with source-aware local or combined results backed by `loadWorkspaceSkillEntries`, or adds an explicit `skills.register` / `openclaw skills register <path>` flow for safe local roots. The implementation should update protocol schema, CLI/UI behavior, docs, and tests while preserving current ClawHub install/update behavior.
