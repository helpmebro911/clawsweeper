## Summary

Keep open. Current main has partial Chrome MCP mitigations, but the reported existing-session default path still goes through `npx -y chrome-devtools-mcp@latest` via the upstream `StdioClientTransport`, `/tabs` still drops the caller's 300ms reachability budget before `listChromeMcpTabs`, and the browser plugin is not using OpenClaw's process-tree-aware stdio transport. The Windows Defender comments describe a related but separate Windows failure mode and do not resolve the original macOS/Node 25 report.

## Best Possible Solution

Keep this issue open and implement a focused browser-plugin fix: propagate timeout and AbortSignal through Chrome MCP reachability/listing and tab-management calls, ensure failed or timed-out pending sessions close and reap the subprocess tree, replace/adapt the browser Chrome MCP transport with a process-tree-aware implementation through a plugin-safe seam or extension-local equivalent, and add a regression that simulates a hung default `npx`/`client.connect`/`list_pages` path through `/tabs` proving bounded response time and cleanup. Keep `mcpCommand`/`mcpArgs` documented as workarounds, not the primary fix.
