## Summary

Keep open. Current main has useful mitigations for Chrome MCP existing-session hangs, including a 30s attach handshake guard, tool-call timeout recovery, and a documented per-profile `mcpCommand`/`mcpArgs` escape hatch. It still does not fully close the reported default-path hang: existing-session still defaults to spawning `npx -y chrome-devtools-mcp@latest`, `/tabs` still enters Chrome MCP reachability without forwarding its 300ms timeout or request abort signal, and the adjacent tests do not prove a hung `client.connect`/`npx` spawn is bounded and cleaned up without orphaned subprocesses. The Windows Defender comments are useful Windows-specific follow-up, but they do not supersede the original macOS/Node 25 report.

## Best Possible Solution

Keep this issue open and implement a focused browser-plugin fix: propagate route timeout and `AbortSignal` through Chrome MCP attach/list calls, ensure failed pending sessions close the spawned transport with process-tree cleanup or move browser MCP to a plugin-safe stdio transport seam, add a regression for a hung `client.connect` or hung `npx chrome-devtools-mcp` spawn that proves requests return within the browser client budget and subprocesses are not orphaned, and document Windows Defender-specific guidance separately if maintainers confirm that path.
