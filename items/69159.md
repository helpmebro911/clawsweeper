## Summary

Keep open. The PR is a plausible focused fix for a real session-lock contention path, and current main has not implemented the key timeout wiring: embedded attempts still acquire the session write lock without passing the run timeout, so `acquireSessionWriteLock` falls back to its 10s default. Current main does have related newer error-classification work, so the PR likely needs rebasing rather than closing.

## Best Possible Solution

Keep the PR open and ask for a rebase onto current main. The likely maintainer path is to pass `params.timeoutMs` into `acquireSessionWriteLock` from `src/agents/pi-embedded-runner/run/attempt.ts`, reuse or intentionally evolve the existing `SessionWriteLockTimeoutError` contract instead of adding a second error class, add a regression test proving lock wait honors `--timeout`, and decide whether the error helper should be exported through `src/plugin-sdk/agent-harness-runtime.ts`.
