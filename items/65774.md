## Summary

Keep this issue open. Current main still has the core safety gaps reported: startup catch-up can replay overdue or missed cron slots immediately, cron removal does not abort an already-started run, gateway shutdown only stops future cron timers, and there is no public cron/gateway emergency stop for active outbound delivery.

## Best Possible Solution

Keep this open and fix it in cron/gateway core. Add schedule-window-aware startup catch-up or explicit deferred replay semantics, maintain a service-level registry of active cron AbortControllers, make cron.remove/disable and gateway shutdown abort matching active executions, add an explicit emergency stop or pause-all outbound surface, and add regression coverage for an America/Guatemala out-of-hours reconnect plus deletion/abort of an active WhatsApp-style isolated delivery. If #55160 becomes the canonical maintenance-window design, link this issue there but do not close this issue before the relevant safety behavior lands and is verified.

## Risks / Open Questions

- Current main may still send delayed WhatsApp or other outbound cron messages outside the user's intended business hours when a due or missed slot is replayed after reconnect.
- Deleting a cron job, disabling cron, or stopping the gateway may not cancel a delivery already started by the cron runtime.
- A fix needs careful run/session/task-ledger handling so aborted isolated WhatsApp-style deliveries do not leave misleading success, failure, or lost-run state.
