## Summary

Current main still does not provide the safety behavior requested here: startup catch-up can execute overdue cron work immediately, deletion/disable do not abort an in-flight cron run, and gateway shutdown stops scheduling rather than active outbound delivery. The related maintenance-window PR is still only a candidate, and the merged gateway startup race fix does not cover this failure mode.

## Best Possible Solution

Keep this open and fix it in cron/gateway core. Add schedule-window-aware startup catch-up or explicit deferred replay semantics, maintain a service-level registry of active cron AbortControllers, make cron remove/disable and gateway shutdown abort matching active executions, add a real emergency outbound pause/stop surface, and cover an America/Guatemala out-of-hours reconnect plus active delivery abort in regression tests. If #55160 becomes the canonical design, land or narrow it and still add the active-run abort path before closing this.

## Risks / Open Questions

- Delayed cron messages may still be sent outside the user's intended contact window after gateway reconnect or restart.
- Deleting/disabling a cron job, or stopping the gateway, may not cancel a WhatsApp-style isolated delivery that has already started.
- A fix needs careful run/session/task accounting so aborted deliveries do not appear as successful runs or get silently lost.
