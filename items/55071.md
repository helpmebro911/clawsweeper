## Summary

Keep this issue open. Current main still does not implement `metadata.openclaw.activates.plugins` or any per-turn skill-driven plugin tool-schema visibility. The current behavior remains static: skill metadata preserves known `requires`/install fields, plugin tools are resolved before model/context-engine assembly, optional plugin tools are exposed only through explicit allowlists, and the context engine can see available tool names but cannot return a revised tool set.

## Best Possible Solution

Keep this open for maintainer design. The likely core path is to add a typed `metadata.openclaw.activates.plugins` skill metadata contract, preserve it in frontmatter parsing and snapshots, document it, and introduce a request-time tool visibility plan before `createOpenClawCodingTools()`/session creation. That plan should merge activated plugin ids into optional plugin tool visibility without bypassing `tools.deny`, define how bundled plugin skills become visible, and add focused parser, snapshot, tool-policy, deny-precedence, and runner/context tests.

## Risks / Open Questions

- The proposed design needs a clear pre-model activation source, because current skill handling exposes a catalog for the model to read rather than semantically matching skills before tool construction.
- Any implementation must preserve security semantics: explicit `tools.deny` and plugin disablement must still win over activation metadata.
- Per-turn tool-set changes need deterministic ordering and cache-aware reporting so prompt-cache behavior remains stable.
