## Summary

Keep open. Current `main` still does not implement the PR's runtime-port discovery for TUI connections, so the custom-port mismatch this PR targets is not closed out on shipped code. The PR thread also reflects substantive follow-up fixes and at least one remaining review concern, which makes conservative auto-closure inappropriate.

## Evidence

- **TUI still resolves from config/env only:** On current `main`, `resolveGatewayConnection()` loads config and immediately builds the gateway URL; there is no pre-resolution step that reads active gateway runtime state or a lock payload to override the port.
  - file: [src/tui/gateway-chat.ts:249](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/tui/gateway-chat.ts#L249)

- **Port resolution has no live-runtime fallback:** `resolveGatewayPort()` only checks `OPENCLAW_GATEWAY_PORT`, then `config.gateway.port`, then falls back to `18789`. That means a separate TUI process still has no built-in way to discover a gateway started with `--port` unless the env/config also matches.
  - file: [src/config/paths.ts:285](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/config/paths.ts#L285)
