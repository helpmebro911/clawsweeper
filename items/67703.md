## Summary

Current main still does not solve the reported `auth.mode="none"` internal probe failure. `auth.mode="none"` succeeds only at the generic auth layer, but the loopback probe still deliberately sends no device identity when no token/password is present, `method: "none"` is not counted as shared auth, and non-Control-UI operator clients without shared auth still hit the missing-device path that closes with `1008: device identity required`. The related #69066 RFC is relevant design work, but it remains open and is not a shipped fix for this concrete probe/followup failure.

## Best Possible Solution

Keep this issue open. The best fix is a narrowly scoped internal-service identity path, or a tightly constrained loopback-only internal caller exception, for Gateway-owned probe/followup/backend RPC clients under `gateway.auth.mode="none"`. Add regression coverage for `probeGateway` with no token/password/device identity and for the affected backend approval/followup RPC path, while preserving fail-closed behavior for remote, browser-origin, node-role, and non-internal clients. Align the durable design with #69066.
