## Summary

Keep this issue open. Current main still accepts `gateway.auth.mode="none"` only at the shared auth layer; unauthenticated loopback probes still omit device identity, and non-Control-UI probe/backend clients without device identity still hit the 1008 `device identity required` path. The related #69066 service-identity RFC is relevant, but it is open design work rather than a shipped fix for this concrete failure.

## Best Possible Solution

Keep this issue open. The best fix is to add explicit internal service identity for Gateway-owned loopback callers, or a narrowly scoped interim auth-none path for internal probe/followup/backend RPC clients, with regression tests proving `probeGateway` and backend `callGateway` work without token/password while remote, browser-origin, node-role, and non-internal clients still fail closed. Align the final shape with #69066 rather than expanding shared-secret exceptions ad hoc.
