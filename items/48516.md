## Summary

Keep open. Current main has useful mitigation work: Android notification forwarding is now opt-in and configurable with allowlist/blocklist policy, and gateway notification system events are marked untrusted. But the central requested behavior is not fully implemented: there is no default native-channel package exclusion for WhatsApp/Telegram/Discord/Signal, the gateway still accepts every `notifications.changed` package into a node session without server-side package filtering, and heartbeat handling can still turn untrusted notification events into a heartbeat run rather than dropping native-channel duplicate notifications. The issue is also only about 40 days old and the comments correctly frame this as a cross-session trust/routing problem, so conservative cleanup should leave it open for maintainer/product judgment.

- Leaving the item open is appropriate because closing as implemented could hide a privacy/trust issue that still depends on operator configuration rather than a default guardrail.
