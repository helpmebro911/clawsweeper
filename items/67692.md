## Summary

Keep this PR open. Current main still does not lazy-initialize ACP for an agent configured with `runtime.type: "acp"` on the TUI/webchat reply-dispatch path when no ACP metadata exists, and the PR is a plausible implementation candidate that needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open and review it as the active implementation candidate. The best path is a narrow, policy-safe lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for agents whose resolved config has `runtime.type === "acp"`, after existing policy/send/no-op gates as appropriate, with normalized ACP fields, unconditional re-resolve after initialization, and focused tests for ACP-configured versus non-ACP agents. Treat CLI lazy-init parity and any branch-history cleanup as separate maintainer review scope unless intentionally included.
