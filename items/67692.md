## Summary

Current main still has the reported ACP lazy-init gap: the reply-dispatch ACP path returns unhandled when no ACP session metadata exists, so a TUI/WebChat turn for an agent configured with `runtime.type: "acp"` still falls through to embedded execution. The open PR remains a plausible implementation candidate rather than obsolete or superseded, with security-sensitive ordering and branch-history questions still needing maintainer review.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best path is a narrow lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for sessions whose resolved agent has `runtime.type === "acp"`, after send-policy, ACP policy, and non-empty-turn gates, with normalized `runtime.acp` values, unconditional re-resolve after initialization, focused tests for ACP and non-ACP agents, policy-denied/init-failure/empty-turn coverage, and explicit maintainer judgment on whether CLI parity belongs in the same PR. Resolve the #68516 branch-history concern before merge.

## Risks / Open Questions

- The PR changes when a normal TUI/WebChat turn can start an external ACP runtime based on agent config; policy checks, send-policy checks, and empty-turn/no-op checks need to remain before any `initializeSession()` side effects.
- The current PR implementation still deserves maintainer scrutiny around inline-image-only turns and policy-denied fallback semantics, because returning `null` may silently fall back to embedded instead of surfacing an ACP policy error.
- The branch-history discussion around changes resembling #68516 should be resolved before merge, even though the current PR head no longer shows workflow, dependency, lockfile, package-publishing, or third-party code-execution changes.
