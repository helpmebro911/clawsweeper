## Summary

Keep open. Current main still lacks the requested browser-specific or per-channel lane routing: channel-triggered agent work defaults into the process-wide main lane unless a trusted caller explicitly supplies a lane, and the config/schema/docs only expose global main/subagent/cron concurrency knobs.

## Best Possible Solution

Keep this issue open. The best implementation path is a core scheduling/config change: add a validated custom-lane concurrency surface, apply it on gateway startup and reload, route inbound runs by channel/account/session policy before runEmbeddedPiAgent enters the global lane, and cover the interaction with per-session serialization, agents.defaults.maxConcurrent, cron/subagent lanes, and browser cleanup. Browser-tool auto-detection could be a follow-up, but pre-run lane assignment is the safer way to prevent main-lane starvation.
