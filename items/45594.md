## Summary

Keep open. Current main still does not honor the PR's central case: `cron.retry.retryOn` accepts/documents `server_error`, but structured Codex/OpenAI `server_error` payloads are still classified as `timeout`, cron retry detection only checks raw `5xx` text for the `server_error` bucket, and cron protocol state cannot surface `lastErrorReason: "server_error"`.

## Best Possible Solution

Keep the PR open and have it updated against current main: add a first-class `server_error` failover reason if maintainers still want that public state, map it to fallback status 500, expose it in cron protocol state, make cron retry matching consult the failover classifier before raw regex fallback, and keep regression coverage for structured Codex payloads plus explicit 4xx precedence.
