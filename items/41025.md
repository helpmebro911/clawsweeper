## Summary

Keep open. Current main still intentionally flattens remote markdown images in Control UI chat, tests and docs assert that no-remote-fetch policy, and #40945 remains open with this PR as a concrete implementation candidate. The PR should not be merged as-is because it only changes the markdown renderer/tests while current Gateway CSP still blocks remote images, and the proposed http/https image rendering needs explicit product/security approval.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945, but require an updated design before merge. A complete solution should align the markdown renderer, Control UI CSP, docs, and tests around an explicit maintainer-approved remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, or documented privacy tradeoff. If maintainers decide the no-remote-fetch policy should remain, close this PR only after recording that product decision on #40945 or the broader inline media issue #64607.

## Risks / Open Questions

- Allowing arbitrary remote markdown images lets model/chat content trigger third-party browser requests, exposing operator IP address, user agent, and read timing; referrerpolicy only reduces Referer leakage.
- The PR allows plain http:// image URLs, which can create mixed-content and privacy problems if CSP is later widened on HTTPS deployments.
- Current Gateway CSP would still block http(s) images, so merging only the renderer/test change may not deliver the visible feature in the served Control UI.
- The change reverses an intentional no-remote-fetch hardening policy documented in docs and changelog without a maintainer-approved policy such as HTTPS-only, allowlist, opt-in, or media proxying.
