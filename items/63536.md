## Summary

Keep this PR open. Current main still routes `loadSessionEntry` and direct history reads through the freshest-`updatedAt` match, so the reported exact-key/legacy-alias problem remains plausible, but the submitted one-file patch is not safe to merge as-is because it would bypass existing freshest duplicate behavior for canonical/case-variant rows.

## Best Possible Solution

Keep this PR open and request changes: prefer exact matches only for explicit non-canonical archived/legacy/compound keys, preserve current freshest duplicate and case-variant behavior for canonical logical keys, and add regression coverage for both the archived-key click case and the existing freshness cases.
