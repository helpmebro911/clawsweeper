## Summary

Keep open. Current main still has the reported behavior shape: wake requests can be queued while a heartbeat handler is running and drained immediately afterward, and targeted/session exec wakes bypass the interval not-due guard before calling the heartbeat runner.

## Best Possible Solution

Keep this issue open and fix the heartbeat scheduler/wake path with targeted regression coverage. The likely maintainer-safe direction is to add an in-window cooldown or dedupe for automatic external exec/session wakes queued during an interval heartbeat, while preserving explicit immediate wake semantics for manual/system/cron/plugin `now` modes.

## Risks / Open Questions

- A blanket skip for all non-interval wakes could regress documented immediate behavior for `openclaw system event --mode now`, cron `--wake now`, and manual/plugin wake APIs; the fix needs to distinguish automatic external wake noise from explicit immediate wake modes.
- Regression coverage should assert that the extra `runOnce()` call is avoided, not only that duplicate outbound payloads are suppressed after the model run.
