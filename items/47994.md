## Summary

Keep open. Current main still has the sticky fallback shape: runtime/fallback provider-model values are written into session state, and later model resolution prefers those stored runtime fields. The later #68991 PR is clearly related, but it is also open/unmerged and there is no maintainer signal in the provided context that it supersedes #47994 enough to auto-close this active, focused bug-fix PR.

## Best Possible Solution

Keep #47994 open for maintainer review while consolidating it with #68991. The best product direction is to make automatic fallback per-request only: usage/cost should record the model that actually ran, but the persisted next-turn session selection should remain the configured or explicitly selected model. Maintainers should pick either #68991's explicit `persistedProvider`/`persistedModel` contract or #47994's fallback-origin flag, then require regression coverage across agent command, auto-reply, followup, cron, status, sessions display, and gateway session rows before closing the other PR as superseded.
