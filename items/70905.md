## Summary

Keep this issue open. Current main still does not expose a stable `agentId` on `ChannelOutboundContext`, even though upstream direct outbound paths already carry agent/session agent metadata and use it for media scoping. This remains an actionable public channel plugin SDK gap rather than a ClawHub-only plugin request or a duplicate of the provided related items.

## Best Possible Solution

Add optional `agentId?: string` to `ChannelOutboundContext`, derive it in outbound delivery from the existing session/mirror context, thread it through `ChannelHandlerParams` and `createChannelOutboundContextBase`, cover every outbound adapter callback with tests, and document that `agentId` is stable routing/attribution metadata distinct from display `identity`.

## Risks / Open Questions

- The implementation needs a documented precedence rule when both `session.agentId` and `mirror.agentId` exist and differ.
- Because this is a public plugin SDK type, it should include focused contract tests and docs, not only internal plumbing.
- All final adapter surfaces should be covered: `sendText`, `sendMedia`, `sendPayload`, `sendFormattedText`, `sendFormattedMedia`, and `renderPresentation`. System or human-initiated sends with no agent in scope should leave the field absent.
