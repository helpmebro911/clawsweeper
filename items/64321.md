## Summary

Keep open. Current `main` still builds every `sessions.list` row before applying `limit`, and row hydration can still fall back to reading each transcript file in full when session usage metadata is missing or stale. That means the reported large-history slowdown remains plausible on current `main`, even though the standalone CLI `openclaw sessions` path now reads only the session store.

## Evidence

- **Issue discussion:** The report measured `openclaw sessions list` at 23s for 124 session files / 112MB on OpenClaw 2026.4.9, and the reporter followed up on 2026-04-14 that they were still seeing about 20s on 100+ sessions with the nightly cleanup workaround. That leaves a recent, unrefuted performance complaint in the issue itself.
- **Gateway sessions.list still hydrates full rows for all sessions:** `sessions.list` still loads the combined session store and immediately delegates to `listSessionsFromStore`, so the gateway/tool path remains the relevant implementation for the issue title.
