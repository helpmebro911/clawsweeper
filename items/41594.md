## Summary

Keep open. Current main has partial mitigations for prompt pressure, including global bootstrap-injection controls and `compat.supportsTools: false` for model/provider entries, but it does not implement the requested per-model or alias-scoped `contextProfile` that independently controls workspace files, tools, and skills. The related open #59168 may be relevant groundwork for distinct named model profiles, but it is broader and does not supersede this specific prompt-injection/tool-visibility request.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible core config design for model or alias scoped prompt profiles, then schema/types/docs/tests and runtime wiring through bootstrap resolution, system-prompt skills inclusion, and effective tool assembly. Coordinate the identity/keying part with #59168 if distinct aliases for the same upstream model are needed.

## Risks / Open Questions

- The existing `compat.supportsTools: false` path partially addresses tool-schema exposure for model entries, so a future implementation should preserve that behavior and avoid creating conflicting tool-disable semantics.
- Alias-scoped profiles may interact with the open internal-model-key work in #59168; duplicating the same upstream API model under multiple user-facing profiles should be designed deliberately.
