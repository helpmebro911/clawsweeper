## Summary

Keep this issue open. Current main has adjacent safeguards, including managed-service config preflight and a separate fix for the cited lossless-claw `info.id` regression, but it still does not expose `openclaw gateway preflight [--json]` or a shared no-bind startup validation routine that resolves plugins/context engines and checks port bindability before serving traffic.

## Best Possible Solution

Keep this issue open and implement `openclaw gateway preflight [--json]` as a first-class Gateway CLI command. The safest product path is to extract a shared no-bind startup validation routine used by both `gateway run` and the new command: parse/read config, run schema and SecretRef/auth checks, load/register startup plugins, resolve the selected context engine factory, check configured port bindability without accepting connections, and return a stable structured report plus exit code. Add docs at `docs/cli/gateway.md` and focused CLI/startup tests for JSON output and failure cases. A later `openclaw update --safe` can then compose stop, package install, preflight, restart, and rollback around that routine.

## Risks / Open Questions

- A future implementation should reuse the real config/auth/plugin/context-engine startup paths; a separate schema-only validator would recreate the drift problem described in the issue.
- Directly invoking full startup as preflight would be unsafe without refactoring because current startup can persist auth/config seeds, run startup maintenance, stage bundled runtime dependencies, and run session migration hooks.
- The existing `openclaw update` preflight wording is for dev worktree build/lint validation, not the requested live Gateway no-bind validation or `openclaw update --safe` flow.
