## Summary

Keep this issue open. Current main still exposes cron and message delivery outcomes through coarse run statuses, delivery booleans, four delivery statuses, and raw error strings; it does not provide the shared normalized delivery reason codes requested for cron runs, status/UI surfaces, JSON output, or messaging actions.

## Best Possible Solution

Keep this issue open. The best path is an additive normalized delivery outcome model shared across cron run history, message action results, background task delivery where applicable, Gateway protocol schemas, Control UI filters/cards, CLI JSON/text output, and docs. Preserve compatibility fields, add stable reason codes such as `not_configured`, `channel_unavailable`, `skipped`, `attempted_failed`, and `rate_limited`, and cover config/routing failures, outbound adapter failures, provider/rate-limit failures, best-effort skips, fallback delivery, and UI/API filtering in focused tests.

## Risks / Open Questions

- A future fix should be additive because existing clients already consume `status`, `deliveryStatus`, `delivered`, `deliveryError`, and task delivery status fields.
- Reason classification should preserve raw provider/plugin diagnostics while adding stable operator-facing reason codes.
- Partial implementation would likely create another inconsistent status surface because the request spans cron state, run logs, Gateway protocol, CLI JSON/text output, messaging actions, Control UI, and docs.
