## Summary

Keep this issue open. Current main still routes direct Google Gemini through tagged reasoning and still injects the `<think>/<final>` prompt wording that the report identifies as interfering with native tool calls. The Google transport has native `functionDeclarations` and `functionCall` parsing, but current tests still assert the tagged reasoning mode rather than proving the reported behavior was fixed.

## Best Possible Solution

Keep this issue open and fix the Google provider/reasoning prompt path. The likely maintainer path is to make direct Google Gemini use native reasoning output when native function calling/tools are active, or amend the tagged reasoning prompt so provider function calls are explicitly outside the `<think>/<final>` text contract and must be emitted through Gemini function calling. Add focused regression coverage for Google `resolveReasoningOutputMode`/system prompt composition with tools, while keeping the existing Google transport tests that prove `functionDeclarations` and parsed `functionCall` blocks still work. If the provider needs tool availability to choose the mode, extend `ProviderReasoningOutputModeContext` additively instead of adding more core Google-specific policy.
