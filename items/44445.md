## Summary

Keep open. Current `main` still writes raw user message text into session-memory without stripping inbound metadata, so the requested behavior is not implemented. The PR also expands scope beyond the title by adding scheduled-reset hook dispatch, and the existing review comments identify credible behavior regressions that are not disproven by current `main` code.

## Risks / Open Questions

- If merged as-is, the scheduled-reset branch would fire `command:new`/`before_reset` hooks on ordinary post-expiry messages, changing behavior beyond the PR title and bypassing the usual manual-command authorization path.
- If the PR strips standalone `[message_id: ...]` lines after metadata blocks, it can delete legitimate user text that intentionally begins with that line shape.
- If message text is normalized before the existing `!text.startsWith("/")` filter, real slash-prefixed user content can disappear from memory snapshots.
