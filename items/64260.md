## Summary

Keep this issue open. Current main still exposes `llm_output` with provider/model identifiers, resolved/harness metadata, assistant text, and usage counts only. The requested context window, model max output token limit, effective context budget, and utilization fields are not present in the hook contract or the current PI, CLI, and Codex emitters, even though model metadata exists elsewhere.

## Best Possible Solution

Keep this issue open and implement a focused plugin hook contract update: add optional native context window, effective context budget, and model max output token fields to `PluginHookLlmOutputEvent`; populate them in the PI embedded runner, CLI harness, and Codex app-server harness; add hook emission and contract tests; and update the public plugin hooks docs. Treat utilization as derived or optional unless the runtime can provide a semantically reliable context-used value, and coordinate naming with #64327.

## Risks / Open Questions

- The hook contract should remain additive and optional so existing third-party plugins keep working.
- Field naming should be coordinated with #64327 so native context window, effective runtime cap, and max output-token limit do not become divergent plugin API concepts.
- A computed utilization field needs a precise numerator definition because aggregate usage can include output and cache tokens and may not equal live prompt-context occupancy.
