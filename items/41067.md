## Summary

Keep this PR open. Current main still lacks the central dashboard reconnect recovery: hello snapshots do not expose active chat runs, the Control UI still clears chatRunId on reconnect, the chat view busy state still ignores abortability, and shutdown does not abort active chat runs with a durable transcript notice. The PR is also paired with the still-open same-author dashboard issue #41064, so it should be reviewed or updated rather than closed as obsolete.

## Best Possible Solution

Keep this PR open as the implementation candidate for #41064 and the related restart-notice work. The best path is to rebase it onto current main, address the remaining review comments by choosing the correct active run and making transcript injection failures fall back safely, preserve the existing SessionManager-based transcript contract, regenerate/check protocol artifacts, and run the targeted gateway/UI/protocol checks before maintainer review.

## Risks / Open Questions

- The PR head still has unresolved review concerns from the provided discussion: recovery chooses the first active run for a session instead of the newest/existing matching run, which can leave the UI tracking a stale run when multiple runs exist for one session.
- The restart-sentinel transcript injection path described in review can drop the fallback system event if filesystem/session transcript injection throws instead of returning false.
- The PR is stale against current main and touches protocol/generated client surfaces, so it needs a rebase plus protocol/generated artifact validation before merge.
- Shutdown transcript injection is a security-sensitive filesystem path; maintainers should verify it preserves 0o600 permissions, idempotency, and the SessionManager parentId transcript contract.
