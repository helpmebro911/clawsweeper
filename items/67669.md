## Summary

Keep this PR open. Current main still does not implement `openclaw models custom-test`, and the existing `models status --probe` flow is related but not equivalent: it probes auth/profile health through the runtime runner using selected candidate models, rather than verifying every configured custom provider model. The latest PR diff is a plausible feature contribution, and it does address the earlier Greptile network-error detail by forwarding `result.error`, but it still has functional and security-sensitive gaps that need maintainer review before merge.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature is accepted, rebuild it around the existing runtime probe path or a shared runtime probe helper, enumerate every configured custom provider model using `model.api ?? provider.api`, include Responses-family APIs, preserve clean JSON/stdout and non-zero failure exits, redact sensitive endpoint fields, honor provider request transport semantics, and add focused tests plus docs for the new CLI surface.

## Risks / Open Questions

- Merging as-is could create false health results for custom providers that rely on runtime request settings such as headers, auth overrides, proxy/TLS, or private-network policy.
- `--json` output can be non-parseable because human progress text is logged before JSON, and JSON mode returns before applying a non-zero exit for failed probes.
- `--json` output includes raw provider base URLs, which this repo treats as potentially sensitive URL fields.
- The command can silently skip supported custom providers using `openai-responses` or `openai-codex-responses`, and can miss or mis-probe models that use model-level `api` overrides.
- There are no tests or docs in the PR diff, so the new CLI behavior, automation contract, redaction behavior, and failure exits are not locked down.
