## Summary

Keep open. Current main only partially mitigates the report: zero-valued poll defaults no longer trip `message.send`, and generic Discord `components` schema exposure appears reduced, but the shared message tool still exposes poll creation fields on the same schema surface as ordinary sends and the send path still rejects populated poll params. The provided GitHub context also shows open PR #46455 explicitly uses closing syntax for this issue, so the issue should remain open until that PR is reviewed, landed, superseded, or closed by maintainers.

## Best Possible Solution

Keep this issue open. Maintainers should review, update, land, or explicitly supersede PR #46455; the likely product/code path is to scope poll creation fields to poll-capable actions or strip model-populated poll noise on `action="send"`, preserve deliberate poll validation on the poll path, and add focused regression tests for Telegram/Feishu/Discord send-with-file plus Discord empty-modal-skeleton behavior. The docs and prompt hints should also be reconciled with the shared `presentation` contract so generic sends do not invite provider-native component payloads by default.
