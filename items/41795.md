## Summary

Keep this PR open. Current main already has robust Telegram support for standard HTTP_PROXY/HTTPS_PROXY env proxy routing, including tests and graceful EnvHttpProxyAgent fallback, but it does not implement the PR's Telegram-specific OPENCLAW_TELEGRAM_PROXY override. The PR also targets older src/telegram paths while Telegram now lives under extensions/telegram, and review comments flagged a real malformed-explicit-proxy failure mode that still needs a deliberate design decision.

## Best Possible Solution

Keep open and ask for a narrow refresh against extensions/telegram. If maintainers want a Telegram-only env override, implement it in the current Telegram transport layer with precedence config proxy > OPENCLAW_TELEGRAM_PROXY > standard HTTPS_PROXY/HTTP_PROXY, wire it through monitor/send/probe/audit via the shared resolver, and add tests for blank values, malformed explicit env proxy handling, standard env fallback, and config precedence. If maintainers decide standard HTTP_PROXY/HTTPS_PROXY is enough, close with an explicit product decision and update docs instead.

## Risks / Open Questions

- The branch is stale against current main because Telegram has moved from src/telegram to extensions/telegram; the PR should not be merged as-is without rebasing or replacing it against the current plugin-owned files.
