## Summary

Keep this issue open. Current main still only warns during ordinary config loads and only hard-blocks selected service/process mutations. The linked auto-downstamp PR #46336 was closed unmerged, and the later GPT-5.4 hardening PR #70743 shipped adjacent runtime fixes but not the reported future-stamped-config delivery gap.

## Best Possible Solution

Keep the issue open until maintainers either extend the future-config guard/remediation to ordinary gateway startup paths that can run channel delivery, or add a focused regression/manual proof showing the current service/process guard fully eliminates the Discord/OpenAI-Codex silent-delivery failure. The follow-up should also make the startup warning actionable for this runtime path, for example by pointing users to the split-brain install troubleshooting and repair flow.

## Risks / Open Questions

- The observed OpenAI-Codex/Discord failure may be a split-brain install symptom rather than `lastTouchedVersion` directly affecting delivery, so the best fix should include a focused regression or manual reproduction before choosing downstamp versus startup refusal.
- Hard-failing every newer config read could break legitimate read-only inspection or intentional downgrade recovery; any remediation should be scoped to gateway/runtime paths that can silently deliver from an older binary.
