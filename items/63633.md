## Summary

Keep this PR open. The underlying Telegram media SOCKS proxy bug is still present on current main, and the linked bug was closed specifically because this PR is the canonical implementation candidate. The PR is not ready as-is because current main's guarded media path still rejects non-http(s) explicit proxies, and the PR still claims unsupported SOCKS variants.

## Best Possible Solution

Keep this PR open and request focused rework against current main: implement dependency-supported socks5:// and socks:// handling in the guarded explicit-proxy media path, preserve Telegram dispatcher HTTP/1 and pool behavior, remove unsupported SOCKS4/SOCKS5H claims unless separately implemented, and add regression coverage for channels.telegram.proxy: socks5:// media downloads plus guard behavior.
