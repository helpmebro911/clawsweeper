## Summary

Keep #10659 open. Current `main` has meaningful SecretRef support, documented credential surfaces, and plugin hooks that explain why the linked `openclaw-credential-vault` and `openclaw-secrets-hardening` workarounds are useful today, but it still does not ship the native masked-secrets behavior requested here: no `{{secret:NAME}}` command/runtime substitution, no `secrets.mask`/`maskedSecrets` config, no native command blocking for `env`/`.env` reads, and no guaranteed redaction of exec/process output before the agent sees it. The related core PRs in the provided context (#9271 and #22744) are closed unmerged, and the open roadmap/RFC items (#11829, #59165) do not yet supersede this issue as shipped behavior.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secrets or credential-broker flow for agent-executed commands/process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue to acknowledge SecretRefs, `openclaw-credential-vault`, and the Linux user-isolation guide as useful mitigations, but do not treat them as closing the core security/product request.
