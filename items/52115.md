## Summary

Current main does not yet fix the reported gateway-tool freshness issue. The builtin backend remains the default; gateway `memory_search` reuses a cached default manager, and after an index exists `search()` starts dirty `onSearch` sync in the background before querying current rows. The sync path would prune deleted or moved files, but because it is not awaited the first search can still return stale or ghost hits. QMD has a separate awaited dirty-search path, but that does not cover default builtin users.

## Best Possible Solution

Keep this issue open and fix the builtin/default `MemoryIndexManager` path used by gateway `memory_search`. The likely maintainer change is to make agent-facing search await pending or newly triggered sync when builtin memory is dirty or sessionsDirty, then add regression tests for a cached default manager where the first search after memory file create/delete/rename reflects disk and returns no nonexistent paths.

## Risks / Open Questions

- A fix that always blocks on full sync could add noticeable latency; it should await only when `dirty`, `sessionsDirty`, or a pending search/watch sync exists, or expose an explicit freshness mode for the tool.
- Regression coverage should target a cached builtin default manager and verify the first search after memory file create, delete, and rename reflects disk state and returns no nonexistent paths.
