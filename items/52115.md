## Summary

Current main still appears to have the reported default builtin `memory_search` freshness gap. The issue discussion already narrowed the symptom to the long-lived gateway manager scheduling dirty search refresh in the background, and current source still does that for the builtin backend: cached managers only force a synchronous sync for an empty index, then fire-and-forget dirty `onSearch` sync while immediately querying existing rows. QMD has a separate awaited dirty-search path, but builtin remains the default backend, so QMD work does not close this issue.

## Best Possible Solution

Keep this issue open and fix the builtin/default memory manager path. The likely maintainer fix is to make agent-facing `MemoryIndexManager.search()` await pending or newly triggered sync when `dirty` or `sessionsDirty` is set, or add an explicit freshness mode used by the `memory_search` tool. Add regression coverage for a long-lived builtin manager where the first search after creating, deleting, and renaming memory files observes the updated index and returns no nonexistent paths; keep QMD behavior covered separately.
