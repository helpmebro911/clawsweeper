## Summary

Keep this issue open. Current main still has the stale-first-search behavior on the default/fallback builtin memory manager: a long-lived manager marks file changes dirty, but `search()` starts search-triggered sync in the background and immediately queries the existing index. QMD has stricter freshness semantics, but the default backend is still builtin and the builtin regression remains covered as non-blocking behavior.

## Best Possible Solution

Keep this issue open and fix the builtin memory manager/tool path. The likely maintainer path is to make `MemoryIndexManager.search()` await pending or newly triggered sync when `dirty` or `sessionsDirty` is true for agent-facing `memory_search`, or add an explicit freshness mode used by the tool, then add regression coverage for a long-lived manager where the first search after creating, deleting, or renaming a memory file observes the updated index and returns no nonexistent paths.

- QMD already awaits dirty search sync, so regression coverage should target the builtin manager and fallback wrapper without regressing QMD behavior.
