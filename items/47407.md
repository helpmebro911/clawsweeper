## Summary

Keep open. Current main still accepts `cron edit --model` by normalizing and persisting the payload model without any edit-time allowlist warning. Runtime cron execution does warn and fall back when `payload.model` is not allowed, and docs describe that runtime behavior, but that does not solve this PR's stated UX gap. The PR is plausible but needs rework to match the runtime resolver and gateway context issues raised in review.

## Best Possible Solution

Keep the PR open, but require a revised implementation that validates the requested model against the same effective gateway/runtime model-selection contract used by isolated cron execution, with focused tests for aliases, bare model names, configured defaults, auth-profile suffixes, and remote gateway behavior. If that cannot be done locally in the CLI, add a small gateway-side validation/preview path rather than duplicating resolver logic in `register.cron-edit.ts`.
