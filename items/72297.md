## Summary

Keep this PR open. Current main still does not solve the linked QMD name-mismatch failure when `qmd collection list --json` only returns collection names, and this PR is a focused implementation candidate for open #71783. However, the current PR approach has an unresolved safety concern because it would parse a collection name from unstructured `qmd collection add` stderr and then remove that collection without verifying path/pattern metadata first.

## Best Possible Solution

Keep this PR or the linked issue active, but require a safer implementation before merge: only rebind a different-named collection when OpenClaw can verify that the stale collection name corresponds to the same managed path and pattern through structured qmd metadata, or keep refusing automatic removal when metadata is unavailable. Add coverage for the #71783 `Name: life-main` conflict and for refusing to remove an unverified stderr-only name.
