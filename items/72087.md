## Summary

Keep #72087 open. The report is recent, detailed, contributor-authored, and describes a concrete CLI entry-path failure that is still plausible on current main. Current main still routes the package binary through `openclaw.mjs` into the `dist/entry.js` main branch before `runCli`, and the existing tests/docs confirm Codex OAuth image generation is intended but do not cover the reported Linux `dist/entry.js`/Codex OAuth/IPv6-unreachable failure boundary. I found no protected-label or maintainer-author reason to block review, but also no strong source/test/history evidence that current main fixes the observable bug well enough to close it.

## Best Possible Solution

Keep the issue open and investigate as a real CLI bootstrap bug. The best path is to reproduce against current main's built `openclaw.mjs`/`dist/entry.js` on Linux with an OpenAI Codex OAuth profile, isolate the differing entry side effect (respawn, profile/env handling, compile cache, warning filter, global Undici dispatcher, or DNS family behavior), then add a regression test that exercises the packaged entry/main path with a mocked Codex SSE response or a controlled network/dispatcher fixture. If current main already fixes it incidentally, maintainers should attach that exact proof and close with the fixing SHA.
