## Summary

Keep this issue open. Current main has meaningful partial session management across Gateway RPCs, Control UI, Web Chat, TUI, and CLI list/cleanup, but it still does not satisfy the broader Topic/Session Management UI request: first-class description/tags/createdAt/messageCount metadata, import/export, deleted-topic archive restore, and create/rename/delete management workflows across the requested UI/CLI surfaces remain incomplete or undefined.

## Best Possible Solution

Keep this issue open, but narrow it into concrete maintainer-owned work: add a first-class Control UI create/manage workflow around sessions.create, sessions.patch, and sessions.delete; decide whether description, tags, createdAt, and messageCount belong in GatewaySessionRow; add TUI/CLI create/rename/delete/archive actions only if maintainers want those surfaces; and separately design deleted-session archive restore plus import/export semantics with docs and tests.

## Risks / Open Questions

- The issue is broad and mixes already-shipped listing/switching/labeling behavior with still-missing product/API decisions, so implementation should be split into narrower follow-ups before coding.
- Adding description, tags, createdAt, and messageCount would change the Gateway session-row contract and should be additive with protocol docs and focused UI/API tests.
- Import/export and deleted-session archive restore need explicit storage, permission, conflict-resolution, and UI semantics before they can be safely implemented.
