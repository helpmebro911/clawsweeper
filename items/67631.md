## Summary

Keep open. Current main still exposes the reported ownership conflict: the macOS app has its own `ai.openclaw.mac` LaunchAgent, but local gateway management still goes through the shared `ai.openclaw.gateway` service, and CLI doctor repair paths can still install or rewrite that gateway service without detecting that the Mac app LaunchAgent is present. The newer launchd fix on main improves plist-vs-launchd supervision diagnostics, but it does not address the duplicate Mac app/CLI gateway ownership boundary described here.

## Best Possible Solution

Keep this issue open and make the macOS launchd ownership boundary explicit. The conservative fix is for CLI doctor/gateway repair to detect the Mac app LaunchAgent (`ai.openclaw.mac`) and either skip installing or rewriting `ai.openclaw.gateway` with a clear note, or coordinate through one canonical owner. Add regression coverage for doctor repair and launchd scanning, preserve the existing attach-existing behavior, and update the macOS/doctor docs with safe guidance for Mac app users.
