## Summary

Keep open. Current main still runs Active Memory through the same blocking before-prompt embedded-agent path, and the source shows timeout/fail-open mitigation plus tuning docs rather than a proven fast lightweight bootstrap. The fresh v2026.4.24 discussion evidence says the symptom persisted after the original report, so this should remain a focused profiling/optimization task.

## Best Possible Solution

Keep this issue open as the focused Active Memory lightweight embedded-runner cold-path task. Add stage timings around lane wait, runtime plugin load, hook/model/harness/auth resolution, context-engine initialization, bootstrap assembly, tool materialization, session manager/resource loader/session creation, and first prompt start; then cache or bypass safe per-run invariants for lightweight blocking runs while preserving the hard timeout/fail-open behavior. Update the public Active Memory docs with measured timeout and latency guidance after the root cause is fixed.
