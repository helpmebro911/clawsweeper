## Summary

Keep open. Current main still runs Active Memory as a blocking before_prompt_build embedded-agent call, and the latest local code shows timeout/fail-open mitigation plus some gateway/cron-adjacent latency work, not a proven fast lightweight embedded bootstrap path for blocking plugins. The issue discussion also includes fresh v2026.4.24 evidence that the symptom persisted after the original report.

## Best Possible Solution

Keep this issue open as a focused Active Memory embedded-runner cold-path profiling task. Add stage timings around lane wait, runtime plugin load, hook/model/harness/auth/profile resolution, session setup, context-engine bootstrap/assembly, tool registry materialization, and prompt start; then cache or bypass safe per-run invariants for lightweight blocking runs. Preserve Active Memory's hard timeout/fail-open behavior and update the public Active Memory docs with measured timeout and latency guidance once the root cause is fixed.
