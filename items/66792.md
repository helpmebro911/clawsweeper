## Summary

Keep #66792 open. Current main has partial mitigations for active-memory timeout behavior, but the central report is still a plausible core/runtime performance bug: active-memory still runs a blocking pre-reply embedded agent through the full `runEmbeddedPiAgent`/`runEmbeddedAttempt` bootstrap path, and there is no source, test, doc, release, or related open canonical item proving the reported lightweight-bootstrap latency is fixed.

## Best Possible Solution

Keep the issue open as a core/runtime performance investigation. The best maintainer path is to add structured timing around the active-memory `runEmbeddedPiAgent` cold path, especially queue wait, model/auth resolution, context-engine bootstrap, tool/runtime materialization, resource loader reload, `createAgentSession`, prompt assembly, and first model-call start. Then either optimize/cache the expensive pre-model phases for lightweight blocking plugin runs or document the realistic blocking-plugin latency budget and timeout guidance. If the hard-timeout mitigation is considered enough for a narrower bug, open or link a separate follow-up for the remaining bootstrap latency before closing this umbrella item.
