## Summary

Keep open. Current main still has Active Memory as a blocking pre-reply embedded Pi run, and the shipped mitigation is a plugin-owned hard timeout/fail-open path. I did not find source, test, docs, or history evidence that `bootstrapContextMode: "lightweight"` now gives the requested fast bootstrap path for blocking plugins.

## Best Possible Solution

Keep this open and narrow it to embedded Pi runner cold-path profiling for blocking plugin runs. Preserve the Active Memory hard timeout/fail-open behavior, then add stage timings around lane wait, runtime plugin load, harness/model/auth/profile resolution, context engine/bootstrap resolution, session locking/session manager setup, tool materialization, bundle runtime setup, resource loader reload, and prompt start. After measuring, cache or bypass safe per-run invariants for lightweight blocking runs and update the public Active Memory docs with realistic latency and timeout guidance.
