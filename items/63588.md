## Summary

Keep open. Current main still strips Slack bearer auth on cross-origin Slack CDN redirects, and the current tests explicitly assert that behavior. This PR is stale and failing CI, but it remains a focused candidate for the unresolved Slack `download-file` redirect-auth bug rather than something already implemented on main.

## Best Possible Solution

Keep this PR open as the implementation candidate, but rebase or replace it with a current-main Slack plugin patch. The best fix belongs in `extensions/slack/src/monitor/media.ts`: retain Authorization only for HTTPS Slack-owned media redirect hosts such as `*.slack.com`, `*.slack-edge.com`, and `*.slack-files.com`; strip or block every external/non-HTTPS redirect; preserve `fetchRemoteMedia`/`fetchWithSsrFGuard` pinned-DNS SSRF protections, redirect limits, dispatcher runtime behavior, and HTML-login-page detection; and update `extensions/slack/src/monitor/media.test.ts` to cover Slack CDN auth retention plus external/HTTP/SSRF-block cases.
