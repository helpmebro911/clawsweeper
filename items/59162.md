## Summary

Keep open. The issue is recent, not protected/maintainer-authored, and the discussion now includes a fresh non-restart repro on 2026.4.14 showing duplicate Telegram/mobile approval resolves within seconds after a successful resolve. I did not find high-confidence evidence that current main or v2026.4.23 specifically fixes the duplicate resolve/idempotency behavior described in #59162.

## Evidence

- **fresh-repro-comment:** SavvyAdvice supplied a non-restart repro on 2026.4.14: first exec.approval.resolve succeeded, exec-approval-followup ran, then three subsequent resolves 4.8s-10.1s later failed with INVALID_REQUEST "unknown or expired approval id". The comment explicitly distinguishes this from restart churn and from related #58479 hand-off behavior.
- **related-surface:** The issue discussion includes another user seeing the same problem on qqbot and another report that Telegram approvals in 4.5 time out/fail, which makes this a plausible cross-channel/native approval bug rather than a one-off stale deployment report.
- **current-doc-contract:** Current public exec approval docs still describe /approve as resolving a pending exec/plugin approval id and state pending exec approvals expire after 30 minutes by default; the reported duplicate failures happen within seconds after a successful resolve, so the documented behavior does not make the report obsolete.
