## Summary

Keep open. Current `main` still routes the system-prompt workspace path through fallback logic that ignores `OPENCLAW_WORKSPACE_DIR`, and the latest released tag shows the same behavior. The linked PR in the issue timeline is not sufficient evidence to close because the shipped/current source still does not implement the requested env-var fallback.

## Evidence

- **System prompt uses the resolved workspace directly:** `buildAgentSystemPrompt()` computes `displayWorkspaceDir` from `params.workspaceDir` (or sandbox container workdir) and emits `Your working directory is: ...`, so any wrong fallback value propagates straight into the prompt text.
  - file: [src/agents/system-prompt.ts:639](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/system-prompt.ts#L639)

  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Gateway/runtime fallback still comes from agent workspace resolution:** Gateway sessions still derive `runtimeWorkspaceDir` from `resolveAgentWorkspaceDir(cfg, sessionAgentId)` when there is no spawned-workspace override, and embedded agent runs still call `resolveRunWorkspaceDir()` before building the prompt/runtime context.
  - file: [src/gateway/server-methods/agent.ts:136](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/gateway/server-methods/agent.ts#L136)
