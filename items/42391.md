## Summary

Keep open. Current main still has typed plugin hooks for before/after tool calls and agent end, but those hook payloads do not expose the requested structured numeric progress metrics. Adjacent plan/item progress events exist for internal/channel callbacks, but they are not the plugin hook contract requested by the Feishu progress-card use case.

## Best Possible Solution

Keep this issue open and design a small backward-compatible progress observation contract. A good maintainer path is to add a shared optional progress metrics type either to the existing tool/agent hooks or to a dedicated progress hook, emit it only when OpenClaw has a defensible step model, document absent/unknown semantics, and add tests for monotonicity, successful completion reaching 100 only when applicable, and no false 100 percent on failure or cancellation.

## Risks / Open Questions

- A numeric percent can be misleading for dynamic plans, retries, subagents, cancellations, and failures unless the API defines when totals are known versus absent.
- This is a public plugin contract change; even optional fields need shared types, docs, and regression tests so existing plugins stay compatible.
- Current structured item/plan events may be useful implementation input, but treating them as equivalent to hook metrics would not satisfy this plugin hook request.
