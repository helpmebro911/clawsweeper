## Summary

Keep open. Current main has the relevant plugin hook surfaces from prior tool/lifecycle hook work, and the issue’s Feishu/live-card validation shows a concrete plugin consumer, but the typed hook contracts and runtime emitters still do not expose structured run progress metrics such as `stepIndex`, `stepTotal`, `percent`, or `etaSec`. This is not a ClawHub-only close because an external progress-card plugin cannot derive stable monotonic run progress from the current hook API alone.

## Best Possible Solution

Keep this issue open and design a small backward-compatible progress contract. A good path is to add a shared optional `PluginHookProgressMetrics` type in `src/plugins/hook-types.ts`, attach it to the relevant existing hooks or introduce a dedicated progress observation hook if step semantics need isolation, emit it consistently from PI and CLI/native harness paths, document it at `docs/plugins/hooks.md`, and add focused tests for monotonicity, success-to-100 only when a stable step model exists, and no false 100% on failure or cancellation.

## Risks / Open Questions

- Progress totals may be hard to define consistently across PI, CLI/native Codex harnesses, dynamic plans, subagents, retries, cancellations, and tool-only turns.
- A naive percent implementation could mislead users if plans change mid-run or failed/cancelled runs are shown as complete.
