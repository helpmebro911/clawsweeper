## Summary

Keep this PR open. Current main still lacks the inbound Feishu reply-card fallback, and this PR is relevant to #43690, but it is not merge-ready because the fallback is too broad and can create duplicate or unintended plain-text deliveries.

## Best Possible Solution

Keep this PR open and request a focused revision: share or align with the table-limit predicate from #70651, fall back only for final user-visible replies when no prior card chunk was delivered, rethrow non-table-limit card errors, preserve block payload suppression, and add regression tests for table-limit fallback, non-table-limit failures, later-chunk failure, and block payload handling.

## Risks / Open Questions

- The PR fallback catches every structured-card failure rather than only Feishu table-limit errors such as 230099 / ErrCode 11310, which can hide authentication, permission, rate-limit, or malformed-card failures.
- If a later chunk fails after an earlier card chunk was delivered, retrying the entire answer as plain text can duplicate content and first-chunk mentions.
- If a block payload reaches the fallback after streaming-card startup fails, the PR can expose preview or reasoning-like content as a normal Feishu message before the final answer.
- The added tests cover first-card failure and success only; they do not cover non-table-limit errors, later-chunk failures, or block payload suppression.
