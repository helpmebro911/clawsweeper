## Summary

Keep this PR open. Current main still lacks the Feishu non-streaming reply-card fallback for the linked table-limit failure, and the related outbound PR does not cover this inbound reply-dispatcher surface. The patch is focused and relevant, but it is not merge-ready without narrowing the fallback and covering duplicate/block cases.

## Best Possible Solution

Keep this PR open and ask for a focused revision: share or align with the table-limit predicate from #70651 if that helper lands, rethrow non-table-limit card errors, limit fallback to final user-visible replies, avoid resending already delivered chunks, and add regression tests for later-chunk failure, block payloads, and non-table-limit failures. Then use the revised inbound reply-dispatcher fix to resolve #43690.

## Risks / Open Questions

- The current PR catches every structured-card failure instead of matching Feishu table-limit errors such as 230099/11310, so auth, permission, or malformed-card failures could be hidden behind text delivery.
- If a multi-chunk card reply fails after one or more chunks are already delivered, the PR can resend the entire answer as plain text, duplicating content and mentions.
- If a block/preview payload reaches this fallback path after streaming/card startup problems, it can be emitted as an ordinary Feishu text message before the final answer.
- The added tests cover first-card failure and card success, but not later-chunk failure, block payloads, or non-table-limit errors.
