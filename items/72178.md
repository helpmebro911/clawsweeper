## Summary

Keep open. Current main still does not implement cursor-based chat.history pagination or the web UI load-older path, so this PR is not obsolete. The latest PR head is also not ready to close or land: it computes a before/start/end cursor window but still projects the full raw history, so older-page requests can return the latest page again. Security review found no obvious supply-chain or secret/permission regression in the changed file set.

## Best Possible Solution

Keep this PR open for maintainer review. Fix the gateway handler so before selects the raw [start, end) page before projection, or carry raw indices through projection/sanitization explicitly. Add handler-level tests for normal pagination, sanitized NO_REPLY/commentary drops, and byte-capped leading trims. Coordinate with open PR #56383 and related large-history work before choosing the implementation to land; consider splitting the partial NO_REPLY prefix fix if pagination remains blocked.

## Risks / Open Questions

- Functional correctness: latest PR head appears to make before affect cursor math but not raw message selection, so loadOlderChatHistory can prepend duplicate latest messages instead of older history.
- Coverage gap: pure helper tests and mocked UI tests do not prove the gateway handler returns the page before a cursor, especially with sanitized NO_REPLY/commentary rows or byte-budget trimming.
- Scope risk: the PR combines cursor pagination with partial NO_REPLY prefix suppression; maintainers may want the smaller NO_REPLY fix split if pagination needs more design work.
- Protocol risk: adding cursor/hasMore to a gateway method is additive, but it should be validated against all clients that consume chat.history before landing.
