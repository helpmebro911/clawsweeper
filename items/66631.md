## Summary

Keep this issue open. Current main has adjacent Feishu topic-session routing and thread-history support, but it still does not resolve a Feishu `thread_id` (`omt_*`) to the original root `om_*` message before selecting the outbound reply target. In the reported `thread_id`-only topic event, normal replies still fall back to the triggering message id, which matches the bug report's failure mode rather than fixing it.

## Best Possible Solution

Implement this in the Feishu plugin. For a group topic/reply-in-thread event with `thread_id` but no usable `root_id`, `parent_id`, or `reply_target_message_id`, resolve the thread container to the earliest/root `om_*` message, cache `thread_id -> rootMessageId`, and use that root for `replyToMessageId`/`rootId` across normal replies, streaming cards, media replies, and ACP/init error notices. Add regression tests for `thread_id`-only topic-group events that assert outbound reply anchoring, not just session routing.
