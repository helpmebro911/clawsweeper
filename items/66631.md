## Summary

Keep this issue open. Current main still parses and routes Feishu `thread_id` for topic sessions, but it does not resolve a `thread_id`-only topic event to the root `om_*` message before choosing outbound reply targets. In the reported shape, normal replies and ACP init error notices still fall back to the triggering `messageId`, which matches the bug report rather than fixing it.

## Best Possible Solution

Implement this in the Feishu plugin. For topic-mode or `replyInThread` group events that have `thread_id` but lack a usable `root_id`, `parent_id`, or `reply_target_message_id`, resolve the thread container to the earliest/root `om_*` message, cache `thread_id -> rootMessageId`, and set the reply target before ACP init error notices and normal dispatch. Cover text, card/streaming, and media reply paths through the existing dispatcher, and add regression tests for the `thread_id`-only topic event while preserving #32980 normal-group reply placement.
