## Summary

Keep this issue open. Current main still treats Slack Socket Mode and inbound reply delivery as bot-token centered, documents/generated setup still subscribes only `bot_events`, and `userTokenReadOnly: false` only permits user-token writes as a fallback when no bot token is available. That does not satisfy the requested dedicated Slack user DM flow.

## Best Possible Solution

Implement this in the bundled Slack plugin: add an explicit user-event subscription/setup surface, carry the configured user identity/token through Slack monitor context, distinguish bot-level versus user-level inbound DM events, route eligible user-level DM replies through the xoxp token when `userTokenReadOnly: false`, and add focused tests for inbound DM routing, token selection, loop prevention, setup manifest output, and docs.

## Evidence
