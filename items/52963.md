## Summary

Keep this issue open. Current main still treats Slack `userToken` as supplemental user-context API access and a write fallback in some action paths, not as a user-level inbound event identity for DM routing or monitor replies. The existing issue discussion already identified this same remaining gap, and current source/docs still match that conclusion.

## Best Possible Solution

Implement this in the bundled Slack plugin: generate and document the required Slack `user_events` subscription for `message.im`, resolve and store the configured user-token identity at monitor startup, distinguish bot-level and user-level DM events, select the xoxp token only for eligible user-level replies when `userTokenReadOnly` is false, prevent self-loops, and add regression coverage for manifest output, event routing, token selection, docs, and reply delivery.

## Risks / Open Questions

- Routing replies through xoxp user tokens broadens credential and identity impact; a fix needs explicit opt-in, clear token selection, auditability, and self-loop prevention.
- Slack user-level Socket Mode envelope metadata and self-event behavior should be validated against a real Slack app configured with `user_events` for `message.im`.
- No current regression coverage was found for user-level `message.im` routing, configured user-token identity resolution, or xoxp monitor reply delivery.
