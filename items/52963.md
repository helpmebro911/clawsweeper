## Summary

Keep this issue open. Current main still configures Slack inbound subscriptions around bot-level events, starts the monitor with bot/app tokens, and sends normal inbound replies with the bot token. The configured `userToken` is present, but current behavior treats it as supplemental user-context API access and a write fallback when the bot token is unavailable, not as a user-level DM event identity.

## Best Possible Solution

Keep this open and implement it in the bundled Slack plugin: document and generate the required Slack `user_events` subscription for `message.im`, resolve the configured user-token identity at monitor startup, distinguish user-level DM events from bot-level DM events, select the xoxp token only for those eligible replies when `userTokenReadOnly` is false, and add regression coverage for inbound routing, setup manifest output, docs, token selection, and self-loop prevention.
