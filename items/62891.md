## Summary

Keep this issue open. Current `main` has the partial #65365-style startup ordering fix that starts scheduled services only after post-attach sidecars complete, and it also has an HTTP readiness checker that can detect disconnected managed channels. But cron activation still does not wait on channel connected readiness, and the hot-reload path can restart cron before restarting configured channels, so the reporter's SIGUSR1/config.patch and Discord WS readiness concern remains actionable.

## Best Possible Solution

Implement a bounded cron startup/reload readiness gate in the gateway/cron boundary. It should delay automatic cron timer execution and startup catch-up until configured delivery channels are connected/healthy, a configurable wait such as `cron.channelReadyWaitMs` expires, or channels are explicitly unhealthy with clear diagnostics. Cover full startup, SIGUSR1/config.patch reloads, Discord-style async readiness, and the existing `/readyz`/channel health semantics with focused gateway and cron tests.
