## Summary

Keep open. Current `main` still does not implement the reported Discord message-tool attachment workflows end-to-end: generic `send` handling ignores `buffer` payloads, Discord action handlers only accept URL/path-style media inputs, and the actual Discord config schema still does not expose `channels.discord.mediaLocalRoots` even though generated metadata advertises it.

## Evidence

- **Generic message `send` does not read `buffer`:** `handleSendAction` only treats `media`, `mediaUrl`, `path`, `filePath`, and `fileUrl` as attachment inputs. It never reads `buffer`, `mimeType`, or `contentType`, so a buffer-only send can fall through as a text-only send.
  - file: [src/infra/outbound/message-action-runner.ts:481](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/infra/outbound/message-action-runner.ts#L481)
  - command: `nl -ba src/infra/outbound/message-action-runner.ts | sed -n '470,610p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Attachment hydration is gated away from `send`:** The shared attachment hydrator that normalizes `buffer` and converts path inputs into base64 only runs for `sendAttachment`, `setGroupIcon`, or `upload-file`. It explicitly returns early for ordinary `send` actions.
  - file: [src/infra/outbound/message-action-params.ts:383](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/infra/outbound/message-action-params.ts#L383)
  - command: `nl -ba src/infra/outbound/message-action-params.ts | sed -n '340,406p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Discord message action only forwards URL/path media:** Discord's `send` action maps attachments from `media`, `path`, or `filePath` into a single `mediaUrl` field before calling the runtime. There is no `buffer` handling in the Discord adapter.
  - file: [extensions/discord/src/actions/handle-action.ts:84](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/extensions/discord/src/actions/handle-action.ts#L84)
  - command: `nl -ba extensions/discord/src/actions/handle-action.ts | sed -n '60,110p'`
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Discord runtime send path is still media-URL based:** `handleDiscordMessagingAction(..., "sendMessage")` reads only `mediaUrl`/`path`/`filePath`, then calls `sendMessageDiscord` with `mediaUrl`. The outbound sender loads the file from that path/URL via `loadWebMedia(...)`; there is no inline buffer parameter on this path.
  - file: [extensions/discord/src/actions/runtime.messaging.ts:347](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/extensions/discord/src/actions/runtime.messaging.ts#L347)

- **`mediaLocalRoots` is advertised but not present in the real Discord schema:** Generated bundled-channel metadata still lists `mediaLocalRoots`, but the actual `DiscordAccountSchema` shows `mediaMaxMb` followed immediately by `retry` with no `mediaLocalRoots` field.
