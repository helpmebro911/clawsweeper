## Summary

Keep #10737 open. Current main still lacks a legacy `channels.imessage` auto-ack or typing-feedback implementation, and the issue discussion already contains a maintainer-side Codex review saying to keep this legacy iMessage request open despite BlueBubbles support.

## Best Possible Solution

Keep #10737 open until maintainers either implement or explicitly decline legacy `channels.imessage` feedback. A concrete fix should add typed config/schema/docs for `autoAck` or a native-typing/text-fallback option, send it after access and mention gating but before `dispatchInboundMessage`, honor `dmOnly`, and add regression coverage proving the acknowledgment is not re-ingested as a final reply. BlueBubbles can remain the recommended iMessage path, but it does not close this legacy imsg-channel request.

## Risks / Open Questions

- A text auto-ack would be a real outbound iMessage and should be integrated with the existing sent-message cache/reflection guards so it is not re-ingested as an inbound user message.
- Native typing support for the legacy imsg path appears unstable upstream, so maintainers should revalidate imsg behavior before choosing native typing, text fallback, or an explicit decline.
