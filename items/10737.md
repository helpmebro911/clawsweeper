## Summary

Keep #10737 open. The current `main` still does not implement legacy `channels.imessage` auto-acknowledgment or native typing feedback, and the provided issue discussion includes a same-day maintainer comment explicitly saying to keep this issue open despite BlueBubbles typing support.

## Best Possible Solution

Keep #10737 open until maintainers either implement or explicitly decline legacy `channels.imessage` feedback. A concrete fix should add typed config/schema/docs for `autoAck` or native typing with text fallback, send it after access and mention gating but before `dispatchInboundMessage`, honor `dmOnly`, and add regression tests proving the acknowledgment is not re-ingested as a final reply. BlueBubbles can remain the recommended iMessage path, but it does not close this legacy iMessage request.

## Risks / Open Questions

- A text auto-ack would be a real outbound iMessage and needs integration with sent-message cache/reflection guards so it is not re-ingested as an inbound user message.
- Native typing support for the legacy imsg path should be revalidated before implementation; a text fallback may still be needed if upstream typing behavior is unavailable or unreliable.
