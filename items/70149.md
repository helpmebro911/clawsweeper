## Summary

Keep open. Current main still routes the WhatsApp WebSocket `agent` through `resolveAmbientNodeProxyAgent()`, which constructs generic `proxy-agent` rather than the explicit `HttpsProxyAgent` path the reporter found working. Dependency source shows `proxy-agent@8.0.1` treats WebSocket upgrades as `wss:` URLs, and `proxy-from-env@2.1.0` then looks for `wss_proxy`/`all_proxy`, not `HTTPS_PROXY`/`HTTP_PROXY`. That matches the report's failure mode and means the existing docs/tests only prove an agent object is passed, not that a WhatsApp WSS handshake actually uses the configured HTTP(S) proxy.

## Best Possible Solution

Keep the issue open and fix the WhatsApp WebSocket proxy path directly. The likely product/code direction is to make `extensions/whatsapp/src/session.ts` resolve the `wss://web.whatsapp.com` socket agent from `HTTPS_PROXY`/`HTTP_PROXY` with `NO_PROXY` respected, probably using `HttpsProxyAgent` like Slack/Discord do for WebSocket-capable clients, then add a focused test that verifies a WSS WhatsApp target with only `HTTPS_PROXY` produces the expected proxy tunnel agent rather than merely checking that some `agent` object exists.
