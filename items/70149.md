## Summary

Keep open. The current WhatsApp login path still uses `resolveAmbientNodeProxyAgent()`, which returns `proxy-agent`'s ambient `ProxyAgent`, and passes that agent into Baileys for the WebSocket handshake. #70149 specifically reports that this exact ambient `proxy-agent` path times out while an explicit `HttpsProxyAgent` succeeds, so the earlier broad env-proxy implementation is not enough evidence to close this newer, narrower bug.

## Best Possible Solution

Keep #70149 open. Reproduce the reported WSS login path against current main, then add a focused WhatsApp WebSocket proxy resolver if the report holds: resolve the concrete env proxy URL with `NO_PROXY` semantics, use an explicit `HttpsProxyAgent` for HTTP(S) proxies on the Baileys `agent` path, preserve fallback for unsupported proxy schemes, and add a regression test that distinguishes `ProxyAgent` from `HttpsProxyAgent` or exercises a mocked CONNECT/ws handshake.
