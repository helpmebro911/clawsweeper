## Summary

Keep open. Current `main` still sends WhatsApp WebSocket traffic through the generic ambient `proxy-agent` path the report calls out, there is no shipped WhatsApp-specific `HttpsProxyAgent` fix on `main`, and the only post-release changes in the touched area are test-only.

## Evidence

- **WhatsApp still uses ambient proxy-agent for its WebSocket agent:** `createWaSocket()` resolves `agent` via `resolveEnvProxyAgent()`, and that helper logs `Using ambient env proxy for WhatsApp WebSocket connection` before returning `resolveAmbientNodeProxyAgent(...)`. This matches the exact path described in the issue rather than an explicit `HttpsProxyAgent` workaround.
  - file: [extensions/whatsapp/src/session.ts:147](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/whatsapp/src/session.ts#L147)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Shared proxy helper still constructs a generic ProxyAgent:** `resolveAmbientNodeProxyAgent()` currently imports `proxy-agent` and returns `new ProxyAgent()` with no WhatsApp-specific override. I did not find a `HttpsProxyAgent` path for WhatsApp on `main`.
  - file: [src/plugin-sdk/extension-shared.ts:237](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/plugin-sdk/extension-shared.ts#L237)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **WhatsApp proxy coverage does not exercise a successful WebSocket handshake:** The WhatsApp session test only asserts that an ambient proxy agent and fetch dispatcher are created when `HTTPS_PROXY` is set. It does not cover opening the WhatsApp WebSocket through the proxy or distinguish `ProxyAgent` from `HttpsProxyAgent` behavior.
  - file: [extensions/whatsapp/src/session.test.ts:190](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/whatsapp/src/session.test.ts#L190)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Other WebSocket-heavy channels use explicit HttpsProxyAgent:** Slack builds an explicit `HttpsProxyAgent(proxyUrl)` for Socket Mode, and Discord builds an explicit `HttpsProxyAgent(proxy)` for its gateway WebSocket. That makes the WhatsApp path stand out as materially different from other known proxy-tunneled WS implementations in this repo.
  - file: [extensions/slack/src/client-options.ts:49](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/slack/src/client-options.ts#L49)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Discord gateway also uses explicit HttpsProxyAgent:** Discord's gateway plugin validates the proxy URL and constructs `new HttpsProxyAgentCtor<string>(proxy)` before wiring the WebSocket path.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:516](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/discord/src/monitor/gateway-plugin.ts#L516)
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **User-facing docs still promise env-proxy support for WhatsApp:** The WhatsApp docs say the transport honors `HTTPS_PROXY`, `HTTP_PROXY`, and `NO_PROXY`. Given the current implementation above and the issue's concrete ambient-vs-explicit agent comparison, this remains a plausible current bug rather than something safe to auto-close.
  - file: [docs/channels/whatsapp.md:153](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/docs/channels/whatsapp.md#L153)
