## Summary

Keep open. Current main still does not start Telegram native typing at the accepted/normalized inbound point. Telegram defines `sendTyping` and wires it into the downstream reply typing pipeline, while generic typing starts from model-loop/text/reasoning/tool signals. Existing tests only cover callback-driven typing, and related #6946 is a broader visible processing-message feature rather than this narrower timing bug.

## Best Possible Solution

Implement a Telegram plugin-owned early native typing handshake after `buildTelegramMessageContext` has accepted and normalized the message, but before slow dispatch, queue, compaction, model, or tool work. Add focused regression coverage proving accepted Telegram messages call `sendChatAction("typing")` before a delayed dispatch path, while ignored, unauthorized, disabled, or suppressed messages do not. Keep #6946 separate as the visible processing-placeholder feature.

## Risks / Open Questions

- Moving typing earlier must remain gated behind Telegram access checks, mention/body normalization, route/thread resolution, binding readiness, and typing suppression settings so ignored or unauthorized messages do not emit visible feedback.
- The fix should integrate with the existing typing controller lifecycle so an early native typing action does not create duplicate keepalive loops or leave typing running after silent, dropped, or failed turns.
