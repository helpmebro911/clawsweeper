## Summary

Current `main` does not contain this PR's hardening for Telegram direct-thread delivery inference. The cron tool still strips `:thread:` suffixes before parsing, so Telegram direct-thread session keys fall back to the parent direct peer instead of `<chatId>:topic:<threadId>`, and the current tests only cover generic threaded keys plus Telegram forum-topic keys, not Telegram direct-thread or multi-account variants.

## Evidence

- **Current cron inference still strips thread suffixes before parsing:** `stripThreadSuffixFromSessionKey()` removes the trailing `:thread:` segment, and `inferDeliveryFromSessionKey()` immediately parses that stripped key. There is no `extractTelegramDirectThreadTarget()`-style recovery on `main`, so Telegram direct-thread session keys still lose thread context during inference.
