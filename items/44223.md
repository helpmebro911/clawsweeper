## Summary

Keep open. Current `main` still lacks the PR’s Ctrl+C active-run abort path and stale-abort recovery, so this is neither implemented on `main` nor safely closeable as obsolete.

## Evidence

- **Ctrl+C logic on main still ignores active runs:** `resolveCtrlCAction` only supports `clear | warn | exit`, and `handleCtrlC` passes only editor-input/exit-window state. There is no active-run abort branch, so pressing Ctrl+C with drafted input during a running TUI session still takes the clear-input path instead of aborting the run.
  - file: [src/tui/tui.ts:260](https://github.com/openclaw/openclaw/blob/24e9924d6abd93560fb7bf721909257de533e57a/src/tui/tui.ts#L260)

  - sha: [24e9924d6abd](https://github.com/openclaw/openclaw/commit/24e9924d6abd93560fb7bf721909257de533e57a)
- **Abort RPC handling on main still leaves stale-run edge cases:** `abortActive()` awaits `client.abortChat(...)` and only sets activity text; it does not capture the run id, does not clear a stale `activeChatRunId`, and does not handle the gateway’s documented `{ ok: true, aborted: false }` response for missing runs. That matches the unresolved review concerns about Ctrl+C getting stuck on repeated abort attempts.
