## Summary

Keep open. Current main still has Ctrl+C wired only to clear/warn/exit behavior, while Esc and `/abort` remain the active-run abort paths. The PR is a plausible focused TUI bug fix rather than obsolete or implemented on main, and the provided review context shows the abort-fallback concerns were actively addressed in the PR branch.

## Best Possible Solution

Keep #44223 open for maintainer review or land an equivalent focused TUI fix. The best path is to make Ctrl+C behavior intentional and tested: if maintainers want Ctrl+C to abort active runs, add an active-run-first resolver branch, guard pending/stale abort states using the `chat.abort` result, reset stale exit confirmation on abort, preserve/finalize partial assistant text on aborted events, and update `docs/web/tui.md` if the user-facing shortcut contract changes. If maintainers prefer the existing Esc/`/abort` model, close this PR only after replacing it with a narrower status-hint fix such as #38502.
