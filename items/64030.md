## Summary

Keep this issue open. Current main has moved new cron job ids to UUIDs, so the original filename coupling is partly reduced, but the user-visible bug remains: explicit `session:<sessionKey>` targets and `current` session binding still pass channel-native session keys through `assertSafeCronSessionTargetId`, which rejects `/` and `\` before the job can be created or run.

## Best Possible Solution

Keep this issue open. Implement a cron-boundary fix that accepts opaque channel-native session keys for explicit `session:<id>` and resolved `current` targets, while keeping run-log filenames UUID-backed or otherwise encoded and preserving NUL/path traversal protections where filesystem paths are built. Add regression coverage for slash-bearing DingTalk-style session keys through normalize, create/update, current-session binding, persisted validation/manual-run, and run-log safety.
