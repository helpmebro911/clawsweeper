## Summary

Keep this issue open. Current main still consumes Discord component registry entries by the clicked component id only: build/registration stores sibling buttons as independent entries, the registry deletes only `params.id`, and the interaction handler consumes only `parsed.componentId` before dispatch. The closed broader #54233 mentions the same remaining bug but is closed and does not supersede this focused issue.

## Best Possible Solution

Keep this issue open as the focused tracker. Fix it in the Discord plugin by assigning a shared non-reusable consumption group for entries created from the same component message/panel, then invalidating sibling non-reusable entries before callback dispatch. Add a regression test with two buttons in one non-reusable component message proving the second sibling click returns the expired response and does not dispatch, while `reusable:true` still permits repeated use.
