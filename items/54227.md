## Summary

Current main still implements Discord component expiry per component id, not per component message/panel. The docs promise single-use components by default, but the builder registers sibling buttons as independent entries, the registry consumes only the clicked id, and the monitor dispatch path consumes only parsed.componentId before delivering the callback. The closed broader #54233 repeats the same bug but does not supersede this focused tracker.

## Best Possible Solution

Keep this issue open as the focused Discord plugin bug. Implement a shared non-reusable consumption group for entries created from the same component message/panel, invalidate all sibling non-reusable entries before callback dispatch, and add a regression test with two buttons proving the second sibling click returns the expired response and does not dispatch while reusable:true still permits repeated use.
