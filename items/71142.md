## Summary

Keep #71142 open. Current main still has a hardcoded 5,000,000-byte attachment cap in the Gateway chat attachment path, and the strict config schema/types expose no Control UI/WebChat upload-size setting. The issue is recent, unprotected, author is not a maintainer, and the related closed image-upload issue only proves upload support exists, not configurable limits.

## Best Possible Solution

Keep the issue open and implement a first-class upload limit configuration, likely under `gateway.webchat` or `gateway.controlUi`, with either a single max attachment byte value or typed media limits. Thread the resolved value through `chat.send`, `agent.request`, node-event attachment parsing, and `parseMessageWithAttachments`; update strict config schema/types/help/docs and add tests proving the default behavior plus a configured larger limit.

## Risks / Open Questions

- Raising only the browser-side picker limit would not fix the server rejection; all relevant Gateway attachment parsing paths need the same resolved limit.
- Decoded upload limits above roughly 18MB can exceed the current 25MB WebSocket payload cap after base64 and JSON overhead unless the design accounts for transport limits or uses a managed-media upload path.
- Adding a public config key requires schema, types, help/labels, docs, and generated config baseline updates, plus tests for default and configured behavior.
