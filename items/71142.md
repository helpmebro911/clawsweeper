## Summary

Keep this issue open. Current main still has no configurable Control UI upload size limit: Control UI sends selected files as base64 `chat.send` attachments, and the server-side `chat.send` path still passes a hardcoded 5,000,000 byte cap to the shared attachment parser. The `gateway.controlUi` config contract remains strict and only exposes hosting/security fields, so proposed keys like `maxUploadSizeMb` or `mediaLimits` would not be accepted. The existing `tools.media.*.maxBytes` documentation is a separate media-understanding policy after media is ingested, not an upload-ingestion limit for Control UI.

## Best Possible Solution

Implement a documented upload policy instead of closing this. A conservative path is to add either `gateway.controlUi.maxUploadSizeMb`/`mediaLimits` for WebChat only, or a shared Gateway chat attachment limit if all `chat.send` callers should share it. Wire it through config types, zod schema, generated schema/help metadata, Control UI runtime config and validation/error display, and the `chat.send` attachment parser. Add tests for the default 5 MB behavior, raised limits, oversized rejection, and the WebSocket/base64 or media-store behavior required for larger uploads.

## Risks / Open Questions

- Raising the parser cap alone may fail for larger raw files because Control UI currently sends base64 attachments over a 25 MiB WebSocket payload limit.
- The product decision still needs to choose whether this should be scoped to `gateway.controlUi` or exposed as a shared Gateway chat attachment policy for all `chat.send` callers.
