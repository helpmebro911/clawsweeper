## Summary

Keep open. Current main still hardcodes the Control UI/WebChat attachment limit in Gateway chat paths and does not expose a `gateway.controlUi` or `gateway.webchat` upload-size/media-limit config. Related prior items cover image upload support and other media/context limits, but not this configurable upload-limit request.

## Best Possible Solution

Keep the issue open and implement a first-class, documented Gateway config for Control UI/WebChat attachment limits. The likely path is to add typed config/schema/help/docs fields, feed the resolved limit into `chat.send`, `agent` send, and node-event attachment parsing instead of hardcoded `5_000_000`, and expose the resolved limit to the Control UI bootstrap if client-side preflight/error copy is desired.

## Risks / Open Questions

- Raising this limit should be enforced server-side, not only in the browser, because Control UI sends base64 over the Gateway RPC and larger values affect memory, WebSocket payload size, media-store offload, and provider caps.
- The final config shape should avoid conflicting with existing `tools.media.*` and OpenAI-compatible HTTP endpoint limits; docs and schema/help/generated config metadata would need to stay aligned.
