## Summary

Current main has a SQLite-backed memory search index plus adjacent promotion, wiki, and LanceDB plugin layers, but it still does not define the requested durable typed SQLite memory record schema, canonical current record model, revision trail, or explicit active/stale/superseded/forgotten/conflicted lifecycle.

## Best Possible Solution

Keep this issue open. The best implementation path is a versioned SQLite durable-memory model in the memory owner surface, likely `extensions/memory-core` plus the memory-host SDK storage contract: canonical `memories` rows, `memory_revisions`, typed record/status/confidence/timestamp/validity/entity fields, migration and contract tests, docs, and integration with write, provenance, review, forget, and conflict flows. Coordinate with #42647, #42648, and #42650; #42649 only covers retrieval.
