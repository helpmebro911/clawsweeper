## Summary

Keep open. Current main still has the ingredients for the reported WhatsApp reconnect/health-monitor collision: the WhatsApp loop marks the account disconnected while retrying, the generic health monitor treats running+disconnected accounts as restartable, and the shared stop path can mark the account manually stopped when the 5s abort wait times out. The existing tests cover pieces of that behavior, but I did not find a coordination fix or a regression test for the WhatsApp reconnect-versus-health-monitor race.

## Best Possible Solution

Keep #70463 open. The likely fix belongs in the WhatsApp/channel lifecycle boundary: make the generic health monitor treat provider-owned `reconnecting` states as in-progress instead of forcibly stopping them, or add an explicit provider/channel coordination seam so WhatsApp can own Baileys reconnects without the health monitor marking the account manually stopped. Add a regression test that simulates WhatsApp status 499/reconnect delay plus a health-monitor tick, and consider exposing documented WhatsApp watchdog timeout/check tuning only after the lifecycle race is closed.
