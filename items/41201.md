## Summary

Current main partially fixes the local-workspace avatar path, but the issue's remote `http(s)` avatar case remains intentionally unsupported in Control UI while the CLI/config contract still accepts remote avatar values. This needs maintainer product/security judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep this open for maintainer product/security judgment. Either implement remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, size, and MIME controls, or make remote avatars explicitly unsupported for Control UI by tightening CLI help, config/docs, validation, and UI feedback while preserving the current local workspace avatar route and diagnostics.

## Risks / Open Questions

- Closing as implemented would hide the remaining mismatch between accepted remote avatar configuration and Control UI rendering behavior.
- Directly rendering arbitrary remote avatar URLs would let operator browsers fetch third-party image URLs, conflicting with the current CSP/privacy hardening.
- A simple remote 302 redirect approach like the closed related PR would broaden the image policy without the proxy/cache, auth, size, MIME, and privacy controls that a core fix likely needs.
