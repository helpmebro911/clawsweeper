## Summary

Keep open. Current main handles local workspace/data/avatar metadata and authenticated local avatar serving, but it still intentionally does not render the issue's explicit remote http(s) avatar example in Control UI. The remaining gap is a product/security decision: either safely proxy/cache remote avatars through same-origin authenticated routes, or make the CLI/docs validation clearer that Control UI only renders local/data/blob/same-origin avatars.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is to either implement safe remote avatar rendering through an authenticated same-origin `/avatar` proxy/cache with CSP and privacy controls, or tighten CLI validation/help/docs so users know Control UI renders workspace-local, data:image, browser-local blob, and same-origin avatar routes only. Preserve the existing local avatar route, auth, workspace-boundary, blob rendering, and oversize regression coverage.

- Rendering arbitrary remote avatars directly from operator browsers has privacy and CSP implications; a safe implementation likely needs an authenticated same-origin proxy/cache rather than direct cross-origin image fetches.
