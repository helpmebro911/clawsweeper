## Summary

Current main still has the Discord configured-ACP parent-thread session collision that this PR targets. The PR remains a focused implementation candidate, with review discussion already narrowing the fix to preserve Feishu canonical conversation remapping.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for #64199. The best path is to rebase and review a surgical Discord change where parent-channel matches materialize the runtime thread ID while preserving `parentConversationId`, keep registry support for channel-canonical matches, and land regression coverage proving sibling Discord threads under one configured parent receive distinct ACP session keys plus the intended reset/reverse-lookup behavior.

## Risks / Open Questions

- The PR should be rebased and reviewed against current main before landing because its head is not present in this checkout and current main has moved since the PR base.
- Changing configured Discord thread sessions from parent-keyed to thread-keyed will start fresh ACP state for threads that previously shared the parent session.
