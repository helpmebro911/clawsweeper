## Summary

Keep open. Current `main` still advertises `.ogg`/`.opus` as supported multimodal memory audio inputs for Gemini, and the indexing/request path still forwards the detected MIME type directly to Gemini without any provider-specific filtering or transcoding. I found no implementation on `main` that would prevent the reported `audio/ogg` rejection.

## Evidence

- **Multimodal memory still classifies OGG/Opus as supported audio:** `MEMORY_MULTIMODAL_SPECS.audio.extensions` still includes `.ogg` and `.opus`, so those files are accepted for indexing on current `main`.
  - file: [packages/memory-host-sdk/src/host/multimodal.ts:10](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/packages/memory-host-sdk/src/host/multimodal.ts#L10)

  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Indexing path preserves detected MIME type for multimodal files:** `buildFileEntry` stores the detected MIME type for multimodal files, and `loadMultimodalEmbeddingInput` later emits that exact MIME type in the `inline-data` part. There is no normalization from `audio/ogg`/`audio/opus` to a Gemini-safe format.
  - file: [packages/memory-host-sdk/src/host/internal.ts:241](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/packages/memory-host-sdk/src/host/internal.ts#L241)

  - file: [extensions/google/embedding-provider.ts:87](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/extensions/google/embedding-provider.ts#L87)
