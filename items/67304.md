## Summary

Keep open. Current main still does not implement `OPENCLAW_JITI_CACHE_DIR`, and this PR targets a real core/plugin-loader operational gap rather than a ClawHub-only feature. The patch is not merge-ready as written because its unset fallback passes explicit `fsCache: true`, which can override Jiti's own `JITI_FS_CACHE` / `JITI_CACHE` handling. ([github.com](https://github.com/unjs/jiti))

## Best Possible Solution

Keep this PR open for contributor revision and maintainer review. The best implementation is a narrow helper that returns a trimmed `OPENCLAW_JITI_CACHE_DIR` only when non-empty and otherwise omits `fsCache` or returns `undefined` so Jiti keeps honoring `JITI_FS_CACHE` / `JITI_CACHE`; wire that through the shared plugin loader, root alias, and Matrix wrapper; add tests for unset, blank, custom path, and Jiti env-disable behavior; include cache-key isolation if the effective option can vary; and add the required changelog/docs entry if maintainers accept the env var.

## Risks / Open Questions

- Merging unchanged would force `fsCache: true` when `OPENCLAW_JITI_CACHE_DIR` is unset, potentially overriding deployments that already use `JITI_FS_CACHE=false` or `JITI_CACHE=false` to avoid filesystem cache writes.
- If the effective cache directory can vary in-process during tests, the cached Jiti loader key may need to include the effective `fsCache` value or use another isolation mechanism.
- A new user-facing env var needs focused tests and a changelog/docs decision before merge so operators can discover it and maintainers can preserve compatibility.
- Security/supply-chain review found no unrelated dependency, workflow, package-publishing, network, or secret-handling changes; remaining risk is limited to env-controlled filesystem cache writes and compatibility with Jiti defaults.
