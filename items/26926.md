## Summary

Keep #26926 open. Current main still handles Synology Chat inbound webhooks as text-only: the payload contract requires `text`, parsing rejects requests missing text, and delivery/inbound context does not carry attachment paths, URLs, MIME types, filenames, or sizes. Existing media support is outbound URL-based file sending only, and the prior maintainer-side Codex comment already identified the same inbound attachment gap.

## Best Possible Solution

Keep #26926 open and implement inbound attachment handling inside the bundled Synology Chat plugin. Confirm Synology Chat's attachment webhook/API shape, extend the typed payload/parser, download or stage accepted images/CSVs/PDFs through existing media-safe helpers, expose `MediaPath(s)`, `MediaUrl(s)`, `MediaType(s)`, and useful filename metadata in the inbound context, update https://docs.openclaw.ai/channels/synology-chat, and add focused webhook plus inbound-context regression tests.
