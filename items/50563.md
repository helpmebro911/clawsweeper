## Summary

Current main still has the reported WhatsApp inbound crash-loss window: the plugin sends a read receipt before enrichment, dedupe claim, debouncer enqueue, and `onMessage` completion, and the WhatsApp inbound dedupe path is memory-only rather than a durable WAL/persist-before-ack queue. The KeepMyClaw comment is useful as complementary state-snapshot discussion, but it does not replace the missing inbound message durability in OpenClaw core/plugin code.

## Evidence

- **Read receipt precedes processing:** `handleMessagesUpsert` awaits `maybeMarkInboundAsRead(inbound)` before append filtering, message enrichment, dedupe claim, and `enqueueInboundMessage`, so a crash after the read receipt can occur before OpenClaw has dispatched or durably recorded the message.

  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '420,710p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Read receipt implementation is explicit:** `maybeMarkInboundAsRead` calls `sock.readMessages([{ remoteJid, id, participant, fromMe: false }])` when read receipts are enabled, confirming this is an acknowledgement/read side effect separate from later handler completion.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:440](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/whatsapp/src/inbound/monitor.ts#L440)
