## Summary

Current main still does not implement automatic forwarding of original Node `agent.request` image attachments when `deliver: true`. Attachments are parsed into agent input, but the original delivery-ready media is not carried into `AgentCommandOpts`, and final outbound delivery only sends agent-produced reply payloads.

## Best Possible Solution

Keep this issue open. The best implementation is an additive Gateway protocol/node-event change that preserves sanitized delivery-ready attachment references only when explicitly requested, carries that intent through `AgentCommandOpts`, and injects media into outbound payloads under narrow conditions such as `deliver: true`, resolved external target, channel media support, and no agent-supplied media. It should include docs and regression tests for text-only replies, no-reply runs, agent-media override, and unsupported channels.

## Risks / Open Questions

- Automatic sender-controlled media fanout needs explicit product and security design: opt-in semantics, target validation, type/size limits, outbound media access scoping, unsupported-channel behavior, and precedence when the agent also returns media.
- A plugin-only implementation is not cleanly sufficient today because the Node/Gateway path consumes attachments into agent input before outbound delivery sees a delivery-ready reference.
