## Summary

Current main still does not satisfy the request. The shared CLI message formatter continues to hard-cap rendered message rows at 25, and the affected `read`, `list-pins`, and `search` display paths still route through that capped formatter. Some Microsoft Teams provider-side pagination exists, but it is partial and does not solve the observable CLI output cap.

## Best Possible Solution

Keep this issue open and implement a narrow message action contract fix: pass the requested display limit and pagination metadata through to `formatMessageCliText`/`renderMessageList`, remove or configure the hardcoded 25-row cap, propagate limits through `list-pins` where providers support them, expose provider cursors or next-page metadata consistently, and add regression tests showing more than 25 rendered rows for `read`, `list-pins`, and `search` while preserving existing Teams `@odata.nextLink` behavior.
