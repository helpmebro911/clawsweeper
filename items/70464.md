## Summary

Keep open. Current main still has the failure surface this PR is trying to address: timestamped dreaming narrative session keys, `idempotencyKey` equal to the session key, cleanup only through `sessions.delete`, and no memory-core-specific delete authorization. The PR is also not safe to close as already implemented or duplicate, because related PR #68364 covers only part of the stable-key work and the scope/lifecycle cleanup bug remains tracked by open reports.

## Best Possible Solution

Keep the PR open and request changes, or pick one canonical PR for the memory-core dreaming cleanup fix. The best fix should make the narrative session key stable per workspace and phase, keep `nowMs` only in the idempotency key, clear stale stable sessions before each run, and provide a narrow memory-core-only cleanup path that works both with fallback contexts and request-scoped clients while preserving non-memory-core `sessions.delete` admin protection. It should also remove unrelated runtime dependency edits and add regression tests for the request-scoped client path and denial cases.
