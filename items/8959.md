## Summary

Keep #8959 open. Current main still has only a shared single-select message presentation contract plus adjacent provider-native or command/wizard multiselect paths; it does not implement the requested portable message-tool multi-select with min/max values and one structured selected-values callback across Slack, Discord, Telegram, and fallback rendering.

## Best Possible Solution

Keep the issue open until OpenClaw adds an additive portable batch-selection contract in the shared message presentation/message tool surface, exports it through openclaw/plugin-sdk/interactive-runtime, documents it on docs.openclaw.ai/plugins/message-presentation, and implements focused render/callback handling for Slack, Discord, Telegram, and fallback text paths. Existing provider-native Discord components and Slack parsing should be treated as implementation references, not as completion of the cross-platform API.
