## Summary

Keep #8959 open. Current main has shared message presentation and provider-native select plumbing, but the shared message tool still lacks a portable multi-select/batch-submit contract. The current `presentation` schema only exposes single `select` blocks, Slack renders them as `static_select`, Telegram turns selects into individual inline buttons, and the related dialog PR #18286 was closed unmerged and is not present on main.

## Best Possible Solution

Keep the issue open until OpenClaw adds an additive cross-platform message presentation contract for multi-select/batch-submit, likely either a new `multiselect` block or a backward-compatible `select` extension with `minValues`, `maxValues`, and structured selected-value callback semantics. The implementation should update the shared message tool schema, `MessagePresentation` types/normalization/fallbacks, Discord/Slack/Telegram renderers, interaction dispatch behavior, docs, and focused contract tests; otherwise a maintainer should make an explicit product decision to narrow or reject the feature.
