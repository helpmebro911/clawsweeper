## Summary

Keep #29907 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still only supports cron payloads for `systemEvent` and `agentTurn`; there is no `exec` or `lobster` cron kind, no condition/onCondition contract, and no deterministic non-LLM cron execution path. Lobster exists, but as an optional agent tool, so cron use still requires an agent turn. Related #18160 remains an open direct-exec subset and does not fully supersede the Lobster pipeline, condition evaluation, silent no-op, or conditional handoff scope requested here.

## Best Possible Solution

Keep #29907 open unless maintainers explicitly supersede it with a broader cron-executor issue. The best product path is either an additive core cron/protocol feature or a stable cron-executor plugin seam covering deterministic `exec` and Lobster pipeline execution, timeout/security policy, bounded output capture, condition evaluation (`exitCode`, stdout, JSON path, always), silent no-op runs, conditional enqueue/delivery of an `agentTurn`, run-log/task-state visibility, CLI and Control UI support, public docs, and protocol/runtime tests.
