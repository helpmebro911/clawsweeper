## Summary

Keep open. Current main still lacks the requested opt-in shared Slack thread/session routing knobs and continues to route Slack thread replies into per-thread session keys. The linked implementation PRs were closed unmerged, while the broader related tracker #19929 remains open rather than superseding this Slack-specific report.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread and channel sessions, then implement it in Slack-owned routing using the existing resolveThreadSessionKeys useSuffix seam plus config type/schema/help/docs updates. The implementation should align inbound and outbound session keys, key thread history by Slack thread identity rather than shared session key, replace session-existence gates with per-thread seen state, preserve MessageThreadId for reply targeting, and add focused regression tests. If maintainers want one umbrella tracker, explicitly make #19929 canonical and migrate the Slack-specific requirements there before closing this as a duplicate.

## Risks / Open Questions

- The requested behavior intentionally reduces conversation isolation, so implementation needs explicit opt-in defaults, privacy/security docs, reset semantics, and regression tests for cross-thread/channel context leakage.
- Inbound and outbound session-key alignment is easy to regress because Slack has both monitor routing and outbound thread-aware routing paths.
- The older linked PRs predate later Slack routing changes such as runtime conversation bindings, so they should be treated as design notes rather than applied directly.
