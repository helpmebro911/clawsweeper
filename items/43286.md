## Summary

Keep this issue open. Current main still preserves Slack thread session isolation by default, has no `session.threadIsolation` or `session.channelIsolation` config surface, and the linked implementation PRs were closed unmerged. The latest Slack routing fix aligned thread roots and follow-up replies, but it did not implement the requested opt-in shared session behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread/channel sessions, then implement it in the Slack routing owner surface using the existing `resolveThreadSessionKeys` `useSuffix` seam plus config schema/help/docs updates, inbound and outbound session-key alignment, per-thread history keys, starter/history gating based on per-thread state rather than shared session existence, and regression tests. If maintainers want one umbrella, explicitly consolidate this into #19929 instead of closing it as stale.
