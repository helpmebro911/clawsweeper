## Summary

Keep #29552 open. Current main still has the same plausible local gateway auth failure path behind the reported pairing failure: local client calls resolve credentials once, ordinary local calls still prefer OPENCLAW_GATEWAY_TOKEN over gateway.auth.token, and the only automatic AUTH_TOKEN_MISMATCH retry is the trusted cached device-token path, not a config-auth fallback. The directly related fallback PR #29676 was closed unmerged, and recent issue comments report the version/token drift symptoms persisted into April 2026.

## Best Possible Solution

Keep this open. The best maintainer path is to add a guarded local-only retry in src/gateway/call.ts: after AUTH_TOKEN_MISMATCH, only when there is no remote mode/URL override and no explicit token/password, retry once using config-resolved gateway.auth.token/gateway.auth.password. Add regression coverage in src/gateway/call.test.ts for stale OPENCLAW_GATEWAY_TOKEN. If env-first local call precedence is intentional, document that explicitly and split the update/version-drift symptoms into a narrower canonical issue.
