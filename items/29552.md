## Summary

Keep #29552 open. Current main has partial split-brain/version-drift hardening, but the reported pairing failure still has a plausible unresolved local gateway auth path: local calls can prefer a stale OPENCLAW_GATEWAY_TOKEN over gateway.auth.token, callGatewayWithScopes resolves credentials once, and the automatic AUTH_TOKEN_MISMATCH retry is limited to cached device-token recovery. The directly related fallback PR #29676 was closed unmerged, so there is no canonical shipped fix for the remaining token-mismatch failure.

## Best Possible Solution

Keep this open and narrow the maintainer fix to the remaining pairing failure: add a guarded local-only retry in src/gateway/call.ts after AUTH_TOKEN_MISMATCH, only when there is no remote mode/URL override and no explicit token/password, retrying once with config-first gateway.auth.token/password. Add regression coverage in src/gateway/call.test.ts for stale OPENCLAW_GATEWAY_TOKEN. If env-first local call precedence is intentional, document that explicitly and split the version/update drift symptoms into a separate canonical issue.
