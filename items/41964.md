## Summary

Keep this PR open. Current main still has the `isControlUiVisible` gates the PR wants to remove, so the proposed behavior is not implemented, and the TUI live-display gap remains plausible. The patch is also not safe to merge as-is because it widens a documented hidden-run visibility boundary for external-channel runs.

## Best Possible Solution

Keep this PR open, but request a revised design before merge. Preserve `isControlUiVisible` for general Control UI and node-session fanout, wire Gateway TUI `--session` mode to the existing `sessions.messages.subscribe` / `session.message` transcript stream, and add a narrowly authorized TUI/session subscriber path for assistant deltas if transcript events are not enough. Coordinate with #38829 and #45417, then add regression tests for non-internal channel sessions plus explicit visibility/fanout authorization tests.

## Risks / Open Questions

- Merging as-is would broaden chat and node-session fanout for external-channel runs deliberately marked non-control-UI-visible, including Telegram-origin sessions.
- Operator-read WebSocket clients receive global `chat` broadcasts by scope rather than explicit per-session subscription, so the one-line guard removal is wider than a TUI-only fix.
- Closing now would discard a focused reproduction and patch attempt for a still-plausible TUI live-display bug; #38829 remains open, and #45417 addresses inbound user-message live tailing rather than this PR's assistant-streaming path.
