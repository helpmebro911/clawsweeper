## Summary

Current `main` still appears to implement the two contamination paths described in #66702: it scans the full runtime-enriched prompt text for image refs, and it intentionally preserves image blocks from the last three completed turns in session history. I did not find a strong fix on `main`, so this is not a conservative close.

## Evidence

- **Runtime-enriched prompt is passed straight into image detection:** `attempt.ts` builds `effectivePrompt` by prepending hook/plugin context and then passes that same string to `detectAndLoadPromptImages(...)` with no block-stripping step in between.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2158](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/pi-embedded-runner/run/attempt.ts#L2158)

- **Image detector scans the entire prompt text for attachment markers and paths:** `detectImageReferences(prompt)` runs regex scans over the full prompt string, including `[media attached: ...]`, message-image markers, file URLs, and path-like text. `detectAndLoadPromptImages` calls it directly on `params.prompt` and treats matches as current-prompt refs.
