## Summary

Current main still has a plausible stale-context path for this report. Manual compaction stores `tokensAfter` as fresh `totalTokens`, and auto-compaction can derive a post-run total from `lastCallUsage`, but the shared status formatter then enables transcript usage fallback and overwrites the smaller stored total whenever the transcript prompt snapshot is larger. That can make `/status` and `session_status` show the pre-compaction prompt size until a later turn writes a fresh smaller usage snapshot.

## Best Possible Solution

Keep this issue open and fix the shared status/session_status formatter so fresh post-compaction `entry.totalTokens` wins over older transcript prompt snapshots. A conservative patch would treat `totalTokensFresh === true` after a compaction as authoritative, or make transcript fallback compaction-aware by ignoring pre-compaction usage when `entry.compactionCount` or a latest compaction checkpoint indicates a newer post-compaction token count. Add a focused regression test with `entry.totalTokens` set to a small fresh post-compaction value and transcript usage containing a larger pre-compaction prompt snapshot, asserting the Context line reports the small fresh value while compaction history remains visible separately.
