## Summary

Current main has partial post-compaction accounting, but the reported stale `/status` path still looks actionable. Manual `/compact` records `tokensAfter` and the immediate compact reply uses it, yet the shared status formatter can still replace that fresh post-compaction store value with a larger transcript-derived prompt snapshot from the same unrotated transcript. The related closed ClawSweeper reports addressed missing/zero usage fallback, but this issue is narrower: the fallback can become too aggressive immediately after compaction.

## Best Possible Solution

Keep the issue open and fix the status accounting boundary: after compaction, `/status` and `session_status` should treat a fresh post-compaction `SessionEntry.totalTokens` as authoritative over older larger transcript-derived usage, or the compaction path should rotate/truncate/annotate transcript usage so fallback cannot select the pre-compaction peak. Add a regression test that simulates a session with large transcript usage, runs/stubs compaction with a small `tokensAfter`, then verifies the next status card shows the small post-compaction context and, if desired, a separate compaction delta such as `last: 106k -> 36`.
