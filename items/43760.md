## Summary

Keep this issue open. Current main still has one shared `memorySearch.remote` object and models `memorySearch.fallback` as a string provider id, so Gemini primary plus an OpenAI-compatible fallback endpoint still cannot be configured independently. The related implementation PR #44878 is closed unmerged, and broader #55590 is also closed.

## Best Possible Solution

Keep this issue open. A good fix should preserve the existing string fallback form for compatibility while adding an object fallback shape such as `{ provider, model, remote }`, then thread fallback-specific remote, model, headers/secrets, output dimensionality where needed, and provider defaults through config validation, `resolveMemorySearchConfig`, `createEmbeddingProvider`, runtime fallback activation, docs/help metadata, and regression tests for Gemini primary plus OpenAI-compatible fallback remote.
