## Summary

Keep PR #39489 open. Current main still does not thread Feishu runtime connection/activity status into the gateway channel health snapshot, so the PR is not obsolete or cleanup-close material. It still needs revision before merge because the current health contract distinguishes app events from transport liveness and uses `lastTransportActivityAt` for stale-socket checks, while this PR only reports `connected`, `mode`, `lastEventAt`, and `lastError` and has no focused Feishu status tests.

## Best Possible Solution

Keep the PR open and ask for a rebase/revision that uses the existing channel status snapshot fields. The Feishu startup path should create/pass a status sink from `ctx.setStatus` through `monitorFeishuProvider`, `monitorSingleAccount`, event handlers, and webhook/websocket transports; report `connected`, `mode`, `lastConnectedAt`, `lastEventAt`, `lastTransportActivityAt`, and `lastError` at validated lifecycle/activity points; preserve the webhook guard-ordering fix already discussed; and add focused Feishu tests for websocket/webhook status behavior. Maintainers should also pick the canonical branch among the overlapping Feishu health PRs before requesting final changes.
