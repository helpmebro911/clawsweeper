## Summary

Current main still does not implement the requested Discord daemon-install bridge. The setup path still supports `DISCORD_BOT_TOKEN` as a default-account env-backed SecretRef, but gateway service env generation only persists durable `.env`/config env vars, auth-profile env refs, and generated service metadata. The linked fix PR #67881 was closed unmerged in the provided context, so this issue remains actionable.

## Best Possible Solution

Keep this issue open. The best fix is to revive or reimplement the #67881 approach in current main: during gateway daemon install, detect the default Discord `channels.discord.token` env-backed SecretRef, copy the resolved install-time `DISCORD_BOT_TOKEN` value into the managed service environment when present, keep the existing dangerous-env filtering, and add a focused regression test in `src/commands/daemon-install-helpers.test.ts`. If maintainers decide not to persist channel SecretRefs, update the Discord setup wizard and docs to direct daemon users to `~/.openclaw/.env`, `env.vars`, or explicit service metadata instead of relying on the invoking shell.
