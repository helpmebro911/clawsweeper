## Summary

Current `main` already auto-sends typing indicators during active Discord reply runs, but the shared message tool still does not expose any `typing` action or capability, and the plugin architecture explicitly warns against adding provider-native UI escape hatches to the generic message tool. That means issue #30529 is not implemented as filed, but it is still coherent and potentially valid, so there is not strong evidence to close it.

## Risks / Open Questions

- The user-facing problem is only partially covered today: automatic typing exists for normal Discord reply runs, but there is still no explicit message-tool action for agents to start/stop typing during custom tool orchestration.
- If maintainers want this, it likely needs a generic cross-channel capability design rather than a Discord-only message-tool escape hatch, because current architecture pushes back on provider-native message-tool actions.
