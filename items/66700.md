## Summary

Keep open. Current main still treats Nextcloud Talk webhook `object.content` as an opaque string and applies only regex-based mention matching, while the upstream bot webhook contract uses JSON-encoded rich-object content with placeholders and `parameters`. The bundled Nextcloud Talk plugin therefore still has no path to resolve `{mention-userN}` placeholders to the configured bot identity before `requireMention` gating.

## Best Possible Solution

Keep this issue open. The fix should live in `extensions/nextcloud-talk`: parse JSON rich-object `object.content` into display text plus `parameters`, preserve mention parameter metadata, resolve `{mention-userN}` entries whose rich-object user ID matches the configured bot identity such as `apiUser` or a new explicit bot-user setting if needed, feed that as an explicit mention fact into the shared mention decision path, and add regression tests for `requireMention: true` group rooms where bot placeholders pass and non-bot placeholders still drop. Coordinate with #49869 if that PR remains the best rich-object parsing base, but do not close this issue until the mention-gating behavior is actually shipped on main.
