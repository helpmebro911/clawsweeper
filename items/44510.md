## Summary

Current `main` does not implement this PR’s lock-loss protections. The watchdog still force-releases session locks without notifying the active holder, `runEmbeddedAttempt` does not install any write-time lock-ownership guard, and the proposed follow-up fixes in the PR discussion were not landed on `main`. This remains a plausible correctness bug, so conservative cleanup should keep the PR open.

## Evidence

## Risks / Open Questions

- The bug is still plausible on long-running or wedged runs: once the watchdog releases the lock, current main has no mechanism to stop the original holder before more session writes occur.
- Current startup repair only recovers malformed JSON lines after corruption; it is not equivalent to preventing concurrent writes in the first place.
