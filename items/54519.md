## Summary

Keep open. Current main has a useful partial `session.sendPolicy` path that can suppress normal auto-reply delivery while still processing inbound messages, but it does not implement the requested per-account channel read-only mode or a hard gateway-level no-outbound guarantee. The shared channel config and WhatsApp account schema still have no `readOnly` field, and direct/gateway sends plus message-tool/plugin action sends can still reach outbound delivery without a first-class account policy gate. PR #71382 is an open WhatsApp-specific candidate, not a merged generic solution for this broader channel-level request.

## Best Possible Solution

Keep this issue open. The best path is a generic channel/account outbound delivery policy, likely exposed on shared channel/account config and schema, enforced before gateway/direct outbound delivery and message action/plugin handoff. It should cover text, media, polls/actions, typing/reactions/read receipts where channels support them, include docs/config metadata/UI hints, and add regression tests for auto-reply, direct gateway sends, message-tool sends, and at least one bundled account-based channel such as WhatsApp. Coordinate with PR #71382 for the WhatsApp-specific transport layer work.

## Risks / Open Questions

- A `message_sending` plugin can be used as a workaround to cancel some outbound delivery, but it is not a first-class per-account gateway guarantee and does not cover every plugin-owned transport side effect by construction.
- PR #71382 may cover a WhatsApp-specific subset if it lands, so maintainers should coordinate with it rather than duplicate the WhatsApp transport work; it does not by itself solve the generic per-account channel policy requested here.
