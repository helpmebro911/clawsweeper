## Summary

Keep #11489 open. Current main still implements pairing for DMs and devices only; group access remains open/allowlist/disabled with explicit group/sender allowlists. There is no `groupPolicy: "pairing"`, pending group approval state, group approve/reject CLI shape, owner-visible group-join notification path, or pending read-only group-agent mode. The already-linked #8495 and PR #42400 are narrower adjacent work, not a canonical replacement for this request.

## Best Possible Solution

Keep #11489 open for maintainer/product design. The likely path is an additive core/channel contract: add `groupPolicy: "pairing"` to config/schema/docs, persist pending group approvals keyed by channel/account/group ID, add CLI approve/reject semantics, emit owner-visible group-join or first-seen-unconfigured-group notifications, teach group-capable channels such as Telegram and WhatsApp to honor that state, and decide whether pending read-only observation belongs in the same change or a narrower follow-up coordinated with #8495 and PR #42400.
