## Summary

Keep open. Current main still has only generic, standard-emoji, queue-only Telegram reaction notifications; it does not include the PR's `reactionSemantics` config, `custom_emoji` normalization, `reaction_key` system-event metadata, or mapped-reaction heartbeat wake behavior. The PR discussion shows useful unmerged work tied to open #47677, with review feedback addressed, so this is not obsolete cleanup.

## Best Possible Solution

Keep the item open. The best path is to continue the Telegram plugin implementation: add a validated `reactionSemantics` config, normalize standard emoji and Telegram `custom_emoji` reactions into stable keys, include those keys in system events, honor auth and reaction notification modes, wake only after a mapped `wake` reaction is successfully enqueued, and cover the behavior with focused schema, helper, handler, docs, and changelog updates while preserving the plugin/core boundary.
