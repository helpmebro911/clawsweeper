## Summary

Current `main` still reports only `PLAINTEXT_FOUND` for resolved config secrets and has no authored-config path to distinguish exact `${ENV_VAR}` templates, so this PR still addresses a live gap rather than something already implemented or obsolete.

## Evidence

- **Audit report types on main still lack migration guidance:** `SecretsAuditCode` only includes `PLAINTEXT_FOUND`, `REF_UNRESOLVED`, `REF_SHADOWED`, and `LEGACY_RESIDUE`, and the summary only counts those buckets. There is no `MIGRATION_RECOMMENDED` code or `migrationRecommendedCount` on current `main`.
  - file: [src/secrets/audit.ts:41](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/secrets/audit.ts#L41)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Current audit logic still collapses authored `${ENV_VAR}` into plaintext:** `runSecretsAudit()` feeds `collectConfigSecrets()` with `snapshot.config`, and `createConfigFileSnapshot()` defines `config` as the runtime config. `resolveConfigForRead()` applies env substitution before that snapshot is exposed. In `collectConfigSecrets()`, any non-ref secret-like value falls through to `PLAINTEXT_FOUND`, so exact authored templates cannot be distinguished once resolved.
