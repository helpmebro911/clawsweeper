## Summary

Keep #37816 open. Current main still does not provide an opt-in Telegram mode that mirrors Web UI activity as durable, separate messages. The implemented surfaces are preview edits, short tool-progress lines in the same edited preview, default tool-progress messages when preview streaming is off, `/verbose` tool bubbles, and `/reasoning` controls; those do not cover full persistent per-activity visibility for reasoning, memory, command/tool output, and other agent activity. The related Telegram issues are narrower closed bugs and do not supersede this feature request.

## Best Possible Solution

Keep #37816 open as bundled Telegram plugin work. Add an explicit opt-in `channels.telegram` mode or flag with schema, docs, and tests that emits durable separate messages for relevant agent activity while preserving current preview-edit behavior by default. Use existing replyOptions callbacks where they are sufficient; if reasoning, memory, or other activity sources are not exposed cleanly, add a narrow additive channel/plugin event contract rather than hard-coding around internals. Suppress NO_REPLY/internal runs and include rate-limit and sensitive-output safeguards.

## Risks / Open Questions

- A durable per-activity Telegram mode could be noisy and may hit Telegram rate limits, especially in groups or tool-heavy turns.
- Reasoning, memory, command output, and tool result snippets can expose sensitive data, so the mode should be explicit opt-in with filtering/redaction.
- Internal housekeeping turns such as NO_REPLY and memory flushes need deliberate suppression so background activity does not leak into user-facing chats.
