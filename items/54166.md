## Summary

Keep this issue open. Current main still lets llm-task choose provider and model independently, only strips a duplicate prefix when the model already starts with the selected provider, and dispatches the unresolved pair to the embedded runner. That does not satisfy the reported cross-provider override cases such as an Anthropic default provider with `model: "google/gemini-3-flash-preview"`, and it also validates `allowedModels` before any shared model-alias/provider-ref resolver is used.

## Best Possible Solution

Keep this issue open. Resolve llm-task `model` and `defaultModel` overrides through the same canonical model-ref/alias resolver used by normal agent model selection before allowlist validation and before `runEmbeddedPiAgent`, then pass the resolved provider plus bare model ID. Add focused llm-task regression coverage for a bare alias such as `gemini-flash`, an explicit cross-provider ref such as `google/gemini-3-flash-preview` while the default provider is Anthropic or OpenAI, an Anthropic alias, and an allowlist case that validates the resolved canonical key. If the bundled plugin cannot use the resolver through the current boundary, add a narrow plugin runtime/SDK resolver seam first.
