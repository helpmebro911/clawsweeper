## Summary

Keep open. Current main still has the llm-task model-routing bug described here: the tool derives `provider` and `model` separately, validates the raw `provider/model` pair, and passes those raw values to the embedded runner. A `model` value such as `gemini-flash` or `google/gemini-3-flash-preview` is not resolved through the model alias/provider-ref path before dispatch, so the selected provider can remain the session/default provider. The related discussion is useful: #48576 is the closest resolver-family bug but covered a different surface, while #26018/#26175 were haiku-alias-specific and do not fix llm-task bypassing alias resolution.

## Best Possible Solution

Keep this issue open. The best fix is to resolve llm-task's requested model through the same alias/provider-ref path used by other model-selection surfaces before calling `runEmbeddedPiAgent`, then pass the resolved provider and bare model id. The allowlist check should run against the resolved canonical `provider/model` key. Because bundled plugin code currently only has the narrow `openclaw/plugin-sdk/llm-task` surface, the maintainable implementation is either to add a narrow plugin-facing/runtime resolver seam or to make the runtime agent helper accept and resolve a model ref explicitly, with focused tests for `model: "gemini-flash"`, `model: "google/gemini-3-flash-preview"`, and an Anthropic alias.
