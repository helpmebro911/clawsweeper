## Summary

Keep open. Current main supports selecting/listing existing sessions, but it does not implement the requested tab system for keeping multiple chat sessions open, creating/closing/reordering tabs, or preserving separate visible chat state per tab.

## Best Possible Solution

Keep the issue open for maintainer/product prioritization. The likely implementation belongs in the Control UI chat surface: add an open-session tab model, active tab switching over existing session keys, create/close/reorder actions, local persistence of open tabs, and targeted tests around switching while drafts, streams, queues, and session-scoped overrides are present. Update user-facing Control UI/session docs if shipped.

## Risks / Open Questions

- A tabbed Control UI would need careful per-session draft, attachment, stream, side-result, queue, and scroll state so switching tabs does not discard work or misroute active-run aborts.
- The existing session picker/list may solve part of the workflow, so the product decision should define whether tabs are purely a UI affordance over existing session keys or require new-session creation semantics.
