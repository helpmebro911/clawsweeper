## Summary

Keep open. Current main still contains the Telegram no-visible-response path that synthesizes a `NO_REPLY` payload, outbound policy still rewrites that silent payload into visible direct-message filler, and the regression test still asserts the unwanted visible fallback. PR #70630 is an open, unmerged implementation candidate linked to this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review or land PR #70630, or an equivalent narrow Telegram adapter fix. The fix should treat successful no-visible-response Telegram direct turns as handled silent no-ops, avoid synthesizing transport-visible `NO_REPLY`, preserve real dispatch-error and non-silent-delivery-failure fallback behavior, and update the Telegram dispatch regression test accordingly.
