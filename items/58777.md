## Summary

Current `main` partially addresses the report: it now records and broadcasts selected-vs-active model/provider fallback state, and the UI can surface that drift. But the issue is not fully implemented because there is still no dedicated `effective_*` telemetry on `main`, and the generic thinking-level clamp still happens only against the initially requested model before fallback, while runtime/request-shaping paths continue to carry the original `thinkLevel` rather than a re-derived effective value for the winning fallback model.

## Evidence

- **Actual fallback model/provider are now persisted:** `agent-runner.ts` computes `modelUsed`/`providerUsed` from run metadata, resolves fallback transition state, and persists `fallbackNoticeSelectedModel`, `fallbackNoticeActiveModel`, and `fallbackNoticeReason` into the session store.
  - file: [src/auto-reply/reply/agent-runner.ts:1279](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/agent-runner.ts#L1279)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Lifecycle events carry selected vs active model/provider:** `agent-runner.ts` emits `lifecycle` events for `fallback`/`fallback_cleared` with `selectedProvider`, `selectedModel`, `activeProvider`, and `activeModel`, so there is now model/provider fallback telemetry on current `main`.
  - file: [src/auto-reply/reply/agent-runner.ts:1503](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/agent-runner.ts#L1503)

- **Session runtime model is reconciled after runs:** `persistSessionUsageUpdate()` writes `modelProvider` and `model` back to the session entry from `providerUsed`/`modelUsed`, which addresses stale selected-vs-active model state after fallback.
  - file: [src/auto-reply/reply/session-usage.ts:135](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/auto-reply/reply/session-usage.ts#L135)
