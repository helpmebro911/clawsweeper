## Summary

Current main still preserves the reported failure shape. The CLI sends an `agent` request through the Gateway with `expectFinal: true`, treats any outer Gateway timeout as an error, and then calls embedded `agentCommand(localOpts, ...)` with the original session selectors. The Gateway can already have accepted and dispatched the run, and the embedded fallback can resolve the same transcript file and hit the same 10s session write-lock timeout. Existing docs still advertise embedded fallback, and the current targeted tests only cover ordinary Gateway failure fallback, not lock-safe timeout fallback.

## Best Possible Solution

Keep this issue open as the focused tracker for Gateway-timeout-to-embedded-fallback session lock contention. Fix `src/commands/agent-via-gateway.ts` so a timed-out accepted Gateway run is handled by waiting via `agent.wait`, aborting and waiting for release, or switching embedded execution to a clearly fresh user-visible session. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before embedded fallback, and update `https://docs.openclaw.ai/cli/agent` if fallback semantics change. Coordinate with #71605 for the broader Gateway dispatch-timeout work.
