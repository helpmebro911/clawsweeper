## Summary

Current main still has the reported timeout-fallback/session-lock failure shape. The CLI sends the original session selectors to the Gateway, waits for a final response with a client-side timeout, and falls back to embedded execution with the same options. The Gateway run can continue after the CLI WebSocket closes, while the embedded runner resolves the same session transcript and still times out on a live session write lock.

## Best Possible Solution

Fix the CLI/Gateway fallback contract in `src/commands/agent-via-gateway.ts`: after a Gateway `agent` request has been accepted and the final wait times out, either wait through `agent.wait`, explicitly abort the Gateway run and wait for lock release before retrying, or start embedded under a clearly fresh user-visible session. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for timeout fallback/session isolation, and update `docs/tools/agent-send.md` or the relevant CLI docs if fallback semantics change.
