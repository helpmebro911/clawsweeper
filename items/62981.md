## Summary

Current main still leaves the reported Gateway-timeout-to-embedded-fallback lock contention open. The CLI waits for the final Gateway `agent` response, times out while the accepted daemon-side run may still be active, then falls back to embedded execution with the original session selectors. The session lock change in current main is only a partial mitigation: it delays lock acquisition until after cold setup, but an active Gateway run can still hold the same transcript lock through the model attempt.

## Best Possible Solution

Keep this issue open as the focused tracker for Gateway-timeout-to-embedded-fallback session lock contention. Fix `src/commands/agent-via-gateway.ts` so an accepted Gateway run that times out is handled by `agent.wait`, abort-and-wait, or an explicitly fresh user-visible session before embedded execution. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before fallback, and coordinate the broader dispatch-timeout behavior with #71605.
