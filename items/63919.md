## Summary

Current main still does not wire `createOpenClawCodingTools` into the Gateway `/tools/invoke` resolver, so this PR is not obsolete or implemented. The PR addresses a real gap supported by the current docs and HTTP deny-list shape, but it remains a security-sensitive implementation candidate that needs maintainer review for hook wrapping, adjusted-param cleanup, and rebase drift before merge.

## Best Possible Solution

Keep this PR open for maintainer review and iteration. The best path is to rebase it on current main, preserve the `surface === "http"` loopback guard and Gateway-tool precedence, pass the current request's owner context where needed, and resolve the wrapper issues by either constructing an HTTP-safe unwrapped coding-tool set or teaching `/tools/invoke` to detect wrapped tools and consume adjusted params correctly. Add regression tests for allowlisted coding-tool availability, default deny-list behavior, loopback non-expansion, single `before_tool_call` execution, adjusted-param cleanup, and owner/scoped-auth behavior before landing.

## Risks / Open Questions

- Merging the PR as-is may run `before_tool_call` and loop detection twice for HTTP coding tool calls because `/tools/invoke` runs the hook and `createOpenClawCodingTools` returns hook-wrapped tools.
- Routing wrapped coding tools through direct HTTP invoke may retain adjusted params unless the HTTP path consumes or clears them after execution.
- The PR head was authored against an older resolver shape and needs rebasing onto current main to preserve current subagent-envelope policy and current HTTP `senderIsOwner` semantics.
- The capability is security-sensitive: once allowlisted, execution and file-mutation tools become reachable over the Gateway HTTP operator surface, so deny-list, owner policy, loopback non-expansion, and hook behavior need targeted regression coverage.
