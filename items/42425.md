## Summary

Current main still does not implement this PR’s central behavior: Gateway startup loads internal hooks once from the default workspace, while docs describe workspace hooks as per-agent. The PR is stale against the current startup file layout and needs rework, but it remains a plausible implementation candidate rather than something to close as obsolete or superseded. Security review did not show CI/dependency/supply-chain file changes in the provided diff, but the functional change expands trusted local hook-code loading at Gateway startup and needs careful policy-preserving review before merge.

## Best Possible Solution

Keep this PR open, but require a rebase and scoped rewrite against current main. The best path is a startup-specific multi-workspace hook loading flow wired from `src/gateway/server-startup-post-attach.ts` using `listAgentWorkspaceDirs`, while preserving the current hook policy: workspace hooks remain explicit opt-in, boundary-checked trusted local code, and unable to override bundled, managed, or plugin hooks. Add focused regression tests for an enabled hook in a non-default agent workspace, workspace isolation, failure isolation, and precedence/security behavior; then review and land the refreshed PR or replace it with a maintainer-owned patch.
