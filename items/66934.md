## Summary

Keep open. Current `main` appears to have addressed the message flicker/desync path, but the sender-label bug described in #66934 is still present in code: user-message sanitization still extracts `senderInfo.label` from untrusted inbound metadata, and the chat renderer still prefers that extracted `senderLabel` over the local fallback `You`. That means webchat history can still show `openclaw-control-ui` for the local sender, so the issue is not fully implemented on `main` and should not be auto-closed.

## Evidence

- **Inbound sender-label extraction still prefers metadata label/id:** `extractInboundSenderLabel()` still returns `senderInfo.label` first, then falls back through `name`, `username`, `e164`, `id`, and conversation sender. For webchat/control-ui messages, that metadata can contain `openclaw-control-ui`, matching the issue report rather than forcing a local `You` label.
  - file: [src/auto-reply/reply/strip-inbound-meta.ts:298](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/auto-reply/reply/strip-inbound-meta.ts#L298)
  - command: `nl -ba src/auto-reply/reply/strip-inbound-meta.ts | sed -n '294,314p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Sanitization still copies extracted senderLabel onto user messages:** `stripEnvelopeFromMessage()` calls `extractMessageSenderLabel()` for `role === user`; that helper returns the extracted inbound label from `content`/`text`, and the sanitizer writes it back to `next.senderLabel`. There is no control-ui/webchat special-case here.
  - file: [src/gateway/chat-sanitize.ts:11](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/gateway/chat-sanitize.ts#L11)

If history supplies `openclaw-control-ui`, the UI will show that string instead of `You`.
