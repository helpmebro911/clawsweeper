## Summary

Current main still does not implement the requested Gateway/channel workflow for `/agents` plus `/agent <name>` to temporarily switch a single conversation to another configured agent without editing global bindings or restarting the Gateway. Existing surfaces cover static binding-based routing, thread/subagent focus, TUI-local agent/session switching, and plugin-specific static overrides, but not the requested per-conversation dynamic route override.

## Best Possible Solution

Keep this open for maintainer-designed Gateway/channel routing work. The likely path is to add authorized `/agents` and `/agent <id|main>` handling that lists configured agents, stores a temporary per-conversation route override, applies it before normal binding fallback for that conversation, and defines reset/expiry semantics. If this should live in ClawHub or a DingTalk/channel plugin, first add a public plugin API for safe per-conversation route overrides rather than requiring plugins to edit global `bindings`.

## Risks / Open Questions

- Closing this now would conflate TUI-local agent/session selection and thread/subagent focus with the requested cross-channel temporary routing command.
- A correct implementation needs explicit authorization, per-conversation scope, reset/expiry semantics, and isolation of per-agent workspace, auth, memory, and session stores.
- If maintainers prefer a plugin-first implementation, core still needs a documented generic route-override API; asking channel plugins to mutate global bindings would be a risky workaround.
