## Summary

Current main still does not implement the requested Gateway/channel workflow for `/agents` plus `/agent <name>` to temporarily switch a single conversation to another configured agent without editing global bindings or restarting the Gateway. Existing surfaces cover static route bindings, TUI-local agent switching, and `/focus` bindings to existing subagent/ACP/session targets, but not a chat command that lists configured agents, creates or selects the target agent's session, and stores a per-conversation agent route override.

## Best Possible Solution

Keep this open for maintainer-designed Gateway/channel routing work. A complete fix should add authorized `/agents` and `/agent <id|main>` handling that lists configured agents, creates or targets the selected agent's main session when needed, stores a temporary per-conversation route override before normal binding fallback, and defines reset/expiry plus workspace/auth/memory isolation semantics. If maintainers prefer this as plugin-first channel behavior, core should still provide a documented way for plugins to create configured-agent route overrides by agent id rather than requiring global `bindings` edits.
