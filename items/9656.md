## Summary

Keep #9656 open. Current main still treats Signal edit envelopes as ordinary inbound messages: it reads only `editMessage.dataMessage`, does not type or preserve `editMessage.targetSentTimestamp`, derives `MessageSid` from the new envelope timestamp, and sends edits through the normal reply dispatch path. The existing issue comment reached the same keep-open conclusion, and the related PR #9701 is closed unmerged in the provided context.

## Best Possible Solution

Implement this in the bundled Signal plugin. Add a typed edit-message envelope with validated `targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, suppress edit-triggered replies unless the edited text explicitly mentions the agent, and add focused Signal tests for edit envelopes, `MessageSid`/dedupe behavior, and mention-gated group edits. PR #9701 can be used as historical context, but it was closed unmerged and should not be treated as shipped.
