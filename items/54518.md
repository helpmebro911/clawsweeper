## Summary

Current main still does not provide hard per-agent workspace confinement by default. The agent workspace is documented and implemented as the default working directory plus optional guards; `tools.fs.workspaceOnly` remains opt-in and tests explicitly preserve outside-workspace host file access when it is false or unset. Runtime surfaces such as `exec`, `process`, plugin tools, MCP/bundle tools, media paths, and sandbox/elevated modes are separate policy domains, so the user-visible request remains unresolved.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. A complete solution should define a per-agent confinement profile or onboarding option that enables workspace-scoped filesystem tools, denies or sandboxes runtime tools such as `exec` and `process`, covers plugin/MCP/media path surfaces, updates skill guidance away from fixed default-workspace paths, and adds regression coverage for per-agent boundaries plus legacy host-file workflows.

## Risks / Open Questions

- Changing the default boundary would be a compatibility and product-policy change because current docs, config defaults, implementation, and tests intentionally allow host file access unless `tools.fs.workspaceOnly` or sandboxing is enabled.
- `tools.fs.workspaceOnly` alone would not fully satisfy the issue because `exec`, `process`, plugin tools, MCP/bundle tools, media paths, sandbox mounts, elevated execution, and channel/provider-specific file paths need separate confinement policy.
- Third-party or shared skills that hard-code `~/.openclaw/workspace` can still steer an agent toward the wrong workspace unless skill guidance, installers, or runtime context make the active workspace explicit.
