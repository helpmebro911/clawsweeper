## Summary

Current main has an opt-in per-agent/global `tools.fs.workspaceOnly` guard and sandbox/tool-policy options, but it still does not make an agent workspace a hard default filesystem boundary. With `workspaceOnly` unset or false, read/write/edit can still reach host paths outside the active workspace, and exec/process remain separate escape surfaces that need sandboxing or denial.

## Best Possible Solution

Keep this issue open for maintainer product judgment. The strongest path is a per-agent confinement profile or setup/wizard option that sets `tools.fs.workspaceOnly: true`, denies or sandboxes runtime tools like `exec` and `process`, and guides skills to use the active agent workspace instead of fixed `~/.openclaw/workspace` paths. If maintainers want workspaces to become hard default boundaries, implement that as an explicit behavior change with migration notes and regression coverage for file tools, exec workdir handling, sandbox modes, plugin/MCP tool surfaces, and per-agent overrides.

## Risks / Open Questions

- Changing the default workspace boundary would be a compatibility/product decision because current main intentionally preserves legacy host-file access when `tools.fs.workspaceOnly` is unset or false.
- `tools.fs.workspaceOnly` alone is not complete confinement; exec/process, elevated mode, plugin tools, and MCP tools still need sandboxing or explicit tool policy restrictions.
- No live end-to-end agent run was performed because this review was required to keep the checkout byte-for-byte clean, but source, tests, docs, and the issue discussion are sufficient to decide the item remains open.
