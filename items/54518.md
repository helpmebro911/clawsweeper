## Summary

Current main still does not provide hard per-agent workspace confinement by default. The code and docs intentionally model an agent workspace as the default cwd plus optional guards; `tools.fs.workspaceOnly` is opt-in, sandboxing is opt-in, and `exec`, `process`, plugin, MCP, and media/path-loading surfaces remain separate policy domains. The prior ClawSweeper comment remains consistent with current main.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. A complete fix should define a per-agent confinement profile or onboarding option that enables workspace-scoped filesystem tools, denies or sandboxes runtime tools such as `exec` and `process`, covers plugin/MCP/media path surfaces, updates skill guidance away from fixed default-workspace paths, and adds regression coverage for per-agent boundaries while preserving intentional trusted-host workflows.

## Risks / Open Questions

- Changing the default workspace boundary would be a compatibility and security/product-policy change because current docs, tests, and implementation preserve unrestricted host file access unless explicit guards are enabled.
- `tools.fs.workspaceOnly` alone would not fully satisfy the reported user problem because `exec`, `process`, plugin tools, MCP/bundle tools, media path loading, sandbox/elevated modes, and skill instructions need coordinated confinement policy.
- Hard-coded workspace paths in shared or third-party skills can still steer an agent toward the default workspace unless skill guidance or runtime context makes the active per-agent workspace explicit, or core enforces a stronger boundary.
