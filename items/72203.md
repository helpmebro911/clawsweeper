## Summary

Keep #72203 open. It is a fresh, specific v2026.4.24 regression report with a plausible current-main path still capable of using a stale shell OPENCLAW_GATEWAY_TOKEN for the post-setup health check, even after QuickStart has generated/written a different gateway.auth.token. The related closed token-field reports cover Control UI signing/storage and older service drift cases, but they do not prove this setup health/dashboard failure is fixed on current main.

## Best Possible Solution

Keep the issue open and fix the setup health/dashboard auth path. The likely maintainer direction is to make finalizeSetupWizard pass the freshly resolved setup credential into the health check, or make local health/status calls prefer the persisted gateway.auth token over ambient OPENCLAW_GATEWAY_TOKEN during onboarding. Add a regression test where QuickStart writes gateway.auth.token=fresh-token while process.env.OPENCLAW_GATEWAY_TOKEN=stale-token, then assert both the post-setup health probe and Control UI/dashboard path authenticate with fresh-token.
