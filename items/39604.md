## Summary

Keep this issue open. Current main still does not expose a general `web_fetch` opt-in for localhost, LAN, or private/internal network targets; it only supports the narrower RFC 2544 fake-IP proxy range. The same-author implementation PR #39630 is open and unmerged with closing syntax for this issue, so the maintainer path is to review, revise, land, or close that PR before closing the issue.

## Best Possible Solution

Keep this issue open while maintainers review #39630 or a replacement implementation. The likely core path is a default-off `web_fetch` SSRF policy option, preferably with explicit dangerous naming, wired through the strict runtime schema, exported config types, generated config labels/help, cache-key isolation, public docs, and SSRF regression tests. If maintainers reject broad private-network access, document that product/security decision and point users to narrower alternatives such as the existing RFC2544 fake-IP policy or plugin-owned service integrations.

## Risks / Open Questions

- A broad private-network opt-in intentionally expands SSRF reach when enabled, so any implementation needs a secure default, explicit dangerous naming or narrow scope, cache-key isolation, docs/help/schema updates, and regression tests.
- The current RFC2544 option only helps `198.18.0.0/15` fake-IP proxy cases; it does not solve the localhost, `10.x`, `172.16-31.x`, or `192.168.x` workflows described in the issue and comments.
