## Summary

Current main still does not expose a `web_fetch` opt-in for localhost/LAN/private-network targets. The schema/type/runtime only wire `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, which addresses the RFC2544 fake-IP proxy range but not the issue’s requested `127.0.0.1`, `10.x`, `192.168.x`, or `172.16-31.x` access. The lower-level SSRF guard already supports `allowPrivateNetwork`/`dangerouslyAllowPrivateNetwork`, so the remaining work is web_fetch config/runtime/docs/test wiring and maintainer security judgment. The related PR #39630 remains the active implementation path, not proof that the issue is obsolete or shipped.

## Best Possible Solution

Keep #39604 open until maintainers either land or explicitly reject a web_fetch private-network opt-in. The likely implementation path is to evaluate or supersede PR #39630 with current-main changes across `src/config/zod-schema.agent-runtime.ts`, `src/config/types.tools.ts`, config labels/help/generated schema, `src/agents/tools/web-fetch.ts`, `src/agents/tools/web-fetch.ssrf.test.ts`, and `docs/tools/web-fetch.md`. Given current naming, a scoped `tools.web.fetch.ssrfPolicy.dangerouslyAllowPrivateNetwork` shape may fit better than the originally proposed flat key, but the user problem remains unsolved on main.

## Risks / Open Questions

- Do not treat `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` as satisfying this issue; it only allows the 198.18.0.0/15 fake-IP proxy range, not localhost/LAN/private networks generally.
- A general private-network web_fetch opt-in widens SSRF reach even if default-off, so the final config shape likely needs explicit maintainer security/product judgment.
