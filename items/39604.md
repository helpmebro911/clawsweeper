## Summary

Keep this issue open. Current main still does not expose the requested `tools.web.fetch.allowPrivateNetwork` opt-in, nor an equivalent `dangerouslyAllowPrivateNetwork`/hostname allowlist for `web_fetch`. The only shipped `web_fetch` SSRF config is the narrower `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange`, and the same-author implementation PR #39630 remains open with closing syntax for this issue.

## Best Possible Solution

Keep this issue open while maintainers review #39630 or a replacement implementation. The likely core path is a default-off `web_fetch` SSRF policy option, preferably with explicit dangerous naming or narrower hostname controls, wired through the strict runtime schema, exported config types, generated config labels/help, cache-key isolation, public docs, and SSRF regression tests. If maintainers reject broad private-network access, document that product/security decision and point users to narrower alternatives such as the existing RFC2544 fake-IP option or plugin-owned service integrations.

## Risks / Open Questions

- A broad private-network opt-in expands SSRF reach when enabled, so any implementation needs secure defaults, explicit naming or narrow host controls, docs/help/schema updates, cache-key isolation, and regression tests.
- The shipped RFC2544 option addresses fake-IP proxy ranges only; it does not cover the localhost, 10.x, 172.16-31.x, or 192.168.x workflows described by this issue and commenters.
