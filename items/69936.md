## Summary

Keep open. Current main still does not implement PR #69936's configured `approvals.{exec,plugin}.targets[].accountId` fallback for native approval account binding, so the no-turn-source/no-session-binding multi-account Telegram case can still take the permissive native-runtime path.

## Best Possible Solution

Keep the PR open or replace it with an equivalent maintainer patch. The best fix belongs in the shared native approval account-binding path, likely `src/infra/approval-request-account-binding.ts`, with conservative fallback logic that consults active `approvals.{exec,plugin}.targets[].accountId` only when unambiguous for the request/channel and with regression coverage in both infra binding tests and Telegram native approval handling.
