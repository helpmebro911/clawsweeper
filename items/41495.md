## Summary

Keep this issue open. Current main can render Telegram inline buttons when the structured `presentation` contract is used, but the reported Gemini failure mode is still plausible: the agent prompt still advertises the old `buttons=[[{text,callback_data,style?}]]` shape, the message tool schema exposes `presentation` instead of top-level `buttons`, and assistant final-text delivery does not parse the reported `[[ {...callback_data...} ]]` text into Telegram reply markup.

## Best Possible Solution

Keep this issue open and fix it as a contract-alignment bug: update the agent prompt and Telegram docs to steer models to `message` with `presentation.blocks[{type:"buttons"}]`, keep the existing Telegram presentation renderer, and add regression coverage for the prompt/tool schema plus Telegram delivery. Consider a conservative legacy parser for the exact reported `[[...]]` button shape only if maintainers explicitly want compatibility for already-generated text.

## Risks / Open Questions

- Adding a parser for arbitrary model-authored `callback_data` text could create unsafe or surprising callback injection unless it is narrowly validated and scoped.
- The current prompt/docs/schema mismatch may continue to steer weaker tool-calling models toward plain-text button JSON even though the Telegram renderer itself supports structured presentation buttons.
