## Summary

Keep open. Current main has partial rollover fixes that should prevent some new stale cron sessionFile pointers, but `sessions cleanup --fix-missing` still trusts one resolved persisted `sessionFile` path and deletes the entry when that file is missing. It does not fall back to the canonical `<sessionId>.jsonl` transcript for already-mismatched stores. The linked PR #69954 is relevant and explicitly targets this gap, but it is open/unmerged in the provided GitHub context.

## Evidence

- **Cleanup still prunes from a single resolved path:** `pruneMissingTranscriptEntries()` resolves `resolveSessionFilePath(entry.sessionId, entry, ...)`, checks only that path with `fs.existsSync`, and deletes the store entry if it is missing. There is no fallback check for `<sessionId>.jsonl` before pruning.

  - command: `nl -ba src/commands/sessions-cleanup.ts | sed -n '139,158p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Path resolver prefers persisted sessionFile when structurally valid:** `resolveSessionFilePath()` returns a stored `sessionFile` candidate when it resolves inside the sessions dir; it falls back to the derived transcript only when the candidate is absent or invalid, not when the stored file is missing or mismatched to `sessionId`.
  - file: [src/config/sessions/paths.ts:267](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/config/sessions/paths.ts#L267)

- Closing now would be premature; #69954 may become the canonical fix if merged, but the issue still represents remaining main-branch behavior today.
