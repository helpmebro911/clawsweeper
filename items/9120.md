## Summary

Keep open. Current main partially addresses the cost concern with timing-based inbound debounce and collect-mode queueing, but it still does not implement the requested user-controlled continuation marker, marker stripping, continuation timeout, or continuation-specific typing behavior. The existing maintainer/Codex comment in the issue reached the same conclusion, and rechecking current `main` at `b565e6e96310c03e81b32caf82894ce89f32fec5` did not find a newer implementation.

## Best Possible Solution

Keep the issue open for maintainer/product judgment. A complete implementation would likely extend the shared inbound message pipeline/config schema with a strict `messages.continuationMarker`/timeout contract, strip the marker before prompt delivery, release the batch on the first non-marker message or timeout, add channel-safe tests, and document how it interacts with `messages.inbound`, `messages.queue.collect`, media/control commands, and typing indicators.
