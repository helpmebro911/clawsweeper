## Summary

Keep open. Current main does not implement scheme-level wildcard matching for `gateway.controlUi.allowedOrigins`, and the related open PR #38290 explicitly covers exact non-standard extension origins only, not `chrome-extension://*`. The closed PR #39867 targeted this issue but was not merged, so it cannot be treated as implemented or superseding evidence.

## Evidence

- **Current matcher only supports global wildcard or exact normalized URL origin:** `checkBrowserOrigin` builds an allowlist and accepts only `allowlist.has("*")` or `allowlist.has(parsedOrigin.origin)`. There is no prefix, suffix, or scheme-level wildcard matching path for entries like `chrome-extension://*`.
  - file: [src/gateway/origin-check.ts:45](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/gateway/origin-check.ts#L45)
  - command: `nl -ba src/gateway/origin-check.ts | sed -n '1,240p'`
  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Chrome extension origins normalize to URL.origin `null` under current runtime behavior:** A live Node URL check returned `{"origin":"null","protocol":"chrome-extension:","host":"abcdefghijklmnop","hostname":"abcdefghijklmnop"}` for `chrome-extension://abcdefghijklmnop`, so current exact `parsedOrigin.origin` matching cannot match either an exact extension origin or `chrome-extension://*`.
  - command: `node -p "const u=new URL('chrome-extension://abcdefghijklmnop'); JSON.stringify({origin:u.origin,protocol:u.protocol,host:u.host,hostname:u.hostname})"`
  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Tests do not cover extension origins or scheme-level wildcard behavior:** The gateway origin tests cover host-header fallback, loopback, exact HTTPS allowlist, global `*`, missing/null/malformed origins, and mismatches, but include no `chrome-extension://...` or `chrome-extension://*` cases proving this request on main.
