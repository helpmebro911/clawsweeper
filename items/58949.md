## Summary

Keep open. Current `main` still uses the early-return implementation in `safeEqualSecret`, so this PR is not implemented on main, and the review discussion identified missing regression coverage for the exact null/empty-string cases that motivate the change.

## Evidence

- **Current main still fast-fails non-string inputs:** `safeEqualSecret` on current `main` returns `false` immediately when either input is not a string, before hashing/comparison.
  - file: [src/security/secret-equal.ts:7](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/security/secret-equal.ts#L7)
  - command: `nl -ba src/security/secret-equal.ts | sed -n '1,80p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Current tests do not cover the critical null/empty-string cases:** The existing table test covers `undefined`/`null` against non-empty secrets, but not `[null, "", false]`, `[undefined, "", false]`, `[null, null, false]`, or `[undefined, undefined, false]`.
  - file: [src/security/audit-extra.sync.test.ts:51](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/security/audit-extra.sync.test.ts#L51)
