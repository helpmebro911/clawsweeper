## Summary

Keep this PR open. Current main still does not deliver internal `message:received` or `message:sent` hook-produced `event.messages`, the linked bug #39688 remains open, and maintainer successor #41375 is open but unmerged. This PR is a plausible implementation candidate, but it is currently dirty against main and has unresolved session-context and hook-loop replay risks that should be consolidated with the maintainer path before merge or closure.

## Best Possible Solution

Keep this PR or an explicit maintainer successor open until #39688 is resolved. Maintainers should consolidate with #41375, decide and document whether `message:sent` is a replyable hook surface, implement a shared reply-drain helper only where a safe target exists, preserve session/media context, persist any hook-loop suppression through recovery if outbound replay is possible, and add regression tests plus hooks docs updates for the final contract.

## Risks / Open Questions

- The PR's `message:received` route-reply path omits `sessionKey`, so hook replies can diverge from normal agent-scoped response-prefix behavior.
- The PR's recursive `message:sent` delivery omits the originating `session`, which weakens agent-scoped media-root preservation if this surface later handles more than text.
- The proposed `skipMessageHooks` loop suppression is not persisted in the current queue/recovery contract, so replayed hook-generated deliveries could re-emit hooks unless the queue schema is extended or the path is guaranteed unqueued.
- Live GitHub reports the PR as `dirty`, so it needs a rebase and reconciliation against current main before it can be evaluated for merge.
- The desired `message:sent` auto-reply contract is still product-sensitive: maintainer successor #41375 intentionally limits auto-delivery to replyable surfaces and does not make all lifecycle hooks reply-capable.
