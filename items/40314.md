## Summary

Keep this PR open. Current main still emits internal `message:received` and `message:sent` hooks but does not drain `event.messages` into user-visible replies, the linked bug #39688 is still unresolved in main, and the maintainer successor #41375 is open/unmerged while intentionally leaving `message:sent` out of scope.

## Best Possible Solution

Keep this PR or an explicit successor open until #39688 is resolved. Maintainers should consolidate it with #41375, decide the intended `message:sent` reply contract, implement a shared reply-drain helper only for surfaces with safe reply targets, preserve session/media context, make loop suppression robust across delivery and recovery, and update regression tests plus the public hooks docs at https://docs.openclaw.ai/automation/hooks to match the final contract.

## Risks / Open Questions

- The proposed inbound `message:received` hook reply route omits `sessionKey`, so response-prefix and agent-scoped reply behavior may diverge from normal routed replies unless maintainers intentionally choose that contract.
- The proposed recursive `message:sent` hook reply delivery omits the originating `session`, so future non-text/media-capable hook replies could lose agent-scoped media local-root policy.
- The proposed `skipMessageHooks` suppression is not persisted by the current delivery queue shape; if an inbound hook reply is queued and replayed after failure/recovery, `message:sent` hooks could be re-enabled and cause unexpected follow-up messages or loops.
- #41375 has maintainer ownership and intentionally narrows the replyable-surface contract, so this PR needs consolidation rather than independent stale closure.
