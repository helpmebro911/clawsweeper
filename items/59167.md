## Summary

Keep open. Current main still unconditionally rejects trusted-proxy authentication from loopback sources after the trusted proxy address check, which matches the reported failure. The linked PR #59190 appears to be the active proposed fix, and the issue discussion includes unresolved deployment/security concerns, so this is not a conservative cleanup close.

## Evidence

- **Current main still rejects loopback trusted-proxy sources:** In authorizeTrustedProxy, after confirming the remote address is in trustedProxies, current main returns trusted_proxy_loopback_source whenever remoteAddr is loopback. That means a config including 127.0.0.1/::1 still cannot authenticate through the trusted-proxy path.
  - file: [src/gateway/auth.ts:353](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/src/gateway/auth.ts#L353)

- **Tests currently encode rejection behavior:** The gateway auth tests include coverage named rejects trusted-proxy identity headers from loopback sources and assert reason trusted_proxy_loopback_source for a request from 127.0.0.1 with trusted-proxy identity headers.
  - file: [src/gateway/auth.test.ts:3864](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/src/gateway/auth.test.ts#L3864)
