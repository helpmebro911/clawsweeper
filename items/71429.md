## Summary

Keep #71429 open. Current main has useful reload deferral, in-process reply tracking, and some Telegram fallback behavior, but the specific normal Telegram auto-reply path is still not durably journaled before Bot API delivery and sendChatAction/network failures are not converted into a hard, visible turn failure. The existing maintainer discussion explicitly reached the same boundary and left the issue open.

## Best Possible Solution

Keep #71429 open and add a focused Telegram regression that simulates a live auto-reply during channel hot reload with sendChatAction and final Bot API delivery failures. Then either route normal Telegram final replies through the existing durable outbound/session-delivery queue before Bot API send, or deliberately fail the turn with visible Telegram/UI status and retry guidance. The current reload deferral and outbound queue primitives are useful, but the normal Telegram same-surface reply path still needs explicit wiring and coverage.
