## Summary

Keep open. Current main still has related restart deferral and durable generic/restart-continuation queues, but ordinary Telegram same-surface auto-replies still go through the in-process reply dispatcher and direct Telegram Bot API delivery. The existing maintainer comments already identify this boundary, and the relevant files have not materially changed since that review.

## Best Possible Solution

Keep this issue open. Add a focused Telegram regression that simulates a live Telegram turn during hot reload with sendChatAction/network delivery failure, then either journal normal same-surface Telegram replies before Bot API delivery or fail the turn visibly in Telegram and UI with retry guidance.
