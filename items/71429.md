## Summary

Keep open. Current main has restart deferral and durable queues on the generic route-reply path, but the normal same-surface Telegram auto-reply path still goes through the buffered dispatcher and Telegram `deliverReplies`/Bot API delivery directly. I did not find durable journaling before final Telegram delivery or conversion of the reported `sendChatAction`/delivery failure into a visible hard turn failure. The maintainer discussion on #71429 also explicitly left this open for the same boundary.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram auto-reply/reload regression that simulates an in-flight turn, channel reload, `sendChatAction` failure, and final Bot API delivery failure. Then either route normal Telegram final replies through a durable outbound/session-delivery queue before Bot API delivery, or deliberately fail the turn with visible Telegram/UI status and retry guidance.

## Evidence
