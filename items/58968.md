## Summary

Keep open. Current `main` added timeout protection for initial `connectOverCDP` attempts, but the actual `listPagesViaPlaywright` enumeration path still has no timeout/circuit-breaker around CDP page inspection and still propagates Playwright failures instead of degrading, so the reported hang/failure mode is not clearly fixed on `main`.

## Evidence

- **Initial connect has timeouts, but only for connect setup:** `connectBrowser` now retries `getChromeWebSocketUrl` and `chromium.connectOverCDP` with 5-9s timeouts. That is a partial mitigation for connection establishment, not for later page-listing commands on an already cached/stuck Playwright browser.
  - file: [extensions/browser/src/browser/pw-session.ts:442](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/browser/src/browser/pw-session.ts#L442)

- **`pageTargetId` still has no timeout wrapper:** The page-listing path still calls `session.send("Target.getTargetInfo")` directly and only detaches in `finally`; there is no `Promise.race`, abort signal, or other timeout protection here.
  - file: [extensions/browser/src/browser/pw-session.ts:530](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/browser/src/browser/pw-session.ts#L530)
