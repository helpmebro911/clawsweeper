## Summary

Keep open. Current main still only supports exec hosts `auto`, `sandbox`, `gateway`, and `node`; it has no `host="cloud"`, no `CloudSandboxProvider`, and no cloud browser target. This PR is a core exec/plugin API proposal, not just a third-party plugin submission, because current plugin services are only generic background services and there is no existing core routing contract for exec/browser to call. The PR also has unresolved security and lifecycle review findings, so the right maintainer action is review/rework rather than conservative cleanup close.

## Best Possible Solution

Keep this PR open as a core exec/plugin API proposal. The best path is maintainer review plus a smaller reworked design that defines a documented cloud sandbox capability contract, keeps vendor implementation in plugins, routes exec/browser through shared helpers, preserves gateway/node approval and environment-safety semantics, and adds focused contract/runtime tests. If maintainers decide cloud sandbox routing should not be in core, close it explicitly as a product/API scope decision rather than as implemented, duplicate, stale, or ClawHub-only.

## Risks / Open Questions

- Do not land the current diff without resolving the approval/allowlist behavior for cloud execution; review comments indicate it can execute without the gateway/node approval checks that comparable host paths enforce.
- Do not forward sanitized inherited gateway process environment to a third-party cloud sandbox by default; review comments identify potential secret exposure through remote `env` output.
- Cloud background sessions need durable provider handles, scoped registry visibility, output cap accounting, timeout semantics, and retryable kill behavior before this can be safe for long-running cloud jobs.
- Cloud browser routing needs a bridge/control API design, not just passing a CDP endpoint into code paths that expect browser-control REST routes.
