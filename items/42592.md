## Summary

Current main still only partially hardens installer failures. The issue requests structured npm error classification, bounded retries for transient failures, clearer remediation, and replacing an explicit Docker config-get `|| true` masking pattern; those behaviors are still not fully implemented in the Unix or Windows installer paths.

## Best Possible Solution

Keep this issue open and implement it as a scoped installer reliability change: add shell-tested npm error-code extraction/classification, bounded retries for known transient network codes, immediate actionable remediation for non-retryable npm codes, replace the Docker config-get `|| true` with explicit optional-config handling, mirror reasonable behavior in `install.ps1`, and add focused tests for Docker config-get failure handling plus npm classification/retry behavior.
