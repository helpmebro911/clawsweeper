## Summary

Current main still does not implement the requested structured installer error handling and smart npm retry policy. The Unix installer has partial npm diagnostics and a Linux prefix fix, but no retryable/non-retryable npm classification table or gated retry behavior; Docker setup still masks the optional config read with `|| true`; Windows still reports a generic npm install failure.

## Best Possible Solution

Keep this issue open and implement a scoped installer reliability change: add tested npm error-code extraction/classification, retry only known transient npm/network failures, print actionable remediation for non-retryable npm codes, make Docker `gateway.controlUi.allowedOrigins` config-get failures explicit without breaking first-run defaults, mirror reasonable behavior in `scripts/install.ps1`, and add focused tests for Docker config-get failure handling plus Unix/Windows npm classification and retry behavior.

## Risks / Open Questions

- A smart retry wrapper should retry only known transient npm/network failures with a small bounded budget, or it could hide permanent installer failures and slow broken installs.
- Docker config-get cleanup needs to preserve quiet first-run behavior for a genuinely missing optional config key while surfacing unexpected CLI/container failures.
- Windows and Unix installers should stay behaviorally aligned enough that remediation messages and retry policy do not diverge by platform.
