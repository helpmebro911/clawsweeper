## Summary

Keep open. Current main still appears to preserve the reported failure mode: the configured reserveTokensFloor flows into memory-flush gating without a context-window-aware clamp, and a non-positive computed threshold returns null so token-based memory flush does not trigger. The related PRs #50694 and #51316 are useful active fix attempts, but the provided context says they are open/unmerged and the fix is not present in this checkout.

## Evidence

- **Token gate still disables flush for non-positive threshold:** resolveMemoryFlushGateState computes threshold as max(0, contextWindow - reserveTokens - softThreshold) and returns null when threshold <= 0. With reserveTokensFloor equal to contextWindowTokens and the default soft threshold, this path returns null before shouldRunMemoryFlush can trigger.
