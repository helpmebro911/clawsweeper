## Summary

Keep #71299 open. Current main has partial Telegram hardening: accepted polling update offsets are persisted before long-running handlers complete, and attempted Telegram API failures are logged. That addresses the restart/replay class discussed by the maintainer, but it does not prove the central reported symptom is fixed: assistant output visible in Web UI while Telegram receives no message and no send log appears. The only related PR that explicitly targeted comprehensive outbound sendMessage logging, #71377, is closed and unmerged.

## Best Possible Solution

Keep the issue open. Ask for retest on a build at or after v2026.4.24/main and logs showing whether `telegram sendMessage failed: ...` appears. If missing replies still have no send failure log, implement a focused Telegram outbound observability fix around final-reply dispatch decisions and sendMessage attempt/success/failure points, using #71377 as prior art but validating the actual no-attempt path before merging.
