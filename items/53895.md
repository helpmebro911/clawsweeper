## Summary

Keep open. Current main has a durable queue for generic direct outbound delivery and limited Telegram direct-send retries, but Telegram interactive final/block reply delivery still calls the Telegram bot API through the reply-delivery path and rethrows transient send failures. The supplied 2026.4.22 repro remains consistent with current source: the assistant answer can exist in OpenClaw while Telegram delivery is lost after `sendMessage` fails.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it into #50040. The fix should make Telegram interactive final/block reply delivery durable, either by routing it through the shared outbound delivery queue or by adding a Telegram-owned persistent reply queue that preserves chat/topic/thread/reply/quote/session metadata, classifies permanent 4xx/auth/routing errors separately from transient network/provider failures, retries with bounded backoff/TTL, drains on startup and Telegram transport recovery, and adds regression coverage for `deliverReplies`/dispatch after `sendMessage` fails once the assistant answer has already been produced.
