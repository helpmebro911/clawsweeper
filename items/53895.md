## Summary

Current main still does not make Telegram interactive final/block replies durable across transient sendMessage failures. The shared outbound delivery queue exists for callers routed through deliverOutboundPayloads, and Telegram send helpers have bounded retry logic, but the interactive Telegram reply dispatcher still sends through deliverReplies directly; deliverReplies emits failure hooks and rethrows without queueing the original reply for later drain. The recent 2026.4.22 reproduction in the issue discussion remains relevant.

## Best Possible Solution

Keep this open. Implement durable Telegram interactive final/block reply delivery by routing that dispatcher path through the shared outbound delivery queue or adding a Telegram-owned persistent reply queue that preserves chat, topic/thread, reply/quote, account, session, media, buttons, pinning, and visibility metadata; separates permanent 4xx/auth/routing failures from retryable and ambiguous network failures; retries with bounded backoff/TTL; drains on startup and Telegram transport recovery; and adds regression coverage for deliverReplies/dispatcher behavior when sendMessage fails after the assistant answer has been produced. If maintainers prefer one tracker, consolidate this into #50040 after copying over the 2026.4.22 reproduction details.
