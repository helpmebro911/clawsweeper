## Summary

Keep this PR open. Current main has strong config clobber guardrails for the severe wipe symptom, but it still does not implement the PR's central serialized/fresh-read ownerDisplaySecret persistence path, and the stale full-config auto-persist remains a plausible lost-update surface.

## Best Possible Solution

Keep this PR open for maintainer review, but require a current-main rebase or narrow rewrite. The best implementation is a per-config-path ownerDisplaySecret auto-persist transaction: serialize the relevant write, read the fresh source snapshot inside the lock with `readConfigFileSnapshotForWrite`, merge only the generated secret, preserve retry behavior, and keep current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration contracts. Add regression coverage proving concurrent startup writes survive while the generated owner secret is persisted.
