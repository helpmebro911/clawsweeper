## Summary

Keep this PR open. Current main has strong partial protection against catastrophic config clobbers, including destructive-write rejection and last-known-good recovery docs/tests, but it still does not implement the PR's core serialization/fresh-read ownerDisplaySecret transaction. The supplied PR discussion and review comments describe a concrete startup race, and the current code still persists a precomputed ownerDisplaySecret config object rather than rereading the latest source config inside a write lock.

## Best Possible Solution

Keep this PR open for maintainer review, but require a current-main rebase or rewrite. The best implementation is a narrow ownerDisplaySecret auto-persist transaction: read the fresh source snapshot inside a per-config write lock, merge only the generated secret, and add regression coverage proving concurrent startup writes such as gateway auth/bootstrap survive. Preserve current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration behavior.
