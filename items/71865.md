## Summary

Keep open. Current main still rejects OpenClaw-owned config writes solely when the rendered JSON shrinks below half of the previous raw file size, and the `models auth login` path still updates `openclaw.json` after writing the auth profile. A PowerShell/BOM/verbose-format file can therefore still trip the same guard unless the auth config write is changed to compare normalized semantics, preserve formatting, or explicitly exempt format-only shrinkage while retaining real clobber protection.

## Best Possible Solution

Keep the issue open and fix the config writer. Best path: add a semantic comparison before treating `size-drop:*` as blocking, likely by parsing the previous raw config and the newly rendered payload and allowing shrinkage when the normalized config only changes expected auth/default fields plus metadata. Add a regression test with a BOM and verbose/PowerShell-style `openclaw.json` that runs the same `models auth login` config-update path or a narrow equivalent using `applyAuthProfileConfig`, while preserving the existing destructive-shrink rejection test.
