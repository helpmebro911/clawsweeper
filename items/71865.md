## Summary

Keep open. Current main still blocks config writes solely on raw byte-size shrink after rewriting `openclaw.json` with 2-space JSON, so a PowerShell/BOM or verbose-format file can still trip the false-positive `size-drop` guard during `models auth login`. The command is less silent on current main because rejected writes throw `CONFIG_WRITE_REJECTED`, but the central formatting-only false-positive remains unhandled.

## Best Possible Solution

Keep the issue open. The best fix is to make config write safety distinguish semantic data loss from formatting-only rewrites, for example by comparing normalized parsed config shape or allowing BOM/indentation-only shrink after validation. Add a focused regression test with a UTF-8 BOM / verbose-indented `openclaw.json` and the `models auth login` updateConfig path, and ensure the CLI reports or avoids partial auth-profile/config persistence if the config write still fails.
