## Summary

Keep #72126 open. The linked issue #72120 is closed, but current main still has the same raw OpenAI Realtime API-key fallback that the PR is trying to fix: both browser session creation and the server-side bridge read `process.env.OPENAI_API_KEY` as a literal string, so `keychain:openclaw:OPENAI_API_KEY` can still be sent as the bearer key. The PR remains a focused, relevant bug fix, though the review comments about not caching failed Keychain lookups indefinitely should be addressed before landing.

## Best Possible Solution

Keep the PR open and finish it as the canonical fix for #72120. The right implementation belongs in the OpenAI realtime voice provider: resolve supported `keychain:<service>:<account>` values before use in both browser-session creation and bridge creation, avoid permanent caching of failed resolutions, and add regression tests for the env fallback path. A shared SecretRef resolver can be considered later, but current main still needs this concrete OpenAI realtime fix.
