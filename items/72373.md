## Summary

Keep open. Current main still sends OpenShell exec text through the SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight, and generic repeated same-tool/same-args loops remain warning-level until the later global no-progress breaker. The existing ClawSweeper comment remains consistent with current main, and the provided related items do not supersede this OpenShell/runtime hardening request.

## Best Possible Solution

Keep this issue open. The best maintainer path is a focused runtime hardening change: add a shared exec/OpenShell preflight before SSH command construction for obvious generated-command hazards and malformed shell structure, then update loop detection so repeated identical failed or no-progress generic calls can become a critical block at a configured threshold. Add regression coverage for OpenShell/shared SSH command construction and before-tool-call loop handling, and separately clean up or lint bundled help/skill examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- Command preflight must be narrow enough not to reject legitimate shell syntax such as redirection, comparisons, command substitution intentionally used by operators, or literal angle brackets in data.
- Hard-aborting repeated generic calls should distinguish failed/no-progress loops from legitimate repeated operations, likely using recorded tool outcomes and configurable thresholds.
