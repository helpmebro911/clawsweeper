## Summary

Keep open. Current main still forwards OpenShell exec text into the shared SSH `/bin/sh -c` wrapper without generated-command preflight, and repeated generic same-tool/same-args calls remain warning-level until the later global no-progress breaker. The provided prior ClawSweeper comment is consistent with current `main` at 8c18df02f3d01f36c1c94ae805724f963f4ac4b9, and the related title-search items do not cover this OpenShell/runtime safety request.

## Best Possible Solution

Keep this issue open. The best maintainer path is a focused runtime hardening change: add a shared exec/OpenShell preflight before SSH command construction for obvious generated-command hazards and malformed shell structure, then adjust loop detection so repeated identical failed/no-progress generic tool calls can become a clear critical block at a configured threshold. Add regression coverage for OpenShell/shared SSH command construction and before-tool-call loop handling, and separately rewrite or lint bundled help/skill examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- Command preflight can reject legitimate shell if the placeholder/syntax checks are too broad; the fix should be narrow and regression-tested.
- Hard-aborting repeated generic calls can break valid repeated workflows unless it is tied to failed or no-progress outcomes.
