## Summary

Current main still appears to preserve the reported post-compaction timeout messaging bug. Successful compaction can retry the prompt, later timeout can enter model fallback, and mixed fallback exhaustion still collapses to the generic external `/new` fallback when verbose failure details are off.

## Best Possible Solution

Keep this open. The best fix is to carry structured compaction-attempt outcome state from `runEmbeddedPiAgent` through model fallback into `runAgentTurnWithFallback`, then format mixed fallback exhaustion after successful compaction as a specific external-chat recovery: compaction succeeded, the retried model call hit the model idle timeout, and fallback candidates were unavailable or exhausted. Add a regression for successful overflow compaction followed by timeout plus unavailable fallback candidates with verbose failure details off.
