## Summary

Keep open. Current main improves restart UX and has restart-sentinel continuation for explicit restart flows, but it still rejects new command enqueues while the gateway is draining, so the original inbound message is not queued or replayed automatically after restart.

## Best Possible Solution

Keep this issue open and implement the remaining core gateway behavior: allow drain-window inbound command tasks to enter a preserved per-session/per-channel FIFO lane without starting until `resetAllLanes()`, or add a durable pending inbound queue if full stop/start replay is in scope. Add regression coverage for multiple messages arriving during drain, order preservation, SIGUSR1 replay, and any durable restart behavior selected.
