## Summary

Keep open. Current main has friendlier restart wording and some restart recovery paths, but the core requested behavior is still missing: new inbound command work is rejected while the gateway is draining, before it can enter the preserved queue and be replayed after restart.

## Best Possible Solution

Keep this issue open. The best fix belongs in core gateway/command queue behavior: allow drain-window inbound command tasks to enter a preserved FIFO lane without starting until `resetAllLanes()`, or introduce a durable pending inbound queue if full stop/start replay is in scope. Add regression coverage for multiple messages during drain, per-channel/session order preservation, SIGUSR1 replay, and any selected durable-restart path. Keep the friendlier restart copy as the fallback UX.

## Risks / Open Questions

- Closing now would conflate the shipped friendlier restart message and explicit restart-continuation paths with the still-missing automatic replay of messages rejected while `gatewayDraining` is true.
