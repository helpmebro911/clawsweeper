## Summary

Current main still has the reported UTF-8 boundary bug in the child-process adapter. The PR is not implemented on main, the failure mode is technically valid, and there is no repo evidence that the report is obsolete or incoherent.

## Evidence

- **Current main still decodes each stdio chunk independently:** `createChildAdapter` still calls `chunk.toString()` separately for each `data` event on both stdout and stderr, with no boundary-safe decoder or end-of-stream flush.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/process/supervisor/adapters/child.ts#L111)
  - command: `nl -ba src/process/supervisor/adapters/child.ts | sed -n '100,140p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Corrupted chunks flow straight into captured and user-visible output:** The supervisor appends each adapter chunk directly into `stdout`/`stderr` capture buffers and forwards it through `input.onStdout`/`input.onStderr`, so any bad decode at the adapter layer propagates unchanged.
  - file: [src/process/supervisor/supervisor.ts:186](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/process/supervisor/supervisor.ts#L186)
  - command: `nl -ba src/process/supervisor/supervisor.ts | sed -n '176,206p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Existing tests do not cover UTF-8 chunk-boundary decoding:** `child.test.ts` exercises kill/wait/platform behavior, but there is no test around split multibyte stdout/stderr data or decoder flush behavior, so current tests do not disprove the report.
  - file: [src/process/supervisor/adapters/child.test.ts:69](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/process/supervisor/adapters/child.test.ts#L69)
  - command: `nl -ba src/process/supervisor/adapters/child.test.ts | sed -n '1,240p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Reasonable reproduction of the exact decoding failure still holds:** A local Node check with split UTF-8 bytes reproduced the underlying failure: `Buffer.from([0xE4,0xBD]).toString()` plus `Buffer.from([0xA0]).toString()` produced replacement characters, while `StringDecoder('utf8')` across the same chunks produced `你`. That matches the report's root cause.
  - command: `node -e 'const chunks=[Buffer.from([0xE4,0xBD]),Buffer.from([0xA0])]; console.log(chunks.map(c=>c.toString()).join("|"));' && node -e 'const {StringDecoder}=require("node:string_decoder"); const d=new StringDecoder("utf8"); const chunks=[Buffer.from([0xE4,0xBD]),Buffer.from([0xA0])]; console.log(chunks.map(c=>d.write(c)).join("|")+"|"+d.end());'`
