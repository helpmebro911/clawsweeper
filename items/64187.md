## Summary

Keep this issue open. Current main still has the reported direct rename-based SQLite atomic reindex swap in memory-core, with no retry/backoff or copy fallback for Windows EBUSY/EPERM/EACCES lock failures, and the provided GitHub context shows open PR #71611 as an unmerged implementation candidate with closing syntax.

## Best Possible Solution

Keep this issue open and treat #71611 as the current implementation candidate. The best path is to review or revise that PR so memory-core's atomic reindex move gets bounded retry/backoff for Windows lock errors, an appropriate fallback where safe, and regression coverage for EBUSY/EPERM/EACCES behavior before merge.
