## Summary

Keep this PR open. Current main still has the unrestricted `tools.fs.workspaceOnly` default and no agent filesystem sensitive-path denylist, so the security hardening is neither implemented nor obsolete. The PR is also not merge-ready as submitted because its denylist does not cover the normal host read tool path and uses lexical path checks that review comments correctly identified as symlink-bypassable.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned superseding hardening patch. The best implementation path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, security docs, changelog, and regression tests together. If a sensitive-path denylist is retained, enforce it consistently for read/write/edit and use symlink-safe realpath checks.

## Risks / Open Questions

- Merging this PR as-is could create false assurance because the primary credential exfiltration path, host `read` with `tools.fs.workspaceOnly=false`, would remain unguarded.
- The proposed denylist uses lexical path resolution rather than symlink-safe realpath checks, so a symlink can bypass the intended sensitive-path block.
- Changing the filesystem default from unrestricted to workspace-only is a compatibility and security-policy change that needs aligned config schema/help, docs, tests, and maintainer review.
- Security review found no unrelated package, lockfile, workflow, release script, or dependency-source changes in the provided PR file list; the observable risk is incomplete filesystem-policy hardening rather than supply-chain expansion.
