## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to unrestricted behavior and has no agent filesystem sensitive-path denylist, while the PR remains a plausible security-hardening candidate that needs changes before merge.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned superseding patch. The best path is to decide the default filesystem policy explicitly, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, security docs, and regression tests together. If a sensitive-path denylist is kept, enforce it consistently for read/write/edit with symlink-safe realpath handling and coordinate with the related open proposal #58947.

## Risks / Open Questions

- Merging the PR as submitted would not fully deliver its stated sensitive-path protection because the host read path remains the primary exfiltration surface when `workspaceOnly=false`.
- The PR's proposed denylist uses lexical path matching; review comments identify a symlink bypass that should be addressed with realpath-aware enforcement before relying on it for credential protection.
- Changing `tools.fs.workspaceOnly` from false to true is a compatibility and security-policy change that needs aligned config schema/help, docs, tests, release notes, and maintainer judgment.
