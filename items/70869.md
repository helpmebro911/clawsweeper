## Summary

Keep open. Current main mitigates the provider-replay failure by filtering OpenClaw `delivery-mirror` assistant rows before prompt submission, but the reported core behavior remains: outbound delivery still persists `role: assistant`, `model: delivery-mirror` rows into the resolved session with no config toggle and no guard against appending after a different assistant turn. Related #70870 is separate LCM/plugin repair scope, #72141 covers a narrower isolated-cron stale-state leak that was closed as implemented, and #63779 remains adjacent but does not supersede this mirror-write issue.

## Best Possible Solution

Keep the issue open. Add a focused core regression for the message-tool/outbound-route path where the target session already ends in an assistant turn, then either add an explicit delivery-mirror policy toggle or skip/convert mirror writes when they would append an unpaired transcript-only assistant row. Coverage should include direct delivery, plugin-handled sends, queued delivery replay, isolated cron/subagent contexts, and Anthropic replay input so provider-side filtering is not the only protection.
