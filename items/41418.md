## Summary

Current main does not implement a global `openclaw agent --dry-run` mode. The agent CLI lacks the option, the CLI option type and gateway/local call path do not carry `dryRun`, and the universal Pi tool adapter still calls `tool.execute(...)` after `before_tool_call` handling. Existing dry-run support remains scoped to other commands or individual tools, and the current plugin hook surface can block/rewrite/require approval but cannot provide the requested structured dry-run tool result. This is actionable core agent execution work, not implemented, superseded, or a ClawHub-only item.

## Best Possible Solution

Keep this issue open as a core agent execution feature request. If maintainers want it, implement a request-scoped CLI-only `openclaw agent --dry-run` path that threads a `dryRun` flag through the agent CLI and embedded attempt context, intercepts both OpenClaw-managed and client tool definitions after existing policy hooks, returns structured preview results to the model, logs sanitized previews to stderr, updates `docs/cli/agent.md`, and adds focused adapter plus CLI registration tests. Keep #13362 and #13617 as separate scoped dry-run tracks.

## Risks / Open Questions

- A global process-level flag would be risky for gateway or concurrent runs; the best implementation should use request-local/CLI-only state unless gateway support is explicitly designed.
- Dry-run logging must sanitize or redact tool params before writing previews to stderr, and it should preserve existing `before_tool_call` blocks and approvals before returning preview payloads.
