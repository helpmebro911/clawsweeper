## Summary

Current main still does not carry request-origin session/lane metadata through the exhausted model-fallback wrapper. Individual fallback attempts retain provider/model details, but FallbackSummaryError, runWithModelFallback, logModelFallbackDecision, and the agent-command call site still have no structured sessionId/sessionKey/lane/session_id path, so this remains a real observability gap rather than a stale or already-implemented issue.

## Best Possible Solution

Keep this issue open and implement a narrow observability fix in the model fallback path: add normalized request-origin metadata such as sessionId/sessionKey or lane to runWithModelFallback, carry it through FallbackSummaryError and logModelFallbackDecision, include attempted providers/models on the exhausted wrapper, wire the existing agent-command/gateway session data into the fallback call, and add focused tests for decision JSON logs plus terminal exhausted-fallback handling.
