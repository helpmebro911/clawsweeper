## Summary

Keep open. Current main has WhatsApp watchdog and stale-status plumbing, but it still does not solve the reported 10-17 minute stale message-flow loss window under default production configuration. Transport activity can keep the socket looking alive while the app-silence detector waits much longer, and related open maintainer PR #72621 explicitly says this related report should remain open unless its exact symptoms are proven fixed.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale message-flow/message-loss bug. The best maintainer path is to add a configurable app-silence or message-flow stale threshold separate from transport activity, wire it through config types/schema/docs, and add a regression test where transport frames continue while inbound app messages stop within the reported 10-17 minute failure window. Coordinate with #63939 and the narrower open #72621 listener/auth-state work, but do not close this until the exact stale inbound delivery symptoms are proven fixed.
