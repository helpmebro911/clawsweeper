## Summary

Keep open. Current main still has the gateway stop-timeout stale-store behavior this PR targets, and the PR is paired with open same-author bug #70024. The PR is narrow and relevant, but the current head is incomplete because clearing `store.tasks` on timeout allows a replacement task while the old task's unguarded lifecycle callbacks can later mutate the replacement runtime state.

## Best Possible Solution

Keep the PR open and ask for a revision that preserves the stop-timeout recovery goal while guarding task-owned status writes, error writes, final stop writes, and restart bookkeeping by current task identity. Pull in the useful approach from the closed-unmerged #71456 fixup or land an equivalent maintainer patch, and add a regression test where a timed-out task eventually settles after a replacement start without clobbering the replacement state.
