## Summary

Keep PR #70056 open. Current main still has the stop-timeout bug the PR is trying to fix, and the PR is a focused bug fix tied to open issue #70024, but the provided review comments identify a real race in the proposed cleanup that should be addressed before merge rather than auto-closing the PR.

## Best Possible Solution

Keep the PR open and ask for a revised gateway lifecycle fix: the timeout path should stop lying about running state and avoid permanent stale store entries, while also preventing the old trackedPromise from overwriting or restarting over a newer task. A good revision would add an identity guard around the running=false/lastStopAt finally path or otherwise keep the old task reserved until it settles, plus a regression test where the timed-out task eventually settles after a restart attempt.
