## Summary

Keep open. Current `main` still has the reported bug: `sessions_history` accepts any positive numeric `limit`, but gateway `chat.history` validates `limit <= 1000` before its internal clamp runs, so a tool call with `limit: 2000` can still fail with `INVALID_REQUEST`. The PR addresses a real gap on `main`, although the review discussion is also right that it should add a direct execute-level regression test for the clamp path.

## Evidence

- **Tool still forwards over-limit values on main:** `sessions_history` declares `limit` with only `minimum: 1` and computes it with `Math.max(1, Math.floor(params.limit))`, so values above 1000 are still accepted and forwarded unchanged.
