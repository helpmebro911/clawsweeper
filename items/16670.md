## Summary

Keep #16670 open. Current main has memory-search config, docs, provider registration, diagnostics, and `openclaw memory status --deep`, but onboarding/configure still lacks the requested dedicated memory embedding setup step for provider/model/API key or SecretRef plus verification. The fresh issue discussion also includes a maintainer/Codex keep-open signal for this exact gap.

## Best Possible Solution

Keep #16670 open until OpenClaw adds a dedicated memory/embedding setup path or a maintainer explicitly decides it should not be mandatory. The likely implementation belongs in onboarding/configure: prompt for provider, model, and API key or SecretRef under `agents.defaults.memorySearch`, use the existing embedding provider registry/config schema, verify readiness with a small embedding probe or `openclaw memory status --deep`, and update public docs for API-key, Codex OAuth, and daemon environment requirements.

- A fix should account for daemon/LaunchAgent environment visibility and SecretRef-backed credentials, not only the interactive shell environment.
