## Summary

Keep #40194 open. Current main propagates Feishu accountId through startup and inbound routing, but the shared group/channel session-key builder still omits account/app identity for non-DM peers, so multiple Feishu bot accounts in the same group can still collapse to the same same-agent group session unless they are routed to different agent ids.

## Best Possible Solution

Implement account-aware Feishu group session isolation, preferably as a Feishu-specific or generic opt-in account-scoped group mode. Use normalized accountId rather than raw app credentials in the key, preserve legacy session continuity where possible, document the behavior, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct AccountId/reply account.

## Risks / Open Questions

- Account-specific bindings may work around deployments that route each Feishu account to a different agent, but they do not solve same-agent session isolation for multiple bot accounts in one group.
- Changing the group session-key shape globally could break existing group session continuity for other channels, so a Feishu-specific or opt-in migration path needs maintainer care.
