## Summary

Current main still does not isolate Feishu group sessions by Feishu account/app when multiple accounts in the same group route to the same agent. Account identity is propagated into routing and reply dispatch, but the non-DM session-key builder still drops accountId, and Feishu group session scopes only vary by group, sender, and topic.

## Best Possible Solution

Keep this issue open and implement account-aware Feishu group session isolation. Prefer a Feishu-specific or generic opt-in account-scoped group mode using normalized accountId rather than raw app credentials, preserve legacy session continuity where feasible, document the behavior in the Feishu channel docs, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.

## Risks / Open Questions

- Changing the global group session-key shape would affect session continuity for other channels, so the fix should likely be Feishu-specific or an explicit opt-in account-scoped group mode with migration/backcompat handling.
- Account-specific route bindings can avoid the collision only when each Feishu account routes to a different agent; they do not solve same-agent same-group transcript isolation.
