## Summary

Current main still does not fully solve the reported Feishu multi-bot group identity collision. Feishu inbound has the account identity and passes it into routing, but the shared session-key builder only uses accountId for direct-message `per-account-channel-peer` keys; group keys remain `agent:<agent>:feishu:group:<peerId>` and can still collide across Feishu accounts when multiple bots in the same group route to the same agent/group peer.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation, preferably through a Feishu-specific or generic opt-in account-scoped group mode using normalized `accountId`, not raw app credentials. Preserve legacy session continuity where feasible, update Feishu/session docs and config metadata, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.

## Risks / Open Questions

- A global change to group/channel session-key shape would affect existing transcript continuity across channels; the safer direction is likely Feishu-specific or an explicit opt-in account-scoped group mode with compatibility handling.
- Current mention-gating and account-aware routing can prevent some duplicate handling and can route different accounts to different agents, but they do not isolate transcripts for the same agent, same Feishu group, and different Feishu bot accounts.
