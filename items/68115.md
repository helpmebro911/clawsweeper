## Summary

Keep open. Current main still lacks a first-class global setup/onboarding hook for credentialed tool-only plugins. Existing setup discovery and runtime registration are provider/CLI-backend shaped, while the generic plugin-config wizard only partially overlaps and explicitly skips sensitive fields, so it does not solve API-key onboarding for a newly installed tool plugin.

## Best Possible Solution

Keep this open as a plugin-SDK/core setup design item. The best path is an additive manifest-first `setup.toolPlugins` descriptor plus a setup-only API/helper that can prompt for SecretInput-backed credentials, persist through `plugins.entries.<pluginId>.config` or another documented plugin-owned path, and add contract tests and public docs alongside the existing provider and channel setup surfaces.

## Risks / Open Questions

- The eventual implementation should stay descriptor-first so global setup does not execute every installed plugin setup module just to discover credentialed tool plugins.
- Credential collection needs explicit SecretInput-safe design: masked/ref prompting, no plaintext leakage, and writes to documented plugin-owned config paths rather than provider auth profiles.
- The existing generic plugin-config wizard overlaps enough that maintainers should decide whether to extend it or add a separate `setup.toolPlugins` step, but current behavior is not sufficient to close this request.
