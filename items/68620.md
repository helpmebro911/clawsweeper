## Summary

Keep #68620 open. Current main has detection and some timeout mitigations, but it still does not implement the requested stuck-session auto-kill/recovery path: the diagnostics heartbeat only emits `session.stuck`, the config surface only exposes a warning threshold, and normal agent turns still default to a 48-hour run timeout. The recent comment is credible and broadens the impact to Telegram plus cron; the related DeepSeek issue is a separate provider-history replay bug, not a canonical replacement for this session-liveness problem.

## Best Possible Solution

Keep this issue as the active user-facing tracker for stuck-session auto-recovery. The best fix is a bounded runtime watchdog for channel/cron agent turns that tracks active runs by session key, aborts the embedded run through existing abort handles after a configurable threshold, records a diagnostic recovery event, marks the session idle, and lets the existing model-fallback/error-delivery path proceed. Add fake-timer tests for a never-resolving model call and queue-drain behavior for Telegram/direct and cron-triggered sessions, plus docs for the new config key.
