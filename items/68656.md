## Summary

Keep open. Current main has better startup diagnostics and cold plugin-registry planning, but the reported Windows gap still maps to synchronous gateway plugin bootstrap before readiness and before the first runtime slash-command registration. The recent related Windows/Discord fix and the broader #70728 report are adjacent, not a clean implementation or superseding item for this plugin-registration stall.

## Best Possible Solution

Keep the issue open and use the startup trace/benchmarking path to capture Windows timings for `plugins.bootstrap`, then either make that phase visibly logged by default or reduce it by deferring nonessential default sidecar command registration/module imports until after listen. If a narrower fix lands, close this with source, test, and release evidence tied to the Windows plugin-bootstrap gap.
