## Summary

Current main still leaves #72345 actionable. The voice-call CLI still constructs a process-local runtime for call/control commands, and runtime construction still starts a webhook listener on the configured bind/port. The existing gateway RPC methods are present, but the CLI is not wired to use the shared gateway RPC client or any `--use-gateway`/remote option, so a separate CLI process can still collide with the gateway-owned webhook port.

## Best Possible Solution

Keep #72345 open and implement a gateway-aware voicecall CLI path. Route `call`/`start`/`continue`/`speak`/`dtmf`/`end`/`status` through the existing `voicecall.*` gateway RPC methods when a running gateway is selected or explicitly requested, preserve standalone local-runtime behavior where appropriate, and add regression coverage proving delegated CLI commands do not call `createVoiceCallRuntime()` or start a webhook listener. At minimum, catch webhook `EADDRINUSE` and emit a targeted message explaining that the gateway voice-call runtime already owns the configured port.
