## Summary

Keep open. Current main still wires `openclaw voicecall call` and related CLI subcommands to a local voice-call runtime, and that runtime still starts a webhook server on the configured serve bind/port, defaulting to `127.0.0.1:3334`. The gateway-side `voicecall.*` RPC methods exist, but the CLI does not delegate to them or expose a gateway/remote option, so the reported EADDRINUSE path remains plausible on current main.

## Best Possible Solution

Keep the issue open and implement a gateway-aware voicecall CLI path: route `call`/`start`/`continue`/`speak`/`dtmf`/`end`/`status` through the existing `voicecall.*` gateway RPC when a running gateway is selected or explicitly requested, preserve a standalone local-runtime mode where appropriate, and add a targeted regression test proving the CLI does not call `createVoiceCallRuntime()` when delegating to the gateway. At minimum, catch `EADDRINUSE` from webhook startup and surface a specific message that the gateway voice-call webhook is already bound on the configured port.

## Risks / Open Questions

- Fixing this by always routing through the gateway could regress standalone `voicecall` CLI use when no gateway is running; the implementation should either auto-detect safely or add an explicit gateway delegation mode with clear fallback behavior.
- The gateway RPC methods already exist, but CLI auth/target selection and error messages need to align with existing `openclaw gateway call` behavior rather than inventing a voice-call-only transport.
