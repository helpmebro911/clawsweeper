## Summary

Keep open. Current main still does not implement the requested behavior: context overflow is detected and handled through compaction/truncation plus a local context_overflow error, while model fallback explicitly bypasses context-overflow errors. The provided related issue context also does not provide a canonical superseding item: #28632 is output-token length, #34005 is graceful Ollama overflow handling, and #29014 is broader overloaded/retry fallback work.

## Best Possible Solution

Leave #9986 open until maintainers either decide the current documented policy is intentional or implement a guarded context-overflow fallback path. A concrete implementation would likely live around the embedded runner's exhausted overflow branch and the model-fallback policy: after compaction/truncation is exhausted, only replay when safe, choose configured fallbacks with sufficient context headroom, log the transition, and update docs/tests for primary smaller-context to fallback larger-context behavior.

## Risks / Open Questions

- Naively falling back after context overflow could replay side-effecting turns or completed tool work; an implementation needs replay-safety and side-effect gating.
- Selecting a larger fallback needs reliable context-window metadata and tests for cases where fallback models are equal or smaller.
