## Summary

Keep #9986 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still treats context overflow as compaction/tool-result-truncation recovery on the active model, and explicitly keeps likely context-overflow errors out of the configured model fallback chain. Related closed issues cover output-token freezes, overloaded/server errors, or graceful overflow messaging, not the requested smaller-primary to larger-fallback routing.

## Best Possible Solution

Keep #9986 open until maintainers either explicitly document a compaction-only policy for context overflow, or implement a guarded fallback path after compaction and oversized-tool-result truncation are exhausted. The likely code path is the embedded runner plus model-fallback contract: classify exhausted context_overflow as eligible only when a configured fallback has known larger context headroom, log the switch, preserve replay safety, and add regression coverage for smaller-primary to larger-fallback behavior.
