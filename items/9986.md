## Summary

Keep #9986 open. Current main still treats context overflow as a compaction/retry problem and explicitly excludes it from configured model fallback. When overflow recovery is exhausted, the runner returns a blocked `context_overflow` payload telling the user to reset or choose a larger-context model, rather than selecting a configured fallback with more headroom.

## Best Possible Solution

Keep #9986 open until maintainers either explicitly choose and document a compaction-only policy for context overflow, or implement a guarded fallback path that runs only after compaction and oversized-tool-result truncation are exhausted. The implementation should select configured fallback candidates with known larger context headroom, log the switch, preserve replay safety, and add regression coverage for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- A context-overflow fallback path needs replay-safety gating so a retried turn cannot duplicate tool side effects or outbound channel messages.
- Selecting a larger fallback requires trustworthy context-window metadata for the active model and configured fallback candidates.
- Current docs intentionally describe compaction-only handling, so implementing this feature would need docs updates as well as tests.
