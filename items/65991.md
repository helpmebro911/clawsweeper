## Summary

Current main still does not accept, persist, document, or propagate a `silent` option on cron `failureAlert`. Telegram and the generic outbound/message path already support silent delivery, so this remains a narrow core cron failure-alert integration gap rather than a ClawHub-only request or duplicate cleanup candidate.

## Best Possible Solution

Implement a narrow additive cron failure-alert contract change on main: add `silent?: boolean` to per-job `CronFailureAlert` and global `cron.failureAlert` config, update TypeBox/Zod/generated config docs, CLI and Control UI forms, then thread it through `resolveFailureAlert`, `emitFailureAlert`, `sendCronFailureAlert`, `sendGatewayCronFailureAlert`, `sendFailureNotificationAnnounce`/`sendCronAnnouncePayloadStrict`, and `deliverOutboundPayloads({ silent })`. Add focused regression coverage proving Telegram cron failure alerts set `disable_notification` when `silent: true` while preserving default behavior when omitted.
