## Summary

Keep open. Current main still allows audio auto-detection to fall through to the generic Gemini CLI `read_many_files` entry, and the public audio/media-understanding docs still document Gemini CLI in the default audio auto-detect order. The linked issue #46588 was closed as superseded by this PR plus #46601, not because the defensive skip is implemented. #46601/#46607 track the longer-term real inline-audio path, but they are adjacent/open follow-ups rather than shipped replacements for this narrow defensive PR.

## Best Possible Solution

Keep #46593 open or ask the author/maintainer to rebase it. The best code path is still the narrow defensive change: make `resolveGeminiCliEntry` return `null` for `capability === "audio"`, keep Gemini CLI auto-detect for image/video, update the audio/media-understanding docs, and add regression coverage proving audio does not auto-select Gemini CLI. Keep #46601/#46607 as the separate long-term track for real binary inline audio attachment in Gemini headless flows.
