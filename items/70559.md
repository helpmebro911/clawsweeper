## Summary

Current main still appears to contain the reported failure path. `runUnsafeReindex()` calls `resetIndex()`, `resetIndex()` drops `chunks_vec` and clears only `vector.dims`, while the subsequent sync setup still prepares `DELETE FROM chunks_vec ...` whenever `vector.enabled && vector.available` is true. The report is recent, concrete, and not protected or maintainer-authored, but the code evidence supports leaving it open rather than closing as fixed or stale.

## Best Possible Solution

Keep this issue open and fix the memory-core reindex path. The narrowest code direction is to make `resetIndex()` preserve and clear `chunks_vec` when sqlite-vec is enabled and available, or otherwise make the subsequent sync setup robust when the vector table has been dropped. Add a regression test that exercises `OPENCLAW_TEST_FAST=1` plus `OPENCLAW_TEST_MEMORY_UNSAFE_REINDEX=1` with vector availability true and confirms a full reindex does not prepare statements against a missing `chunks_vec` table.
