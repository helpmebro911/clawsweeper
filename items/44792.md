## Summary

Current `main` already threads Feishu inbound media through the actual saved file path, preserves inbound filename metadata when saving, and has regression coverage for that path. The released `v2026.4.22` tag also contains the same implementation.

Closing as `implemented_on_main` after a Codex review. Current OpenClaw no longer matches the reported Feishu path-mismatch behavior: `extensions/feishu/src/bot-content.ts:427` downloads the resource, saves it via `saveMediaBuffer(..., result.fileName || mediaKeys.fileName)`, and stores `saved.path`; `src/plugin-sdk/agent-media-payload.ts:13` forwards that same saved path into `MediaPath`/`MediaPaths`; and `extensions/feishu/src/bot.ts:1015` passes the media payload straight into the inbound agent context. The media store also preserves `originalFilename` metadata in `src/media/store.ts:407`, with regression coverage in `extensions/feishu/src/bot.test.ts:1441` and `src/media/store.test.ts:637`. I also verified that the released `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains the relevant Feishu/media-store implementation. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with the exact `[feishu]` log lines from download through dispatch and whether multi-channel sync is enabled.
