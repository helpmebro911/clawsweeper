## Summary

Keep #71696 open. Current main still lacks Telegram chat-type-based reply mode, still resolves and passes a single Telegram replyToMode through message and native-command delivery, still drops explicit/current reply threading in the relevant Telegram delivery path, and still does not expose provider message ids in plugin command context. The PR is recent and appears to supersede the earlier closed, unmerged #70579 rather than being obsolete.

## Best Possible Solution

Keep #71696 open for maintainer review. The best path is to rebase it onto current main, preserve runtime replyToMode fallback while adding Telegram replyToModeByChatType to schema/types/generated metadata/docs, resolve reply mode per direct/group for message and native-command delivery, pass currentMessageId through outbound planning/delivery, expose Telegram message ids to plugin command handlers, and run the changed Telegram/config/plugin/outbound checks.
