## Summary

Keep open. Current main partially improves the symptom by throwing a typed session write-lock timeout error, but the reported `openclaw agent --session-id ... --timeout 30 --json` path still does not pass the resolved run timeout into the session write lock. The remaining work is directly tracked by open PR #69159, which proposes passing `params.timeoutMs` to `acquireSessionWriteLock`; the issue should stay open until that fix or an equivalent is landed.

## Best Possible Solution

Keep #69112 open until PR #69159 is refreshed and merged or an equivalent fix lands. The narrow product/code path is to pass the resolved agent run timeout into `acquireSessionWriteLock`, preserve the typed lock-timeout classification, and add a regression test proving concurrent `--session-id` callers receive a distinguishable busy/lock-timeout result instead of silently timing out.
