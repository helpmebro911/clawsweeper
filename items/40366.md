## Summary

Current `main` at `9f4ef6162dbde384797abc002cd4241080eb65e3` does not implement this PRâ€™s behavior. The agent command path still normalizes payloads and then gates the no-reply case on raw `payloads.length`, so normalized-empty outputs can still fall through to an empty outbound send with no fallback. The Slack monitor path still clears preview drafts and returns when preview/progress happened but no final/block reply was delivered; it does not send the proposed keepalive/status fallbacks.

## Evidence

- **Agent delivery still uses raw payload emptiness, not normalized deliverable emptiness:** `deliverAgentCommandResult` builds `outboundPayloadPlan`/`normalizedPayloads`, but then only treats the turn as no-reply when `!payloads || payloads.length === 0`. A whitespace-only or directive-stripped payload can therefore survive this check even when normalization produces nothing deliverable.

## Risks / Open Questions

- Normalized-empty agent outputs can still produce silent no-op delivery instead of a user-visible fallback.
- Slack turns that emit preview/progress but end without a final reply can still look like ghost interactions because the draft is cleared and the handler returns quietly.
- Current main lacks the regression tests described in the PR for normalized-empty fallback and Slack missing-final fallback behavior.
