## Summary

Keep open. Current main still implements only a timeout-based TUI streaming watchdog, not the requested prompt transition after visible response content finishes. Chat deltas still mark the TUI as `streaming`, normal idle still depends on `final`/lifecycle terminal handling, and gateway chat final remains lifecycle-end driven after async flush work. The issue discussion also includes actionable named-worker reproduction context that should inform a real fix.

## Best Possible Solution

Keep this issue open as the TUI UX tracker. The best fix is to decouple visible content completion from full run lifecycle completion, either by adding an additive gateway/TUI content-complete or finalizing signal, or by moving the TUI from `streaming` to a distinct non-streaming finalizing/idle state after visible deltas end while preserving active-run ownership. Add regression coverage for delayed lifecycle final events, concurrent runs, hidden/tool-only deltas, and the named worker-agent path from the discussion.

## Risks / Open Questions

- The existing watchdog reduces indefinite stuck-status impact, so the eventual fix should preserve it as recovery while avoiding a misleading immediate idle state if tools, delivery, or persistence are still legitimately active.
- A TUI-only silence heuristic could regress concurrent-run or hidden/tool-only output behavior; a backend content-complete/finalizing signal or carefully tested state-machine transition would be safer.
