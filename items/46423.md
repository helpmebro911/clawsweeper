## Summary

Keep open. Current main still does not implement PR #46423's central behavior: transient retry is only wired for one-shot `at` jobs, while recurring `cron`/`every` errors still schedule the later of the natural next run and the generic backoff floor. The PR is useful but still needs maintainer review and cleanup because the discussion contains substantive unresolved retry-contract questions and the branch picked up unrelated CI/main-sync edits.

## Best Possible Solution

Keep the PR open for maintainer handling, but ask for a focused cron-only revision or consolidation with the open recurring-retry tracking items. The best path is to implement recurring transient retries before the next natural schedule, preserve non-transient scheduling and forced-run cadence semantics, define the config contract explicitly, keep startup catch-up/backoff behavior consistent, clamp unsafe zero-delay backoffs, and add focused regression coverage for cron, every, timeout, custom backoff, and exhausted retry cases.
