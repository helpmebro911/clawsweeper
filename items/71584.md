## Summary

Keep #71584 open. Current main still has the reported local validation gap for `ollama/*`: `max` normalizes as a thinking level, but the Ollama provider does not publish a provider thinking profile containing `max`, so explicit `/think max` falls back to the base `off, minimal, low, medium, high` profile and is rejected before the request reaches Ollama. The native Ollama wrapper also only sends top-level `think: true/false`, not `options.reasoning_effort: "max"`.

## Best Possible Solution

Keep the issue open and fix it in the Ollama provider boundary. The likely maintainer path is to add an Ollama-owned `resolveThinkingProfile` for models/providers that support `max` (or a generic profile bridge from configured `compat.supportedReasoningEfforts`), then add focused tests that `ollama/deepseek-v4-flash:cloud` lists/accepts `/think max` and that the outgoing Ollama payload maps it to the provider-supported highest reasoning effort without regressing existing native `think: true/false` behavior.
