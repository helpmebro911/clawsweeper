## Summary

Keep open. Current `main` does not already implement the requested human-readable cron timestamp fields, and the proposed PR still has a credible runtime-safety gap: its new formatter would throw on oversized but finite epoch values that current cron parsing accepts.

## Evidence

- **Current main still logs only raw epoch fields:** `applyJobResult` logs `nextRunAtMs` only for retry/backoff, and `armTimer` logs `nextAt` only; there is no human-readable companion field on `main`.
  - file: [src/cron/service/timer.ts:481](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/cron/service/timer.ts#L481)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '470,560p'; nl -ba src/cron/service/timer.ts | sed -n '628,690p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Current main event type also lacks the added human-readable field:** `CronEvent` on `main` exposes `nextRunAtMs?: number` but no `nextRunAt?: string`, so the PR is not redundant with current code.
  - file: [src/cron/service/state.ts:16](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/cron/service/state.ts#L16)
  - command: `nl -ba src/cron/service/state.ts | sed -n '1,60p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Current cron parsing accepts oversized finite numeric timestamps:** `parseAbsoluteTimeMs` accepts any digits-only string that converts to a finite positive number and returns `Math.floor(n)`, with no ECMAScript Date range guard.
  - file: [src/cron/parse.ts:18](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/cron/parse.ts#L18)
  - command: `nl -ba src/cron/parse.ts | sed -n '1,80p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Oversized finite timestamps would make the PR's formatter throw:** A direct reproduction on this checkout showed `Number('9000000000000000')` is finite/positive, `new Date(n)` becomes `Invalid Date`, and `toISOString()` throws `RangeError: Invalid time value`. That matches the review concern against the PR's `formatEpochMs` helper.
  - command: `node -e "const n=Number('9000000000000000'); console.log(JSON.stringify({finite:Number.isFinite(n), positive:n>0, parsed:new Date(n).toString()})); try { console.log(new Date(n).toISOString()); } catch (e) { console.log(e.name + ': ' + e.message); }"`
- **PR discussion already contains a concrete blocking review point:** The provided review timeline includes two Codex review comments on PR #58908 noting that `formatEpochMs` only guards non-finite values and would throw for out-of-range finite timestamps, turning a logging/event-path enhancement into a scheduler failure path.
  - sha: [3b5fd442d049](https://github.com/openclaw/openclaw/commit/3b5fd442d049278fa72348a94c5db43a35191d84)
