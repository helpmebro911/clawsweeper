## Summary

Keep #70682 open. Current main has not shipped this PR, and the linked privacy issue is still not cleanly resolved by main or by the PR as written: host `read` is only workspace-guarded when `tools.fs.workspaceOnly` resolves true, while the documented/default config remains false. The PR also bundles unrelated changes and has a review finding, so it needs maintainer/security judgment rather than automated cleanup.

## Best Possible Solution

Keep #70682 open but require maintainer/security triage. The best path is to split unrelated changes, decide the intended security semantics for `tools.fs.workspaceOnly` and per-agent workspace privacy, then land a focused read-tool fix with regression tests for cross-agent memory reads. If the policy stays opt-in, docs should clearly state that separate workspaces do not block direct host reads unless workspace-only FS tools or sandboxing are enabled.
