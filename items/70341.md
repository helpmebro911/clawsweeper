## Summary

Keep #70341 open. Current main has the broader #52952 command-rebuild/exact-command approval path, but it does not contain this PR's narrower Windows PowerShell cmdlet allowlist fallback: unresolved bare cmdlets still cannot match non-wildcard allowlist entries or persist a PowerShell-path allow-always pattern. The PR also has an unresolved correctness concern from review: once it makes a bare cmdlet look allowlisted, the node-host Windows execution path would still rewrite the shell payload to `segments[0].argv` and try to spawn `Get-ChildItem` directly.

## Best Possible Solution

Keep the PR open as needs-changes or maintainer product judgment. A mergeable fix should either implement cmdlet allowlist matching together with an execution plan that preserves or synthesizes the PowerShell host argv, plus regression coverage for the node-host/gateway execution path, or explicitly supersede the PR with a maintainer decision that exact-command durable approvals are the supported Windows cmdlet model.

## Evidence

A PowerShell cmdlet such as `Get-ChildItem` has no filesystem path, so current main has no way for it to match a PowerShell executable path entry with an `argPattern`.
