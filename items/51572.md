## Summary

Keep open. Current main has newer plugin lifecycle hooks for session reset/delete, but the bundled `session-memory` hook still only runs for explicit `command:new` and `command:reset`. Automatic idle/daily rollover in `initSessionState()` and maintenance pruning still do not invoke that hook path, so the feature request is not implemented on `main`.

## Evidence

- **Bundled session-memory hook still only subscribes to reset/new commands:** The bundled hook metadata advertises only `command:new` and `command:reset`, and the handler immediately returns unless the event is a `command` with action `new` or `reset`. That excludes idle resets, daily resets, and maintenance pruning.

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Session-memory handler hard-gates on explicit reset/new command events:** `saveSessionToMemory` checks `event.action === "new" || event.action === "reset"` and `event.type === "command"`; otherwise it returns without doing anything.
  - file: [src/hooks/bundled/session-memory/handler.ts:51](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/hooks/bundled/session-memory/handler.ts#L51)

After rollover it emits plugin `session_end`/`session_start` hooks only; there is no `createInternalHookEvent()`/`triggerInternalHook()` call here for the internal `session-memory` hook path.

## Risks / Open Questions

- Current main partially addresses the general area with plugin lifecycle hooks (`before_reset`, `session_end`, `session_start`), so a future maintainer could choose to solve this by adapting `session-memory` to those hooks rather than extending internal command hooks.
- GitHub CLI inspection was unavailable in this environment because `GH_TOKEN` was not set, so this review relies on the checked-out `main` code and local docs only.
