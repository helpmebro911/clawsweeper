## Summary

Keep this PR open. Current main still routes a configured Control UI `basePath` only when the request path is the base path itself or a child of it; `GET /` still falls through to the gateway 404 path. The PR remains a narrow candidate for the linked Control UI root `Not Found` behavior, with a small security review note around basePath normalization/open-redirect shape to resolve before landing.

## Best Possible Solution

Keep this PR open for maintainer review and rebase/reapply. The best implementation is the narrow gateway routing change: when `basePath` is non-empty and `pathname === "/"`, redirect `GET` and `HEAD` to `${basePath}/${search}`, preserve write-method fallthrough for plugin/webhook handling, keep regression tests for root redirect/query preservation/write fallthrough, and either harden or explicitly document the accepted `basePath` shape before landing.
