## Summary

Current main still does not implement a hard SKILL.md instruction-isolation boundary. It limits startup exposure by rendering an escaped skills catalog and supports load-time visibility filters, but selected skills are still model-facing instructions read with the normal read tool and followed by prompt guidance. There is no role locking, behavioral monitoring, or enforceable per-skill capability model for role, memory, or tool access.

## Best Possible Solution

Keep #12512 open as core security hardening. The best path is to define a documented instruction-isolation model in core prompt assembly and SKILL.md consumption, add enforceable skill capability/permission metadata for role, memory, and tool influence, and cover malicious SKILL.md cases with regression tests and user-facing docs. This should not be redirected to ClawHub unless core first provides an API that lets external skills rely on the boundary.
