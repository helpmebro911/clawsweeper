## Summary

Keep #12512 open. Current main still makes skills model-facing: the system prompt lists available skills, tells the model to read the selected SKILL.md, and then follow it. Existing controls are visibility, allowlists, prompt size limits, installer scanning, plugin prompt-hook policy, and sandbox guidance; they do not implement SKILL.md instruction isolation, role locking, memory/tool capability gates, or behavioral monitoring requested by the issue.

## Best Possible Solution

Keep #12512 open as core security hardening. Define and document a SKILL.md instruction-isolation and precedence model, add enforceable per-skill capability metadata for role, memory, and tool influence, apply it through prompt assembly/tool policy/memory/sandbox behavior, and add malicious-skill regression tests. ClawHub moderation and scanners can supplement that boundary but should not replace it.

## Risks / Open Questions

- Closing this would drop tracking for a core prompt/security boundary that current plugin, ClawHub, installer-scanning, or visibility controls do not enforce.

- The current docs correctly warn users about untrusted skills, but warnings and sandbox guidance do not provide the requested role/memory/tool instruction-isolation model.
