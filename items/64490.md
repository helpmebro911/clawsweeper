## Summary

Keep open. Current `main` still contains the zsh completion description escaping that the PR is trying to fix, and the corresponding regression coverage is not present on `main` either.

## Evidence

- **Current zsh arg generation still uses the old escaping:** `generateZshArgs` on `main` still interpolates descriptions into double-quoted zsh specs after escaping only `\\`, `"`, `'`, `[` and `]`. It does not escape `$` or backticks before emitting `"[${desc}]"`, so the PR's core fix is not already present.
  - file: [src/cli/completion-cli.ts:193](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/cli/completion-cli.ts#L193)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Main still has a completion description containing `$OPENCLAW_STATE_DIR`:** The completion command defines `--write-state` with the description `Write completion scripts to $OPENCLAW_STATE_DIR/completions (no stdout)`, which is exactly the kind of text the PR discussion called out as needing zsh-safe escaping.
  - file: [src/cli/completion-cli.ts:88](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/cli/completion-cli.ts#L88)
