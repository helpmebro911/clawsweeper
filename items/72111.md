## Summary

Keep open. Current main does not appear to implement the reported spawned streamable-http lifecycle, and it also does not clearly reject the mixed `command` + `url` + `transport: "streamable-http"` configuration. The resolver checks stdio first, so a config like the issue body is treated as a stdio server, while the reported child process is actually starting an HTTP MCP endpoint. That makes the 30000ms timeout plausible on current main rather than obsolete or already fixed.

## Best Possible Solution

Keep the issue open. The best fix is either to add an explicit managed streamable-http child-process mode with port readiness probing, HTTP client connection, lifecycle cleanup, and regression coverage, or to make current behavior explicit by rejecting mixed `command` + `url`/HTTP transport configs before runtime with an actionable error. The current silent stdio precedence should not remain the observable timeout path.
