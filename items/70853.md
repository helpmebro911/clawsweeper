## Summary

Keep #70853 open. Current main still has the automatic reply-session reset path rebinding the same stable session key to a fresh sessionId/sessionFile without any visible replay, hydration, or logical injection of prior user/assistant transcript history. The related PR #70898 is a plausible candidate fix, but it is open in the provided context and its proposed replay helper is not present on current main.

## Best Possible Solution

Keep #70853 open until a focused fix lands. The best path is to merge or supersede PR #70898 with a tested implementation that preserves a bounded recent user/assistant tail, or logically injects that tail, when silently rotating an existing DM session key; tests should cover continuity after silent rotation and confirm explicit `/new`/`/reset` still start fresh.
