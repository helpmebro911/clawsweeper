## Summary

Keep open. Current main still lacks SKILL.md signature/hash verification, persisted trusted skill integrity metadata, runtime re-verification, and unsigned-skill capability gating. ClawHub plugin archive integrity exists, but skill-family packages are intentionally routed away from that verifier into the native skills install/load path.

## Best Possible Solution

Keep #12507 open as the canonical core/ClawHub skill integrity issue. A complete fix should add trusted hash/signature/publisher metadata to ClawHub skill versions, persist expected digest/signature state in .clawhub origin/lock metadata, verify native skill downloads before extraction/install, reverify SKILL.md or full skill contents during skill snapshot/load, expose verification status on SkillEntry/status surfaces, and add maintainer-approved config/capability policy for unsigned or unverified skills with docs and regression tests.

## Risks / Open Questions

- ClawHub plugin archive integrity can be mistaken for a complete fix, but current code explicitly routes skill-family packages away from that verifier.
- Install-time hashing alone would not satisfy the issue unless trusted metadata is persisted and skills are rechecked at runtime or snapshot build time to detect post-install local tampering.
- The current docs and threat model acknowledge untrusted skill risk, but they do not provide the requested enforcement mechanism.
