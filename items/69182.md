## Summary

Current main still implements active-memory as a blocking `before_prompt_build` pass that is awaited before the main reply prompt is built. The newer hard-timeout behavior appears to address the related #70481 symptom where timeoutMs was not enforced, but it does not address #69182's central performance request: avoid delaying first visible model output by the configured timeout.

## Best Possible Solution

Keep #69182 open for maintainer product/engineering judgment. A real fix likely belongs in `extensions/active-memory/index.ts`: either make the recall pass asynchronous and inject/use results after first output when available, add a much smaller fail-open budget for first-token latency, or introduce adaptive timeout behavior with tests proving the main reply no longer waits for the full configured `timeoutMs`. The existing hard-timeout tests from the #70481 path should remain as regression coverage.
