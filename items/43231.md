## Summary

Keep open. Current main has stronger partial coverage than the earlier review noted: Telegram topic names are cached, persisted, and surfaced for the current forum topic. It still does not implement the requested human-readable topic-name to topic_id resolution, topic catalog/list action, `forum_topics` map, or channel metadata endpoint needed for routing to another topic by name.

## Best Possible Solution

Implement this in the Telegram plugin as an agent-visible topic catalog/resolver. Reuse the persisted service-message topic-name cache, merge explicit config-provided aliases where needed, expose a safe `topic-list` or directory-style action returning known `topic_id <-> title` entries with source/confidence, and teach routing to use exact unique matches while falling back to numeric `:topic:<id>` targets for ambiguity or unknown topics. Add focused tests for listing, duplicate names, stale/closed topics, persistence, and message routing, then document the Telegram API limitation and cache semantics.
