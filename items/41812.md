## Summary

Keep open. Current main still routes `openclaw models set` through parse/alias resolution and config mutation only; it does not consult the model catalog, warn, or reject catalog misses. The PR is a focused implementation candidate, but the latest revision changed the stated behavior from rejection to warning and still needs maintainer/product review plus regression tests before merge.

## Best Possible Solution

Keep the PR open and ask for a focused update: decide whether `models set` should warn or reject on catalog misses, keep custom/stale-catalog allowlist behavior intact, use the established config mutation path unless there is a proven reason not to, update the PR body to match the chosen behavior, and add regression tests for catalog hit, catalog miss, catalog-unavailable, and configured custom-model cases.
