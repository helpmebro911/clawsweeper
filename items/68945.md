## Summary

Current main still does not pass `humanDelay` into the Telegram block reply dispatcher. The shared dispatcher only applies pacing when `ReplyDispatcherOptions.humanDelay` is supplied, and other channel implementations still wire `resolveHumanDelayConfig(...)` explicitly while Telegram does not. Related PRs in the provided timeline are candidate fixes, but the canonical issue should remain open until one lands on main.

## Best Possible Solution

Keep #68945 open and land a small Telegram-owned fix: expose/import `resolveHumanDelayConfig` through the existing plugin SDK/runtime seam as appropriate, pass `humanDelay: resolveHumanDelayConfig(cfg, route.agentId)` into Telegram's `dispatcherOptions`, and add a focused Telegram regression test proving configured humanDelay reaches the buffered block dispatcher. The open candidate PRs can be used as references, but the issue should not close until the fix is on main.
