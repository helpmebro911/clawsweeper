## Summary

Current `main` still has the exact boundary mismatch described in PR #60155: the workspace-skill audit treats the workspace root as the containment boundary, while the skill loader enforces containment under the configured skills root. The proposed fix is therefore not implemented on `main`, and the PR should stay open.

## Evidence

- **Audit still checks workspace root:** `collectWorkspaceSkillSymlinkEscapeFindings` resolves `workspaceRealPath` and suppresses warnings whenever `isPathInside(workspaceRealPath, skillRealPath)` is true. Its finding text and remediation still say 'workspace root', so a symlink that escapes `workspace/skills` but stays inside the workspace is still treated as clean.
  - file: [src/security/audit-extra.async.ts:487](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/security/audit-extra.async.ts#L487)
  - command: `nl -ba src/security/audit-extra.async.ts | sed -n '476,570p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Loader enforces skills-root containment:** `resolveContainedSkillPath` rejects any candidate whose realpath is not inside `rootRealPath`. Workspace skill loading passes `rootDir = <workspace>/skills` and then reuses the resolved `baseDirRealPath` when validating each skill directory and `SKILL.md`, so the loader boundary is the skills directory root, not the workspace root.
  - file: [src/agents/skills/workspace.ts:268](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/skills/workspace.ts#L268)

There is no test for the PR's reported case where `skills/**/SKILL.md` resolves inside the workspace but outside `workspace/skills`, so nothing on `main` demonstrates the mismatch was addressed.
