## Summary

Keep this issue open. Current main still emits the reported embedded `read tool called without path` warning without the run/session/agent correlation needed to identify the malformed embedded caller. Current read-parameter validation and heartbeat prompt guidance are adjacent improvements, but they do not close the diagnostic/root-cause gap described here.

## Best Possible Solution

Keep this open as a diagnostics and root-cause task. Add structured correlation to the missing-path warning using available `runId`, `sessionId`, `sessionKey`, and `agentId`; pass trigger/provider/channel/plugin context into the subscription if needed; then reproduce the Feishu timeout/reconnect/failover path and fix or reject the owner that constructs a malformed `read` call before it reaches the embedded tool-start handler.
