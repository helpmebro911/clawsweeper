## Summary

Keep this issue open. Current main still uses fixed-size `chunkMarkdown(content, { tokens, overlap })` for memory-core indexing, exposes no heading/section chunking strategy in config/docs, and cannot persist a strategy change in reindex metadata. The related open PR #68019 is an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review #68019 or a smaller focused PR that adds a documented heading/section-aware chunking strategy in the memory-host SDK path used by memory-core, exposes it in config/schema/help/docs, persists the effective strategy or config fingerprint in reindex metadata, and adds tests proving adjacent `##`/`###` sections do not blend into one embedding/FTS chunk. Track entity extraction and graph memory as a separate memory-core or memory-wiki design item unless maintainers intentionally keep this as the umbrella.
