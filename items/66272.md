## Summary

Keep open. Current main still has the reported loopback `gateway.trusted_proxies_missing` behavior, so the underlying cleanup is not implemented. The submitted PR is also stale against the current split gateway-audit implementation and should be revised to classify exposure from the effective resolved bind host/proxy context rather than the raw `gateway.bind` string.

## Best Possible Solution

Keep this PR open and request a revision against `src/security/audit-gateway-config.ts`. The fix should classify `gateway.trusted_proxies_missing` from effective gateway exposure and explicit reverse-proxy context, stop warning for definitely local-only loopback-resolving configs, preserve warnings for real reverse-proxy/exposed cases, and add regression coverage for loopback, LAN, custom loopback, custom exposed, and auto/container behavior.

## Risks / Open Questions

- Merging the submitted patch unchanged is stale against current main because the active gateway audit logic now lives in `src/security/audit-gateway-config.ts`.

- Suppressing the warning for every literal loopback config could hide same-host reverse-proxy deployments where the docs still require `gateway.trustedProxies` for forwarded-client-IP handling.
