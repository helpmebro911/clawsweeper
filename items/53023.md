## Summary

Keep open. Current main still serializes every session lane by default, exposes only global/subagent concurrency config, and does not implement config-backed auto-yield or a configurable lane-wait warning threshold. The open related PR #63864 may inform the session-concurrency portion, but it is unmerged and does not cover the whole request.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review or rework PR #63864 for the session-lane concurrency portion, wire any accepted setting through config types, strict schema, defaults, startup/reload application, docs, and queue/runner tests, then separately decide whether auto-yield belongs in the embedded runner/tool loop or queue policy. Lane-wait warning configurability should be tracked in the same config surface or a narrower canonical follow-up if maintainers want to split the work.

## Risks / Open Questions

- Increasing per-session concurrency may violate current assumptions around transcript/session-file mutation, so the implementation should include focused queue and runner tests.
- Automatic yield needs a product and runtime design decision because forced interruption across tool execution is different from the existing explicit `sessions_yield` flow.
