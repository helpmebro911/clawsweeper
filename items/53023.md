## Summary

Keep this issue open. Current main still lacks config-backed per-session lane concurrency, automatic yield after elapsed tool-loop time, and a configurable lane-wait diagnostic threshold. Existing queue modes and explicit `sessions_yield` are adjacent mitigations, but they do not implement the requested defaults. The related open PR #63864 appears to cover only the session-concurrency slice, while #14747 separately tracks the lane-wait warning threshold.

## Best Possible Solution

Keep this issue open as the umbrella tracker. Review or rework #63864 for the per-session concurrency portion, wiring the accepted setting through AgentDefaults types, zod schema, defaults, generated metadata/docs, startup/reload lane application, and queue/runner tests. Separately design `autoYieldAfterMs` in the embedded runner/tool loop, and either implement `laneWaitWarnAfterMs` here or explicitly delegate that slice to #14747.

## Risks / Open Questions

- Increasing per-session concurrency can conflict with transcript ordering and session mutation assumptions unless the accepted design defines locking and lifecycle semantics clearly.
- Auto-yield is a separate runner/tool-loop design problem: forced interruption during tool execution differs from the existing explicit `sessions_yield` callback and queue interrupt modes.
- If #63864 merges, this issue should be reassessed or narrowed, but the auto-yield and lane-wait warning settings would still need implementation or separate canonical tracking.
