## Summary

Keep this issue open. Current main still serializes session lanes at the queue default, exposes only global/subagent concurrency in `agents.defaults`, keeps the lane wait warning default hardcoded at 2 seconds unless a caller passes an override, and implements `sessions_yield` only as an explicit tool action. The related open PR #63864 is an implementation candidate for the session-concurrency slice only; #14747 separately tracks the warning-threshold slice, and neither covers auto-yield on current main.

## Best Possible Solution

Keep this issue open as the umbrella tracker. Review or rework #63864 for the per-session concurrency portion, including AgentDefaults types, zod schema, defaults/generated metadata/docs, startup and reload lane application, and queue/runner tests. Separately design `autoYieldAfterMs` in the embedded runner/tool loop, and either implement `laneWaitWarnAfterMs` here or explicitly delegate that slice to #14747 once there is a canonical plan.

## Risks / Open Questions

- Increasing per-session concurrency needs an explicit design for transcript ordering, session mutation, and write-lock interactions; the current docs say the session lane prevents those races.
- Auto-yield is a separate runner/tool-loop behavior from the existing explicit `sessions_yield` tool and would need clear cancellation/interruption semantics for in-flight tool calls.
- Closing this umbrella issue before #63864 merges and before auto-yield/warning-threshold work is tracked elsewhere would drop still-missing requested behavior.
