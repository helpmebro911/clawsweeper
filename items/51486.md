## Summary

Keep open. Current `main` still contains the exact global `schtasks /Query` preflight that this PR removes, so the change is not implemented on `main` and the PR still addresses a live gap in the Windows runtime-status path.

## Evidence

- **Runtime read path still does the broad preflight:** `readScheduledTaskRuntime()` on current `main` first calls `assertSchtasksAvailable()` and only then performs the task-scoped `schtasks /Query /TN ... /V /FO LIST` lookup. That is the behavior this PR proposes to remove.
  - file: [src/daemon/schtasks.ts:1018](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/daemon/schtasks.ts#L1018)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '1018,1044p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Preflight helper is still the global `/Query` call:** `assertSchtasksAvailable()` still shells out with `execSchtasks(["/Query"])`, so a failure in the broad probe can still block runtime reads before the task-scoped query runs.
  - file: [src/daemon/schtasks.ts:288](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/daemon/schtasks.ts#L288)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '286,305p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Mainline tests still encode the preflight path:** The startup-fallback test helper on `main` still queues an initial successful schtasks response before the failing task lookup, which matches the existing preflight behavior. The supporting fixtures/mocks also have no thrown-error queue for `execSchtasks`, so the extra regression coverage described in the PR is not present on `main`.
  - file: [src/daemon/schtasks.startup-fallback.test.ts:123](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/daemon/schtasks.startup-fallback.test.ts#L123)
