## Summary

Keep #68793 open. Current main has already moved the raw WebSocket DISPATCH payload log to `debug`, but the central privacy/noise problem remains because normal inbound QQ Bot messages are still logged at `info` with the full `event.content`, and the plugin wires that logger into the OpenClaw channel logger used for gateway/file logs.

## Best Possible Solution

Keep the issue open and fix QQ Bot logging in the bundled plugin: demote the `Processing message ... event.content` path to debug or replace it with metadata such as message id, sender/account, chat type, content length, and attachment counts. Add focused tests with an injected logger proving info logs do not contain inbound message bodies while debug can still expose payload details when explicitly enabled.
