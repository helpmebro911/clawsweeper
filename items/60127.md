## Summary

Keep this issue open. Current main has multi-agent routing, device roles/scopes, plugin enablement, and cooperative shared-inbox hardening, but it still explicitly does not support the requested shared-server tenant isolation model with tenant identity propagation, tenant RBAC, tenant quotas, tenant-scoped plugin enablement, or tenant-attributed audit boundaries.

## Best Possible Solution

Keep this issue open for maintainer product and security triage. If shared-server multi-tenancy is in scope, design tenant identity propagation through Gateway auth/request context and protocol, tenant-scoped state/config/plugins/skills/model auth, per-tenant quotas, tenant-attributed audit logs, and cross-tenant denial tests/docs. If it is out of scope, maintainers should document that hostile shared-Gateway tenancy is unsupported and split narrower issues only for specific plugin/API seams external managed-service implementations need.

## Risks / Open Questions

- This is broad product and security architecture work; maintainers may decide shared hostile multi-tenancy is intentionally out of scope rather than implementing it.

- Current docs use both cooperative multi-agent sharing language and strict security disclaimers, so any future direction should keep cooperative multi-agent use distinct from adversarial tenant isolation.
