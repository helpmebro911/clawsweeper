## Summary

Keep this PR open. Current main still lacks the requested configurable image-tool compression quality/max-side controls, the related core feature request #41771 is closed without a shipped implementation, and the previous related implementation #42058 is also closed unmerged. This PR is a useful implementation candidate, but it should not merge as-is because the provided review comments identify constraint-violating fallback behavior and the branch bundles unrelated docs/gateway/provider-link changes.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a focused rebase before merge or supersede it with a clean PR. The best patch should add a typed and documented `agents.defaults` image-tool compression contract, thread quality/max-side options from `createImageTool` through `loadWebMedia` and `optimizeImageToJpeg`, honor out-of-ladder values such as quality 95/85 and max side 600, add regression tests for those edge cases, preserve current media access/SSRF guards, and split unrelated docs/gateway/provider-link edits into separate work if still wanted.

## Risks / Open Questions

- Do not merge the PR as-is: the proposed fallback behavior can silently violate configured compression constraints for out-of-ladder values like quality 85 or max side 600.
- The branch bundles unrelated README, debugging docs, gateway invalid-config messaging, doctor wording, and provider-link changes with the image compression feature.
- The branch is stale against current main's media path layout: current main uses `src/media/web-media.ts`, while the PR body/diff refer to the older `src/web/media.ts` surface.
- Higher quality or larger max-side settings can increase payload size, model token usage, latency, and cost if applied globally without clear opt-in semantics.
