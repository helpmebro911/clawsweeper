## Summary

Keep this issue open. Current main has related route-preservation safeguards, but it still does not implement a force-origin or immutable persisted-origin reply behavior for Telegram/Discord/WhatsApp-style channels when WebChat/dashboard is involved. The linked implementation PR #57802 is useful prior art, but it was closed unmerged.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is a narrow reply-routing design at the session/reply boundary, either an explicit opt-in force-origin setting or an immutable persisted-origin route based on the prior PR #57802 approach, with tests covering Telegram/Discord-style inbound sessions while dashboard/WebChat is active and safeguards against stale-route or wrong-channel delivery.

## Risks / Open Questions

- A broad force-origin default could misdeliver replies across channels or stale sessions; current main intentionally protects WebChat/UI clients from silently inheriting external routes.
- Existing route-preservation fixes for dashboard, subagent, and inter-session delivery should not be regressed while adding any opt-in or persisted-origin behavior.
