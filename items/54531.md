## Summary

Keep open. Current main still blocks WebChat/UI turns from routing replies to an external originating channel unless explicit delivery is set, and it does not contain the requested force-origin-channel setting or immutable persisted-origin fields. The closed unmerged #57802 remains useful prior art, not a shipped fix.

## Best Possible Solution

Keep this issue open for a narrow maintainer-reviewed reply-routing design. The likely fix belongs at the session/reply boundary: either an explicit force-origin setting or a carefully scoped immutable first external origin route, with tests for messaging-channel sessions while WebChat/dashboard is active and with safeguards against stale or wrong-channel delivery. #57802 can be used as prior art, but it should be revived or replaced against current route-preservation behavior rather than treated as shipped.
