## Summary

Current main still does not implement SKILL.md or plugin skill metadata model routing. The existing code supports model selection at default, agent, cron payload, live session, subagent-default, and plugin hook levels, but there is no parsed skill-level model/preferredModel/modelProfile contract or runtime routing path tied to skill invocation.

## Best Possible Solution

Keep this issue open as the canonical feature request. The best path is to review, revise, land, or supersede #58142 with a maintainer-approved implementation that adds a documented skill metadata contract, parses and carries preferred model/fallback/profile fields through skill snapshots or runtime lookup, applies routing at the selected skill invocation/read boundary with clear precedence, reuses existing model alias/allowlist resolution, and adds focused tests for parsing, slash/read invocation behavior, fallbacks, and plugin hook interaction.

## Risks / Open Questions

- This is a public skill metadata/API decision; maintainers still need to settle field names, precedence, fallback behavior, allowlist enforcement, and interaction with explicit user/session model choices, cron payload models, live /model switching, subagents, and before_model_resolve hooks.
- A plugin can override provider/model via before_model_resolve, but that hook currently receives prompt/attachment context rather than parsed skill identity and skill metadata, so closing this as ClawHub/plugin work would leave the requested core contract unresolved.
- The open PR #58142 may be useful, but it needs maintainer review for behavior, security/cost policy, and session-state design before the issue can close as implemented.
