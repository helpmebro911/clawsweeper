## Summary

Keep #13583 open. Current main has adjacent enforcement surfaces, including fail-closed pre-tool hooks, one-pass final-answer revision, and outbound message cancellation, but it still does not implement the requested fail-closed pre-response gate for mandatory tool-call accounting, typed validators, retry/fallback policy, and audit records. The latest maintainer/Codex discussion also explicitly kept this issue open as a remaining core/plugin API gap.

## Best Possible Solution

Keep #13583 open as the canonical response-gate policy-as-code design item. The best maintainer path is a versioned core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records; domain-specific policy packs can then live in plugins or ClawHub.

## Risks / Open Questions

- Existing before_tool_call and message_sending hooks solve narrower guardrail problems, so the remaining issue should stay scoped to response-gate policy-as-code rather than all governance ideas.
- A fail-closed response gate changes final-answer semantics across harnesses, providers, channels, retries, and audit logging, so it needs a versioned core/plugin API with focused tests and docs.
