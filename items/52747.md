## Summary

Keep this PR open. Current main still lacks a timeout/deadline for ACP session-actor lane tasks that hang before active turn execution, including initialize and ensure-runtime paths. The PR addresses a real control-plane gap, but its proposed timeout design still needs maintainer review because it can release the per-session queue while the original operation continues running.

## Best Possible Solution

Keep this PR open as an implementation candidate for a real ACP control-plane bug. The best path is a session-lane deadline design that covers initialize/status/cancel/close/idle-eviction actor tasks, preserves per-session serialization or fences stale post-timeout side effects, propagates cooperative cancellation/abort support to runtimes where available, keeps runtime cache/concurrency accounting until close is confirmed, clamps and unrefs timers safely, and wires `acp.sessionLane.taskTimeoutMs` through config schema/help/docs plus regression tests for a hung `ensureSession` followed by queued work.

## Risks / Open Questions

- The proposed timeout rejects the queue-visible promise while `op()` continues, which can violate per-session serialization and let stale side effects race with newer same-session work.
- Applying the timeout to idle eviction is risky while the cache entry is cleared before `runtime.close()` finishes; a hung close can orphan a runtime and undercount active sessions.
- The proposed `taskTimeoutMs` accepts very large values in the schema and should be clamped to safe timer bounds with clear semantics for disable/default.
- The PR does not appear to introduce workflow, package, lockfile, dependency-source, publishing, or lifecycle-script changes; supply-chain risk looks low from the diff, but ACP concurrency behavior is security-sensitive.
