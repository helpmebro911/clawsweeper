## Summary

Keep #8508 open. Current main still exposes and consumes ack reactions as a single configured value, not an array, random/contextual selector, or model-driven per-message decision. The prior maintainer/Codex comment on #8508 matches the current code state, and the related items cover ack cleanup, timing, account routing, or slash-command feedback rather than configurable/dynamic emoji selection.

## Best Possible Solution

Keep #8508 open for a backward-compatible core/channel design. Preserve the existing `ackReaction` string contract, then add a clear `messages.ackReactions` array or selector contract, define whether inbound message context is available for selection, and update exported config types, zod schemas, generated schema/help metadata, public docs, shared channel-feedback helpers, SDK surfaces if needed, and focused bundled-channel tests.

## Risks / Open Questions

- Channel validation differs: Slack uses shortcode names, Telegram supports a constrained Unicode reaction set, Discord may involve Unicode or custom emoji, WhatsApp uses a channel-specific object, and BlueBubbles tapbacks are constrained.
