## Summary

Keep #8508 open. Current main still models ack reactions as a single configured string and sends that one value through the shared and bundled channel ack paths. There is no array, selector, random/contextual chooser, or model-driven pre-processing hook for per-message ack emoji selection.

## Best Possible Solution

Keep #8508 open for a backward-compatible core/channel design. Preserve the existing `ackReaction` string contract, add a clear `messages.ackReactions` array or selector contract if maintainers want this feature, define whether inbound message context is available before the ack is sent, then update exported config types, zod/generated schemas, schema help/metadata, docs, shared channel-feedback helpers, SDK exports if needed, and focused bundled-channel tests.

## Risks / Open Questions

- Product/API direction is still needed: deterministic array selection, random selection, rule-based selection, or model-driven selection have different latency and predictability tradeoffs because ack reactions are sent before normal reply generation.
- Channel reaction validation differs across Telegram, Slack, Discord, Matrix, WhatsApp, and BlueBubbles, so the eventual contract should define channel-specific fallback behavior.
