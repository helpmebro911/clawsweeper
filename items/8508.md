## Summary

Keep #8508 open. Current main still supports a configurable single ack reaction string plus lifecycle status reactions, but it does not support the requested `ackReactions` array or agent/context-selected ack reaction per inbound message. A same-day maintainer/Codex comment also explicitly kept this issue open for that missing behavior.

## Best Possible Solution

Keep the issue open for maintainer product direction. A good implementation would add an explicit config contract such as `messages.ackReactions` or a channel/account override equivalent, preserve the existing `ackReaction` behavior for compatibility, define whether selection is deterministic/random/model-driven, and update the resolver, bundled channel ack paths, schema/help/docs, and targeted tests across at least Telegram plus one shared/global contract test.
