## Summary

Keep this issue open. Current main still implements WebChat/Control UI as an internal/operator chat surface selected by an existing session key, while `identityLinks` canonicalization is applied only when routing has a concrete direct-message peer id. The docs still state both that `identityLinks` links the same person across channels and that WebChat defaults to the selected agent's main session, so the requested runtime mode or docs clarification remains unresolved.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. The likely product path is an explicit WebChat/Control UI session mode or identity setting that can bind a configured WebChat identity such as `webchat:default` to the same canonical direct key as linked Telegram DMs. If maintainers decide current operator semantics are intentional, update the public Session Management, Channel Routing, and WebChat docs to state that `identityLinks` applies to direct channel peers and WebChat remains main-session/operator scoped unless the user manually switches sessions.
