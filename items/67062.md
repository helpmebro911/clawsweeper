## Summary

Keep this issue open. Current main covers the reporter's example host through the fixed `*.qq.com.cn` default, but it still does not expose or wire a supported `channels.qqbot.mediaAllowHosts` setting for QQ Bot media downloads. The bundled QQ Bot plugin continues to use a hardcoded SSRF hostname policy for inbound attachments and outbound fallback downloads, while Microsoft Teams already has the configurable pattern the reporter cited.

## Best Possible Solution

Implement this in the bundled QQ Bot plugin: add `mediaAllowHosts` to the QQ Bot account/channel config type, Zod schema, plugin manifest schema, generated metadata, and docs; normalize configured hosts with the existing SSRF allowlist helpers; merge them with the conservative QQ/Tencent defaults before inbound and outbound fallback media downloads; and add focused tests for defaults, custom host merging, HTTPS-only behavior, and continued unsafe/private-network rejection.
