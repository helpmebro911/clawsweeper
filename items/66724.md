## Summary

Current `main` still appears vulnerable to the reported Feishu/Lark regression. The streaming reply path forwards every non-empty partial text snapshot into the card accumulator, and `mergeStreamingText()` still appends unrelated text like `"..."` ahead of the final reply. I did not find a current fix or a regression test covering placeholder-only partials, and the Feishu streaming files are unchanged between the latest release `v2026.4.23` and `main`.

## Evidence

- **Partial streaming text is accepted without placeholder filtering:** `onPartialReply` only skips falsy `payload.text`, then forwards every non-empty partial snapshot into `queueStreamingUpdate(...)`. There is no special-case for placeholder-only text such as `"..."`.
  - file: [extensions/feishu/src/reply-dispatcher.ts:495](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/feishu/src/reply-dispatcher.ts#L495)
  - command: `nl -ba extensions/feishu/src/reply-dispatcher.ts | sed -n '400,520p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Final close path preserves previously accumulated partial text:** When the streaming card is active, final delivery merges the final text into `streamText` with `mergeStreamingText(...)` before closing. Any earlier placeholder text already accumulated into `streamText` is therefore retained unless a later snapshot fully subsumes it.
  - file: [extensions/feishu/src/reply-dispatcher.ts:416](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/feishu/src/reply-dispatcher.ts#L416)
