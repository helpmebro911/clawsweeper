## Summary

Keep open. Current main still treats an unmarked external SIGTERM as a clean gateway stop, unit coverage pins that behavior, the LaunchAgent plist relies on launchd KeepAlive for respawn, and the macOS launchd integration tests still do not exercise the reporter's direct SIGTERM -> new PID -> healthy endpoint path. Related closed launchd items cover adjacent restart, recovery, and PID issues, not this raw-SIGTERM on-demand-only repro.

## Best Possible Solution

Keep this issue open. Add a macOS launchd integration reproduction that installs a test Gateway LaunchAgent, reads the launchd runtime PID, sends SIGTERM to that PID, and asserts `launchctl print` reaches `state = running` with a different PID and the health endpoint returns. If that fails, fix the macOS launchd lifecycle or document the exact GUI-session/on-demand-domain limitation; if it passes on current main, close later with that direct proof.
