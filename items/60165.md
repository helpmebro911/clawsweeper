## Summary

Do not close. Current main only backfills `PATH` for gateway exec by probing a non-interactive login shell; it does not generally load interactive shell rc files like `~/.zshrc`, which is the core request in #60165. The docs and tests also scope the behavior to PATH merging, not full shell-profile loading, so there is not strong evidence that the reported macOS/zsh+nvm case is fully implemented.

## Evidence

- **Gateway exec only merges PATH from a shell probe:** `exec` applies `getShellPathFromLoginShell(...)` only for `host === "gateway"`, then merges that PATH into the child env via `applyShellPath(env, shellPath)`.
  - file: [src/agents/bash-tools.exec.ts:1590](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/bash-tools.exec.ts#L1590)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '1588,1598p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **The shell probe is a non-interactive login-shell env dump:** The implementation runs the shell as `-l -c 'env -0'` and then reads `PATH` back from that environment. That is a login-shell probe, not a general interactive rc-file load.
  - file: [src/infra/shell-env.ts:81](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/shell-env.ts#L81)
  - command: `nl -ba src/infra/shell-env.ts | sed -n '69,90p;277,297p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Actual exec commands still run with plain `-c`:** Non-Windows exec chooses the shell with args `['-c']`, and runtime execution builds argv as `[shell, ...shellArgs, execCommand]`. That means the command itself is not launched as an interactive/login shell.
  - file: [src/agents/shell-utils.ts:54](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/shell-utils.ts#L54)

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **User-facing docs describe PATH merge only:** The exec docs say `host=gateway` "merges your login-shell PATH into the exec environment" and describe node/sandbox PATH behavior; they do not claim to source `~/.zshrc` or other interactive shell config for exec runs.
  - file: [docs/tools/exec.md:124](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/tools/exec.md#L124)

- **Tests cover PATH merge, not full shell-profile loading:** The dedicated test asserts that gateway exec merges login-shell PATH entries into `echo $PATH`. There is no corresponding test demonstrating interactive rc-file sourcing or zshrc-loaded version managers.
  - file: [src/agents/bash-tools.exec.path.test.ts:146](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/bash-tools.exec.path.test.ts#L146)

## Risks / Open Questions

- Some user setups may already work if their PATH is exported from login-shell files such as `.zprofile`/`.profile`, so the remaining gap is configuration-dependent.
- I did not run a live macOS/zsh reproduction in this environment, so the conclusion rests on code/docs/test semantics rather than end-to-end reproduction.
