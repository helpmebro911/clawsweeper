## Summary

Keep open. Current `main` still implements `openclaw cron runs` as a flag-only command with `--id`, with no positional `<jobId>` argument or alias, and it forwards only `opts.id` to the RPC layer. The same implementation was already present in `v2026.3.8`, so I did not find evidence that this issue was fixed on `main`.

## Evidence

- **CLI still requires `--id` and defines no positional argument:** `cron runs` is registered with `.requiredOption("--id <id>", "Job id")` and no `.argument("<id>")` on current `main`.
  - file: [src/cli/cron-cli/register.cron-simple.ts:137](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/cli/cron-cli/register.cron-simple.ts#L137)

  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **CLI forwards only the option value to the gateway:** The action computes `const id = String(opts.id)` and sends `{ id, limit }` to `cron.runs`; there is no code path that reads a positional argument for this command.
  - file: [src/cli/cron-cli/register.cron-simple.ts:145](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/cli/cron-cli/register.cron-simple.ts#L145)
