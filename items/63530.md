## Summary

Keep this PR open. Current main still does not implement the PR’s session-store recovery behavior, and the PR remains a plausible but unfinished gateway/session-store hardening candidate with unresolved review findings rather than something that should be closed as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to decide the intended corrupt session-store contract, fix the threshold and key-target recovery issues, and add focused regression coverage for invalid JSON, `.prev` restoration, transcript fallback, `sessions.list`, and `sessions.resolve`. If maintainers decide this recovery behavior is not wanted, close it manually after that product judgment.

## Risks / Open Questions

- Merging as-is could add avoidable transcript filesystem scans for healthy one-session stores because the PR uses `<=` instead of `<` for the minimum-entry threshold.
- The key-based recovery path appears incomplete because `resolveGatewaySessionStoreTarget` can read a corrupt store before the PR’s try/catch runs.
- Changing `loadSessionStore` from graceful `{}` fallback to throwing reverses an explicitly tested current contract and needs a complete gateway error/recovery design.
- The PR body is still mostly the untouched template, so repro, root cause, security impact, and human verification are not documented by the author.
