## Summary

Keep this PR open for maintainer review. Current main does not contain the added devcontainer, benchmark workflow, or release SBOM workflow, and the existing benchmark/SBOM surfaces only partially overlap. The PR is actionable in this repository, but it is not merge-ready because the latest diff still introduces security-sensitive workflow behavior and duplicates existing CI patterns.

## Best Possible Solution

Keep the PR open, but require a focused maintainer revision before merge: either split devcontainer, benchmark, and SBOM into separate PRs, or update this PR to reuse .github/actions/setup-node-env/action.yml, call the existing test:startup:bench/test:startup:bench:check harness where appropriate, avoid or checksum-verify any privileged hyperfine install, and generate the SBOM from the actual shipped npm tarball or production dependency set. If maintainers do not want these repo-level workflows, close it explicitly as CI/release policy rather than as implemented or superseded.

## Risks / Open Questions

- The benchmark workflow would download and install a third-party .deb with sudo without integrity verification.
- The SBOM workflow may publish a materially misleading SBOM because it scans the full checkout after a full install instead of the shipped production package/artifact.
- The release workflow grants contents: write and uses release-attachment actions, so it needs explicit maintainer review of permissions, trigger scope, and artifact policy.
- The benchmark workflow overlaps existing startup benchmark scripts and docs; accepting it as-is would create a second performance surface to maintain.
