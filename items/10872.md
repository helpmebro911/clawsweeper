## Summary

Keep #10872 open. Current main still handles iMessage inbound reply context only when the imsg payload already contains resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`; it does not parse or resolve `thread_originator_guid` / `reply_to_guid`, nor does it do a chat.db fallback lookup before dispatch. The issue discussion and open PR #42683 identify the same remaining Layer 3 gap, but that work is not shipped on main.

## Best Possible Solution

Keep the issue open until the bundled iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The likely maintainer path is in `extensions/imessage/src/monitor/*`: add a narrow resolver via supported imsg metadata/RPC or a scoped read-only chat.db lookup when `dbPath` is configured, then add parser/monitor regression tests for GUID-only inline replies. PR #42683 can be used as the starting point or superseded by a maintainer-approved replacement.
