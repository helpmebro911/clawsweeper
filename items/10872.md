## Summary

Keep #10872 open. Current main still only carries iMessage reply context when the incoming imsg payload already contains resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`. It does not model or resolve `thread_originator_guid` / `reply_to_guid`, does not perform a chat.db fallback lookup before dispatch, and the existing tests cover resolved fields only. The related open PR #42683 is a useful implementation candidate, but it is not merged or shipped, and the April 26 real-world testing comment reports the bug still occurring on released OpenClaw/imsg setups.

## Best Possible Solution

Keep the issue open until the bundled iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The likely path is in `extensions/imessage/src/monitor/*`: add a narrow resolver using supported imsg metadata/RPC or a scoped read-only chat.db lookup when `dbPath` is configured, then add regression tests for GUID-only inline replies. PR #42683 can be used as the starting point or superseded by a maintainer-approved replacement.
