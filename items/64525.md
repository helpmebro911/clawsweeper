## Summary

Keep open. Current `main` still emits `message_received` only from `dispatchReplyFromConfig`, while active-session inbound messages are enqueued into the followup queue and later drained through `runEmbeddedPiAgent` without re-entering that dispatch hook path. The linked fix PR #64559 still exists as a separate head ref, which is consistent with the issue not being resolved on `main`.

## Evidence

- **`message_received` still fires only in dispatch path:** Current `main` emits both the plugin `message_received` hook and the internal `message:received` hook inside `dispatchReplyFromConfig`; there is no parallel emission moved up to the shared ingestion path.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:601](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/auto-reply/reply/dispatch-from-config.ts#L601)

- **Busy sessions still enqueue followups instead of redispatching:** When a session is active, `agent-runner.ts` chooses `enqueue-followup`, pushes the message into `enqueueFollowupRun(...)`, and returns early. That path does not call `dispatchReplyFromConfig` again for the queued inbound message.
  - file: [src/auto-reply/reply/agent-runner.ts:1008](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/auto-reply/reply/agent-runner.ts#L1008)
