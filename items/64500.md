## Summary

Keep this issue open. Current main still lacks pair-scoped circuit breaker enforcement: the global breaker is scoped to the current tool/signature/stable result hash, while ping-pong critical blocking still requires stable result hashes on both sides. The reported variable-result memory_search <-> memory_get loop remains plausible and is not superseded by the related loop-detection issues.

## Best Possible Solution

Keep this issue open and fix the agent tool-loop guard. The best path is to make ping-pong protection pair-aware in src/agents/tool-loop-detection.ts and the before-tool-call enforcement path: either allow sustained A/B alternation to escalate without identical per-side result hashes, add a pair-scoped threshold before the current per-tool global breaker, or persist short-lived session-level blocks for both tools/signatures when a critical pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the offending pair; update config/docs if a new threshold is introduced.
