## Summary

Keep this issue open. Current main still implements the loop guard as current-tool/current-signature enforcement, while ping-pong critical blocking still depends on stable per-side result hashes and the before-tool-call hook does not persist a block for both tools in an alternating pair.

## Best Possible Solution

Keep this issue open and fix the agent tool-loop guard in src/agents/tool-loop-detection.ts plus the before-tool-call enforcement path. The best implementation should make ping-pong protection pair-aware by either allowing sustained A/B alternation to escalate without identical per-side result hashes, adding a pair-scoped threshold before the current per-tool global breaker, or persisting short-lived session-level blocks for both tool signatures when a critical pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the offending pair; update docs/config only if a new threshold or user-facing behavior is introduced.
