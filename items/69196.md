## Summary

Keep open. Current main contains the later same-origin `Permissions-Policy` microphone fix, but this issue's follow-up explicitly narrowed the failure to Chrome's `webkitSpeechRecognition` returning `not-allowed` even with microphone permission granted and after a local header change. The Control UI dictation button still depends on browser `SpeechRecognition`/`webkitSpeechRecognition`, and its error path clears recording state without surfacing the error or providing a fallback, so the reported user-visible behavior is not clearly fixed.

## Best Possible Solution

Keep the issue open and treat it as a Control UI dictation reliability bug distinct from the shipped Permissions-Policy fix. The likely maintainer path is to make the STT path surface SpeechRecognition errors, add regression coverage for `not-allowed` handling, document any Chrome/local-origin limitation if it is browser-imposed, and consider a provider-backed or `getUserMedia`-based transcription fallback if browser SpeechRecognition remains unreliable on loopback Chrome/macOS.
