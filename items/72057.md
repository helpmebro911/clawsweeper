## Summary

Keep #72057 open. The current main Google LLM path still does not automatically honor plain `HTTP_PROXY`/`HTTPS_PROXY` environment variables for `google-generative-ai` requests. Google LLM streaming uses `buildGuardedModelFetch`, and that wrapper calls the SSRF guard without enabling trusted env-proxy mode unless a model-provider `request.proxy` override is configured. Existing docs expose an explicit `models.providers.*.request.proxy.mode: "env-proxy"` workaround, but that is not the same as fixing the reported regression where unchanged Docker env proxy settings stopped working after upgrading.

## Best Possible Solution

Keep the issue open and fix the Google/model-provider LLM transport path. A conservative implementation would teach `buildGuardedModelFetch` or its resolved request policy to use `TRUSTED_ENV_PROXY` only for operator-controlled provider requests when `shouldUseEnvHttpProxyForUrl(url)` is true, while preserving strict DNS-pinned behavior for `NO_PROXY`, `ALL_PROXY`-only cases, and untrusted web fetches. Add focused regression coverage for `google-generative-ai` requests to `generativelanguage.googleapis.com` with `HTTPS_PROXY` set, plus bypass coverage for `NO_PROXY`; optionally document `models.providers.google.request.proxy.mode: "env-proxy"` as a temporary workaround.
