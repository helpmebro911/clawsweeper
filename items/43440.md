## Summary

Keep this issue open. Current main still implements legacy iMessage by spawning `imsg rpc` as a stdio child process from the OpenClaw gateway runtime, and the docs/setup text still require Full Disk Access for the process context running OpenClaw/`imsg`. I found no current launchd/XPC/signed helper handoff that would make the `imsg` binary's own TCC grant sufficient after a Node binary path change.

## Best Possible Solution

Keep this open for maintainer product/security handling in the iMessage plugin boundary. The likely fix is to run legacy `imsg` database access through a stable OpenClaw-owned or imsg-owned macOS process context, document/support a launchd, signed-app, or XPC bridge that survives Node upgrades, or explicitly retire the legacy path in favor of BlueBubbles with maintainer approval. Any implementation should update `extensions/imessage/src/client.ts`, monitor/probe/send behavior, setup/doctor guidance, and public iMessage docs with macOS TCC reproduction proof.

## Risks / Open Questions

- A complete fix needs macOS validation of TCC attribution before and after a Homebrew Node upgrade; static review can confirm the current process model but not OS-level permission prompts.
- The legacy iMessage path is documented as deprecated in favor of BlueBubbles, so maintainers may choose an explicit retirement/docs path instead of adding an imsg-owned helper process.
