## Summary

Keep this issue open. Current main still runs legacy iMessage through a child `imsg rpc` process from the gateway/plugin runtime, and the checked-in setup/docs still tell users to grant Full Disk Access to the OpenClaw/`imsg` process context. I found no bundled iMessage helper, launchd/XPC handoff, or macOS app-owned bridge that would make a stable `imsg` permission identity sufficient after a Node/Homebrew path change.

## Best Possible Solution

Keep this issue open for maintainer product/security handling in the bundled iMessage plugin. If legacy `imsg` remains supported, move Messages database access behind a stable macOS process identity such as a signed OpenClaw-owned helper, imsg-owned daemon, launchd/XPC bridge, or another explicit helper boundary. Otherwise, document and execute a retirement path toward BlueBubbles. The implementation should update `extensions/imessage/src/client.ts`, probe/send/monitor behavior, setup copy, public iMessage docs, and include macOS TCC reproduction proof.
