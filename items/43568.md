## Summary

Keep this issue open. Current main has provider-config-scoped proxy support through `models.providers.*.request.proxy`, but auth-profile credential records and auth resolution still do not carry profile-scoped transport metadata. The requested `proxyUrl` per auth profile remains unimplemented, and the plugin/runtime seams currently reject proxy/TLS transport injection from runtime auth hooks.

## Best Possible Solution

Keep the issue open. The best path is a deliberate core/provider-transport design: add a documented, redacted auth-profile transport field or equivalent structured proxy setting; flow it through auth-profile resolution into the selected model request transport for the active profile; preserve existing provider-level request proxy behavior; and add regression tests for multiple Anthropic/OpenAI profiles under one provider using distinct proxies. If maintainers decide separate custom provider IDs are the supported model, document that explicitly before closing.

## Risks / Open Questions

- Provider-level `models.providers.*.request.proxy` may be an acceptable workaround if users split accounts into separate custom provider IDs, but that is not equivalent to per-auth-profile routing under one provider and needs maintainer product judgment.
- A future profile-scoped proxy feature would need explicit redaction and security handling because proxy URLs may contain credentials and would affect outbound provider traffic routing.
