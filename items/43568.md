## Summary

Keep open. Current main supports provider-config-scoped proxy transport through `models.providers.*.request.proxy`, but it still does not provide an auth-profile-scoped `proxyUrl` or equivalent. Auth profile types/config only carry credential identity and routing metadata, auth resolution returns no transport data, and runtime auth request overrides explicitly reject proxy/TLS, so this cannot be closed as implemented or moved to ClawHub without a new or blessed core/plugin transport seam.

## Best Possible Solution

Keep this issue open. The best maintainer path is to design a provider-transport extension that either adds a documented, redacted auth-profile transport/proxy field or an equivalent structured profile setting; flows the selected profile's transport through auth resolution into model request construction; preserves existing provider-level `models.providers.*.request.proxy`; updates schema/help/docs/redaction; and adds regression coverage for multiple Anthropic/OpenAI profiles under one provider using distinct proxies. If maintainers prefer separate custom provider IDs as the intended model, document that explicitly before closing.

## Risks / Open Questions

- A profile-scoped proxy field can contain proxy credentials and changes outbound provider routing, so implementation needs explicit validation, redaction, and likely SecretRef/security design.
- Splitting accounts into separate custom provider IDs with `models.providers.*.request.proxy` may be a workaround, but it is not equivalent to multiple auth profiles under one provider using distinct proxies.
- Because runtime auth proxy/TLS overrides are intentionally rejected today, closing this as ClawHub/plugin work would be premature unless maintainers first add or bless a plugin API for profile-scoped transport.
