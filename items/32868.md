## Summary

Keep open. Current `main` still has the same architectural gap the report describes: same-channel block replies are queued through the dispatcher without awaiting actual delivery before tool-boundary flushes, and `waitForIdle()` is only consulted during end-of-turn finalization.

## Evidence

- **ACP block flush only awaits projector delivery callback:** `createAcpReplyProjector()` wires block streaming to `await params.deliver("block", payload)`, so any pre-tool flush is only as strong as `deliver()` for same-channel sends.
  - file: [src/auto-reply/reply/acp-projector.ts:186](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/auto-reply/reply/acp-projector.ts#L186)
  - command: `nl -ba src/auto-reply/reply/acp-projector.ts | sed -n '180,205p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Same-channel direct block path is still fire-and-forget:** In the non-routed path, `deliver()` calls `params.dispatcher.sendBlockReply(ttsPayload)` and immediately returns the boolean result. There is no `await params.dispatcher.waitForIdle()` before resolving the block delivery promise.
  - file: [src/auto-reply/reply/dispatch-acp-delivery.ts:370](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/auto-reply/reply/dispatch-acp-delivery.ts#L370)
  - command: `nl -ba src/auto-reply/reply/dispatch-acp-delivery.ts | sed -n '350,395p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Dispatcher separates enqueue from delivery completion:** `sendBlockReply` is just `enqueue("block", payload)`, while delivery completion is exposed separately as `waitForIdle()`. That matches the issue's claim that queueing and actual channel delivery are distinct chains.
  - file: [src/auto-reply/reply/reply-dispatcher.ts:271](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/auto-reply/reply/reply-dispatcher.ts#L271)
  - command: `nl -ba src/auto-reply/reply/reply-dispatcher.ts | sed -n '260,278p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Tool-boundary flush does not wait for dispatcher idle:** `onBlockReplyFlush` still only does `await blockReplyPipeline.flush({ force: true })`. Because the projector's block callback resolves after enqueue on same-channel sends, tool execution can continue before actual outbound delivery completes.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:1222](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/auto-reply/reply/agent-runner-execution.ts#L1222)
  - command: `nl -ba src/auto-reply/reply/agent-runner-execution.ts | sed -n '1216,1230p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Dispatcher idle is only awaited at end-of-turn finalization:** ACP finalization calls `await params.delivery.settleVisibleText()`, and that helper is where `await params.dispatcher.waitForIdle()` happens. Separately, ACP turn handling invokes finalization only after `await projector.flush(true)` at the end of the turn.
  - file: [src/auto-reply/reply/dispatch-acp.ts:187](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/auto-reply/reply/dispatch-acp.ts#L187)
