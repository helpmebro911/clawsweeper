## Summary

Current `main` does not implement `old_text`/`new_text` edit-tool aliases. The edit tool now validates only `path` plus a canonical `edits` array, and the runtime parser only accepts `edits[].oldText`/`edits[].newText`. Current tests also explicitly assert that legacy alias-style edit params are rejected, so there is not strong evidence to close this as already implemented or obsolete.

## Evidence

- **Edit validation only accepts canonical `edits` payload:** `REQUIRED_PARAM_GROUPS.edit` requires `path` and an `edits` group validated by `hasValidEditReplacements`; that validator checks only canonical `oldText`/`newText` fields on each edit entry.
  - file: [src/agents/pi-tools.params.ts:66](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/agents/pi-tools.params.ts#L66)
  - command: `nl -ba src/agents/pi-tools.params.ts | sed -n '1,260p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Runtime parser has no `old_text`/`new_text` alias normalization:** `readEditReplacements()` returns edits only when `record.edits` exists and each entry has string `oldText` and `newText`; `readEditToolParams()` reads only `path` plus that parsed `edits` array.
