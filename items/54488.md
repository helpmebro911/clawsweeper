## Summary

Keep open. Current main has useful mitigations around heartbeat skipping and background-capable context maintenance, but it still does not fix the reported starvation path: followup drains can keep running until the queue is empty, each followup performs a full embedded run on the same per-session FIFO lane, and fresh normal inbound messages are not prioritized ahead of queued followup work.

## Best Possible Solution

Keep this issue open and fix it in the core reply/session scheduler. The best path is a bounded/fair followup drain, explicit priority or preemption for fresh user messages over accumulated followup/system-event work on the same session key, a phase-aware aggregate watchdog for queue/preflight/provider/post-turn time, and focused regression tests that enqueue multiple followups followed by a later inbound user message on the same session lane.
