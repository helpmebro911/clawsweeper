## Summary

Current main still has the reported scheduler shape: followup drains run until the followup queue is empty, followup work is appended without priority, each followup executes a normal embedded agent turn, and that turn serializes through the same per-session FIFO lane as fresh inbound work. The linked #65504 and #66638 discussions are adjacent responsiveness/notification work, but neither supersedes this specific followup-drain fairness bug.

## Best Possible Solution

Keep this open and fix it in the core reply/session scheduler. The likely path is a bounded or fair followup drain, explicit priority or preemption for fresh user messages over accumulated followup/system-event work on the same session key, phase-aware aggregate timing for queue/preflight/provider/post-turn work, and a focused regression test that enqueues multiple followup turns plus a later fresh inbound message on the same session key and proves the fresh turn is not starved behind the entire backlog. Keep #65504 and #66638 linked as related responsiveness and notification-delivery work, but do not treat them as superseding this scheduler fairness bug.
