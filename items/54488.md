## Summary

Keep this open. Current main still has the reported fairness problem: followup drains run the queued backlog to empty, new busy-session inbound messages are appended to the same followup queue, and every followup runs a full embedded agent turn through the same per-session FIFO lane. Adjacent issues #65504 and #66638 are related responsiveness/notification work, but they do not supersede this specific followup-drain starvation bug.

## Best Possible Solution

Keep this issue open and fix it in the core auto-reply/session scheduler. The likely product/code direction is a bounded or fair followup drain, explicit priority/yielding for fresh user messages over accumulated followup/system-event work on the same session key, phase-aware aggregate timing for queue/preflight/provider/post-turn work, and a focused regression test that enqueues multiple followups plus a later fresh inbound message on the same session key and proves the fresh turn is not starved behind the entire backlog. Keep #65504 and #66638 linked as related work, but do not use them as the canonical replacement for this scheduler fairness bug.
