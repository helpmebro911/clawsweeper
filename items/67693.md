## Summary

Keep this PR open. Current main still forwards wizard select and multiselect labels directly to Clack and has no regression coverage for blank labels. The related onboarding crash work covers channel metadata/docs-link/text-input trim paths, not this prompt-adapter label fallback. The PR is narrow and not security-sensitive, but it still needs normal maintainer review for the exact crash evidence and searchable prompt test coverage.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is a narrow prompt-adapter helper in src/wizard/clack-prompter.ts that normalizes blank select, searchable select, multiselect, and searchable multiselect labels to a safe value-derived fallback, with focused unit coverage and either a precise crash reproduction or corrected wording that describes the change as defensive display hardening.

## Risks / Open Questions

- The PR title/body frame this as a crash, but @clack/prompts 1.2.0 visibly treats a blank string label as present; maintainers should ask for an exact crash reproduction or reframe this as defensive fallback/display hardening if no direct crash path exists.
- The current PR patch does not mock the imported autocomplete export while importing createClackPrompter, and it lacks explicit searchable select coverage; add an autocomplete mock and searchable-path assertions before landing.
- Current channel setup selection assembly already sanitizes channel labels before reaching the prompter, so the remaining malformed-label source should be validated if this is merged as a bug fix rather than generic adapter hardening.
