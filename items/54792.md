## Summary

Keep open. Current main has a runtime/startup backstop and user-facing docs for `hooks.enabled=true` requiring `hooks.token`, but the requested shared schema/config-write validation is still missing. The generic config validators and write paths still do not reject `hooks.enabled: true` with no non-empty token before persistence.

## Best Possible Solution

Keep this issue open. Add a shared cross-field config validation rule, preferably in the raw validation layer used by `validateConfigObjectRaw` and the plugin-aware wrappers, that rejects `hooks.enabled=true` unless `hooks.token` normalizes to a non-empty string. That should make `config.set`, `config.apply`, `config.patch`, `writeConfigFile`, CLI dry-run/write paths, and plugin/runtime config writes fail before persistence. Keep `resolveHooksConfig` as the startup/reload backstop, and extend `config.schema.lookup` or UI hints with dependency/warning metadata if clients are expected to preflight this before edits.
