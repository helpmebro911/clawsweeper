## Summary

Keep open. Current main has a runtime/startup backstop and docs for `hooks.enabled` requiring `hooks.token`, but the shared schema/config-write validation requested here is still missing, so `config.patch`/`config.apply`/`config.set` can still persist the invalid cross-field state before restart.

## Best Possible Solution

Keep this issue open. Add a shared cross-field config validation rule that rejects `hooks.enabled=true` unless `hooks.token` is a non-empty string, route it through `validateConfigObjectRaw` / `validateConfigObjectWithPlugins` so `config.set`, `config.apply`, `config.patch`, `writeConfigFile`, CLI writes, and plugin runtime config writes all reject the invalid config before persistence, and keep `resolveHooksConfig` as the startup/reload backstop. If clients rely on `config.schema.lookup`, add dependency or warning metadata for `hooks.enabled` there too.

## Risks / Open Questions

- Adding a hooks-only check only inside one Gateway RPC handler would leave CLI writes, direct persistence, and plugin/runtime config writes inconsistent.
- A broad dependency metadata framework may be larger than needed for the immediate outage path; the minimal useful fix is a shared cross-field validator plus optional schema lookup metadata.
