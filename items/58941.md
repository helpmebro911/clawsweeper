## Summary

Current `main` does not show this issue as fixed. The Discord approval path still requires explicit `channels.discord.execApprovals.enabled` and still does not infer approvers from Discord `allowFrom`/`dm.allowFrom`, so an older working config can still lose Discord exec approvals after upgrade. The latest docs also conflict with the implemented behavior, which argues against closing this as obsolete or resolved.

## Evidence

- **Core approval helper still requires explicit enablement:** `isChannelExecApprovalClientEnabledFromConfig` returns true only for `enabled === true` or `enabled === "auto"`; omitted `enabled` stays off even when approvers exist.
  - file: [src/plugin-sdk/approval-client-helpers.ts:40](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/plugin-sdk/approval-client-helpers.ts#L40)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Discord regression is encoded in current tests:** Current Discord tests assert that exec approvals stay disabled when `enabled` is omitted, even with explicit approvers or `commands.ownerAllowFrom`, and also assert that Discord does not infer approvers from `allowFrom`, legacy `dm.allowFrom`, or `defaultTo`.
  - file: [extensions/discord/src/exec-approvals.test.ts:25](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/discord/src/exec-approvals.test.ts#L25)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **User-facing fallback text still directs users to turn the feature on explicitly:** The Discord approval setup text says to configure approvers or `commands.ownerAllowFrom` and set `channels.discord.execApprovals.enabled` to `auto` or `true`, which matches the reported workaround rather than indicating a migration/fix for older configs.
  - file: [extensions/discord/src/approval-native.ts:183](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/discord/src/approval-native.ts#L183)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Docs still claim auto-enable on unset config:** The Discord docs say native exec approvals auto-enable when `enabled` is unset or `"auto"`, but that does not match the current helper/test behavior above.
  - file: [docs/channels/discord.md:891](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/channels/discord.md#L891)
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Latest release still carries the same behavior:** Inspecting `v2026.4.23` shows the same `isChannelExecApprovalClientEnabledFromConfig` logic (`enabled === true || enabled === "auto"`), so this is not a main-only fix waiting to ship.
  - command: `git show v2026.4.23:src/plugin-sdk/approval-client-helpers.ts | sed -n '35,55p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Issue discussion matches the implemented workaround:** The 2026-04-04 comment reports that explicitly setting `channels.discord.execApprovals.enabled true` and `channels.discord.execApprovals.approvers` restores Discord approvals on 2026.4.2, which lines up with the current code path rather than a shipped migration or automatic compatibility fix.

## Risks / Open Questions

- Closing this would hide a still-real upgrade/migration problem for existing Discord installs.
- Docs currently describe different behavior from the code, so users can still be misled even on current builds.
- A separate Discord crash in #58760 may be distinct, but it does not negate the approval-path regression described here.
