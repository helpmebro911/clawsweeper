## Summary

Keep this issue open. Current main still queues or serializes ordinary same-session messages while a run is active, but it does not send the requested immediate channel-visible busy/waiting reply with optional cancel/new-session guidance. The reporter's Telegram-side processing-indicator ticket is useful context, and the prior Codex review comment correctly identified the same remaining OpenClaw UX gap.

## Best Possible Solution

Keep this issue open. The best fix belongs in the generic active-session queue/runner path around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent`: add a configurable channel-safe busy/status reply for ordinary messages that arrive during an active run, include optional `/stop` and `/new` guidance, preserve the existing queue modes, and add focused runner plus channel-level regression tests.

## Risks / Open Questions

- A future implementation must preserve collect, followup, steer, steer-backlog, and interrupt semantics so a fast busy/status reply does not double-reply, drop queued content, or break followup drains.
- Existing `/stop`, `/new`, `/queue interrupt`, typing indicators, and internal queued-followup prompts are adjacent features, but treating them as the requested implementation would leave ordinary chat users without the immediate busy explanation requested here.
