## Summary

Current main still does not implement the requested immediate, channel-visible busy/waiting reply for ordinary messages sent while the same session has an active long-running run. The existing behavior enqueues or serializes follow-up work, exposes adjacent `/stop` and `/new` commands, and has internal busy/status tracking, but none of those paths send the scripted busy reply with cancel/new-session guidance requested by this issue.

## Best Possible Solution

Keep this issue open and implement a generic active-session busy/status path in the auto-reply queue policy and runner. The likely change belongs around `resolveActiveRunQueueAction` and `runReplyAgent`: return a configurable channel-safe busy payload while preserving queued follow-up behavior, then add focused runner tests plus at least one channel-level test proving an ordinary message sent during an active turn receives the busy/status reply and still preserves intended follow-up handling.

## Risks / Open Questions

- A future implementation must preserve `collect`, `followup`, `steer`, and `steer-backlog` semantics so a fast busy/status reply does not double-reply, drop queued content, or break follow-up drains.
- Channel status indicators and `/stop`/`/new` commands are adjacent behavior but not equivalent to the requested immediate busy reply; treating them as already sufficient would leave the reported UX gap.
