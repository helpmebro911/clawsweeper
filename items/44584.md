## Summary

Current `main` appears to have addressed the thread-context duplication side of the report, but I did not find strong evidence that the CLI `openclaw agent --channel discord --to ...` path now resolves Discord thread-bound sessions correctly. The session-resolution code still derives the session key from `to` alone and does not use Discord target normalization or `channel`/`accountId` context in that lookup, so this issue should stay open.

## Evidence

- **Follow-up turns suppress repeated thread context:** `runPreparedReply` now strips `ThreadStarterBody` on non-new sessions and also drops `InboundHistory`/`ThreadStarterBody` from the inbound user-context prelude when `ThreadHistoryBody` is available, which matches the issue's metadata re-injection complaint.
  - file: [src/auto-reply/reply/get-reply-run.ts:298](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/auto-reply/reply/get-reply-run.ts#L298)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '260,520p'`
- **Regression tests cover non-duplicated thread context:** Tests assert follow-up prompts prefer `[Thread history - for context]` and do not repeat `[Thread starter - for context]` when history/plain-text prelude is present.
  - file: [src/auto-reply/reply/get-reply-run.media-only.test.ts:328](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/auto-reply/reply/get-reply-run.media-only.test.ts#L328)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.media-only.test.ts | sed -n '300,420p'`
- **CLI/gateway session resolution still ignores channel/account context:** Both CLI entrypoints resolve `sessionKey` from `resolveSessionKeyForRequest({ cfg, agentId, to, sessionId })` / `resolveSession({ cfg, to, sessionId, sessionKey, agentId })`; neither passes `channel`, `replyAccountId`, or normalized Discord target context into session-key lookup.
  - file: [src/commands/agent-via-gateway.ts:113](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/commands/agent-via-gateway.ts#L113)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '1,240p'`
- **Underlying resolver only builds context from `to` and falls back to main:** `resolveSessionKeyForRequest` constructs `ctx` as `{ From: opts.to }` only. `resolveGroupSessionKey` needs a provider/group hint to derive a channel session, and `resolveSessionKey` otherwise collapses to `agent:main:main`. Discord has a target normalizer (`normalizeDiscordMessagingTarget`) for bare IDs, but this resolver path does not use it.
  - file: [src/agents/command/session.ts:130](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/agents/command/session.ts#L130)
