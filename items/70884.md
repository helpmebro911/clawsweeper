## Summary

Keep PR #70884 open. The PR documents real current gateway behavior: device-less trusted-proxy/token/password WebSocket connects can have self-declared scopes cleared, while trusted HTTP requests can honor `x-openclaw-scopes`. Current main still does not explain the WebSocket scope-clearing consequence or explicitly limit the header guidance to HTTP API endpoints, and the PR's review feedback appears to have been addressed in follow-up commits. No conservative close reason fits.

## Best Possible Solution

Keep the PR open as a focused docs-only fix. The best maintainer path is to review and merge revised wording in `docs/gateway/trusted-proxy-auth.md` and `docs/gateway/protocol.md` that explicitly says `x-openclaw-scopes` is an HTTP API header, explains WebSocket scope clearing without device identity, preserves the `allowInsecureAuth` exception, and frames `dangerouslyDisableDeviceAuth` as critical break-glass guidance.
