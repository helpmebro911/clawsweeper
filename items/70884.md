## Summary

Keep PR #70884 open. Current main still has the documentation gap this PR targets: trusted-proxy docs describe `x-openclaw-scopes` as an operator scopes header without saying it is HTTP-only, and the protocol docs list device-less trusted-proxy Control UI as an exception without documenting the scope-clearing consequence. The runtime and tests support the requested clarification, including the local `allowInsecureAuth` preservation exception and the critical nature of `dangerouslyDisableDeviceAuth`.

## Best Possible Solution

Keep #70884 open and land a focused docs clarification after final maintainer review. The docs should state that `x-openclaw-scopes` is an HTTP API header, explain device-less WebSocket scope clearing and the resulting `missing scope` symptom, include the local `allowInsecureAuth` token/password exception, and frame `dangerouslyDisableDeviceAuth` only as a critical break-glass workaround.
