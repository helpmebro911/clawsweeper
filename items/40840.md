## Summary

Keep open. Current main has stronger active-run cleanup than the reporter's 2026.3.7 baseline, but it still does not provide a guaranteed hard-kill/delete boundary for normal `agent` lifecycle writers. The remaining gap is core Gateway/session-store behavior, not ClawHub scope or an already-shipped duplicate.

## Best Possible Solution

Keep this open and fix it in core Gateway/session-store semantics. A good implementation would add a regression test that simulates a normal `agent` run holding a stale session-store snapshot, calls `sessions.delete`, and proves the late `updateSessionStoreAfterAgentRun` cannot resurrect the deleted key. The code fix should add a per-session deletion generation/tombstone or equivalent write fence, and/or make `sessions.delete` fail while any tracked normal agent finalizer for that key can still persist state. Then document whether `sessions.delete` is a hard-kill boundary or introduce a separate native `sessions.kill` primitive with that guarantee.

## Evidence

- **Issue context:** The report is concrete and recent: Builderz Mission Control observed `sessions.delete` returning success and archiving the transcript, followed by the same `sessionKey` being written back after a normal `gateway call agent` run. The related #12297 feature request is closed, but this report narrows the remaining problem to same-key re-registration after normal agent lifecycle writes.
- **Delete removes the row but records no hard-delete barrier:** `sessions.delete` calls cleanup, deletes the current store row, archives transcripts, emits lifecycle hooks, and returns `{ ok: true, deleted }`. There is no tombstone, generation token, or write barrier that later session-store writers must honor.
