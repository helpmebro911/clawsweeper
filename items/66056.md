## Summary

Keep this issue open. Current main still hardcodes user-facing runtime and sanitizer failure text in English, the `messages` config contract has no locale/errorLocale field, Control UI i18n is browser-scoped, and the related pairing-locale PR was closed unmerged.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely path is a small core user-facing error catalog with English fallback, a documented `messages.locale` or `messages.errorLocale` config field wired through types, zod schema, generated config help/docs, and focused tests for `sanitizeUserFacingText` plus runner fallback paths. If maintainers want localization delegated to plugins, core should first expose structured error keys and locale context to hooks or a formatter seam.
