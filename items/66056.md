## Summary

Keep open. Current main still lacks the requested locale-aware user-facing error system: runtime failure paths and the sanitizer emit hardcoded English, `messages` config/schema/help has no locale field, Control UI i18n is browser-local only, and the related pairing-locale PR #66055 was closed unmerged. A plugin-only close would be premature because the available message hook exposes only string content, not structured error keys or locale context.

## Best Possible Solution

Keep this issue open for maintainer product/API design. The likely core path is a small keyed user-facing error catalog with English fallback, a documented `messages.locale` or `messages.errorLocale` config/schema/help surface, locale-aware formatting wired through sanitizer and reply-runner failure paths, and focused regression tests. If maintainers prefer plugin-owned localization, first add a structured error-key plus locale-context hook so plugins can translate without matching English literals.
