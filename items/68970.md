## Summary

Keep open. Current main has adjacent improvements for TUI startup history retries and restart-continuation durability, but it does not implement the specific TUI reconnect/event-gap backfill requested by #68970. The Web UI actively reconnects on a sequence gap, while the TUI only updates the status line for `onGap`; a missed post-restart final event still has no TUI-specific forced reload/replay path.

## Best Possible Solution

Keep #68970 open and implement a TUI-specific recovery path: on event gap and/or immediately after gateway reconnect, force a guarded `loadHistory()` for the current session, ideally with a short delayed second reload for restart-continuation timing. Add a focused TUI test that simulates reconnect history loading before the recovery reply, then a missed/gapped final event, and verifies the reply is backfilled into the TUI just as Web UI can recover via its seq-gap reconnect path.
