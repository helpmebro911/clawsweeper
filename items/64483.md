## Summary

Current main still appears to have the reported provider-precedence bug. `modelstudio` is globally normalized to `qwen`, the bundled Qwen plugin still treats `modelstudio` as a compatibility alias, and built-in Qwen suppression still runs before the configured inline/custom provider fallback can resolve the reported `modelstudio/qwen3.6-plus` configuration. The external setup/workaround comment is useful context, but it does not close the core precedence issue.

## Best Possible Solution

Keep this issue open and fix the provider-precedence contract. A conservative path is to let an exact user-configured provider id such as `models.providers.modelstudio` take precedence for explicit `modelstudio/...` model refs, limit Qwen `modelstudio` aliasing to legacy compatibility when no explicit custom provider owns that id, move custom configured-provider resolution ahead of Qwen built-in suppression for this case, and add a regression test for `modelstudio/qwen3.6-plus` with `api: "openai-completions"` and the Coding Plan base URL. Update the public Qwen docs once the compatibility rule is settled.
