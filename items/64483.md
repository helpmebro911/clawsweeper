## Summary

Keep open. Current `main` still treats `modelstudio` as a Qwen alias across provider normalization, provider discovery, and the Qwen plugin itself, so a distinct custom `models.providers.modelstudio` entry is not preserved as a separate provider. The latest release `v2026.4.23` still contains the same behavior.

## Evidence

- **Qwen plugin still declares modelstudio as legacy provider id and alias:** `extensions/qwen/index.ts` hard-codes `LEGACY_PROVIDER_ID = "modelstudio"`, accepts `aliases: ["modelstudio", "qwencloud"]`, and reads configured base URLs from either `qwen` or `modelstudio`, so the bundled plugin still intentionally owns both ids.
  - file: [extensions/qwen/index.ts:16](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/qwen/index.ts#L16)
  - command: `nl -ba extensions/qwen/index.ts | sed -n '1,260p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Global provider normalization still collapses modelstudio into qwen:** `normalizeProviderId()` maps `modelstudio` to `qwen`, and `findNormalizedProviderValue()` compares provider configs through that normalization. A custom `models.providers.modelstudio` entry therefore resolves as the same provider key as `qwen` rather than a distinct provider.
  - file: [src/agents/provider-id.ts:3](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/provider-id.ts#L3)

  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Provider discovery writes alias-owned catalog results under the normalized key:** When a plugin returns a single provider catalog result, `normalizePluginDiscoveryResult()` emits entries for the provider id and all aliases, but normalizes each key first. For Qwen, `modelstudio` normalizes to `qwen`, so discovery does not preserve a separate `modelstudio` provider namespace.
  - file: [src/plugins/provider-discovery.ts:80](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/plugins/provider-discovery.ts#L80)

- **Tests currently lock in modelstudio-owned Qwen discovery:** The shared discovery contract for the Qwen plugin still has a `"modelstudio provider discovery contract"` case and expects a `models.providers.modelstudio.baseUrl` config to produce the bundled Qwen catalog. That is consistent with the reported conflict, not a fix.
  - file: [test/helpers/plugins/provider-discovery-contract.ts:527](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/test/helpers/plugins/provider-discovery-contract.ts#L527)

That matches the current implementation and does not document a separate custom-provider escape hatch.
