## Summary

Keep open. Current main still embeds BOOT.md verbatim into an ordinary agent prompt, sends fallback attempts through the same prompt body, and does not provide a boot-specific guard that can suppress literal BOOT.md echoes through the message tool while preserving legitimate BOOT.md outbound sends.

## Best Possible Solution

Keep this open. Implement a narrow internal boot-run context that tags BOOT.md executions, detects and suppresses substantial literal echoes of the generated BOOT.md prompt before both final reply projection and message-tool delivery, and add deterministic tests for primary-model failure followed by fallback-model echo while preserving legitimate BOOT.md-requested sends.

## Risks / Open Questions

- The symptom is model-dependent, so regression coverage should simulate a primary failure followed by a fallback model echoing the BOOT.md prompt rather than rely on live Gemini behavior.
- A blanket message-tool disable for BOOT.md would break documented startup notifications; the fix needs to distinguish legitimate BOOT.md-requested outbound messages from substantial literal prompt echoes.
