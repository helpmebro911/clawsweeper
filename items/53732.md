## Summary

Keep this issue open. Current main still runs BOOT.md by embedding its contents directly in an agent prompt and suppresses the normal final reply path with `deliver: false`, but I did not find a boot-internal marker or literal-echo guard that would prevent a fallback model from sending echoed BOOT.md text through the message tool.

## Best Possible Solution

Keep this issue open and add a narrow internal boot-md guard. Mark boot-md runs as internal, suppress or reject substantial literal echoes of the generated boot prompt before both final reply projection and message-tool delivery, and add regression coverage where the primary fails, a fallback returns or sends BOOT.md text verbatim, and no user-visible payload is emitted while legitimate BOOT.md message-tool sends still work.
