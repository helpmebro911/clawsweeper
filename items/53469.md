## Summary

Keep this issue open. Current main has partial private-network support: SearXNG web_search can use private/loopback SearXNG endpoints, and model provider request transport supports allowPrivateNetwork. The central request is still missing for generic web_fetch and Gateway OpenResponses URL inputs, and image/media tools only inherit the currently narrow web_fetch RFC2544 policy rather than a private-network opt-in.

## Best Possible Solution

Keep this issue open and scope the remaining work around a narrow explicit private-network opt-in for tools.web.fetch.ssrfPolicy, remote media/image references that inherit that policy, and Gateway OpenResponses URL inputs such as gateway.http.endpoints.responses.files/images. Preserve default blocking, add schema/help/docs, decide separately whether gateway.mode=local should auto-enable anything, and cover web_fetch, redirects, Gateway URL inputs, cache behavior, and image/media reference handling with SSRF tests. Treat SearXNG web_search and models.providers[].request.allowPrivateNetwork as partial shipped precedent, not full resolution.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should remain default-off unless maintainers explicitly approve a local-mode auto-enable rule.
- web_fetch cache keys must include any private-network policy bit so denied and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, explicit proxies, hostname allowlists, Gateway input_file/input_image URL fetching, and remote image/media references need regression coverage for both denied-by-default and explicit opt-in behavior.
