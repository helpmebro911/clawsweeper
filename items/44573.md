## Summary

Keep open. Current `main` still computes a per-run cron session key, but isolated cron execution and descendant-run follow-up logic continue to use the stable job-scoped `agentSessionKey`, so the PR's requested fix is not implemented on `38caa6832d4e1d344736d635b0c5fdad37e4430b`.

## Evidence

- **Per-run key exists but is only attached to returned run metadata:** `prepareCronRunContext` builds `runSessionKey` as `agentSessionKey:run:<sessionId>` and returns it via `withRunSession`, which matches the PR description of persistence/result bookkeeping already being per-run.
  - file: [src/cron/isolated-agent/run.ts:492](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/cron/isolated-agent/run.ts#L492)
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Execution still runs on the stable session lane:** `createCronPromptExecutor` passes `params.agentSessionKey` into both `runCliAgent` and `runEmbeddedPiAgent` instead of a per-run key, so repeated isolated cron runs still share the stable execution lane.
  - file: [src/cron/isolated-agent/run-executor.ts:127](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/cron/isolated-agent/run-executor.ts#L127)
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Top-level cron runner never forwards `runSessionKey` into execution:** `runCronIsolatedAgentTurn` calls `executeCronRun` with `prepared.context.agentSessionKey`; there is no `runSessionKey`/`executionSessionKey` parameter on the execution path.
  - file: [src/cron/isolated-agent/run.ts:952](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/cron/isolated-agent/run.ts#L952)
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Descendant accounting is still keyed to the stable lane:** Interim-ack retry and delivery follow-up both query descendant runs with `params.agentSessionKey`, so subagent tracking has not been moved to a run-scoped requester key either.
  - file: [src/cron/isolated-agent/run-executor.ts:367](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/cron/isolated-agent/run-executor.ts#L367)
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
- **Current tests still encode the old execution behavior:** `isolated-agent.session-identity` expects the embedded runner call to use `agent:ops:cron:job-ops`, while separately expecting returned `res.sessionKey` values to be per-run `...:run:<id>`, confirming the mismatch remains on `main`.
  - file: [src/cron/isolated-agent.session-identity.test.ts:100](https://github.com/openclaw/openclaw/blob/38caa6832d4e1d344736d635b0c5fdad37e4430b/src/cron/isolated-agent.session-identity.test.ts#L100)
  - sha: [38caa6832d4e](https://github.com/openclaw/openclaw/commit/38caa6832d4e1d344736d635b0c5fdad37e4430b)
