## Summary

Keep open. Current main implements important parts of the request: atomic JSON writes, cross-agent same-provider/profile refresh serialization, refresh-token-reused recovery, and bootstrap-only Codex CLI behavior. The remaining hardening is still actionable because OAuth-producing auth writes still use the unlocked upsert path, and duplicate enrollment protection is still keyed to provider/profileId rather than a stable enrollment or documented single-writer boundary.

## Best Possible Solution

Keep this issue open and narrow it to the remaining hardening: route OAuth-producing auth writes through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, or make the default OAuth upsert path lock-safe without breaking the public SDK contract; add practical duplicate-enrollment detection within one store where feasible; and document the operator boundary for copied auth-profiles.json files, multiple hosts/stores, and Codex CLI coexistence. Preserve the shipped atomic writes, same-profile refresh serialization, refresh-token-reused recovery, and Codex bootstrap-only behavior.
