## Summary

Keep open. Current main has implemented important hardening pieces, but the central request is not fully done: OAuth-producing auth writes still use the unlocked upsert path, and duplicate enrollment protection remains scoped to provider/profile/store coordination rather than a stable enrollment or copied-store/single-writer boundary.

## Best Possible Solution

Keep this issue open and narrow it to the remaining hardening: route OAuth-producing writes such as writeOAuthCredentials and provider auth-result persistence through updateAuthProfileStoreWithLock or a lock-safe OAuth-specific upsert; add feasible duplicate-enrollment detection within one store using stable identity metadata when present; and document the single-writer boundary for copied auth-profiles.json files, multiple hosts/stores, and Codex CLI coexistence. Preserve the shipped atomic JSON writes, Codex bootstrap-only behavior, same-profile refresh serialization, and refresh_token_reused recovery.

## Risks / Open Questions

- Closing now would drop tracking for the remaining unlocked OAuth credential mutation paths that can overwrite fresher auth-store state during concurrent login/setup/refresh activity.
- The current refresh lock prevents same-provider/profile refresh storms, but it cannot detect the same OAuth enrollment copied under another profileId, another auth-profiles.json path, or another host.
- A fix must account for public plugin SDK compatibility because upsertAuthProfile and writeOAuthCredentials remain exported surfaces used by provider plugins.
