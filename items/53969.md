## Summary

Keep open. Current main has shipped several important pieces of the request, including atomic JSON writes, per-provider/profile OAuth refresh serialization, refresh-token-reused recovery paths, and Codex CLI bootstrap-only behavior. The remaining request is still actionable: provider OAuth login paths still persist through an unlocked upsert, and there is no duplicate-enrollment guard for the same refresh-token chain under different profile IDs, stores, or hosts.

## Best Possible Solution

Keep this issue open and narrow it to the remaining hardening: route OAuth-producing auth mutations such as models auth login and writeOAuthCredentials through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, add a practical duplicate-enrollment guard or stable-identity refresh-chain strategy where feasible, and document the single-writer boundary for auth-profiles.json and Codex CLI coexistence. Preserve the already-shipped atomic writes, same-profile refresh serialization, refresh-token-reused recovery, and Codex bootstrap-only behavior.
