## Summary

Keep open. Current `main` still rejects the reported custom redaction regex and the session-persistence path still does not apply redaction before writing messages. The PR discussion also surfaced an unresolved `toolCall.arguments` leak in the proposed patch, so this is not conservative auto-close material.

## Evidence

- **Current tokenizer still rejects the sample safe regex:** On `main`, `tokenizePattern` handles `\\` before checking `inCharClass`, so escaped tokens inside `[...]` are counted as standalone tokens. Reproduced with `node --input-type=module -e 'import { compileSafeRegex } from "./src/security/safe-regex.ts"; const pattern="([\\\\w]|[-.])+@([\\\\w]|[-.])+\\\\.\\\\w+"; const re=compileSafeRegex(pattern,"gi"); console.log(re===null?"NULL":re.toString());'`, which printed `NULL`.
  - file: [src/security/safe-regex.ts:143](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/security/safe-regex.ts#L143)

  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
