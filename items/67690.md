## Summary

Keep open. Current main has useful partial coverage for context awareness and cache-TTL tool-result pruning, but it still does not implement the core request for pruning visibility/control: pruning silently rewrites the in-memory prompt, returns no pruning metadata, emits no pruning hook/event, and the Control UI only shows high context pressure plus compaction status, not what pruning removed or when pruning happened. Related open reports #8606 and #9409 cover narrower missing pieces, but they do not supersede the broader UI/control request.

## Best Possible Solution

Keep this issue open and narrow it into a maintainer-approved pruning observability/control design. The likely implementation belongs around `src/agents/pi-hooks/context-pruning/*` plus the agent event/UI surfaces: have the pruner return safe metadata, emit an internal/plugin-visible pruning lifecycle event, surface a concise Control UI/chat notice when pruning happens, document the new and existing knobs, and coordinate the hook shape with #8606 and overflow diagnostics with #9409.
