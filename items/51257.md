## Summary

Keep open. Current `main` still does not implement this feature, and the PR's current `displayName ?? label ?? key` approach still has unresolved usability and terminal-safety issues.

## Evidence

- **CLI status still shows raw session keys:** `buildStatusSessionsRows()` still renders `Key: params.shortenText(sess.key, 32)`, and `SessionStatus` on `main` has no `label`/`displayName` fields to carry friendlier names through the status pipeline.
  - file: [src/commands/status.command-sections.ts:309](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/status.command-sections.ts#L309)
  - command: `nl -ba src/commands/status.command-sections.ts | sed -n '296,320p'; nl -ba src/commands/status.types.ts | sed -n '1,80p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **TUI status summary also still uses only the raw key:** `formatStatusSummary()` prints `entry.key`, and `GatewayStatusSummary.sessions.recent` does not currently expose `label` or `displayName` on `main`.
  - file: [src/tui/tui-status-summary.ts:79](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/tui/tui-status-summary.ts#L79)
  - command: `nl -ba src/tui/tui-status-summary.ts | sed -n '65,95p'; nl -ba src/tui/tui-types.ts | sed -n '92,108p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Display names are not valid session selectors:** Gateway resolution accepts only `key`, `sessionId`, or `label`; there is no `displayName` selector in either the resolver or protocol schema, so showing `displayName` in the main status identifier would not match follow-up command inputs.
  - file: [src/gateway/sessions-resolve.ts:79](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/sessions-resolve.ts#L79)
  - command: `nl -ba src/gateway/sessions-resolve.ts | sed -n '79,99p'; nl -ba src/gateway/sessions-resolve.ts | sed -n '150,218p'; nl -ba src/gateway/protocol/schema/sessions.ts | sed -n '71,82p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Friendly names are untrusted text and need sanitization:** `displayName` can come from thread/group metadata, while terminal-safe rendering elsewhere uses `sanitizeTerminalText()` to strip ANSI/control characters and escape newlines. The status paths under review do not apply that sanitizer.
  - file: [src/auto-reply/reply/session.ts:669](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/auto-reply/reply/session.ts#L669)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '669,672p'; nl -ba src/config/sessions/metadata.ts | sed -n '166,176p'; nl -ba src/terminal/safe-text.ts | sed -n '1,20p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)

## Risks / Open Questions

- The feature is not implemented on current `main`, so closing as already done would be incorrect.
- The PR diff is stale against current status rendering code: CLI session rows now flow through `buildStatusSessionsRows()` in `src/commands/status.command-sections.ts`, not the older inline table mapping in `status.command.ts`.
- Merging the PR unchanged would still leave two open concerns from review: `displayName` is not a resolvable selector for follow-up commands, and `label`/`displayName` are not sanitized before terminal rendering.
