## Summary

Current main still does not implement the requested opt-in session wall-clock duration cap or cumulative per-session token cap. The existing surfaces cover daily/idle reset freshness, parent-fork context guards, per-cron-run timeouts, context-window usage snapshots, and session disk maintenance, but none enforce `session.maxDurationMinutes` or `session.maxTokensPerSession` for active runaway sessions. The discussion includes a useful implementation sketch and an external Helm documentation workaround, but no shipped OpenClaw core behavior or canonical merged replacement.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and cumulative usage-budget state, enforce before costly LLM/tool work and on lifecycle checks, emit deterministic user/operator feedback, and add focused regression tests for normal chat, cron/heartbeat, provider-owned CLI sessions, compaction/reset interactions, and budget-exceeded actions.

## Risks / Open Questions

- Budget-exceeded behavior still needs maintainer product/API judgment: stop, warn, compact, reset, or channel/runtime-specific fallback semantics.
- A correct cumulative token cap likely needs explicit per-session usage accounting because current `totalTokens` is deliberately a context snapshot, not cumulative spend.
- Cron, heartbeat, always-on, and provider-owned CLI session paths need explicit coverage because existing per-run timeouts and daily/idle freshness do not cap active runaway sessions.
