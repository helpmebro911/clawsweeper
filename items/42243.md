## Summary

Keep open. Current main still turns fatal isolated cron failures into `isError: true` delivery payloads and passes them through the normal announce delivery path. The raw JSON text may now be sanitized into safer error text, but failed announce runs can still post a user-facing channel message rather than only recording the failure and using explicit failure-notification paths.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron run or dispatch boundary so fatal/error run outcomes do not perform normal `delivery.mode: "announce"` delivery of synthesized `isError` payloads, while still recording `lastRunStatus: "error"`, preserving error counters/backoff/run history, and leaving explicit failureAlert/failureDestination notifications as the only channel-visible failure path. Add regression coverage for announce mode with `bestEffort: false`, run-level provider errors, structured `isError` payloads, `lastRunStatus: "error"`, and no outbound delivery attempt.

## Risks / Open Questions

- Current main appears to reduce raw JSON exposure by formatting some provider failures into safer text, so the narrow raw-JSON symptom may be partially mitigated while the broader failed-run announce behavior remains.
- There is a product-policy distinction to preserve: normal fatal run output should be suppressed, while explicit cron failure alerts or configured failure destinations may still intentionally send sanitized notifications.
