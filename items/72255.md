## Summary

Keep open. Current main still has a plausible credential-selection path where stored per-agent auth profiles can be used even when they are no longer declared in `openclaw.json`, and I did not find a runtime warning, doctor warning, or strict allowlist that would close the reported Pi fallback billing/quota issue.

## Best Possible Solution

Keep the issue open and implement a bounded auth-selection change: make `auth.profiles` declarations authoritative for runtime selection when a provider has declared profiles, or add an explicit allowlist passed into `resolveAuthProfileOrder`/Pi auth resolution. Preserve only narrowly proven legacy drift repairs, and add `openclaw doctor`/`models status` visibility for stored profile ids present in `auth-profiles.json` but absent from `openclaw.json`. Add a regression test for an undeclared `anthropic:*` token in the store after the config declaration is removed.

## Evidence

That can reintroduce a stored same-provider profile that is not declared in current config.

## Risks / Open Questions

- Changing this needs care because the current stored-profile fallback is intentionally covering legacy config/profile-id drift, especially OAuth identity migration. A fix should preserve known migration repairs while preventing removed profile declarations from silently becoming usable again.
- The exact live billing behavior depends on runtime auth availability and provider credentials, but source inspection is enough to show the requested strict/loud behavior is not implemented on current main.
