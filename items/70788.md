## Summary

Keep open. The reported Startup-folder login item behavior still exists on current main: OpenClaw generates `OpenClaw Gateway.cmd` with `start "" /min cmd.exe /d /c`, writes only that `.cmd` fallback item, and tests explicitly assert the old launcher. The related closed #57682 covers a different update/restart path and does not supersede this startup-folder issue.

## Best Possible Solution

Keep #70788 open and fix the Windows Startup-folder fallback in `src/daemon/schtasks.ts`: generate a truly hidden login launcher, preserve the existing `.cmd` detection surface or update detection deliberately, remove any sidecar on uninstall, and update `src/daemon/schtasks.startup-fallback.test.ts` plus Windows gateway docs. The linked #57682 should remain treated as related background, not the canonical tracker for this startup-login bug.
