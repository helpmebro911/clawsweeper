## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants `userTokenReadOnly: false` to make bot token primary for Slack reads, rebase the change into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path and bot-denied fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close this PR manually with that product/security rationale and document the intended workaround or a separate explicit config for bot-token read preference.

- A simple `botToken ?? userToken` preference can regress workspaces where a configured bot token lacks membership or scopes for a channel that the user token could read.
- The PR cannot land as-is because it modifies pre-plugin Slack action files that are absent from current main.
