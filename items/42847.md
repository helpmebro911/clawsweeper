## Summary

Keep this PR open. Current main still prefers the Slack user token for read operations whenever it is configured, so the proposed behavior is not implemented. The linked #42752 is closed and this PR remains the canonical implementation candidate, but it needs maintainer review because the Slack code has moved into the plugin path and the bot-token read preference has unresolved security and fallback-policy concerns.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants `userTokenReadOnly: false` to make the bot token primary for Slack reads, rebase the implementation into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path and bot-denied fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close this PR manually with that product/security rationale and document the intended workaround or add a separate explicit config for bot-token read preference.

## Risks / Open Questions

- Switching Slack reads from user-token-first to bot-token-first can broaden access for tool-supplied channel, thread, and file targets unless maintainers explicitly accept and document that policy.
- A simple `botToken ?? userToken` read preference can regress setups where a configured bot token lacks channel membership or read scopes while the user token could read successfully.
- The PR cannot land as-is because it targets the old pre-plugin Slack action path.
