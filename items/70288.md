## Summary

Keep open. Current main still has the exact plugin-bound `handled` branch returning `queuedFinal: false`, and current tests still assert that old behavior for plugin-owned bindings. Telegram still fires its silent-reply fallback when `queuedFinal` is false and nothing was delivered, so this PR is not obsolete or implemented on main.

## Best Possible Solution

Keep PR #70288 open. The best path is to land a corrected fix that makes a plugin-bound `inbound_claim` result of `{ handled: true }` an explicit committed-response signal through the Telegram-facing dispatch path, while leaving missing-plugin, no-handler, declined, and error paths eligible for fallback. Add focused coverage for the core dispatch branch and a Telegram no-silent-reply regression using an async plugin-claim shape like the linked `openclaw-codex-app-server` reproduction.
