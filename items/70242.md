## Summary

Keep open. Current `main` still appears to match the report: `memory-core` registers `publicArtifacts`, `wiki bridge import` reads them from global memory plugin state, but the plugin CLI path still builds plugin commands through a non-activating snapshot load and does not later force a full plugin activation for `wiki` commands. That leaves a plausible path where `listActiveMemoryPublicArtifacts()` returns `[]` during `openclaw wiki bridge import` unless some other command already activated the runtime registry in-process.

## Evidence

- **memory-core does register publicArtifacts:** `memory-core` registers a unified memory capability including `publicArtifacts.listArtifacts = listMemoryCorePublicArtifacts`, so the reported surface is expected to exist when the memory plugin is active.
  - file: [extensions/memory-core/index.ts:33](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/extensions/memory-core/index.ts#L33)
  - command: `nl -ba extensions/memory-core/index.ts | sed -n '1,140p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **wiki bridge import depends on active memory public artifacts:** `syncMemoryWikiBridgeSources()` calls `listActiveMemoryPublicArtifacts({ cfg })` and derives both `artifactCount` and `workspaces` from that returned list. If global memory capability state is empty, the command will report `0`/`0` exactly as in the issue.
  - file: [extensions/memory-wiki/src/bridge.ts:224](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/extensions/memory-wiki/src/bridge.ts#L224)
  - command: `nl -ba extensions/memory-wiki/src/bridge.ts | sed -n '210,270p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **plugin CLI registration still uses snapshot loads:** Plugin CLI commands are still loaded via `loadOpenClawPlugins(... activate: false, cache: false ...)`, i.e. a non-activating snapshot path rather than the normal runtime activation path.
  - file: [src/plugins/cli-registry-loader.ts:115](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/plugins/cli-registry-loader.ts#L115)
  - command: `nl -ba src/plugins/cli-registry-loader.ts | sed -n '1,220p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **snapshot loads restore prior memory state instead of leaving new capability registered:** During non-activating loads, the loader restores the previous memory plugin state after plugin registration. If no memory capability was active before the CLI snapshot load, the `memory-core` capability registered during command discovery is discarded again.
  - file: [src/plugins/loader.ts:2887](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/plugins/loader.ts#L2887)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2868,2918p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **wiki commands do not get automatic plugin bootstrap:** CLI startup policy defaults `loadPlugins` to `never`, and the explicit command-catalog overrides only mark a few built-in command roots such as `agent`, `message`, `channels`, `directory`, and `agents` for plugin loading. There is no `wiki` override in that catalog region, so `wiki` commands do not obviously trigger a later full plugin activation before action execution.
  - file: [src/cli/command-path-policy.ts:5](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/cli/command-path-policy.ts#L5)
  - command: `nl -ba src/cli/command-path-policy.ts | sed -n '1,80p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **startup policy honors the default no-plugin-load behavior:** CLI startup derives `loadPlugins` from command-path policy and only bootstraps plugin loading when that policy says to. For a plugin command root like `wiki`, current code does not show a built-in policy override that would force activation before the command runs.
  - file: [src/cli/command-startup-policy.ts:19](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/cli/command-startup-policy.ts#L19)
  - command: `nl -ba src/cli/command-startup-policy.ts | sed -n '1,120p'`

## Risks / Open Questions

- No end-to-end CLI test currently covers `openclaw wiki bridge import` with `memory-core` public artifacts present through the real plugin-command bootstrap path, so this regression can remain undetected.
- Recent loader tests prove preservation of an already-existing memory capability across snapshot loads, but they do not prove that plugin CLI startup for `wiki` establishes that capability when starting from a clean process.
