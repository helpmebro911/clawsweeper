## Summary

Current main still does not provide the reported one-click cross-machine Chrome extension relay workflow. The browser plugin intentionally removed the legacy relay/profile path, current profile creation rejects `driver: "extension"`, legacy configs are migrated to `existing-session`, and the documented replacements require host-local Chrome MCP, raw/remote CDP, or a browser-capable node host rather than the browser-extension-only managed-hosting flow described by the reporter and confirming commenters.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The best path is either to restore an explicitly opt-in, hardened relay inside the browser plugin with schema, docs, tests, and security review, or to add a narrow public browser transport/profile-driver API so the linked ClawHosters relay or a ClawHub plugin can integrate with the existing browser tool cleanly.

## Risks / Open Questions

- Restoring a browser relay would reintroduce a remote browser-control surface and needs explicit maintainer product and security review for authentication, origin validation, token handling, public exposure, and operator consent.
- The linked ClawHosters relay is useful prior art and a ClawHub plugin may be viable long term, but current main does not expose an obvious public browser profile-driver or transport API for integrating it into the bundled browser tool without private internals or replacing the whole tool.
- The documented node-host, existing-session, and remote CDP paths cover some remote-browser cases, but they do not resolve the specific no-Node, no-special-Chrome-flags, Chrome-extension-only managed-hosting workflow.
