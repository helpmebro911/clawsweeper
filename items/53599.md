## Summary

Current main has not restored the Chrome extension relay or added a replacement that covers the reported one-click cross-machine browser-control workflow. The old relay path is still intentionally disabled, `driver: "extension"` is still rejected or migrated away, and the documented replacements are host-local Chrome MCP, node-host `browser.proxy`, or remote CDP. The reporter's external relay implementation is useful context, but current plugin surfaces do not expose a browser profile/transport-driver capability that would let it wire into the browser tool as a clean ClawHub-only solution today.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The likely paths are either restoring an opt-in extension relay inside the browser plugin with schema/docs/tests/security review, or first adding a small public browser transport/profile-driver capability so the reporter's relay or a ClawHub plugin can wire into the browser tool without private reach-ins.

## Risks / Open Questions

- Restoring an extension relay would reintroduce remote browser-control WebSocket behavior, so a fix needs explicit auth, origin, token-handling, exposure, and operator-consent review.
- If maintainers want this to live on ClawHub, core likely needs a narrow typed browser transport/profile-driver capability first; otherwise an external plugin would need private browser internals or would only expose a parallel HTTP route, not the existing browser tool profile contract.
