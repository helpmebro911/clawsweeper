## Summary

Keep #70309 open. Current main stores and renders MCP server definitions, and session runtimes can connect to MCP servers when tools are materialized, but there is still no Control UI MCP connectivity/status panel or Gateway RPC that reports per-server connected/error state, tool names, or a manual reconnect/refresh path. The request is a core Control UI/Gateway observability feature, not a ClawHub-only plugin idea, because it needs access to OpenClaw's global `mcp.servers` config and a first-party status surface.

## Best Possible Solution

Add a first-party Gateway MCP status/control surface, likely `mcp.status` plus a bounded refresh/reconnect action, that reads configured `mcp.servers`, probes each server through the existing transport/runtime helpers, returns redacted per-server connected/error state plus tool names/counts, and always cleans up probe transports. Then add an Infrastructure -> MCP Control UI panel showing those states, refresh controls, and redacted error details, with focused Gateway/UI tests and docs updates under the public Control UI/MCP docs.

## Risks / Open Questions

- Implementing probes must preserve secret redaction for URLs, headers, environment variables, and error messages.
- Status refreshes can spawn stdio servers or call remote HTTP/SSE endpoints, so they need bounded timeouts, cleanup, and clear user-triggered semantics to avoid side effects.
- A status panel should avoid reusing or mutating active session MCP runtimes in a way that changes tool availability during running agent turns.
