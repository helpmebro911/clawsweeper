## Summary

Keep this issue open. Current main stores MCP server definitions, hot-disposes cached session MCP runtimes after MCP config changes, and can show session-effective tools, but it still lacks the requested Control UI MCP connectivity panel with per-server health, per-server tool names, manual reconnect/refresh, and surfaced redacted error details. The existing ClawSweeper comment remains accurate against current main SHA 348728c28c1ea9ae7c5824350baba35ee08a275e.

## Best Possible Solution

Keep this issue open. The best implementation is an additive Gateway operator diagnostics API backed by the MCP config/runtime loader, returning each configured server's name, transport summary, connected/error state, listed tool names/count, and redacted last error, plus a scoped refresh/reconnect action. Then render it under Control UI Infrastructure > MCP and document the behavior on the public docs site.

## Risks / Open Questions

- A future diagnostics panel must redact tokens, headers, URL userinfo, and environment-derived secrets before surfacing MCP errors in the browser.
- The product design should decide whether pre-flight checks are allowed to spawn stdio MCP processes outside an active session, or whether diagnostics stay session-scoped to avoid surprising side effects.
