## Summary

Keep open. Current main still does not implement the requested Telegram 421 retry behavior: the shared channel retry regex omits 421, visible Telegram sends use a strict non-idempotent retry predicate limited to pre-connect errors or 429, and the docs/tests still have no 421 coverage. The related PR #48908 is useful prior work but remains unmerged, so it does not supersede the issue on main.

## Evidence

- **Shared retry policy omits 421:** `CHANNEL_API_RETRY_RE` matches 429, timeout, connect/reset/closed, unavailable, and temporarily; it does not match `421` or `Misdirected Request`, so the default channel retry runner still would not retry a bare 421 error.
