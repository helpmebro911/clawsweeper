## Summary

Keep open. Current main has narrower service-scan behavior than the report implies, but it still does not meet the issue's requested contract of only flagging actual gateway conflicts by port or OpenClaw gateway binary. The macOS LaunchAgent scanner can still classify a non-canonical plist as gateway-like solely because its contents mention OpenClaw or Clawdbot, and there is no macOS regression test covering unrelated LaunchAgents like the reported labels.

## Evidence

- **LaunchAgent marker is broad:** Launchd scanning calls detectMarker(contents), and detectMarker returns a marker when the entire plist contains "openclaw" or "clawdbot" anywhere. It does not require a gateway command, matching port, or OpenClaw gateway binary for macOS LaunchAgents.
  - file: [src/daemon/inspect.ts:56](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/daemon/inspect.ts#L56)

  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Non-marker plists are skipped, but marker plists are still reported:** scanLaunchdDir skips plists with no marker unless the label is legacy, but when a marker exists it reports the service unless it is the ignored default label or a canonical ai.openclaw gateway service. That leaves false positives for unrelated non-ai.openclaw LaunchAgents whose plist content mentions OpenClaw.
