## Summary

Keep open. Current main still has a plausible macOS LaunchAgent false-positive path: the launchd scanner treats any plist containing the broad markers `openclaw` or `clawdbot` as a candidate, then reports non-`ai.openclaw.*` labels even when the plist does not run an OpenClaw gateway or bind the gateway port. I did not find a current automatic unload path from this scanner, but the core reported problem, unrelated LaunchAgents being reported as gateway-like, is not fixed. The prior ClawAid/workaround comment is useful support context, not an OpenClaw core fix.

## Best Possible Solution

Keep this issue open. Narrow `src/daemon/inspect.ts` launchd detection so a plist is treated as gateway-like only when it has explicit OpenClaw gateway service markers (`OPENCLAW_SERVICE_MARKER=openclaw` plus `OPENCLAW_SERVICE_KIND=gateway`), a recognized `ai.openclaw.*` gateway label with a gateway command, or a legacy gateway signature. Add macOS scanner tests for unrelated `com.holland.*` LaunchAgents that mention `.openclaw` paths/comments but do not run `gateway`, plus positive tests for real legacy/current gateway services. Consider making cleanup hints refer to the actual extra service or suppressing canonical gateway removal hints for non-legacy extras.
