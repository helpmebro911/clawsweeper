## Summary

Keep #8061 open. Current main still only supports a global subagent concurrency cap through `agents.defaults.subagents.maxConcurrent`; `agents.list[]` has no accepted or typed per-agent `maxConcurrent`, and both native and ACP `sessions_spawn` dispatch onto the shared `subagent` lane.

## Best Possible Solution

Keep #8061 open. Add a validated per-target-agent subagent concurrency setting, preferably `agents.list[].subagents.maxConcurrent`; update types, strict schema, generated config schema/help, and docs; then enforce it alongside the existing global `CommandLane.Subagent` cap in both native and ACP `sessions_spawn`. Add focused tests proving a capped target such as `tts-agent` serializes while uncapped targets can still run concurrently.

## Risks / Open Questions

- Maintainers still need to choose the public config shape; `agents.list[].subagents.maxConcurrent` appears clearer than a top-level `agents.list[].maxConcurrent` because the requested cap is about spawned subagent runs.
- Runtime semantics should define whether the cap applies to runs targeting that agent id, runs spawned by that agent id, or both, and must cover native plus ACP `sessions_spawn` paths.
