## Summary

Keep open. Current main has auth-profile ordering, lastUsed-based round-robin ordering, cooldown/failure rotation, status probes, and a narrow provider-owned profile preference hook, but it does not implement the requested opt-in provider-pool routing layer with strategies, per-profile in-flight/concurrency limits, batch/task stickiness, routing state, or `models routing` CLI controls. The current docs explicitly describe session stickiness and say OpenClaw does not rotate on every request, which is the central behavior this issue asks to extend.

## Best Possible Solution

Keep this as an open design/feature issue. The best implementation path is an additive auth-routing contract: define config/schema for pool mode, strategy, stickiness, profile list, and concurrency limits; add runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; expose routing state through `models status --json`; and decide whether provider-specific behavior uses the existing `ProviderPlugin.resolveAuthProfileId` seam or a new generic pool-routing API.
