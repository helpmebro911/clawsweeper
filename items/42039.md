## Summary

Keep this issue open. Current main still keeps `maxMissedJobsPerRestart` and `missedJobStaggerMs` as internal cron service dependency knobs, not public `cron` config fields accepted by `openclaw.json`, documented, or wired from gateway config. The related implementation PR #42069 is closed and unmerged, and #48345 is also closed, so there is no active canonical replacement for the remaining request.

## Best Possible Solution

Implement a narrow public cron catch-up configuration path: add either `maxMissedJobsPerRestart` and `missedJobStaggerMs`, or a clearer `skipMissedRuns` control, to `CronConfig`, zod validation, generated schema metadata, and cron docs; wire values from gateway config into `CronService`; define disabled catch-up semantics explicitly; and add focused tests for config acceptance, gateway passthrough, default compatibility, stagger timing, and skipped catch-up behavior.

## Risks / Open Questions

- Defining `maxMissedJobsPerRestart: 0` needs explicit product semantics: skip overdue runs, defer all overdue runs, or advance to the next future schedule.
- Changing default catch-up behavior could regress users who rely on missed jobs running after sleep or downtime; the conservative path should preserve existing defaults and make opt-out behavior explicit.
