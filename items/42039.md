## Summary

Keep this issue open. Current main still keeps `maxMissedJobsPerRestart` and `missedJobStaggerMs` as internal cron service dependency knobs only; they are not exposed through public config types, strict validation, generated metadata, gateway wiring, or cron docs. The related implementation PR was closed unmerged, and the later skip-missed-runs issue is also closed, so there is no active canonical replacement that solves the remaining request.

## Best Possible Solution

Keep this issue open and implement a narrow public cron config path: add `maxMissedJobsPerRestart` and `missedJobStaggerMs` to `CronConfig`, strict zod validation, generated schema labels/help, and public cron docs; wire those fields from `buildGatewayCronService` into `CronService`; define and test the intended `0` behavior explicitly, likely as skip catch-up or advance to the next scheduled slot rather than deferred replay; add focused tests for config passthrough, default compatibility, stagger timing, and disabled catch-up.

## Risks / Open Questions

- Exposing `maxMissedJobsPerRestart` verbatim may not deliver the requested `0` equals no catch-up behavior: current logic treats `0` as zero immediate candidates and then defers all missed jobs to future ticks, so disabling catch-up needs explicit skip or advance-to-next-slot semantics.
- Changing defaults would affect users who rely on missed cron jobs running after sleep or downtime; the safer implementation preserves current defaults and only changes behavior when the new config is explicitly set.
