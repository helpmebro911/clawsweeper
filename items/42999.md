## Summary

Keep this issue open. Current main still uses fixed 16k/32k context-window guard thresholds, tests explicitly lock that behavior, and the runtime call site passes the same static thresholds. Existing `contextTokens` support caps the effective model budget, but it does not implement the requested percentage-based warn/block thresholds. The related PRs in the provided GitHub context are closed and unmerged, so they do not supersede the remaining work.

## Best Possible Solution

Implement a focused threshold resolver in `src/agents/context-window-guard.ts` that derives warning and hard-min thresholds from the resolved effective context window, preserves explicit override support and safe floors, then update the guard tests, Pi runtime call site, onboarding assumptions, and user-facing docs so small and large model behavior is intentional.
