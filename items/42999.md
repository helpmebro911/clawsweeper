## Summary

Current main still implements the reported context-window guard with fixed 16k hard-min and 32k warning thresholds. Adjacent dynamic budgeting exists for compaction, and `contextTokens` can cap the effective runtime window, but neither implements the requested percentage-based warning/block threshold resolver. The related proposal PRs in the provided GitHub context are closed and unmerged, so they do not supersede the remaining work.

## Best Possible Solution

Keep this open. The best implementation is a focused threshold resolver in `src/agents/context-window-guard.ts` that derives warning and hard-min thresholds from the resolved effective context window, preserves explicit `contextTokens` cap behavior and safe floors, then updates guard tests, Pi runtime setup expectations, onboarding/default behavior, and public docs so small and large model behavior is intentional.
