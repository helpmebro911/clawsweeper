## Summary

Keep #8355 open. Current main still batches the classic voice-call STT/agent/TTS path: it waits for one completed agent response string, then one completed telephony TTS buffer, then chunks that completed buffer to Twilio. Realtime voice support exists, but it is documented and configured as a separate full-duplex mode, not the requested sentence-level LLM-to-TTS pipeline. Maintainer discussion in the provided context also explicitly keeps #8355 as the canonical item after #42187 closed unmerged and #42159 was closed as the narrower duplicate.

## Best Possible Solution

Keep #8355 open as the canonical provider-generic voice-call feature. The best implementation path is to add a streaming telephony TTS capability with buffered fallback, feed embedded-agent partial replies or sentence chunks into that capability, stream synthesized audio chunks through Twilio and Telnyx media paths where supported, and preserve queue clearing, abort, timeout, and barge-in semantics for in-flight synthesis and playback.
