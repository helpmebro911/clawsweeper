## Summary

Keep open. Current main still batches the classic voice-call STT -> LLM -> TTS path: response generation resolves to one final text result, telephony TTS returns one complete audio buffer, and Twilio only chunks that completed buffer after synthesis. The newer realtime voice path is documented and validated as a separate architecture, and the provided issue discussion already treats #8355 as the canonical provider-generic sentence-level streaming TTS request after PR #42187 closed unmerged.

## Best Possible Solution

Keep #8355 open as the canonical voice-call feature request. The best implementation path is a provider-generic sentence-level pipeline in the voice-call plugin: feed embedded-agent partial replies into a sentence splitter, add a streaming telephony TTS contract for speech providers, stream audio chunks through the Twilio/Telnyx media handlers with abort/barge-in support, and preserve a buffered fallback for providers that only return full audio buffers.
