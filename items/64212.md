## Summary

Keep this issue open. Current main has generic custom-provider image routing and configurable image-tool timeouts, but I found no current source, test, docs, changelog, history, or discussion evidence proving the NVIDIA NIM Kimi K2.5 image abort is fixed. The prior ClawSweeper comment already reached the same conclusion after the generic timeout work; current main still has the same NVIDIA/Kimi-specific evidence gap.

## Best Possible Solution

Keep this issue open. Add a focused regression or live/manual proof for NVIDIA NIM Kimi K2.5 through the same configured-provider image path used by Feishu and Telegram. First verify whether tools.media.image.timeoutSeconds above the observed provider latency fixes it; if yes, document the required config or revisit the default. If it still fails, fix the NVIDIA/Kimi OpenAI-compatible image payload or streaming behavior, or surface a clear unsupported-provider error.
