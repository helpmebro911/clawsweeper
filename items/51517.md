## Summary

Current main already provides the requested observe/read-only behavior through the generic session send-policy path: inbound messages are still processed, automatic channel delivery is suppressed, and explicit sends use the separate message-tool outbound path.

## Risks / Open Questions

- The equivalent is implemented as generic session send-policy behavior, not as a literal `dmPolicy: "observe"` enum, so discoverability is lower than the issue proposed.
- This review was code-and-docs based only; I did not run a live WhatsApp reproduction because the task required a read-only checkout.

## Close Comment

Codex review: current main already implements the requested read-only / observe behavior through the generic session send-policy path, so a new `dmPolicy: "observe"` enum is not required. `session.sendPolicy` can deny delivery by `channel` + `chatType` (`docs/gateway/config-agents.md:1138`, `docs/gateway/config-agents.md:1164`), and owners can toggle it live with `/send off` (`docs/tools/slash-commands.md:134`). In the reply pipeline, `sendPolicy: "deny"` still processes inbound turns but suppresses outbound reply/notice/indicator delivery (`src/auto-reply/reply/dispatch-from-config.ts:472`), and that behavior is covered by tests (`src/auto-reply/reply/dispatch-from-config.test.ts:3420`, `src/auto-reply/reply/dispatch-from-config.test.ts:3483`, `src/auto-reply/reply/dispatch-from-config.test.ts:3506`, `src/auto-reply/reply/dispatch-from-config.test.ts:3538`). Explicit sends still use the separate `message` tool outbound path (`src/agents/tools/message-tool.ts:807`, `src/infra/outbound/message-action-runner.ts:824`). Closing as implemented on main.
