## Summary

Keep this PR open. Current main still uses a fixed two-hour stale cron running-lock cleanup threshold in timer maintenance even though cron execution timeouts are resolved per job, so the PR addresses a real remaining reliability gap. The current PR head should not merge as-is because its final clamp can clear locks for explicit long-running jobs before their configured timeout has elapsed.

## Best Possible Solution

Keep this PR open as a real cron reliability candidate, but request changes before merge: rebase on current main; derive stale-lock cleanup from `resolveCronJobTimeoutMs(job) + buffer` for normal finite positive timeouts; preserve a conservative fallback for `undefined` no-timeout jobs; handle timer-overflow-sized values with an explicit policy that does not shorten valid long configured timeouts; and add regression coverage for short custom timeouts, systemEvent defaults, no-timeout jobs, and explicit long-timeout preservation.
