## Summary

Keep this PR open. Current main still has the fixed 2-hour cron stale-lock maintenance threshold while cron execution timeouts are resolved per job, so the PR targets a real remaining reliability gap. Do not merge the current PR head as-is: its final clamp caps every positive timeout-derived stale-lock threshold at 2 hours, which can clear locks for legitimate explicit long-running jobs before their configured timeout has elapsed.

## Best Possible Solution

Keep the PR open as a real cron reliability candidate, but request changes before merge: rebase on current main, derive the stuck-lock threshold from `resolveCronJobTimeoutMs(job) + buffer` for normal positive timeouts, preserve a conservative fallback for `undefined` no-timeout jobs, handle only timer-overflow-sized values with an explicit compatible policy, and add regression coverage for explicit long timeouts plus the short-timeout, systemEvent, and no-timeout cases.
