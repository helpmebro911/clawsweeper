## Summary

Keep this PR open. Current main still uses the strict MCP SDK `client.listTools()` path, so the linked MCP compatibility bug is not implemented on main. The latest PR revision is a plausible implementation candidate and addresses the earlier schema-normalization and provider-payload concerns, but it still replaces `Client.listTools()` with raw `client.request()`. In the pinned MCP SDK 1.29.0, `listTools()` caches tool metadata after listing, and `callTool()` uses that cached metadata for output-schema validation and task-required routing, so the PR needs rework or maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rework bundle MCP discovery so it accepts tools whose `inputSchema.type` is missing while preserving the MCP SDK 1.29.0 `listTools()` metadata behavior, keep the latest catalog-time rejection of explicit non-object root schemas, and add a regression that proves both typeless-schema discovery and metadata-dependent `callTool()` behavior still work.
