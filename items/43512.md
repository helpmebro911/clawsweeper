## Summary

Keep this PR open. Current main has an abort-only Feishu control lane for `/stop`-style text, but this PR and the linked open regression #42803 target broader text command immediacy for `/new`, `/status`, `/help`, `/model`, and similar command-only messages. The PR is stale against the current handler split and needs maintainer review, but it is not obsolete cleanup material.

## Best Possible Solution

Keep this PR open as a stale but plausible implementation candidate for #42803. The best path is to rebase or rewrite it on the current `createFeishuMessageReceiveHandler` and `getFeishuSequentialKey` design, decide which Feishu command classes should use a control lane, preserve authorization before destructive side effects where feasible, and add regression tests proving active-run immediacy for read-only commands plus ordering/drop behavior for `/stop`, `/new`, and `/reset`.

## Risks / Open Questions

- The PR likely needs a rebase or rewrite because current main has split Feishu receiving into `monitor.message-handler.ts` plus `sequential-key.ts`, while the PR patch targets the older `monitor.account.ts` receive-handler shape.
- Broadly routing every text command around the same-chat lane can reorder command handling relative to buffered or queued user text; `/status` and `/help` may be safe, while `/new`, `/reset`, `/model`, and other state-changing commands need explicit semantics.
- Destructive pre-dispatch cleanup is security-sensitive because unauthorized `/stop`, `/new`, or `/reset` attempts could drop pending buffered text before command authorization is checked unless the implementation moves or gates that side effect.
- The shared `createInboundDebouncer` API addition (`dropKey`) would need focused contract tests if retained, because it changes a core debounce utility used outside Feishu.
