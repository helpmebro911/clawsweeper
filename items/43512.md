## Summary

Keep this PR open. Current main only partially supersedes the PR by routing Feishu abort text such as `/stop` to a separate control lane; it still does not implement the broader control-command bypass described by this PR for `/new`, `/status`, `/help`, `/model`, and other command-only messages. The PR is stale against the newer Feishu handler split and has behavior questions from review, but it remains a plausible implementation candidate for the still-open Feishu regression tracked by #42803.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate. The best path is to rebase or rewrite it on the current `createFeishuMessageReceiveHandler` and `getFeishuSequentialKey` design, decide which command classes should bypass the per-chat lane, preserve authorization before destructive side effects, and add focused Feishu tests for `/status` immediacy plus `/new`, `/reset`, and `/stop` ordering during an active run.

## Risks / Open Questions

- The PR targets the older `extensions/feishu/src/monitor.account.ts` receive/debounce shape; current main moved that logic into `extensions/feishu/src/monitor.message-handler.ts` and `extensions/feishu/src/sequential-key.ts`, so it likely needs a rebase or rewrite before review.
- The PR discussion still leaves authorization and ordering tradeoffs: destructive commands that drop debounce buffers or clear queued chat work before authorization can cause message loss for unauthorized command attempts.
- Broadly bypassing the same-chat lane can reorder user text relative to informational commands and destructive session commands; maintainers should choose separate semantics for `/status`/`/help` versus `/new`/`/reset`/`/stop` and require regression tests.
- No supply-chain or CI/workflow expansion is evident from the supplied PR files; the review risk is the Feishu command-dispatch behavior, not third-party code execution.
