## Summary

Keep this PR open. Current main still does not refresh the TUI when the active session is reset externally, and this PR remains a focused implementation candidate for the linked #38966 bug. It needs rebasing/adaptation to the current subscriber-scoped `sessions.changed` lifecycle model rather than landing the older generic `session` event shape as-is. Security pass found no dependency, workflow, script, lockfile, secret, or third-party code changes in the PR diff; the main security/protocol concern is preserving the current event scope/subscription boundary.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch adapted to current main: add a TUI backend path to subscribe to session lifecycle events, handle `sessions.changed` for `reason: "new"` or `"reset"` when `sessionKey` matches the current TUI session, clear stale active/finalized/session run state and local run IDs, recreate streaming state, reload history/session info, ignore unrelated sessions, and add focused TUI handler plus gateway/TUI contract tests. If maintainers intentionally choose a new generic event instead, update the advertised event list, broadcast scope guard, subscription/broadcast semantics, protocol docs, and tests together.
