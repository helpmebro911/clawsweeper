## Summary

Keep this PR open. Current main has moved the gateway side of the fix to the subscriber-scoped `sessions.changed` lifecycle event, but the TUI still does not subscribe to or handle that event, so the linked external-reset TUI bug remains actionable. The PR is stale against the current protocol shape because it proposes a generic `session` event, but the core TUI work is still useful and should be adapted rather than closed.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch adapted to current main: have the TUI backend call `sessions.subscribe` after connecting, handle `sessions.changed` with `reason: "new"` or `"reset"` when `sessionKey` matches the active TUI session, clear active/finalized/session run state plus local run IDs and streaming/watchdog state, reload history/session info, ignore unrelated sessions, and add focused TUI handler plus gateway/TUI contract tests. Preserve the current subscriber-scoped `sessions.changed` protocol unless maintainers deliberately introduce a new documented event family.
