## Summary

Keep open. Current main still routes native `sessions_spawn` sub-agent startup through `callGateway()` and a WebSocket-backed GatewayClient before the request reaches the `subagent` queue lane. Recent docs now clarify the expected backend gateway-client/shared-auth path, and existing tests cover some stale paired-scope cases, but the requested same-process shortcut is not implemented and the related open PR #42497 covers embedded gateway tool dispatch, not this exact sub-agent spawn path. The issue also contains concrete downstream reproduction context and workaround links (`sidekyk-verticals` polling workaround), so it is not stale or incoherent cleanup material.

## Best Possible Solution

Keep #70053 open as the focused tracker for native sub-agent spawn dispatch. The best implementation path is to add a same-process sub-agent dispatch seam that can invoke the gateway agent handler or agent ingress directly when the Gateway runtime is local, while preserving task registration, lane selection, lifecycle hooks, delivery context, timeout, and cleanup semantics. Add regression coverage for fresh server-side installs, shared-token/password backend calls, and `gateway.auth.mode="none"` behavior.
