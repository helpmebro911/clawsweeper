## Summary

Keep #19929 open. Current main at e672b614 still lacks selective shared-session routing for chosen Discord/Slack group, channel, or thread chats. Core routing only threads `dmScope` and `identityLinks` into session-key construction, and only for direct peers; group/channel peers still resolve to isolated keys, with thread suffixing layered on top. Strict config schemas, public types, generated help, and docs expose no `channelGroups`, `groupScope`, `includeThreads`, or equivalent. The linked PR #20078 is closed and unmerged, and #7524 is an open but narrower group-to-main request rather than a replacement for selective cross-channel/thread grouping.

## Best Possible Solution

Keep #19929 open as the concrete tracker for selective shared sessions across chosen group/channel/thread chats. A good implementation belongs in core routing/config: add a backward-compatible grouping config, thread it through `buildAgentSessionKey`/`resolveAgentRoute` and Discord/Slack thread/outbound paths, update strict schemas/types/help/docs/generated metadata, and add regression tests for grouped routing, ungrouped compatibility, reply delivery, queue serialization, thread inheritance, and sandbox/trust-boundary behavior. Reuse any useful design from closed PR #20078 only after re-reviewing it against current main.
