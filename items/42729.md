## Summary

Keep open. Current `main` still appears vulnerable to the duplicate-after-restart case described in the PR: startup catch-up selects runnable jobs from `collectRunnableJobs()`, and `isRunnableJob()` still treats any stale `nextRunAtMs <= now` slot as runnable without checking whether `lastRunAtMs` already passed that slot. The newer maintenance recompute guard only runs in paths that have no due jobs, so it does not close the gap this PR targets.

## Evidence

- **Current runnable check still accepts stale expired slots:** On current `main`, `isRunnableJob()` returns `true` whenever `nextRunAtMs <= now` and does not compare `lastRunAtMs` to that scheduled slot first.
  - file: [src/cron/service/timer.ts:894](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/cron/service/timer.ts#L894)

  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Startup restart catch-up uses the same runnable filter:** `runMissedJobs()` -> `planStartupCatchup()` calls `collectRunnableJobs()` during startup, so the stale-slot case would be picked up before regular timer maintenance runs.
  - file: [src/cron/service/timer.ts:989](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/cron/service/timer.ts#L989)

  - file: [src/cron/service/jobs.ts:519](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/cron/service/jobs.ts#L519)
