## Summary

Keep this PR open. Current main still treats a persisted past-due `nextRunAtMs` as runnable before checking whether `lastRunAtMs` already covers that scheduled slot, and the PR is a narrow scheduler fix with a matching regression test. The discussion also includes a concrete v2026.3.31 confirmation of the restart-duplicate behavior.

## Best Possible Solution

Keep this PR open and review or land it, or port the same strict stale-slot guard into current main's `isRunnableJob`/startup catch-up path with the PR's regression test. The linked bug is closed, but current main still appears to need this focused scheduler fix.
