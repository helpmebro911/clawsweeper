## Summary

Keep open. Current `main` still appears to have the reported bug: `sandbox explain` leaves any colon-containing session key unchanged, but its channel inference only works for agent-scoped keys and its session-store lookup does not canonicalize non-agent-scoped keys. The PR’s fix is not present on `main`.

## Evidence

- **Colon-containing keys are returned unchanged:** `normalizeExplainSessionKey` returns `raw` for any session containing `:`, so `telegram:slash:<id>` is not wrapped to an agent-scoped key.
  - file: [src/commands/sandbox-explain.ts:49](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.ts#L49)
  - command: `nl -ba src/commands/sandbox-explain.ts | sed -n '37,59p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Channel inference requires an agent-scoped key:** `inferProviderFromSessionKey` immediately calls `parseAgentSessionKey`; that parser returns `null` unless the key starts with `agent:` and has at least three parts, so `telegram:slash:<id>` cannot infer `telegram`.
  - file: [src/commands/sandbox-explain.ts:65](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.ts#L65)
  - command: `nl -ba src/commands/sandbox-explain.ts | sed -n '61,88p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Parser rejects non-agent-scoped slash keys:** `parseAgentSessionKey` requires `parts[0] === "agent"`; non-agent-scoped keys like `telegram:slash:200` are rejected.
  - file: [src/sessions/session-key-utils.ts:35](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/sessions/session-key-utils.ts#L35)
  - command: `nl -ba src/sessions/session-key-utils.ts | sed -n '28,47p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Store lookup uses the unnormalized key:** `resolveActiveChannel` indexes the session store with `store[params.sessionKey]` directly, so an unscoped `telegram:slash:<id>` lookup misses agent-scoped stored entries.
  - file: [src/commands/sandbox-explain.ts:100](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.ts#L100)

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Current tests do not cover the reported case:** `src/commands/sandbox-explain.test.ts` on `main` only covers basic JSON shape and sandbox tool-policy behavior; it has no test for non-agent-scoped colon keys or downstream `elevated.channel` / `allowedByConfig` behavior.
  - file: [src/commands/sandbox-explain.test.ts:16](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.test.ts#L16)
