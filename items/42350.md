## Summary

Keep this PR open. Current main still lacks the proposed shared Swift Adaptive Card parser/SwiftUI chat renderer, the same contributor has an open Android companion PR for the same feature set, and the PR still needs maintainer product and security review rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer product and security review. If native iOS/macOS Adaptive Card rendering is wanted in core, rebase it on current main, fix the parser/layout/fallback/action-data review items, add focused Swift tests or app smoke coverage, explicitly review remote image and URL-opening behavior, and coordinate the platform decision with open Android companion PR #42304. The external `openclaw-adaptive-cards` plugin is useful prior work for producing the markers, but current OpenClaw plugin APIs do not provide a native Swift chat renderer extension point that would make this PR obsolete as ClawHub-only work.

- The current PR adds no Swift parser/view tests or app smoke coverage for the new renderer path.
- The docs addition lists a third-party plugin in `docs/plugins/community.md`, while current docs say ClawHub is the canonical discovery surface; that docs piece may need to be removed or handled separately even if the native renderer is accepted.
