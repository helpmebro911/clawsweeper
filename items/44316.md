## Summary

Keep open. Current main has moved the Control UI chat pipeline to tool-card rendering and final-payload sanitization, but it does not fully implement this PR's requested behavior. Tool results are still included when `showToolCalls` is enabled, tool payloads are intentionally preserved in `chat.history`, and there is no current UI-side sanitizer/test proving `SECURITY NOTICE` or `EXTERNAL_UNTRUSTED_CONTENT` markers are stripped from expanded tool output. The PR is useful but needs rework against the current architecture rather than auto-close.

## Best Possible Solution

Keep the PR open but do not merge it as-is. Rework the fix against current main by adding a targeted tool-result display sanitizer, preserving intentional Tool output cards, decoupling behavior from `showThinking`, and adding focused tests for `showThinking`, `showToolCalls`, assistant tool-call-only blocks, web_fetch `SECURITY NOTICE`/`EXTERNAL_UNTRUSTED_CONTENT` payloads, and the history-limit-after-filtering case.

## Risks / Open Questions

- The PR patch targets the old `ui/src/ui/views/chat.ts` chat-item path; current main routes chat construction through `ui/src/ui/chat/build-chat-items.ts`, so the change needs a rebase/reimplementation.
- The proposed `extractText`-level external-marker stripping is probably too broad for current main because it can redact legitimate user or assistant discussion of those literal strings; the sanitizer should be scoped to tool-result/display projection paths or clearly wrapped payloads.
- Current `buildChatItems` applies the 200-message window before filtering hidden tool rows, so any refreshed fix should handle the existing review concern that hiding tool calls can show fewer real chat turns.
