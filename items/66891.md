## Summary

Current `main` has replaced the old Discord reply-delivery implementation with the shared outbound pipeline, and that pipeline does route interactive/channel-data replies through a Discord component send path. But the PR discussion surfaces reply-specific edge cases that are still not clearly handled on `main` today, especially bound-thread webhook persona delivery and forum-style channel fallback. That makes an auto-close unsafe; this PR should stay open or be rebased to the new outbound adapter architecture.

## Evidence

- **Reply delivery now goes through shared outbound routing:** `deliverDiscordReply()` no longer contains the old bespoke send logic; it forwards replies into `deliverOutboundPayloads()`, and the outbound runtime invokes `handler.sendPayload()` whenever a payload carries `interactive`, `presentation`, or `channelData`. That means the original code path from the PR body is obsolete, but also means the right question is whether the new adapter path fully covers the discussed edge cases.
