## Summary

Keep open. Current main still exposes Kimi Coding, still converts Kimi tagged tool-call text into executable tool calls, and `sessions_yield` still intentionally ends the turn by aborting the run and surfacing `end_turn`. I found tests for normal Kimi tagged tool parsing and for intended subagent yield fallback, but no source, test, release, or discussion evidence that current main fixes unwanted Kimi-generated `sessions_yield` interruptions or the reported need to manually send another message to resume.

## Best Possible Solution

Keep this issue open. The best maintainer path is to reproduce current main with a captured Kimi Coding transcript around the first `sessions_yield`, including whether there are pending child sessions or expected completion events. If Kimi calls `sessions_yield` without a pending child/result, add a guard or recovery path in the session-yield execution/runner flow and regression coverage in `extensions/kimi-coding/stream.test.ts` plus embedded runner or QA-lab coverage. If the root cause is missing delivery of a real yielded subagent completion, extend the direct-fallback path and cover the Kimi Coding flow specifically.
