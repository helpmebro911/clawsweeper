## Summary

Keep this PR open. Current main still lacks a built-in hard approval boundary for destructive tool calls in external-content hook sessions, and the PR is not merge-ready because its guard is not production-wired, its session predicate likely misses agent-scoped hook sessions, and its tool classifier is incomplete relative to current mutating tool metadata. Security review found no new dependency, workflow, or package-publishing surface in the PR, but merging as-is would create false assurance for a security control that does not run.

## Best Possible Solution

Keep this PR open, but request changes or supersede it with a production-wired core policy. The best implementation is a before-tool-call/tool-dispatch guard that carries external-content provenance into runtime context, uses current mutating-tool metadata such as `isMutatingToolCall` where possible, routes required approvals through existing plugin approval UI/channel flows, honors deliberate unsafe-content bypass flags, and updates tests/docs for any intentional `web_search` warning change.

## Risks / Open Questions

- Merging the PR as-is would imply a hard security boundary while the new guard is not invoked in production.
- A raw `hook:*` session check likely misses current agent-scoped hook sessions such as `agent:main:hook:webhook:42`.
- The PR’s destructive-tool classifier is incomplete for current outbound, process, session, gateway, cron, and plugin-owned mutating tools.
- Changing `web_search` to always include the long warning may be desirable, but it currently contradicts established tests and should be handled as an intentional behavior change.
