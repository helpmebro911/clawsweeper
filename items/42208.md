## Summary

Keep this PR open. Current main still provides prompt-level wrapping and sanitization for external hook content, but it does not implement the requested hard approval boundary for destructive tool calls. The PR is a real core security-hardening direction, but the submitted diff is not mergeable as-is because the guard is not wired into production dispatch, the session-key predicate is likely mismatched with current agent-scoped execution keys, and the web_search warning change conflicts with current tests.

## Best Possible Solution

Keep the PR open, but request changes or supersede it with a production-wired implementation. The right path is a core before_tool_call/tool-dispatch policy that recognizes external hook provenance in both raw and agent-scoped session keys, requires approval for mutating/exec/control-plane/outbound actions using existing tool policy metadata where possible, preserves deliberate allowUnsafeExternalContent behavior, and updates tests/docs for any intentional web_search warning change.

## Risks / Open Questions

- Security review: the provided PR diff only adds a security helper/test and changes external-content wrapping; it does not add dependencies, workflow changes, package metadata, install scripts, downloaded artifacts, generated vendor code, or broader secret-handling changes. The main security risk is functional false assurance from an unused guard.
- The proposed raw hook:* session check would likely miss current agent-scoped tool contexts unless the implementation preserves or resolves hook provenance explicitly.
- The destructive-tool regex list may be incomplete relative to OpenClaw's real tool catalog, mutation metadata, message tools, subagent controls, and plugin-owned tools; a production fix should reuse existing policy/classification surfaces where possible.
- Changing web_search to include the full external-content warning may be desirable, but it currently conflicts with established tests and should be handled as an intentional behavior change.
