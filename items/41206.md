## Summary

Keep this PR open. Current main still uses the single `jobs.json` cron store, and the PR is a plausible implementation candidate for the still-open per-job cron storage request. It should not be closed as implemented or obsolete, but it also should not land as-is because the posted review comments identify real filename-keying/data-integrity bugs that need author or maintainer follow-up.

## Best Possible Solution

Keep the PR open as a useful implementation candidate for per-job cron storage, but require a revision that uses a stable id-based filename contract or otherwise loads/removes by persisted job id, adds tests where job id and name differ plus rename/remove/collision cases, and updates the user-facing cron docs/config/doctor behavior before maintainer merge review.
