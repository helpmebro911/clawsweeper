## Summary

Keep open. Current `main` still logs config warnings on every successful `loadConfig()` with warnings, so the requested dedupe is not implemented on `main`. The proposed PR also has an unresolved edge case from review: its fingerprint cache is not cleared on non-warning failure paths, so warning logging can be suppressed incorrectly after transitions like warning config -> invalid/missing config -> same warning config.

## Risks / Open Questions

- Leaving this PR open is appropriate because the underlying log-spam behavior still exists on `main` and can still flood `gateway.log` during repeated config validation cycles.
- Merging the PR as written would likely introduce a false-suppression edge case after warning -> invalid/missing -> warning transitions, so it needs revision rather than closure.
