## Summary

Keep this PR open. Current main still has no direct config-warning dedupe in config IO, so the work is not implemented on main. The PR remains a relevant implementation candidate, but its current patch needs revision for stale dedupe state, the write path, and overlap with the broader open maintainer PR #38295.

## Best Possible Solution

Keep this PR open for maintainer review unless maintainers explicitly choose #38295 as the canonical path. The best fix is a centralized config-warning fingerprint helper in `src/config/io.ts` that clears state on every no-warning, failed, and early-return load path, re-logs when raw config content or formatted warning details change, decides whether `writeConfigFile()` shares the same guard, and adds tests for repeated load plus warning -> no-warning/invalid/missing -> same-warning transitions.

## Risks / Open Questions

- The proposed module-level warning fingerprint can remain stale across missing, invalid, parse-failed, or other early non-warning load paths, suppressing a re-emerged identical warning after the file changed away and back.
- The PR only guards the load path while current main also emits write-side `Config warnings:` unconditionally.
- There is implementation overlap with open maintainer PR #38295, so merging both without coordination could create conflicting or duplicate config-log state handling.
