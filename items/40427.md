## Summary

Keep this PR open. Current main still does not implement direct config-warning dedupe in config IO: `createConfigIO().loadConfig()` and `writeConfigFile()` both emit `Config warnings:` unconditionally whenever validation returns warnings. Runtime snapshot pinning and gateway persisted-hash reload dedupe reduce some production rereads, but they do not replace the warning-fingerprint behavior this PR is trying to add. The PR is focused and security-benign, but it needs revision for the existing review-commented state-clearing edge case and likely a scope decision on the write path.

## Best Possible Solution

Keep this PR open for maintainer review, but request a revised implementation that centralizes warning fingerprinting per config path, clears the fingerprint on every non-warning or failed-load path, re-logs when raw config content or warning details change, and adds focused tests for repeated loads plus warning/non-warning/warning transitions. Maintainers should also decide whether the same guard should cover `writeConfigFile()`. If the intended product boundary is only runtime snapshot and gateway reload dedupe, close only after documenting that decision in a canonical linked issue or PR.

## Risks / Open Questions

- The current PR patch should not land as-is until warning dedupe state is cleared on all non-warning and failed-load paths, with regression coverage for warning -> invalid/missing/non-object/no-warning -> same-warning transitions.
- Main still has two `Config warnings:` callsites. This PR guards only the load path, while related #57979 attempted to guard both load and write; maintainers should decide whether write-side warning dedupe belongs in the same fix.
- Runtime snapshot pinning and gateway persisted-hash dedupe can reduce gateway log spam, but they are not a full replacement for config IO warning dedupe and should not be used as the sole reason to close this PR.
