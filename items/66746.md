## Summary

Keep open. Current `main` does not provide strong evidence that the reported Windows sqlite-vec load failure is fixed or obsolete. The report's `FTS-only` wording is partly outdated because current memory search falls back to in-process cosine scoring when embeddings exist, but the underlying complaint about `sqlite-vec` failing to initialize on Windows while standalone `node:sqlite` + `sqlite-vec` works is not disproven from the current codebase or release history available in this checkout.

## Evidence

- **Current vector init path still just loads sqlite-vec and marks it unavailable on failure:** `ensureVectorReady()` calls `loadVectorExtension()`, which calls `loadSqliteVecExtension(...)`; failures only set `vector.available=false` and `vector.loadError` and log `sqlite-vec unavailable`, with no Windows-specific branch or recovery in this path.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:202](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/memory-core/src/memory/manager-sync-ops.ts#L202)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Loader still uses the bundled `sqlite-vec` module or a configured extension path:** `loadSqliteVecExtension()` imports `sqlite-vec`, enables extension loading, then either calls `db.loadExtension(extensionPath)` or `sqliteVec.load(db)`. That matches the area the issue points at and does not itself show a post-report fix for Windows runtime loading.
  - file: [src/memory-host-sdk/host/sqlite-vec.ts:18](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/memory-host-sdk/host/sqlite-vec.ts#L18)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current main no longer means 'FTS-only' when sqlite-vec is unavailable:** If embeddings were produced but sqlite-vec is unavailable, `searchVector()` falls back to listing chunks and scoring `chunks.embedding` with cosine similarity in-process. The docs also state that sqlite-vec unavailability falls back to in-process cosine similarity automatically.
  - file: [extensions/memory-core/src/memory/manager-search.ts:208](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/memory-core/src/memory/manager-search.ts#L208)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Status output can still show `Vector: unknown` without a probe:** `memory status` only probes vector availability in the deep path; otherwise the rendered status shows `unknown` when `status.vector.available` is still undefined. That explains one symptom in the report, but it is not evidence that the Windows loader bug itself is fixed.
  - file: [extensions/memory-core/src/cli.runtime.ts:737](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/memory-core/src/cli.runtime.ts#L737)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **No touched-surface proof of a fix after the latest release in this checkout:** `git diff --name-only v2026.4.23..HEAD -- extensions/memory-core src/memory-host-sdk` does not list `extensions/memory-core/src/memory/manager-sync-ops.ts`, `extensions/memory-core/src/memory/manager-search.ts`, `extensions/memory-core/src/memory/manager-db.ts`, or `src/memory-host-sdk/host/sqlite-vec.ts`, so there is no current-main change here that would justify closing as already fixed.
