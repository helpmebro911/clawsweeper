## Summary

Keep open. Current `main` does not show the reconnect-stall watchdog behavior this PR adds, and the existing tests only cover a single open-without-READY timeout, not repeated open/close stall cycles. That is not strong enough evidence to close as already implemented or obsolete.

## Evidence

- **Current lifecycle has only the per-open READY timeout:** `provider.lifecycle.ts` imports no stall-watchdog helper and defines only the 15s startup READY timeout and 30s runtime READY timeout. There is no separate reconnect-stall deadline in the current file.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:1](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/discord/src/monitor/provider.lifecycle.ts#L1)

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Open/close debug handling resets only the short ready watch:** On `Gateway websocket opened`, current main starts `startReadyWatch()`. On `Gateway websocket closed`, it only calls `clearReadyWatch()` and updates disconnected status. There is no cumulative watchdog preserved across repeated reconnect cycles.
