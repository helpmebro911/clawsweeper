## Summary

Current `main` still appears to skip the primary `agents.defaults.workspace` during memory dreaming whenever `agents.list` is non-empty, so the reported bug remains plausible and should stay open.

## Evidence

- **Dreaming workspace resolver omits the default workspace once `agents.list` exists:** `resolveMemoryDreamingWorkspaces()` collects agent IDs only from `cfg.agents.list` and adds `resolveDefaultAgentId(cfg)` only when that list is empty. If the main workspace lives at `agents.defaults.workspace` and is not represented by a listed agent entry, it is excluded.
  - file: [src/memory-host-sdk/dreaming.ts:595](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/memory-host-sdk/dreaming.ts#L595)
  - command: `nl -ba src/memory-host-sdk/dreaming.ts | sed -n '595,628p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Managed dreaming sweep uses that incomplete workspace list:** `runShortTermDreamingPromotionIfTriggered()` builds `workspaceCandidates` from `resolveMemoryDreamingWorkspaces(params.cfg)` and only falls back to `params.workspaceDir` when the resolved list is empty, so a partial list still skips the primary workspace instead of adding it.
  - file: [extensions/memory-core/src/dreaming.ts:503](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/dreaming.ts#L503)
  - command: `nl -ba extensions/memory-core/src/dreaming.ts | sed -n '503,517p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Another workspace helper already includes the default agent explicitly:** `listAgentWorkspaceDirs()` adds all listed agent workspaces and then always adds `resolveDefaultAgentId(cfg)`, which contrasts with the dreaming-specific resolver and supports the report's claim of inconsistent enumeration.
  - file: [src/agents/workspace-dirs.ts:4](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/src/agents/workspace-dirs.ts#L4)
  - command: `nl -ba src/agents/workspace-dirs.ts | sed -n '1,16p'`
  - sha: [2011de69d3dc](https://github.com/openclaw/openclaw/commit/2011de69d3dc8f1a11ebc774ba4af23f54892461)
- **Current regression coverage does not cover the reported configuration:** The multi-workspace dreaming test only exercises two explicit `agents.list` workspaces (`alpha` and `beta`) and asserts exactly two workspaces processed. There is no test for a separate primary `agents.defaults.workspace` plus listed sub-agents.
  - file: [extensions/memory-core/src/dreaming.test.ts:2084](https://github.com/openclaw/openclaw/blob/2011de69d3dc8f1a11ebc774ba4af23f54892461/extensions/memory-core/src/dreaming.test.ts#L2084)

## Risks / Open Questions

- Nightly dreaming can continue to miss the user's primary workspace while still processing listed sub-agent workspaces, leaving `DREAMS.md` and durable promotion state stale for the main agent.
- Doctor/status surfaces that rely on the same resolver can undercount dreaming data for the primary workspace in the same configuration.
