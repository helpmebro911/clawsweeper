## Summary

Keep open. Current main does not contain either fix from PR #42482: cron job timeouts still start before isolated execution actually begins, and cron model-fallback attempts still reuse the parent abort signal instead of a fresh per-attempt controller.

## Evidence

- **Timeout still arms immediately:** `executeJobCoreWithTimeout` still starts `setTimeout(...)` as soon as it enters `Promise.race(...)` and passes the same abort signal into `executeJobCore(...)`. There is no deferred arming callback such as `onExecutionStart` in current main.
  - file: [src/cron/service/timer.ts:79](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/cron/service/timer.ts#L79)

- **Queue wait is still inside the timed section:** For isolated jobs, `executeJobCore` calls `runIsolatedAgentJob(...)` directly under that outer timeout. The embedded runner then enqueues onto the session lane and global lane before real execution starts, so queue time still consumes the cron timeout budget.
  - file: [src/cron/service/timer.ts:1283](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/cron/service/timer.ts#L1283)
