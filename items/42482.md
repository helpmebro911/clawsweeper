## Summary

Keep this PR open. Current main still lacks both central behaviors the PR is trying to add: the cron timeout is armed before isolated execution starts, and isolated cron model fallback attempts still reuse the same parent abort signal. The PR is stale against current file layout, but it remains plausible useful bug-fix work rather than cleanup-close material.

## Best Possible Solution

Keep this PR or a canonical replacement open. The best maintainer path is to rebase the implementation onto current main, adapt the per-attempt AbortController logic in src/cron/isolated-agent/run-executor.ts, defer or split cron queue-wait versus execution timeout accounting in src/cron/service/timer.ts, and add regression tests proving queued isolated execution does not consume the execution budget and fallback attempts receive fresh non-aborted signals.
