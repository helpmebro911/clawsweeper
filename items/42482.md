## Summary

Keep open. Current `main` still has both behaviors this PR is trying to fix: cron job timeout starts immediately before execution begins, and isolated cron model-fallback attempts still share the same abort signal. I did not find evidence that an equivalent fix landed separately on `main` or in the latest release.

## Evidence

- **Timeout still starts before execution:** `executeJobCoreWithTimeout` still arms `setTimeout(...)` immediately inside `Promise.race(...)` and passes only `runAbortController.signal` into `executeJobCore`, with no deferred arming callback or safety backstop.
