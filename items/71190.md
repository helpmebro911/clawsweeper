## Summary

Keep #71190 open. Current main still prepends the full orphan text into the next prompt without stripping prior queued-message markers or the abort hint, so an orphan that is already an accumulated repaired prompt can be wrapped again. The existing tests prove first-merge, duplicate-text, and structured-content behavior, but not the reported repeated post-abort snowball case.

## Best Possible Solution

Keep the issue open. Fix the helper in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts` so it preserves legitimate orphan delivery but canonicalizes or strips prior `[Queued user message that arrived while the previous turn was still active]` markers and the abort hint before merging. Add focused regression tests for repeated post-abort merges and a negative test that legitimate cross-channel orphan content is still delivered rather than dropped.
