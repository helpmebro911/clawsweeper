## Summary

Current main still preserves and re-merges the entire orphan leaf text. That keeps the earlier #65388 behavior that prevents dropped mid-run user text, but it does not address this issue’s snowball case where the orphan leaf already contains OpenClaw-generated queued-message markers and the aborted-run note.

## Best Possible Solution

Keep this issue open. The best fix is in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts`: normalize only OpenClaw-generated queued-message markers and aborted-run hints out of `orphanText` before merging, preserve the latest real orphan content, and add focused regression tests for an already-marked orphan leaf plus the existing remove-leaf behavior.

## Evidence

- **Merge helper still prepends the full orphan text:** `mergeOrphanedTrailingUserPrompt` extracts the orphan text, checks only whether that full text is already present in the next prompt, then returns `[QUEUED_USER_MESSAGE_MARKER, orphanText, "", params.prompt].join("\n")`. There is no normalization step for an orphan that already contains the queued marker or aborted-run note.
