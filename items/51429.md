## Summary

Keep open. Current `main` still persists the resolved workspace path into config during onboarding/local setup, so the portability bug described in the issue is not implemented away on `main` even though the reporter's `wangtao` hypothesis is likely incorrect.

## Evidence

- **Current onboard wizard still bakes resolved workspace into config:** `runSetupWizard` resolves the user/default workspace with `resolveUserPath(...)` and immediately passes that resolved absolute path into `applyLocalSetupWorkspaceConfig(baseConfig, workspaceDir)`.
  - file: [src/wizard/setup.ts:485](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/wizard/setup.ts#L485)
  - command: `nl -ba src/wizard/setup.ts | sed -n '476,488p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Config helper persists workspace value verbatim:** `applyLocalSetupWorkspaceConfig` writes its `workspaceDir` argument directly to `agents.defaults.workspace` without converting the canonical default back to a portable `~/.openclaw/...` form.
  - file: [src/commands/onboard-config.ts:8](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/onboard-config.ts#L8)

There is no hardcoded `wangtao` string in the repo, but the persisted value is absolute for the current host.
  - file: [src/agents/workspace.ts:17](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/workspace.ts#L17)
