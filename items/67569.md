## Summary

Keep this PR open. Current main still does not populate the synchronous context-window cache from the gateway model catalog, and the session-listing path still builds UI rows through read-only context lookups that do not run model discovery. The PR remains a plausible implementation candidate for #67530, with useful reporter confirmation and author follow-up on provider alias normalization, but unresolved review comments still need maintainer handling before merge.

## Best Possible Solution

Keep this PR open and request a focused revision: populate session-row context windows from precise gateway catalog or persisted runtime data, avoid unconditional 200k defaults before catalog load, retain provider alias normalization, make cache keys collision-safe for slash-containing model IDs, and reset any new global context-cache side effects through existing gateway test helpers.

## Risks / Open Questions

- The proposed unconditional `DEFAULT_CONTEXT_TOKENS` fallback can make the UI show a confident but wrong percentage for uncached discovery-only models whose real window is not 200k.
- Provider-qualified cache writes need to avoid collisions between synthetic provider/model keys and raw slash-containing model IDs such as OpenRouter model IDs.
- Adding gateway catalog writes into the global context-token cache should be paired with reset coverage so gateway tests do not leak model windows across cases.
