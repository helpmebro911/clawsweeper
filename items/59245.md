## Summary

This issue should stay open. Current main has useful voice-call building blocks, including outbound calls, DTMF, realtime/streaming voice, manual continue/speak/end/status controls, and closed-loop turn tests, but it still lacks the requested autonomous outbound task-call workflow. The maintainer discussion already reached the same conclusion on current main, and pchero's VoIPBin/media-offloading comment is useful provider design input rather than a reason to close.

## Best Possible Solution

Keep this issue open and implement the next slice in `extensions/voice-call` as a documented task-call orchestration layer over the existing outbound call, DTMF, realtime voice, transcript, and status primitives. The maintainer design should define a task objective schema, call state machine, IVR/DTMF strategy, hold detection or provider hold events, trusted owner escalation over a configured channel with timeout behavior, allowed-information policy, max duration/cost controls, transcript/reporting, and callback/reschedule semantics. Evaluate VoIPBin-style media offloading as a provider option, not as a substitute for the OpenClaw product contract.

## Risks / Open Questions

- A small patch that only adds another voice-call command would likely under-solve the feature without a real task-call state machine, owner escalation path, and policy model.
- The requested behavior has product and safety design dependencies: legal/AI disclosure, recording consent, cost and duration caps, authorization boundaries, provider reliability, and callback semantics.
- VoIPBin-style media offloading may be valuable, but it still needs an OpenClaw-side task workflow contract and should not be treated as a complete replacement for the requested feature.
