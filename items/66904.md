## Summary

Keep open. Current `main` has partial reconnect handling for WhatsApp via `messages.upsert` `append` events, but it does not listen for `messaging-history.set`, does not expose the proposed reconnect-sync config surface, and explicitly drops stale offline backlog instead of making it available to agent history.

## Evidence

- **No `messaging-history.set` handling in the WhatsApp inbound monitor:** `handleMessagesUpsert` only accepts `notify` and `append`, and the socket listeners wired here are only `messages.upsert` and `connection.update`. There is no `messaging-history.set` listener on current `main`.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:639](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/whatsapp/src/inbound/monitor.ts#L639)
  - command: `nl -ba extensions/whatsapp/src/inbound/monitor.ts | sed -n '639,708p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Current reconnect path skips stale history instead of ingesting it:** The current tests assert that stale `append` messages are marked read but never passed to `onMessage`, which means older offline backlog is not fed into the normal agent/message handling path.
  - file: [extensions/whatsapp/src/monitor-inbox.allows-messages-from-senders-allowfrom-list.test-support.ts:340](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/whatsapp/src/monitor-inbox.allows-messages-from-senders-allowfrom-list.test-support.ts#L340)

## Risks / Open Questions

- Closing as implemented would overstate current behavior: current `main` only covers a narrow recent-`append` reconnect case and still drops older offline backlog.
- The issue still needs product/behavior decisions about whether reconnect history should only populate context buffers or also trigger agent handling for missed messages.
