## Summary

Keep #16121 open. Current main still has only adjacent Telegram custom-emoji handling, such as forum topic icon custom emoji IDs and filtering reaction metadata; it does not provide first-class inline custom emoji rendering for normal Markdown-rendered Telegram messages, captions, replies, previews, or agent action sends.

## Best Possible Solution

Keep #16121 open. Implement inline Telegram custom emoji in the bundled Telegram plugin with a structured Markdown IR/entity representation or explicit Markdown-level syntax carrying `emoji-id` plus fallback text; render safe Telegram HTML as `<tg-emoji emoji-id="...">fallback</tg-emoji>`; preserve plain-text fallback; and cover normal messages, media captions, replies/quotes, preview streaming, action sends, docs, and formatter/delivery tests.

## Risks / Open Questions

- There is a narrow trusted raw-HTML path via `textMode: "html"`, so the remaining work should be scoped to first-class Markdown/action/reply/caption support rather than claiming no caller can ever pass `<tg-emoji>` manually.
- A simple allowlist for raw `<tg-emoji>` in normal model output could weaken the deliberate HTML escaping and plain-text fallback behavior; implementation should preserve safe fallback text and Telegram parse-error recovery.
