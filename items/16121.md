## Summary

Keep #16121 open. Current main still has Telegram custom emoji support only in adjacent/partial places, such as forum topic icon custom emoji IDs and filtering incoming custom-emoji reaction metadata. It does not implement first-class inline `<tg-emoji emoji-id="...">` rendering for normal Markdown-rendered Telegram messages, captions, replies, previews, or agent action sends. The existing 2026-04-26 Codex/maintainer comment already kept this issue open, and the relevant Telegram formatter/send/action/docs files are unchanged since that reviewed SHA.

## Best Possible Solution

Keep #16121 open. Implement inline Telegram custom emoji in the bundled Telegram plugin with a structured Markdown IR/entity representation or explicit Markdown-level syntax carrying `emoji-id` plus fallback text; render safe Telegram HTML as `<tg-emoji emoji-id="...">fallback</tg-emoji>`; preserve plain-text fallback; and cover normal messages, media captions, replies/quotes, preview streaming, action sends, docs, and formatter/delivery tests.

## Risks / Open Questions

- There is a narrow trusted raw-HTML path via `textMode: "html"`, so the remaining work should be scoped to first-class Markdown/action/reply/caption support rather than assuming no caller can ever pass `<tg-emoji>` manually.
- A naive allowlist for raw `<tg-emoji>` in normal model output could weaken the current deliberate HTML escaping and plain-text fallback behavior.
