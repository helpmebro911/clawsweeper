## Summary

Keep open. Current main still lacks the PR's central behavior, and the submitted patch remains a plausible but not-ready implementation because review comments identify concrete correctness issues. Related #41957 is open and same-author, so it is a consolidation target rather than a safe automatic close reason.

## Best Possible Solution

Keep this PR open until maintainers either review it directly or consolidate it with #41957 or the broader user-feedback work in #43204. The implementation should stay narrow: compute typing TTL from the actual fallback candidate budget including configured-primary append and transient retry replay, use a narrow timeout predicate such as `isTimeoutError` for timeout-final messaging, and add regression coverage for true timeout failures versus bare network/DNS errors.
