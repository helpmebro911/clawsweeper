## Summary

Keep this PR open. Current main still lacks the requested provider request concurrency limiter for OpenAI Responses, Azure OpenAI Responses, and OpenAI chat completions, and there is no `LLM_MAX_CONCURRENT` or equivalent shipped config surface. The PR remains relevant, but its current head is incomplete and not mergeable as-is. Security review found no new workflow, dependency, lockfile, script, publishing, or secret-permission changes in the provided diff; the remaining risk is behavioral reliability in provider request scheduling.

## Best Possible Solution

Keep this PR open for revision or maintainer replacement. A mergeable implementation should add a tested provider request limiter at the provider transport/stream boundary, cover OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, hold slots until the streamed response is fully consumed or aborted, reject queued entries promptly on abort, parse invalid configuration safely, and document the chosen config surface. If the same capability lands in another PR, close this PR as superseded by that implementation.

## Risks / Open Questions

- The PR should not merge as-is: it misses the chat completions surface claimed in the body and has formatting that would fail the repo formatter gate.
- The proposed limiter releases slots after SDK stream creation rather than after stream consumption, so long-running streams can still overlap beyond the intended cap.
- Invalid `LLM_MAX_CONCURRENT` values can silently disable throttling because `Math.max(1, NaN)` yields `NaN`.
- Queued work does not reject promptly when the caller's `AbortSignal` fires, which can leave cancelled requests waiting behind active streams.
- An env-only `LLM_MAX_CONCURRENT` surface needs maintainer judgment and documentation to align with the existing provider/config model.
