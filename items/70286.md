## Summary

Keep open. Current `main` still has the reported behavior: a plugin-owned `inbound_claim` handled result returns `queuedFinal: false`, Telegram still uses `!queuedFinal && !deliverySummary.delivered` to send the silent-reply fallback, and the existing tests assert the problematic false result. The linked PR #70288 is useful context, but the fix is not merged into current `main`; in fact, the current dispatch wrapper also collapses `queuedFinal` to false when no final reply count exists, so a robust fix should preserve an explicit async-claim/response-committed signal through the wrapper and Telegram fallback path.

## Best Possible Solution

Keep #70286 open and either merge an updated #70288 or implement the same fix directly with an explicit async plugin-claim/response-committed signal. The change should update `dispatch-from-config`, preserve the signal through `src/auto-reply/dispatch.ts`, and add regression coverage for both plugin-bound `inbound_claim` and Telegram silent-reply suppression.

## Risks / Open Questions

- A narrow `queuedFinal: true` change may not be enough unless the async-claim signal survives `finalizeDispatchResult` and reaches channel handlers.
- The issue comment correctly notes a behavior tradeoff: if a plugin claims a turn and then never responds, suppressing the fallback can leave the user with no feedback. A follow-up timeout or explicit claimed-async state would handle that better.
