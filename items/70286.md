## Summary

Keep open. Current main still contains the reported behavior: a plugin-owned inbound claim with `status: "handled"` returns `queuedFinal: false`, while the Telegram dispatch path still sends the silent-reply fallback when `queuedFinal` is false and no delivery has been observed. The linked PR #70288 appears to target the right code path, but it is open/unmerged in the provided GitHub context, so the issue is not implemented on main or safe to close yet.

## Best Possible Solution

Keep #70286 open until PR #70288 or an equivalent maintainer-reviewed fix lands. The likely code change is to treat `plugin-bound-handled` as a committed async response for channel fallback purposes, with regression coverage spanning `dispatch-from-config` and the Telegram silent-reply fallback path. The broader async-claim timeout/typing contract noted in the issue comment can be tracked separately after the immediate regression is fixed.
