## Summary

Keep open. Current main partially remaps agent events only when an internal runId is explicitly linked to a chat client runId, but the collect/followup queue path still creates a fresh execution runId and does not preserve or expose the original chat.send runId(s). The issue is recent, concrete, unprotected, and describes a remaining gateway/WebSocket contract gap.

## Evidence

- **Agent event schema has no original/source run ids:** The Gateway agent event protocol schema only allows runId, seq, stream, ts, and data, with additionalProperties false; there is no originalRunIds/sourceRunIds field for merged queued requests.
  - file: [src/gateway/protocol/schema/agent.ts:27](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/gateway/protocol/schema/agent.ts#L27)

This covers the direct run path but does not record a future queued execution id or a set of source ids.
