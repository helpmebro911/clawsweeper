## Summary

Keep #71216 open. Current main has adjacent pieces, but it does not implement the requested core config contract: sandbox mode exists under `agents.defaults.sandbox` / `agents.list[].sandbox`, not top-level `sandbox`; there is no `routing.rules` or `instances` top-level config; and `gateway.nodes` still supports command allow/deny only, not path deny rules. The multiple-gateway docs recommend separate profiles/config/state rather than a single `instances.*` declaration, so the central request remains actionable product/config work rather than cleanup-close material.

## Best Possible Solution

Keep the issue open and ask maintainers to decide the core config shape. A pragmatic path is to split the work: document/use existing `agents.defaults.sandbox.mode` for the sandbox part if that is sufficient, design a canonical path-deny policy if core wants `gateway.nodes.denyPaths`, keep multi-gateway `instances.*` as a separate config/doctor validation proposal, and separately evaluate tag-based provider routing against existing agent bindings and model selection.

## Risks / Open Questions

- The issue bundles four separable config/product changes; maintainers may want to split it into narrower issues before implementation.
- `gateway.nodes.denyPaths` would need careful canonicalization across host, node, sandbox, symlink, and mounted-path boundaries to avoid a false sense of enforcement.
- `routing.rules` needs product design around tags, agent/runtime boundaries, provider auth, and whether this belongs in core model selection or a plugin/runtime hook.
