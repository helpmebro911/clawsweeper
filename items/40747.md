## Summary

Keep this PR open. Current main still cannot express Feishu document insertion at the beginning of a document because `feishu_doc(action: "insert")` still requires `after_block_id` and the runtime computes insertion only relative to that existing block. The PR is a focused implementation candidate, but it still needs the reviewed schema validation fix and targeted tests/docs before merge.

## Best Possible Solution

Keep this PR open, or replace it with an equivalent maintainer patch in `extensions/feishu`: make `after_block_id` optional only for `action: "insert"`, add `index: Type.Optional(Type.Integer({ minimum: 0, ... }))`, pass the index through `insertDoc`, default missing `after_block_id` to root insertion at `index ?? 0`, preserve the current after-block behavior, and add focused Feishu doc tests plus skill/docs updates for the new insert mode.

## Risks / Open Questions

- The PR currently validates `index` as a generic number, so negative or fractional values could pass tool validation and fail later at the Feishu API boundary with a less useful error.
- The PR does not add targeted regression coverage for index-only insertion at document root or for preserving the existing `after_block_id` branch.
- The user-facing Feishu doc skill guidance would remain incomplete unless the new insert-by-index mode is documented.
