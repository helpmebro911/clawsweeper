## Summary

Keep open. Current main still starts the remote skill bin probe from the node connect handler before the gateway sends the successful connect response, while the Swift client only starts its normal event listener after `sendConnect()` returns. The original “no handler” diagnosis is partly outdated because macOS node mode now advertises and handles `system.which` and `system.run`, matching the April comment that manual `system.which` can work, but the connect-time probe race and lack of a co-located-node skip remain unaddressed.

## Best Possible Solution

Keep this issue open and fix the gateway/macOS node handshake path. The most direct fix is to defer the initial `refreshRemoteNodeBins()` call until after the node connect response has been sent and the Swift listener is active, or to make the Swift client buffer/dispatch early event frames while waiting for the connect response. A complete fix should also decide whether loopback/co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, then add regression coverage around connect-time remote-bin probing.
