## Summary

Current main still leaves the reported macOS app-backed remote skill bin probe race open. The original “macOS app does not support system.which/system.run” diagnosis is now outdated, but the gateway still starts the automatic remote-bin probe during the node connect path before sending the successful connect response, while the Swift client only starts normal event listening after connect completes. The probe also still has no loopback or co-located-node skip.

## Best Possible Solution

Keep this issue open. The best fix is to defer the initial refreshRemoteNodeBins call until after the connect response is sent and the Swift node listener can process node.invoke.request events, or buffer/dispatch early event frames during waitForConnectResponse. Maintainers should also decide whether loopback/co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, then add regression coverage for connect-time remote-bin probing.
