## Summary

Keep #12590 open. Current main still contains the reported memory-flush bookkeeping path: after a memory-triggered compaction, `runMemoryFlushIfNeeded` can persist the post-increment `compactionCount` as `memoryFlushCompactionCount`, while the flush gate skips when those two values are equal. The current tests still encode that synchronized state, and the supplied discussion includes repeated confirmations plus an explicit maintainer Codex review on 2026-04-26 saying to keep the issue open.

## Best Possible Solution

Keep #12590 open. Fix the bookkeeping in `src/auto-reply/reply/agent-runner-memory.ts` so a flush-triggered compaction does not persist the post-increment `compactionCount` as the marker that suppresses the next eligible flush. Preserve the pre-increment count or replace the equality-based marker with clearer state, then update the current synchronized-counter tests and add regression coverage proving consecutive eligible auto-compaction cycles run `memoryFlush` each time. PR #51421 and the earlier closed fix attempts are useful references, but the issue should close only after a fix lands on main.
