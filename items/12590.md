## Summary

Keep #12590 open. Current main still has the reported counter-locking path: when a memory-flush run observes compaction completion, `runMemoryFlushIfNeeded` increments `compactionCount`, assigns the returned post-increment value to `memoryFlushCompactionCount`, and persists it. The flush gate still skips when those counters match, and current tests still assert that synchronized state.

## Best Possible Solution

Keep #12590 open until `runMemoryFlushIfNeeded` stops recording the post-increment compaction count as `memoryFlushCompactionCount`, or maintainers deliberately change the dedup contract. The likely fix belongs in `src/auto-reply/reply/agent-runner-memory.ts`, with regression coverage proving consecutive eligible auto-compaction cycles run `memoryFlush` each time and tests updated so they no longer assert synchronized counters after flush-triggered compaction. Review or supersede #51421/#69138 as the canonical fix path.
