## Summary

Current main still has the reported stale-display path. Heartbeat state is known in the reply runners, but session usage persistence has no heartbeat-preservation parameter and still writes the just-used model/modelProvider into the persisted session entry that gateway/UI surfaces resolve for display. The direct fix PR referenced from the issue context, #62987, is closed unmerged.

## Best Possible Solution

Keep this issue open and land a narrow persistence fix: add an isHeartbeat parameter to persistSessionUsageUpdate/persistRunSessionUsage, pass it from runReplyAgent and createFollowupRunner, and when true preserve entry.model and entry.modelProvider in both persistence branches while still recording usage, context tokens, cost, prompt metadata, and normal non-heartbeat behavior. Add a focused regression test proving a heartbeat run can execute on the heartbeat model while the persisted session display model remains the normal session/default model.
