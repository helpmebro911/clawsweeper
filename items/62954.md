## Summary

Current main still has the reported stale-display path: heartbeat-aware callers do not pass any heartbeat preservation signal into session usage persistence, and persistence still writes the just-used model/modelProvider into the session entry that gateway and UI display surfaces resolve. The direct fix PR #62987 is closed unmerged in the provided GitHub context, so this issue still tracks real remaining work.

## Best Possible Solution

Keep this issue open and land a narrow persistence fix: add an isHeartbeat or preservation-mode parameter to persistSessionUsageUpdate/persistRunSessionUsage, pass it from runReplyAgent and createFollowupRunner, and preserve entry.model and entry.modelProvider during heartbeat turns in both persistence branches while retaining normal usage, cost, token, context, CLI binding, and non-heartbeat behavior. Add a focused regression test proving a heartbeat run can execute on the heartbeat model without changing the session model shown by resolveSessionModelRef/chat.history.
