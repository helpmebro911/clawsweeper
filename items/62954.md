## Summary

Keep this issue open. Current main still persists the just-used heartbeat provider/model into the session entry fields that gateway and UI surfaces read for the session's displayed model, and the direct fix PR referenced in the issue context is closed unmerged.

## Best Possible Solution

Keep this issue open and implement the narrow persistence fix in the auto-reply session usage path: add an isHeartbeat parameter to persistSessionUsageUpdate/persistRunSessionUsage, pass it from runReplyAgent and createFollowupRunner, and when true preserve entry.model and entry.modelProvider in both persistence branches while still recording usage, context, cost, and prompt metadata. Add a focused regression test with default model != heartbeat model proving heartbeat execution uses the heartbeat model but persisted session display identity remains on the normal session model.
