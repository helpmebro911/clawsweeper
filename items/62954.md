## Summary

Keep open. Current main still has the reported persistence path: heartbeat turns can run with a heartbeat-specific provider/model, but session usage persistence has no heartbeat guard and still writes the just-used `modelUsed` / `providerUsed` into the persisted session entry. Gateway and chat-history surfaces still resolve display identity from those persisted runtime fields. The related cost snapshot work addressed #69347, while the direct fix attempt #62987 is closed unmerged.

## Best Possible Solution

Keep this open and implement the narrow persistence fix: thread `isHeartbeat` through `persistRunSessionUsage` / `persistSessionUsageUpdate` from both `runReplyAgent` and `createFollowupRunner`; when true, preserve the existing `entry.model` and `entry.modelProvider` in both persistence branches. Add a focused regression test with default model != heartbeat model proving the heartbeat run can use the heartbeat model while the persisted session row/chat-history model remains the normal session model. Treat heartbeat CLI bindings and heartbeat cost/token accounting as explicit follow-up decisions unless maintainers want them in the same patch.

- Heartbeat CLI session binding and heartbeat token/cost accounting are adjacent policy questions; they should be evaluated deliberately rather than silently bundled unless maintainers choose that scope.
