## Summary

Keep #38568 open. Current main still renders the system prompt Runtime line from static runtime metadata only; no context-window used/limit/percent value is accepted by runtime prompt params or passed from the embedded or CLI prompt wiring. The existing status/session_status path does expose context usage, but that remains the workaround described in the issue rather than automatic per-turn Runtime prompt injection.

## Best Possible Solution

Implement this in core prompt assembly. Add an explicit per-turn context usage input to runtime prompt params, populate it from the current prompt token snapshot and effective model context cap before system prompt construction in embedded and CLI paths, render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)`, and add focused tests for `buildRuntimeLine` plus prompt wiring while keeping `/status` and `session_status` behavior intact.

## Risks / Open Questions

- The correct implementation should use a fresh pre-request prompt/context snapshot, not stale after-turn provider usage.
- The denominator should use the effective runtime context cap, including configured contextTokens, rather than only native model contextWindow.
- The field is intentionally volatile, so prompt-cache impact should be considered and tested.
