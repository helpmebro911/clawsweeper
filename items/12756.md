## Summary

Keep #12756 open. Current main still has the shared `read` message action and Slack/Discord/Matrix implementations, but Telegram does not advertise, map, dispatch, document, or test `message read`. The related PR #14367 is useful prior art for an inbound-store approach, but the provided GitHub context says it was closed unmerged and the current checkout does not contain that implementation.

## Best Possible Solution

Keep #12756 open and implement Telegram `message read` end-to-end in the bundled Telegram plugin: advertise `read` in discovery, map shared `read` to a Telegram runtime `readMessages` handler, maintain a bounded Telegram-safe inbound message store or equivalent source, align CLI/tool parameters (`target`/`to`, `limit`, `before`, `after`, and topic/thread handling), document platform limitations, and add focused adapter/runtime/contract tests. Treat closed PR #14367 as prior art rather than shipped proof.

## Risks / Open Questions

- A correct Telegram implementation needs a product decision for Telegram Bot API limitations: bots cannot fetch arbitrary historical chat messages, so the likely design is a bounded inbound message store with clear retention, restart, privacy, and pagination semantics.
- The closed #14367 implementation may be useful prior art, but its own review noted end-to-end adapter and parameter wiring gaps, so it should not be treated as ready-to-land without rework.
