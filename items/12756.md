## Summary

Keep #12756 open. Current main still has the shared `read` message action, but the bundled Telegram plugin does not advertise, map, gate, document, or implement a Telegram `read`/`readMessages` path. The prior implementation PR #14367 was closed unmerged, and the latest issue discussion already treats #12756 as the canonical remaining tracker.

## Best Possible Solution

Keep #12756 open and implement the feature in the bundled Telegram plugin: add `read` discovery, action mapping, runtime handling, config/docs/tests, and CLI parameter compatibility for `openclaw message read --channel telegram --target ...`. PR #14367 is useful prior art, but the current main still needs an end-to-end Telegram `read`/`readMessages` path, likely backed by bounded inbound retention rather than direct Bot API history reads.
