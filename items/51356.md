## Summary

Current `main` does not contain this PR’s core changes. The model config schema/type still rejects per-model `options`, model resolution does not carry `options` through discovered/fallback paths, and there is no Ollama tool-call recovery implementation on `main` to match the PR description.

## Evidence

- **Per-model model schema still rejects `options`:** `ModelDefinitionSchema` is strict and only includes `id`, `name`, `api`, `baseUrl`, `reasoning`, `input`, `cost`, `contextWindow`, `contextTokens`, `maxTokens`, `headers`, `compat`, and `metadataSource`; there is no `options` field, so the PR’s sample config would still fail validation on `main`.
  - file: [src/config/zod-schema.core.ts:307](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/config/zod-schema.core.ts#L307)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Model config type still lacks `options`:** `ModelDefinitionConfig` on `main` has no `options?: Record<string, unknown>` property, so the accepted config surface described by the PR is not implemented in the current typed contract either.
  - file: [src/config/types.models.ts:76](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/config/types.models.ts#L76)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Model resolution does not propagate configured `options`:** The discovered-model override path rebuilds the resolved model from configured/discovered fields but only forwards `api`, `baseUrl`, `reasoning`, `input`, `cost`, `contextWindow`, `contextTokens`, `maxTokens`, `headers`, and `compat`. The fallback model builder likewise omits any `configuredModel.options`. That means even if config accepted `options`, current `main` would still drop them before runtime.
  - file: [src/agents/pi-embedded-runner/model.ts:388](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/pi-embedded-runner/model.ts#L388)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **No Ollama recovery/runtime helpers from the PR exist on `main`:** A repo search for the PR’s runtime helpers returned no matches for `recoverToolCallsFromText`, `createConfiguredOllamaStreamFn`, or `resolveOllamaContextWindowTokens`, and there is no `src/agents/ollama-stream.ts` file in the current checkout. That is strong evidence the PR’s Ollama tool-call recovery and context-window plumbing were not merged separately.
  - command: `rg -n "recoverToolCallsFromText|createConfiguredOllamaStreamFn|resolveOllamaContextWindowTokens" src/agents src/config && rg --files src | rg 'ollama-stream'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)

I did not find a current-main path that turns text-mode `<tool_call>...</tool_call>` output into executable tool calls, so the PR’s second major fix is also absent.
