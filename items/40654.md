## Summary

Keep this PR open. Current main still does not implement automatic Discord thread archival after `/acp close`; the PR is a plausible focused implementation candidate, but it needs to be rebased/ported to the current `extensions/discord` plugin layout before maintainer review can finish.

## Best Possible Solution

Keep this PR open as a useful implementation candidate. The best path is to port it onto current main by storing ACP-created Discord child-thread ownership metadata in the Discord conversation binding flow, making `/acp close` request archive-after-reply only for the current ACP-created Discord thread, adding the archive-after-reply Discord outbound path in `extensions/discord`, preserving runtime notices and send overrides, and keeping focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, and outbound payload delivery.

## Risks / Open Questions

- The PR was written against older Discord/core paths, so a maintainer-safe merge requires a rebase or port to the current `extensions/discord` plugin architecture.

- The review-fixed edge cases still matter during any port: degraded close `runtimeNotice` must be preserved, archive failure must not undo close/unbind, and outbound send overrides must be honored.
