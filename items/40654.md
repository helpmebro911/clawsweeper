## Summary

Current main still does not implement archiving ACP-created Discord threads on `/acp close`. The PR remains a plausible implementation candidate, but it needs to be ported from the old `src/discord` / `src/channels/plugins/outbound` layout to the current `extensions/discord` plugin layout and reviewed for a guarded archive command surface.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best maintainer path is to port it onto current main by storing ACP-created Discord child-thread ownership metadata during ACP thread binding, having `/acp close` request archive-after-reply only after close/unbind succeeds for the current owned Discord thread, implementing the post-reply archive in `extensions/discord`, and adding focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, degraded close notices, and outbound send overrides.

## Risks / Open Questions

- The PR patch is not directly current-main-ready because Discord moved into `extensions/discord`; a mechanical apply would miss the active runtime paths.
- Archiving is a persistent, user-visible Discord side effect; the port should fail closed unless the removed binding is the current Discord thread and has reliable ACP-created ownership metadata.
- The proposed `channelData.discord.archiveCurrentThreadAfterReply` shape needs to be internal or guarded. If arbitrary assistant/plugin outbound payloads can set it, they could archive Discord threads outside the intended `/acp close` flow.
- The review-fixed details need to survive the port: degraded close `runtimeNotice`, archive failure partial-success text, and injected send override behavior.
