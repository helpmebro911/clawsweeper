## Summary

Keep this PR open. Current main still implements one-time shared pairing replies for existing pending DM requests, so the requested UX improvement is not already shipped. The PR targets a real gap, but the submitted one-line guard change is incomplete because it would make `onCreated` fire on repeat pending requests, leave current tests/docs inconsistent, and introduce an unthrottled repeated-code reply policy without maintainer approval. Security review found no CI, dependency, lockfile, secret, publishing, or install-script changes in the PR diff; the security-sensitive concern is the access-control behavior change itself.

## Best Possible Solution

Keep this PR open for a complete implementation: resend the existing code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it only fires for `created === true` or introduce a separately named reply-sent/reminder callback, add throttling if maintainers want repeated replies, then update shared pairing tests, affected channel tests, and the public pairing/security docs.

## Risks / Open Questions

- Merging the current one-line patch would make `onCreated` fire for repeat pending requests, producing duplicate logs and potentially repeated one-time side effects in channel code.
- Always resending still-valid pairing codes to unapproved senders without a cooldown or maintainer-approved policy could increase DM spam and code exposure during the one-hour validity window.
- Current shared and channel tests plus public docs lock the no-resend behavior, so this patch would leave the implementation, tests, and user-facing contract inconsistent.
