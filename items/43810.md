## Summary

Keep this PR open. Current main still implements the one-time pairing reply behavior, so the requested repeated-code resend behavior is not already implemented. The submitted one-line patch is also not safe to merge as-is because it changes `onCreated` semantics and conflicts with current shared/channel tests and public docs.

## Best Possible Solution

Keep this PR open for a complete implementation: resend the existing code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it only fires for `created === true` or add a separately named callback for reply-sent behavior, then update shared pairing tests, affected channel tests, and the public pairing/security docs if repeated reminders become the intended behavior.

## Risks / Open Questions

- The submitted patch would make `onCreated` fire for repeat pending requests, producing duplicate logs and possible repeated one-time side effects in channel code.
- Repeatedly resending pairing codes may be a valid UX improvement, but it needs an explicit maintainer decision on spam/throttling behavior for unapproved senders.
- Merging the current patch without test and docs updates would leave shared pairing tests, channel tests, and public pairing/security documentation inconsistent with runtime behavior.
