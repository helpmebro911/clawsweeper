## Summary

Keep #71899 open. Current main still resolves a non-default agent's workspace by appending the agent id to any configured `agents.defaults.workspace`, and the `agents add` wizard uses that resolver as the visible default. That still contradicts the docs' peer-level `workspace-<agentId>` convention and matches the linked #71889 bug; the PR remains a focused, relevant fix rather than obsolete cleanup.

## Best Possible Solution

Keep #71899 open and review/land it or an equivalent fix. The right product/code direction is to make `resolveAgentWorkspaceDir` treat setup-managed main workspace defaults as main-agent workspaces, fall back to peer-level `workspace-<agentId>` for newly added agents, preserve explicit per-agent workspaces and intentional custom shared bases, and keep the resolver, wizard prompt, docs, and changelog covered by regression tests.
