## Summary

Keep this issue open. Current main has useful partial mitigations for local memory and QMD timeouts, but it does not yet provide official Apple Silicon Metal/GPU guidance, a supported CPU/GPU backend option for local embeddings, crash isolation for the in-process local `node-llama-cpp` provider, or the requested serialization of local GGUF embedding batches. The recent issue discussion also adds fresh April 2026 reproductions, so this is not stale.

## Best Possible Solution

Keep this issue open and treat it as the canonical Apple Silicon local-embedding stability/guidance tracker until maintainers either ship documented CPU-only/Metal guidance plus config support, serialize or otherwise constrain local GGUF batches, and contain native embedding failures away from the gateway; or explicitly move the GPU-backend portion into a reviewed narrower PR such as #60347 while keeping crash isolation and docs tracked separately.

## Evidence

- **Recent reproduction remains active:** The provided discussion includes fresh April 24 and April 27, 2026 reports of Apple Silicon `node-llama-cpp`/Metal failures, including one where `NODE_LLAMA_CPP_GPU=false` did not eliminate the teardown assertion and QMD embedding/search loops continued. That is direct current user evidence against stale closure.
- **Local provider still runs node-llama-cpp in process:** The built-in local embedding provider lazy-loads `node-llama-cpp`, calls `getLlama`, `loadModel`, and `createEmbeddingContext` directly, then serves query and batch embeddings from the same provider object. There is no process isolation boundary here that would contain a native ggml abort away from the gateway/plugin runtime.
