## Summary

Keep open. Current `main` still matches the PR's reported bug: `sendRequest()` starts a 10-second timeout without retaining its handle, and neither the successful-response path nor `disposeSession()` clears that timer.

## Evidence

- **Pending request entries still have no timer field:** On current `main`, `LspSession.pendingRequests` stores only `{ resolve, reject }`, so there is nowhere to retain a timeout handle for later cleanup.
  - file: [src/agents/pi-bundle-lsp-runtime.ts:20](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-bundle-lsp-runtime.ts#L20)
  - command: `nl -ba src/agents/pi-bundle-lsp-runtime.ts | sed -n '12,185p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **`sendRequest()` still creates a fire-and-forget timeout:** `sendRequest()` calls `setTimeout(...)` at lines 97-103, but does not assign the handle or add it to the pending request entry. That leaves the timeout alive after a successful response.
  - file: [src/agents/pi-bundle-lsp-runtime.ts:98](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-bundle-lsp-runtime.ts#L98)
