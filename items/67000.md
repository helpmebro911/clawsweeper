## Summary

Keep this issue open. Current main still has active-memory latency mitigations, but it does not implement warm standby or persistent embedded-agent session reuse. Each uncached active-memory lookup still creates a fresh sub-agent session id/session key, usually a fresh transcript file, and then calls the embedded runner, which performs queueing, workspace/plugin/model/auth/hook setup before constructing a new attempt session stack. The related #50963 remains open and broader; the provided context does not show a maintainer decision that it supersedes this active-memory-specific request.

## Best Possible Solution

Keep the issue open and treat it as active-memory embedded-agent warm reuse work. A maintainer-quality fix likely belongs either in the active-memory plugin as a bounded per-parent-session memory sub-agent reuse pool, or in a generic embedded-runner/session API if multiple plugin-owned subagents need the same warm standby semantics. The fix should define TTL/invalidation, preserve isolation/security boundaries, and add tests proving repeated active-memory turns reuse the intended initialized state or skip the expensive setup path.
