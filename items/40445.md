## Summary

Keep open. Current `main` has adjacent tool-loop guardrails, but not the requested passive `ExecutionHealthMonitor`. The shipped feature is `tools.loopDetection`: it is disabled by default, focuses on repeated tool-call patterns, and can block execution. I did not find `agents.defaults.executionHealth`, file-burst detection, or turn-level no-real-side-effect monitoring in current `main`.

## Evidence

- **Existing feature is different and disabled by default:** `src/agents/tool-loop-detection.ts` defines `DEFAULT_LOOP_DETECTION_CONFIG` with `enabled: false` and only `genericRepeat`, `knownPollNoProgress`, and `pingPong` detectors.
  - file: [src/agents/tool-loop-detection.ts:33](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/agents/tool-loop-detection.ts#L33)
- **Current guard operates on repeated tool calls, not passive per-turn execution health:** `detectToolCallLoop(...)` only checks unknown-tool repeats, global no-progress streaks, known polling loops, ping-pong loops, and generic same-tool/same-args repeats. It returns warning/critical loop results and can block execution; there is no file-burst or turn-level side-effect analysis here.
  - file: [src/agents/tool-loop-detection.ts:433](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/agents/tool-loop-detection.ts#L433)
- **Session diagnostic state lacks file-write or side-effect counters:** `SessionState` stores `toolCallHistory`, `toolLoopWarningBuckets`, and `commandPollCounts`, but nothing for file creations, commits/pushes/sends, or consecutive no-effect turns.
  - file: [src/logging/diagnostic-session-state.ts:3](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/logging/diagnostic-session-state.ts#L3)
- **Docs describe the shipped feature as optional tool-loop safety checks:** `docs/tools/loop-detection.md` says the guard is disabled by default and exists to detect repeated tool-call patterns; it also states loop detection can block or dampen the next tool cycle.
  - file: [docs/tools/loop-detection.md:10](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/docs/tools/loop-detection.md#L10)
- **Config surface is `tools.loopDetection`, not `agents.defaults.executionHealth`:** The runtime schema exposes `ToolLoopDetectionSchema` and docs wire it under `tools.loopDetection`; I found no `executionHealth` config key on current `main`.
  - file: [src/config/zod-schema.agent-runtime.ts:500](https://github.com/openclaw/openclaw/blob/171322644dece55402569c52b26370fd81769ca6/src/config/zod-schema.agent-runtime.ts#L500)
  - command: `rg -n "ExecutionHealth|executionHealth|agents\.defaults\.executionHealth" -S .`

- `main` now has partial mitigation for the same class of runaway behavior, so maintainers may want to re-scope this issue toward the still-missing pieces: passive/default-on monitoring, file-burst detection, and no-effect-turn tracking.
