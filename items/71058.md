## Summary

Keep open. Current main still models Microsoft Teams as a single `channels.msteams` object with one bot identity, one credential set, one webhook listener context, and one default account. The existing multi-account routing/account helpers in core provide a plausible path, but the bundled Teams plugin has not adopted them, and the related Feishu multi-account cleanup does not solve Teams.

## Best Possible Solution

Keep the issue open and implement this in the bundled Microsoft Teams plugin. A conservative design would preserve the current single-object config, add the repo-standard `channels.msteams.accounts.<id>` plus `defaultAccount` shape, resolve credentials/webhook/JWT validation per account, account-scope stored conversation references and proactive sends, and route named Teams bot identities to agents through existing `bindings` account matching rather than inventing a separate `agentBinding` contract.

## Risks / Open Questions

- A schema-only change would be incomplete: Teams webhook auth, SDK app/token provider selection, conversation reference storage, proactive sends, probes, Graph calls, and docs/tests all need account-aware handling.
- The proposed `agentBinding` field should probably be reconciled with OpenClaw's existing `bindings[].match.accountId` routing model instead of adding a parallel routing mechanism.
