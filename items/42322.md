## Summary

Keep this PR open. Current main still does not expose a Feishu replyToMode config surface, and inbound Feishu replies still pass a reply target into the dispatcher, so the requested behavior control is not obsolete. The PR is useful but not merge-ready as-is: it needs a rebase onto the current Feishu reply-target refactor, docs/manifest/config metadata, tests, and a complete first-mode implementation for streaming cards plus media.

## Best Possible Solution

Keep the PR open and ask for a Feishu-owned implementation pass: rebase onto current main, add replyToMode to schema, manifest/config metadata, docs, and generated metadata; define precedence across group, account, and top-level config; apply the policy consistently to streaming cards, static cards, text chunks, and media; and add focused tests for off, first, and all, including a streaming-plus-media case. The shared reply policy helpers in src/infra/outbound/reply-policy.ts are a good implementation reference.
