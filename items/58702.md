## Summary

Keep open. The report is concrete, non-stale, from a non-maintainer, has no protected labels, and current main still shows the same actionable WebChat/control UI mismatch described in the issue discussion: the CSS reserves space only for `.chat-bubble.has-copy`, while the grouped message renderer builds `bubbleClasses` without adding `has-copy` even when copy/expand actions are rendered.

## Evidence

- **Issue discussion identifies the same root cause:** Commenter 1168784 pointed out that `.chat-bubble.has-copy` exists in CSS but the JS/renderer never applies `has-copy`, so copy/expand actions can overlap message text. That matches the reporter's screenshot and reproduction.
  - command: `provided GitHub context: issue comment 4169559798`
- **CSS still depends on has-copy for action spacing:** Current main defines `.chat-bubble.has-copy { padding-right: 70px; }` and positions `.chat-bubble-actions` absolutely at the top-right, hidden until hover except on non-hover devices.
  - file: [ui/src/styles/chat/grouped.css:1580](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/ui/src/styles/chat/grouped.css#L1580)
  - command: `GitHub blob inspection of current main`
  - sha: [afd6b5d6fce1](https://github.com/openclaw/openclaw/commit/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386)
- **Renderer still omits has-copy from bubbleClasses:** Current main computes `canCopyMarkdown`, `canExpand`, and `hasActions`, renders action buttons when `hasActions` is true, but `bubbleClasses` is built only from `chat-bubble`, optional `streaming`, and `fade-in`; `has-copy` is not included.
  - file: [ui/src/ui/chat/grouped-render.ts](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/ui/src/ui/chat/grouped-render.ts)
