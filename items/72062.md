## Summary

Keep #72062 open. Current main still has no `agents.list[].tools.deny_wrappers` or equivalent per-agent exec script/path deny policy. `tools.allow`/`tools.deny` remains a tool-name filter, the typed and strict exec config schemas expose only host/security/ask/node/path/safe-bin/inline-eval knobs, and host exec approvals still model per-agent allowlists without a negative wrapper-path layer. Related #69570/#69386/#57568 cover approval-loop symptoms or an unmerged `/approve` loop PR, not this specific core exec policy gap.

## Best Possible Solution

Keep the issue open and design an additive core exec policy, likely under `agents.list[].tools.exec` and optionally global `tools.exec`, that denies resolved script/wrapper paths before approval/allowlist success. Reuse the existing shell wrapper script candidate resolution and exec glob matcher, make deny win over allowlist/durable approvals, add gateway/node tests plus config-schema/docs coverage, and cross-link #69570 only as a related approval-cascade symptom rather than the canonical tracker.
