## Summary

Current main still lacks memory-lancedb agent scoping, so this PR is not obsolete. Keep it open as a plausible bundled-plugin implementation candidate, but it is draft/stale against current main and needs correctness and security-hardening fixes before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate for bundled memory-lancedb per-agent isolation. The best path is to rebase it onto current main, keep the fix inside the memory-lancedb plugin, use trusted ctx.agentId consistently in tool factories and before_prompt_build/agent_end hooks, add a backward-compatible agentScoping config/manifest surface, fix the LanceDB filtering and migration observability issues, and add regression coverage for agent A/agent B isolation, legacy global memories, duplicate detection, forget/delete scoping, and default unscoped compatibility.

## Risks / Open Questions

- The current main behavior still permits cross-agent memory recall/store/forget through one shared memory-lancedb table.
- The PR is stale against current main: current main uses before_prompt_build and live runtime config handling, while the PR diff is based on older before_agent_start-era code.
- The PR’s LanceDB scoped search may post-filter after limiting ANN candidates, starving valid results for the requested agent.
- The PR stores raw agentId but filters/deletes with a stripped value, which can make scoped memories unrecoverable and undermines the stated validation mitigation.
- The PR silently swallows migration failures, so operators could believe agent scoping is active when the agentId column was not added.
- The PR’s security-sensitive filter interpolation needs a maintainer-approved validation/escaping contract for trusted ctx.agentId before merge.
