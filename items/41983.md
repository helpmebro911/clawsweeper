## Summary

Keep this PR open. Current main still scopes control-plane write limits by `deviceId+clientIp` when either identity value exists, while the PR proposes a real gateway behavior change to isolate budgets by `connId`. That change is plausible but needs maintainer security/product judgment and a rebase that preserves current bucket pruning and hard-cap safeguards.

## Best Possible Solution

Keep this PR open for maintainer review or author update. The best path is to rebase onto current main, preserve pruning and the bucket hard cap, rename/update the stale keying test, add a regression test proving two concurrent sessions from the same device/IP get isolated budgets, and update gateway docs/changelog if maintainers choose per-connection write budgets as the intended behavior.
