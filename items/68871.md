## Summary

Keep open. Current main has several env-proxy paths, including Google auth/gaxios transport and shared undici fetch helpers, but they only decide from proxy environment variables and NO_PROXY. They do not probe whether the configured proxy endpoint is listening, and failures after EnvHttpProxyAgent/ProxyAgent creation still propagate rather than falling back to direct connection. The related Telegram polling stall report in the provided context is narrower and does not supersede this env-proxy reachability request.

## Best Possible Solution

Keep this issue open and implement a bounded proxy-reachability policy in the shared env-proxy layer, then wire it through the Google auth/gaxios-compatible transport and guarded fetch paths. The safest direction is a cached, timeout-bounded TCP probe or request retry for local/operator proxy URLs, with tests for a closed loopback proxy, NO_PROXY behavior, malformed proxy URLs, and no silent bypass of mandatory corporate proxies.

## Risks / Open Questions

- Current behavior can still produce ECONNREFUSED when HTTPS_PROXY/HTTP_PROXY points at a closed local proxy port.
- A naive automatic direct fallback could bypass an intentionally required proxy, so the implementation should treat reachability fallback as an explicit, well-scoped operator convenience with tests and clear security semantics.
