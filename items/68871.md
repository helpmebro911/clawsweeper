## Summary

Keep #68871 open. Current main still decides to use environment or explicit proxies from configuration and NO_PROXY matching, but I did not find source or tests that probe whether the proxy socket is reachable and then retry/direct-route after a closed proxy port causes ECONNREFUSED. The closest fallbacks only cover malformed/constructor-time dispatcher failures or Telegram DNS/IP fallback cases, not an unavailable local proxy service.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A good implementation would add a narrow, well-tested proxy reachability/fallback policy in the shared proxy resolution/dispatcher layer, then thread it through model-provider, Google auth, media, and Telegram paths without bypassing intentional proxy requirements. Tests should cover valid proxy env with closed 127.0.0.1 port, malformed proxy URLs, NO_PROXY bypass, explicit-proxy behavior, and a policy case where direct fallback must not occur.

## Risks / Open Questions

- A naive automatic direct fallback could unexpectedly bypass an operator-required corporate proxy or weaken SSRF assumptions. The implementation should be scoped, probably opt-in or limited to local loopback proxy-manager cases, with short timeouts, clear logging, NO_PROXY parity, and regression tests for both env-proxy and explicit-proxy paths.
