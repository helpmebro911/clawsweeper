## Summary

Keep open. The report is recent and concrete, and current main does not provide high-confidence evidence that this exact Bailian/OpenAI-compatible memory_search failure is fixed. Main has later proxy-aware remote embedding transport, but the documented/current memory config still resolves provider names as registered memory embedding adapter ids, not arbitrary `models.providers` entries.

## Evidence

- **Issue discussion remains unresolved:** The provided issue context reports `memory_search` returning `disabled: true`, `unavailable: true`, and `error: "fetch failed"` for a Bailian/DashScope embedding setup on OpenClaw 2026.3.2. The only comment asked the reporter to compare runtime config, curl the endpoint, check env shadowing, and inspect gateway logs; no follow-up evidence confirmed a fixed root cause.
- **Current docs list fixed memory adapter ids:** The memory config reference describes `agents.defaults.memorySearch.provider` as an embedding adapter id and lists built-in ids such as `bedrock`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, and `voyage`, with custom OpenAI-compatible endpoints configured via `memorySearch.remote.baseUrl`/`apiKey`.
