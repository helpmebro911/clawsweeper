## Summary

Current main still treats billing failures as auth-profile disabled state, not model-scoped state. LiteLLM is a bundled proxy-style provider, and a single `litellm:default` profile in `disabledUntil` still blocks sibling LiteLLM models except for limited primary probe cases. Related #66315/#66314 fixed billing guidance for fallback-summary errors, but did not change cooldown scope or add a supported billing-backoff disable/zero option.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate proxy-aware or model-scoped billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented supported opt-out for operators who manage failover inside their proxy. The likely code path is auth-profile usage state plus model fallback, with a regression test using one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other; update config schema/help/docs if a public opt-out is added.

## Risks / Open Questions

- A model-scoped billing policy needs to distinguish true provider-key exhaustion from proxy/upstream-model billing failures; making every billing failure model-scoped could repeatedly retry an actually exhausted account.
- A supported `billingBackoffHours: 0` or disable flag would need clear operator-managed failover semantics to avoid retry storms and unexpected provider cost or rate-limit pressure.
