## Summary

Keep this PR open. Current main still lacks Discord cross-instance channel ownership, and this PR is a recent, targeted implementation candidate for the same-token multi-instance routing bug described in the linked discussion. It should not be closed as obsolete or implemented, but it still needs maintainer review before merge because the proposed shared claims file affects routing and has unresolved correctness/security-sensitive edge cases.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best product path is a Discord-plugin-owned fix under `extensions/discord/src/monitor` that performs robust per-instance and per-bot ownership checks before local allowlist denial in message preflight, autocomplete, and native slash-command paths, with tests for same-token multi-instance routing, guild-wide and channel-scoped allowlists, bot scoping, denied/disabled channels, wildcard and name-based channel entries, resolved channel IDs, shared-file concurrency or per-instance claim files, and lifecycle cleanup.

## Risks / Open Questions

- Closing now would discard the only open implementation candidate found for a still-missing Discord same-token multi-instance routing bug.
- The PR still has maintainer-review issues: shared-file read/modify/write is not serialized, the temp path is process-pid based, stale or dropped claims can affect routing, and file permissions/privacy for stable Discord bot/guild/channel IDs should be deliberate.
- The latest PR diff still appears to omit `botId` from native command/autocomplete claim lookups and uses `message.channelId` rather than the resolved `messageChannelId` in message preflight, matching unresolved review concerns about multi-bot scoping and partial payloads.
- The claim builder only records numeric channel keys and does not clearly exclude denied/disabled channel entries, so name-based allowlists and explicit denies may still route incorrectly unless addressed before merge.
