## Summary

Current `main` and the released `v2026.4.22` tree already contain cron fixes that clear stuck `runningAtMs`, recover stale markers after restart, persist delivery state, and bound isolated runs with a safety timeout when no explicit timeout is set.

## Evidence

- **Run completion clears running marker and writes delivery state:** `applyJobResult()` unconditionally clears `job.state.runningAtMs` and persists `lastDeliveryStatus`/`lastDeliveryError` on completion.
  - file: [src/cron/service/timer.ts:416](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/service/timer.ts#L416)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Safety timeout for isolated agent-turn cron jobs:** `resolveCronJobTimeoutMs()` gives isolated `agentTurn` jobs a 60 minute safety timeout when `timeoutSeconds` is omitted, preventing indefinite hangs from wedging cron forever.
  - file: [src/cron/service/timeout-policy.ts:16](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/service/timeout-policy.ts#L16)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Startup clears stale running markers:** Cron startup now clears persisted `runningAtMs` markers before catch-up, so a gateway restart no longer leaves recurring jobs stuck in `running` until the 2 hour maintenance sweep.
  - file: [src/cron/service/ops.ts:93](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/service/ops.ts#L93)

  - file: [src/cron/service.restart-catchup.test.ts:118](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/service.restart-catchup.test.ts#L118)

## Risks / Open Questions

- An explicit `payload.timeoutSeconds: 0` still disables the timeout, so a deliberately no-timeout job can still remain running until other cleanup paths intervene.
- The original report included no logs or job config, so a separate delivery-target/configuration problem on an older build cannot be ruled out from the issue text alone.

## Close Comment

Closing from a Codex maintainer review because this is already implemented on current `main`, and the released `v2026.4.22` tree contains the same cron behavior. The current code clears `runningAtMs` when a run finishes in `src/cron/service/timer.ts:416`, persists delivery status there, adds a safety timeout for isolated agent-turn jobs in `src/cron/service/timeout-policy.ts:16`, and clears stale persisted `runningAtMs` markers on startup in `src/cron/service/ops.ts:93`. The repo also has regression coverage for restart recovery in `src/cron/service.restart-catchup.test.ts:118` and timeout cleanup in `src/cron/service/ops.regression.test.ts:200`. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with `openclaw cron runs --id <jobId>` output, the job's delivery mode/target, whether `timeoutSeconds` was set (or `0`), and any relevant gateway restart timing so maintainers can isolate the remaining case.
