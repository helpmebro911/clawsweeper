## Summary

Keep this issue open. Current main still returns the explicit-token failure before trying the bound-thread ACP resolver, so the reported bound-thread `/acp close <non-session-token>` path remains broken. The matching fix PR #66338 was closed unmerged, and existing coverage only proves bound-thread close when no bad token is present.

## Best Possible Solution

Keep this issue open and land a focused resolver fix: when explicit token resolution returns null, try `resolveBoundAcpThreadSessionKey` before returning the explicit-token error. Preserve the existing bad-token error when no bound thread target resolves, and add regression coverage for `/acp close <non-session-token>` inside a bound thread plus the no-binding bad-token case.
