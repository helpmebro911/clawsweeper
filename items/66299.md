## Summary

Keep this issue open. Current main still returns `Unable to resolve session target: <token>` as soon as a non-empty ACP target token fails gateway resolution, so the bound-thread resolver is not reached for the reported Discord slash-command/thread-id case. The related fix PR #66338 was closed unmerged, and the current tests only cover bound-thread ACP commands without the bad-token fallback case.

## Best Possible Solution

Keep this issue open and land a focused resolver fix equivalent to the closed PR #66338: when token resolution returns null, try `resolveBoundAcpThreadSessionKey` before returning the explicit-token error. Preserve the existing bad-token error when no thread binding resolves, and add regression coverage for `/acp close <non-session-token>` inside a bound thread plus the no-binding bad-token case.
