## Summary

Keep this PR open. Current main still does not implement a pre-dispatch Signal typing indicator, and the Signal group-history work described in the PR still happens before dispatch. The PR is a focused UX fix candidate, but it should not be merged as written because its unconditional pre-dispatch `sendTypingSignal` would run before current delivery and typing suppression policy is resolved.

## Best Possible Solution

Keep this PR open and rework it into a policy-aware implementation. The best path is to start Signal typing earlier through the dispatch/reply pipeline after route/session `sendPolicy` and `typingMode` have been resolved, or add an equivalent Signal-local guard that respects `sendPolicy: "deny"`, `typingMode: "never"`, and the intended group-versus-DM scope. Add regression coverage proving early group feedback happens while denied or disabled sessions do not emit typing.

## Risks / Open Questions

- Merging the PR as written could emit a Signal typing indicator when outbound delivery is denied, leaking bot presence despite the `sendPolicy: "deny"` contract.
- Merging the PR as written could ignore `typingMode: "never"` or other typing suppression paths because it runs before the reply pipeline resolves typing policy.
- The implementation scope is unclear: the PR is framed around group latency, but `ctxPayload.To` exists for both Signal DMs and groups.
