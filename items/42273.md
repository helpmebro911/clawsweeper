## Summary

Keep open. Current `main` still backs up the entire resolved state directory as a single asset, passes that whole tree directly to `tar`, and exposes no exclude or smart-exclude flags. The linked proposed fix remains open, so there is not strong evidence that #42273 is implemented or obsolete on `main`.

## Evidence

- **Backup plan always includes the full state dir:** `resolveBackupPlanFromPaths()` seeds backup candidates with the resolved state directory first, then optional config/credentials/workspaces. There is no logic here to exclude large subdirectories like `logs/`, `browser/`, `session-index/`, or `media/` from the state backup.
  - file: [src/commands/backup-shared.ts:146](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/commands/backup-shared.ts#L146)
  - command: `nl -ba src/commands/backup-shared.ts | sed -n '146,158p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Archive creation tars every included asset root as-is:** `createBackupArchive()` writes a manifest, then calls `tar.c()` over `[manifestPath, ...result.assets.map((asset) => asset.sourcePath)]`. That means the whole state tree is archived once selected; there is no per-subpath filtering or smart exclusion in the create path.
  - file: [src/infra/backup-create.ts:353](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/infra/backup-create.ts#L353)
  - command: `nl -ba src/infra/backup-create.ts | sed -n '353,369p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **CLI still lacks exclude-related flags:** The `backup create` command currently exposes `--output`, `--json`, `--dry-run`, `--verify`, `--only-config`, and `--no-include-workspace`. There is no `--smart-exclude`, `--exclude`, `--exclude-file`, `--exclude-logs`, or `--exclude-media` option on `main`.
  - file: [src/cli/program/register.backup.ts:23](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/cli/program/register.backup.ts#L23)

That matches the reported gap rather than a shipped fix for large `.openclaw` installs.

## Risks / Open Questions

- I did not run a live reproduction because this review is read-only and the report depends on a multi-GB local state directory, so the conclusion is based on current code/docs rather than an end-to-end stall repro.
- `tar.c({ file, gzip: true })` already writes to a file path, so one suggested cause in the issue body ('buffering in memory') is not proven by the current code; the unresolved part is the large-install failure mode, not that specific diagnosis.
