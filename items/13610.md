## Summary

Keep #13610 open. Current main has the shipped SecretRef baseline for env/file/exec providers and docs for wrapping Vault/1Password via exec, but it still does not implement native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager provider sources. The issue discussion explicitly preserves #13610 as the umbrella for that remaining first-class provider work after #26155 shipped the baseline.

## Best Possible Solution

Keep #13610 open as the native secrets-manager provider umbrella. The maintainer path is to either add first-class AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager support across SecretRef types, config schema, resolver dispatch, CLI configure/audit/apply flows, docs, and tests, or make an explicit product/API decision that exec wrappers or a new public secret-provider plugin/ClawHub API are the supported route.

## Risks / Open Questions

- Exec providers can operationally call Vault, 1Password, cloud CLIs, or custom resolvers, but that is not the same as native provider source types, provider-specific schema, resolver dispatch, caching/audit semantics, CLI UX, and tests.
- OpenClaw’s VISION.md prefers optional capabilities as plugins when the API supports them, so maintainers may still decide the best long-term path is a public secret-provider plugin API rather than bundling each vendor in core.
