Keep this PR open. Current main still rejects unknown config keys through the strict schema/read path and startup invalid-config handling, while this PR remains a plausible focused implementation candidate. The open related issue #40268 and open related PR #47654 overlap, but the provided context does not show a maintainer decision making either one canonical or superseding this PR.

## Best Possible Solution

Keep this PR open for maintainer judgment. The best path is to rebase or consolidate it with #40268 and #47654, then land a single implementation that tolerates unknown keys on the read/startup path while preserving fail-closed validation for invalid known fields, legacy keys, and union/discriminator mismatches. Before merge, review the `.secrets.baseline` changes and run the targeted config validation/write/migration tests plus the changed gate required for the final touched files.
