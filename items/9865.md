## Summary

Keep #9865 open. Current main still has no first-class OpenAI/Anthropic Batch API routing for delayed background LLM turns; existing batch code is limited to memory embeddings, media/STT, message coalescing, or config batching. The latest issue discussion also explicitly treats #9865 as the canonical tracker rather than a duplicate or ClawHub-only request.

## Best Possible Solution

Keep #9865 open as the canonical product/API tracker. The best path is an additive provider/runtime batch-turn contract with submit, poll, cancel, timeout, failure, usage, and transcript reconciliation semantics, then explicit opt-in fields for cron agentTurn jobs, heartbeat/background runs, sessions_send, and sessions_spawn/subagents, with OpenAI/Anthropic plugin implementations, docs, and focused regression coverage.

## Risks / Open Questions

- Provider Batch APIs need explicit lifecycle semantics before implementation: delayed delivery, polling cadence, cancellation, timeout/fallback behavior, failure mapping, usage accounting, and transcript reconciliation.
- Existing references to batching in embeddings, media, config, and message coalescing can look related but do not solve delayed provider Batch API routing for LLM agent turns.
