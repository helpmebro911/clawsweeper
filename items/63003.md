## Summary

Current main still lacks the requested `sandbox.mounts` JSON field and `Effective mounts:` text section. The linked issue #62988 is closed as superseded by this PR, so this PR remains the active implementation candidate, but the current diff still needs maintainer cleanup because mount output is security-boundary introspection.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Revise it to compute sandboxed status through `resolveSandboxRuntimeStatus`, derive effective mounts from the selected sandbox backend or a shared backend-aware mount contract, keep both JSON and text output, add focused coverage for JSON, text, Docker binds, direct runtime, session aliases, agent workspace overrides, and remote roots, and remove unrelated test-file churn before maintainer review.

## Risks / Open Questions

- Merging the current diff can report mounts for sessions that canonical runtime status would treat as direct, especially main/global aliases.
- Merging the current diff can report Docker-style container roots for SSH/OpenShell-style backends where effective roots come from remote runtime paths.
- Mount metadata is additive output, not code execution, but it is security-relevant because users may rely on it to understand filesystem exposure.
- The PR includes unrelated import-order churn in `test/image-generation.runtime.live.test.ts`, which should be removed before maintainer review.
