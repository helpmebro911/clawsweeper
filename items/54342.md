## Summary

Keep this issue open. Current main addresses adjacent pieces, including ACP background task tracking and hook delivery/message-tool duplicate suppression, but the central channel-less ACP persistence path is still missing: `sessions_spawn(runtime:"acp", mode:"session")` still requires `thread:true`, the accepted spawn result still does not expose the ACP backend/agent session identity, and `/hooks/agent` still executes through the isolated cron embedded-agent path rather than dispatching known ACP session keys into ACP `runTurn`.

## Best Possible Solution

Keep this issue open and narrow it to a conservative headless ACP persistence contract. The likely path is to add an explicit headless persistent ACP mode, or narrowly relax the `mode:"session"` thread guard when `streamTo:"parent"` and lifecycle cleanup are well-defined; expose the ACP backend/agent session identity or a documented status lookup path; add regression coverage for channel-less hook-triggered ACP `mode:"session"` spawns; and separately decide whether `/hooks/agent` should route known ACP session keys into `acpManager.runTurn` or whether a dedicated `sessions_send`/ACP send tool should own that follow-up path.

## Risks / Open Questions

- Closing now would leave hook-triggered/headless ACP workflows without a supported persistent `mode:"session"` path for resume or multi-turn continuity.
- A future fix should not simply remove the thread guard; headless persistent ACP sessions need explicit cleanup, visibility, delivery, ownership, and timeout semantics.
- Exposing backend resume identities or adding hook-to-ACP `runTurn` routing should preserve existing ACP policy checks and task/session visibility boundaries.
