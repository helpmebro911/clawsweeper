## Summary

Keep #38687 open. Current main still resolves a proxy-aware media fetch before it knows the configured audio/video backend baseUrl, so it cannot skip the wrapper for a NO_PROXY-covered self-hosted STT target. Target-aware proxy helpers exist elsewhere, but they are not wired into this runner-level fetch selection, and the current runner tests still cover only the old env-only behavior.

## Best Possible Solution

Keep #38687 open and adapt it to current main: resolve the media provider `baseUrl` before choosing `fetchFn`, gate the wrapper with `shouldUseEnvHttpProxyForUrl` or a deliberate extension of the shared proxy matcher, and add focused runner/proxy tests proving NO_PROXY-covered self-hosted audio backends do not receive the proxy fetch while non-excluded backends still do.
