## Summary

Keep this issue open. Current main still leaves `tools.web.search.provider` as an arbitrary string in the base schema and does not add plugin-aware provider-id validation before or during config validation. Runtime behavior has improved for explicitly passed unknown provider IDs, but config-sourced typos still fall through to provider fallback behavior, which is the core gap reported here.

## Best Possible Solution

Keep this issue open. Add plugin-aware validation for `tools.web.search.provider` in the config validation path before the no-plugins early return, deriving accepted provider IDs from bundled and installed plugin manifest `contracts.webSearchProviders`. Reject clear typos with an actionable validation issue, preserve warning-only behavior for stale missing-plugin configs, and add focused tests for built-in providers, manifest-declared plugin providers, invalid typos, and missing-plugin compatibility.

## Risks / Open Questions

- Typos in `tools.web.search.provider` still pass config validation and can silently select a different provider through runtime fallback.
- A fix needs to preserve stale or removed plugin compatibility as a warning path rather than blindly blocking gateway startup for every unknown provider string.
- The validator should derive accepted IDs from manifest `contracts.webSearchProviders` for bundled and installed plugins, not restore a hardcoded enum that blocks third-party plugins.
