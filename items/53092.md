## Summary

Keep open. Current main still does not validate `tools.web.search.provider` against manifest-registered web search provider IDs in the config validation path. It accepts arbitrary strings in the base schema, the generated schema still exposes a plain string, and runtime handling warns/falls back for invalid config values rather than rejecting clear typos during config validation.

## Best Possible Solution

Keep this issue open. Add plugin-aware validation for `tools.web.search.provider` in the config validation path, deriving accepted provider IDs from bundled and installed plugin manifests via `contracts.webSearchProviders`. Reject clear unknown provider typos with an actionable config issue, preserve warning-only behavior for stale/missing-plugin compatibility, and add focused tests for built-in providers, extension-registered providers, invalid typos, and stale missing-plugin configs.
