## Summary

Keep open. Current main still accepts any `tools.web.search.provider` string at schema/config-validation time, while runtime and tests preserve fallback behavior for unknown configured provider IDs. The plugin manifest contract metadata needed for a real fix exists, so this remains actionable OpenClaw core/plugin-boundary work rather than cleanup-close material.

## Best Possible Solution

Keep this issue open. Add plugin-aware config validation for `tools.web.search.provider` in the config validation layer, deriving accepted provider IDs from manifest/runtime plugin web-search provider metadata, rejecting or clearly reporting typos, and preserving a warning-only path for stale missing-plugin configs. Cover built-in providers, manifest-declared third-party providers, and invalid provider typos with focused regression tests.
