## Summary

Keep PR #71503 open. Current main at 5c4c33c7ded3afe04ed9a710ac13bb9a486ea008 still has the old LAN address selection behavior: gateway display prefers only exact en0/eth0 and then first external IPv4, while pairing setup and the bundled device-pair plugin still take the first private IPv4 in interface iteration order. The PR directly addresses the closed #71493 symptom and has active follow-up addressing Greptile's CNI/default-route review points, so it is not obsolete or superseded.

## Best Possible Solution

Keep the PR open and continue review toward landing a narrow fix: update current main's gateway/status LAN picker, pairing setup URL resolution, and bundled device-pair plugin behavior to prefer default-route or physical LAN interfaces before Docker/bridge/CNI candidates, with regression tests for Docker bridge, CNI overlay, default-route, bridge-only fallback, and pairing URL consistency. A later plugin-facing SDK helper can remove the mirrored device-pair logic if maintainers want a public contract for this selection behavior.
