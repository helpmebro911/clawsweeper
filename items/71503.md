## Summary

Keep this PR open. Current main still uses order-dependent LAN IPv4 selection for gateway dashboard/status, setup-code QR pairing, and the bundled device-pair plugin. The PR remains the active implementation candidate for linked #71493, and the discussion shows the author already addressed the CNI-prefix review note and added default-route preference while preserving bridge/container fallback behavior.

## Best Possible Solution

Keep this PR open for maintainer review or land an equivalent fix. The best path is consistent LAN address selection across gateway dashboard/status, core setup-code QR pairing, and device-pair: prefer default-route or physical/default-looking LAN interfaces, deprioritize Docker/bridge/veth/CNI/container interfaces as fallback candidates, preserve explicit URL/config overrides, and keep regression coverage for Docker bridge, CNI overlay, and default-route cases.
