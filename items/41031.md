## Summary

Keep open. Current main still resolves heartbeat runs to the main session when neither `opts.sessionKey` nor `heartbeat.session` is set; explicit `heartbeat.target`/`heartbeat.to` is only applied later for delivery. The PR is a narrow, still-relevant implementation candidate rather than something already implemented or superseded.

## Best Possible Solution

Keep this PR open. The best path is to review and adapt the patch on current main so explicit `heartbeat.target` plus `heartbeat.to` can derive the run session through `resolveHeartbeatDeliveryTarget` and `resolveOutboundSessionRoute` when no forced or configured heartbeat session is present, while preserving existing fallbacks for `target: last`, `target: none`, global scope, subagent keys, and explicit `heartbeat.session`. The regression test should cover the explicit Telegram group case plus enough fallback coverage to protect existing behavior.

## Risks / Open Questions

- The PR likely needs a rebase/adaptation because current main has additional heartbeat preflight, isolated-session, and event-drain behavior around the same functions.
- The semantics should be confirmed by a maintainer: current docs describe heartbeat as a main-session turn, while this PR intentionally makes explicit channel targets select the corresponding peer session when no session override is set.
- Session routing affects transcript/history selection. Review should ensure the final patch preserves same-agent restrictions, account handling, `directPolicy`, `target: last`, `target: none`, subagent guards, and global-scope fallback behavior.
