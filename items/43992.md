## Summary

Keep open. Current main still does not implement the PR's central behavior: the gateway stores an agent run `result` in the terminal dedupe payload, but the `agent.wait` snapshot/response path drops it, and the subagent registry/announce paths still depend on transcript/history capture rather than using an `agent.wait` result fallback. The supplied PR discussion remains relevant: the author addressed the overwrite concern, but the open functional review concerns need to be handled before merge.

## Best Possible Solution

Keep this PR open or replace it with an updated patch against current main. The best fix is to thread structured `result` through `AgentWaitTerminalSnapshot`, cached/live `agent.wait` responses, and `AgentWaitResult`; extract text from normal agent result shapes; use it only as a fallback when no chat-history `frozenResultText` exists; handle the lifecycle-before-dedupe race; and add focused gateway plus subagent regression coverage.
