## Summary

Keep this issue open. Current main still has the reported split: direct CLI send can be handled by a plugin before the core hook-bearing delivery path, Telegram's plugin send action calls its own sender directly, hook cancellation still produces no explicit blocked delivery result, and gateway send still turns empty delivery results into `No delivery result`. The related implementation PR #63596 was closed unmerged, and #38340 was a closed Discord-specific attempt rather than a shipped canonical outbound send contract.

## Best Possible Solution

Keep the issue open and implement a core canonical outbound send contract. The likely direction is a shared send wrapper or delivery-layer discriminated result that preserves agent/session/request identity, returns either delivered or blocked with a denial reason, and is consumed consistently by direct CLI `message.send` and gateway `send`. Plugin-owned send actions should either route through that same wrapper or get an explicit plugin API contract that preserves equivalent `message_sending`, delivery, and audit semantics. Add regression coverage for no grant blocked, granted delivery result, and revoked grant blocked across direct and gateway entrypoints; #63596 may be useful prior art but is not shipped.
