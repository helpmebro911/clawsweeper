## Summary

Current main already supports gateway-loaded plugins spawning subagents after the request-scoped context is gone. `agent_end` hooks are still fired asynchronously, but gateway plugin runtimes are late-bound to the gateway subagent runtime and `dispatchGatewayMethod(...)` falls back to the server's gateway context when no request scope is present, which covers the reported `agent_end` case. The same code is present in released tag `v2026.4.22`.

## Close Comment

Closing as implemented on main. Codex review of current `main` found that gateway-loaded plugin runtimes already bind `api.runtime.subagent` to the gateway subagent runtime at startup (`src/gateway/server-plugin-bootstrap.ts:39`), and subagent dispatch now falls back to the server gateway context when there is no active request scope (`src/gateway/server-plugins.ts:259`, with the resolver installed in `src/gateway/server.impl.ts:767`). That is the missing piece for `agent_end`, which still runs fire-and-forget after completion (`src/agents/pi-embedded-runner/run/attempt.ts:2655`). The same implementation is present in released tag `v2026.4.22`, so there is no remaining core cleanup item here. If you still hit the old request-scope error on a newer build, please open a fresh issue with the exact plugin, hook path, and current version.
