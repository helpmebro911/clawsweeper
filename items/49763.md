## Summary

Keep open. Current main still has the reported Ctrl+O/tool-output trigger: Ctrl+O maps to onCtrlO, toggles tool expansion, then replaces the busy activity status with a non-busy "tools expanded/collapsed" status. The status renderer treats only sending/waiting/streaming/running as busy; leaving that set stops and clears the Loader, so an active run can lose its spinner until another run event changes status again. The issue comment's proposed root cause aligns with this code path, and there is no regression test covering Ctrl+O preserving the active-run busy indicator.

## Evidence

- **Issue discussion identifies Ctrl+O/tool-output toggle as a trigger:** The issue body says pressing Ctrl+O can also trigger the spinner freeze, and the sole comment attributes this to tool-output visibility causing a full rerender/status reset while an active run is still in progress.
- **Ctrl+O is wired to the tool-output toggle:** CustomEditor dispatches Ctrl+O to the TUI onCtrlO handler.
