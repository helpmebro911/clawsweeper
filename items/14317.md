## Summary

Keep #14317 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still does not implement Slack `groupPolicy: "private"`: the shared config type/schema reject that value, Slack config still uses the shared three-value schema, Slack message and slash routing still evaluate only `open`, `allowlist`, and `disabled` without private-channel metadata, and docs/tests still describe only the existing modes. The existing 2026-04-26 Codex/steipete review comment remains consistent with the current checkout.

## Best Possible Solution

Keep this issue open. Implement it in the bundled Slack plugin and aligned config/security contract: accept a Slack-scoped `groupPolicy: "private"`, carry or fetch Slack channel privacy metadata for message and slash routing, allow private channels where the bot is present, preserve explicit public-channel allowlist and deny behavior, update security warnings/audit handling, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` directly to the shared `GroupPolicy` union could accidentally impose Slack-only semantics on other channels unless scoped carefully.
- Slack private channels can still include external participants through Slack Connect/shared-channel behavior, so docs and security audit wording should avoid treating `is_private` as an internal-only trust boundary.
