## Summary

Keep #14317 open. Current `main` at e672b61417af5c45b0431df6d9109a1f4b618ef5 still does not implement Slack `groupPolicy: "private"`: the shared config type and schema reject that value, Slack config still uses the shared three-value schema, Slack message and slash routing still evaluate only `open`, `allowlist`, and `disabled`, and the fetched Slack conversation metadata is not carrying `is_private` into policy evaluation. The prior 2026-04-26 review comment keeping the issue open remains accurate against the current checkout.

## Best Possible Solution

Keep this issue open. Implement it in the bundled Slack plugin and aligned config/security contract: accept a Slack-scoped `groupPolicy: "private"`, carry or fetch Slack channel privacy metadata for message and slash routing, allow private channels where the bot is present, preserve explicit public-channel allowlist and deny behavior, update security audit/fix guidance so `private` is not treated like `open`, and add schema, runtime, docs, and tests covering private channels, public fallback allowlists, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` directly to the shared `GroupPolicy` union could leak Slack-only semantics into other channels unless it is scoped carefully.
- Slack private channels can include external participants through Slack Connect/shared-channel behavior, so the implementation and docs should not present `is_private` as an internal-only trust boundary.
