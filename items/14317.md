## Summary

Keep open. Current main still does not support Slack `groupPolicy: "private"`: the shared config contract rejects it, Slack policy evaluation only handles `open | disabled | allowlist`, message and slash routing do not pass private-channel metadata into authorization, and docs/tests still cover only the existing three policies. The existing maintainer-side Codex comment on #14317 reached the same conclusion, and the current checkout at cf834e2a does not change that surface.

## Best Possible Solution

Implement this in the bundled Slack plugin and aligned config/security contract: add a supported Slack `private` groupPolicy mode, carry or fetch Slack `is_private`/private-channel metadata during message and slash routing, allow private channels while preserving explicit public-channel allowlist entries and explicit `enabled: false` denies, update security audit/fix guidance so `private` is not treated like `open`, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` to the shared `GroupPolicy` union globally could expose the mode to channels that cannot classify room privacy; the implementation should either define generic semantics or scope the value to Slack carefully.
- Slack Connect/shared-channel behavior needs an explicit caveat because Slack private channels can still include external participants.
