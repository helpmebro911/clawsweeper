## Summary

Keep this PR open. Current main still has the stale-history behavior the PR targets: `getNoProgressStreak` skips over non-matching or missing-result records with `continue`, so an intervening tool call does not break the no-progress count before known-polling and global circuit-breaker blocking paths evaluate it. The PR remains a focused implementation candidate with matching regression coverage in the provided diff and no protected-label or maintainer-author reason to auto-close it.

## Best Possible Solution

Keep the PR open for maintainer review. The best product/code path is to land a scoped fix that makes no-progress streak counting use contiguous current-tail semantics, with regression coverage for both known polling no-progress detection and the global no-progress circuit breaker.
