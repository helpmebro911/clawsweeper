## Summary

Keep this issue open. Current main still lacks an explicit opt-in TUI observe or mirror mode: the CLI exposes only a single session selector, TUI runtime state has no mirror/observe flag, `/new` still creates an isolated `tui-<uuid>` session, live chat/BTW events are filtered to the current session, and history loading fetches one session at a time. The same-author broader #41548 remains open, but it does not clearly supersede this narrower TUI read-only monitoring workflow.

## Best Possible Solution

Keep this open as a core TUI/Gateway UX feature. Implement an explicit `openclaw tui --observe` or `tui.mirrorChannels` mode that preserves default `/new` isolation, aggregates recent same-agent activity through a deliberate Gateway/session event or history surface, labels channel and session origin, and defines read-only versus interactive behavior. Add focused tests for same-agent different-session events, channel-originated final and side-result events, aggregate history loading, and read-only behavior; coordinate with #41548 only if maintainers choose that broader issue as the canonical umbrella.
