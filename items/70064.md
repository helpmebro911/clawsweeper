## Summary

Keep open. Current main has good Telegram audio transcript plumbing, but the issue reports a downstream stale/contaminated reply after `message:preprocessed`; I did not find a direct fix, release note, or regression test proving two sequential Telegram DM audio turns cannot still produce a stale model-facing prompt or reply.

## Best Possible Solution

Keep #70064 open and add a focused regression or diagnostic capture for two sequential Telegram DM audio messages with distinct media paths/transcripts. Capture `followupRun.prompt`, `transcriptCommandBody`, and final delivery for the second turn; if the current transcript is present and not overwritten, document the conversation-history/model-behavior boundary or close with that proof. If stale content appears, fix the handoff between Telegram context, session state, queueing, and prompt assembly.
