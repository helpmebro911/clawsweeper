## Summary

Keep #14051 open. Current main still lacks an activity-based heartbeat idle timeout: the heartbeat config/schema/docs only expose interval, active-hours, delivery, timeout, light-context, isolation, and reasoning options, while the runner remains interval-driven and does not gate execution on recent allowed inbound activity. The provided discussion also keeps this as the canonical tracker: #59567 was closed as a duplicate in favor of #14051, and #40611 references this as the more complete fix rather than superseding it.

## Best Possible Solution

Keep #14051 open as the canonical heartbeat idle-timeout tracker. A proper fix should add a maintainer-approved heartbeat idle/activity config key to exported types, strict zod schema, config labels/help, docs at https://docs.openclaw.ai/gateway/heartbeat, config-honor inventory, and focused tests. Runtime should gate heartbeat scheduling or execution on recent allowed inbound activity at the correct agent/session/channel boundary, likely using lastInteractionAt or an explicit activity ledger, while preserving existing heartbeat scheduling, active-hours, wake, retry, and isolated-session semantics.

## Risks / Open Questions

- The exact product/API shape still needs maintainer judgment: idleTimeoutMinutes, idleOnly, triggerWhen, or another activity-window model could be preferable.
- The implementation needs a precise activity boundary so allowed inbound user/channel messages reset the heartbeat window while heartbeat, cron, exec, gateway bookkeeping, and other system events do not.
- The change should preserve existing interval, activeHours, requests-in-flight retry, wake-triggered events, lightContext, isolatedSession, and per-agent merge behavior.
