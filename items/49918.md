## Summary

Keep #49918 open. Current main still has the reported TUI /new gap: /new only switches to a unique local tui-* session via setSession, while the hook-emitting path remains sessions.reset. The related PR #49945 tracks the intended fix but is open/unmerged, so the regression is not implemented on main.

## Evidence

- **Current TUI /new path still skips resetSession:** The /new handler creates a unique tui-* key and calls setSession(uniqueKey), then logs the new session. It does not call client.resetSession(uniqueKey, "new"), so it does not enter the gateway reset path that emits command hooks.

That locks in the current missing /new reset call rather than proving the requested behavior.
