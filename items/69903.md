## Summary

Keep open. Current main has an opt-in bundled diagnostics-otel path for OpenTelemetry logs/traces/metrics, but it does not implement this PR's direct Sentry DSN error-tracking behavior. I also do not have high-confidence ClawHub grounds because the exact logger/crash behavior depends on internal diagnostics and crash-path seams that are not generally exposed to external plugins.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not merge it as-is. The best path is to align it with the existing diagnostics-otel architecture: either document how Sentry-compatible backends should use OTLP if that satisfies the product goal, or add a narrow, redacted, public diagnostics/log/crash export API before moving Sentry-specific forwarding to a plugin. Any core/bundled implementation should address the P1 entrypoint initialization gap and the PII/token redaction concern before merge.
