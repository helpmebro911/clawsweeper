## Summary

Current main still has gateway-first `devices approve` behavior and lacks the explicit token/password guard proposed by this PR. The PR remains a focused CLI/test implementation candidate, with no observed supply-chain-sensitive changes, but it needs rebase/adaptation and a maintainer decision on gateway log/broadcast semantics.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The maintainer path is to rebase it onto current main, preserve the current `readConnectPairingRequiredMessage` parser, keep the explicit token/password no-fallback guard if accepted, decide whether local-first approval must emit gateway-equivalent log/broadcast behavior, and keep/add regression tests for direct local approve, unknown local request fallback to gateway approve, and explicit auth flags suppressing local fallback.
