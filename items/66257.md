## Summary

Keep this PR open. Current main still does not contain the PR's narrower CLI behavior: explicit `devices approve <requestId>` calls the gateway first, local approval only runs after a qualifying gateway pairing error, and the local fallback guard still does not reject explicit `--token` or `--password`. The PR is open, draft, unmerged, and focused on the CLI/test surface rather than obsolete or superseded work.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The maintainer path is to rebase it onto current main, adapt the fallback guard to `readConnectPairingRequiredMessage`, decide explicitly whether direct local-first approval is still desired alongside gateway-side silent local pairing, preserve or revise the explicit token/password no-fallback guard, and add regression coverage for direct local approve, unknown-local-request gateway fallback, and explicit auth flag behavior.
