## Summary

Current main still matches the reported behavior: `session.parentForkMaxTokens` resolves to a static 100000 default, oversized parent sessions are skipped in `initSessionState()` with only a `session-init` log warning, and the child starts fresh. The related compaction-provider and checkpoint work is useful infrastructure, but it is not wired into the parent-fork overflow branch, and docs/tests still preserve the skip behavior.

## Best Possible Solution

Keep open. Implement the fix in the session owner path: update `src/auto-reply/reply/session.ts` and likely `src/auto-reply/reply/session-fork.ts` so an oversized parent either gets compacted/checkpointed before fork, emits a one-time visible/operator warning when inheritance is skipped, or resolves a model-aware cap from existing context-window metadata. Update the focused session tests plus session/config docs to cover the chosen behavior, reusing the shipped compaction-provider and checkpoint infrastructure where appropriate.
