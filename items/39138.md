## Summary

Keep #39138 open. Current main still permits gateway startup delivery recovery to replay old pending outbound queue entries: startup invokes recovery, recovery loads and sorts all pending `.json` entries, and the retry eligibility rule still allows `retryCount=0` entries with no `lastAttemptAt` immediately, without a stale-age cutoff or replay-count cap.

## Best Possible Solution

Implement a conservative outbound recovery policy in `src/infra/outbound`: expire or move stale pending deliveries to `failed/` before replay, preserve recent crash recovery, and add focused tests for old `retryCount=0` entries without `lastAttemptAt`, `maxRecoveryMs` interaction, startup recovery, and reconnect drains. If operator control is desired, add documented config/schema for max pending age and possibly a max replay count.
