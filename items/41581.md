## Summary

Keep this issue open. Current main still forces Telegram DM partial streaming through message-preview transport, while the lower-level draft stream still supports DM-preferring `sendMessageDraft` when `previewTransport` is left on `auto`. The user reports and related closed/unmerged fix attempts point at the same owner-owned Telegram dispatcher behavior, not a stale or already-shipped fix.

## Best Possible Solution

Keep this issue open. The best path is a targeted Telegram plugin dispatcher fix that allows safe DM answer partial previews to use `auto`/`sendMessageDraft`, preserves final materialization and duplicate-final suppression, decides whether reasoning previews should remain message-based, updates the Telegram streaming docs, and adds dispatcher regression tests for smooth DM partial preview transport without duplicate final messages.

## Risks / Open Questions

- A naive switch back to `previewTransport: "auto"` could reintroduce the duplicate-final/materialization problem noted by the dispatcher comment, so the fix needs lifecycle regression coverage.
- Reasoning and answer preview lanes share the dispatcher lane factory; the intended transport for reasoning previews should be decided explicitly.
- No live Telegram DM smoke was run in this read-only review; the source, tests, docs, and related PR context are enough to keep the issue open, not to prove a user-visible UX fix.
