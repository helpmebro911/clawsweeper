## Summary

Keep this issue open. Current main still forces Telegram DM partial preview streaming through the message/editMessageText transport, while the lower-level draft stream still supports DM-preferring auto transport. Current tests and docs also lock in the edit-message behavior, and the related fix attempts #62207 and #62210 were closed unmerged.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate Telegram dispatcher change that restores auto/sendMessageDraft transport for DM answer partial previews when safe, preserves duplicate-final suppression and reasoning-lane behavior, updates the Telegram docs to match the chosen product behavior, and adds regression coverage proving DM answer previews are smooth without reintroducing duplicate final sends.

- A naive restoration of sendMessageDraft for every DM lane could reintroduce the duplicate-final behavior that the forced message transport was guarding against.
- Reasoning-stream DM behavior may need a separate transport decision from answer-stream DM behavior because current tests cover separate answer and reasoning preview lanes.
