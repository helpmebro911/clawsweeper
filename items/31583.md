## Summary

Current main has code and tests showing skill env overrides are applied before agent/tool execution and inherited by child exec environments for declared skill env keys. But the exact reported `gog` case is still not clearly fixed: sensitive names like `GOG_KEYRING_PASSWORD` are blocked unless the skill metadata explicitly declares them, and `skills/gog/SKILL.md` still does not declare any required env keys. The original root cause appears partly resolved, but the concrete reproduction in this issue can still plausibly fail on current main, so there is not strong enough evidence to close conservatively.

## Evidence

- **Skill env overrides are applied for embedded runs before tools execute:** Embedded runs call `applySkillEnvOverrides` / `applySkillEnvOverridesFromSnapshot` before building the tool set, so exec sees the run-scoped env state.

  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '468,488p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Host exec inherits the sanitized host environment:** The exec tool builds its child env from the inherited host environment (`process.env`) plus per-call overrides. That means run-scoped skill env injected into `process.env` is available to exec children.
  - file: [src/agents/bash-tools.exec.ts:1580](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/agents/bash-tools.exec.ts#L1580)
