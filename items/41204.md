## Summary

Keep this PR open rather than auto-closing it. Current main already has stronger subagent orphan/recovery behavior and installer validation than the PR branch, and the PR diff has serious functional regressions, but the session-store lock queue cap requested in the PR is still not implemented on main and no canonical replacement issue or PR was found.

## Best Possible Solution

Keep the PR open only as a signal for the remaining queue-cap idea, but do not merge this branch. The maintainable path is to ask for a rebased, narrow PR that drops the superseded installer and subagent changes, adds a well-justified session-store queue limit in `src/config/sessions/store.ts`, and covers it with production-path tests. If maintainers do not want a queue cap, close after opening or linking a narrower tracking issue that records that decision.
