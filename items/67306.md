## Summary

Keep this issue open. Current main still has JSON output modes, but no first-class JSON Schema input contract, validation, or repair/retry semantics on the requested `openclaw infer model run` and `openclaw agent` surfaces. The existing `llm-task` plugin is useful prior art and was correctly called out in the prior review, but it is optional tool behavior and throws on invalid JSON/schema rather than defining the requested core CLI contract.

## Best Possible Solution

Keep this issue open. The best maintainer path is to implement a narrow provider-neutral JSON Schema contract on `openclaw infer model run` first, likely `--json-schema <path-or-inline>`, parse and validate final JSON with a shared validator, define bounded repair/retry behavior, return structured validation and repair-exhausted errors in `--json` mode, and add an additive gateway protocol field with matching tests/docs. After that lands, maintainers can decide whether `openclaw agent` should expose the same contract or whether workflow-only use cases should stay with plugin tools like `llm-task`.

## Risks / Open Questions

- Closing this as implemented would conflate optional `llm-task` tool validation with the requested provider-neutral CLI contract, machine-readable failure shape, and repair-exhaustion behavior.
- A real implementation needs an additive gateway protocol field plus docs/tests so local and gateway transports behave consistently.
- Provider-native schema modes can be used opportunistically, but OpenClaw still needs one stable CLI/output contract that does not vary by provider.
