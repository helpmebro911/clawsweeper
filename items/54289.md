## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction path: it finds a local Gemini CLI install, reads oauth2.js or bundle files, regex-parses client ID and secret values, caches the result, and uses it as the fallback OAuth client config when explicit env vars are absent. Existing docs and onboarding warnings disclose the unofficial OAuth risk but do not explicitly disclose or gate local Gemini CLI source/bundle credential extraction.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI source/bundle credential extraction by default, or require an explicit extraction-specific opt-in before filesystem discovery. Update extensions/google/oauth.credentials.ts, extensions/google/gemini-cli-provider.ts, Gemini OAuth tests, and the public Google/Gemini provider docs so the behavior and consent model are explicit. Review #41991 against that direction because it preserves the fallback while improving diagnostics.

## Risks / Open Questions

- Security severity is still a maintainer/product-policy call because the credentials are third-party/user-controlled and OpenClaw's security policy has a trusted-host, one-operator model.
- Removing automatic extraction outright may break existing Gemini CLI OAuth onboarding for users who rely on local Gemini CLI client-config discovery.
- Landing #41991 as-is would improve diagnostics while preserving the same fallback behavior, so it should be reviewed against the consent/hardening direction before merge.
