## Summary

Keep open. Current main still contains the reported Gemini CLI OAuth client-config extraction path, uses it as the fallback when explicit OAuth environment variables are absent, and has tests that preserve extraction from Gemini CLI install layouts. The added onboarding warning/confirmation reduces surprise for OAuth setup, but it does not remove or specifically disclose the local client ID/secret extraction behavior.

## Best Possible Solution

Keep this open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI source/bundle credential extraction by default, or require a narrowly worded explicit opt-in before any filesystem extraction, then update extensions/google/oauth.credentials.ts, the Gemini OAuth tests, and the public provider docs. The protect-mcp comment is useful broader credential-isolation input, but it does not replace deciding or fixing this bundled-plugin behavior.

## Risks / Open Questions

- Removing the fallback outright could break Gemini CLI OAuth onboarding for users who currently rely on the installed CLI's embedded client config.
- The existing confirmation gate reduces surprise during normal onboarding, but it is not specific consent for filesystem extraction of another tool's OAuth client config.
