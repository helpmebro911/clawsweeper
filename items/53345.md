## Summary

Keep this issue open. Current main and v2026.4.24 already ship the Korean Control UI locale, picker registration, docs, and tests, but the remaining default agent language-persistence behavior and automatic Korean search localization requested here are not implemented; the paired same-author PR #53346 is closed and unmerged.

## Best Possible Solution

Keep this issue open but narrow it to the remaining agent/search behavior. Preserve the shipped Korean Control UI locale, then design and test a default primary-language response policy in the agent prompt or conversation layer with explicit override and literal-text preservation rules. Add Korean search localization only through provider-aware web-search parameters or configuration, not as an assumed universal `search_lang` field.

## Risks / Open Questions

- Closing now would lose the still-unimplemented agent-side language consistency request even though the Control UI localization portion is shipped.
- A default primary-language policy needs explicit precedence for user overrides and should preserve code, logs, identifiers, paths, quotes, and raw tool output rather than blindly translating everything.
- Automatic search localization should be provider-aware because `search_lang` is documented and implemented as a Brave-specific parameter, while other providers use different or unsupported fields.
