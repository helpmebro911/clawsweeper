## Summary

Keep this issue open. Current main still implements the macOS paperclip attachment flow as image-only: the native picker allows only `UTType.image`, the shared Swift view model routes selected files through `addImageAttachment`, and non-image UTTypes are rejected before the existing generic chat attachment payload can reach the Gateway. The provided prior Codex comment is still consistent with current main, and the related items supplied do not show a merged PR, canonical duplicate, protected maintainer decision, or external-plugin boundary that would justify closing.

## Best Possible Solution

Keep this issue open. The fix should be in the shared Swift chat UI used by the macOS app: broaden the `NSOpenPanel` allowed types to the supported non-video attachment set, replace the `addImageAttachment`-only file-loading path with a generic attachment loader, keep image previews when possible with a file/document fallback for non-images, align size and type policy with `ui/src/ui/chat/attachment-support.ts` and the Gateway `chat.send` attachment contract, and add focused Swift coverage or smoke verification for PDF/TXT/DOCX plus image parity.
