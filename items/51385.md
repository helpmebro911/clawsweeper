## Summary

Keep open. Current `main` has adjacent short-term dreaming/promotion work that tracks recall frequency and consolidation signals for promotion into `MEMORY.md`, but it does not implement this issue's requested memory-search lifecycle changes well enough. The indexed chunk schema still has no retrieval/impact fields, `memory_search` still ranks via vector/BM25 merge plus optional temporal decay and MMR, and the exposed CLI/docs do not provide the requested prune/forgetting flow for indexed chunks.

## Evidence

- **Chunk schema still lacks retrieval/impact metadata:** The `chunks` table on current `main` still only stores `id, path, source, start_line, end_line, hash, model, text, embedding, updated_at`; there is no `retrieval_count`, `last_retrieved_at`, `impact`, or similar field, and the embedding cache likewise has no ranking metadata columns.

- **Search ranking pipeline is still weighted merge plus decay/MMR:** `mergeHybridResults()` still computes `vectorWeight * vectorScore + textWeight * textScore`, then applies optional temporal decay and optional MMR. There is no frequency-aware or impact-aware multiplier in the retrieval path.
