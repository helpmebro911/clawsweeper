## Summary

Keep open. Current `main` and the latest shipped release still contain the same restart-health and followup code paths, there is no evidence of a fix for the reported false-negative restart exit, and the issue body is concrete enough to investigate further rather than auto-close.

## Evidence

- **Restart still hard-fails on post-restart health failure:** `runDaemonRestart()` waits for `waitForGatewayHealthyRestart()`, then calls `fail(...)` and throws if health never becomes `healthy`. A false-negative restart would still surface as a command failure from this path on current `main`.
  - file: [src/cli/daemon-cli/lifecycle.ts:241](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/cli/daemon-cli/lifecycle.ts#L241)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Linux systemd restart path is only the service-manager handoff:** `restartSystemdService()` just runs `systemctl --user restart <unit>` and returns `{ outcome: "completed" }`. That matches the reporter's claim that systemd restart itself can succeed while later CLI logic still decides failure.
  - file: [src/daemon/systemd.ts:633](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/daemon/systemd.ts#L633)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Expired approval followup warnings are logged, not propagated:** `sendExecApprovalFollowupResult()` catches followup delivery errors and only logs `exec approval followup dispatch failed ...`; it does not rethrow. That weakens the reporter's specific suspicion about approval followups being the direct cause of the non-zero exit, but it does not resolve the restart false-negative itself.
  - file: [src/agents/bash-tools.exec-host-shared.ts:402](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/bash-tools.exec-host-shared.ts#L402)
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Existing tests cover adjacent restart states, not the reported healthy-after-failure case:** Current tests cover stale-PID retry, timeout, stopped-free early exit, and lagging runtime/probe cases, but I did not find a test for 'restart command fails even though immediate post-check `status`/`probe` are healthy afterward'.
  - file: [src/cli/daemon-cli/lifecycle.test.ts:226](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/cli/daemon-cli/lifecycle.test.ts#L226)

- **No shipped fix identified in latest release:** The relevant `lifecycle.ts`, `restart-health.ts`, `bash-tools.exec-host-shared.ts`, and restart tests in latest release `v2026.4.23` match the current logic I inspected, so this does not look implemented-on-main or fixed-in-release already.
