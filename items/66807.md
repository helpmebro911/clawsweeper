## Summary

Keep open. Current `main` still has the peer-blind account selection the PR is trying to address for plain `openclaw message send`, but the proposed implementation is not safe: it bypasses route-binding filtering and would also misroute configs where multiple agents share the same peer.

## Evidence

- **Current runner is still peer-blind:** `runMessageAction` resolves `accountId` from an explicit param/default first, then falls back only to `buildChannelAccountBindings(cfg).get(channel).get(normalizeAgentId(resolvedAgentId))`. There is no peer-id lookup in this path.
  - file: [src/infra/outbound/message-action-runner.ts:848](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/infra/outbound/message-action-runner.ts#L848)
  - command: `nl -ba src/infra/outbound/message-action-runner.ts | sed -n '820,900p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Binding map ignores peer ids:** `buildChannelAccountBindings` indexes bindings only by `resolved.channelId` and `resolved.agentId`, and stores account ids in insertion order. It never keys by `peer.id`.
  - file: [src/routing/bindings.ts:54](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/routing/bindings.ts#L54)

- **Plain CLI still forces default-agent context:** `messageCommand` still calls `runMessageAction` with `agentId: resolveDefaultAgentId(cfg)`, so `openclaw message send` remains tied to the default agent unless the caller passes an explicit account.
  - file: [src/commands/message.ts:68](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/commands/message.ts#L68)

- **Repo now has a safer peer-aware helper:** `resolveFirstBoundAccountId` already filters through `listRouteBindings(params.cfg)` and matches on agent, peer id, peer kind, and scope. That is closer to the right abstraction than the PR's raw `cfg.bindings` scan.

## Risks / Open Questions

- If merged as-is, the PR can pick ACP bindings during outbound account resolution.
- If merged as-is, the PR can misroute when multiple agents intentionally share the same peer on one channel provider.
- If closed, the remaining plain-CLI `openclaw message send` peer-routing gap would be left unresolved on `main`.
