## Summary

Keep this PR open. Current main still resolves outbound message accounts from explicit/default accountId and then an agent-keyed channel binding fallback, so it does not use the outbound target peer when choosing the account. The PR is a focused fix attempt for a plausible multi-account Slack routing bug, but its submitted implementation is not safe as-is because it scans raw bindings, can include ACP bindings, and can choose an arbitrary first binding when multiple route bindings share the same peer.

## Best Possible Solution

Keep this PR open and request a revised implementation. The best path is a typed outbound account resolver in or near src/infra/outbound/message-action-runner.ts that preserves explicit --account/defaultAccountId precedence, uses route-only bindings, derives peer id/kind from the normalized outbound target when available, safely handles the reported unique peer-bound account case without arbitrary first-match behavior, and adds regression tests for the Slack multi-account reproduction, ACP-binding exclusion, and same-peer collision cases.

## Risks / Open Questions

- Current main still preserves the reported Slack multi-account failure mode for CLI-originated sends that omit --account and rely on peer-scoped bindings owned by a non-default agent/account.
- The PR as submitted can select ACP bindings because it scans raw cfg.bindings instead of route-only bindings, which can route an outbound message through the wrong account.
- The PR as submitted matches only channel plus peer.id before the existing agent-keyed fallback, so same-peer multi-account configurations can become first-match/order-dependent and misroute messages.
- No regression test is included for the reported Slack peer-binding scenario, ACP-binding exclusion, or same-peer collision behavior.
