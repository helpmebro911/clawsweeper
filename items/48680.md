## Summary

Keep open. Current main has added several fallback classifiers for specific provider failures, but it does not yet implement the issue's central request: a generic assistant-layer error / HTTP-200 business rejection should force the outer model fallback loop to continue. Unclassified `stopReason: "error"` payloads can still return normally from `runEmbeddedPiAgent`, which lets `runWithModelFallback` mark that candidate successful.

## Evidence

- **Fallback loop still treats normal returns as success unless an explicit result classifier reports an error:** `runFallbackAttempt` only converts a successful return into a failed candidate when `classifyResult` returns a classification; otherwise it builds success. The candidate success path then logs `candidate_succeeded`.
