## Summary

Keep open. Current `main` has picked up the missing Feishu convert-scope documentation, but the normal `feishu_doc` write/append path still contains the descendant-payload bug described in the thread: `children` is normalized, then the raw block is spread back over it, which can restore a string-valued `children` field and still plausibly yield Feishu 400s on valid markdown.

## Evidence

- **String-valued children remain part of the block contract:** `FeishuDocxBlock.children` is still typed as `string[] | string`, so callers must handle both shapes.
  - file: [extensions/feishu/src/docx-types.ts:21](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/feishu/src/docx-types.ts#L21)
  - command: `nl -ba extensions/feishu/src/docx-types.ts | sed -n '1,60p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Non-batch descendant conversion still overwrites normalized children:** `toDescendantBlock()` normalizes `block.children`, adds `{ children }`, then spreads `...block` afterward. If `block.children` is a string, that later spread can overwrite the normalized array and recreate the invalid payload shape mentioned in the issue discussion.
  - file: [extensions/feishu/src/docx.ts:170](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/feishu/src/docx.ts#L170)
  - command: `nl -ba extensions/feishu/src/docx.ts | sed -n '166,178p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Write and append both use the buggy non-batch path for typical documents:** For documents at or below `BATCH_SIZE`, both `writeDoc()` and `appendDoc()` call `insertBlocksWithDescendant()`, which maps blocks through the buggy `toDescendantBlock()` helper.
  - file: [extensions/feishu/src/docx.ts:983](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/feishu/src/docx.ts#L983)
  - command: `nl -ba extensions/feishu/src/docx.ts | sed -n '969,1020p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Batch path has the intended spread order, but only there:** `docx-batch-insert.ts` builds descendant blocks as `{ ...block, ...(children ? { children } : {}) }`, preserving normalized `children`. That contrast suggests the single-request path in `docx.ts` was not fixed the same way.
  - file: [extensions/feishu/src/docx-batch-insert.ts:33](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/feishu/src/docx-batch-insert.ts#L33)
  - command: `nl -ba extensions/feishu/src/docx-batch-insert.ts | sed -n '25,45p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Permission guidance now includes the missing convert scope:** The Feishu doc skill now lists `docx:document.block:convert` as required, which aligns with the reporter follow-up that missing scope was one real cause in at least some setups. That improves docs, but it does not eliminate the code-path bug above.
  - file: [extensions/feishu/skills/feishu-doc/SKILL.md:209](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/extensions/feishu/skills/feishu-doc/SKILL.md#L209)

- **Issue discussion points to mixed causes, not a clean obsolete report:** The thread contains both a code-level diagnosis (`children` normalization lost in descendant payloads) and a reporter confirmation that adding `docx:document.block:convert` fixed their own environment. Together that means the report is not safely obsolete: current docs cover one cause, while current code still exposes the other.
