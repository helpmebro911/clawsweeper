## Summary

Current `main` still has the reported `NO_REPLY` fallback rewrite, so the underlying Telegram duplicate-delivery risk is not implemented away. This PR should stay open because it addresses a real bug, but the existing review also identified a real regression: as written, preserving raw `NO_REPLY` here would make API streaming surfaces emit the internal token unless stream suppression is handled too.
