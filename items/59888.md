## Summary

Keep open. The report is recent, got a second cross-channel confirmation on Discord, and I did not find strong evidence on current `main` that this specific stale/cached-reply bug was fixed or obsoleted.

## Evidence

- **Issue discussion points to a real shared-path bug, not a one-off WhatsApp report:** The original issue reports a second distinct inbound message getting the first reply back with a 5ms turnaround and the same `web-auto-reply` runId; a later comment reports the same stale-reply behavior on Discord DM in `2026.4.5`, which makes the problem look cross-channel rather than channel-specific.
- **WhatsApp current main still routes auto-replies through the shared buffered reply dispatcher:** `dispatchWhatsAppBufferedReply()` still calls `dispatchReplyWithBufferedBlockDispatcher(...)` and treats queued block/final replies as visible output. That is the same shared dispatch layer the report suspected.
  - file: [extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:276](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts#L276)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Discord current main still uses the same shared dispatcher/typing lifecycle primitives:** Discord message handling still builds a `createReplyDispatcherWithTyping(...)`, runs `dispatchInboundMessage(...)`, and only then marks run-complete/dispatch-idle in `finally`. That keeps the shared reply-dispatch lifecycle directly relevant to the Discord corroboration comment.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:778](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/discord/src/monitor/message-handler.process.ts#L778)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Shared dispatcher cleanup exists, but the checked-in regression coverage I found is about cleanup/idle semantics, not stale previous-reply reuse across separate inbound turns:** `dispatchInboundMessageWithBufferedDispatcher()` now guarantees `markRunComplete()` and `markDispatchIdle()` in `finally`, and `src/auto-reply/dispatch.test.ts` verifies that typing cleanup happens. I did not find a targeted regression test proving that distinct sequential inbound messages cannot reuse an earlier final reply.
  - file: [src/auto-reply/dispatch.ts:71](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/auto-reply/dispatch.ts#L71)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
