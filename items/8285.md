## Summary

Keep #8285 open. Current main still does not provide a supported configurable pre-processing text acknowledgement for ordinary inbound messages; it has reaction acknowledgements, status reactions, typing indicators, and an audio-transcript echo special case, but no `messages.ackText`-style contract. The existing plugin hooks do not fully close the gap because the usable pre-dispatch/reply-dispatch hooks are claiming/terminal paths rather than a non-terminal interim-send API that continues into normal model dispatch.

## Best Possible Solution

Keep #8285 open as a core/product or plugin-API feature request. The best path is to add a supported non-terminal pre-processing acknowledgement contract, likely `messages.ackText` with scope and per-channel/account overrides or a documented plugin/channel hook that can send interim text and then continue normal model dispatch. Update config types, zod schema, generated schema/help/labels, public docs, and channel tests for affected messaging surfaces.

## Risks / Open Questions

- Adjacent features can be mistaken for this request: ack/status reactions, typing indicators, Slack streaming fallback text, and audio transcript echo reduce latency perception in some cases but do not provide a supported configurable text acknowledgement before ordinary agent processing.
- The remaining decision is product/API design, not evidence of completion: this could be implemented as core `messages.ackText` or as a documented non-terminal plugin/channel hook that sends an interim text and then continues normal dispatch.
