## Summary

Keep this issue open. Current main still resolves the TUI Gateway URL from config/env/defaults and does not read the active Gateway service/runtime port before building the local loopback URL. The provided GitHub context also shows an open, unmerged implementation candidate, #42496, with closing syntax, so this issue should remain open until that PR is reviewed and landed or superseded.

## Best Possible Solution

Keep this issue open and review #42496, or supersede it with an equivalent focused patch. The fix should make `resolveGatewayConnection` apply the active Gateway runtime port before building the local URL, ideally by reusing the daemon lifecycle/status service-command port resolver, and add a regression test proving a Gateway running on port 48789 makes TUI resolve `ws://127.0.0.1:48789`. Update the TUI and troubleshooting docs once the behavior ships.
