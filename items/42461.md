## Summary

Keep this issue open. Current main still builds the TUI Gateway target from explicit URL, environment/config, or the default port; it does not inspect the active running Gateway service port before resolving the local loopback URL. The related in-repo fix PRs are closed and unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open and land or supersede the prior focused fix attempts with a small implementation in the TUI Gateway connection path. The fix should resolve the active running Gateway port before `buildGatewayConnectionDetails` chooses the local URL, reuse existing service-command/status port parsing where appropriate, and add a regression test proving a Gateway running on port `48789` makes `openclaw tui` resolve `ws://127.0.0.1:48789` without breaking explicit URL/auth precedence.
