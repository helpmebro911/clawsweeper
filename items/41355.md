## Summary

Keep open. Current main still passes Discord ThreadStarterBody into inbound context without the Slack-style previous-session guard, and the same-author PR #50522 is still open as an implementation candidate that explicitly attempts to close this issue. Maintainers should review or supersede that PR before closing the issue.

## Best Possible Solution

Keep this issue open as the canonical tracker. Review and land #50522 or an equivalent Discord-source fix that only supplies ThreadStarterBody for the first thread session turn, add a focused Discord regression test for existing thread sessions, and consider whether the generic session fallback should ignore ThreadStarterBody on non-new sessions unless a channel intentionally supplies safe history context.
