## Summary

Keep this issue open. Current main still supports MCP credentials only on the shared `mcp.servers` definitions and the internal OpenClaw loopback MCP bridge; it does not provide a schema, type contract, runtime merge layer, or cache key for per-agent env/header overlays for an external MCP server such as Plane. Related #65612 and #4834 do not supersede this request: they cover MCP tool policy/native MCP support, not same-server per-agent credential attribution.

## Best Possible Solution

Keep this open as a core MCP runtime/config feature. Add a schema-backed per-agent MCP identity or credential overlay, merge it before embedded Pi and CLI backend MCP config creation, include the effective overlay in runtime/cache fingerprints, redact secrets consistently, and add tests proving two agents can use the same MCP server id with different effective env/header credentials.

## Risks / Open Questions

- A real implementation needs careful cache fingerprinting and runtime lifecycle handling so credentials for one agent cannot bleed into another agent's MCP process or HTTP client.
- The duplicate-server operational workaround can preserve attribution, but it changes server/tool IDs and forces per-agent tool-policy maintenance rather than solving same-server per-agent credentials.
