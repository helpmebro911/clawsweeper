## Summary

Current main still does not implement the requested per-agent MCP credential/env overlay. MCP credentials remain scoped to top-level `mcp.servers` entries, the agent schema has no `mcp` override surface, and embedded Pi runtime creation/stdio launch do not receive an active-agent env overlay. The newer OpenClaw loopback MCP agent-id env/header path and the related #65612 tool-policy fix are useful, but they do not solve Plane-style per-agent API key attribution for the same external MCP server id.

## Best Possible Solution

Keep this open as a focused core MCP runtime/config feature. The likely implementation is a schema-backed per-agent MCP identity or credential overlay applied before MCP runtime creation, included in the runtime cache fingerprint, and covered by tests proving two agents can use the same MCP server id with different effective env credentials. Until then, document the duplicate `mcp.servers` plus per-agent tool-policy workaround as a workaround, not a replacement.

## Risks / Open Questions

- A duplicate-server workaround can preserve attribution operationally, but it is not transparent: it changes MCP server ids/tool names and requires per-agent tool policy maintenance.
- A real implementation needs cache and lifecycle design so effective MCP env affects runtime fingerprints, hot reload invalidation, session reuse, and secret redaction.
