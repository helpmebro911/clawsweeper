## Summary

Keep #38945 open. Current main still has the Unicode tokenizer gaps the PR targets: MMR tokenization is ASCII plus CJK-only, FTS query construction omits combining marks and NFC normalization, and the shared keyword-search planner has the same regex/normalization gap. Related CJK work does not cover accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining marks, or NFC/NFD equivalence.

## Best Possible Solution

Keep the PR open and port the narrow tokenizer fix to current `extensions/memory-core`: update MMR tokenization, `buildFtsQuery()`, and the shared FTS search token planner to use NFC-normalized Unicode letter/mark/number tokens while preserving CJK bigram/unigram and trigram fallback behavior. Add focused regression tests for ASCII, CJK, Finnish/accented Latin, Cyrillic, Arabic/Hebrew, Thai/Hindi combining marks, and NFC/NFD equivalence.
