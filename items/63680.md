## Summary

Current main still lacks the requester/resolver self-approval guard that this PR proposes. The PR is not obsolete: `exec.approval.request` records `requestedByConnId`, but `exec.approval.resolve`, the shared approval resolver, and `ExecApprovalManager.resolve` do not reject non-deny resolutions from the same connection. The PR also contains no unrelated supply-chain or workflow changes; its security-sensitive diff is focused on gateway approval logic and tests.

## Best Possible Solution

Keep this PR open and review it as an active implementation candidate. The best path is to land a focused gateway fix that rejects non-`deny` `exec.approval.resolve` decisions when the resolver connection matches `snapshot.requestedByConnId`, preserves self-deny and different-connection approvals, updates the current bypass test to use a separate resolver connection, keeps focused regression coverage, and runs the changed gateway checks before merge.
