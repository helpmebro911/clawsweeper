## Summary

Keep this PR open. Current main still has the zero-output dispatch path this PR targets: an undefined resolver result becomes an empty replies array, dispatch records completion, and the function returns with no queued final reply. The PR is a plausible focused fix, but it should be updated/reviewed against sendPolicy suppression and the canonical chat-type policy before merge.

## Best Possible Solution

Keep this PR open and review/update it against current main. The best fix is a narrow dispatch-level fallback after final queued counts are known, only when user delivery is allowed, the turn is direct-style under canonical chat-type policy, there is no routed-origin delivery, no queued tool/block/final output, no block-stream/TTS output, and no intentional handled/no-text or plugin/suppressed-delivery branch. Add focused regression tests for the direct empty-output case and for contexts that must stay silent.

## Risks / Open Questions

- As proposed, the fallback needs an explicit no-suppressed-delivery guard so `sendPolicy: deny`, plugin-owned suppressed paths, and other intentional no-delivery flows cannot leak a final reply.
- The PR should use or extend canonical chat-type normalization instead of a local `direct|dm|private` helper, otherwise direct/private semantics may drift from the rest of auto-reply policy.
- Regression coverage should include direct empty-output fallback, group/channel no-op behavior, sendPolicy-deny no-op behavior, routed-origin turns, block-stream/TTS output, and handled-without-text hooks.
- Security review found no CI workflow, dependency source, lockfile, lifecycle script, secret handling, package publishing, or downloaded-artifact changes in the provided diff; the remaining risk is functional delivery behavior.
