## Summary

Keep this PR open. Current main still has the zero-output dispatch completion path the PR targets, and the PR is a plausible focused fix. The author addressed the Greptile wording concern and narrowed the fallback toward direct/private contexts, but the current PR patch still needs maintainer review around sendPolicy suppression and the canonical chat-type contract before it should merge or be closed.

## Best Possible Solution

Keep this PR open and review/update it against current main. The best fix is a narrow dispatch-level fallback after final queued counts are known, guarded by delivery policy, direct-style conversation policy, no routed-origin delivery, no queued final/tool/block output, no block-stream/TTS output, and no intentional handled/no-text or plugin-handled branch. Use `normalizeChatType` or deliberately extend the canonical chat-type contract, then add focused regression tests for the allowed direct empty-output path and the suppressed/no-op cases.

## Risks / Open Questions

- Security review: the provided diff touches only `src/auto-reply/reply/dispatch-from-config.ts` and its test file, with no workflow, dependency, lockfile, release, or package-publishing changes. The main security-sensitive risk is policy bypass: a fallback sent outside `suppressDelivery` could leak an outbound message when a session explicitly denies delivery.
- The local `direct|dm|private` helper in the PR may drift from the canonical `normalizeChatType` contract unless `private` is intentionally added as a supported alias with tests.
- The PR's regression coverage should be broadened before merge to include sendPolicy deny, routed-origin delivery, block-stream/TTS output, group/channel silence, and handled-without-text no-op cases.
