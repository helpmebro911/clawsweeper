## Summary

Current main does not implement the requested discovery-time pruning. The loader still discovers the full bundled plugin root and builds a manifest registry for all discovered candidates before applying plugin enablement/allowlist decisions. Related startup work reduced other costs, but it does not supersede this narrower request.

## Best Possible Solution

Keep this issue open and implement a loader/discovery optimization rather than close it as already fixed. The likely direction is a cheap bundled-plugin candidate prefilter or generated metadata index that can narrow manifest reads for restrictive `plugins.allow`/activation scopes while preserving default-enabled bundled plugins, explicit channel/provider activation, slots, aliases, and duplicate-precedence rules. Add regression coverage that an allowlisted boot path does not read/parse manifests for unrelated bundled plugin directories.

## Risks / Open Questions

- A naive directory-name filter could break bundled defaults, configured channels, provider/model auto-enable, memory/context slots, or legacy aliases; the fix needs to preserve the existing activation contract.
- The report says `tools.allow`, while the current loading gate is `plugins.allow`; maintainers should clarify whether any new optimization is based on plugin inventory policy, tool policy, or a generated bundled-plugin metadata index.
