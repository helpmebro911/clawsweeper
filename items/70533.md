## Summary

Keep #70533 open. Current main has useful plugin-registry work that reduces some cold metadata reads and #69250-style duplicate registration, but the gateway runtime loader still discovers all plugin candidates and builds a full manifest registry before applying `onlyPluginIds` at import time. The report's `tools.allow` wording is likely imprecise because current plugin loading is gated by `plugins.allow`; however, the central requested behavior, discovery-time narrowing before manifest reads/validation for a restrictive plugin set, is not fully implemented.

## Best Possible Solution

Keep the issue open and implement a narrow startup/discovery plan that computes the required plugin ids from `plugins.allow`, explicit configured-channel exceptions, slots, provider/harness triggers, and setup needs, then applies that scope before manifest parsing/validation. Existing persisted plugin-registry work should be reused where possible, but the runtime loader should also avoid full `dist/extensions/*` manifest materialization when it is handed a scoped startup plugin list. Add regression coverage that proves skipped plugin manifests are not read, not only that skipped runtime modules are not imported.
