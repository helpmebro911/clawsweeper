## Summary

Keep open. Current main has partial adjacent improvements for gateway status, LaunchAgent supervision, service entrypoint drift, and Telegram 409 conflict hints, but `openclaw doctor` still runs gateway RPC health and daemon/runtime checks as separate phases and can still emit the confusing mixed path the report describes instead of one consolidated healthy/degraded/down verdict with ordered next steps or machine-readable diagnosis.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. Reuse the stronger `gateway status` gather/probe model where possible, then collect service installed/loaded state, LaunchAgent supervision, runtime PID/status, port listener ownership, RPC/read-probe result, supervisor config drift, extra gateway-like services, recent channel-specific signals such as Telegram getUpdates 409 conflicts, and last gateway errors. Print one final `healthy` / `degraded` / `down` verdict with deterministic next steps and an optional compact JSON diagnosis block. Add regression tests for mixed RPC failure plus running LaunchAgent runtime, entrypoint drift, missing LaunchAgent supervision, port ownership mismatch, and Telegram duplicate-poller hints.
