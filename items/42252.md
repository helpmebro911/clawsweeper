## Summary

Keep open. Current main has partial improvements for adjacent cases, including LaunchAgent split-brain detection, service entrypoint drift reporting, Telegram 409 duplicate-poller log hints, and stronger `gateway status` semantics. The central doctor UX request remains unresolved: doctor still emits gateway RPC health and daemon/runtime state in separate sequential passes, so a failed RPC can still print `Gateway not running.` before the later daemon pass reports `Runtime: running (pid ...)`, without a final consolidated healthy/degraded/down verdict or doctor JSON diagnosis.

## Best Possible Solution

Keep this issue open and implement a doctor-owned consolidated gateway diagnosis. The implementation should collect service loaded state, runtime PID/status, port listener ownership, RPC/read-probe result, service config drift, extra gateway-like services, and recent channel-specific signals such as Telegram `getUpdates` 409 conflicts into one final `healthy` / `degraded` / `down` verdict with deterministic next steps. Add an optional machine-readable doctor diagnosis block and tests for mixed LaunchAgent/runtime/RPC states.
