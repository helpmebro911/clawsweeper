## Summary

Keep PR #71329 open. Current main still gates Slack lifecycle status reactions through the ack-reaction decision, so `messages.statusReactions.enabled: true` does not independently activate status reactions in cases where ack reactions are disabled by scope or configuration. The PR is a focused Slack behavior fix, with a minor review/test assertion gap still to address rather than a reason to close.

## Best Possible Solution

Keep the PR open and have the Slack owner/maintainer review the small behavior change. The best path is to land a scoped prepare-layer fix that treats `messages.statusReactions.enabled === true` as an explicit lifecycle-status opt-in independent of ack-reaction scope, while preserving the existing implicit ack-gated behavior for unset status reactions and adding regression tests for explicit enable without ack plus explicit disable with plain ack fallback.
