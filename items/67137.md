## Summary

Keep this PR open for maintainer review. Current main still has the exact `WizardPrompter.text` boundary and downstream `.trim()` assumptions that the PR targets, and the broader canonical cleanup role for this PR is reinforced by related duplicate triage. The original linked user crash appears likely fixed through the docsPath/channel-selection guard, and locked Clack currently finalizes undefined text input to an empty string, but that is not strong enough to auto-close a focused hardening PR with regression coverage.

## Best Possible Solution

Keep this PR open. The best path is maintainer review of a small boundary fix that makes `createClackPrompter().text` always satisfy `Promise<string>`, plus focused regression tests for undefined text results; downstream guards should be kept only where they preserve existing required-field semantics. If maintainers decide the locked `@clack/prompts` behavior is sufficient, that should be an explicit product/code review decision rather than an automated close.
