## Summary

Keep #17931 open. Current main still mirrors eligible skills into writable sandbox workspaces for non-rw modes and does not add automatic read-only per-skill Docker bind overlays. The related #22669 custom-bind ordering fix provides a manual workaround path, but it does not implement the requested default sandbox hardening.

## Best Possible Solution

Keep #17931 open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that sandbox-hardening direction. A robust implementation should derive eligible skill source directories, add deterministic read-only Docker overlays after broader workspace mounts for rw mode, avoid writable copied skills where the backend can mount sources directly, update sandbox config hashing and fs-bridge mount resolution as needed, and cover the behavior in tests and sandboxing docs. Treat #22669 as a useful manual workaround, not the canonical fix.

## Risks / Open Questions

- Sandboxed agents can still mutate mirrored skill copies in workspaceAccess "ro" and "none" because those copies live under the sandbox workspace.
- In workspaceAccess "rw", workspace-local skills remain writable through the normal /workspace mount unless an operator manually configures overlapping read-only custom binds.
- Closing as implemented would conflate the shipped #22669 custom-bind precedence fix with the still-missing automatic read-only per-skill mount behavior.
