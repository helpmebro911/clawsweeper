## Summary

Keep this PR open. Current main still does not implement `--daemon-env` for `openclaw onboard --install-daemon`, and this PR remains a focused implementation candidate. It should not be cleanup-closed, but it needs maintainer/security review because the proposed merge point can bypass the existing daemon host-env filtering boundary.

## Best Possible Solution

Keep this PR open for maintainer/security review. The best path is to either implement a narrow reviewed opt-in for the known `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` bootstrap case, or add a shared daemon-env parser/normalizer that validates portable keys, rejects dangerous host-env names using `host-env-security`, warns or fails on malformed input, and has regression coverage for the CLI plus both onboarding install paths. If accepted, document the supported flag and security limits in `docs/cli/onboard.md` and `docs/reference/wizard.md`.
