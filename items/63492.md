## Summary

Keep open. Current main still implements session persistence as local filesystem metadata plus local JSONL transcript files. The config/schema surface exposes a path-based `session.store`, the gateway and CLI session paths resolve and scan `sessions.json` files, and plugin runtime helpers can read/write the current file-backed store but do not provide a backend registration contract. The related PostgreSQL request #57497 is narrower and still open, so it is neither a shipped implementation nor a canonical duplicate for this broader stateless unified storage design.

## Best Possible Solution

Keep this issue open for a core session persistence design. The likely path is a versioned session persistence contract that covers both `sessions.json` metadata and JSONL transcript/history semantics, keeps the filesystem backend as the default, adds config/schema/docs and migration/doctor behavior, proves at least one multi-instance-safe backend, and only then exposes a plugin-facing backend registration API for additional stores.

## Risks / Open Questions

- A plugin-only implementation is not currently enough because core has no stable backend registration contract and transcript/history operations still assume local files.
- The design needs maintainer decisions for metadata plus transcript semantics, distributed locking/concurrency, TTL and pruning behavior, migration, docs/schema compatibility, and secure credential handling.
