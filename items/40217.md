## Summary

Keep this PR open. Current main still lacks stale-history pruning for tool-loop detection, the production path still detects before recording the next call, and the linked bug #40144 plus overlapping PR #40798 remain open. The PR is a plausible focused fix, but it still needs maintainer review and consolidation rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate it with #40144 and #40798. The best implementation should clear or scope loop history before detection at the real heartbeat/turn boundary, use activity/outcome-aware cutoff semantics that preserve slow-tool loop protection, reset or scope warning dedupe consistently, and add detect-then-record regression coverage plus aligned config/schema/docs updates.

## Risks / Open Questions

- Merging the current PR shape may weaken loop detection for slow or backed-off repeated tools because the proposed cutoff uses the call-start timestamp and recordToolCallOutcome does not refresh that timestamp.
- Clearing only toolCallHistory would need coordinated handling for toolLoopWarningBuckets, otherwise warning dedupe can suppress later warning-level loops after a stale-history reset.
- There are multiple open fix attempts for #40144, so maintainers should choose a canonical implementation path instead of letting overlapping PRs diverge.
