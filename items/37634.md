## Summary

Keep #37634 open. Current main still gives `workspaceAccess: "none"` an isolated sandbox workspace, but treats that workspace as read-only/write-blocked in container mounts, host and remote filesystem bridges, tests, and docs. Related fix PRs #37276 and #38154 were closed unmerged, and #46300 is still only an open implementation path, not shipped behavior.

## Best Possible Solution

Keep #37634 open as the focused tracker unless maintainers explicitly choose the documented read-only semantics. A complete fix should consistently make the isolated `none` sandbox workspace writable across Docker/browser mounts, host and remote filesystem bridges, mutation gates, memory flush behavior, tests, and docs; #46300 can remain the current implementation candidate until merged or superseded.
