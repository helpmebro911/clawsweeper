## Summary

Current main still does not implement manifest-level install-time `configPatch`. The manifest type and loader do not carry such a field, install result shapes cannot propagate it, install persistence only enables/records plugins, and docs describe setup/package metadata without a config-patch install contract. The existing provider-auth `configPatch` path is real but narrower and does not solve the service/tool plugin install case raised by #6792 and the comments.

## Best Possible Solution

Keep #6792 open for maintainer design and core implementation. A complete solution should add a typed manifest-level config intent or equivalent install-time plugin config API, sanitize and validate patches, propagate them through linked, copied/archive, npm, marketplace, ClawHub, and bundled install flows, define array and conflict semantics, show user-visible diff/skip output, add uninstall guidance for applied patches, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/plugins/building-plugins.

## Risks / Open Questions

- Install-time config mutation is trust-sensitive and needs validation, prototype-pollution protection, conflict handling, and a clear diff/skip UX before it is safe to ship.
- Array semantics remain unresolved: the issue proposes replacement, while the `tools.alsoAllow` use case in comments likely needs additive union behavior to avoid clobbering other plugins.
- Related issue #31251 shows plugin install/uninstall config writes have previously risked unrelated config drift, so any implementation needs tests proving includes and unrelated keys are preserved.
