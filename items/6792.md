## Summary

Current main still lacks manifest-level install-time `configPatch`. The existing `configPatch` contract is provider-auth scoped, while plugin manifest parsing, install result payloads, CLI install branches, persistence, docs, and tests still only cover normal plugin enablement/allowlist/install-record state. The issue discussion remains relevant: the claude-search-proxy use case and `tools.alsoAllow` example describe real remaining friction, and PR #43421 was closed unmerged rather than shipping the feature.

## Best Possible Solution

Keep this issue open for maintainer product/API decision. A complete fix likely belongs in core plugin install and manifest contracts: add a documented `configPatch` field to `PluginManifest`, parse and validate it, propagate it through all install result types and install sources, apply a sanitized deep merge using the existing provider-auth merge helper or a documented variant, validate before writing config, handle array semantics such as `tools.alsoAllow` deliberately, show the applied diff or skip reason, and add regression coverage for local path, linked path, npm, ClawHub/marketplace, bundled fallback, and invalid-patch cases.
