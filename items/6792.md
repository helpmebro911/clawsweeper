## Summary

Current main still does not implement manifest-level install-time `configPatch`. The manifest type/loader omit it, install results cannot carry it, plugin install persistence only enables/records plugins, and public docs do not document such a contract. The existing provider-auth `configPatch` path is narrower and does not solve service/tool plugin install configuration. PR #43421 attempted this but was closed unmerged, and the latest maintainer-side issue comment explicitly kept #6792 open.

## Best Possible Solution

Keep #6792 open for maintainer design and implementation in core. A complete solution should add a typed manifest-level config intent or equivalent install-time plugin config API, sanitize and validate it, propagate it through linked, copied/archive, npm, marketplace, ClawHub, and bundled install flows, define array/conflict semantics, show a diff or skip output, add uninstall messaging for applied patches, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/tools/plugin.

## Risks / Open Questions

- Install-time config mutation is trust-sensitive and needs validation, prototype-pollution protection, conflict behavior, and clear user-visible diff/skip UX before shipping.
- Array merge semantics remain unresolved: the issue proposes replacement, while the `tools.alsoAllow` use case in comments likely needs additive union behavior to avoid clobbering other plugins.
- The related clobbering report #31251 shows plugin install/uninstall config writes can be fragile, so any implementation should prove unrelated config keys and includes are preserved.
