## Summary

Keep open. Current main still does not implement the requested `brave.http` diagnostics flag: Brave builds and sends both Brave Search API requests and uses the shared cache, but there is no Brave diagnostics flag check, subsystem logger, request/status/timing log, or cache hit/miss log. The generic diagnostics flag API and Telegram `telegram.http` pattern already exist, and the web-search provider context includes full config, so this is an actionable bundled Brave plugin observability improvement rather than a cleanup close.

## Best Possible Solution

Implement this in `extensions/brave`: pass the full config or precomputed diagnostics state from `createTool(ctx)` into the Brave runtime, gate logs on `brave.http` using `openclaw/plugin-sdk/diagnostic-runtime`, log sanitized request URL/query/params, HTTP status and duration, and cache hit/miss through a Brave diagnostics subsystem logger without exposing `X-Subscription-Token`, then add focused Brave provider tests and docs updates in `docs/diagnostics/flags.md` and `docs/tools/brave-search.md`.
