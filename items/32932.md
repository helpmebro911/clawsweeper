## Summary

Keep #32932 open. Current main still lacks a scoped private-network opt-in for scheduled-task webhook egress: finished webhooks, failure-alert webhooks, and failure-destination webhooks all use the SSRF-guarded fetch helper without passing any policy, while config, protocol, UI, CLI, docs, and tests still expose webhook URL/token settings but no private-network control.

## Best Possible Solution

Keep the issue open until maintainers either implement a scoped cron webhook private-network opt-in or explicitly decide that scheduled-task webhook destinations must always block intranet targets. A likely implementation adds a narrow policy field in cron config and/or per-job delivery/failure-destination contracts, passes that policy into postCronWebhook for finished, failure-alert, and failure-destination webhook sends, preserves default SSRF blocking, and updates strict config schema, protocol schemas, generated help/docs, Control UI/CLI surfaces, and regression tests for default-blocked and opted-in private-network behavior.
