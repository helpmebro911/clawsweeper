## Summary

Keep open. Current `main` still has the reported macOS-node inconsistency: `exec host=node` always calls `system.run.prepare`, but the macOS companion app advertises and implements `system.run`/`system.which` without any `system.run.prepare` command.

## Evidence

- **Exec path requires prepare:** `exec host=node` only checks for `system.run` on the selected node, then unconditionally invokes `system.run.prepare` before forwarding `system.run`. A node that advertises `system.run` but not `system.run.prepare` will fail exactly on this path.
  - file: [src/agents/bash-tools.exec-host-node.ts:91](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/agents/bash-tools.exec-host-node.ts#L91)
  - command: `nl -ba src/agents/bash-tools.exec-host-node.ts | sed -n '88,122p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Core command contract expects prepare:** The shared node system command set includes `system.run.prepare`, `system.run`, and `system.which`, and the gateway macOS/linux/windows default system command allowlist is built from that set.
  - file: [src/infra/node-commands.ts:1](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/infra/node-commands.ts#L1)
  - command: `nl -ba src/infra/node-commands.ts | sed -n '1,12p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **macOS app advertises no prepare:** The macOS node coordinator advertises `system.notify`, `system.which`, `system.run`, and exec-approvals commands, but not `system.run.prepare`.
  - file: [apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift:139](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift#L139)
  - command: `nl -ba apps/macos/Sources/OpenClaw/NodeMode/MacNodeModeCoordinator.swift | sed -n '136,162p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **macOS app implements no prepare handler:** The macOS node runtime handles `system.run`, `system.which`, `system.notify`, and exec-approvals commands. There is no `system.run.prepare` case in the invoke switch.
  - file: [apps/macos/Sources/OpenClaw/NodeMode/MacNodeRuntime.swift:47](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/apps/macos/Sources/OpenClaw/NodeMode/MacNodeRuntime.swift#L47)
  - command: `nl -ba apps/macos/Sources/OpenClaw/NodeMode/MacNodeRuntime.swift | sed -n '34,84p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Gateway rejection matches report:** When a node does not declare a requested command, the gateway returns `node command not allowed: the node (platform: ...) does not support "<command>"`, which matches the issue's observed failure for `system.run.prepare`.
  - file: [src/gateway/server-methods/nodes.ts:1156](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/gateway/server-methods/nodes.ts#L1156)
  - command: `nl -ba src/gateway/server-methods/nodes.ts | sed -n '1154,1168p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Docs still describe the inconsistent surface:** Current docs say the macOS node exposes `system.run`, `system.notify`, and `system.execApprovals.get/set`, while also documenting that shell execution uses the exec path and that `system.run.prepare` stays on that path only. That matches the code inconsistency rather than a shipped fix.
  - file: [docs/nodes/index.md:321](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/docs/nodes/index.md#L321)
  - command: `nl -ba docs/nodes/index.md | sed -n '321,336p'`
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Issue discussion strengthens reproduction:** The thread already includes a failed restart/workaround attempt from the reporter and a second reporter confirming the same behavior on macOS companion app 2026.4.15. I found no code on current `main` showing a later macOS-side fix for `system.run.prepare`.

## Risks / Open Questions

- Remote macOS execution and Peekaboo/UI automation remain blocked for companion-app nodes that advertise `system.run` without `system.run.prepare`.
- The CLI/docs currently mislead users into thinking macOS node exec should work when the app-side command surface still lacks the required prepare step.
