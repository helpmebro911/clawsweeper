## Summary

Current main still has adjacent runtime controls, but not the requested capability-based permission model. Tool policies are name-based, plugin/skill manifests do not declare fs/net/secrets/messages capabilities, grants/revocations are not persisted as a first-class permission layer, and audit logging is not tied to high-risk capability invocations.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. Design a manifest-declared capability schema, persist grant/revoke state with CLI/UI management, default-deny high-risk capabilities before tools are listed or invoked across agent/HTTP/MCP surfaces, and write durable audit records for high-risk capability use. Treat @vainplex/openclaw-governance as valuable before_tool_call governance prior art, not a replacement for API-layer authorization.
