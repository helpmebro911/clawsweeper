## Summary

Current main still does not implement #12678's requested manifest-declared, default-deny capability permission model for skills/tools. Main has narrower controls: tool-name allow/deny profiles, per-sender/group policies, owner-only filtering, Gateway HTTP default-deny for selected dangerous tool names, exec/plugin approval prompts, and before_tool_call hooks. Those do not provide pre-exposure capability grants/revocations or immutable high-risk invocation audit semantics.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. The best path is to add a manifest-declared capability schema, enforce default-deny grants before risky tools are exposed or invoked, provide grant/revoke UI and CLI flows, and record high-risk capability invocations in a durable audit log. The linked @vainplex/openclaw-governance plugin should be treated as useful prior art for call-time governance, not as a replacement for the missing API-layer authorization model.
