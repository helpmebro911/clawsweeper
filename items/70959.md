## Summary

Keep open. The issue is recent, has concrete reproduction/log/session evidence, and current main does not fully close the reported product problem: webchat image submissions still rely on per-run idempotency, same-content duplicate sends with new run IDs are not rejected, and typical pasted images below the offload threshold can still be forwarded inline rather than resized/offloaded before provider submission.

## Best Possible Solution

Keep the issue open and fix it in the webchat/gateway/runner boundary: add a client in-flight guard for the exact submitted draft+attachments, reject or coalesce short-window duplicate `chat.send` payloads server-side even when run IDs differ, resize/offload webchat inbound images before model submission, and ensure overflow-compaction retries do not append the same user image turn repeatedly.

## Evidence

- **Issue evidence:** The issue body reports two same-day incidents on OpenClaw 2026.4.22 where one pasted PNG was persisted 4 and 9 times, producing 413 request-body failures and repeated compaction attempts. There are no maintainer comments or protected labels that would make this auto-closeable.
- **UI idempotency is per generated run only:** The browser `chat.send` request uses `idempotencyKey: params.runId`; `sendChatMessage` generates a fresh UUID for each send call, so this only dedupes repeats that reuse the same run ID, not accidental re-entry of the same draft/attachment as a new send.
