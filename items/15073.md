## Summary

Keep #15073 open. Current main still supports per-agent model fallback lists only; fallback attempts carry provider/model candidates through the active run and reuse the same agentId, workspaceDir, agentDir, sessionFile, and skills snapshot. The requested fallback-agent/workspace/context-profile behavior is absent from config types, strict schemas, runtime candidate metadata, docs, and tests. The issue discussion also already includes a 2026-04-26 maintainer/Codex comment explicitly keeping it open.

## Best Possible Solution

Keep #15073 open until maintainers choose and implement a first-class design: either fallbackAgent/workspace semantics or a narrower model-specific context-budget/profile mechanism. A complete implementation likely needs core config types and strict schemas, docs/schema help, fallback candidate metadata carrying alternate context, reply-runner/session plumbing that safely swaps workspace/agentDir/skills/context for fallback attempts, and regression tests proving fallback retries use the intended alternate context without corrupting the original session.

## Risks / Open Questions

- Current per-agent fallback support can be mistaken for this request: it selects fallback model strings, but does not switch agent identity, workspace, agentDir, session ownership, or skills snapshot.
- Bootstrap hooks can partially reshape prompt files, but without active fallback provider/model metadata they are only a workaround and do not satisfy the requested first-class behavior.
