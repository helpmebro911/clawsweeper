## Summary

Keep open. Current main still classifies tool-result failures only from `details.status`, while bundled MCP materialization only writes `details.status` for `isError === true`, so status-less MCP content errors are not implemented on main. The PR is small and agent-scoped, with no supply-chain-sensitive file changes, but the final behavior still needs maintainer judgment because the latest PR commit trusts explicit `isError: false` and only applies the `Error:` fallback when `isError` is absent.

## Best Possible Solution

Keep this PR open for maintainer review. Decide whether OpenClaw should strictly trust MCP `isError` or provide a compatibility fallback for non-conformant servers; then either land a revised agent-scoped implementation with tests for `isError: true`, `isError: false`, omitted `isError`, `Error:` false positives, and `after_tool_call.error`, or close the PR after explicitly rejecting content-prefix error inference.
