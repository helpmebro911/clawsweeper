## Summary

Keep PR #69910 open. Current main still passes native Ollama `response.message.tool_calls[].function.arguments` through unchanged in `buildAssistantMessage`, so JSON-string arguments can still reach downstream tool execution. Existing normalization covers replay/outbound paths, not this inbound assistant-message path, and the current tests do not cover stringified Ollama tool-call arguments.

## Best Possible Solution

Keep the PR or an equivalent narrow fix open: update `buildAssistantMessage` in `extensions/ollama/src/stream.ts` so native Ollama response tool-call arguments are normalized with the existing `ensureArgsObject` helper before constructing the assistant `toolCall`, widen the local response type to allow unknown/string arguments, and keep regression tests for JSON-string arguments, unsafe integer preservation, malformed-string fallback, and existing object pass-through.
