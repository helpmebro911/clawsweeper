## Summary

Keep open. Current main partially normalizes Ollama tool-call arguments on replay/outbound paths, but the inbound `buildAssistantMessage` path still copies provider-returned `tool_calls[].function.arguments` through unchanged. PR #69910 addresses that remaining path and includes useful follow-up for the bot review comments; the related PRs in the provided context are closed unmerged, not a shipped superseding fix.

## Best Possible Solution

Keep #69910 open and land a rebased minimal variant that normalizes `toolCall.function.arguments` inside `extensions/ollama/src/stream.ts` `buildAssistantMessage`, reusing `ensureArgsObject()` so string JSON, malformed strings, non-object values, and unsafe integer preservation behave consistently with the replay path. Keep the regression coverage for inbound Ollama string `tool_calls`.
