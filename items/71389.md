## Summary

Keep #71389 open. Current main still loads sharp through the bundled media-understanding-core image ops path and the requested graceful fallback/disable behavior is not implemented for resizeToJpeg, convertHeicToJpeg, or resizeToPng. The related closed non-AVX2 Docker report does not supersede this sharp-specific runtime crash report.

## Best Possible Solution

Implement this in the core/bundled media image path, not ClawHub: add a documented image backend opt-out or CPU/libvips capability gate before sharp is loaded, treat sharp import/operation failures as image optimization unavailable, make resize/HEIC/PNG paths pass through or return clear recoverable errors where pass-through is unsafe, and add targeted tests by injecting/mocking the image ops loader to throw. If SIGILL can occur during native module load, prefer a pre-import CPU check or isolated probe over relying only on try/catch.
