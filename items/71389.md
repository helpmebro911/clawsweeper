## Summary

Keep open. Current main still has Linux/Node image-processing paths that load Sharp in-process through the media-understanding-core public artifact, and the resize/HEIC/PNG entry points still do not provide the requested CPU-feature guard, disabled backend mode, or pass-through fallback. The related #47419 report addressed a different non-AVX2 Docker publish regression, not this Sharp runtime native-load crash path.

## Best Possible Solution

Keep the issue open. The best product path is to make image optimization failure non-fatal on Linux/Node: add a documented image-backend disable/pass-through mode, avoid in-process native Sharp loads on unsupported CPUs where feasible, and add focused regression coverage around Sharp load failure plus the resize/HEIC/PNG call sites. If native SIGILL cannot be caught safely in-process, isolate the probe/work in a child process or require a preflight CPU/library compatibility check before the gateway loads Sharp.
