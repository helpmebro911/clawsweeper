## Summary

Keep #19075 open. Current main still does not implement a configurable Brave Search `baseUrl`: the bundled Brave runtime constructs both Brave API requests from fixed endpoint constants, the Brave plugin schema/UI config exposes only `apiKey` and `mode`, top-level search config remains strict without a Brave endpoint override, and docs do not document any Brave proxy/base URL setting. The related PR attempts in the provided context are closed unmerged, while duplicate issues #45729 and #51658 were closed back to #19075, so this remains the canonical tracker.

## Best Possible Solution

Keep #19075 open as the canonical bundled Brave Search `baseUrl` tracker. Implement it in the Brave plugin-owned surface by adding `plugins.entries.brave.config.webSearch.baseUrl` to the manifest, UI hints, config metadata, and shared config type; normalize and validate it; derive both `/res/v1/web/search` and `/res/v1/llm/context` from the configured base while preserving the default Brave endpoint; include the resolved base in cache identity; update https://docs.openclaw.ai/tools/brave-search; and add focused runtime/schema tests. The linked PR attempts and duplicate reports are useful context, but none shipped on current main.

## Risks / Open Questions

- A future implementation should include the resolved Brave base URL in cache identity to avoid cross-proxy cache collisions.
- Endpoint validation needs an explicit security choice for http loopback/private-network proxy URLs while preserving the existing trusted web-search endpoint guard.
