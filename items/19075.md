## Summary

Keep #19075 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still hardcodes Brave Search endpoints, exposes no Brave `baseUrl` in the plugin manifest/UI hints, keeps the top-level web search schema strict without a Brave endpoint override, and documents only `apiKey` plus `mode` for Brave. The related PR attempts are closed unmerged/not present in main, and the duplicate issues in the provided context were closed back to #19075, so this remains the canonical tracker.

## Best Possible Solution

Keep #19075 open as the canonical bundled Brave Search `baseUrl` tracker. Implement it in the Brave plugin-owned surface by adding `plugins.entries.brave.config.webSearch.baseUrl` to the manifest, UI hints, config metadata/help, and shared config type; normalize and validate the URL; derive both `/res/v1/web/search` and `/res/v1/llm/context` from the configured base while preserving the default Brave endpoint; include the resolved base in cache identity; update https://docs.openclaw.ai/tools/brave-search; and add focused runtime/schema tests.

## Risks / Open Questions

- A future fix should include the resolved Brave base URL in cache identity to avoid cross-proxy cache collisions.
- Endpoint validation needs an explicit choice for `http://` loopback/private-network proxy support while preserving the existing web-search SSRF guard.
- Prior PR attempts used both top-level and plugin-scoped shapes; the best fit for current main is the plugin-owned `plugins.entries.brave.config.webSearch.baseUrl` surface.
