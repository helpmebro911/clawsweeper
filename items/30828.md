## Summary

Current `main` has reactive rate-limit handling, not the requested operator-configurable per-model request/token budgets or header-driven scheduling. The codebase documents cooldown/failover after 429-like failures and a cap on long SDK `Retry-After` sleeps, but I found no config or runtime path that lets users set `tokens per interval` / `requests per interval`, and no runtime consumption of OpenAI `x-ratelimit-*` headers.

## Evidence

- **Docs describe cooldown/failover, not proactive rate-budgeting:** The model-failover docs say rate-limit errors put a profile into cooldown, optionally scoped to a model, with fixed exponential backoff (1m, 5m, 25m, 1h). That is reactive failover behavior, not configurable per-interval request/token throttling.
  - file: [docs/concepts/model-failover.md:120](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/docs/concepts/model-failover.md#L120)
  - command: `nl -ba docs/concepts/model-failover.md | sed -n '118,170p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Published config surface lacks request/token interval knobs:** `auth.cooldowns` exposes backoff and rotation settings such as `overloadedBackoffMs` and `rateLimitedProfileRotations`, but nothing for user-set `requests per N` or `tokens per N` limits.
  - file: [docs/gateway/configuration-reference.md:3511](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/docs/gateway/configuration-reference.md#L3511)
  - command: `nl -ba docs/gateway/configuration-reference.md | sed -n '3511,3548p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Runtime stores cooldown windows after failures:** The auth-profile usage code computes `backoffMs`, writes `cooldownUntil`, and tracks `cooldownReason` / `cooldownModel` after a rate-limit failure. It does not maintain token/request counters or interval buckets for model requests.
  - file: [src/agents/auth-profiles/usage.ts:749](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/agents/auth-profiles/usage.ts#L749)
  - command: `nl -ba src/agents/auth-profiles/usage.ts | sed -n '732,806p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Provider transport only parses Retry-After headers:** The guarded model fetch path reads `retry-after-ms` and `retry-after`, then suppresses long SDK-managed retries via `x-should-retry:false`. There is no handling here for `x-ratelimit-limit-*`, `x-ratelimit-remaining-*`, or `x-ratelimit-reset-*` headers.
  - file: [src/agents/provider-transport-fetch.ts:13](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/agents/provider-transport-fetch.ts#L13)
  - command: `nl -ba src/agents/provider-transport-fetch.ts | sed -n '1,90p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **No non-test runtime references to OpenAI x-ratelimit headers:** A repo search over `src/` and `extensions/` found no non-test runtime matches for `x-ratelimit`, which supports that the header values are not currently consumed to drive scheduling or throttling.
  - command: `rg -n "x-ratelimit" src extensions --glob '!**/*.test.ts'`
