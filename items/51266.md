## Summary

Current `main` already handles bad internal hook entries without taking down gateway startup. Hook loading now skips unreadable/missing/bad handlers, logs an error, and continues; gateway startup also wraps internal hook loading so failures do not crash the process.

## Evidence

- **Per-hook load failures are caught and skipped:** Directory-discovered hooks are loaded inside a per-entry `try/catch`; unreadable base dirs, boundary-check failures, bad exports, and import errors all log and continue instead of rethrowing.
  - file: [src/hooks/loader.ts:113](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/hooks/loader.ts#L113)

- **Legacy config hook paths no longer crash startup:** Legacy `hooks.internal.handlers[].module` entries are validated with `realpath`; missing paths log `Handler module path could not be resolved with realpath` and continue, and any later import failure is also caught.
  - file: [src/hooks/loader.ts:184](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/hooks/loader.ts#L184)

## Close Comment

Closing as implemented on `main` after a Codex review. Current hook loading already degrades gracefully instead of crashing gateway startup: per-hook load/import failures are caught and skipped in `src/hooks/loader.ts:113` and `src/hooks/loader.ts:184`, and gateway startup has its own outer guard in `src/gateway/server-startup-post-attach.ts:226`. There is also regression coverage showing missing or invalid hook modules are treated as non-loadable rather than fatal in `src/hooks/loader.test.ts:319`. `openclaw doctor` also warns that legacy `hooks.internal.handlers` is no longer the public path in `src/commands/doctor-config-flow.ts:104`. If you can still reproduce a startup crash on a current build, please open a new issue with the exact hook config shape and the startup log output.
