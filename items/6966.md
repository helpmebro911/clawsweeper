## Summary

Keep open. Current main has strong adjacent rate-limit failover work, including model-scoped cooldowns, ordered fallback, and primary cooldown probes, but it still does not implement the issue's requested dynamic model-tier policy or upstream the fork's round_robin/model-health strategy.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely implementation belongs in core model fallback/auth state: a documented policy for rate-limit-triggered downgrade, health/cooldown tracking per provider/model tier, observable decisions, and a conservative upgrade-back probe that does not wait for the active fallback to fail. The aashari fork's round_robin/model-health design is useful prior art but should not be treated as shipped upstream behavior.
