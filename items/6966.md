## Summary

Current main already implements rate-limit-triggered model fallback with model-scoped cooldown handling, which covers the request well enough.

## Evidence

- **Shipped release note:** The `2026.4.1` changelog explicitly says OpenClaw now caps same-provider auth-profile retries for rate-limit failures before cross-provider model fallback and adds `auth.cooldowns.rateLimitedProfileRotations`.
  - file: [CHANGELOG.md:1592](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/CHANGELOG.md#L1592)
- **Runtime escalation to model fallback:** The embedded runner increments rate-limit profile rotations and, once the configured/default cap is exceeded, throws a `FailoverError` with reason `rate_limit` to trigger model fallback.
  - file: [src/agents/pi-embedded-runner/run.ts:504](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/agents/pi-embedded-runner/run.ts#L504)
- **Model-scoped cooldown support:** Auth-profile cooldown checks bypass a rate-limit cooldown for a different requested model, and failures record `cooldownModel` so sibling models can still be attempted.
  - file: [src/agents/auth-profiles/usage.ts:240](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/agents/auth-profiles/usage.ts#L240)
- **Documented fallback behavior:** The docs state that if provider profiles fail, OpenClaw moves to the next configured fallback model for rate limits, and that same-provider fallback siblings may still be attempted during transient cooldowns.
  - file: [docs/concepts/model-failover.md:215](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/docs/concepts/model-failover.md#L215)
- **Tests cover rate-limit switching:** Tests verify cross-provider fallback after rate-limit rotation caps, immediate fallback when the cap is `0`, and same-provider fallback attempts during rate-limit cooldown.
  - file: [src/agents/model-fallback.run-embedded.e2e.test.ts:708](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/src/agents/model-fallback.run-embedded.e2e.test.ts#L708)

## Risks / Open Questions

- The current implementation is configured fallback-on-rate-limit plus cooldown/probe logic; it is not a broader metrics-driven or header-driven adaptive policy engine.

## Close Comment

Closing from a ClawSweeper/Codex review because current `main` already implements this well enough. This shipped in `v2026.4.1` per `CHANGELOG.md:1592`, so I’m closing this as implemented on `main`.
