## Summary

Keep open. Current main has ordered model fallback, auth-profile rotation, model-scoped cooldown metadata, and cooldown probing, but it still lacks the requested dynamic rate-limit-driven model routing policy: no model strategy config, no per-model health state, no threshold-based downgrade, and no gradual upgrade-back behavior. The aashari fork linked in the issue is useful prior art, not upstream behavior.

## Best Possible Solution

Keep this open as a core model-routing design issue. A good upstream solution would define a documented policy such as failover versus health-aware/round-robin, add per-provider/model/profile health state with success/failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema and docs, and cover rate-limit scenarios with regression tests. Treat the aashari fork linked in the discussion as prior art rather than shipped upstream behavior.
