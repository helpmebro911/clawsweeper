## Summary

Keep open. Current `main` has partial equivalents for the requested workflows, but not the generic three-part session API in the issue: native child-session forking exists, ACP/Codex-specific resume-by-id exists, and generic `continue most recent` behavior is not exposed as a public session option.

## Evidence

- **Native transcript forking exists, but only as child/thread session behavior:** The sub-agent docs describe `context: "fork"` as branching the requester transcript into a child session, and the session setup code only forks when `ParentSessionKey` is present for a new related session.
  - file: [docs/tools/subagents.md:80](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/docs/tools/subagents.md#L80)

- **Generic spawn schema does not offer a general forkSession(id) surface:** `sessions_spawn` exposes `context: "fork"` for native subagents, not an API to fork an arbitrary session by id. The runtime helper is wired to parent-session branching, not a general public fork endpoint.

## Risks / Open Questions

- The repo already contains overlapping session controls across native subagents, ACP, Codex, and session tools, so the issue may need a tighter scope to avoid duplicating existing surfaces.
- If maintainers interpret 'resume' broadly, they may consider `sessions_send` plus ACP/Codex resume sufficient; the remaining gap is mainly a unified public API and a 'continue most recent' convenience path.
