## Summary

Keep open. Current main has DeepSeek V4 replay payload handling and runtime-deps loop hardening, but there is still no source-backed proof that these changes resolve the reported gateway 100% CPU startup loop with DeepSeek V4 models. The issue discussion and linked PR both explicitly treat this as unconfirmed pending logs/profile or a latest-main reproduction.

## Best Possible Solution

Keep #71915 open. Ask for a latest-main or next-release retry with gateway logs and a CPU profile/stack sample. If the logs show repeated bundled runtime-deps staging, close or link it to the #71818 fixed class; otherwise investigate the DeepSeek V4 stream/payload path and gateway startup interaction directly, with PR #72132 treated as supplemental test coverage rather than closure proof.
