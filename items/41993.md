## Summary

Current main still rejects mixed DNS answers when any resolved address is blocked, before IPv4 preference or pinned lookup creation. The issue remains actionable: the SSRF pinning path should filter blocked DNS answers, fail only when none remain, and pin only the allowed candidates.

## Best Possible Solution

Keep this issue open. Implement the narrow SSRF pinning fix in `src/infra/net/ssrf.ts`: classify and filter DNS lookup results, throw only when no allowed answers remain, and pass only allowed results into IPv4-first dedupe and pinned lookup creation. Add focused tests for public IPv4 plus `::`, public IPv4 plus Teredo/reserved IPv6, all-blocked DNS still failing, and pinned results excluding blocked addresses.

## Risks / Open Questions

- The fix is security-sensitive: allowing mixed DNS answers is only safe if blocked answers are removed before the pinned lookup/dispatcher can use them.
- The existing RFC2544 opt-in and trusted proxy paths address adjacent fake-IP/proxy scenarios, but they do not solve the reported default public-IPv4 plus blocked IPv6 DNS response.
