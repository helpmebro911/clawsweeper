## Summary

Current main still appears to have the reported mixed DNS SSRF bug: the DNS result guard throws when any resolved address is blocked, before IPv4-first dedupe and pinned lookup creation. The related fix PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in only addresses fake-IP IPv4 ranges, not mixed public IPv4 plus blocked IPv6 answers.

## Best Possible Solution

Keep this issue open. Implement a narrow fix in `src/infra/net/ssrf.ts` that classifies resolved DNS answers, returns only allowed answers, throws only when no allowed answer remains, and passes only that filtered set into IPv4-first dedupe and pinned lookup creation. Add focused regression tests for public IPv4 plus `::`, public IPv4 plus reserved/Teredo-style blocked IPv6, all-blocked DNS still failing, and pinned results excluding blocked addresses.

## Risks / Open Questions

- This is security-sensitive: allowing mixed DNS answers is only safe if blocked answers are removed before any pinned lookup, dispatcher override, round-robin retry, or fallback path can use them.
- A config-only RFC2544/fake-IP workaround does not solve the default case described here, where a public IPv4 answer is mixed with a blocked IPv6 answer.
