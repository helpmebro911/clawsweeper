## Summary

Keep open. Current `main` does not show the issue's exact reported reproduction path through `acp`/`agents.defaults`, but it also does not contain the proposed runtime type guard. `resolveConfigForRead()` still calls `applyConfigEnvVars()` before schema validation, and `env.vars` entries are still forwarded to `containsEnvVarReference()` without a runtime `typeof value === "string"` check, so the same `value.includes is not a function` failure remains plausible for malformed `env.vars` content.

## Evidence

- **Current helper still has no runtime type guard:** `containsEnvVarReference` immediately calls `value.includes("$")` and is typed as `value: string`; there is no defensive runtime guard for non-strings.
  - file: [src/config/env-substitution.ts:137](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/env-substitution.ts#L137)
  - command: `nl -ba src/config/env-substitution.ts | sed -n '120,170p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Config env application still passes raw `env.vars` values through:** `collectConfigEnvVarsByTarget` copies `env.vars` entries without checking `typeof value === "string"`, and `applyConfigEnvVars` then calls `containsEnvVarReference(value)` on each collected entry.
  - file: [src/config/config-env-vars.ts:13](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/config-env-vars.ts#L13)

  - file: [src/config/io.ts:1050](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/io.ts#L1050)
