## Summary

Keep open. Current main still has the Telegram transport behavior described by this issue: a qualifying aggregate `fetch failed` error can arm sticky IPv4 fallback, the fallback dispatcher explicitly forces IPv4, and the resolved fetcher keeps using the successful fallback dispatcher on later calls. The issue discussion is also still relevant: a later user reported the same sticky fallback log shape on OpenClaw 2026.4.5, and the existing code/tests do not show a loopback-IPv4 `ECONNREFUSED` exception for the `/etc/hosts` IPv6 workaround.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin transport layer. Add a focused regression test for an aggregate `fetch failed` containing loopback IPv4 `ECONNREFUSED` plus IPv6 timeout under `dnsResultOrder: "verbatim"`, then adjust `extensions/telegram/src/fetch.ts` so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when IPv6 is configured or known healthy. Also decide explicitly, with tests, whether `UND_ERR_SOCKET` should continue to trigger sticky fallback.
