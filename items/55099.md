## Summary

Keep this issue open. Current main still classifies the reported transport failures as Telegram fallback triggers, still builds a sticky IPv4-only fallback dispatcher, and still has regression coverage asserting sticky IPv4 reuse rather than the requested loopback-ECONNREFUSED/IPv6-override exception.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin transport layer. Add a focused regression test for an aggregate `fetch failed` containing loopback `ECONNREFUSED` plus IPv6 timeout under `dnsResultOrder: "verbatim"`, then adjust `extensions/telegram/src/fetch.ts` so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when the configured or previously healthy route is IPv6. The fix should also decide, with coverage, whether the reported `UND_ERR_SOCKET` case should still trigger fallback.
