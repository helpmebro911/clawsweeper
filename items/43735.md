## Summary

Keep open. Current main loads workspace skills and hot-refreshes active watcher changes, but the CLI and Gateway session paths still decide whether to reuse a persisted skillsSnapshot using process-local version counters. A cached snapshot with version 0 can still look current after restart when skills changed while no watcher was active. The related implementation PR #48089 is closed unmerged, and #44898 is closed but does not provide an open canonical fix for this remaining path.

## Best Possible Solution

Keep this open until maintainers add restart-safe invalidation for persisted skillsSnapshot data or explicitly narrow the documented contract. A strong fix would persist and compare a deterministic inventory fingerprint for visible skill roots, loaded skill metadata, relevant filters/config, eligibility inputs, and prompt limits; then add regression tests for CLI and Gateway sessions where a cached version-0 snapshot survives a process restart after a workspace skill is added while no watcher was active. If remaining reports are frontmatter/filter/budget issues, add diagnostics or docs guidance separately.

## Risks / Open Questions

- Some missing-skill reports may be caused by intended filtering, invalid frontmatter, disable-model-invocation, allowlists, or prompt budgets. The remaining fix should target stale cached sessions rather than promising every skill directory always appears in <available_skills>.
- The current docs also say a new session refreshes skill snapshots. Maintainers should decide whether existing-session refresh after process restart is a guaranteed contract or a best-effort watcher behavior, then align code, tests, and docs.
