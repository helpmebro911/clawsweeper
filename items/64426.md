## Summary

Keep this issue open. Current main still has only opt-in generic tool-loop detection keyed on exact tool plus params hashes and diagnostic-only warning events; it does not implement the requested first-class read-tool same-turn reread guard with normalized path/set tracking, progress-aware suppression after edit/write/test, or a model-visible soft nudge. The related #63352 remains complementary because it is exec-scoped, not a canonical replacement for this read-tool path.

## Best Possible Solution

Keep this open and implement a read-specific warning layer on the existing tool-loop path: normalize first-class read path identities per run/session, detect repeated same-file and same-small-file-set rereads within one turn, reset or suppress after edit/write/test or broadened-context progress, emit a model-visible soft nudge plus diagnostic event, and add focused tests and docs/config guidance.

## Risks / Open Questions

- Existing opt-in genericRepeat and pingPong detection partially overlap exact duplicate read calls, so the eventual implementation should extend or coordinate with that state instead of adding conflicting loop accounting.
- Read-reread heuristics can false-positive during legitimate debugging, so the first implementation should remain warning-only and reset or suppress after clear progress such as edits, writes, tests, or materially broadened context.
