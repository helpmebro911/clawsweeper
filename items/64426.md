## Summary

Keep open. Current `main` has generic, opt-in tool-loop detection that can warn/block repeated identical `read` calls, but it is disabled by default and its heuristics do not clearly cover the issue's reported same-turn orbit across a small rotating file set (9/7/6/2 rereads) without hitting the per-file thresholds.

## Evidence

- **Loop detection is opt-in, not active by default:** `ToolLoopDetectionConfig` documents `enabled?: boolean` with default `false`, and the public loop-detection doc repeats that the guard is disabled by default.
  - file: [src/config/types.tools.ts:165](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/config/types.tools.ts#L165)

There is no heuristic here for a 3-4 file reread orbit or reread-ratio-based warning.

## Risks / Open Questions

- Closing as implemented would overstate coverage: the issue's concrete 9/7/6/2 reread distribution can evade the current per-file `warningThreshold` and non-ping-pong heuristics.
- Because the guard is disabled by default, the reported 'no warning' behavior remains plausible on current main without extra config.
- The GitHub discussion supplied no technical resolution beyond the issue body; the only comment in context is unrelated spam, so there is no maintainer/user confirmation that current behavior solves this report.
