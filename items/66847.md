## Summary

Keep this PR open. Current main and the latest release still lack the proposed protected-instruction-file mutation guard, and the PR is a plausible core security-hardening candidate rather than obsolete cleanup. The existing PR discussion is also useful: reviewers identified apply_patch, path-alias, and canonical-target bypass classes, and the PR author added follow-up commits addressing some of those areas.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with an equivalent core implementation. The best path is to define the approval/product boundary for agent edits to bootstrap and instruction files, then implement one central mutation policy covering host write/edit, sandbox write/edit, apply_patch add/update/delete/move, normalized path aliases, symlinks/hardlinks, and any intended exec/process scope, with regression tests and docs aligned to the chosen main-session memory/persona workflow.

## Risks / Open Questions

- The PR is security-sensitive: a partial guard that covers write/edit but misses apply_patch, sandbox bridge behavior, canonical aliases, symlinks, hardlinks, or exec-created writes can create a false sense of protection.
- The requested blanket block conflicts with documented main-session workflows that edit MEMORY.md and SOUL.md, so maintainers should decide the explicit approval path, exceptions, and docs before merge.
- The protected basename set is product policy, not just mechanics; maintainers should confirm whether CLAUDE.md, TOOLS.md, BOOT.md, TASKS.md, MEMORY.md, and SOUL.md should all share the same runtime guard semantics.
