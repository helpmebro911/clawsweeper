## Summary

Keep this PR open. Current main and the latest release still do not implement an equivalent protected instruction-file mutation guard, and the PR remains a focused core security-hardening candidate rather than conservative cleanup. The PR diff is limited to agent mutation code and tests, with no CI workflow, dependency, lockfile, package publishing, or external code-execution changes, but it still needs maintainer security/product review before merge because the policy must be complete across mutation paths and compatible with documented persona/memory workflows.

## Best Possible Solution

Keep this PR open for maintainer security review or replace it with an equivalent core implementation. The best path is one central agent-file mutation policy covering host write/edit, sandbox write/edit, apply_patch add/update/delete/move, normalized path aliases, symlink/hardlink canonical targets, and any intended exec/process scope, with an explicit operator approval/exception model for legitimate SOUL.md and MEMORY.md workflows. Add regression tests for those bypass classes and update the public docs so persona, bootstrap, and memory-write behavior remains clear.

## Risks / Open Questions

- A partial guard could create a false sense of protection if any mutating surface remains uncovered, especially apply_patch add/update/delete/move, sandbox bridge writes, exec/process-created writes, path aliases, symlinks, or hardlinks.
- The PR head still appears to need careful canonical-target review: write/edit symlink defense uses fs.realpathSync on the raw tool path, which may not match workspace-relative resolution, and apply_patch still checks hunk strings before final target resolution rather than enforcing a central post-resolution mutation policy.
- The protected basename set is product policy, not just mechanics; maintainers should confirm whether SOUL.md, MEMORY.md, IDENTITY.md, CLAUDE.md, TOOLS.md, BOOT.md, and TASKS.md should all share identical runtime guard semantics.
- The requested blanket block conflicts with documented main-session workflows that intentionally update SOUL.md and with memory promotion flows that write MEMORY.md, so an explicit operator approval path and exceptions are needed before merge.
- No local tests were executed because this review was constrained to read-only inspection; CI or maintainer-local validation is still needed before landing.
