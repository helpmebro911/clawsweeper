## Summary

Current main still leaves the actionable Race C unresolved. The plugin binding approval path still performs an un-serialized read-modify-write and updates the in-memory approvals cache before the async atomic write completes. Race A and Race B still match the issue's low-severity mitigated descriptions, but they do not eliminate the persistent approval data-loss risk.

## Best Possible Solution

Keep this issue open. Fix Race C in src/plugins/conversation-binding.ts by serializing the persistent approval read-modify-write path, preferably with the existing createAsyncLock pattern or an equivalent promise queue stored with plugin binding global state. The lock should cover loading/merging approvals and writing plugin-binding-approvals.json so concurrent allow-always decisions cannot persist stale snapshots. Add a focused regression test that forces two concurrent persistent approval writes to complete out of order, then verifies both the in-memory cache and a fresh reload from plugin-binding-approvals.json reflect the intended final state. Treat Race A and Race B separately as accepted mitigated behavior or document them if maintainers want that contract explicit.
