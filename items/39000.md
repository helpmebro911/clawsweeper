## Summary

Keep #39000 open. Current main still exposes gateway method registration and external gateway dispatch, but it does not provide a plugin-facing `api.callGatewayMethod()` or equivalent generic in-process request/response seam for plugin-to-plugin calls.

## Best Possible Solution

Keep #39000 open for plugin SDK/runtime design. The best path is a documented inter-plugin request/response seam, whether named `api.callGatewayMethod()` or expressed as a typed service/capability contract, with contract tests for plugin isolation, caller identity and scopes, plugin-registered handler dispatch, errors, recursion/lifecycle behavior, deferred-load timing, and plugin author docs.

## Risks / Open Questions

- Maintainer design is still needed for caller identity, scopes, authorization, missing-method behavior, error propagation, recursion, and startup/deferred-load lifecycle semantics.
- Existing `callGateway`/`GatewayClient` helpers go through Gateway auth and connection semantics, so treating them as the answer would leave the in-process plugin caller contract undefined.
