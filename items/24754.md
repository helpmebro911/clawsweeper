## Summary

Keep this issue open. Current main has browser Control UI/WebChat over Gateway WebSocket, browser device identity for pairing/auth, Tailscale/HTTPS access, PWA/web push, and OpenAI Realtime Talk over browser WebRTC, but it still does not implement the requested native no-third-party browser E2EE chat transport with browser-held per-device/per-agent keys and isolated encrypted channels. The maintainer-associated prior review comment reached the same conclusion, and the current source/docs still support it.

## Best Possible Solution

Keep this open as a product/security architecture item. The likely path is a maintainer-led design spanning Gateway protocol/auth, Control UI/WebChat, session/channel routing, transcript storage, docs, and tests. If maintainers want this to start plugin-first, first define the missing core/plugin API needed for an external browser-native E2EE transport to integrate without bypassing pairing, auth scopes, session isolation, or transcript semantics.
