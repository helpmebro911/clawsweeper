## Summary

Keep this issue open. Current main still provides browser chat through authenticated Gateway WebSocket RPC and provider-backed WebRTC voice, but not the requested native no-third-party browser E2EE chat transport with browser-held per-device/per-agent keys, encrypted envelopes, or isolated encrypted channel routing. The prior maintainer-associated Codex review already kept it open, and the current code/docs still support that conclusion.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. A strong path would define an additive Gateway protocol for encrypted chat envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated channel/session routing, transcript/storage semantics, relay/fallback behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers want plugin-first delivery, first design the missing plugin/SDK seams so an external browser-native E2EE transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.
