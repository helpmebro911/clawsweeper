## Summary

Keep this PR open. Current main already avoids the narrow empty-output rerun path, but it still does not recover arbitrary successful in-run toolResult text as the final cron output. The submitted patch is also not safe to land as-is because it suppresses retries from prompt wording alone and reads transcript tool results with cross-run correctness risks.

## Best Possible Solution

Keep the PR open, but require a rebase/rewrite on current main. The best fix is a structural cron output recovery path in the current run-executor/run outcome flow: recover only a verified successful current-run tool result, gate retry suppression on that recovered result instead of prompt keywords, reject stale/malformed/errored transcript entries, and add focused regression tests for raw stdout recovery, exact NO_REPLY, acknowledgement-without-tool-result retry, stale/older transcript entries, malformed timestamps, and errored tool results.

## Risks / Open Questions

- Functional: landing the PR as-is could suppress the only interim-ack retry for a raw-stdout-shaped prompt even when no successful current-run tool result exists yet.
- Correctness/data exposure: the transcript scan accepts entries with malformed timestamps because it only skips finite timestamps older than runStartedAt, so stale toolResult text could be surfaced as a fresh cron result.
- Coverage: current tests prove empty text is no longer treated as interim and fatal empty-payload errors are delivered, but they do not cover arbitrary raw stdout recovery from a successful toolResult.
- Security review: the PR does not touch CI, dependencies, package resolution, release scripts, permissions, secrets, or third-party code execution; the security-sensitive concern is the new transcript-to-output path potentially exposing stale or unrelated tool output if not narrowed to a verified current run.
