## Summary

Current main still does not implement the requested live provider/model-attempt update for the CLI TUI footer. The fallback loop still logs and records attempts but does not emit a `model_switch`-style agent event, the agent command path persists the successful fallback model only after the fallback run returns, and the TUI footer still renders from `sessionInfo` refreshed via session/list state rather than an in-flight fallback signal. Existing fallback lifecycle events are for auto-reply/web UI fallback notices after a fallback transition, not this real-time TUI status-bar path.

## Best Possible Solution

Keep this issue open. Implement a narrow, typed live fallback-attempt status path: emit an agent event before each actually attempted fallback candidate with runId, sessionKey, provider, model, attempt, and total; make the CLI TUI handle that stream with session/run filtering before the current unknown-run drop when needed; update `state.sessionInfo.modelProvider`/`model` and the footer immediately; and add regression coverage for both fallback emission and TUI state updates.

## Risks / Open Questions

- A fix should sessionKey-filter any live fallback-attempt event before mutating TUI footer state, otherwise another active session could change the visible provider/model.
- Existing post-success fallback lifecycle and persisted runtime model fields may be useful building blocks, but using them alone still leaves the in-flight stale-footer window reported here.
