## Summary

Keep open. Current main still lacks a live provider/model-attempt signal from the model fallback loop, and the CLI TUI still renders the footer from sessionInfo without handling any fallback/model-switch stream. The related status-bar fixes cover selected/default/persisted model display, not the in-flight fallback candidate path reported here.

## Best Possible Solution

Implement a narrow typed fallback-attempt status path: emit a runId/sessionKey/provider/model/attempt/total event immediately before each actually attempted fallback candidate, have the CLI TUI accept it with correct session/run filtering, update state.sessionInfo.modelProvider and state.sessionInfo.model, refresh the footer, and add regression coverage for both the fallback emission and TUI state update.
