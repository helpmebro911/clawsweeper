## Summary

Keep open. Current main still does not appear to send attempt-time provider/model changes from the model fallback loop to the terminal TUI, and the TUI footer still renders only from `state.sessionInfo.modelProvider`/`model`, which is refreshed from session listing rather than live fallback attempts. Existing fallback lifecycle handling is adjacent but post-run/Control UI oriented and does not solve the described terminal footer staleness.

## Best Possible Solution

Implement the issue’s core/TUI path rather than closing it: emit a session-scoped attempt event from the actual fallback candidate loop before each candidate that reaches `runFallbackAttempt`, then teach the terminal TUI to accept that event before the known-run filter when it matches the current session, update `state.sessionInfo.modelProvider`/`model`, refresh footer/autocomplete/render, and add focused regression coverage in `src/agents/model-fallback.test.ts` or a caller test plus `src/tui/tui-event-handlers.test.ts`. Existing Control UI fallback toast behavior can remain separate unless maintainers decide to unify the event shape.
