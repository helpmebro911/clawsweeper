## Summary

Keep open. Current main has a newer `preferOver`-based channel replacement path, but it does not implement this PR's central guard: an already-enabled same-channel alternative such as `openclaw-lark` suppressing Feishu auto-enable when no `preferOver` metadata is present. The PR itself needs rebasing because the auto-enable code/tests were split after it was opened, but the underlying regression path is still plausible.

## Best Possible Solution

Keep the item open, but ask for a rebase onto the current split auto-enable files. The best fix is either a regression test in `src/config/plugin-auto-enable.channels.test.ts` plus implementation in `src/config/plugin-auto-enable.shared.ts` that respects enabled same-channel alternatives, or an explicit maintainer decision that `openclaw-lark` must declare `channelConfigs.feishu.preferOver: ["feishu"]` in its manifest/catalog and this PR should be replaced by that narrower metadata fix.
