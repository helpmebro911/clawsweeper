## Summary

Keep open. Current `main` still lacks the requested alternative-plugin guard, and the PR as proposed is not conservative enough because it can suppress bundled auto-enable based on `plugins.entries.<id>.enabled=true` even when that alternative plugin is blocked by a restrictive `plugins.allow`.

## Evidence

- **Current main still auto-enables the bundled channel without an alternative-plugin guard:** In `materializePluginAutoEnableCandidatesInternal`, the loop only skips on denylist, explicit disable, `preferOver`, or already-enabled state before calling `registerPluginEntry`/`ensurePluginAllowlisted`. There is no check for another enabled plugin handling the same channel, so the reported Feishu/Lark conflict is not implemented on `main`.
  - file: [src/config/plugin-auto-enable.shared.ts:739](https://github.com/openclaw/openclaw/blob/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3/src/config/plugin-auto-enable.shared.ts#L739)
  - sha: [a4fc6c2409f9](https://github.com/openclaw/openclaw/commit/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3)
- **Allowlists stay authoritative over plugin-entry enablement:** Current tests assert that `plugins.entries.telegram.enabled=true` does not activate a bundled plugin when it is missing from `plugins.allow`; activation resolves to `not in allowlist`. That means the PR's proposed 'enabled alternative plugin' check is insufficient unless it also verifies allowlist eligibility.
  - file: [src/plugins/config-state.test.ts:306](https://github.com/openclaw/openclaw/blob/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3/src/plugins/config-state.test.ts#L306)
  - sha: [a4fc6c2409f9](https://github.com/openclaw/openclaw/commit/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3)
- **Only explicit channel enablement bypasses restrictive allowlists:** A separate test shows the supported bypass is `channels.<id>.enabled=true`, not merely `plugins.entries.<id>.enabled=true`. Skipping bundled Feishu auto-enable because `openclaw-lark` is marked enabled in `plugins.entries` can therefore leave the channel with no effective handler in allowlisted setups.
  - file: [src/plugins/config-state.test.ts:335](https://github.com/openclaw/openclaw/blob/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3/src/plugins/config-state.test.ts#L335)
  - sha: [a4fc6c2409f9](https://github.com/openclaw/openclaw/commit/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3)
- **No current-main regression coverage for the Feishu/Lark case:** A repo search under `src/config` and `src/plugins` found no `openclaw-lark` references or alternative-plugin regression coverage on `main`, so there is no evidence the reported scenario was already handled elsewhere.
  - command: `rg -n "openclaw-lark|feishu.*enabled.*alternative|alternative plugin" src/config src/plugins`
  - sha: [a4fc6c2409f9](https://github.com/openclaw/openclaw/commit/a4fc6c2409f9c28e55e04e22c2e243c6900d66a3)
- **PR discussion already identified the unsafe edge case:** The PR review comment correctly points out that treating an alternative plugin as active based only on `plugins.entries[record.id].enabled === true` ignores restrictive `plugins.allow`, which can suppress bundled auto-enable while the alternative still resolves disabled at load time.
  - sha: [887c17d2b678](https://github.com/openclaw/openclaw/commit/887c17d2b678196dd46b431c02e7daf41f2dc4d9)

## Risks / Open Questions

- Closing this PR would drop a still-unfixed bug report: current `main` does not implement the Feishu/Lark conflict guard.
- Merging the PR as-is risks a regression for allowlisted deployments, where `doctor --fix` could skip bundled channel enablement based on an alternative plugin entry that will not actually load.
