## Summary

Keep PR #66551 open. Current main still has the underlying host-read gap for HTML/XML/CSS/XML fallback documents: the MIME map recognizes these extensions, but the host-read guard only has validated fallback handling for CSV/Markdown and still rejects fallback document MIME. The PR is not fully superseded by #51562 or #67047; #51562 shipped MIME/outbound fallback coverage, and #67047 shipped CSV/Markdown host-read fallback only. The current PR likely needs revision because its review comments correctly flag that extension-only HTML/XML/CSS fallback should be gated by content validation before merge.

## Best Possible Solution

Keep #66551 open and ask for a safer revision: extend the validated text fallback model to HTML/XML/CSS/XML instead of returning solely on extension MIME, require no disallowed sniffed binary type, add regression tests for real `.html`/`.xml`/`.css` host-read acceptance and renamed binary rejection, and keep #51562/#67047 as related context rather than treating either as the canonical replacement.
