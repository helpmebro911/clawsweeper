## Summary

Keep open. Current main still handles Feishu `card.action.trigger` as a void path: it awaits or schedules `handleFeishuCardAction`, relies on token dedupe, and never returns an instant callback card/toast or an `ocf1` disable-state contract.

## Best Possible Solution

Keep this issue open and implement it in the bundled Feishu plugin: extend the typed `ocf1` interaction envelope with an opt-in disable-state or equivalent metadata, return a valid immediate Feishu callback response with a transitional disabled card/toast while synthetic command dispatch continues asynchronously, and add targeted tests for WebSocket/webhook callback payloads plus existing quick-action, approval-card, and duplicate-token behavior.
