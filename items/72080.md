## Summary

Keep #72080 open. The report is fresh, specific, and not protected by maintainer labels, but current main still appears to trust ACP runtime turn completion as the authoritative signal for task success. The code still writes the same synthetic OpenClaw ACP transcript shape with zero usage and `stopReason: "stop"`, and I did not find a current-main guard that distinguishes that transcript/proxy artifact from real harness completion before waking the parent with `Background task done`. Related ClawSweeper items are adjacent ACP lifecycle/session-visibility issues, not a canonical duplicate for this premature-completion bug.

## Best Possible Solution

Keep #72080 open and fix it in the ACP runtime/task lifecycle boundary. The best direction is to make ACP child task success depend on a real ACP/harness terminal condition, not the synthetic OpenClaw `acp-runtime` transcript record or an early zero-usage `stop` placeholder. Add a regression test that simulates an ACP child emitting initial text/placeholder completion while the backing harness/tool stream is still active, and assert `sessions_yield`/task delivery does not wake the parent until the authoritative runtime completion signal arrives.
