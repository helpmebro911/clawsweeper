## Summary

Keep #72080 open. Current main still treats ACP runtime turn completion as terminal for parent notification: `agent-command` emits ACP lifecycle end immediately after `acpManager.runTurn` resolves, `task-registry` maps lifecycle `phase: "end"` to a succeeded task and auto-delivers `Background task done`, and ACP transcript persistence still writes `provider: "openclaw"`, `model: "acp-runtime"`, `stopReason: "stop"`, zero-usage assistant entries. I found no current guard that distinguishes that synthetic/zero-usage ACP transcript shape from real harness completion or waits for an explicit harness/session-completion signal before waking the parent.

## Best Possible Solution

Keep this issue open. The best fix is to harden the ACP completion contract so parent task delivery only fires after true harness turn completion: make the acpx adapter or `AcpSessionManager.runTurn` wait for a real terminal event/process state, ignore transcript-only `acp-runtime` zero-usage placeholder records as completion evidence, and add a regression test where an ACP child emits early text/done-like output while the harness remains active so `sessions_yield` does not wake the parent prematurely.
