## Summary

Keep this issue open. Current main still trusts the ACP runtime iterator as the authoritative completion boundary: once `runtime.runTurn(...)` finishes, OpenClaw marks the ACP background task succeeded, emits the child run lifecycle end, persists the synthetic `openclaw` / `acp-runtime` zero-usage transcript entry, and lets the task registry deliver the parent wake. The reporter’s concrete timeline and the follow-up analysis about a possible downstream Claude ACP idle/result race remain actionable, and I did not find a current-main defensive guard or regression coverage for premature ACP completion.

## Best Possible Solution

Keep this issue open. Maintainers should capture the raw Claude SDK / ACP event order for a failing long run, then fix the completion boundary so OpenClaw marks ACP child tasks terminal only after an authoritative terminal ACP result, not merely after a downstream iterator returns on a transient idle. Add regression coverage in the ACP manager/runtime contract for text/tool progress followed by premature downstream completion, and ensure the task registry delivery, lifecycle event, transcript persistence, and `sessions_yield` parent wake fire only after true completion.
