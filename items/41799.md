## Summary

Keep this PR open. Current main has changed the outbound normalizer path so a runtime object result is discarded instead of flowing directly into `params.to`, but it still does not implement the PR's compatibility behavior of extracting `to` from legacy `{ ok, to }` normalizer results. The work is still plausible and focused, though it needs a rebase onto the current split outbound test files and maintainer review of the test-only mock/baseline churn.

## Best Possible Solution

Keep the PR open and rework it against current main: add a small runtime-only legacy `{ ok, to }` extraction helper in `src/infra/outbound/target-normalization.ts`, add focused regression coverage in `target-normalization.test.ts` plus the current target-resolver or core-send runner tests, and remove or justify the unrelated `.secrets.baseline` and OAuth test-mock churn. If maintainers decide not to support out-of-contract legacy normalizer objects, close with that explicit product/API decision rather than as obsolete.

- Current main may avoid the exact `to required` symptom for some inputs by discarding non-string normalizer results, so the PR should be revalidated around the narrower remaining behavior: extracting the legacy object's `to` value instead of falling back to raw input.

- The `.secrets.baseline` churn and file-scoped OAuth test mock are not runtime supply-chain changes, but they should be dropped, scoped, or clearly justified before merge.
