## Summary

Current main still matches the reported Nostr restart-loop failure. The DM subscription still passes an array-cast filter to `nostr-tools@2.23.3`, whose `subscribeMany` contract accepts a single `Filter`; the gateway account start path still resolves immediately after creating the Nostr bus, so the channel supervisor treats a successful start as task exit and schedules auto-restarts. Related PR #42832 remains open and is an implementation candidate, but no fix has landed on main.

## Best Possible Solution

Keep this issue open as the canonical current Nostr restart-loop/subscription tracker. Review and rebase PR #42832 or prepare a fresh patch on current main that passes a plain `Filter` object to `SimplePool.subscribeMany`, removes the unsafe cast, keeps `startNostrGatewayAccount` pending until abort using the existing plugin SDK lifecycle helper, closes the subscription and SimplePool relay connections during cleanup, and adds regression coverage for the serialized REQ filter shape plus gateway lifecycle so a successful Nostr start is not treated as an exit.

## Risks / Open Questions

- Closing this now would lose a concrete, currently reproducible channel bug before a fix has landed on main.
- A narrow DM-only patch should also consider cleanup: the current Nostr plugin has no `gateway.stopAccount` hook, and the returned `{ stop }` handle is not used when the task resolves immediately, so previous bus connections can remain alive during restart loops.
- There is a similar array-cast `subscribeMany` call in `extensions/nostr/src/nostr-profile-import.ts:125`; maintainers should decide whether to fix all `nostr-tools` filter-shape misuse in the same patch.
