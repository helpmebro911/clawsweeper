## Summary

Keep PR #45783 open. Current main does not implement the requested LLM-boundary privacy filter: there is no privacy config/module, and existing redaction remains log/tool-summary oriented. The PR is large and security-sensitive, but it is coherent, has substantial tests/review history, and the remaining work is not safely covered by current plugin/ClawHub surfaces.

## Best Possible Solution

Keep this open for maintainer judgment. The best path is either a smaller core implementation that adds a documented `privacy` config and a well-tested LLM-boundary transform across embedded runner, BTW, TTS summarization, cache traces, logs, and stream/result restoration, or a deliberate plugin-API design that exposes a safe dynamic bidirectional message-transform seam before moving the detector/replacer implementation to ClawHub.
