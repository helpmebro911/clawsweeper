## Summary

Keep this issue open. Current main has useful partial session-memory behavior for `/new` and `/reset`, plus startup hydration from recent memory files, but it still does not implement the requested automated synthesis workflow, `.session_memory_preserve.json`, full transcript exports with anchored receipts, or daily continuation packages. The issue discussion’s layered-memory concern remains valid, and related PR #50584 is still an open implementation candidate while #69943 remains an active safety concern around expanding automatic raw persistence.

## Best Possible Solution

Keep this open until maintainers either land a scoped core implementation or explicitly redirect the feature to a ClawHub/context-engine plugin design. A strong path should coordinate with #50584 and #69943: opt-in synthesis, sanitization before persistence, compact daily continuation packages, receipt links to full transcripts, clear daily-file behavior, timezone-aware filenames, and regression coverage for `/new`/`/reset` preservation plus startup hydration.

## Risks / Open Questions

- Expanding automatic persistence before the safety work in #69943 is resolved could amplify raw transcript/token reinjection problems.
- PR #50584 covers important pieces of the request, but it is open/unmerged and does not cover every requested artifact, especially `.session_memory_preserve.json` and anchored full-transcript receipts.
- A ClawHub/context-engine plugin may be the right long-term product shape, but maintainers should decide that explicitly because current main already ships a bundled command-scoped session-memory hook.
