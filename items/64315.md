## Summary

Keep open: the request is coherent, recent, and still not implemented on current `main`. Current status surfaces favor truncation or hiding text in constrained layouts rather than exposing overflow via scrolling/marquee behavior, and the repo vision keeps CLI/web UX ergonomics in core scope.

## Evidence

- **In-scope core UX work:** `VISION.md:31` lists "Ergonomics across CLI and web frontend" as a project priority, so compact status-surface behavior is core product work rather than an obvious ClawHub-only plugin item.
  - file: [VISION.md:31](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/VISION.md#L31)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)
- **Current task status formatting truncates long text:** `src/tasks/task-status.ts:12` sets `TASK_STATUS_TITLE_MAX_CHARS = 80` and `TASK_STATUS_DETAIL_MAX_CHARS = 120`, and `truncateTaskStatusText()` at `src/tasks/task-status.ts:41` hard-truncates overflow with an ellipsis.
  - file: [src/tasks/task-status.ts:41](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/tasks/task-status.ts#L41)
  - sha: [11804a484ded](https://github.com/openclaw/openclaw/commit/11804a484ded89374afa55020813cfa265aa143a)

  - file: [src/tasks/task-status.test.ts:61](https://github.com/openclaw/openclaw/blob/11804a484ded89374afa55020813cfa265aa143a/src/tasks/task-status.test.ts#L61)
