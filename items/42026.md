## Summary

Keep this issue open. Current main has adjacent work around runtime plans, embedded/plugin harness selection, hot config reload, per-agent workspaces/auth/session stores, and headless node hosts, but it still does not implement the requested distributed split between a lightweight gateway control plane and separately registered per-agent compute runtimes. The requested `openclaw-runtime` service, gateway-to-runtime registration/streaming protocol, distributed/local-split deployment modes, and per-agent lifecycle CLI remain missing. The related K8s worker-node item #12429 is narrower and still open, while the closed Agents Plane discussion #17299 is a different provisioning layer rather than a replacement for this runtime architecture RFC.

## Best Possible Solution

Keep this issue open for maintainer architecture review. The best path is an additive design that preserves embedded Gateway mode, first defines a stable gateway-to-agent-runtime protocol and local process split with per-agent lifecycle commands, then layers container/Kubernetes backends and stricter per-agent secret/resource isolation on top. The implementation should reuse the existing runtime plan/harness, config reload, gateway protocol, node-host connection/auth, and per-agent workspace/auth/session-store boundaries where they already solve part of the problem.

## Risks / Open Questions

- The RFC is broad and should likely be decomposed into smaller milestones before implementation: protocol boundary, local process split, per-agent lifecycle commands, secret scoping, and orchestration backends.
- Some restart/config pain is already partly addressed by hot reload and per-agent isolation surfaces, so future work should build on those rather than adding a parallel control path.
- #12429 may become a useful Kubernetes-specific subtask, but closing this broader RFC in favor of it would drop the local-split/runtime-protocol and per-agent lifecycle portions.
