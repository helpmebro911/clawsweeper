## Summary

Keep this open. Current main has adjacent infrastructure, including multi-agent routing, embedded agent harness selection, runtime-plan policy plumbing, node transport, hot config reload, and gateway service lifecycle commands. It still does not implement the requested Gateway control-plane plus separately registered per-agent compute runtime split, local-split/distributed modes, or per-agent start/stop/restart lifecycle. TinyTb's comment usefully narrows a Phase 0 to per-agent lifecycle without a full distributed backend, and that remains missing too.

## Best Possible Solution

Keep this open for maintainer architecture review. The best path is an additive design that preserves embedded Gateway mode, defines a stable Gateway-to-agent-runtime protocol and runtime role/capability handshake, adds a local per-agent process split with start/stop/restart and health, then layers container/Kubernetes backends and stronger per-agent secret/resource isolation while reusing existing harness, runtime-plan, node auth, sandbox, config reload, and session routing pieces.

## Risks / Open Questions

- The RFC is broad and should probably be split into milestones: protocol boundary, local per-agent process lifecycle, runtime registration and health, per-agent secrets/resource isolation, then orchestration backends.
- Current node transport, agent harnesses, runtime-plan policy, sandboxing, hot reload, and session routing are useful building blocks but could become competing control paths if a future design does not integrate them deliberately.
- Related #12429 remains relevant for Kubernetes worker pairing, but it is narrower infrastructure work and should not be treated as the canonical tracker for this agent compute runtime split.
