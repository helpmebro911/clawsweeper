## Summary

Current main still exposes the freshly rotated device token in the `device.token.rotate` RPC success payload. The behavior is also preserved by gateway unit/integration tests, consumed by the Control UI, and documented by the CLI docs, while the related redaction PR is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and handle it as a gateway protocol/security design fix. The likely path is to change `src/gateway/server-methods/devices.ts` so normal `device.token.rotate` responses do not expose `entry.token`, add regression tests proving the secret is absent or explicitly constrained to a safer bootstrap flow, and update Control UI, CLI behavior, and docs to use a safer retrieval/provisioning path if plaintext delivery remains necessary in a narrow flow.
