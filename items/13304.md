## Summary

Keep #13304 open. Current main still lacks declarative content-based agent routing in bindings: the binding type/schema/docs only allow static channel/account/peer/guild/team/roles constraints, the route resolver has no message body input or transformed BodyForAgent output, and current plugin hooks can override models, observe/claim/short-circuit messages, or synthesize replies but cannot return a normal target agent plus stripped prompt through binding resolution. The related #10969/#11220 work is model-routing, not a canonical replacement for this agent-routing request.

## Best Possible Solution

Keep #13304 open. Implement either `bindings[].match.content` with deterministic prefix/regex matching plus optional `stripMatch`, or a documented pre-routing plugin hook that can return `agentId`, session behavior, and transformed `BodyForAgent` before normal dispatch. Update AgentBindingMatch, strict Zod schema, generated schema/help/docs, resolver input/cache keys/precedence, representative channel call sites, and route/channel regression tests. Treat #10969/#11220 as related model-routing work, not a superseding tracker.

## Risks / Open Questions

- Adding regex/prefix matching needs deterministic precedence with existing peer/guild/team/account tiers and safe regex validation.
- If route results depend on message text, resolver cache keys and all channel/plugin route call sites need careful updates so stale cached routes cannot ignore content.
- A hook-based alternative would need an explicit pre-routing contract that can return agentId, session behavior, and transformed BodyForAgent without forcing synthetic replies or middleman sessions.
