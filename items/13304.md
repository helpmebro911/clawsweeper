## Summary

Keep #13304 open. Current main still routes bindings by static channel/account/peer/guild/team/role context only, and the newer plugin hooks can observe, claim, or synthesize replies but do not provide a documented way to select a different configured agent plus strip/transform the inbound body for the normal channel turn. The linked #10969 and #11220 are related model-routing requests, not a canonical open tracker for content-based agent routing.

## Best Possible Solution

Keep the issue open. Implement either a first-class bindings[].match.content contract or a documented pre-routing plugin hook that can return target agentId, session behavior, and transformed BodyForAgent. The design should define safe prefix/regex semantics, deterministic precedence with existing binding tiers, stripMatch behavior, schema/docs updates, and resolver plus representative channel regression tests.
