## Summary

Keep #13304 open. Current main still does not implement declarative content-based agent routing in `bindings`: binding types, strict config validation, generated schema/help, docs, and the route resolver all only cover static channel/account/peer/guild/team/role constraints. Existing plugin hooks and related #10969/#11220 model-routing work do not supersede this agent-routing request because they cannot return a normal resolved target agent plus stripped/transformed inbound body through binding resolution.

## Best Possible Solution

Keep #13304 open. The best path is to either implement `bindings[].match.content` with deterministic prefix/regex matching and optional `stripMatch`, or design a documented pre-routing hook that can return `agentId`, session behavior, and transformed `BodyForAgent` before normal dispatch. The implementation should update `AgentBindingMatch`, strict Zod config schema, generated schema/help/docs, resolver input/cache keys/precedence, representative channel route call sites, and regression tests. Treat #10969/#11220 as related model-routing work, not a superseding tracker.

## Risks / Open Questions

- Adding content-dependent routing would need deterministic precedence with existing peer/guild/team/account/channel tiers.
- If route results depend on message text, resolver caching and channel/plugin call sites need updates so cached static routes cannot mask content changes.
- Regex support would need safe validation and clear docs for matching, stripping, and failure behavior.
