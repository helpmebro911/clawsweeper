## Summary

Keep open. Current `main` does not implement the Windows PATH bootstrap from this PR, and the proposed approach conflicts with the repo’s newer PATH-hardening policy around user-writable directories. This needs maintainer judgment/rework, not conservative auto-close.

## Evidence

- **Current main still lacks win32 bootstrap logic:** `candidateBinDirs()` on current `main` hardens PATH by prepending only trusted system dirs and appending non-system fallbacks; there is no `win32` branch for `%LOCALAPPDATA%`, `%APPDATA%`, `%ProgramFiles%`, or `%SystemRoot%` in the current implementation.
  - file: [src/infra/path-env.ts:95](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/infra/path-env.ts#L95)
  - command: `nl -ba src/infra/path-env.ts | sed -n '1,240p'`
  - sha: [bc2d53dacd4e](https://github.com/openclaw/openclaw/commit/bc2d53dacd4eda5488af5f884791ecfb80eb06ee)
- **Current tests enforce anti-hijack ordering:** The new `path-env` regression tests on `main` explicitly require user-writable/home bin directories to stay after trusted system dirs, including a test named `places ~/.local/bin AFTER /usr/bin to prevent PATH hijack` and another asserting all user-writable home dirs come after `/usr/bin`.
  - file: [src/infra/path-env.test.ts:254](https://github.com/openclaw/openclaw/blob/bc2d53dacd4eda5488af5f884791ecfb80eb06ee/src/infra/path-env.test.ts#L254)

That direction cuts against prepending user-writable Windows bootstrap dirs ahead of the existing PATH.

That concern remains material given current main’s anti-hijack policy.

- Merging the PR as-is would likely need redesign because several proposed prepends are user-writable Windows directories that conflict with current PATH-hardening and could alter command resolution unexpectedly.
