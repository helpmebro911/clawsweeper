## Summary

Keep #7338 open. Current main still does not implement opt-in, domain-scoped OpenClaw agent attestation for arbitrary external HTTP, skills, or tool-driven requests. The existing provider-scoped attribution/custom-header paths and plain web_fetch headers are adjacent, not a replacement, and the recent issue discussion already recorded the same keep-open conclusion.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether agent attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped configuration, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other domains. Treat #11137/provider-header work as adjacent, not a replacement.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, but that does not solve the requested shared OpenClaw attestation contract for built-in web_fetch, skills, and arbitrary tool HTTP clients.
- Any implementation needs explicit security and privacy design for signing keys, verifier trust, replay expiry, audience/domain scoping, opt-in defaults, and cross-service tracking risk.
