## Summary

Keep #7338 open. Current main still has only provider-scoped attribution/custom-header support for model/provider traffic and plain built-in web_fetch headers; it does not implement the requested opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary external HTTP, skills, or tool-driven requests. The prior issue discussion also explicitly treated #11137 as adjacent rather than a replacement.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether agent attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other domains. Treat #11137/provider-header work as adjacent, not a replacement.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, and provider plugins can add provider-specific headers, but that does not solve the requested shared OpenClaw trust-root problem for built-in web_fetch, skills, and arbitrary tool HTTP clients.
- Any implementation needs explicit security/privacy design for signing keys, verifier trust, replay expiry, audience/domain scoping, opt-in defaults, and cross-service tracking risk.
