## Summary

Keep this PR open. Current main still does not persist bundled session-memory on automatic daily/idle rollover: the bundled handler, metadata, public docs, and tests remain command-only, while automatic stale rollover is exposed through plugin session lifecycle hooks. The PR is still a plausible focused bug fix, but it needs rebase/maintainer review for the canonical integration path and the ACP duplicate-write concern raised in review.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical automatic-rollover integration path, then update bundled `session-memory` metadata/docs/tests so daily and idle automatic rollovers persist the prior session once while explicit `/new` and `/reset`, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- Functional risk if merged as-is: the proposed `isNewSession && !resetTriggered` rollover predicate may double-write memory for stale ACP-bound explicit reset/new commands unless it also accounts for the ACP in-place reset path and `resetHookTriggered` behavior.
- Integration risk: current main already has plugin-level `session_end`/`session_start` lifecycle hooks for automatic rollover, so maintainers should choose whether bundled `session-memory` should consume that lifecycle payload or whether a new typed internal `session:rollover` event is the intended contract.
- Rebase risk: the PR base predates current main, and the session lifecycle path has changed enough that targeted tests around daily rollover, idle rollover, explicit `/new`/`/reset`, and ACP-bound resets should be rerun after rebase.
