## Summary

Keep open. Current main still treats only an exact single silent token or a JSON NO_REPLY envelope as silent; doubled `NO_REPLY\n\nNO_REPLY` is not suppressed and normalization can reduce it to a deliverable `NO_REPLY`. The reporter's Quiubo-side guard in bitlabs-com/quiubo#463 is a useful workaround, but the shared OpenClaw suppression predicate still lacks this core behavior.

## Best Possible Solution

Keep this issue open until main adds shared repeated silent-token detection in `src/auto-reply/tokens.ts` or lands an equivalent implementation. The fix should update `isSilentReplyPayloadText` and the delivery/normalization path, add regression coverage for `NO_REPLY\n\nNO_REPLY` and custom tokens such as `HEARTBEAT_OK`, and preserve the existing #19537 substantive-reply guard. Review PR #69472 as the likely consolidation path, but do not close this issue before the relevant fix is merged.
