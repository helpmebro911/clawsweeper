## Summary

Current main still has the reported core suppression gap: repeated bare silent-token payloads such as `NO_REPLY\n\nNO_REPLY` do not match the shared silent predicate, and normalization can strip only one trailing token and leave a visible `NO_REPLY`. The Quiubo plugin-side guard mentioned in the issue is a useful workaround, but plugins relying on OpenClaw host suppression still inherit the shared core behavior.

## Best Possible Solution

Keep this issue open. The best fix is to update the shared silent-token predicate or normalization path in `src/auto-reply/tokens.ts` and `src/auto-reply/reply/normalize-reply.ts` so token-only sequences like `NO_REPLY\n\nNO_REPLY` are suppressed while preserving #19537 mixed-content behavior, then add focused regression tests in `src/auto-reply/tokens.test.ts` and the reply normalization/runtime delivery surface.
