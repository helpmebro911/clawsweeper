## Summary

Keep open. Current main still leaves the session-memory slug generator on the embedded runner's shared auth-profile failure policy, so a billing-classified helper failure can still write `disabledReason: "billing"`/`disabledUntil` into shared auth state. The helper-local failure policy exists and is already used by active-memory, but the slug-generator call site has not adopted it.

## Best Possible Solution

Keep the issue open. Fix `src/hooks/llm-slug-generator.ts` by passing `authProfileFailurePolicy: "local"` to `runEmbeddedPiAgent`, then add a focused regression test in `src/hooks/llm-slug-generator.test.ts` asserting the slug-helper embedded run uses local failure policy and falls back without recording shared auth-profile failure state. Treat #33962 as related slug-generator performance/model-selection work, not as the canonical fix for this auth-state poisoning bug.

## Risks / Open Questions

- Current main has partial recovery behavior for some billing-disabled primary profiles via cooldown probes, so the exact "no upstream call for 5 hours" symptom may differ in some routing shapes; the underlying shared-state poisoning from the slug helper remains visible in source.
- A broad classifier change for Anthropic extra-usage messages could regress legitimate billing/quota handling. The narrower, lower-risk fix is to make the slug helper use the existing local auth-profile failure policy.
