## Summary

Keep PR #45270 open. Current main still has the exact stale-token fallback this PR is trying to remove, now in `src/status/status-message.ts` after the status builder moved out of `src/auto-reply/status.ts`. The merged Control UI freshness PR #71297 explicitly left #45270 open because `/status` is outside that PR's scope.

## Best Possible Solution

Keep the PR open and rebase/port its narrow behavior into `src/status/status-message.ts`: only trust stored context occupancy when `totalTokensFresh !== false`, keep transcript-usage promotion as the recovery path, and add the missing `src/auto-reply/status.test.ts` regressions for both missing stale totals and explicit stale totals.

## Evidence

(entry?.inputTokens ?? 0) + (entry?.outputTokens ?? 0)` without checking `entry.totalTokensFresh`, so a stale entry with missing totalTokens can still render accumulated input+output as context usage.
  - file: [src/status/status-message.ts:574](https://github.com/openclaw/openclaw/blob/b979f2964c9609a4d2ce9e96d643ed020e3fc00e/src/status/status-message.ts#L574)
