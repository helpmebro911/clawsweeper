## Summary

Keep open. Current main still lacks the recovery mechanism described by #41064: the Gateway hello snapshot does not carry active chat runs, and the Control UI explicitly clears `chatRunId` on reconnect, which can make the composer render idle controls while a run may still be active. The linked PR #41067 is relevant and appears to target this issue, but it is still open in the provided context, so the bug should remain tracked rather than closed.

## Evidence

- **Reconnect path clears active run state:** `connectGateway` handles `hello` by setting `host.chatRunId = null`, clearing `chatStream`, and resetting tool stream state under the comment `Reset orphaned chat run state from before disconnect.` This is the behavior the issue reports as losing busy state after reconnect/tab return.

  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Hello snapshot has no active chat run recovery field:** `SnapshotSchema` includes presence, health, stateVersion, uptime/config fields, sessionDefaults, authMode, and updateAvailable, but no `activeChatRuns` or equivalent session-to-run state that the UI could use to recover `chatRunId`.
  - file: [src/gateway/protocol/schema/snapshot.ts:46](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/gateway/protocol/schema/snapshot.ts#L46)
