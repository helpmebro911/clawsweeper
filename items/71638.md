## Summary

Keep this issue open. Current main still has the metadata escape hatch and Telegram reply/forward normalization for agent context, but the received hook path still cannot carry Telegram forward, via-bot, or reply provenance into MessageReceivedHookEvent metadata. The prior Codex review comment remains accurate against current main.

## Best Possible Solution

Keep this issue open and implement a Telegram-owned additive fix. Add a bounded provenance metadata path to the canonical received-hook context, derive metadata.forward, metadata.viaBot, and metadata.replyTo from inbound Telegram messages, apply it to both normal dispatch and silent-ingest emits, reuse existing forward/reply visibility normalization where possible, and add focused mapper plus Telegram emit-path tests.
