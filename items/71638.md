## Summary

Keep #71638 open. Current main still has the generic `MessageReceivedHookContext.metadata` escape hatch and Telegram already derives reply/forward provenance for agent prompt/session context, but that provenance is not threaded into `message:received` hook metadata. I found no `via_bot`/`viaBot` handling or `metadata.forward`/`metadata.viaBot`/`metadata.replyTo` implementation in the Telegram hook path.

## Best Possible Solution

Implement a Telegram-owned additive fix: normalize bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from inbound Telegram `msg`, thread it through the canonical received-hook mapper path used by both normal dispatch and mention-skip ingest, preserve existing group supplemental-context visibility rules, truncate reply text, and add focused tests for `toInternalMessageReceivedContext` plus the Telegram received-hook emit paths.
