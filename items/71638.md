## Summary

Keep #71638 open. Current main still has the generic `MessageReceivedHookContext.metadata` escape hatch, and Telegram already derives reply/forward provenance for prompt context, but the canonical received-hook context and both plugin/internal received-hook mappers still only carry generic routing/sender metadata. I found no `via_bot`/`viaBot` implementation in the Telegram hook path and no tests asserting `metadata.forward`, `metadata.viaBot`, or `metadata.replyTo`.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix: normalize inbound Telegram `msg.forward_origin`/legacy forward fields, `msg.via_bot`, and `msg.reply_to_message` into bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo`; thread that through `CanonicalInboundMessageHookContext`; update `toInternalMessageReceivedContext` and `toPluginMessageReceivedEvent`; and add focused mapper plus Telegram received-hook tests.
