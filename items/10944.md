## Summary

Keep #10944 open. Current main at e672b61417af5c45b0431df6d9109a1f4b618ef5 still has no `channels.telegram.parseMode` or per-account parse-mode config, and Telegram outbound delivery remains HTML-first with plain-text fallback only after HTML parse/splitting failures. The prior steipete/Codex issue comment already kept this open, and related #10758 is narrower custom-emoji/raw-HTML work, not a canonical tracker for configurable parse mode.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface. A conservative path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired consistently through sends, captions, edits, draft previews, and delivery helpers. Update strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and tests for HTML plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji support as explicit maintainer/security choices or separate follow-ups.
