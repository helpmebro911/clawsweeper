## Summary

Keep #10944 open. Current main still has no `channels.telegram.parseMode` or per-account parse-mode config, and Telegram outbound paths remain HTML-first with parse-error plain-text fallback only. The prior issue discussion also includes a maintainer/Codex review explicitly keeping this item open; related #10758 is a narrower closed custom-emoji/raw-HTML request, not a canonical tracker for this config work.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface. A conservative path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired consistently through sends, captions, edits, draft previews, and delivery helpers. Update strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and tests for HTML plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji support as an explicit maintainer/security choice or separate follow-up.
