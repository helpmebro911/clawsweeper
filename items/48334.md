## Summary

Keep PR #48334 open: current main has related context-token work, but it does not implement the PR’s central registry-aware cron persistence fix or the stale 200k status-summary repair.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase/adaptation: add a registry-aware awaited context-window resolver or equivalent cron hydration before persisting session `contextTokens`, add a guarded status-summary repair for stale `DEFAULT_CONTEXT_TOKENS` values that preserves explicit caps, and keep focused regression tests for profile cron persistence and stale status display.

This does not await registry discovery before writing the session entry.
