## Summary

Current main still acknowledges Telegram webhook updates with `200` before any durable handoff, then processes the update asynchronously in memory. The linked contributor PRs proposed the right class of disk-backed queue/replay fix, but they are closed unmerged and no equivalent Telegram webhook persistence exists on main.

## Best Possible Solution

Keep this issue open and rework the contributor queue approach into the Telegram plugin. The likely fix belongs in `extensions/telegram`: validate the secret and JSON body, durably enqueue the raw update in the OpenClaw state directory before ACK, process through the existing bot pipeline, dequeue only after all synchronous and deferred update work completes, and replay pending entries on webhook startup with focused tests for restart replay, ordering, dedupe, malformed entries, and security handling.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after the current `200` response and before or during in-memory `bot.handleUpdate` completion.
- A future fix needs careful at-least-once semantics: validate the Telegram secret before persistence, use path-safe queue keys and restrictive file permissions, replay pending entries on startup, and dequeue only after deferred work completes to avoid spoofing, traversal, or duplicate-processing regressions.
