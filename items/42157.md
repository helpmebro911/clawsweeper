## Summary

Current main still acknowledges Telegram webhook payloads before durable handoff or processing completion. The code validates the request and JSON body, returns 200, then runs bot.handleUpdate in a detached in-memory async task; failures are logged after ACK and there is no Telegram inbound webhook persistence/replay integration. The related contributor PRs #40056, #42185, and #42523 were closed unmerged, so they are useful design evidence but not shipped fixes.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: validate request guards, Telegram secret, and JSON body first; durably enqueue the raw update in OpenClaw state before returning 200; process through the existing Telegram bot pipeline; remove the queue entry only after synchronous and deferred work has completed; and replay pending entries before accepting new webhook traffic on startup. The closed contributor PRs #40056, #42185, and #42523 are useful prior art, but a maintainer-reviewed implementation still needs to land.

## Risks / Open Questions

- Users can still lose Telegram webhook messages when the gateway exits after the 200 response and before or during in-memory bot.handleUpdate completion.
- A proper fix needs at-least-once semantics tests for duplicate update handling, ordering, malformed queue files, disk write failures, and secret validation before persistence.
