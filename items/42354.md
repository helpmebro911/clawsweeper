## Summary

Keep this issue open. Current main still lacks a shipped Feishu WebSocket startup/long-connection repair: the WSClient factory does not pass heartbeat config, the monitor starts one client once without an OpenClaw-owned retry/backoff loop, and the documented/default Feishu transport is still WebSocket. Related maintainer PR #72411 is open and unmerged, so it is an implementation candidate rather than evidence that the issue is resolved.

## Best Possible Solution

Keep this issue open and use #72411 or a successor maintainer-approved PR as the active fix path. The fix should validate the current @larksuiteoapi/node-sdk behavior, repair startup/reconnect handling in the Feishu plugin without global SDK mutation, add focused regression coverage for heartbeat config and startup/retry behavior, and document any Feishu-vs-Lark domain/webhook limitation if that remains a user-facing workaround.
