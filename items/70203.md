## Summary

Keep open. Current `main` still flags any `startedAt < createdAt` as `inconsistent_timestamps` with no jitter tolerance, and the cron task creation path still records `createdAt` from a fresh `Date.now()` while accepting an earlier captured `startedAt`, so the reported 1 ms false positive remains plausible on current code. The linked closed PR #70216 is not reflected in the checked-out `main` or the latest release files inspected here.

## Evidence
