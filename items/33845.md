## Summary

Keep this PR open. Current main still loses Slack thread participation across gateway restarts because the Slack cache is only a process-local dedupe cache, while Slack mention gating still depends on that cache to keep replying in previously joined threads without another mention. The PR is a focused implementation candidate with useful tests, but it needs maintainer review and storage/security cleanup before merge.

## Best Possible Solution

Keep this PR open for maintainer review rather than close it as stale or implemented. The best path is to preserve the Slack restart-survival behavior, then rework the storage onto the accepted persistent primitive, preferably current `openclaw/plugin-sdk/persistent-dedupe` or the `persistent-keyed-store` direction in #70626 if that lands. The final implementation should remove production-exported test hooks, constrain storage to the state-dir contract, use bounded locked atomic storage, preserve recency/LRU semantics, and keep regression coverage for restart round-trip, TTL expiry, missing/corrupt state, clear semantics, max-entry behavior, and split module instances. Maintainers should also coordinate with the overlapping open Slack persistence work in #58731 rather than close either implementation prematurely.

## Risks / Open Questions

- Closing this PR now would drop a focused fix while current main still has the restart-survival bug for Slack thread participation.
- Merging the PR as-is would expose `_resetForTests` and `_flushPersist` from production Slack API surface, allowing same-process consumers to redirect the persistence path and then write, rename, or unlink files via exported helpers and `clearSlackThreadParticipationCache`.
- The PR-local JSON persistence reimplements storage behavior already covered by SDK helpers and still has security/correctness concerns: predictable temp path, no explicit restrictive file/directory mode, no file lock or cross-process serialization, unbounded synchronous file read/JSON parse, and future timestamps accepted as non-expiring until wall clock catches up.
- The PR caps the serialized array to `MAX_ENTRIES`, but the shadow `persistState.entries` map itself is not trimmed and updates use `Map.set` without reinsertion, leaving possible memory growth and recency-order drift in busy workspaces.
