## Summary

Keep this PR open. Current main still uses a process-local Slack thread participation cache, so the restart-survival bug this PR targets is not implemented. The PR is a real implementation candidate, but it needs maintainer review and storage/security cleanup before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to preserve the Slack restart-survival behavior, then rework it onto the accepted persistent storage primitive, preferably current `openclaw/plugin-sdk/persistent-dedupe` or `openclaw/plugin-sdk/persistent-keyed-store` if #70626 lands. The final implementation should remove production-exported test hooks, constrain persistence to the state-dir contract, use bounded locked atomic storage, preserve recency/LRU semantics, and keep regression coverage for restart round-trip, TTL expiry, missing/corrupt state, clear semantics, max-entry behavior, and split module instances.

## Risks / Open Questions

- Merging the PR as-is would introduce production-exported test helpers that can override the persistence path and then write or unlink files through `_flushPersist` / `clearSlackThreadParticipationCache`.
- The PR’s Slack-local JSON store reimplements storage semantics already covered by SDK helpers and still has unresolved storage-correctness concerns: predictable temp path, no cross-process locking, unbounded read/parse before validation, future timestamp acceptance, and a shadow map that can grow or preserve stale insertion order under high volume.
- Closing the PR now would drop a focused bug-fix candidate while current main still loses Slack thread participation across gateway restarts.
