## Summary

Keep open. Current main still does not implement `fallbackChannel` or `fallbackChannels` for hook mappings; existing fallback behavior is limited to `last`/session target resolution, tool-context selection, best-effort handling, or cron failure notifications, not an alternate-channel retry chain after primary hook delivery fails.

## Best Possible Solution

Keep #7281 open until maintainers either implement the hook delivery fallback feature or make an explicit product decision not to. A proper implementation would add typed `fallbackChannel`/`fallbackChannels` config and schema support, propagate it through hook mapping resolution and cron delivery dispatch, retry alternate channels only after primary delivery failure, document the behavior at https://docs.openclaw.ai/gateway/configuration-reference, and add focused tests distinguishing this from existing `last`, best-effort, and failure-notification behavior.
