## Summary

Keep open. Current main still has the reported failure mode for non-empty or true poll-looking fields on `action="send"`: the shared message schema still exposes poll creation fields, the send path still throws before dispatch when those fields are present, and Feishu still advertises no poll support. The only implemented mitigation I found is narrower: zero/empty poll defaults no longer trigger the guard. Open PR #46455 remains an unmerged implementation candidate with closing syntax, so this issue should not be closed before that work is reviewed, landed, or replaced.

## Best Possible Solution

Keep the issue open and review, land, or replace PR #46455. The durable fix should either sanitize accidental poll creation params before the send-path guard or narrow the LLM-facing schema so poll creation fields are only visible for poll-capable actions/channels, while preserving `action="poll"` behavior and adding regression coverage for Feishu and Telegram send-with-attachment calls that include auto-populated poll fields.
