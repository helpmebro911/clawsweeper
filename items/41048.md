## Summary

Keep this PR open. Current main still has the reported Telegram null-media path: getFile failures return null, the single-message handler only sends user-facing warnings for thrown media errors, and the inbound body builder can still emit media placeholders when no media was attached. The PR is a plausible focused fix, but it needs to be rebased from the old src/telegram paths to extensions/telegram and should address the review concerns around warning wording, rate limiting, and media-group all-fail behavior before landing.

## Best Possible Solution

Keep this PR open as a valid implementation candidate, but rework it on current main under extensions/telegram. The best landing path is to add a tagged or otherwise distinguishable media-resolution failure result, suppress or clearly mark unavailable-media placeholders when allMedia is empty, surface a bounded user warning for single messages and all-failed media groups, and add focused regression tests for video/photo null getFile paths plus normal successful attachment behavior.
