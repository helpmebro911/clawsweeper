## Summary

Keep open. Current main no longer fully strips Slack files: it parses Slack file fields, attempts downloads, stages media paths, and adds filename/fileId placeholders. But it still does not deliver the requested structured file metadata to agents. The code drops MIME type and size from the placeholder path, `SlackMediaResult` only carries path/contentType/placeholder, and the finalized inbound context only exposes `BodyForAgent`, `RawBody`, and legacy media fields such as `MediaPath(s)`/`MediaType(s)`. The existing issue comment correctly reframes this as a remaining shared inbound attachment metadata contract gap. The related open PR #62792 is about Slack media fetch/hydration reliability, not the requested metadata envelope.

## Best Possible Solution

Keep this issue open. Implement an additive inbound attachment metadata contract in `MsgContext`/plugin SDK, populate it from Slack `message.files` while preserving existing `BodyForAgent` placeholders and `MediaPath(s)` behavior, document the new field, and add Slack plus shared channel contract tests. Review #62792 separately for the narrower Slack media hydration bug.

## Risks / Open Questions

- A Slack-only metadata field would create contract drift; the durable fix should be an additive shared inbound attachment schema or structured context contract usable by Slack, Discord, Telegram, and third-party channel plugins.
- Agent-visible metadata must remain untrusted user/provider content. Avoid exposing Slack private download URLs or tokens directly; prefer safe provider IDs, display name, MIME type, size, local path when downloaded, and download status.
