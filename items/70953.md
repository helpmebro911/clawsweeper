## Summary

Keep #70953 open. Current main still implements exec approvals as an asynchronous approval-pending flow: gateway and node exec paths register the approval, start a detached wait/execute follow-up, and return `status: "approval-pending"` to the agent immediately. The docs also explicitly describe that behavior. Related items in the provided context are adjacent approval UX/policy issues, not a canonical duplicate or a shipped fix for pausing the agent turn.

## Best Possible Solution

Keep the issue open and implement this as an exec approval flow change: foreground gateway/node exec should await `exec.approval.waitDecision` with timeout and abort handling before returning control to the model, then execute or deny inline. Explicit background mode can keep the async follow-up behavior. Update `docs/tools/exec.md`, gateway/node exec tests, and runner tests so multiple simultaneous foreground approval requests are prevented.

## Risks / Open Questions

- Changing this behavior would affect the documented async follow-up model for gateway/node exec approvals and must preserve explicit background execution semantics.
- The runner currently suppresses visible assistant text after deterministic approval prompt delivery, but that is not the same as awaiting the approval before continuing tool/model execution.
