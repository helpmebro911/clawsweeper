## Summary

Keep open. Current main still does not provide the requested direct single-job cron read path. The existing CLI single-job display is still implemented by paging `cron.list` and filtering locally, while the gateway method registry, protocol schemas/validators, method scopes, server handlers, and agent cron tool do not expose `cron.get` or a `get` action. The related implementation PR #67677 is closed unmerged, so it is useful prior art but not shipped behavior.

## Best Possible Solution

Keep this issue open. The best product path is to implement the additive read API in core: add `cron.get` to the gateway method list, read scopes, protocol schema/types/validators, and `src/gateway/server-methods/cron.ts`, backed by `CronService.getJob(id)` with a clean not-found error. Then update the CLI to provide `openclaw cron get <id>` or make `cron show` use the direct RPC, add an agent cron `get` action if agent workflows remain in scope, and cover CLI/protocol/server behavior with focused tests. #67677 should be treated as prior art, not shipped behavior.
