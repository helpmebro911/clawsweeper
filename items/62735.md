## Summary

Keep open. Current main documents and implements custom Ollama base URLs through the canonical `ollama` provider, but the issue asks for arbitrary custom provider ids using `api: "ollama"` and/or explicit `injectNumCtxForOpenAICompat` override. Source inspection shows those requested paths are still constrained by provider ownership and Ollama-specific heuristics.

## Evidence

- **Native Ollama stream registration is provider-owned:** `registerProviderStreamForModel` asks `resolveProviderStreamFn` for the model's provider id, then registers the model API only when a provider-owned stream function is found. A custom provider id such as `toolbridge` therefore needs a matching provider runtime plugin before `api: "ollama"` gets registered.
  - file: [src/agents/provider-stream.ts:15](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/agents/provider-stream.ts#L15)
  - command: `nl -ba src/agents/provider-stream.ts | sed -n '1,70p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Provider stream lookup requires runtime provider match:** `resolveProviderStreamFn` calls `resolveProviderRuntimePlugin(params)?.createStreamFn`, and `resolveProviderRuntimePlugin` filters loaded provider plugins by `matchesProviderId(plugin, params.provider)`. The Ollama plugin is not automatically treated as owner for arbitrary provider ids.
  - file: [src/plugins/provider-runtime.ts:530](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/plugins/provider-runtime.ts#L530)
  - command: `nl -ba src/plugins/provider-runtime.ts | sed -n '528,542p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Ollama manifest owns only canonical provider id:** The bundled Ollama plugin manifest declares `providers: ["ollama"]`; it does not declare arbitrary custom provider aliases or activation for all providers with `api: "ollama"`.
  - file: [extensions/ollama/openclaw.plugin.json:1](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/extensions/ollama/openclaw.plugin.json#L1)
  - command: `nl -ba extensions/ollama/openclaw.plugin.json | sed -n '1,20p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Explicit num_ctx flag is still gated by Ollama heuristics:** `shouldInjectOllamaCompatNumCtx` returns false before reading `injectNumCtxForOpenAICompat` unless `isOllamaCompatProvider` accepts the provider/baseUrl. That helper still requires provider id `ollama`, localhost port `11434`, or an id containing `ollama` on port `11434` with `/` or `/v1`.
  - file: [extensions/ollama/src/stream.ts:83](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/extensions/ollama/src/stream.ts#L83)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '80,150p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Docs cover canonical Ollama provider custom base URL, not arbitrary provider ids:** The public Ollama docs show custom base URL configuration under `models.providers.ollama` with `api: "ollama"`, which is a useful workaround for replacing the canonical Ollama endpoint but does not implement the issue's arbitrary custom-provider request.
  - file: [docs/providers/ollama.md:280](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/docs/providers/ollama.md#L280)
