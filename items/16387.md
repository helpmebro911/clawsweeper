## Summary

Keep #16387 open. Current main still treats configured model-provider headers as static config values, and the shipped Fireworks plugin does not add Fireworks `x-session-affinity` or provide a generic `{{session.key}}` / `{{agent.id}}` / `{{run.id}}` header-template path for custom providers. The newer provider-owned transport hook can attach per-turn headers for provider plugins, but it is not a dynamic resolver for arbitrary `models.providers.*.headers` or `models.providers.*.request.headers` config, so the Fireworks prompt-cache and LiteLLM/Langfuse use cases from the issue discussion remain real remaining work.

## Best Possible Solution

Keep the issue open. The best maintainer path is a narrow, documented dynamic header resolver for `models.providers.*.headers` and `models.providers.*.request.headers`, evaluated at request dispatch with explicit supported placeholders for session key, agent identity, and per-turn/run identity. Add regression coverage for Fireworks `x-session-affinity`, LiteLLM/Langfuse-style headers, main turns, compaction, retries, and OpenAI-compatible transports; if maintainers prefer provider-owned behavior only, document that product decision and the plugin/API path that replaces custom-provider config templating.
