## Summary

Keep open. Current `main` still reports package installs via `detectPackageManager()`, which reads the checkout/package metadata rather than the actual global installer, so the reported bug is still present. The proposed PR also adds a narrower Bun-path heuristic than the existing global-install detection used elsewhere, so there is not strong evidence to close this as already fixed.

## Evidence

- **Current status path still uses repo package-manager detection for package installs:** `checkUpdateStatus()` resolves `pm` by calling `detectPackageManager(root)` before it decides `installKind`, then returns that value for both git and package installs.
  - file: [src/infra/update-check.ts:395](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/infra/update-check.ts#L395)

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **The detector still reads package.json.packageManager and lockfiles:** `detectPackageManager()` prefers `package.json.packageManager`, then falls back to repo lockfiles. That matches the PR's reported root cause for globally installed packages.
  - file: [src/infra/detect-package-manager.ts:6](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/infra/detect-package-manager.ts#L6)

## Risks / Open Questions

- If this PR is closed now, the reported `update status` misdetection remains on `main` for packaged installs.
- If this PR is merged as-is, Bun installs outside `~/.bun` (for example custom `BUN_INSTALL`) can still be misclassified, and OpenClaw will keep two separate global-manager detection paths.
