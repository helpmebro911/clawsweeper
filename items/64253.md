## Summary

Keep this issue open. Current main still has the reported direct-first subagent completion announce path: completion delivery calls the Gateway `agent` RPC with the 120000 ms announce timeout, treats `gateway timeout` as transient, and test coverage preserves four direct retries before giving up. Related reports #47975 and #71699 are adjacent and open, but neither is a merged canonical fix or a true duplicate for this Windows alive-but-unresponsive completion-timeout case.

## Best Possible Solution

Keep this issue open as the focused Windows alive-but-unresponsive Gateway/subagent completion-timeout report. The maintainer path is to reproduce or add a stress regression for concurrent subagent completion announcements and saturated Gateway `agent` calls, then decide whether completion delivery should fail over faster, avoid direct parent-agent RPCs under saturation, isolate announce delivery from busy lanes, add an unresponsive-Gateway watchdog path for Windows service installs, or address provider/lane starvation. Cross-link #47975 and #71699 as related but distinct.
