## Summary

Keep #71545 open. Current main supports `streamable-http` MCP servers, but the session runtime still caches one MCP client/transport and forwards `callTool` directly with no lost-session detection, reconnect, catalog refresh, or retry path. The PR is a focused, recent core-runtime bug fix with maintainer follow-up on the head branch, and the related stale-session reports do not supersede this streamable-http path.

## Best Possible Solution

Keep the PR open and continue maintainer review. The best path is to land or refine a core `createSessionMcpRuntime` fix that, for `streamable-http` only, detects lost session errors from `callTool`, rebuilds the MCP transport/client via the existing resolver, refreshes SDK tool metadata/catalog state as needed, retries once, preserves validation behavior, and has regression tests covering successful reconnect, failed reconnect retryability, and concurrency/false-positive boundaries.
