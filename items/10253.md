## Summary

Keep open. Current main still lacks a hook/webhook session lifecycle policy. The newer cron session retention only prunes canonical cron run aliases, while webhook dispatch still runs isolated hook sessions under `hook:` keys and persists them without hook-specific cleanup, archive timing, or transcript-retention controls.

## Best Possible Solution

Keep this issue open. The likely product fix belongs in core hook session lifecycle: add a maintainer-approved `hooks` cleanup/retention policy, validate and document it, wire it through hook dispatch and isolated cron finalization, define transcript retention/archive semantics, and add focused tests for Gmail/per-hook keys, delayed cleanup, session-list visibility/status, and interaction with existing cron/session maintenance.
