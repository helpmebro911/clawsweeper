## Summary

Keep #10253 open. Current main still has no hook/webhook-specific lifecycle policy for on-complete archival, delayed cleanup, deletion, or transcript-retention control. Generic `session.maintenance` can age/count-prune ordinary session entries, and `cron.sessionRetention` prunes only canonical cron run-session aliases, but hook sessions such as `agent:main:hook:gmail:*` are still dispatched as isolated cron-style runs and persisted as normal session rows without a hook cleanup contract. The prior Codex review comment on the issue also kept this open, and the adjacent related items in the provided context do not supersede this request.

## Best Possible Solution

Keep #10253 open. The best maintainer path is a core hook-session lifecycle policy under `hooks` with config types, strict schema/help/docs, and runtime wiring through direct `/hooks/agent`, mapped hooks, Gmail preset handling, and isolated-run finalization. It should define archive/delete/no-op modes, delayed cleanup, transcript retention, sessions_list visibility, interaction with generic session maintenance and cron retention, and focused tests for `agent:<id>:hook:gmail:*` keys.

## Risks / Open Questions

- A fix needs explicit precedence with `session.maintenance` and `cron.sessionRetention` so hook cleanup does not unexpectedly delete non-hook sessions or audit transcripts.
- Generic session pruning can reduce storage growth eventually, but treating it as a full fix would miss the requested hook-specific completion semantics and UI clutter behavior.
