## Summary

Keep this issue open. Current main still validates known plugin entry config against plugin schemas as a hard config error, skips last-known-good recovery when the invalidity is scoped to plugins.entries.*, and then throws during gateway startup with the doctor --fix hint. Doctor repair covers stale unknown plugin references and registry state, but not quarantine/disablement or deterministic repair for a known third-party plugin whose config is invalid. Related #70371 and open PR #70394 are useful overlap, but they are not merged and do not cover this issue's full doctor/safe-mode recovery scope on current main.

## Best Possible Solution

Keep this issue open and fix it in core. Add a narrow startup degradation path that disables or quarantines only the bad third-party plugin entry while keeping the gateway bootable, surface the failed plugin and schema path in plugin/status/doctor diagnostics, and extend openclaw doctor --fix or a safe-mode/plugin-quarantine command to repair plugins.entries, plugins.allow, and related plugins.load.paths deterministically. Coordinate with #70371/#70394, but do not close this issue until the startup degradation plus doctor/safe-mode recovery behavior is merged with regression tests.
