## Summary

Keep this issue open. Current main still treats plugin-local config/schema failures as invalid config, skips whole-file last-known-good recovery for that class, and then throws during gateway startup. Doctor repair still only removes stale unknown plugin IDs and bundled load-path aliases; it does not quarantine or disable a known third-party plugin with bad config/schema/runtime state.

## Best Possible Solution

Keep this issue open and fix it in core. Add a narrow plugin-local recovery path that keeps the gateway bootable by quarantining or disabling only the bad third-party plugin entry, while preserving the current protection against rolling back unrelated user config. Extend `openclaw doctor --fix` or a plugin quarantine command to deterministically repair or quarantine `plugins.entries`, `plugins.allow`, and related `plugins.load.paths`, with focused startup, validation, doctor, and plugin-registry tests. Coordinate with #70371/#70394 if that work lands first, but do not close this issue until the doctor/safe-mode recovery path is covered.
