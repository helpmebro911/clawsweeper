## Summary

Keep this issue open. Current main still validates configured plugin entries as fatal config issues, explicitly skips last-known-good recovery for plugin-local invalidity, and then rejects gateway startup. Doctor repair only removes stale unknown plugin IDs and redundant bundled load paths; it does not quarantine, disable, or repair a known third-party plugin with invalid config.

## Best Possible Solution

Keep this open and fix it in core. Add a narrow startup degradation path for bad third-party plugin entries, surface the failing plugin and config path in diagnostics/status/doctor, and extend doctor --fix or a safe-mode/plugin-quarantine command to repair plugins.entries, plugins.allow, and relevant plugins.load.paths deterministically. Coordinate with #70371/#70394, but close only after startup degradation plus doctor/safe-mode recovery behavior lands with regression tests.
