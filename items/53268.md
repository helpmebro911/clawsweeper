## Summary

Keep open. Current main still lacks a user-configurable URL or hostname deny list for `web_fetch`; the config types, strict runtime schema, generated config schema, labels/help, docs, and runtime fetch plumbing expose only existing fetch limits plus a narrow `ssrfPolicy.allowRfc2544BenchmarkRange` override. The existing SSRF guard has hardcoded internal-host blocking and allowlist-only policy, which does not satisfy blocking a few unwanted public domains while allowing the rest.

## Best Possible Solution

Keep this issue open and implement a core `web_fetch` hostname deny-list surface if maintainers want the behavior. A solid path is to add a typed config key such as `tools.web.fetch.denyHosts`, validate it in the strict config and generated schemas, add labels/help/docs, and enforce exact plus `*.domain` matching in the guarded fetch path before DNS and again on redirects. Focused tests should cover config schema rejection/acceptance, SSRF/fetch guard behavior, redirect handling, and `web_fetch` execution errors.
