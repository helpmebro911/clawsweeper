## Summary

Keep this PR open. Current main still routes Signal inbound text into raw model-facing fields, and the submitted patch wraps only legacy `formatInboundEnvelope(...)` body strings. The security hardening goal is relevant, but the PR needs rework before merge.

## Best Possible Solution

Keep this PR open, but require a rework or replacement before merge. The fix should protect the actual Signal model-facing path by using wrapped untrusted user text in `BodyForAgent` or structured `UntrustedContext`, while preserving raw `CommandBody`/`BodyForCommands` for command parsing. It should wrap pending history once or use stable marker IDs, keep generated message IDs outside untrusted blocks, and add focused Signal coverage for direct messages, group messages, debounced bodies, quote-only context, and pending group history.

## Risks / Open Questions

- Merging as-is would create a false sense of Signal prompt-injection protection while raw `BodyForAgent` still feeds the normal model prompt path.
- The history change in the submitted patch would call `wrapExternalContent` during prompt rendering, creating fresh random marker IDs for unchanged history and undermining deterministic prompt bytes.
- The submitted history wrapping places generated `[id:...]` metadata inside the untrusted block instead of keeping trusted application metadata outside it.
- No new third-party dependency, workflow, permission, lockfile, package-publishing, or artifact-download surface is evident from the provided PR diff; the security concern is functional correctness of the mitigation, not supply-chain expansion.
