## Summary

Keep this PR open. Current main still routes Signal inbound message text into raw `BodyForAgent`/`RawBody`, so the security hardening topic is not implemented. The submitted patch is also not merge-ready because it wraps only legacy `formatInboundEnvelope(...)` body strings while the active reply path prefers `BodyForAgent` and turns that into `BodyStripped` for the model prompt.

## Best Possible Solution

Keep this PR open, but require a rework or replacement before merge. The fix should protect the actual Signal model-facing path, likely by deriving `BodyForAgent` from wrapped untrusted user text or using `UntrustedContext`/structured context consistently, while preserving `CommandBody`/`BodyForCommands` for command parsing. It should wrap pending group-history user text once or use stable marker IDs, keep generated message IDs outside untrusted blocks, and add focused Signal tests for direct messages, group messages, debounced bodies, quote-only context, and pending group history.

## Risks / Open Questions

- Merging the PR as submitted would create a false sense of Signal prompt-injection protection while the normal model prompt still comes from raw `BodyForAgent`.
- Wrapping pending group history inside the render-time `formatEntry` callback would call `wrapExternalContent` repeatedly; that helper generates a fresh random marker ID per call, making stable history bytes non-deterministic.
- The submitted history change places generated `[id:...]` metadata inside the untrusted block, which blurs trusted application metadata with attacker-controlled text.
- A correct fix needs to keep command parsing on deliberate raw command fields while marking only model-facing user content or structured supplemental context as untrusted.
