## Summary

Keep open. Current main still does not implement the requested prompt transition when visible assistant content has stopped. It has the merged watchdog mitigation from #67401, but the TUI still shows `streaming` from chat deltas until a terminal chat/lifecycle event arrives or the 30s watchdog fires. The gateway/protocol still has no content-complete or finalizing state, and agent lifecycle terminal emission can still wait on post-response flush work.

## Best Possible Solution

Keep this issue open until OpenClaw decouples the TUI's visible `streaming` status from full lifecycle/chat-final completion. The best fix is either an additive content-complete/finalizing signal emitted when model-visible content is done, or TUI logic that moves from `streaming` to a distinct non-streaming finalizing/idle state after visible deltas stop while still tracking the active run. Add regression coverage for delayed lifecycle finals, concurrent active runs, tool-only or hidden deltas, and the watchdog fallback.

## Risks / Open Questions

- The shipped watchdog reduces the worst stuck-status case, so a final fix should preserve it as recovery while avoiding misleading immediate idle state during legitimate tool, delivery, or persistence work.
- A purely TUI-side silence heuristic could regress concurrent runs or hidden/tool-only output; a backend content-complete or finalizing signal looks cleaner if it can be added compatibly.
