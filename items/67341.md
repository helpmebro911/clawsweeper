## Summary

Keep this PR open. Current main still lacks the requested `amazon-bedrock-mantle` `config.discovery.enabled` opt-out, and the linked #67288 behavior is still supported by the source path: Mantle catalog discovery runs without reading plugin config and can fall through to IAM bearer-token generation. The PR is a focused implementation candidate, but it needs updating before merge to match the current runtime config pattern and to add tests/docs.

## Best Possible Solution

Keep this PR open as an implementation candidate for #67288. Update it against current main to add Mantle manifest schema/uiHints and public docs, resolve current plugin config using the established provider pattern, pass config into `resolveImplicitMantleProvider`, gate before IAM bearer-token generation, and add regression tests covering `discovery.enabled: false`, `true`, and unset/default behavior. Then land this PR or an equivalent replacement.

## Risks / Open Questions

- The PR currently gates only the catalog handler; passing plugin config into `resolveImplicitMantleProvider` would protect direct resolver callers and make the behavior easier to test.
- The PR captures `api.pluginConfig` at registration time, while the current sibling provider resolves live plugin config from `ctx.config` for runtime behavior.
- The PR has no regression tests for disabled, enabled, and unset/default discovery states, and no docs update for the new user-facing config path.
