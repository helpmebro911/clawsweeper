## Summary

Current `main` still resolves exec approvals from `OPENCLAW_HOME`/home-relative `~/.openclaw`, not from `OPENCLAW_STATE_DIR`, so the reported split-state bug is still present rather than implemented or obsolete.

## Evidence

- **Exec approvals path is still hardcoded to home-relative defaults:** `DEFAULT_FILE` and `DEFAULT_SOCKET` remain `~/.openclaw/...`, and `resolveExecApprovalsPath()` / `resolveExecApprovalsSocketPath()` only call `expandHomePrefix(...)`.
  - file: [src/infra/exec-approvals.ts:173](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/infra/exec-approvals.ts#L173)
  - command: `nl -ba src/infra/exec-approvals.ts | sed -n '166,189p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Home expansion does not consult OPENCLAW_STATE_DIR:** `resolveEffectiveHomeDir()` and `resolveRawHomeDir()` only consider `OPENCLAW_HOME`, `HOME`, `USERPROFILE`, and `os.homedir()`. There is no state-dir fallback in the home-resolution path used by exec approvals.
  - file: [src/infra/home-dir.ts:16](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/infra/home-dir.ts#L16)
  - command: `nl -ba src/infra/home-dir.ts | sed -n '16,46p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **State-dir semantics are implemented separately elsewhere:** `resolveStateDir()` explicitly honors `OPENCLAW_STATE_DIR`, which confirms exec approvals are not following the repo’s normal state-root resolution path.
  - file: [src/config/paths.ts:60](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/config/paths.ts#L60)
  - command: `nl -ba src/config/paths.ts | sed -n '55,68p'`
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **CLI and gateway both route through the same home-based approvals helpers:** The local CLI path reads approvals via `readExecApprovalsSnapshot()`, and the gateway RPC handlers call `ensureExecApprovals()` plus `readExecApprovalsSnapshot()`. Those helpers ultimately use `resolveExecApprovalsPath()`, so both surfaces inherit the same path behavior.
  - file: [src/cli/exec-approvals-cli.ts:85](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/cli/exec-approvals-cli.ts#L85)

- **Tests still lock in OPENCLAW_HOME-based behavior:** The store helper test asserts that the default approvals file/socket resolve under `OPENCLAW_HOME/.openclaw/...`, which matches the reported bug rather than a fix to state-dir semantics.
  - file: [src/infra/exec-approvals-store.test.ts:83](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/infra/exec-approvals-store.test.ts#L83)
