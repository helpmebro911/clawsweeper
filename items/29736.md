## Summary

Keep #29736 open. Current main still has the TypeScript exec approvals store defaulting file and socket resolution to home-relative `~/.openclaw`, while `OPENCLAW_STATE_DIR` and `--profile` state-root machinery exists separately. The macOS companion app now has a state-dir-aware approvals store, but the Linux/headless gateway, CLI, and node-host paths implicated by the issue still use the home-backed TypeScript helpers.

## Best Possible Solution

Keep the issue open and fix the core TypeScript exec approvals path contract. Default approvals file and socket resolution should derive from the active state root/profile semantics, likely through `resolveStateDir()`, for CLI, gateway, and headless node-host paths, while preserving a compatible migration/fallback for existing home-backed approvals. Update docs and regression tests for `OPENCLAW_STATE_DIR`, `--profile`, container writable-state deployments, `OPENCLAW_HOME`, and symlink safety.

## Risks / Open Questions

- Changing the default TS approvals path needs compatibility or migration handling for existing `~/.openclaw/exec-approvals.json` files and socket paths.
- The existing symlink and directory-safety behavior in the TS store must be preserved if approvals move under the active state root.
- Swift and TypeScript approvals stores now appear to have different path semantics, so the fix should align cross-platform behavior deliberately.
