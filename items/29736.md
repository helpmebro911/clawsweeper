## Summary

Keep #29736 open. Current main still resolves the TypeScript exec approvals file and socket from home-relative ~/.openclaw constants, while OPENCLAW_STATE_DIR/profile state-root handling exists separately. CLI, gateway, and headless node-host management still call those TypeScript helpers, so the Linux/headless path reported here remains unresolved even though the macOS app store is state-dir-aware.

## Best Possible Solution

Keep the issue open and fix the core TypeScript exec approvals path contract. Default approvals file and socket resolution should derive from the active state root/profile semantics, likely via resolveStateDir(), for CLI, gateway, runtime enforcement, and headless node-host paths. Add migration/fallback behavior for existing home-backed approvals, preserve symlink/destination safety, and update regression tests/docs for OPENCLAW_STATE_DIR, --profile, container writable-state deployments, and OPENCLAW_HOME compatibility.

- The prior comment that it works in 2026.4.11 appears consistent with the macOS app path, but not with the Linux/headless TypeScript paths implicated by this issue.
