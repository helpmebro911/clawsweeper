## Summary

Current main still has the memory-lancedb behavior this PR targets: the visible `memory_forget` candidate list truncates IDs to 8 characters while direct deletion validates a full UUID. The PR is not obsolete, but it needs maintainer judgment on the security/product tradeoff and should drop or justify unrelated generated baseline churn before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Maintainers should either land the focused memory-lancedb fix with its regression test, or revise it to use non-authoritative handles or explicit confirmation for candidate deletion; before merge, remove or justify the unrelated generated docs baseline change.
