## Summary

Keep #70515 open. Current main still has the reported async and sync suspicious-config restore behavior: failed backup copy errors are swallowed, the warning always says the backup was restored, and the observe audit record is always written with `valid: true`. The PR is a focused bug fix with matching async/sync tests in the provided PR context, and no related item supersedes it.

## Best Possible Solution

Keep #70515 open and either merge it after the normal maintainer review/rebase and changed-lane validation, or apply the same focused fix on main: capture async and sync backup-copy failures, log a distinct restore-failure warning, write `valid: false` when the restore copy fails, add nullable restore error details to observe audit records, and keep the async/sync regression tests.
