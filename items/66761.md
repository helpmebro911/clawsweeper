## Summary

Current main still has the unguarded `provider?.models?.some(...)` call in provider policy lookup, so the linked HTTP chat-completions crash is still reachable when runtime provider data supplies `models` as a non-array value. This PR is a focused implementation candidate with matching regression coverage, and the provided discussion, Greptile review, and earlier Codex review all support the same narrow guard.

## Best Possible Solution

Keep this PR open for maintainer review and land the `Array.isArray(provider?.models)` guard with the included regression tests, or apply an equivalent main-branch fix that prevents non-array runtime provider model values from crashing provider policy lookup. Treat provider normalization as a follow-up, not a blocker for the narrow HTTP chat-completions regression fix.
