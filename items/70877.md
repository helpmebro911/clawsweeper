## Summary

Keep this PR open. Current main still defaults tool-loop detection off in the detector and production hook path, and the PR is a plausible but incomplete implementation candidate rather than something already shipped or superseded.

## Best Possible Solution

Keep this PR open for maintainer review or consolidation with a designated loop-detection PR. The best fix is a minimal core change that enables loop detection by default in the authoritative runtime path, preserves explicit opt-out and global-to-agent inheritance semantics, updates source zod/schema help/docs plus generated schema baselines, and adds focused tests for no-config default-on, explicit false, global false with partial agent overrides, production hook behavior, and UI/schema default consistency.

## Risks / Open Questions

- Closing now would discard a still-unfixed runtime bug: current main allows repeated identical tool calls to continue unless loop detection is explicitly enabled.
- Merging the current PR as-is appears likely to fail its own added direct detector test, because `detectToolCallLoop()` still defaults omitted config to disabled in the unchanged detector implementation.
- A runtime-only default can create schema/docs/UI drift: docs and generated metadata still say default false, and the config UI falls back to false when a boolean schema default is absent.
- Default-on loop detection can block legitimate repeated tool calls, so explicit `enabled: false`, global false plus partial agent override inheritance, and UI-visible defaults need focused coverage.
