## Summary

Keep PR #70877 open. Current main still ships tool-loop detection disabled by default, so the user-visible bug the PR targets remains real. The PR is a plausible focused fix with useful repro context, but the latest patch shown in the PR context is not ready to close or merge: it defaults only the higher-level resolver, leaves the detector's own no-config default disabled, and adds a no-config detector test that would contradict current detector behavior.

## Best Possible Solution

Keep the PR open and ask for a revision rather than cleanup-close it. The best fix is to enable loop detection by default at the actual runtime boundary consistently, preserve explicit global or agent disabled semantics during merge, update schema/help/generated config docs from the non-generated source so the UI does not display a false default, and add a passing regression test for the no-config default plus explicit enabled:false opt-out. Maintainers should also decide whether #70877, #56403, or another overlapping loop-detection PR is the canonical implementation before merging.
