## Summary

Current main still matches the reported behavior: manual `sessions_spawn(runtime="acp")` is exposed when ACP is enabled and a backend exists, but the initial spawned ACP turn is sent through the normal `agent` path, where `acp.dispatch.enabled=false` raises `ACP_DISPATCH_DISABLED` before `runTurn`. The related ACP reports cover transcript persistence, placement, timeouts, media dispatch, and action support; none supersede this manual-vs-automatic policy split.

## Best Possible Solution

Keep this issue open. Implement a focused ACP policy split: carry an explicit manual or parent-owned spawn context from `sessions_spawn` and `spawnAcpDirect` into the initial `agentCommand` ACP turn, bypass only `acp.dispatch.enabled` for that context, keep global ACP/backend/sandbox/allowlist gates intact, and keep `src/auto-reply/reply/dispatch-acp.ts` blocking automatic bound-thread follow-ups when dispatch is disabled. Add regression coverage for manual ACP spawn succeeding with dispatch disabled and automatic ACP dispatch still returning `ACP_DISPATCH_DISABLED`, then update the ACP docs at `https://docs.openclaw.ai/tools/acp-agents` and `https://docs.openclaw.ai/gateway/configuration-reference`.

## Risks / Open Questions

- Changing this without a narrow manual-spawn context could weaken an intentional global ACP execution gate. The safer fix should bypass only the dispatch-only gate for explicit manual or parent-owned spawn turns, while preserving `acp.enabled`, backend availability, sandbox, ownership, and `acp.allowedAgents` checks.
- Docs currently describe `acp.dispatch.enabled=false` as blocking execution, so a code fix needs matching docs and regression tests to avoid leaving conflicting operator guidance.
