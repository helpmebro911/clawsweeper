## Summary

Current main still matches the reported split-provider auth bug. Runtime auth, session auto-selection, and status labeling choose auth profiles from canonical provider auth order before considering `models.providers.<id>.apiKey`, and that provider `apiKey` field is documented and typed as a credential/SecretRef, not an auth-profile binding.

## Best Possible Solution

Keep this issue open. Maintainers should define an explicit split-provider auth contract, likely a dedicated provider-entry auth profile field or documented profile-reference semantics, then update runtime auth resolution, session auto-selection, status labels, docs, and focused regression tests for multiple provider entries sharing one upstream auth provider.

## Risks / Open Questions

- A narrow runtime-only fix would still leave session auto-pinning and `/status` labels inconsistent unless those surfaces use the same provider-entry-aware auth rule.
- Overloading `models.providers.*.apiKey` to sometimes mean an auth-profile id could conflict with the documented credential/SecretRef contract and could accidentally send a profile id as a bearer token.
