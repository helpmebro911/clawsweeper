## Summary

Keep open. Current main partially covers the PR by normalizing Telegram reply payloads before delivery, but the same-line reply-tag plus MEDIA case described in the PR/review is still not fixed: media extraction still runs before reply-tag stripping, and media extraction only recognizes lines whose trimmed start is MEDIA:. The PR should be refreshed for the current extensions/telegram layout and adjusted to avoid the whitespace regression noted in review.

## Best Possible Solution

Refresh the PR against current main, keep the existing Telegram delivery outbound-plan normalization, and fix parseReplyDirectives so reply tags are stripped before MEDIA extraction only when needed, preserving intentional whitespace for ordinary text. Add focused regression coverage for [[reply_to:42]] MEDIA:./photo.jpg in both the shared directive/outbound normalization path and Telegram delivery, plus any needed current-message threading coverage for direct Telegram/native-plugin reply paths.
