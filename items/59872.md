## Summary

Keep open. Current `main` still exposes `web_fetch` as a generic URL-fetch tool, has no `tools.web.fetch` domain routing/blocklist surface, and the runtime fetch path does not special-case or redirect `x.com`/`twitter.com`. Docs now point users toward `x_search` for exact X post URLs, but that guidance has not been turned into tool-level behavior, so the reported gap remains real.

## Evidence

- **Tool prompt still strongly advertises generic URL fetching:** The agent system prompt still summarizes `web_fetch` as `Fetch and extract readable content from a URL`, matching the issue's root-cause description.
  - file: [src/agents/system-prompt.ts:489](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/system-prompt.ts#L489)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Tool implementation description is still generic:** The runtime tool definition still describes `web_fetch` as lightweight page access without browser automation, with no warning or routing for X/Twitter domains.
  - file: [src/agents/tools/web-fetch.ts:625](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/tools/web-fetch.ts#L625)
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
