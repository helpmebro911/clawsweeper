## Summary

Keep this issue open. Current main still forces synchronous `sessions.json` disk rereads inside the session-store write lock for the main mutation helpers, and the exact watchdog warning path reported by the issue is still present. The related open PR #68554 is the most direct implementation candidate, while #71903 addresses a different lock-orphan hardening path and does not remove the active over-held lock symptom.

## Best Possible Solution

Keep this issue open and review/land #68554 or an equivalent maintainer-owned fix that removes redundant `skipCache` disk rereads while preserving session-store cache coherency and write serialization. Add focused regression or performance coverage where feasible, then verify startup/RPC responsiveness on current main with a large session store before closing. Treat #71903 as a separate related lock-hardening candidate, not as the closure path for this active over-held lock report.
