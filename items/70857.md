## Summary

Keep #70857 open. The report is recent, concrete, and not covered by protected maintainer ownership/labels. Current main still contains the reported `sessions.json.lock` store-lock path with a 10s acquisition timeout and 15s watchdog max hold, and the watchdog still force-releases long-held locks with the exact warning shape the reporter saw. Related #69674 is fixed only for Gateway probe responsiveness, and open PR #71903 addresses a related lock-reclamation/lost-history scenario rather than the Windows startup latency plus active `sessions.json.lock` hold reported here.

## Best Possible Solution

Keep the issue open and route it to a focused Windows/session-store-lock investigation: identify which `updateSessionStore` or `saveSessionStore` mutation holds `sessions.json.lock` for minutes during startup/reply handling, add diagnostics or startup trace attribution around the lock holder, then narrow the session-store critical section or raise/specialize the watchdog only if the long hold is intentional. Track #71903 separately for the inherited-PID lock-reclamation bug and keep #69674 closed as the Gateway probe fix.
