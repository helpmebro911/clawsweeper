## Summary

Keep #70857 open. Current main still contains the session-store lock behavior matching the reported `sessions.json.lock` warning: the store lock defaults to a 10s timeout, derives a 15s max hold, and the watchdog force-releases locks that exceed that max. Startup/doctor cleanup also only targets transcript `.jsonl.lock` files, not the reported `sessions.json.lock` path. The linked #71903 PR is still open and only partially overlaps the remaining lock-safety work.

## Best Possible Solution

Keep the issue open as a live Windows/session-store lock regression. The maintainer path should profile Windows startup and reply latency, then make `sessions.json.lock` handling safer: avoid watchdog-forced release of active store writes, narrow the store lock critical section where possible, decide whether doctor/startup should inspect `sessions.json.lock`, and coordinate with the narrower #71903 lock-safety PR and the broader #13744 lock-scope/configuration work.
