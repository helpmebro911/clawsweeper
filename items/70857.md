## Summary

Keep this issue open. Current main still has the reported session-store write path: both main mutation helpers reread `sessions.json` with `skipCache: true` while holding `withSessionStoreLock`, which bypasses the object/serialized cache and falls through to synchronous disk read and JSON parse. The lock watchdog warning signature from the report is still present. The provided discussion also has a fresh macOS corroboration and identifies open PR #68554 as the direct fix candidate, while #71903 and stale-lock cleanup are adjacent hardening rather than a closure path for this active long-held lock report.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes redundant `skipCache` disk rereads from `updateSessionStore` and `updateSessionStoreEntry` while preserving serialized writes and cache coherency. Add a focused regression or performance guardrail where feasible, then verify startup/RPC responsiveness with a large session store and cron/channel activity before closing. Track #71903 separately as adjacent lock hardening.
