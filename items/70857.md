## Summary

Keep #70857 open. The report is recent, concrete, and still maps to current core agent/session-lock code; current main has stale-lock cleanup and watchdog behavior, but it still holds the embedded runner session write lock across the run and releases it only during final cleanup. The related #71903 PR is open and related, not merged or sufficient to prove this Windows startup/reply-latency issue is fixed.

## Best Possible Solution

Keep #70857 open as an actionable core regression until maintainers either fix the Windows startup/reply-latency path or deliberately consolidate it under a canonical Windows/session-lock tracker. #71903 may be relevant as a partial session-lock fix, but it should not close this issue unless it lands and there is evidence that it resolves the Windows active-lock hold and startup latency symptoms.
