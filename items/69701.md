## Summary

Keep PR #69701 open. Current main still resolves loopback gateway binds to both `127.0.0.1` and `::1` when IPv6 is bindable, and the gateway startup path still listens on every resolved bind host. The later main commit that mentions #69674 addresses probe route ordering, not the Windows loopback dual-bind behavior this PR changes.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either merge/adapt the Windows-specific `resolveGatewayListenHosts` guard with targeted regression coverage and, if feasible, live Windows confirmation, or explicitly supersede the PR only after proving the current probe-ordering fix eliminates the reported Windows hang and the dual-bind diagnosis is invalid.
