## Summary

Keep this issue open. Current main still does not carry Discord's native thread parent channel id into ordinary reply `modelByChannel` resolution when `channels.discord.thread.inheritParent` is false, so a thread with no explicit override can still miss the parent channel override. The related closed #57960 work covers parent-channel route/binding inheritance, and older #30760 was closed stale rather than shipped as this model-selection fix.

## Best Possible Solution

Implement a narrow Discord model-routing fix: preserve thread-specific `modelByChannel` precedence, then use the resolved Discord native parent channel id as a fallback candidate for ordinary thread replies independently of `channels.discord.thread.inheritParent`. Add regression coverage for a Discord thread with no thread override but a parent channel override, plus a thread-specific override-wins case.

## Risks / Open Questions

- When `channels.discord.thread.inheritParent` is explicitly true, the existing `ParentSessionKey` path may provide a parent override fallback, but that setting also changes transcript/session inheritance semantics and does not satisfy the requested default Discord thread model routing behavior.
- The auto-thread title generation path already has a parent override fallback; a fix should avoid treating that as sufficient proof for ordinary reply model selection.
