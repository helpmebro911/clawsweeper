## Summary

Keep #70451 open. Current main has source and test evidence that the broad `status`/`health` startup path no longer preloads plugins, which likely addresses part of the related #64306 failure class. But #70451 specifically reports `openclaw hooks enable/list`, and current hooks CLI still builds its report through `buildPluginDiagnosticsReport`, which loads plugin modules (`loadModules: true`) via `loadOpenClawPlugins`. That leaves the reported Windows/Node 24 hook-command hang insufficiently proven fixed or superseded.

## Best Possible Solution

Keep #70451 open and narrow it to the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` use a config/workspace/manifest-only plugin hook snapshot instead of `buildPluginDiagnosticsReport` module loading, or otherwise bound and diagnose that load on Windows. Add regression coverage that hooks CLI actions do not invoke full plugin module loading for ordinary hook visibility/toggle operations, and update the public hooks docs to mention the config-only fallback clearly.
