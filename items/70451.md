## Summary

Current main still does not prove the Windows hooks timeout is fixed. The broader CLI bootstrap preload class from #64306 appears mitigated for commands like status/health through command-path policy, but the reported `openclaw hooks list/enable` path still builds its report through plugin diagnostics with module loading enabled, so hooks commands can still exercise the risky plugin module-loading path.

## Best Possible Solution

Keep this issue open and narrow it to the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` derive plugin-managed hook visibility from a manifest/config/workspace snapshot, or otherwise bound and diagnose plugin diagnostics loading on Windows, with regression coverage proving hook visibility and toggles do not import full plugin runtime modules unnecessarily while preserving plugin-managed hook visibility documented in the hooks docs.
