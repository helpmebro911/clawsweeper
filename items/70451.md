## Summary

Keep open. Current main fixes the broader #64306 bootstrap path for status/health-style commands, but this issue is specifically about Windows `openclaw hooks list/enable`, and those hook actions still build hook state through full plugin diagnostics that can load plugin runtime modules before listing or enabling hooks.

## Best Possible Solution

Keep #70451 open. The likely product/code fix is to make `openclaw hooks list/info/check/enable/disable` avoid full runtime plugin diagnostics for ordinary hook management, using persisted plugin metadata or a lightweight snapshot for plugin-managed hooks and workspace/bundled hook discovery for local toggles. Add regression coverage that hooks list/enable do not call `loadOpenClawPlugins()` for config/workspace-only paths, plus a Windows smoke when feasible.
