## Summary

Current main still leaves the hooks CLI on the full plugin diagnostics path, so the Windows/Node plugin module-loading hang described here remains plausible for `openclaw hooks list/info/check/enable/disable`. The closed broader startup issue #64306 does not supersede this hooks-specific path.

## Best Possible Solution

Keep this open and fix the hooks CLI path directly. `openclaw hooks list/info/check/enable/disable` should get plugin-managed hook visibility from manifest/config/workspace snapshot data or a bounded hook-specific metadata API, without importing full plugin runtime modules. Add regression coverage proving the hooks commands avoid the `loadModules: true` diagnostics path while still showing plugin-managed hooks as documented.
