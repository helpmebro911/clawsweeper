## Summary

Keep this issue open. Current main has useful partial mitigations, but the reported default crash-restart/OOM loop is still not fully addressed: the active byte guard is opt-in, repair/recovery/run paths can still materialize the full transcript before any quarantine or bounded successor path, and there is still no count-based circuit breaker for accumulated full-bootstrap markers.

## Best Possible Solution

Keep this issue open and implement a core pre-open transcript recovery guard. The guard should stat/classify active JSONL files before repair, `readSessionMessages`, or `SessionManager.open`; rotate, quarantine, or compact pathological transcripts into a bounded successor/degraded session; cap repeated `openclaw:bootstrap-context:full` markers; reuse media offload/dedup where applicable; and add focused regression tests for restart loops and marker accumulation.

## Risks / Open Questions

- Default configurations remain exposed because `maxActiveTranscriptBytes` is disabled unless users opt in and also enable transcript rotation.
- Even configured byte-trigger compaction can still fail on an already pathological transcript because the compaction path opens and repairs the same full active file before it can rotate to a successor.
- Repeated full-bootstrap completion markers are detected only opportunistically by a bounded tail scan; there is no count-based degraded-session breaker for marker accumulation.
