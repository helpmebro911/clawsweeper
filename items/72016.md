## Summary

Current main has some plugin-adjacent doctor support, but it is narrower than this request. The shipped surfaces cover core hard-coded doctor contributions, channel-scoped doctor adapters, and plugin config compatibility migrations. I did not find a generic supported API for an external plugin to add arbitrary environment-specific `openclaw doctor` checks and `--fix` repairs that survive npm upgrades. The related #68348 report is only about optimizing doctor/status contract loading and does not supersede this feature request.

## Best Possible Solution

Keep this open for maintainer/product design. The best path is a small, documented doctor contribution API for trusted code plugins, e.g. a `registerDoctorCheck`/manifest-declared doctor contribution with stable ids, read-only check output, optional repair gated by `--fix`, deterministic ordering, non-interactive policy, and tests/docs. In parallel, consider a built-in Telegram/channel doctor check for multi-instance allowlist/pairing drift if that specific failure is common enough.

## Risks / Open Questions

- The existing channel-scoped doctor adapter may solve a narrower Telegram/channel repair if implemented in the Telegram plugin itself, so maintainers should decide whether the first step is a Telegram-specific built-in check or a generic plugin API.
- A generic doctor extension API would execute local repair logic, so it needs explicit trust boundaries, deterministic ordering, non-interactive behavior, and clear `--fix` semantics before being exposed to third-party plugins.
