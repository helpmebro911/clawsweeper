## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not pass MiniMax baseUrl or region context into usage fetching. This PR addresses a real remaining bug, but it overlaps with open #65054/#65059 and still needs maintainer review or consolidation rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or consolidate it with #65059. The best fix is a MiniMax-owned, region-aware usage endpoint resolver that preserves `GET /v1/api/openplatform/coding_plan/remains`, derives routing from `minimax`, `minimax-portal`, and `MINIMAX_API_HOST` where appropriate, defines the safe default and token-routing trust policy, updates MiniMax docs, and adds regression tests for URL, method, global, CN, invalid/custom baseUrl, API-key, and OAuth configurations.

## Risks / Open Questions

- Merging this PR as-is may set the default MiniMax usage host to global, while current docs and #65059 imply a CN fallback for backward compatibility; maintainers should choose the default intentionally.
- The PR derives a usage bearer-token destination from provider config or `MINIMAX_API_HOST`; maintainers should decide whether to restrict usage polling to trusted MiniMax global/CN hosts or explicitly support custom origins.
- This PR overlaps with #65059/#65054, so landing both independently could duplicate or conflict in the MiniMax usage routing path.
