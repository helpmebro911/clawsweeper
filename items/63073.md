## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and the MiniMax plugin does not pass configured baseUrl or region context into usage fetching. The PR addresses a real remaining gap, but it still needs maintainer review and likely consolidation with the open #65054/#65059 work rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or consolidation with #65059. The best fix is a MiniMax-owned, region-aware usage endpoint resolver that preserves `GET /v1/api/openplatform/coding_plan/remains`, derives global/CN routing from the same credential/config source used for MiniMax auth (`minimax`, `minimax-portal`, and `MINIMAX_API_HOST` where appropriate), validates token-routing safety, updates the MiniMax docs, and adds regression tests for URL, method, global, CN, invalid/custom baseUrl, and mixed API-key/OAuth configurations.

## Risks / Open Questions

- Merging this PR as-is can route a MiniMax usage bearer token to a configured or env-derived origin through the plain `fetchJson` helper; maintainers should either constrain the resolver to trusted MiniMax global/CN hosts or route through the provider HTTP trust policy before merge.
- There is overlapping open work in #65054/#65059, so leaving both PRs open duplicates review until a maintainer chooses one implementation or asks contributors to consolidate.
- Default host behavior still needs product judgment: current docs name the CN usage endpoint, global setup uses `api.minimax.io`, and API-key versus OAuth/portal baseUrl alignment needs explicit tests.
