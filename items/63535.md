## Summary

This PR should stay open. Current main still does not implement stale cron session-row recovery, so the idea is not obsolete, but the submitted patch is not ready: it only scans canonical `:run:` rows, is skipped when retention pruning is disabled, and does not cover the stable base cron session row that current isolated cron persistence writes as the visible mutable session entry. Security review found no supply-chain, CI, dependency, secret, network, or publish-surface changes in the fetched PR diff.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is a rebased cron lifecycle/session recovery patch that runs independently of retention pruning, repairs the stable base cron session row plus any legacy `:run:` rows, deliberately chooses the terminal status for interrupted work, and adds focused regression coverage for `sessionRetention: false`, stale versus active running sessions, base-key and legacy-run-key shapes, and short-retention defer-before-prune behavior. A final review should run the focused cron tests and changed-check lane when writable validation is allowed.

## Risks / Open Questions

- Merging the PR as-is would silently skip orphan recovery for users who set `cron.sessionRetention` to `false`.
- Updating only canonical `:run:` rows can leave the stable base cron session row visible as `status: "running"`.
- A fixed two-hour orphan threshold and `done` terminal status may misclassify legitimate long-running or interrupted cron work unless the intended product semantics are settled and tested.
- The PR's added test does not prove defer-before-prune behavior under a short retention window, because the sample rows are younger than the default 24-hour retention cutoff.
