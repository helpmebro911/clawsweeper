## Summary

Keep this issue open. Current main still sends embedded Anthropic Messages runs through the boundary-aware Anthropic transport, and that transport still parses each SSE `data:` frame with raw `JSON.parse` without a repair or graceful recovery path for the reported bad-control-character failure. I did not find a merged fix, regression test, docs/changelog evidence, or canonical duplicate that resolves this pure-text Telegram/Anthropic streaming abort.

## Best Possible Solution

Keep this issue open. Add focused regression coverage around `parseAnthropicSseBody` and the embedded Anthropic Messages runner path for long text with emoji/non-ASCII content, split UTF-8 chunk boundaries, and an invalid raw-control-character-in-string SSE frame matching the reported `JSON.parse` failure. If reproduced, harden the Anthropic SSE parser or its caller with a deliberately scoped recovery/failure path that preserves terminal lifecycle delivery, then separately decide whether the same parser change resolves #68442 or that webchat/tool-call freeze needs its own fix.
