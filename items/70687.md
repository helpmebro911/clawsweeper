## Summary

Keep #70687 open. Current main has several related improvements: clearer scope-upgrade errors, non-mutating diagnostic probes, subagent admin-scope pinning, and documented `openclaw devices list` / `approve` recovery. But I did not find a source/test/history proof that the reported upgrade path from 2026.4.15/2026.4.21/2026.4.23 no longer leaves devices approved only for `operator.read` or that the Telegram native approvals handler and spawn flows recover without manual scope repair. Related scope-upgrade/native-approval and post-approval loop reports remain open, so this is not a high-confidence close.

## Best Possible Solution

Keep this open until maintainers either make #68634/#70787/#71609 the explicit canonical tracker and close this as a duplicate, or add a focused fix/regression test for the upgrade path: existing paired operator devices should not be silently downgraded to `operator.read`, `openclaw devices approve <requestId>` should persist the expanded approved scope baseline across restarts/upgrades, and Telegram native approval/subagent connections should recover without manual `paired.json` edits.
