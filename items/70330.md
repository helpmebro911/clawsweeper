## Summary

Keep #70330 open. Current main still has the core risky behavior described by the report: Control UI/WebChat New Session is still implemented as a `/new` chat send, WebChat requests admin scope, and the gateway still treats admin-scoped `/new` or `/reset` in `chat.send` as a session reset that rotates the `sessionId` and can strip the reset command from the stored post-reset message. Related issues #59839 and #70472 are open and adjacent, but #70330 contains a distinct restart/reconnect reproduction that is not fully superseded.

## Best Possible Solution

Keep the issue open and fix it in core WebChat/gateway session handling: replace the UI button’s `/new` chat-send implementation with an explicit confirmed session reset/create RPC, add reset audit metadata with old/new session ids and trigger source, make reset archives/checkpoints discoverable or restorable from Control UI, and add a regression test for gateway restart/reconnect followed by a normal WebChat message preserving the intended active session unless an explicit reset was confirmed.
