## Summary

Keep #70330 open. Current main has related hardening, but it does not fully solve the reported product problem: Control UI/WebChat still routes New Session through generic `chat.send`, admin-scoped WebChat text can still be interpreted as `/new`/`/reset` session control, there is no explicit `allowSessionControl`/trigger audit field in the `chat.send` protocol, reconnect handling does not clear reset/new queue state except for seq-gap recovery, and reset archives remain outside the normal session list/history surface. The provided related context supports this as a distinct still-open continuity issue: #59839 tracks broader hidden history after New Session, and #70472 tracks a similar silent rotation under a different trigger, but neither fully supersedes this restart/reconnect failure mode.

## Best Possible Solution

Keep this issue open and fix it in core WebChat/gateway session control. The best path is to move WebChat New Session/Clear History to explicit session-control RPCs or an additive `chat.send` session-control flag, persist an auditable reset event with trigger/old/new session ids, clear stale reset/new queue state on reconnect, and expose reset archives/checkpoints through sessions/history UI or a dedicated restore surface. Link #59839 for the broader archive discoverability work and #70472 for the compaction/orphaning variant, but keep #70330 as the restart/reconnect regression until that path is covered by source and regression tests.
