## Summary

Keep this issue open. Current main still leaves the core WebChat/session-continuity gap: Control UI WebChat requests admin scopes, the visible New Session and local `/new`/`/reset` flows still go through generic `chat.send`, and the gateway can still rotate the session key after parsing slash reset text without a first-class WebChat reset provenance/audit field. Related work has improved label preservation and compaction checkpoint restore paths, but it does not solve the reported silent rotation/reconnect diagnosis problem.

## Best Possible Solution

Keep this in OpenClaw core. Replace the WebChat New Session button and local `/new`/`/reset` handling with an explicit session-control RPC or an explicit `allowSessionControl` flow carrying trigger metadata; persist an auditable reset/rotation event with old/new session ids and archive path; clear stale reset/draft/idempotency state across reconnect; and add regression coverage for gateway restart/reconnect followed by a normal visible send so `agent:main:main` cannot silently rotate without provenance.
