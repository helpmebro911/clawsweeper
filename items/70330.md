## Summary

Keep this issue open. Current main still has the reported WebChat/session-continuity product gap: the Control UI New Session path routes through `/new` on generic `chat.send`, authorized reset text can still rotate the active session id and archive the old transcript, and the protocol/UI do not provide explicit reset provenance or first-class reset-archive recovery for the active WebChat row.

## Best Possible Solution

Keep this in OpenClaw core. Wire WebChat New Session to an explicit session create/reset RPC instead of generic `chat.send` slash text, require an explicit session-control flag or confirmed UI action for `/new`/`/reset` over WebChat, persist a visible reset/rotation audit record with old/new session ids and trigger, clear queued reset/draft state across reconnect, and expose reset archives/checkpoints or a restore path in the Sessions/WebChat UI. Add regression coverage for gateway restart/reconnect followed by a normal visible send so `agent:main:main` does not silently rotate without provenance.
