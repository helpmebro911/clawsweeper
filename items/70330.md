## Summary

Keep open. Current main still preserves the risky surface described by #70330: WebChat's New Session path is still implemented by sending `/new`, generic `chat.send` still recognizes `/new`/`/reset` and can strip a post-reset message into an ordinary-looking first message, and reset still rotates the session id plus archives the prior transcript without a first-class reset-audit/history surface in the UI. The related reports in the provided context are open or older/adjacent, not a shipped canonical fix for this restart/reconnect failure mode.

## Best Possible Solution

Keep #70330 open and fix it in the WebChat/gateway session-control path: move the UI New Session action off generic `chat.send("/new")`, require explicit reset intent for chat-driven session control, persist an auditable reset event with old/new session ids and trigger source, expose reset archives/checkpoints for recovery in Control UI, and add a reconnect/regression test that a normal post-restart visible message cannot silently rotate `agent:main:main`.
