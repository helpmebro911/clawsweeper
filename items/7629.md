## Summary

Keep open. Current main still has the bundled `memory-lancedb` plugin on vector-only LanceDB retrieval: it opens/creates the `memories` table without an FTS index, calls `vectorSearch(vector)`, drops the original query text before database search, and exposes no `hybrid`, BM25, FTS, reranker, `vectorWeight`, or `textWeight` config. The separate bundled `memory-core` engine already has BM25+vector hybrid search, but that does not implement the requested LanceDB plugin parity. The provided GitHub context also shows PR #7636, which attempted this work, was closed unmerged, and a maintainer Codex review on April 26, 2026 explicitly kept #7629 open for a focused `extensions/memory-lancedb` enhancement.

## Best Possible Solution

Keep #7629 open as a focused bundled `memory-lancedb` parity enhancement. The best implementation path is in `extensions/memory-lancedb`: add typed `hybrid` config and manifest/docs entries, create or maintain an FTS index on `text`, thread original query text through recall/forget/CLI/hooks and duplicate checks, use verified LanceDB hybrid search or a tested fallback/reranker when enabled, preserve vector-only fallback, and add tests for config validation, FTS setup, exact-keyword recall, fallback behavior, and query-text propagation.
