## Summary

Current `main` already closes Discord gateway sockets on channel stop/restart via the generic abort path, so the stale-WS condition described by PR #51297 is not reproducible from the current code. The proposed `stopAccount()` hook in `extensions/discord/src/channel.ts` would duplicate existing shutdown behavior rather than add a missing cleanup path.

## Close Comment

Closing after Codex review of current `main` at `608c08fc54da2f86338366f8eeafd7e2a02713aa`.

I could not reproduce a missing Discord gateway shutdown path in the current code. Health-monitor restarts already go through `channelManager.stopChannel(...)` in `src/gateway/channel-health-monitor.ts:160-165`, and `stopChannel` aborts the running account task even when a plugin has no custom `stopAccount` hook in `src/gateway/server-channels.ts:528-585`. On the Discord side, that abort path calls `gateway.disconnect()` inside `waitForDiscordGatewayStop` in `extensions/discord/src/monitor.gateway.ts:18-75`, then unregisters the gateway in `extensions/discord/src/monitor/provider.lifecycle.ts:456-460`. There is also a direct test for the abort-driven disconnect path in `extensions/discord/src/monitor.gateway.test.ts:80-84`.

Given that current `main` already disconnects and unregisters the Discord gateway during stop/restart, this PR's added `stopAccount()` hook is redundant on top of the existing lifecycle and does not appear to fix a reproducible gap anymore.
