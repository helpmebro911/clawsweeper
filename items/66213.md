## Summary

Keep this PR open. Current main still lacks a post-publish macOS release asset verifier and appcast CODEOWNERS protection, while the reported release/appcast failure mode is substantiated and tied to open tracker #48306. The patch is a plausible release guard but touches maintainer-owned publishing policy and still has reviewable workflow behavior, so the best cleanup action is release-owner review, not closure.

## Best Possible Solution

Keep the PR open for maintainer release-owner review. If maintainers want this guard, revise the workflow to fail fast on non-retryable GitHub API errors, recheck before timeout, fully anchor tag validation to the intended stable-release format, decide whether appcast absence should warn or fail, and align the /appcast.xml CODEOWNERS owners with release policy while linking the work to #48306.

## Risks / Open Questions

- Release workflow and appcast ownership changes affect maintainer-controlled macOS publishing and should be reviewed by release owners.
- The proposed polling loop treats all GitHub API failures as zero assets, which can turn auth, rate-limit, or outage errors into a misleading missing-assets timeout.
- The loop sleeps after the last check and then fails without a final asset recheck, so a private publish completing during the final minute can still produce a false failure.
- The tag validation is prefix-only; exposure is limited by release/workflow-dispatch permissions, but the accepted format should be tightened before landing.
- Whether appcast.xml absence should warn or fail is a release-policy decision, especially because stable macOS readiness docs require the feed to point at the new stable zip after publish.
