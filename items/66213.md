## Summary

Keep this PR open. Current main still does not contain a post-publish macOS asset verifier or root appcast CODEOWNERS protection, and the reported release failure mode is supported by current release data and open tracker #48306. The PR is focused, but it changes release workflow policy and has reviewable workflow/security edge cases, so release-owner review is the best path.

## Best Possible Solution

Keep the PR open for release-owner review. If maintainers want this guard, revise the verifier to fail fast on non-retryable GitHub API errors, recheck before timeout, fully anchor tag validation to the intended stable/correction-release formats, decide whether appcast absence should warn or fail, and align /appcast.xml CODEOWNERS ownership with release policy while linking the work to #48306.

## Risks / Open Questions

- The proposed polling loop treats every gh api error as 0 assets, so auth, rate-limit, and GitHub outage failures would be reported as missing assets after a long timeout instead of failing with the real cause.
- The loop sleeps after the last poll and then fails without a final asset recheck, so assets that arrive during the final minute can produce a false failure.
- The tag validation regex is prefix-only and looser than the repo's release tag format in .github/workflows/macos-release.yml.
- The PR makes appcast.xml absence a warning even though release docs describe appcast advancement as part of stable macOS readiness; that is a release-owner policy choice.
- The CODEOWNERS addition names individual owners for /appcast.xml instead of the existing openclaw-release-managers team, so ownership should be confirmed by maintainers.
