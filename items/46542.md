## Summary

Keep open. Current main still does not implement the proposed shared `_shared` memory store, `memorySearch.sharedPaths`, shared-origin search results, or `openclaw memory ... --shared` CLI flow. The PR remains potentially useful, but it is stale against the current memory-core plugin layout and still needs the unresolved security/data-scope review findings handled before maintainer review can proceed.

## Best Possible Solution

Keep this PR open as a real implementation candidate, but require a rebase/port into `extensions/memory-core` and `packages/memory-host-sdk`, update the public config/schema/docs for `sharedPaths` or an equivalent memory-core plugin contract, and resolve the outstanding security/data-scope review comments with targeted tests before maintainer review. If maintainers decide shared document memory should be external instead, the narrower alternative is a memory plugin or memory corpus supplement, but the current PR should not be closed automatically because current main still lacks the requested shared-store behavior and no canonical replacement item tracks it.

## Risks / Open Questions

- The proposed feature broadens local file-search/read scope for every agent that can use shared memory; blank or whitespace `sharedPaths` entries need explicit rejection so a typo cannot become broad filesystem read access.
- The PR currently appears stale against the memory-core plugin split, so applying it as-is would miss the current `extensions/memory-core` and `packages/memory-host-sdk` ownership boundaries.
- The PR discussion leaves product/API ambiguity around whether `sharedPaths` are global-only index sources or per-agent additions with weights; that needs a documented contract and tests before merge.
- The eager shared-path directory creation described in review could create a directory where an operator intended a file path, causing confusing filesystem side effects.
