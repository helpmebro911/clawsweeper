## Summary

Keep this PR open, but do not land it unchanged. Current main still does not fail fast before channel startup when the configured default model provider lacks usable auth; it only prewarms model resolution and logs channel-start failures. The PR remains a plausible implementation candidate, but it is stale against the current startup split and uses a narrower auth helper than runtime model execution, with unresolved review feedback on false blockers and circular remediation text.

## Best Possible Solution

Keep this PR open for revision. The best path is to implement the startup guard in the current post-attach sidecar startup path before channel startup, still honoring OPENCLAW_SKIP_CHANNELS and OPENCLAW_SKIP_PROVIDERS, and base the check on the same non-mutating auth contract runtime model execution uses, likely hasAvailableAuthForProvider or a resolveApiKeyForProvider wrapper supplied with cfg and agentDir. The revision should avoid provider-circular suggestions and add focused tests for missing auth, valid config-backed apiKey auth, Bedrock/aws-sdk auth, synthetic local/plugin/native auth, and skip-channel bypass behavior.

## Risks / Open Questions

- Merging the PR as-is could block valid gateway configurations that authenticate through models.providers.<provider>.apiKey, Bedrock/aws-sdk default-chain auth, plugin-owned synthetic/native auth, or local custom-provider synthetic auth.
- The hardcoded remediation example can tell users already on openai-codex/gpt-5.4 to switch to the same failing model, which is misleading.
- The PR needs a rebase or rewrite onto the current server-startup-post-attach.ts startup path before it can be evaluated as a current-main patch.
