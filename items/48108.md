## Summary

Keep open. Current main still unconditionally re-adds the active session key to the Control UI chat session dropdown even when the refreshed sessions.list result no longer contains that key, so the reported deleted-direct-session path remains possible. The related PR #48190 described the intended fix but was closed unmerged in the provided GitHub context.

## Evidence

- **Dropdown source still adds active key unconditionally:** resolveSessionOptionGroups builds options from sessions.list rows, then calls addOption(sessionKey) regardless of whether that key exists in the latest sessionsResult. For a deleted active direct session, row lookup returns undefined but the key is still inserted into the dropdown.

- **Fresh sessions refresh does not normalize missing active session:** loadSessions replaces state.sessionsResult with the gateway response and invalidates checkpoint cache, but it does not change state.sessionKey when the active key is absent from the response.
