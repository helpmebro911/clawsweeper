## Summary

Keep this issue open. Current main still has a credible Telegram-versus-WebUI divergence path for skill-backed tool availability: Telegram can pass per-group/per-topic skill filters into reply execution, including an explicitly empty topic list, while WebUI chat.send dispatches through the internal webchat path without that Telegram filter. Current vLLM/local-model docs explain one possible provider-side tool-call failure mode, but they do not prove the reported Telegram/WebUI parity bug is fixed.

## Best Possible Solution

Keep this issue open and reproduce on current main with the same agent/session, vLLM gpt-oss-20b model, himalaya skill, and Telegram route. Capture Telegram group/topic config, replyOptions.skillFilter, merged skill filter, skillsSnapshot, effective runtime tools, provider/model metadata, and structured-versus-text tool-call events for both WebUI chat.send and Telegram. If Telegram loses himalaya through skill filtering, fix Telegram filter inheritance or diagnostics and add a Telegram dispatch regression test; if vLLM emits non-structured tool calls, keep the provider diagnostic path and document the exact model setup; if parity is proven, close later with source-backed cannot-reproduce or implemented-on-main evidence.
