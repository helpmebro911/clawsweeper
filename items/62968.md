## Summary

Current main still matches the reported bug. Completion-mode announce dispatch preserves the failed direct-send error only in the `direct-primary` phase, then returns the successful queue/steer fallback as the top-level delivered result. The caller and cleanup lifecycle both treat delivered results as successful and do not surface phase errors, so the recovered direct-send failure remains silent except for retry-attempt logs.

## Best Possible Solution

Keep this issue open. Add a narrow recovered-delivery diagnostic: when completion dispatch records a failed `direct-primary` phase with an error and a later `queue-fallback` phase delivers, warn once without marking the announce failed. The likely implementation belongs in `runSubagentAnnounceFlow` after `delivery` is returned, or behind a small dispatch logger/callback seam, with a focused test covering `delivered: true`, `path: "queued"` or `"steered"`, and a failed direct-primary phase.
