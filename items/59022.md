## Summary

Keep open. The issue is recent, has no protected labels, and reports a concrete `canvas.a2ui.push` failure on OpenClaw 2026.3.31. Current docs explain that a direct unauthenticated `/__openclaw__/a2ui/` request can return Unauthorized under non-loopback binds, and that paired nodes should receive node-scoped capability URLs, but I could not verify the current source/tests or reproduce the node failure because the local command sandbox failed before starting a shell. That is not enough evidence for a high-confidence close.

## Evidence

- **Issue report:** Reporter says `canvas a2ui_push` fails with `UNAVAILABLE: A2UI_HOST_UNAVAILABLE: A2UI host not reachable` while other Canvas commands work; direct curl to `/__openclaw__/a2ui` returns Unauthorized.
- **Current docs describe expected auth/capability model:** Gateway canvas host serves both `/__openclaw__/canvas/` and `/__openclaw__/a2ui/`; non-loopback binds require Gateway auth, and paired nodes should receive node-scoped capability URLs because WebViews typically do not send auth headers.
  - file: [docs/gateway/configuration-reference.md:727](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/docs/gateway/configuration-reference.md#L727)
