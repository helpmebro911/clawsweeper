## Summary

Keep #70003 open. Current main has useful mitigations for long-running work, including `chat.abort`, `/stop`, and a hard `/redirect`/`sessions.steer` path, and the older triage comment is partly outdated because main-lane default concurrency is now 4. But the central requested behavior is still not implemented: normal same-session follow-up messages are still queued while a run is active, and embedded agent runs are wrapped in a per-session command lane that serializes turns for that session.

## Best Possible Solution

Keep the issue open and narrow it to the remaining product gap: decide whether normal inbound follow-ups during an active run should preempt, steer, or start a concurrent same-session turn. A likely implementation would add an explicit queue/turn policy for same-session active runs, wire it through `chat.send` or `sessions.steer`, make cancellation cooperative for blocking tools, and add a gateway/UI regression test where a long-running tool is active and a later user message is processed immediately according to the selected policy.

Since session lanes are ordinary command-queue lanes created with `maxConcurrent: 1`, same-session turns are still serialized even when the global main lane can run more than one task.
