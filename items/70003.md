## Summary

Keep open. Current main has useful partial controls (`/stop`, `chat.abort`, `sessions.steer`, queue modes), and related ClawSweeper reports correctly identify those narrower shipped paths. This issue asks for a stronger product capability: handling a new user turn during a long-running in-session tool/run. Current main still serializes each session, default inbound messages collect/follow up, and documented steering is delivered only after the current assistant turn finishes its tool calls.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely path is a core Gateway/session runtime design that either preempts active turns without waiting for a tool boundary, or moves long-running tool/subagent results onto an async event/background-result channel, with tests for foreground `exec`, queued/collect defaults, `sessions.steer`, and transcript consistency.

## Risks / Open Questions

- True per-session concurrent turns can corrupt transcript ordering or tool-call result invariants unless the design explicitly models active turns, cancellation, and async tool results.
- Existing `/stop`, `chat.abort`, queue `interrupt`, and `sessions.steer` behavior should be preserved as narrow control paths while adding any stronger concurrent-turn capability.
