## Summary

Keep #68645 open. Current main has broader cron timeout work, including a single outer cron timeout and manual/startup task-ledger timeout coverage, but it does not implement this PR's explicit `runTimeoutSeconds` field and still only closes timer-driven task ledger rows inside the post-run locked apply pass. The PR discussion includes a concrete production stale/lost-task report and follow-up comments tying the wall-clock kill path to daily isolated-session cron jobs and related reports #68156/#68157, so this remains plausible maintainer-reviewable work rather than conservative cleanup.

## Best Possible Solution

Keep the PR open for maintainer review or a rebased replacement. The best path is to reconcile it with current `src/cron/service/timeout-policy.ts`: either add a documented `runTimeoutSeconds` run-level budget, or explicitly decide that `timeoutSeconds` remains the only outer cron budget. Independently, add timer-driven regression coverage proving `onTimer` closes the cron task ledger as `timed_out` before or outside the locked store-apply pass when execution times out.
