## Summary

Keep open. Current `main` has inbound external-content wrapping and provenance for hook/cron payloads, plus docs recommending a separate reader agent, but it does not implement the requested built-in dual-LLM privilege-separation architecture or a worker-manager JSON handoff for cron jobs.

## Evidence

- **Cron hooks still run in one agent turn with wrapped text:** `runCronIsolatedAgentTurn` detects external hook provenance and replaces the prompt body with `buildSafeExternalPrompt(...)`, then sends that same `commandBody` into the isolated agent turn. This is prompt wrapping/sanitization inside one run, not a separate low-privilege worker model feeding a privileged manager.
  - file: [src/cron/isolated-agent/run.ts:582](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/cron/isolated-agent/run.ts#L582)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Security layer is marker-based wrapping, warnings, and token stripping:** The external-content helper defines boundary markers and a warning banner for untrusted content. It does not define a second model, privilege-separated execution path, or structured worker-to-manager JSON contract.
  - file: [src/security/external-content.ts:57](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/security/external-content.ts#L57)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Cron payload surface only tracks provenance and unsafe-wrap opt-out:** The cron agent payload includes `allowUnsafeExternalContent` and `externalContentSource`, but no fields for a reader/worker model, validator model, or manager/worker routing policy.
  - file: [src/cron/types.ts:116](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/cron/types.ts#L116)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Tests cover wrapping behavior, not privilege separation:** The cron tests assert that external hook content is wrapped with `EXTERNAL, UNTRUSTED` / `SECURITY NOTICE` before the embedded agent sees it. That confirms the current mitigation is wrapping provenance into the prompt, not dual-LLM isolation.
  - file: [src/cron/isolated-agent.hook-content-wrapping.test.ts:21](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/src/cron/isolated-agent.hook-content-wrapping.test.ts#L21)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **Docs describe a reader agent as operator guidance, not shipped cron architecture:** The security docs recommend using a read-only or tool-disabled reader agent to summarize untrusted content before passing it to the main agent. That is advice for reducing blast radius, not evidence that OpenClaw cron already enforces the architecture requested in the issue.
  - file: [docs/gateway/security/index.md:616](https://github.com/openclaw/openclaw/blob/c6684af682bcb59bd1ee85cb49f72d4074007e2c/docs/gateway/security/index.md#L616)
  - sha: [c6684af682bc](https://github.com/openclaw/openclaw/commit/c6684af682bcb59bd1ee85cb49f72d4074007e2c)
- **No built-in dual-LLM hook/plugin surface found:** A repository search for `after_external_content_wrap`, `blockrun`, `dual-LLM`, and `dual llm` returned no hits on current `main`, so the cross-referenced idea from PR #30046 is not present under those implementation names.
