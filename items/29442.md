## Summary

Keep #29442 open. Current main has meaningful prompt-injection hardening through external-content wrapping, special-token sanitization, tool policies, sandbox/read-only guidance, and plugin hooks that can block tools or claim a cron run, but it still does not implement the requested privilege-separated reader/manager architecture or a plugin seam that can replace/block wrapped external content and then continue with a sanitized privileged manager turn. The linked ClawMoat plugin is useful defense-in-depth, and closed PR #30046 was directly related, but neither makes the core security/product item obsolete.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is to add either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built and then continue safely. ClawHub plugins such as https://github.com/darfaz/clawmoat could then provide scanner/sanitizer implementations on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate wrapper/special-token hardening and tool policy with the requested privilege-separated reader/manager architecture.
- Closing as clawhub would be premature because plugins can observe, block tools, or short-circuit cron, but current main still lacks a clean external-content replace/block-and-continue seam for sanitized manager turns.
- The final design still needs maintainer security/product judgment on fail-open versus fail-closed behavior, cost/latency, default cron/web-tool policy, and whether the right core shape is first-class reader-worker flow or an additive plugin API seam.
