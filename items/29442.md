## Summary

Keep #29442 open. Current main has meaningful prompt-injection hardening through external-content wrapping, special-token sanitization, tool policy guidance, and plugin tool-call guards, but it still does not implement the requested privilege-separated reader/manager architecture or a narrow external-content replacement/blocking hook before cron submits the prompt to the privileged agent turn. The linked ClawMoat plugin is useful defense-in-depth, and closed PR #30046 was a plausible direction, but neither makes the core security/product item obsolete.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is to add either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built. ClawHub plugins such as https://github.com/darfaz/clawmoat could then provide scanner/sanitizer implementations on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate marker/special-token wrapping and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- Closing as clawhub would be premature because ClawMoat-style scanners can help today, but current main still lacks a focused external-content replace/block seam for cron hook payloads before privileged prompt submission.
- The final design needs maintainer security/product judgment on fail-open versus fail-closed behavior, cost/latency, default cron/web-tool policy, and whether the right core shape is a first-class reader-worker flow or a documented plugin API seam.
