## Summary

Current `main` does not implement this feature end-to-end. OpenClaw can preserve provider-supplied retry/reset text when the upstream message already contains it, but its generic fallback for rate limits is still `⚠️ API rate limit reached. Please try again later.` and it does not appear to derive user-facing wait time from internally tracked cooldown/retry data.

## Evidence

- **Generic fallback is still hard-coded:** `formatRateLimitOrOverloadedErrorCopy()` returns the hard-coded generic rate-limit string when no provider-specific retry text is available.

## Risks / Open Questions

- Conclusion is based on code/docs/tests inspection rather than a live rate-limit reproduction, but the current expectations and formatter logic are direct and unambiguous.
- A subset of providers already surface retry/reset time via their raw upstream error text, so the remaining gap is specifically internal cooldown-derived messaging, not all rate-limit messaging.
