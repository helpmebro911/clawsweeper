## Summary

Keep open. Current `main` still does not implement per-agent WhatsApp ack emoji resolution, and the generic ack resolver still assumes flat string `ackReaction` values rather than WhatsApp's object-shaped config.

## Evidence

- **WhatsApp ack path still uses only channel emoji:** `maybeSendAckReaction` reads `params.cfg.channels?.whatsapp?.ackReaction` and sends `(ackConfig?.emoji ?? "").trim()`. It never resolves per-agent fallback from `agentId`, so different agents sharing one WhatsApp account still cannot emit different ack emoji.
  - file: [extensions/whatsapp/src/auto-reply/monitor/ack-reaction.ts:36](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/whatsapp/src/auto-reply/monitor/ack-reaction.ts#L36)
  - command: `nl -ba extensions/whatsapp/src/auto-reply/monitor/ack-reaction.ts | sed -n '1,200p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Agent id reaches the WhatsApp ack path but is not used for emoji resolution:** `process-message.ts` passes `route.agentId` into `maybeSendAckReaction`, but the current ack helper does not use that value when choosing the reaction emoji.
  - file: [extensions/whatsapp/src/auto-reply/monitor/process-message.ts:167](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/whatsapp/src/auto-reply/monitor/process-message.ts#L167)

- **Generic resolver on main still expects string ackReaction values:** `resolveAckReaction` casts both account-level and channel-level `ackReaction` to `string | undefined`. That does not match WhatsApp's configured type, where `ackReaction` is an object with `emoji/direct/group` fields.
  - file: [src/agents/identity.ts:19](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/identity.ts#L19)

  - file: [src/config/types.whatsapp.ts:38](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/config/types.whatsapp.ts#L38)

## Risks / Open Questions

- Closing this PR would drop a still-unimplemented behavior change: current `main` cannot select different WhatsApp ack emoji per responding agent.
- The current code path still hardcodes channel-level WhatsApp emoji lookup, so the reported multi-agent behavior gap remains plausible and unresolved.
- The current mainline tests do not lock the WhatsApp object-format/per-agent cases, so there is no landed regression proof that the request is already covered.
