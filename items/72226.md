## Summary

Current main does not yet close #72226. Main-session cron `systemEvent` delivery is implemented, but heartbeat-triggered cron runs still turn the cron prompt into runtime-only context with an empty submitted prompt. When the replay history already ends in an assistant message, the current code path has no visible user turn to force Anthropic's Messages API out of assistant-prefill shape, and the existing role-ordering recovery does not match Anthropic's reported "conversation must end with a user message" error.

## Best Possible Solution

Keep #72226 open. Fix the OpenClaw core cron/heartbeat embedded-run path so a main-session cron `systemEvent` produces a valid Anthropic turn sequence when replay history ends with an assistant message, either by submitting the cron event as a real user turn for that run or by inserting a synthetic user boundary before runtime-only event context. Add a regression covering direct `anthropic/*` replay ending on an assistant turn plus a main-session cron `systemEvent`.

## Risks / Open Questions

- A naive fix that records all heartbeat runtime context as a normal user transcript could leak internal/runtime-generated context into user-authored history; the boundary should be targeted to cron/systemEvent or assistant-tail provider shaping.
- Error-only recovery would still let direct Anthropic fail once per affected run and may exhaust fallback chains; the better fix is to shape the request before provider submission.
