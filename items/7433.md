## Summary

Keep open. Current `main` does not implement the Baileys reliability hooks requested in #7433: the WhatsApp socket is created without `getMessage` or `cachedGroupMetadata`, group metadata is still handled by a local on-demand cache in the inbox monitor, the connect-time `groupFetchAllParticipating()` result is only logged, and there are no `groups.upsert` / `groups.update` / `group-participants.update` / `lid-mapping.update` listeners on current `main`.

## Evidence

- **Socket creation lacks requested Baileys callbacks:** `createWaSocket()` passes auth, version, logger, browser, and proxy options into `makeWASocket()`, but there is no `getMessage` or `cachedGroupMetadata` option in that constructor call.
  - file: [extensions/whatsapp/src/session.ts:149](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/extensions/whatsapp/src/session.ts#L149)
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Group metadata is still cached only inside the inbox monitor:** The WhatsApp monitor creates its own `groupMetaCache` and resolves groups by calling `sock.groupMetadata(jid)` on demand. That cache is separate from socket construction and is not exposed back to Baileys as `cachedGroupMetadata`.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:234](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/extensions/whatsapp/src/inbound/monitor.ts#L234)
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Connect-time group hydration is not persisted into a cache:** On connect, the code calls `sock.groupFetchAllParticipating()` only to log how many groups were hydrated; the returned groups are not stored into `groupMetaCache` or any socket callback path.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:717](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/extensions/whatsapp/src/inbound/monitor.ts#L717)
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **LID handling is lookup-only, not update-event persistence:** Current LID resolution uses `sock.signalRepository?.lidMapping` at read time and falls back to reverse-mapping files, but there is no code here persisting fresh mappings from a `lid-mapping.update` event.
  - file: [extensions/whatsapp/src/targets-runtime.ts:122](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/extensions/whatsapp/src/targets-runtime.ts#L122)
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **No matching Baileys event listeners or socket options on current main:** A repo search for `cachedGroupMetadata`, `getMessage`, `groups.upsert`, `groups.update`, `group-participants.update`, and `lid-mapping.update` returned no matches in the OpenClaw WhatsApp implementation.
  - command: `rg -n "cachedGroupMetadata|getMessage\\s*:|groups\\.upsert|groups\\.update|group-participants\\.update|lid-mapping\\.update" extensions/whatsapp src test -g '!**/dist/**'`
  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)

## Risks / Open Questions

- Static inspection only; this read-only review did not run a live WhatsApp/Baileys reproduction path.
- Some adjacent reliability work exists (`makeCacheableSignalKeyStore`, local group metadata TTL cache, LID lookup helper), so future implementation should verify which Baileys hooks are still necessary versus redundant.
