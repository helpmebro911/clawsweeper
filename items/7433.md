## Summary

Keep open. Current `main` still does not implement the central WhatsApp/Baileys reliability requests: `createWaSocket` does not pass Baileys `getMessage` or `cachedGroupMetadata`, and WhatsApp production source has no `groups.upsert`, `groups.update`, `group-participants.update`, or `lid-mapping.update` handlers. The code has partial related behavior, including cacheable signal keys, an inbound-only group metadata cache, and LID resolution through Baileys/auth mapping, but those do not close the requested socket-level retry and group-cache work. The existing issue discussion also includes a maintainer Codex review from April 26, 2026 explicitly keeping this open, and the broader related #66917 does not supersede these concrete recommendations.

## Best Possible Solution

Keep #7433 open and implement the missing reliability work in the bundled `@openclaw/whatsapp` plugin: add a bounded message-content store and pass a Baileys `getMessage` callback, create a shared group metadata cache passed as `cachedGroupMetadata`, seed/update it from group hydration and group events, and then decide whether existing Baileys `signalRepository.lidMapping` persistence is sufficient or whether an explicit `lid-mapping.update` listener is still needed. Cover the socket options and cache update behavior with focused WhatsApp plugin tests.
