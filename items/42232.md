## Summary

Keep open. Current `main` still rejects `chat.send` payloads that include `model`, and the only programmatic model-override path is `sessions.patch`, which is a separate admin-scoped RPC and is blocked for ordinary webchat clients. That is not equivalent to the requested write-scoped `chat.send` support for per-session model selection.

## Evidence

- **`chat.send` schema still has no `model` field:** `ChatSendParamsSchema` defines `sessionKey`, `message`, `thinking`, delivery/origin fields, attachments, timeout, provenance, and `idempotencyKey`, with `additionalProperties: false`. There is no optional `model` property, so `chat.send` still rejects it as an unexpected property.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/gateway/protocol/schema/logs-chat.ts#L35)
  - command: `nl -ba src/gateway/protocol/schema/logs-chat.ts | sed -n '1,140p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **`chat.send` handler enforces that schema before doing any routing:** The `chat.send` handler immediately runs `validateChatSendParams(params)` and returns `invalid chat.send params: ...` on validation failure, so a caller cannot sneak a `model` field through runtime handling.
  - file: [src/gateway/server-methods/chat.ts:2037](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/gateway/server-methods/chat.ts#L2037)

- **There is a separate session-override RPC for models:** `SessionsPatchParamsSchema` includes an optional `model` field, and `applySessionsPatchToStore` resolves the requested model through `resolveAllowedModelRef(...)` before applying the session override and marking a live switch pending.
  - file: [src/gateway/sessions-patch.ts:388](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/gateway/sessions-patch.ts#L388)

- **That alternate RPC is not equivalent in scope:** Gateway method scopes put `chat.send` under `WRITE_SCOPE`, but `sessions.patch` under `ADMIN_SCOPE`. A client that can send chat messages does not automatically have permission to change the session model via `sessions.patch`.
  - file: [src/gateway/method-scopes.ts:120](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/gateway/method-scopes.ts#L120)

- **Ordinary webchat clients are explicitly blocked from `sessions.patch`:** `rejectWebchatSessionMutation(...)` returns an error for webchat clients trying to patch sessions unless the client is the Control UI: `webchat clients cannot patch sessions; use chat.send for session-scoped updates`. That leaves non-Control-UI web dashboards without an equivalent programmatic model-selection path.
