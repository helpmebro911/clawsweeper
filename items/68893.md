## Summary

Current main still has the reported behavior. `openclaw onboard` reaches `setupSkills`, and the Homebrew prompt is gated only by `process.platform !== "win32"`, so FreeBSD remains eligible for the "Homebrew recommended" note whenever a brew-backed skill is selected and `brew` is missing. The related PRs in the provided context propose the right allowlist-style fix, but none is merged into the checked-out main.

## Best Possible Solution

Keep #68893 open until a small fix lands. The likely best implementation is to gate the Homebrew recommendation/install-command prompt on supported Homebrew platforms, e.g. `process.platform === "darwin" || process.platform === "linux"`, and add a regression test that stubs `process.platform` to `freebsd` and asserts no `Homebrew recommended` note is emitted for a brew-backed skill. If maintainers want stronger behavior, also avoid surfacing brew-only install actions on unsupported platforms unless the skill metadata explicitly opts in.
