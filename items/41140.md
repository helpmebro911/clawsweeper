## Summary

Keep #41140 open. Current main has stronger allowlist parsing, exact-command allow-always reuse, Windows-only `argPattern`, and strict inline-eval prompting, but it still does not implement pre-allowlist command-content deny patterns. The related PR #47848 is open/unmerged, and the external CRE links in the comments are useful prior art rather than shipped OpenClaw core behavior.

## Best Possible Solution

Keep the issue open and treat it as actionable core security hardening. The best path is to add first-class deny rules to exec approvals, evaluate them before allowlist/ask fallback decisions against a canonical command string and argv/shell payload, document precedence, expose management through the existing approvals CLI/UI surfaces, and add regression tests for shell-wrapper bypasses plus non-shell direct invocations. PR #47848 can be mined for ideas, but current main should not be considered fixed until a reviewed, merged implementation ships.

## Risks / Open Questions

- This is a core exec-approval security boundary, not a ClawHub-only add-on: the enforcement point must run before host exec policy allows a command.
- Closing as duplicate/superseded on the open PR would be premature because the PR is unmerged and this issue remains the focused canonical report for command-content deny patterns.
- A complete fix needs schema, CLI/UI/docs, gateway/node-host evaluation, and tests for shell-wrapper bypass cases such as `zsh -c "openclaw gateway stop"` with `/bin/zsh` allowlisted.
