## Summary

Keep #70185 open. Current main still does not implement the PR's central fix: the bundled memory-core CLI metadata entry registers only CLI descriptors, and the CLI metadata loader only wires `registerCli`, so `registerMemoryCapability` remains a no-op in this path. The related wiki bridge path still depends on active memory public artifacts, so the observable `wiki bridge import` zero-artifact problem remains actionable even though pruning is now guarded when no memory capability is present.

## Best Possible Solution

Keep the PR open and review it as the canonical fix for the remaining CLI bridge import gap. The best path is a narrow main change that lets CLI metadata loading register the selected active memory plugin's `publicArtifacts` provider, with the same memory-slot and rollback safeguards used by full plugin registration, plus focused regression coverage for `openclaw wiki bridge import`/CLI metadata mode.
