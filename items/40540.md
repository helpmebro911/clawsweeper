## Summary

Keep open. Current `main` still performs the package-manager self-update from the running CLI process and only considers gateway restart handling afterward, so there is no strong evidence that the Windows `EBUSY` self-update failure in #40540 is fixed on `main`. The current user-facing recovery hints also cover `EACCES` and native optional-dependency failures, but not `EBUSY`/self-lock handling.

## Evidence

- **Package self-update still runs before any restart handling:** `runPackageInstallUpdate` immediately invokes the global package-manager install (`global update`) against the detected install root. The gateway-service loaded check and restart-script preparation happen later in `updateCommand`, after the update flow completes.
