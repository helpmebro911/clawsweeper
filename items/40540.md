## Summary

Current main still runs Windows npm global self-updates from the active OpenClaw process, and the gateway update path only schedules restart after the package update succeeds. That leaves the reported Windows EBUSY rename failure unresolved; the detached-helper implementation candidate remains open and unmerged.

## Best Possible Solution

Keep this issue open until OpenClaw ships a Windows-safe global npm update flow. The likely path is to review and revise #44614 or land an equivalent design that delegates package replacement to a detached helper or otherwise stops/releases the gateway and active CLI before npm renames the global package root, with regression coverage for EBUSY/rename output and gateway update.run plus a Windows smoke/manual verification note.
