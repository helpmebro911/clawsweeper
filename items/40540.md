## Summary

Current main still runs Windows npm global self-updates through the active OpenClaw process and only restarts the gateway after a successful package replacement. No detached Windows updater or EBUSY rename recovery is present; #44614 remains the open implementation candidate, while #40724 was closed unmerged and #48157 was closed as a duplicate report.

## Best Possible Solution

Keep this issue open as the canonical Windows npm global self-update bug. Review and revise #44614, or land an equivalent Windows-safe update design that releases the active CLI/gateway locks before npm renames the global package root, with regression coverage for EBUSY/rename output, gateway `update.run`, and a Windows smoke/manual path that does not pre-stop the gateway.
