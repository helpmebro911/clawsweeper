## Summary

Keep open. Current main still does not implement the proposed `acp_send` tool, `sessions_cancel` tool, or `sessions.cancel` Gateway method. Main has related overlap through `sessions_send`, `/acp cancel`, and `sessions.abort`, but the PR remains a potentially useful ACP lifecycle rework candidate rather than obsolete or implemented. Its current diff also needs security and product/API fixes before merge.

## Best Possible Solution

Keep this PR open only as a rework candidate: rebase it onto current main, decide whether an ACP-specific send tool is still needed beyond current `sessions_send` ACP handling, and if an agent cancel tool is desired, implement it as a guarded surface over the existing `sessions.abort`/ACP cancellation path with visibility/ownership checks, ACP target validation, protocol/docs/catalog updates, and focused tests.

## Risks / Open Questions

- Security: the PR's proposed `sessions_cancel` path forwards a raw caller-provided session key to ACP cancellation without reusing the existing session visibility, ownership, and A2A policy guards used by session tools.
- Functional: the PR's `acp_send` path disables normal delivery and relies on ACP callbacks; without an ACP-only target check, non-ACP target replies can be accepted but effectively lost.
- Product/API: adding `sessions.cancel` beside the shipped `sessions.abort` and `/acp cancel` would fragment session-control semantics unless naming and behavior are reconciled.
- Validation: the PR needs focused tests for tool registration, cancel authorization/visibility, ACP-only send targeting, and interaction with existing `sessions.abort` and ACP cancellation paths.
