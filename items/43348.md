## Summary

Current main still does not implement the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC. Main has partial overlap through `sessions_send`, `sessions.abort`, and `/acp cancel`, but that does not make this PR obsolete. The PR is a plausible implementation candidate, but it needs rework because the proposed cancel and ACP fire-and-forget paths still carry authorization, target-validation, and API-fragmentation risks.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate ACP-only send tool is still needed beyond `sessions_send`, and if an agent cancel tool is desired, implement it through the existing `sessions.abort`/ACP cancellation model or a clearly documented new RPC with visibility and ownership checks, ACP target validation, protocol/docs/tool-catalog updates, and focused regression tests.

## Risks / Open Questions

- The proposed `sessions_cancel` path could let an agent cancel ACP turns for any session key it can learn unless it reuses the current session resolution, visibility, ownership, and A2A policy checks.
- The proposed `acp_send` path can silently lose output for non-ACP targets because it dispatches fire-and-forget while relying on ACP callback delivery.
- Adding `sessions.cancel` beside shipped `sessions.abort` and `/acp cancel` would fragment session-control semantics unless naming, scope, behavior, and docs are reconciled.
- The PR has no accompanying current-main tests in the provided diff for tool registration, cancel authorization, ACP-only targeting, callback delivery, or interaction with existing abort/cancel surfaces.
