## Summary

Keep open. Current main still does not implement cron pre-hooks, and PR #45129 is a real implementation attempt for the still-open broader request #9465 rather than an obsolete or superseded change. The PR needs substantial rework before merge, especially persistence/update handling and shell-runner safety, but the underlying feature gap remains real.

## Best Possible Solution

Keep the PR open as useful prior art for #9465, but require a maintainer-shaped redesign before merge: define the cron hook contract in types/schema/docs, persist and patch hooks in `src/cron/service/jobs.ts`, add protocol/service/CLI tests, consume or ignore stdout, handle signal exits, bound `timeoutMs`, decide started/skipped/task semantics, and document the shell security model. If maintainers prefer externalization, first add a core/plugin API that can gate a cron run before the model/API call.

## Risks / Open Questions

- Leaving the PR open keeps a stale, currently incomplete implementation visible; maintainers should require rework before any merge.
- Shell-based pre-run hooks are a security and UX boundary because they execute local commands from persisted cron configuration; this needs explicit validation, limits, documentation, and owner review.
- The broader #9465 asks for pre, post, onError, agentTurn, systemEvent, shell, and webhook hooks; this PR only addresses shell pre-hooks, so merging it as-is would only partially satisfy the canonical request.
