## Summary

Keep open. Current main has source and unit-test coverage for the basic `workspaceOnly=false` host-tool expectation, but it does not prove the reported direct-session/tool-host mismatch is fixed. The active FS policy is still only a boolean `workspaceOnly`, direct `read` assembly still starts from a single workspace root, and the reusable guard still validates against exactly one root. The issue is recent, concrete, and the existing comment already narrowed it to a direct-session policy-wiring gap rather than a stale or duplicate read-tool report.

## Best Possible Solution

Keep the issue open. A maintainer should trace the direct-session `read` tool construction from gateway/session runtime through provider tool normalization, decide the intended contract for non-workspace read roots, then add a regression that exercises a direct-session `read` against an approved non-workspace path when `tools.fs.workspaceOnly=false` or an equivalent expanded policy is active. The fix should thread one canonical FS policy/resolver through the outer tool-host layer or explicitly document that broader roots are not supported.
