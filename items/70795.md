## Summary

Keep open. The current main source supports the narrow helper-level case where host `read` is unwrapped when `tools.fs.workspaceOnly=false`, but that does not close #70795’s reported direct-session/tool-host mismatch. The issue discussion already narrows the problem the same way: the existing unit test passes, while the remaining ask is to trace the active direct-chat read path and add regression coverage for an approved non-workspace root.

## Best Possible Solution

Keep #70795 open and fix it by tracing the actual direct-session read-tool host path, then either thread one canonical effective filesystem policy with approved read-only roots through read/sandbox/OpenShell/tool-host layers, or document and test the intended sandbox mount requirement. Add a regression test that exercises direct-session `read` against an approved non-workspace root with `tools.fs.workspaceOnly=false`.
