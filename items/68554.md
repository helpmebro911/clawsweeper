## Summary

Keep this PR open. Current main still has the two targeted `loadSessionStore(..., { skipCache: true })` calls inside the serialized session-store update paths, and `loadSessionStore` bypasses the object cache by doing a synchronous disk read and JSON parse when `skipCache` is set. The PR is a plausible focused fix for a real current performance/outage symptom also reported in related issue #70857, but it now needs maintainer review because current main also contains a test that explicitly expects `updateSessionStoreEntry` to re-read disk inside the lock to avoid clobbering an external writer with unchanged mtime.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main, reconcile the event-loop blocking fix with the current stale-cache/external-writer test, and add or adjust focused session-store tests so the accepted behavior is explicit before landing.
