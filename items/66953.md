## Summary

Keep open. Current `main` still derives outbound mirror routing from the caller-provided `agentId`, Feishu outbound session routing still bakes that `agentId` directly into the session key, and mirrored transcript appends still resolve their store from `agentId` rather than from the routed session key. I found no main-branch regression coverage for the reported Feishu multi-account multi-agent `delivery-mirror` case, so there is not strong evidence that #66953 is already fixed or obsolete.

## Evidence

- **runMessageAction still roots routing/account fallback in resolvedAgentId:** `runMessageAction()` resolves `resolvedAgentId` from `input.agentId` or `input.sessionKey`, then uses that value for bound-account fallback. There is no later guard here that re-derives the agent from the outbound route/session key for the send path.
  - file: [src/infra/outbound/message-action-runner.ts:824](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/infra/outbound/message-action-runner.ts#L824)

  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **send path passes agentId straight into outbound route and mirror metadata:** The send handler passes `agentId` into `prepareOutboundMirrorRoute(...)`, then mirrors the delivered transcript with `{ sessionKey: outboundRoute.sessionKey, agentId }`. That keeps mirror agent identity tied to the caller-provided agent value, not to the routed session key.
