## Summary

Keep open. Current main still has the reported failure class: delivery-mirror writes use the current or effective request agent id, while the transcript session key can come from an outbound route or provided session key. The issue is recent, has a concrete Feishu multi-account reproduction and a corroborating comment, and the related closed duplicate-delivery issues do not cover wrong-agent transcript ownership.

## Best Possible Solution

Keep this issue open. Add a focused regression around Feishu or the generic outbound route with session.dmScope="per-account-channel-peer", a writer account/agent binding, and a delivery-mirror append where the current/default request agent differs from the writer route. Then either derive mirror transcript ownership from the outbound route/session key or ensure every caller propagates the writer agent before route construction, and update the gateway precedence test to encode the chosen contract.
