## Summary

Keep this issue open. Current main still sends OpenRouter/OpenAI-compatible model requests through the shared guarded Undici fetch path and still maps generic fetch/connection failures to the reported `LLM request failed: network connection error.` message. The useful proxy workaround in the discussion explains some reporters' failures, and related #58150 was closed as a duplicate of this issue, but I did not find a shipped RISC-V64-specific fix, fallback, doctor check, or narrower canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64 provider transport tracker. Maintainers should ask for exact board and distro, Node build source, `process.arch`, `process.versions`, direct `fetch` and `node:https` results against the same provider, redacted gateway service proxy environment, and logs that preserve the underlying provider failure cause. If stale proxy injection is confirmed, the fix likely belongs in doctor/docs or service environment handling; if RISC-V64 Undici/fetch is confirmed, the fix likely belongs in the shared provider transport path or an explicit compatibility diagnostic.
