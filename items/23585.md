## Summary

Keep #23585 open. Current main still has no trusted-proxy fallback/default user path: the auth implementation requires the configured identity header, the public config/schema surfaces do not accept `defaultUser` or `fallbackUser`, tests assert `trusted_proxy_user_missing`, and the related discussion explicitly says #25293 did not fix this and #17761 is a separate trusted-proxy fallback problem.

## Best Possible Solution

Keep #23585 open. If maintainers accept the feature, implement an explicit trusted-proxy fallback/default user option across `GatewayTrustedProxyConfig`, zod and generated schemas, config wizard/help/docs, security audit guidance, and auth tests. Define that it only applies after trusted proxy source and required-header checks pass, decide how `allowUsers` applies to the fallback identity, and settle the current loopback/non-loopback Cloudflare Tunnel policy. If maintainers reject fallback identity on security grounds, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to a supported token/password, Tailscale Serve, or non-loopback identity-header setup.
