## Summary

Keep open. Current main still lacks a hard fail-open timeout around the full memory_search manager-init/search path, so a never-resolving manager creation or search can still block the tool result path. Open PR #49794 is the active proposed fix, but it is not merged into main and does not make this issue closable yet.

## Evidence

- **memory_search awaits manager init before any catch/timeout guard:** createMemorySearchTool awaits getMemoryManagerContext({ cfg, agentId }) directly. If manager resolution never settles, the later catch block cannot return the unavailable tool result requested by the issue.
  - file: [extensions/memory-core/src/tools.ts:211](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/memory-core/src/tools.ts#L211)
  - command: `nl -ba extensions/memory-core/src/tools.ts | sed -n '183,340p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **memory_search awaits manager.search without a wall-clock timeout:** The handler awaits memory.manager.search(query, ...) directly and only converts thrown/rejected errors into buildMemorySearchUnavailableResult. There is no Promise.race, AbortSignal deadline, or outer tool-level timeout around the search call.
  - file: [extensions/memory-core/src/tools.ts:252](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/memory-core/src/tools.ts#L252)
