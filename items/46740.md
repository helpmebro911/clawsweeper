## Summary

Keep open. Current main still does not implement the PR's requested classification: post-start `Error("acpx exited with code 1")` failures remain `ACP_TURN_FAILED`, and the checked-out ACPX plugin now delegates runtime behavior to `acpx/runtime`, so the PR is stale but still represents useful unresolved behavior work.

## Best Possible Solution

Keep the item open, but rework it against current main: either add a narrow OpenClaw manager-side classifier for truly silent ACPX/runtime process exits, or upstream the classification into `acpx/runtime` so it emits an OpenClaw-recognized `ACP_BACKEND_UNAVAILABLE` code. Preserve session-init `ACP_SESSION_INIT_FAILED`, avoid classifying stderr/user-input failures as backend outages, and add regression tests for final error classification plus `/acp doctor` guidance.
