## Summary

Keep open. Current main still sends the visible `<media:audio>`/`<media:${kind}>` placeholder as the Signal JSON-RPC message body for media-only sends, so the reported caption leak remains. The related discussion correctly identifies the boundary leak, but no current code path or regression test shows the requested invisible/non-rendered Signal placeholder behavior is implemented.

## Evidence

- **Signal media-only send still creates visible placeholder:** For a mediaUrl with no text, the Signal sender resolves the media kind and sets `message` to `<media:image>` or `<media:${kind}>`, including `<media:audio>` for audio.
  - file: [extensions/signal/src/send.ts:162](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/extensions/signal/src/send.ts#L162)
  - command: `nl -ba extensions/signal/src/send.ts | sed -n '120,225p'`
  - sha: [d228463120a5](https://github.com/openclaw/openclaw/commit/d228463120a5057f7ccf9a5c31af62fe4a922088)
- **Placeholder is sent as the Signal RPC message body:** The computed `message` is assigned into the RPC params before attachments are added and the `send` RPC request is issued, so the placeholder remains part of the outbound Signal payload.
  - file: [extensions/signal/src/send.ts:188](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/extensions/signal/src/send.ts#L188)
  - command: `nl -ba extensions/signal/src/send.ts | sed -n '120,225p'`
  - sha: [d228463120a5](https://github.com/openclaw/openclaw/commit/d228463120a5057f7ccf9a5c31af62fe4a922088)
- **Signal reply adapter passes empty captions into the same sender:** Signal reply delivery calls `sendMessageSignal(target, caption ?? "", { mediaUrl, ... })`, so audio-only or media-only replies with an empty caption flow into the placeholder-producing code path.
  - file: [extensions/signal/src/monitor.ts:344](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/extensions/signal/src/monitor.ts#L344)

- **Core audio-only hook behavior does not fix Signal-specific send body:** Core tests cover keeping audio-only spoken text out of the media caption for generic plugin delivery, but Signal's channel-local sender still replaces the resulting empty text with `<media:${kind}>`.
  - file: [src/infra/outbound/deliver.test.ts:942](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/infra/outbound/deliver.test.ts#L942)

- **Issue discussion supports remaining bug scope:** The issue body reports the Signal Android visible caption and Ryce's comment identifies the same boundary-state leak, recommending a zero-width-space style fix. Current main has not applied that fix in the Signal sender.
