## Summary

Keep open. Current main still lacks a task-level timeout in the command lane queue, so the permanent lane-jam failure mode remains possible. The PR is relevant, but it still needs maintainer-guided rework because the fixed fallback timeout conflicts with current agent and cron runtime budgets.

## Best Possible Solution

Keep the PR open, but require a revised core fix: add a queue-level stuck-task safety net that is configurable per enqueue, wire agent and cron callers to their existing timeout policies including disabled/long budgets, and add regression tests for lane unblocking, stale reset behavior, diagnostics, and long/disabled timeout cases.

## Risks / Open Questions

- Closing this PR would leave the current command queue without automatic recovery for never-settling lane tasks.
- Merging the PR as-is could introduce false `CommandLaneTaskTimeoutError` failures for legitimate long-running agent or cron work unless caller timeout budgets are preserved.
- A bare `Promise.race` timeout can unblock the lane while the original task continues running, so the final fix should account for abort/cancellation and overlap risks.
