## Summary

Keep open. Current main now documents that Codex OAuth cannot satisfy embedding requests and offers memory-core/Active Memory alternatives, but the reported `memory-lancedb` behavior remains: its config still requires an embedding API key, the manifest requires `embedding.apiKey`, and runtime constructs an OpenAI embeddings client directly from that key before auto-recall/auto-capture hooks can run.

## Best Possible Solution

Keep the issue open unless maintainers decide the documented memory-core plus Active Memory path is the intended replacement. The concrete product path is either to retrofit `memory-lancedb` onto the memory embedding adapter surface so it can use supported non-OpenAI/OAuth-compatible providers, or to document `memory-lancedb` as API-key-only and provide an explicit OAuth-compatible recipe using `memory-core`/`active-memory` plus local, GitHub Copilot, Bedrock, or other supported memorySearch providers.
