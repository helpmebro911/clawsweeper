## Summary

Keep open. Current main partially covers the WorkingDirectory side by defaulting macOS gateway install plans to the state directory, but it still does not emit `ProcessType=Interactive` in the generated LaunchAgent plist. This PR is also paired with the same-author open issue it closes, and the later merged #72616 explicitly left this work separate, so conservative cleanup should not close it as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase it on current main, keep or rework the `ProcessType=Interactive` plist change with focused `src/daemon/launchd.test.ts` coverage, and align any `WorkingDirectory` behavior with the already-landed state-dir default rather than overwriting it with `/`. The paired open same-author issue should remain linked until this PR is either merged or explicitly declined by a maintainer.

## Risks / Open Questions

- Applying the PR unchanged after current main needs care because main now uses the OpenClaw state directory as the macOS gateway working directory default; changing fallback behavior to `/` could regress that newer intent in direct template/rewrite paths.
- `ProcessType=Interactive` changes launchd resource classification for the gateway. It is a documented plist key, but maintainers should confirm that foreground-class scheduling is the desired product policy for the macOS gateway.
- No supply-chain or secret-handling regression was visible in the supplied diff; the remaining review risk is functional macOS launchd behavior, not CI/dependency security.
