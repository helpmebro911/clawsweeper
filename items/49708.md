## Summary

Current main still appears to preserve the reported bug: session path resolution canonicalizes through the real sessions directory, while doctor orphan detection compares those resolved references against logical directory-entry paths. The linked PR #49796 directly tracks a fix, but it is still open/unmerged in the provided GitHub context, so the issue should remain open.

## Evidence

- **Session resolver returns realpath-backed paths:** `resolvePathWithinSessionsDir` computes `realBase` with `fs.realpathSync` and returns `path.resolve(realBase, normalized)`, so a symlinked sessions directory still produces the physical target path rather than the caller-visible logical path.
  - file: [src/config/sessions/paths.ts:185](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/config/sessions/paths.ts#L185)
  - command: `nl -ba src/config/sessions/paths.ts | sed -n '1,330p'`
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Persisted sessionFile uses resolver output:** `resolveAndPersistSessionFile` writes the `resolveSessionFilePath` result back into the session store as `sessionFile`, so the realpath-backed value can still be persisted.
  - file: [src/config/sessions/session-file.ts:30](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/config/sessions/session-file.ts#L30)

- **Doctor orphan comparison still mixes canonical and logical paths:** Doctor builds `referencedTranscriptPaths` from `path.resolve(resolveSessionFilePath(...))`, then compares it to `path.resolve(path.join(sessionsDir, entry.name))` for files listed under the logical sessions directory.
