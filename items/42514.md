## Summary

Keep open. Current `main` does not show a shipped fix for the reported `:19001` -> `ws://localhost:18789` behavior. The browser Control UI still documents that it should be served by the gateway and talk to the gateway on the same port, while the UI client still contains a special Vite-dev fallback that rewrites a page served on another port to `ws://<host>:18789`. I found no corresponding main-branch change after `v2026.4.22` that removes or alters that fallback, and the Android gateway changes since that release are unrelated loopback/TLS validation hardening.

## Evidence

- **Control UI docs still describe same-port gateway serving:** The docs say the Control UI is served by the gateway and speaks directly to the gateway WebSocket on the same port, with the default local URL on `:18789`. That matches the issue's expectation that a dashboard page should connect to the gateway endpoint it was served with, not silently jump to another port.
