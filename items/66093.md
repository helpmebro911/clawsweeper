## Summary

Keep open. Current main still matches the reported auto-reply CLI fallback auth-profile gap: the fallback CLI path can execute `google-gemini-cli` while auth scoping drops the profile id, and CLI session reuse still invalidates when stored and current `authProfileId` differ.

## Best Possible Solution

Keep this issue open. The maintainer fix should update the auto-reply fallback CLI path to use provider/runtime-aware auth planning, or otherwise resolve a `google-gemini-cli`-appropriate auth profile before CLI session reuse, while preserving the cross-provider auth isolation from #1156. Add regression coverage for `google-gemini-cli` fallback session reuse and stale `cliSessionBindings` where `authProfileId` is missing on one side.
