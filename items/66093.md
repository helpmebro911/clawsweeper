## Summary

Keep open. Current main still has the reported contract gap in the auto-reply fallback path: provider-scoped auth intentionally drops an authProfileId when fallback switches providers, while CLI session reuse still invalidates on any stored/current authProfileId mismatch. The issue is recent and includes concrete logs plus a stale-binding follow-up, and I did not find current coverage that resolves google-gemini-cli fallback session reuse.

## Best Possible Solution

Keep this issue open and fix the CLI auth/session contract. The likely maintainer path is to resolve a provider-appropriate auth profile for fallback CLI providers without reintroducing #1156 cross-provider auth leakage, then add regression coverage for google-gemini-cli fallback reuse and stale `cliSessionBindings` where authProfileId is missing on one side.

## Evidence

- **Issue discussion is concrete and recent:** The supplied issue body gives reproduction steps for google-gemini-cli as a fallback provider, OAuth auth profiles, repeated `cli session reset: provider=google-gemini-cli reason=auth-profile` logs, and tiny prompt sizes. The follow-up comment adds a stale `cliSessionBindings` case where a binding created without authProfileId later invalidates after the provider becomes primary.
- **Provider-scoped fallback auth still drops cross-provider profile ids:** `resolveProviderScopedAuthProfile` forwards the run auth profile only when the selected provider auth-aliases to the primary provider; otherwise it returns undefined authProfileId/authProfileIdSource. That preserves the #1156-era behavior that can drop a google-gemini-cli profile when reached from a different primary provider.
