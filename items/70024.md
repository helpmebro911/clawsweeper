## Summary

Current main still has the reported channel stop-timeout behavior: the timeout branch marks the account running, returns before deleting store entries, and startChannel skips accounts with an existing task. PR #70056 directly tracks the same proposed fix, but it is still open and unmerged, so the bug is not implemented on main yet.

## Best Possible Solution

Keep the issue open until #70056 or an equivalent gateway fix lands. The best path is to make the stop-timeout state truthful and recoverable while preserving protection against duplicate live channel tasks, then update the current server-channels regression test and health-monitor expectations accordingly.
