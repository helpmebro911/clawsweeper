## Summary

Keep open. Current main has useful Slack Socket Mode mitigations for this symptom, but I do not see a source-backed fix that makes the reported valid-token `account_inactive` startup failure connect successfully. The remaining core issue is still actionable: identify which Slack Web API call returns `account_inactive` in this multi-bot startup path and decide whether OpenClaw should handle it differently from a real inactive account.

## Evidence

That means current main is designed to fail/skip on this error, not reconcile a valid-token false positive.
