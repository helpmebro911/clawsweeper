## Summary

Keep open. Current main supports the corrected Google Gemma 4 model id `gemma-4-26b-a4b-it`, but it still accepts the reported `gemma-4-26b` as a forward-compatible Gemma model, normalizes it unchanged, and would send it to Google as `models/gemma-4-26b`. The issue discussion identified the likely correct id, and the reporter said the bad id came from terminal setup/selection, so the remaining maintainer work is validation, aliasing, or setup/model-picker cleanup rather than closing as user error.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Google provider/setup surface: verify whether onboarding, model selection, `/models set`, or docs can lead users to `gemma-4-26b`; replace that choice with `gemma-4-26b-a4b-it`, or add provider-owned validation/alias handling that fails before dispatch with a clear hint. Add regression coverage for the reported `gemma-4-26b` path separately from existing corrected-id Gemma 4 coverage. Review #68996 independently for native-routing behavior, but do not close this issue from that PR unless it merges and also resolves the reported bad-id path.
