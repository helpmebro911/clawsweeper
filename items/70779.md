## Summary

Keep open. The report is recent, concrete, and current main still has the likely regression surface: every OpenAI-compatible streaming request sends `stream_options.include_usage`, even though the compat layer knows generic custom providers may not support streaming usage and preserves explicit false overrides. Related usage-accounting items implemented the behavior that likely exposed this provider-specific Gemini regression; they do not supersede this issue.

## Best Possible Solution

Keep #70779 open and treat it as a focused core transport compatibility bug. The best path is to make `openai-completions` request shaping honor `compat.supportsUsageInStreaming` or add a documented opt-out for custom/proxy backends, preserve streamed-usage behavior for known compatible local/native providers and explicit true overrides, and add regression coverage for a custom Gemini-compatible backend that streams content incrementally without `stream_options.include_usage`.

## Risks / Open Questions

- Changing the default too broadly could regress token accounting for custom providers that now rely on unconditional `include_usage`; the fix should preserve known-compatible local/native endpoints and explicit opt-ins.
- A live custom Gemini-compatible backend fixture would still be needed to prove whether `include_usage`, final SSE chunk handling, or another Gemini proxy behavior is the direct trigger.
