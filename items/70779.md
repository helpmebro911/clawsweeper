## Summary

Keep #70779 open. The report is recent, concrete, cross-surface, and not covered by the related ClawSweeper matches. Current main still unconditionally sends `stream_options.include_usage` for OpenAI-compatible completions, including custom proxy-style providers, while the existing `supportsUsageInStreaming` compat flag is computed but not used to omit that payload field. The stream parser will only produce UI/channel incremental text when upstream chunks contain `choice.delta.content`, so the reporter's no-intermediate-chunks logs remain a plausible current regression rather than an implemented fix.

## Best Possible Solution

Keep the issue open and treat it as an OpenAI-compatible completions transport regression. The best path is a targeted regression test for a custom Gemini-compatible OpenAI endpoint that loses incremental chunks when `stream_options.include_usage` is sent, then a narrow opt-out or capability gate, likely honoring `compat.supportsUsageInStreaming: false` or an explicit per-model payload option, while keeping `include_usage` enabled for known-compatible providers. Verify through the shared stream parser plus one channel/control-UI delivery path so Feishu is not incorrectly blamed.
