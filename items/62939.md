## Summary

Keep open. Current main has meaningful partial prompt-injection hardening for web/external content, inbound metadata, special-token sanitization, and owner-based tool policy, but it still does not implement the central request: a shared trust-boundary contract for non-owner current message bodies and generic/open-world tool results, plus stable system-prompt guidance for those boundaries. The linked implementation PR #62973 is closed unmerged.

## Best Possible Solution

Keep this issue open and implement a core-owned, tested trust-boundary strategy: add stable system-prompt guidance before the prompt-cache boundary, wrap or otherwise label non-owner current message bodies during prompt assembly, classify/wrap open-world tool results at a shared transport or tool-result boundary, preserve existing external-content sanitizer behavior, and add regression coverage across owner/non-owner ingress plus OpenAI, Anthropic, and Google payload shapes.

## Risks / Open Questions

- Security-relevant behavior remains split across source-specific wrappers, metadata labeling, sanitizer helpers, and owner-only tool policy rather than a shared trust-boundary contract for all current non-owner messages and open-world tool results.
- A future implementation needs careful provider-shape tests to avoid double wrapping, cache instability, broken multimodal tool results, or weakening the existing `EXTERNAL_UNTRUSTED_CONTENT` marker and special-token sanitizer.
