## Summary

Current main still appears to reject the reported legacy `plugins.entries.memory-core.config.dreaming.storage: "both"` shape before runtime dreaming normalization can use it. I found the current object-only manifest schema and the plugin config validation path, but no memory-core doctor contract or raw config migration that converts the legacy string into `{ mode, separateReports }`. The issue should stay open for a targeted compatibility migration or explicit friendly diagnostic.

## Best Possible Solution

Keep #70407 open and fix it in the memory-core/plugin config compatibility path. The conservative implementation is to add a memory-core doctor/compat contract or core legacy migration that rewrites legacy string storage values (`inline`, `separate`, `both`) to `{ mode: <value>, separateReports: false }` before plugin schema validation, plus regression coverage for `readConfigFileSnapshot`/`openclaw doctor --fix` on an old `dreaming.storage` string. If maintainers intentionally do not want auto-migration, add a path-specific validation hint so users are not left with a generic `must be object` error.
