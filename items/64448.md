## Summary

Current main still does not provide a stable Feishu media-send idempotency key on the image upload/send path, so the reported duplicate image-send behavior remains plausible and unresolved.

## Best Possible Solution

Fix this in the Feishu plugin media path. Verify the Feishu/Lark message idempotency contract, thread a stable outbound send retry key through the CLI/gateway/channel outbound context into sendMediaFeishu, pass it to message.create/reply where supported, and add regression coverage that simulates a duplicate retry/reconnect without blocking intentional later sends.

## Risks / Open Questions

- The reporter suggested idempotent_id, while the pinned SDK exposes message-level uuid; maintainers should verify current Feishu/Lark API semantics before choosing the exact field and scope.
- A local in-flight lock keyed only by media URL could suppress intentional repeated sends; a stable per-send retry key is the safer implementation direction.
