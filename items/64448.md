## Summary

Current main still lacks a stable Feishu outbound media idempotency key on the image upload/send path, so the reported duplicate image sends during retry/reconnect remain plausible and unresolved.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin media send path. Add or propagate a stable outbound send idempotency key through CLI/gateway/channel outbound context into Feishu sendMediaFeishu, map it to the Feishu/Lark message idempotency field for image create/reply where supported, and add regression coverage that simulates a duplicate retry/reconnect while still allowing intentional later sends of the same image.

## Risks / Open Questions

- The reporter's exact provider field name should be verified against the current Feishu/Lark API and the pinned @larksuiteoapi/node-sdk version before implementing.
- A local in-flight lock keyed only by mediaUrl could suppress intentional repeated sends; a per-send retry key propagated to the provider is the safer direction.
- The bug is intermittent and tied to reconnect/retry timing, so the regression test should simulate duplicate provider retry/reconnect behavior rather than rely on live network timing.
