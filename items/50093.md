## Summary

Keep open. Current main has a related WhatsApp reconnect mitigation for recent Baileys `append` upserts, but it does not implement the requested cursor/history backfill for the outage window and intentionally drops append history older than 60 seconds before reconnect. The BlueBubbles analog #66721 is useful context, but it does not supersede the remaining WhatsApp-specific work.

## Evidence

- **Issue discussion asks for cursor/history backfill:** The issue and comments describe storing a per-chat last-seen cursor and querying WhatsApp/Baileys history on reconnect, with a suggested gap window such as the last 2 hours; that is broader than simply accepting recent post-reconnect append events.
- **Current implementation only consumes Baileys upsert events:** The inbound monitor handles `messages.upsert` only when the upsert type is `notify` or `append`; there is no explicit history fetch in this path.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:639](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/extensions/whatsapp/src/inbound/monitor.ts#L639)

  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **Append catch-up is capped to recent messages:** For `append` upserts, current main uses `APPEND_RECENT_GRACE_MS = 60_000` and skips messages whose timestamp is older than 60 seconds before the socket connected.
