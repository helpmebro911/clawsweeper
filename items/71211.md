## Summary

Keep open. Current main still has the reported foreground exec path: stdout/stderr is accumulated into `outcome.aggregated`, then copied directly into the tool result text and `details.aggregated`. I also found the same raw-output pattern in background `process` poll/log surfaces. Existing redaction is present for logs/transcript persistence/tool parameter previews, but I did not find a redaction pass on live exec result content before it becomes model-visible tool output.

## Best Possible Solution

Keep this open for a security fix in the exec/process output pipeline. The likely implementation is to run `redactSensitiveText` with the existing logging redaction config over stdout/stderr before any live tool result/update/notification is emitted, and to store redacted `aggregated`/`tail` values in tool result details exposed to the agent. Add regression coverage for env-style secrets, bearer/API keys, provider-key prefixes, failed exec reasons, foreground success, background poll/log, and LLM-boundary/provider conversion. If maintainers want raw diagnostic access, make it an explicit owner-only opt-in rather than the default tool-result behavior.

## Risks / Open Questions

- A redaction pass can over-redact legitimate command output that users expect the agent to inspect exactly; maintainers should decide whether any raw reveal path must be explicit, privileged, and auditable.
- The fix needs to cover foreground exec, running updates, background `process` poll/log, completion notifications, details metadata, transcript persistence, and provider conversion so secrets do not survive through a side path.
