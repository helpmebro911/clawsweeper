## Summary

Keep open. Current main still has the reported exec-output exposure path: foreground exec returns `outcome.aggregated` directly as tool-result text/details, live updates use the same session output tail, and background `process poll/log` can also return stored aggregate output. Existing redaction helpers cover logs, diagnostic payloads, approval display, and tool-result details stripping, but I did not find a secret-redaction pass on exec stdout/stderr content before it reaches tool-result text.

## Best Possible Solution

Keep this issue open and implement a central exec-output redaction boundary, preferably before any stdout/stderr aggregate becomes tool-result content. Reuse or harden the existing `src/logging/redact.ts` patterns, apply it consistently to foreground results, live updates, process poll/log, finished-session aggregates exposed to agents, and add regression tests using fake env assignments, bearer/API-key/JWT-style strings, failure output, and background logs. If maintainers want an explicit raw-output escape hatch, it should be operator-controlled and documented rather than the default agent-visible path.

## Risks / Open Questions

- A naive blanket redaction pass could hide legitimate command output or interfere with debugging, so the fix should define a clear policy for default redaction, explicit raw reveal, and how details/session logs are stored.
- The same mitigation likely needs coverage for foreground results, live `onUpdate` events, background `process poll`, background `process log`, failed outcomes, and persisted session history.
