## Summary

Keep open. Current main still returns exec stdout/stderr-derived `aggregated` text directly into model-visible tool results and details, and the background `process` plus node-host paths expose the same raw output shape. The provided related reports cover env inheritance, write-deny semantics, and exec security config, but do not supersede this output-redaction issue.

## Best Possible Solution

Keep #71211 open for a core security fix. The best path is to apply the existing `redactSensitiveText`/resolved redaction machinery at the exec-output boundary before tool-result text/details are emitted, and preferably before `session.aggregated` is persisted or exposed through `process poll`/`log` and node-host exec. Add focused regression tests using fake env-style/API-key/Bearer/PEM outputs for foreground success, foreground failure, background poll/log, live update/command-output events, and node-host payloads.

## Risks / Open Questions

- A fix must preserve useful command output while masking credential-like values consistently across foreground exec, background process polling/logging, node-host exec, live updates, persisted details, and command-output events.
- Hard-blocking secret-file reads would be a larger product/security policy decision; the narrower minimum fix is a central output redaction pass with regression coverage for stdout, stderr, failure reasons, details, and process sessions.
