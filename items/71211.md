## Summary

Current main still returns stdout/stderr-derived exec output through foreground results, runtime outcomes, background process polling/logging, command-output events, and node-host formatting without applying the existing secret redaction helper at those exec-owned boundaries. This is a recent security report with no merged implementation candidate in the provided context, so it should stay open.

## Best Possible Solution

Keep this issue open and patch the exec-owned output boundary. Apply configured redactSensitiveText or an equivalent resolved redaction pass before stdout/stderr-derived text or details leave exec: foreground buildExecForegroundResult, runtime outcome construction, process poll/log/list details, command-output events, notify/follow-up summaries, and node-host formatNodeRunToolResult. Add focused regression tests for env-style assignments, bearer tokens, and common API-key formats while preserving existing output caps and transcript persistence safeguards.

## Risks / Open Questions

- Secrets printed by an allowed exec command can still enter agent-facing live tool result text/details before transcript persistence redaction or detail capping can help.
- Background process poll/log, command-output events, notify/follow-up summaries, and node-host exec formatting share the same stdout/stderr-derived exposure surface.
- A fix needs focused tests so redaction catches env-style assignments, bearer tokens, and common API-key formats without breaking output caps, truncation, and process polling behavior.
