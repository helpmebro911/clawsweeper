## Summary

Keep #44997 open. Current main has a base-hash conflict guard that can prevent silent clobbering, but it does not implement this PR's read/transform/write-under-lock behavior for concurrent `openclaw agents add`; the command still builds `agents.list` from a pre-read snapshot and writes through `replaceConfigFile`. The PR discussion is also useful: the initial lock-only flaw was identified, then the author added `updateConfigFile` and addressed subsequent review concerns, so this remains plausible work rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open for maintainer review, but rebase/rework it against current main as a narrow config transaction for `agents add`: acquire a per-config lock or equivalent serialization, re-read the latest source config inside the critical section, apply the agent/binding transform there, and preserve existing base-hash, plugin-install, include, env-ref, runtime-snapshot, destructive-write, and hardened-directory behavior. Add regression coverage showing several concurrent non-interactive `openclaw agents add` calls all persist distinct agents without silent loss or stale-conflict failures.
