## Summary

Keep open. Current main still does not implement the requested `sendMessageTelegram` adapter guard, even though shared silent-reply helpers and Slack parity behavior exist. Higher-level Telegram delivery normalization covers many final-reply paths, but direct Telegram send/action paths can still pass a bare `NO_REPLY` into `sendMessageTelegram`; PR #68304 is only an open fix attempt, not shipped main behavior.

## Best Possible Solution

Keep #38603 open and review or replace PR #68304 with a focused Telegram plugin change: import `isSilentReplyText` from `openclaw/plugin-sdk/reply-chunking`, return a synthetic suppressed `TelegramSendResult` before Telegram API work when `text` is exactly silent and there is no `mediaUrl` or inline keyboard, and add `extensions/telegram/src/send.test.ts` regression coverage proving `api.sendMessage` is not called. Track any broader streaming-buffering fix separately.
