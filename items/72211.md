## Summary

Keep PR #72211 open. Current main still has only generic dashboard connectivity and gateway service troubleshooting; it does not include the PR's specific local-dashboard-after-update recovery flow with `lsof`/`curl`, LaunchAgent reload recovery context, and stale browser state guidance. The proposed commands map to real CLI/runtime behavior, and the existing review feedback is an actionable docs wording fix rather than a reason to close.

## Best Possible Solution

Keep the PR open and ask for a small docs revision: keep the useful local-dashboard recovery runbook, but split the flow so users first check whether `127.0.0.1:18789` is already serving the dashboard, then run `openclaw gateway restart` or `openclaw gateway start` only if the gateway is not listening or not serving HTML. This belongs in `docs/gateway/troubleshooting.md` near the existing dashboard connectivity and gateway service sections.
