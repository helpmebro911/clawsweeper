## Summary

Keep #70578 open. Current main has important guardrails for the reported data-loss class, including source-shape preservation, pre-validation persistence, suspicious write rejection, last-known-good recovery, and the #67557 startup hash guard. But the issue’s requested root config compatibility and bundled plugin CLI allow-list behavior are not fully resolved: the root schema remains strict, tests still assert unknown config fields are rejected, and plugin-owned CLI commands still return a `plugins.allow` exclusion when the parent plugin is not allowed.

## Best Possible Solution

Keep the issue open, or split it into narrower follow-ups. The best product path is to add a focused regression test around `openclaw plugins install` or an equivalent config write with unknown/legacy source-owned top-level data, then decide explicitly whether root unknown keys should be preserved via passthrough/catchall or rejected with a non-destructive, repairable error. Separately, decide whether bundled plugin CLI command surfaces should be exempted from `plugins.allow`, auto-seeded into restrictive allow lists, or kept blocked but documented as intentional plugin policy.
