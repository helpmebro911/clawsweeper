## Summary

Keep open. Current `main` still bypasses standard `HTTP_PROXY`/`HTTPS_PROXY` handling for the OpenAI Responses WebSocket path, so the reported fallback-to-HTTP behavior remains plausible and matches the implementation.

## Evidence

- **Fallback log matches the report:** The websocket stream path still logs `WebSocket connect failed ... falling back to HTTP`, matching the issue symptom when the WS connect attempt fails.
  - file: [src/agents/openai-ws-stream.ts:786](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/openai-ws-stream.ts#L786)
  - command: `nl -ba src/agents/openai-ws-stream.ts | sed -n '770,792p'`
  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **WebSocket manager does not apply provider proxy config:** `OpenAIWebSocketManager` resolves provider request policy, but when opening the socket it only passes handshake headers, an optional debug proxy agent, and TLS client options. It does not translate `request.proxy` or ambient `HTTP_PROXY`/`HTTPS_PROXY` into the websocket agent.
  - file: [src/agents/openai-ws-connection.ts:413](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/agents/openai-ws-connection.ts#L413)

  - sha: [11cffb230013](https://github.com/openclaw/openclaw/commit/11cffb230013c2d8bd1e0e18da04b0b985b8abd5)
- **Only debug-proxy env vars create a websocket proxy agent:** The websocket proxy helper only enables an `HttpsProxyAgent` when `OPENCLAW_DEBUG_PROXY_ENABLED` and `OPENCLAW_DEBUG_PROXY_URL` are set. Standard proxy env vars are only written by the debug-proxy helper, not read as inputs for normal websocket dialing.
  - file: [src/proxy-capture/env.ts:79](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/src/proxy-capture/env.ts#L79)
