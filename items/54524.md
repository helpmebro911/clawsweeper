## Summary

Keep open. Current main still allows memory search to index both workspace memory files and session transcripts, while the bundled session-memory hook still writes session-derived Markdown into workspace memory. The related PR #40618 is open and unmerged, and even as described it is only a partial mitigation rather than shipped duplicate-indexing guidance or provenance-aware dedupe.

## Best Possible Solution

Keep this issue open. The lowest-risk maintainer path is to add public docs guidance in memory search/config and session-memory hook docs: use sources ["memory"] when relying on the bundled session-memory hook, or disable/redirect the hook when indexing transcripts directly. A stronger code fix would add explicit session provenance or cross-source dedupe keys so OpenClaw can avoid duplicate embedding/index rows without suppressing real long-term memory notes.

## Risks / Open Questions

- A docs-only fix should be precise: the hook summary and raw transcript chunks overlap semantically but are not guaranteed byte-identical, so guidance should not hide legitimate durable memory notes.
- MMR can reduce repetitive top results when enabled, but it is optional and does not avoid duplicate indexing or duplicate embedding work.
- Review #40618 as related work, but it should not close this issue unless merged and paired with guidance or provenance-aware dedupe.
