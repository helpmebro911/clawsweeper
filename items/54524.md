## Summary

Current main still permits the reported overlap. The session-memory hook writes recent transcript content into indexed workspace memory files, while memory search can separately index both `memory` and `sessions`; current result merging only deduplicates identical chunk IDs, not near-duplicate content across source/path boundaries. The public docs still describe these paths separately without warning users to choose one or configure separation.

## Best Possible Solution

Keep this issue open. The lowest-risk maintainer path is to document the supported choice in the public memory-search and hooks docs: use `sources: ["memory"]` when relying on the bundled `session-memory` hook, or disable/redirect that hook when indexing transcripts via `sources: ["memory", "sessions"]`. A code fix would need an explicit provenance marker or cross-source dedupe key so it avoids hiding legitimately distinct long-term memory notes while also avoiding duplicate embedding work.
