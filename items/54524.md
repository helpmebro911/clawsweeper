## Summary

Current main still allows the reported overlap: session-memory writes session-derived Markdown into workspace memory, memory search can index both `memory` and `sessions`, and dedupe is keyed to chunk ids rather than cross-source conversation provenance. The open related PR #40618 is a partial mitigation for configurable hook output, not a merged fix or docs guidance for this issue.

## Best Possible Solution

Keep this issue open. The lowest-risk maintainer path is to add public docs guidance in memory search/config and hook docs: use `sources: ["memory"]` when relying on the bundled `session-memory` hook, or disable/redirect the hook when indexing transcripts directly. A stronger implementation would add explicit session provenance or cross-source dedupe keys so duplicate embedding work can be avoided without hiding real long-term memory notes. Review #40618 as a partial mitigation for configurable hook output, but it should not close this issue until merged and paired with the duplicate-indexing guidance or provenance-aware behavior.
