## Summary

Keep open. Current `main` still hardcodes the WhatsApp idle watchdog timeout to 30 minutes, does not thread any config-derived `messageTimeoutMs` into `monitorWebChannel`, and the public config/docs only expose `web.heartbeatSeconds` and `web.reconnect.*`. The report is recent, coherent, and there is already a cross-referenced open PR (#63939), so there is no strong basis to auto-close it.

## Evidence

- **Gateway start path does not pass timeout config:** The WhatsApp gateway start path calls `monitorWebChannel(...)` with tuning that only includes `statusSink` and `accountId`; no `messageTimeoutMs` or `watchdogCheckMs` is sourced from config here.
  - file: [extensions/whatsapp/src/channel.ts:296](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/whatsapp/src/channel.ts#L296)

- **Monitor still uses hardcoded 30 minute default:** `monitorWebChannel` still resolves `messageTimeoutMs` as `tuning.messageTimeoutMs ?? 30 * 60 * 1000`, so current main still defaults to a 30 minute idle reconnect unless a caller injects an override.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:183](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/whatsapp/src/auto-reply/monitor.ts#L183)
