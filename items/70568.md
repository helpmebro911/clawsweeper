## Summary

Keep open. Current main still has the Telegram-sourced ambiguous-account gap this PR targets: when `turnSourceChannel` is `telegram` and no account binding resolves, the matcher accepts any eligible Telegram account. The PR is useful and has relevant discussion, but it should be revised before merge because review comments flagged a real `mode: "both"` regression risk and the latest PR diff also changes shared cross-channel session-binding behavior that current main still explicitly tests.

## Best Possible Solution

Keep the PR open and ask for a narrower revision: update `extensions/telegram/src/exec-approvals.ts` so Telegram-sourced requests with no resolved account binding are treated as ambiguous in multi-account configs, add the PR's regression coverage for multi-account rejection, single-account pass-through, and explicit `turnSourceAccountId`, and avoid applying explicit target-account gating to `mode: "both"` session/native delivery. Split or separately justify the shared `resolveApprovalRequestChannelAccountId` cross-channel fallback change before merging.
