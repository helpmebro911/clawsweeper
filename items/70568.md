## Summary

Keep open. Current main still has the Telegram-sourced unbound-account hole this PR is trying to address: the ambiguity guard only applies to non-Telegram turn sources, so a Telegram approval request with no bound account can still match every eligible Telegram account. The PR is not ready to close or merge as-is because the provided review comments identify a real `mode: "both"` regression risk, and the branch also includes unrelated local check runtime changes.

## Best Possible Solution

Keep the PR open, but require a narrower revision: fix `extensions/telegram/src/exec-approvals.ts` so Telegram-sourced approvals without a concrete `turnSourceAccountId` or session binding are treated as ambiguous unless exactly one eligible Telegram account can handle them; honor explicit `approvals.exec.targets[].accountId` without disabling session/native delivery in `mode: "both"`; add focused regression tests for multi-account fan-out, single eligible account pass-through, explicit account binding, and `both` mode; remove or split the unrelated local-heavy-check runtime change.
