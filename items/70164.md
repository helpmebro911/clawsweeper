## Summary

Keep open. The reporter’s follow-up narrows the symptom to OpenClaw Gateway itself emitting tool/lifecycle/chat-final frames without any assistant output, and current main still has code paths where a lifecycle end can produce a `chat` final with no `message` when no assistant text was buffered. There are related fixes for throttled/segmented assistant text and empty-turn retry handling, but I did not find source/test evidence that the exact no-assistant-event WebSocket path is fully fixed on main.

## Best Possible Solution

Keep the issue open and add a focused regression around `chat.send` or the embedded runner/Gateway bridge: simulate a tool result followed by an empty assistant stop for a direct session, verify current behavior, and ensure the WebSocket client receives either recovered assistant text or an explicit error final rather than a bare `state:"final"`. If the failure reproduces, the likely fix belongs at the runner-to-Gateway boundary so synthetic incomplete-turn/error payloads are reflected into the live chat event stream before lifecycle finalization.
