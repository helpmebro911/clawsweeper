## Summary

Keep open. The issue is recent, reporter supplied concrete OpenClaw gateway websocket logs showing tool/lifecycle events followed by a bare `chat` final, and current `main` still has a source-level path that emits `state:"final"` with no `message` whenever no assistant text event populated the live chat buffer. Adjacent fixes exist for throttled deltas, silent tokens, MiniMax thinking defaults, and some empty-response retries, but I did not find source/test/release evidence that the reported MiniMax/tool-run path is fixed on main.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add a focused regression around an embedded `chat.send`/gateway run where a MiniMax-like provider emits tool events and lifecycle end without assistant text, then decide the intended product behavior: retry through the empty-response path, surface an incomplete-turn error, or guarantee a final chat message from the terminal reply payload instead of emitting a bare `state:"final"`. The likely investigation areas are `src/agents/pi-embedded-runner/run/incomplete-turn.ts`, MiniMax stream/result normalization, and `src/gateway/server-chat.ts` finalization.
