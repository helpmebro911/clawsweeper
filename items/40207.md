## Summary

Current main still generates the macOS Gateway LaunchAgent stdout/stderr paths under the OpenClaw state directory, so a symlinked `~/.openclaw` can still be written into `StandardOutPath` and `StandardErrorPath`. The issue remains a concrete, actionable daemon/CLI bug.

## Best Possible Solution

Keep this issue open. Fix the CLI-managed macOS LaunchAgent path in `src/daemon`: write `StandardOutPath` and `StandardErrorPath` to a launchd-safe non-symlinked user log location such as `~/Library/Logs/openclaw`, or canonicalize the generated log directory before writing the plist. Add regression coverage for symlinked state directories and plist regeneration, update runtime/status/docs hints, and consider an install-time health check that reports launchd EX_CONFIG failures clearly.
