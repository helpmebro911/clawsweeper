## Summary

Keep #71770 open. Current main has reactive model fallback, auth-profile cooldown/rotation, and usage/quota reporting surfaces, but it does not implement proactive token/quota/cost-threshold routing before a model attempt. The request remains a concrete product/runtime feature rather than a duplicate, shipped behavior, or pure ClawHub item.

## Best Possible Solution

Keep the issue open for product/design work in the model selection and fallback path. A solid implementation would add a documented config surface for provider/model token, quota, or cost thresholds; evaluate usage snapshots and expected request cost before selecting each candidate; integrate with runWithModelFallback without weakening existing cooldown/error fallback; emit user-visible lifecycle notifications; and define recovery/reverse-failover semantics when quota windows reset. If maintainers prefer a plugin-first approach, the best path is to first add a generic pre-call model-routing/plugin API rather than closing this as already possible in ClawHub.

## Risks / Open Questions

- Provider usage data is uneven across providers; some expose quota windows, some expose only post-response usage, and some expose no reliable quota API.
- Automatic provider switching can increase costs or surprise users unless thresholds, fallback order, reverse-failover behavior, and notifications are explicit and auditable.
- This overlaps existing reactive rate-limit/cooldown fallback and usage/cost reporting, so the implementation should be designed as a coherent routing policy rather than another one-off fallback branch.
