## Summary

Keep this PR open. Current main still has the misleading plugin allowlist fallback for arbitrary missing primary commands, and the latest PR patch remains a useful implementation candidate, though it needs rebase/adaptation to the current `run-main-policy.ts` split.

## Best Possible Solution

Keep this PR open and finish maintainer review. The best path is to rebase or apply the same behavior on current main: gate `plugins.allow` / `plugins.entries.<id>.enabled=false` diagnostics on a real bundled plugin CLI root, map aliased roots such as `voicecall` back to their owning plugin id, keep unknown commands such as `run` on the normal unknown-command path, and preserve the denylist/probe-failure regression coverage.
