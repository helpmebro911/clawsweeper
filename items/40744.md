## Summary

Keep this PR open. Current main at f0b327cf686e3a957fef8c94985ee6509953a3b7 still does not strip own non-array `required` values from the shared tool-schema normalization path, and the Gemini cleaner still only drops empty `required` arrays while copying other `required` values through. The PR remains a focused implementation candidate for the Bailian GLM-5 failure discussed in #40568, but it needs a rebase/adaptation to the current `src/agents/pi-tools-parameter-schema.ts` layout and a maintainer decision on the existing Gemini key-copy hardening note.

## Best Possible Solution

Keep this PR open and request a rebase onto current main. The best implementation is to strip own non-array `required` fields in the shared tool-parameter schema normalization path before provider-specific cleaning, preserve valid `required` arrays, add regression tests for top-level strict OpenAI-compatible/Bailian-style schemas and nested Gemini schemas, and explicitly resolve whether the Gemini dynamic-key hardening is in this PR or a tracked follow-up.
