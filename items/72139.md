## Summary

Keep #72139 open. Current main still reconnects the Control UI browser client after AUTH_TOKEN_MISMATCH when no device-token retry was available, and a current test explicitly preserves that behavior. The Gateway chat.history path also still performs synchronous transcript reads and JSON processing in the WebSocket request path, so the reported Slack Socket Mode starvation remains plausible. A prior close for the narrower dashboard token-mismatch item covered the cached device-token retry path, but it does not resolve this no-retry/stale-token plus chat.history blocking report.

## Best Possible Solution

Keep the issue open and fix it in core/UI. Treat shared-token AUTH_TOKEN_MISMATCH as terminal unless a bounded trusted device-token retry is actually available, surface a clear Control UI re-auth/token prompt, and move or bound expensive chat.history transcript work so history loads cannot starve Slack/Telegram/WebSocket pongs. Add regression coverage for stale-token reconnect suppression and a concurrency/performance guard around chat.history.
