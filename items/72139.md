## Summary

Keep #72139 open. Current main only stops AUTH_TOKEN_MISMATCH reconnects after a device-token retry budget has been consumed; it still intentionally reconnects after a token mismatch when no retry was attempted. The `chat.history` path also remains a synchronous gateway handler that reads/parses transcript files and performs JSON byte-budget work inline, so the reported event-loop starvation remains plausible and is not fixed on main.

## Best Possible Solution

Keep the issue open and fix it in core. The likely direction is to make Control UI auth mismatch terminal after the bounded trusted device-token fallback is unavailable or exhausted, surface an explicit re-auth/token-drift prompt, and move or chunk the `chat.history` transcript read/projection/JSON-budget work so one Control UI request cannot starve other gateway WebSocket connections such as Slack Socket Mode.
