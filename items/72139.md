## Summary

Keep this issue open. Current main still matches the reported core failure mode: Control UI treats AUTH_TOKEN_MISMATCH as recoverable unless a cached-device-token retry has already failed, and chat.history still synchronously reads and parses the full transcript before applying visible-message limits. The provided related items are adjacent but not canonical duplicates, and the existing Codex comment already identified the same remaining work.

## Best Possible Solution

Fix this in core Control UI and Gateway history handling. Make AUTH_TOKEN_MISMATCH terminal when no trusted cached-device-token retry can be attempted, and after the single retry is exhausted. Separately, make chat.history non-starving by avoiding synchronous full-transcript reads/parses on the event loop before limits are applied, then add regression coverage for the no-retry auth loop and large-history responsiveness while channel timers remain healthy.
