## Summary

Keep #60542 open. Current main has lifecycle persistence and restart recovery for the narrower running-row corruption family, but I did not find a shipped reconciliation path for this issue’s distinct terminal-but-stale main-session row versus newer transcript case. The related PR #60642 was closed unmerged in the provided GitHub context, and the current source does not contain its described transcript-mtime stale-row check.

## Best Possible Solution

Keep the issue open and add a targeted main-session reconciliation path before row reuse or during gateway startup. For a main session whose persisted lifecycle fields are terminal but older than the referenced transcript, OpenClaw should either clear/advance lifecycle state consistently or mint a fresh session row under the same key, using the locked session-store update path. The fix should also ensure reset/recovery paths do not carry stale `status`, `startedAt`, `endedAt`, or `runtimeMs` into a fresh session, and should add a regression test for the #60542 stale-terminal-row/newer-transcript scenario.
