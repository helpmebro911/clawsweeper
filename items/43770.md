## Summary

Keep this PR open, but it should not merge as-is. The linked bug is real, and the second PR commit addressed Greptile's test-preservation gap, but the implementation only lets unknown top-level plugin-entry keys pass base Zod validation. Current runtime validation, normalization, docs, and plugin API all route plugin-owned settings through `plugins.entries.<id>.config`, so this patch would likely turn a clear startup error into ignored or unvalidated configuration.

## Best Possible Solution

Request changes and keep this PR open for maintainer review. The best fix is to preserve an explicit core plugin-entry shape, decide any documented entry-level fields such as `apiKey` and `env` intentionally, and keep plugin-owned settings under `plugins.entries.<id>.config` with regression coverage proving base validation, plugin schema validation/defaulting, normalization, and `api.pluginConfig` runtime delivery. If maintainers want legacy top-level compatibility, implement an explicit migration/normalization path with warnings rather than raw passthrough.

## Risks / Open Questions

- Merging as-is would likely convert actionable `Unrecognized keys` failures into silent ignored configuration for misplaced top-level plugin settings.
- Broad top-level passthrough would expand the accepted core config surface outside plugin-owned `configSchema` validation, which can hide unsupported fields or misplaced credentials.
- There is still a documented metadata mismatch around `plugins.entries.*.apiKey` and `plugins.entries.*.env` appearing in generated/help metadata while `PluginEntrySchema` and `PluginEntryConfig` do not accept them; that should be resolved deliberately rather than by accepting every unknown key.
