## Summary

Keep this PR open. Current main still does not add the requested media-server headers beyond `X-Content-Type-Options: nosniff`; the PR is a focused security hardening candidate, but it targets an older handler shape and needs maintainer review of the exact cache and CSP policy before merge.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase/adapt the hardening onto `createMediaRequestHandler`, add shared assertions for success and error responses, and make an explicit media-server cache/CSP decision that accounts for active-content attachment handling and one-shot URL fallback behavior.

## Risks / Open Questions

- `Cache-Control: no-store` may make one-shot hosted media URLs more fragile when an integration, such as Tlon, falls back to the original temporary media URL after upload/fetch failure.
- The proposed CSP includes `style-src 'unsafe-inline'`; current main's active-content attachment handling may allow a stricter policy, so the exact CSP should be deliberately chosen instead of merged verbatim.
- The PR currently targets the older `attachMediaRoutes` implementation and only asserts the added headers on the 200 response path, leaving current error-path coverage to be added during rebase.
