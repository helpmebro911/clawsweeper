## Summary

Keep PR #50506 open. Current main still does not implement the requested media-server response headers, although the media handler has since been rewritten and now mitigates active HTML/SVG/XML content by serving it as opaque attachment bytes. The PR is stale against the current handler shape and has a real cache-policy review concern, but the central hardening request remains plausible and is not implemented or superseded.

## Best Possible Solution

Keep this open and rework it against `src/media/server.ts` on current main: add deliberate media-route-specific security headers where they still improve the standalone temporary media server, preserve the existing active-content download behavior, expand tests to cover success and error paths, and explicitly decide whether to omit or alter `Cache-Control` because of one-shot media URL consumers such as Tlon.

## Evidence

It does not set `X-Frame-Options`, `Referrer-Policy`, `Content-Security-Policy`, or `Cache-Control` on this path.

## Risks / Open Questions

- The PR patch targets an older Express-style media route (`attachMediaRoutes`) while current main uses `createMediaRequestHandler`, so it needs rebasing/reimplementation rather than direct merge.
- The proposed `Cache-Control: no-store` may conflict with current one-shot media URL behavior and should be resolved in product/security review before landing.
