## Summary

Keep this PR open. Current main still lacks the requested media-server response headers beyond `X-Content-Type-Options: nosniff`, and the PR is a focused security-hardening candidate, but it is stale against the current handler shape and needs maintainer policy review for CSP and caching before merge.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase or rework the hardening onto `createMediaRequestHandler`, add shared assertions for successful and error responses, and make an explicit media-server CSP/cache policy decision that accounts for active-content attachment handling and one-shot URL fallback behavior.

## Risks / Open Questions

- The PR has no observed dependency, workflow, lockfile, package-script, generated/vendor, or publishing-surface change in the provided diff, so the security review concern is policy behavior rather than supply-chain execution.
- `Cache-Control: no-store` may regress one-shot hosted media URLs for integrations that fall back to the original temporary URL after upload/fetch errors.
- The proposed CSP includes `style-src 'unsafe-inline'`; current active-content attachment handling may support a stricter CSP, so this should be deliberately chosen rather than merged verbatim.
- The patch targets the old `attachMediaRoutes` shape and only adds a 200-path assertion; current main needs coverage for success and error paths on `createMediaRequestHandler`.
