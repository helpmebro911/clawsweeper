## Summary

Keep #70191 open. Current main does not implement `/health/deep` or an equivalent authenticated HTTP model-contract probe. The shipped HTTP probe surface is limited to `/health`, `/healthz`, `/ready`, and `/readyz`; readiness details only expose ready/failing/uptime channel-readiness state. The existing health RPC summary also lacks model/provider/fallback/routing-error fields. `openclaw models status --json` exposes resolved model and fallbacks, but that is a CLI/config-auth surface, not the requested cheap in-process Gateway HTTP endpoint with live auth/readiness semantics and last-routing/error counters. Plugin HTTP routes exist, but closing as ClawHub would be premature because the requested root health path, readiness-auth policy, and active routing/error state are core Gateway design questions.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is to decide whether this belongs as a core Gateway probe or a plugin-owned route after clarifying the contract. If core, implement a small authenticated HTTP endpoint in the Gateway probe surface, reuse or formalize the readiness-detail auth policy, return resolved primary provider/model/fallbacks without inference, explicitly document status semantics, and add focused `server-http.probe.test.ts` coverage plus public docs. If maintainers prefer plugin ownership, first expose any missing SDK/runtime surface for active model routing decisions and recent gateway error state, then steer the external endpoint to a plugin-specific route rather than closing this proposal outright.
