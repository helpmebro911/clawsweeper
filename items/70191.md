## Summary

Keep open. Current `main` does not implement a native HTTP deep-health/model-contract probe: gateway probes are limited to `/health`, `/healthz`, `/ready`, and `/readyz`, and the readiness payload only exposes `ready`, `failing`, and `uptimeMs`. Public docs instead point operators to CLI/WS health surfaces and local `models status`, which do not satisfy the requested external HTTP verification use case. The issue is only 3 days old and is actionable: the reporter supplied a concrete schema, auth model, and linked overlay prior art, so it is not stale, incoherent, or ready for a conservative cleanup close.

## Evidence

- **Gateway HTTP probe surface on main:** `handleGatewayProbeRequest()` only recognizes `/health`, `/healthz`, `/ready`, and `/readyz`. For readiness, it returns either the full `ReadinessChecker` result or `{ ready }`; liveness stays shallow with `{ ok: true, status }`. There is no `/health/deep` route or model/provider/fallback payload on current `main`.
