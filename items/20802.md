## Summary

Keep open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as observe-only/void hook surfaces, so this PR's central behavior is not implemented. The open #61485 overlaps the `llm_input`/`llm_output` part, but it does not supersede this PR's broader adapter-side `after_tool_call` result-mutation proposal.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split or redesign adapter-side `after_tool_call` result mutation as a separate plugin API decision that preserves exactly-once hook dispatch and validates the replacement result shape.

## Risks / Open Questions

- This PR overlaps #61485 for `llm_input`/`llm_output`; maintainers should avoid landing two independent hook-contract implementations.
- Adapter-side `after_tool_call` mutation conflicts with the current exactly-once dispatch model unless the API defines how it composes with the subscription handler, run/session/tool-call context, and persisted tool results.
- The PR's `after_tool_call` result type is `unknown` and the adapter path casts it to `AgentToolResult<unknown>` without validation, so malformed plugin output could fail later in the tool-result path.
- Prompt and `systemPrompt` mutation is security-sensitive and should be reviewed against `allowPromptInjection=false`, audit logging, payload cloning, image re-resolution, failover/block semantics, and original-content leakage.
