## Summary

Keep #20802 open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the PR still contains unique unresolved `after_tool_call` result-mutation work that is not implemented or fully superseded by the related focused `llm_input`/`llm_output` PR discussion.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the broader direction or lands a canonical replacement. Let #61485 or an equivalent focused PR handle mutable `llm_input`/`llm_output`, then make a separate maintainer decision on whether plugin authors should get adapter-side `after_tool_call` result mutation or use `tool_result_persist` / runtime tool-result middleware instead.
