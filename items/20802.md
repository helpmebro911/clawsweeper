## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observe-only hooks, and the runner/adapter paths do not apply returned prompt, response, block, or tool-result mutations. Open #61485 is a narrower candidate for the `llm_input`/`llm_output` portion, but it does not supersede this PR's broader adapter-side `after_tool_call` proposal.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API decision that preserves exactly-once hook dispatch and defines how it composes with `tool_result_persist` and native/Codex tool bridges.

## Risks / Open Questions

- This PR overlaps open #61485 for mutable `llm_input`/`llm_output`; maintainers should choose one canonical focused implementation rather than merging both independently.
- Adapter-side `after_tool_call` result mutation conflicts with the current exactly-once dispatch model and needs an explicit API/ordering decision around the subscription-handler path, `tool_result_persist`, and native/Codex tool bridges.
- Allowing plugins to mutate `systemPrompt`, prompts, assistant text, and model-visible tool results is security-sensitive. Review should cover `allowPromptInjection` policy, audit logging, payload cloning, result validation, failover/block handling, and embedded/CLI/native harness parity before landing.
