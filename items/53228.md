## Summary

Keep this issue open. Current main has useful Web Push support from merged #44590, a low-level `openclaw gateway call` escape hatch, `chat.inject`, `openclaw system event`, and `openclaw nodes notify`, but it still does not provide a first-class `openclaw notify <message>` command or a production-named admin-scoped Gateway/UI broadcast method for visible notifications to connected Control UI sessions without using the agent/event queue.

## Best Possible Solution

Keep this issue open and either add a production-facing `openclaw notify <message>` wrapper or explicitly design a Gateway method such as `ui.notify` or `notifications.broadcast`. The implementation should define the required operator scope, decide whether connected Control UI WebSocket clients receive an in-app toast event in addition to Web Push, reuse the Web Push infrastructure where appropriate, and add CLI docs plus focused gateway/CLI tests for the shell-script workflow.
