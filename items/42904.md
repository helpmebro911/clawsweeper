## Summary

Current `main` does not implement the midnight log-file fallback for `openclaw channels logs`. The command still reads only the configured file for today, while the repo docs and logging helper code show daily rolling logs and an existing fallback resolver used elsewhere. This PR is not obsolete enough to close conservatively.

## Evidence

- **`channels logs` still reads only today's configured file:** `channelsLogsCommand` calls `getResolvedLoggerSettings().file` directly and passes that path into `readTailLines(...)`, with no fallback to the active prior-day rolling log when today's file does not exist yet.
  - file: [src/commands/channels/logs.ts:88](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/commands/channels/logs.ts#L88)
  - command: `nl -ba src/commands/channels/logs.ts | sed -n '72,100p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **OpenClaw docs still describe daily rolling gateway logs:** The public logging docs say the gateway writes rolling files under `/tmp/openclaw/openclaw-YYYY-MM-DD.log`, which matches the date-boundary failure mode described in the PR.
  - file: [docs/logging.md:22](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/docs/logging.md#L22)
  - command: `nl -ba docs/logging.md | sed -n '8,28p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Main already has a resolver for missing rolling log files:** `src/logging/log-tail.ts` contains `resolveLogFile(file)`, which stats the configured file and, if it is a rolling log path and missing, scans the directory and chooses the newest matching log file by mtime.
  - file: [src/logging/log-tail.ts:26](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/logging/log-tail.ts#L26)
  - command: `nl -ba src/logging/log-tail.ts | sed -n '18,52p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **The fallback resolver is wired for `openclaw logs`, not `openclaw channels logs`:** `readConfiguredLogTail()` uses `resolveLogFile(getResolvedLoggerSettings().file)`, but `openclaw channels logs` still dispatches to the separate `channelsLogsCommand` path, so the existing fallback behavior does not cover this surface.
  - file: [src/logging/log-tail.ts:155](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/src/logging/log-tail.ts#L155)
