## Summary

Current main already implements the requested hard wall-clock timeout path for Active Memory. The plugin still uses the blocking memory sub-agent rather than a new single-shot mode, but the issue explicitly offered hard timeout as an acceptable alternative, and main now races the sub-agent against an abort-sentinel timeout so the reply path returns `status=timeout` near `config.timeoutMs` even if the sub-agent does not cooperatively stop.

## Best Possible Solution

Close this issue against current main and keep the shipped hard-timeout implementation. If maintainers still want a lower-latency single-shot Active Memory mode, that should be tracked as a narrower enhancement separate from this timeout-enforcement bug.
