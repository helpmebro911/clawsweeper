## Summary

Keep #28025 open. Current main still has no Slack `replyOnToolUse` or equivalent tool-use-gated threading policy; Slack reply placement is still driven by `replyToMode`, existing thread context, and delivery plans created before tool callbacks run.

## Best Possible Solution

Keep #28025 open. If accepted, add a default-false typed Slack plugin config option, carry a reliable per-turn tool-use signal into Slack reply planning before streaming or final delivery commits to a target, preserve explicit thread replies plus existing `replyToMode` and `thread.inheritParent` semantics, and add Slack tests for DMs, channels, existing threads, reply-mode interactions, streaming, and tool/progress callbacks.
