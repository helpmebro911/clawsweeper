## Summary

Keep this PR open. Current main still does not expose `OPENCLAW_AGENT_ID` or `OPENCLAW_SESSION_KEY` on the requested `exec`, `acp-client`, or `tui-local` child-process paths, and the PR remains a plausible implementation candidate, but it needs rebasing and maintainer security/product review before landing.

## Best Possible Solution

Keep this PR open and ask for an update. The best implementation is a shared runtime env resolver that injects deterministic `OPENCLAW_AGENT_ID` plus optional `OPENCLAW_SESSION_KEY`, threads it through exec direct/sandbox/PTY, gateway approval follow-up execution, ACP client spawn helpers in the current `client-helpers.ts` layout, and TUI local-shell commands. It should add focused tests for explicit-agent precedence, session-key fallback, missing/legacy context behavior, and gateway approval execution, with a maintainer security/product decision on exposing session keys to subprocess environments.

## Risks / Open Questions

- The PR exposes `OPENCLAW_SESSION_KEY` to spawned shells/bridge processes. Session keys are routing/session identifiers, not obviously bearer tokens, but they may include agent/channel/account/peer context and can leak through child process environments, logs, crash reports, or shell debugging. Maintainers should explicitly approve this public env contract.
- The PR is stale against current main: ACP spawn-env logic now lives in `src/acp/client-helpers.ts`, while the fetched patch modifies the older helper location in `src/acp/client.ts`.
- The fetched patch still does not update the gateway approval follow-up `runExecProcess` call, so approved gateway commands can miss explicit agent identity unless that path is included before landing.
