## Summary

Keep this PR open. Current main still does not expose `OPENCLAW_AGENT_ID` or `OPENCLAW_SESSION_KEY` for the requested exec, ACP client, and TUI local-shell child-process paths, and the PR remains useful but stale/incomplete. It needs a rebase plus fixes for the gateway approval/safeBin propagation gap and explicit security review of exposing session keys to child environments.

## Best Possible Solution

Keep this PR open and ask for a rebase/update. The best path is a shared runtime env resolver that injects deterministic `OPENCLAW_AGENT_ID` plus optional `OPENCLAW_SESSION_KEY`, then threads it through exec direct/sandbox/PTY, gateway approval and safeBin execution, ACP client spawn helpers, and TUI local-shell commands. Add focused tests for explicit-agent-id precedence, session-key fallback, missing-context behavior, and the gateway approval path, plus docs for the public environment contract and a maintainer security check on session-key exposure.

## Risks / Open Questions

- The PR intentionally exposes `OPENCLAW_SESSION_KEY` to spawned child processes. Maintainers should confirm session keys are routing context rather than bearer/capability secrets on every covered path before landing.
- The PR diff does not update `src/agents/bash-tools.exec-host-gateway.ts`, so approved gateway/safeBin-style execution can still miss explicit `agentId` propagation if landed as shown.
- The PR's exec helper falls back through `resolveAgentIdFromSessionKey`; missing or legacy session contexts need explicit tests so they do not incorrectly report `main` when a caller had a different explicit agent identity.
- The PR is stale against current main and likely needs a rebase, especially around the ACP client helper split.
