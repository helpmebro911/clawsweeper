## Summary

Current main still leaves the concrete webchat/openclaw-tui outbound delivery hook gap open. The webchat `chat.send` path builds its own reply dispatcher and uses a local `deliver` callback that records/broadcasts replies, but it does not route delivery through `deliverOutboundPayloads`, does not install the existing `message_sending` before-delivery hook wrapper, and does not emit `message_sent`. openclaw-tui uses the same gateway `chat.send` surface in gateway mode, and local embedded TUI runs through the internal/session-only agent command path rather than a hook-emitting channel delivery path. Some related hooks now have coverage elsewhere (`agent_end`, guarded transcript writes), so the issue text is broader than the remaining proven gap, but the central outbound lifecycle problem is not fixed. Related #12584 is closed unmerged, and #50126 remains an open umbrella, not a shipped fix for this specific surface.

## Best Possible Solution

Keep #70928 open and fix the gateway chat outbound path. The likely implementation is to put webchat/openclaw-tui replies through the same lifecycle observer as shared outbound delivery, or minimally add a `beforeDeliver` plus post-delivery hook layer around the `chat.send` dispatcher with session/run/channel context. Add regression tests for `message_sending` rewrite/cancel and `message_sent` success/failure for webchat and openclaw-tui clients, and document any intentionally unsupported hook/channel combinations until #50126's broader unification is complete.
