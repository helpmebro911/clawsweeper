## Summary

Keep #68944 open. Current main and the latest release still contain the post-challenge client path shape discussed in the issue: `connect.challenge` calls `sendConnect()` inside a broad message-handler `try`, while synchronous device-signing/public-key failures are not surfaced through `onConnectError` and can be logged only as a debug parse error. The related PR #68614 was explicitly acknowledged in the issue discussion as an unrelated Google Chat timeout PR, and a later commenter reported the same handshake-hang pattern on Linux with OpenClaw 2026.4.22.

## Best Possible Solution

Keep the issue open for a focused core Gateway/CLI fix. The likely maintainer path is to narrow `handleMessage()` so only JSON/frame parsing is caught as parse failure, wrap the synchronous device-auth work in `sendConnect()` with an explicit `onConnectError`/close path, wire CLI gateway calls to surface connect errors clearly, and add a regression test where signing or public-key derivation throws after `connect.challenge` so the command fails fast instead of waiting for gateway/outer timeouts.
