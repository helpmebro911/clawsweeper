## Summary

Current main does not yet implement the requested cross-run OpenAI prompt-cache affinity for isolated recurring cron jobs. Cron now has a stable job-derived session key, but isolated runs still force a fresh run session id, and OpenAI prompt_cache_key is still derived from the session id rather than the stable cron job key.

## Evidence

- **Isolated cron uses a stable job session key only at the cron service boundary:** Gateway cron builds sessionKey as `cron:${job.id}` for isolated jobs and passes it to `runCronIsolatedAgentTurn`; this is useful routing state, but it is not itself the OpenAI prompt_cache_key path.
  - file: [src/gateway/server-cron.ts:335](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/gateway/server-cron.ts#L335)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '318,360p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Isolated cron still forces a new run session id:** Cron run preparation calls `resolveCronSession` with `forceNew: input.job.sessionTarget === "isolated"`, then records `runSessionId` from the resulting session entry. `resolveCronSession` creates a fresh UUID whenever forceNew is true.
  - file: [src/cron/isolated-agent/run.ts:482](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/cron/isolated-agent/run.ts#L482)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '460,510p' && nl -ba src/cron/isolated-agent/session.ts | sed -n '121,150p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Embedded cron execution receives both ids, but sessionId remains the fresh run id:** The embedded agent invocation passes `sessionId: params.cronSession.sessionEntry.sessionId` and `sessionKey: params.agentSessionKey`, so the stable cron key is available separately but does not replace the per-run session id.
  - file: [src/cron/isolated-agent/run-executor.ts:161](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/cron/isolated-agent/run-executor.ts#L161)
  - command: `nl -ba src/cron/isolated-agent/run-executor.ts | sed -n '110,214p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **OpenAI prompt_cache_key still comes from options.sessionId:** Direct OpenAI Responses sets `prompt_cache_key` to `options?.sessionId` when cache retention is enabled; the completions path likewise sets `params.prompt_cache_key = options.sessionId` when supported. No current code path found maps cron sessionKey/job id into that field.
  - file: [src/agents/openai-transport-stream.ts:866](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/agents/openai-transport-stream.ts#L866)

- **Docs still require reusing the same prompt_cache_key for OpenAI cache reuse:** The prompt caching troubleshooting docs say low OpenAI cacheRead should check that the same `prompt_cache_key` is reused for turns that should share a cache, matching the issue's requested stable cross-run affinity.
