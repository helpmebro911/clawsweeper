## Summary

Keep this PR open. Current main still does not implement the proposed `google-genai` provider, `google-genai-sdk` auth mode, or `models.providers.google-genai.vertexai` config contract, and the PR remains a plausible but unfinished implementation path for linked Google Vertex/ADC work.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either revise it so unified Google GenAI auth fits the current bundled Google plugin/provider architecture and existing transport behavior bar, or supersede it with a narrower PR that fixes the remaining `google-vertex` ADC/proxy behavior in the existing provider path with focused tests and docs.

## Risks / Open Questions

- The PR changes shared model-provider config/auth contracts, including making `baseUrl` optional and adding a keyless `google-genai-sdk` mode, so it needs maintainer design review rather than automated cleanup closure.
- The keyless auth mode must be propagated through every API-key requirement path, not just embedded runs; current main has relevant `requireApiKey` call sites in `/btw` and media/tool runtime paths.
- A new SDK transport must preserve the existing Google transport behavior for system prompts, tools, generation controls, cancellation, proxy/TLS/request guards, usage, and finish metadata.
- Importing `@google/genai` from core without a root dependency declaration would be a build/package-resolution risk unless the dependency ownership is changed deliberately.
