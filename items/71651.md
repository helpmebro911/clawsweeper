## Summary

Keep #71651 open. Current main still lacks the PR's cron reload invalidation behavior: forced timer reloads merge persisted runtime state and then skip recompute, with no comparison against the previous in-memory schedule or enabled state. The PR is also not superseded by the linked issue closure; the provided context shows the PR is open/unmerged and contains maintainer follow-up commits addressing review feedback.

## Best Possible Solution

Keep the PR open and proceed with maintainer review/landing if it still applies cleanly: preserve the semantic schedule comparator, invalidate stale `nextRunAtMs` only when schedule or enabled state changes, keep the regression coverage for cron/every/at and key-order-preserving cases, and update the cron docs/changelog in repo style. Track broader non-timer external edit reload behavior separately under the existing #41614 path.
