## Summary

Keep PR #71651 open. Current main still has the stale reload shape the PR is trying to fix: timer ticks force-reload the cron store with `skipRecompute: true`, `ensureLoaded` replaces `state.store` without comparing the reloaded job schedule/enabled fields against the previous in-memory job, and due-job collection then trusts the merged `state.nextRunAtMs`. The PR discussion is also active and maintainer-reviewed: the original bot review concerns were addressed by maintainer fixups and additional tests/docs in the PR head, but those changes are not on current `main`.

## Best Possible Solution

Keep the PR open and either land the maintainer-fixup version after normal review/CI, or merge an equivalent main-side fix that compares reloaded schedule/enabled inputs semantically and clears stale `nextRunAtMs` only when scheduling inputs actually changed. Preserve the PR’s negative coverage for unchanged and key-reordered schedules so reload formatting changes do not skip pending slots.

## Evidence

There is no previous-jobs snapshot, semantic schedule comparison, or clearing of `hydrated.state.nextRunAtMs` when `schedule`/`enabled` changed on disk.
