## Summary

Keep this PR open. Current main still has the cron reload path described by the PR: timer ticks force-reload the split cron store with recomputation skipped, `loadCronStore` merges sidecar runtime state by job id, and due-job checks still trust the merged `state.nextRunAtMs` without invalidating it when scheduling inputs change. The PR discussion also shows maintainer fixups that addressed the review feedback with semantic schedule comparison, broader tests, docs, and changelog coverage.

## Best Possible Solution

Keep this PR open and either review/land its maintainer-fixup shape or explicitly supersede it with an equivalent merged follow-up. The best implementation is to invalidate stale `nextRunAtMs` after `loadCronStore` merges `jobs-state.json` runtime state but before due-job collection, using semantic comparison for the `at`, `every`, and `cron` schedule union plus `enabled`, while preserving unchanged and key-order-only pending slots. Carry forward the PR's regression tests and cron docs/changelog updates.
