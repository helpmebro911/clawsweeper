## Summary

Keep open. Current `main` still saves LINE inbound media into the temp directory, while sandbox staging only allows files rooted under OpenClaw's media directory, so the sandbox-access bug described by the PR still exists. The PR's approach is directionally correct, but the patch is stale because LINE now lives under `extensions/line/` and should be rebased onto the plugin-era code paths and SDK seams.

## Evidence

- **LINE downloader still writes inbound media to temp files:** `downloadLineMedia` on `main` still calls `buildRandomTempFilePath(...)` and writes the downloaded buffer with `fs.promises.writeFile(...)`, so inbound LINE media lands under the temp root instead of `~/.openclaw/media/inbound/`.
  - file: [extensions/line/src/download.ts:39](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/extensions/line/src/download.ts#L39)
  - command: `nl -ba extensions/line/src/download.ts | sed -n '1,140p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **LINE message handling has not adopted the filename/MIME-path part of the PR:** The LINE webhook handler still calls `downloadLineMedia(message.id, account.channelAccessToken, mediaMaxBytes)` without forwarding `message.fileName`, so the PR's `originalFilename` plumbing is not present on `main`.
  - file: [extensions/line/src/bot-handlers.ts:490](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/extensions/line/src/bot-handlers.ts#L490)
