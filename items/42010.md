## Summary

Keep open. Current main still has the LINE sandbox media-storage bug this PR is trying to fix: LINE downloads are written to a temp path outside the OpenClaw media directory, while sandbox staging only accepts local inbound media from the media directory. The PR remains a useful bug-fix candidate, but it is stale against the current `extensions/line/*` plugin layout and needs to be rebased through the plugin SDK media-store seam.

## Best Possible Solution

Keep this PR open as a valid bug-fix candidate. The best path is to rebase it onto the current LINE plugin: update `extensions/line/src/download.ts` to use `getMessageContentWithHttpInfo` and `saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename)` through `openclaw/plugin-sdk/media-store`, update `extensions/line/src/bot-handlers.ts` to pass `message.fileName` for LINE file messages, and add/update tests in `extensions/line/src/download.test.ts` and `src/media/store.test.ts` for sandbox-accessible storage, Content-Type propagation, filename MIME fallback, sanitization, and size-limit behavior.

## Risks / Open Questions

- The PR is stale against current main: its provided diff targets the old `src/line/*` layout, while LINE now lives under `extensions/line/*`.
- A rebased implementation must preserve the extension boundary by importing `saveMediaBuffer` from `openclaw/plugin-sdk/media-store`, not by reaching into core internals from plugin code.
- `message.fileName` is user-controlled; it is acceptable only if it continues to flow through `saveMediaBuffer` sanitization and has regression coverage for traversal-like names, unusual extensions, and oversized media.
- The dependency-backed switch to `getMessageContentWithHttpInfo` should be covered by unit tests around the SDK response shape, content-type fallback, and missing-header behavior.
