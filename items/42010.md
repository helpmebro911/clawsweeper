## Summary

Keep this PR open. Current main still writes LINE media downloads to the OpenClaw temp directory and exposes those paths to the inbound context, while sandbox staging only accepts local paths under the OpenClaw media directory. The PR is stale against the current bundled-plugin layout, but the underlying sandbox-access bug remains a valid focused fix candidate.

## Best Possible Solution

Keep this PR open as a valid stale bug-fix candidate. Rebase it onto the current plugin layout by updating extensions/line/src/download.ts to use getMessageContentWithHttpInfo and saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename) via openclaw/plugin-sdk/media-store, update extensions/line/src/bot-handlers.ts to pass message.fileName for LINE file messages, and add focused tests for sandbox-accessible storage, Content-Type propagation, filename-aware MIME fallback, filename sanitization, and streaming size-limit behavior.

## Risks / Open Questions

- The PR targets the obsolete src/line layout; current LINE code lives under extensions/line, so it cannot merge cleanly without a rebase.
- A rebased implementation must import saveMediaBuffer through openclaw/plugin-sdk/media-store, not by deep-importing core media internals from the LINE plugin.
- message.fileName is user-controlled. It is acceptable only if it continues through saveMediaBuffer sanitization and has regression coverage for traversal-like names, odd document extensions, generic ZIP-container document MIME fallback, and oversized media.
- Current saveMediaBuffer uses originalFilename for ID/extension fallback but does not pass originalFilename as detectMime({ filePath }), so a rebased fix should deliberately decide whether LINE needs MIME contentType accuracy or only a correct saved extension.
