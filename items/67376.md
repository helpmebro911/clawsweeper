## Summary

Keep this PR open. Current main has shipped partial MCP loopback hardening by deriving `senderIsOwner` from owner/non-owner bearer tokens and filtering owner-only tools, but it still does not implement this PR's per-backend token-bound scope model. `sessionKey`, `accountId`, and `messageProvider` are still read from mutable loopback request headers, backend spawn still injects those scope env vars, and the linked security issue remains open.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it with a narrower security patch for #64993. The best path is to rebase onto current main, preserve the shipped owner/non-owner bearer mitigation, add a per-backend token registry that binds `sessionKey`, `accountId`, `messageProvider`, and `senderIsOwner`, stop emitting scope headers/env vars, invalidate scoped tool-cache entries on token revocation, add negative tests for all spoofable scope fields, and resolve the review comments plus the unrelated plugin-loader diff before merge.
