## Summary

Keep #70347 open. Current main still has the reported cron outer-timeout gap: `executeJobCoreWithTimeout` aborts and rejects the race with a plain timeout error, while scheduled/manual timeout result handling can drop `sessionId`/`sessionKey`. The gateway session-store finalization path exists, but it is lifecycle-event driven, and I found no cron-side terminal lifecycle emission for the outer timeout path. Related #71756 was closed as a broad superseded umbrella while explicitly leaving #70347 as the narrower canonical cron-timeout tracker.

## Best Possible Solution

Keep the issue open and implement a cron-owned timeout finalization path. Preserve the active isolated cron session identity once the session is prepared, then emit or synthesize a terminal lifecycle error/end event with timeout metadata through the existing gateway lifecycle persistence path. Add regression coverage for a timeout after session start that proves the matching sessions.json row becomes terminal immediately, without relying on watchdog repair.

## Risks / Open Questions

- A fix should be idempotent with any later embedded-runner lifecycle event so timeout handling does not double-finalize or downgrade a stronger terminal state.
- The cron base session key and per-run session identity can differ; the implementation needs to preserve the actual row key before the outer timer can win.
