## Summary

Keep open. Current main still has the cron outer timeout race abort and reject the run without emitting an agent lifecycle terminal event, while the existing gateway session persistence path only finalizes sessions when lifecycle start/end/error events arrive. The newer #71756 is a broader related report, but it references #70347 as a distinct timeout subcase rather than clearly superseding this focused bug.

## Best Possible Solution

Implement the focused cron timeout fix in core: when the outer cron wall timer fires after an isolated session has started, preserve the active run session identity and emit or synthesize a terminal lifecycle event with timeout metadata so the existing gateway lifecycle persistence updates `sessions.json` immediately. Add a regression test that proves the matching session row gets a terminal status and `endedAt` without waiting for watchdog/cleanup repair. Keep #71756 as the broader lifecycle umbrella if maintainers want, but do not close this narrower issue until that behavior is covered.
