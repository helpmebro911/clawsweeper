## Summary

Keep this PR open. Current main still has the Telegram model-picker no-override fallback bug: `resolveTelegramSessionState()` routes the callback to the correct agent and builds the provider/model list for that agent, but when there is no stored override or persisted model it still uses `agents.defaults.model` instead of the routed agent default, so the picker can miss the checkmark for `agents.list[].model`.

## Best Possible Solution

Keep this PR open for maintainer review. The best implementation is this patch or an equivalent one that computes the routed agent default in `resolveTelegramSessionState()`, uses it for the final no-override fallback, keeps the regression test for an agent default that differs from the global default, and validates the Telegram extension before merge. A small cleanup could store the resolved default once to avoid the duplicate resolver call noted in review.
