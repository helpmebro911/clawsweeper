## Summary

Keep this PR open. Current main has already implemented the Feishu configured-binding path through the newer registry-backed conversation-binding architecture, but the PR is not obsolete: `acp.defaultChannels` is still absent, catch-all ACP bindings are still intentionally rejected/skipped, and QQ Bot inbound handling still does not resolve or initialize configured ACP bindings. The author also linked a companion QQ Bot plugin-side PR in the discussion, which remains useful context for the remaining QQ Bot work.

## Best Possible Solution

Keep this PR open as stale-but-useful work to rebase or split. Port the remaining QQ Bot work onto current main by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through the current conversation-binding runtime SDK, with route/provider tests and docs. Treat `acp.defaultChannels` and catch-all ACP bindings as a separate maintainer product/security decision because current main intentionally requires explicit `match.peer.id`.

## Risks / Open Questions

- Landing the PR as-is would likely conflict with the current registry-backed configured-binding architecture and current Plugin SDK subpath policy.
- `acp.defaultChannels` and catch-all bindings are broad routing controls: a small config change can send every conversation on a channel/account through a host-side external ACP CLI runtime, so this needs explicit product/security review and focused tests.
- The QQ Bot work now belongs in the bundled `extensions/qqbot` plugin on current main, not only in external-plugin snippets from the original PR discussion.
