## Summary

Keep #29384 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still does not expose Docker sandbox `--gpus`, `--ipc`, `capAdd`, `--device`, `--privileged`, or raw extra Docker create-arg support. The existing OpenShell `gpu` option is a separate plugin/backend path, and the related #53108 capAdd-only PR is open/unmerged in the provided context and would not satisfy the reporter's central Docker GPU/IPC request.

## Best Possible Solution

Keep the issue open until a maintainer-approved Docker sandbox extension lands. The best implementation path is a constrained, explicit core Docker config contract such as `agents.defaults.sandbox.docker.gpus` and a carefully reviewed `ipc` policy or dangerous override, not a broad raw-args passthrough by default. Update the TypeScript config type, strict Zod schema, resolver/hash behavior, runtime validation/audit messaging, Docker create-arg builder, generated schema/help/docs, and focused regression tests. Treat #53108 as related capAdd prior work only, not closure for GPU/IPC support.

## Risks / Open Questions

- The requested Docker flags can weaken sandbox isolation, especially `--ipc=host`, `--privileged`, devices, and raw pass-through args, so this needs explicit maintainer security/product review rather than an unconstrained passthrough.
- A capAdd-only change, including the related open #53108 described in the provided context, would still leave the reporter's stated `--gpus all` and `--ipc=host` workflow unresolved.
- OpenShell GPU support may be a workaround for some users, but it requires switching backend/plugin behavior and should not be treated as implementing Docker sandbox parameters.
