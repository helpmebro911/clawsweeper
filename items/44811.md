## Summary

Keep PR #44811 open. Current main still projects runtime-derived writes onto the stale runtime source snapshot before the inner write path sees the current disk snapshot, so the dmScope/data-loss scenario described by #15146 is still plausible on main. The PR is a focused fix with a regression test, but the bot review comments identify edge cases the contributor or maintainer should resolve before merge.

## Best Possible Solution

Keep the PR open and have maintainers review it as a focused config writeback bug fix. The best path is to update the writeback projection so runtime-derived writes use the current on-disk snapshot when it exists, preserve missing-file fallback behavior, avoid an unnecessary second read by threading the base snapshot where appropriate, and keep the dmScope stale-snapshot regression test.
