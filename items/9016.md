## Summary

Keep #9016 open. Current main has estimated cost surfaces, transcript aggregation, and `/usage cost`, but it still does not capture OpenRouter's provider-reported `usage.cost` / `cost_details` from OpenAI-compatible responses and expose that exact per-message cost to the agent runtime. The provided discussion already distinguishes this from #9696 and #7006, and current source still matches that assessment.

## Best Possible Solution

Keep #9016 open. The best implementation is to capture OpenRouter's raw provider usage at the OpenAI-compatible response/stream boundary, preserve provider-reported cost separately from OpenClaw's local pricing estimate, carry typed provider cost details and cache-write tokens through transcript/session usage and `EmbeddedPiAgentMeta`, then expose per-turn and cumulative values through `session_status` or a dedicated usage/cost tool. Telegram and other reply footers can render exact OpenRouter cost when available and fall back to existing estimates otherwise.
