## Summary

Current main still does not implement inter-session progress streaming for `sessions_send`. The tool remains fire-and-forget or synchronous `agent.wait`, timeout still returns `status: "timeout"`, A2A announce remains completion-only, and gateway event delivery still depends on WebSocket/session subscribers or the target session rather than an internal-RPC caller. Related ACP parent streaming exists, but it is scoped to `sessions_spawn({ runtime: "acp", streamTo: "parent" })`, not arbitrary `sessions_send` delegation.

## Best Possible Solution

Keep this issue open and implement an additive core session-event observation path. A good direction is a permission-checked `agent.events` long-poll/read API backed by bounded per-run buffering, plus either a `sessions_subscribe` tool or a `sessions_send` progress option. Tests should cover long-running `sessions_send`, timeout-vs-accepted semantics, cross-session event visibility, target-session routing isolation, and Telegram/topic-bound progress delivery.

## Risks / Open Questions

- A future implementation needs explicit session visibility and privacy checks so a caller can only observe runs it is allowed to inspect.
- The UX should distinguish successful delivery from wait timeout so long-running delegated work is not reported as a failed send.
