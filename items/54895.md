## Summary

Current main still does not implement inter-session progress streaming for `sessions_send`. The issue remains specific and actionable: `sessions_send` has only fire-and-forget or synchronous wait semantics, `agent.wait` can still return `timeout` after delivery, A2A announce remains final-only, and gateway event routing is still WebSocket/session-subscriber oriented rather than an agent-callable run-event stream. Adjacent work exists, including Task Flow/background task status and ACP `sessions_spawn` `streamTo: "parent"`, but those do not cover observing a long-running run initiated by `sessions_send`.

## Best Possible Solution

Keep this issue open. The best path is an additive core implementation: a permission-checked `agent.events` long-poll/read API backed by bounded per-run event buffering, plus either a `sessions_subscribe` tool or a `sessions_send` progress option. Also adjust `sessions_send` result semantics so successful delivery is not confused with wait timeout, and add regression coverage for long-running sends, event visibility, routing isolation, and Telegram/topic-bound delivery. This belongs in core gateway/session tooling rather than ClawHub because it requires a protected run-event observation contract.

## Risks / Open Questions

- A future implementation needs explicit authorization checks so one session can only observe runs it is allowed to inspect.
- Forwarded tool and assistant progress may expose sensitive tool args/results unless payloads are redacted or gated by verbosity policy.
- Messaging-channel progress forwarding needs throttling and delivery policy so Telegram/topic updates do not become noisy or rate-limit prone.
