## Summary

Current main still lacks the requested inter-session progress streaming for `sessions_send`. The tool supports fire-and-forget or synchronous wait, but not a subscription/progress stream; wait expiry can still return `status: "timeout"` after accepted delivery; A2A announce remains completion-only; and gateway event routing is still WebSocket/session-subscriber or target-session oriented rather than an agent-callable run-event stream. Adjacent Task Flow and ACP `sessions_spawn` parent-streaming work does not cover observing a long-running run initiated via `sessions_send`.

## Best Possible Solution

Keep this issue open. The best implementation path is an additive, permission-checked core run-event observation surface such as `agent.events` backed by bounded per-run event buffering, plus either a `sessions_subscribe` tool or an explicit `sessions_send` progress option. `sessions_send` should also separate accepted delivery from wait timeout, and coverage should include long-running sends, event visibility, routing isolation, ACP/Codex behavior, and Telegram/topic-bound delivery.

## Risks / Open Questions

- A future run-event observation API needs explicit authorization so one session cannot inspect unrelated runs.
- Forwarding tool args, results, or assistant deltas could expose sensitive intermediate data unless payloads are gated or redacted.
- Messaging-channel progress forwarding needs throttling and delivery policy, especially for Telegram topics and other rate-limited channels.
