## Summary

Keep this open. The report is concrete and current main has adjacent reliability improvements, but the central four-agent Telegram scenario is not proven fixed and several requested owner-layer changes remain unresolved: Telegram runner concurrency is still tied to agent maxConcurrent, the main command lane still uses the same global cap, LaunchAgent service env does not default UV_THREADPOOL_SIZE, and LLM provider transport does not expose the Telegram-style dnsResultOrder=ipv4first control.

## Best Possible Solution

Keep this as a current reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, command-lane queue wait, session-lock hold/acquire time, LLM connect/header/body/idle timing, abort reasons, provider retry behavior, and event-loop/threadpool pressure. Then fix the measured layer: decouple Telegram update processing from agent-run concurrency if polling is starved, add a generic configurable LLM network policy if Undici/DNS is implicated, tune fallback/timeout policy if attempts still burn too much time, and add regression coverage for fallback timeout behavior plus a focused concurrency/queue test for the reported all-agents-timeout pattern.
