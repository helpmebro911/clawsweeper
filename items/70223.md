## Summary

Keep open. Current main still has the missing launchd WorkingDirectory plumbing described in #70223: the LaunchAgent template only emits WorkingDirectory when supplied, the packaged gateway install path still returns only programArguments, and relative configured agentDir values still resolve against process.cwd(). Fresh setup now appears to avoid writing the reported relative default agentDir, but existing configs with a relative agentDir remain exposed to the launchd cwd=/ failure discussed in the issue comments.

## Best Possible Solution

Implement rather than close: ensure packaged macOS gateway service installs set a deterministic WorkingDirectory, preferably HOME or the OpenClaw state/config directory, before rendering the launchd plist; add regression coverage that the non-dev `openclaw gateway install` path produces a LaunchAgent with WorkingDirectory. As defense-in-depth, either anchor relative `agents.list[].agentDir` to the loaded config directory or add doctor/config repair guidance for existing relative agentDir configs.
