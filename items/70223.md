## Summary

Keep open. Current `main` still leaves a real launchd install gap: packaged service installs can write a plist with no `WorkingDirectory` because `resolveGatewayProgramArguments()` only sets `workingDirectory` for dev/bun branches, and the install path forwards that `undefined` value unchanged into the launchd plist writer. The issue body's broader claim that the default config now ships a relative `agentDir` is not supported by current `main`, which resolves the default agent directory under the state dir, so the exact reported failure mode may be narrower than stated. That mismatch argues against auto-closing, not for it.

## Evidence

- **Packaged gateway install still omits `workingDirectory`:** In `resolveCliProgramArguments()`, the `runtime === "node"` branch and the normal non-dev branch return only `programArguments`; only bun/dev branches populate `workingDirectory`. That means the packaged `openclaw ... install` path can still produce no working directory at all.
  - file: [src/daemon/program-args.ts:189](https://github.com/openclaw/openclaw/blob/24e9924d6abd93560fb7bf721909257de533e57a/src/daemon/program-args.ts#L189)

  - sha: [24e9924d6abd](https://github.com/openclaw/openclaw/commit/24e9924d6abd93560fb7bf721909257de533e57a)
- **Install plan forwards missing `workingDirectory` unchanged:** `buildGatewayInstallPlan()` destructures `{ programArguments, workingDirectory }` from `resolveGatewayProgramArguments()` and returns `workingDirectory` directly, with no fallback for packaged installs.
  - file: [src/commands/daemon-install-helpers.ts:257](https://github.com/openclaw/openclaw/blob/24e9924d6abd93560fb7bf721909257de533e57a/src/commands/daemon-install-helpers.ts#L257)

That aligns with current code inspection.
