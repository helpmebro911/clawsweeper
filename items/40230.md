## Summary

Keep this PR open. Current main still rejects symlinked workspace bootstrap files that resolve outside the workspace boundary and marks them missing, while #40210, #38622, #64472, and multiple PR comments show a concrete user-facing bug. The PR remains a plausible implementation candidate, but it changes security-sensitive boundary behavior and needs maintainer/security review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for explicit maintainer/security review or replacement. The best implementation is a narrow change in `src/agents/workspace.ts` that supports symlinked fixed top-level bootstrap files only, preserves the existing boundary-reader safety properties, adds regression tests for external absolute and relative symlinks plus failure cases, and updates the public workspace/system-prompt docs if the behavior is accepted. Consolidate #40210, #38622, #64472, and #38650 around the chosen patch once it lands or is rejected.

## Risks / Open Questions

- The PR intentionally creates an exception to the workspace boundary-path model, so maintainers should explicitly decide whether fixed top-level bootstrap symlinks may point outside the workspace.
- The latest PR patch reads through an fd after `realpath`, but an open review comment still asks for pinning the resolved symlink target across resolve/open; that remaining same-file identity concern should be resolved or consciously accepted.
- The final fix should preserve regular-file, max-size, hardlink, dangling symlink, non-regular target, relative/absolute symlink, and extra-bootstrap traversal protections.
- There are overlapping open reports and another open implementation PR; maintainers should consolidate on one canonical fix path rather than closing this implementation candidate without replacement.
