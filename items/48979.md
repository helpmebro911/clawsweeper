## Summary

Keep open. Current main has a working Telegram media sender when a media URL/path is present, but the specific `read` image path still does not appear to be bridged into outbound media. The read tool returns inline image content/metadata, while the tool-to-reply media extractor only queues `details.media`, `MEDIA:` directives, or `details.path`; the tests explicitly keep base64 image-only read results as non-deliverable media.

## Evidence

- **Read wrapper does not add outbound media metadata:** `createOpenClawReadTool` executes the upstream read, strips truncation details, normalizes/sanitizes image blocks, and returns that result; there is no added `details.media`, `MEDIA:` directive, or `details.path` for the original image file.
  - file: [src/agents/pi-tools.read.ts:674](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/agents/pi-tools.read.ts#L674)

  - sha: [d228463120a5](https://github.com/openclaw/openclaw/commit/d228463120a5057f7ccf9a5c31af62fe4a922088)
- **Media extractor requires a path or directive:** Tool result media extraction only supports structured `details.media`, `MEDIA:` text, or legacy `details.path`; the inline comment says Pi SDK `read` image results with base64 data and no file path return no media and need another delivery path.
  - file: [src/agents/pi-embedded-subscribe.tools.ts:237](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/agents/pi-embedded-subscribe.tools.ts#L237)
