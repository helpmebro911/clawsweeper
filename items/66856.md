## Summary

Keep open. Current `main` still resolves memory embedding providers from `resolveMemorySearchConfig()` without threading the active config, so `status`/`doctor` can still reach the compat plugin-loading path with `cfg === undefined` and emit the false `plugins.allow is empty` warning described in #66856.

## Evidence

- **Memory search still omits cfg for provider resolution:** `mergeConfig()` calls `getMemoryEmbeddingProvider(provider)` and `getMemoryEmbeddingProvider(fallback)` without passing `cfg`, and `resolveMemorySearchConfig()` later re-checks `getMemoryEmbeddingProvider(resolved.provider)` the same way. Those are the exact config-resolution paths used by CLI/doctor.
  - file: [src/agents/memory-search.ts:145](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/agents/memory-search.ts#L145)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '130,180p'; nl -ba src/agents/memory-search.ts | sed -n '380,410p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Missing cfg triggers compat reload path:** `getMemoryEmbeddingProvider(id, cfg?)` forwards `cfg` into `resolvePluginCapabilityProvider(...)`. When that `cfg` is undefined, `resolvePluginCapabilityProvider()` builds compat config from `params.cfg` and resolves a fresh runtime registry from that reduced view.
  - file: [src/plugins/memory-embedding-provider-runtime.ts:33](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/plugins/memory-embedding-provider-runtime.ts#L33)
  - command: `nl -ba src/plugins/memory-embedding-provider-runtime.ts | sed -n '1,120p'; nl -ba src/plugins/capability-provider-runtime.ts | sed -n '120,170p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **CLI and doctor still hit resolveMemorySearchConfig:** `openclaw doctor` calls `resolveMemorySearchConfig(cfg, agentId)` in `noteMemorySearchHealth()`, and the CLI embedding/status-related code also resolves memory search config before provider inspection. That keeps the buggy config-less provider lookup on live CLI paths.
  - file: [src/commands/doctor-memory-search.ts:322](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/commands/doctor-memory-search.ts#L322)

- **Other call sites already pass cfg explicitly:** The gateway embedding HTTP path calls `getMemoryEmbeddingProvider(params.provider, params.cfg)`, which shows the intended pattern is to thread config through provider resolution rather than rely on an empty compat fallback.
  - file: [src/gateway/embeddings-http.ts:161](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/gateway/embeddings-http.ts#L161)

- **Issue discussion matches current code and linked an unfinished fix:** The timeline/comments identify the memory embedding fallback path as the source of the false warning and cross-reference open PR #63848 as the minimal fix. Current `main` still matches that traced path, so the report remains actionable rather than obsolete.
