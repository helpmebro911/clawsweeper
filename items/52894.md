## Summary

Keep this issue open. Current main still only displays the Homebrew install command during brew-backed skill onboarding, then continues into the normal skill dependency install path that fails when no brew executable is available. The standalone shell installer can bootstrap Homebrew during initial OpenClaw install, but that does not cover later `openclaw configure`, gateway `skills.install`, or Skills UI dependency installs.

## Best Possible Solution

Keep this open. A conservative fix should live in the skill dependency onboarding/gateway path: when a selected brew-backed skill needs Homebrew and `brew` is missing on macOS, ask for explicit install confirmation, run the Homebrew installer with the same admin/failure expectations as `scripts/install.sh`, reload/re-detect `brew`, then continue or fail with actionable guidance. If maintainers decide skills onboarding must never install Homebrew, update the prompt and docs to say Homebrew is a manual prerequisite instead of presenting the command as an install step.

## Risks / Open Questions

- Auto-installing Homebrew from skills onboarding would execute an external installer and can require administrator privileges, so a fix needs explicit confirmation, macOS gating, timeout handling, logging, failure guidance, and post-install brew re-detection.
- Changing only `scripts/install.sh` would leave `openclaw configure`, gateway `skills.install`, and the Skills UI unchanged.
- Linux/Docker Homebrew behavior has different user, PATH, and writability constraints and should stay coordinated with #14593 rather than being silently folded into a macOS onboarding fix.
