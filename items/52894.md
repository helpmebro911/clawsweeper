## Summary

Keep this issue open. Current main still treats missing Homebrew during skill onboarding as a recommendation/manual-command prompt, then continues into the same brew-backed skill install path that fails when no brew executable is available. The top-level shell installer can install Homebrew on macOS, but that is a separate install flow and does not fix skill onboarding or later `configure`/Skills UI dependency installs. The prior ClawSweeper comment and related #14593/#57555 are useful adjacent context, but they do not supersede this macOS Homebrew auto-install request.

## Best Possible Solution

Keep this issue open. A conservative implementation should live in the skill dependency onboarding path: when a selected brew-backed skill needs Homebrew and `brew` is missing, offer an explicit macOS-only install confirmation, run the Homebrew installer with the same admin/failure expectations as `scripts/install.sh`, re-detect `brew`, then continue or fail with actionable guidance. If maintainers decide onboarding must never install Homebrew, update the prompt and docs to state that Homebrew is a manual prerequisite instead of presenting the command as an install step.

## Risks / Open Questions

- Auto-installing Homebrew from onboarding would execute an external installer and may require administrator privileges, so a fix needs explicit confirmation, macOS/non-interactive gating, timeout/error handling, and clear logging.
- Changing only `scripts/install.sh` would not help users who reach skill setup after OpenClaw is already installed through `openclaw configure`, gateway `skills.install`, or the Skills settings UI.
