## Summary

Keep #8321 open. Current main still lacks a per-job cron `disableCache` or `contextPruning` override: the cron job types, persisted base shape, gateway protocol schemas, CLI/Control UI creation paths, and isolated runner all continue to use config-level/per-agent defaults for context pruning. The related PR #8341 attempted the feature but was closed unmerged, and #24597’s direct script/no-LLM direction is a separate cost-reduction path rather than job-level cache/pruning control for agent-run cron jobs.

## Best Possible Solution

Keep #8321 open. Implement an additive per-job cron `contextPruning?: AgentContextPruningConfig` override using the existing `mode: "off" | "cache-ttl"` contract, thread it through cron create/update/list schemas, normalization, persistence, CLI and Control UI editor surfaces, docs, and agent-tool/protocol clients, then merge it into the effective `cfgWithAgentDefaults` before isolated cron invokes the embedded or CLI runner. Add regression coverage proving a job-level `mode: "off"` overrides global or per-agent `agents.defaults.contextPruning.mode: "cache-ttl"`. Track direct script/no-LLM cron execution separately.

## Risks / Open Questions

- The implementation will cross public cron persistence/types, gateway protocol validation, generated/client models, CLI, Control UI, docs, and isolated-run regression coverage.
- Direct script/no-LLM cron execution may reduce costs for deterministic shell jobs, but it would not satisfy per-job cache/pruning control for cron jobs that still run through the agent.
