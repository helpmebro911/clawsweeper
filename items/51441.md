## Summary

Current `main` does not implement this request. OpenAI-compatible assistant metadata is still built from the requested `model.id`/`provider`, session persistence only stores `modelUsed`/`providerUsed`, and gateway/status surfaces expose only `model`/`modelProvider` plus the existing fallback notice text, not separate `resolvedModel`/`resolvedProvider`/`fallbackUsed` fields.

## Evidence

- **Agent meta has no resolved-model fields:** `EmbeddedPiAgentMeta` only defines `provider` and `model`; there is no `resolvedModel` or `resolvedProvider` field to carry backend resolution through the runtime.
  - file: [src/agents/pi-embedded-runner/types.ts:5](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/pi-embedded-runner/types.ts#L5)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **OpenAI-compatible response conversion keeps requested model identity:** `buildAssistantMessageFromResponse()` receives a caller-supplied `{ provider, id }` and writes that into the assistant message. The caller in `openai-ws-stream.ts` passes `provider: model.provider` and `id: model.id`, so the response object's own model identifier is not surfaced here.
  - file: [src/agents/openai-ws-message-conversion.ts:498](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/openai-ws-message-conversion.ts#L498)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **OpenAI transport output starts with requested model id:** The streaming transport initializes assistant output with `provider: model.provider` and `model: model.id`, which matches the configured/requested route rather than a resolved backend model.
  - file: [src/agents/openai-transport-stream.ts:689](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/agents/openai-transport-stream.ts#L689)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Session persistence only stores modelUsed/providerUsed:** `persistSessionUsageUpdate()` accepts `modelUsed` and `providerUsed` and persists them into `SessionEntry.model` / `modelProvider`. There are no persisted `resolvedModel`, `resolvedProvider`, or `fallbackUsed` fields.
  - file: [src/auto-reply/reply/session-usage.ts:71](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/auto-reply/reply/session-usage.ts#L71)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Gateway session rows do not expose the requested new fields:** `GatewaySessionRow` only includes `modelProvider` and `model`. `buildGatewaySessionRow()` ultimately returns those fields and, when an override exists, prefers the selected model for the row, not a separate resolved-backend field.
  - file: [src/gateway/session-utils.types.ts:18](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/gateway/session-utils.types.ts#L18)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Status surfaces still show selected model, not separate resolved backend fields:** `/status` text renders `🧠 Model: <selected>` plus an optional fallback notice line, and `status.summary` computes its model from `resolveSessionModelRef(...)`. Neither surface emits `resolvedModel`, `resolvedProvider`, or `fallbackUsed` as distinct outputs.
  - file: [src/status/status-message.ts:851](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/status/status-message.ts#L851)
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
