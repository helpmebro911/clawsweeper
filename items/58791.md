Keep open. Current `main` does not implement `registerStreamFnWrapper`, and the PR still has unresolved correctness/safety concerns against its own advertised use case, so there is not strong evidence for a conservative close.

## Risks / Open Questions

- The advertised per-call model-routing use case can misroute/authenticate against the wrong backend because provider transport is chosen before the proposed plugin wrapper runs.
- A generic plugin wrapper that can replace `streamFn` after provider/core wrapping creates a real chance of bypassing safety, auth, or telemetry wrappers unless the contract is narrowed.
- The PR body's test plan remains unchecked, and the provided diff does not include regression tests for wrapper composition, invalid return values, or the cross-provider routing case.
