## Summary

Keep #70133 open. Current main still has a Feishu-owned per-chat sequential queue that awaits the full message dispatch promise before releasing the lane. The only current lane split is for abort-like control text and `/btw`; ordinary P2P messages in the same chat still share the same key. I did not find a Feishu queue timeout/release guard that would solve the reported stuck-dispatch behavior, and the linked #58953/#43512 context does not supersede this exact bug.

## Best Possible Solution

Keep the issue open. A good fix likely belongs in the Feishu inbound queue/dispatch path: add a configurable per-chat dispatch timeout or release guard that uses an abort-aware downstream run, logs the timeout, releases the chat lane, and has regression tests showing a hung first Feishu P2P task does not block a later same-chat P2P message indefinitely. It should also preserve the existing control and `/btw` lane behavior.
