## Summary

Current main still appears to have the reported Feishu per-chat blocking behavior. Feishu inbound messages are put on a sequential key derived from account and chat, the queue advances only when the prior task promise settles, and the task awaits the full agent reply dispatch. I did not find a dispatch-timeout or lane-release guard that would let later same-chat P2P messages proceed after a stuck turn. Related #58953 is a different closed symptom, while the local related reports #43512 and #54409 support that Feishu per-chat queue behavior still has unresolved edge cases rather than superseding this report.

## Best Possible Solution

Keep the issue open and fix it in the Feishu plugin. The likely maintainer path is to add a bounded dispatch/queue-release guard around the per-chat lane, make timeout behavior explicit in logs and config if configurable, ensure processing claims and reply dispatcher/typing state settle, and add a regression test where the first same-chat task never resolves but a later message proceeds after the configured guard.

## Risks / Open Questions

- A fix needs to fence or cancel the still-running agent turn; simply releasing the queue after a timer could allow late replies from the old turn to interleave with the next message.
- The review was source/docs/test based, not a live Feishu repro, but the current implementation still matches the reporter's logged queue behavior.
