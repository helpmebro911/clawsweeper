## Summary

Keep open. Current main still has the reported Signal doctor repair behavior: the shared doctor normalizer will create `channels.signal.accounts.default` from top-level `cliPath`, `dmPolicy`, and `groupPolicy` when named accounts already exist, and Signal has no promotion contract that preserves those values at the channel root or targets the configured named default account. The related #62764 context is useful as a co-occurring `doctor --fix` deployment failure, but it does not supersede this separate Signal config corruption report.

## Evidence

- **Doctor still creates accounts.default:** `seedMissingDefaultAccountsFromSingleAccountBase` iterates channels with existing account maps, moves matching top-level keys into a new `accounts.default` entry, deletes those top-level keys, and records the exact reported message.
  - file: [src/commands/doctor/shared/legacy-config-core-normalizers.ts:121](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/commands/doctor/shared/legacy-config-core-normalizers.ts#L121)
  - command: `nl -ba src/commands/doctor/shared/legacy-config-core-normalizers.ts | sed -n '118,190p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Reported keys still qualify for promotion:** The static single-account promotion key set includes `cliPath`, `dmPolicy`, and `groupPolicy`; when named accounts exist, keys are only narrowed if a channel supplies `namedAccountPromotionKeys`, otherwise all matched keys are returned.
  - file: [src/channels/plugins/setup-promotion-helpers.ts:15](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/channels/plugins/setup-promotion-helpers.ts#L15)
  - command: `nl -ba src/channels/plugins/setup-promotion-helpers.ts | sed -n '15,150p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Signal has no narrowing promotion contract:** Signal's setup adapter is just `createPatchedAccountSetupAdapter({ channelKey: "signal", ... })`, and the shared Signal plugin exposes that setup object without `namedAccountPromotionKeys` or `resolveSingleAccountPromotionTarget`.
  - file: [extensions/signal/src/setup-core.ts:221](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/extensions/signal/src/setup-core.ts#L221)
  - command: `nl -ba extensions/signal/src/setup-core.ts | sed -n '221,235p'`
  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Phantom default account can be treated as configured:** Signal account resolution marks an account configured if merged config has `cliPath`, `httpHost`, `httpPort`, `httpUrl`, `account`, or `autoStart`. A doctor-created `accounts.default` containing only `cliPath` is therefore configured, and Signal's plugin uses `account.configured` for `isConfigured`.
  - file: [extensions/signal/src/accounts.ts:47](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/extensions/signal/src/accounts.ts#L47)

- **Gateway starts every listed channel account:** Gateway startup uses `plugin.config.listAccountIds(cfg)` when no specific account is requested, then checks enabled/configured and starts each account. That means a doctor-created `accounts.default` is part of the runtime start set.
  - file: [src/gateway/server-channels.ts:299](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/src/gateway/server-channels.ts#L299)
