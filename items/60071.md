## Summary

Keep open. Current `main` still wires the Twitch gateway in the same way that causes passive providers to look like they exited: `monitorTwitchProvider()` returns a `{ stop }` handle immediately, `startAccount()` awaits it directly, and the gateway supervisor auto-restarts any account whose `startAccount()` promise resolves. I did not find a Twitch-side lifecycle wrapper or test proving this was fixed, and the relevant files are unchanged from `v2026.4.23`.

## Evidence

- **Twitch monitor still resolves with a stop handle:** `monitorTwitchProvider()` connects, registers the message handler, installs an abort listener, and then returns `{ stop }` instead of staying pending until abort.
  - file: [extensions/twitch/src/monitor.ts:195](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/twitch/src/monitor.ts#L195)

- **Twitch gateway does not keep the passive monitor alive:** `startAccount()` marks the account running, lazy-imports `monitorTwitchProvider()`, and `await`s it directly. There is no `runStoppablePassiveMonitor()`/`waitUntilAbort()` wrapper here.
  - file: [extensions/twitch/src/plugin.ts:169](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/twitch/src/plugin.ts#L169)
