## Summary

Keep open. Current main already ships the ID-prefix approval lookup portion, including the exact-resolved grace-period safety that review comments requested, but it does not implement the reply-to-message `/approve <decision>` flow. This PR still contains useful product work, though it needs a rebase/rewrite against the current approval command handler before it is mergeable.

## Best Possible Solution

Keep this PR open but ask for a rebase onto current main. The best path is to preserve the shipped `lookupPendingId` implementation, add reply-to ID extraction inside the current `handleApproveCommand` while keeping existing exec/plugin approval authorization and routing, parse the full `ID:` line from `ReplyToBody`, update the approval hint text, and add focused tests for single-token reply approvals plus whitespace/custom-ID edge cases.

## Risks / Open Questions

- The PR diff predates the current `commands-approve.ts` plugin approval routing and fallback behavior; applying it as-is would likely regress plugin approval handling by routing only to `exec.approval.resolve`.
- The PR's reply-body extractor captures only `\S+` after `ID:`, while current exec approval IDs are schema-level non-empty strings and can contain whitespace; a revised implementation should parse and trim the full ID line.
- The prefix portion is already shipped, so the remaining review should be narrowed to reply-to extraction, hint text, and tests instead of reintroducing the older manager changes.
