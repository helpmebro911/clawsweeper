## Summary

Keep this issue open. Current main still lacks the requested shared `ToolFreshnessMetadata` type, standardized `freshness` envelope, or agent-facing freshness-aware tool-result contract. The issue is recent, coherent, author/labels are not protected, and the related context does not show a canonical merged or open implementation that supersedes this specific freshness metadata RFC.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, documented for plugin authors, then targeted adoption in `session_status`, `memory_search`, cron status/list/runs, and cached web providers. Because `details` is not model-visible, the design should explicitly decide how freshness becomes available to agents without breaking existing `content[]`/`details` consumers.
