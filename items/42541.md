## Summary

Keep open. Current main partially addresses the report by wrapping OAuth refresh failures and updating runtime auth snapshots, but it still suppresses profile-resolution refresh errors in active auth-selection paths and still does not special-case `google-gemini-cli` cooldown handling.

## Evidence

- **OAuth refresh failures are now wrapped with a user-facing message and preserved cause:** `src/agents/auth-profiles/oauth.ts:378-390` builds `OAuth token refresh failed for <provider>: ...` and rethrows with `{ cause: error }`. This is an improvement over the silent failure described in the issue, but it only applies if callers let the error propagate.
  - file: [src/agents/auth-profiles/oauth.ts:385](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/agents/auth-profiles/oauth.ts#L385)
  - command: `nl -ba src/agents/auth-profiles/oauth.ts | sed -n '378,392p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Default provider auth selection still swallows profile refresh errors:** `src/agents/model-auth.ts:505-537` catches any `resolveApiKeyForProfile()` error, logs only at debug, and continues. If no other source resolves, `src/agents/model-auth.ts:593-601` throws the generic `No API key found for provider` error instead of surfacing the refresh failure. That means the core runtime path is still not clearly fixed for unlocked/default profile selection.
  - file: [src/agents/model-auth.ts:505](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/agents/model-auth.ts#L505)
