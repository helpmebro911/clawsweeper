## Summary

Keep open. Current main has partial fixes for the original report: OAuth profile resolution now wraps refresh failures, refreshed auth-store saves update runtime snapshots, and Gemini project discovery has explicit environment fallback coverage. But the remaining Gemini refresh path still falls back to @mariozechner/pi-ai getOAuthApiKey v0.70.2, whose published implementation drops the original refresh error cause; google-gemini-cli has no provider-local refreshOAuth hook; google-gemini-cli is still subject to generic auth cooldown; and an auxiliary usage/status OAuth resolver still has a bare catch. The 2026-04-24 Windows comment is recent, concrete, and adjacent PR #71076 explicitly did not address this refresh/projectId stack.

## Best Possible Solution

Keep the issue open and narrow the remaining maintainer path: add a Gemini-owned refreshOAuth hook or upstream pi-ai fix that preserves refresh causes, replace/log the remaining silent provider-usage auth catch, decide whether Gemini refresh failures should bypass or use a shorter auth cooldown, and add focused regression coverage for Gemini refresh error details plus projectId persistence during code-assist login. #8673 can continue tracking generic retry policy, but it does not fully cover this Gemini-specific refresh diagnostics/cooldown/projectId report.
