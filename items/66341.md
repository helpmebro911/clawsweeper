## Summary

Keep this PR open. Current main still gates cron.add/update/remove/run behind operator.admin and has no persisted creatorScopes or cron execution scope ceiling, so the requested authorization change is not implemented or obsolete. The PR is plausible core security/authorization work, but its branch still needs maintainer security review and cleanup of unrelated browser and QA-lab changes before it should land.

## Best Possible Solution

Keep this PR open for maintainer review. Ask the author to split unrelated browser and QA-lab changes, then review a focused cron-only branch that lowers only the intended cron methods, persists creatorScopes at creation time, enforces the ceiling in embedded and CLI/MCP execution paths, and includes regression coverage proving write-scoped callers cannot update or run admin-created jobs to gain admin tool access.
