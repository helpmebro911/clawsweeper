## Summary

Keep #10519 open. Current main still drops fenced-code language metadata before Telegram rendering, so Telegram outbound HTML remains `<pre><code>...</code></pre>` rather than `<pre><code class="language-...">...`. The related WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram by retaining optional fence language metadata through the shared markdown IR/render path, or by adding a Telegram-specific fence-aware renderer if maintainers prefer not to widen the shared IR. Update Telegram formatter, chunking, and outbound tests to expect `class="language-js"` for language fences while preserving current HTML escaping and chunk splitting behavior.
