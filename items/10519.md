## Summary

Keep #10519 open. At the checked-out main SHA, Telegram still renders fenced Markdown code blocks as `<pre><code>` without the fence language, the Telegram test suite still asserts that output, and the shared Markdown IR currently drops `token.info` before Telegram rendering. The related WebChat copy-button/language-class work is separate UI code and does not affect Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation either by carrying fence metadata through the shared Markdown IR/render path or by adding a Telegram-specific fence-aware renderer. Update Telegram format/chunking/file-reference tests to expect `class="language-js"` while preserving HTML escaping, chunk limits, protected code regions, and plain-text fallback behavior.
