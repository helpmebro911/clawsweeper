## Summary

Keep #71973 open. Current main has the broad cache-stable system prompt work, but the narrower report is still valid: heartbeat, cron, WhatsApp heartbeat, reset, memory, and post-compaction paths still inject exact `Current time:` lines into runtime prompt content, and tests explicitly preserve that behavior.

## Best Possible Solution

Keep the issue open. The best maintainer path is a narrow cache-aware runtime-time policy: preserve the timezone-only stable system prompt, move exact wall-clock time into late-bound context that is not included in provider cache-marked prefixes when possible, and inject exact time only for flows that demonstrably need it. Update heartbeat, cron, WhatsApp heartbeat, post-compaction, reset, and memory tests to prove cached prefixes remain byte-identical while date-dependent behavior still works.

## Risks / Open Questions

- Removing or gating the timestamp blindly could regress date-sensitive startup, memory, reset, cron, and post-compaction flows that currently rely on exact date/time availability.
- The optimization should be validated per provider/cache path because Anthropic cache-control, OpenAI prompt-cache keys, Google cachedContent, Bedrock cache points, and CLI-backed sessions do not share identical cache-boundary semantics.
