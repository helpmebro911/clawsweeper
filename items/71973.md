## Summary

Keep #71973 open. Current main has the earlier system-prompt cache-stability work, but the narrower automation concern remains: heartbeat, cron, session-reset, memory flush, and post-compaction paths still inject an exact wall-clock `Current time:` line into per-turn prompt content, and tests explicitly codify that behavior.

## Best Possible Solution

Keep the issue open and implement a focused prompt-composition change: keep stable timezone metadata in the system prompt, preserve date/time correctness via `session_status` or late/runtime-only context, and stop placing exact wall-clock `Current time:` near the start of cache-sensitive heartbeat, cron, reset, memory-flush, and post-compaction prompt bodies. Add regression tests that prove exact time is no longer in the cache-sensitive prompt prefix while affected workflows still have a dependable way to obtain current time.
