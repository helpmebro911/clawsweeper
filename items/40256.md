## Summary

Keep open. Current `main` still places channel- and session-dependent system-prompt sections above the stable project-context block and above the cache boundary, so the reported prefix-cache invalidation path is still present.

## Evidence

- **Dynamic messaging text still varies by channel/capability:** `buildMessagingSection()` still emits different prompt text based on `inlineButtonsEnabled` and `runtimeChannel`, including the `Inline buttons supported` vs `Inline buttons not enabled for <channel>` branch.
  - file: [src/agents/system-prompt.ts:370](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/agents/system-prompt.ts#L370)
  - command: `nl -ba src/agents/system-prompt.ts | sed -n '330,410p'`
  - sha: [d4e93e791bc5](https://github.com/openclaw/openclaw/commit/d4e93e791bc5093ea14666bf4d8202b45a5132b4)
- **Dynamic sections still precede project context and cache boundary:** `buildAgentSystemPrompt()` appends Messaging and Voice first, then Reactions and Reasoning Format, then stable `# Project Context`, and only later inserts `SYSTEM_PROMPT_CACHE_BOUNDARY` before dynamic project context / group context / heartbeat. That means channel-dependent prompt bytes still sit in the stable prefix.
  - file: [src/agents/system-prompt.ts:871](https://github.com/openclaw/openclaw/blob/d4e93e791bc5093ea14666bf4d8202b45a5132b4/src/agents/system-prompt.ts#L871)

Current code does not do that for Messaging/Reactions.
