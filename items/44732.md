## Summary

Current main already implements the requested proxy-aware Telegram media download path. Telegram media downloads now carry the Telegram transport dispatcher policy into the guarded fetch, explicit proxies opt into trusted proxy-side DNS handling, and SSRF tests cover skipping target DNS pinning while preserving hostname policy checks. The changelog shows this shipped in v2026.4.14.

## Close Comment

Closing as implemented on current `main` after a Codex review. The current Telegram media download path now forwards the Telegram transport’s dispatcher policy into the guarded media fetch and opts into trusted explicit-proxy DNS handling for channel-level proxies (`extensions/telegram/src/bot/delivery.resolve-media.ts:235`). The shared media fetch path switches those requests into `trusted_explicit_proxy` mode (`src/media/fetch.ts:133`), and the SSRF guard keeps the hostname allowlist checks while skipping target DNS pinning so the configured proxy performs final resolution instead of OpenClaw opening a direct pinned connection (`src/infra/net/fetch-guard.ts:348`). There are regression tests for both behaviors in `src/media/fetch.test.ts:294` and `src/infra/net/fetch-guard.ssrf.test.ts:1119`. The checked-in changelog shows this shipped in `v2026.4.14` (`CHANGELOG.md:680`). If you can still reproduce this on `v2026.4.14` or newer, please open a fresh issue with the exact proxy type, config, logs, and current version.
