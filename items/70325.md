## Summary

Keep #70325 open. Current main still applies the fixed 10-minute cron safety timeout to non-agent `systemEvent` payloads, which is the main-session job shape in the report. The configurable `timeoutSeconds` path exists only for `agentTurn` payloads, the cron config/schema do not expose a global job timeout, and the public cron docs do not document the main-session 10-minute limit. Related open PRs #42482 and #42680 cover adjacent cron timeout semantics but do not supersede this main-session configurability/documentation request.

## Best Possible Solution

Keep the issue open and fix the core cron contract. The best path is to add an explicit execution timeout setting that covers main `systemEvent` jobs as well as isolated `agentTurn` jobs, wire it through the cron type/gateway schema/CLI/tool surfaces, update `resolveCronJobTimeoutMs()`, and add regression coverage for a main-session heartbeat that completes after the old 10-minute window without recording a false error. If maintainers intentionally want main-session jobs capped, document the 10-minute limit on https://docs.openclaw.ai/automation/cron-jobs and recommend isolated `agentTurn` jobs with `timeoutSeconds` for long-running background work.
