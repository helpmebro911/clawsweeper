## Summary

Keep this PR open. The branch is a plausible focused fix attempt, but it is not safely obsolete or closeable: the author's follow-up says live WhatsApp testing still missed active-session messages, and current main still has the generic `message_received` emission tied to `dispatchReplyFromConfig` rather than a shared inbound/enqueue boundary. The PR also changes when inbound content becomes plugin-visible, so it needs maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review, but request a revision or replacement that defines the real generic inbound hook boundary, covers active-session enqueue/drain behavior with regression tests, and explicitly reconciles plugin `message_received` and internal `message:received` semantics with WhatsApp's opt-in privacy behavior. If maintainers decide the WhatsApp-specific main implementation fully supersedes the linked bug, close this PR only with that canonical evidence.
