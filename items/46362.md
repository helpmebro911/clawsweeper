## Summary

Keep open. Current main still does not implement the PR's central feature: spoken Discord voice phrases that persistently escalate/de-escalate the session model. Main only supports a static `channels.discord.voice.model` override, while the unrelated plugin HTTP route registry fix from this PR has already been solved separately on main.

## Best Possible Solution

Keep this PR open only as a feature candidate. The best path is a fresh, narrow Discord plugin change on current main: drop the obsolete gateway registry edits, implement explicit low-false-positive voice model commands in `extensions/discord/src/voice/manager.ts`, ensure matched command phrases are consumed instead of sent to the agent, create/update the session entry before first-turn persistence, honor configured/allowed model references instead of hardcoding policy-sensitive defaults where possible, and add focused e2e/unit coverage plus Discord voice docs if maintainers want the feature.

## Risks / Open Questions

- The PR branch is stale against current main: Discord voice now lives under `extensions/discord/src/voice/manager.ts`, not the old `src/discord/voice/manager.ts` path shown in the PR diff.
- The PR bundles an already-solved gateway registry fix with the voice feature, so rebasing as-is would risk reintroducing obsolete runtime-state code.
- The proposed voice UX needs maintainer/product judgment because hardcoded cloud model targets and broad natural-language triggers can surprise users or bypass configured model policy expectations.
- The existing review comments identify real correctness issues that should be fixed before merge: false positives, command text being sent to the agent, and first-utterance persistence.
