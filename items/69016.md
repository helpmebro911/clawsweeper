## Summary

Keep open. Current main has partial CLI transcript compaction work, but the gateway/auto-reply CLI backend path still bypasses the context-engine lifecycle that this issue reports: it calls `runCliAgent` directly, persists usage/session metadata, and does not resolve or run the active context engine’s `assemble()`, `afterTurn()`, or ingest fallback for normal CLI backend turns.

## Best Possible Solution

Keep the issue open and implement one shared CLI backend context-engine finalization path used by both `openclaw agent` and gateway/auto-reply CLI runs. It should persist the OpenClaw-side turn transcript, pass CLI usage/current token data into the active engine, run `afterTurn()` or ingest fallback after successful CLI turns, invoke `compact()` when the CLI transcript is over budget, update compaction counts, and clear/reseed provider-owned CLI session state after compaction. If `assemble()` cannot be meaningfully projected into provider-owned CLI history, document that limitation and provide an explicit startup/status warning when a selected context engine cannot fully operate with a CLI backend.
