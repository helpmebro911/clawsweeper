## Summary

Keep this issue open. Current main still has adjacent Telegram update-offset hardening and generic session-store maintenance, but it does not implement the requested status diagnosis or safe Telegram-specific reset/quarantine flow for malformed or oversized persisted session state. The Keep My Claw backup skill mentioned in the discussion is a useful safety net, not first-party detection or recovery UX.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status surface first. Add a read-only Telegram state inspector used by channel status and/or doctor that checks the resolved Telegram update-offset path plus relevant session-store entries for existence, size, parse success, schema sanity, and mtime. Surface a clear ChannelStatusIssue with the affected account and recovery guidance. Then add a safe recovery command that backs up or quarantines only the affected Telegram artifact and either reinitializes the channel or tells the operator to restart the gateway. Use a generic core seam only if multiple channel plugins need the same persisted-state diagnostic API.

## Risks / Open Questions

- The exact bad artifact in the report is ambiguous between Telegram-owned update-offset JSON and the shared agent sessions.json store, so the implementation should inspect both relevant Telegram artifacts and generic session-store impact before choosing recovery UX.
- A corrupt-file reproduction was not run because this review was explicitly read-only; the conclusion is based on current source, docs, tests, history searches, and the supplied issue discussion.
- Any recovery command should be non-destructive by default: report exact artifact paths, create timestamped backups or quarantine files, and avoid deleting unrelated state under the OpenClaw state directory.
