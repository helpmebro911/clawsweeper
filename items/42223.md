## Summary

Keep open. Current `main` still contains the same collapse logic the PR removes: a nav group stays visually expanded whenever it contains the active tab, so the reported bug is not already implemented/fixed on `main` or in the latest release snapshot I inspected.

## Evidence

- **Current main still forces active-tab groups open:** `ui/src/ui/app-render.ts:1308-1313` computes `hasActiveTab` and then `showItems = navCollapsed || hasActiveTab || !isGroupCollapsed`; the `nav-section--collapsed` class is only applied when `showItems` is false, so a collapsed group with the active tab will still render as expanded.
  - file: [ui/src/ui/app-render.ts:1308](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/ui/src/ui/app-render.ts#L1308)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '1288,1336p'`
  - sha: [7d9172d5e093](https://github.com/openclaw/openclaw/commit/7d9172d5e093435d37ed3e00e476a4eaf84a252b)
- **Collapsed class is what actually hides children:** `ui/src/styles/layout.css:496-497` sets `.nav-section--collapsed .nav-section__items { display: none; }`, so if `showItems` stays true because of `hasActiveTab`, child items remain visible exactly as described in the PR.
  - file: [ui/src/styles/layout.css:496](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/ui/src/styles/layout.css#L496)
