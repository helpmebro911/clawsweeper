## Summary

Keep open. Current `main` still has the reported fire-and-forget Signal daemon shutdown pattern in the moved plugin code (`extensions/signal/src/daemon.ts`), and the restart/teardown paths still proceed after channel task resolution rather than waiting for the `signal-cli` child to exit. The issue body's old `src/signal/*` path is stale, but the underlying bug appears to remain on `main` and in the latest release commit.

## Evidence

- **Signal daemon stop is still synchronous fire-and-forget on main:** `SignalDaemonHandle.stop` is typed as `() => void`, and `spawnSignalDaemon()` still only sends `SIGTERM` without awaiting process exit or escalating to `SIGKILL`. That matches the race described in the issue.
  - file: [extensions/signal/src/daemon.ts:16](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/extensions/signal/src/daemon.ts#L16)
  - command: `nl -ba extensions/signal/src/daemon.ts | sed -n '1,260p'`
  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Monitor teardown does not wait for daemon exit:** `createSignalDaemonLifecycle().stop()` just flips a flag and calls `daemonHandle?.stop()`. In `monitorSignalProvider()` cleanup, `daemonLifecycle.stop()` is called in abort/finally paths with no await on `handle.exited`, so the monitoring task can finish before `signal-cli` has released resources.
  - file: [extensions/signal/src/monitor.ts:118](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/extensions/signal/src/monitor.ts#L118)
  - command: `nl -ba extensions/signal/src/monitor.ts | sed -n '90,180p'; nl -ba extensions/signal/src/monitor.ts | sed -n '390,540p'`
  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Channel restart waits for task completion, not child-process exit:** `stopChannel()` aborts the account task and waits for its promise via `waitForChannelStopGracefully()`. Config hot reload then does `await stopChannel(name); await startChannel(name);`. Because the Signal monitor task resolves after `daemonLifecycle.stop()` returns, a restarted Signal account can be started before the old `signal-cli` process exits.
  - file: [src/gateway/server-channels.ts:547](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/gateway/server-channels.ts#L547)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '80,140p;528,595p'; nl -ba src/gateway/server-reload-handlers.ts | sed -n '160,188p'`
  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Gateway restart path can move on once server close completes:** On restart, the run loop awaits `server.close(...)`, and server close stops all channel plugins via `await params.stopChannel(plugin.id)`. After that, the run loop may spawn a fresh process. Since Signal channel shutdown does not await `signal-cli` exit, this leaves the reported orphan/port-lock race plausible during restart.
  - file: [src/cli/gateway-cli/run-loop.ts:204](https://github.com/openclaw/openclaw/blob/a958b6e72328d1d922a6d133b1a70e866faf4013/src/cli/gateway-cli/run-loop.ts#L204)

  - sha: [a958b6e72328](https://github.com/openclaw/openclaw/commit/a958b6e72328d1d922a6d133b1a70e866faf4013)
- **Latest release still contains the same shutdown code:** The latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` still shows the same `stop: () => { child.kill("SIGTERM") }` implementation in `extensions/signal/src/daemon.ts` and the same non-awaited lifecycle stop in `extensions/signal/src/monitor.ts`.
  - file: [extensions/signal/src/daemon.ts:141](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/signal/src/daemon.ts#L141)
