## Summary

Keep open. The issue is recent, specific, and still describes an actionable supported surface: Slack Socket Mode multi-account inbound dispatch with account-scoped bindings after a 2026.3.23-2 config write. Current main has account-aware routing and independent per-account startup, but there is still no issue-specific confirmation, linked fixing PR, or focused regression test proving the reported config-taint/non-dispatch scenario is resolved.

## Best Possible Solution

Keep this issue open and narrow it with maintainer evidence: ask for sanitized current channels.slack.channels, channels.slack.accounts, channels.slack.defaultAccount, bindings, and verbose Slack routing/drop logs on the latest release. Add a focused Slack inbound regression test covering two configured named Slack accounts, an unconfigured or disabled default account, account-scoped bindings, an allowed stable channel ID, and an @mention dispatching to the bound agent. If the reporter used channel names, resolve it through the stable-ID docs and the #46327 workaround pattern.
