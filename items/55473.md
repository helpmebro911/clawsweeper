## Summary

Keep this PR open. Current main still treats startup secret-resolution failures as fatal, and active channel secret refs are still included in startup resolution, so the underlying bug is not implemented on main. The PR is a plausible focused fix, but it is not ready to close or merge because maintainer and bot review comments identify unresolved correctness issues around degraded startup, recovery, and tests.

## Best Possible Solution

Keep this PR open as the implementation candidate, but require a rework on current main that keeps non-channel and non-availability secret failures fatal, preserves write/recovery invariants, restarts recovered channels when appropriate, and includes stable regression tests for active channel env-ref degradation plus negative/fatal cases.

## Risks / Open Questions

- The PR changes secret-resolution failure handling; if the availability classifier is too broad, malformed refs, allowlist violations, or permanent provider/config errors could be hidden by disabling a channel instead of failing startup.
- Review comments still flag recovery correctness risks: stale `recoveryConfig`, non-durable explicit disable writes during degraded startup, and recovered channels not restarting immediately after `secrets.reload`.
- Maintainer feedback reports a full-file `src/gateway/server.reload.test.ts` failure on the PR head, so the regression coverage needs stabilization before this can be considered mergeable.
