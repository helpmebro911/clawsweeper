## Summary

Keep this issue open. Current main still has local-only `sessions_send` and `sessions_spawn`: there are no tool params or protocol fields to select a peer gateway, config exposes a single `gateway.remote` rather than `gateway.peers` or remote agents, and the server `sessions.send` path loads a local session entry before dispatch. The issue is also paired with same-author open PR #43656 as the implementation candidate, so the maintainer path is to review, revise, land, or close that PR rather than close the tracker. The related `xihe-jianmu-ipc` MCP bridge reports are useful external workaround evidence, but they do not supersede native cross-gateway session routing with OpenClaw session history and policy semantics.

## Best Possible Solution

Keep this issue open as the canonical native cross-gateway session messaging tracker. Maintainers should review, revise, land, or close PR #43656; if that PR is not the right shape, a replacement should add a documented, additive, secure peer/remote-gateway contract across config schema, gateway protocol, session tool schemas, auth/token resolution, local/remote history handling, tests, and user-facing docs.
