## Summary

Keep open. Current main has partial fixes for adjacent startup isolation and Feishu multi-account burst behavior, but it does not fully implement this issue's core request: the visible `ready` log still precedes channel/sidecar startup, there is no per-channel recovery timing summary, Feishu startup probes can still spend the 30s timeout before falling back, and Telegram polling still awaits/retries webhook cleanup before polling can begin.

## Best Possible Solution

Keep this open as the canonical combined startup UX issue unless maintainers create a narrower replacement. The likely fix belongs in core gateway/channel startup status plus bundled Telegram/Feishu recovery: distinguish HTTP/core readiness from channel recovery in logs and `/readyz`, add a per-channel timing/degraded summary, keep channel recovery fault-isolated, and bound or background Telegram webhook cleanup and Feishu identity probes without hiding degraded channel state.
