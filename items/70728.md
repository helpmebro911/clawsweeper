## Summary

Keep #70728 open. Current main has partial improvements: channel plugin startup is fault-isolated, Feishu multi-account bot probes are serialized, and `/readyz` is stricter than `/healthz`. But the central user-facing issue is not fully implemented: startup still logs `ready (...)` before channel/sidecar startup begins, there is no clear per-channel recovery timing summary, Telegram polling still awaits and retries `deleteWebhook` before polling, and Feishu can still spend the 30s startup probe timeout path before background identity recovery.

## Best Possible Solution

Keep the issue open and narrow the maintainer path: preserve the existing channel fault isolation and Feishu sequential preflight, then improve startup UX by changing the early `ready` log to distinguish HTTP/gateway availability from sidecar/channel recovery, add a concise per-channel startup/recovery timing summary, surface degraded channel states while recovery continues, and consider making Telegram webhook cleanup failures transition Telegram into a clear recovering/degraded state instead of silently extending perceived startup.
