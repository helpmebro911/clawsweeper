## Summary

Current main still has the reported main-session `systemEvent` cron fast-return path, and I did not find a shipped ghost-run warning, configurable threshold, or run-log annotation. The related implementation PR #63111 is closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep this issue open. The best maintainer path is to implement a scoped ghost-run signal in `src/gateway/server-cron.ts` or make the cron service distinguish queued heartbeat handoff from confirmed agent processing, then add threshold config/schema/docs and regression coverage for fast successful main-session `systemEvent` next-heartbeat runs.

## Risks / Open Questions

- A duration-only warning could false-positive on legitimate no-op or very fast main-session events; the implementation should scope detection to `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `wakeMode: "next-heartbeat"`, with a configurable threshold.
- Maintainers should decide whether the best product semantics are a warning/run-log annotation or a distinct queued-vs-processed cron outcome; the current behavior still records the handoff as `ok`.
