## Summary

Keep open. Current main still intentionally ignores explicit HTTP(S) image URLs in native prompt image detection, so the mixed inline-image plus remote-image behavior this PR targets is not implemented. This PR is also paired with same-author open bug #62514 and has unresolved security/behavior review feedback, so the maintainer path is review or contributor rework rather than cleanup closure.

## Best Possible Solution

Keep this PR and #62514 open for maintainer review or contributor rework. A safe implementation should detect only clearly intended remote image refs, avoid fetching arbitrary hyperlinks, thread the existing `ssrfPolicy` through native remote preloading, handle dedupe and trailing punctuation correctly, account for the shared CLI detector surface, and add focused regression coverage for mixed inline plus remote image prompts, non-image URL no-fetch behavior, and SSRF-policy propagation.

## Risks / Open Questions

- The proposed PR should not merge as-is if it preloads every HTTP(S) URL found in a prompt; that expands outbound gateway fetches to ordinary hyperlinks and creates latency/privacy exposure beyond explicit image refs.
- The proposed remote preload path needs to thread the existing remote-media SSRF policy contract; current tool paths already do this for remote media, and dropping it would bypass operator-configured policy.
- Because the detector is shared with CLI prompt image loading, the final implementation should either cover CLI behavior explicitly or constrain the change to the WebUI/native vision path it intends to fix.
