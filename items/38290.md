## Summary

Keep PR #38290 open. Current main still normalizes browser origins with URL.origin, so chrome-extension://<id> becomes the opaque string "null" and cannot match an exact gateway.controlUi.allowedOrigins entry. The PR remains a narrow live gateway hardening fix; scheme wildcard support is separate in #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in src/gateway/origin-check.ts that symmetrically normalizes incoming Origin headers and configured gateway.controlUi.allowedOrigins for opaque-origin schemes using an exact protocol//host fallback. Keep/add unit and gateway WebSocket regression coverage for accepted and rejected extension origins, and leave scheme wildcard support to #39739.
