## Summary

Current main does not implement the reported workflow. The loader now sees symlinked child directories, but it then resolves each skill path and rejects any skill directory or SKILL.md whose realpath escapes the configured skills root. A symlink from ~/.openclaw/skills/<name> to an arbitrary external folder would still be skipped. Docs point users to skills.load.extraDirs for custom shared skill folders, and related #60145/#60513 track audit feedback for rejected symlink escapes, but they do not implement this issue's requested external symlink loading behavior.
