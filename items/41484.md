## Summary

Keep open. Current main has a useful mitigation because the agent-facing cron tool is owner-only, so non-owner chat senders should not be able to create or edit cron jobs through that tool. However, the requested per-job elevated flag/trust model is not implemented in the cron job schema, and heartbeat elevated behavior still resolves from provider allowFrom rather than HEARTBEAT.md modification provenance or an explicit heartbeat trust marker.

## Risks / Open Questions

- Closing would lose the remaining heartbeat trust-model request and the explicit per-job elevated scoping requested by the issue.
- The owner-only cron mitigation is useful but is not equivalent to allowing owner-approved elevated cron jobs while preventing later unapproved jobs from inheriting elevated privileges.
