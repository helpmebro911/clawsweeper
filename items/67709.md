## Summary

Keep this open. Current main still has a plausible shared startup stall path for the reported symptom: Control UI starts `chat.history` and `models.list` together, `models.list` waits for the model catalog, and `chat.history` can wait on the same catalog when the stored session has no thinking level. Existing startup-gating only handles the sidecar-unavailable window and predates the reported version.

## Best Possible Solution

Keep the issue open and fix the gateway/UI startup responsiveness path: make `chat.history` return without waiting on full model catalog discovery when only a thinking default is missing, make Control UI model catalog loading backgrounded or timeout-bounded, and add a regression test with a deferred `loadGatewayModelCatalog()` proving initial chat history/rendering remains responsive while `models.list` is still loading.
