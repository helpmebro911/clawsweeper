## Summary

Keep this issue open. Current main has a startup-unavailable guard for `chat.history` and `models.list`, but after that guard clears the reported equal-duration stall is still plausible: Control UI startup still runs history and model loading together, both gateway handlers can still wait on the shared model-catalog load, and the browser RPC client still has no in-flight request timeout.

## Best Possible Solution

Keep the issue open and fix the Control UI startup responsiveness path. `chat.history` should not wait on full model-catalog discovery merely to fill a missing thinking default, `models.list` should be backgrounded or timeout-bounded for initial Control UI startup, and a regression test should use a deferred `loadGatewayModelCatalog()` to prove initial chat rendering and send readiness are not blocked while catalog loading is still pending.

## Risks / Open Questions

- The exact 352-second Windows timing was not reproduced in this read-only review, but current source still matches the reported equal-duration stall shape.
- A fix that lets `chat.history` return before full model-catalog discovery must preserve or deliberately redefine session `thinkingLevel` default semantics.
- Adding a client-side RPC timeout alone would improve UI responsiveness but would not stop server-side work unless paired with a cancellation or backgrounding strategy.
