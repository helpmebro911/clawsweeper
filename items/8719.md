## Summary

Keep #8719 open. Current main has narrower security hardening, audit checks, sandbox/tool policy knobs, plugin trust checks, and an optional `tools.fs.workspaceOnly` guard, but it still does not implement the requested native data-centric Security Profile v1.1. The issue discussion includes an explicit steipete keep-open review and useful external prior art via ClawMoat; neither current code nor the open filesystem-only PathGuard PR supersedes this broader umbrella.

## Best Possible Solution

Keep #8719 open as the product/security umbrella until maintainers either define a native OpenClaw Security Profile or split it into canonical narrower issues. Track filesystem PathGuard work separately, use ClawMoat as complementary external prior art, and create or keep follow-ups for data classification/security levels, protected OS presets, cross-tool FS/exec/export/network policy, sensitive-action audit logs, approval/TTL elevation, session isolation, config integrity, and plugin/skill verification.

## Risks / Open Questions

- Closing as implemented would conflate optional `workspaceOnly`, sandbox defaults, exec approvals, config audit checks, and plugin allowlist warnings with the broader native data-centric profile requested here.
- Closing as duplicate/superseded would lose the umbrella: #60981 and #7722 are filesystem-focused, while #8719 covers cross-tool filesystem/exec/export/network policy, data classification, audit, approval/elevation, session isolation, config integrity, and plugin verification.
- Closing as clawhub would be premature because the remaining ask includes core config schema, tool policy semantics, exec/export/network enforcement, sensitive-action audit events, approval/elevation, session isolation, and plugin verification behavior that cannot be reduced to an external plugin without product/security design work.
