## Summary

Current main still has the focused heartbeat outbound-session routing gap this PR targets. Isolated heartbeat runs switch the agent turn to `runSessionKey`, but outbound delivery is still built with the original base `sessionKey`, and existing main tests do not assert the outbound delivery session key. The PR is not obsolete; it should stay open for maintainer review, with one caveat that the transcript-mirror part of the PR rationale needs confirmation because current delivery appends transcripts only when an explicit `mirror` object is provided.

## Best Possible Solution

Keep this PR open for maintainer review. The likely direction is to route heartbeat outbound delivery metadata to the isolated `runSessionKey` while preserving base-session delivery policy/target context where intended, and to land a regression test that asserts `deliverOutboundPayloads(...).session.key` is the stable `:heartbeat` key for fresh-base and wake re-entry paths. If transcript mirroring is also the intended observable fix, the PR should add or adjust the explicit mirror path rather than relying only on `session.key`.

## Risks / Open Questions

- The PR body says the change fixes transcript mirroring, but current `deliverOutboundPayloads` appends transcript text only when `params.mirror` is present; heartbeat outbound calls currently pass `session` but no explicit `mirror`. Maintainer review should confirm whether the desired fix is outbound session metadata/internal hooks, transcript appending, or both.
- Because `deliverOutboundPayloads` uses `params.session?.policyKey ?? params.session?.key` for payload planning, changing `session.key` to the isolated heartbeat key may also change policy resolution unless the runner preserves the base key as `policySessionKey`. That is reviewable and does not make the PR obsolete.
