## Summary

Keep this issue open. Current main partially fixed the older unconditional `config.patch` SIGUSR1 path, but the specific `browser.profiles.<name>.cdpUrl` workflow remains unresolved because the bundled browser plugin still registers the whole `browser` prefix as restart-required. In `hot` mode the RPC handler still schedules a direct SIGUSR1 for restart-planned paths, and in default `hybrid` mode the config watcher queues the same restart plan.

## Best Possible Solution

Keep this issue open as the canonical tracker for browser-profile config reload behavior. Implement granular browser reload registration so safe profile-scoped paths such as `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` hot-reload or no-op, while preserving restart or reconcile behavior for service-affecting browser settings. Add reload-plan and `config.patch` regression coverage for `hybrid` and `hot` modes, then align the gateway/browser docs with the shipped behavior.

## Risks / Open Questions

- A fix should be granular. Marking all `browser.*` changes hot/no-op could miss service-affecting browser settings, local managed browser launch invariants, or security-sensitive runtime behavior.
- Closing this now would leave a mismatch between docs that classify browser config as no-restart and code/tests that still classify the whole browser prefix as restart-required.
