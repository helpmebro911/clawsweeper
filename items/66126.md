## Summary

Keep this issue open. Current main still handles overflow compaction by retrying the same runner prompt/cron command body, with bounded retry and replay-safety metadata but without the durable turn identity, original inbound message ledger, or stable resume contract requested here. Related #59618 and #63216 remain broader open compaction-resume reports; closed #66128 only mitigated one exec-preflight trigger path, not this retry contract.

## Best Possible Solution

Keep this issue open and fix the embedded runner/subscriber compaction retry contract. Preserve a stable current-turn identity across compaction, carry retry attempt and original inbound message metadata into the resumed attempt, prevent cron/system prompt reconstruction from being treated as fresh input, and add regression coverage for a cron tool-failure cascade that compacts without re-executing deterministic failed tool calls. Keep #59618 and #63216 cross-linked as related compaction-resume work; treat #66128 as only a trigger-path mitigation.
