## Summary

Keep this issue open. Current main has guardrails that bound some retry paths, but it still retries overflow compaction by rebuilding the same prompt and starting another attempt, and the requested durable retry/turn-state marker or reply-obligation ledger is not present. The issue is also recent and the related broader reports are still open, so there is no strong duplicate or shipped-fix basis for cleanup.

## Best Possible Solution

Keep the issue open and fix the compaction retry contract in the Pi runner/subscriber path: preserve a stable current-turn identity across compaction, carry retry attempt metadata into the resumed attempt, avoid treating cron/system prompt reconstruction as a fresh user turn, and add regression coverage showing a failed tool cascade does not re-execute as new input after compaction. Related #59618 and #63216 should stay cross-linked as broader compaction-resume/retry-path reports, but neither is a clean canonical replacement for this narrower cron/system re-injection case.
