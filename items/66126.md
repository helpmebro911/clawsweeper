## Summary

Keep this issue open. Current main still handles overflow compaction by rebuilding the attempt prompt from the original run params and continuing the same retry loop after successful compaction. The retry loop is bounded, and newer replay metadata can mark side-effect risk for some failure paths, but the durable turn identity / original message ledger requested here is still not implemented. The linked exec-preflight bug #66128 was only one trigger path; related #59618 and #63216 remain broader compaction-resume/retry reports, not superseding fixes.

## Best Possible Solution

Keep this issue open and fix the embedded runner/subscriber compaction retry contract. Preserve a stable current-turn identity across compaction, carry retry attempt and original inbound message metadata into the resumed attempt, ensure cron/system prompt reconstruction is not treated as a fresh user turn, gate deterministic tool cascades from being re-executed after compaction, and add regression coverage for a cron/tool failure cascade that compacts without re-injecting the original prompt as a new task. Keep #59618 and #63216 cross-linked as related compaction-resume work; treat closed #66128 as only a trigger-path mitigation.
