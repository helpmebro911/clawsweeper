## Summary

Keep #72157 open. Current main still has the WebChat textarea Enter path guarded only by composition state and connection, while the send button separately disables during `sending`. The host send path clears the draft and marks runs busy synchronously, which may mitigate some double-submit cases, but there is no source-level keyboard guard or regression test covering rapid Enter presses, and the latest release tag has the same Enter-handler shape. The related ClawSweeper items cover configurable Enter behavior or other duplicate-message causes, not this specific rapid keyboard submit path.

## Best Possible Solution

Keep the issue open for a focused Control UI fix and regression test. The best path is to make the textarea Enter handler share the same send eligibility policy as the toolbar controls, while preserving intentional queue behavior for active runs if maintainers still want Enter to queue follow-up messages; add a DOM-level chat view test that fires rapid Enter keydowns and proves no duplicate `chat.send` can be initiated while a send is already pending.
