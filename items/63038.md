## Summary

Keep this PR open. Current main still has the shutdown gap the PR is trying to address, but the submitted fix is not merge-ready: its `flushAll()` snapshots buffered keys once, does not wait for active key chains that no longer have buffer entries, and has no WhatsApp regression test proving `reply`/`sendMedia` finish before `sock.ws.close`. Security review found no unexplained supply-chain, CI, dependency, lockfile, script, secret, or artifact-execution changes in the PR diff.

## Best Possible Solution

Keep this PR open for revision. The best fix is a real debouncer drain/quiescence API in `src/auto-reply/inbound-debounce.ts` that repeatedly flushes current buffers and waits for active keyed chains until stable, then call it from `extensions/whatsapp/src/inbound/monitor.ts` after detaching inbound listeners and before `closeInboundMonitorSocket(sock)`. Add focused WhatsApp regression coverage proving pending debounced messages can still send `reply` and `sendMedia` before close, including late-enqueue and in-flight keyed-task cases, and keep the plugin SDK return shape plus bundled mocks aligned.

## Risks / Open Questions

- Merging the current one-shot `flushAll()` can still close the WhatsApp socket while late debounced messages or active `onFlush` reply/media work are pending.
- Regression coverage is missing for `reply` and `sendMedia` completing before `sock.ws.close`, late enqueue after shutdown starts, and active keyed work with no buffer entry.
- Because `createInboundDebouncer` is exposed through plugin SDK subpaths and runtime types, adding a new return method should keep SDK contracts, bundled plugin wrappers, and mocks aligned.
