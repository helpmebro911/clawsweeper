## Summary

Keep this PR open. Current main still has no debouncer-wide drain before WhatsApp socket teardown, and the PR is a plausible fix path, but the latest PR shape still needs a real quiescence/drain API and focused regression coverage before merge.

## Best Possible Solution

Keep this PR open for revision. The best path is to add a debouncer drain/quiescence API in `src/auto-reply/inbound-debounce.ts` that repeatedly flushes current buffers and waits for active keyed chains until stable, then call it from WhatsApp close after detaching inbound listeners and before socket teardown. Add focused tests proving pending debounced WhatsApp messages can still send `reply` or `sendMedia` before close, including late-enqueue and in-flight keyed-task cases, and keep plugin SDK/mocks aligned with the new return shape.

## Risks / Open Questions

- A one-shot `flushAll()` can still drop messages enqueued by already-running WhatsApp upsert callbacks after the snapshot.
- A `flushAll()` that only looks at current buffers can return while active keyed `onFlush` work is still running, allowing socket close to overtake reply or media sends.
- The PR expands the plugin-facing debounce helper return shape; the final version should keep bundled plugin mocks/types and SDK contract checks aligned.
