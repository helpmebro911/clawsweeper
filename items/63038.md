## Summary

Keep this PR open. Current main still has the reported WhatsApp close-path gap: inbound debounce exposes only `enqueue` and `flushKey`, WhatsApp closes the socket without draining pending debounced work, and the existing close regression only proves `onMessage` eventually receives a batch after timers advance, not that replies/media are sent before socket teardown. The PR is a plausible fix direction, but its `flushAll` design still needs hardening for late enqueues and active keyed flushes as the review comments noted.

## Best Possible Solution

Keep the PR open for author or maintainer follow-up. The best fix is a debouncer-wide drain/quiescence API that flushes all current buffers, waits for active keyed work, handles late enqueues from already-running upsert callbacks, and is called from WhatsApp close after detaching inbound listeners but before `closeInboundMonitorSocket(sock)`. Add focused regression coverage proving pending debounced WhatsApp messages can still perform `reply` or `sendMedia` before socket teardown, including late-enqueue and in-flight keyed-task cases.
