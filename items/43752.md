## Summary

Current main still does not persist the effective gateway port in the lock-file payload: the lock payload type/schema and writer omit `port`, while the CLI/run loop already forwards the resolved port into `acquireGatewayLock`. This PR remains a focused bug-fix candidate, but it needs to be rebased onto current main and updated for the current Zod-backed `LockPayloadSchema`. The provided diff is narrow and does not show a security or supply-chain regression.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `port?: number` to `LockPayload` and `LockPayloadSchema`, persist only valid ports with the existing 1-65535 guard, and land focused `gateway-lock` tests proving provided-port persistence and absent-port backward compatibility.
