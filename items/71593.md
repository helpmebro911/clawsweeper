## Summary

Keep #71593 open. Current main has SecretRef support for built-in env/file/exec providers and manifest-declared plugin config SecretInput paths, but it still lacks a plugin-defined SecretRef provider contract and does not cover all requested core schema surfaces such as `skills.entries.*.env`. The linked AWS Secrets Manager PR #43643 was closed unmerged per the provided GitHub context, and no equivalent provider appears in current main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either explicitly document env/file/exec, especially exec, as the supported extensibility model, or add an additive, versioned plugin SecretRef-provider contract with updates to config types, Zod schemas, plugin SDK/runtime registration, resolver dispatch, Gateway protocol schemas, docs, tests, and any remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- The existing exec provider may be the intended extensibility answer, but it is not the same as a versioned plugin-defined provider contract and needs maintainer product/security judgment.
- Adding plugin-defined providers would touch public config, SDK, resolver, Gateway protocol, docs, and tests, so this should not be auto-closed as conservative cleanup.
