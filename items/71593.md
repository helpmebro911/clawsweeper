## Summary

Keep #71593 open. Current main supports built-in SecretRef sources (`env`, `file`, `exec`) and manifest-declared plugin config SecretInput paths, but it still lacks a plugin-defined SecretRef provider API and does not provide universal SecretInput coverage for core schemas such as `skills.entries.*.env`. The related AWS Secrets Manager PR #43643 was closed unmerged, and no equivalent AWS secret provider appears on current main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported extensibility model, or add a versioned plugin SecretRef-provider contract with matching changes to core types/Zod schemas, plugin SDK registration, resolver dispatch, Gateway protocol schemas, docs, tests, and remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would change public config, plugin SDK, runtime resolver, and Gateway protocol contracts, so this needs explicit maintainer design.
- The existing exec provider may be an acceptable extensibility workaround for some users, but it is not the plugin-defined provider contract requested in #71593.
