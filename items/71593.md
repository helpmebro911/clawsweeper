## Summary

Keep this issue open. Current main still has a closed SecretRef source/provider model (`env`, `file`, `exec`) and no plugin-defined SecretRef provider contract. Manifest `secretInputs` can mark selected `plugins.entries.<id>.config` paths as SecretInput surfaces, but that is not a plugin provider/resolver API and it does not cover the requested `skills.entries.*.env` surface. The engram authToken comment is a useful real-world data point for the remaining gap.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should decide whether `exec` is the supported universal secret-provider extension point and document that clearly, or add a versioned plugin-defined SecretRef provider contract. If adding the contract, align core SecretRef types and schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch, docs, tests, and remaining SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- The existing `exec` provider can wrap many external secret managers today, so maintainers may decide the best product answer is to document `exec` as the supported universal extension point rather than add a plugin-defined provider API.
- A true plugin SecretRef-provider contract would touch public config validation, plugin SDK/runtime registration, resolver dispatch, Gateway protocol schemas, config materialization ordering, docs, and tests.
