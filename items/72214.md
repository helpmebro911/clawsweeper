## Summary

Keep #72214 open. Current main has useful Telegram transport mitigations, request-level timeouts for a few Bot API methods, polling liveness/status tracking, and documented proxy/IPv4 workarounds, but it does not yet prove the central requested behavior: Telegram api.telegram.org/undici timeouts must be isolated so gateway health/RPC remains responsive and the Telegram account is clearly marked degraded. The report is new, detailed, author association is NONE, and has no protected labels, so it remains actionable rather than stale or duplicate cleanup.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin/gateway boundary: add a regression test with a hanging or ETIMEDOUT Telegram fetch/runner stop, bound provider startup and shutdown at the channel-account level, publish a degraded Telegram runtime status with a clear lastError/transport diagnostic, and prove `openclaw health` and `openclaw gateway status` remain responsive while Telegram retries or is disabled/restarted.
