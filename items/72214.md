## Summary

Keep #72214 open. Current main has several Telegram network mitigations, but they do not fully cover the reported product problem: Telegram Bot API transport failure can still affect health/status paths, command-menu startup calls are not all covered by the hard Telegram request timeout list, and the issue’s requested channel-level degraded diagnostics/transport-mode reporting are not implemented.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin plus gateway health surface: add hard timeouts for all Telegram startup/control Bot API calls including deleteMyCommands and setMyCommands, ensure health/status never block on Telegram network startup beyond a short isolated probe budget or cached degraded state, surface transport mode/fallback and last network error in channel status, document apiRoot/proxy/local Bot API workarounds, and add regression tests with a hanging/ETIMEDOUT Telegram fetch proving gateway health/status remain responsive.
