## Summary

Keep this PR open. Current main still lacks a pre-git-update guard or stop/restart preparation for a same-install running gateway, so the reported in-place `dist` replacement race remains real. The PR is a plausible implementation candidate, but the current patch still needs revision because it can miss wrapper-based service commands and can leave a previously running gateway stopped when the git update skips or fails.

## Best Possible Solution

Keep this PR open and request a revised implementation in `src/cli/update-cli/update-command.ts`: detect a same-install running gateway before in-place git checkout/build work, handle direct dist entrypoints and wrapper-backed service commands, stop only when restart/restoration is guaranteed, and restart or restore the prior gateway on success, skipped updates, and errors. Add focused `src/cli/update-cli.test.ts` coverage for absolute, relative, wrapper, different-install, `--no-restart`, dirty/skipped, and failed-update paths.

## Risks / Open Questions

- Merging the current patch can leave a managed gateway stopped if the git update later returns `error` or `skipped`, because current error/skipped paths return before restart handling.
- The current PR matcher can miss wrapper-backed service commands such as `[wrapper, "gateway", "--port", ...]`, allowing the same install's gateway to keep running during in-place dist replacement.
- Closing this PR now would drop a focused fix for a remaining git-mode update race that current main still does not cover.
