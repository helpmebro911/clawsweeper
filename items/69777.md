## Summary

Keep open. Current main still does not provide a strong enough implemented or cannot-reproduce basis for this recent dashboard TTS-after-compaction report. The code supports the prior triage note that compaction does not directly flip `ttsAuto`, but automatic audio replies can still come from session/local/config TTS state or model/tool-directed TTS, and the reporter followed up that it happened twice.

## Best Possible Solution

Keep the issue open and ask for a focused reproduction bundle: redacted `messages.tts` config, `~/.openclaw/settings/tts.json`, affected session entry fields including `ttsAuto` and `compactionCount`, `/tts status` before and after compaction, and a short transcript around the first audio-only reply. If reproduced, add a dashboard `chat.send` plus compaction regression and decide whether internal webchat should suppress automatic TTS unless explicitly requested, or whether the TTS prompt/tool guidance should be stricter.
