## Summary

Keep this PR open. Current main still lacks the dedicated root `--log-file` option and `OPENCLAW_LOG_FILE` file-sink override, while the linked #61295 fix was explicitly scoped to config-backed `logging.file`. The PR is a plausible implementation candidate, but its current diff still has route-first and daemon-status environment issues to resolve before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate, but require revisions before merge: add the root `--log-file` parser/help surface, promote it before both route-first and Commander execution, add an explicit `OPENCLAW_LOG_FILE` resolver with precedence over `logging.file`, thread an env parameter through log-path resolution so daemon status uses the inspected service environment, and keep focused docs/tests for CLI parsing, logger resolution, route-first commands, and daemon status. Maintainers should also confirm that a dedicated CLI flag is the desired product surface now that #61295's config-backed bug is fixed on main.

## Risks / Open Questions

- The PR promotes `--log-file` only in Commander preAction, so route-first commands such as `status`, `gateway status`, `config get`, and `config unset` can silently ignore the new flag.
- The PR's `resolveConfiguredLogFilePath()` consults current `process.env`, but daemon status is supposed to report the inspected service environment; status can therefore display the CLI process log path instead of the service log path.
- The selected log path is a local file-write surface. That is not a supply-chain issue in this diff, but it should remain under the same local operator/config trust model as existing `logging.file`.
