## Summary

Keep open. Current main still sends `tool_choice: "auto"` by default for vLLM-style OpenAI-compatible completions with tools, does not add a vLLM/Qwen compatibility policy, and only offers a generic `extra_body` pass-through workaround rather than a documented or scoped fix for the reported failure mode.

## Best Possible Solution

Keep this issue open for maintainer work. The best path is a vLLM/Qwen-scoped compatibility option, first-class per-model forwarding for a `toolChoice` style setting, or explicit vLLM docs showing the supported `agents.defaults.models["vllm/<model>"].params.extra_body: { tool_choice: "required" }` workaround, with regression coverage proving the override reaches the local vLLM OpenAI-compatible payload without changing native OpenAI behavior globally.

- No live vLLM plus Qwen server was exercised in this read-only review; implementation should include payload-builder regression coverage and ideally a captured or live vLLM verification path.
