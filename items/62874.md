## Summary

Keep open. Current main still defaults local/proxy OpenAI-compatible completions requests with tools to `tool_choice: "auto"`, and the focused vLLM test now explicitly locks that behavior in. There is a generic `params.extra_body` workaround path that can overwrite generated payload keys, but vLLM/Qwen is not fixed by default and the vLLM docs do not document the reporter's `tool_choice: "required"` workaround.

## Best Possible Solution

Keep this issue open for a focused maintainer decision. The likely fix is a vLLM/Qwen-scoped compatibility option, per-model `params.toolChoice` support, or documented `params.extra_body: { tool_choice: "required" }` guidance with tests proving local vLLM/Qwen payloads can force required tool calls without changing native OpenAI behavior globally.
