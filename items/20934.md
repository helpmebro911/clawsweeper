## Summary

Keep #20934 open. Current main still does not expose REST-native /v1/sessions reset/status/compact/model/reasoning endpoints for HTTP chat-completions integrations. The underlying session operations exist on the Gateway RPC/control plane, and there are separate HTTP kill/history and tools/invoke surfaces, but those do not satisfy the requested REST session-management API. Related #23941 was a direct REST attempt but is closed unmerged; #29985 was scoped away from REST and is not evidence that this issue is implemented on current main.

## Best Possible Solution

Keep #20934 open as the canonical REST session-management tracker. The best product/code path is to add authenticated Gateway HTTP endpoints for reset, status/list-or-resolve, compact, and model/reasoning patch operations that reuse the existing sessions.reset, sessions.list/resolve, sessions.compact, and sessions.patch semantics, with route/auth/scope tests and public docs under the OpenAI-compatible HTTP API.

## Risks / Open Questions

- Closing as implemented would hide a real gap for HTTP-only integrations that cannot use the Gateway WebSocket/RPC control plane.
- Closing as duplicate would be incorrect because the direct REST PR is closed unmerged and the open related passthrough issue tracks a different storage/statelessness problem.
- The generic tools/invoke surface may help some automations, but it is not the standardized authenticated REST session-management surface requested here.
