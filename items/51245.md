## Summary

Keep open. Current `main` appears to have fixed the live elevated gating path for Telegram native commands by carrying `Provider: "telegram"` into elevated permission checks, but the issue's `sandbox explain --session telegram:slash:<id>` diagnostic still looks broken on `main`, and the docs still tell users to use that raw Telegram slash key. That is not strong enough evidence to close conservatively.

## Evidence

- **`sandbox explain` still leaves raw slash keys unnormalized:** `normalizeExplainSessionKey()` returns any colon-containing `--session` value unchanged, so `telegram:slash:<id>` is not rewritten to the stored agent-scoped key.
  - file: [src/commands/sandbox-explain.ts:49](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.ts#L49)
  - command: `nl -ba src/commands/sandbox-explain.ts | sed -n '37,59p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Channel inference still only works for agent-scoped session keys:** `inferProviderFromSessionKey()` immediately returns `undefined` when `parseAgentSessionKey()` fails, so a raw `telegram:slash:<id>` key still resolves to `(unknown)` unless a matching store entry already exists under that exact raw key.
  - file: [src/commands/sandbox-explain.ts:65](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/commands/sandbox-explain.ts#L65)
  - command: `nl -ba src/commands/sandbox-explain.ts | sed -n '61,89p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Telegram native commands persist session metadata under agent-scoped slash keys:** Telegram slash command tests assert `recordSessionMetaFromInbound()` is called with `agent:main:telegram:slash:200`, not `telegram:slash:200`. That means `sandbox explain --session telegram:slash:<id>` will miss the stored metadata path it needs.
  - file: [extensions/telegram/src/bot-native-commands.session-meta.test.ts:432](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/telegram/src/bot-native-commands.session-meta.test.ts#L432)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.session-meta.test.ts | sed -n '419,433p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Current Telegram slash session construction is agent-scoped:** `resolveNativeCommandSessionTargets()` builds unbound native command session keys as `agent:<agentId>:telegram:slash:<userId>`, matching the test above.
  - file: [src/channels/native-command-session-targets.ts:15](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/src/channels/native-command-session-targets.ts#L15)
  - command: `nl -ba src/channels/native-command-session-targets.ts | sed -n '12,22p'`
  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Live elevated gating no longer depends on `sandbox explain` channel resolution:** `get-reply-directives` derives `messageProviderKey` from `sessionCtx.Provider`/`ctx.Provider` and passes it straight into `resolveElevatedPermissions()`, and Telegram native commands set `Provider: "telegram"` on the inbound context. This suggests the original runtime block may be obsolete even though the CLI diagnostic remains wrong.

  - file: [docs/tools/slash-commands.md:356](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/docs/tools/slash-commands.md#L356)
