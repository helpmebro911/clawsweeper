## Summary

Keep open. Current main has adjacent Telegram polling watchdog hardening, including a 120s configurable threshold and non-getUpdates API liveness tracking, but it does not contain the PR's specific active-update-handler bypass or pre-handler middleware seam. The PR is stale because Telegram moved from `src/telegram` to `extensions/telegram`, so it needs a current-main port/review rather than closure.

## Best Possible Solution

Keep the PR open for maintainer judgment or revision. The best path is to port only the still-missing active-update-handler protection into the current `extensions/telegram` plugin, likely by adding a narrow handler-lifecycle hook or extending `TelegramPollingLivenessTracker`, with regression coverage for a long-running handler that performs no Telegram API call, bounded bypass expiry, middleware ordering after sequentialize and before handlers, and single restart logging.
