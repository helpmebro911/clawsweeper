## Summary

Keep this PR open. Current main has overlapping Telegram polling watchdog hardening, including a configurable getUpdates/API-liveness threshold, but it still does not implement this PR's central active update-handler bypass or a pre-handler middleware hook that would let polling liveness distinguish a long-running local handler from a dead poller. The branch is stale against the pluginized Telegram layout, but the focused runtime change remains a plausible transplant candidate, and the same author still has the paired split PR #41957 open from #41883.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to transplant the active update-handler watchdog bypass into extensions/telegram/src, likely by integrating bounded active-handler state into TelegramPollingLivenessTracker or adding a narrow bot-core middleware seam after sequentialization and before handler registration. Add a regression test for a long-running update handler that makes no Telegram API call. If maintainers decide the shipped pollingStallThresholdMs/API-liveness implementation is the intended answer, close manually with that product decision.
