## Summary

Keep open. Current main has related Telegram polling watchdog hardening, but it does not implement this PR's central active update-handler bypass or the pre-handler registration hook/order needed to count active handlers precisely. The PR is stale against the pluginized Telegram paths, but the branch still contains plausible focused bug-fix work and the provided discussion shows the only bot-raised logging concern was addressed.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to rebase or transplant the focused active-handler watchdog bypass onto extensions/telegram/src, decide whether it belongs inside TelegramPollingLivenessTracker or behind a narrow pre-handler middleware seam, and add a regression test for a long-running update handler that performs no recent Telegram API call. If maintainers decide the shipped configurable threshold is the intended product answer, close manually with that explicit product decision.
