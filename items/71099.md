## Summary

Keep open. This is a recent Telegram runtime bug report with an explicit maintainer comment saying not to close yet, and current main still does not show a targeted Unicode/ByteString fix or regression test for U+25C6 in the Telegram outbound reply path. The current implementation continues to render/chunk reply text and pass it to grammY sendMessage, while ByteString handling found in the repo is limited to secret/header normalization rather than Telegram response text delivery.

## Best Possible Solution

Keep this issue open. Ask for the stack frame around the ByteString throw or add a minimal current-main repro that sends an embedded-agent response containing U+25C6 through the Telegram reply path. Once the failing boundary is identified, fix the specific header, method, token-like field, or transport adapter receiving non-Latin1 text, and add a Telegram regression test proving valid Unicode response text is delivered without stripping.
