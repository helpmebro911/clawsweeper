## Summary

Keep open. Current main has generic outbound delivery diagnostics and cron run-log delivery state, but I did not find the requested per-attempt structured `cron.delivery.sent` log carrying `cronJobId`, `cronRunId`, `agentId`, `accountId`, and platform message IDs. The companion #69054 is separate lint/config work; this issue’s forensic send-attribution gap remains open.

## Best Possible Solution

Keep the issue open and implement a core cron delivery telemetry path: thread a small privacy-bounded cron delivery context from `dispatchCronDelivery` into the outbound send layer, emit exactly one structured success/error record per attempted cron-owned send with platform message IDs where available, and add focused tests for success, failure, bestEffort partial failure, and retry/idempotency behavior.

## Risks / Open Questions

- The implementation needs to define exactly-once semantics across direct message-tool sends, runner fallback delivery, best-effort partial failures, retries, and cached idempotent deliveries.
- The log shape must preserve the existing privacy boundary: no prompt text, message body, tokens, or sensitive payload content.
