## Summary

Keep open. Current `main` at `7e16e3d07723cc3903f40ce94f53a3920c88b1fc` still implements Telegram processing feedback as typing plus ack/status reactions, not as a transient sent message like `⌛️` that is deleted or edited on completion/failure. I found no `processingIndicator`-style config or code path on current `main`.

## Evidence

- **Telegram docs describe only ack reactions:** The Telegram channel docs still document `ackReaction` as the processing feedback mechanism: an acknowledgement emoji while OpenClaw is processing an inbound message. There is no documented `processingIndicator` message option here.
  - file: [docs/channels/telegram.md:644](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/docs/channels/telegram.md#L644)

  - sha: [7e16e3d07723](https://github.com/openclaw/openclaw/commit/7e16e3d07723cc3903f40ce94f53a3920c88b1fc)
- **Config reference exposes reactions, not indicator messages:** The messages config on current `main` lists `ackReaction`, `ackReactionScope`, `removeAckAfterReply`, and `messages.statusReactions.enabled`. It does not expose `processingIndicator`, `deleteOnReply`, or `showErrorOnFailure`.
  - file: [docs/gateway/configuration-reference.md:2055](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/docs/gateway/configuration-reference.md#L2055)

- **Telegram inbound context sets reactions on the user message:** `extensions/telegram/src/bot-message-context.ts` resolves `ackReaction`, optionally creates a status-reaction controller, and otherwise calls `reactionApi(chatId, msg.message_id, ...)`. This is a reaction lifecycle on the inbound message, not a separate temporary outbound message.
  - file: [extensions/telegram/src/bot-message-context.ts:452](https://github.com/openclaw/openclaw/blob/7e16e3d07723cc3903f40ce94f53a3920c88b1fc/extensions/telegram/src/bot-message-context.ts#L452)

There is no corresponding send/delete/edit lifecycle for a dedicated processing-indicator message in this flow.
