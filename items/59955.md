## Summary

Keep open. One hunk is already superseded on current `main`, but the remaining `forward`/`copy` action addition is incomplete and would introduce broken runtime behavior if merged as proposed.

## Evidence

- **Windows symlink fallback already landed separately:** Current `main` already has a broader Windows fallback in `scripts/stage-bundled-plugin-runtime.mjs`: it copies files or directories when symlink creation fails with `EPERM`, `EINVAL`, or `UNKNOWN`, and threads the fallback through `ensureSymlink(...)`. That supersedes the PR's narrower script-only hunk.
  - file: [scripts/stage-bundled-plugin-runtime.mjs:15](https://github.com/openclaw/openclaw/blob/4d1ee3a7/scripts/stage-bundled-plugin-runtime.mjs#L15)
  - command: `nl -ba scripts/stage-bundled-plugin-runtime.mjs | sed -n '1,120p'`
  - sha: [4d1ee3a7](https://github.com/openclaw/openclaw/commit/4d1ee3a7)
- **Public action list still does not define `forward` or `copy` on main:** `CHANNEL_MESSAGE_ACTION_NAMES` on current `main` does not include `forward` or `copy`, so the PR would be extending the public action union rather than merely syncing an already-supported capability.
  - file: [src/channels/plugins/message-action-names.ts:1](https://github.com/openclaw/openclaw/blob/4d1ee3a7/src/channels/plugins/message-action-names.ts#L1)
  - command: `nl -ba src/channels/plugins/message-action-names.ts | sed -n '1,120p'`
  - sha: [4d1ee3a7](https://github.com/openclaw/openclaw/commit/4d1ee3a7)
- **Target-mode table and target application would disagree for the new actions:** `MESSAGE_ACTION_TARGET_MODE` has no `forward`/`copy` entries. `actionRequiresTarget()` treats any mapped non-`none` action as target-required, but `applyTargetToParams()` falls back unknown actions to `none` and throws `Action <name> does not accept a target.` when `target` is present. Merging only the PR hunk would create exactly that contradiction.
  - file: [src/infra/outbound/message-action-spec.ts:11](https://github.com/openclaw/openclaw/blob/4d1ee3a7/src/infra/outbound/message-action-spec.ts#L11)
  - command: `nl -ba src/infra/outbound/message-action-spec.ts | sed -n '1,180p'`
  - sha: [4d1ee3a7](https://github.com/openclaw/openclaw/commit/4d1ee3a7)
- **Normalization path would still fail the new actions:** `normalizeMessageActionArgs` auto-fills or validates targets based on `actionRequiresTarget(...)`, then calls `applyTargetToParams(...)`, then throws `Action ${action} requires a target.` if no usable target remains. With the PR's new action names but no matching target-mode support, requests can fail in either direction depending on whether a target is supplied.
  - file: [src/infra/outbound/message-action-normalization.ts:36](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/infra/outbound/message-action-normalization.ts#L36)
  - command: `nl -ba src/infra/outbound/message-action-normalization.ts | sed -n '30,90p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **PR discussion already identified the same blocker:** The existing Codex review comment on PR #59955 flagged the missing `MESSAGE_ACTION_TARGET_MODE` entries for `forward` and `copy` as a P1 because the actions would become unusable at runtime. Current `main` still matches that analysis.

## Risks / Open Questions

- Closing this PR now would discard the only remaining unique change without landing a correct end-to-end design for `forward`/`copy` message actions.
- Merging as-is would expose new public action names that current outbound normalization and target handling do not support consistently.
