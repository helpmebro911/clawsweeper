## Summary

Keep open. Current `main` already filters heartbeat/cron/generated transcript noise out of the dreaming corpus, so part of the reported root-cause analysis is outdated. But the core complaint still stands: REM material selection is still driven by recurring concept-tag counts rather than a semantic-importance pass, and there is no user-facing `dreaming.promptStyle` or similar control on current `main`.

## Evidence

- **REM reflections still rank recurring tags by count/strength:** `buildRemReflections()` aggregates `entry.conceptTags`, increments a per-tag count, computes `strength` from tag frequency relative to entry count, sorts by `strength` then `count`, and renders lines like "Theme: `<tag>` kept surfacing across N memories." That matches the issue's claim that REM selection is still recurrence-driven rather than semantically selecting the most meaningful events.
