## Summary

Keep this issue open. Current main still contains the VoiceWakeOverlay reentrant update path described in the report: layout measurement synchronously mutates observable overlay state, while the SwiftUI view still calls window-frame updates directly from the attributed-text onChange handler. The related open menubar PR is a different surface and does not resolve this overlay render-loop report.

## Best Possible Solution

Keep this issue open and fix it in the macOS app owner code. The likely path is to remove synchronous observable-state mutation from measuredHeight(), guard and defer or coalesce isOverflowing updates, defer/coalesce the attributed-text window-frame update, and add focused regression coverage or manual macOS verification for the VoiceWakeOverlay update cycle.
