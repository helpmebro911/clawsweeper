## Summary

Keep open. Current main still lacks Feishu text keyword triggers under requireMention, so this PR is not obsolete. The PR is a plausible implementation candidate, but it needs revision before merge because the current patch does not update broadcast active-agent selection, does not make keyword matches an effective WasMentioned signal, allows empty triggers, and lacks docs/tests/generated config metadata updates.

## Best Possible Solution

Keep the PR open and ask for a revision in the Feishu bundled plugin. The implementation should add a validated non-empty mentionTriggers config surface, resolve it consistently for top-level, account, wildcard group, and exact per-group Feishu config, treat a keyword match as an effective mention for both single-agent and broadcast dispatch, update activeAgentId and WasMentioned semantics, and add focused tests plus Feishu docs/config metadata updates.

## Risks / Open Questions

- Merging the PR as-is would likely make keyword-triggered messages fail silently in Feishu broadcast groups because activeAgentId remains null when ctx.mentionedBot is false and requireMention is true.
- The proposed z.array(z.string()) schema allows empty strings; ctx.content.includes("") is always true, which would bypass requireMention for every group message if a user configures an empty trigger.
- The PR introduces a public Feishu config field without docs, generated config metadata, or focused tests, so setup/config tooling and user-facing documentation would drift.
- Security review pass: the diff does not touch CI workflows, dependencies, lockfiles, package publishing metadata, secrets handling, or downloaded artifacts; the security-relevant concern is the empty-trigger mention-gate bypass.
