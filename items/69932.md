## Summary

Keep PR #69932 open. Current main still has the BlueBubbles monitor typing start/restart/stop calls that the PR targets, and the existing related #65447 work only covers send/attachment Private API refresh behavior, not monitor typing callbacks. The PR discussion also contains a real maintainer-facing tradeoff about treating unknown Private API state as disabled, so this should stay open for review rather than be auto-closed.

## Best Possible Solution

Keep the PR open for maintainer review. The likely path is to land the narrow `monitor-processing.ts` guards plus the disabled-path monitor test if the unknown-state tradeoff is accepted; otherwise ask the author to adjust the implementation to distinguish explicit `false` from unknown or to refresh capability before starting typing.

## Risks / Open Questions

- The PR intentionally treats unknown cached Private API state the same as disabled for monitor typing, which may suppress the first typing bubble after a cold or expired cache on servers where Private API is actually enabled. Maintainers should decide whether that tradeoff is acceptable or whether a tri-state helper or lazy probe is preferred.
