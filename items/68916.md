## Summary

Keep open. Current main still does not close child ACP sessions when a parent session is reset: the reset cleanup path stops subagent runs and closes only the target ACP runtime key, while ACP child sessions spawned via `sessions_spawn(runtime: "acp", mode: "run")` are persisted with `spawnedBy` metadata under their own `agent:<id>:acp:<uuid>` keys. The current docs also still list ACP completion stalls requiring manual process cleanup as an active troubleshooting case, so this should not be auto-closed as implemented or obsolete.

## Best Possible Solution

Keep #68916 open and fix it in OpenClaw’s gateway/session cleanup path. The likely maintainer path is to add a read-only helper that finds ACP session entries across relevant agent stores where `spawnedBy` or `parentSessionKey` matches the reset parent, close/cancel those child ACP runtimes before or alongside the parent reset, and add a regression test that seeds parent plus child ACP entries and proves `acpManager.closeSession()` is called for each child key. Process-tree hardening in `acpx` can be tracked separately, but it does not replace the missing parent-reset cleanup contract.
