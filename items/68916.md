## Summary

Keep this issue open. Current main still closes ACP runtime state only for the session being reset, while ACP sessions spawned through `sessions_spawn({ runtime: "acp" })` are persisted as child sessions with requester ownership metadata. Related duplicate closures now point back to this issue as the canonical remaining tracker, and the docs still describe manual stale ACP process cleanup as troubleshooting, so the requested parent-reset child ACP cleanup is not shipped yet.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely fix belongs in `src/gateway/session-reset-service.ts`: enumerate ACP child sessions for the reset parent using existing session ownership metadata, cancel/close their ACP runtimes before or alongside parent mutation, and add regression coverage proving parent reset closes child ACP sessions created by `sessions_spawn({ runtime: "acp", mode: "run" })`. If live validation still shows adapter descendants surviving close, extend the ACP/acpx process cleanup path with stored PID or process-group fallback and document the shipped behavior.

## Risks / Open Questions

- Current main still appears vulnerable to the reported parent-reset leak: child ACP runtime sessions with `spawnedBy` or `parentSessionKey` matching the reset parent are not cancelled or closed during reset.
- The bundled `acpx` dependency has direct child close logic, but that does not prove the OpenClaw parent-reset cleanup gap is fixed, and process-tree cleanup for adapter descendants should still be live-verified before this can close.
