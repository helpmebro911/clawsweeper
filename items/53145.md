## Summary

Keep open. Current main still leaves the general `sessions_send` A2A announce path unresolved: the exact placeholder is still injected as an inter-session user turn, `NO_REPLY` is not one of the A2A skip tokens, and normal peer sessions still run ping-pong plus announce delivery. Newer chat display suppression helps assistant-side control-token leaks, but it is not a fix for user-provenance inter-session messages or the A2A forwarding loop described here.

## Best Possible Solution

Keep this issue open. The likely fix belongs in `sessions_send` A2A plumbing: mark the announce placeholder as internal/non-displayable or avoid persisting it as a user turn, treat exact silent/control replies such as `NO_REPLY` before any ping-pong or announce delivery, and add focused regression coverage for `sessions_send({ timeoutSeconds: 0 })` peer-session delivery with placeholder, `NO_REPLY`, and acknowledgement follow-up cases.
