## Summary

Current `main` does not implement configurable batching/aggregation of outbound Telegram notifications. OpenClaw has outbound retry/persistence and some heartbeat/system-event coalescing, but Telegram sends still execute immediately per payload/chunk and there is no Telegram config surface for batch windows, grouping strategy, or flush controls.

## Evidence

- **No Telegram batching config:** The Telegram channel config type exposes `replyToMode`, `retry`, heartbeat visibility, silent error replies, and error policy, but no batching/aggregation settings such as interval, max batch size, grouping, or flush-on-shutdown.
  - file: [src/config/types.telegram.ts:122](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/config/types.telegram.ts#L122)
  - command: `sed -n '1,280p' src/config/types.telegram.ts`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Docs mention retry, not outbound batching:** Telegram docs describe limits, chunking, retry, and action gating. The documented outbound control here is `channels.telegram.retry`; there is no documented outbound digest/batching feature.
  - file: [docs/channels/telegram.md:690](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/docs/channels/telegram.md#L690)
  - command: `nl -ba docs/channels/telegram.md | sed -n '430,760p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Outbound delivery sends each payload/chunk immediately:** `deliverOutboundPayloadsCore` iterates normalized payloads and text chunks, awaiting `handler.sendText` / `handler.sendPayload` for each one. This is immediate per-message delivery, not time-window aggregation.
  - file: [src/infra/outbound/deliver.ts:779](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/infra/outbound/deliver.ts#L779)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '760,980p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Telegram send helper is retry-based, not batch-based:** `sendMessageTelegram` builds a non-idempotent retry wrapper and then loops through message chunks, calling `api.sendMessage` for each chunk. This adds retries/backoff behavior, but no outbound queue window or digest formatting.
  - file: [extensions/telegram/src/send.ts:646](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/extensions/telegram/src/send.ts#L646)
  - command: `nl -ba extensions/telegram/src/send.ts | sed -n '617,840p'`
  - sha: [d29eaeafc1fc](https://github.com/openclaw/openclaw/commit/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132)
- **Existing system-event queue is internal coalescing only:** System events are queued in-memory per session with a max of 20 entries and drained into later heartbeat handling. That is internal prompt/event coalescing, not a Telegram outbound notification aggregator.
  - file: [src/infra/system-events.ts:24](https://github.com/openclaw/openclaw/blob/d29eaeafc1fc1ca86f0ccb5c6b2b3d3668502132/src/infra/system-events.ts#L24)

## Risks / Open Questions

- Heartbeat/system-event batching could look superficially related, but it does not provide Telegram-specific digest delivery or user-configurable batch controls.
- There is related demand around Telegram send queuing/rate limiting, but that would still be narrower than this issue's requested aggregation/digest feature.
