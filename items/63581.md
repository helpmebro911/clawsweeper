## Summary

Keep this PR open. Current main still normalizes iMessage echo-cache text with CRLF replacement plus trim only, so leading NUL bytes remain in the cache key. The provided PR is a narrow implementation candidate for the already-linked iMessage NUL-prefix echo bug, and it is not implemented or superseded on main.

## Best Possible Solution

Review and land this narrow iMessage echo-cache normalization fix, or land an equivalent focused patch with regression coverage for single and repeated leading NUL bytes. Broader U+FFFD/control-prefix normalization can stay in a separate maintainer decision such as #62191, but current main does not yet solve this PR's NUL-prefix behavior.
