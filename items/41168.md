## Summary

Current main does not yet implement the core requested TUI behavior. With no `--session`, the TUI still resolves the initial session to the configured main/global key and loads history for that key on connect; it does not select the latest user-active session from `sessions.list`. There are partial mitigations for related storage/context concerns, including compact skill-list docs, session maintenance, and isolated heartbeat session handling, but those do not close the reported startup-session bug.

## Best Possible Solution

Keep this issue open. The best path is to implement a TUI startup resume policy for the no-`--session` case: query sessions for the current agent, choose the latest user-active non-synthetic session using `lastInteractionAt` or an equivalent exposed field, exclude heartbeat/cron/system-only rows, and add TUI plus gateway tests. In parallel, clarify or split the storage concern into session-maintenance/compaction behavior if current maintenance and compact skill-list behavior do not cover the reporter's case.

## Risks / Open Questions

- Choosing by `updatedAt` alone could still select heartbeat, cron, or system-bookkeeping rows; the implementation should use or expose user-activity metadata such as `lastInteractionAt` and filter synthetic heartbeat sessions.
- Changing the default TUI session from `main` to latest-active would alter documented behavior and needs explicit docs and regression tests.
- The storage-bloat half of the report may need a separate focused issue if the remaining problem is transcript/session retention policy rather than TUI session selection.
