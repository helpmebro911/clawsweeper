## Summary

Keep this PR open. Current main still has an unbounded WhatsApp initial connection wait, and the PR remains a plausible focused fix, but it needs to be updated for the current controller-based gateway startup path and the socket cleanup/QR-login constraints already raised in review.

## Best Possible Solution

Keep this PR open and ask for a rebase/update: add an explicit initial WhatsApp connection timeout to waitForWaConnection or the controller openConnection API, use it in the Gateway monitor startup path, preserve unbounded QR login waits unless the login layer supplies its own deadline, ensure timed-out sockets/listeners/timers are cleaned up, and add focused tests for timeout rejection, listener cleanup, socket close on startup timeout, and QR-login behavior.

## Risks / Open Questions

- Applying the PR exactly as shown would likely miss the current WhatsApp Gateway monitor path because monitorWebChannel now starts through WhatsAppConnectionController.openConnection.
- A non-opt-in default timeout would regress QR login flows that can validly stay pending while the user scans or while QR codes refresh.
- Any timeout implementation must close or otherwise transfer ownership of timed-out Baileys sockets so repeated reconnect attempts do not accumulate orphaned sockets.
- The PR’s added timeout tests are directionally useful but should be extended to cover controller startup timeout cleanup and the QR-login no-global-timeout case.
