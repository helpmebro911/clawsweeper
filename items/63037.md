## Summary

Keep this PR open. Current main still has the reported unbounded WhatsApp connection wait, and the PR is aimed at a real availability failure mode, but the submitted branch is not complete enough to close or land as-is because it does not cover the controller-based Gateway startup path and needs explicit socket cleanup on timeout.

## Best Possible Solution

Keep the PR open and request an update: add an opt-in initial WhatsApp connection timeout through waitForWaConnection and WhatsAppConnectionController.openConnection, pass it from both the Gateway monitor startup path and the legacy direct inbound path, close timed-out sockets, preserve unbounded QR login waits unless the login layer supplies its own deadline, and add focused tests for timeout rejection, listener cleanup, controller socket close on startup timeout, direct monitor socket cleanup, and QR-login behavior.

## Risks / Open Questions

- Landing the PR as-is would likely leave the main controller-based WhatsApp Gateway monitor startup path able to hang indefinitely.
- The direct monitor timeout path needs explicit socket close on timeout before rethrowing; otherwise repeated reconnect attempts can leave orphaned WhatsApp sockets.
- A non-opt-in default timeout on waitForWaConnection would regress QR login flows that already have separate QR/login deadlines.
