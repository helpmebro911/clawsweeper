## Summary

Keep this PR open. Current main still has the unbounded WhatsApp connection wait that the PR targets, so the work is relevant, but the submitted patch is not complete enough to close or land as-is because the main controller-based Gateway startup path remains unbounded and the direct monitor timeout path needs explicit socket cleanup on timeout.

## Best Possible Solution

Keep this PR open and request a focused update: add an opt-in initial WhatsApp connection timeout to waitForWaConnection, plumb it through WhatsAppConnectionController.openConnection and the Gateway monitor startup path, pass it from the legacy direct inbound path, close timed-out sockets before retrying, preserve unbounded QR login waits unless the login layer supplies its own deadline, and add regression tests for timeout rejection, listener cleanup, controller socket close on startup timeout, direct monitor socket cleanup, and QR-login behavior.

## Risks / Open Questions

- Landing the PR as-is would likely leave the main controller-based WhatsApp Gateway monitor startup path able to hang indefinitely.
- The direct monitor change in the PR passes a timeout before attachWebInboxToSocket owns the socket; without a catch that closes the just-created socket, repeated timeout/reconnect attempts can orphan WhatsApp sockets.
- A global default timeout for waitForWaConnection would regress QR login flows; the PR’s final default of 0 avoids that, but the complete fix still needs caller-supplied monitor startup deadlines and tests.
