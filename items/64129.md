## Summary

Keep open. Current main still allows the Google Gemini API-key setup path to select a Gemini model as the primary default, and heartbeat still runs as a normal agent turn using the resolved default model unless a heartbeat-specific model is configured. The existing empty/comment-only HEARTBEAT.md skip and optional isolated/lightweight heartbeat settings reduce cost in some configurations, but they do not prevent the actionable scenario reported here: existing non-empty heartbeat instructions can inherit a newly selected paid Gemini provider without an explicit paid-background-use boundary.

## Best Possible Solution

Keep this issue open and fix the paid-background-use boundary explicitly. A conservative maintainer fix should either preserve the existing primary model during Google/provider setup unless the user explicitly confirms switching defaults, or require explicit confirmation/disablement before heartbeat can inherit a newly configured paid external provider. Add regression coverage for setup with an existing primary model plus an actionable HEARTBEAT.md so paid heartbeat usage cannot begin silently.

## Risks / Open Questions

- Users with existing actionable HEARTBEAT.md content can still generate paid Gemini background turns after setup changes the primary model to Google.
- The empty/comment-only heartbeat skip is only a partial mitigation; it does not protect populated heartbeat checklists or missing HEARTBEAT.md runs.
- The current heartbeat `model`, `lightContext`, and `isolatedSession` settings are manual mitigations, not setup-time confirmation or default-preservation boundaries for paid provider auth.
