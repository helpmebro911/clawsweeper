## Summary

Keep open. Current `main` still matches the reported false-positive path: a successful SSH tunnel target is prepended to the configured targets and only exact URL duplicates are removed, while `multiple_gateways` is emitted whenever more than one probed target is reachable. The code already extracts `self` identity from probe presence, but the warning logic does not use it to collapse same-gateway transports. The provided related items cover other gateway-probe false negatives/timeouts/scope behavior and do not supersede this same-identity false-positive report.

## Best Possible Solution

Keep the issue open and fix the gateway-probe warning in `src/commands/gateway-status/output.ts`, likely by grouping reachable probes by a stable gateway identity derived from `self` presence when available, while preserving conservative behavior for probes without identity. Add a focused regression test covering `sshTunnel` plus `configRemote` targets with matching `self` identity and a separate test proving distinct identities still emit `multiple_gateways`. Update the CLI/gateway docs if the warning copy changes.
