## Summary

Keep open. Current main fixes the broad timeout classification from #68700, but it still does not implement the central requested fail-fast/session-busy behavior for direct session write-lock timeouts. A direct `SessionWriteLockTimeoutError` remains unclassified and `runWithModelFallback()` intentionally continues across remaining model candidates for unrecognized errors, so the same locked session can still be retried through the configured fallback chain.

## Best Possible Solution

Keep this issue open and add a targeted non-model-failover path for `SessionWriteLockTimeoutError` or nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` errors at the model fallback wrapper or agent attempt boundary. Add a regression where a fallback chain has multiple candidates and the run callback throws a session write-lock timeout; assert the callback is invoked exactly once and the surfaced error is a clear session-busy/infrastructure error. Queue/retry/backoff can be handled at the session lane or caller boundary, while #69030 remains a separate raw-inference follow-up for one trigger path.

## Risks / Open Questions

- Closing this based only on #68700 would likely miss the remaining direct-lock cascade path: the error is no longer labeled `timeout`, but it can still be retried as an unrecognized error across remaining model candidates.
- The fix should preserve existing fallback behavior for generic unknown provider errors while treating `SessionWriteLockTimeoutError` and nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` errors as non-model infrastructure pressure.
