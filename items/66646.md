## Summary

Keep open. Current main fixed the original broad timeout classifier problem for session write-lock timeouts, but it still does not implement the central requested fail-fast/session-busy behavior. `SessionWriteLockTimeoutError` is now unclassified, yet `runWithModelFallback()` still continues past unrecognized errors when there are remaining fallback candidates, so a lock timeout can still spend attempts on additional models before surfacing.

## Best Possible Solution

Keep this issue open and implement a targeted non-model-failover path for `SessionWriteLockTimeoutError` or nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` errors at the fallback wrapper or agent attempt boundary. The regression should configure multiple model candidates, throw a session write-lock timeout from the run callback, assert the callback is invoked exactly once, and surface a clear session-busy/infrastructure error. Queue/retry/backoff can be handled at the session lane or caller boundary; #69030 remains a narrower raw-inference trigger follow-up, not a replacement.
