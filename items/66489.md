## Summary

Current main still does not classify the reported local-media allowlist rejection as a permanent delivery error. The delivery queue classifier lacks both requested patterns, the local media guard still emits the exact reported error, and queue failures still increment retryCount unless the classifier matches. A newer agent CLI media-path normalizer fixes one upstream relative-MEDIA source, but it does not cover already queued delivery payloads or the delivery-queue permanent-error policy requested here.

## Best Possible Solution

Keep this issue open. The narrow fix belongs in src/infra/outbound/delivery-queue-recovery.ts and its policy/recovery tests: classify the exact local-media allowlist rejection as permanent so recovery moves poisoned entries directly to failed, then separately verify whether Telegram `message is too long` still reaches the queue before adding that pattern. Enqueue-time validation and a lastErrorClass field can be follow-up improvements, but they are not present on current main.

## Risks / Open Questions

- A broad media-error regex could incorrectly skip retries for fixable local-media failures such as temporary staging, missing files, or changing allowed roots; the implementation should match the deterministic allowlist rejection precisely.
- The proposed Telegram `message is too long` classifier should be verified against current Telegram chunking and media-caption behavior before adding it, because text delivery now has channel chunking safeguards and the exact failure surface may differ.
