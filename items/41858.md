## Summary

Keep this PR open. Current main still starts chokidar watchers for both skills and memory without a `depth` limit, and the linked FD-exhaustion problem is not fully implemented on main. The PR is stale against the current memory-core file layout, but it remains the canonical focused PR after #43056 was closed as a duplicate/superseded variant.

## Best Possible Solution

Keep this PR open as the canonical watcher-depth/resource-hardening thread. Rebase or replace it with a current-main patch against `src/agents/skills/refresh.ts` and `extensions/memory-core/src/memory/manager-sync-ops.ts`, choose a memory watcher resource boundary that preserves documented recursive memory indexing, handle symlinks and broad `extraPaths` deliberately, and add focused watcher-option/regression tests plus manual FD-pressure proof if feasible.

## Risks / Open Questions

- The PR likely will not apply cleanly until it is rebased from `src/memory/manager-sync-ops.ts` to `extensions/memory-core/src/memory/manager-sync-ops.ts`.
- A naive memory depth cap can silently miss deeply nested indexed memory or `extraPaths` files, so the final fix needs explicit semantics and tests for nested note trees.
- The current PR does not fully address symlink traversal or accidentally broad `extraPaths`; maintainers should decide whether `followSymlinks: false`, extra path validation, or a depth cap is the right resource boundary.
