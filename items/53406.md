## Summary

Keep this issue open. Current main has ACP thread binding, streaming output, explicit `/acp steer`, and native embedded-agent queue steering, but ordinary messages in a bound ACP chat/thread still enter the serialized ACP turn path instead of reaching the active runtime before the current turn completes. Native `/subagents steer` is also restart-based, so the requested same-turn input behavior is still not fully implemented.

## Best Possible Solution

Add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP runtime/manager contract, and route bound chat/thread messages to that path while a turn is active. Add regression coverage where a second bound Discord/chat message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, make the product policy explicit and either implement live same-turn steering for bound thread replies or document/test the remaining queued/restart behavior.

## Risks / Open Questions

- ACP backends may not all support live same-turn input, so the eventual contract needs explicit capability/fallback behavior to avoid dropping messages or surprising users.
- Maintainers still need to choose the native sub-agent policy: ordinary thread replies can use live queue steering where supported, remain follow-up turns, or keep explicit restart-based `/subagents steer`; that policy is not settled by current docs/tests.
