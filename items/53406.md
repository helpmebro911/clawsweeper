## Summary

Keep this issue open. Current main has substantial related ACP/session-control work, including bound ACP dispatch, `/acp steer`, `/subagents steer`, and configurable queue steering for normal embedded runs, but ordinary messages in a bound Discord/chat thread still do not become live input to an active ACP turn. The ACP path sends normal bound replies as prompt turns, and the ACP manager serializes turns per session, so a second message waits behind the running turn instead of steering it mid-run.

## Best Possible Solution

Keep the issue open. The best path is to add a true active-turn input path for bound chat sessions: for ACP, introduce a manager/runtime capability that can deliver steering/input to the active runtime without waiting behind the per-session actor queue, with backend support or a documented fallback; for native sub-agent/thread sessions, make an explicit product decision on whether ordinary bound thread replies should default to live steering or respect `messages.queue`, then add Discord/thread-bound regression tests for active ACP and native sub-agent runs.
