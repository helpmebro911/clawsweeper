## Summary

Keep #71803 open. Current main still has no-output watchdog paths that are driven only by CLI stdout/stderr activity, with no pending Monitor/MCP/tool-call expected-idle state. The issue is recent, detailed, unprotected, and distinct from the broader open watchdog-timeout items in the provided related context.

## Best Possible Solution

Keep open. The best product/code path is to teach the CLI backend watchdog about pending Monitor/MCP tool-call liveness: suspend or heartbeat the no-output timer while a bounded Monitor task is pending, keep the overall run timeout enforced, and add a regression test that stays silent past noOutputTimeoutMs during a pending Monitor wait and then completes when the monitor emits output.
