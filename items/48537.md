## Summary

Keep open. Current main still has the broad Telegram fetch abort behavior this PR is trying to narrow: the polling-cycle abort signal is wired into every Telegram API fetch, and current tests still assert that a generic client fetch aborts when that signal is aborted. The PR discussion also contains real user-facing logs matching the proposed fix direction, so this is not obsolete cleanup.

## Best Possible Solution

Keep the PR or equivalent work open, rebase it onto the current bot-core.ts/polling-session.ts split, and scope the polling-cycle abort signal to getUpdates only. Add regression coverage that exercises the actual bot API transformer/fetch path, including the throttler, so getUpdates still aborts on restart while non-polling sends are allowed to finish.
