## Summary

Keep this PR open. Current main still does not expose `gateway.mcpLoopback.port` or any operator-facing way to pin the bundled MCP loopback port. The original session-reset symptom is partly addressed by `mcpResumeHash` canonicalizing OpenClaw loopback port changes, but the configurable-port feature requested by this PR remains unimplemented. The PR needs a rebase because current main starts MCP loopback lazily from the CLI backend preparation path, not the older eager gateway startup path patched by the PR.

## Best Possible Solution

Keep this PR open but require a rebase against current main. If maintainers still want stable operator-configured MCP URLs, add `gateway.mcpLoopback.port` to the gateway config type, strict Zod schema, generated schema/help/labels, and public gateway configuration docs, then pass the configured value into the current lazy CLI backend `ensureMcpLoopbackServer(configuredPort)` path with tests for configured-port binding and bind-failure diagnostics. If maintainers decide the stable `mcpResumeHash` behavior fully removes the product need, close only after recording that product decision or linking a canonical follow-up.

## Risks / Open Questions

- Merging the PR as-is would not wire the configured port into the current lazy CLI-backend `ensureMcpLoopbackServer()` startup path.
- The provided PR diff appears to use a `configuredMcpLoopbackPort` variable in `catch` that is declared inside the preceding `try` block in `src/gateway/server.impl.ts`, so the branch likely needs a compile-fix during rebase.
- A fixed predictable local port can collide during restart; the rebased implementation should keep useful configured-port diagnostics and preserve loopback-only binding plus bearer-token protection.
- Because `mcpResumeHash` already avoids session invalidation when only the OpenClaw loopback port changes, maintainers should confirm whether a new public gateway config key is still desired as an operator feature.
