## Summary

Keep open. Current main still has the direct `buffer` attachment false-success path: the shared `message.send` schema advertises `buffer`, `filename`, `contentType`, and `mimeType`, but ordinary `send` is not hydrated and the Discord send bridge only forwards media path/URL fields into the Discord upload runtime. The path/filePath Discord upload route appears implemented and tested, so the remaining issue should be narrowed to buffer-backed ordinary `message.send`.

## Best Possible Solution

Keep this issue open and narrow it to direct buffer attachments on ordinary Discord `message.send`. The best fix is to make `buffer`, `filename`, and `contentType`/`mimeType` flow end to end through shared send hydration and the Discord multipart upload path with regression coverage, or explicitly reject those fields for ordinary send if direct buffers are not intended to be supported. Review #57335 and #63578 as candidate fixes, but do not close until one is merged and covered on main.
