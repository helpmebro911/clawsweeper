## Summary

Keep open. Current main appears to have fixed the Discord filePath/path upload and multipart payload shape, but not the buffer-backed `message.send` path reported here. The public send schema still advertises `buffer`/`filename`/`contentType`/`mimeType`, while the shared send runner and Discord action runtime only collect URL/path-style media and forward `mediaUrl` plus `filename`. A buffer send can still be accepted without being uploaded or explicitly rejected.

## Best Possible Solution

Keep this issue open and narrow it to direct buffer attachments on Discord `message.send`. The best fix is to either hydrate and deliver `buffer` + `filename` + `contentType`/`mimeType` end to end for ordinary send, with Discord regression coverage that a buffer-backed image reaches the multipart `files` upload path, or remove/fail-fast those fields from the send schema if buffers are intentionally unsupported.
