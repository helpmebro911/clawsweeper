## Summary

Keep #70050 open. Current main still performs default enforced session-store maintenance synchronously during `loadSessionStore()`: it reads/parses the file, normalizes/migrates entries, then prunes and caps oversized stores before returning. The cap path still sorts all removable keys. Tests assert this load-time prune/cap behavior, and the only later changelog evidence I found addresses honoring configured maintenance settings, not deferring or bounding the startup work that #70050 reports.

## Best Possible Solution

Keep the issue open and fix the load-time maintenance path without losing #69404's OOM protection: defer cleanup until after the gateway is responsive, chunk/yield long prune/cap work, replace full sorting with bounded selection where practical, and add a regression/performance test around a large `sessions.json` proving startup/`chat.history` are not blocked for minutes. Also reconcile docs that still describe warn-mode defaults with the enforced default in code/tests.
