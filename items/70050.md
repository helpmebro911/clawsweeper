## Summary

Keep #70050 open. Current main still performs default enforced session-store maintenance synchronously inside `loadSessionStore`, and `chat.history` still reaches that path through `loadSessionEntry`. The provided issue discussion also contains a concrete triage comment pointing at the same code path. There is not enough evidence to close this as fixed or obsolete.

## Best Possible Solution

Keep the issue open and fix the startup path directly. The likely maintainer direction is to keep bounded session storage but avoid blocking gateway readiness: defer load-time maintenance to a background pass, chunk/yield the prune and cap work, or move enforcement to an explicit/write-path cleanup while preserving a documented `session.maintenance.mode: "warn"` workaround. Add a regression or performance-oriented test around an oversized `sessions.json` plus `chat.history`/gateway startup, then update docs/config help so defaults and mitigation guidance match runtime behavior.
