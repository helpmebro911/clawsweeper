## Summary

Keep open. Current `main` still does synchronous load-time session maintenance for oversized stores, the default mode is still `enforce`, and gateway/session-history paths still call that loader directly. I found no evidence on current `main` of an async/deferred startup fix for this report.

## Risks / Open Questions

- I did not reproduce the full 20-minute delay locally because the report depends on the reporter's Windows/QQ environment and store size; this is a code-path review against current `main` rather than a live timing repro.
- `gh` inspection was unavailable in this environment because `GH_TOKEN` was not set, so GitHub-side verification was limited to the issue/timeline context provided in the prompt.
- There is a docs/code mismatch on the default maintenance mode (`docs/concepts/session.md` says `warn` by default, while code/tests show `enforce`), which increases the chance of user confusion but does not weaken the keep-open conclusion.
