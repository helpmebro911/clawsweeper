## Summary

Keep open. Current main still implements memory-flush write authority as a single daily-memory target, and the write wrapper still rejects `SESSION_HANDOFF.md` or any other configured extra target. The linked implementation attempt, #66765, is closed and unmerged, while adjacent open reports #63664 and #68995 overlap with the same write-policy area but do not fully supersede this specific handoff-file request.

## Best Possible Solution

Keep this issue open and fix the memory-flush contract deliberately: add validated additional write targets to the memory flush plan or config, thread them through the embedded runner and tool creation, allow full overwrite for an explicit `SESSION_HANDOFF.md` target while preserving append-only behavior for the daily memory file, and add focused tests for allowed handoff writes, unrelated-path rejection, and path escapes.

## Risks / Open Questions

- Allowing arbitrary paths inferred from free-form prompt text would widen memory-flush write authority; the safer fix is explicit, validated plan/config targets with path traversal rejection.
- The bundled default prompt now targets only `memory/YYYY-MM-DD.md`, so the remaining confirmed bug is for custom or recommended prompts that explicitly ask the flush turn to update `SESSION_HANDOFF.md`.
- Related open issues #63664 and #68995 overlap with memory-flush write restrictions, so the final contract should be coordinated rather than patched narrowly in one call site.
