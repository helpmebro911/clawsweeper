## Summary

Keep open. Current main still gives memory-flush runs exactly one append-only write target, `memory/YYYY-MM-DD.md`; a flush prompt that asks for `SESSION_HANDOFF.md` would still be rejected by the write wrapper. The bundled default/docs now point only at the daily memory file, but the configurable-prompt case in this issue remains real. The prior implementation PR #66765 is closed and unmerged, and related open issues #63664 and #68995 overlap without superseding this specific handoff-file request.

## Best Possible Solution

Keep this issue open and fix the memory-flush write contract deliberately: add an explicit validated additional-write-target mechanism to the memory flush plan/config or plugin contract; thread it through the embedded runner and tool creation; allow full overwrite for `SESSION_HANDOFF.md` while preserving append-only behavior for the daily memory file; reject unrelated paths and path escapes; and add regression tests for allowed handoff writes, daily-memory append behavior, unrelated-path rejection, and traversal attempts.

## Risks / Open Questions

- Allowing arbitrary paths inferred from free-form prompt text would widen flush-time write authority; a fix should use explicit, validated additional targets with workspace-root and traversal checks.
- The default bundled prompt no longer mentions `SESSION_HANDOFF.md`, so maintainers should frame this as a configurable additional-write-target bug rather than a current default-prompt mismatch.
- Related open issues #63664 and #68995 touch the same write-policy area, so the final contract should be coordinated instead of patching only the error check.
