## Summary

Keep open. The issue is recent, externally reported, and plausibly still actionable: the discussion includes both an original abandoned-answer report and a later 2026.4.9 reproduction where post-compaction work ran but client delivery disappeared. A related mainline fix (#64300/#64232) appears to make replay/liveness failures observable, but I did not find high-confidence evidence that current main universally resumes and delivers ordinary post-compaction output for the generic/openai-completions path described in the comment.

## Evidence

- **Issue discussion remains concrete and recent:** The provided issue body describes compaction occurring before the final assistant synthesis, and PlaidJohn's Apr 11 comment reports a 2026.4.9 openai-completions/vLLM reproduction where retry work produced real file writes but the TUI/websocket pipeline never received the output.
  - command: `provided GitHub context for issue #59618`
- **Current docs do not prove the requested universal resume/delivery behavior:** The session-management compaction docs describe overflow recovery as compact-then-retry and threshold maintenance as after a successful turn. That supports existing compaction semantics, but it does not establish that mid-turn auto-compaction always resumes and delivers the interrupted response.
  - file: [docs/reference/session-management-compaction.md:18](https://github.com/openclaw/openclaw/blob/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386/docs/reference/session-management-compaction.md#L18)
  - command: `opened docs/reference/session-management-compaction.md via GitHub raw/web because local sandbox commands failed`
  - sha: [afd6b5d6fce1](https://github.com/openclaw/openclaw/commit/afd6b5d6fce1dac98539d8b45bd0f9db9ff25386)
- **Related fix is narrower than this report:** PR #64300 merged on Apr 11 and adds replay/liveness state surfacing, classifying outcomes as working/paused/blocked/abandoned and preserving replay-invalid truth across compaction retries. Its stated non-goals also say it does not change generic runner behavior outside provider-owned hooks and replay/liveness surfacing, so it is not enough evidence to close this generic compaction resume/delivery issue.
  - command: `opened https://github.com/openclaw/openclaw/pull/64300`
- **Maintainer-authored follow-up exists but does not make this issue redundant with high confidence:** Issue #64232 was opened Apr 10 to classify replay/liveness failures and prevent silent long-task abandonment, including stale continuation/compaction abandonment handling. Its acceptance criteria focus on observable state transitions and truthful progress state, not necessarily universal same-turn resume or client delivery for the openai-completions/vLLM scenario described here.
