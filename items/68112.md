## Summary

Keep this PR open. Current main still has the startup catch-up call before the first cron timer arm with no try/catch/finally, and the catch-up path can still propagate persistence failures. The PR is a focused implementation candidate for a real remaining cron scheduler resilience gap, not an obsolete cleanup item.

## Best Possible Solution

Keep this PR open for maintainer review. Either land this PR after validating the latest head against startup catch-up persistence failures, one-shot interruption semantics, timeout-disabled jobs, and final startup persist failures, or replace it with an equivalent main-branch fix that guarantees `start()` arms the cron timer even when startup catch-up or repair persistence fails.
