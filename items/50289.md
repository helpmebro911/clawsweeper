## Summary

Keep open. Current main partially addresses the reported config-parse portion through pinned runtime config snapshots, but the specific large `commands.ownerAllowFrom` hot path still normalizes the owner allowlist on each authorization pass and still uses linear array membership checks. That leaves the central performance concern unresolved enough that this should not be auto-closed.

## Evidence

- **Owner allowlist normalization still walks every configured entry:** `resolveOwnerAllowFromList` reads `params.allowFrom ?? params.cfg.commands?.ownerAllowFrom` and iterates `for (const entry of raw)`, normalizing each entry before calling `formatAllowFromList`.
  - file: [src/auto-reply/command-auth.ts:270](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/auto-reply/command-auth.ts#L270)
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
- **The config and context owner lists are still resolved during command authorization:** `resolveOwnerAuthorizationState` calls `resolveOwnerAllowFromList` for both `configOwnerAllowFrom` and `contextOwnerAllowFrom`, and `resolveCommandAuthorization` passes `cfg.commands?.ownerAllowFrom` into that state builder.
  - file: [src/auto-reply/command-auth.ts:382](https://github.com/openclaw/openclaw/blob/a1090b604312ac7a49cf2d2ce7276b334786bb79/src/auto-reply/command-auth.ts#L382)
  - sha: [a1090b604312](https://github.com/openclaw/openclaw/commit/a1090b604312ac7a49cf2d2ce7276b334786bb79)
