## Summary

Keep this issue open. Current main narrows the session write-lock scope around slow setup, but MCP catalog materialization and MCP tool execution can still block the reply path without per-list/per-call budgets, circuit state, degraded empty-tool fallback, or user-facing diagnostics. Related #65560 improved a narrower Chrome MCP attach diagnostic, but it does not supersede this broader resilience request.

## Best Possible Solution

Keep this issue open and implement a core resilience layer around external MCP/plugin work: configurable short catalog and tool-call budgets, repeated-failure cooldown/circuit state, degraded empty-tool materialization for optional unhealthy MCP capability, status/log diagnostics that distinguish channel health from external capability health, and regression tests for slow catalog listing, slow callTool, repeated failure cooldown, and blocking plugin hook phases.
