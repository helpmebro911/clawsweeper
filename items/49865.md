## Summary

Keep open. Current main has implemented substantial mitigations for several parts of #49865, including lighter gateway status probes, degraded SecretRef diagnostics, cron retry/failure notification support, ACPX packaging improvements, and orphan transcript cleanup. However the central SecretRef parity ask is not fully implemented: current code still treats auth SecretRefs as potentially unresolved in the CLI command path and probes without configured auth credentials rather than resolving from the already-running Windows gateway context. Because that remaining capability is a meaningful part of the issue and no single narrower canonical issue covers the whole remainder, this should stay open for maintainer judgment.

## Evidence

- **SecretRef parity remains a degraded path:** Gateway status auth resolution catches unresolved SecretRef errors and returns empty probe auth plus a warning: `${path} SecretRef is unresolved in this command path; probing without configured auth credentials.` This is a graceful diagnostic path, not parity with the gateway process that already resolved the secret.
