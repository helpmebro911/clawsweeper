## Summary

Keep #9912 open. Current main still lacks the requested global or per-agent hard budget for model/tool turns, total tool calls, or timeout-after-first-tool-call behavior. Existing `tools.loopDetection`, run timeouts, and accepted-but-ignored OpenResponses `max_tool_calls` are related mitigations, but they do not implement the requested config contract or forced final-response behavior.

## Best Possible Solution

Keep #9912 open as the canonical global/per-agent hard iteration budget request. A complete fix should add typed config, strict schema/help, generated config docs, and public docs for max model-tool turns and/or total tool calls at `agents.defaults` and per-agent scope; define timeout-after-first-tool-call semantics; enforce the limits in the embedded/tool-call runtime; add focused tests for limit exhaustion and final-response behavior; and keep #13271 separate for per-channel budgets and auto-delegation.

## Risks / Open Questions

- Existing `tools.loopDetection` can mitigate some runaway repeated-call patterns, so an eventual implementation needs clear interaction rules between hard budgets, loop detection warnings/blocks, whole-run timeout, and LLM idle timeout.
- Maintainers still need to choose exhaustion semantics: block the next tool call, inject a model-visible warning and allow one final response, return the last useful tool result directly, or combine those behaviors.
