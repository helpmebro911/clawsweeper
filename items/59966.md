## Summary

Current `main` still contains a hardcoded 120s timeout in the Synology Chat webhook reply path, while the core agent timeout code already respects `agents.defaults.timeoutSeconds`. The issue is therefore still plausible on current `main`, not something to close as already fixed or obsolete.

## Evidence

- **Hardcoded 120s timeout still exists on main:** `processAuthorizedSynologyWebhook()` races the channel delivery promise against `setTimeout(..., 120_000)` and throws the exact error text `Agent response timeout (120s)`.
  - file: [extensions/synology-chat/src/webhook-handler.ts:562](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/synology-chat/src/webhook-handler.ts#L562)
  - command: `nl -ba extensions/synology-chat/src/webhook-handler.ts | sed -n '520,590p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **The hardcoded timeout wraps the real inbound agent dispatch:** The Synology gateway passes webhook messages into `dispatchSynologyChatInboundTurn()`, and that function awaits `dispatchReplyWithBufferedBlockDispatcher(...)` before returning. The 120s race therefore caps the end-to-end reply wait for this channel path.
  - file: [extensions/synology-chat/src/gateway-runtime.ts:180](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/extensions/synology-chat/src/gateway-runtime.ts#L180)
  - command: `nl -ba extensions/synology-chat/src/gateway-runtime.ts | sed -n '168,205p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Core agent timeout honors config:** `resolveAgentTimeoutSeconds()` reads `cfg?.agents?.defaults?.timeoutSeconds`, and `resolveAgentTimeoutMs()` uses that configured value as the default runtime timeout.
  - file: [src/agents/timeout.ts:9](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/timeout.ts#L9)
  - command: `nl -ba src/agents/timeout.ts | sed -n '1,220p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **120s is only the idle-timeout fallback in current core logic:** `resolveLlmIdleTimeoutMs()` first prefers `agents.defaults.llm.idleTimeoutSeconds`, then `runTimeoutMs`, then `agents.defaults.timeoutSeconds`; it only falls back to the default 120s value when none of those are set.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:45](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts#L45)

  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Docs match the core behavior, not the channel override:** The docs say agent runtime timeout comes from `agents.defaults.timeoutSeconds`, and 120s is the fallback only for LLM idle timeout when no explicit timeout is configured.
  - file: [docs/concepts/agent-loop.md:166](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/docs/concepts/agent-loop.md#L166)
