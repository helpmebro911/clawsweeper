## Summary

Keep #71382 open. Current main at 4a3030df does not implement a WhatsApp `readOnly` config or transport-level outbound gate, and the generic `session.sendPolicy` path only suppresses reply delivery rather than all WhatsApp transport sends such as pairing replies, read receipts, presence, typing, polls, reactions, or explicit send APIs. The PR is still relevant, but its review thread identifies actionable follow-ups before merge.

## Best Possible Solution

Keep the PR open and have the author or a maintainer update the WhatsApp plugin implementation so `readOnly` is a documented channel/account config field, enforced at the WhatsApp transport boundary for all outbound paths, while still preserving non-sending pairing request creation. Then rerun the targeted WhatsApp tests and extension typecheck listed in the PR.
