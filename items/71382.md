## Summary

Keep #71382 open. Current main does not implement a WhatsApp channel/account `readOnly` mode, and the existing generic `session.sendPolicy` only suppresses session reply delivery, not WhatsApp transport-level sends such as presence, read receipts, pairing replies, reactions, polls, or direct outbound sends. The PR remains a focused implementation path, but the discussion still has concrete review follow-ups before merge.

## Best Possible Solution

Keep the PR open. The best path is for the branch owner or a maintainer with push access to finish the transport-layer WhatsApp `readOnly` implementation, including the triaged fixes for empty-content sends, composing behavior, and pairing request creation without outbound pairing replies, then validate with targeted WhatsApp tests and extension typecheck. If the branch stalls, a maintainer replacement patch should preserve the same transport-level safety boundary rather than relying only on `session.sendPolicy`.
