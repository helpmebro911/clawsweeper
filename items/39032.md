## Summary

Keep open. Current main still has the same unfixed extractor path: `toolResult` / `tool` messages are converted into completion fallback text without checking top-level `isError`, block-level `is_error`, or status-based failed-tool markers, and that text can flow into requester-visible subagent completion delivery. The direct PR #39367 was closed unmerged, and the merged adjacent hardening in #59649 does not appear to cover this extractor path.

## Best Possible Solution

Keep the issue open and implement the narrow extractor fix in `src/agents/subagent-announce-output.ts`: detect failed tool results before populating `latestRawText`, reuse or mirror the existing `isToolResultError` semantics plus `isError`/`is_error` marker handling, and add focused regression tests proving failed tool results are ignored while successful tool-only completion fallback still works.

## Risks / Open Questions

- A fix must preserve the intentional successful tool-result fallback while dropping only failed tool results.
- The guard should cover the marker shapes present across the codebase: top-level `isError`, content/block-level `is_error`, and status-based failures such as `details.status: "error"` or `"timeout"`.
- If the subagent writes debugging text as a normal assistant reply rather than only as failed tool output, the longer-term assistant-result separation may still need a separate follow-up.
