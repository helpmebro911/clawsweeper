## Summary

Current main still does not implement the requested ACP binding notifyPolicy config or a core bound-thread duplicate guard. The issue discussion already identified the same two gaps; current code still has task notify policies only at task creation/update level, persistent ACP binding config still cannot carry notifyPolicy, ACP spawn task records still omit notifyPolicy and therefore fall back to done_only, and existing duplicate suppression is limited to parent-owned one-shot ACP children or the optional Slack thread-ownership plugin rather than active persistent ACP-bound threads.

## Best Possible Solution

Keep this issue open. Add typed notifyPolicy support to AgentRuntimeAcpConfig and/or AgentAcpBinding.acp, update strict Zod schemas, generated/help docs, public docs, binding normalization/materialization, and ACP task creation so persistent ACP-bound conversations can resolve silent notifications before task records are created. Separately design a channel/account/conversation/thread ownership guard for active ACP-bound sessions, with Slack persistent-bound-thread tests that distinguish duplicate automated replies from intentional manual/operator replies.
