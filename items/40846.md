## Summary

Keep this PR open. Current main does not implement automatic macOS idle-sleep prevention for the gateway, and the PR is a focused candidate for a real gateway lifecycle gap. It should be revised before merge because the submitted patch introduces new process execution and has an acknowledged async spawn-error bug with no test coverage.

## Best Possible Solution

Keep this PR open for maintainer review and revision. The best path is a small macOS-owned gateway lifecycle change that starts a trusted `caffeinate -i -w <gateway pid>` assertion only while the gateway is alive, handles async child `error`/`exit` events, cleans up in the existing run-loop finally path, and adds focused tests that mock platform and child_process behavior. If maintainers prefer not to change default power behavior, document the existing wrapper/manual caffeinate path instead of closing this as already implemented.

## Risks / Open Questions

- The PR is not safe to close as obsolete: current main still lacks the automatic macOS idle-sleep prevention it proposes.
- The submitted patch should not be merged as-is because a missing or inaccessible `caffeinate` binary can emit an unhandled async error unless the child process has an `error` listener.
- The patch adds new process execution during gateway startup. A revision should prefer a trusted macOS path or explicit resolution, keep stdio inert, and document/test the lifecycle behavior.
- The power-management behavior is user-visible and may need an opt-out or explicit maintainer product decision if preventing idle sleep by default is too broad.
