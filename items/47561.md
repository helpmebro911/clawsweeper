## Summary

Keep open. Current main still derives the active Telegram DM session key from the current `session.dmScope`, so switching from `per-channel-peer` back to `main` changes the conversation from `agent:main:telegram:direct:<peer>` to `agent:main:main`. `/new`/`/reset` rotate only the targeted session entry; I did not find a migration, rebinding, quarantine, or warning that would keep the fresh per-channel session authoritative after that routing-mode change. Related #69599 and #47504 are open and adjacent, but they do not supersede this specific Telegram routing transition report.

## Evidence

- **Route resolution still keys DMs from current dmScope:** `resolveAgentRoute` reads `input.cfg.session?.dmScope ?? "main"`, passes that into `buildAgentSessionKey`, then derives `lastRoutePolicy` from whether the resulting `sessionKey` equals `mainSessionKey`. That means a config change from isolated DM scope to main changes the resolved session key.
