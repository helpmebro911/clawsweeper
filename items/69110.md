## Summary

Keep open. Current main still matches the reported gap: `message_sending` delivery hooks receive content, routing fields, channel/account/media metadata, and a minimal context, but no resolved provider/model. The agent loop does know the actual fallback-selected provider/model for usage, response-prefix interpolation, and session persistence, and `llm_output` events expose provider/model, but that observation hook is not a synchronous delivery enforcement point. Related PR #65149 is a plausible narrower implementation path, but it is still open and unmerged, so the issue should remain visible.

## Best Possible Solution

Keep the issue open and resolve it with a single additive hook/context design: thread resolved `providerUsed`/`modelUsed` or stronger provenance from the agent result through final reply assembly and the delivery decision point, update the public plugin hook types/docs, and add focused regression tests for a fallback-selected model whose outbound text claims a different model. If PR #65149 becomes the canonical path, it should explicitly cover this issue or leave a narrower follow-up for server-side verification/overwrite behavior.
