## Summary

Current main still routes the pre-compaction memoryFlush embedded run without a lane override, so runEmbeddedPiAgent resolves it onto the default main global lane. The related implementation attempt, #53154, was closed unmerged, and there is no current regression coverage proving memoryFlush uses a non-main lane.

## Best Possible Solution

Keep this open and implement focused memoryFlush lane isolation. The likely core fix is to pass an explicit non-main global lane, or add a dedicated compaction lane, when runMemoryFlushIfNeeded invokes runEmbeddedPiAgent; then add regression coverage in src/auto-reply/reply/agent-runner-memory.test.ts that asserts the memory flush call carries that lane. Maintainers should also decide whether same-session messages should bypass the session lane during background memoryFlush or whether only global-lane starvation is in scope.

## Risks / Open Questions

- A lane parameter on the embedded run would isolate the global lane, but same-session interactivity may still be serialized by the session lane; maintainers should decide whether memoryFlush may bypass per-session ordering or must only avoid cross-session/global starvation.
- The compaction timeout is configurable and currently defaults to 900 seconds, so timeout tuning alone does not address the reported lane-blocking behavior.
