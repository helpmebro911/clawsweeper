## Summary

Current main still does not isolate pre-compaction memoryFlush work from the normal embedded-run queue. The memory flush dispatch omits an explicit lane, omitted embedded-run lanes still default to the main global lane, and the related implementation candidate #53154 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best fix belongs in the core auto-reply memory flush dispatch path: route memoryFlush through an explicit non-main background or dedicated compaction lane with its own timeout/concurrency policy, and add regression coverage that fails if the captured runEmbeddedPiAgent call falls back to the default main lane. PR #53154 is useful prior art, but it was closed unmerged, so there is no shipped implementation to close against.

## Risks / Open Questions

- The current default agents.defaults.maxConcurrent value reduces cross-session starvation compared with a single-lane deployment, but it does not isolate memoryFlush and deployments can still configure main-lane concurrency to 1.
- A simple global-lane override may still leave same-session messages behind the session lane; maintainers should decide whether background compaction may bypass per-session serialization or only avoid process-wide main-lane starvation.
