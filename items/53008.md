## Summary

Keep open. Current main still runs pre-compaction memoryFlush through runEmbeddedPiAgent without an explicit non-main lane, and omitted embedded-run lanes still resolve to CommandLane.Main. The related implementation candidate #53154 was closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Keep this issue open. The best fix is to change the core auto-reply memory flush dispatch path so pre-compaction memoryFlush runs on an explicit background or dedicated compaction lane with its own timeout/concurrency policy, then add regression coverage proving the captured runEmbeddedPiAgent call cannot fall back to CommandLane.Main. PR #53154 is useful prior art, but because it was closed unmerged, maintainers should either revive a corrected version or implement the same product goal directly on current main.

## Risks / Open Questions

- Default agents.defaults.maxConcurrent is now 4, which reduces process-wide starvation compared with a single-concurrency deployment, but memoryFlush still has no isolation and deployments may configure main concurrency to 1.
- Moving only the global lane may not solve same-session responsiveness, because runEmbeddedPiAgent also serializes on the session lane; maintainers should decide whether memory flush may bypass, yield, or preempt per-session serialization.
