## Summary

Keep #7648 open. Current main still implements bottom-follow behavior with a bottom-oriented “New messages” affordance, but it does not provide the requested Control UI/WebChat option to disable auto-scroll, keep new responses anchored at the top, or jump to the top of a long response.

## Best Possible Solution

Implement this as an explicit Control UI/WebChat chat-scroll preference rather than closing the request. A good path is to add a persisted UI setting or documented gateway config for chat auto-scroll mode, thread it through app render state into scheduleChatScroll, support at least bottom-follow and off/top behavior, add a Scroll to Top affordance if product wants that alternative, and cover send/stream/refresh behavior in app-scroll and chat-view tests.
