## Summary

Keep #7648 open. Current main still auto-follows/bottom-scrolls Control UI/WebChat chat content, exposes only a downward “New messages” affordance, and has no persisted UI setting or gateway/webchat config option to disable auto-scroll or jump to the top of a long response. The related #44130 is TUI-specific and does not supersede this browser/native WebChat request.

## Best Possible Solution

Keep #7648 open. A conservative fix should add an explicit Control UI/WebChat chat scroll preference, likely persisted in UI settings and optionally backed by documented gateway/webchat config, then thread bottom-follow/off/top behavior through scheduleChatScroll, renderChatView/OpenClawApp state, and OpenClawChatView. Add focused tests for send, streaming, initial load, manual-scroll, refresh, and the visible Scroll to Top or disabled-auto-scroll state. Keep TUI-specific work tracked separately in #44130.
