## Summary

Keep open. Current main has adjacent Active Memory, QMD startup, and session-lock mitigations, but the broad Windows reply-latency regression is not proven fixed and is not cleanly covered by one narrower tracker. Active Memory disabled config is honored, but enabled Active Memory is still a documented blocking pre-reply sub-agent, QMD startup is only partially deferred, and the related open issues cover overlapping but different failure modes.

## Best Possible Solution

Keep this open as a Windows reply-latency investigation. Ask the reporter to retest current main or the next release and attach one slow-turn trace with Windows version, OpenClaw version, install method, channel/UI, model/provider, Active Memory plugin entry and config.enabled values, /status output, gateway logs with hook timing/model-call timing/QMD memory startup timing, and any session-write-lock warnings. If the trace matches #66792, #69182, #70857, or #72015 exactly, close as a duplicate of the best canonical tracker; otherwise fix the implicated owner path directly.
