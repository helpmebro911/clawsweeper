## Summary

Keep #13615 open. Current main has adjacent reactive 429 handling, auth-profile cooldown/failover, inbound gateway/auth throttles, provider usage snapshots, and Codex-plugin account rate-limit readouts, but it still lacks the requested configurable outbound API rate limiting/throttling layer with provider/request/token windows, queues, token budgets, warnings, and an `openclaw limits` CLI. Maintainer discussion already explicitly kept this open, with closed PR #13686 as prior art and #14206 as a narrower Feishu follow-up rather than a replacement.

## Best Possible Solution

Keep #13615 open as the canonical core product/design item. The best path is an opt-in outbound policy layer at generic model-provider, tool/web-search, and channel request boundaries, with config/schema/docs for per-provider, per-model, and global RPM/TPM/day/month windows, queue/backpressure controls, 429 retry/backoff behavior, token budget enforcement, threshold warnings/status reporting, and an `openclaw limits` CLI. Treat closed PR #13686 as prior art and #14206 as a narrower Feishu channel follow-up, not a replacement.
