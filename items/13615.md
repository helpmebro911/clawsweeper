## Summary

Keep open. Current `main` still has only adjacent handling for provider 429 classification, auth-profile cooldown/fallback, SDK retry suppression for long 429/Retry-After responses, and inbound gateway/auth limiters. It does not implement the issue’s requested configurable outbound API rate limiting, per-provider request/token/day windows, queue/backpressure, token budgets, warnings/metrics, or `openclaw limits` CLI. The directly related PR #13686 was closed unmerged, and the issue discussion already contains a Codex review from steipete saying to keep this open.

## Best Possible Solution

Keep #13615 open as a core product/design item. The best path is an opt-in outbound policy layer at generic provider/channel request boundaries, with config/schema/docs, RPM/TPM/day windows, queue/backpressure and 429 retry behavior, token budget enforcement, warning/status reporting, and an `openclaw limits` CLI. Treat closed PR #13686 as useful prior art, not as shipped or superseding work.
