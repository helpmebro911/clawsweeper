## Summary

Keep #13615 open. Current main has adjacent reactive 429 handling, auth-profile cooldown/failover, inbound gateway/auth throttles, and provider usage visibility, but it still does not implement the requested configurable outbound API rate limiting/throttling layer with provider/request/token windows, queues, token budgets, warning thresholds, or an `openclaw limits` CLI.

## Best Possible Solution

Keep #13615 open as the canonical core product/design item. The best path is an opt-in outbound policy layer at generic model-provider, tool/web-search, and channel request boundaries, with config/schema/docs for per-provider/per-model/global RPM, TPM, daily/monthly windows, queue/backpressure controls, 429 retry/backoff behavior, token budget enforcement, threshold warnings/status reporting, and an `openclaw limits` CLI. Treat closed PR #13686 as prior art and #14206 as a narrower Feishu follow-up rather than a replacement.
