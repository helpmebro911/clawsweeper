## Summary

Keep #8130 open. Current main has model fallback, internal attempt tracking, lifecycle events, Control UI fallback status, and verbose-only final fallback notices, but it still lacks proactive retry/failover notifications to the user's active channel and lacks the requested notification config knobs. The issue discussion also explicitly keeps #8130 as the canonical remaining UX gap after related duplicates were closed.

## Best Possible Solution

Keep #8130 open and implement this in the core reply-runner/model-fallback path. Add a configurable retry/failover notice policy, wire failed-attempt and successful fallback transitions to the existing active-channel block reply delivery pipeline, sanitize provider error summaries, support delayed retry notices, and add tests proving non-verbose users receive retry/fallback notices without duplicating verbose or Control UI observability.

## Risks / Open Questions

- Provider/API failure summaries must be sanitized before sending to user channels.

- Delivery needs careful threading and streaming behavior for block replies, preview streaming, silent/system events, and notice delivery failures.
