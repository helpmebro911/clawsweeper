## Summary

Keep open. Current main has useful partial fallback visibility: fallback state is persisted, lifecycle events are emitted, Control UI/TUI can show fallback status, and verbose mode can prepend a final “Model Fallback” notice. It still does not implement the requested proactive active-channel notification during retry/failover, and the existing user-facing notice remains verbose-only and appended to the final reply path rather than sent during the silent retry window.

## Best Possible Solution

Keep #8130 open as the canonical core UX gap. The best implementation is a gateway/reply-runner fallback notification path wired near `runWithModelFallback`: emit sanitized retry/failover notices through the existing reply/block delivery channel during candidate failures and successful fallback selection, make it configurable with a clear default/product decision, preserve existing lifecycle events for UI surfaces, and add regression tests proving notifications work with verbose off without duplicating the existing final verbose notice.

## Risks / Open Questions

- A core implementation should avoid double-notifying users who already enable verbose fallback notices or Control UI/TUI fallback indicators.
- Notification delivery needs to respect existing channel threading, block streaming, silent/system-event policies, and avoid leaking sensitive raw provider error text.
