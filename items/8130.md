## Summary

#8130 should stay open. Current main still has model fallback, attempt tracking, lifecycle events, and verbose-only fallback notices, but it does not proactively notify the user's active channel during retry/failover and it does not implement the requested notification config knobs.

## Best Possible Solution

Keep #8130 open and implement this in the core reply-runner/model-fallback path. Add a configurable retry/failover notice policy, wire failed-attempt and successful fallback transitions into the active-channel block reply delivery pipeline, sanitize provider error summaries, support delayed retry notices, and add tests proving non-verbose users receive retry/fallback notices without duplicating existing verbose or lifecycle observability.
