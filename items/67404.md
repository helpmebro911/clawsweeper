## Summary

Keep this PR open. Current main still lacks a QMD transcript-update listener, so the central behavior is not implemented; however, the PR branch is not ready as-is because its transcript-triggered refresh uses a non-forced `runUpdate("session-delta")` path that can be swallowed by an already-running QMD update, and the proposed `agent-harness` SDK export needs maintainer boundary review against existing runtime event seams.

## Best Possible Solution

Keep this PR open and request revision or maintainer review. The best implementation is to add a QMD session transcript listener only when session export is enabled, filter to the manager's agent session directory, debounce updates, clean up timers/subscriptions on close, and schedule the refresh through a forced or queued update path so in-flight QMD updates get a follow-up export/index pass. Prefer the existing memory-core host foundation or `api.runtime.events.onSessionTranscriptUpdate` seam unless maintainers intentionally want `openclaw/plugin-sdk/agent-harness` expanded, and add focused tests for debounce, agent filtering, cleanup, and the in-flight update race.

## Risks / Open Questions

- Merging the PR as-is can still leave QMD session exports stale when transcript updates arrive after `exportSessions()` has already run in an in-flight QMD update, especially with periodic updates disabled.
- The `agent-harness` export broadens a public trusted-plugin SDK surface; current runtime event and memory-core host seams should be evaluated before adding another way to subscribe to transcript updates.
- I did not run tests because the requested review is read-only and the environment lacks `pnpm`; this does not block the source-level decision to keep the PR open.
