## Summary

Keep open. Current main has useful update-channel/rollback documentation, an unreleased opt-in active-transcript byte trigger, and a lock-scope improvement, but the concrete freeze described in the discussion is still not robustly fixed for default gateway/channel users. The embedded gateway run path still performs whole-file transcript repair and then opens the same active JSONL through Pi's SessionManager without a default size cap, fail-fast recovery, or proof that HTTP/channel loops remain responsive for an oversized active transcript.

## Best Possible Solution

Keep this issue open and implement bounded large-active-transcript recovery in the embedded gateway run path. Add a pre-open size guard before whole-file repair or SessionManager.open, then either stream/yield repair, rotate/compact safely without opening the full transcript on the gateway event loop, or fail fast with an actionable recovery message while keeping HTTP/channel processing responsive. Add regression coverage for an oversized JSONL proving gateway/channel work is not starved, and document the default recovery path plus the opt-in byte threshold on the public docs site.

That is actionable and not just a broad stability complaint.

## Risks / Open Questions

- Closing now would leave default gateway users dependent on manual transcript removal or opt-in compaction/rotation settings after an active JSONL grows large enough to stall the process.
- The opt-in byte threshold may still perform expensive whole-file repair and SessionManager.open work on the oversized transcript before it can rotate or compact it.
- I did not find regression coverage proving an oversized active session JSONL cannot starve gateway HTTP or channel handlers.
