## Summary

Current main still has the remaining source-dependent prompt and tool-schema drift described by this issue. Inbound metadata has been partially stabilized, but `sessions_send` still injects requester/target-specific A2A text through `extraSystemPrompt`, OpenAI Codex Responses still sends the resulting system prompt as top-level `instructions`, and owner-only tools such as `cron` are still removed for non-owner or unknown senders before request tools are built.

## Best Possible Solution

Keep this issue open. The best fix is to move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering message, provenance, or another non-prefix-critical turn structure; then add a regression test that compares same-session OpenAI Codex Responses requests across direct and inter-session turns with the same `prompt_cache_key`. Separately, make an explicit security/product decision for owner-only tools: either keep a stable exposed schema with runtime authorization denial, or document and test a deliberate cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- A stable exposed tool schema for owner-only tools would be security-sensitive; any fix must preserve execution-time authorization denial for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- The current cache observability can detect system prompt and tool-set changes, but the product bug remains because the request payload can still differ by dispatch source for the same target session.
