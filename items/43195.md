## Summary

Keep this PR open. Current main already globalizes the unhandled-rejection handler registry, but it still does not make installUnhandledRejectionHandler() idempotent: repeated calls still attach additional process.on("unhandledRejection") listeners. The PR remains useful if rebased onto current main and covered by a focused listener-count regression test. Security review found no CI, dependency, script, permission, release, or secret-handling changes in the PR context; the meaningful risk is process-level error-handler behavior.

## Best Possible Solution

Keep this PR open, or replace it with an equivalent maintainer patch, by adding a shared/global installed guard for installUnhandledRejectionHandler() on top of the current main structure. The patch should preserve the existing global handler registries, add a focused idempotency regression test using listener counts or controlled process stubbing, and confirm plugin-registered rejection handlers remain shared.

## Risks / Open Questions

- The PR needs a rebase because current main already uses separate globalThis-backed handler Sets for unhandled rejections and uncaught exceptions; the final patch should preserve that structure while adding only the missing installed guard.
- A regression test should assert that repeated installUnhandledRejectionHandler() calls do not increase process.listenerCount("unhandledRejection"), and should avoid leaking process listeners between tests.
- Because this is process-level error handling, maintainers should review that idempotency does not change fatal/config/transient rejection classification or plugin-registered rejection handler behavior.
