## Summary

Keep this PR open. Current main already shares the unhandled-rejection handler registry through globalThis, but installUnhandledRejectionHandler() still lacks an installed guard and still calls process.on("unhandledRejection", ...) every time. The direct entrypoint and runCli() can both call it, so the PR remains a plausible focused fix. Security review found the PR limited to src/infra/unhandled-rejections.ts with no CI, dependency, lockfile, package, publish, or secret-handling surface, but it should be rebased and covered by a listener-count regression test.

## Best Possible Solution

Keep this PR open, or replace it with an equivalent maintainer patch, by adding a shared/global installed guard for installUnhandledRejectionHandler() on top of the current main structure. Preserve the existing global handler registries, add a focused repeated-install listener-count regression test, and verify plugin-registered rejection handlers remain shared.
