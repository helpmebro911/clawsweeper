## Summary

Keep #14747 open. Current main still does not expose a configurable lane wait diagnostic threshold: the queue fallback remains hardcoded at 2,000 ms, cron manual-run enqueue still passes a fixed 5,000 ms warning threshold, and the diagnostics config/schema/docs expose `stuckSessionWarnMs` but no `laneWaitWarnMs` or equivalent. Related PR #16366 targeted the request, but the provided context shows it was closed unmerged and its head commit is not present in this checkout.

## Best Possible Solution

Keep the issue open until a maintainer-approved implementation adds a configurable lane wait diagnostic threshold. A focused fix should add the setting to the config contract (`src/config/types.base.ts`, `src/config/zod-schema.ts`, generated schema/help/labels, Control UI metadata, and public docs), then wire `src/process/command-queue.ts` so per-call `warnAfterMs` still wins while the default can come from config. The cron enqueue path should inherit that default or expose a documented cron-specific threshold, with regression coverage for the configured default and long-running cron queue scenario.
