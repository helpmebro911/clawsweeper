## Summary

Keep #14747 open. Current main still only supports per-call lane wait thresholds and still hardcodes the queue default and cron enqueue threshold; there is no documented `diagnostics.laneWaitWarnMs` or equivalent config/schema surface. The related implementation PR #16366 was closed unmerged, and the latest issue discussion already reached the same keep-open conclusion.

## Best Possible Solution

Keep #14747 open until a focused maintainer-approved patch adds a documented configurable lane wait diagnostic threshold. The likely path is a `diagnostics.laneWaitWarnMs` config field in the exported types, strict zod schema, labels/help/generated metadata, and public docs, then wiring the command queue so explicit per-call `warnAfterMs` still wins while the default comes from config. The cron enqueue path should either inherit that configured default or expose a documented cron-specific threshold, with regression coverage for long-running cron queue waits.
