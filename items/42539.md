## Summary

Keep this issue open. Current main still lacks a configurable Telegram TTS delivery mode that always sends readable text separately before the voice note. Normal auto-TTS final replies keep visible text on the reply payload while adding audio, and Telegram’s media delivery still turns short text into a voice/audio caption, splitting only on caption overflow or special video-note paths.

## Best Possible Solution

Keep this issue open. Implement a Telegram-owned delivery policy or shared messages.tts.delivery: "bundled" | "separate" contract, then update strict config types/schema, Telegram bot and direct send paths, tests, generated/public config docs, and https://docs.openclaw.ai/tools/tts / https://docs.openclaw.ai/channels/telegram. In separate mode, send visible text as a normal Telegram message first, then send the voice note without a caption.

## Risks / Open Questions

- There is a partial TTS-only path after successful block streaming, but it does not provide a configurable default for normal Telegram final replies and does not satisfy the requested always-separate Telegram mode.
- A correct fix needs product/API judgment on whether the knob belongs in shared messages.tts, Telegram channel/account config, or a broader outbound delivery policy.
- Implementation needs to preserve Telegram reply/thread routing, inline buttons, silent delivery, caption-overflow fallback, video-note behavior, and VOICE_MESSAGES_FORBIDDEN fallback behavior.
