## Summary

Current `main` does not implement a configurable Telegram TTS delivery mode for separate text + voice messages. The Telegram send path still uses a caption by default for audio/voice sends and only falls back to a separate text message when the caption exceeds Telegram's limit (or for captionless media like video notes). No `messages.tts.delivery` or equivalent config key exists in the current TTS config type/schema, so the issue remains a valid feature request and should stay open.

## Evidence

- **Telegram send path still captions voice/audio by default:** `sendMessageTelegram` derives `caption` from `splitTelegramCaption(text)` and sends `sendVoice`/`sendAudio` with that caption. It only sets `needsSeparateText` and sends a follow-up text when `followUpText` exists, which happens only when the caption is too long (or when video notes force captionless delivery).
  - file: [extensions/telegram/src/send.ts:842](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/telegram/src/send.ts#L842)
  - command: `nl -ba extensions/telegram/src/send.ts | sed -n '835,990p'`
- **Regression tests confirm split happens only on caption overflow:** The Telegram tests assert that 1100 characters produce media without caption plus a follow-up `sendMessage`, while 1024 characters stay as a single captioned media send.
  - file: [extensions/telegram/src/send.test.ts:780](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/telegram/src/send.test.ts#L780)
  - command: `nl -ba extensions/telegram/src/send.test.ts | sed -n '780,850p'`
- **Bot delivery fallback is overflow-only, not a configurable mode:** The runtime delivery path logs `telegram sendVoice caption too long; resending voice without caption + text separately` only inside the `isCaptionTooLong` error branch for `sendVoice` failures.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:412](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/extensions/telegram/src/bot/delivery.replies.ts#L412)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '400,445p'`
- **Docs describe voice-note sending, but only video notes are documented as always separate:** The Telegram docs say audio can be forced to a voice note with `[[audio_as_voice]]`, and separately state that video notes do not support captions so text is sent separately. There is no documented Telegram TTS delivery-mode option.
  - file: [docs/channels/telegram.md:525](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/docs/channels/telegram.md#L525)
  - command: `nl -ba docs/channels/telegram.md | sed -n '520,565p'`
- **TTS config has no delivery-mode field:** `TtsConfig` and the strict `TtsConfigSchema` expose `auto`, `enabled`, `mode`, `provider`, `summaryModel`, `modelOverrides`, `providers`, `prefsPath`, `maxTextLength`, and `timeoutMs`, but no `delivery`/`bundled`/`separate` option.
  - file: [src/config/types.tts.ts:28](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/src/config/types.tts.ts#L28)
  - command: `nl -ba src/config/types.tts.ts | sed -n '1,60p'; nl -ba src/config/zod-schema.core.ts | sed -n '499,525p'`
- **Changelog matches current behavior: overflow fallback only:** The changelog records prior fixes for splitting long captions and a `sendVoice` caption-overflow fallback, which aligns with the current code but does not indicate a general default-separate delivery mode.
  - file: [CHANGELOG.md:3658](https://github.com/openclaw/openclaw/blob/7d9172d5e093435d37ed3e00e476a4eaf84a252b/CHANGELOG.md#L3658)
