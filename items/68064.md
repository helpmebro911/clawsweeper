## Summary

Keep open. Current main still treats `responseUsage` as session-entry state, and the Discord parent-thread inheritance path passes `ParentSessionKey` for transcript forking without copying the parent session's usage-footer preference or offering a global/thread default config surface.

## Best Possible Solution

Keep this issue open. The best fix is a small session/core change with targeted tests: either add a validated global default such as `session.defaults.responseUsage`, add a Discord thread-specific default, or copy `parentEntry.responseUsage` when creating a new thread session from `ParentSessionKey`. Update the token-use and Discord/config docs to describe the chosen behavior.
