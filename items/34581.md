## Summary

Keep this PR open. Current main still strict-parses authenticated Microsoft Teams webhook bodies with `express.json()` and only handles 413 parser errors specially, so the reported invalid JSON escape path remains unresolved. The PR is a plausible focused fix, but it is stale against later Teams webhook hardening and needs maintainer review before landing.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase in `extensions/msteams/src/monitor.ts` that preserves the current bearer and Bot Framework JWT gates before body parsing, switches authenticated webhook bytes to `express.raw()` or an equivalent raw-body seam with the existing size limit, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated middleware tests for repair success, unrecoverable acknowledgement, adapter call/no-call behavior, ordering, and payload limits.

## Risks / Open Questions

- The PR head is stale relative to current main’s pre-parse bearer/JWT hardening; landing the old middleware shape without adaptation could parse and 200-ack malformed bodies before authentication.
- The current PR tests mirror the private helper instead of exercising the real Express middleware path, so they do not prove parser ordering, payload limits, adapter invocation on repaired payloads, or no adapter invocation on unrecoverable payloads.
- The latest PR regex shown in the review comments uses a first branch that can consume invalid escapes unchanged, so a state-machine repair or better-tested equivalent is safer before merge.
