## Summary

Keep this PR open. Current main still strict-parses authenticated Microsoft Teams Bot Framework webhook bodies with `express.json()` and only handles 413 parser errors, so invalid JSON escape sequences still fail before the adapter route can acknowledge the activity. The PR addresses a plausible live bug, but it needs maintainer rebase/review because it changes security-sensitive webhook body parsing and HTTP 200 acknowledgement behavior after later Teams auth hardening on main.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase in `extensions/msteams/src/monitor.ts` that preserves the current bearer and Bot Framework JWT gates before body parsing, keeps `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated middleware tests for repair success, unrecoverable acknowledgement, adapter invocation/no-invocation, ordering, and payload limits.

## Risks / Open Questions

- Security review: the PR changes an internet-facing webhook parser from strict JSON parsing to raw-body parsing plus repair and HTTP 200 fallback. A stale merge could weaken the current bearer/JWT-before-body-parse ordering or body-size limit if not rebased carefully.
- Returning HTTP 200 for unrecoverable malformed JSON is intentional for Azure backoff avoidance, but it should only happen after Bot Framework authentication succeeds and should avoid invoking the adapter on unparseable activity bodies.
- Regex-based JSON repair has correctness risk around valid doubled backslashes mixed with invalid bare escapes; the PR discussion already identified edge cases that need maintainer review and integrated tests on current main.
