## Summary

Keep this PR open. Current main still detects Microsoft Teams implicit mentions only through `replyToId` plus the sent-message cache; it does not use the existing `conversation.id;messageid=` thread-root parser for mention gating, and it does not record thread-root anchors when sending replies.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused Microsoft Teams plugin fix that computes implicit mentions from either `replyToId` or `conversation.id;messageid=`, records the thread-root anchor when the bot replies, keeps the documented RSC requirement clear, and includes regression tests against the shared production helper.

## Risks / Open Questions

- The PR broadens no-@mention wakeups for Teams thread replies after existing Teams access and allowlist gates; maintainers should still review malformed or surprising Teams payload boundaries before merge.
- The PR reuses `sent-message-cache` for inbound thread-root anchors as well as bot-sent message ids. That is targeted for this bug, but the naming/semantics should be accepted, documented, or renamed to avoid future misuse.
- This was a read-only static review, so targeted Microsoft Teams tests and any available live Teams verification should still run before merge.
