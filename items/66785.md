## Summary

Keep this issue open. Current main partially improves the large model-list path with provider filtering/searchable selection and OpenRouter onboarding is present, but the installer still logs that an existing config skips onboarding and then immediately continues into `openclaw onboard` when a TTY is available. That matches the forced-wizard part of the report.

## Best Possible Solution

Keep this open and fix the installer or Umbrel setup entrypoint so an existing `$OPENCLAW_CONFIG_PATH` or legacy config exits the onboarding path after doctor/dashboard handling instead of continuing to `openclaw onboard`. Add a focused regression test for the existing-config install path, then verify the packaged Umbrel wizard can filter large model catalogs and select OpenRouter.
