## Summary

Keep open. Current main still creates schema v1 backup manifests without per-asset SHA-256 fields, and `backup verify` still performs structural archive checks only. The PR is a focused core backup integrity hardening change, not a ClawHub/plugin item, not implemented on main, and not superseded by the related Windows tar-entry mismatch report.

## Best Possible Solution

Keep the PR open and have a maintainer either rebase/review it or land an equivalent mainline implementation: schema v2 manifests with per-asset SHA-256 checksums generated from archived bytes, fail-closed schema v2 verification, streamed/single-pass checksum computation, docs updates, and regression tests for tampering, v1 compatibility, empty/symlinked directories, and non-ASCII path ordering.
