## Summary

Keep open. The discussion points to a real vLLM tool-calling/configuration incompatibility: one commenter fixed the 400 by restarting vLLM with tool-choice/parser flags, and vLLM's own docs describe those flags as required for auto tool choice. Current main has a vLLM OpenAI-compatible provider and generic escape hatches, but I did not find a current-main fix or vLLM-specific docs that close this exact 400/no-body report conservatively.

## Evidence

- **Issue discussion identifies a vLLM tool-call configuration workaround:** The provided comments show the reporter using vLLM v0.6.3 with direct curl working, another user reporting the same OpenClaw 400/no-body behavior, and BigFaceBoy resolving it by restarting vLLM with `--enable-auto-tool-choice --tool-call-parser hermes`. This aligns with vLLM tool-calling documentation that marks auto tool choice setup as required for auto tool selection. ([docs.vllm.ai](https://docs.vllm.ai/usage/tool_calling/?utm_source=openai))
