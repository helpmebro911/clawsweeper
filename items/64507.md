## Summary

Keep this issue open. Current main still lacks a typed, validated per-agent delivery default, and the message tool still falls back only to explicit arguments or trusted run/session channel context. Adjacent cron and session-delivery work helps captured routes, but does not satisfy the requested agent-level default for fresh programmatic, webchat-routed, or cross-session runs.

## Best Possible Solution

Keep this issue open for a core product/design fix. Add a typed, documented per-agent delivery default contract, likely under agents.list[].delivery using DeliveryContext-like fields rather than only target/channel strings; validate it in the zod schema and config labels/docs; carry it through resolveAgentConfig; and apply it only as a low-precedence fallback when building trusted message tool context. Define precedence clearly: explicit message args first, trusted per-run/session deliveryContext next, then agent default, with accountId/threadId and multi-channel ambiguity covered by tests for sessions_send, cron-created runs, webchat-routed runs, and ordinary channel-triggered sessions.

## Risks / Open Questions

- Adding a broad default target without explicit precedence can silently route messages to the wrong chat, especially with multi-channel, multi-account, thread/topic, and stored-session context interactions.
- Current behavior still leaves fresh programmatic sessions dependent on explicit target strings or previously captured deliveryContext, so the long-ID copy/hallucination failure mode described by the reporter remains plausible.
