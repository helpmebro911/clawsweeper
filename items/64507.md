## Summary

Keep this issue open. Current main still does not provide a typed per-agent delivery default, and the message tool still infers a missing send target only from trusted current run/session channel context. Adjacent session deliveryContext, cron announce, and channel defaultTo behavior help captured routes, but they do not satisfy the requested agent-level default for fresh programmatic, webchat-routed, or cross-session runs.

## Best Possible Solution

Keep this issue open for a core product/design fix. Add a typed, documented per-agent delivery default contract, likely under agents.list[].delivery using DeliveryContext-style fields such as channel, to, accountId, and threadId rather than only target/channel strings. Validate it in the zod schema and config docs, carry it through resolveAgentConfig, and apply it only as the lowest-precedence trusted fallback when building message tool context. Define precedence as explicit message args first, trusted per-run/session deliveryContext next, then agent default, with tests for message tool normalization, sessions_send, sessions.send/webchat-routed runs, cron-created runs, and normal channel-triggered sessions.

## Risks / Open Questions

- Adding a broad default target without clear precedence could silently route messages to the wrong chat in multi-channel, multi-account, DM/group, thread/topic, or stored-session contexts.
- Channel-level defaultTo, cron announce fallback, and session deliveryContext can look similar but do not currently solve message-tool target inference for a fresh agent run with no trusted currentChannelId.
