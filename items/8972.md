## Summary

Keep #8972 open. Current main still resolves Discord guild channel allowlists by exact channel id/name/slug, thread-parent fallback, and the literal `*` wildcard only. A config key like `ws-*` remains a literal key and will not match `ws-task-1`; the category alternative is separately represented by open issue #60582.

## Best Possible Solution

Keep #8972 open. Implement a documented, opt-in Discord guild channel pattern form such as `ws-*`, preserving exact-match and thread-parent precedence, keeping literal `*` as the final fallback, updating schema/types/docs/status/audit behavior, and adding regression tests proving dynamically created matching Discord channels are accepted without per-channel config edits. Track category inheritance through #60582 unless maintainers choose to merge both designs.

## Risks / Open Questions

- Adding glob matching changes Discord access-control semantics; exact id/name/slug matches, normalized matches, thread-parent fallback, pattern matches, and literal `*` fallback need explicit precedence and regression tests.
- Implementing this in the shared matcher could affect other channel/config consumers, so maintainers should either add an explicit opt-in matcher mode or scope the pattern logic to Discord guild channel allowlists.
- The category-based alternative overlaps #60582 and should remain separate unless maintainers intentionally combine the two designs.
