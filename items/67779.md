## Summary

Keep this issue open. Current main still has a plausible WebChat live-update mismatch for mixed-case non-default agent session links: the UI keeps the raw `?session=agent:DB_Dude:main` value, while the gateway canonicalizes session keys to lowercase and emits live/final events with the canonical key. The UI then compares session keys with exact string equality, so the completed response can be ignored until a refresh reloads persisted history. The related #39743 is broader and closed stale, not a canonical active fix for this specific case.

## Best Possible Solution

Normalize or canonicalize session keys consistently at the Control UI boundary, or compare canonical session keys in the chat/session event handlers. Add a focused UI regression test using a mixed-case non-default agent deep link such as `agent:DB_Dude:main` receiving lowercase gateway `chat` and `session.message` events, while keeping gateway storage/broadcast behavior canonical.
