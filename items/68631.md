## Summary

Keep open. Current main already has tool-name deny policy, including per-agent `agents.list[].tools.deny`, and the matcher applies deny before allow. The requested path-scoped `path.allow` / `path.deny` policy is still missing: config types and strict Zod schema expose only `tools.fs.workspaceOnly`, and runtime write/edit guards are workspace-wide rather than glob-scoped.

## Best Possible Solution

Keep the issue open but narrow the maintainer path: acknowledge that `agents.list[].tools.deny` is already implemented, then implement the remaining core path policy as a typed, validated config surface, likely under the existing `tools.fs` / `agents.list[].tools.fs` guard area. Enforcement should canonicalize targets and apply allow/deny globs to write/edit/apply_patch and any supported exec/bash write paths, with tests covering deny precedence, backwards-compatible configs, malformed pattern validation, and agent-specific overrides.
