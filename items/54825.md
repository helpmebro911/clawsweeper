## Summary

Keep this issue open. Current main still turns provider billing/rate-limit failures into ordinary final reply text, WhatsApp has no suppress/custom/owner-only error policy surface, and Telegram’s existing errorPolicy only handles Telegram delivery errors rather than the provider-failure final-reply path described here.

## Best Possible Solution

Implement a shared outbound provider/runtime error-reply policy before channel delivery. A likely maintainer path is to classify final provider/runtime failures on ReplyPayload or the reply pipeline, then add per-channel/account/group/topic controls for normal reply, silent suppression, custom text, and owner-only routing. Cover WhatsApp groups and DMs, Telegram groups/topics/DMs, billing and rate-limit provider failures, and unchanged tool-error behavior while preserving full diagnostics in logs and operator surfaces.

## Risks / Open Questions

- A silent or group-wide default could hide owner-actionable billing, auth, quota, or provider setup failures unless logs, Control UI diagnostics, and owner/admin routing remain explicit.
- The existing messages.suppressToolErrors setting is narrower than provider/runtime failures; reusing it without new classification could suppress useful tool-error behavior or still miss final provider-failure replies.
- Telegram’s docs currently describe reply/silent values while the schema implements always/once/silent, so any implementation should align docs and schema rather than adding a second conflicting policy model.
