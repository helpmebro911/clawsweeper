## Summary

Keep this issue open. Current main still has the reported Discord inbound worker timeout behavior and still lacks the requested diagnostic details in the user-facing timeout reply, timeout log, run/session tracking, and queue/status surface. The prior ClawSweeper comment correctly identified the same gap, and the current checkout at 348728c28c1ea9ae7c5824350baba35ee08a275e has not made that issue obsolete.

## Best Possible Solution

Keep this open as Discord plugin runtime diagnostics and hardening work. The likely fix belongs in `extensions/discord/src/monitor/inbound-worker.ts` and `extensions/discord/src/monitor/message-handler.process.ts`: track safe account/session/timeout-source details, wire `onAgentRunStart` so timeout handling can include the run id when available, capture best-known stage/progress from existing callbacks, enrich the timeout log and fallback reply, and add focused queue tests. Add a small generic queue-depth/status API only if maintainers want queue depth in the timeout report.
