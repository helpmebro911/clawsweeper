## Summary

Keep open. Current main still has the Discord inbound worker timeout behavior described by the reporter: the timeout is configurable and tested, but the Discord fallback reply remains the generic `Discord inbound worker timed out.` and logs only include channel/message context. The related ClawSweeper items cover narrower listener-timeout, websocket, and attachment-wedge fixes, not this newer diagnostics/unblockability request against v2026.4.24.

## Best Possible Solution

Keep this open as Discord plugin diagnostics/runtime hardening. The likely fix belongs around `extensions/discord/src/monitor/inbound-worker.ts`, `message-handler.process.ts`, and the queue/status helpers: capture account id, queue key/session key, timeout source, run id when the agent run starts, and best-known stage state; include safe context in the timeout log and Discord fallback reply; add focused queue tests for the enriched timeout output; consider a small production queue-depth/status API only if maintainers want queue depth in the timeout report.
