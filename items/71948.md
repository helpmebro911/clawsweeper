## Summary

Keep open. Current main still sends the exact generic Discord timeout reply after the inbound worker timeout and logs only limited channel/message context. Prior related fixes cover listener blocking, attachment stalls, and gateway lifecycle churn, but this report asks for timeout diagnostics and stuck-run/session visibility that are not implemented on main.

## Best Possible Solution

Keep this issue open and implement focused Discord inbound timeout diagnostics: capture the agent run id via the existing reply options hook, carry accountId/agentId/sessionKey/queueKey and timeout config source into the timeout log/reply, expose bounded queue depth or wait-state data where feasible, and add regression coverage in the Discord inbound worker queue tests. Investigate the underlying session-lock/qmd/subagent/gateway stalls separately rather than closing this as covered by older listener or attachment fixes.
