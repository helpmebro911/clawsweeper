## Summary

Keep open. Current main has already fixed the older Feishu mention-prefixed command bug, but it still does not implement this PR's requested bare slash-command bypass: Feishu group messages with `requireMention=true` return before command probing when the bot is not mentioned. The PR is a plausible focused fix and the discussion shows the broadcast scoping review gap was addressed, but it needs maintainer review, tests, and clear authorization semantics before landing.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is a Feishu-owned mention-gate fix that computes normalized command presence and effective command authorization before the group mention return, bypasses only authorized command-only traffic, carries the bypass state into broadcast active-agent selection, and adds focused tests for single-agent, broadcast, and unauthorized/no-allowFrom requireMention group cases.

## Risks / Open Questions

- The PR should not be merged as a stale drive-by without tests: the provided diff changes production Feishu behavior but adds no regression coverage for single-agent or broadcast requireMention groups.
- The proposed bypass is based on command detection before the later authorization value is resolved. Maintainers should ensure Feishu bypasses mention requirements only for authorized command traffic, consistent with the shared channel contract and slash-command security docs.
- Related Feishu group-command work remains open around no-allowFrom command authorization (#49915/#49958), so this PR may need to be coordinated with that path rather than treated as a complete fix for every Feishu slash-command failure.
