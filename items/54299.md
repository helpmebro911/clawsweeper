## Summary

Current main still matches the reported Google/Gemini CLI OAuth behavior: an env-provided client ID may be used without a matching client secret, token exchange only sends client_secret when present, and the browser callback/listener remains fixed to localhost:8085. The manual fallback is useful mitigation, but it does not make the redirect port configurable or dynamic.

## Best Possible Solution

Keep this open for the bundled Google plugin owner. Make the Gemini CLI OAuth client-secret contract explicit in code, tests, and docs: reject or warn when an env client ID is provided without a matching secret if confidential-client mode is intended, or document and test public-client mode as intentional. Separately, add supported redirect URI/port configuration if compatible with Google OAuth registration, or document and regression-test the fixed localhost:8085 plus manual paste fallback as the intended behavior.

## Risks / Open Questions

- The issue's attacker-impact claim is stronger than the source alone proves because this flow uses PKCE and changing OAuth environment variables implies local process/environment control; the actionable problem is the ambiguous and untested client-secret contract.
- A dynamic port may be constrained by registered Google OAuth redirect URIs, so the practical fix may need explicit redirect URI/port configuration or documented/tested manual fallback rather than server.listen(0).
- The existing manual paste fallback reduces breakage when port 8085 is busy, but users still have no documented supported way to choose another redirect port.
