## Summary

Current main still has both reported Gemini CLI OAuth surfaces: an environment client id can resolve without a matching client secret, token exchange only sends `client_secret` when present, and the redirect URI/listener remain fixed to localhost:8085. The existing manual fallback reduces port-conflict breakage but does not implement configurable or dynamic redirect-port behavior.

## Best Possible Solution

Keep this issue open for the bundled Google plugin owner. The best fix is to make the Gemini CLI OAuth client-secret contract explicit: either fail or warn when an env client id is provided without a matching secret, or document and test supported public-client mode. Separately, add a supported redirect URI/port configuration path or document the existing manual paste fallback as the intended recovery for port conflicts, with focused tests around listener `EADDRINUSE`.
