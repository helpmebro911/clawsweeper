## Summary

Keep open. Current `main` still rejects any Feishu external key containing `".."`, so the reported image/file-key bug remains plausible and is still present in the shipped code path, but this PR also has unresolved review concerns because its latest patch would allow the literal `".."` token through validation without adding coverage for that case.

## Evidence

- **Current main still blocks any key containing '..':** `normalizeFeishuExternalKey()` on current `main` returns `undefined` for any key containing `/`, `\\`, or `..`, so legitimate opaque Feishu keys with `..` are still rejected before any download attempt.
  - file: [extensions/feishu/src/external-keys.ts:15](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/extensions/feishu/src/external-keys.ts#L15)
  - command: `nl -ba extensions/feishu/src/external-keys.ts | sed -n '1,20p'`
  - sha: [459d27707648](https://github.com/openclaw/openclaw/commit/459d277076488d5833a181cca7d8bf5c0f9696dc)
- **Rejected keys are dropped from the inbound media path:** `parseMediaKeys()` normalizes `image_key` and `file_key`, and `resolveFeishuMediaList()` returns early when both normalize to falsy. That means a valid Feishu media key containing `..` would be discarded and media analysis would not run.
  - file: [extensions/feishu/src/bot-content.ts:294](https://github.com/openclaw/openclaw/blob/459d277076488d5833a181cca7d8bf5c0f9696dc/extensions/feishu/src/bot-content.ts#L294)
