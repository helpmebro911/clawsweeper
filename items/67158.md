## Summary

Keep open. Current main has OpenAI Codex catalog/model-resolution work for gpt-5.4/gpt-5.5, but it does not prove or implement a fix for the ChatGPT/Codex account entitlement rejection. The latest discussion also reports the same runtime rejection on OpenClaw 2026.4.24 with gpt-5.5, so this remains active provider-boundary behavior to handle or document.

## Best Possible Solution

Keep this issue open. The best maintainer path is an OpenAI Codex provider-boundary fix: detect the backend unsupported-ChatGPT-account model error, classify it as an entitlement/model-unavailable condition for fallback and status, avoid presenting synthesized rows as definitely runnable when current account probing says otherwise, and update OpenAI provider docs/model-list diagnostics. Close only after current-main live proof or regression coverage shows the rejected-model path degrades correctly, or after upstream support is restored and verified.
