## Summary

Keep this PR open. Current main still does not support `explicit` as an active-memory chat type: the manifest schema, runtime config types, normalization filter, resolver, docs, and tests remain limited to direct/group/channel. The PR is a focused implementation candidate, but the current diff should be adjusted before landing because its broad `sessionKey.includes(...)` ordering can misclassify explicit session IDs that contain other chat-type marker text.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Update the resolver to parse the structured `agent:<agent>:explicit:<sessionId>` prefix before broad marker checks, or parse session-key segments directly, then land the equivalent active-memory patch with schema/types/normalization/docs updates and regression coverage for explicit sessions, including marker-containing explicit session IDs.

## Risks / Open Questions

- Landing the current diff as-is could classify `agent:<agent>:explicit:<sessionId>` as group/channel/direct when the user-supplied session ID contains `:group:`, `:channel:`, `:direct:`, or `:dm:`. That would make explicit-only configs unreliable and could run or skip active memory under the wrong chat-type gate.
- Closing this PR now would leave the linked active-memory behavior unresolved on current main, even though the related issue is already closed.
