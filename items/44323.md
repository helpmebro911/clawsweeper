## Summary

Keep PR #44323 open. Current main has memory-search doctor diagnostics, but it does not implement the PR’s static validation goal: `memorySearch.provider` is still an optional string, `validateConfigObjectRaw` has no memory-search provider validator, and explicit unknown providers still reach runtime provider creation before failing. The PR should not be merged as-is because the branch is XL and includes unrelated cron/agent lifecycle/watchdog work plus review-discussed false-positive risks.

## Best Possible Solution

Keep the item open, but ask for a focused refresh: drop unrelated cron/infra work, implement a narrow memory-search validation or doctor-diagnostic improvement that respects provider defaults and auth fallback, validate defaults and per-agent overrides where the runtime has enough plugin metadata, and add regression tests for disabled memory search, QMD, `auto`, Ollama defaults, env/auth-store credentials, gateway-ready CLI-missing-key cases, and unknown explicit providers.

## Risks / Open Questions

- A static validator must not reject supported env/auth-profile/provider-config credential fallback, QMD backend mode, gateway-ready deployments, `provider: "auto"`, or Ollama’s default endpoint/model behavior.
- Provider ids are plugin-registered and third-party plugins exist, so provider-existence validation needs a plugin-aware path or a clear runtime/doctor diagnostic boundary rather than a closed core enum.
- The current PR branch contains unrelated cron and agent lifecycle files, increasing review and regression risk beyond the stated memorySearch validation change.
