## Summary

Keep open. Current main still has the reported gap: `/tools` builds its reply from `resolveEffectiveToolInventory()` only, while the runtime run path separately materializes and policy-filters MCP server tools before adding them to the callable tool set. The PR is a plausible fix direction, but it should be narrowed and validated before merge.

## Best Possible Solution

Keep the PR open, but narrow it to the MCP inventory bug. The best implementation is to make `/tools` append MCP tools from an existing session MCP runtime without starting a new runtime for a read-only command, materialize them with the same reserved-name collision behavior as `attempt.ts`, pass them through `applyFinalEffectiveToolPolicy` with matching session/channel/sender context, add an `mcp` inventory source/group, and add focused tests for compact and verbose `/tools` output. Split the plugin install peer-dependency changes into a separate PR or drop them from this one.
