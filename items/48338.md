## Summary

Keep open. Current main partially improves generic missing model metadata defaults, but the reported custom-provider setup UX is still unresolved: the custom provider onboarding path still writes an explicit 16,000-token contextWindow for non-Azure custom models, the interactive flow does not ask for or validate contextWindow/maxTokens, and the narrower PR that proposes metadata prompts (#43493) is still open rather than merged.

## Evidence

- **Custom provider onboarding still writes 16K:** Non-Azure custom provider setup uses DEFAULT_CONTEXT_WINDOW, which is defined as CONTEXT_WINDOW_HARD_MIN_TOKENS, and writes that value into new custom model entries as contextWindow.
  - file: [src/commands/onboard-custom-config.ts:16](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/commands/onboard-custom-config.ts#L16)

  - sha: [d228463120a5](https://github.com/openclaw/openclaw/commit/d228463120a5057f7ccf9a5c31af62fe4a922088)
- **Behavior is covered as current expected behavior:** The test suite explicitly expects newly added custom models to use the hard-min context window, so this is not just a stale code path.
  - file: [src/commands/onboard-custom-config.test.ts:107](https://github.com/openclaw/openclaw/blob/d228463120a5057f7ccf9a5c31af62fe4a922088/src/commands/onboard-custom-config.test.ts#L107)

- **Related implementation remains open:** The provided GitHub context shows PR #43493 is still open and directly targets this remaining UX by adding custom-provider metadata prompts for contextWindow, maxTokens, and related fields after provider setup.
