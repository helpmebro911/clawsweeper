## Summary

Keep this PR open. Current main still starts the cron timeout before the isolated execution path reaches the embedded runner lanes, and main has no `onExecutionStart`/`onLaneAcquired` hook or tests proving lane queue wait is excluded. The PR is a plausible focused fix, but it is stale against the current `run-executor` split and must address the unresolved CLI-provider timeout-bypass review before merge.

## Best Possible Solution

Keep this PR or a canonical replacement open. The best maintainer path is a narrow cron timeout fix rebased onto current main that defines a shared execution-start boundary for isolated cron runs, excludes embedded lane queue wait from the execution timeout budget, preserves equivalent timeout enforcement for CLI-provider runs, retains an explicit hard backstop if desired, and adds regression coverage for embedded queue wait, CLI-provider execution, manual `cron.run`, and startup catch-up.

## Risks / Open Questions

- The PR as written is stale against current main's `src/cron/isolated-agent/run-executor.ts` split and cannot be reviewed as a direct clean merge without rebasing/adaptation.
- The callback-only timer design can disable the outer cron timeout for CLI-provider isolated runs unless the callback is fired from a shared execution-start boundary or the CLI path gets equivalent enforcement.
- Deferring the only timeout until embedded lane acquisition changes the hard wall-clock safety contract; maintainers should decide whether to keep a separate backstop for queue deadlocks or pre-run hangs.
- The PR does not introduce dependency, workflow, lockfile, or release-script changes, but it does touch timeout enforcement, which is a scheduler availability/security-sensitive path.
