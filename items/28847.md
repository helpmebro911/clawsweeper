## Summary

Keep open. Current main has meaningful adjacent behavior: multiple auth profiles, per-profile cooldown/failover, Gemini RESOURCE_EXHAUSTED classification as rate_limit, and a generic API-key rotation helper. It still does not implement the issue’s central request: first-class provider key pools with Gemini project/quota-bucket metadata, bucket-level cooldown fanout, and selection away from every key in the exhausted project bucket.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add explicit quotaBucket/quotaGroup metadata for auth profiles or provider-owned external profiles, then update auth-profile failure handling, profile ordering, and runtime model auth selection so a Gemini project quota failure cools and skips every profile in the same bucket before trying another bucket. Add Gemini multi-project tests and docs, or make an explicit product decision to expose a stable plugin/sidecar API for shared auth-health state.
