## Summary

Keep #70879 open. Current main still sends Feishu text/card messages with single direct SDK calls and no retry/backoff around message create/reply/card send failures. Existing Feishu backoff handling is limited to typing indicator reactions, and the current reply-fallback test explicitly expects a thrown Feishu rate-limit error to be re-thrown rather than retried.

## Best Possible Solution

Keep the issue open and implement bounded retry/backoff inside the Feishu plugin send surface, likely around `extensions/feishu/src/send.ts` create/reply/card calls. Reuse the existing plugin SDK retry helpers with a strict Feishu-specific predicate for retryable SDK/Axios/non-throwing response shapes, add tests for codes such as 11232, 2200, 429, and existing Feishu rate-limit codes, and consider idempotency/duplicate-delivery behavior before retrying non-idempotent sends.
