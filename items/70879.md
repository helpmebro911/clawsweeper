## Summary

Keep #70879 open. Current main still lacks bounded retry/backoff for Feishu outbound message sends: text, card, and media paths call the Feishu SDK once and fail on non-zero responses or thrown SDK errors. Existing Feishu backoff handling is scoped to typing indicators, not delivery reliability for ordinary message sends.

## Best Possible Solution

Keep the issue open and implement a Feishu-owned bounded retry layer around outbound create/reply/media send operations in `extensions/feishu`. Centralize retryable error classification for HTTP 429 and confirmed Feishu flow-control/rate-limit codes, preserve withdrawn-reply fallback semantics, avoid unsafe retries for ambiguous post-delivery failures, and add focused tests for non-throwing API responses plus thrown SDK/Axios error shapes.

## Risks / Open Questions

- A naive retry can duplicate Feishu messages if the original request succeeded but the SDK response was lost; retry should be bounded and restricted to well-classified rate-limit or pre-delivery failures.
- Feishu code 2200 is broad/internal, so maintainers should verify Feishu/Lark error taxonomy before treating it as always retryable.
