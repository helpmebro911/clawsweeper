## Summary

Keep this PR open. Current main still does not implement Telegram inline buffer/filename/contentType delivery for message sends, and the PR remains a plausible implementation candidate, but it needs a refresh for the current bundled plugin layout and a stronger pre-decode size guard before it can be merged safely.

## Best Possible Solution

Keep the PR open, request a refresh against current main, and either coordinate with #63578 or fold the generic hydration change into one canonical patch. The mergeable fix should update `src/infra/outbound/message-action-params.ts` for `send` hydration, update `extensions/telegram/src/action-runtime.ts` and `extensions/telegram/src/send.ts` to forward and deliver inline buffers with filename/contentType, add focused regression tests in those current files, and enforce a base64 length limit before decoding.

## Risks / Open Questions

- The PR’s proposed inline buffer path still decodes base64 with `Buffer.from(..., "base64")` before any pre-decode length check; the later decoded-byte check does not avoid large allocation/CPU work. This matches the Aisle/Greptile review concern and should be fixed before merge.
- The patch is structurally stale for current main because Telegram moved to `extensions/telegram/src/*`; applying old core-path changes could miss the live runtime path or conflict with plugin ownership.
- #63578 may solve part of the generic hydration problem, so maintainers should avoid merging two divergent implementations of the shared `send` hydration layer.
