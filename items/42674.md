## Summary

Keep this PR open. Current main still exposes inline attachment fields on the shared message send tool, but the active outbound and Telegram plugin paths do not hydrate plain send buffers, do not treat buffer-only sends as media, and do not forward inline buffer/filename/contentType into Telegram transport. The PR is stale against the pluginized Telegram layout and has a remaining pre-decode base64 allocation concern, but it is still a relevant implementation candidate rather than something already fixed or fully superseded.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. A mergeable version should be refreshed onto current main, move Telegram-specific changes into `extensions/telegram/src/action-runtime.ts` and `extensions/telegram/src/send.ts`, coordinate the generic `send` media hydration with #63578, add focused regression tests for buffer-only sends and data-URL normalization, and enforce a pre-decode base64 size bound before `Buffer.from`.
