## Summary

Keep this issue open. Current main still routes `openclaw models set` through the generic config replacement path, and the config writer still blocks a raw more-than-half size drop with `CONFIG_WRITE_REJECTED` unless a caller explicitly opts into destructive writes. I found no current regression test or command-specific policy that accepts legitimate default-model updates while preserving the clobber guard.

## Best Possible Solution

Keep the issue open. Add a regression test that exercises a real `openclaw models set` write against a valid config that legitimately shrinks below the current raw-byte threshold, then fix the models/config mutation path with a narrow semantic policy for intended `agents.defaults.model.primary` updates while preserving the size-drop guard for actual clobbers. Coordinate with any related size-drop reports so the generic false positive is addressed instead of only this command.
