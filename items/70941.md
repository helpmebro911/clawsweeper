## Summary

Keep open. Current main still has the reported failure surface: `openclaw models set` updates the model by writing a full replacement config, while the shared writer still rejects large size drops with `CONFIG_WRITE_REJECTED`. The issue is recent, has no protected labels, and the related context does not provide a canonical duplicate that covers this exact model-set/size-drop path.

## Best Possible Solution

Keep this open and fix it in the model/config mutation path. The likely maintainer direction is to make `openclaw models set` perform a narrow, source-preserving config mutation that cannot be mistaken for a destructive clobber, or refine the size-drop guard so legitimate single-field model changes are allowed while true root truncations stay blocked. Add a regression test using a large authored config that would currently trip `CONFIG_WRITE_REJECTED`, then verify `models set` preserves unrelated config and writes the intended `agents.defaults.model.primary`/allowlist change.
