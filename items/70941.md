## Summary

Keep #70941 open. The issue is recent and describes a concrete CLI/write-safety failure. Current main still sends `openclaw models set` through the generic config replacement path, and that writer still blocks any normalized output under half the previous raw file size as `CONFIG_WRITE_REJECTED`. I found ordinary model-set tests and destructive-write guard tests, but no regression test or code path proving this specific model-only size-drop false positive is fixed.

## Best Possible Solution

Keep #70941 open and reproduce it with the reporter's config if available, or with a synthetic oversized valid `openclaw.json` that shrinks after normal serialization. If confirmed, add a regression test around `modelsSetCommand` or the shared model `updateConfig` path, then refine the write-safety heuristic so narrow validated model updates are not rejected solely because formatting or normalization reduces raw byte size, while preserving real destructive-clobber protection.
