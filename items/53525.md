## Summary

Keep open. Current main has generic empty-error retry and TUI fallback behavior, but I did not find the requested Qianfan/Baidu error normalization. Qianfan still routes through the generic OpenAI-compatible transport, and no code or test extracts Baidu top-level error fields such as `code`/`message` or `error_code`/`error_msg` into the assistant `errorMessage`.

## Best Possible Solution

Keep this open and fix it in the Qianfan provider plugin or a narrow shared OpenAI-compatible error-normalization seam. The fix should preserve Qianfan/Baidu top-level `code`/`message`/`requestId` and legacy `error_code`/`error_msg` from non-2xx responses and stream error events into `errorMessage`, with regression coverage for the Qianfan shape. The existing TUI fallback and empty-error retry should remain fallback behavior, not the primary fix.
