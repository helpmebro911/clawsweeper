## Summary

Keep #29100 open. Current main still lacks a supported pricing overlay for auto-discovered Amazon Bedrock models: the plugin-owned discovery schema is strict and has no costs field, the legacy Bedrock discovery type/schema also have no costs field, discovered foundation models are assigned zero DEFAULT_COST, inference profiles only inherit that base/default cost, and docs still tie pricing config to explicit provider model entries. The linked implementation attempt, PR #31160, is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep #29100 open. The best implementation is a backward-compatible pricing overlay, preferably either plugin-owned Bedrock discovery costs or a general discovered-model cost overlay, with manifest schema, TS types, zod/help/UI metadata, config migration/doctor behavior where relevant, deterministic cache-key inclusion, application to both foundation models and inference profiles, focused tests, and docs updates for Bedrock discovery plus token-use/cost surfaces. The fix should also decide and test how estimated costs appear for aws-sdk/IAM Bedrock in /status, /usage full, /usage cost, and transcript-backed summaries.

## Risks / Open Questions

- The remaining product design choice is whether pricing should be Bedrock-plugin-owned under plugins.entries.amazon-bedrock.config.discovery.costs or a general models.costs overlay that can patch any discovered model by provider/model id.
- The user-facing display policy for aws-sdk-authenticated Bedrock needs an explicit decision; adding metadata alone may populate session/log cost calculations but not necessarily /status or /usage full footers under current auth gating.
