## Summary

Keep #29100 open. Current main still has no supported way to attach cost metadata to auto-discovered Amazon Bedrock models: the plugin discovery schema and legacy config schema lack a costs field, discovered foundation models still receive zero DEFAULT_COST, inference profiles only inherit that zero/default cost, and the docs still describe pricing as explicit provider model metadata only. The related implementation PR #31160 is closed and unmerged in the provided GitHub context, and the relevant current-main files have not changed since the prior keep-open review.

## Best Possible Solution

Keep #29100 open. The best implementation is a backward-compatible pricing overlay for auto-discovered Bedrock models, preferably plugin-owned unless maintainers choose a general models.costs overlay. It should update the manifest schema, TS/zod/config metadata, deterministic discovery caching, foundation-model and inference-profile cost resolution, focused tests, and docs for https://docs.openclaw.ai/providers/bedrock plus https://docs.openclaw.ai/reference/token-use. It should also explicitly decide how estimated costs appear for aws-sdk/IAM Bedrock in /status, /usage full, /usage cost, and transcript-backed summaries.

## Risks / Open Questions

- The adjacent cached-pricing system does not satisfy this request: it collects configured model refs and cached lookup does not normalize amazon-bedrock into a Bedrock pricing catalog key, while the issue asks for an operator-controlled overlay for discovered models.
- A complete fix must decide whether pricing belongs under plugins.entries.amazon-bedrock.config.discovery.costs or a general discovered-model overlay, then apply it consistently to foundation models and inference profiles.
- Even after metadata exists, /status and per-response /usage cost visibility for aws-sdk/IAM Bedrock may remain hidden unless maintainers intentionally change the current non-api-key display policy.
