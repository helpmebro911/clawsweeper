## Summary

Keep this PR open. Current main still limits `deleteAfterRun` deletion to `schedule.kind === "at"`, while this PR is a narrow implementation candidate for the still-open linked bug. The earlier overlapping PR is closed unmerged and is not a shipped replacement.

## Best Possible Solution

Keep this PR open as the focused implementation candidate. Maintainers should review or rebase it, decide the durable-audit and user-facing wording questions separately, then land this guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention before closing the linked bug.

## Risks / Open Questions

- Maintainers still need to decide whether recurring self-delete is intended product behavior versus keeping `deleteAfterRun` semantically one-shot-only; current schema and tools allow the flag generally, but UI copy still describes it as best for one-shot reminders.
- Bot review comments surfaced a broader durable-audit/replay hardening concern for self-deleting jobs. That behavior predates this PR for `at` jobs, but broadening deletion to recurring schedules may deserve a separate storage/audit follow-up.
- I did not run tests because this was explicitly a read-only review. The PR author reported targeted cron tests passing, and the patch adds focused regression coverage.
