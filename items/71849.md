## Summary

Keep #71849 open. Current main still routes realtime phone-call context questions through the shared `openclaw_agent_consult` embedded-agent path, does not define `realtime.fastContext`, and ignores non-final transcript chunks in the voice-call realtime handler. The maintainer comment already reached the same conclusion, and related PR #72237 is only an open consult tuning PR, not the requested bounded memory/session-context hot path.

## Best Possible Solution

Implement this in the bundled `voice-call` plugin: add a default-off `realtime.fastContext` config block, use the active memory search manager or memory-core search primitive under a short hard deadline, accumulate/normalize partial user transcript chunks for trigger detection, inject either top snippets or an explicit miss/timeout result into the realtime session, keep `openclaw_agent_consult` as fallback, and update `docs/plugins/voice-call.md` plus focused config/runtime/realtime-handler tests.
