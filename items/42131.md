## Summary

Keep this PR open. Current main still does exact-case trusted safe-bin directory matching while doctor lowercases configured safe-bin entries before resolving and checking them, so the reported false warning remains plausible. The PR is a focused fix candidate, but its latest tmpdir-based filesystem probe still has an unresolved security-sensitive mixed-volume case-sensitivity concern, so it should be reviewed or revised rather than closed as cleanup.

## Best Possible Solution

Keep this PR open for maintainer review or a small revision. The best fix is a shared, testable path-normalization helper that compares trusted-dir entries and resolved executable dirs consistently on genuinely case-insensitive paths without collapsing distinct paths on case-sensitive or mixed-sensitivity volumes. Add deterministic regression coverage for doctor/runtime safe-bin trusted-dir matching with differing path case, plus a trust-boundary test showing a mixed-case sibling is not accepted on case-sensitive paths.

## Risks / Open Questions

- The PR's latest implementation uses one global `os.tmpdir()` case-sensitivity probe. If temp is on a case-insensitive volume but `safeBinTrustedDirs` points at a case-sensitive volume, lowercasing can collapse distinct trusted and untrusted sibling directories and weaken the safe-bin trust boundary.
- The added tests mirror the runtime probe and mostly validate the current branch behavior on the host filesystem; they do not deterministically cover mixed-volume case sensitivity or prove a case-sensitive sibling path remains untrusted.
